# 系统设计题：让你设计一个终端 AI Agent，你会选 REPL 循环还是应用架构，为什么？

这是一道系统设计题，没有唯一正确答案，但有明显更成熟的答案。

面试官想听到的不是技术细节，而是你对「这个系统的本质是什么」的判断。

---

## 打印机模型：为什么直觉上合理

最自然的终端工具设计是这样的：

```
while (true) {
  const input = await readline()
  const result = await agent.process(input)
  console.log(result)
}
```

读输入、处理、打印输出，再等下一条。这是 shell、Python REPL、大多数 CLI 工具的做法。每一轮交互都是独立的，不需要跨轮次共享状态。

对简单工具来说，这完全够用。

---

## 但 Claude Code 需要同时处理哪些事情

想象用 Claude Code 做一个代码重构任务，屏幕上可能同时存在：

- 主对话流：Claude 的消息和工具调用结果
- 任务面板：正在运行的 50 个子任务
- 权限请求：等待用户批准的文件修改
- 状态栏：当前权限模式、模型名称、token 计数
- 团队视图：4 个并行 teammate 的进度 spinner
- 通知：某个子 agent 发来的消息
- Bridge 连接：远端 claude.ai viewer 的状态

这七种东西同时存在，**它们的状态相互影响**。批准了权限请求，任务面板里的任务就会继续，状态栏里的 token 计数跟着更新，通知区里可能出现新消息。

打印机模型里，每个子系统各自维护自己的状态，靠事件总线或 props 传递同步。这条路走到后面会是什么样？每个子系统都有一份「自己认为的真相」，不一致时你不知道信哪一个。

## 应用架构的核心决定：共享现实

Claude Code 的 `App.tsx` 用三层 Provider 回答了这个问题：

```tsx
export function App({ getFpsMetrics, stats, initialState, children }) {
  return (
    <FpsMetricsProvider getFpsMetrics={getFpsMetrics}>
      <StatsProvider store={stats}>
        <AppStateProvider
          initialState={initialState}
          onChangeAppState={onChangeAppState}
        >
          {children}
        </AppStateProvider>
      </StatsProvider>
    </FpsMetricsProvider>
  )
}
```

在 `children`（所有 UI）之前，先装配了三个基础设施层：

1. **FPS 观测**：以 getter 形式提供，不推送数据，不触发重渲染
2. **统计系统**：可替换的仪表，进程退出时落盘
3. **AppState**：整台应用的共同现实，带副作用总闸门

关键决定是 `AppStateProvider`——它的存在让所有子系统订阅**同一份现实**，而不是各自维护。

---

## 应用 vs 脚本的三个核心差异

**状态的生命周期不同**

脚本的状态随调用消失。应用的状态在整个会话里持续存在并演化。Claude Code 的 `AppState` 从进程启动就存在，直到进程结束，中间承载了所有的权限变更、任务创建、模型切换。

**并发源的数量不同**

脚本通常串行。应用需要同时处理多个事件源：用户键盘输入、子任务完成回调、远端 WebSocket 推送、权限请求到达、文件系统变化。

**可观测性要求不同**

脚本要么跑完要么报错。应用需要持续监控自身健康——这就是 `FpsMetricsProvider` 和 `StatsProvider` 存在的原因。

## 一个说明问题的对比

假设权限模式从 `default` 切换到 `bypassPermissions`：

**打印机模型**：需要把这个变化通知给消息区、状态栏、输入区、权限按钮……每个组件自己监听，自己更新。如果有一个组件忘了监听，就会显示过时的状态。

**应用架构**：`setAppState(prev => ({ ...prev, toolPermissionContext: newContext }))` 一行。所有订阅了这个字段的组件通过 `useSyncExternalStore` 自动收到通知并重渲染。没有人会漏。

而且权限模式的副作用——通知远端、写配置——都在 `onChangeAppState` 里统一处理，不会散落在各处。

## 什么时候选打印机，什么时候选应用

面试里这是一个好问题。

**选打印机（REPL 循环）** 的场景：
- 单次查询工具（git blame 包装器、代码格式化器）
- 无状态管道：输入 → 处理 → 输出
- 不需要跨次交互保持状态

**选应用架构**的场景：
- 长期会话，状态跨轮次演化
- 多个子系统需要观察同一份现实
- 并发事件源（用户 + 后台任务 + 远端推送）
- 需要可观测性和统计
- 副作用需要集中管理

Claude Code 需要后者。

## 面试指导

**高频追问：为什么不用 Redux？**

Redux 同样可以实现共享现实。Claude Code 没有用 Redux，因为它不需要 Redux 的时间旅行和 DevTools——终端 UI 不是浏览器环境，这些能力用不上。自己实现一个 50 行的 `createStore` 更轻量，对这个场景足够。

**高频追问：AppState 太大了，会有性能问题吗？**

关键不在 AppState 有多大，而在于订阅粒度。`useAppState(s => s.verbose)` 只在 `verbose` 字段变化时才重渲染。大状态 + 细粒度 selector = 不会有无谓的重渲染。这是下一节 `useSyncExternalStore` 要展开的核心。
