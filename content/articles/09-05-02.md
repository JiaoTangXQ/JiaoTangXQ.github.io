# 锁文件和重读是在给并发写入和恢复力兜底

Claude Code 的 mailbox 不是简单“把 JSON 写进文件”，它在每次写和标记已读时都加锁并重读。

## 实现链

`writeToMailbox()` 会先确保 inbox 文件存在，然后通过 `proper-lockfile` 风格的锁获得独占访问，再重新读取最新消息，追加新消息，最后写回。`markMessageAsReadByIndex()` 和 `markMessagesAsRead()` 也走同样套路。

这说明它从一开始就承认：多个 agent 可能同时写，恢复和重试也可能发生，不能假设内存里那份就是最新。

## 普通做法

普通做法常常直接 `read -> push -> write`，甚至干脆只 append，不考虑并发碰撞。

## 为什么不用

Claude Code 不敢这么乐观，因为团队通信本来就跨成员、跨时机、跨后端。没有锁和重读，一旦两个写入撞上，消息丢失会非常隐蔽。对团队系统来说，这种 bug 比慢一点更可怕。

## 代价

代价是磁盘消息通道更慢，代码更啰嗦，还得处理 `ENOENT` 和锁失败等边角情况。但这些都在换稳定性。
