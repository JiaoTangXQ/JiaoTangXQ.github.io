# 深度分析：多 agent 团队为什么是组织系统，不是并发调度器

## 两种理解 swarm 的方式

**理解方式一：并发执行框架**

多 agent 就是多个并行运行的任务。每个 agent 是一个执行单元，任务分解后分配给多个 agent 并行处理，最后收集结果。这是"工程师视角"：问题的本质是并发和资源分配。

**理解方式二：组织系统**

多 agent 是一群有各自身份的协作者，他们需要协调行动、处理冲突、在某个成员出现问题时能够接续工作。这是"管理者视角"：问题的本质是如何让多个独立个体协同工作而不出乱子。

Claude Code 的 swarm 实现选择了第二种理解方式，虽然它也需要解决并发和资源问题。

**面试题：为什么说这是个组织问题？从代码里能看到哪些证据？**

## 证据一：身份系统

在 `src/utils/swarm/reconnection.ts` 里：

```typescript
const isLeader = !agentId  // agentId 缺失意味着这是 leader

return {
  teamName,
  teamFilePath,
  leadAgentId: teamFile.leadAgentId,
  // ...
}
```

每个 agent 有明确的身份（`agentId`、`agentName`）、所属团队（`teamName`）、以及角色（leader vs worker）。

这不是并发框架里的"线程 ID"——那只是为了调度区分，没有组织含义。这是组织里的"职位"：leader 代表团队，worker 在 leader 的协调下工作。

身份系统意味着：系统需要维护"谁是谁"的关系，不只是"有多少个执行单元在跑"。

## 证据二：邮箱通信而不是共享内存

如果是纯并发框架，agent 之间通过共享内存通信是最自然的（最高效的）。

实际上用的是文件邮箱（`mailbox.ts`）：每条消息都写到文件系统，发送者把消息写入收件人的邮箱目录，接收者定期轮询自己的邮箱。

这是有意识选择效率更低的通信方式，因为文件邮箱有一个共享内存没有的特性：**可恢复性**。

某个 agent 崩溃重启后，可以读取邮箱里未处理的消息，恢复到崩溃前的状态。共享内存在进程崩溃时数据丢失，没有恢复的起点。

组织系统需要可恢复性——成员可能临时退出，但组织的工作应该能继续。纯并发框架不关心这个。

## 证据三：权限同步而不是本地决策

在 `permissionSync.ts` 的设计里，worker 遇到权限请求时不自己决定，而是把请求发给 leader，由 leader（代理用户）做决定。

这是组织决策模式：下级遇到超出自己权限的事情，上报给上级，等待指示。

如果是纯并发框架，每个线程遇到权限问题要么自己决定（本地策略），要么走全局锁（所有线程共享一个权限决策点）。

组织模式更复杂，但它保留了"人插进来的位置"——用户通过 leader UI 参与权限决策，而不是被系统自动处理掉。

## 证据四：teamContext 是组织背景，不是执行参数

`AppState` 里有 `teamContext`，包含：
- `teamName`：这个 agent 属于哪个团队
- `teamFilePath`：团队配置文件的位置
- `leadAgentId`：团队的 leader 是谁

这些信息是"组织背景"：它们不直接影响任务的执行逻辑，但影响"如何与其他成员协调"、"谁的权限高于自己"、"重连时找谁对齐状态"。

纯并发框架不需要这些信息——线程只需要知道自己该做什么，不需要知道自己在哪个组织里。

## 为什么组织设计比并发设计更难

并发的挑战主要是技术的：数据竞争、死锁、饥饿。这些有成熟的解决方案（锁、原子操作、无锁数据结构）。

组织的挑战包含了并发的挑战，还多出了：

- **身份和信任**：谁的决定可以信赖？谁有权限做什么？
- **失败和接续**：某个成员退出，工作如何继续？
- **人的参与**：人要能看懂发生了什么，要能干预和纠正

这些没有通用算法，需要针对具体场景设计协议（比如 permissionSync 协议），而且协议本身会随业务需求演进。

Claude Code 的 swarm 实现承认了这个复杂性，而不是试图把"多 agent 协作"简化成"更强的并发执行"。

## 面试要点

1. **身份系统是组织的基础**：agentId、agentName、teamName 是组织成员的标识，不只是并发调度的编号。
2. **文件邮箱牺牲性能换可恢复性**：组织需要成员临时退出后能恢复协作，这比纯性能更重要。
3. **权限上报是组织决策的保留**：worker 不自己处理超权决策，上报给 leader，保留了人介入的位置。
4. **团队背景是组织信息，不是执行信息**：teamContext 影响协调方式，不直接影响任务逻辑，这正是"先是组织问题"的体现。
