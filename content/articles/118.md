# 面试题：in-process MCP 服务为什么要假装自己是 stdio 服务？

## 问题情境

MCP（Model Context Protocol）有几种服务类型：

- **stdio 服务**：通过标准输入输出通信，适合独立进程的工具
- **HTTP/SSE 服务**：通过网络通信，适合远端服务
- **in-process 服务**：服务代码直接运行在 Claude Code 进程内，无需网络

in-process 服务是性能最好的方式：没有进程切换、没有序列化、没有网络延迟，工具调用就是普通的函数调用。

但 Claude Code 的 in-process 服务对外暴露的接口和 stdio 服务一样——同样的工具发现、同样的调用协议、同样的错误格式。

**面试题：in-process 服务已经可以直接暴露更高效的接口，为什么还要伪装成 stdio？**

## 伪装的代价与收益

先算代价：

in-process 服务"假装是 stdio"意味着工具调用要经历一套没有实际意义的序列化/反序列化——工具输入从对象序列化成 JSON 字符串，调用完再从 JSON 字符串反序列化回对象。这比直接传对象浪费时间和内存。

还意味着代码更复杂：要实现 MCP 协议的握手、工具列表查询、工具调用协议，而不是直接 `await tool.call(input)`。

所以"伪装成 stdio"是有成本的。为什么愿意付这个成本？

## 统一接口的实际好处

**Claude Code 所有处理工具调用的代码都只需要写一次**。

拿工具调用的错误处理来说：如果 in-process 服务有自己的调用接口，错误处理就要写两套——一套处理 in-process 服务的错误，一套处理 stdio 服务的错误。

如果接口一致，错误处理只写一套，对两种服务都有效。

拿工具发现来说：Claude Code 的 `ToolSearch` 功能查找可用工具，提示词生成时枚举工具能力，都走同一套工具列表 API。如果 in-process 服务暴露不同的接口，这两个功能就要分别处理两种接口，或者 in-process 工具无法被发现。

拿调试来说：如果所有 MCP 服务都用同一种协议，记录工具调用日志只需要一套代码，日志格式一致，开发者查日志时不用区分"这是 in-process 服务的调用还是 stdio 服务的调用"。

## 接口是长期稳定的语义约定

一个更深层的理由：**接口代表长期稳定的语义约定，实现可以高效，但约定不应该随实现变化**。

MCP 的工具接口定义了"什么叫一个工具"：有名字、有输入 schema、有调用方式、有结果格式。这个定义是 Claude Code 和所有工具之间的约定。

in-process 服务遵守这个约定，意味着今天的 in-process 服务明天可以被替换成 stdio 服务、HTTP 服务或者完全不同的实现，调用方不需要任何修改。

这不是假设场景——Claude Code 确实在演进。某些今天是 in-process 的服务，可能因为安全原因以后要独立进程。如果接口不统一，这个迁移会很痛苦。

## 克制的另一面：内部可以高效

"表面统一"不意味着放弃性能。

接口层（对外暴露的 API）遵守 MCP 协议，但实现层（调用工具的内部代码）可以走捷径。

比如，in-process 服务的 JSON 序列化可以使用高性能实现（Claude Code 有 `slowOperations.ts` 这个名字的文件，暗示对序列化性能是有意识的）；工具查找可以用内存索引而不是文件系统搜索；错误对象可以是富类型而不是字符串。

"克制"不是"放弃内部优化"，而是"内部优化不应该泄漏到接口上"。接口保持简单一致，实现保持高效，这两个目标不冲突。

## 类比：HTTP 服务器的设计

这个设计很类似好的 HTTP 服务器设计原则。

一个 HTTP 服务器的外部接口是标准的 HTTP 协议，但内部可以用任何语言、任何数据结构、任何缓存策略。客户端不知道也不需要知道服务器内部是 Node.js 还是 Go 还是 Rust。

in-process MCP 服务遵守的是同样的原则：对外是标准的 MCP 接口，内部怎么高效怎么来。

## 面试要点

1. **接口一致性降低全系统维护成本**：错误处理、工具发现、日志记录只需要写一套，对所有 MCP 服务类型有效。
2. **接口是长期约定，实现是短期细节**：in-process 服务今天可以换成独立进程，接口保持不变，调用方零修改。
3. **克制不等于放弃性能**：内部实现可以走捷径，但这些优化不应该泄漏到接口上。
4. **"假装成 stdio"的代价是真实的**：多出来的序列化开销不是免费的，但这个代价换来的是长期可维护性，是合理的取舍。
