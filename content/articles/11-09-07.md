# warmSearchIndex 为什么要分批 + sleep(0)？主线程让步的时机选择

全文搜索在大量消息时的性能问题，以及 JavaScript 单线程下如何避免 UI 冻结。

## warmSearchIndex 的任务

`JumpHandle.warmSearchIndex()` 做的事是：遍历所有消息，提前调用 `extractSearchText(msg)` 并把结果缓存到 `fallbackLowerCache`（WeakMap）。

为什么要预热？第一次按 `/` 输入搜索词时，`setSearchQuery` 要在所有消息上跑 `extractSearchText`，这个函数对每条消息做字符串 toLowerCase（可能还有 Markdown 解析、ANSI 剥离），是非常内存密集的操作。

对于有 700 条消息的 transcript，每条消息平均 500 字节的文本，一次性 toLowerCase 需要：
- 700 次 string 操作
- 约 350KB 的新字符串分配
- 几十毫秒的 CPU 时间

在用户按 `/` 的那一刻同步执行这些，界面会冻结几十毫秒，输入会有明显的延迟感。

## sleep(0) 的原理

`sleep(0)` 是 `await new Promise(resolve => setTimeout(resolve, 0))` 的简写。

`setTimeout(fn, 0)` 不是"立刻执行 fn"，而是"把 fn 放进宏任务队列，等当前调用栈清空后，在下次宏任务循环时执行"。

在 `async` 函数里 `await sleep(0)` 的效果：

1. 当前函数在 `await` 处暂停
2. 控制权交还给事件循环
3. 事件循环处理其他待定的任务（用户键盘输入、滚动事件、React 渲染等）
4. setTimeout 回调触发，函数从 `await` 处继续

这让长时间的计算任务可以主动"让路"，在计算间隙处理 UI 事件，避免连续占用主线程。

## 实际实现

```typescript
warmSearchIndex: async () => {
  if (indexWarmed.current) return 0  // 已经热过，跳过
  const start = Date.now()
  
  const BATCH_SIZE = 50  // 每批处理 50 条
  for (let i = 0; i < messages.length; i++) {
    extractSearchText(messages[i]!)  // 提取 + 写入缓存
    
    if (i % BATCH_SIZE === BATCH_SIZE - 1) {
      await sleep(0)  // 每 50 条让步一次
    }
  }
  
  indexWarmed.current = true
  return Date.now() - start  // 返回耗时（毫秒）
}
```

BATCH_SIZE = 50：每 50 条消息让步一次。对于 700 条消息，共让步 14 次。总耗时会比一次性计算稍长（每次 sleep 增加约 0-4ms 的 macrotask 延迟），但 UI 在这 14 个间隙里都有机会处理输入事件。

## 调用时机

`warmSearchIndex` 在用户按 `/` 时被调用（来自键盘绑定），调用者收到 Promise 之后可以先显示 "indexing..." 状态：

```typescript
// SearchPalette 或 KeybindingHandler 里
const elapsed = await jumpRef.current.warmSearchIndex()
if (elapsed > 0) {
  showStatus(`indexed in ${elapsed}ms`)
}
```

`elapsed === 0` 表示已经预热过，不显示状态。`elapsed > 0` 说明这次真的做了预热，显示耗时让用户了解系统状态。

## 为什么不用 Web Worker

理想的解法是把搜索索引预热放进 Web Worker，完全不占主线程。但 Claude Code 运行在 Node.js 里（基于 ink 的终端应用），没有浏览器的 Worker API。

在 Node.js 里有 worker_threads，但引入它会带来：
1. 消息序列化/反序列化开销（传几百条消息的文本）
2. Worker 线程的启动开销
3. 与 React 渲染的同步问题（Worker 完成时如何通知主线程）

对于这个规模（几百条消息，几十毫秒），分批 + sleep(0) 的成本更低，代码更简单。

## 面试指导

这道题考察 JavaScript 事件循环机制和 UI 性能优化的实践知识。

**核心知识点**：JavaScript 是单线程的，长时间的同步计算会阻塞 UI 响应。解决方案是把长任务切成小块，在小块之间主动让步（yield）。`sleep(0)` 是最简单的 yield 方式，但在真正的浏览器/Node 性能优化里，`requestAnimationFrame`（浏览器）或 `setImmediate`（Node.js）在某些场景更合适。

**比较不同的 yield 方式**：
- `setTimeout(fn, 0)`：进入 macrotask 队列，延迟 0-4ms，让 microtask（Promise）先跑完
- `Promise.resolve().then(fn)`：进入 microtask 队列，在当前 macrotask 结束后立即运行，不让出 UI 事件处理
- `requestAnimationFrame(fn)`：在下次 paint 前运行，适合动画
- `scheduler.yield()`（Chrome 实验性 API）：精确让步给 UI，比 setTimeout 更高优先级

**追问**：BATCH_SIZE 是 50，有没有更好的值？答：理想的批大小取决于每条消息的处理时间。如果每条处理 0.5ms，50 条 = 25ms，正好在"用户感受到延迟"（>16ms）的边缘。更好的做法是动态调整：先测量一批的时间，如果超过 16ms（1 帧），缩小批大小；如果远低于 16ms，增大批大小。这是 React 的 time slicing 策略的简化版。
