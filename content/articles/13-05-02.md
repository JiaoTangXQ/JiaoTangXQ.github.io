Claude Code 这层复杂度不是无目的膨胀，它大部分都在给“不中断地继续工作”买保险。`QueryEngine` 会把消息和文件状态持续留在会话里，而不是每轮都重新开始；`query.ts` 针对流式中断、缺失工具结果、重试和补结案准备了额外逻辑；`compact.ts` 不只是缩短上下文，还在努力把压缩后的线索重新拼回可继续工作的形状；`LocalAgentTask.tsx` 则把子 agent 的输出、回看和回收时机全部正式化，避免后台工作一结束就只剩一坨不可追踪的文本。

更普通的实现，是把失败恢复留给人：断了就重跑，context 爆了就清空，子任务结束了就看日志，远端掉线了就重新进。这条路的代码更薄，但每次出问题都会把复杂度重新推回用户身上。

Claude Code 选择的是相反的分摊方式：系统多背一些状态和恢复链，用户少背一点“出错后该怎么办”的临场负担。这种选择大体上是合理的，特别适合长期会话和复杂工作流；只是它也会让代码越来越像产品基础设施，而不是一套轻便脚本。

## 实现链
`QueryEngine`、`query()`、`AppStateStore` 和 compact/resume 链条一起保证长会话不中途散架。

## 普通做法
只为当前这轮成功负责。

## 为什么不用
因为工作台一旦要恢复、继续、压缩和回放，就不能只顾眼前。

## 代价
复杂度会持续占据维护成本。
