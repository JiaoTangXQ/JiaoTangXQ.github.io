# QueryEngine 为什么不每轮重新初始化，而要保持一整场会话？

## 两种架构的对比

设计一个对话系统，有两种基本方式：

**方式 A**：每条消息来了，创建一个上下文对象，调模型，返回结果，销毁上下文。轻，无状态，好测试。

**方式 B**：维护一个持久的会话对象，每条消息追加进去，历史永远在那里，状态在整个会话生命周期内有效。重，有状态，但恢复和继续变得容易。

Claude Code 选了方式 B，`QueryEngine` 就是那个持久的会话对象。

## QueryEngine 的持久状态

`QueryEngine` 在整个会话里持续维护：

- **消息历史**：每一轮的输入和输出，包括工具调用和工具结果
- **文件状态快照**：当前会话涉及到的文件的最后已知状态
- **权限拒绝记录**：用户在本次会话里拒绝过哪些工具调用，避免反复询问
- **token 使用量**：本次会话累计消耗了多少，用来判断是否需要触发 compact

这些状态如果每轮重新初始化，就会消失。用户下一轮说"把刚才那个文件改回去"，系统就不知道"刚才那个文件"是什么了。

## 对恢复的影响

`query.ts` 的主循环在执行时可能被中断：网络断了、用户按了 Ctrl+C、工具超时。如果 `QueryEngine` 是无状态的，中断就意味着这轮的所有上下文都丢失，用户需要重新描述任务。

因为 `QueryEngine` 是持久的，中断后的恢复点很清晰：从 transcript 里找到最后一条已经记录的消息，判断中断发生在哪一步，从那里继续，而不是从头开始。

这不是免费的——`QueryEngine` 需要在每一步操作完成时立刻把状态写回 transcript，而不是等整轮结束后再写。这让单步操作略微变慢，但让恢复变得可能。

## 对 compact 的影响

当会话 token 接近上限时，`compact.ts` 需要生成一段摘要，让模型在新边界里继续工作。

这件事能做成的前提，是 `QueryEngine` 持有完整的消息历史。如果每轮消息处理完就丢弃，compact 就没有可以压缩的材料——你只能知道这一轮发生了什么，不知道前面发生了什么。

压缩后，`QueryEngine` 的消息历史被替换成摘要 + 新边界，但会话对象本身继续存在，指向同一个工作流。这就是"压缩后还能继续干活"的底层保障。

## 对子任务的影响

当主会话里触发了一个 subagent 时，`LocalAgentTask` 会被创建，但它的消息历史是独立的。主会话的 `QueryEngine` 记录了"我发起了一个 subagent"，subagent 有自己的消息管道。

两者通过 `AppStateStore` 里的任务状态同步，而不是直接读对方的 `QueryEngine`。这保证了子任务的消息历史不会污染主任务的上下文。

## 代价

会话对象本身会越来越大。长时间运行的会话，消息历史可能有几百条，文件快照可能有几十个文件的状态。这些都在内存里。

这就是为什么 compact 是必须的功能，而不是可选的优化——没有定期的压缩，长会话最终会撑爆上下文窗口。

## 面试指导

被问"你会怎么设计一个支持长期会话的 agent"时，`QueryEngine` 这套设计是一个有深度的参考答案：

选择持久的会话对象而不是无状态的请求处理器，是为了让恢复、压缩和上下文传递有可操作的基础。代价是内存占用更多、状态管理更复杂，但收益是系统能承担真正的长期工作，而不只是一次性的问答。
