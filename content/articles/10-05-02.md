# compact 后为什么还要重注入文件、plan 和技能？

compact 完成了——消息从 180K token 压缩到了 8K 的摘要。但这还没结束。

`compactConversation` 里有一段并行执行的重注入逻辑，把几类"被压缩掉的关键上下文"重新塞回来。这是为什么？

---

## compact 清空了什么

compact 开始时有一段状态清理：

```typescript
// Store the current file state before clearing
const preCompactReadFileState = cacheToObject(context.readFileState)

// Clear the cache
context.readFileState.clear()
context.loadedNestedMemoryPaths?.clear()
```

`readFileState` 是一个 map，记录"模型这轮已经读过哪些文件"。compact 会清空它——因为新的摘要上下文里没有原始的文件读取记录，再用旧的 cache 状态会产生不一致（系统以为文件已经"在上下文里了"，实际上已经在摘要中被概括掉了）。

清空 `readFileState` 意味着：compact 后模型不再"记得"自己读过哪些文件。如果下一轮需要用到那些文件，系统会重新读取。

但等等——如果某个文件在摘要里被引用了，模型"知道"它的内容，但系统没有这个文件的 cache 记录，会发生什么？

这就是为什么需要重注入。

## 重注入的内容和预算

```typescript
const [fileAttachments, asyncAgentAttachments] = await Promise.all([
  createPostCompactFileAttachments(
    preCompactReadFileState,   // compact 前的文件读取状态
    context,
    POST_COMPACT_MAX_FILES_TO_RESTORE,  // = 5
  ),
  createAsyncAgentAttachmentsIfNeeded(context),
])
```

重注入的预算：
- 最多恢复 **5 个文件**（`POST_COMPACT_MAX_FILES_TO_RESTORE`）
- 文件总 token 预算 **50,000**（`POST_COMPACT_TOKEN_BUDGET`）
- 单个文件上限 **5,000 token**（`POST_COMPACT_MAX_TOKENS_PER_FILE`）
- 技能总预算 **25,000 token**（`POST_COMPACT_SKILLS_TOKEN_BUDGET`）
- 单个技能上限 **5,000 token**（`POST_COMPACT_MAX_TOKENS_PER_SKILL`）

为什么要有这些上限？因为文件重注入本身也会消耗 token，如果不加限制，重注入内容可能和压缩节省的内容一样多，compact 就没有意义了。

选择"最近使用的 5 个文件"而不是所有文件，基于一个合理假设：compact 之后继续工作，最可能需要的是最近在用的文件，而不是早期读过的历史文件。

## plan 的特殊处理

plan mode 下，当前的任务计划需要重注入：

```typescript
const planAttachment = createPlanAttachmentIfNeeded(context.agentId)
if (planAttachment) {
  postCompactFileAttachments.push(planAttachment)
}

const planModeAttachment = await createPlanModeAttachmentIfNeeded(context)
if (planModeAttachment) {
  postCompactFileAttachments.push(planModeAttachment)
}
```

plan 不在 `POST_COMPACT_MAX_FILES_TO_RESTORE` 的 5 个名额里——它是单独注入的。原因是 plan 的重要性高于普通文件：它描述的是"当前任务的执行计划"，是 compact 后继续工作的路线图。如果 plan 消失了，模型可能不知道接下来要做什么。

## 工具列表和 agent 列表的重注入

compact 之后，之前通过 delta 机制注入的工具信息也会失效：

```typescript
for (const att of getDeferredToolsDeltaAttachment(
  context.options.tools,
  context.options.mainLoopModel,
  [],  // 传空数组，相当于"对比空历史"，重新发送完整工具列表
  { callSite: 'compact_full' },
)) {
  postCompactFileAttachments.push(createAttachmentMessage(att))
}

for (const att of getAgentListingDeltaAttachment(context, [])) {
  postCompactFileAttachments.push(createAttachmentMessage(att))
}

for (const att of getMcpInstructionsDeltaAttachment(..., [])) {
  postCompactFileAttachments.push(createAttachmentMessage(att))
}
```

传 `[]`（空数组）作为 delta 基线，相当于"从零开始"重新宣告所有工具。这确保模型在 compact 后的第一轮仍然知道可以用哪些工具、有哪些 agent、MCP 服务的配置是什么。

## 技能的特殊处理

```typescript
const skillAttachment = createSkillAttachmentIfNeeded(context.agentId)
if (skillAttachment) {
  postCompactFileAttachments.push(skillAttachment)
}
```

注意代码注释（来自源码）：
> Intentionally NOT resetting sentSkillNames: re-injecting the full skill_listing (~4K tokens) post-compact is pure cache_creation with marginal benefit.

`sentSkillNames` 不重置——技能列表（skill_listing）不在 compact 后重注入。理由是：skill listing 大约 4K token，重新注入主要是增加 `cache_creation` 成本，边际价值低。模型仍然知道有 SkillTool 可以用，需要时会主动调用，不需要提前把所有技能内容灌进上下文。

这是一个有意识的 token 经济决策：有些东西"重新注入的边际价值"不高，宁愿按需加载。

---

## session start hooks 在 compact 里执行

```typescript
const hookMessages = await processSessionStartHooks('compact', {
  model: context.options.mainLoopModel,
})
```

compact 后会触发一次 session start hooks，`trigger` 类型是 `'compact'`（区别于普通会话启动的 `'start'`）。

这让外部系统有机会在 compact 完成后注入内容——比如在每次 compact 后重新注入当前日期、某些项目状态、或者工作环境信息。

## 面试指导

"compact 后为什么要重注入内容"考察的是对 compact 的深层理解。

浅层理解："compact 生成了摘要，用摘要替换历史。"

深层理解："compact 后，模型的记忆是从摘要重建的。但工具 schema、文件 cache、plan 状态这些不在对话历史里的东西，也在 compact 时被清除了。重注入是把这些'环境状态'带回新的上下文，确保模型在压缩后的新窗口里能继续工作，而不只是知道'做过什么'，还知道'接下来用什么工具、参考什么文件'。"

这两者之间的差距，就是候选人对 compact 理解的深度差距。
