# Claude Code settings 治理地图

## 先说结论：settings 在这里不是偏好文件，而是制度装配线

Claude Code 不是读一份 `settings.json` 就结束。它的 settings 系统在做的是真正的治理工作：多来源叠加、优先级、校验、监听、缓存、变更前拦截、对运行时的制度回写。

## 第一层：设置来源天然分层，不是所有人同权

### 实现链

从代码和文档都能看出，设置来源至少包括：

1. 用户级
2. 项目级
3. 本地私有级
4. CLI flag
5. policy / managed

### 普通做法

更普通的项目会只有用户配置加项目配置两层，或者让后写入的全部覆盖前面的。

### 为什么不用

Claude Code 要同时服务个人习惯、项目约束和组织政策。它必须先承认“谁有资格定规则”这件事不是平的。

### 代价

来源越多，解释越难，配置冲突也更难读；但现实世界就是多层级的。

## 第二层：`settings.ts` 做的是装配，不只是读取

### 实现链

`settings.ts` 会处理来源读取、合并、managed drop-ins、校验和稳定输出接口。

### 普通做法

很多系统只是读几个 JSON，merge 一下就完。

### 为什么不用

Claude Code 的设置会直接影响权限模式、hooks、plugins、远端和很多运行时行为，所以它需要一条更正式的装配线，而不是轻量读档函数。

### 代价

设置系统本身会长成一个大子系统；但这是运行时可治理的前提。

## 第三层：managed drop-ins 说明它在借系统治理世界的经验

### 实现链

除了主 managed 文件，还支持按顺序覆盖的 drop-in 目录。

### 普通做法

更简单的做法是只保留一个集中式托管配置文件。

### 为什么不用

Claude Code 需要让企业和团队政策可拆分、可局部增补，而不是逼所有规则都进一个巨型文件。

### 代价

加载顺序和冲突解释会更复杂；但维护更现实。

## 第四层：校验不是“错一处整份作废”

### 实现链

validation 相关逻辑会尽量把错误局部化，尤其是 permission rules 之类内容，不会轻易因为局部错误把整份设置完全废掉。

### 普通做法

很多项目的配置校验是一票否决，任何错误都让文件整体失效。

### 为什么不用

Claude Code 更在意系统还能不能继续稳定工作。真实环境里，设置文件经常会被人手改、半改、复制错。局部容错更实用。

### 代价

错误模型更复杂，也要把“哪些生效、哪些没生效”解释清楚；但用户体验更稳。

## 第五层：变更探测说明设置是活的运行时制度

### 实现链

`changeDetector` 不是普通文件监听器，它要处理写入稳定、删除重建、内部写/外部写、轮询来源等现实情况。

### 普通做法

更普通的做法是配置只在启动时读一次，或者文件变了就简单重载。

### 为什么不用

Claude Code 的设置变化会影响当前会话，不只是下次启动。既然要在运行时生效，就必须认真处理真实文件系统行为，而不是假设每次改动都像理想保存。

### 代价

监听和刷新链会更重，也更容易牵涉缓存问题；但这是“设置会活着改写运行时”的代价。

## 第六层：`ConfigChange` hook 说明“改规则”本身也要受规则约束

### 实现链

设置变化不是立刻生效，而是可以先经过 `ConfigChange` hook 再决定要不要进入当前会话。

### 普通做法

很多系统把配置变化视为绝对事实，检测到就直接应用。

### 为什么不用

Claude Code 承认：修改制度本身也是高影响动作，值得再被制度检查一层。

### 代价

配置生效链更长，也更难跟；但治理能力更强。
