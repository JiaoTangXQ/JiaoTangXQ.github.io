# commands.ts 里那些 feature gate 是设计之美，还是历史债？

## 工程反思：诚实地看代码

真正的工程反思不是把所有复杂都包装成"优雅的设计"，而是能分清楚：哪些复杂是合理的选择，哪些复杂只是现实压力留下的形状。

`commands.ts` 是最好的例子。

## commands.ts 里的 feature gate

打开 `commands.ts`，会看到一层层 feature gate 和条件 import：

```
if (feature('SOME_EXPERIMENTAL_FLAG')) {
  // 注册这个命令
}
```

这些 gate 存在的原因，通常不是"架构上这里需要一个开关"，而是：
- 这个功能正在 A/B 测试
- 这个功能只对内部用户开放
- 这个功能还没有稳定到可以给所有人
- 这个功能在某些平台上不可用

这些是**产品现实**，不是架构设计。它们的存在说明 Claude Code 在真实的发布节奏和真实的用户类型下工作——内部用户、外部用户、headless 模式、不同平台，都在同一份代码里。

## main.tsx 和 query.ts 里的同类痕迹

`main.tsx` 里有些路径只在特定 build 开启：协调器模式、远端模式、Chicago MCP。这些不是"架构上对应了三种不同的世界"——其中有些确实对应了不同的用户场景，但有些只是因为某个功能还没有准备好对所有人开放。

`query.ts` 里也有不少"为了照顾 headless 场景"或"为了照顾旧版客户端"的分支。这些分支很多时候不是你想要的那种精心设计的扩展点，而是"当时必须这样写才能让旧的东西不断"。

## 如何区分两种复杂

有一个实用的判断方式：问"如果没有产品历史和发布压力，这段代码会不会存在？"

- 如果答案是"会"，那它大概率是设计选择——比如权限制度层、任务终态协议，这些去掉了系统就会变弱。
- 如果答案是"不会，是因为要兼容旧版本/支持实验开关/照顾某个特殊用户类型"，那它大概率是现实债——去掉了系统会更干净，但现实不允许去掉。

`commands.ts` 里的大多数 feature gate 是后者。

## 这不是批评，而是诚实

Claude Code 可贵的地方，不是它没有这些现实债，而是它大多数时候能把这些债务压在边缘，不让它们轻易夺走主干结构的话语权。

主干——分层、交棒、制度先行、接缝零件——这些是真正的设计选择，在代码里确实成立。

边缘——feature gate、兼容路径、为特殊用户类型写的分支——这些是真实产品必须背的重量，在代码里也确实存在。

两者都是真的，都不该被忽视。

## 为什么这个判断很重要

如果读者把所有复杂都神化成"设计感"，就会形成一种错误的学习路径：以为自己也需要一开始就写满 feature gate 和兼容逻辑。这不对。这些是产品到了一定阶段、背负一定用户基础之后不得不增加的重量，不是你应该一开始就模仿的风格。

如果读者把所有复杂都降级成"运气好没散"，就会错过真正值得学的部分。

诚实地区分两类复杂，是读代码的基本功。

## 面试指导

被问"这段代码有什么问题，或者你会怎么改"时，能指出"这是产品历史债，不是架构选择"的候选人，比只能说"这里有一些 if/else"的候选人，展示出了更高层次的代码阅读能力。

工程判断力的一部分，就是能区分"该改的复杂"和"不得不背的复杂"。
