不是所有复杂处都该被夸成“设计感”。只要看 `commands.ts` 里那一层层 feature gate 和条件 import，就知道 Claude Code 也在和发布现实、用户类型、实验开关、历史形态谈判。`main.tsx` 和 `query.ts` 里也有不少这种痕迹：某些路径只在特定 build 开启，某些分支是为了照顾 headless、remote、coordinator 或 Chicago MCP 这类条件世界。这些代码未必丑，但很多时候它们存在，不是因为这是最纯的设计，而是因为真实产品就是要在同一份代码里容纳不同阶段、不同入口和不同实验。

更理想主义的实现，是删掉所有条件世界，只保留一条最美的主线。那样架构当然更容易讲，也更容易让人觉得“这套系统真干净”。问题是，一旦系统已经面向真实用户、真实平台和真实发布节奏，这种纯净很容易靠牺牲产品现实换来。

Claude Code 的可贵之处，不是它没有现实污点，而是它大多数时候能把这些污点压在边缘，不让它们轻易夺走主干结构的话语权。缺点也同样存在：边缘层一多，读者很容易误把“为了兼容而加的形状”当成“设计者本意”。这就需要分析时保持诚实，别把所有复杂都神化。

## 实现链
`main.tsx`、CLI 入口和 `query()` 里有不少是被环境、用户类型、兼容场景逼出来的条件分支。

## 普通做法
把这些分支都包装成“高级设计”。

## 为什么不用
因为其中一部分只是宿主现实和产品压力留下的形状，不是架构之美。

## 代价
系统会保留历史包袱，难以做到绝对整洁。
