# onToolCallDisconnected() 的错误文案是一份现场排查清单

`onToolCallDisconnected()` 返回的字符串不是在打太极：

```
Browser extension is not connected. Please ensure the Claude browser extension 
is installed and running (https://claude.ai/chrome), and that you are logged 
into claude.ai with the same account as Claude Code. If this is your first 
time connecting to Chrome, you may need to restart Chrome for the installation 
to take effect. If you continue to experience issues, please report a bug: 
https://github.com/anthropics/claude-code/issues/...
```

这段话依次覆盖了浏览器桥断线最常见的几个真实原因：

1. 扩展没装或没在运行
2. 账号不一致（浏览器里登的是另一个 claude.ai 账号）
3. 刚安装后 Chrome 还没重启，扩展还没生效
4. 以上都不是，去报 bug

顺序不是随机的，是按「最可能」到「兜底」排列的。装了但没生效，比装了但账号错更常见；账号错，比真正的 bug 更常见。

## 错误文案是文档，不是占位符

系统越复杂，断线的原因越难被单一错误码覆盖。浏览器桥横跨 CLI、Chrome 扩展、OAuth 账号三个独立系统，任何一段脱节都会让工具调用失败。抽象的「连接失败，请重试」只会让用户原地打转，因为它没有告诉用户应该去检查什么。

`onToolCallDisconnected()` 把工程团队积累的真实失败场景直接写进了错误文案。用户看到这段话时，不用猜「为什么」，只需要沿着清单逐项对照。

## 末尾的 bug 报告地址是一个谦逊的承认

文案最后给出了 GitHub Issues 链接。这个细节说明工程团队知道自己的清单不是全集——还有清单覆盖不到的情况，那就请用户报告，帮助团队继续学习。

好的错误处理不是「保证不出错」，而是「出了错，用户知道下一步该往哪走」。
