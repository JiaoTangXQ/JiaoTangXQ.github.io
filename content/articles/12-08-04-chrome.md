# 断线文案先讲同账号和重启 Chrome，说明失败处理也得面对真实世界

`onToolCallDisconnected()` 最难得的地方，不是它会报错，而是它报得很现实。它不会只丢一句“扩展未连接”，而是明确告诉用户去检查扩展是否安装、浏览器里的 `claude.ai` 账号是否和 Claude Code 是同一个、第一次安装后也许需要重启 Chrome，实在不行再去提 bug。

这说明 Claude Code 在处理浏览器失败时，不装神秘，也不把责任都推给“网络异常”这种空话。它承认浏览器世界里真正会出问题的是账号、安装、生效时机和产品接入链。成熟系统的失败处理，本来就应该这样正视现实。

## 实现链
`createChromeContext()` 的 `onAuthenticationError()` 和 `onToolCallDisconnected()` 不是只回一个抽象错误码，而是直接把“同一个 claude.ai 账号”“可能需要重启 Chrome”“扩展是否安装”“去哪里报 bug”这些真实排障信息写进文案。

## 普通做法
普通做法是统一返回“连接失败，请重试”。

## 为什么不用
Claude Code 不满足于空错误，因为浏览器桥的问题往往根本不在代码，而在账号、扩展状态和宿主浏览器。

## 代价
代价是错误文案会更长、更产品化，也带上了当前实现的具体假设。
