# 哪些副作用被故意挂到了 preAction 之后

一旦跨过 `preAction`，Claude Code 才允许一批真正会改写系统状态的动作开始发生。这里面最典型的有几类: 日志 sink 开始接管事件，`--plugin-dir` 被正式写进当前会话的插件世界，迁移开始改写本地配置版本，远端受管设置和 policy limit 开始异步加载，用户设置同步也可能在后台启动。

这些动作为什么不能更早做？因为它们都已经不属于“解释命令”的范围，而属于“让系统进入真实运行状态”的范围。它们会改变后面的配置来源、插件集合、日志落点、远端规则，甚至可能触发网络请求。把这些东西挂在闸门之后，本质上是在承认一件很重要的事: 只有当用户真的要执行动作时，系统才有资格开始动用这些现实世界里的权力。

这正是 `preAction` 这道门真正高级的地方。它不只是省掉了帮助路径上的初始化成本，更把副作用从介绍层里剥了出去。成熟系统最怕的不是“慢一点”，而是边界脏掉。一旦帮助、解析、执行、副作用全混在一起，后面就只剩下一团越来越重的启动泥球。Claude Code 明显是在很早就防这个问题。

## 实现链
这页对应的实现链都落在 `preAction` 前后的职责切分上。代码把必须早做但很轻的判断留在闸门前，把完整环境、交互世界和重副作用留到闸门后，借这道线把不同成本的路径分开。

## 普通做法
更直觉的做法是把所有前置工作都塞进同一个大 handler，先后顺序只靠注释维持。

## 为什么不用那种做法
Claude Code 不这么混着写，是因为它既要处理帮助/版本这类轻路径，又要处理完整会话这类重路径。如果没有这道闸门，轻路径会一直给重路径买单。

## 代价与回报
回报是不同路径各付各的成本。代价是控制流被切成多阶段，入口阅读会更费心。
