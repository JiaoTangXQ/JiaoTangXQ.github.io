# 大型工具结果怎么处理？Claude Code 的工具输出持久化机制

模型调用 `GrepTool` 搜索代码库，得到了 5000 行匹配结果。如果把这 5000 行全部放进 context window，不仅浪费 token，也可能超出限制。

`utils/toolResultStorage.ts` 实现了一套**工具结果持久化**机制：超过阈值的工具输出自动写到磁盘，模型收到的是指向文件的引用而不是完整内容。

## 持久化阈值的计算

每个工具有自己声明的 `maxResultSizeChars`，这个值和全局默认值取较小值：

```typescript
export function getPersistenceThreshold(
  toolName: string,
  declaredMaxResultSizeChars: number,
): number {
  // Infinity = 硬性禁用持久化
  // FileReadTool 的 maxResultSizeChars = Infinity
  // 原因：持久化 Read 的输出 → 文件路径 → 模型再次 Read → 循环
}
```

`FileReadTool` 的阈值是 `Infinity`（永不持久化），因为它读文件然后把内容给模型，如果把内容持久化到文件再让模型读那个文件，就形成了死循环。这个特殊情况在注释里写得很清楚。

## 持久化的格式

超大输出被写到会话目录下的 `tool-results/` 子目录，然后模型收到的内容是：

```
<persisted-output>
/path/to/session/tool-results/abc123.txt
</persisted-output>
```

`PERSISTED_OUTPUT_TAG` 和 `PERSISTED_OUTPUT_CLOSING_TAG` 常量定义了这个 XML 标签。模型可以通过 `FileReadTool` 读取这个路径获取完整内容，但通常它会先处理这个摘要或者在需要时才读全部内容。

## GrowthBook 动态阈值覆盖

```typescript
// 标志名 'tengu_satin_quoll'
// 值是 Record<toolName, thresholdChars>
// 允许运营时调整特定工具的持久化阈值
```

通过 GrowthBook feature flag，可以在不发版的情况下调整特定工具的持久化阈值。比如如果发现某个工具的输出普遍超大，可以直接降低它的阈值让更多结果被持久化，而不需要修改代码。

## 每条消息的总体 token 预算

除了单个工具结果的限制，还有跨工具的总体预算：

```typescript
export const MAX_TOOL_RESULTS_PER_MESSAGE_CHARS = // 某个上限
```

一条 user 消息里可能包含多个工具结果（并行工具调用的情况）。如果它们的总大小超过这个限制，最旧的工具结果内容会被清空，替换为：

```
[Old tool result content cleared]
```

这个 `TOOL_RESULT_CLEARED_MESSAGE` 常量是模型能理解的占位符——它告诉模型"这里有个工具结果，但内容太老了被清理掉了"。

## 为什么不用截断而用持久化？

截断有一个问题：如果模型知道内容被截断了，它可能会要求看完整输出，但我们给不出来。

持久化方案让完整内容永远可达——模型可以通过 `FileReadTool` 的 `offset` 和 `limit` 参数读取特定片段。这保留了"完整信息可访问"的性质，只是用按需加载代替了全量传输。

## 代码归因：模型名防泄漏

`commitAttribution.ts` 里有一个有意思的安全机制——模型内部代号防泄漏：

```typescript
const INTERNAL_MODEL_REPOS = [
  'github.com:anthropics/claude-cli-internal',
  // ...几十个 Anthropic 内部 repo
]

// 在公开 repo 里，git commit trailer 里的模型名会被替换成公开名
// 在私有 repo 里，可以保留内部代号
export const isInternalModelRepo = sequential(async () => {
  const remoteUrl = await getRemoteUrlForDir(cwd)
  return INTERNAL_MODEL_REPOS.some(repo => remoteUrl.includes(repo))
})
```

当 Claude Code 生成 git commit 时，commit message 里会包含"Co-authored-by Claude"这样的 trailer。内部 repo 可以包含内部模型代号，公开 repo 必须用公开模型名。这个检查防止了内部模型代号通过 git history 泄漏到公开仓库。

## 面试角度

工具结果持久化是大型 context 管理的典型方案：

1. **soft limit vs hard limit**：持久化是 soft limit（内容还在磁盘上），截断是 hard limit（内容丢失）；根据场景选择
2. **阈值按工具设置**：不是全局统一阈值，因为不同工具的输出大小特征差异很大
3. **动态调整不发版**：GrowthBook 覆盖让运营可以响应式调整，不需要等待发版周期
4. **循环依赖要显式处理**：`FileReadTool` 的 Infinity 阈值是明确处理循环依赖的例子，值得在代码注释里说清楚
