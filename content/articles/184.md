# 第十三卷回收索引：外延执行里对外统一、对内高效说明接口层在守长期稳定语义

## 实现链
外延执行里还有一条更偏接口层秩序的线，不在“看起来统一”本身，而在统一语义能不能长期站得住。

这条线真正逼出来的，是第十三卷里关于接口层工程修养的判断。  
统一不是装饰，还是要能在长期里把语义守住。

这页实际回收的实现链是：
- `12-04-02` 把“对外统一、对内高效”写成成熟工程常见取舍，这会继续支撑 `13-03-01`，说明接口层不是在伪装，而是在守长期稳定语义。

好的接口层不会只追求表面整齐。  
它更在意的是，统一外观能不能一直把长期语义托住。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
