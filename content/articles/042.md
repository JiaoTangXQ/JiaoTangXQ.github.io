# Coordinator Mode：Claude Code 的多 Agent 编排是怎么设计的？

Claude Code 的 Coordinator Mode 是一种让 Claude 扮演"调度者"角色的运行模式——不直接执行任务，而是把工作分发给 Worker Agent，汇总结果，向用户报告进展。

`coordinator/coordinatorMode.ts` 是这套机制的核心。

## 模式检测：环境变量 + feature gate 双重条件

```typescript
export function isCoordinatorMode(): boolean {
  if (feature('COORDINATOR_MODE')) {
    return isEnvTruthy(process.env.CLAUDE_CODE_COORDINATOR_MODE)
  }
  return false
}
```

Coordinator Mode 的激活需要两个条件都满足：构建时的 `feature('COORDINATOR_MODE')` gate 开启，以及运行时环境变量 `CLAUDE_CODE_COORDINATOR_MODE` 为真值。这是 feature gate 和环境变量双重保护的典型模式。

## 会话恢复时的模式匹配

会话恢复时存在一个微妙问题：如果存档的会话是 coordinator 模式，但用户恢复时的环境变量没有设置（或者反过来），怎么办？

```typescript
export function matchSessionMode(
  sessionMode: 'coordinator' | 'normal' | undefined,
): string | undefined {
  const currentIsCoordinator = isCoordinatorMode()
  const sessionIsCoordinator = sessionMode === 'coordinator'

  if (currentIsCoordinator === sessionIsCoordinator) {
    return undefined
  }

  // Flip the env var — isCoordinatorMode() reads it live, no caching
  if (sessionIsCoordinator) {
    process.env.CLAUDE_CODE_COORDINATOR_MODE = '1'
  } else {
    delete process.env.CLAUDE_CODE_COORDINATOR_MODE
  }

  return sessionIsCoordinator
    ? 'Entered coordinator mode to match resumed session.'
    : 'Exited coordinator mode to match resumed session.'
}
```

`matchSessionMode` 通过**直接修改 `process.env`** 来让模式和存档会话一致。因为 `isCoordinatorMode()` 每次调用时实时读取环境变量（没有缓存），直接写入 env 就能立即生效。这是一个副作用型的"状态修复"：不需要重启，不需要传参，直接改变进程的环境。

## Worker 可用工具的动态构建

Coordinator 需要告诉 Claude 哪些工具 Worker 可以使用：

```typescript
const INTERNAL_WORKER_TOOLS = new Set([
  TEAM_CREATE_TOOL_NAME,
  TEAM_DELETE_TOOL_NAME,
  SEND_MESSAGE_TOOL_NAME,
  SYNTHETIC_OUTPUT_TOOL_NAME,
])

export function getCoordinatorUserContext(
  mcpClients: ReadonlyArray<{ name: string }>,
  scratchpadDir?: string,
): { [k: string]: string } {
  const workerTools = isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE)
    ? [BASH_TOOL_NAME, FILE_READ_TOOL_NAME, FILE_EDIT_TOOL_NAME]
        .sort()
        .join(', ')
    : Array.from(ASYNC_AGENT_ALLOWED_TOOLS)
        .filter(name => !INTERNAL_WORKER_TOOLS.has(name))
        .sort()
        .join(', ')
  // ...
}
```

Worker 工具列表有两个版本：简化模式（`CLAUDE_CODE_SIMPLE`）只给 Bash/Read/Edit；完整模式过滤掉 `INTERNAL_WORKER_TOOLS`（这些是 coordinator 专用的内部工具，Worker 不应该有）。MCP 工具也动态拼进来。

Scratchpad 目录也在这里注入——Worker 可以在这个目录下读写文件而无需权限提示，作为 Worker 间共享知识的"持久化黑板"（通过 feature gate `tengu_scratch` 控制）。

## Coordinator 系统提示的结构

`getCoordinatorSystemPrompt()` 返回的系统提示是整个 coordinator 机制的"宪法"，规定了 6 个核心部分：

**1. 角色定义**：你是调度者，不是执行者。直接回答能处理的问题，其余全部委派给 Worker。

**2. 工具语义**：
- `AgentTool`：启动新 Worker
- `SendMessageTool`：向已有 Worker 发送续任务（利用其已有上下文）
- `TaskStopTool`：停止走错方向的 Worker

**3. 任务通知协议**：Worker 完成时，结果以 `<task-notification>` XML 格式出现在 user 角色消息里。Coordinator 必须能识别这种特殊格式（不是真实用户消息）：

```xml
<task-notification>
<task-id>{agentId}</task-id>
<status>completed|failed|killed</status>
<summary>{human-readable status summary}</summary>
<result>{agent's final text response}</result>
<usage>...</usage>
</task-notification>
```

**4. 并发策略**：
- 只读任务（研究）：可以无限并发
- 写任务（实现）：同一批文件只能一个 Worker 同时写
- 验证：可以和实现并行（不同文件区域）

**5. Worker Prompt 写法的硬性要求**：系统提示明确禁止"lazy delegation"——不能写"based on your findings, fix it"，必须把研究结果综合后写成包含具体路径、行号、类型签名的规格说明。这是防止 coordinator 把"理解"工作也委托出去。

**6. Continue vs. Spawn 决策矩阵**：以表格形式给出什么情况下用 `SendMessage` 续用已有 Worker、什么情况下用 `AgentTool` 启动新 Worker。核心原则：上下文重叠度高 → continue，低 → spawn fresh。

## 面试角度

Coordinator Mode 的设计体现了多 Agent 系统的几个核心问题：

1. **工具访问权的分层**：Coordinator 和 Worker 看到的工具集不同，内部协调工具（SendMessage/TaskStop）只有 coordinator 有，防止 Worker 越权干涉调度逻辑
2. **通知协议的 role 混用**：Worker 通知以 user 消息格式出现，但有 XML 标签标识。这是一个"假借 user 角色传系统信息"的设计，因为 Anthropic API 只有 user/assistant/tool 三种 role
3. **会话恢复的状态修复**：直接修改 `process.env` 是一个有意为之的副作用——`isCoordinatorMode()` 没有缓存、读取是实时的，所以修改环境变量立即生效
4. **系统提示作为行为约束**：大量的"不能写 based on your findings"这类具体禁令，说明 coordinator 的系统提示不只是描述功能，而是通过负面示例来约束具体的不良行为
