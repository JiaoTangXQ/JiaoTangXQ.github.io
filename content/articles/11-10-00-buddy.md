# 体验层导读：为什么 buddy 要在骨架之后登场

如果你先看 `buddy`，会觉得它很可爱：一个像素风格的小精灵，出现在终端右侧，偶尔说几句话，在你工作时陪着你。

但这不是 `buddy` 真正值得理解的地方。

## buddy 不是独立模块

`buddy` 接进了四条线：

- `PromptInput.tsx`：给 `/buddy` 命令做彩虹高亮，预留列宽
- `CompanionSprite.tsx`：渲染精灵和气泡，处理宽窄屏切换、全屏浮层
- `useBuddyNotification.tsx`：控制启动 teaser 的出现时机
- `buddy/prompt.ts` + `utils/attachments.ts`：生成 `companion_intro` 附件，送进消息流

它没有自己的状态系统，没有自己的事件循环，没有独立的 UI 层级。它是沿着现有输入区、消息系统和配置接入的，是一个「认真遵守宿主规则的体验层」，不是一个「另起炉灶的独立功能」。

## 为什么要在骨架之后讲

`buddy` 太容易让人聚焦在角色感上，忽视它底下的工程结构。如果先讲 `buddy`，读者会记住「Claude Code 有个小精灵」，但看不见输入区的布局约束、消息附件机制、配置的多开关协同、以及「功能必须服从宿主结构」这条更重要的原则。

把 `buddy` 放在工作流、消息编排、虚拟滚动这些骨架之后，是在做一个优先级的声明：工程结构先于体验装饰，理解骨架才能理解装饰为什么是这个形状。

## 这章真正想讲的

`buddy` 的每一个设计决策，都在回答同一个问题：**一个体验层功能，如何在不破坏主系统结构的前提下存在？**

列宽预算、窄屏降级、只介绍一次、气泡自动消失、静音一键退场……这些不是「buddy 的功能设计」，而是「一个成熟的体验层功能该怎么和宿主共存」的示范。
