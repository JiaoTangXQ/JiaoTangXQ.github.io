# 面试题：/compact 为什么不让模型来决定怎么压缩？

## 反直觉问题

Claude Code 是一个 AI 编程助手，里面有个核心命令 `/compact` 用来压缩对话历史。

为什么 `/compact` 是 `local` 型命令，不让模型来执行？

## 看 /compact 的实现

`/compact` 命令的 `call` 函数最终返回一个 `compact` 类型的结果：

```typescript
// 来自 commands/compact/ 的实现
return {
  type: 'compact',
  compactionResult: ...,  // 精确的压缩方案
  displayText: '...'
}
```

在 `getMessagesForSlashCommand` 的 `local` 分支里：

```typescript
case 'local': {
  const mod = await command.load()
  const result = await mod.call(args, context)
  
  if (result.type === 'compact') {
    const slashCommandMessages = [
      syntheticCaveatMessage,
      userMessage,
      ...(result.displayText ? [createUserMessage({ content: `<local-command-stdout>${result.displayText}</local-command-stdout>` })] : [])
    ]
    const compactionResultWithSlashMessages = {
      ...result.compactionResult,
      messagesToKeep: [...(result.compactionResult.messagesToKeep ?? []), ...slashCommandMessages]
    }
    resetMicrocompactState()
    return {
      messages: buildPostCompactMessages(compactionResultWithSlashMessages),
      shouldQuery: false,  // ← 不触发模型推理
      command
    }
  }
}
```

`compact` 结果里包含 `messagesToKeep`——一个精确指定要保留哪些消息的列表。这是一个程序化决策，不是模型推理的产物。

## 为什么压缩必须是程序决策

压缩操作有几个不适合交给模型的特性：

**1. 结构正确性**

消息历史有严格的格式约束（user/assistant 必须交替，tool_use 必须有对应 tool_result，compact 边界消息有特定格式）。程序可以完全保证这些约束，模型不能。让模型「决定怎么压缩」会产生格式不合规的消息列表。

**2. 确定性**

用户每次 `/compact` 应该得到一致的结果（相同的消息历史产生相同的压缩方案）。模型输出有随机性，不满足这个要求。

**3. 速度**

压缩操作经常在会话上下文将满时触发，用户希望立刻看到结果。如果还要等一轮模型推理，延迟会很明显。

**4. 不需要推理**

「哪些消息应该保留」这个判断，Claude Code 有一套明确的规则（保留技能调用、保留 compact 边界消息、保留最近 N 条消息等）。这是可以被算法精确实现的，不需要模型的自然语言理解能力。

## `skip` 类型：命令什么都不留

`local` 命令还有第三种返回类型：

```typescript
if (result.type === 'skip') {
  return {
    messages: [],
    shouldQuery: false,
    command
  }
}
```

`skip` 表示这次命令执行后不在 transcript 里留下任何消息。这用于那些「改了内部状态但不需要告知模型」的命令——比如切换某个内部标志位，不需要让模型知道。

## `text` 类型：最常见的情况

大多数 `local` 命令返回 `text`：

```typescript
return {
  messages: [
    userMessage,
    createCommandInputMessage(`<local-command-stdout>${result.value}</local-command-stdout>`)
  ],
  shouldQuery: false,
  command,
  resultText: result.value
}
```

注意使用 `createCommandInputMessage` 而不是 `createUserMessage`——command input 消息是 system 子类型，在 UI 里渲染成系统消息样式，不是对话气泡。用户看到的是「`/cost` → `Session cost: $0.12`」这样的系统反馈，不是模型回复。

## local 命令的设计原则

`local` 命令的存在说明一件事：**好的 agent 知道什么时候不该调用模型**。

把所有操作都发给模型处理，表面上是「充分利用 AI」，实际上是把程序能精确完成的工作外包给概率性推理，结果更慢、更贵、更不可靠。

`local` 命令是 Claude Code 对这个边界的明确承认：「这件事由程序做更好。」

---

*面试指导：被问到「什么时候应该用程序逻辑而不是 AI 推理」时，`local` 命令的设计是一个很好的例子。核心判断标准：结果需要确定性、有明确算法实现、不依赖自然语言理解，这三点同时满足时应该用程序，不是模型。*
