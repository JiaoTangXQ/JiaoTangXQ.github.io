# 为什么 local 命令为什么代表本地程序直接完成事情

如果 `prompt` 命令是在改写会话工作方式，那么 `local` 命令做的就是另一件事：它承认有些事情根本不该绕模型一圈。

`getMessagesForSlashCommand()` 处理 `local` 命令时，会懒加载本地模块，直接调用它的实现，然后把结果作为本地输出写回当前会话。它既可以返回普通文本，也可以返回 `compact` 这种会直接重写消息结构的结果，甚至还能选择 `skip`，表示这次命令根本不该留下消息。整个过程里，系统并没有先把命令翻译给模型，再等模型决定怎么做，而是由本地程序直接把事办完。

这背后的判断很成熟。像清空、压缩、列文件、读状态这类能力，很多时候更适合由程序精确执行，因为它们的结果应该稳定、直接、可预测。强行让模型参与，只会把简单事情变得更慢、更松散，还可能引入不必要的歧义。

所以 `local` 命令的重要意义，不只是“有些命令在本地跑”。更重要的是，Claude Code 在这里公开承认：不是所有能力都值得被包装成一次模型推理。真正好的 agent，知道哪里该交给模型，哪里该交给程序。

## 实现链
`local` 命令直接 `load()` 模块并本地执行，返回的可能是文本、compact 结果或者 `skip`。也就是说，有些事根本不需要走模型，只要本地程序自己做完再把结果写回 transcript。

## 普通做法
另一种常见设计是所有命令都统一转成 prompt，让模型决定后续动作。

## 为什么不用
因为像重命名、配置、清理这种操作，走模型只会增加延迟和不确定性。本地能确定完成的事，没必要让模型绕一圈。

## 代价
命令系统必须同时维护“模型型命令”和“程序型命令”两套心智模型；但这避免了把所有事情都做成模型代理造成的臃肿。
