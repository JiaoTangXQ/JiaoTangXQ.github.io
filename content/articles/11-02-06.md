# 副作用不散落回调里而是收回总闸门

真正成熟的地方，不在状态字段多，而在副作用没有散落。`onChangeAppState()` 把权限模式同步、模型持久化、配置写回、环境变量重打、认证缓存清理这些善后动作收回一个总闸门。

这样无论状态是从键盘、命令、桥接还是对话框被改掉，后面的世界都按同一套秩序跟上。系统不会因为入口变多就越长越乱。

## 实现链

`onChangeAppState()` 会比较新旧状态：权限模式变化时发 `notifySessionMetadataChanged()` 和 `notifyPermissionModeChanged()`，模型变化时回写 settings，`settings.env` 变化时重跑 `applyConfigEnvironmentVariables()`。副作用不是散在每个按钮 handler 里，而是统一收回这个 diff 闸门。

## 普通做法

更常见的写法，是哪个交互改了状态，就在那个地方顺手做持久化、通知和环境同步。

## 为什么不用

Claude Code 不这么分散，是因为这里的状态来源很多，可能来自键盘、配置文件、远端元数据或别的系统入口。副作用要是跟着入口走，迟早会漏同步或顺序不一致。

## 代价

代价是 `onChangeAppState()` 会变成一条必须小心维护的总闸门；但它至少把“状态变了以后哪些世界要跟着变”集中在一个地方说清楚。
