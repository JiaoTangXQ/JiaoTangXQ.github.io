# 一个子 Agent 从创建到回收经历了什么？

跟踪一个 `local_agent` 任务从出生到消亡的完整路径，是理解 Claude Code 任务框架最直接的方式。

## 第一步：出生——获得骨架

任务的生命从 `createTaskStateBase()` 开始：

```typescript
export function createTaskStateBase(
  id: string,
  type: TaskType,
  description: string,
  toolUseId?: string,
): TaskStateBase {
  return {
    id,
    type,
    status: 'pending',    // 出生即 pending
    description,
    toolUseId,
    startTime: Date.now(),
    outputFile: getTaskOutputPath(id),
    outputOffset: 0,
    notified: false,
  }
}
```

注意初始状态是 `pending` 而不是 `running`。这个区别很重要：任务已经存在于系统中，但还没真正开始执行。`outputOffset: 0` 表示还没有任何输出被读取过。`notified: false` 表示完成通知还没发出。

对于 `local_agent` 任务，`LocalAgentTask.tsx` 在这个骨架上追加执行字段，然后**直接把 status 改成 `running`**（不经过 `pending` 阶段）：

```typescript
const taskState: LocalAgentTaskState = {
  ...createTaskStateBase(agentId, 'local_agent', description, toolUseId),
  type: 'local_agent',
  status: 'running',  // 立刻进入运行态
  agentId,
  prompt,
  // ...更多字段
  pendingMessages: [],
  retain: false,
  diskLoaded: false
}
```

## 第二步：登记——进入 AppState

`registerTask(taskState, setAppState)` 把任务放进全局状态字典：

```typescript
export function registerTask(task: TaskState, setAppState: SetAppState): void {
  let isReplacement = false
  setAppState(prev => {
    const existing = prev.tasks[task.id]
    isReplacement = existing !== undefined
    // 如果已存在同 ID 任务，保留 UI 持有的字段
    const merged = existing && 'retain' in existing
      ? {
          ...task,
          retain: existing.retain,
          startTime: existing.startTime,
          messages: existing.messages,
          diskLoaded: existing.diskLoaded,
          pendingMessages: existing.pendingMessages,
        }
      : task
    return { ...prev, tasks: { ...prev.tasks, [task.id]: merged } }
  })

  if (isReplacement) return  // 替换不算新建，跳过 task_started 事件

  enqueueSdkEvent({ type: 'system', subtype: 'task_started', ... })
}
```

关键细节：如果同 ID 任务已存在（`resumeAgentBackground` 会重新注册同一个 Agent），系统会**保留** `retain`、`messages`、`diskLoaded`、`pendingMessages`。用户正在查看的 transcript 不会被新的注册冲掉。

## 第三步：运行——进度更新和输出写入

运行期间，两条路并行：

**进度路径**：每次 Agent 发出 assistant 消息，`updateProgressFromMessage()` 更新 `ProgressTracker`（工具调用次数、token 数、最近活动），然后 `updateAgentProgress()` 把结果写回任务状态的 `progress` 字段。

**输出路径**：`initTaskOutputAsSymlink()` 在任务创建时就把 `outputFile` 指向 Agent 的 transcript 文件（软链接）。框架的轮询循环通过 `getTaskOutputDelta(taskId, outputOffset)` 增量读取新内容，更新 `outputOffset`。

## 第四步：终止——先改状态再善后

任务可以以三种方式结束：

- `completeAgentTask()`：正常完成
- `failAgentTask()`：出错失败
- `killAsyncAgent()`：被强制终止

三者的代码结构几乎相同，都遵循同一个顺序：

```typescript
export function completeAgentTask(result, setAppState) {
  updateTaskState<LocalAgentTaskState>(taskId, setAppState, task => {
    if (task.status !== 'running') return task  // 幂等检查

    task.unregisterCleanup?.()  // 注销清理钩子

    return {
      ...task,
      status: 'completed',           // 1. 先改状态
      result,
      endTime: Date.now(),
      evictAfter: task.retain        // 2. 设驱逐时间（若未被持有）
        ? undefined
        : Date.now() + PANEL_GRACE_MS,
      abortController: undefined,    // 3. 清空执行资源
      unregisterCleanup: undefined,
      selectedAgent: undefined
    }
  })
  void evictTaskOutput(taskId)       // 4. 最后才清理磁盘输出
}
```

**先改状态，再做善后**。这个顺序防止别的模块在清理期间还认为任务是活的。

## 第五步：通知——回到消息队列

任务完成后，`enqueueAgentNotification()` 生成一段 XML 送进主消息队列：

```xml
<task-notification>
<task-id>a8kj2m9x</task-id>
<output-file>/tmp/.../a8kj2m9x.output</output-file>
<status>completed</status>
<summary>Agent "分析日志" completed</summary>
<result>发现 3 个异常请求...</result>
<usage><total_tokens>4521</total_tokens><tool_uses>12</tool_uses></usage>
</task-notification>
```

`notified` 标志在发出通知前原子性置为 `true`，防止重复通知。

## 第六步：驱逐——体面退场

任务不会在终止后立刻消失。`evictAfter` 字段保存的是驱逐截止时间（`Date.now() + PANEL_GRACE_MS`，默认 30 秒）。只有当：

1. 任务处于终态（`completed`/`failed`/`killed`）
2. `notified === true`
3. `evictAfter` 已过期
4. `retain === false`（用户没有持续查看）

这四个条件同时满足，`evictTerminalTask()` 才会从 `AppState` 里删掉这个任务。

## 面试要点

这个生命周期回答了一个系统设计问题：**如何让任务的"逻辑结束"和"物理清理"解耦？**

Claude Code 的答案是两个字段：`notified`（通知已发）和 `evictAfter`（驱逐时间）。任务逻辑上结束（`status: 'completed'`）之后，它在内存中还会多活一段时间，直到用户读完、通知确认、宽限期结束。这让"任务结束"和"任务消失"成为两个独立事件，中间留出了人类阅读的时间窗口。
