# 上下文治理为什么是分层修复而不是一次性整理？

## 问题的起点

当对话越来越长，token 用量接近模型上限，该怎么办？

最简单的答案是「压缩历史」。但「压缩」其实是一个大类，包含截然不同的操作：有的很便宜，有的代价很高；有的保留细节，有的会丢失信息。把它们全归成「压缩」是一种认知懒惰。

Claude Code 把上下文治理拆成了三层，每层对应不同的代价和适用场景。

## 第一层：collapse_drain_retry

`contextCollapse.recoverFromOverflow()` 是第一层。它处理的是「已经被 staged 但还没有排干的 collapse」。

什么叫 staged collapse？上下文折叠是预先标记好的——系统在某些时机会把一段历史标记为「可以被折叠」，但不立即执行。当真的超限时，先把这些已标记的折叠排干（drain），就能在不重写历史的情况下腾出空间。

```typescript
// query.ts
if (contextCollapse && state.transition?.reason !== 'collapse_drain_retry') {
  const drained = contextCollapse.recoverFromOverflow(...)
  if (drained) {
    state.transition = { reason: 'collapse_drain_retry' }
  }
}
```

注意这里的防重入判断：`state.transition?.reason !== 'collapse_drain_retry'`。如果上一轮已经跑过 drain 了，就不再重复，否则会死循环。

**代价**：低。排干已标记的 collapse 不需要调用模型，不需要生成摘要，只是把预先标记好的折叠真正执行。

## 第二层：reactive_compact_retry

第一层排干还不够时，才走第二层：`reactiveCompact.tryReactiveCompact()`。

这一层会真正去生成新的 post-compact messages——通常意味着调用一次模型来压缩历史，代价明显高于 drain。

```typescript
// query.ts
if ((isWithheld413 || isWithheldMedia) && reactiveCompact) {
  const compacted = await reactiveCompact.tryReactiveCompact({...})
  if (compacted) {
    state.transition = { reason: 'reactive_compact_retry' }
  }
}
```

**代价**：中高。需要一次额外的模型调用，有延迟和 token 成本。但压缩后的历史更精简，可以让当前对话继续更长时间。

## 第三层：token_budget_continuation

第三层不是处理超限，而是处理「预算快用完了但还没超」的情况。`checkTokenBudget()` 基于百分比和 diminishing returns 判断是否注入续写提示：

```typescript
// query/tokenBudget.ts
const isDiminishing =
  tracker.continuationCount >= 3 &&
  deltaSinceLastCheck < DIMINISHING_THRESHOLD  // 500 tokens
```

如果每一轮输出越来越少（递减收益），即使还有预算，系统也会判断「继续下去意义不大」而停止续写。

**代价**：低，但有精度损失风险。系统注入 meta 消息指导模型接着写，但不保证连贯性。

## 为什么顺序不能颠倒

这三层的顺序不是随意的：

1. drain 最便宜，先试
2. drain 不够再 compact，代价更高
3. token_budget 是运营时的续写控制，和前两层的触发条件不同

如果把 reactive_compact 放在第一层，大量可以通过 drain 解决的情况都会去调用一次模型，无谓增加延迟和成本。

## 面试要点

被问「你们怎么处理 context window 超限」时，这个分层设计是一个很好的答案框架：

1. 先分清楚超限的类型（硬截断 vs 预算不足 vs staged 折叠未排干）
2. 对不同类型用不同代价的修复手段
3. 防重入——不让同一种修复在同一轮里触发两次

Claude Code 的上下文治理不是「一把刀砍到底」，而是根据代价从低到高依次尝试的修复链。
