# 第十二卷导读：外延执行不是第二运行时

## 这卷真正要回答什么

第十二卷不是在炫 Claude Code 又多了几个“能操作设备”的功能，而是在回答一个更重要的问题：当能力真的伸到桌面、浏览器和真实账号世界时，系统为什么没有因此长出第二套运行时。

代码里的答案很具体。`setupComputerUseMCP()` 和 `setupClaudeInChrome()` 都不是直接把新能力裸接进主循环，而是先包装成动态 MCP 配置、统一命名的工具集合和配套 system prompt；`createChromeContext()` 又把浏览器桥、配对身份、权限模式、日志上报和 sideQuery 推理塞回同一套宿主语言里。也就是说，新世界进来时，Claude Code 先做的是翻译和收编，不是另建一套私有协议。

更容易想到的做法，是把 browser / desktop / remote capability 做成一个独立 agent 子系统：自己管理连接、自己定义命令、自己处理权限、自己收日志。那样看起来更直接，但最后会出现两套心智模型。Claude Code 没选这条路，所以这一卷要看的重点从来不是“它能点哪里”，而是“它为什么坚持先说 MCP、先过权限、先接回原工作台”。

## 为什么这一卷不能写成“新能力巡礼”

如果只按功能表来写，第十二卷会很像两样东西：一堆桌面工具，外加一套浏览器自动化。但源码根本不是这么组织的。`computerUse` 的关键不是鼠标键盘本身，而是它为什么以 `mcp__computer-use__*` 的名字进入系统，并且用整场 `request_access` 授权替代每一步弹窗。Claude in Chrome 的关键也不是它会点网页，而是 `setup.ts` 和 `mcpServer.ts` 为什么宁可多写 wrapper、manifest、socket、bridge 和持久身份，也不愿意把浏览器世界做成一条独立旁路。

所以这一卷的中心判断只有一句：边界确实在扩张，但骨架没有重写。看懂这一点，后面的每一页才不会被误读成“又多了一套花哨功能”。

## 实现链
真正落地时，这卷对应的是两条接入链。`main.tsx` 先决定要不要把 `setupComputerUseMCP()` 和 `setupClaudeInChrome()` 产出的动态 MCP 配置并进当前 session；随后 `createChromeContext()`、`wrapper.tsx` 和各自的 `toolRendering.tsx` 再把浏览器、桌面、权限、日志和 UI 展示接回原来的工具运行时。也就是说，新能力没有绕开主循环，而是先被翻译成现有系统听得懂的 MCP 工具、权限模式和状态回写。

## 普通做法
普通产品更容易把“桌面控制”和“浏览器控制”各做成独立子系统，自己有连接协议、自己有权限逻辑、自己有一套提示词。

## 为什么不用
Claude Code 没这么做，是因为它更看重“同一工作台里的连续性”。一旦外延能力说的是另一种语言，`QueryEngine`、权限链、task、恢复链和 UI 就都要重新适配第二遍。

## 代价
代价是接入层会明显变厚，`setup.ts`、`mcpServer.ts`、`common.ts`、`wrapper.tsx` 这些文件都在替外部世界做翻译；但这比长期维护第二套运行时便宜。
