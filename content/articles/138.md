# 深度分析：工具发现机制如何让所有工具说同一种语言

## 工具多了以后的发现问题

Claude Code 早期只有十几个核心工具（BashTool、ReadTool、EditTool 等）。AI 直接把所有工具放进上下文，每次对话都能看到全部工具。

随着 MCP 工具、浏览器工具、插件工具不断加入，工具数量可能达到几十甚至上百。如果所有工具都放进上下文，会消耗大量 token，而且 AI 要在几百个工具里找到最合适的那个，效果并不好。

这就是需要 `ToolSearch` 的背景：**按需发现工具，而不是一次加载所有工具**。

## ToolSearch 的工作原理

`ToolSearch` 的基本模型是：

1. AI 在生成响应时，如果需要某类工具，发出一个工具搜索请求
2. 系统根据搜索词（工具名、功能描述、工具类别）找到最匹配的工具
3. 把匹配的工具定义注入 AI 的上下文
4. AI 用这些工具完成任务

这个模型让 AI 不需要在对话开始时就了解所有工具，而是在需要时才加载相关工具。

**工具前缀在这里的作用**：当 AI 搜索"浏览器相关工具"时，带有 `_mcp` 后缀或 `computer_use_` 前缀的工具名，让搜索可以通过前缀过滤快速定位到相关工具，而不需要逐一检查工具描述。

## 技能提示（Skill Hints）

除了按需搜索，Claude Code 还有"技能提示"机制：当用户提到某些关键词（比如"浏览器"、"网页"、"截图"），系统可以主动把相关工具类别的提示加入上下文，告诉 AI"这个任务可能需要浏览器操作工具，你可以搜索 tabs_context_mcp 类工具"。

技能提示和 ToolSearch 配合：
- 技能提示告诉 AI"这里有某类工具可用"（降低工具发现的门槛）
- ToolSearch 是 AI 主动查找工具的机制（精确定位需要的工具）

外延工具要融入这个体系，需要被包含在技能提示的映射里：当用户的输入匹配某些关键词时，技能提示里能出现"你可以使用浏览器工具"这样的引导。

## 工具前缀让发现可扩展

没有工具前缀时，新增一个工具类型需要：
1. 更新技能提示配置（加入新类型的工具）
2. 更新 ToolSearch 的搜索逻辑（支持新类型的搜索）
3. 更新权限检查（识别新类型的工具）
4. 更新 UI 渲染（展示新类型工具调用的样式）

有了工具前缀，新增工具类型：
1. 确定前缀约定（比如 `ai_search_` 用于 AI 搜索工具）
2. 实现工具并用这个前缀命名

其他地方通过检查前缀自动支持新类型，不需要逐一更新。

这就是"发现方式即骨架"的含义：工具的命名约定（前缀）让整个发现、权限、展示体系能自动适应新工具，发现机制本身就是骨架的一部分。

## 连远了以后系统还像一个整体

当工具数量从 10 个扩展到 100 个，工具覆盖从终端扩展到浏览器再到桌面，系统"还像一个整体"的感觉来自哪里？

**来自统一的发现机制**：无论是核心工具还是浏览器工具，都通过 ToolSearch 被发现，都有统一的工具定义格式，都通过相同的 API 调用。

**来自统一的语言**：工具前缀约定让所有工具的命名遵循可预期的模式，新成员加入时能快速理解工具体系的结构。

**来自统一的权限体系**：所有工具的权限检查走同一条路，不存在"这个工具的权限在这里检查，那个工具的权限在那里检查"。

这种统一性不是偶然的，而是"先确定命名约定、先确定接入标准、先确定发现机制"的工程纪律带来的结果。

## 面试要点

1. **ToolSearch 是按需发现工具的机制**：解决了工具数量增长后"全量加载"效率低的问题，工具前缀让搜索更高效。
2. **技能提示是主动引导，ToolSearch 是主动查找**：两者配合，降低 AI 发现工具的门槛，又支持精确定位。
3. **工具前缀让新类型自动集成**：前缀约定让发现、权限、展示体系通过检查前缀自动支持新工具类型，不需要逐一修改。
4. **"还像整体"来自统一基础设施**：发现机制、命名约定、权限体系的统一，是系统扩展到 100 个工具后仍然可管理的工程基础。
