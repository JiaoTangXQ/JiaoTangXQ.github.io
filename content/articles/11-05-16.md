# 状态栏压缩掉的是次要细节，说明它只保留当前世界里最该先知道的那一截

同一条状态里，真正值得长期露出来的不是所有变化，而是那些会改写人下一步行动的变化。模式、权限、远端、PR、工作目录这些会直接影响行为的信息，才值得占住那一行。

这也是摘要层的边界：它能压缩，但不能把决定行动的事实压没。压缩之后还剩下的那一截，必须仍然够人立刻判断自己现在在哪。

## 实现链

无论是 `Byline` 的单行拼接，还是 `Text wrap="truncate"` 的尾部截断，底部代码都在表达同一件事：状态栏默认只保留当前最该先知道的一截，比如模式、权限、远端、任务、PR 这类会改写行动判断的事实，次要细节则延后或折叠。

## 普通做法

更普通的设计会努力把所有状态都塞进底部，结果变成一条谁也看不完的流水账。

## 为什么不用

Claude Code 不把状态栏做成流水账，是因为摘要层的任务不是“尽量全”，而是“先告诉你现在最需要知道什么”。底部只有一行，必须先保会改写行动的事实。

## 代价

代价是细节天然会被牺牲，用户得去别的界面层补全；但这一行因此还能承担真正的摘要职责。
