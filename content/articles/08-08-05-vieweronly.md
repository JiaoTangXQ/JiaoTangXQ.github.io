# 面试题：为什么历史分页只在 viewerOnly 模式下启用？

## 问题的表面

`useAssistantHistory.ts` 开头有一个硬性检查：

```typescript
if (config?.viewerOnly !== true) {
  return // no-op
}
```

非 viewerOnly 模式，函数直接返回，什么都不做。

表面上，这看起来是一个功能限制：live 远端控制模式下，用户不能翻历史，只能看当前的实时流。这似乎不合理——同样是远端消息，为什么控制模式下看不了历史？

## 深层原因：两种模式的状态管理复杂度不同

viewerOnly 模式的消息流是**单向的**：

```
服务端推送消息 → 本地接收 → 转换 → 渲染
```

没有本地发出的消息（不能控制），没有 echo（不会收到自己发出的消息），没有权限审批的状态机。消息列表是一个相对干净的"读取-显示"管线。

live 远端控制模式的状态机更复杂：

```
本地发消息 → 服务端接收 → echo 回来 → 需要去重
服务端请求权限 → 本地审批 UI → 回复响应 → 服务端恢复执行
工具开始执行 → 进度状态 → 工具完成 → 清除进度 → 显示结果
实时消息持续到来 → 消息列表持续更新
```

在这个复杂的状态机里，加入历史前插会带来什么问题？

### 问题一：去重集合的干扰

live 模式维护了两个去重集合：`recentPostedUUIDs`（过滤 echo）和 `recentInboundUUIDs`（过滤重放）。

如果加入历史前插，历史消息的 UUID 也会进入 `recentInboundUUIDs`。问题来了：

历史中可能包含**本地之前发出去的消息**（在 `recentPostedUUIDs` 里的消息，在 echo 过滤后应该显示在本地，但也会被记录进历史）。如果这些 UUID 后来又被实时流重新推送（服务端重放），会被去重过滤掉——但这次是合理的新消息，不是重放。

在 viewerOnly 里，因为没有本地发消息，这类交叉不会发生。

### 问题二：滚动锚定与实时消息的竞争

live 模式下，实时消息可能在用户翻历史的同时到来，被追加到列表末尾。同时，历史消息在被前插到列表最前面。

两个方向同时更新列表，滚动锚定的基准（sentinel 位置）会因为末尾追加而改变，影响前插的偏移量计算。

在 viewerOnly 里，虽然实时消息也会到来，但因为没有权限审批状态机，消息流相对稳定，干扰更可控。

### 问题三：不一致的滚动行为

live 模式有一个"新消息到来时，自动滚到底部"的行为（如果用户当前在底部），这是即时通信的标准 UX。

如果同时有历史前插（把视口往下推）和自动滚底（把视口推到最底），这两个行为会互相竞争，导致视口位置不稳定。

## 功能隔离换清晰度

Claude Code 的选择是：**不在 live 模式里加历史分页，保持 live 模式的状态机简洁**。

代价：live 远端控制的用户只能看实时消息，不能往回翻历史。

收益：两种模式各自的行为是清晰的，不会互相干扰。live 模式可以安全地假设"消息列表只会被实时消息追加"，不需要处理前插的情况。

这是一个**功能隔离（feature isolation）**的工程决策：把两个性质不同的功能（实时控制 vs 历史阅读）分配给两种不同的模式，让每种模式只处理自己范围内的复杂度。

## 产品层面的合理性

这个技术决策在产品层面也有支持：

- **live 远端控制的用户**：关注的是"现在正在发生什么"，不是"之前发生了什么"
- **viewerOnly 的用户**：关注的是"完整的会话上下文"，包括历史和现在

两种用户的需求不同，功能设计自然也不同。"因为技术上难以同时支持"和"因为产品上不同用户有不同需求"，这两个理由刚好在同一个方向上指向了相同的设计结论。

## 面试指导

"为什么不把功能做全"是面试中考验技术判断力的好题。

答题要点：

1. **说出技术约束**：两种模式同时支持历史分页的具体技术难点（去重干扰、滚动竞争）
2. **说出产品逻辑**：两种模式的用户需求本就不同，功能不对称是合理的
3. **承认取舍**：live 模式用户失去了历史翻页能力，这个损失需要被明确承认，而不是假装"不需要"

好的系统设计，是对"做什么"和"不做什么"都有清晰的理由，而不只是"我们做到了什么"。
