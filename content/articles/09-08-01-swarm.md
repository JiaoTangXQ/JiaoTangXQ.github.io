# 任务系统和 Swarm 系统的边界：两个都能"让 agent 工作"，职责区别在哪里？

Claude Code 里存在两套管理 agent 工作的基础设施：

**任务系统**：`Task.ts`、`framework.ts`、`LocalAgentTask.tsx`、`InProcessTeammateTask.tsx`

**Swarm 系统**：`utils/swarm/`、`teammateMailbox.ts`、`permissionSync.ts`、`leaderPermissionBridge.ts`

两套系统都能"让 agent 工作"。那它们的职责边界在哪里？

## 任务系统回答的问题

任务系统关心的核心问题：**这件工作的状态是什么，结果怎么处理？**

具体来说：
- 任务怎么被注册（`registerTask()`）
- 任务的生命周期状态（running / completed / killed）
- 进度怎么追踪（`progressTracker`）
- 完成后怎么通知（`onIdleCallbacks`）
- 任务对象怎么在 AppState 里存储和检索
- 任务结束后怎么回收（`evictTerminalTask()`）

任务系统的抽象是"工作单元"：一个工作单元开始、进行、完成、结束。它不关心做工作的主体是谁、怎么协调、权限怎么批。

## Swarm 系统回答的问题

Swarm 系统关心的核心问题：**谁在这个组织里，他们怎么协作，秩序怎么维持？**

具体来说：
- 成员身份（`TeammateIdentity`：agentId、团队、颜色、planModeRequired）
- 成员通信（`teammateMailbox.ts`：文件信箱，显式消息传递）
- 权限协调（`permissionSync.ts`：pending/resolved 目录；`leaderPermissionBridge.ts`：跨层桥接）
- 执行后端（`registry.ts`：tmux vs in-process vs iTerm2）
- 空间布局（`TmuxBackend.ts`：pane 创建、颜色、位置）
- 退场秩序（shutdown request/response 协议）

Swarm 系统的抽象是"组织制度"：多个工作主体如何有秩序地共存和协作。

## 两套系统的交叉点

`InProcessTeammateTask.tsx` 是两套系统的交叉点，它同时需要两者：

**从任务系统继承**：
- `InProcessTeammateTaskState` extends task state（有 id、status、description、toolUseId 等任务字段）
- 通过 `registerTask()` 注册到 AppState.tasks
- 通过 `evictTerminalTask()` 在完成后清理
- 通过 `onIdleCallbacks` 通知等待者

**从 Swarm 系统增加**：
- `identity: TeammateIdentity`（谁在做这个工作）
- `permissionMode`（以什么权限模式工作）
- `shutdownRequested`（是否被请求优雅退出）
- `pendingUserMessages`（来自 mailbox 的消息队列）

一个 `InProcessTeammateTaskState` 对象，既是任务系统里的一个"工作单元"，也是 Swarm 系统里的一个"成员档案"。

## 为什么不合并成一套系统

有人可能会问：既然 `InProcessTeammateTaskState` 同时包含两套系统的字段，为什么不干脆合并成一套系统？

答案：**它们的演进方向不同**。

任务系统未来可能要支持新的任务类型（代码生成任务、搜索任务、长时间运行的后台任务），这些变化和 Swarm 无关。任务系统的核心不变：工作单元的注册、状态管理、回收。

Swarm 系统未来可能要支持新的后端（比如 SSH 远程 agent、容器化 agent），新的通信协议（比如 WebSocket 而不是文件），新的权限模型（比如细粒度的工具级别权限）。这些变化和任务系统的生命周期管理无关。

两套系统的关注点不同，演进轨迹不同，分开维护让两边都能独立修改而不影响对方。`InProcessTeammateTaskState` 是两者的合并点，但它是特定场景的合并，不是整个系统的合并。

## 类比：操作系统的调度器和网络栈

操作系统里有调度器（决定 CPU 时间片怎么分配）和网络栈（决定数据包怎么路由）。两者都运行在同一个内核里，都影响进程如何工作，但职责完全不同：

- 调度器关心"哪个进程现在跑，跑多久"
- 网络栈关心"数据怎么在进程间和跨网络传输"

一个网络请求的处理涉及调度器（调度网络中断处理）和网络栈（协议解析）的协同，但这不代表两者应该合并。它们是不同层次的抽象，独立设计让每一层都能被独立优化和替换。

Claude Code 的任务系统和 Swarm 系统是同样的关系：不同层次的抽象，交叉点是 `InProcessTeammateTaskState`，但系统边界是清晰的。

## 面试怎么答

如果面试题是"多 agent 系统应该有几套基础设施，怎么划分职责"：

**思考框架**：先问"系统需要回答哪些根本不同的问题"。如果这些问题用同一套抽象都能表达，就一套系统；如果它们需要不同的抽象（比如"工作状态"和"组织成员关系"），就分开。

**Claude Code 的划分**：任务系统的核心抽象是"工作单元"（生命周期、状态、回收），Swarm 系统的核心抽象是"组织成员"（身份、通信、权限、空间）。这两个抽象服务不同的问题域，分开是合理的。

**判断依据**：如果合并成一套系统，当你只需要修改"工作状态管理"时，你是否需要理解"组织通信协议"？如果是，说明合并过度，职责边界模糊了。Claude Code 的分法让每套系统可以独立理解和修改。
