# PR 状态只在空间和主项都允许时才露脸，说明摘要层懂得屏幕稀缺

底部区不是发现有 PR 就硬贴一个 badge。它会先看当前已经有多少主项要显示，再看终端宽度够不够。也就是说，PR 信息虽然重要，但它仍然要服从眼前这块屏幕的预算，不能为了多报一点，把真正的主现实挤乱。

成熟摘要层不是“能显示就显示”。它知道窄屏里谁该先活下来，谁该稍后再说。

## 实现链

`PromptInputFooterLeftSide.tsx` 里的 `shouldShowPrStatus` 不只看有没有 PR，还要求 `primaryItemCount < 2`，并且在有主项时 `columns >= 80`。PR 状态只有在主项和屏幕预算都允许时，才有资格露脸。

## 普通做法

更普通的做法会觉得“既然有 PR 信息就应该一直显示”，最多让它和别的标签一起挤在底部。

## 为什么不用

Claude Code 不这么执着，是因为底部一行是硬预算，先保模式、任务和团队这类主现实，比补充 PR 状态更重要。摘要层懂得屏幕稀缺，不会把所有事实都抢成第一屏。

## 代价

代价是某些真实会被故意延后，不是任何时候都能第一眼看到；但这正是它在做摘要，而不是原样转录。
