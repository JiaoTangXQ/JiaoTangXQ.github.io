# 对比分析：远端消息适配器，是「翻译」还是「降级」？

## 适配器的两种工作模式

`sdkMessageAdapter.ts` 把远端的 `SDKMessage` 转成本地的 `Message`。这个转换有两种模式：

**等价翻译**：源语义和目标语义完全对应，只是格式不同。`user` 消息、`assistant` 消息、结果消息，都是等价翻译——字段名称可能不同，但信息量保持不变。

**有损降级**：源格式有本地格式无法完整表达的语义，翻译时需要丢弃一部分信息。`tool_progress` → `system informational message` 是最典型的有损降级。

## tool_progress 的降级

远端的 `tool_progress` 消息带有丰富的进度信息：

- 工具 ID（哪个工具实例在运行）
- 进度描述（文本形式的当前状态）
- 可能还有进度百分比或中间输出

本地的进度消息（`ProgressMessage`）是深度集成的：它和本地工具的执行上下文绑定，有专门的渲染组件，可以显示精细的进度条和动态输出。

但本地工具执行上下文，只有在**本地实际执行工具**时才存在。远端工具在 CCR 容器里执行，本地根本没有这个上下文。

怎么办？降级处理：把进度信息转成普通的 `system informational message`——一段文字说明"工具 X 正在做 Y"，然后以普通系统提示的形式显示出来。

代价：失去了专用进度 UI（进度条、实时输出流），显示效果比本地执行更朴素。

收益：这段进度信息能进入现有消息渲染流，而不需要为远端进度单独开发一套渲染组件。

## 翻译而不是并行体系

另一种可能的设计是：**为远端场景建立一套独立的消息类型**，然后写独立的渲染组件。

这样的好处是远端消息可以保留完整的语义，不需要适配本地格式。坏处是：

- 消息搜索、历史分页需要同时理解两套类型
- 工具结果显示、压缩分割线等功能需要重复实现
- 新增本地功能时，需要同步维护两套实现

Claude Code 选择了"强制翻译"的策略：**所有消息必须转成本地 Message 格式才能进入 UI**。这个约束让 UI 层保持了单一的数据模型，代价是部分远端消息需要有损降级。

## 翻译的不对称性

值得注意的是，翻译是不对称的：

- **入方向（远端 → 本地）**：`SDKMessage → Message`，由 adapter 负责
- **出方向（本地 → 远端）**：`Message → SDKMessage`，由 `toSDKMessages` 函数负责，并通过依赖注入传入 bridge core

出方向的翻译有一个额外的过滤步骤（`isEligibleBridgeMessage`）：本地的 virtual 消息、display-only 消息、内部状态消息，都不应该发给远端——远端不关心本地 UI 的内部状态。

这个过滤说明：**"能进入本地 UI 的消息"和"应该发给远端的消息"是两个不同的集合**。Adapter 负责入方向的映射，filter 负责出方向的筛选。

## `useAssistantHistory.ts` 的复用

设计正确的翻译层，最直接的红利是代码复用。

`useAssistantHistory.ts` 是 viewerOnly 模式（只读观看另一个会话）的历史加载逻辑。它同样使用 `sdkMessageAdapter`，把从服务端拉取的历史消息转成本地 Message，然后投喂给同一套渲染组件。

这意味着：

- 主会话的消息（实时推送）
- viewerOnly 的历史消息（HTTP 拉取）

使用完全相同的渲染路径。Adapter 层的正确性，保证了两种来源的消息在 UI 里表现一致。

## 面试指导

"如何设计消息格式的兼容层"是后端/全栈面试的常见题。

几个值得讨论的点：

1. **等价翻译 vs 有损降级**：判断哪些语义可以保留，哪些需要放弃，放弃的代价是什么
2. **单一数据模型 vs 并行数据模型**：强制翻译 vs 双轨并行，各自的维护成本
3. **翻译的方向性**：入站和出站翻译的逻辑往往不对称，需要分开设计
4. **复用的证明**：好的翻译层设计，体现在"不同来源的消息能共享同一套渲染逻辑"

Claude Code 的 adapter 是"单一数据模型"策略的一个具体实现，以有损降级为代价，换取 UI 层的简洁性和可复用性。这个取舍是否正确，取决于业务对进度消息精细度的实际需求——在 Claude Code 当前阶段，显然判断为"够用"。
