# Claude Code 的权限治理体系总结：六个关键设计决策

读完整个第六章，可以把 Claude Code 的权限体系归纳为六个核心设计决策。每个决策都有其明确的工程动机和取舍。

## 决策一：制度快照（ToolPermissionContext）

**决策**：所有权限检查读内存快照，不实时读磁盘。

**动机**：agent 运行时需要一致的权限状态。如果工具装配时读到 allow，运行时磁盘被修改变成 deny，行为不一致会导致安全隐患。

**取舍**：快照可能短暂过期（settings 变化后 1.5 秒才刷新），但对大多数场景影响可忽略。

**核心类型**：
```typescript
type ToolPermissionContext = DeepImmutable<{
  mode: PermissionMode
  alwaysAllowRules: ToolPermissionRulesBySource
  alwaysDenyRules: ToolPermissionRulesBySource
  strippedDangerousRules?: ToolPermissionRulesBySource  // auto 模式暂存
}>
```

## 决策二：双层权限控制

**决策**：装配层做工具可见性控制（工具级 deny），运行时做命令级审批。

**动机**：两层解决不同维度的问题。装配层缩小 Claude 的认知世界，减少规划阶段的无效尝试；运行时处理上下文相关的细粒度判断。

**取舍**：看起来像”重复判断”，实际上是关注点分离。删掉任何一层都会把另一层的问题硬塞给错误的模块。

## 决策三：模式作为状态机

**决策**：七种权限模式（default/plan/acceptEdits/bypassPermissions/dontAsk/auto/bubble）形成有向状态图，切换通过 `transitionPermissionMode` 完成。

**动机**：模式切换不是 UI 换色，而是真正重写当前制度。进入 auto 剥离危险规则，离开 auto 恢复规则，进入 plan 存储前置模式——每条边都有副作用。

**核心逻辑**：
```typescript
export function transitionPermissionMode(fromMode, toMode, context) {
  if (fromMode === toMode) return context
  handlePlanModeTransition(fromMode, toMode)
  handleAutoModeTransition(fromMode, toMode)
  // ... 危险规则剥离/恢复 ...
  return context
}
```

## 决策四：auto 模式的危险规则剥离

**决策**：进入 auto 模式时，`Bash(*)`、`Agent` allow 规则等危险授权被从内存 context 中剥离，存入 `strippedDangerousRules`；离开 auto 时恢复。

**动机**：auto 模式的核心保证是”所有非白名单工具调用都要经过分类器”。预先存在的宽松规则会绕过分类器，破坏这个保证。

**关键设计**：剥离是内存操作，不修改磁盘文件。`isAutoModeActive()` 是权威状态信号，`strippedDangerousRules` 只是数据口袋。

## 决策五：settings 变化的集中传播

**决策**：`fanOut` 函数集中重置缓存再通知监听者，而不是让每个监听者各自重置。

**动机**：N 个监听者各自重置 = N 次磁盘读取（O(N) 读取）。集中重置后，第一个监听者填充缓存，后续监听者命中缓存，总计 1 次磁盘读取。

**性能数据**：”5 loadSettingsFromDisk calls in 12ms when remote managed settings resolved at startup.”

```typescript
function fanOut(source: SettingSource): void {
  resetSettingsCache()       // 先重置（O(1)）
  settingsChanged.emit(source)  // 再通知（1 次磁盘读取）
}
```

## 决策六：ConfigChange hook 的前置阻断权

**决策**：每次 settings 文件变化（包括删除）先执行 ConfigChange hook，如果 hook 返回 exit code 2，这次变化永久不生效，不重试。

**动机**：settings 变化是高风险操作（可能引入危险权限规则）。给管理员一个在变化生效前的最后否决权。

**关键边界**：被阻断的变化不会被记忆或重试。内存状态可能短暂与磁盘不一致（文件删除被阻断时）。重启后重新读磁盘，恢复一致性。

## 整体权力结构

Claude Code 的权限体系有明确的权力等级：

```
policySettings（MDM/企业管控）
  ↓ 可以锁定
userSettings / projectSettings / localSettings
  ↓ 会影响
ToolPermissionContext（内存快照）
  ↓ 决定
工具调用是否被允许
```

这个结构不是线性的：policySettings 通过 `allowManagedPermissionRulesOnly` 和 `allowManagedHooksOnly` 这样的专属字段，主动**忽略**下级来源，而不只是优先级更高。

## 面试总结

如果被问”设计一个 AI agent 的权限系统你会怎么做”，这六个决策是可以直接拿来讨论的框架：

1. **快照 vs 实时**：选快照，解释一致性的价值
2. **装配 vs 运行时**：两层都要，解释各自的责任边界
3. **全局模式 vs 零散开关**：全局模式，解释”制度感知”的价值
4. **宽松规则的危险性**：auto 模式要剥离，解释”自动化不等于无监管”
5. **缓存与通知的顺序**：先重置再通知，解释 N 次读取问题
6. **变化拦截**：ConfigChange hook 的否决权，解释”修改规则本身也需要程序”
