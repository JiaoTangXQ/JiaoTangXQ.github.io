# 第十三卷回收索引：外延执行里同一名字被后端、提示词和渲染共认，能力簇先长出公共现实

## 实现链
外延执行里更深的一层，不只是“认出同一个对象”，  
而是这件事能不能被后端、提示词和渲染一起认成一块公共现实。

这条线真正逼出来的，是第十三卷里关于跨层协同的判断。  
真正成熟的系统，不会让名字只停在一层，而是把它做成各层共同承认的现实。

这页实际回收的实现链是：
- `12-03-02` 把同一种名字写成后端、提示词和渲染共同认出的能力簇，这会继续支撑 `13-03-01`，说明命名不是标签，而是跨层协同的最小公共现实。

很多系统真正稳住的，不只是名字本身。  
而是这名字一旦落下去，后端、提示词和渲染就都知道自己在说同一个能力簇。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
