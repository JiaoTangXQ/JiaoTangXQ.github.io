# 跨层命名：后端、提示词和渲染如何共认同一个能力对象

面试题版本：**"Claude Code 为什么要让命名跨越后端、提示词和渲染三层？一个简单的字符串标识不够用吗？"**

## 问题从哪里来

想象一个叫 `computerUse` 的能力。后端用它调用截图和鼠标操作，提示词里用它告诉模型"你现在可以操作浏览器"，前端渲染时用它决定显示哪个图标和权限提示。

如果这三层各自维护一套说法——后端叫 `computer_use_tool`，提示词里叫"浏览器交互能力"，前端叫 `browserCapability`——系统还能跑，但每次扩展都要同步三处，每次出错也很难定位是哪层的命名漂移了。

这就是 Claude Code 里"命名不是标签"这个判断的来源。

## 源码里的证据

在 `src/Tool.ts` 里，工具定义不只是一个函数加一个名字，而是一套完整的身份描述：

```typescript
export type Tool = {
  name: string
  // ...
  isConcurrencySafe(): boolean
  interruptBehavior(): string
  contextModifier?: (context: ToolUseContext) => ToolUseContext
}
```

`name` 被 `findToolByName()` 用来在工具列表里查找，被权限检查用来决定是否放行，被提示词组装器用来生成系统 prompt，被渲染层用来显示工具执行状态。同一个字符串，横穿四条路径。

在 `src/services/mcp/` 目录里，MCP server 进来的工具会经过 `normalization.ts` 被规范化成同一种 `Tool` 形状。不管来源是内建工具、技能目录、还是外部 MCP server，最后都说同一种语言。

## 为什么不能让各层各说各话

**从工程角度看**：分层调试时，如果后端报了一个工具名，你必须能在提示词日志里找到对应的上下文，在前端日志里找到对应的渲染记录。跨层追踪的前提是名字一致。

**从系统演进角度看**：Claude Code 的能力在持续扩展——computerUse、MCP 工具、skill 工具都是后来加入的。每次新增能力，如果命名不统一，就等于给系统埋了一个新的分裂点。统一命名让扩展变成了往同一套协议里加条目，而不是同时维护三套说法。

**从 LLM 的特殊性看**：提示词层不像普通代码，它是自然语言或半结构化文本，LLM 的理解边界和代码的边界不完全对齐。如果后端能力的名字和提示词里的描述不能被对应起来，模型在推理时会产生歧义——它不确定"你现在有浏览器能力"和实际可调用的 `computerUse` 工具是否是同一件事。

## 代价是真实的

统一命名不是免费的。在 `src/services/mcp/normalization.ts` 里能看到，MCP server 传来的工具名需要被翻译成内部统一格式，包括前缀处理、命名冲突检测、能力合并等。每多一种来源，翻译层就要多一套规则。

但这个代价是值得的：翻译代价集中在接入层，而不是散布在每次调用、每次渲染、每次权限检查里。集中的笨重比分散的混乱好处理。

## 面试角度的总结

如果面试官问这个问题，核心答案是：**命名是系统在多层间共享意图的最小公共现实**。当后端、提示词、渲染三层都能认出同一个名字，能力才是真正的整体而不是三个松散的零件凑在一起。Claude Code 的架构选择是把命名一致性当成系统约束而不是约定，用统一的接口类型和规范化流程来强制执行。
