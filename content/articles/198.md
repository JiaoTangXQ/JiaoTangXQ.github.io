# 扩张边界不等于重写骨架：外部能力接入后的秩序保持

面试题版本：**"Claude Code 的能力在持续扩张（computerUse、MCP、浏览器 bridge、remote agent），但主循环和工具系统的结构基本稳定。这是如何做到的？扩张时有什么明确的约束？"**

## 骨架的稳定性来自哪里

Claude Code 的核心骨架是：
1. 主循环（`query.ts`）：接收用户输入，发给模型，执行工具，循环
2. 工具系统（`Tool.ts` + `tools.ts`）：工具注册、查找、执行
3. 状态机（`transition`）：管理每轮的继续/停止决策
4. 权限系统（`permissions`）：控制什么操作可以执行

这四个部分在 Claude Code 从只支持文件操作，到支持 MCP、computerUse、remote agent 的整个演进过程中，接口基本没有变。

骨架稳定的原因是：**所有扩展都通过接入点进入系统，而不是直接修改骨架**。

## 接入点 vs 骨架修改

**骨架修改的样子**：
```typescript
// 反面例子：在主循环里加特殊逻辑
async function query(...) {
  // 原有逻辑
  const tools = getAllTools()
  
  // computerUse 特殊路径
  if (hasComputerUseTools(tools)) {
    return await queryWithComputerUse(...)
  }
  
  // MCP 特殊路径
  if (mcpTools.length > 0) {
    // MCP 的特殊处理...
  }
  
  // 原有路径
  return await normalQuery(...)
}
```

**接入点的样子**：
```typescript
// Claude Code 实际的方式：扩展通过接入点进入，主循环不变
async function query(...) {
  // 主循环不需要知道哪些工具来自 MCP，哪些是内建的
  const tools = getAllTools()  // 统一获取，屏蔽来源
  // 原有流程继续...
}

// computerUse 能力通过工具注册进入，不改主循环
const computerUseTools = createComputerUseTools()
registerTools(computerUseTools)
```

骨架只看到统一的接入点（`getAllTools()`），不需要知道哪些工具是新加的、哪些是旧的。

## 扩张时的约束条件

扩张边界不等于重写骨架，依赖于扩张时遵守几个约束：

**约束一：新能力必须适配工具接口**
所有新能力都要实现 `Tool` 接口——`name`、`description`、`inputSchema`、`isConcurrencySafe()`、`call()` 等。不管这个能力多复杂（比如 computerUse 需要截图、鼠标操作、键盘输入），对外都是一个工具调用接口。

**约束二：不在消费层加来源判断**
主循环不允许出现 `if (tool.source === 'computerUse') { specialPath() }`。如果 computerUse 工具需要特殊处理，这个处理要在工具内部（`call()` 方法里），或者通过工具属性（`interruptBehavior`、`contextModifier`）告知系统。

**约束三：状态变化通过已有机制传播**
工具执行后需要改变上下文状态（比如 computerUse 截图后需要把截图结果放进下一轮的消息里）。这通过 `contextModifier` 机制传播，而不是在主循环里加特殊的截图结果处理逻辑。

## 边界变宽，秩序不散

"扩张边界不等于重写骨架"的另一个维度是权限系统。每次加新能力，权限系统不是把这个新能力的权限规则手动嵌入，而是让新能力通过权限系统的标准声明机制来表达自己的权限需求。

比如，computerUse 需要截图权限。这个权限通过工具的 `toolInputPermissionMapping`（或类似机制）声明，权限检查器统一处理所有工具的权限请求，不需要知道这是 computerUse 还是其他工具。

这保证了权限系统的推理边界：用户可以通过统一的权限界面看到所有工具的权限请求，不需要针对每种新能力学习新的权限模型。

## 面试角度的总结

能力扩张而骨架稳定，依赖的核心机制是：**扩展点是明确设计的，修改骨架是不被允许的**。接入点（工具注册、权限声明、状态修改接口）是骨架专门留出的扩展槽，新能力通过这些槽进入，不改骨架的主流程。这个约束在短期内可能感觉"太麻烦了，为什么不直接改主循环"，但长期来看，它是系统保持可推理性的基础——骨架的行为是可预测的，因为它不会因为每次新能力的加入而悄悄改变。
