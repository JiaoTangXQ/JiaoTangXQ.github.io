# 第十三卷回收索引：外延执行里扩张边界不等于重写骨架

## 实现链
外延执行里另一半更难的问题，不是接进来多少能力，  
而是外部能力进来以后，系统骨架会不会被顺手改写。

这条线真正逼出来的，是第十三卷里关于边界和秩序的判断。  
边界可以继续变宽，但原来的骨架仍然要保持原样。

这页实际回收的实现链是：
- `12-08-01` 把“扩张边界不等于重写骨架”讲清，这会继续支撑 `13-04-02`，说明外部能力进来以后，边界可以变宽，但骨架不能跟着散。

真正稳的扩张，不是把边界改成第二套制度。  
而是让外部能力进来以后，秩序仍然保持原样。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
