# 为什么不同来源的能力最后都要说同一种 Command 语言

Claude Code 里能变成“命令”的东西，来源非常杂。内建命令是一类，技能目录里长出来的是一类，插件带来的又是一类，MCP 过来的能力还是一类。如果这些来源各说各话，系统很快就会出现四套显示逻辑、四套筛选规则、四套权限边界和四套缓存方式，表面上是灵活，实际上是失控。

`Command` 抽象的高明之处，就在于它先把这些来源翻译成同一种表面语言。名字、描述、参数提示、可见性、执行类型、来源标记、是否能给模型调用，这些共同身份一旦统一，排序、展示、索引、远端安全过滤、桥接安全过滤、技能枚举这些工作才能落在同一条管线上。于是系统面对的就不再是“四种来历不明的能力”，而是一批可以被同一套制度处理的命令对象。

但它又没有因此把差异完全抹平。`prompt`、`local`、`local-jsx` 仍然保留不同执行世界；来源字段、插件信息、可用环境、是否允许模型调用，也都还在。换句话说，它统一的是接口，不是现实。

这是很成熟的抽象方式。差的抽象喜欢用统一来掩盖差异，最后把真实世界压扁；好的抽象先给出共同语言，再把真正重要的不同郑重保留下来。Claude Code 的命令系统之所以能一路接住技能、插件和 MCP，靠的正是这层分寸感。

## 实现链
代码里真正统一的是 `Command` 的形状：同样有名字、类型、加载方式、可见性和执行上下文，然后由 `findCommand()`、`hasCommand()`、格式化函数和 slash 处理器统一消费。

## 普通做法
更容易想到的是内建命令一套结构、插件命令一套结构、技能再来一套结构，最后在入口层写很多 if/else 适配。

## 为什么不用
因为入口层不想知道“你来自哪里”才能决定怎么显示、怎么追踪、怎么过权限。统一成 `Command` 以后，来源差异被压到更后面去处理。

## 代价
抽象会让最简单的命令也显得规格很重；但这是用统一成本换来源扩张能力。
