# 面试题：为什么不要求远端和本地共享同一份工具注册表？

## 紧耦合方案的诱惑

一种看起来"严谨"的设计是：

> 远端只能执行本地已知的工具。如果远端请求的工具不在本地注册表里，直接拒绝。这样系统的行为是可预测的，审批 UI 总有完整信息。

这个方案的优点：
- 本地 UI 总能展示完整的工具描述
- 不需要处理"未知工具"的边缘情况
- 系统行为明确，没有歧义

但它有一个致命缺陷：**它把远端的工具能力锁死在本地已知的工具集合上**。

## 锁死的代价

Claude Code 的远端架构支持 MCP（Model Context Protocol）工具扩展。用户可以在 CCR 容器里安装额外的 MCP 服务器，配置只有远端才用得到的工具——比如公司内部的部署工具、远端环境的专用 CLI 包装。

如果要求远端-本地工具同步：

1. 用户在远端安装 `my-company/internal-deploy`
2. 本地必须也安装同一个 MCP 服务器，配置相同的连接参数
3. 如果本地 claude 版本不支持这个 MCP 服务器，远端任务就无法进行
4. 每次远端添加新工具，都需要同步更新本地配置

在实际使用中，远端环境可能有几十个工具，本地根本不需要知道所有这些工具的细节——它只需要在工具要求权限时，让用户看到足够的信息做决定。

## stub 的"最小信息集"设计

`createToolStub` 的设计哲学是：**审批 UI 需要的最小信息集是什么？**

用户在审批时需要知道：
1. 这个工具叫什么（`userFacingName()`）
2. 它要做什么（通过 `input` 参数推断）
3. 这个操作需要权限（`needsPermissions: () => true`）

这三件事，stub 都能提供：

```typescript
{
  name: toolName,
  userFacingName: () => toolName,
  renderToolUseMessage: (input) => {
    // 显示前三个参数，够用了
    return Object.entries(input).slice(0, 3)
      .map(([k, v]) => `${k}: ${v}`).join(', ')
  },
  needsPermissions: () => true,
}
```

失去的是什么：工具的详细描述、专用渲染（rich UI）、本地特定的安全检查逻辑。

这些失去的东西，在"陌生工具"的场景下，本来就不存在——你不可能为一个不认识的工具写出详细描述。stub 只是诚实地表达了"我不认识这个工具，但我能告诉你它叫什么、想做什么"。

## 与接口分离原则的关系

这个设计实际上体现了 SOLID 里的**接口分离原则（ISP）**。

本地审批 UI 依赖的是 `Tool` 接口的一个子集：

```typescript
interface MinimalToolForReview {
  name: string
  userFacingName: () => string
  renderToolUseMessage: (input: Record<string, unknown>) => string
  needsPermissions: () => boolean
}
```

stub 实现了这个接口，其他字段（`call`、`description`、工具特定的 schema）填上空实现或 stub 实现。

审批 UI 不关心工具能不能真正执行（`call` 方法），它只关心能不能展示足够的信息让用户做决定。工具的执行发生在远端，本地不需要实现 `call`。

## 版本兼容性的视角

从系统版本演进的角度看：

**紧耦合方案**：本地 v1.5 和远端 v1.5 必须使用完全相同的工具集。当远端升级到 v1.6，添加了新工具，本地 v1.5 的用户就无法使用远端 v1.6 的新功能。

**stub 方案**：本地 v1.5 可以连接远端 v1.6，新工具通过 stub 审批。本地用户看到的 UI 比较简单，但功能不被卡死。当本地也升级到 v1.6，UI 自动变得丰富。

这是**渐进式升级**的设计：系统不要求所有组件同步升级，不同版本的组件可以协同工作，只是部分功能可能降级。

## 面试指导

"为什么不要强同步"这个问题，在微服务架构面试里很常见，通常以"服务 A 依赖服务 B 的数据，如何保证两者一致"的形式出现。

优秀的回答需要区分：

- **强一致性要求**（必须同步）：涉及事务性操作，如支付、库存
- **最终一致性够用**：涉及展示和辅助信息，如工具描述、UI 细节

权限审批属于前者（必须经过审批），工具 UI 的丰富程度属于后者（有就好，没有也能用）。stub 方案让这两者的依赖关系得以独立演进。
