# seek effect 最多重试一次就放弃，重试上限是如何设计的？

重试是处理"我期望它在这里但它还没出现"的标准手段。关键是重试上限设多少合适，以及放弃时做什么。

## 重试发生的场景

`scrollToIndex(idx)` 调用后，`useVirtualScroll` 会根据 offsets 估算 idx 对应的 scrollTop，并把 range 调整为包含 idx 的窗口。理论上下次渲染后，第 idx 条消息应该被挂载。

但有时候估算不准确：

1. **offsets 有误差**：之前未测量的消息用了默认高度估算，实际高度可能更大或更小，导致 scrollTop 算偏，range 的起点不对，idx 没有被包含在范围里

2. **React 批量更新**：scrollToIndex 触发的状态更新可能在同一批次里被合并，range 更新后的渲染还没完成，seek effect 的第一次运行已经开始了

3. **极端情况**：接近 transcript 末尾的消息，其 offsets 估算误差累积最大（因为前面所有未测量消息的误差都在这里叠加）

## 重试逻辑

```typescript
const el = getItemElement(req.idx)
const h = el?.yogaNode?.getComputedHeight() ?? 0

if (!el || h === 0) {
  // 目标没挂载或高度为 0
  if (req.tries > 1) {
    // 超过一次重试，放弃
    scanRequestRef.current = null
    logForDebugging(`seek(i=${idx}): no mount after scrollToIndex, skip`)
    stepRef.current(req.wantLast ? -1 : 1)  // 自动前进到下一条
    return
  }
  // 第一次失败，重试
  scanRequestRef.current = { ...req, tries: req.tries + 1 }
  scrollToIndex(req.idx)
  bumpSeek()  // 触发下次 seek effect
  return
}
```

`tries` 初始值是 0，第一次 seek effect 时 tries = 0，`0 > 1` 为 false，重试（tries 变成 1）。第二次 seek effect 时 tries = 1，`1 > 1` 为 false，再重试（tries 变成 2）。第三次 tries = 2，`2 > 1` 为 true，放弃。

所以实际上是"最多重试两次"（tries: 0 → 1 → 2 → 放弃）。注释里说的"retry once"是指最多一次额外的重试尝试（从 tries=0 到 tries=1 算第一次 seek，tries=1 到 tries=2 算重试）。

## 放弃时的行为

放弃不是"什么都不做"，而是 `stepRef.current(wantLast ? -1 : 1)`——自动前进到下一条匹配。

这个行为很关键：如果目标消息在极端情况下确实无法挂载，最差的结果也不是"搜索卡住了"，而是"跳过这一条，继续找下一个"。用户的搜索体验从"停滞"变成"可能错过一条"，后者更可接受。

## tries 上限为什么是 1（实际重试两次）

更多次重试理论上可以提高精度，但 Claude Code 选择最低的合理上限。原因：

**两次通常足够**：第一次 scrollToIndex 用旧的 offsets，可能偏差；seek effect 触发后，VirtualMessageList 有机会更新 offsets（新挂载的消息完成测量后更新 heightCache，重建 offsets）。第二次 scrollToIndex 用更新的 offsets，通常能命中。

**避免循环放大**：每次重试都会触发 React 重渲（bumpSeek → seekGen++ → useEffect 重跑），如果目标真的无法挂载，无限重试会导致无限的 React 更新循环。上限是防止这种情况的保险。

**幂等性**：第二次 scrollToIndex 和第一次用的是相同的 idx，如果真的卡住（目标不可达），两次结果相同，第三次也会相同，多重试没有意义。

## phantomBurstRef 的独立上限

这是和 tries 相关但不同的另一个保护机制：

`phantomBurstRef.current` 计数连续"幽灵匹配"（引擎匹配到但 scanElement 找不到高亮位置的消息）的次数，超过 20 次时停止自动前进。这防止的是幽灵匹配的无限自动导航，而不是挂载失败的无限重试。

两个机制都有上限，都在防止相同的本质问题：无上界的自动操作。

## 面试指导

重试策略是分布式系统里的核心话题，在 UI 系统里同样重要。

**重试设计的三要素**：
1. 重试触发条件（什么情况下值得重试）
2. 重试上限（最多试几次）
3. 放弃时的 fallback（放弃后做什么）

Claude Code 的答案：
1. 元素挂载失败（预期应该在那里但没有）
2. 最多 2 次（tries > 1 放弃）
3. 自动前进到下一条匹配

**常见错误答案**：增加重试次数（比如 10 次）以提高成功率。这会在目标真的不可达时造成 10 次无意义的渲染循环，用户感受到 UI 卡顿。对于"估算偏差"这类问题，2 次通常足够，额外的重试是边际收益递减的。

**追问**：为什么不用 Promise.retry 这类库？答：这里的重试不是"异步操作失败后重试"，而是"同步设置 scrollTop 后等待下一次 React 渲染"。重试的节奏由 React 的渲染周期决定，不是 setTimeout 延迟决定。bumpSeek（setState）保证了重试在下次渲染后触发，这是 React 特定的时序控制，用通用的 retry 库反而更复杂。
