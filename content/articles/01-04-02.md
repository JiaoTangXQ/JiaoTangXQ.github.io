# 权限与设置为什么在定义它被允许怎样动

只讲主流程，很容易把 Claude Code 误会成一台“收到输入就一路往前跑”的机器。真正让它不像失控自动机的，不只是流程，还有一层更深的制度: 它在动之前，先知道自己被允许怎样动。

这里至少有两种力量同时在起作用。设置决定这台系统采用什么工作方式，用哪种模型、哪些能力、哪些环境前提；权限决定某种动作在当前世界里到底能不能做，是应该直接允许、必须询问，还是提前禁止。Claude Code 甚至把这些规则拆成多种来源，一层层叠进同一个权限上下文里，而不是让每个模块各自偷偷做主。

这件事非常重要。因为一个 agent 一旦能写文件、跑命令、开子任务、连接外部系统，它最危险的地方就不在“会不会想”，而在“谁来定义它什么时候能动”。Claude Code 的成熟，不在于它敢做很多事，而在于它先把允许、不允许、需要确认这些边界做成正式结构。

所以设置和权限不是附属菜单，也不是用户体验上的小弹窗。它们更像这台系统的宪法。流程告诉它接下来想做什么，制度告诉它哪些事根本不该越线。

## 实现链
这页落到的实现链都属于制度世界：权限装配、设置来源、hooks 介入、连续性设施或条件能力 gating。代码的共同特点是，它们不是在主循环失败以后补救，而是在主循环真正推进前后就提前改写资格和边界。

## 普通做法
更普通的做法是先把 happy path 主流程跑通，再把权限、外部规则和跨回合记忆当辅助功能补上。很多系统都是这样长出来的。

## 为什么不用那种做法
Claude Code 没这么长，是因为对 agent 来说，真正危险的往往不是下一步怎么走，而是这一步有没有资格走、走完后有没有留下可继续工作的状态、外部制度有没有权利插手。它把这些问题提前写进了运行时。

## 代价与回报
回报是系统更不容易越权和失忆。代价是相关逻辑分散，阅读时不能只盯着一条主流程。
