# 面试追问：toolPermissionContext 存在 AppState 里，而不是在每次调用时重新算，为什么？

## 第一层问题：为什么不每次重算？

`ToolPermissionContext` 的类型定义里有一堆字段：

```typescript
type ToolPermissionContext = DeepImmutable<{
  mode: PermissionMode
  additionalWorkingDirectories: Map<string, AdditionalWorkingDirectory>
  alwaysAllowRules: ToolPermissionRulesBySource
  alwaysDenyRules: ToolPermissionRulesBySource
  alwaysAskRules: ToolPermissionRulesBySource
  isBypassPermissionsModeAvailable: boolean
  isAutoModeAvailable?: boolean
  strippedDangerousRules?: ToolPermissionRulesBySource
  shouldAvoidPermissionPrompts?: boolean
  awaitAutomatedChecksBeforeDialog?: boolean
  prePlanMode?: PermissionMode
}>
```

如果每次工具调用前都重新从 settings 文件读取并计算这个对象，有什么问题？

最直接的代价是性能：settings 文件需要磁盘 IO，多个来源需要合并，企业 MDM 策略可能需要从系统 keychain 读取。在高频工具调用场景下，这个开销不可忽略。

但性能不是最重要的理由。最重要的理由是：**有些权限规则根本不存在于任何静态文件里**。

## 第二层问题：动态权限规则存在哪里？

当用户在权限弹窗里点击"记住这个决定"时，这条规则被写入什么地方？

答案是 AppState 里的 `toolPermissionContext.alwaysAllowRules`（或 `alwaysDenyRules`/`alwaysAskRules`）。`ToolPermissionRulesBySource` 是一个按规则来源分组的结构——有 `settings` 来源（来自 `~/.claude/settings.json`），有 `command` 来源（来自用户当前会话里手动批准的"记住"决定）。

在 `QueryEngine.submitMessage` 里有一段：

```typescript
setAppState(prev => ({
  ...prev,
  toolPermissionContext: {
    ...prev.toolPermissionContext,
    alwaysAllowRules: {
      ...prev.toolPermissionContext.alwaysAllowRules,
      command: allowedTools,
    },
  },
}))
```

这里 `allowedTools` 是从 `processUserInput` 返回的——用户的斜杠命令（比如 `/allow-tools BashTool`）可以在会话中途更新允许的工具列表，这个更新被写进 AppState 里的 `toolPermissionContext`，不会写回 settings 文件（除非用户明确选择持久化）。

如果每次工具调用都重新从 settings 文件算权限上下文，这些会话中动态产生的规则就会丢失。

## 第三层问题：alwaysAllowRules 被剥除？

`ToolPermissionContext` 里有一个字段：`strippedDangerousRules`。这个字段只在自动模式（`--auto-approve` 或 plan 模式）下有值。

它的含义是：系统在发现用户配置了一条宽泛的 `alwaysAllow` 规则（比如 `Bash(*)`）时，不是直接应用这条规则，而是把它"剥除"——暂时不让它生效，把它存到 `strippedDangerousRules` 里，同时告知用户"某些规则在当前模式下被禁用以保障安全"。

这个设计针对的是一个具体的安全漏洞：在自动执行模式下，如果存在一条"所有 Bash 命令都允许"的规则，攻击者可以通过构造一个能触发这条规则的工具调用，绕过所有权限检查。把危险规则剥除，即使在自动模式下也需要逐个确认高风险操作。

但这个剥除不是永久的——当用户退出自动模式时（`prePlanMode` 存的是进入 plan 模式之前的权限模式），被剥除的规则需要恢复。这个恢复也发生在 AppState 里，通过更新 `toolPermissionContext` 实现。

## 第四层问题：mode 的变化如何传播？

`PermissionMode` 有几个值：`default`、`auto`、`plan`、`acceptEdits` 等。当用户用 `/plan` 进入计划模式时，`mode` 从 `default` 变成 `plan`，这个变化需要立刻对所有后续工具调用生效。

如果权限上下文不在 AppState 里，而是在调用时重算，这个模式切换需要额外的全局状态来广播。AppState 里持有它，意味着模式变化就是一次 `setAppState` 调用，下次任何组件读 `getAppState().toolPermissionContext.mode` 时就会看到新值。

这也是为什么 `QueryEngine` 的 `config` 里有 `getAppState` 和 `setAppState` 两个函数，而不是直接接收一个状态对象——权限上下文需要在整个会话里被多方读写，而不仅仅是在初始化时配置好。

## 面试指导

**这道题的层次**：

表层：性能优化（避免重复磁盘 IO）

中层：动态规则（会话中手动批准的决定不在静态文件里）

深层：权限模式切换传播（mode 变化需要对所有后续调用立即生效）

**最考验理解的追问**：`ToolPermissionRulesBySource` 按来源分组有什么意义？答：可以按来源优先级合并规则（settings 文件 vs 命令行 vs 会话内手动），也可以在重置时只清除某个来源的规则（比如会话结束时清除 `command` 来源的临时规则，保留 `settings` 来源的持久规则）。

**一句话总结答法**：权限上下文在 AppState 里，是因为权限规则的来源是多层叠加的，其中运行时动态产生的那一层根本不存在于任何静态文件里，必须有个地方存它。
