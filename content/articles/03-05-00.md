# 面试题：三种执行模型（local、local-jsx、prompt）为什么不能统一成一种？

## 假设统一的后果

**如果所有命令都走 `prompt` 模型（都发给模型）**：

`/clear` 要清空屏幕，变成 `prompt` 就得让模型决定「是否应该清空，然后怎么清空」。这不合理——清空屏幕是一个精确的本地操作，不存在需要推理的歧义。更严重的是，`/compact` 要重写消息历史，这个操作的正确性必须由程序保证，不能靠模型「猜测」应该保留哪些消息。

**如果所有命令都走 `local` 模型（本地程序直接执行）**：

`/review`（代码 review skill）需要把代码发给模型分析，`/commit` 需要模型生成 commit message。这类命令的核心价值在于调用模型推理，不能退化成本地程序。

**如果所有命令都走 `local-jsx` 模型（都弹界面）**：

`/compact` 压缩上下文不需要 UI 交互，`/clear` 清屏不需要确认面板。强迫每个命令都弹界面，会产生大量不必要的模态交互，破坏终端工具的流畅感。

## 三种模型各自解决什么问题

**`local`**：

解决「程序可以精确完成，不需要推理，不需要 UI」的问题。

典型命令：`/clear`（清屏）、`/compact`（压缩历史，结果确定性强）、`/cost`（读取本地统计数据）。

特征：执行同步，结果立即可知，返回 `text | compact | skip` 三种类型之一。

**`local-jsx`**：

解决「需要短暂的交互式 UI，但不应该重新发起模型推理」的问题。

典型命令：`/config`（配置选择面板）、`/theme`（主题选择器）、`/session`（会话历史列表）。

特征：返回一个 JSX 组件，挂到当前终端 UI，等用户操作完成后 `onDone` 回调触发，然后继续或结束。在整个交互过程中，输入框被隐藏（`shouldHidePromptInput: true`）。

**`prompt`**：

解决「需要模型参与，命令的价值在于调用推理能力」的问题。

典型命令：`/review`（代码审查）、`/commit`（生成 commit message）、技能目录里大多数 skill。

特征：把技能内容注入当前会话，设置 `shouldQuery: true`，让主循环继续推进，触发真实的 API 请求。

## 为什么「命令系统」能同时支持这三种

从用户视角，`/clear`、`/config`、`/review` 看起来都是「输入斜杠命令」。

从系统视角，`getMessagesForSlashCommand` 里是一个 switch：

```typescript
switch (command.type) {
  case 'local-jsx': return new Promise<SlashCommandResult>(resolve => ...)
  case 'local':     return { messages: [...], shouldQuery: false, command }
  case 'prompt':    return await getMessagesForPromptSlashCommand(...)
}
```

用户看到的统一外观，在代码里是三条完全不同的路径。外观统一是为了降低用户学习成本；路径分离是为了让每种命令都走最合适的执行机制。
