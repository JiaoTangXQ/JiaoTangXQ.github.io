# 执行模型导读：统一外观不等于统一内部机制

这一章真正重要的，不是系统里恰好有三种命令模型，而是 Claude Code 为什么拒绝把不同执行世界硬揉成一种机制。统一入口不等于统一内部。

所以这一章的四片叶子，要连起来读成“统一外观下的三种执行世界”。先看 [03-05-01-prompt命令更像会话工作方式.md](/Users/xxx/tmp/cc/docs/claude-code-book/03-用户一句话怎样进入系统/03-05-三种命令执行模型并存/03-05-01-prompt命令更像会话工作方式.md)，再顺着 local、local-jsx 和最后这页“统一外观不等于统一内部机制”往下读，这一章才会从“命令类型列表”变成“为什么执行世界必须并存”的入口。

## 实现链
同样是 `/命令`，系统内部会分成 `local`、`local-jsx`、`prompt`、`context:fork` 几种机制：有的直接在本地执行，有的先挂 UI，有的生成新 prompt 继续进主循环，有的交给子 agent。

## 普通做法
更常见的命令系统会强迫所有命令都走同一执行模型，比如全都本地执行，或者全都重新喂给模型。

## 为什么不用
因为这里的命令要解决的问题差异太大。用一种机制硬包所有命令，要么浪费模型，要么做不出交互式 UI，要么把长任务堵在主线程。

## 代价
从用户眼里命令长得一样，代码里却有多条执行链，理解起来会更费脑子；但这正是统一外观下保住工程现实的代价。
