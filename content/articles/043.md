# BashTool 的安全解析层：为什么要引入 tree-sitter AST？

bash 命令安全分析是 Claude Code 里工程复杂度最高的模块之一。早期实现用正则 + shell-quote 字符串解析，但随着攻击向量增多，这条路越走越窄。`utils/bash/ast.ts` 引入了 tree-sitter 作为底层解析器，重新构建了这套分析体系。

## 核心设计原则：fail-closed

```typescript
/**
 * The key design property is FAIL-CLOSED: we never interpret structure we
 * don't understand. If tree-sitter produces a node we haven't explicitly
 * allowlisted, we refuse to extract argv and the caller must ask the user.
 */
```

这个注释解释了为什么引入 AST 而不是继续扩展正则。正则的问题是"遇到不认识的模式，要么通过要么报错"——但安全分析需要第三条路：**遇到不认识的结构，拒绝自动通过，交给用户决定**。

tree-sitter 可以精确识别节点类型。代码维护了一个**结构节点白名单**：

```typescript
const STRUCTURAL_TYPES = new Set([
  'program',
  'list',
  'pipeline',
  'redirected_statement',
])
```

只有在这些白名单节点里递归，才能产出 `SimpleCommand`（可信的命令解析结果）。遇到任何白名单外的节点，返回 `too-complex`。

## ParseForSecurityResult：三种可能的分析结果

```typescript
export type ParseForSecurityResult =
  | { kind: 'simple'; commands: SimpleCommand[] }
  | { kind: 'too-complex'; reason: string; nodeType?: string }
  | { kind: 'parse-unavailable' }
```

- `simple`：成功解析出所有简单命令的 argv，可以和权限规则匹配
- `too-complex`：遇到了无法安全解析的结构（subshell、coproc、select、function、trap 等），需要走权限请求流程
- `parse-unavailable`：tree-sitter 未初始化或崩溃，降级处理

这个三值返回比布尔值"安全/不安全"更精确——`too-complex` 不是"危险"，而是"分析能力不足，无法判断"。

## SimpleCommand：解析结果的数据结构

```typescript
export type SimpleCommand = {
  argv: string[]          // 完整参数列表，引号已解析
  envVars: { name: string; value: string }[]  // 前置环境变量赋值
  redirects: Redirect[]   // 输出/输入重定向
  text: string            // 原始文本（用于 UI 展示）
}
```

一旦得到 `SimpleCommand`，就可以用 `argv[0]` 匹配权限规则：
- `Bash(git:*)` 规则只有 argv[0] 是 `git` 时才匹配
- `Bash(npm:*)` 规则只有 argv[0] 是 `npm` 时才匹配

重定向也被单独提取——`Redirect` 有 `op` 和 `target` 字段，可以单独验证"这个命令是否向某个危险路径写文件"。

## 占位符技术处理命令替换

命令替换（`$(...)` 或反引号）是个特殊情况：外层命令的 argv 里包含一个运行时才知道的值。

```typescript
const CMDSUB_PLACEHOLDER = '__CMDSUB_OUTPUT__'
```

当遇到 `$(...)` 时，用占位符替换，同时递归分析内层命令。这样：
1. 外层命令的 argv 保持"可处理"状态（不是 `too-complex`）
2. 内层命令独立走一遍权限检查

比如 `git checkout $(cat branch.txt)`：
- 外层 argv = `['git', 'checkout', '__CMDSUB_OUTPUT__']`
- 内层 argv = `['cat', 'branch.txt']`
- 两组 argv 都能和权限规则匹配

## 引号上下文分析

```typescript
export type QuoteContext = {
  withDoubleQuotes: string    // 单引号内容移除（双引号保留）
  fullyUnquoted: string       // 所有引号内容移除
  unquotedKeepQuoteChars: string  // 内容移除但保留引号字符
}
```

这三个视图解决了一个具体问题：正则检查 `$()` 时，在单引号里的 `'$(evil)'` 不是命令替换，但字符串 `"$(evil)"` 是。三个不同的去引号视图让检查逻辑可以明确说"在哪个上下文里有这个模式"。


sed 是一个功能极强的工具，`w` 可以写文件、`e` 可以执行命令。白名单只允许两种安全用法（行打印和简单替换），黑名单作为"纵深防御"检查即使通过白名单的命令里有没有危险模式（Unicode 同形字、反斜杠花招、negation 操作符等）。

## 破坏性命令的提示信息

除了权限检查，`destructiveCommandWarning.ts` 还实现了一层纯信息性的警告——不影响权限决策，只是在 UI 上展示提示：

```typescript
const DESTRUCTIVE_PATTERNS: DestructivePattern[] = [
  { pattern: /\bgit\s+reset\s+--hard\b/, warning: 'Note: may discard uncommitted changes' },
  { pattern: /\bgit\s+push\b[^;&|\n]*[ \t](--force|--force-with-lease|-f)\b/, 
    warning: 'Note: may overwrite remote history' },
  { pattern: /(^|[;&|\n]\s*)rm\s+-[a-zA-Z]*[rR][a-zA-Z]*f/, 
    warning: 'Note: may recursively force-remove files' },
  // ... 数据库 DROP/TRUNCATE/DELETE，kubectl delete，terraform destroy ...
]
```

这些正则是专门设计的，比如 `git clean` 的正则带了 `(?!.*--dry-run)` 排除干跑模式，区分了"真正危险"和"只是查看"的情况。

## 面试角度

bash 安全分析的工程复杂度揭示了几个普遍原则：

1. **白名单比黑名单更安全**：只允许已知安全结构，而不是禁止已知危险结构。新的攻击向量出现时，黑名单需要持续更新，白名单自动拒绝未知情况
2. **fail-closed 而不是 fail-open**：遇到无法解析的命令结构，选择"要求用户确认"而不是"默认允许"。这让安全边界随工程能力边界一起收缩
3. **多层防御**：AST 解析 + 白名单 + 黑名单 + 破坏性提示是四个独立的层，各自有各自的关注点，任何一层失效不会导致其他层也失效
4. **语义感知的命令分析**：`grep exit 1` 不是错误，`diff exit 1` 也不是错误。命令语义表（COMMAND_SEMANTICS）让工具对不同命令的退出码有正确预期
