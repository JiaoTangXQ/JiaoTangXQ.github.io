# isBackgrounded 是状态位还是运行模式？前后台统一模型的工程逻辑

`LocalAgentTaskState` 里有一个字段：

```typescript
export type LocalAgentTaskState = TaskStateBase & {
  type: 'local_agent'
  // ...
  isBackgrounded: boolean   // false = 前台运行，true = 后台运行
  // ...
}
```

前台 Agent（`registerAgentForeground`）初始化时 `isBackgrounded: false`，后台 Agent（`registerAsyncAgent`）初始化时 `isBackgrounded: true`。

这个字段决定了 UI 是否显示 Agent 的实时输出、是否有 `backgroundSignal` 机制、是否需要显示"转后台"按钮。

但关键是：**两种 Agent 用的是同一个类型**。不是 `ForegroundAgentState` 和 `BackgroundAgentState` 两种类型，而是 `LocalAgentTaskState` 加一个 `isBackgrounded` 布尔位。

## 如果用两种类型会怎样

假设把前台和后台做成两个类型：

```typescript
// 假设的两类型设计
type ForegroundAgentState = TaskStateBase & {
  type: 'local_agent_foreground'
  isBackgrounded: false
  backgroundSignalResolver: () => void
  // ...
}

type BackgroundAgentState = TaskStateBase & {
  type: 'local_agent_background'
  isBackgrounded: true
  // 没有 backgroundSignalResolver
  // ...
}
```

当用户把一个前台 Agent 转到后台时，你需要：

1. 把 `ForegroundAgentState` 对象销毁
2. 创建一个新的 `BackgroundAgentState` 对象
3. 把两个对象之间的状态迁移过去（`progress`、`messages`、`pendingMessages`、`retain` 等）
4. 更新 `AppState` 里的字典键（如果类型作为键的话）
5. 所有持有引用的地方（面板、通知系统、kill 逻辑）都需要感知到这次迁移

这是一次**对象迁移**，而不是**状态更新**。对象迁移的复杂度远高于状态更新，因为需要复制字段、保证不丢失任何中间状态。

## 统一模型的实际代价

统一模型的代价是：`LocalAgentTaskState` 要同时携带前台和后台两种模式的所有字段，其中部分字段在某种模式下没有意义：

- 后台 Agent 没有 `backgroundSignal`（resolver 存在外部 Map 里，不在对象上，但这本身也是为了统一）
- 前台 Agent 的 `isBackgrounded` 在转后台前始终是 `false`，这个字段在"纯前台"场景下显得多余

这叫**形式上的冗余**，但在实践中，这种冗余被认为是可接受的，因为它换来了：

1. **转场无需迁移**：把 `isBackgrounded` 从 `false` 改成 `true`，就完成了前台到后台的切换
2. **kill 逻辑统一**：`killAsyncAgent` 对前台和后台 Agent 一视同仁，不需要分支
3. **面板显示统一**：`isPanelAgentTask(task)` 只需要检查 `task.type === 'local_agent'`，不需要区分前后台
4. **`retain` 和 `evictAfter` 统一**：驱逐逻辑不需要知道任务是前台还是后台启动的

## backgroundAgentTask 里的转场

```typescript
export function backgroundAgentTask(
  taskId: string,
  getAppState: () => AppState,
  setAppState: SetAppState,
): boolean {
  const state = getAppState()
  const task = state.tasks[taskId]
  if (!isLocalAgentTask(task) || task.isBackgrounded) {
    return false  // 已经是后台或者不是 local_agent，忽略
  }

  // 更新状态：isBackgrounded = true
  setAppState(prev => {
    const prevTask = prev.tasks[taskId]
    if (!isLocalAgentTask(prevTask)) return prev
    return {
      ...prev,
      tasks: {
        ...prev.tasks,
        [taskId]: { ...prevTask, isBackgrounded: true },
      },
    }
  })

  // Resolve backgroundSignal，通知执行循环可以交棒了
  const resolver = backgroundSignalResolvers.get(taskId)
  if (resolver) {
    resolver()
    backgroundSignalResolvers.delete(taskId)
  }
  return true
}
```

整个转场过程：改一个字段（`isBackgrounded: true`），resolve 一个 Promise，从 Map 里删一个 resolver。不需要创建新对象，不需要迁移状态。

## isLocalAgentTask 的类型守卫作用

```typescript
export function isLocalAgentTask(task: unknown): task is LocalAgentTaskState {
  return (
    typeof task === 'object' &&
    task !== null &&
    'type' in task &&
    task.type === 'local_agent'
  )
}
```

这个类型守卫不检查 `isBackgrounded`，不检查 `backgroundSignal`。对系统来说，前台和后台的 `local_agent` 任务是同一个类别，`isLocalAgentTask` 对两者都返回 `true`。

如果用两种类型，就需要 `isForegroundAgentTask` 和 `isBackgroundAgentTask` 两个守卫，所有使用者都需要决定"我要检测哪种"，很容易造成漏处理。

## 面试要点

**"前台/后台"是运行模式，不是本质属性**。这个区分指导了设计选择：

- **运行模式**：同一个对象在不同时间点可以采用不同的执行方式，用布尔字段表示
- **本质属性**：对象从出生到销毁都不会改变的特征，用独立类型表示

`local_agent` 和 `local_bash` 是本质属性不同（执行方式完全不同，字段不同，kill 逻辑不同）。`isBackgrounded` 是运行模式不同（同一种 agent，只是当前是否在前台显示）。

把运行模式做成类型差异，会导致本来应该是"翻个状态位"的操作变成"迁移对象"，这是过度建模的表现。
