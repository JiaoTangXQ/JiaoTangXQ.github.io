# ToolPermissionContext 为什么不是运行时检查，而是运行时骨架？

如果你见过只在"删文件前问一下用户"这种地方加权限检查的系统，你就知道那种系统的权限层是事后补丁。Claude Code 的权限设计和这个完全不同。

## 从 query.ts 的接缝看起

`query.ts` 是主查询循环所在，它会在调用 LLM 前先获取 `ToolPermissionContext`。这个上下文对象不是简单的布尔开关集合，而是一个多层制度对象，包含了当前 session 的权限规则、hooks、settings，以及从哪里继承这些规则的完整溯源链。

为什么在进主循环之前就要拿这个？因为**能力的出现本身就受权限控制**。不是"工具先跑，跑完再检查"，而是"这个工具现在能不能出现在这一轮的可用工具列表里"，这个判断发生在 LLM 看到工具列表之前。

## 制度对象的四个层

Claude Code 的权限制度由四类对象协同工作：

- **permissions**：什么工具在什么条件下允许执行
- **rules**：允许执行前需要经过哪些规则验证
- **hooks**：工具执行前后触发哪些副作用
- **settings**：用户级和项目级的配置覆盖

这四类对象不是四个独立的 if/else 分支，而是运行时持续存在的语境层。当你在 `auto` 模式下运行时，危险工具默认被剥离出可用列表；当你退出 `auto` 模式时，这个剥离需要被撤销，旧的权限层需要被"归还"。这种"归还"的设计说明了一件事：**权限状态有生命周期，不是静态配置文件。**

## 为什么不用零散 if/else

最常见的权限实现是这样的：在 `runBashCommand` 前加一个 `checkPermission('bash')` 调用，返回 true/false。

Claude Code 没这么做的根本原因是：**它的工具调用是动态的，工具列表本身就是运行时构建的**。每一轮 query 传给 LLM 的工具列表，是根据当前的 `ToolPermissionContext` 筛选过的。如果权限层只是分散的检查点，就无法在 LLM 看到工具之前就把危险能力从视野里清除掉。

更重要的是，`hooks` 和 `rules` 允许权限层在运行时被修改。用户可以在一次 session 中动态调整哪些路径是可写的，哪些命令需要提前确认。这种动态性需要制度对象在内存中持续存活，而不是每次从配置文件里重读。

## 自动模式的危险剥离

当系统进入 `auto` 模式（`dangerously allow all`），它不是把所有检查清空，而是执行一套明确的危险工具剥离策略。具体来说，`tools.ts` 的筛选逻辑会根据当前 `ToolPermissionContext` 决定哪些工具在 auto 模式下不该出现——即使用户声明了 allow all。

这是一个非常重要的设计判断：**"用户授权放行"和"系统认为安全"是两个独立的维度**。auto 模式下的危险剥离，是系统在用户信任之上还保留的最后一层制度防护。

当你从 auto 模式切回普通模式，之前被剥离的工具需要重新出现，已经被临时许可的规则需要回退。这个回退不是自动发生的——它需要权限层知道"我什么时候进入了 auto 模式，我在 auto 模式里做了哪些临时变更"。

## 权限是运行时骨架的证明

`ToolPermissionContext` 被初始化在整个 query 循环的最外层，不是每次工具调用时临时构建，而是贯穿整个会话持续存在。这意味着：

1. 你可以在会话中途安全地修改权限规则，修改会立刻在下一轮 query 里生效
2. hooks 可以在工具执行后更新权限状态（比如：某个文件被写入后，自动限制对相关路径的后续写操作）
3. 跨会话的权限记忆（用户之前批准过的操作类型）可以安全地合并进来

## 面试答题要点

被问到"你的工具权限系统是怎么设计的"时，区分两种答案的高下：

**弱答**："我们在每个敏感操作前做权限检查，有一个全局的 permission map。"

**强答**："权限层是运行时骨架的一部分，不是检查点。能力是否出现在工具列表里，本身就受权限控制。权限对象在整个会话生命周期内持续存活，支持在运行中被动态修改，修改立刻对下一轮 query 生效。我们特别区分了'用户授权放行'和'系统层安全保底'，两者独立维护。"

追问"怎么处理权限回退"时，关键点是：**权限修改有溯源**，系统知道一个规则是从哪个层（用户配置/项目配置/auto模式临时许可）引入的，回退时只需要移除对应层的条目，不会影响其他层的状态。
