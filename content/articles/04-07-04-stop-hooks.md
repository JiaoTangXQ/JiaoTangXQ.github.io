# stop hooks 阻塞为什么也会变成继续下一轮的理由

很多系统把 hooks 当成会话结束后的尾声动作，跑完最好，跑不完就算了。Claude Code 对 stop hooks 的态度更严肃。它承认有些外部制度不是尾声，而是下一轮还能不能继续的重要前提。

当 stop hooks 产出阻塞错误时，系统不会简单把它们当日志丢掉，而是把这些错误写回消息流，再带着 `stop_hook_blocking` 这样的继续理由进入下一轮。换句话说，它是在对模型说：刚才那轮并没有真正收尾，因为制度层还有问题没有过关，你得在这个新事实之上继续工作。更强的一种情况是 hook 直接要求阻止 continuation，那系统就真的停下。

这件事很关键。它说明 Claude Code 的“失败与恢复世界”并不只来自模型和上下文，也来自制度层。外部验证、团队约束、清理脚本，只要被正式接进来，就有资格改写主循环的去向。

所以 stop hooks 不是结束以后的小挂件，而是主循环的关口之一。

代码里，stop hooks 只在拿到有效 assistant 响应后才跑；如果最后一条消息其实是 API error，系统会直接跳过 hooks，避免“错误 → hook 阻塞 → 重试 → 再错误”的死循环。真有 blocking errors 时，它会把这些错误消息追加到 `messages`，把 `transition.reason` 设成 `stop_hook_blocking`，再启动下一轮。更常见的做法是 hook 一旦阻塞就整轮失败退出，或者相反，对任何情况都强行跑 hook。

Claude Code 的做法更像“制度参与续写”，而不是“制度负责终结”。这更贴近真实工作流，因为很多阻塞其实是在给下一轮补前提，而不是宣布任务结束。代价是 hook 不再是独立前后置插件，而要和主循环状态机互相理解，调试起来也更绕。

## 实现链
stop hooks 只在有效 assistant 响应后跑，阻塞时把错误追加进消息链并设置 `stop_hook_blocking`。

## 普通做法
hook 阻塞就整轮失败，或相反任何情况都强跑 hook。

## 为什么不用
因为有些 hook 阻塞是在给下一轮补前提，不是在宣布结束。

## 代价
hook 和主循环状态机耦合更深。
