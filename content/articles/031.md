# Compact 是怎么压缩对话历史的？forked Agent 方案和 microcompact 的区别

Context window 快满了需要压缩——但压缩本身也要消耗 context（把几十条消息压成摘要，需要让模型先读完这几十条）。这是一个自我指涉的问题：压缩者也需要空间工作。

`services/compact/compact.ts` 的解法是用一个 **forked Agent** 来做压缩。

## forked Agent 方案

```typescript
// compact.ts 里的核心流程
const result = await runForkedAgent({
  messages: messagesToCompact,
  systemPrompt: COMPACT_SYSTEM_PROMPT,
  // forked agent 有独立的 context，不占用主线程的 context window
  maxOutputTokens: COMPACT_MAX_OUTPUT_TOKENS,
})
```

`runForkedAgent` 创建一个完全独立的 API 调用，带着需要压缩的消息序列，让模型生成摘要。这个调用不占用主线程的 context window——它是一次全新的 API 请求，有自己的 token 计算。

结果是：即使主线程的 context 已经 99% 满了，compact 任务仍然有完整的 context 空间生成高质量摘要。

## 压缩前的状态保存

压缩前要做几件重要的事：

```typescript
// 执行 pre-compact hooks（用户可以在这里自定义压缩前逻辑）
await executePreCompactHooks(context)

// 保存文件状态缓存的快照（压缩后要恢复）
const fileStateCacheSnapshot = cacheToObject(context.readFileState)

// 记录 compact 边界时刻的 token 统计
const tokenStats = await analyzeContext(messages)
```

`analyzeContext` 生成当前 context 的 token 统计报告，用于 Statsig 指标上报，也用于在 UI 里展示"本次压缩节省了多少 token"。

## 压缩后的恢复

压缩之后，新的消息历史是：

```
[compact 摘要消息]
[CompactBoundaryMessage]  ← 标记压缩边界
[compact 之后的新消息]
```

`CompactBoundaryMessage` 是一个关键标记：`getMessagesAfterCompactBoundary` 函数只返回这个边界之后的消息，发给 API 时不会把压缩掉的历史重新发送过去。

压缩后还要执行 `post-compact hooks` 和重新注入一些重要的 attachment（比如 MCP 指令、deferred tools 的增量更新）：

```typescript
await executePostCompactHooks(context)
// 重新注入需要在每次 compact 后刷新的 attachment
const mcpDelta = getMcpInstructionsDeltaAttachment(...)
const deferredDelta = getDeferredToolsDeltaAttachment(...)
```

## microcompact：轻量级压缩变体

`services/compact/microCompact.ts` 和 `apiMicrocompact.ts` 是另一种压缩策略，处理更轻量的情况：

```typescript
// microcompact 针对的场景：
// - 不需要完整压缩，只需要缩短工具结果
// - 单个工具调用输出过大
// - 快速减少 context，不需要摘要质量保证
```

Microcompact 不启动 forked agent，而是直接截断或替换过大的工具结果。适用于"某个 bash 命令输出了 10 万行日志"这样的情况——不需要模型理解这些日志，直接截断就好。

## reactive compact

`services/compact/reactiveCompact.ts` 是一个更智能的变体：

```typescript
const reactiveCompact = feature('REACTIVE_COMPACT')
  ? require('./reactiveCompact.js')
  : null
```

Reactive compact 监控 context 使用率，在合适的时机主动触发压缩，而不是等到快溢出再处理。这让 Claude 在长会话中有更稳定的可用 context，避免"context 突然溢出然后紧急压缩"的颠簸体验。

## 面试角度

Compact 系统是 LLM 应用工程的经典问题。几个值得讨论的点：

1. **forked agent 避免了"用 context 压缩 context"的悖论**：独立 API 调用有完整 context 空间
2. **压缩前后的状态一致性**：fileStateCache 快照保证压缩不丢失文件读取记录
3. **CompactBoundaryMessage 是软断点**：不是物理删除消息，而是逻辑标记"API 请求从这里开始"
4. **多层次压缩策略**：微压缩（截断单个输出）→ 标准压缩（摘要历史）→ reactive 压缩（主动预防溢出）
