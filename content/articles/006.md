# swarm 的权限请求是怎么从 worker 流转到 leader 的？

## 场景：worker 遇到需要权限确认的操作

一个 swarm 场景：leader 分派了 3 个 worker 并行工作，其中一个 worker 要写文件。正常情况下，写文件会弹一个"是否允许"的确认框。但 worker 没有独立的 UI，它不能直接弹框——弹了用户也不知道是哪个 agent 请求的，搞不好 3 个 worker 同时各弹一个框，用户完全不知道该点哪个。

`swarmWorkerHandler.ts` 里的 `handleSwarmWorkerPermission()` 解决了这个问题。

## 执行路径

```
worker 遇到权限请求
↓
tryClassifier() → 尝试自动批准（bash 命令的 classifier）
↓ 没有自动批准
createPermissionRequest() → 创建请求对象（含 requestId）
↓
registerPermissionCallback() → 先注册回调
↓
sendPermissionRequestViaMailbox() → 再发送到 leader
↓
setAppState({ pendingWorkerRequest }) → 在 UI 上显示"等待 leader"
↓
await promise → 等待 leader 响应
```

有一行注释很关键：
```typescript
// Register callback BEFORE sending the request to avoid race condition
// where leader responds before callback is registered
```

这是经典的"注册-发送"顺序问题。如果先发送请求，leader 的速度极快，回调还没注册完就来了响应，那个响应就会丢失。所以必须先把监听器挂上，再把消息发出去。

## 四种结果路径

`registerPermissionCallback()` 注册了两个回调：

**`onAllow(allowedInput, permissionUpdates, feedback, contentBlocks)`**
Leader 批准了，可能还附带了修改后的输入（leader 可能编辑了 worker 提出的命令）、权限更新（比如"这条命令以后不用再问"）、反馈文字、额外内容块。

**`onReject(feedback, contentBlocks)`**
Leader 拒绝了，附带拒绝理由。

还有第三种情况——abort signal 触发：
```typescript
ctx.toolUseContext.abortController.signal.addEventListener(
  'abort',
  () => {
    if (!claim()) return
    clearPendingRequest()
    ctx.logCancelled()
    resolveOnce(ctx.cancelAndAbort(undefined, true))
  },
  { once: true }
)
```

如果 worker 的会话被取消（比如用户按了 Ctrl+C），等待中的权限请求不应该永远挂着。abort signal 监听器确保了这个 promise 不会泄漏。

**`claim()`** 是一个原子性的"有没有人先应答"检查。防止 abort 和 leader 响应同时到来时，两个处理路径都走——只有第一个调用 `claim()` 成功的路径会继续，另一个直接 return。

## 邮箱文件结构

权限请求通过 `sendPermissionRequestViaMailbox()` 写入文件系统。邮箱在 `.claude/teams/{team_name}/inboxes/` 下，每个 agent 有自己的 inbox 文件。

```typescript
const LOCK_OPTIONS = {
  retries: {
    retries: 10,
    minTimeout: 5,
    maxTimeout: 100,
  },
}
```

邮箱写入使用了文件锁，重试 10 次，最长等待 100ms。多个 worker 同时写 leader 的 inbox，这个锁防止了并发写入导致的 JSON 损坏。

## 为什么用文件而不是内存？

同一进程内完全可以用内存 channel。但 `teammateMailbox.ts` 开头的注释说明了意图：

> 结构是 `.claude/teams/{team_name}/inboxes/{agent_name}.json`

文件路径里有 team_name 和 agent_name。这意味着 worker 可以是不同进程，甚至是不同机器上的独立会话，只要挂载了同一个文件系统就能通信。内存通信无法支持这种扩展性。

## 面试指导

说 swarm 权限时，重点是两个：1）先注册再发送的竞态处理；2）`claim()` 防止双重处理。这两点都是并发编程中的经典问题，能说清楚代表你真的读了代码。补充说"用文件而不是内存的原因是为跨进程/跨机器做准备"，这是架构层面的判断，加分点。
