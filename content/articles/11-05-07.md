# 面试题：为什么全屏模式下底部区域没有内容时也要保留一行高度？

这道题要求你理解终端渲染的底层机制——为什么"多一行空白"有时候比"精确省掉一行"更有价值。

## 问题现象

在 `PromptInputFooterLeftSide.tsx` 里有这样一段代码：

```typescript
// In fullscreen, always occupy at least one line so the footer height
// doesn't jump from 0→1 when content first appears, displacing the
// scrollable content above.
if (isFullscreenEnvEnabled() && !parts.length && !tasksPart && !modePart) {
  return <Text> </Text>;
}
```

三个条件都为空（没有 parts、没有任务提示、没有 mode 提示），但仍然返回一个含空格的 `<Text>`，而不是 `null`。

## 终端渲染的高度语义

在终端界面里，行高不像浏览器那样是 CSS 属性——它是真实的字符行。一行就是一行，没有"过渡动画"，没有"平滑高度变化"。

**高度变化的代价**

当底部区域从 0 行变成 1 行时，Yoga 布局引擎会重新计算整个屏幕的行分配：

```
[消息区：25行] → [消息区：24行]
[底部：0行]       [底部：1行]
```

这不只是底部"多了一行"，而是消息区被挤掉了一行。如果用户正在阅读第 24 行的内容，这行内容会瞬间消失，换成第 25 行的内容上来。这就是"抖动"。

**为什么全屏比普通模式更敏感**

在普通模式下，消息区高度本来就会随内容增减而变化，用户已经习惯了这种动态布局。但在全屏模式下，整个屏幕都是消息区，用户建立了一种"这块区域稳定"的心理预期。这时底部突然出现一行，对阅读体验的打击更大。

## 两个场景对比

**场景 A：没有空格占位符**

1. 用户打开全屏，底部没有内容，消息区占满整个屏幕
2. 状态栏 hook 命令执行完成，返回文本
3. 底部从 0 行变成 1 行，消息区被压缩 1 行
4. 用户的阅读位置跳了一行

**场景 B：有空格占位符（当前实现）**

1. 用户打开全屏，底部已经占了 1 行（空格），消息区比"场景 A 之前"少 1 行
2. 状态栏 hook 命令执行完成，返回文本，填进那行占位
3. 底部仍然是 1 行，消息区没有变化
4. 用户的阅读位置保持稳定

代价是全屏时有一行空白，好处是阅读位置不抖动。

## 这个技巧的通用性

这种"提前占位防抖动"的模式在 UI 开发里很常见，有几个名字：

- **Layout stabilization**：提前为将来可能出现的内容预留空间
- **Skeleton loading**：在内容加载前先显示结构骨架
- **Min-height / min-rows**：组件声明自己至少需要多少空间

在浏览器端，这通常用 `min-height` 或 skeleton 组件实现。在终端里，最轻量的做法就是放一个空格字符。

## StatusLine 里的类似处理

```typescript
// StatusLine.tsx
if (statusLineText === undefined) {
  // Don't render an empty element — the footer's min-height handles the
  // placeholder row. If fullscreen, keep a blank row so the footer height
  // doesn't jump from 0→1 when the command returns.
  return isFullscreenEnvEnabled() ? <Text> </Text> : null;
}
```

状态栏命令还没返回结果时，也做了同样的处理：全屏下渲染空格占位，非全屏下返回 `null`。

这里的逻辑非常清楚：`statusLineText === undefined` 意味着命令还在跑，这是个临时状态，很快就会变成有内容的状态。在这个临时窗口里用空格占位，防止从 0→1 的突变。

## 面试指导

**直接问法**："为什么没有内容时还要返回 `<Text> </Text>` 而不是 `null`？"

答：防止底部高度从 0 变 1 时把消息区压缩一行，导致阅读位置抖动。全屏模式下用户对布局稳定性的期望更高，一行的突然出现会直接中断阅读体验。

**追问**："为什么非全屏模式不需要这个处理？"

答：非全屏模式下，消息区高度本身就是动态的（随消息增减而变化），用户习惯了这种动态布局，不会把底部多出一行当成打断。全屏模式建立了"布局稳定"的预期，才需要这个保护。

**设计追问**："这种提前占位的模式有什么代价？"

答：全屏时总是有一行底部空白，即使真的什么都没有。用户可能会注意到这个空行，觉得奇怪。但相比阅读位置抖动的体验损失，这是值得的 trade-off。这类 trade-off 在 UI 开发里很常见，需要能清楚说出为什么选择了"稳定性"而不是"精确性"。

**代码细节追问**："这里用空格字符而不是空字符串，有什么区别？"

答：空字符串 `""` 在 Ink 里可能被视为空渲染，不会产生实际的行高。空格 `" "` 是一个实际的字符，确保 Yoga 布局引擎为这个 Text 组件分配至少一行高度。这是利用了"非空内容的 Text 至少占一行"的 Yoga 行为。

---

*核心考察点：终端渲染的行高语义、布局稳定性 vs 精确性的 trade-off、skeleton 占位模式的通用性。*
