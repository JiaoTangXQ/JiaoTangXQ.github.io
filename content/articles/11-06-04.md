# 团队存在时底部信息主动分层，说明组织关系会改写布局优先级

teammate pills 在别处把“人”压成短标签，这一页则把团队信息拆进总览层和行内层。只要团队上下文出现，任务区就不能再只认单一列表。

## 实现链

`TaskListV2.tsx` 先从 `teamContext.teammates` 建出 `teammateColors`，再从 `appState.tasks` 里找 `in_process_teammate` 的运行态，拼出 `activeTeammates` 和 `teammateActivity`。真正渲染时，standalone 模式会先给一行 `tasks / done / in progress / open` 的总览，再在下面挂具体任务；普通模式则直接只挂任务列表。底层逻辑说明，这里已经默认把“团队状态”和“任务状态”分到不同层了。

## 普通做法

更普通的界面会把所有信息揉成一行，或者只保留任务行本身，让团队信息另起一个弹窗去看。

## 为什么不用

Claude Code 不是只在展示任务，而是在展示一个正在协作的运行时。团队一旦存在，布局优先级就得跟着改，否则任务、人员和活动会互相挤压，最后谁都看不清。

## 代价

代价是界面必须让人区分“总览层”和“行内元数据层”。读者需要先认识哪一层是状态汇总，再去看每个任务行上的 owner 和 activity，不然会觉得信息被拆得有点碎。
