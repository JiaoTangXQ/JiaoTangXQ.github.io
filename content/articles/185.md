# 任务进度为什么要住在 Task 对象里，而不是外挂一个状态变量

面试题版本：**"Claude Code 的 TaskStateBase 里有 startTime、outputOffset、status 等进度字段。为什么这些字段要在 Task 对象里？用一个全局 Map 或 EventEmitter 不是更灵活？"**

## 看看 Task 对象的结构

`src/Task.ts` 里的 `TaskStateBase` 不是一个简单的任务描述：

```typescript
export type TaskStateBase = {
  id: string
  type: TaskType
  status: TaskStatus          // 'pending' | 'running' | 'completed' | 'failed' | 'killed'
  description: string
  toolUseId?: string
  startTime: number
  endTime?: number
  totalPausedMs?: number
  outputFile: string          // 后台输出写到哪里
  outputOffset: number        // 已经读取到的输出偏移量
  notified: boolean           // 是否已经通知用户
}
```

这里有几个字段特别值得注意：`outputFile` + `outputOffset` 组合，`totalPausedMs`，和 `notified`。

## outputFile + outputOffset 是可中断接续的关键

`outputFile` 记录后台任务的输出写到哪个文件，`outputOffset` 记录主会话最后读到了文件的哪个位置。

为什么需要 offset？因为任务可能在运行中被中断（用户切走了、网络断了、Claude Code 崩溃了），下次恢复时不需要从头重读所有输出，从 offset 位置接着读即可。

这个设计的前提是进度必须和任务对象绑定。如果 offset 放在外部 Map 里：

```typescript
// 反面例子：外挂状态
const taskOutputOffsets = new Map<string, number>()
```

会话结束时这个 Map 消失，下次恢复时 offset 信息丢失，要么从头重放输出，要么接续点不对。而 `outputOffset` 在 TaskStateBase 里，AppState 持久化时它跟着走，恢复时它跟着回来。

## totalPausedMs 说明了什么

`totalPausedMs` 记录任务被暂停的总毫秒数。这个字段的存在说明 Claude Code 的任务有显式的暂停/恢复语义，而不只是 start 和 end。

为什么要记暂停时间？两个原因：
1. 计算真实的活跃执行时间（而不是 wall time）
2. 支持资源计费——如果任务在等用户确认时暂停，这段时间不应该计入 LLM token 消耗或 API 调用时间

这个字段只有在任务有持久状态对象时才有意义。EventEmitter 方案里每个暂停/恢复只是一个事件，没有地方自然聚合成"总暂停时长"。

## 进度在对象里，劳动才"有人认领"

原文里提到"劳动进行中也持续被认领"，这句话的工程含义是：任何时刻系统都能回答"任务 X 现在在哪个阶段？"这个问题。

Claude Code 的 AppState 维护了 `tasks: Record<string, TaskState>`，这意味着：

- UI 可以随时轮询所有运行中任务的状态
- 用户切换到任务列表时不需要等任务"汇报"，直接读当前状态
- 后台任务和前台会话的生命周期解耦，任务可以在前台会话结束后继续运行，状态不会丢失

如果用全局 EventEmitter，"UI 要显示任务当前进度"这个需求就变成：要么 UI 自己维护一个镜像状态，要么每次都触发一次查询事件等异步响应——两种方案都比直接读 AppState.tasks 复杂。

## 什么时候这种设计会变重

TaskStateBase 的字段数量意味着每个任务对象都有一定的内存占用，大量并发任务时会有压力。另外，状态字段越多，状态迁移的合法路径也越多，测试矩阵就越大。

Claude Code 的对策是 `isTerminalTaskStatus()` 这个函数：

```typescript
export function isTerminalTaskStatus(status: TaskStatus): boolean {
  return status === 'completed' || status === 'failed' || status === 'killed'
}
```

终态任务可以被清理出 AppState，避免无限积累。这是用显式的生命周期边界来控制状态对象的膨胀。

## 面试角度的总结

"进度住在任务对象里"这个设计选择的本质是：**任务是第一类公民，不是一个临时的异步 handle**。只有当任务有稳定的状态对象时，系统才能支持中断恢复、跨会话接续、实时监控和资源计费这些特性。全局 Map 或 EventEmitter 方案在任务少且简单时足够，但在 Claude Code 这种需要前后台切换、长期运行、可接续的场景下，状态对象的额外开销是必要的。
