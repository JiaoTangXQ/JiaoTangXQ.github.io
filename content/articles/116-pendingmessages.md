# 面试题：pendingMessages 为什么要先排队，不能直接注入当前轮次？

## 问题场景

设想这样的情况：主会话正在进行第 N 轮对话——用户已经发送了消息，AI 正在生成响应，工具调用正在执行。

这时，一个后台任务（比如一个子 agent）完成了，它有结果要汇报给主会话。

**如果直接注入当前轮次，会发生什么？**

轮次 N 的上下文（用户消息、AI 的部分响应、已执行的工具结果）已经在 AI 的视野里了。后台任务的结果突然插进来，会破坏这个轮次的完整性：AI 可能已经根据截至目前的信息开始了某个推理方向，突然多出一段信息，AI 的响应可能变得不一致，甚至产生逻辑矛盾。

更严重的情况：如果两个后台任务同时完成，同时注入，消息的顺序是不确定的，这可能导致 AI 看到的历史消息顺序和实际发生顺序不匹配。

## 轮次边界是什么

Claude Code 的对话是按"轮次（turn）"组织的。每一轮包括：
1. 用户消息进入
2. AI 处理，可能触发工具调用
3. 工具调用执行
4. AI 继续生成，直到本轮结束（stop_reason 达成）
5. 等待下一个用户消息

**轮次边界**就是上一轮结束、下一轮开始之间的那个时间点。

这个时间点是"安全"的注入窗口：上一轮的状态已经稳定，下一轮的 AI 响应还没开始，历史消息的顺序不会被破坏。

## pendingMessages 的等待策略

后台任务的消息进入 `pendingMessages` 队列后，等待条件满足才被消费。

等待的条件是：当前不在任何轮次的"中间状态"。具体来说，主会话的 AI 响应生成已经完成，工具调用已经执行完毕，状态已经稳定。

这有点像数据库的事务隔离：你不能在另一个事务的中间插入数据，只能等那个事务提交后，下一个事务开始前，再写入。

## 为什么不用锁

一个替代方案是给消息注入加锁：当主会话在处理某一轮时，加锁阻止后台任务注入；轮次结束后释放锁，后台任务竞争获取锁然后注入。

这样可以保证消息不会在轮次中间插入，但有几个问题：

**锁的粒度很难把握**：对话轮次的边界不是一个时间点，而是一段持续时间（AI 生成响应可能需要几秒）。持锁太久会阻塞后台任务汇报结果。

**优先级不明确**：多个后台任务同时持有待注入消息，释放锁后谁先注入？锁不能表达这个顺序。

**与 React 状态管理不兼容**：Claude Code 的 UI 层用 React 管理状态，React 的更新也有自己的批处理和调度机制，直接加锁会和 React 的调度冲突。

`pendingMessages` 队列比锁更适合这个场景，因为它：
- 不阻塞后台任务（后台任务可以随时写入队列）
- 保证注入顺序（FIFO 队列）
- 消费时机由主会话控制（在轮次边界检查队列，决定哪些消息进入下一轮）

## 消息的生命周期

一条后台任务消息的完整生命周期：

1. **后台任务完成**，生成结构化通知消息
2. **写入 pendingMessages**，消息进入等待队列
3. **主会话结束当前轮次**，检查 pendingMessages
4. **决定是否消费**：有些消息有时效性，如果等待太久可能不再适合注入（比如任务已经被用户取消）
5. **注入到下一轮的对话历史**，AI 在下一轮开始时看到这条消息
6. **AI 决定如何响应**，可能触发新的工具调用或直接回复用户

步骤 4 的"有些消息有时效性"是这个机制里比较微妙的部分。等太久才注入，消息可能已经过时了——用户可能已经手动查看了那个后台任务的状态，重复注入反而造成混乱。

## 与 backgroundSignal 的关系

`backgroundSignal` 是另一个后台通信字段，但它和 `pendingMessages` 的用途不同。

`backgroundSignal` 用于后台任务向主会话发送"我需要你注意一下"的信号，是一种主动触发（让主会话提前结束当前轮次，来处理后台消息）。

`pendingMessages` 是消息的实际载体，存储待注入的内容。

两者配合：后台任务完成时，先把消息放进 `pendingMessages`，然后发送 `backgroundSignal` 告知主会话有待处理的消息。主会话收到信号后，在合适的时机消费 `pendingMessages`。

## 面试要点

1. **轮次边界是安全注入窗口**：在 AI 响应生成过程中注入消息会破坏轮次完整性，等到轮次结束后再注入是安全的选择。
2. **队列比锁更适合这个场景**：不阻塞生产者，保证消费顺序，消费时机由消费者控制。
3. **消息有时效性**：等待时间过长的消息可能已经过时，注入前需要检查有效性。
4. **pendingMessages 和 backgroundSignal 分工明确**：前者是消息载体，后者是触发信号，配合工作。
