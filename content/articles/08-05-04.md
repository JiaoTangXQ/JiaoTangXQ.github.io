# 分布式系统分析：重连时清空本地状态，是在用保守一致性换用户体验

## 重连后的状态不确定性

当 WebSocket 断线，然后重连成功，本地缓存了一些状态：

- 若干工具正在进行中（`inProgressToolUses`）
- 若干工具有进度显示（进度条/动画）
- 后台任务计数（"2 tasks running"）

这些状态在断线前是准确的。但在断线期间，远端 CCR 继续运行，可能：
- 工具已经执行完了
- 后台任务完成了或新增了
- 某些工具因为超时被取消了

本地不知道这段时间发生了什么。问题来了：**应该继续展示旧状态（乐观），还是清空等待新数据（保守）？**

## CAP 定理的影子

这个问题和分布式系统里的 CAP 定理有相同的结构。

CAP 定理说：在网络分区（P）发生时，你只能在一致性（C）和可用性（A）之间选一个：

- 选 AP：分区期间继续提供服务，接受数据可能不一致
- 选 CP：分区期间拒绝服务，保证数据一致性

把这个模型套到"断线重连"场景：

- **乐观策略（AP 类比）**：继续展示旧状态，让 UI 保持可用，接受状态可能是过期的
- **保守策略（CP 类比）**：清空旧状态，等待新数据，让 UI 暂时为空

Claude Code 选择了保守策略：

```
断线 → 清空工具进度
断线 → 清空进行中的 tool use 集合
断线 → 清空远端任务计数
重连 → 等待服务端推送新的状态
```

## 为什么不选乐观策略？

乐观策略更"顺滑"——用户不会看到状态突然消失。但它有一个致命的问题：**状态漂移可能是永久的**。

考虑这个场景：

1. 工具 A 在执行（本地显示进度条）
2. WebSocket 断线
3. 工具 A 在远端完成了，但没有 `tool_result` 消息发到本地
4. WebSocket 重连
5. **本地进度条还在转**，但工具早就结束了

如果服务端不会重播已完成的工具结果，这个进度条就会永远转下去。用户以为工具还在执行，但实际上任务早就结束了。

更糟的情况：工具 A 结束后，远端开始执行工具 B。本地因为还"以为"工具 A 在运行，可能做出错误的 UI 决策——比如阻止新的工具发起。

清空旧状态避免了这种"幽灵状态"（ghost state）——状态看起来存在，实际上对应的操作早就结束了。

## 保守策略的代价

清空状态的用户体验影响：

- 断线重连时，进度条消失，然后重新出现（如果工具还在运行）
- 任务计数归零，然后重新更新
- 视觉上有一个短暂的"空白期"

这对实时感要求高的场景是一个明显的体验损失。但 Claude Code 的判断是：**工作台的可信度比视觉流畅度更重要**。

一个显示错误进度的工具状态，比短暂空白更令人困惑，也更难解释。"进度消失了然后重新出现"是可理解的（"断线重连了"），而"进度一直在转但实际上早就结束了"是令人费解的。

## 与 `from_sequence_num` 的配合

服务端有一个 `from_sequence_num` 机制：断线重连时，客户端带上上次收到的 seq-num，服务端从那个位置开始重放消息，补齐断线期间的内容。

这个机制和清空策略是互补的：

- 清空：解决"旧状态可能是错的"
- `from_sequence_num`：解决"断线期间的新状态需要补全"

两者组合，断线重连后本地状态经历：空白 → 从断线点开始填充 → 最终一致。

## 实现细节：BoundedUUIDSet

与去重相关的 `BoundedUUIDSet`，也体现了"保守而不是激进"的设计风格：

- 不是无限 Set（防止内存泄漏）
- 但保留足够多的历史（防止断线重连后的消息被误认为重复）
- 大小是可配置的（根据场景调整保留历史的量）

## 面试指导

"分布式系统中，客户端如何处理短暂断线"是系统设计面试的常见场景。

优秀的回答需要：

1. **识别状态的时效性**：哪些状态在断线后可能已经变化（进度、计数），哪些不会变化（历史消息）
2. **说清楚一致性策略的取舍**：乐观（AP）vs 保守（CP），各自的代价
3. **补充技术机制**：seq-num 重放、状态清空、keep-alive，这三者如何配合实现"最终一致"
4. **讨论用户感知**：哪种不一致对用户更难以接受（永久漂错 vs 短暂空白）

Claude Code 的保守选择，是一种有意识的产品决策，不是技术上的偷懒。在设计文档中说明这个决策和理由，比只说"断线重连"要有说服力得多。
