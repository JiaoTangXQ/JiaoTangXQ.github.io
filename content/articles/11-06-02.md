# 面试题：TaskListV2 的任务优先级排序算法——为什么最近完成要排在进行中前面？

很多候选人会本能地认为"进行中"应该排最前面，因为那是"最需要关注的活跃任务"。这道题要考察你能不能打破这个直觉，理解不同任务状态的可见时间窗口逻辑。

## 排序优先级的全貌

```typescript
const prioritized = [
  ...recentCompleted,   // 30s 内刚完成
  ...inProgress,        // 正在进行
  ...pending,           // 待处理（未阻塞先，已阻塞后）
  ...olderCompleted,    // 更早完成
];
visibleTasks = prioritized.slice(0, maxDisplay);
```

**最高优先级：最近 30s 内完成的任务**

这个排序看起来反直觉，但背后有清晰的逻辑：

完成反馈有时效性。任务刚完成的那一刻，是用户最需要看到"这件事确实收尾了"的时刻。如果这一刻任务区里全是 in_progress 任务，用户可能根本没看到某个重要任务已经完成。

**in_progress 排第二，而不是第一**

进行中的任务是"持续可见"的——它们今天在，明天还在，用户多看几眼就会记住。但"30s 内完成"是转瞬即逝的时间窗口，错过了就是错过了。

这是一个**信息时效性驱动的优先级**，而不是"重要性驱动"。

## recentCompleted 的边界判断

```typescript
const recentCompleted: Task[] = [];
const olderCompleted: Task[] = [];
for (const task of tasks.filter(t => t.status === 'completed')) {
  const ts = completionTimestampsRef.current.get(task.id);
  if (ts && now - ts < RECENT_COMPLETED_TTL_MS) {  // RECENT_COMPLETED_TTL_MS = 30_000
    recentCompleted.push(task);
  } else {
    olderCompleted.push(task);
  }
}
recentCompleted.sort(byIdAsc);
olderCompleted.sort(byIdAsc);
```

`completionTimestampsRef` 记录的是任务**第一次**进入 completed 状态的时间戳，不是最后一次更新时间。这个细节很重要：如果任务反复从 completed 切回其他状态再完成，时间戳不会重置（因为 `set` 操作只在"从未记录"时执行）。

等等，实际代码是这样的：

```typescript
for (const id of currentCompletedIds) {
  if (!previousCompletedIdsRef.current.has(id)) {
    completionTimestampsRef.current.set(id, now);  // 只有新完成时才记录
  }
}
```

只有"这次 completed 集合里有、上次没有"的任务才记录时间戳——这保证了时间戳是任务首次完成的时间。

## pending 的内部排序：阻塞感知

```typescript
const pending = tasks.filter(t => t.status === 'pending').sort((a, b) => {
  const aBlocked = a.blockedBy.some(id => unresolvedTaskIds.has(id));
  const bBlocked = b.blockedBy.some(id => unresolvedTaskIds.has(id));
  if (aBlocked !== bBlocked) return aBlocked ? 1 : -1;  // 被阻塞的往后排
  return byIdAsc(a, b);  // 同级的按 ID 升序
});
```

未被阻塞的 pending 任务排前面，被阻塞的排后面。逻辑是：**能推进的任务比不能推进的任务更值得被看见**。

`unresolvedTaskIds` 是所有非 completed 任务的 ID 集合。一个 pending 任务的 `blockedBy` 里如果有任何任务还在这个集合里，就说明它还不能开始。

## 为什么 olderCompleted 排最后而不是直接不显示

你可能会问：已经完成很久的任务，为什么还要显示？

几个原因：

1. **一致性**：用户配置了 10 个任务，前 3 个快速完成了，如果 `maxDisplay` 是 5，把那 3 个旧完成项踢掉会让用户觉得"任务消失了"
2. **总任务数的参考**：看到所有 10 个任务（包括已完成的），用户能更好地评估整体进度
3. **当 in_progress 和 pending 都很少时**：旧完成项填充剩余槽位，避免任务面板看起来很空

但它们是最低优先级的——当空间紧张时，旧完成项是第一批被推进 `hiddenTasks` 的。

## 这个排序设计的 UX 含义

如果你把自己想象成一个协调多个 agent 工作的用户：

- 刚完成的任务 → 需要确认，可能需要根据结果调整下一步
- 进行中的任务 → 需要监控，出问题了需要干预
- 可推进的 pending → 需要了解，将来可能需要调整优先级
- 被阻塞的 pending → 需要知道，但暂时无法行动
- 旧的完成项 → 历史记录，参考用

这个 1-2-3-4-5 的优先级，精确地映射了"我现在最该把注意力放在哪里"的人类决策逻辑。

## 面试指导

**直接问法**："为什么最近完成的任务会排在进行中的任务前面？"

答：因为两者的可见时间窗口不同。进行中的任务是持续可见的（在整个任务周期内都在）；最近完成的任务只有 30 秒的"确认窗口"，之后会变成 olderCompleted 被降级。在这 30 秒内，用户需要看到完成反馈来做决策，这比看到进行中任务更紧急。

**追问**："如果用户有 20 个任务，maxDisplay 是 5，5 个都是刚完成的，进行中的一个都看不到，这合理吗？"

答：这是一个边界情况，但实际上在 30 秒内同时完成超过 maxDisplay 个任务是少见的。更重要的是，用户看到 5 个刚完成的任务，立刻就能评估本轮的结果，这个反馈时刻是高价值的。进行中的任务在状态稳定后会自然浮上来（30s 后刚完成的降级）。如果这成为问题，可以给 recentCompleted 加一个上限（比如 maxDisplay/2），但当前没有这个限制。

**代码细节**："completionTimestampsRef.current.set 只在新完成时执行——如果任务完成后重置为 pending，再次完成会发生什么？"

答：第一次完成的时间戳仍然存在（`set` 操作只在 `!previousCompletedIdsRef.current.has(id)` 时执行，但 previousCompletedIdsRef 只记录上一次渲染时的完成集合，不是永久记录）。如果任务从 pending 重新完成，因为 `previousCompletedIdsRef` 里没有这个 ID，会重新设置时间戳。所以重新完成会重置 30s 窗口——这在语义上是对的，用户需要对"重新完成"这件事有确认期。

*核心考察点：信息时效性驱动的优先级设计、任务生命周期的视觉映射、边界情况的合理性判断。*
