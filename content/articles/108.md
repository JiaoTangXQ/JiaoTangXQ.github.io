# outputOffset 和延迟驱逐：后台任务的"可续读"和"体面退场"怎么配合？

后台任务面对两个时间段的问题：**运行中怎么随时可查**，和**结束后怎么不立刻消失**。

## 续读：outputFile + outputOffset

`LocalAgentTaskState` 继承了 `TaskStateBase` 的两个字段：

```typescript
outputFile: string    // 任务输出写入的文件路径
outputOffset: number  // 当前已消费到哪里（字节偏移量）
```

这两个字段构成了一套"外部痕迹系统"。

任务在运行时，输出会持续写入 `outputFile`。不管前台是否有人在看这个任务，写入不会停止。当用户切回来查看这个任务时，系统从 `outputOffset` 位置开始读文件，把这期间积累的所有输出呈现出来。

这比"保存所有 output 在内存里"更可靠：内存里的状态如果在切换时被清理掉了（比如 AppState 的 GC），文件还在；而且文件读取可以做流式懒加载，不需要把整个输出一次性加载到内存。

`diskLoaded` 字段配合这个机制工作：

```typescript
diskLoaded: boolean  // Bootstrap 是否已经从 sidechain JSONL 读取并合并进 messages
```

第一次重新进入一个长时间运行的任务时，需要先把历史输出从磁盘加载到 messages 里（"bootstrap"），之后的新增内容才能做流式追加。`diskLoaded` 是这个初始化步骤的完成标记。

## 留存与驱逐：retain + evictAfter

这是 `LocalAgentTaskState` 特有的两个字段：

```typescript
retain: boolean         // UI 正在持有这个任务（阻止驱逐）
evictAfter?: number     // 驱逐时间戳（毫秒）：超过这个时间可以清理
```

注释里对 `retain` 的描述很精确：

> "UI is holding this task: blocks eviction, enables stream-append, triggers disk bootstrap. Set by enterTeammateView. Separate from viewingAgentTaskId (which is 'what am I LOOKING at') — retain is 'what am I HOLDING.'"

"正在看"和"正在持有"是两件不同的事。`viewingAgentTaskId` 表示用户界面当前聚焦在哪个任务，`retain` 表示系统应该为这个任务维持完整的运行时状态。一个任务可以被"持有"但用户视线已经移走，或者被"看着"但还没有显式持有（比如刚刚切换过来，bootstrap 还没完成）。

`evictAfter` 是延迟驱逐的实现机制。任务进入终态（completed/failed/killed）时，不是立刻从 AppState 里删除，而是设置一个未来时间戳。只有当当前时间超过 `evictAfter`，并且没有 `retain` 标记，这个任务才会被 GC 清理。

为什么要延迟？因为后台任务刚完成时，主会话可能还没有处理完任务的完成通知。如果任务对象在完成的瞬间就消失了，通知处理逻辑可能会找不到对应的任务元数据（比如任务描述、toolUseId 等），无法生成完整的完成消息。

## PANEL_GRACE_MS 常量

`utils/task/framework.ts` 里有 `PANEL_GRACE_MS` 常量，它是 evictAfter 的默认延迟时间。这个值的大小需要在两个目标之间取平衡：

- 太短：任务刚完成主会话还没处理好通知，任务对象就消失了
- 太长：AppState 里积累了大量已完成的任务状态，占用内存

"Grace period"这个命名很准确：给刚完成的任务一段宽限期，让它的后续处理有时间完成。

## 统一基础设施的好处

`outputFile + outputOffset + retain + evictAfter` 这四个字段，是所有有"续读"需求的任务类型共享的基础设施。无论是 `local_agent`、`remote_agent` 还是 `in_process_teammate`，只要它们继承了 `TaskStateBase`（并对于 agent 类型扩展了 `LocalAgentTaskState`），就自动拥有了这套续读和延迟驱逐的能力。

不需要每种任务类型各自实现自己的"输出持久化"和"体面退出"逻辑，因为这是统一的基础设施层提供的。

## 面试答题要点

被问"你们的后台任务用户切走了再切回来，怎么恢复状态"，关键要说清楚：

1. **输出不丢失**：写入 outputFile，不依赖内存状态
2. **增量续读**：outputOffset 记录读取位置，不需要从头加载
3. **初始化区分**：diskLoaded 区分"第一次进入"和"后续追加"两种模式

追问"任务完成后多久会被清理"：evictAfter 控制延迟时间，retain 标记可以阻止清理（比如用户正在查看已完成任务的结果时，不应该被 GC）。这是"体面退场"的具体实现：不是立刻消失，而是给足时间让主会话和 UI 完成善后。
