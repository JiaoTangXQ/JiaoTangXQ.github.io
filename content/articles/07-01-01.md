# 对比分析：VS Code 插件系统 vs Claude Code 扩展三层架构

## 面试题引入

> 你会怎么设计一个"可扩展的 AI 工具"的插件系统？参考 VS Code 的扩展模型是否合适？

这道题没有标准答案，但能否说清"参考哪里合适、哪里不合适"，考察的是真正的架构判断力。

---

## VS Code 的选择：单一 Extension API

VS Code 的插件系统非常经典。所有插件都通过同一套 `vscode` API 接入：注册命令、提供 TreeView、响应文档变更、贡献语言特性……不管插件来自 Marketplace、本地路径还是打包进 VS Code 本身，都走同一条 `ExtensionContext` 生命周期。

这套设计的优点显而易见：
- 作者只需学一个 API
- 宿主只需维护一套激活/停用逻辑
- 能力发现统一在 `package.json contributions` 字段里声明

适合 VS Code 的原因是：所有插件本质上都在"为编辑器添加功能"，约束是同质的——都要在 Node.js 进程里运行，都要响应文件事件，都要渲染 UI。

---

## Claude Code 的选择：三套独立路径

`loadSkillsDir.ts` 的注释不说"扩展"，说的是"技能"（skills）。`pluginLoader.ts` 处理的是 marketplace 和安装来源。`client.ts` 处理的是 MCP 连接状态。三者几乎没有共享的抽象基类。

**为什么不用单一 Extension API？**

因为三种来源的生命周期根本不同：

| | 技能 | 插件 | MCP |
|---|---|---|---|
| 发现时机 | 启动时扫描磁盘 | 安装后启用 | 连接后握手 |
| 失败处理 | 跳过单文件，继续加载 | 整包不进场 | needs-auth 缓存 + 重连 |
| 能力动态变化 | 否 | 否 | 是（list_changed） |
| 依赖网络 | 否 | 可选（来源校验） | 是 |
| 认证 | 否 | 否 | 是（OAuth / XAA） |

如果强行抽象成一套 Extension API，要么接口因为最复杂的那一类（MCP）而变得臃肿，要么最简单的那一类（本地技能）也被迫背负它用不到的状态机。

这就是为什么 Claude Code 选择三套路径而不是一个"插件 API"——不是工程懒惰，而是三种约束本来就不同构。

---

## 统一点在哪里？

统一发生在最终出口，而不是入口。`getCommands(cwd)` 把所有来源的产物都收成 `Command[]`：

```typescript
const allCommands = await loadAllCommands(cwd)
// loadAllCommands 并发加载：
//   getSkills(cwd) → skillDirCommands + pluginSkills + bundledSkills + builtinPluginSkills
//   getPluginCommands()
//   getWorkflowCommands(cwd)
// 然后合并进 COMMANDS()（内置命令）
```

VS Code 的统一点在入口（Extension API），Claude Code 的统一点在出口（Command 类型）。前者更便于插件作者，后者更便于系统内部消费者（补全、权限、调度）。

---

## 一个值得记住的工程原则

**同质约束 → 统一入口更好；异质约束 → 统一出口更好。**

VS Code 的插件约束是同质的（都在编辑器里运行），所以统一 Extension API 合理。Claude Code 的扩展约束是异质的（本地文件、市场安装、远程协议），所以统一 Command 出口更合理。

---

## 面试指导

被问到"如何设计可扩展系统"时，先问清楚扩展来源的约束是否同质。如果是，单一插件 API 是好选择；如果不是，分层入口 + 统一出口往往更稳。

说出"我会先分析各种扩展来源的生命周期差异，再决定统一层放在入口还是出口"——这比直接说"参考 VS Code 的 Extension API"高一个层次。
