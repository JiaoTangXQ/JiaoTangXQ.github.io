# 深度分析：抽象层次决定了什么是稳定的，什么是可变的

## 后端可切换的含义

在 swarm 模式里，"后端"是指每个 agent 实际运行在哪里：

- `in_process_teammate`：所有 agent 在同一个进程里运行（最简单）
- `local_agent`：每个 agent 是本机上独立的进程
- `remote_agent`：agent 运行在远端容器里

从用户的角度，这三种后端的体验应该是一致的：
- 看到的任务面板是一样的
- 权限请求的处理方式是一样的
- agent 的输出展示方式是一样的
- 团队里的身份（leader、worker）是一样的

**面试题：这个"体验一致"是靠什么保证的？如果后端可以替换，骨架是什么？**

## 骨架是接口，不是实现

TaskType 的定义里有三种 agent 类型：

```typescript
| 'local_agent'
| 'remote_agent'
| 'in_process_teammate'
```

但从任务面板的角度看，这三种都是"正在后台运行的任务"。面板的渲染逻辑不需要知道任务是在本地还是远端——它只需要 `TaskStateBase` 里的信息：id、status、description、outputFile。

`TaskStateBase` 就是骨架的一部分：**无论后端如何实现，都必须提供这组字段，这是"参与这个系统"的最低要求**。

## 骨架的其他组成部分

除了 `TaskStateBase`，骨架还包括：

**权限协议**：无论是本地 agent 还是远端 agent，遇到需要权限的操作时，都必须走相同的 permissionSync 流程。这个流程不依赖 agent 在哪里运行，只依赖"有没有 leader 的邮箱"和"leader 能不能看到这个请求"。

**邮箱接口**：agent 间通信走邮箱协议（发送、接收消息），不管 agent 是本地的还是远端的（远端 agent 可以通过网络代理接入同一个邮箱系统）。

**身份约定**：所有 agent 都有 agentId、agentName、teamName，这些身份信息在所有后端类型里是一致的。

这些是骨架：如果某种后端实现违反了这些骨架约定（比如不通过邮箱通信，或者不实现 TaskStateBase），它就无法融入这个系统，或者会破坏依赖这些骨架的其他组件。

## 后端实现细节应该被隔离的原因

想象 remote_agent 的连接实现细节泄漏到了 UI 层：

```typescript
// UI 组件里出现了这种代码
if (task.type === 'remote_agent') {
  // 显示连接状态
  showConnectionStatus(task.websocketState)
} else {
  // 普通状态显示
  showTaskStatus(task.status)
}
```

现在 UI 组件依赖了 `remote_agent` 的特定字段（`websocketState`）。

以后 `remote_agent` 改变了连接实现（从 WebSocket 换成 gRPC），UI 组件也要跟着改。以后新增一种 `remote_agent_v2` 类型，UI 组件要加一个新的 if 分支。

这就是"被后端实现细节带走"：UI 的稳定性依赖于后端实现的稳定性，而后端可以变化的东西多于 UI 应该关心的东西。

**正确的做法**：`remote_agent` 在 `TaskStateBase` 里提供 `status` 字段，UI 只依赖 `status`。连接的 WebSocket 是否正常，翻译成 `status: 'running'` 还是 `status: 'failed'` 之后再暴露给 UI。

## 当后端需要在 UI 里展示特殊信息时

有时后端的特殊信息需要展示给用户——比如 `remote_agent` 的延迟状态。

正确的做法不是让 UI 直接读后端的内部字段，而是让后端把这个信息**翻译成骨架能表达的形式**。

比如，延迟状态可以翻译成 `description` 字段的内容（"远端 agent，当前延迟 120ms"），或者通过标准的任务进度字段（`AgentProgress.summary`）展示。

这样，UI 仍然只依赖骨架，但通过骨架能展示后端想展示的特殊信息。

## 面试要点

1. **骨架是接口约定，后端是实现细节**：TaskStateBase 是骨架，具体的 WebSocket 连接是后端细节；骨架稳定，后端可变。
2. **隔离后端细节防止 UI 被拖着走**：UI 依赖骨架（status、description），不依赖后端特定字段，后端实现变化不影响 UI。
3. **后端特殊信息通过翻译进入骨架**：延迟状态、连接信息等，通过翻译成 description 或 summary 字段，在不破坏骨架的前提下展示给用户。
4. **多种后端类型的统一体验来自骨架的完整性**：所有后端都实现了相同的骨架接口，这是统一体验的工程基础。
