# VS Code 扩展权限 vs Claude Code：两种治理模型的对比

两种系统都在解决”给外部代码授权”的问题，但选择了截然不同的架构。对比阅读能帮助你理解 Claude Code 的设计为什么做成现在这样。

## VS Code 的权限模型：静态声明式

VS Code 扩展在 `package.json` 里声明所有需要的权限：

```json
{
  “contributes”: {
    “commands”: [...],
    “configuration”: {...}
  },
  “activationEvents”: [“onLanguage:typescript”]
}
```

核心特征：
- **安装时审批**：用户在安装扩展时看到权限列表，一次性同意
- **静态范围**：权限在 manifest 里声明，运行时不能动态扩大
- **沙箱隔离**：扩展运行在独立进程，通过消息传递访问 VS Code API
- **无上下文感知**：扩展拥有的权限与当前打开的项目无关

这个模型非常适合**工具扩展**（language server、格式化器、linter），因为这类扩展的权限边界明确且不变。

## Claude Code 的权限模型：动态快照式

Claude Code 的权限模型完全相反：

```typescript
// 权限不是静态声明的，而是在会话建立时从多来源汇成快照
export type ToolPermissionContext = DeepImmutable<{
  mode: PermissionMode               // 当前全局安全姿态
  alwaysAllowRules: ToolPermissionRulesBySource   // 多来源 allow 规则
  alwaysDenyRules: ToolPermissionRulesBySource    // 多来源 deny 规则
  strippedDangerousRules?: ToolPermissionRulesBySource  // 动态剥离的规则
}>
```

核心特征：
- **运行时构建**：权限不在代码里声明，而是在运行时从 settings 文件、CLI 参数、session 状态汇聚
- **会话级可变**：在同一会话里可以切换模式（default → auto），每次切换都重写当前权限状态
- **上下文感知**：同一条规则 `Bash(ls *)` 在不同工作目录下可能触发不同的运行时检查
- **动态剥离和恢复**：进入 auto 模式时主动剥离危险规则，离开时恢复

## 两者的核心差异

| 维度 | VS Code 扩展 | Claude Code |
|------|------------|-------------|
| 权限声明时机 | 安装前，静态 | 运行时，动态 |
| 权限粒度 | 功能级（能写文件/不能） | 命令级（`Bash(git *)` 允许，`Bash(rm *)` 拒绝） |
| 可变性 | 不可变（需要重装） | 可变（模式切换） |
| 来源数量 | 1（扩展自身声明） | 6+（多层来源） |
| 环境感知 | 无 | 有（policySettings 随部署环境变化） |

## 为什么 Claude Code 不用 VS Code 的声明式模型

Claude Code 面临的核心挑战是**命令级权限的指数爆炸**。VS Code 扩展的权限粒度是”能不能写文件”这个层面，有限可枚举。但 Claude Code 要控制的是 shell 命令的子集：

- `Bash(ls *)` 允许
- `Bash(git status)` 允许
- `Bash(git push origin main)` 需要询问
- `Bash(rm -rf /)` 拒绝

这种粒度的权限无法在安装时静态声明，只能在运行时结合具体命令和当前规则做匹配。

另一个原因是**多租户/多来源的必要性**。Claude Code 被用在企业环境、个人开发、CI/CD pipeline 等不同场景，每个场景的权限需求完全不同。静态声明式模型要求所有场景共享同一份权限声明，这在多来源场景下不可行。

## VS Code 扩展权限的优势

VS Code 的模型不是”差”，而是”适合工具”：
- 审计简单：看 package.json 就知道扩展要什么权限
- 信任建立简单：安装时一次审批，后续无打扰
- 沙箱隔离强：扩展进程天然隔离，权限逃逸面小

Claude Code 的动态模型在灵活性上占优，但付出的代价是：
- 权限状态不再静态可审计
- 快照过期问题（settings 变化后的窗口期）
- 学习成本：用户需要理解 default/plan/auto 等模式的语义

## 这道题考什么

**对比分析型问题**的考察重点通常不是”谁更好”，而是”各自解决了什么问题，做了什么取舍”。

好的回答应该包含：Claude Code 的动态快照模型是为了处理命令级权限的指数爆炸和多来源/多场景需求，VS Code 的静态声明模型适合边界明确的工具扩展。两者都是合理的设计，只是在不同的约束条件下做出了不同的权衡。

如果面试官进一步追问”Claude Code 的模型有什么弱点”，正确答案是：权限状态不如 VS Code 的 manifest 透明易审计，快照的短暂过期窗口在高频场景下可能引入规则不一致。
