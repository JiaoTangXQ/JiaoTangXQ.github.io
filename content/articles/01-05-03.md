# 场景推演：从"用户输入一句话"开始，追踪它在 Claude Code 里经过了哪些设计决策

## 起点：用户按下了回车

用户在 Claude Code 的 REPL 里输入：`帮我找出这个项目里所有未处理的 TODO`，然后按了回车。

接下来发生了很多事。如果一路追下去，每个停靠点都会遇到一个可以追问的设计决策。这篇文章沿着这条线走，在每个关键节点停下来，问"为什么要这样做"。

## 停靠点 1：processUserInput 里的输入分类

用户的输入首先进入 `processUserInput` 函数（在 `utils/processUserInput/processUserInput.ts`）。

这个函数做的第一件事是判断这次输入是什么类型：是 `/command`（斜杠命令）、还是普通的自然语言提示词、还是特殊的工具结果类型（在某些 SDK 调用模式下，"用户输入"可能是上一轮工具调用的结果）。

**设计决策：为什么要在一个函数里统一处理这三种类型？**

因为它们虽然处理路径不同（斜杠命令不经过模型、普通提示词要走 API、工具结果要追加到消息列表），但它们都需要"消费当前消息状态、更新消息状态、决定是否触发模型调用"。如果三种类型分开处理，每条路径都要自己管理消息状态，会出现各种状态不一致的边角情况。统一入口，统一出口（一个 `{ messages, shouldQuery, allowedTools, model, resultText }` 对象），让后面的 `QueryEngine` 不需要关心输入是哪种类型来的。

## 停靠点 2：未知斜杠命令的处理

假设用户输入的不是 `/compact` 这种已知命令，而是 `/some-custom-command`。

Claude Code 不会报错"命令不存在"。它会把这个未知斜杠命令当成普通的自然语言提示词处理——`/some-custom-command` 被原样保留在消息内容里发给模型。

**设计决策：为什么不在未知命令时报错？**

因为用户可能有自定义的提示词习惯，喜欢用斜杠前缀来表示某种格式。也因为未知命令的"正确"处理是"让模型去理解"，而不是系统报错后让用户重新输入。用户目的是完成某件事，系统报告"命令不存在"只是增加了一次交互往返而没有产生价值。

这个设计体现了一种具体的取舍：宁愿偶尔"把命令名字发给模型看看模型怎么响应"，也不要让一个打错字的用户看到一条冷冰冰的错误消息。

## 停靠点 3：fetchSystemPromptParts 的并行装配

在模型被调用之前，`QueryEngine.submitMessage` 调用 `fetchSystemPromptParts`：

```typescript
const [defaultSystemPrompt, userContext, systemContext] = await Promise.all([
  customSystemPrompt !== undefined
    ? Promise.resolve([])
    : getSystemPrompt(tools, mainLoopModel, additionalWorkingDirectories, mcpClients),
  getUserContext(),
  customSystemPrompt !== undefined ? Promise.resolve({}) : getSystemContext(),
])
```

三个来源被并行请求：工具描述（`getSystemPrompt`）、用户上下文（CLAUDE.md + 日期，`getUserContext`）、系统上下文（git 状态，`getSystemContext`）。

**设计决策：为什么 customSystemPrompt 设置后跳过 getSystemContext？**

注释里写得很清楚：自定义系统提示**替换**默认系统提示，`getSystemContext` 里的 git 状态会被附加在默认系统提示后面——但如果默认系统提示都被替换了，把 git 状态附加在一个不再使用的提示上没有意义。SDK 调用方用 `customSystemPrompt` 时，是在明确表达"我要完全控制系统提示的内容"，不应该有任何默认内容悄悄混进来。

## 停靠点 4：tool_use 的闭环要求

模型返回一个 `tool_use` 块，比如 `GrepTool` 调用，参数是 `{ pattern: "TODO", path: "." }`。

工具执行完毕后，必须在消息历史里追加一条 `tool_result` 消息。这不是可选的——Anthropic API 要求每个 `tool_use` 块都必须有对应的 `tool_result` 才能发出下一条请求。如果工具执行过程中发生了错误，`tool_result` 里的内容是错误信息，而不是省略这条消息。

**设计决策：为什么不能让工具失败后直接继续？**

因为 API 的消息历史必须是"闭合的"：每个 assistant 消息里的 tool_use 都必须有对应的 tool_result 响应，否则下次 API 请求会被拒绝（协议错误）。这个要求不是 Claude Code 自己的设计选择，而是 Anthropic API 的协议约束。Claude Code 在这里做的事情是：把这个协议约束转化为本地的不变式，确保任何代码路径（包括工具抛异常的路径）都会生成有效的 tool_result。

## 停靠点 5：maxTurns 的计数逻辑

`QueryEngine` 支持 `maxTurns` 参数——最多允许多少轮工具调用。但"一轮"的定义是什么？

答案是：一次模型响应算一轮。如果模型在一次响应里调用了 5 个工具，这算 1 轮，不是 5 轮。只有模型发出新一次 API 请求，轮数才加一。

**设计决策：为什么用"模型响应次数"而不是"工具调用次数"来计算轮数？**

因为模型每次响应的工具调用数量是模型自己决定的（并发工具调用），外部不应该假设这个数量。用"工具调用次数"作为 maxTurns 的单位，会让 maxTurns 的实际效果依赖于模型的并发调用习惯，这是不稳定的。用"模型响应次数"，每次 API 往返算一轮，单位稳定且可预测。

## 这条路的价值

把用户输入到模型响应这条链路走一遍，会发现每个停靠点都有一个值得追问的设计选择。这些选择不是"优化"或"功能"，而是面对具体问题后做出的判断：

- 未知命令不报错 → 对用户意图的宽容解读
- 系统上下文在自定义提示时跳过 → 对调用方控制权的尊重
- 工具结果必须闭环 → 把 API 约束内化为本地不变式
- maxTurns 用响应次数 → 选择稳定可预测的单位

这些判断合起来，构成了这个系统的工程品位。技术文档的作用，是把这些判断说清楚，让读者不只知道系统能做什么，还知道它为什么被设计成今天这个样子。

## 面试指导

**考法**：`走一遍 Claude Code 里一次完整用户请求的执行路径，并在你认为值得深入的地方停下来解释`。

**满分标准**：路径要走得完整，不能只说"进 main，调模型，返回结果"；每个停靠点要有对应的"为什么这里要这样做"，而不是只描述"这里做了什么"；最后能总结出几个贯穿整条路径的一致性主题（比如"让模型有能力做更好决策"、"把协议约束内化为本地不变式"）。

**一句话区别高低分**：低分答案描述执行顺序，高分答案在执行顺序里看见设计判断。
