# 为什么这本书必须沿流程和设计问题一起下钻

最合适的写法，不是目录树，也不是功能表，而是把两件事绑在一起: 先沿真实流程往前走，再在每个关键节点停下来，追问那里到底在解决什么设计问题。

流程负责给读者方向感。程序怎样启动，输入怎样进来，上下文怎样装配，主循环怎样思考，工具怎样开动，失败后怎样续上，下一轮怎样接住上一轮。只要主线不丢，普通读者就不会迷路。

设计问题负责把深度找回来。走到某个节点时，真正要讲的不是“这里有一个类”或“这里调用了一个函数”，而是“为什么这里必须分层”“为什么这里不能贪快”“为什么这里要先记账再行动”“为什么这里宁愿多一道恢复机制”。这样写，读者看到的就不再是一堆程序零件，而是一连串被认真解决过的难题。

Claude Code 本身也是这样长出来的。开发者不是先发明一张漂亮目录，再把代码塞进去；他们是在一个个真实问题前做选择。顺着流程走，再顺着问题下钻，才最接近这套系统真正的生成方式。

## 实现链
从代码映射看，目录树首先服务开发，不直接服务理解。`main.tsx`、`QueryEngine.ts`、`commands.ts`、`Tool.ts`、状态存储这些文件分别落在不同目录，但它们回答的其实是入口、运行时骨架、制度和状态问题。文档如果只照目录平铺，就看不见这些跨目录的实现链。

## 普通做法
最直觉的写法是按目录讲文件，或者按产品功能讲菜单。这两种写法都很常见，也最省心。

## 为什么不用那种做法
Claude Code 这本书没有只选其中一种，因为这套系统的精彩之处大多发生在交叉处：流程走到哪里，为什么要用这种边界处理。只按目录会让普通读者迷路，只按功能又会把实现判断糊成大块。

## 代价与回报
回报是文档能真正贴着代码里的设计问题下钻。代价是文档结构和目录结构不再一一对应，维护时必须不断做导读、回收和索引。
