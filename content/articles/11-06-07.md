# 面试题：hiddenSummary 为什么不写"+N more"而是分类写出状态构成？

这道题考察的是信息密度和信息质量的区别。"+N more"有信息，但是最低密度的信息。分类摘要每个字符的信息量高出几倍。

## 代码实现

```typescript
let hiddenSummary = '';
if (hiddenTasks.length > 0) {
  const parts: string[] = [];
  const hiddenPending = count(hiddenTasks, t => t.status === 'pending');
  const hiddenInProgress = count(hiddenTasks, t => t.status === 'in_progress');
  const hiddenCompleted = count(hiddenTasks, t => t.status === 'completed');
  if (hiddenInProgress > 0) parts.push(`${hiddenInProgress} in progress`);
  if (hiddenPending > 0) parts.push(`${hiddenPending} pending`);
  if (hiddenCompleted > 0) parts.push(`${hiddenCompleted} completed`);
  hiddenSummary = ` … +${parts.join(', ')}`;
}
```

最终渲染：

```typescript
{maxDisplay > 0 && hiddenSummary && <Text dimColor>{hiddenSummary}</Text>}
```

`dimColor` 让这行摘要在视觉上区别于具体任务行，是"补充信息"而不是"主要内容"。

## "+N more" vs 分类摘要的信息量对比

**场景**：15 个任务，maxDisplay=5，显示了 5 个，还有 10 个隐藏。

"+10 more" 告诉用户：
- 还有 10 个任务

"… +2 in progress, 3 pending, 5 completed" 告诉用户：
- 还有 2 个活跃的任务（in progress）
- 还有 3 个待推进的任务（pending）
- 还有 5 个已完成的任务（历史记录）

第二种信息让用户能评估：**后台还有多少工作量**。`2 in progress + 3 pending = 5 个未完成的任务`，说明工作还远未结束。`5 completed` 说明已经完成了相当部分。

如果只看 "+10 more"，用户无法判断这 10 个是已经完成的还是还在跑的，也无法估计剩余工作量。

## 为什么 in_progress 排在 pending 前面

```typescript
if (hiddenInProgress > 0) parts.push(`${hiddenInProgress} in progress`);
if (hiddenPending > 0) parts.push(`${hiddenPending} pending`);
if (hiddenCompleted > 0) parts.push(`${hiddenCompleted} completed`);
```

输出：`+2 in progress, 3 pending, 5 completed`（而不是按字母顺序）

这个顺序和可视区域的优先级顺序是一致的：in_progress > pending > completed。摘要行的信息顺序，和上面具体任务行的视觉优先级保持一致，用户建立的认知框架是统一的。

## 只写非零项

```typescript
if (hiddenInProgress > 0) parts.push(...)
if (hiddenPending > 0) parts.push(...)
if (hiddenCompleted > 0) parts.push(...)
```

"0 completed" 不会出现在摘要里。这是信息系统里的"零值不传递"原则：告诉用户"没有 X"和不提 X，信息价值完全不同。"0 completed" 是噪音，用户不需要被告知后台没有已完成的任务——没提到就意味着没有。

## dimColor 的语义

```typescript
<Text dimColor>{hiddenSummary}</Text>
```

`dimColor` 不是随意的样式选择：
- 具体任务行：主要信息，正常亮度
- hiddenSummary：辅助信息，降低亮度

这在视觉上建立了层级：眼睛先看具体任务，后看摘要，而不是两者竞争注意力。

## maxDisplay > 0 的守卫

```typescript
{maxDisplay > 0 && hiddenSummary && <Text dimColor>{hiddenSummary}</Text>}
```

`maxDisplay > 0` 是关键守卫。当 `maxDisplay = 0`（终端太矮，整个任务区退出）时，不显示 hiddenSummary。

逻辑上这很合理：如果连具体任务行都不显示了，在小屏幕上再显示一行"还有 N 个任务"反而是噪音。

## 和 standalone 总览行的区别

```typescript
// standalone 模式的总览（在 hiddenSummary 之上）
<Text dimColor>
  <Text bold>{tasks.length}</Text>{' tasks ('}
  <Text bold>{completedCount}</Text>{' done, '}
  ...
</Text>
```

standalone 总览（如果启用）是**全量数据**的摘要，包括所有任务不只是隐藏的。hiddenSummary 是**仅隐藏部分**的摘要。

两者都在帮助用户建立整体感知，但维度不同：standalone 说"全部是什么样的"，hiddenSummary 说"你没看到的那部分是什么样的"。

## 面试指导

**直接问法**："为什么不用'+N more'而是分类写？"

答："+N more" 只告诉数量，不告诉构成。"2 in progress, 3 pending, 5 completed"让用户能评估后台的活跃度和剩余工作量。对于需要协调多个 agent 的用户来说，知道"后台还有 2 个活跃任务"比知道"还有 10 个未显示"信息量高很多。

**追问**："如果全是 completed，会显示什么？"

答：`… +8 completed`，只有一项。in_progress 和 pending 都是 0，条件判断跳过，不写零值。用户看到这一行，知道后台还有一些完成的历史记录但没有活跃工作，可以专注于已显示的任务。

**设计追问**："如果这行摘要太长超出终端宽度怎么办？"

答：这行在 `<Text dimColor>` 里，Ink 默认会截断或换行（取决于父容器的 wrap 设置）。如果是截断，最坏情况是摘要被截断——但这是低优先级信息，截断比换行更好（不占额外行高）。实际上，`N in progress, M pending, K completed` 的格式相当紧凑，在标准 80 列终端里通常不会超宽。

**工程追问**："count 函数是怎么实现的？"

答：是 `../utils/array.js` 里的工具函数，`count(arr, predicate)` 等价于 `arr.filter(predicate).length`，但避免创建中间数组，更节省内存。这类数组工具函数在 Claude Code 里被广泛使用。

---

*核心考察点：信息密度 vs 信息量、零值不传递原则、视觉层级的语义、摘要行的边界守卫逻辑。*
