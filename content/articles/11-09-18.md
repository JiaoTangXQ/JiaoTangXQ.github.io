# selectedIndex 的 scrollTo 为什么通过 jumpState ref 而不是直接依赖 offsets？

实现"选中项保持可见"有两种方式，微小的实现差异会带来截然不同的滚动行为。

## 错误的实现：offsets 放进 deps

```typescript
// ❌ 有问题的写法
useEffect(() => {
  if (selectedIndex === undefined) return
  const top = offsets[selectedIndex]
  scrollRef.current?.scrollTo(top - HEADROOM)
}, [selectedIndex, offsets, scrollRef])  // offsets 在 deps 里
```

`offsets` 在每次消息高度更新时重建（flow：消息高度测量完成 → heightCache 更新 → offsets 重建）。在流式输出期间，每次 Claude 输出新的文字，正在生成的消息高度就会变化，offsets 就会更新，这个 effect 就会重跑。

效果：用户选中了第 100 条消息，但每次流式输出都把视口滚回选中位置。用户想手动滚下去看后面正在生成的内容，但 effect 不停把视口拉回去。

## 正确的实现：通过 ref 读取

```typescript
// ✅ 实际实现
const jumpState = useRef({
  offsets,
  getItemTop,
  getItemElement,
  messages,
  scrollToIndex
})
jumpState.current = { offsets, getItemTop, getItemElement, messages, scrollToIndex }

useEffect(() => {
  if (selectedIndex === undefined) return
  const s = jumpState.current
  const el = s.getItemElement(selectedIndex)
  if (el) {
    scrollRef.current?.scrollToElement(el, 1)
  } else {
    s.scrollToIndex(selectedIndex)
  }
}, [selectedIndex, scrollRef])  // 只有 selectedIndex 和 scrollRef
```

`jumpState.current` 每次渲染时都更新为最新的 offsets、getItemTop 等，但 effect 的 deps 里没有 `jumpState`（ref 对象的引用永远不变）。

这意味着：effect 只在 selectedIndex 变化（用户按 j/k 导航）时触发，不在 offsets 更新（高度测量）时触发。但 effect 执行时，通过 jumpState.current 读到的是最新的布局数据。

## getItemElement vs getItemTop 的优先级

effect 先尝试 `getItemElement(selectedIndex)`：

- 如果元素已挂载（在当前虚拟滚动窗口内），直接 `scrollToElement(el, 1)`
  - `scrollToElement` 精确：知道元素在 DOM 里的实际位置，scrollTo 不会因为 offsets 估算误差而产生跳动
  - `1` 是 headroom 行数：元素顶部距视口顶部 1 行，让用户看到完整的消息上下文

- 如果元素未挂载（在虚拟窗口外），回退到 `scrollToIndex(selectedIndex)`
  - scrollToIndex 用 offsets 估算位置，可能不精确，但 seek effect 会在挂载后做精确调整

这个优先级策略是"能精确就精确，不能精确就先到邻近位置"的务实路线。

## 面试指导

这道题是"effect deps 该包含什么"的进阶版本，考察你是否理解"频繁变化的数据不应该放进 deps"。

**判断标准**：effect 的 deps 里只放"需要 effect 重跑"的变量。如果某个变量变化时 effect 不需要重跑（只需要在 effect 执行时读到最新值），就通过 ref 访问，不放进 deps。

**offsets 的特性**：高频更新（每次高度测量都更新），但选中项的滚动不需要跟着每次更新触发。所以 offsets 走 ref，不走 deps。

**selectedIndex 的特性**：每次变化时 effect 必须重跑（用户导航到新位置）。所以 selectedIndex 放 deps。

**一个更细的区分**：`scrollRef` 也在 deps 里，但 scrollRef 是 ref 对象，ref 对象引用本身不变，实际上是 mutable container。把它放进 deps 是因为 scrollRef 可能指向不同的 ScrollBox 实例（不同的会话），这时需要重新绑定。
