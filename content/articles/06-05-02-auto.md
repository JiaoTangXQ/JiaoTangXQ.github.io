# auto 模式的分类器是什么？安全工具白名单背后的设计逻辑

`auto` 模式的核心是分类器。但分类器不会处理所有工具调用——有一批工具被直接判定为安全，跳过分类器直接放行。这个白名单的设计体现了对”什么样的操作绝对安全”的工程判断。

## SAFE_YOLO_ALLOWLISTED_TOOLS：被完全信任的工具

```typescript
// classifierDecision.ts
const SAFE_YOLO_ALLOWLISTED_TOOLS = new Set([
  // 只读文件操作
  FILE_READ_TOOL_NAME,        // 读文件
  GREP_TOOL_NAME,             // 搜索
  GLOB_TOOL_NAME,             // 文件名匹配
  LSP_TOOL_NAME,              // 语言服务器查询

  // 任务管理（只修改元数据，不改代码）
  TODO_WRITE_TOOL_NAME,
  TASK_CREATE_TOOL_NAME,
  TASK_GET_TOOL_NAME,
  // ...

  // Plan 模式控制
  ASK_USER_QUESTION_TOOL_NAME,
  ENTER_PLAN_MODE_TOOL_NAME,
  EXIT_PLAN_MODE_TOOL_NAME,

  // Swarm 协调（子 agent 有自己的权限检查）
  TEAM_CREATE_TOOL_NAME,
  TEAM_DELETE_TOOL_NAME,
  SEND_MESSAGE_TOOL_NAME,

  // 安全工具
  SLEEP_TOOL_NAME,
  YOLO_CLASSIFIER_TOOL_NAME,  // 分类器工具本身
])

export function isAutoModeAllowlistedTool(toolName: string): boolean {
  return SAFE_YOLO_ALLOWLISTED_TOOLS.has(toolName)
}
```

白名单里的共同特征：**要么只读，要么只改本地元数据，要么是协调工具（子 agent 有自己的权限检查）**。

注意 `TODO_WRITE_TOOL_NAME` 在白名单里——写 todo list 只是修改内部状态，不影响文件系统或外部服务。

## 不在白名单里的：Write、Edit、Bash

文件写入工具（`Write`、`Edit`）和 shell 执行工具（`Bash`）不在白名单里，每次调用都要经过分类器评估。

注释里有一句关键说明：
> “Does NOT include write/edit tools — those are handled by the acceptEdits fast path (allowed in CWD, classified outside CWD).”

有一个 `acceptEdits` fast path：在当前工作目录内的写操作会被快速通过，工作目录外的写操作才走完整分类器流程。这是一个合理的经验判断：修改当前项目的代码是预期行为，修改其他地方的文件是异常。

## 分类器的工作原理

分类器（`yoloClassifier.ts`）向 Claude API 发一个独立的请求，提供：
1. 系统提示（`auto_mode_system_prompt.txt`）：描述 auto 模式的安全规则
2. 权限模板（`permissions_external.txt` 或 `permissions_anthropic.txt`）：当前的 allow/deny 规则描述
3. 当前工具调用的内容

Claude 作为分类器，输出 `allow` 或 `deny`，并附上理由。

这里有一个有趣的架构设计：**用 Claude 来审批 Claude 的操作**。分类器 Claude 和主线程 Claude 是独立的 API 调用，不共享状态。分类器看到的是工具调用的语义，不知道主线程在做什么任务。

## 分类器失败时怎么处理

分类器是外部 API 调用，可能失败（超时、API 错误、token 超限）。`yoloClassifier.ts` 里有 fail-closed 设计：

```typescript
const CLASSIFIER_FAIL_CLOSED_REFRESH_MS = 30 * 60 * 1000 // 30 分钟
```

当分类器不可用时，系统会降级到需要用户确认，而不是自动放行。这个 30 分钟的 refresh 计时器控制何时重新尝试恢复分类器。

## auto 模式的双重门禁

进入 auto 模式需要通过两个检查：

```typescript
function canCycleToAuto(ctx: ToolPermissionContext): boolean {
  if (feature('TRANSCRIPT_CLASSIFIER')) {
    const gateEnabled = isAutoModeGateEnabled()
    const can = !!ctx.isAutoModeAvailable && gateEnabled
    // ...
    return can
  }
  return false
}
```

1. `ctx.isAutoModeAvailable`：context 层面的可用性（启动时由 GrowthBook 检查决定）
2. `isAutoModeGateEnabled()`：实时检查（会话中途也可能变化，比如 circuit breaker 触发）

注释解释了为什么需要实时检查：
> “The live check prevents transitionPermissionMode from throwing, which would silently crash the shift+tab handler and leave the user stuck at the current mode.”

如果只检查 `ctx.isAutoModeAvailable`（启动时的快照），但会话中途 circuit breaker 断开了 auto 模式，Shift+Tab 切换时会抛出异常，导致用户界面卡住。实时检查确保了状态转换的安全性。

## 这道题考什么

**白名单边界的判断逻辑**：为什么 TODO_WRITE 在白名单里但 Write 不在？考察能否从”是否可能对外部状态造成不可逆影响”这个维度来分析工具。

**fail-closed 设计**：分类器失败时降级到人工确认而不是自动放行，这是 auto 模式最重要的安全保证。能说清楚这个设计意图的候选人理解了”自动化 ≠ 无监管”。

**双重门禁的必要性**：启动时快照 + 实时检查的组合，以及为什么只靠快照会导致 UI 卡住。这是一个典型的”缓存可能过期”问题，在权限系统里需要格外谨慎。
