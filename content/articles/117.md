# 深度分析：为什么 computerUse 工具命名要服务三个层次

## 命名不只是标识符

大多数时候，函数名和变量名的职责是"让人能找到这段代码"。

但在一个多层系统里，某些名字的职责远不止于此。它们是**跨层的协议**：同一个字符串在不同的层里有不同的含义，但这些含义必须保持一致。

Claude Code 的 `computerUse` 工具簇就是这样的例子。

## 三个层次各自的需求

**后端逻辑层**需要知道：这个工具调用是否需要特殊权限处理？是否属于"computer use"这个能力簇，从而适用该簇的权限规则？如果工具名字不统一，权限逻辑就要为每个具体操作单独判断，维护起来很复杂。

**AI 提示词层**需要知道：这些工具能做什么，什么时候该用哪个？提示词里的工具描述需要把 `screenshot`、`click`、`type`、`scroll` 等操作组织成一个连贯的"计算机操作"概念，而不是一堆零散工具。如果命名没有揭示这些工具的关系，AI 在生成提示词时就无法形成这个整体概念。

**前端渲染层**需要知道：这次工具调用是计算机操作，需要展示什么样的 UI（可能是截图预览、操作步骤展示）？如果工具名里没有类别信息，渲染层要么对所有工具用统一的展示方式（缺乏信息），要么维护一个"哪些工具名属于计算机操作"的列表（容易过时）。

## computerUse 命名约定如何解决这三个需求

具体的命名约定可以从源码里推断（虽然 computerUse 相关的具体实现文件在摘录的源码里不完整，但命名模式是一致的）：

工具名使用 `computer_use_*` 的前缀约定。比如 `computer_use_screenshot`、`computer_use_click`、`computer_use_type`。

这个前缀让：
- 后端权限逻辑可以通过检查前缀判断工具类别，不需要维护工具名列表
- 提示词生成可以把所有 `computer_use_*` 工具组织成一个工具组，描述它们作为整体的能力
- 渲染层可以对 `computer_use_*` 工具统一应用"计算机操作"的展示样式

## 命名是接缝的第一步

一个系统里，不同模块之间的交界面叫"接缝"。接缝处的设计决定了模块的耦合程度。

命名是接缝设计的第一步：在两个模块的边界上，双方需要约定用什么名字称呼同一个东西。

如果约定好了，每个模块可以独立演化——后端可以改权限逻辑，不影响 AI 怎么使用这些工具；前端可以改展示样式，不影响后端怎么处理工具调用。约定的名字是接缝的稳定锚点。

如果没有约定，或者约定不明确，接缝会变成紧耦合：任何一处改动都可能要同步修改多个层次，因为大家不确定"改这个名字会不会影响别的地方"。

## 命名约定失效时会怎样

想象 computer use 工具命名没有统一约定：

- `screenshot` 是截图工具
- `mouse_click` 是点击工具  
- `keyboard_input` 是键盘输入工具

后端想给"计算机操作"加一个统一的日志记录，要写：

```typescript
const isComputerUseTool = 
  toolName === 'screenshot' || 
  toolName === 'mouse_click' || 
  toolName === 'keyboard_input' || ...
```

每次新增一个计算机操作工具，要找到所有这样的列表并更新。这是维护负担，更是 bug 的温床——总有人忘记更新某个地方。

有了 `computer_use_` 前缀约定：

```typescript
const isComputerUseTool = toolName.startsWith('computer_use_')
```

新增工具只要遵循命名约定，所有层次自动认出它是计算机操作工具。

## 命名约定的代价

统一命名约定的代价是：约定一旦确立，很难修改。

如果某天需要把 `computer_use_screenshot` 改名为 `screen_capture`，会影响到：
- 已有的权限规则
- 提示词里对工具的引用
- 前端渲染规则
- 使用这些工具的历史会话记录

这不是不能改，但改动的代价比命名本身更大。所以命名约定一旦确立，必须认真对待，不能随意变更。

这也是为什么在系统设计阶段就应该确定命名约定：早期改动代价小，后期改动代价大。

## 面试要点

1. **命名是跨层协议**：工具名不只是标识符，它是后端、提示词、渲染层三个层次共同认可的"这是什么类型的工具"的约定。
2. **前缀是类别信息**：`computer_use_` 前缀把工具类别编码进了名字，让每个层次可以通过检查前缀而不是维护列表来判断工具类别。
3. **命名是接缝设计的第一步**：约定好名字，各层才能独立演化，不需要每次改动都同步多个地方。
4. **约定确立后难以修改**：早期认真设计命名约定，比后期重构代价更小。
