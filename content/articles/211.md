# 第十三卷回收索引：连续性世界里的附件、计划和技能线索重新带回工作台

## 实现链
连续性世界里最实际的一半，不是整理动作本身，  
而是整理以后，工作台上还能不能重新看见那些真正要继续往下做的线索。

这条线真正逼出来的，是第十三卷里关于“压缩不是清空，而是把桌面重新摆成还能继续工作”的判断。

这页实际回收的实现链是：
- `10-05-02` 把附件、计划和技能线索重新带回讲清，这会继续支撑 `13-07-02`，说明整理不是把桌面清空，而是把真正要继续工作的线索重新摆回现场。

好的连续性设计，不是让历史消失。  
它是让附件、计划和技能线索重新回到工作台，让下一轮还能顺着这些线继续做下去。

## 普通做法
更普通的做法，是一发现上下文变长就立刻整理，或者把当前状态和长期记忆统统塞回主会话，省掉额外的插槽和外置存储。

## 为什么不用
Claude Code 不这么做，是因为整理太早会打断主回合，记忆不分层会把短期噪音和长期知识混在一起，最后让连续性系统同时失去可读性和可恢复性。

## 代价
代价是需要额外维护 compact boundary、回合后插槽、外置 memory 和记忆回灌链，系统会更厚。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
