# 搜索预热分块 + sleep(0)：后台工作不能冻住前台交互

打开一个长 transcript，系统需要为搜索功能建立索引缓存——扫描所有消息，提取可搜索文本，建立快速查找结构。这个工作如果同步执行，会直接冻住 JavaScript 主线程，用户打开 transcript 的那几秒里什么都做不了。

`warmSearchIndex()` 的解法是分块处理 + 主动让出执行权。

## 具体机制

```typescript
async function warmSearchIndex(messages) {
  const chunks = chunk(messages, CHUNK_SIZE);
  for (const batch of chunks) {
    await sleep(0);  // 让出主线程
    for (const msg of batch) {
      extractAndCacheSearchText(msg);
    }
  }
}
```

`sleep(0)` 本质上是 `setTimeout(resolve, 0)`——让当前执行暂停，把控制权交还给事件循环，让其他待处理的任务（用户输入、滚动事件、渲染更新）有机会执行，然后再继续下一批预热。

## 为什么不用 Web Worker

Web Worker 是处理 CPU 密集任务的理想选择，可以完全不影响主线程。但它有代价：消息需要序列化传递，DOM 访问不可用，错误边界和取消逻辑都更复杂。

对于搜索预热这个任务——主要是遍历消息对象和提取文本——分块 + sleep(0) 是一个「足够好且简单」的方案。每批处理几十条消息，中间让出一次，主线程从未被连续占用超过一小段时间，用户感受到的是「偶尔有点微小延迟」，而不是「完全卡住了」。

## 预热是异步的，搜索仍然可以开始

分块预热带来的另一个好处是：用户不需要等预热全部完成才能开始搜索。即使缓存只建了一半，已经预热的部分可以立即被搜索利用；未预热的部分则在搜索时即时计算（慢一点，但不是不可用）。后台预热和前台搜索可以并行进行。
