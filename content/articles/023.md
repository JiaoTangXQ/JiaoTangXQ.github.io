# Auto Compact 的 context window 边界是怎么计算的？

当对话越来越长，Claude 的 context window 终将触顶。Claude Code 有一套 auto compact 系统，在 context 快满时自动压缩历史消息。但"快满"到底是什么数字？怎么在不丢内容的情况下留出足够空间？

`services/compact/autoCompact.ts` 里的代码给出了精确答案。

## 有效 context window 的计算

首先，"context window 大小"不是直接拿来用的。模型需要输出空间，这部分必须预留：

```typescript
const MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20_000

export function getEffectiveContextWindowSize(model: string): number {
  const reservedTokensForSummary = Math.min(
    getMaxOutputTokensForModel(model),
    MAX_OUTPUT_TOKENS_FOR_SUMMARY,
  )
  let contextWindow = getContextWindowForModel(model, getSdkBetas())

  // 支持环境变量覆盖（用于测试和调试）
  const autoCompactWindow = process.env.CLAUDE_CODE_AUTO_COMPACT_WINDOW
  if (autoCompactWindow) {
    const parsed = parseInt(autoCompactWindow, 10)
    if (!isNaN(parsed) && parsed > 0) {
      contextWindow = Math.min(contextWindow, parsed)
    }
  }

  return contextWindow - reservedTokensForSummary
}
```

`MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20_000` 的来源有明确说明：**基于 compact summary 输出的 p99.99 是 17,387 tokens**，向上取整到 20,000 作为预留。这是实际测量值，不是拍脑袋的数字。

有效 context window = 模型上限 - min(模型最大输出, 20,000)。

## 四个 token 阈值

```typescript
export const AUTOCOMPACT_BUFFER_TOKENS = 13_000    // 自动触发压缩的提前量
export const WARNING_THRESHOLD_BUFFER_TOKENS = 20_000  // 显示警告的阈值
export const ERROR_THRESHOLD_BUFFER_TOKENS = 20_000    // 显示错误的阈值
export const MANUAL_COMPACT_BUFFER_TOKENS = 3_000      // 手动压缩的提前量
```

这几个阈值的语义：

- **AUTOCOMPACT_BUFFER**（13,000）：当剩余空间 < 13,000 tokens 时，自动触发 compact。这个值相当保守，确保有足够空间完成当前轮次的工具调用
- **WARNING_THRESHOLD**（20,000）：显示"context 快满了"的黄色警告，让用户知道
- **ERROR_THRESHOLD**（20,000）：显示红色错误提示，情况更紧急
- **MANUAL_COMPACT_BUFFER**（3,000）：用户手动 `/compact` 时用的是更小的提前量，因为用户主动操作，风险由用户承担

## 连续失败熔断器

这是整个系统里最有工程价值的设计：

```typescript
export type AutoCompactTrackingState = {
  compacted: boolean
  turnCounter: number
  turnId: string
  consecutiveFailures?: number  // 连续失败次数，成功时重置
}

// 最多连续失败 3 次后停止重试
const MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3
```

注释里有一段真实的数据：

> BQ 2026-03-10：1,279 个会话有 50+ 次连续失败（最多 3,272 次），在单个会话内浪费约 250K API 调用/天

这说明在加熔断器之前，系统会对 context 已经无法压缩的情况（比如 `prompt_too_long` 错误）不断重试，造成巨大的资源浪费。加了熔断器后，连续失败 3 次就停止重试，直到有新的用户输入才重置。

这是典型的**熔断器模式（Circuit Breaker Pattern）**：检测到系统处于失败状态时，主动停止请求，避免雪崩。

## Token 预估的 fallback

`tokenCountWithEstimation` 的命名很有意思——包含"Estimation"意味着 token 计数有时是估算的，不是精确的。

实际上，精确 token 计数需要调用模型的 tokenizer，这有不可忽略的延迟。对于 auto compact 的触发判断，精确值不是必须的——用字符数估算 token 数（通常 1 token ≈ 4 个英文字符）在大多数情况下足够准确，误差在可接受范围内。估算失败顶多触发一次提前压缩，不会导致系统错误。

## 面试角度

这套设计体现了 LLM 应用工程的几个重要实践：

1. **用实测数据驱动阈值**：`p99.99 = 17,387 tokens` 是真实数据，不是猜测
2. **为边界情况留缓冲**：不是等到 100% 再处理，而是提前 13,000 token 就触发
3. **熔断器必须有重置条件**：`consecutiveFailures` 在有新输入时重置，不然熔断了就再也不 compact
4. **环境变量允许测试覆盖**：`CLAUDE_CODE_AUTO_COMPACT_WINDOW` 让集成测试可以用小 context window 触发压缩，而不需要真的撑满 200K tokens
