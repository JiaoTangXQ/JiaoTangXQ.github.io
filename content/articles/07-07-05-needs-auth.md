# 串行写needs-auth缓存说明总线也怕并发自相踩踏

## 实现链
- `client.ts` 里对 `mcp-needs-auth-cache.json` 的读写不是随手 `writeFile` 一把，而是通过串行化的缓存 promise 链去更新。
- 原因很现实：多个 server 几乎同时失败时，如果每个异步分支都独立读旧文件、改一点、再写回去，最后很容易互相覆盖。

## 普通做法
- 更普通的做法是每次状态变化就各写各的 JSON，默认操作系统文件写入能帮你兜住。
- 在低并发下它看起来也经常“没问题”。

## 为什么不用
- Claude Code 没赌这个运气，因为这里写的是全局共享缓存，不是单个请求的临时文件。并发覆盖一旦发生，系统会错误忘掉某些 server 的 `needs-auth` 状态。
- 它选择把写入排成单列，本质上是在承认“共享缓存就是会踩踏”。

## 代价
- 代价是实现更绕，而且串行更新在极端情况下会比乱写慢一点。
- 但这种慢换来的是状态正确性；对平台总线来说，这比追求写入瞬时吞吐更重要。
