# 分组和折叠是在替人压缩注意力

Claude Code 会折叠后台 bash 通知、hook 摘要、read/search 结果和 teammate 关闭消息。这不是为了省几行屏幕, 而是在承认一个事实: 长会话里真正稀缺的不是数据, 而是人的注意力。

## 实现链

`Messages.tsx` 在 `normalizeMessages()` 和 `reorderMessagesInUI()` 之后, 会继续跑 `applyGrouping()`、`collapseBackgroundBashNotifications()`、`collapseHookSummaries()`、`collapseReadSearchGroups()`、`collapseTeammateShutdowns()`。也就是说, 这套界面不是逐行决定“这条要不要折”, 而是先把整段消息链看完, 再把明显属于同一动作簇、同一摘要簇、同一噪声簇的东西压成更短的可读结构。

## 普通做法

更普通的做法是把所有事件原样列出来, 或者只在每一行里放一个“展开/收起”按钮, 让用户自己承担第一轮筛选工作。

## 为什么不用

Claude Code 不这么做, 是因为它面对的不是十几行聊天记录, 而是会不停长大的工作流水。要是每一种后台通知、hook 成功、read/search 明细都和主消息抢同等位置, 用户很快就看不见真正推进工作的那几行。这里先做分组和折叠, 等于系统先替你做第一轮“哪些值得占眼前位置”的判断。

## 代价

代价是消息区会带上明显的“政策味”。有些细节不会第一眼就露出来, 需要进 transcript、verbose 或展开视图才能看到原始形状。也就是说, 它换来了可读性, 但牺牲了“屏幕就是原始账本”的直观感。
