# 工程细节：翻页加载历史时，如何保持滚动位置不被打飞？

## 无限向上滚动的经典问题

"无限向上滚动加载历史"是 Web 消息应用里的常见功能。聊天应用、邮件客户端、社交 feed 都需要这个功能。

看起来简单：当用户滚动到列表顶部，加载更多历史消息，前插到列表里。

但这会产生一个 UI 问题：**前插内容会把所有现有内容向下推**，导致用户正在看的内容的位置发生变化。

假设用户在读消息 M100（当前视口的位置），这时加载了 20 条更早的历史（M80 到 M99），这 20 条消息被插入到列表最前面。M100 往下移动了"20 条消息的高度"，用户的视口停留在原来的像素位置，看到的内容变成了 M86 或 M87——视口"跳上去了"。

用户的感受：看着看着，内容突然往下跳了。

## layout effect 的时序优势

普通的 `useEffect` 在浏览器绘制之后执行。如果在这里补偿滚动位置，执行时浏览器已经渲染了新内容（包含位移），然后才调整滚动位置——用户会看到一个瞬间的"跳跃然后复位"的闪烁。

`useLayoutEffect` 在 DOM 更新之后、浏览器绘制之前执行。在浏览器还没有把新内容渲染到屏幕的时候，就先调整好滚动位置——用户根本看不到跳跃，新内容的出现是无感的。

`useAssistantHistory.ts` 使用这个时序：

```typescript
// 简化的实现思路
const prevScrollHeight = useRef(0)

// 在内容被前插之前，记录当前的 scrollHeight
beforeInsert(() => {
  prevScrollHeight.current = container.scrollHeight
})

// 使用 layout effect，在 DOM 更新后、绘制前执行
useLayoutEffect(() => {
  const newScrollHeight = container.scrollHeight
  const heightAdded = newScrollHeight - prevScrollHeight.current
  // 把滚动位置向下补偿"新增内容的高度"
  container.scrollTop += heightAdded
}, [messages])
```

这里的关键：`scrollTop` 的变化量 = 新增内容的高度（`newScrollHeight - prevScrollHeight`）。无论新增了多少内容，视口始终停在用户原来正在读的那条消息上。

## 为什么不整页重刷？

最简单的实现：加载历史后，把整个消息列表重新渲染，滚动条回到顶部。

这在技术上最容易实现，但用户体验很差：

- 用户当前的阅读位置丢失
- 如果历史很长，渲染时间长，有明显的白屏或 loading 状态
- 每次翻页都是一次完整的"重新开始"，不是"继续看"

对于 viewerOnly 这个场景——用户在看一个长时间运行的会话，可能需要往回翻几十条消息——保持阅读位置是核心体验需求，不是可选的优化。

## 前插的顺序问题

除了滚动位置，前插还有一个顺序问题：历史消息需要被插入到列表的最前面，但保持它们自己内部的时间顺序。

API 返回的历史消息通常是倒序的（最新的先返回），或者需要按时间排序。前插时需要先把新页的消息按正确顺序排列好，然后整体插入到列表最前面。

如果顺序弄错，用户会看到"最新的消息在上面，最旧的在下面"这样反直觉的排列。

## convertSDKMessage 的角色

历史消息从 API 返回的是 `SDKMessage` 格式，需要通过 `convertSDKMessage()` 转成本地 `Message` 格式，才能进入同一套渲染管线。

这和实时消息的处理路径一致：实时流收到 SDKMessage → 转成 Message → 渲染。历史消息：API 返回 SDKMessage → 转成 Message → 前插 → 渲染。

两条路径使用同一个转换函数，保证了"历史消息"和"实时消息"在渲染时外观一致——用户看不出哪条是从历史加载的，哪条是实时收到的。

## 面试指导

"无限滚动列表的性能和 UX 优化"是前端面试里的高频题。

滚动位置保持问题的解决方案：

1. **方案 A**：虚拟列表（virtual scrolling）——只渲染视口内的元素，前插不会影响已渲染元素。但实现复杂，对动态高度的消息处理困难。

2. **方案 B**：layout effect 补偿——简单、可靠，对大多数场景足够。代价是在内容量很大时有性能压力（因为实际渲染了所有消息）。

3. **方案 C**：CSS `overflow-anchor`——浏览器原生支持的滚动锚定，通过 CSS 属性声明。但浏览器支持不一致，在某些场景下行为不可预测。

`useAssistantHistory.ts` 选择方案 B，因为对话消息数量通常是几百条级别，layout effect 的实现简单且可靠，虚拟列表的实现复杂度是不必要的。

在面试中，能说清楚"不同方案的适用场景和代价"，比直接给出一个方案更有说服力。
