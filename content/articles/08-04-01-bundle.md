# 工程取舍：Bridge core 为什么用依赖注入而不是直接 import？

## 一个注释里的 1300 个模块

`bridgeApi.ts` 里有一个注释，值得仔细读：

```typescript
type BridgeApiDeps = {
  baseUrl: string
  getAccessToken: () => string | undefined
  runnerVersion: string
  onDebug?: (msg: string) => void
  /**
   * Called on 401 to attempt OAuth token refresh. Returns true if refreshed,
   * in which case the request is retried once. Injected because
   * handleOAuth401Error from utils/auth.ts transitively pulls in config.ts →
   * file.ts → permissions/filesystem.ts → sessionStorage.ts → commands.ts
   * (~1300 modules). Daemon callers using env-var tokens omit this — their
   * tokens don't refresh, so 401 goes straight to BridgeFatalError.
   */
  onAuth401?: (staleAccessToken: string) => Promise<boolean>
}
```

不直接 import `handleOAuth401Error` 的理由：它会传递拉入约 1300 个模块。

这不是夸大。在 Node.js 的静态分析视角，`import A` 意味着 A 的所有 `import` 都会被带进来，递归直到叶子模块。`commands.ts` 是命令注册中心，几乎 import 了所有 REPL 命令，而这些命令又 import 了 React 组件、Ink UI、各种 hook……整棵 REPL 的依赖树。

## Bundle 边界的工程含义

Bridge core 需要跑在三种入口：

1. **REPL 入口**：用户运行 `claude` 命令，完整的本地工作台
2. **Daemon 入口**：后台服务进程，没有 UI，使用 env-var token
3. **SDK 入口**：第三方集成，轻量 bundle，不需要完整命令系统

如果 bridge core 直接 import `commands.ts`，那么：
- Daemon bundle 里会包含完整的 React UI
- SDK bundle 会包含只有 REPL 用得到的权限系统

这不只是"bundle 太大"的性能问题，而是**入口边界失真**——daemon 进程包含了它永远不会执行的 UI 代码，SDK 包含了只在交互模式下有意义的组件。

## 注入的设计

`createBridgeApiClient(deps: BridgeApiDeps)` 采用工厂函数 + 依赖注入：

```typescript
export function createBridgeApiClient(deps: BridgeApiDeps): BridgeApiClient {
  // 所有行为都通过 deps 参数提供
  // 内部不 import 任何业务模块
}
```

调用方（REPL）注入完整实现：

```typescript
createBridgeApiClient({
  getAccessToken: getBridgeAccessToken,
  onAuth401: handleOAuth401Error,  // REPL 才有这个
  getTrustedDeviceToken: getTrustedDeviceToken,
})
```

调用方（Daemon）省略不需要的依赖：

```typescript
createBridgeApiClient({
  getAccessToken: () => process.env.BRIDGE_TOKEN,
  // onAuth401 省略 — daemon 用 env-var token，不支持刷新
  // 401 直接抛 BridgeFatalError
})
```

每个调用方只带入它需要的依赖，bridge core 本身对 REPL 专有模块一无所知。

## 可信设备 token 的条件注入

同一个模式在 `getTrustedDeviceToken` 上也体现了：

```typescript
/**
 * Returns the trusted device token to send as X-Trusted-Device-Token.
 * Optional — when absent or returning undefined, the header is omitted.
 */
getTrustedDeviceToken?: () => string | undefined
```

当 CCR v2 的服务端安全级别要求受信设备时，bridge 需要在每个请求里附上设备 token。但不是所有环境都有设备 token 的概念——环境直接访问的 bridge 就没有。

把它设计成可选注入，让 bridge core 不必知道"我现在跑在哪种环境里"，调用方自己决定要不要提供。

## 与 mapper 注入的相似模式

`EnvLessBridgeParams` 里的 `toSDKMessages` 注入是同一个模式：

```typescript
/**
 * Converts internal Message[] → SDKMessage[] for writeMessages() and the
 * initial-flush/drain paths. Injected rather than imported — mappers.ts
 * transitively pulls in src/commands.ts (entire command registry + React tree)
 * which would bloat bundles that don't already have it.
 */
toSDKMessages: (messages: Message[]) => SDKMessage[]
```

消息格式转换函数通过参数注入，而不是直接 import。原因一样：`mappers.ts` 会拉入整棵命令树。

## 面试指导

"为什么用依赖注入"是面试中很常见的问题，但大多数人的回答停留在"方便测试"。

更深层的理由是**bundle 边界控制**——在有多个打包入口的系统里，直接 import 会导致依赖扩散到所有入口，即使某个入口根本用不到这个依赖。依赖注入让每个入口只带入它真正需要的依赖。

这个理由在前端生态里很常见（tree-shaking 的前提是依赖可以被静态分析），在 Node.js 后端里不那么常提，但同样重要——尤其是当一个核心模块需要同时服务多个用途时。
