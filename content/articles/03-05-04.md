# 面试题：从用户视角看命令都一样，从代码视角看执行路径完全不同，这是设计缺陷吗？

## 面试官挑战

「你之前说 `/clear`、`/config`、`/review` 从用户视角看都是斜杠命令，但实现完全不同——`/clear` 同步本地执行，`/config` 等待 UI 交互，`/review` 发 API 请求。这不是破坏了统一性吗？」

## 两种统一

这个问题混淆了两种不同的统一：

**入口统一（用户界面层面的统一）**：用户只需要学一种语法——`/命令名 [参数]`。不管命令背后是本地程序、UI 交互还是模型推理，输入方式是一致的。

**机制统一（执行层面的统一）**：所有命令走同一条执行路径，产生同样类型的效果。

Claude Code 追求前者，**有意拒绝后者**。

## 为什么机制统一是个坏主意

想象一个强制机制统一的系统——所有命令都走 `prompt` 模型，最终都变成一轮模型推理：

```
用户: /clear
模型: 好的，我现在要清空上下文。[清空操作]
```

这里有几个问题：

1. **不确定性**：模型输出有随机性。`/clear` 的结果应该是确定的——清空就是清空，不需要模型「决定」。
2. **延迟**：每次 `/clear` 都要等一轮 API 请求，明显变慢。
3. **成本**：每次 `/clear` 消耗 token，完全没必要。
4. **格式可靠性**：`/compact` 需要精确控制消息历史的格式，模型不能保证这一点。

如果强制走 `local` 模型：

```
/review → 本地程序执行 → 输出字符串
```

那 `/review` 就只能是一个固定的脚本，不能利用模型的推理能力，失去了作为 AI 工具的核心价值。

## 统一外观需要什么

入口统一所需要的，只是：

1. **统一的触发语法**：`/命令名 [参数]`
2. **统一的 Command 类型**：所有命令都有名字、描述、可见性控制
3. **统一的入口处理**：`parseSlashCommand`、`findCommand`、`hasCommand` 对所有命令一视同仁

这三件事 Claude Code 都做到了。执行层的差异（`switch (command.type)`）被封装在 `getMessagesForSlashCommand` 里，外层（`processSlashCommand`、`processUserInput`）不感知。

## 统一外观的真正价值

用户只需要记住一件事：「如果想做什么，就试试输入 `/` 加上你想做的事」。

这是 discoverability（可发现性）。用户不需要知道 `/clear` 和 `/review` 的实现完全不同，他们只需要知道「打斜杠就能找到相关命令」。

这种可发现性来自统一外观，不依赖统一机制。

## 判断标准

什么时候该统一，什么时候该保留差异？

**统一**：当统一带来的好处（用户学习成本降低、入口逻辑简化）超过成本（需要为某些命令强加不适合的约束）。

**保留差异**：当某个执行机制是该类命令正确工作的必要条件，而不只是实现细节时。

`/compact` 正确工作需要程序化的确定性 → `local` 是必要条件，不是可选的。
`/review` 正确工作需要模型推理 → `prompt` 是必要条件，不是可选的。
`/config` 正确工作需要 UI 交互 → `local-jsx` 是必要条件，不是可选的。

统一外观服务于用户，分离机制服务于系统正确性。两个目标不冲突，各自在正确的层面被满足。

---

*面试指导：被问到「你们系统的 X 在不同地方行为不同，这是设计问题吗」时，先区分「外部行为一致性」和「内部实现统一」，再说明哪个层面需要一致、哪个层面保留差异更正确。这种分层思维比直接辩护「没问题」或承认「有缺陷」更有说服力。*
