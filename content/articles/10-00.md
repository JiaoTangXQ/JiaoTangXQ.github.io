# 200K token 快溢出了，Claude Code 的四级压缩体系是怎么设计的？

这是一道关于 LLM Agent 工程的系统设计题。面试官不只在问你"如何压缩上下文"，更在考察你对长会话连续性这个根本矛盾的理解：**越成功的会话，积累越多，越接近崩溃边缘**。

---

## 问题背景

Claude Code 的上下文窗口最大约 200K token。一个认真工作的长会话——读文件、跑工具、规划、写代码——每轮可以轻松消耗数千 token。跑上一两个小时，窗口就满了。

这时候你有两个极端选择：
1. 硬撑——买更大的窗口，等到爆再说
2. 摘要——爆了就总结一下，从头开始

Claude Code 都没选。它的回答是：**连续性管理是一种时间制度，不是一次性的善后动作**。

---

## 五层架构概览

从 `query.ts` 的主循环可以看到，每次进入下一轮 API 调用之前，消息流会依次经过：

```
原始消息流
  → applyToolResultBudget（单条工具结果预算）
  → snipCompact（HISTORY_SNIP，裁尾）
  → microcompact（局部工具结果清除）
  → contextCollapse（CONTEXT_COLLAPSE，语义折叠）
  → shouldAutoCompact → compactConversation（系统级重写）
  → API 调用
```

这五个环节不是备选方案，而是从轻到重的串联过滤器。越靠前的机制越轻量、越局部；越靠后的越重、越整体。

压缩完成后，外置记忆系统（session memory、auto-memory）在回合结束后的 stop hooks 里异步写入，作为第六层的长期存储。

---

## 第一级：snip — 裁掉历史尾巴

`HISTORY_SNIP` feature flag 控制的 `snipCompact` 是最轻量的机制。它不做摘要，只是把超出一定回溯深度的老轮次整个裁掉，返回被裁掉的 token 数 `snipTokensFreed`。

这个数值非常关键——它会被直接传给 `shouldAutoCompact()`，用于修正 token 估算。因为 snip 把老的 assistant message 删掉了，但留下来的最新 assistant message 里的 `usage.input_tokens` 仍然反映着 pre-snip 的上下文大小。如果不减掉这个差值，autocompact 会虚高估计当前上下文，误触发一次不必要的系统级压缩。

```typescript
// query.ts ~L403
let snipTokensFreed = 0
if (feature('HISTORY_SNIP')) {
  const snipResult = snipModule!.snipCompactIfNeeded(messagesForQuery)
  messagesForQuery = snipResult.messages
  snipTokensFreed = snipResult.tokensFreed
}
```

---

## 第二级：microcompact — 清除工具结果，不动对话

microcompact 的目标是精准的。它只清除已经用完、不再需要的工具调用结果（file read、bash、grep、glob、web_fetch 等），不碰用户消息、不碰 assistant 的思考块，不生成摘要。

从 `microCompact.ts` 可以看到，可清除工具集合是静态枚举的：

```typescript
const COMPACTABLE_TOOLS = new Set<string>([
  FILE_READ_TOOL_NAME,
  ...SHELL_TOOL_NAMES,
  GREP_TOOL_NAME, GLOB_TOOL_NAME,
  WEB_SEARCH_TOOL_NAME, WEB_FETCH_TOOL_NAME,
  FILE_EDIT_TOOL_NAME, FILE_WRITE_TOOL_NAME,
])
```

Ant 内部版本还有 `CACHED_MICROCOMPACT`——通过 cache editing API 直接在服务端删除已缓存的工具结果，不用让这些内容参与 prompt cache rewrite，更省钱。

microcompact 有一个时间触发条件：如果距离上次 assistant 消息超过阈值（缓存可能已失效），会先做一次 time-based microcompact 清除，然后跳过 cached 路径（因为服务端缓存已经是冷的了）。

---

## 第三级：autocompact — 系统级熔断

当 microcompact 之后 token 数仍然超过阈值，`shouldAutoCompact()` 会返回 true，触发 `compactConversation()`——用一个 fork 出来的子 agent 对整段历史生成摘要，然后用 `buildPostCompactMessages()` 重组消息流。

阈值计算方式（`autoCompact.ts`）：

```
effectiveContextWindow = contextWindow - reservedForSummaryOutput(20K)
autoCompactThreshold = effectiveContextWindow - AUTOCOMPACT_BUFFER_TOKENS(13K)
```

为防止"自相残杀"（compact agent 自身也触发 autocompact），`shouldAutoCompact()` 对 `querySource === 'session_memory'` 和 `querySource === 'compact'` 直接返回 false。

还有熔断机制：连续失败 3 次后停止重试：
```typescript
const MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3
```

---

## 第四级：手动 compact — 主导权保留给用户

`/compact` 命令走 `commands/compact/compact.ts`。它先尝试 session memory 路径（轻量），再跑 microcompact，最后调 `compactConversation()`。用户可以追加自定义指令，告诉系统"保留哪些内容"、"按什么方式摘要"。

---

## 压缩后重组：compact boundary 不是删除线

`buildPostCompactMessages()` 返回的不是一个光秃秃的摘要，而是有结构的重组：

```typescript
export function buildPostCompactMessages(result: CompactionResult): Message[] {
  return [
    result.boundaryMarker,       // compact 边界标记
    ...result.summaryMessages,   // 摘要内容
    ...(result.messagesToKeep ?? []),  // 保留的原始消息片段
    ...result.attachments,       // 重注入：文件、plan、skill、工具列表
    ...result.hookResults,       // session start hooks 结果
  ]
}
```

文件 cache、plan、技能附件都会在 compact 后重新注入，确保模型在新窗口里仍然能"找到"它之前用过的东西。

---

## 外置记忆：stop hooks 里的异步后台任务

回合结束后，`handleStopHooks()` 异步触发：
- `executeExtractMemories`：auto-memory 长期记忆提炼
- `executeAutoDream`：自动反思/规划
- `executePromptSuggestion`：提示词建议

这些任务不参与当前回合的 token 预算，不阻塞用户响应，但它们把"值得长期记住的知识"外置到 `~/.claude/projects/<path>/memory/` 文件系统，作为跨会话的持久状态。

---

## 面试考察点

**系统设计角度**：五级机制从局部到全局，从轻到重。这是"多防线"的工程思维，每一层尽量不惊动更重的机制。

**边界条件**：task_budget 跨 compact 的延续（query.ts 用 `taskBudgetRemaining` 累减），snipTokensFreed 修正 autocompact 误判，compact agent 的递归保护——这些都是细节，但都是真实 bug 的防御。

**异步解耦**：把整理动作推到 stop hooks 而不是插入主回合，这是"先让主线程落稳，再做后台整理"的架构选择。

**记忆分层**：session memory（当前工作台状态）vs auto-memory（项目长期知识），两种不同的粒度和触发时机。

这整套设计的核心哲学：**连续性是运行时的时间制度，不是偶尔需要的善后工具**。
