# 第十卷导读：连续性不是善后，而是工作台的时间制度

## 实现链

这一页重点看“第十卷导读-连续性不是善后，而是工作台的时间制度”。对应源码主要是 `src/query.ts`、`src/services/compact/microCompact.ts`、`src/services/compact/autoCompact.ts`、`src/services/compact/compact.ts`、`src/services/SessionMemory/sessionMemory.ts`、`src/services/extractMemories/extractMemories.ts`、`src/query/stopHooks.ts`。
这一卷真正落在 `query.ts`、`microCompact.ts`、`autoCompact.ts`、`compact.ts`、`SessionMemory/sessionMemory.ts`、`extractMemories.ts`、`stopHooks.ts` 这些连续性模块上。Claude Code 不把“上下文太长了”当成单次事故，而是把减重、重组、外置记忆和回合后整理写成一整套时间制度。

## 普通做法

更常见的做法，是把连续性问题理解成 token 不够了再压一下，或者干脆靠更大上下文窗口硬扛。这样主线更简单，也不用在系统里长出那么多整理机制。

## 为什么不用

Claude Code 没这么做，因为长会话真正会积累的不只是 token 数，还有附件、工具结果、计划、技能和未来还要继续工作的线索。单纯“撑更大”或“最后再总结”都保不住这些接缝。

## 代价

这样做的好处是连续性真的进入了运行时骨架。代价是这整卷的代码看起来会比普通聊天系统复杂得多，因为整理时机、整理粒度和记忆边界都被拆开管理了。
