# 你要做一个 agent，第一步应该建哪一层？

## 面试综合题：第一步做什么

面试官问"如果让你从零设计一个代码助手 agent，你会怎么开始"。

最常见的答案是"先搞清楚需求，再设计工具列表，然后写主循环"。这个回答不算错，但它跳过了一个更根本的问题：这些东西分别由谁负责？

Claude Code 的代码里，这个问题的答案是一套反复坚持的四分法。

## 四分法：入口、行动、制度、显示

**入口层**解决的是"用户和系统怎么开始说话"。`commands.ts` 是入口层的典型。它定义了哪些命令可以被输入，每个命令的参数格式是什么，触发后交给谁处理。入口层不自己执行动作，只负责解析和分发。

**行动层**解决的是"系统能伸哪些手、这些手的协议是什么"。`Tool.ts` 和 `tools.ts` 是行动层。它们定义每个工具的输入格式、权限需求和执行方式。行动层不管任务如何组织，也不管结果如何显示。

**制度层**解决的是"哪些行动现在有资格发生"。`permissionSetup.ts`、settings 模块和 hooks 是制度层。它们在会话开始时就建立规则，不是等工具跑起来再拦。制度层不执行任何动作，只建立边界。

**显示层**解决的是"界面当前认定什么是真的"。`AppStateStore` 是显示层的核心。所有组件从这里读取状态，不自己推断，不订阅分散的数据源。

这四件事分开以后，模型能力才有稳定的落脚点——行动层能伸手，但只在制度层允许的范围内；显示层展示结果，但只读一份一致的事实。

## 为什么这套分工可迁移

这套四分法不只适用于 Claude Code，它适用于几乎所有需要长期工作的 agent 系统。

你可以用不同的技术栈实现：入口层可以是 CLI、HTTP API 或 WebSocket；行动层可以是 MCP 工具、函数调用或自定义执行环境；制度层可以是规则引擎、策略配置或 prompt 前置检查；显示层可以是 React 状态、事件流或日志系统。

变的是实现方式，不变的是"这四件事必须分开"这个原则。

## 为什么混在一起会出问题

更常见的失败写法，是把这四件事混在一个 agent 主循环里：

```
// 糟糕的模式
function handleMessage(input) {
  // 入口解析
  const command = parse(input)
  // 制度检查（但混在行动里）
  if (command.needsPermission) askUser()
  // 行动执行
  const result = executeTool(command)
  // 顺手改UI
  updateUI(result)
  // 顺手改状态
  updateState(result)
}
```

这样一开始写得快，因为任何新功能都能"顺手加进去"。但功能一多，就会出现：
- 制度检查分散在多个地方，容易漏掉
- UI 和状态更新耦合在行动里，测试和复用都困难
- 没有一层真正对"系统状态是否一致"负责

Claude Code 选择了更慢理解、但更抗增长的路：四层分开，靠明确的接口交接。

## 何时开始建制度层

一个常见的误判是"制度层可以等功能做稳了再加"。Claude Code 的代码给出了相反的答案：制度先行，能力后进场。

不是因为这样更容易（制度层让开发变慢），而是因为能力一旦变强，不先上制度就会先上事故。等出了问题再加制度，比一开始就有制度，成本要高很多。

## 面试指导

回答"如果你从零做一个 agent"时，先说四分法：把入口、行动、制度、显示分清楚，再往里填能力。

这个回答展示的不是技术细节，而是系统设计的成熟度——知道"分清楚谁管什么"比"先堆能力"更重要。这种判断力，比熟悉任何具体工具都更难培养，也更值钱。
