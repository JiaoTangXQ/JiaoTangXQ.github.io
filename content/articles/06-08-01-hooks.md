# hooks 快照和实时读取的区别：disableAllHooks 能禁用哪些 hooks？

`hooksConfigSnapshot.ts` 里的 `getHooksFromAllowedSources` 函数展示了 hooks 系统的权力结构，值得逐行分析。

## getHooksFromAllowedSources 的决策树

```typescript
function getHooksFromAllowedSources(): HooksSettings {
  const policySettings = settingsModule.getSettingsForSource('policySettings')

  // 1. 管理员完全禁用所有 hooks（包括管理员自己的）
  if (policySettings?.disableAllHooks === true) {
    return {}
  }

  // 2. 管理员锁定：只用管理员的 hooks
  if (policySettings?.allowManagedHooksOnly === true) {
    return policySettings.hooks ?? {}
  }

  // 3. strictPluginOnlyCustomization：只允许插件来源的 hooks
  if (isRestrictedToPluginOnly('hooks')) {
    return policySettings?.hooks ?? {}
  }

  const mergedSettings = settingsModule.getSettings_DEPRECATED()

  // 4. 用户/项目级 disableAllHooks：只能禁用非管理员 hooks
  if (mergedSettings.disableAllHooks === true) {
    return policySettings?.hooks ?? {}  // 管理员 hooks 仍然运行
  }

  // 5. 普通情况：所有来源的 hooks 合并
  return mergedSettings.hooks ?? {}
}
```

这个决策树里有一个非对称性：**`disableAllHooks` 在不同地方有不同的效力**。

- **policySettings 里的 `disableAllHooks`**：禁用**所有** hooks，包括管理员自己配置的
- **非 policySettings 里的 `disableAllHooks`**（用户设置、项目设置）：只能禁用非管理员 hooks，管理员 hooks 仍然运行

这个设计的逻辑是：用户说”我不想要任何 hooks 打扰我”，但管理员配置的 hooks 可能是安全审计、数据保护等合规要求，不能因为用户偏好而绕过。

## 为什么管理员 hooks 不能被用户禁用

注释里解释了这个不对称性的来源：

```typescript
// If disableAllHooks is set in non-managed settings, only managed hooks still run
// (non-managed settings cannot override managed hooks)
if (mergedSettings.disableAllHooks === true) {
  return policySettings?.hooks ?? {}
}
```

非管理员 settings 能控制用户自己配置的 hooks，但不能控制管理员层面的 hooks。这是权力边界，不是 bug。

类比：普通员工可以关掉自己电脑上的广告拦截插件，但不能关掉 IT 部门安装的企业安全监控软件。

## 快照 vs 实时读取

`captureHooksConfigSnapshot()` 在会话初始化时调用一次，把 hooks 配置冻结在 `initialHooksConfig` 变量里：

```typescript
let initialHooksConfig: HooksSettings | null = null

export function captureHooksConfigSnapshot(): void {
  initialHooksConfig = getHooksFromAllowedSources()
}

export function getHooksConfigFromSnapshot(): HooksSettings | null {
  if (initialHooksConfig === null) {
    captureHooksConfigSnapshot()  // 懒初始化
  }
  return initialHooksConfig
}
```

`updateHooksConfigSnapshot()` 在 settings 变化时被 `applySettingsChange` 调用，更新快照：

```typescript
export function updateHooksConfigSnapshot(): void {
  resetSettingsCache()  // 先清缓存，确保读到新鲜数据
  initialHooksConfig = getHooksFromAllowedSources()
}
```

注意 `resetSettingsCache()` 的调用——注释解释了为什么需要它：

> “Without this, the snapshot could use stale cached settings when the user edits settings.json externally and then runs /hooks - the session cache may not have been invalidated yet”

用户通过 `/hooks` 命令手动触发 updateHooksConfigSnapshot 时，如果 settings 文件刚刚被修改但 changeDetector 的 1 秒稳定窗口还没过，settings 缓存可能是旧的。所以 updateHooksConfigSnapshot 里要手动清一次缓存，确保读到磁盘上的新状态。

## resetHooksConfigSnapshot 和 SDK 初始状态

```typescript
export function resetHooksConfigSnapshot(): void {
  initialHooksConfig = null
  resetSdkInitState()  // 同时重置 SDK 初始化状态
}
```

重置 hooks 快照时，同时重置 SDK 初始化状态（`resetSdkInitState`）。这两个状态耦合的原因是：SDK 模式下，hooks 配置是初始化序列的一部分；如果 hooks 快照被重置（比如测试里），SDK 的初始化状态也需要同步重置，否则 SDK 会误以为已经初始化好了，跳过重新读取 hooks 的步骤。

## shouldAllowManagedHooksOnly 的双重判断

```typescript
export function shouldAllowManagedHooksOnly(): boolean {
  const policySettings = settingsModule.getSettingsForSource('policySettings')
  if (policySettings?.allowManagedHooksOnly === true) {
    return true
  }
  // 用户级 disableAllHooks 等价于管理员 hooks only
  if (
    settingsModule.getSettings_DEPRECATED().disableAllHooks === true &&
    policySettings?.disableAllHooks !== true  // 排除管理员完全禁用的情况
  ) {
    return true
  }
  return false
}
```

这个函数的第二个判断很有意思：当用户设置了 `disableAllHooks: true`（不是管理员设置的），系统认为这等价于”只允许管理员 hooks”——因为管理员 hooks 是不受用户控制的，不管用户怎么设置都会运行。

## 这道题考什么

**权力非对称性**：能否解释”为什么用户的 `disableAllHooks` 不能禁用管理员 hooks”，这是理解 policySettings 权威地位的核心。

**快照 + 手动刷新的必要性**：`updateHooksConfigSnapshot` 里的 `resetSettingsCache()` 处理了 `/hooks` 命令直接调用时的缓存过期问题，不同于 `applySettingsChange` 路径（那条路上 fanOut 已经清了缓存）。能区分两条更新路径的候选人，对 settings 系统有系统性理解。
