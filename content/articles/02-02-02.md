# claude ssh、cc:// 和 assistant 为什么在 main() 最开头就被识别？

`main()` 函数开头有一大段代码，专门在 Commander 解析之前处理几种特殊入口。每一种都有类似的模式：识别特征参数，从 argv 里提取信息，然后重写 `process.argv`。

这个设计看起来有点奇怪——为什么不直接在 Commander 里注册这些子命令？

## 三种特殊入口及其处理方式

### 1. `cc://` 协议 URL

```typescript
const ccIdx = rawCliArgs.findIndex(a => a.startsWith('cc://') || a.startsWith('cc+unix://'));
if (ccIdx !== -1 && _pendingConnect) {
  const ccUrl = rawCliArgs[ccIdx]!;
  const { parseConnectUrl } = await import('./server/parseConnectUrl.js');
  const parsed = parseConnectUrl(ccUrl);
  _pendingConnect.url = parsed.serverUrl;
  _pendingConnect.authToken = parsed.authToken;
  // 从 argv 里删掉这个 URL
  const stripped = rawCliArgs.filter((_, i) => i !== ccIdx);
  process.argv = [process.argv[0]!, process.argv[1]!, ...stripped];
}
```

处理方式：把 URL 解析结果存入 `_pendingConnect` 对象，然后把 URL 从 argv 里删掉。Commander 不会看到这个 URL，但后续代码（REPL 启动前）会读取 `_pendingConnect`。

### 2. `claude ssh <host> [dir]`

```typescript
if (rawCliArgs[0] === 'ssh' && rawCliArgs[1] && !rawCliArgs[1].startsWith('-')) {
  _pendingSSH.host = rawCliArgs[1];
  if (rawCliArgs[2] && !rawCliArgs[2].startsWith('-')) {
    _pendingSSH.cwd = rawCliArgs[2];
  }
  // 重写 argv，去掉 ssh 子命令
  process.argv = [process.argv[0]!, process.argv[1]!, ...rest];
}
```

注意：`claude ssh` 还需要提前解析几个特有的 flag——`--permission-mode`、`--dangerously-skip-permissions`、`--local`，以及需要转发到远端 CLI 的参数（`--resume`、`--continue`、`--model`）。这些 flag 都是 ssh 专用的，Commander 不认识，必须手动提取。

### 3. `claude assistant [sessionId]`

```typescript
if (rawArgs[0] === 'assistant') {
  const nextArg = rawArgs[1];
  if (nextArg && !nextArg.startsWith('-')) {
    _pendingAssistantChat.sessionId = nextArg;
    rawArgs.splice(0, 2);
  }
  process.argv = [process.argv[0]!, process.argv[1]!, ...rawArgs];
}
```

## 为什么不在 Commander 里注册子命令？

表面看，这三种入口都可以用 Commander 的子命令机制处理：

```typescript
program.command('ssh <host> [dir]').action(async (host, dir, options) => { ... })
program.command('open <url>').action(async (url) => { ... })
```

但这样做有几个结构性问题：

**问题一：会话模式影响整条主链**

`claude ssh` 不是一个独立的子命令——它是"用 SSH 后端替换本地文件系统后端，然后启动完整 REPL"。`cc://` URL 也类似——它是"连接到指定的 direct connect server，然后启动 REPL"。

如果作为子命令实现，就会出现两套初始化路径：普通 REPL 路径和 SSH/cc:// 路径。而实际上这两条路径在 setup、trust、REPL 启动等阶段是高度共享的，只是会话后端不同。

共享的正确方式是：在同一条主链里传入"这次用什么后端"的参数，而不是两套独立路径。

**问题二：headless 模式需要不同处理**

`cc://` URL 在 headless（`-p`）模式下需要重写为 `open` 子命令：

```typescript
if (rawCliArgs.includes('-p') || rawCliArgs.includes('--print')) {
  // Headless: rewrite to internal `open` subcommand
  process.argv = [process.argv[0]!, process.argv[1]!, 'open', ccUrl, ...stripped];
} else {
  // Interactive: strip cc:// URL and flags, run main command
  _pendingConnect.url = parsed.serverUrl;
}
```

这个分支必须在最早期完成，因为 `isNonInteractive` 的判断（影响 telemetry、trust 等）依赖于 argv 的最终形态。

**问题三：早期 argv 重写是有状态的**

多个特殊入口的 argv 处理是有顺序的，而且会相互影响（例如，ssh 的 flag 提取必须在 ssh 模式识别后完成）。如果通过 Commander 的命令分发来处理，这些顺序依赖难以表达。

## _pending 模式的优雅之处

每种特殊入口对应一个 pending 对象：

```typescript
const _pendingConnect: PendingConnect | undefined
const _pendingAssistantChat: PendingAssistantChat | undefined  
const _pendingSSH: PendingSSH | undefined
```

这些对象在 `main()` 最开头被初始化（仅当对应 feature flag 开启时），在 argv 解析阶段被填充，在后续的 REPL 启动逻辑里被消费。

它相当于一个"意图暂存区"：识别到了特殊入口，但不立即执行，先把意图记下来，等整条主链走到合适的时机再拿出来用。

## 面试指导

这道题考查的是对"框架集成的边界"的理解。

Commander（或 argparse、click 等 CLI 框架）是很好的工具，但它有一个隐含假设：所有参数都在框架解析之后才被消费。

有些参数打破了这个假设——它们需要在框架解析之前就改变系统状态（设置来源）、重写参数（特殊入口），或者影响框架本身的初始化（比如设置 commander 是否严格模式）。

对于这类参数，正确的做法不是强行把它们塞进框架，而是在框架之前用简单的手动解析处理，然后让框架接手剩余的参数。能识别这个边界，是架构设计能力的体现。
