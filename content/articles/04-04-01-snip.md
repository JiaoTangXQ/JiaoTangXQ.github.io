# 为什么工具结果预算和 snip 要先于真正的压缩

很多人一听“上下文太长”，第一反应就是做摘要。Claude Code 的顺序更讲究。它会先做两件更轻的事：限制工具结果体积，以及 `snip` 这种局部裁剪。

这背后的判断很成熟。因为很多上下文压力，并不是“整段对话已经老到该总结了”，而只是最近某些工具结果太胖、某些局部片段太臃肿。如果这种时候直接上重型压缩，就等于还没分清病因，先做大手术。Claude Code 选择先减掉最显眼、最便宜、最少破坏连续性的那一层重量。

这就是为什么工具结果预算会先跑，`snip` 也会在 `microcompact` 和 `autocompact` 之前触发。它们像主循环里的前线止血，不是为了漂亮总结，而是为了尽量避免后面更重的上下文重组。

所以这一步的优雅不在“压得多高级”，而在“先问有没有更小的修复”。成熟系统很少一上来就动大刀。

代码上，`query()` 在进入更重的 compact 逻辑前，会先扣掉 snip 已经释放的 token，并让 `toolResultStorage` 这一层优先处理那些体积大、信息密度低的结果块。更常见的做法是：只要 token 超标，就把最近几轮消息整体总结掉，因为那条路径最统一、最省脑子。问题是这样会把真正有推理价值的近因也一起磨平。

Claude Code 先打工具结果的主意，是因为这些内容往往最臃肿、最接近“原始回显”，删减它们对会话逻辑的伤害最小。代价也有：系统必须记住哪些 token 是 snip 释放的、哪些是 compact 真释放的，否则恢复和 `/resume` 会出现“显示已经变短，实际统计还按旧长度算”的一致性问题。

## 实现链
先在 `toolResultStorage` 和 token 预算层处理最臃肿的工具回显，再考虑整体压缩。

## 普通做法
一超限就直接总结最近几轮。

## 为什么不用
因为工具回显最胖，且删它们对推理伤害通常最小。

## 代价
必须维护 snip 与 compact 的一致性统计。
