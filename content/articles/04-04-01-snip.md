# 边界条件分析：工具结果预算和 snip 为什么要在压缩之前跑？

`query()` 内部有一段顺序值得仔细看：

```typescript
// 1. 工具结果预算
messagesForQuery = await applyToolResultBudget(messagesForQuery, ...)

// 2. snip（局部裁剪）
if (feature('HISTORY_SNIP')) {
  const snipResult = snipModule!.snipCompactIfNeeded(messagesForQuery)
  messagesForQuery = snipResult.messages
  snipTokensFreed = snipResult.tokensFreed
}

// 3. microcompact
const microcompactResult = await deps.microcompact(messagesForQuery, ...)

// 4. context collapse
if (feature('CONTEXT_COLLAPSE') && contextCollapse) {
  const collapseResult = await contextCollapse.applyCollapsesIfNeeded(...)
}

// 5. autocompact
const { compactionResult } = await deps.autocompact(messagesForQuery, ...)
```

工具结果预算和 snip 排在最前面——比 microcompact 还早。这个顺序不是偶然的，背后有几个具体的边界条件在驱动。

## 工具结果预算：单条工具结果超大的场景

`applyToolResultBudget()` 处理的是这样的场景：模型读了一个 50000 行的日志文件，`Read` 工具返回了一个 500KB 的结果。这一条工具结果就能把 token 预算消耗大半。

这种情况下，问题不是"整段对话历史太长"，而是"这一条结果太大"。最便宜的修复是对这条结果截断或替换，而不是对整段历史做摘要。

```typescript
// 工具结果预算的逻辑：
// 如果某个工具的 maxResultSizeChars 没有设置（Number.isFinite 返回 false），
// 就把它排除在预算之外（不限制）
// 对于有限制的工具，在 applyToolResultBudget 里截断过大的结果
const excludedTools = new Set(
  toolUseContext.options.tools
    .filter(t => !Number.isFinite(t.maxResultSizeChars))
    .map(t => t.name),
)
messagesForQuery = await applyToolResultBudget(
  messagesForQuery,
  toolUseContext.contentReplacementState,
  persistFn,
  excludedTools,
)
```

代价极低：本质上是字符串截断，没有 API 调用，不改变对话结构。

## snip：局部臃肿的场景

snip 处理的是另一类场景：最近几轮产生了一些特别冗长的中间步骤（比如一次搜索输出了大量候选，或者一段反复修改的代码历史），这些内容在历史里占用大量 token，但对后续推理的贡献已经很小。

snip 做局部裁剪——只裁剪特定的、被标记为可剪切的片段，不触碰其他内容。它会返回 `tokensFreed`，这个值在后续的 autocompact 阈值判断里被减去：

```typescript
// autocompact 阈值判断时，减去 snip 已经释放的 token
const { isAboveAutoCompactThreshold } = calculateTokenWarningState(
  tokenCountWithEstimation(messagesForQuery) - snipTokensFreed,
  toolUseContext.options.mainLoopModel,
)
```

**这里有一个重要的边界条件**：如果不减去 `snipTokensFreed`，可能出现这样的情况：snip 已经把 token 压到了 autocompact 阈值以下，但 `calculateTokenWarningState` 用的是压缩前的旧 token 估算（因为它从 API response 的 `usage` 字段读数据，而 API 还没被调用），所以会误判"还在阈值以上"，错误地触发 autocompact。

这是一个细节很微妙的一致性问题：**读 API 返回的 token 统计** 和 **客户端估算的 token 统计** 之间有时间差，snip 在 API 调用之前修改了消息，但最新的精确计数要等 API 返回后才能拿到。`snipTokensFreed` 是这个时间差的补偿。

## 为什么不直接跳到 autocompact？

一个合理的问题：既然有 autocompact，为什么还要在前面加这些轻量处理？直接检测到 token 超限就全量压缩，不是更简单吗？

几个原因：

**1. autocompact 是有成本的**：autocompact 需要调用一次 LLM（通常是 Haiku），花时间、花钱。在会话高频进行的情况下，每次都触发 autocompact 成本很高。

**2. autocompact 是破坏性的**：全量摘要会丢失历史细节，缓存失效。如果当前任务还在密集使用工具，下一轮很可能需要重新读文件、重新搜索——这些信息在摘要里被删掉了，但实际上可能还需要。

**3. 很多时候根本不需要全量压缩**：一次 `Read` 返回了 50000 行，只需要截断那一条结果，不需要压缩整段对话历史。用大炮打蚊子，除了浪费还破坏现场。

## 时序图

```
query() 进入
  ↓
applyToolResultBudget()   ← 截断超大单条工具结果（无 API 成本）
  ↓
snipCompactIfNeeded()     ← 局部裁剪冗长片段（无 API 成本）
  ↓
microcompact()            ← 清除历史低密度工具结果（利用 cache edit）
  ↓
applyCollapsesIfNeeded()  ← 提交暂存折叠（本地投影，无摘要成本）
  ↓
autocompact()             ← 全量 LLM 摘要（有 API 成本，破坏性）
  ↓
API 调用
```

从左到右，代价递增，破坏性递增。设计的核心是：**先用最便宜的工具处理最常见的问题**。

## 面试指导

这个设计模式在系统设计面试里有直接类比：**写放大优化（Write Amplification Reduction）**。在数据库、存储系统里，总是先用代价小的局部操作处理局部问题，不到必要不做全量写入。

如果面试官问你"LLM 上下文管理系统里有哪些性能优化点"，这个分层处理的思路是一个很好的切入点：从截断单条大结果，到局部裁剪，到缓存层删除，再到全量摘要——每一层解决不同规模的问题，代价递增。
