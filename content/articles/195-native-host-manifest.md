# 第十三卷回收索引：外延执行里的产品接入底座

## 实现链
外延执行里真正落地的那一步，不是“功能能不能点开”，而是产品接入链能不能在真实系统里先长出一副笨重但必要的底座。

这条线真正逼出来的，是第十三卷里关于工程现实的判断。  
它现在自然分成两条更窄的线。

这页现在作为父级入口，继续分出去的实现分支是：
### Native Host、manifest、目录与注册表本来就是接入链的底座

- [235-第十三卷回收索引-外延执行-Native-Host-manifest-目录与注册表本来就是产品接入链的一部分.md](/Users/xxx/tmp/cc/docs/claude-code-book/appendices/235-第十三卷回收索引-外延执行-Native-Host-manifest-目录与注册表本来就是产品接入链的一部分.md)

### 检测与装配也属于接入链，笨重不是多余而是替真实用户把现场接稳的代价

- [236-第十三卷回收索引-外延执行-检测与装配也属于接入链笨重不是多余而是替真实用户把现场接稳的代价.md](/Users/xxx/tmp/cc/docs/claude-code-book/appendices/236-第十三卷回收索引-外延执行-检测与装配也属于接入链笨重不是多余而是替真实用户把现场接稳的代价.md)

它不再负责细讲接入底座和检测装配的两条回收线。  
它只负责保住一个入口判断：

成熟的产品接入，不会把 Native Host、manifest、目录、注册表、检测和装配当成附带手续。  
它们本来就是接入链的一部分。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。 对父级入口页来说，更常见的写法就是继续把两三条不同链挤在一页里，读起来省页数，但判断会糊。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。 这些父级页现在存在，就是为了先把分支关系讲清，再让更细页分别承担细节。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 作为父级入口，它还额外带来导航变深、读者要多跳几页的成本。
