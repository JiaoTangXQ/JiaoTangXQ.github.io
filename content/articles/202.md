# 连远了以后系统还像一个整体：层级语言统一的外部证明

面试题版本：**"一个用户同时运行了本地 bash 任务、一个远程 agent、一个 MCP 工具调用、还有浏览器 bridge 操作。这四件事在 Claude Code 里如何保持一致的用户体验？如果其中某件事的 UX 设计和其他三件不同，说明了什么？"**

## "拉远视角"的含义

技术测试通常聚焦单点：这个工具能不能调用成功、那个 API 响应是否正确。但系统整体性的测试是：用户把视角拉远，同时看到所有在运行的东西，整个系统是否还像一个协调工作的整体，而不是几个独立系统的拼凑。

具体问题：
- 本地 bash 任务和远程 agent 任务的进度显示格式是一样的吗？
- 被中断的工具调用和被中断的 agent 任务，用户看到的反馈一致吗？
- 切换前后台时，本地任务和远程任务的切换体验相同吗？
- 错误信息的格式、措辞、解决建议在不同任务类型里是统一的吗？

## 统一 TaskStateBase 的体验意义

前面讨论的 `TaskStateBase` 统一设计，在这里的意义是：所有任务类型（`local_bash`、`local_agent`、`remote_agent`、`dream`）都用同一个状态数据结构。

这意味着 UI 渲染任务状态时，不需要为每种任务类型写单独的渲染逻辑。任务列表、进度条、完成通知——这些 UI 元素从同一个数据源渲染，外观和行为自然一致。

对比：如果 remote agent 有自己的状态格式，进度显示逻辑要单独写。时间久了，remote agent 的进度条和本地任务的进度条在细节上（格式、刷新频率、错误显示）会出现差异。用户会感觉"这两个东西不太像一家的"。

## 统一工具接口的体验意义

`Tool.ts` 的统一接口让每种工具在执行时都有相同的状态转换（`queued → executing → completed`）。`StreamingToolExecutor` 对所有工具统一管理执行状态，进度消息统一从 `pendingProgress` 队列里推送。

用户看到的：截图工具在执行时和 bash 命令在执行时，进度显示方式是一样的——都有执行中的状态、都有输出流、都有完成或错误的终态。

对比：如果 computerUse 工具的执行是特殊路径，它的进度显示可能用了完全不同的机制，用户看到截图工具的 UI 和 bash 工具的 UI 明显不同。

## 什么时候整体感会被破坏

整体感被破坏的典型迹象：

**错误信息不一致**：bash 工具的错误说"命令执行失败，退出码 1"，MCP 工具的错误说"Tool execution failed with error code 1"，remote agent 的错误说"远程任务异常终止"。三种不同的措辞，用户无法建立统一的心智模型。

**操作反馈不一致**：取消一个本地任务立刻生效，取消一个远程任务需要等 30 秒才能确认。用户会对"取消"这个操作产生不确定感。

**状态显示不一致**：本地任务有详细的进度百分比，MCP 工具只有一个转圈加载动画，remote agent 只有"运行中"的文字。用户无法用统一的信息模型理解所有正在进行的工作。

## 整体性是架构约束的最终外化

"系统拉远了还像一个整体"不是 UX 设计师的功劳，而是架构约束的外部体现。如果底层的状态结构、执行接口、通知机制是统一的，用它们构建的 UI 自然倾向于一致。如果底层是各自为政的，再好的 UX 设计也无法掩盖拼凑的感觉——因为 UX 要用不同的适配器去"假装统一"。

## 面试角度的总结

"拉远了还像一个整体"是一个系统层级语言是否真正统一的外部验证指标。Claude Code 通过统一 Task 状态格式、统一 Tool 接口、统一通知机制，让不同类型的任务和工具在 UI 层面自然表现一致，而不需要 UX 层做大量的特殊适配。如果某个地方的体验明显不统一，那是底层架构没有统一的信号，不是表面风格问题。
