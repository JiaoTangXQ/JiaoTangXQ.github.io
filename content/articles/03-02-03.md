# 为什么附件不是原文的一部分，而是另一类系统消息

人类说话时，会很自然地把“正文”和“附带材料”混在一起理解。但对一个运行中的 agent 来说，这两类东西最好别混成一锅。文件、路径、选中的代码、队列里的通知、记忆、规则提醒、任务输出，这些内容虽然都和当前这一轮有关，却并不等于用户刚才那句原话。

Claude Code 因此把它们组织成独立的 attachment message。`getAttachmentMessages()` 不是简单地把附带内容拼进字符串尾巴，而是把不同来源的上下文翻译成一类单独消息，再在合适时机注入会话。这么做的好处很大：系统可以分别控制哪些附件该出现、什么时候出现、有没有预算限制、会不会在后续被重新注入，也能区分哪些是人亲口说的，哪些是运行时补给给这一轮的背景材料。

这正是“会话”与“原文”分层的体现。附件世界承认一件事：模型需要知道的，不一定都该被伪装成用户说过的话。把它们独立出来，既让系统更诚实，也让后面的预算治理、压缩、恢复和显示策略更容易做对。

很多 agent 产品一到这里就开始偷懒，把所有额外上下文都塞进同一大段提示词里。Claude Code 则选择了更笨也更稳的路: 承认不同消息种类确实不同。正因为这样，它后面的主循环才有机会精细地管理“这一轮到底带着哪些世界进入”。

## 实现链
附件走的是 `getAttachmentMessages()` / `createAttachmentMessage()`，生成的是独立 attachment message，而不是把 `@文件`、MCP 资源或 IDE 选择直接揉进用户原句里。

## 普通做法
更普通的产品会把附件内容提前展开成大段文字，直接塞进 prompt 文本。

## 为什么不用
因为附件并不只是“额外文字”，它们有类型、来源和后续处理方式。压扁成纯文本以后，系统和模型都很难再稳定地区分“这是一段附件上下文”还是“这是用户自然语言”。

## 代价
消息类型和渲染逻辑都会变复杂；但这样做后，附件才能在恢复、过滤和后续注入时保持自己独立的身份。
