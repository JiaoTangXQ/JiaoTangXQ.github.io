# 面试题：@引用文件和用户说的话，为什么要分成两类消息？

## 架构对比

这是两种常见设计的对比：

**设计 A（常见）**：把附件内容展开，拼接到用户消息里
```
用户消息：「帮我看这个错误
---文件内容：src/main.ts---
[文件全文]
---结束---
帮我看这个错误」
```

**设计 B（Claude Code）**：独立的 AttachmentMessage
```
用户消息：「帮我看这个错误」
AttachmentMessage { type: 'file', content: [...] }
AttachmentMessage { type: 'ide_selection', content: [...] }
```

Claude Code 选了 B，而且这个选择影响了系统里至少三个地方的工作方式。

## getAttachmentMessages 做了什么

`getAttachmentMessages()` 从输入字符串中提取各种引用，生成独立的 attachment 消息。它识别的来源包括：

- `@文件路径`：读取文件内容
- IDE 选中的代码（通过 `ideSelection` 参数传入）
- MCP 资源引用
- 记忆文件（CLAUDE.md 系统读取的规则）
- Agent 提及（`@agent-name` 语法）
- 队列中的命令（`queuedCommands`）

每一类来源都生成一个 `AttachmentMessage`，携带类型标记 `attachment.type`。

## 为什么要分开

**原因 1：生命周期管理**

附件消息有独立的生命周期控制。`filterDuplicateMemoryAttachments()` 可以去重，防止记忆文件在每轮都重复注入。`getMessagesAfterCompactBoundary()` 在压缩后可以选择性地重新注入某些附件类型。如果附件只是正文字符串的一部分，这种精细管理就不可能实现。

**原因 2：预算控制**

token 预算系统可以对不同来源的附件设置不同限制。一个很长的文件引用和用户的自然语言意图，在压缩优先级上应该有所不同——用户真实意图优先保留，大文件可以先裁。独立的消息类型才能独立地设置优先级。

**原因 3：来源可追溯**

如果用户说「这段代码有 bug」，是因为他自己在消息里贴了代码，还是因为 IDE 选中了代码被系统注入？这两种情况在后续处理（显示、日志、遥测）上有不同需求。分开成不同类型的 attachment，来源不会丢失。

**原因 4：显示层不同**

有些附件在 UI 里有特殊的折叠展示（文件引用显示为可展开的卡片），有些完全隐藏（hooks 上下文），有些部分显示。统一压进正文就失去了这种分层显示的能力。

## AttachmentMessage 的类型清单

从 `createAttachmentMessage` 的调用点可以归纳出系统里使用的 attachment 类型：

- `file` / `directory`：@文件引用
- `ide_selection`：IDE 选中内容
- `agent_mention`：@agent-name 语法
- `hook_additional_context`：hooks 执行结果
- `hook_success`：hook 成功结果
- `command_permissions`：prompt 命令的工具许可声明
- `memory`：CLAUDE.md / 记忆文件

每种类型都有独立的渲染逻辑和压缩策略。

## 什么情况下不提取附件

`processUserInputBase` 里有这个判断：

```typescript
const shouldExtractAttachments =
  !skipAttachments &&
  inputString !== null &&
  (mode !== 'prompt' || effectiveSkipSlash || !inputString.startsWith('/'))
```

slash 命令走自己的路径（在 `getMessagesForSlashCommand` 内部单独处理附件），不在这里提取。这避免了同一批附件被重复注入。

---

*面试指导：被问到「附件是怎么处理的」时，重点说明 AttachmentMessage 不是字符串拼接而是独立消息类型，并解释这个设计的三个实际好处：生命周期管理、预算控制、来源可追溯。这三点比「更清晰」「更模块化」这样的泛泛表述有说服力得多。*
