# 面试题：TaskItem 的 showOwner 条件是什么？为什么离线的 teammate 不显示署名？

这道题表面上是问一个显示条件，实质上是考察你对"实时状态 vs 历史记录"在界面上的不同处理方式。

## showOwner 的完整条件

```typescript
const showOwner = columns >= 60 && task.owner && ownerActive;
```

三个条件，缺一不可：
1. `columns >= 60`：终端宽度足够
2. `task.owner`：任务有 owner 字段
3. `ownerActive`：owner 对应的 teammate 仍在运行中

## 条件三的设计动机：为什么需要 ownerActive

**"接过任务"和"正在场"是不同的事**

想象这样一个工作流：
1. agent `researcher` 领取了 task-5："分析认证模块的安全漏洞"
2. `researcher` 执行了一段时间，然后退场了（任务委派或主动停止）
3. task-5 仍然在 `in_progress` 状态（已经开始了，但还没完成）

这时，task-5 的 `owner` 字段仍然是 `"researcher"`，但 `researcher` 已经不在线了。

**显示离线 owner 的危害**

如果 `(@researcher)` 仍然显示，用户可能会：
- 以为 `researcher` 仍然在处理这个任务（误解当前状态）
- 向不存在的 `researcher` 发指令（行动上的错误）
- 对任务的当前执行者产生混淆

**不显示的收益**

不显示 `(@researcher)` 时，用户知道：这个任务有 owner，但 owner 不在线了。这是一个诚实的状态表达——任务有历史负责人，但当前没有活跃执行者。

## 宽度条件的计算逻辑

```typescript
const ownerWidth = showOwner ? stringWidth(` (@${task.owner})`) : 0;
const maxSubjectWidth = Math.max(15, columns - 15 - ownerWidth);
const displaySubject = truncateToWidth(task.subject, maxSubjectWidth);
```

**为什么是 60 列？**

60 列是一个经验阈值，代表"显示 owner 后还能保留足够的任务 subject 可读性"的最小宽度。

计算一下：`60 - 15（图标+状态）- stringWidth(" (@owner)")` 留给 subject 的宽度。一个 6 字符的 owner 名字 `(@owner)` 大约占 10 列，60 - 15 - 10 = 35 列给 subject。35 列可以显示大约 35 个 ASCII 字符或 17-18 个中文字符，勉强够用。

如果终端只有 50 列，`50 - 15 - 10 = 25`，subject 只有 25 列，很可能被截断到看不出语义。所以 60 列是 owner 显示的最小要求。

**ownerWidth 提前计算的意义**

```typescript
const ownerWidth = showOwner ? stringWidth(` (@${task.owner})`) : 0;
```

`stringWidth()` 是一个精确计算字符串显示宽度的函数（处理宽字符，比如中文字符宽度是 2）。这里提前算好 ownerWidth，再用它来计算 maxSubjectWidth，确保 subject 和 owner 的总宽度不超过 `columns - 15`。

这是一个"先做宽度预算，再截断内容"的正确顺序，避免了"先截断 subject，但加上 owner 后仍然超宽"的问题。

## ownerActive 的数据来源

```typescript
ownerActive={task.owner ? activeTeammates.has(task.owner) : false}
```

`activeTeammates` 是前面分析的 Set，包含所有状态为 running 的 InProcessTeammateTask 的 `agentName` 和 `agentId`。

这个集合在每次渲染时从 `appStateTasks` 实时计算，不是缓存的。所以当 teammate 退场时，下一次渲染（很快，因为状态变化会触发重渲染）`ownerActive` 就会变成 false，署名就消失了。

## 任务行的视觉语义

完整的任务行有多种视觉状态：

```
✓ [done] task subject (strikethrough, dimColor)
■ [in progress] task subject (bold, ownerColor @owner, activity...)
□ [pending] task subject
□ [pending, blocked] task subject ▸ blocked by #2, #3 (dimColor)
```

`showOwner` 只在 in_progress 和某些 pending 的情况下有意义（因为 `ownerActive` 需要 teammate 在运行中）。完成的任务里 teammate 通常已经不在线，所以完成任务的行上通常也不会有 owner 显示——这是自然的，完成了就完成了，不需要知道是谁做的（至少在任务行这个维度上）。

## 面试指导

**直接问法**："showOwner 需要满足哪些条件？"

答：三个：终端宽度 >= 60（保证 subject 可读性）、任务有 owner 字段（有负责人）、owner 仍在运行（ownerActive）。缺一不可，特别是第三个——离线的 owner 不显示，避免误导用户以为有活跃执行者在场。

**追问**："如果 teammate 退场了但任务还在 in_progress，界面会怎样？"

答：`activeTeammates.has(task.owner)` 变成 false，`showOwner` 变成 false，`(@owner)` 从任务行消失。任务行仍然显示 in_progress 状态（任务还在），但没有 owner 署名（执行者不在了）。这是一个"任务还活着，但执行者已经走了"的诚实状态表达。

**设计追问**："你觉得应该在离线 owner 的地方显示什么，比如显示一个灰色的 owner 名，说明"曾经是由他负责"？"

答：有道理，但实现成本和认知成本都更高。灰色 owner 需要用户理解"active = 彩色，inactive = 灰色"的视觉语言，而且还要处理"这是谁上次做的"这个历史状态的价值——大多数情况下用户不关心历史负责人，只关心"现在谁在管"。所以当前的"不显示"是更保守也更清洁的选择。如果历史追溯是一个真实需求，应该在详情视图里提供，而不是在主列表上。

---

*核心考察点：实时状态 vs 历史状态的界面处理、多条件显示逻辑的设计动机、宽度预算计算的正确顺序。*
