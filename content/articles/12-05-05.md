# 状态、扩展、用法和站点权限被摆在同一面板，说明入口不是开关而是现场面板

`/chrome` 这块面板并没有只给一个“启用”按钮就结束，而是把当前连接状态、扩展是否装好、命令行开关用法，以及站点权限该去哪里管，统统摆在同一块地方。这说明它把自己理解成一个现场面板，而不是单纯功能开关。

这种设计很成熟，因为浏览器能力真正难的从来不是“有没有这个功能”，而是人此刻知不知道它在哪、通没通、受谁管、下一步该往哪走。把这些信息放在一起，就是在替用户接住真实现场。

## 实现链
`ClaudeInChromeOnboarding.tsx` 既讲能力范围，也直接告诉用户站点权限继承自扩展设置；`toolRendering.tsx` 还会把 tab 现场做成 `[View Tab]` 链接。代码关注的是“当前浏览器现场能不能继续做事”，而不是只给一个布尔开关。

## 普通做法
普通做法会把“启用/禁用”做成开关，其他用法和权限文档另放角落。

## 为什么不用
Claude Code 不只给开关，是因为浏览器能力真正的故障点都发生在现场层：当前 tab、当前站点权限、当前扩展状态。

## 代价
代价是产品入口会更像一个现场面板，而不是一眼看完的简洁设置项。
