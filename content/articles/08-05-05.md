# 回声消息先被当心跳，再被当重复，这说明稳定性靠顺序细节

## 实现链
- `useRemoteSession.ts` 里处理消息的顺序很讲究：任何 WebSocket 消息先清除超时计时器，再检查是不是本地刚 POST 过去的 echo；而且去重不是普通 Set，而是 `BoundedUUIDSet`，因为同一个 UUID 可能被回声两次。
- 这个顺序表达的是：echo 首先证明远端还活着，其次才是重复内容需要被过滤。

## 普通做法
- 更直觉的实现是先 dedup，发现是自己发过的消息就立刻丢掉。
- 代码会更简单，也更符合“重复消息应尽早过滤”的常规思路。

## 为什么不用
- Claude Code 没先过滤，是因为远端慢流式、compaction 或冷启动时，echo 本身就是心跳。如果先丢掉再清 timeout，本地可能会错误判定远端卡死。
- 这不是算法层的大思想，而是稳定性靠顺序细节一点点抠出来的典型例子。

## 代价
- 代价是逻辑变得不直观，新读者很容易误以为“为什么不先去重”。
- 但这里的复杂度是有回报的，它直接换掉了一类很难解释的误报重连。
