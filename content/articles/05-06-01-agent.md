# 同步子 Agent 为什么在开跑前先拿 backgroundSignal？

`AgentTool.tsx` 里，一个子 Agent 要在当前对话轮次里同步执行时，会走这样一段代码（简化）：

```typescript
// 同步执行路径
const {
  taskId,
  backgroundSignal,
  cancelAutoBackground,
} = registerAgentForeground({
  agentId,
  description,
  prompt,
  selectedAgent,
  setAppState,
  autoBackgroundMs,  // 可选：超时自动转后台
  toolUseId,
})

// 然后才真正开始执行
const result = await Promise.race([
  runAgent(...),
  backgroundSignal.then(() => BACKGROUND_SENTINEL),
])
```

`registerAgentForeground()` 在 `runAgent()` 之前调用，在 Agent 还没开始跑之前就准备好了一个 `backgroundSignal` Promise。

这看起来很奇怪：Agent 还没开始，为什么要先准备"转后台"的机制？

## 因为"同步"只是当前意图，不是本质属性

一个子 Agent 被标记为"同步执行"，意思是：**现在**的意图是在当前轮次里等它跑完。但这个意图可能被打断：

- 用户等了 30 秒觉得太慢，想把它推到后台
- 系统配置了 `autoBackgroundMs`，超时后自动转后台
- 将来可能有其他触发条件

如果不在开始前就准备好转场机制，一旦需要转后台，只有两个选择：

**选项 A：强制中断**。把正在运行的 Agent 直接 abort，重新开一个后台任务从头来过。代价：丢失当前进度、需要重新发 API 请求、用户体验差。

**选项 B：提前准备，优雅交棒**。在开始前就准备好 `backgroundSignal`，一旦决定转后台，通过这个信号让执行循环感知到，然后同一个任务对象继续在后台跑。代价：开始前多一步注册，稍微重一点。

Claude Code 选了选项 B。

## registerAgentForeground 的实现

```typescript
export function registerAgentForeground({
  agentId,
  description,
  prompt,
  selectedAgent,
  setAppState,
  autoBackgroundMs,
  toolUseId,
}): {
  taskId: string
  backgroundSignal: Promise<void>
  cancelAutoBackground?: () => void
} {
  void initTaskOutputAsSymlink(agentId, getAgentTranscriptPath(asAgentId(agentId)))

  const abortController = createAbortController()
  const unregisterCleanup = registerCleanup(async () => {
    killAsyncAgent(agentId, setAppState)
  })

  const taskState: LocalAgentTaskState = {
    ...createTaskStateBase(agentId, 'local_agent', description, toolUseId),
    type: 'local_agent',
    status: 'running',
    agentId,
    prompt,
    selectedAgent,
    agentType: selectedAgent.agentType ?? 'general-purpose',
    abortController,
    unregisterCleanup,
    retrieved: false,
    lastReportedToolCount: 0,
    lastReportedTokenCount: 0,
    isBackgrounded: false,   // 前台任务：isBackgrounded = false
    pendingMessages: [],
    retain: false,
    diskLoaded: false,
  }

  // 创建 backgroundSignal Promise，把 resolver 存起来
  let resolveBackgroundSignal: () => void
  const backgroundSignal = new Promise<void>(resolve => {
    resolveBackgroundSignal = resolve
  })
  backgroundSignalResolvers.set(agentId, resolveBackgroundSignal!)

  registerTask(taskState, setAppState)

  // 如果配置了超时自动转后台
  let cancelAutoBackground: (() => void) | undefined
  if (autoBackgroundMs !== undefined && autoBackgroundMs > 0) {
    const timer = setTimeout((setAppState, agentId) => {
      // 时间到了：改 isBackgrounded，resolve Promise
      setAppState(prev => {
        const prevTask = prev.tasks[agentId]
        if (!isLocalAgentTask(prevTask) || prevTask.isBackgrounded) return prev
        return {
          ...prev,
          tasks: {
            ...prev.tasks,
            [agentId]: { ...prevTask, isBackgrounded: true },
          },
        }
      })
      const resolver = backgroundSignalResolvers.get(agentId)
      if (resolver) {
        resolver()
        backgroundSignalResolvers.delete(agentId)
      }
    }, autoBackgroundMs, setAppState, agentId)

    cancelAutoBackground = () => clearTimeout(timer)
  }

  return { taskId: agentId, backgroundSignal, cancelAutoBackground }
}
```

关键点：
1. `backgroundSignalResolvers` 是一个全局 Map，把 `agentId` 映射到 `resolveBackgroundSignal` 函数
2. 这个 resolver 在任务注册时就准备好，但不会被调用，除非有人显式调用 `backgroundAgentTask()` 或者超时触发
3. 执行循环（`runAgent()`）通过 `Promise.race` 感知这个信号

## Promise.race 的转场逻辑

```typescript
const result = await Promise.race([
  runAgent(...),
  backgroundSignal.then(() => BACKGROUND_SENTINEL),
])

if (result === BACKGROUND_SENTINEL) {
  // backgroundSignal 赢了：Agent 继续在后台跑
  // 但执行控制权已经从当前轮次释放了
  return handleBackgroundedResult(...)
}
// 正常完成
return handleCompletedResult(result)
```

`BACKGROUND_SENTINEL` 是一个特殊的哨兵值，用来区分"Agent 正常返回结果"和"被转后台了"这两种情况。

当 `backgroundSignal` 被 resolve，`Promise.race` 立刻返回 `BACKGROUND_SENTINEL`，当前轮次的等待结束。但 Agent 本身（`runAgent()`）仍然在后台继续运行——因为 `Promise.race` 只是停止了等待，并没有取消另一边的 Promise。

## registerAsyncAgent 的对比

后台 Agent（直接进入后台，不经过前台）使用 `registerAsyncAgent()`：

```typescript
const taskState: LocalAgentTaskState = {
  // ...
  isBackgrounded: true,   // 直接标记为后台
  // 不创建 backgroundSignal
  // 不存到 backgroundSignalResolvers
}
```

`registerAsyncAgent` 不创建 `backgroundSignal`，因为后台任务从一开始就是后台的，不需要"转场"机制。

前台和后台任务的区别只在这一个字段的初值，以及是否有 `backgroundSignal`。其他所有字段（`outputFile`、`progress`、`pendingMessages`、`retain`）完全相同。

## 面试要点

这是一个**防御性初始化**的经典案例：在你确信不需要某个机制之前，先把它准备好。

对于子 Agent，"同步执行"只是初始意图。真正的运行时间和条件是不确定的（用户可能等不及，系统可能超时）。提前准备好转场机制，成本是每次任务注册多执行一次 `new Promise` 和一次 Map 插入；收益是当真的需要转场时，有一个优雅的路径，而不是强制中断重来。

这种"多准备一点，换取优雅降级能力"的思路，在系统设计中非常值得借鉴。
