# 任务完成通知为什么要变成 XML 消息投回消息队列？

`LocalAgentTask.tsx` 里的 `enqueueAgentNotification()` 生成这样一段消息：

```typescript
const message = `<${TASK_NOTIFICATION_TAG}>
<${TASK_ID_TAG}>${taskId}</${TASK_ID_TAG}>${toolUseIdLine}
<${OUTPUT_FILE_TAG}>${outputPath}</${OUTPUT_FILE_TAG}>
<${STATUS_TAG}>${status}</${STATUS_TAG}>
<${SUMMARY_TAG}>${summary}</${SUMMARY_TAG}>${resultSection}${usageSection}${worktreeSection}
</${TASK_NOTIFICATION_TAG}>`

enqueuePendingNotification({
  value: message,
  mode: 'task-notification',
})
```

实际的 XML 大概长这样：

```xml
<task-notification>
<task-id>a8kj2m9x</task-id>
<tool-use-id>toolu_01abc</tool-use-id>
<output-file>/tmp/project/.../tasks/a8kj2m9x.output</output-file>
<status>completed</status>
<summary>Agent "分析日志" completed</summary>
<result>共发现 3 个异常请求模式...</result>
<usage><total_tokens>4521</total_tokens><tool_uses>12</tool_uses><duration_ms>8430</duration_ms></usage>
</task-notification>
```

然后这段文本通过 `enqueuePendingNotification()` 送进消息队列，等待被主 Agent 消费。

**为什么要这么绕？** 子 Agent 完成了，直接把结果塞给主 Agent 的上下文不行吗？

## 直接回调的问题

最直接的实现：

```typescript
// 假设的直接回调实现
function onAgentComplete(result) {
  mainAgentContext.addMessage({
    role: 'user',
    content: `子 Agent 完成了：${result}`
  })
  // 或者直接修改某个共享变量
  sharedResults.push(result)
}
```

这个方案的问题：

**1. 破坏回合原子性**。主 Agent 正在处理一个 API 请求时，它的消息历史是"冻结的"。如果这时直接往历史里插入新消息，就把正在进行中的回合"污染"了——Claude API 收到的历史变成了一个中途被修改过的奇怪序列，可能导致上下文混乱或 API 错误。

**2. 并发通知顺序不可控**。如果有多个子 Agent 同时完成，直接回调会导致多条消息以不可预测的顺序插入主上下文，没有任何去重或排序机制。

**3. 与主 Agent 的轮次边界不对齐**。主 Agent 是以"轮次"为单位推进的（一个 API 请求 = 一个轮次）。后台任务的完成不与轮次边界对齐，直接注入会打乱这个节奏。

## 消息队列的作用

`enqueuePendingNotification()` 把通知放进一个队列，而不是直接投给主 Agent。这个队列有几个重要属性：

**1. 在正确的时机被消费**。消息队列里的通知会在主 Agent 的下一个轮次开始时（即主 Agent 准备好接收新 user 消息时）被取出，作为 user 消息送给 Claude API。这样通知总是在轮次边界对齐的地方插入。

**2. 排序和合并**。如果多个后台任务同时完成，它们的通知都进队列，按顺序被消费，不会竞争或乱序。

**3. 去重保护**。`notified` 标志在通知入队前就被原子性设为 `true`，防止同一个任务发多条通知。

## XML 格式的选择

通知使用 XML 而不是 JSON 或者结构化对象，是因为这个通知会直接出现在 Claude API 的 user 消息文本里：

```
Human: <task-notification>
<task-id>a8kj2m9x</task-id>
...
</task-notification>
```

Claude 能理解 XML 格式的结构化文本，可以从中提取任务 ID、状态、结果等信息，并据此决定下一步操作（比如"子 Agent 完成了，现在我可以用它的结果..."）。

XML 比 JSON 更容易直接嵌入文本（不需要转义，分隔符不容易和普通文本混淆），也是 Claude 在训练数据里更熟悉的格式。

## notified 的原子检查

```typescript
let shouldEnqueue = false
updateTaskState<LocalAgentTaskState>(taskId, setAppState, task => {
  if (task.notified) {
    return task  // 已通知，不改状态
  }
  shouldEnqueue = true
  return { ...task, notified: true }  // 原子性设置 notified
})
if (!shouldEnqueue) {
  return  // 不发通知
}
```

`shouldEnqueue` 标志和 `notified: true` 在同一个 `updateTaskState` 回调里完成：状态更新和"是否需要发通知"的决定是原子的。

为什么要这样？如果先判断 `task.notified`，再改状态，再发通知，在多并发场景下可能有两个路径同时通过了 `!task.notified` 的检查，然后各自改状态、各自发通知，导致重复通知。把判断和状态更新绑定在同一个原子操作里，从结构上杜绝了这种竞争。

## tool_use_id：让通知能匹配回工具调用

```typescript
const toolUseIdLine = toolUseId
  ? `\n<${TOOL_USE_ID_TAG}>${toolUseId}</${TOOL_USE_ID_TAG}>`
  : ''
```

`toolUseId` 是触发子 Agent 的那个 `tool_use` 块的 ID（从 Claude API 的 assistant 消息里来）。把它放进通知里，主 Agent 可以把这个通知和当初的工具调用关联起来。

这对 Claude 模型的上下文理解很重要：它能知道"当初我调用了 Agent 工具，现在这个任务完成了，我应该继续"，而不是把这条通知当成一条无来源的消息。

## 面试要点

**通知系统设计**是一个经典的系统设计话题。关键决策点：

**推送 vs 拉取**：后台任务完成时"推送"通知（直接回调），还是主系统"拉取"（轮询队列）？

Claude Code 选择的是一个混合模式：后台任务完成时**主动入队**（推送语义），但消息在队列里等待，由主 Agent 的轮次推进**拉取**消费。

这个设计避免了纯推送（时机不可控，可能打断正在进行的操作）和纯拉取（延迟高，需要频繁轮询）的缺点，在及时性和时序控制之间取得了平衡。
