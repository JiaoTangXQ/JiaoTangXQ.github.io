# 通知为什么必须重新回到消息队列

子 agent 做完事以后，不是“直接把结果塞给主 agent”就算完。Claude Code 坚持让它重新回到消息队列报到。

## 实现链

`enqueueAgentNotification()` 会生成带 `task_id`、`output_file`、`status`、`summary` 的 XML 片段，再通过 `enqueuePendingNotification()` 送进统一消息队列。`notified` 标志会先被原子化设置，避免重复发送。也就是说，子 agent 不是直接改主会话上下文，而是先变成一条正式通知消息。

这让“任务完成”成为消息世界里可理解的一件事，而不是后台私下发生的一件事。

## 普通做法

普通做法会在任务结束时直接回调父逻辑，或者把结果塞进某个共享数组，让主循环下次自己取。

## 为什么不用

Claude Code 不直接硬塞，是因为主会话和后台任务之间需要有明确边界。通知重新回到消息队列，意味着它重新接受主系统的排序、消费和去重规则。这样主对话不会被后台任务偷偷改写，只会收到一条正式事件。

这对小白来说很好理解：做完事要先回前台窗口报到，而不是直接闯进老板脑子里。

## 代价

代价是多了一层封装和消费流程，完成结果到达主会话不会是最短路径。但这层缓冲换来了更稳的秩序。
