# 面试题：为什么 mcp__computer-use__* 这个工具名前缀是跨系统的「通关文牒」？

## 一个看似简单的命名问题

面试中偶尔会遇到这类问题："你们系统里的工具命名有什么规范？为什么？"

初级工程师的回答通常是：保持可读性、遵循 snake_case 或 camelCase 约定、加前缀避免冲突。

但 Claude Code 的工具命名背后有更深层的设计：名字本身是一份**跨系统协议契约**，改名不只是"重命名"，而是破坏多个独立子系统之间的隐式约定。

## COMPUTER_USE_MCP_SERVER_NAME 被固定的原因

在 `utils/computerUse/common.ts` 里，server name 是一个硬编码常量：

```ts
export const COMPUTER_USE_MCP_SERVER_NAME = 'computer-use'
```

工具名通过 `buildMcpToolName` 生成：

```ts
buildMcpToolName(COMPUTER_USE_MCP_SERVER_NAME, t.name)
// → mcp__computer-use__screenshot
// → mcp__computer-use__click
```

`setup.ts` 的注释解释了为什么这个名字不能随便改：

> The API backend detects `mcp__computer-use__*` tool names and emits a CU availability hint into the system prompt. Built-in tools with different names wouldn't trigger it.

这里有三个系统在依赖同一个字符串：

**后端 API**：扫描工具列表，找到 `mcp__computer-use__*` 前缀，注入 computer use 的 system prompt 提示。名字不对，提示不注入，模型不知道自己有桌面能力。

**权限系统**：`allowedTools` 里存的是精确的工具名，permission checker 靠字符串匹配决定是否放行。名字变了，权限匹配失效。

**UI 渲染层**：`toolRendering.tsx` 识别 computer use 工具并做特殊 UI 处理。名字不对，渲染层不知道这是桌面操作。

## 命名漂移的连锁反应

假设有工程师觉得 `mcp__computer-use__` 太长，想改成 `cu__`：

1. 后端 API 不再检测到前缀，不注入 CU hint → 模型不使用桌面能力
2. `allowedTools` 匹配失败 → 每次桌面操作都触发权限确认弹窗
3. `toolRendering.tsx` 不识别 → 桌面工具调用显示成通用工具格式

三个系统同时静默失效，而且没有任何编译错误提醒你。这就是"命名即协议"带来的风险。

## 命名约定如何在多个 Repo 之间传递

注释还提到了一个更微妙的跨 repo 依赖：

> Cowork uses the same names for the same reason (apps/desktop/src/main/local-agent-mode/systemPrompt.ts:314).

`Cowork`（Claude 的另一个产品）也在用完全相同的工具名前缀，原因完全一样。这意味着这个命名约定已经跨越了两个独立产品代码库。

在大型工程组织里，这类隐式约定是最难管理的：没有类型检查，没有接口定义，只有注释里提到"另一个地方也用这个名字"。一旦有人在一边改名，两个产品之间的行为差异就会开始出现，而且很难被快速发现。

## 统一命名和 Cowork 对齐的工程价值

同一套工具名在两个产品里都被使用，还有一个额外好处：测试和 dogfooding 数据可以跨产品复用。

如果 Cowork 的用户发现某个 computer use 工具调用路径有问题，这个问题在 Claude Code 里很可能也存在。因为它们使用的是同一个后端 API，在同一套 CU availability hint 下工作，命名约定完全一致。

反过来，如果两边命名不同，同样的 bug 可能在一边被发现，另一边因为触发路径不同而被漏掉。

## buildMcpToolName 的设计

`buildMcpToolName` 是生成完整工具名的工具函数：

```ts
// 伪代码，实际实现在 services/mcp/mcpStringUtils.ts
function buildMcpToolName(serverName: string, toolName: string): string {
  return `mcp__${serverName}__${toolName}`
}
```

注意这个函数是集中的：所有工具名都经过同一个函数生成，而不是各自字符串拼接。这样有几个好处：

- 格式约定集中在一处，修改规则只需改一个地方
- 类型系统可以约束参数类型
- 任何格式变更都是显式的，不会静默漂移

## 面试考察点

这道题考察的是对**隐式接口约定**和**多系统耦合**的理解。

软件系统里的接口耦合有两种：
- **显式接口**：通过类型、抽象类、接口定义约束，编译期可发现
- **隐式约定**：通过约定（命名规则、字段含义、调用顺序）耦合，只能靠注释或文档传递

`mcp__computer-use__*` 这个前缀是典型的隐式约定。它的价值在于多系统对齐，它的风险在于变更时没有编译器保护。

资深工程师面对这类设计时，会问：**有没有办法把这个隐式约定变成显式接口？** 比如把前缀检测逻辑集中到一个类型安全的枚举，让编译器强制所有消费方都更新。

Claude Code 的解法是通过集中的常量（`COMPUTER_USE_MCP_SERVER_NAME`）和工厂函数（`buildMcpToolName`）来降低漂移风险，但仍然是隐式约定，不是显式接口。这是一个务实的妥协，而不是理想架构。
