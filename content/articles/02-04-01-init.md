# init() 为什么先做配置系统，而不是先做网络或认证？

## init() 的开头

```typescript
export const init = memoize(async (): Promise<void> => {
  const initStartTime = Date.now()
  logForDiagnosticsNoPII('info', 'init_started')
  profileCheckpoint('init_function_start')

  try {
    enableConfigs()                    // 第一步
    applySafeConfigEnvironmentVariables() // 第二步
    applyExtraCACertsFromConfig()      // 第三步
    setupGracefulShutdown()            // 第四步
    // ...然后才是网络相关
    configureGlobalMTLS()
    configureGlobalAgents()
    preconnectAnthropicApi()
```

这个顺序不是随意的。

## 依赖链：为什么 enableConfigs() 必须第一

`enableConfigs()` 初始化配置系统，之后代码才能调用 `getGlobalConfig()`、`getCurrentProjectConfig()` 等函数。

`applySafeConfigEnvironmentVariables()` 读取配置里的环境变量设置，写入 `process.env`。如果配置系统没初始化，这里读到的是默认空值，写入的是错误数据。

`applyExtraCACertsFromConfig()` 读取配置里的 `NODE_EXTRA_CA_CERTS` 路径，应用到进程 TLS 配置。

```typescript
// 注释非常直接
// Apply NODE_EXTRA_CA_CERTS from settings.json to process.env early,
// before any TLS connections. Bun caches the TLS cert store at boot
// via BoringSSL, so this must happen before the first TLS handshake.
applyExtraCACertsFromConfig()
```

**关键约束**：Bun 在 BoringSSL 层面缓存 TLS cert store。一旦第一次 TLS 握手发生，cert store 就被锁定了。`NODE_EXTRA_CA_CERTS` 必须在任何 TLS 连接之前写入 `process.env`，否则新证书不会生效。

这个约束决定了：
1. `enableConfigs()` 必须第一（配置系统就绪）
2. `applyExtraCACertsFromConfig()` 紧随其后（读配置，应用证书）
3. 任何网络操作（包括 `configureGlobalAgents()` 和 `preconnectAnthropicApi()`）必须在此之后

## "安全"环境变量 vs 完整环境变量

注意 `applySafeConfigEnvironmentVariables()` 里的"Safe"。

这不是说其他环境变量"不安全"，而是说：在 trust 确认之前，只能应用部分环境变量。

具体地说，某些环境变量的内容可能来自项目级配置（`.claude/settings.json`）。如果项目是不可信的（用户还没有确认 trust），应用这些环境变量可能会引入恶意项目注入的配置。

所以 `init()` 里只调用 "Safe" 版本：只应用那些来自用户全局配置（`~/.claude/settings.json`），不依赖项目本地配置的部分。

完整版本（包含项目配置）在 trust 确认后才通过 `applyConfigEnvironmentVariables()` 应用：

```typescript
// initializeTelemetryAfterTrust() 里
void waitForRemoteManagedSettingsToLoad()
  .then(async () => {
    applyConfigEnvironmentVariables()  // 完整版，trust 之后
    await doInitializeTelemetry()
  })
```

## 代理和 mTLS 的约束

```typescript
configureGlobalMTLS()   // 先
configureGlobalAgents() // 后
preconnectAnthropicApi() // 最后
```

mTLS 配置必须先于代理配置，因为某些代理场景下（mTLS + 代理的组合），代理 agent 的创建需要知道 mTLS 证书。

代理配置必须先于 API 预连接，因为预连接使用的 HTTP agent 必须是已经配置好代理的那个。

如果 `preconnectAnthropicApi()` 在代理配置之前执行，它会建立一个"不走代理"的连接。后续真正的 API 请求使用代理，就无法复用这个预热连接——预热白做了。

## setupGracefulShutdown() 为什么是第四步？

```typescript
setupGracefulShutdown()
profileCheckpoint('init_after_graceful_shutdown')
```

这个位置很有意思。`setupGracefulShutdown()` 注册 `SIGTERM`、`SIGINT` 等信号处理器，确保进程退出时能清理资源。

放在配置系统之后、网络操作之前的原因：一旦任何可能需要清理的资源被创建（网络连接、子进程、临时文件），退出处理就必须能清理它们。在创建这些资源之前注册退出处理，保证了不会有"创建了资源但忘记注册清理"的情况。

## 面试指导

初始化顺序面试题的解题框架：

**第一步**：识别哪些操作有显式依赖（B 依赖 A 的输出）
**第二步**：识别哪些操作有时机约束（A 必须在某个系统状态变化之前）
**第三步**：识别哪些操作有安全约束（A 不能在信任确认之前执行）
**第四步**：识别哪些操作是幂等的（可以多次执行），哪些不是

Claude Code 的 `init()` 顺序体现了一个原则：**越底层的基础设施越先初始化**。配置系统是其他一切的基础；证书/代理是网络的基础；进程退出处理是所有资源创建的元基础。

这个"越底层越先"的原则，比"按重要性排序"或"按代码行顺序"更有说服力，也更容易在设计阶段就保持一致。
