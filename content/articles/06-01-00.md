# 为什么 agent 越强，越不能靠零散弹窗管权限？

面试官问：如果你来设计一个能执行 shell 命令、读写文件、调用外部 API 的 AI agent，你会怎么做权限控制？

大多数人的第一反应是：哪里危险，哪里弹窗确认。这个答案直觉上合理，但 Claude Code 的设计选了一条完全不同的路。

## 弹窗模型的根本缺陷

弹窗确认有一个隐含假设：**每次操作都是独立事件**。这对传统 CLI 工具成立——你执行 `rm -rf`，系统问你一次，你选是或否，完事。

Claude Code 的 agent 运行时不是这样工作的。它会：
- 在一次任务里连续调用 20-50 个工具
- 多个子 agent 并行执行，每个都有自己的权限需求
- 中途根据分析结果动态决定是否需要更高权限
- 在 headless/SDK 模式下根本没有 UI 可以弹窗

如果用弹窗模型，这套系统在"系统永远处于什么安全姿态"这个问题上根本没有答案。工具筛选时看到的规则，和工具调用时审批用的规则，可能是完全不同的两套——因为中间那几毫秒里用户点了某个弹窗。

## 制度化安全姿态

Claude Code 的解法是：**先声明当前会话处于哪种权限模式，再让所有组件围着这个模式对齐**。

这个设计的核心实体在 `src/types/permissions.ts`，被 `PermissionMode.ts` 重新导出：

```typescript
// PERMISSION_MODES 是全部合法模式的枚举
const PERMISSION_MODES = [
  'default',
  'plan',
  'acceptEdits',
  'bypassPermissions',
  'dontAsk',
  'auto',     // 仅在 TRANSCRIPT_CLASSIFIER feature flag 开启时有效
  'bubble',   // 内部用
] as const

type PermissionMode = typeof PERMISSION_MODES[number]
```

每种模式不只是个字符串，而是一个完整的配置包：

```typescript
const PERMISSION_MODE_CONFIG: Partial<Record<PermissionMode, PermissionModeConfig>> = {
  default:          { title: 'Default',            color: 'text',       external: 'default' },
  plan:             { title: 'Plan Mode',           color: 'planMode',   external: 'plan' },
  acceptEdits:      { title: 'Accept edits',        color: 'autoAccept', external: 'acceptEdits' },
  bypassPermissions:{ title: 'Bypass Permissions',  color: 'error',      external: 'bypassPermissions' },
  dontAsk:          { title: "Don't Ask",           color: 'error',      external: 'dontAsk' },
  auto:             { title: 'Auto mode',           color: 'warning',    external: 'default' },
}
```

注意 `auto` 的 `external` 字段是 `'default'`——这是 Anthropic 内部模式，外部用户看不到它的真实名称。

## 模式是会话级状态，不是 UI 装饰

`PermissionMode` 被嵌入 `ToolPermissionContext`，而 `ToolPermissionContext` 是整个权限体系的核心快照对象：

```typescript
export type ToolPermissionContext = DeepImmutable<{
  mode: PermissionMode
  alwaysAllowRules: ToolPermissionRulesBySource
  alwaysDenyRules: ToolPermissionRulesBySource
  alwaysAskRules: ToolPermissionRulesBySource
  isBypassPermissionsModeAvailable: boolean
  isAutoModeAvailable?: boolean
  strippedDangerousRules?: ToolPermissionRulesBySource
  shouldAvoidPermissionPrompts?: boolean
  // ...
}>
```

这个对象随每次工具调用的上下文链传递。工具装配层、运行时审批、设置变更处理，都对着同一份快照做决定。这就是"制度化姿态"的工程含义：**不是事到临头再协商，而是事前已经存在一个所有人都能读到的制度现实**。

## 模式切换是立法行为，不是按钮

模式切换通过 `transitionPermissionMode()` 完成。这个函数做的不只是把 `mode` 字段改一下，而是：

1. 触发 plan 模式进出的 attachment 处理（`handlePlanModeTransition`）
2. 触发 auto 模式的激活/停用（`handleAutoModeTransition`、`setAutoModeActive`）
3. 进入 auto 时剥离危险放行规则（`stripDangerousPermissionsForAutoMode`）
4. 离开 auto 时恢复被剥离的规则（`restoreDangerousPermissions`）

```typescript
export function transitionPermissionMode(
  fromMode: string,
  toMode: string,
  context: ToolPermissionContext,
): ToolPermissionContext {
  if (fromMode === toMode) return context

  handlePlanModeTransition(fromMode, toMode)
  handleAutoModeTransition(fromMode, toMode)

  if (fromMode === 'plan' && toMode !== 'plan') {
    setHasExitedPlanMode(true)
  }

  if (feature('TRANSCRIPT_CLASSIFIER')) {
    const fromUsesClassifier = fromMode === 'auto' || (fromMode === 'plan' && isAutoModeActive())
    const toUsesClassifier = toMode === 'auto'

    if (toUsesClassifier && !fromUsesClassifier) {
      autoModeStateModule?.setAutoModeActive(true)
      context = stripDangerousPermissionsForAutoMode(context)
    } else if (fromUsesClassifier && !toUsesClassifier) {
      autoModeStateModule?.setAutoModeActive(false)
      context = restoreDangerousPermissions(context)
    }
  }
  return context
}
```

这就是为什么本章叫"治理"而不是"弹窗"：模式不是一个 UI 开关，而是一套让强能力敢于存在的制度骨架。

## 这道题考什么

面试时如果被问到"如何设计 AI agent 的权限系统"：

**浅层回答**：用 allow list/deny list，危险操作前弹窗确认。

**深层回答**：需要区分两层问题——装配时的工具可见性（deny 规则在工具池就裁掉），和运行时的行为合法性（结合现场上下文和当前模式做审批）。两层之间需要一份一致的制度快照（`ToolPermissionContext`），而不是分散读取。

**加分回答**：聊到 headless/SDK 场景下没有 UI 弹窗的降级处理，以及模式切换本身作为立法行为的设计意义——为什么进入 auto 模式要主动剥离用户之前设的宽松规则，而不是照单全收。
