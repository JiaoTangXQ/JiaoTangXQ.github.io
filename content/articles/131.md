# 深度分析：前后台切换为什么是姿态变化，而不是对象替换

## 一个直觉上的困惑

你正在前台查看一个任务的执行详情。你按下快捷键把它发到后台，继续做别的事。过了一会儿，你从任务面板调出来，重新看它的详情。

**问题：你看到的还是"同一个任务"吗？**

从用户角度，答案应该是：当然是。任务的历史进度还在，之前完成的步骤没有消失。

但从实现角度，这个"还是同一个"需要被明确保证。如果前台和后台用的是不同对象，"调回前台"实际上是在展示一个新创建的对象，之前在前台看到的状态就丢失了。

## isBackgrounded 字段

在 `src/tasks/types.ts` 里，`isBackgroundTask` 函数检查的是：

```typescript
if ('isBackgrounded' in task && task.isBackgrounded === false) {
  return false  // 这是前台任务，不作为后台任务处理
}
```

注意这里的逻辑：`isBackgrounded === false` 表示"这个任务目前是前台任务"。`isBackgrounded === true` 或者字段不存在（`!('isBackgrounded' in task)`），则被认为是后台任务。

这个字段是同一个任务对象上的一个布尔值。**前后台不是两种不同的对象，而是同一对象上一个字段的值。**

## 同步子 agent 也先注册成前台任务

根据 `src/tasks/LocalAgentTask/LocalAgentTask.tsx` 的设计，即使是同步执行的子 agent（不是后台运行的那种），也会先通过 `registerTask` 注册进 AppState，再开始执行。

这意味着：系统先认对象（任务被注册，有了 ID，有了可追踪的状态），再认姿态（这个任务现在是前台的还是后台的）。

这个顺序很关键：**先有对象，再有姿态。**

如果反过来——先决定姿态（这是后台任务），再创建对象——那么当姿态发生变化时，就需要在"后台任务对象"和"前台任务对象"之间做迁移，这正是引起状态丢失的操作。

## 前台和后台共享同一个 outputFile

任务切换到后台后，它的执行仍然在继续，输出仍然在写入 `outputFile`。

当用户把它调回前台时，前台的进度展示从 `outputFile` 里读取——读的是同一个文件，看到的是同一份进度记录，包括在后台期间产生的新输出。

这是"同一个对象"在技术层面的支撑：`outputFile` 的路径不变，内容持续更新，无论任务在前台还是后台。

## 任务面板 vs 主会话

Claude Code 里有两个"显示任务"的地方：

1. **主会话**：当任务是前台任务时，它的输出直接在主对话窗口里展示
2. **任务面板**：所有任务（前台和后台）都在任务面板里有条目

`isBackgroundTask` 函数控制的是"这个任务是否出现在后台任务指示器里"。前台任务（`isBackgrounded === false`）不出现在后台面板里，因为它已经在主会话里展示了。

当任务切换到后台（`isBackgrounded` 变为 `true`），它开始出现在后台面板里，主会话里的展示消失。

这个切换是 UI 层面的变化，不是数据层面的变化——任务对象本身没有任何东西消失。

## 为什么同步子 agent 也要这么设计

同步子 agent 是在主会话等待的情况下运行的——用户发出指令，主 agent 调用子 agent，等待子 agent 完成，再继续。

从用户角度看，这和"前台任务"是一样的：用户在等待，任务在执行，执行完了主会话继续。

如果同步子 agent 不注册成任务，它就是一个"看不见的操作"：用户不知道 AI 正在做什么，面板里没有这个任务的条目，如果任务失败了也没有历史记录。

注册成前台任务后，即使是同步子 agent，也在系统的视野里：用户能看到它在执行，面板里有条目，失败了有记录。

这是"系统先认对象"原则的另一个应用：所有执行过的工作都应该在系统里有身份，哪怕是同步执行的、几乎即时完成的工作。

## 面试要点

1. **isBackgrounded 是姿态字段，不是类型字段**：同一个任务对象通过修改这个字段在前后台之间切换，不是创建新对象。
2. **先注册再执行**：所有任务在执行前先注册进 AppState，确保从一开始就有身份，姿态变化不会导致身份重建。
3. **outputFile 是跨姿态的连续性基础**：无论前台还是后台，输出写入同一个文件，调回前台时能看到完整的历史进度。
4. **同步子 agent 也注册**：即使是同步执行的工作，也应该在系统里有可见的身份，这是"所有劳动都应该被认领"原则的体现。
