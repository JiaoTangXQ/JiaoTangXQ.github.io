# context: fork 是命令世界通往任务世界的桥

面试题版本：**"Claude Code 里有些命令会创建后台任务（比如启动一个长时间运行的 agent），这个从命令到任务的转换是怎么发生的？`context: fork` 在这里起什么作用？"**

## 命令世界和任务世界的区别

**命令世界**：用户输入命令，系统立即执行，在当前会话上下文里直接产生结果。命令是无状态的、一次性的。

**任务世界**：工作单元有完整的生命周期（pending → running → completed），有独立的状态对象，有 outputFile，可以在后台运行，可以被监控和中断。

有些工作本质上属于任务世界——它们持续时间长、需要独立追踪、可能需要在后台运行。但这些工作的入口可能来自命令世界（用户输入一个命令触发）。

## fork 的机制

`context: fork` 是一个上下文切换标记，表示"从当前会话上下文 fork 一个新的执行上下文，在那个上下文里运行后续工作"。

```typescript
// 概念描述：命令触发任务的流程
async function processCommand(command: Command, context: Context) {
  if (command.type === 'local-agent') {
    // fork: 创建新的任务上下文
    const taskContext = context.fork({
      taskId: generateTaskId(),
      parentId: context.sessionId,
    })
    
    // 在任务上下文里运行工作
    createAndRegisterTask(taskContext, command.definition)
    // 返回给用户"任务已创建"，不等待任务完成
    return { type: 'task_created', taskId: taskContext.taskId }
  }
}
```

fork 不是"复制整个状态"，而是"建立一个新的执行线程，有自己的 ID 和生命周期，但知道自己的父级是谁"。

## 为什么需要显式的 fork

不用 fork，直接在命令执行上下文里运行工作，会发生什么？

**命令上下文的生命周期问题**：命令上下文和用户请求绑定，请求处理完就结束。如果工作直接在命令上下文里跑，命令返回时工作也必须结束，无法在后台继续。

**追踪困难**：在命令上下文里运行的工作，系统不知道它的进度、状态、是否完成。用户看不到它的执行情况。

**中断传播**：如果用户取消了当前请求，命令上下文的 AbortController 会触发，在命令上下文里运行的所有工作都会被取消，即使用户只是想取消等待，而不是取消工作本身。

fork 建立了一个独立的执行上下文，有自己的 AbortController、自己的日志上下文、自己的生命周期。命令完成（返回"任务已创建"），工作继续在自己的上下文里运行。

## 层间交棒的含义

"命令世界通往任务世界的桥"说的是：fork 是两个世界之间的正式交棒点。

**命令世界负责**：解析用户意图、验证参数、决定是否应该创建任务、初始化任务配置。

**任务世界负责**：管理工作的生命周期、追踪进度、处理后台化、结果通知。

fork 在两者之间做了清晰的交接——命令世界把所有需要的信息（任务定义、权限配置、初始状态）传给任务世界，然后退出，不再参与任务的运行。任务世界从这个起点独立管理后续工作。

## 面试角度的总结

`context: fork` 是 Claude Code 里实现"命令触发任务"这个模式的具体机制。它建立了明确的层间边界：命令世界处理到"创建任务"为止，任务世界从"任务创建完成"开始接管。这种明确的交棒避免了命令逻辑和任务管理逻辑混在一起，让每一层的代码只关注自己的职责。
