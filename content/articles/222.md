# max_output_tokens 截断时，系统先升上限还是先插续写消息？

## 截断的两种情况

当模型输出被截断（`stop_reason: max_tokens`），有两种完全不同的原因：

1. **默认上限太低**：Claude API 对某些用户默认 max_output_tokens 是 8192，如果实际需要输出更长的内容，8k 很容易撞到
2. **已经用了更高上限还是不够**：即使升到 64k，模型还是被截断了

这两种情况需要不同的处理策略，不能混为一谈。

## 第一策略：max_output_tokens_escalate

```typescript
// query.ts
const capEnabled = getFeatureValue_CACHED_MAY_BE_STALE('tengu_otk_slot_v1', false)
if (
  capEnabled &&
  maxOutputTokensOverride === undefined &&  // 还没升过
  !process.env.CLAUDE_CODE_MAX_OUTPUT_TOKENS
) {
  state.transition = { reason: 'max_output_tokens_escalate' }
  state.maxOutputTokensOverride = ESCALATED_MAX_TOKENS  // 64k
  continue
}
```

条件：
- Feature flag 打开
- 这轮还没有手动覆盖过 max_output_tokens（`maxOutputTokensOverride === undefined`）
- 没有通过环境变量指定

如果满足这些条件，系统会**用同样的消息重发请求**，只是把 max_output_tokens 从默认值升到 `ESCALATED_MAX_TOKENS`（64k）。不插任何额外消息，模型不知道发生了什么。

这是最干净的处理方式：升限后重试，对模型完全透明。

## 第二策略：max_output_tokens_recovery

如果升限后还是截断了（或者第一次就已经用了高上限），走第二策略：

```typescript
// query.ts
const recoveryMessage = createUserMessage({
  content:
    `Output token limit hit. Resume directly — no apology, no recap of what you were doing. ` +
    `Pick up mid-thought if that is where the cut happened. Break remaining work into smaller pieces.`,
  isMeta: true,
})

state.transition = {
  reason: 'max_output_tokens_recovery',
  attempt: maxOutputTokensRecoveryCount + 1,
}
```

注意这条消息：
1. `isMeta: true` — 这是一条系统注入的消息，不是用户说的话，但模型会看到它
2. 明确要求「直接续写，不要道歉，不要回顾」— 这是为了防止模型花大量 token 说「我刚才因为输出限制被截断了，现在接着说…」
3. 「Break remaining work into smaller pieces」— 教模型调整策略，避免下一轮又被截断

这个策略最多重试 `MAX_OUTPUT_TOKENS_RECOVERY_LIMIT`（3）次。超过后，withheld 的 max_output_tokens 错误消息才会被真正 yield 出来让用户看到。

## 为什么先升限，再续写？

这两种策略有明显的优劣分级：

- **升限（escalate）**：完全透明，不影响上下文，代价最小
- **续写消息（recovery）**：会在 transcript 里留下系统注入的 meta 消息，增加上下文长度，有一定的连贯性风险

如果先注入续写消息，后来发现其实只是默认上限太低，那这条多余的 meta 消息就白白占用了 context 空间，而且可能让模型的输出变得奇怪（它在解释为什么要续写一个其实不存在的截断）。

所以顺序是：先升限（便宜、透明），不行再续写（有代价、会留痕）。

## isWithheldMaxOutputTokens 的作用

注意到这里有一个「withheld」的概念：

```typescript
function isWithheldMaxOutputTokens(msg: Message): msg is AssistantMessage {
  return msg?.type === 'assistant' && msg.apiError === 'max_output_tokens'
}
```

当检测到 max_output_tokens 截断时，Claude Code 不会立即把这个错误 yield 给上层（SDK 调用方）。它会先「扣押」这条消息，尝试 escalate 和 recovery。只有所有恢复策略都用尽后，才把扣押的错误消息释放出来。

原因是：如果立即 yield 错误，某些 SDK 调用方（如 Desktop 端）会看到 error 字段就直接终止会话——但实际上续写可能完全没问题。

## 面试问法

「你们系统里模型输出被截断了怎么处理？」

能答出：
1. 先区分默认上限不足 vs 真正超长（两种不同原因）
2. 升限策略（transparent retry）和续写策略（meta message injection）的区别
3. 为什么要 withheld 错误消息，而不是立即暴露给调用方

就是理解了这个机制的核心。
