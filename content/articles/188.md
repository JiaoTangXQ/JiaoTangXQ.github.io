# 任务对象先于前后台之分：为什么身份不能跟着姿态散

面试题版本：**"Claude Code 里任务在前台运行和在后台运行，使用的是同一个 Task 状态对象吗？如果是，前后台切换是怎么实现的？如果不是，两个对象之间如何保持同步？"**

## 答案是同一个对象

Claude Code 的前后台切换不会创建新的任务对象，也不会复制状态。前台和后台是同一个 TaskStateBase 的不同显示姿态。

看 `src/Task.ts` 里的状态类型：

```typescript
export type TaskStateBase = {
  id: string
  type: TaskType
  status: TaskStatus
  // ... 其他字段
}

export type TaskStatus = 'pending' | 'running' | 'completed' | 'failed' | 'killed'
```

注意 `status` 里没有 `foreground` 或 `background`。前后台是显示层的概念，不是任务生命周期的概念。

## 前后台切换实际改的是什么

当一个任务从前台切到后台，AppState 里的变化是：
1. `appState.currentTaskId`（当前前台聚焦的任务）切换到另一个任务或 null
2. TaskStateBase 本身不变，继续在 `appState.tasks` 里，`status` 仍然是 `'running'`

当任务从后台切回前台，方向反过来，但 Task 对象依然是同一个。

## 为什么这个顺序很重要

**如果反过来——先有前后台之分，再决定任务对象**，会发生什么？

假设系统把"前台任务"和"后台任务"实现为两种不同的数据结构：

```typescript
// 反面例子
type ForegroundTask = { id: string; status: TaskStatus; /* 前台专属字段 */ }
type BackgroundTask = { id: string; status: TaskStatus; /* 后台专属字段 */ }
```

前后台切换就必须进行状态迁移：
```typescript
// 前台转后台
const bgTask = convertToBackground(fgTask)
delete foregroundTasks[id]
backgroundTasks[id] = bgTask
```

问题是：迁移期间任务的状态处于什么状态？如果迁移出错，任务丢失了怎么办？如果两种结构的字段不完全一致，迁移后数据是否完整？

Claude Code 选择先有统一的任务对象，再在 UI 层决定显示姿态，这样前后台切换是 O(1) 的纯状态更新，没有数据迁移，没有一致性问题。

## 层间交棒的代价转移

"身份先站稳，显示姿态才有切换空间"这句话的反面是：如果每次切换都需要重建身份，就没有稳定的交棒点。

具体来说，假设一个子 agent 任务在运行中，用户想把它推到后台：

```
切换前：ForegroundAgentTask.id = "a_xyz"
切换后：BackgroundAgentTask.id = "a_xyz"（同一个对象，只是显示层面的变化）
```

因为 ID 不变，所有引用这个 ID 的地方（日志、outputFile 路径、通知记录）都不需要更新。

如果切换时要创建新对象，所有这些引用都要同步更新，或者两套 ID 之间要维护映射关系——这个隐性成本会随着系统规模增长而指数级增加。

## backgroundSignal 是哪来的

`src/Task.ts` 里有 `backgroundSignal` 这个字段用于前后台切换信号。这是一个 AbortController-like 的信号机制，不是一个布尔值，也不是一个状态枚举。

为什么用信号而不是直接修改 status？因为任务本身仍然在运行，`status` 还是 `'running'`。`backgroundSignal` 不是在改变任务的生命周期状态，而是在告知任务"你现在可以停止向前台报告进度了，切换到更安静的后台模式"。这是一种协议通信，不是状态覆盖。

## 面试角度的总结

Claude Code 的前后台切换能如此轻量，根本原因是**任务对象的身份先于显示姿态确立**。对象建好以后，前台还是后台只是 UI 层的关注点，不影响任务对象本身的完整性。这避免了状态迁移的复杂性，让所有依赖任务 ID 的系统（日志、通知、outputFile、历史查询）无需感知前后台切换的存在。
