# substitutePluginVariables：插件为什么需要变量替换？静态分发与运行时现场的桥梁

## 问题：一个插件安装在不同机器上，路径怎么可能是一样的？

```
Alice 的机器：/Users/alice/.claude/plugins/my-plugin/
Bob 的机器：   /home/bob/.claude/plugins/my-plugin/
CI 环境：     /var/jenkins/.claude/plugins/my-plugin/
```

如果插件正文里硬编码了路径，那它只能在一台机器上工作。

解决方案：变量替换。插件正文里写 `${CLAUDE_PLUGIN_ROOT}`，系统在装配时替换成实际安装路径。

---

## substitutePluginVariables 的工作方式

```typescript
// loadPluginCommands.ts 里的简化逻辑
function substitutePluginVariables(
  content: string,
  pluginRoot: string,
  pluginName: string,
): string {
  return content
    .replace(/\$\{CLAUDE_PLUGIN_ROOT\}/g, pluginRoot)
    .replace(/\$\{CLAUDE_PLUGIN_NAME\}/g, pluginName)
}
```

替换在**加载阶段**发生，不是调用阶段。这意味着：
- 替换后的内容被存进 `Command` 对象的 `markdownContent` 字段
- 每次调用直接使用替换后的结果，不重复计算
- 如果插件被移动到新路径，需要重新加载才能更新

## substituteUserConfigInContent：用户级配置注入

除了插件根路径，`loadPluginCommands.ts` 还支持注入用户级配置变量：

```typescript
function substituteUserConfigInContent(
  content: string,
  userConfig: Record<string, string>,
): string {
  for (const [key, value] of Object.entries(userConfig)) {
    content = content.replace(
      new RegExp(`\\$\\{${escapeRegExp(key)}\\}`, 'g'),
      value
    )
  }
  return content
}
```

这让插件可以使用用户在 Claude Code 配置里设置的变量：

```yaml
# 用户配置
plugins:
  variables:
    MY_API_ENDPOINT: https://api.internal.company.com
```

```markdown
<!-- 插件内容 -->
调用接口：${MY_API_ENDPOINT}/v1/review
```

装配后：
```
调用接口：https://api.internal.company.com/v1/review
```

这让同一个插件可以在不同公司/团队的环境下工作，只需修改用户配置，不需要修改插件本身。

## 替换顺序的重要性

如果用户配置里有一个变量叫 `CLAUDE_PLUGIN_ROOT`，会和内置变量冲突吗？

安全的设计是：先替换内置变量，再替换用户配置变量。这样内置变量不会被用户配置覆盖。（如果顺序反过来，用户可以通过设置 `CLAUDE_PLUGIN_ROOT` 为任意值来"污染"插件的路径替换。）

---

## allowed-tools 也要替换

变量替换不只发生在正文内容里，`allowed-tools` 字段也要替换：

```typescript
// 替换前
const allowedToolsRaw = frontmatter['allowed-tools'] as string | undefined
// 值可能是: "run:${CLAUDE_PLUGIN_ROOT}/scripts/lint.sh"

const allowedTools = allowedToolsRaw
  ? substitutePluginVariables(allowedToolsRaw, pluginRoot, pluginName)
  : []
// 替换后: "run:/home/alice/.claude/plugins/my-plugin/scripts/lint.sh"
```

这很关键：权限系统用 `allowedTools` 判断这个技能是否有权调用某个工具。如果 `allowed-tools` 里有路径变量但没有被替换，权限检查就会失败——技能声明允许运行 `${CLAUDE_PLUGIN_ROOT}/scripts/lint.sh`，但实际命令是 `/home/alice/...`，二者不匹配。

---

## 调试难点：磁盘上的文件和运行时的内容不一样

这是变量替换引入的最大调试成本：用户看到的命令内容（运行时 prompt）和磁盘上的文件内容不一样。

一个技能正文写的是：
```markdown
运行：${CLAUDE_PLUGIN_ROOT}/scripts/check.sh
```

但模型看到的是：
```markdown
运行：/Users/alice/.claude/plugins/my-plugin/scripts/check.sh
```

当技能行为异常时，用户打开磁盘文件查看"怎么写的"，却看不到运行时实际传给模型的内容。这种"代码和行为脱节"是模板系统的通病，变量替换只是其中最温和的一种。

## 面试指导

**"如何设计一个可分发的插件系统，使同一个插件包在不同环境下都能正常工作？"**

关键点：

1. **分离静态内容和动态路径**：插件包里只放可移植的内容，路径在安装/激活时注入
2. **替换时机的选择**：加载时替换（性能好，但需要重加载才能更新）vs 调用时替换（灵活，但每次都有开销）
3. **权限字段也要替换**：不只是正文内容，所有可能包含路径的字段都要统一处理
4. **调试可观察性**：系统应该提供"查看当前替换后的实际内容"的手段，否则调试困难

Claude Code 的选择是加载时替换——对技能加载频率和插件移动频率的权衡来说，这是合理的。
