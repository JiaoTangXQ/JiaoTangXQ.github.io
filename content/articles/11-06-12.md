# showActivity 的三个条件：热度只给真正还能推进的任务

`TaskListV2.tsx` 里：

```typescript
const showActivity = isInProgress && !isBlocked && activity;
```

三个条件同时成立，才显示任务的活动摘要。

## 为什么被阻塞的任务不显示活动

被阻塞的任务（`openBlockers.length > 0`）处于等待状态——它有依赖，那些依赖还没完成，它自己不能继续推进。

在这种状态下，显示「它最近在做什么」的活动摘要会传递错误信号。用户看到一段活动文本，自然会认为这个任务还在推进。但实际上它被卡住了，那段活动摘要描述的是它停下来之前的状态。

这种信息不对等比没有信息更危险。

## 注意力是一种分配

任务面板展示的是当前工作的整体状态。用户扫一眼，应该能快速判断：哪些任务还在推进，哪些需要我关注，哪些已经完成了。

如果被阻塞的任务也有活动热度，「热度」就失去了信号价值。每条任务都在发光，就等于没有任务在发光，因为用户无法区分「真的在跑」和「看起来在跑但其实停了」。

`!isBlocked` 这个条件在做的就是保持「活动热度」的信号纯度：看到活动摘要的地方，就是还在推进的地方。没有活动摘要的地方，要么已经完成，要么被卡住了——后者会显示阻塞信息，这是另一种信号。

## 两种信号不应该混用

`isBlocked` 为 true 时，任务行会有自己的阻塞标记（变灰、显示阻塞原因）。这是一种完全不同的信号。如果同时还显示活动摘要，用户就得自己判断「这个任务到底是活跃的还是被卡住的」。

两种状态使用不同的视觉语言，用户的认知负担才能保持在可接受的水平。
