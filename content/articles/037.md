# stale-while-revalidate 在 Claude Code 里怎么用？memoizeWithTTL 源码解析

大多数缓存是"过期了就阻塞等新值"。但如果缓存失效发生在请求关键路径上，这个阻塞会让用户感受到延迟。

`utils/memoize.ts` 实现了三种缓存策略，其中 `memoizeWithTTL` 和 `memoizeWithTTLAsync` 用的是 **stale-while-revalidate（SWR）** 模式：缓存过期后立即返回旧值，同时在后台刷新，下次调用时拿到新值。

## 同步版 memoizeWithTTL

```typescript
export function memoizeWithTTL<Args, Result>(
  f: (...args: Args) => Result,
  cacheLifetimeMs: number = 5 * 60 * 1000,  // 默认 5 分钟
): MemoizedFunction<Args, Result> {
  const cache = new Map<string, CacheEntry<Result>>()

  return (...args) => {
    const key = jsonStringify(args)
    const cached = cache.get(key)
    const now = Date.now()

    if (!cached) {
      // Cold miss: 同步计算并缓存
      const value = f(...args)
      cache.set(key, { value, timestamp: now, refreshing: false })
      return value
    }

    if (now - cached.timestamp > cacheLifetimeMs && !cached.refreshing) {
      // Stale: 标记刷新中，后台异步刷新，立即返回旧值
      cached.refreshing = true
      Promise.resolve()
        .then(() => {
          const newValue = f(...args)
          if (cache.get(key) === cached) {  // Identity guard
            cache.set(key, { value: newValue, timestamp: Date.now(), refreshing: false })
          }
        })
        .catch(e => {
          logError(e)
          if (cache.get(key) === cached) {
            cache.delete(key)  // 刷新失败则删除缓存，下次 cold miss
          }
        })
      return cached.value  // 立即返回旧值
    }

    return cached.value  // 缓存新鲜，直接返回
  }
}
```

三个路径：
1. **Cold miss**：首次调用，同步计算，不走 stale 路径
2. **Stale hit**：缓存过期，立即返回旧值 + 后台刷新
3. **Fresh hit**：缓存新鲜，立即返回

## identity guard 防止 race condition

注释里提到的 identity guard (`cache.get(key) === cached`) 很重要：

场景：缓存过期了，开始后台刷新。在刷新的 microtask 执行之前，用户调用了 `cache.clear()`。然后刷新的 microtask 完成了，它想写回缓存。

如果不检查：刷新结果覆盖了 `clear()` 之后的新缓存（或者写入到一个已经被清空的 Map 里，重新填入了旧数据）。

如果检查 `cache.get(key) === cached`：`clear()` 后 `cache.get(key)` 返回 `undefined`，不等于 `cached`，刷新结果被丢弃。

注释对两种错误的权重也有明确说明：

> `.then` 用旧刷新结果覆盖更糟糕——错误数据会持续整个 TTL。
> `.catch` 删除缓存更好——自我修正，下次会重新计算。

## 异步版的 in-flight deduplication

`memoizeWithTTLAsync` 有一个同步版没有的问题：并发 cold miss。

如果 10 个调用同时发现缓存为空，同步版因为是同步函数，第一个调用会先执行完并写入缓存，后续调用直接命中。但异步版在 `await f(...)` 前就 `return`，第二个调用来的时候缓存还没填好，会触发第二次 `f()` 调用。

```typescript
const inFlight = new Map<string, Promise<Result>>()

const memoized = async (...args) => {
  const key = jsonStringify(args)
  if (!cache.get(key)) {
    const pending = inFlight.get(key)
    if (pending) return pending  // 复用已有的 in-flight Promise
    const promise = f(...args)
    inFlight.set(key, promise)
    try {
      const result = await promise
      // ...写入缓存
      return result
    } finally {
      inFlight.delete(key)
    }
  }
  // ...
}
```

`inFlight` Map 用 Promise 去重：10 个并发 cold miss，第一个创建 Promise 并存入 `inFlight`，后续 9 个拿到同一个 Promise 直接返回。结果只有一次真正的 `f()` 调用。

注释里还举了一个具体的真实案例：如果不做去重，并发的 `refreshAndGetAwsCredentials` 调用会各自触发 `aws sso login` 进程，用户会看到 N 个登录窗口弹出。

## 面试角度

这几个缓存工具体现了实际工程里的缓存设计层次：

1. **SWR 适合读多写少的配置缓存**：feature flags、凭证、模型设置——这些每次启动都需要，但过期数据通常也可以用
2. **identity guard 是处理异步 race condition 的标准模式**：在异步操作完成后写回状态时，总要检查"我想写的地方还是我想写的那个地方吗"
3. **in-flight deduplication 防止并发 stampede**：N 个并发 cold miss 只应该触发 1 次实际请求
4. **`refreshing` 标志防止并发刷新**：不加这个标志，多次调用 stale 数据会各自触发刷新，浪费资源
