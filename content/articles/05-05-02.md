# progress.summary 的两个写入者：为什么要原地保留而不是覆盖？

`LocalAgentTaskState.progress` 字段的类型是 `AgentProgress`：

```typescript
export type AgentProgress = {
  toolUseCount: number
  tokenCount: number
  lastActivity?: ToolActivity
  recentActivities?: ToolActivity[]
  summary?: string          // 这个字段有两个来源
}
```

`summary` 这一个字段有两个互相独立的写入者：

1. **`updateAgentProgress()`**：从实时 assistant 消息流里更新进度数字和活动记录
2. **`updateAgentSummary()`**：后台周期性摘要服务写入的 1-2 句人类可读概述

如果这两个函数都直接覆盖 `progress` 对象，它们就会互相踩掉对方的数据。Claude Code 的解决方案是：每个写入者只覆盖自己负责的部分，显式保留另一方写入的数据。

## updateAgentProgress：保留已有的 summary

```typescript
export function updateAgentProgress(
  taskId: string,
  progress: AgentProgress,
  setAppState: SetAppState,
): void {
  updateTaskState<LocalAgentTaskState>(taskId, setAppState, task => {
    if (task.status !== 'running') return task

    // 保留后台摘要服务已经写入的 summary
    const existingSummary = task.progress?.summary

    return {
      ...task,
      progress: existingSummary
        ? {
            ...progress,
            summary: existingSummary,  // 用旧 summary 覆盖新 progress 的 summary（可能为 undefined）
          }
        : progress,
    }
  })
}
```

传入的 `progress` 是从消息流新算出来的，里面不含 `summary`（因为摘要是后台服务单独生成的）。如果不做保留，每次进度更新都会把 `summary` 字段清掉。

## updateAgentSummary：只更新 summary

```typescript
export function updateAgentSummary(
  taskId: string,
  summary: string,
  setAppState: SetAppState,
): void {
  updateTaskState<LocalAgentTaskState>(taskId, setAppState, task => {
    if (task.status !== 'running') return task

    return {
      ...task,
      progress: {
        ...task.progress,          // 保留所有现有进度数据
        toolUseCount: task.progress?.toolUseCount ?? 0,
        tokenCount: task.progress?.tokenCount ?? 0,
        summary,                   // 只覆盖 summary
      },
    }
  })
}
```

摘要服务只关心 `summary` 字段，其他进度数字它不动。

## 两条数据流的频率和来源

**实时进度流**（`updateAgentProgress`）：
- 触发时机：每次 Agent 的 assistant 消息到来
- 数据来源：消息里的 `usage` 字段（token 数）和 `content` 里的 `tool_use` 块
- 频率：可能每几秒就更新一次（取决于 Agent 的回合速度）
- 粒度：精细，每次工具调用都记录

**后台摘要流**（`updateAgentSummary`）：
- 触发时机：后台摘要服务周期性调用（`startAgentSummarization()`）
- 数据来源：专门的摘要模型，读取任务进度后生成自然语言概述
- 频率：低频（可能每隔几十秒才更新一次）
- 粒度：粗略，一两句话的人类可读概述

这两种数据的粒度和频率完全不同，无法合并成一条流。实时进度适合显示"刚用了 12 个工具，token 共 4521"，摘要适合显示"它目前已经分析了 3 个文件，正在查找相关的函数调用"。

## SDK 事件发送：摘要对外可观察

`updateAgentSummary` 里还有一段代码，在更新本地状态的同时，把摘要发送给 SDK 消费者：

```typescript
if (captured && getSdkAgentProgressSummariesEnabled()) {
  const { tokenCount, toolUseCount, startTime, toolUseId } = captured
  emitTaskProgress({
    taskId,
    toolUseId,
    description: summary,
    startTime,
    totalTokens: tokenCount,
    toolUses: toolUseCount,
    summary,
  })
}
```

这个"发给 SDK 消费者"的逻辑只在 `updateAgentSummary` 里，不在 `updateAgentProgress` 里。原因是：SDK 消费者（比如 VS Code 插件的子 Agent 面板）订阅的是概括性进度，而不是每一次工具调用的原始数据。频繁发送实时进度会造成 SDK 消费者端的不必要重渲染。

## 面试要点

这是一个典型的**多写入者共享状态**问题。解决方案不是加锁，也不是引入消息队列，而是**按字段划分写入所有权**：

- 实时进度流拥有 `toolUseCount`、`tokenCount`、`recentActivities` 的写入权
- 摘要流拥有 `summary` 的写入权
- 每个写入者在更新时显式保留另一方的数据

这种模式简单、直接，没有引入额外的同步机制，代价是每个写入者都需要知道"还有谁也在写这个对象"，并主动做出保留。

当多个写入者共享一个对象时，比起让最后一个写入者覆盖所有数据（简单但危险），更健壮的做法是明确每个写入者的"所有权字段"，并在写入时保留非自己所有权的字段。
