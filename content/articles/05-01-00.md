# 7 种任务类型的 discriminated union：Claude Code 如何统一管理异构任务

`src/Task.ts` 的第 6 行是整个多 Agent 框架的地基：

```typescript
export type TaskType =
  | 'local_bash'
  | 'local_agent'
  | 'remote_agent'
  | 'in_process_teammate'
  | 'local_workflow'
  | 'monitor_mcp'
  | 'dream'
```

7 种类型，字符串字面量联合，用作 discriminated union 的判别字段。不是枚举，不是类继承，不是接口约束。为什么？

## 类型系统深潜

这 7 种类型各自背后的执行方式完全不同：

- `local_bash`：fork 一个子进程跑 shell 命令
- `local_agent`：在当前进程里启动一个完整 Agent 会话
- `remote_agent`：发到远端服务器执行的 Agent
- `in_process_teammate`：进程内以 Team 模式组建的队友 Agent
- `local_workflow`：脚本化工作流（feature flag 保护，按需加载）
- `monitor_mcp`：监控 MCP 协议的特殊任务
- `dream`：用途看名字就能猜到是某种实验性任务

它们的执行环境、生命周期、需要的额外字段各不相同。但它们都必须共享同一套基础字段：`id`、`status`、`outputFile`、`notified`。

这正是 discriminated union + 共享 base type 的适用场景：

```typescript
export type TaskStateBase = {
  id: string
  type: TaskType
  status: TaskStatus
  // ... 其他公共字段
}
```

具体任务用 `TaskStateBase & { type: 'local_agent'; agentId: string; ... }` 的形式扩展。TypeScript 编译器在 switch on `task.type` 时能自动收窄类型，无需强制转换。

## 为什么不用继承

面向对象的第一反应是定义 `abstract class BaseTask`，然后 `class LocalAgentTask extends BaseTask`。这有几个问题：

**第一，React 状态不喜欢类实例。** Claude Code 把任务状态存在 `AppState` 里，而 `AppState` 是 React 风格的不可变状态树。类实例不能安全地被 spread、被 immer 之类的工具处理，也很难做引用相等性比较来决定要不要重渲染。

**第二，运行时 discriminant 比继承链更直接。** `task.type === 'local_agent'` 是 O(1) 字符串比较，`instanceof LocalAgentTask` 需要原型链查找，在序列化/反序列化（比如从磁盘加载任务状态）后还会断掉。

**第三，接口比实现更稳定。** `Task` 接口只约定 `name`、`type`、`kill` 三个字段，任务实现可以完全不同，只要满足这个最小契约。继承反而会引入更强的耦合。

## `getAllTasks()` 的注册模式

`src/tasks.ts` 展示了另一个有趣设计：

```typescript
export function getAllTasks(): Task[] {
  const tasks: Task[] = [
    LocalShellTask,
    LocalAgentTask,
    RemoteAgentTask,
    DreamTask,
  ]
  if (LocalWorkflowTask) tasks.push(LocalWorkflowTask)
  if (MonitorMcpTask) tasks.push(MonitorMcpTask)
  return tasks
}
```

注意注释："Returns array inline to avoid circular dependency issues with top-level const"。这不是随手写法，而是为了避免模块循环依赖。`tasks.ts` 导入各任务实现，各任务实现可能导入 `Task.ts` 的类型，如果用顶层常量而不是函数，初始化顺序会出问题。

`LocalWorkflowTask` 和 `MonitorMcpTask` 用 `feature()` 函数做 feature flag 判断，动态 `require()`。这是 Bun bundle 的能力，允许在生产构建时按 flag 剔除未启用的代码路径，减小包体积。

## 完整生命周期：一个任务的旅程

理解 7 种任务类型的意义，得看它们在系统里走的是同一条路：

1. 任何任务创建时都调用 `createTaskStateBase(id, type, description, toolUseId)`，拿到统一骨架
2. `registerTask(taskState, setAppState)` 把它放进 `AppState.tasks` 字典
3. 框架用 `isTerminalTaskStatus(task.status)` 判断它是否结束
4. 结束后 `evictTerminalTask()` 根据 `notified` 和 `evictAfter` 决定何时从内存里删掉

整条路上，`task.type` 是 7 种之一，但路本身是同一条。

## 面试要点

被问到"如何设计一个支持多种任务类型的系统"时：

**答案的关键不是选择哪种数据结构，而是先想清楚两件事：**

1. 所有任务类型有哪些公共管理需求（ID、状态、输出、通知）？把它们提到 base type。
2. 各类型的差异在哪里？用 discriminated union 让编译器帮你在类型层面保证不同类型的处理完整性。

继承可以解决代码复用，但 discriminated union 解决的是**类型安全的异构集合管理**，在状态管理场景下通常更合适。
