# SAFE_BRIDGE_STRING_KEYS：三个词决定了什么信息能离开现场

```typescript
const SAFE_BRIDGE_STRING_KEYS = new Set([
  'bridge_status',
  'error_type',
  'tool_name',
])
```

这个集合只有三个元素。但它的意义超过它的体积。

`trackEvent()` 在处理遥测 metadata 时做的事情是：布尔值和数字直接通过，字符串只有键名在这个集合里才通过，其他字符串静默丢弃。另外，`status` 这个键名会被改写成 `bridge_status`，避免触发 Datadog 的保留字段。

## 为什么字符串比数字危险

布尔值告诉你「成功还是失败」，数字告诉你「用了多少毫秒」，这些信息是结构化的，它们描述系统状态，不携带现场内容。

字符串不同。字符串可以是任何东西：错误消息里可能嵌有用户的文件路径，状态描述里可能带着页面标题，调试字段里可能有 URL 片段。浏览器桥比主进程更靠近真实世界——它接触的是用户正在浏览的页面内容，是 DOM 元素，是 JavaScript 执行结果。这里产生的字符串携带隐私信息的概率远高于系统内部。

`SAFE_BRIDGE_STRING_KEYS` 的逻辑是：除非字段名明确在白名单里，否则假设字符串值是不安全的。这是一种防御性假设，比「假设安全直到证明不安全」更保守，也更可靠。

## 白名单里的三个词意味着什么

`bridge_status` 描述连接状态（`connected`、`disconnected`），`error_type` 描述错误分类，`tool_name` 描述工具名称。这三种信息的共同特征：它们是枚举型的，取值空间有限，不会携带用户现场的任意文本。

其他一切——错误消息、URL、页面内容摘要、DOM 路径——统统不在白名单里。即使是排障时非常有用的信息，在没有明确分类为安全之前，也不进入分析系统。

这是「观测便利」和「隐私边界」之间做选择时，选了后者的一个具体例子。
