# 进场身份先决：Claude Code 是如何在启动时决定「这次我是谁」的？

## 问题的形状

同一行命令 `claude`，在以下三种情况下会启动出不同的东西：

1. 本地终端直接运行
2. 通过 `--remote` 标志连接远程会话
3. 通过 mobile bridge 发来的请求触发

这三种情况下，可用的命令集合不同，允许执行的操作范围不同，甚至「我能不能运行 /model 这个命令」的答案都不同。

**这不是运行时权限检查的结果，而是进场时的身份就已经决定好了。**

## REMOTE_SAFE_COMMANDS 是如何过滤的

`src/commands.ts` 里有一个 `REMOTE_SAFE_COMMANDS` 集合：

```typescript
export const REMOTE_SAFE_COMMANDS: Set<Command> = new Set([
  session,
  exit,
  clear,
  help,
  theme,
  color,
  vim,
  cost,
  usage,
  copy,
  btw,
  feedback,
  plan,
  keybindings,
  statusline,
  stickers,
  mobile,
])
```

当 Claude Code 以 `--remote` 模式启动时，`filterCommandsForRemoteMode()` 会在 REPL 渲染之前就把命令列表过滤到这个集合。这不是每次命令执行时检查权限——而是在进场的那一刻，就把可用命令集合固定下来。

## bridgeOrigin 与 BRIDGE_SAFE_COMMANDS

Bridge 场景（mobile/web client 通过远程控制桥发来的命令）有另一套约束。`isBridgeSafeCommand()` 的判断逻辑是：

```typescript
export function isBridgeSafeCommand(cmd: Command): boolean {
  if (cmd.type === 'local-jsx') return false  // 需要终端 UI，不适合 bridge
  if (cmd.type === 'prompt') return true       // 展开成文本，安全
  return BRIDGE_SAFE_COMMANDS.has(cmd)         // local 命令需要显式列入白名单
}
```

`local-jsx` 命令会弹出终端界面（Ink UI），在 bridge 场景下没有渲染目标，所以直接阻断。`prompt` 类型的命令（技能文件展开成提示词）是安全的，不需要检查。只有 `local` 类型的命令需要显式列入 `BRIDGE_SAFE_COMMANDS`。

## 目录决定项目身份

`State` 里有两个路径字段，名字类似但语义不同：

- `originalCwd`：启动时的工作目录，永不改变
- `projectRoot`：项目根目录，启动时设定，mid-session 不被更新

注释里明确：「Stable project root — set once at startup... Use for project identity (history, skills, sessions) not file operations.」

这个区分的含义是：文件操作可以跟随 `cwd` 变化（用户可以在会话中切换目录），但历史记录、技能发现、会话恢复——这些基于「项目」语义的功能，始终锚定在启动时确定的 `projectRoot`。

进场时确定的目录，决定了「这次会话属于哪个项目」，这是会话整个生命周期内不会改变的身份要素。

## 连接方式改写能力集合

不只是命令集合，连接方式也会影响能力边界。

当 Claude Code 作为远程会话服务端运行时，某些需要本地终端的能力（比如弹出终端 UI 进行模型选择）就无法使用。当它作为 bridge 服务端运行时，文件系统操作需要额外确认当前目录是否在允许范围内。

这种「能力边界随连接方式改变」不是一种动态的运行时限制，而是在建立连接时就确定的系统形态。

## 为什么身份要先于能力讨论

一种常见的系统设计是：先把所有能力都加载起来，运行时再按权限决定哪些可以用。

Claude Code 的做法相反：在进场的那一刻（启动参数处理、连接类型确定、目录分析），就把这次会话的「身份」固定下来，再由身份决定可用的能力集合。

这样设计的好处是可预测：用户或脚本看到的能力集合，是在进场时就已经稳定的，不会因为某个中间状态的权限检查而出现「有时能用有时不能用」的情况。

代价是需要在多个地方维护这些白名单（`REMOTE_SAFE_COMMANDS`、`BRIDGE_SAFE_COMMANDS`、`effectiveSkipSlash` 等），每次新增能力都要考虑它在不同进场身份下的表现。

## 面试角度

「你们怎么做能力隔离」是一个常见的系统设计题。

浅层答案是「运行时权限检查」。Claude Code 的答案更深一层：**身份在进场时确定，能力集合是身份的属性，不是每次操作都重新评估的结果。**

这个区别在长期运行、多入口的系统里很重要——运行时检查容易出现时序漏洞，进场身份固定则让系统的边界在任意时刻都是明确的。
