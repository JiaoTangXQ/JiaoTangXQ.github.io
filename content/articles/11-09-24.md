# sticky 提示先截首段再限长：线索不该膨胀成正文副本

Sticky header 的职责是：当用户往下滚动时，告知「当前上下文的出发点是什么」，让用户不必滚回去才能想起这段对话从哪里开始。

这个职责对应的是一条**线索**，不是一份**副本**。

## 两步压缩

`StickyTracker` 推送 prompt 文本前做两件事：

**截首段**：取换行前的第一段。很多 prompt 都是分段的：第一段描述任务，后面的段落提供细节。第一段通常是最能召回语境的部分——「优化这个函数」比「具体地说，我希望优化循环里的内存分配，另外…」更适合出现在 header 里。

**压到 STICKY_TEXT_CAP 以内**：即使第一段也可能很长，比如一个详细的任务描述。`STICKY_TEXT_CAP` 确保 header 不会因为这一行过长而挤占整个屏幕顶部。

## 不压缩会怎样

想象一个有 5 行长的 prompt，完整出现在 sticky header 里：
- 它会占据 5 行屏幕空间
- 消息区实际可见内容减少
- 用户仍然在「读」prompt，而不是在用 prompt 作为上下文背景快速定向

当 sticky header 变成另一块内容区，它就失去了「线索」的功能，反而成为了新的阅读负担。

## 线索的信息密度

好的线索高度压缩、高度可辨识。「优化这个函数的性能」这 11 个字就能让人想起「哦，我在让 Claude 优化代码」，不需要完整的 50 字 prompt。

`trimStart + 首段 + STICKY_TEXT_CAP` 这三步在共同维护线索应有的信息密度：刚好足够召回语境，不多余，不膨胀。
