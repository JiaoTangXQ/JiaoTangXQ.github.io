# 治理字段先于执行字段：TaskStateBase 的设计优先级

`TaskStateBase` 只有 8 个字段：

```typescript
export type TaskStateBase = {
  id: string
  type: TaskType
  status: TaskStatus
  description: string
  toolUseId?: string
  startTime: number
  endTime?: number
  totalPausedMs?: number
  outputFile: string
  outputOffset: number
  notified: boolean
}
```

没有 `prompt`，没有 `model`，没有 `messages`，没有任何"这个任务具体在做什么"的字段。全是**治理字段**：这个任务是什么类型、现在处于什么状态、输出在哪里、通知发了没有。

这个优先级是 Claude Code 架构的核心选择。

## 为什么治理字段先于执行字段

执行字段（prompt、model、messages）只服务一种任务类型，`local_agent` 有 prompt，`local_bash` 有 command，`remote_agent` 有不同的执行上下文。如果把这些字段放进 base type，每种任务都得带上一堆用不到的字段。

治理字段不同——它们服务的不是"这个任务怎么跑"，而是"所有任务怎么被统一管理"：

**`outputFile` + `outputOffset`**：决定了日志在哪里、读到第几个字节。框架的增量读取逻辑 `getTaskOutputDelta(taskId, outputOffset)` 只依赖这两个字段，不需要知道任务是什么类型。

**`notified`**：决定了完成通知是否已发。`enqueueAgentNotification()` 在发通知前先检查 `task.notified`，通过一个原子更新同时设置标志并判断是否需要真正发送，防止重复通知问题。

**`status`**：决定了系统还能不能对任务做事。`isTerminalTaskStatus(task.status)` 是整个系统里的"终态判断器"，消息注入保护、回收逻辑、面板驱逐全依赖它。

**`startTime` + `endTime`**：决定了任务列表的排序，以及任务运行了多久。

## 两层叠加而不是一层包含

这个设计的实现方式是**类型叠加**：

```typescript
// 骨架层：所有任务共有
export type TaskStateBase = { id, type, status, outputFile, ... }

// 执行层：子 Agent 特有
export type LocalAgentTaskState = TaskStateBase & {
  type: 'local_agent'     // 覆盖为字面量类型，收窄 discriminant
  agentId: string
  prompt: string
  progress?: AgentProgress
  pendingMessages: string[]
  retain: boolean
  evictAfter?: number
  // ...
}
```

`createTaskStateBase()` 负责填骨架：

```typescript
export function createTaskStateBase(id, type, description, toolUseId): TaskStateBase {
  return {
    id,
    type,
    status: 'pending',
    description,
    toolUseId,
    startTime: Date.now(),
    outputFile: getTaskOutputPath(id),
    outputOffset: 0,
    notified: false,
  }
}
```

具体任务在这个骨架上追加自己的执行字段：

```typescript
const taskState: LocalAgentTaskState = {
  ...createTaskStateBase(agentId, 'local_agent', description, toolUseId),
  type: 'local_agent',
  status: 'running',  // 立刻覆盖成运行态
  agentId,
  prompt,
  // 执行层专有字段...
}
```

## 这个顺序产生的连锁效应

把治理字段放在 base type 的直接结果，是框架层代码**完全不需要了解执行细节**。

`framework.ts` 里的 `generateTaskAttachments()` 在轮询所有任务时，只操作 `taskState.outputOffset`、`taskState.status`、`taskState.notified` 这些公共字段：

```typescript
for (const taskState of Object.values(tasks)) {
  if (taskState.notified) {
    if (isTerminalTaskStatus(taskState.status)) {
      evictedTaskIds.push(taskState.id)
      continue
    }
  }
  if (taskState.status === 'running') {
    const delta = await getTaskOutputDelta(taskState.id, taskState.outputOffset)
    if (delta.content) {
      updatedTaskOffsets[taskState.id] = delta.newOffset
    }
  }
}
```

这段代码对 `local_bash`、`local_agent`、`remote_agent` 一视同仁。它不需要知道任务是在跑 shell 命令还是在调模型，只需要知道输出偏移在哪里、任务是否已结束。

## 面试要点

被问到"如何设计通用基类"时，Claude Code 的做法提供了一个反常识的思路：**基类应该包含管理接口，而不是执行接口**。

执行接口的共性往往是假的——`local_bash` 和 `local_agent` 的"执行"方式天差地别，硬塞进同一个接口只会产生大量空实现或者 `if` 分支。

管理接口的共性是真实的——所有任务都需要被识别（`id`）、被追踪（`status`）、被观察（`outputFile`）、被通知（`notified`）。这些字段放进基础类型，才是真正意义上的公共抽象。
