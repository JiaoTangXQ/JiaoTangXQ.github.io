# Ultrathink 和 Extended Thinking：Claude Code 的思考模式是怎么管理的？

Claude Code 的扩展思考（Extended Thinking）功能不是简单的开/关开关，而是一套多层次的配置和检测体系。`utils/thinking.ts` 定义了这套体系的核心逻辑。

## ThinkingConfig 类型系统

```typescript
export type ThinkingConfig =
  | { type: 'adaptive' }
  | { type: 'enabled'; budgetTokens: number }
  | { type: 'disabled' }
```

三种思考模式：
- `adaptive`：让模型自己决定是否思考、思考多久（Adaptive Thinking 特性）
- `enabled`：强制启用，给定 token 预算上限
- `disabled`：关闭思考

`adaptive` 和 `enabled` 的区别很重要：`enabled` 会传 `thinking.budget_tokens` 参数给 API，模型会尽量使用这个预算；`adaptive` 不传预算，模型按需决定。

## Ultrathink：关键词触发高预算思考

```typescript
export function hasUltrathinkKeyword(text: string): boolean {
  return /\bultrathink\b/i.test(text)
}
```

当用户输入包含 `ultrathink` 关键词时，Claude Code 会切换到最高思考预算。这个关键词不区分大小写，用 `\b` 词边界避免"ultrathinking"这样的词触发。

feature gate + GrowthBook 双重控制：

```typescript
export function isUltrathinkEnabled(): boolean {
  if (!feature('ULTRATHINK')) {
    return false
  }
  return getFeatureValue_CACHED_MAY_BE_STALE('tengu_turtle_carbon', true)
}
```

`feature('ULTRATHINK')` 是构建时的 dead code elimination gate（外部构建可能关闭），`tengu_turtle_carbon` 是运行时的 GrowthBook feature flag（可以远程推送开关）。注意 GrowthBook 默认值是 `true`——在网络不可用时默认开启。

## findThinkingTriggerPositions：用于 UI 高亮

```typescript
export function findThinkingTriggerPositions(text: string): Array<{
  word: string
  start: number
  end: number
}> {
  // 每次调用都新建 /g 正则，避免共享 lastIndex 导致的状态污染
  const matches = text.matchAll(/\bultrathink\b/gi)
  // ...
}
```

这个函数返回 `ultrathink` 在用户输入里的所有位置，供 UI 用彩虹色高亮显示（RAINBOW_COLORS 数组：红、橙、黄、绿、蓝、靛、紫循环）。

注释解释了一个 JavaScript 正则的细节：用 `String.prototype.matchAll` 共享一个带 `/g` 的正则实例时，上一次调用的 `lastIndex` 会泄漏到下一次调用。函数里每次都创建新的正则字面量，避免了这个状态污染问题。

## 模型支持检测的两层逻辑

不是所有 Claude 模型都支持思考：

```typescript
export function modelSupportsThinking(model: string): boolean {
  const supported3P = get3PModelCapabilityOverride(model, 'thinking')
  if (supported3P !== undefined) {
    return supported3P
  }
  const canonical = getCanonicalName(model)
  const provider = getAPIProvider()
  // 1P 和 Foundry：所有 Claude 4+ 模型（包括 Haiku 4.5）
  if (provider === 'foundry' || provider === 'firstParty') {
    return !canonical.includes('claude-3-')
  }
  // 3P（Bedrock/Vertex）：只有 Opus 4+ 和 Sonnet 4+
  return canonical.includes('sonnet-4') || canonical.includes('opus-4')
}
```

两层逻辑：
1. 第三方模型可以通过 `get3PModelCapabilityOverride` 显式覆盖（能力探测结果存储）
2. 没有覆盖时，1P/Foundry 采用"4+ 全支持"规则，Bedrock/Vertex 采用更保守的"只有特定高端型号"规则

**注释里的工程约束**："`IMPORTANT: Do not change thinking support without notifying the model launch DRI and research.`" 说明这里的逻辑不只是功能判断，还涉及模型评估和质量保证——改错了会影响内部测试结果。

## Adaptive Thinking 的更严格限制

```typescript
export function modelSupportsAdaptiveThinking(model: string): boolean {
  const canonical = getCanonicalName(model)
  // 只有 4-6 系列支持
  if (canonical.includes('opus-4-6') || canonical.includes('sonnet-4-6')) {
    return true
  }
  // 对未知模型字符串，1P 和 Foundry 默认 true
  const provider = getAPIProvider()
  return provider === 'firstParty' || provider === 'foundry'
}
```

Adaptive Thinking 比普通 Extended Thinking 更新，所以只有 4-6 系列明确支持。对未知新模型，1P 和 Foundry 默认 `true`（因为新模型训练时包含了 adaptive thinking，不启用会降低模型质量）；Bedrock/Vertex 因为模型字符串格式不同，不做默认。

## shouldEnableThinkingByDefault：默认启用

```typescript
export function shouldEnableThinkingByDefault(): boolean {
  if (process.env.MAX_THINKING_TOKENS) {
    return parseInt(process.env.MAX_THINKING_TOKENS, 10) > 0
  }
  const { settings } = getSettingsWithErrors()
  if (settings.alwaysThinkingEnabled === false) {
    return false
  }
  return true
}
```

思考**默认是开启的**。用户可以通过 `alwaysThinkingEnabled: false` 设置关闭，也可以用 `MAX_THINKING_TOKENS` 环境变量覆盖（设为 0 关闭）。

## 面试角度

思考模式管理的几个工程设计点：

1. **ThinkingConfig 类型而不是布尔值**：三种模式（adaptive/enabled/disabled）表达了不同的语义，用类型系统约束了合法的组合
2. **关键词触发 vs. 设置控制是两个独立的入口**：用户可以通过设置永久开启、也可以通过 ultrathink 关键词临时切换到最高预算
3. **模型能力检测的两层兜底**：先查显式覆盖表，再用规则推断。前者优先是因为可以修正规则的误判
4. **UI 需要的信息和逻辑信息分开**：`findThinkingTriggerPositions` 只服务于 UI 高亮，和权限/配置逻辑完全解耦
5. **注意 `/g` 正则的状态问题**：`matchAll` 配合共享 `/g` 正则是 JS 的经典陷阱，每次调用新建正则是正确的防御方式
