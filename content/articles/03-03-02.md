# 面试题：用户输入 /var/tmp/foo 会发生什么？

## 边界条件

终端环境里最容易发生的一件事：用户把一个文件路径贴进输入框，然后回车。

`/var/tmp/foo` 以 `/` 开头，会进入 slash command 分支。`parseSlashCommand` 解析后，`commandName` 是 `var`，`args` 是 `tmp/foo`。然后 `hasCommand('var', commands)` 返回 false。

普通 CLI 的做法：输出「Unknown command: var」。

Claude Code 的做法：再想一想。

## looksLikeCommand 函数

```typescript
export function looksLikeCommand(commandName: string): boolean {
  // Command names should only contain [a-zA-Z0-9:_-]
  // If it contains other characters, it's probably a file path or other input
  return !/[^a-zA-Z0-9:\-_]/.test(commandName)
}
```

`var` 通过了这个检查（只包含字母），所以还不能直接判断是误输入的命令名。

接下来是 stat 检测：

```typescript
let isFilePath = false
try {
  await getFsImplementation().stat(`/${commandName}`)
  isFilePath = true
} catch {
  // Not a file path — treat as command name
}
```

这里拼回了 `/var` 做 stat 检测。如果 `/var` 是一个真实存在的目录（在大多数 Unix 系统上是的），`isFilePath` 被设为 true。

然后是最终判断：

```typescript
if (looksLikeCommand(commandName) && !isFilePath) {
  // 报 Unknown skill
} else {
  // 退回普通 prompt，把 /var/tmp/foo 当成普通文本处理
}
```

在 macOS/Linux 上，`/var` 存在，stat 成功，`isFilePath = true`，所以 `/var/tmp/foo` 不会被当成未知命令，而是退回普通 prompt 路径，模型会把这段文本当作用户想要解释的路径来理解。

## 追问：为什么是 stat(`/var`) 而不是 stat(`/var/tmp/foo`)?

代码里 stat 的是 `/${commandName}`，也就是 `/var`，不是完整路径。

原因：`parseSlashCommand` 把输入按第一个空格分割成 `commandName` 和 `args`。`/var/tmp/foo` 没有空格，整个会被解析为……

等等，让我重新看一下 `parseSlashCommand`：

```typescript
const withoutSlash = trimmedInput.slice(1)  // "var/tmp/foo"
const words = withoutSlash.split(' ')       // ["var/tmp/foo"]
let commandName = words[0]                  // "var/tmp/foo"
```

没有空格时，整个 `var/tmp/foo` 是 commandName。然后 `looksLikeCommand('var/tmp/foo')` 会检查它：

```
/[^a-zA-Z0-9:\-_]/.test('var/tmp/foo')
```

`/` 不在允许字符集 `[a-zA-Z0-9:\-_]` 里，所以这个正则匹配成功，`looksLikeCommand` 返回 false。

这意味着：包含 `/` 的路径字符串，`looksLikeCommand` 会直接返回 false，甚至不需要做 stat 检测，就会被当成普通文本处理。

`/var/tmp/foo` 在 Claude Code 里不会报「Unknown command」，它会被当成普通输入传给模型。

## 什么样的输入才会报「Unknown skill」

必须同时满足：
1. `looksLikeCommand(commandName)` 为 true（commandName 只包含 `[a-zA-Z0-9:_-]`）
2. `!isFilePath`（对应的路径在文件系统不存在）

比如 `/foobar`，commandName 是 `foobar`，`looksLikeCommand` 返回 true，stat `/foobar` 失败，才会报「Unknown skill: foobar」。

## 日志里的证据

```typescript
logEvent('tengu_input_slash_invalid', {
  input: commandName
})
```

还有一个 heuristic 检查：

```typescript
const looksLikeFilePath = inputString.startsWith('/var') || 
                           inputString.startsWith('/tmp') || 
                           inputString.startsWith('/private')
if (!looksLikeFilePath) {
  logEvent('tengu_input_slash_invalid', ...)
}
```

系统对几个常见的 Unix 路径前缀做了特殊豁免，不触发「invalid slash input」的遥测事件。这说明这个边界条件已经在生产中真实出现过，被专门处理了。

*面试指导：「输入系统的边界条件」是一个很好的追问方向。用 `/var/tmp/foo` 这个例子，能说明系统设计者对「slash 命令和路径的视觉相似性」有明确的意识，并通过 looksLikeCommand + stat 的组合检测来优雅地处理歧义，而不是粗暴地报错。*
