# 为什么未知命令先要判断它是不是别的东西

很多产品一看到用户输入以 `/` 开头，就会条件反射地进入“命令模式”。如果命令不存在，就立刻回一句“未知命令”。这种做法很干脆，但也很傲慢，因为它默认自己比用户更早知道这串字符到底是什么。

Claude Code 在这件事上明显更谨慎。`processSlashCommand()` 发现命令不存在后，并不会立刻武断地下结论，而是先做两层判断：这串东西看起来像不像一个真正的命令名；它会不会其实是一个文件路径或别的输入。只有在它既像命令、又不像路径时，系统才把它当成未知技能来处理。否则，它宁可把这段输入交回普通提示链路。

这种处理看上去只是多了几行分支，背后却是完全不同的产品哲学。它承认用户输入是有歧义的，系统不应该因为看见一个斜杠就过度自信。尤其在终端环境里，路径、命令、自然语言和复制粘贴来的片段本来就很容易长得相像。这里如果一味追求“果断”，结果往往就是误伤。

真正优雅的输入系统，不是总能最快抢答，而是知道什么时候该先怀疑自己。Claude Code 在这个小地方表现出的克制，和它在权限、并发、恢复上的克制，其实是同一种气质。

## 实现链
`processSlashCommand()` 里先 `parseSlashCommand()`，再 `hasCommand()` 看命令是否存在；如果不存在，还会用 `looksLikeCommand()` 和一次 `stat()` 去分辨它更像未知命令，还是更像文件路径，再决定报“Unknown skill”还是退回普通 prompt。

## 普通做法
常见写法是只要没命中命令表，就一律报“Unknown command”。

## 为什么不用
因为像 `/var/tmp/foo` 这种输入更像路径，不该被系统自信地当成用户在敲错命令。它宁可多判断一步，也不愿把真实输入误判成错误指令。

## 代价
入口分支多了一次文件系统探测和一层启发式判断；但这比把路径、命令和普通文本全混成一个错误提示更像成熟工具。
