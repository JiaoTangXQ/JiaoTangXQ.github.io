# 深度分析：为什么"把后台能力翻译成可管理劳动"是长期工作台的前提

## Promise 的边界

JavaScript 的 `Promise` 是一个很好的后台工作抽象：
- 表示一个将来会完成的操作
- 可以链式处理结果
- 有标准的错误处理方式

对绝大多数后台工作来说，Promise 已经足够了。

但 Promise 有一个根本性的限制：**它只知道"操作会完成"，不知道"这是哪一份工作"**。

你无法从一个 Promise 对象里知道：
- 这个操作已经运行了多长时间
- 它做了什么（有哪些工具调用）
- 它消耗了多少 token
- 现在是谁在负责它
- 完成后应该通知谁

这些信息在 Promise 抽象里根本不存在。

## 任务对象加了什么

Claude Code 的 `TaskStateBase` 在 Promise 的基础上加了什么？

```typescript
export type TaskStateBase = {
  id: string              // 身份：这份工作叫什么
  type: TaskType          // 类型：这是什么种类的工作
  status: TaskStatus      // 状态：现在在哪个阶段
  description: string     // 描述：人能读懂的工作说明
  startTime: number       // 时间：什么时候开始的
  endTime?: number        // 时间：什么时候结束的
  outputFile: string      // 输出：结果在哪里
  outputOffset: number    // 进度：已读到什么位置
  notified: boolean       // 通知：是否已经告知系统
}
```

每个字段都在回答"这份工作"的某个问题：

- `id` 解决的是：在系统里，这份工作如何被唯一认出来？
- `status` 解决的是：现在可以对这份工作做什么操作？（pending 状态不能 kill，completed 状态不能注入消息）
- `outputFile` 解决的是：这份工作的产出在哪里，怎么读取？
- `notified` 解决的是：是否已经通知相关方？（防止重复通知）

## "劳动"是比"异步操作"更丰富的概念

为什么用"可管理劳动"这个说法，而不是"结构化的异步操作"？

因为"劳动"暗含了人和时间的维度：劳动是有开始和结束的，是可以被委托和接手的，是可以被中途停止的，是会产生记录的。

Promise 描述的是"一个结果"，任务对象描述的是"一段过程"。

在一个真正的长期工作台里，你关心的不只是结果，而是整个过程：
- 用户启动了一个长时间任务，离开去做别的事
- 任务在后台运行，用户偶尔想看看进展
- 任务完成后，用户想查看整个执行过程
- 某些情况下，用户想把一个正在运行的任务交给另一个会话继续

这些场景都需要"过程"，而不只是"结果"。任务对象正是对"过程"的建模。

## 可接手性：任务对象最重要的特性

任务对象最重要的特性不是进度展示，而是**可接手性**。

什么叫"可接手"？一个人做到一半的工作，另一个人来了能继续，而不是必须重头开始。

Claude Code 的任务设计里，`outputFile` 是可接手性的基础设施。任务的所有输出都写到磁盘上的一个文件。进程重启后，新进程通过 `outputFile` 路径找到之前的输出，通过 `outputOffset` 知道已经读到哪里了，可以继续读取后续内容。

这和 Promise 有本质区别：Promise 的结果在内存里，进程结束就消失了。文件系统的输出在磁盘上，可以跨进程、跨会话保持。

## 注册即宣告存在

在 `src/utils/task/framework.ts` 里有 `registerTask` 函数。每种任务在开始执行前，都要先调用 `registerTask` 把自己写进 AppState。

这个"注册"动作的语义是：**我宣告我存在于这个系统中**。

未注册的任务不会出现在任务面板，不会被通知系统追踪，不会被清理逻辑纳管。注册是任务"进入管理体系"的入口。

这和操作系统的进程模型类似：一个新进程启动后，OS 在进程表里创建一个条目，分配 PID。这个条目就是进程在 OS 管理体系里的存在证明。任务注册就是 Claude Code 版的"进程表条目"。

## 面试要点

1. **Promise 只知道结果，任务对象知道过程**：任务对象的字段是对"一段有开始有结束的工作"的建模，比单纯的异步操作携带更多信息。
2. **outputFile 是可接手性的基础**：写到磁盘的输出让任务能跨进程持续，这是长期工作台的必要条件。
3. **注册是进入管理体系的入口**：未注册的异步操作不在系统的视野里，注册后才能被面板显示、被通知追踪、被清理纳管。
4. **"劳动"比"异步操作"更丰富**：劳动有委托、接手、中止、记录的概念，这些在 Promise 模型里根本不存在。
