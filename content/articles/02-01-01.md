# 为什么模块刚求值的几十毫秒也值得被认真利用

很多程序把“启动”理解成 `main()` 开始之后的事情，前面的模块加载只是不得不经历的门廊。Claude Code 的厉害之处在于，它连这段门廊都不肯浪费。`main.tsx` 最上面那一串注释写得很坦白：在大量 import 还没跑完之前，它就先打出 `profileCheckpoint`、启动 MDM 读取、启动 keychain 预取，让这些慢操作和后面上百毫秒的模块求值并行重叠。

这看起来像是在抠时间，其实是在尊重用户的等待。终端里的 agent 和网页不一样，没有一堆华丽界面来掩护冷启动，用户感受到的就是“我敲下命令之后，它多久像个活物”。Claude Code 因此把启动看成一条完整的时间河流，而不是只盯着函数体内部的那一小段。

更妙的是，它抢跑并不鲁莽。抢先启动的，都是后面几乎必然会用到、而且可以安全并行的慢操作；真正依赖这些结果的步骤，仍然会在合适的闸门前等待它们收口。也就是说，它不是乱并行，而是提前把等待藏进本来就存在的时间缝里。

这种设计最能体现工程品味。真正成熟的系统知道，优雅不只在架构图上，也在那些肉眼几乎看不见、但每次启动都在默默节省用户耐心的几十毫秒里。

## 实现链
这页对应的实现链都发生在 `main.tsx` 模块刚开始执行的阶段：先记启动点，再提前打出 MDM 读取、keychain 预取或其他可安全重叠的等待项，真正依赖这些结果的地方再稍后收口。它优化的不是某个函数体，而是整段启动时间河流。

## 普通做法
更普通的做法是把模块加载当成不可控前奏，等进入主逻辑以后再优化；或者一见到慢操作就一股脑并发开跑。

## 为什么不用那种做法
Claude Code 两条路都没选死。它不愿意浪费模块求值阶段的等待，但也不把依赖关系抹平，所以才形成了先开跑、后会合的两段式启动。

## 代价与回报
回报是把等待藏进原本就存在的时间缝。代价是理解入口时要同时记住什么已经启动、什么还不能消费结果。
