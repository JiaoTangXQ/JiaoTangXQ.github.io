# 面试题：如何设计一个支持长会话的 LLM Agent 架构？

面试官给了你这样一道题：设计一个 LLM agent，要求支持多轮对话、会话恢复（用户断开后能接着聊）、上下文压缩（会话太长时自动摘要）、工具调用（调用外部 API/命令行）。你会怎么设计？

大多数候选人的第一反应是一条大 while 循环：接收输入 → 拼上下文 → 调模型 → 执行工具 → 继续下一轮。这在 demo 里完全没问题，但面试官通常会追问：如果进程在"调模型"阶段崩掉，重启后怎么恢复？如果上下文满了触发压缩，压缩期间用户发来了新消息怎么办？

这时候，单循环的问题才会暴露出来——它无法区分"整场会话要持续跟踪的事实"和"这一轮临时要推进的任务"。

## Claude Code 的答案：两层分工

Claude Code 用 `QueryEngine` + `query()` 解决了这个问题。

**`QueryEngine`** 是会话的长期容器，跨越所有轮次存活：

```
mutableMessages       // 完整对话历史
permissionDenials     // 哪些工具被拒绝过
readFileState (FileStateCache)  // 文件缓存
totalUsage            // 累计 token 消耗
abortController       // 会话级的取消信号
loadedNestedMemoryPaths  // 加载过的嵌套记忆
discoveredSkillNames  // 本轮已发现的技能
```

**`query()`** 是单轮推进引擎，每次调用只负责当前这一轮：

```
State.transition      // 继续的理由（next_turn / collapse_drain_retry / ...）
autoCompactTracking   // 本轮压缩状态
maxOutputTokensRecoveryCount  // 截断恢复次数
streamingToolExecutor // 流式工具调度器（本轮新建）
```

两者的边界很清晰：`QueryEngine.submitMessage()` 先把用户输入整理好落进 transcript，再把当前会话状态打包交给 `query()`；`query()` 运行完（无论成功、失败还是被压缩中断），将结果返回给 `QueryEngine`，后者更新 `mutableMessages` 然后等待下一次调用。

## 为什么不能合并成一个循环？

来看一个具体场景：用户正在进行一次超长代码重构会话，会话进行到第 40 轮时，上下文压缩（auto-compact）触发了。压缩本身是一次独立的 API 调用，需要用已有的上下文生成摘要，再用摘要替换原来的消息。

如果状态在单循环里：压缩期间，`messages` 数组正在被重写，同时循环还想用 `messages` 调用模型——这是一个难以处理的并发写问题。

如果两层分工：`query()` 拿到压缩前的 `messagesForQuery`，在内部独立完成压缩逻辑，压缩后的消息在 `continue` 点被赋给 `state.messages`，`QueryEngine` 的 `mutableMessages` 只在整轮结束后才被更新。每个时刻，每一份消息数组的所有权都是清晰的。

## 状态机视角

`query()` 内部维护了一个显式的 `State` 结构，`transition` 字段记录每次循环继续的理由：

```
next_turn                  // 正常工具执行后继续
collapse_drain_retry       // 上下文折叠后重试
reactive_compact_retry     // 应急压缩后重试
max_output_tokens_escalate // 升级输出 token 上限
max_output_tokens_recovery // 截断恢复
stop_hook_blocking         // stop hook 产生阻塞
token_budget_continuation  // token 预算未用完，继续
```

这不是装饰性注释，而是测试钩子——测试代码可以断言"这一轮因为什么原因继续"，不需要解析消息内容。面试里，能提出用 `transition` 字段让状态机转移可被观测的候选人，会让面试官眼前一亮。

## 两层分工的代价

设计没有免费午餐。两层分工的代价是：

- `messages`、`toolUseContext`、`transition` 在两层之间传递，增加了理解成本
- `QueryEngine` 和 `query()` 之间的接口需要维护（`QueryParams` 和 `QueryEngineConfig` 两套配置）
- 对新读者来说，入口路径比单循环复杂得多

但收益是：长会话恢复、工具重试、上下文压缩、fallback 模型切换都可以在 `query()` 的 `State` 里干净地处理，不污染 `QueryEngine` 的会话状态。

## 面试指导

如果你在面试中被问到这道题，一个好的回答框架是：

1. **先指出两类不同的状态**：跨轮持久状态（消息历史、权限记录）vs 单轮推进状态（当前压缩阶段、retry 次数）
2. **用恢复场景测试设计**：进程崩了怎么恢复？压缩失败了怎么继续？这会暴露单循环的缺陷
3. **引入显式 transition**：不要用 boolean flag 表示"为什么继续"，用枚举类型的 transition 字段——可测试、可调试、可解释

这三点都能在 `QueryEngine.ts` 和 `query.ts` 的源码里找到直接对应。
