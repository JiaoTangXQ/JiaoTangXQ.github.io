# MAX_MESSAGES_WITHOUT_VIRTUALIZATION：非虚拟路径的安全阀

`Messages.tsx` 里：

```typescript
const MAX_MESSAGES_WITHOUT_VIRTUALIZATION = 200;
```

在不走虚拟滚动的路径上，消息条数有一个上限。`computeSliceStart()` 用 `sliceAnchorRef` 记录当前可见窗口的起始位置，确保这个窗口稳定地跟随阅读位置移动。

## 为什么非虚拟路径也需要上限

虚拟滚动（`VirtualMessageList`）通过只挂载当前可见区域的消息来处理长列表。但不是所有情况都走虚拟路径——terminal 渲染环境、特定的 transcript 视图或测试场景可能使用非虚拟路径。

在非虚拟路径下，所有消息都真实挂载到 React 树里。会话一长，消息一多，主线程会越来越慢：滚动卡顿、输入延迟、动画掉帧。当消息数量到达几百条时，挂载整个列表对主线程来说是一个相当重的负担。

200 这个上限不是精确的性能阈值，而是一个「不让事情变得不可用」的保守上界。

## sliceAnchorRef 的作用

上限只是「最多显示多少」的约束，`sliceAnchorRef` 解决的是「显示哪一段」的问题。

当消息超过 200 条时，应该显示最新的 200 条，还是用户当前正在看的那 200 条？`sliceAnchorRef` 存储的是 `uuid + idx` 的双重锚点，让可见窗口尽量跟随用户当前阅读位置，而不是每次都跳回最新消息。

这两个机制合在一起：上限保住了主线程性能，锚点保住了阅读连续性。用户不会因为会话变长而突然卡住，也不会因为触发了上限就被硬跳到列表末尾。
