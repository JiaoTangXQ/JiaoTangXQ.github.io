# 深度分析：新能力接入时为什么先做语言统一，后做功能实现

## 一个反直觉的顺序

工程师拿到一个新需求："把浏览器操作能力加进 Claude Code"。

直觉上的工作顺序是：
1. 设计浏览器 API
2. 实现截图、点击等功能
3. 接入权限系统
4. 更新提示词
5. 测试

但一个更有经验的工程师会在步骤 1 之前加一步：**确定命名约定**。

这一步看起来是"命名规范"这种细节，但实际上它在做的是：**让这组新能力成为系统语言的一部分，而不是一个孤立的扩展**。

## 为什么命名先于实现

**实现可以改，命名很难改**

功能实现可以迭代：截图算法可以优化，点击的精确度可以提升，错误处理可以完善。这些改动在系统内部，调用者不感知。

命名一旦确定，系统里所有引用了这个名字的地方都要跟着改：权限规则里的工具名匹配、提示词里的工具描述、日志分析里的过滤条件、文档里的示例、用户配置文件里的 allow-list。

这就是为什么要先想清楚命名，再开始写代码。早期改命名代价小（还没有被引用），晚期改命名代价极大。

**名字是跨层通信的协议**

这一点在 117 文章里分析过，但在"浏览器世界先说 MCP 语言"这个具体场景里，理解更深入。

浏览器工具的名字需要：
- 被后端权限系统认出（这是一个浏览器操作，需要 browser_access 权限）
- 被 AI 提示词引用（描述工具能力时，提到 tabs_context_mcp 和 screenshot_mcp）
- 被前端渲染层识别（这个工具调用展示为"浏览器操作"类型）
- 被用户配置引用（用户设置里"允许 Claude 使用 *_mcp 工具"）

一个名字，四个消费者。如果命名方式不一致，四个消费者要用不同的方式识别"这是浏览器工具"，系统的维护成本成倍增加。

## MCP 后缀是语言统一的具体手段

选择 `_mcp` 后缀，是在说：

"这些工具遵循 MCP 协议，走 MCP 路径处理，不要用非 MCP 的方式访问它们。"

这个后缀不是装饰，而是类型信息。它让代码里出现 `toolName.endsWith('_mcp')` 这样的判断成为可能，而不需要维护一个"哪些工具名是 MCP 工具"的列表。

## 原语言守住骨架的另一面

"浏览器世界先说 MCP 语言"还有另一层含义：**新能力进来时，是新能力适应系统，而不是系统为新能力改变**。

如果浏览器工具要求系统改变权限处理方式（比如"浏览器工具需要会话级权限，所以要改 ToolPermissionContext 的接口"），那么"加浏览器工具"就会影响所有工具。

如果浏览器工具适应系统（"我用 request_access 机制获取会话级权限，这个机制已经存在，我直接用"），那么加浏览器工具不影响已有的工具和逻辑。

这是大型系统演进的一条基本原则：**新功能尽量适应现有接口，而不是要求修改现有接口**。只有当现有接口真的不够用时，才去扩展它——但扩展要对所有调用者向后兼容。

## 语言统一的长期价值

在系统存在了一两年后，语言统一的价值开始真正显现：

**新成员快速上手**：看到 `tabs_context_mcp`，新成员知道这是 MCP 工具，去 MCP 工具相关的代码找答案，而不是去猜测这是什么类型的工具。

**日志和监控**：监控系统里，"MCP 工具调用的平均延迟"、"权限被拒绝的 MCP 工具"这类指标，只需要用 `_mcp` 后缀过滤，不需要维护工具名列表。

**规则的统一表达**：安全团队想要限制某类操作时，"禁止在沙盒环境里使用 MCP 工具"这条规则可以用一行配置表达，不需要列举所有工具名。

## 面试要点

1. **命名在实现之前，因为名字难以更改**：命名确定后被大量引用，晚期修改代价极高；早期命名代价小，可以仔细斟酌。
2. **MCP 后缀是类型信息**：让工具类型判断从维护列表变成字符串检查，减少维护负担。
3. **新能力适应系统，而不是系统为新能力改变**：浏览器工具用已有的 `request_access` 机制，而不是要求添加新的权限接口。
4. **语言统一的价值随时间增长**：初期是命名规范，中长期是日志分析、监控配置、安全规则的简化基础。
