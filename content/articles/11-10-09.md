# 全屏时气泡换位置：体验层服从宿主布局的物理限制

Claude Code 有两种主要的 UI 模式：普通模式和全屏模式（xterm.js 嵌入环境、桌面 app 等）。这两种模式的布局结构完全不同。

`CompanionSprite.tsx` 在普通模式下，气泡直接内联在精灵旁边：精灵在右侧，气泡在精灵左边，两者在同一个 flex 容器里并排。这在有终端 inline 布局的环境里工作得很好。

但全屏模式里，精灵的父容器可能会被滚动区域裁剪，inline 气泡会被视口边缘切掉。或者全屏模式的布局根本不允许在输入区旁边放一个多行的气泡组件。

## bottomFloat 槽位

`FullscreenLayout.tsx` 专门提供了一个 `bottomFloat` 槽位——一个渲染在视口底部、不受滚动内容影响的固定位置。全屏模式下，`CompanionFloatingBubble` 就挂到这里，作为浮层出现在视口底部角落。

`REPL.tsx` 根据 `isFullscreenEnvEnabled()` 和 `companionNarrow` 决定走哪条路：

```typescript
if (isFullscreenEnvEnabled()) {
  // 精灵显示在某处，气泡交给 bottomFloat 槽位
} else {
  // 精灵和气泡内联在一起
}
```

## 「服从物理限制」是体验层的基本义务

体验层功能总有一个诱人的冲动：想在所有环境下呈现完整效果。但每种宿主环境都有自己的布局约束和物理限制，强行用同一套渲染方案覆盖所有情况，结果通常是：在某些环境里内容被切掉、布局打架、或者挤压了主要的工作区域。

Claude Code 的处理方式是：为每种主要环境维护一条渲染路径，在合适的时机选合适的路径。体验层可以出现，但它的出现形式必须由宿主布局的实际约束决定，而不是由「我想怎么显示」决定。
