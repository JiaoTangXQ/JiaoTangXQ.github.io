# Plan Mode 是怎么实现的？EnterPlanMode 和 ExitPlanMode 的状态机

Plan Mode 让 Claude 在执行任何破坏性操作之前先出一个完整计划，等用户确认后再执行。这个模式在 Claude Code 里有具体的实现：`EnterPlanModeTool` 和 `ExitPlanModeV2Tool`，以及 `PermissionMode` 状态机。

## Plan Mode 的核心机制

在 Plan Mode 下，`PermissionMode` 被设为 `'plan'`：

```typescript
// PermissionMode 类型（types/permissions.ts）
type PermissionMode =
  | 'default'           // 正常模式
  | 'plan'              // 计划模式：只能读，不能写
  | 'acceptEdits'
  | 'bypassPermissions'
  | 'dontAsk'
  | 'auto'
```

当 `permissionMode === 'plan'` 时，权限检查在所有写操作（文件编辑、bash 命令执行）面前都会返回拒绝。模型可以读文件、搜索代码、分析结构，但无法修改任何东西。

## 进入计划模式

`EnterPlanModeTool` 是一个让模型主动请求进入 plan mode 的工具。当模型认为某个任务需要先规划再执行时，它可以调用这个工具：

```typescript
// EnterPlanModeTool 的 call 实现
// 1. 把当前 permissionMode 保存到 prePlanMode
// 2. 把 permissionMode 设为 'plan'
// 3. 在 UI 上展示当前处于计划模式
```

`prePlanMode` 字段保存了进入 plan mode 之前的状态。这样退出 plan mode 时可以恢复到之前的状态，而不是总是回到 'default'——如果进入之前是 'acceptEdits'，退出后应该还是 'acceptEdits'。

## 退出计划模式

`ExitPlanModeV2Tool` 更复杂。它不是简单地退出 plan mode，而是要配合 **AllowedPrompt** 系统：

```typescript
export type AllowedPrompt = {
  text: string          // 计划的文本
  approved: boolean     // 用户是否批准
  toolUseId: string     // 关联的工具调用 ID
}
```

当模型调用 `ExitPlanModeTool` 时，它会提交一个计划（`text`）。Claude Code 把这个计划展示给用户，用户选择批准或拒绝：

- **批准**：`approved = true`，恢复到 `prePlanMode`，模型继续执行计划
- **拒绝**：`approved = false`，保持在 plan mode，模型需要修改计划重新提交

这个交互是专门设计的——不是"模型说退出就退出"，而是"模型提交计划，用户审批，用户决定是否退出"。

## Plan 文件：计划的磁盘持久化

`utils/plans.ts` 管理计划文件：

```typescript
export function getPlanSlug(sessionId?: SessionId): string {
  const cache = getPlanSlugCache()
  let slug = cache.get(id)
  if (!slug) {
    // 生成易读的 slug（如 "brave-whale-compute"）
    for (let i = 0; i < MAX_SLUG_RETRIES; i++) {
      slug = generateWordSlug()
      // 检查文件不冲突
      if (!existsSync(join(plansDir, `${slug}.md`))) break
    }
    cache.set(id, slug)
  }
  return slug
}
```

每个会话的计划被写到 `~/.claude/plans/<slug>.md`，`slug` 是自动生成的可读词组（"brave-whale-compute" 这样的格式）。这让计划可以：
1. 会话中断后恢复（通过 slug 找到对应计划文件）
2. 用户查看历史计划（可读的 slug 而不是 UUID）

## 状态转换的完整图

```
default ──[EnterPlanMode]──→ plan
                                │
                          [ExitPlanMode + 用户批准]
                                │
                                ↓
                          prePlanMode（通常是 default）
```

如果是用户手动通过 UI 切换 plan mode（比如快捷键），不经过工具调用，而是直接修改 `permissionMode`。工具调用路径是让**模型**能主动请求进入 plan mode，这是两种不同的触发方式。

## 面试角度

Plan Mode 的实现体现了几个设计思路：

1. **状态保存和恢复**：`prePlanMode` 字段让模式切换是可撤销的，而不是"进了 plan mode 就丢失了之前的状态"
2. **用户审批嵌入工具协议**：`AllowedPrompt.approved` 让工具调用本身包含了用户决策的结果，不需要额外的副作用通道
3. **磁盘持久化计划**：计划文件支持会话恢复，readable slug 比 UUID 对人更友好
4. **工具触发 vs 用户触发**：同一个状态变更（进入 plan mode）可以由模型工具调用或用户 UI 操作触发，两条路都要维护
