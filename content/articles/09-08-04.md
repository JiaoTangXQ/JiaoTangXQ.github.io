# registry.ts 缓存了哪些东西，为什么都要缓存到进程生命周期？

`registry.ts` 里有 5 个模块级缓存变量：

```typescript
let cachedBackend: PaneBackend | null = null
let cachedDetectionResult: BackendDetectionResult | null = null
let cachedInProcessBackend: TeammateExecutor | null = null
let cachedPaneBackendExecutor: TeammateExecutor | null = null
let inProcessFallbackActive = false
```

它们有一个共同点：**一旦设置，在整个进程生命周期里不会变化**（除非调用 `resetBackendDetection()` ——这个函数只用于测试）。

为什么每一个都需要这种稳定性？逐一分析。

## cachedBackend 和 cachedDetectionResult

这两个是最明显的缓存理由：`detectAndGetBackend()` 需要运行 shell 命令（`tmux display-message`、`which tmux` 等）来探测环境，每次调用都重新运行这些命令不仅慢（每次几十毫秒），而且可能返回不同的结果（如果 tmux 会话在探测过程中改变了状态）。

缓存探测结果，保证所有 `detectAndGetBackend()` 调用看到同一个环境快照。这是性能优化，也是一致性保证。

**为什么要一致性**：如果第 1 个 teammate 检测到 tmux，用 tmux 创建了 pane，第 3 个 teammate 检测到 tmux 不可用了（因为 tmux 会话关闭了），用 in-process 模式……团队的两个成员用了不同的后端，空间语义碎裂，无法统一管理。

## cachedInProcessBackend

`InProcessBackend` 实例如果每次都新建，看起来无所谓——它的代码里没有什么复杂的初始化逻辑。

但 `InProcessBackend` 内部持有 `context: ToolUseContext | null`，通过 `setContext()` 设置后才能使用。如果每次都新建实例，上一次 `setContext()` 的设置就丢失了，新实例是未初始化的。

```typescript
// InProcessBackend 的简化版
class InProcessBackend {
  private context: ToolUseContext | null = null
  
  setContext(context: ToolUseContext): void {
    this.context = context
  }
  
  async spawn(config: TeammateSpawnConfig) {
    if (!this.context) {
      return { success: false, error: 'call setContext() first' }
    }
    // ...
  }
}
```

缓存同一个实例，保证 `setContext()` 的设置在整个会话里有效。

**为什么只用一个实例**：整个会话里，in-process 模式的上下文（访问 AppState、设置状态的函数）应该是一致的。多个 `InProcessBackend` 实例持有不同的 context 引用，可能导致状态更新路径分叉。

## cachedPaneBackendExecutor

`PaneBackendExecutor` 比 `InProcessBackend` 更关键：它内部持有 `spawnedTeammates` Map，记录了所有已经启动的 teammate 的 `{paneId, insideTmux}` 映射。

```typescript
class PaneBackendExecutor {
  private spawnedTeammates: Map<string, { paneId: string; insideTmux: boolean }>
  
  async kill(agentId: string) {
    const info = this.spawnedTeammates.get(agentId)
    if (!info) return false  // 找不到，无法杀掉
    
    const killed = await this.backend.killPane(info.paneId, !info.insideTmux)
    if (killed) this.spawnedTeammates.delete(agentId)
    return killed
  }
}
```

如果每次 `getTeammateExecutor()` 都新建 `PaneBackendExecutor`，新实例的 `spawnedTeammates` Map 是空的——之前启动的 teammate 的 paneId 信息丢失了。

尝试对旧 teammate 调用 `kill()` 或 `terminate()`，会因为找不到 paneId 而失败。清理流程（SIGHUP 时通过 cleanup handler 关闭所有 pane）也会失效，因为新实例不知道有哪些 pane 需要清理。

## inProcessFallbackActive

这是 5 个缓存里最有意思的一个。它不是缓存一个计算结果，而是记录一个已发生的事件："本次会话里，因为 pane backend 不可用，某次 spawn 回退到了 in-process 模式"。

一旦这个标志被设置，`isInProcessEnabled()` 在 auto 模式下直接返回 `true`，不再每次都重新检测环境。

**为什么不每次重新检测**：如果每次都重新检测，可能在某次检测时 pane backend 恢复可用，系统就会切换回 pane 模式，用 tmux 创建新的 pane。但此时有些 teammate 已经在 in-process 模式下运行，有些将要用 pane 模式启动——混合模式，一致性破坏。

`inProcessFallbackActive` 是一个"已发生事实的记录"：既然这次会话里已经回退了，整个会话就承认这个现实，不再反复试探。

## 统一的缓存重置接口

```typescript
export function resetBackendDetection(): void {
  cachedBackend = null
  cachedDetectionResult = null
  cachedInProcessBackend = null
  cachedPaneBackendExecutor = null
  backendsRegistered = false
  inProcessFallbackActive = false
}
```

这个函数只在测试里使用（注释里写明了）。它存在的原因：测试需要在每个测试用例之间重置状态，避免测试间互相污染。

这是一个重要的设计模式：**生产代码假设缓存状态是进程级别稳定的；测试代码通过显式重置接口恢复初始状态**。两种使用场景的需求不同，分别满足，不互相妥协。

## 面试怎么答

如果面试题是"什么时候应该用模块级缓存，什么时候不该用"：

**用模块级缓存的场景**：
1. 结果的计算有外部依赖（I/O、进程调用），重复计算有明显成本
2. 结果需要在同一进程/会话里保持一致，多次计算可能得到不同值
3. 对象持有状态（如 `spawnedTeammates` Map），需要在多次调用间保持

**不该用模块级缓存的场景**：
1. 计算结果可能会频繁变化，过期数据比没有缓存更危险
2. 多个实例的独立状态是需要的（比如多租户系统里每个租户需要独立的实例）

**测试兼容性**：模块级缓存必须有显式的重置接口供测试使用，否则测试间会互相污染，产生难以排查的偶发失败。`resetBackendDetection()` 是这个原则的实践。

**一致性和灵活性的权衡**：缓存带来一致性，但牺牲了灵活性（无法在会话中途切换）。Claude Code 的判断是：对多 agent 系统而言，一致性比灵活性更重要，因为一致性破坏的代价（团队状态碎裂、行为不可预期）远高于灵活性缺失的代价（偶尔无法使用更好的后端）。
