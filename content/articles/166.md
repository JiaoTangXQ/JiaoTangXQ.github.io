# 第十三卷回收索引：任务世界里先改状态再善后是后台系统的保命顺序

## 实现链
任务世界里制度先行的另一半，不是状态本身，而是先把收尾顺序写死。

这条线真正逼出来的，是第十三卷里关于后台保命顺序的判断。  
对 `子 agent/后台系统` 来说，先改状态再善后不是口号，而是避免把自己卡死的顺序。

这页实际回收的实现链是：
- `05-05-03` 把“先改状态再善后”写成子 agent 收尾硬原则，这会在 `13-04-01` 里继续成立，说明后台系统必须先改状态，再做后面的善后动作。

很多系统在收尾时会先忙着做善后，结果状态还卡在旧位置。  
Claude Code 更克制，它先改状态，再把后事收干净，这样后台系统才不会被自己绊住。

## 普通做法
更普通的做法，是把后台动作当成松散的异步句柄，跑起来就算开始，结束了再临时补状态和通知。

## 为什么不用
Claude Code 不这么做，是因为没有统一任务对象，就没有稳定的生命周期、输出、通知和回看链，前后台切换也会变成一串彼此不认账的临时处理。

## 代价
代价是任务状态字段、输出文件、通知协议和前后台转场逻辑都会变厚，维护者要记住更多接缝。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
