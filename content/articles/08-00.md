# 面试题：如果让你设计 Claude Code 的远端架构，你会分几层？

这是一道系统设计题，在大厂面试中常以这样的形式出现：

> "你需要让一个本地 AI 编码助手支持远端会话——用户在网页上发起任务，在远端服务器上执行，结果实时同步回来，权限审批也能双向流动。你会怎么设计？"

大多数人的第一反应是 WebSocket + 消息队列。这没错，但只答了一层。Claude Code 源码里，这套系统分成了四个清晰的层，每层解决一类不同性质的问题。

## 四层架构

**第一层：Bridge（桥接层）**

位于 `src/bridge/*`。负责把本地 REPL 会话整体连到远端。它要处理的问题是：如何在本地进程和远端 CCR（Claude Code Remote）之间建立一个"会话级通道"，而不是一次性的请求通道。

关键文件：`bridgeApi.ts`（环境注册、轮询、heartbeat）、`bridgeMain.ts`（主循环）、`remoteBridgeCore.ts`（env-less 路径）。

**第二层：Remote（远端翻译层）**

位于 `src/remote/*`。负责把远端过来的消息、权限请求和重连信号翻译成本地能理解的形式。核心是 `remotePermissionBridge.ts`，它在远端发来权限请求时，构造一条合成的 `AssistantMessage`，让本地的审批 UI 能正常渲染——即便本地根本没有执行那个工具的能力。

**第三层：Assistant（观察层）**

涉及 `viewerOnly` 模式。远端会话可以被其他客户端以只读方式"观看"，这不是简单的日志流，而是完整的消息历史回放，需要保持和主会话一致的上下文顺序。

**第四层：Teleport（传送层）**

位于 `src/utils/teleport/*`。负责跨设备的会话迁移：在机器 A 上开启的任务，可以被另一台机器 B "接管"继续。涉及 git bundle 打包、环境选择、历史消息反序列化。

## 为什么不是一层？

常见的反驳是："这四件事不都是网络 I/O 吗？能不能合并？"

不能。原因是这四层面对的**失败模式完全不同**：

- Bridge 层失败 → 需要轮询重连，heartbeat 续约，environment 重注册
- Remote 层失败 → 需要重建合成消息，审批回调清理，权限状态回滚
- Assistant 层失败 → 历史消息顺序错乱，seq-num 校对
- Teleport 层失败 → git bundle 上传中止，分支冲突，历史消息部分丢失

如果合并成一层，这四种失败都需要在同一个状态机里处理，互相干扰。

## 核心设计原则

源码中贯穿这四层的一个原则：**"不要把长会话塞进短调用模型"**。

Bridge 层持有的是会话级句柄（能写消息、发控制请求、更新标题、teardown），而不是一次性函数。这意味着初始化很重——需要做 entitlement 检查、OAuth 验证、组织策略、git 上下文准备，但这是必要代价。

另一个原则：**"跨边界后，每项制度都要重新声明谁来执行"**。权限审批原来由本地确认，远端模式下需要明确：本地发起？服务端发起？用户在哪里点击"允许"？这不是实现细节，是架构决策。

## 面试答题框架

如果在面试中遇到这类题，推荐的答题路径是：

1. **先分层**：按失败模式和状态所有权划分，不要按技术协议（WebSocket/HTTP）划分
2. **说清楚每层的边界**：谁持有状态，谁负责恢复
3. **讨论 edge case**：连接断开时 pending 的权限请求怎么处理？token 过期时进行中的任务如何续约？
4. **说出取舍**：分层带来复杂性，换来的是每层可以独立演进和独立测试

Claude Code 选择了四层，而不是一个大 WebSocket 客户端，正是因为每一层在演进时的速度和方向都不同——Bridge 在优化轮询间隔，Teleport 在优化 bundle 大小，两者不应该被同一个状态机绑死。
