# 深度分析：TaskType 和 TaskStatus 为什么是"宪法"，不只是枚举

## 枚举和宪法的区别

普通的 TypeScript 枚举是一种类型安全的字符串集合：

```typescript
type TaskStatus = 'pending' | 'running' | 'completed' | 'failed' | 'killed'
```

这只是五个值。

但当这个定义变成**整个任务系统所有模块共同遵守的基础**时，它就是宪法了：

```typescript
export function isTerminalTaskStatus(status: TaskStatus): boolean {
  return status === 'completed' || status === 'failed' || status === 'killed'
}
```

这个函数告诉系统："这三个状态是终态，一旦到达这三个状态，任务就不再转换了。"

所有依赖这个函数的代码——清理逻辑、通知逻辑、注入消息的防卫检查——都依赖这个宪法成立。如果某个任务类型偷偷创建了第六个"实质上是终态但不是这三个之一"的状态，上述所有防卫都会失效。

## 宪法的内容

`Task.ts` 里的"宪法"包含几个关键条款：

**条款一：任务类型穷举**

```typescript
export type TaskType =
  | 'local_bash'
  | 'local_agent'
  | 'remote_agent'
  | 'in_process_teammate'
  | 'local_workflow'
  | 'monitor_mcp'
  | 'dream'
```

这个列表是封闭的（closed union）。系统里不存在"未被登记的任务类型"——任何新的任务类型都必须显式加入这个列表，这个行为本身会引发代码审查，确保新类型被系统的各个部分正确处理。

**条款二：状态转换规则**

五种状态不是平等的：

- `pending` 可以转换到 `running`（任务开始执行）
- `running` 可以转换到 `completed`、`failed` 或 `killed`（正常完成、执行失败、被强制终止）
- `completed`、`failed`、`killed` 不能转换到任何其他状态（这是 `isTerminalTaskStatus` 的保证）

这个转换规则就是任务的生命周期协议。任何任务类型都不能绕过这个协议——比如不能从 `completed` 恢复到 `running`（如果要"重试"，应该创建新任务，而不是修改旧任务的状态）。

**条款三：ID 前缀约定**

任务 ID 不是随机的 UUID，而是带类型前缀的标识符：

```typescript
const TASK_ID_PREFIXES: Record<string, string> = {
  local_bash: 'b',
  local_agent: 'a',
  // ...
}
```

这个约定让 ID 携带类型信息，在日志里一眼能看出这是什么类型的任务。这不是强制的（类型信息也存在 `type` 字段里），但作为约定，它提高了系统的可读性。

## 为什么先写宪法，再写模块

如果反过来——先分别实现 `LocalShellTask`、`LocalAgentTask` 等，最后才抽出公共部分——会发生什么？

每种任务会定义自己的状态枚举：
- `LocalShellTask.status: 'starting' | 'active' | 'done' | 'error' | 'terminated'`
- `LocalAgentTask.status: 'queued' | 'executing' | 'finished' | 'crashed' | 'aborted'`

这些枚举在语义上是一样的（都有"开始、运行中、正常结束、失败、被终止"），但名字不同。

当需要一个"判断任务是否处于终态"的函数时，要么每种任务写一个，要么写一个超大的 switch-case：

```typescript
function isTerminal(task: AnyTask): boolean {
  if (task.type === 'local_bash') {
    return task.status === 'done' || task.status === 'error' || task.status === 'terminated'
  } else if (task.type === 'local_agent') {
    return task.status === 'finished' || task.status === 'crashed' || task.status === 'aborted'
  } // ...
}
```

这个函数每次新增任务类型就要更新，是维护负担的持续来源。

先写宪法，`isTerminalTaskStatus` 一次写成，永远有效。

## 宪法的约束力

宪法只有在所有模块都遵守的情况下才有效。

TypeScript 的类型系统在这里提供了部分约束力：任务状态必须是 `TaskStatus` 类型，任务类型必须是 `TaskType` 类型，任何违反这些约定的代码在编译时就会报错。

但有些约束是不能靠类型系统保证的：比如"任务不能从终态转回运行态"——类型系统看来，`status = 'running'` 的赋值是合法的，即使任务已经是 `completed` 状态。

这类约束需要靠代码审查和测试来保证：在 PR 审查时，如果发现有代码修改了已处于终态任务的状态，需要明确说明理由或者拒绝这个修改。

## 面试要点

1. **宪法是封闭约定，不是开放列表**：TaskType 的枚举列表是封闭的，新任务类型必须显式加入，触发代码审查。
2. **终态约定让清理逻辑可靠**：isTerminalTaskStatus 的正确性依赖所有模块遵守终态不转换的约定，这个约定越早确立越好。
3. **ID 前缀是编码进 ID 的类型信息**：一个可读的约定，在日志和调试时比查数据库里的 type 字段更直观。
4. **先写宪法避免后期协调成本**：各自定义状态枚举再协调，比先统一枚举再分别实现，代价高得多。
