# 关闭transport必须顺手打断挂起调用说明清缓存不算真正收场

## 实现链
- `client.ts` 在包 transport 的关闭逻辑时，会显式处理 in-flight `callTool()` 的 abort，让挂起请求能真正结束，而不是只把连接对象置空。
- 这里的重点不是“清缓存”，而是把正在等待结果的调用也一起收场。否则连接虽然关了，Promise 还可能永远挂着。

## 普通做法
- 普通做法是断开 socket、删除客户端引用，剩下的请求让它们自己超时。
- 很多接口客户端都这么做，因为超时机制看起来像天然兜底。

## 为什么不用
- Claude Code 不愿这么敷衍，是因为 MCP 里有长调用、流式调用和 server 自己不一定会主动结束的情况。只关 transport 不中断调用，等于留下了一地悬挂状态。
- 从代码看，它更重视“收尾完整性”，而不是“表面上已经断开”。

## 代价
- 代价是关闭路径要更深地介入 SDK 和 transport 行为，代码也更难一眼看懂。
- 但这能少掉一类最难排查的问题：界面已经显示断开，后台却还有永不返回的请求。
