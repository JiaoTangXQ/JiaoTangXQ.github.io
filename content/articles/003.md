# Claude Code 的工具系统是怎么做并发控制的？

## 问题场景

模型在一轮输出里调用了 5 个工具：读 3 个文件、写 1 个文件、跑 1 条 bash 命令。Claude Code 会怎么执行这 5 个工具？全并发？全串行？还是混合？

答案是混合，而且分组逻辑有明确的依据。

## 看 toolOrchestration.ts

`runTools()` 接到一批 `ToolUseBlock`，第一步不是调用，而是分区：

```typescript
for (const { isConcurrencySafe, blocks } of partitionToolCalls(
  toolUseMessages,
  currentContext,
)) {
  if (isConcurrencySafe) {
    // 并发执行
    for await (const update of runToolsConcurrently(blocks, ...))
  } else {
    // 串行执行
    for await (const update of runToolsSerially(blocks, ...))
  }
}
```

`partitionToolCalls` 的作用是把工具调用序列切成若干批次。每个批次要么"并发安全"，要么"串行"。重点：批次是有序的，前一批全部完成后才开始下一批。

## 什么叫「并发安全」？

直觉是：只读操作可以并发，有副作用的操作必须串行。Claude Code 的工具协议里，每个工具通过 `isReadonly` 或 `isConcurrencySafe` 字段声明自己的属性。

读文件工具（ReadTool）是只读的，可以并发——3 个读文件调用可以同时发起。

写文件工具（WriteTool/EditTool）有副作用，不能并发——因为多个写操作可能修改同一个文件，后写会覆盖前写的中间状态。

BashTool 更复杂。运行 bash 命令可能读也可能写，系统默认把它标为非并发安全。

## 并发批次里的上下文修改问题

并发执行有一个微妙的问题：工具执行时可能要修改 `ToolUseContext`（比如更新权限状态、记录文件缓存）。如果两个工具并发运行都想修改同一个 context，就会产生竞争。

`toolOrchestration.ts` 的解决方法是"延迟 context 修改"：

```typescript
const queuedContextModifiers: Record<string, ContextModifier[]> = {}

// 并发执行期间，收集所有 contextModifier，不立即应用
for await (const update of runToolsConcurrently(blocks, ...)) {
  if (update.contextModifier) {
    queuedContextModifiers[toolUseID].push(modifyContext)
  }
  yield { message: update.message, newContext: currentContext }
}

// 并发批次全部完成后，串行应用所有 context 修改
for (const block of blocks) {
  for (const modifier of queuedContextModifiers[block.id]) {
    currentContext = modifier(currentContext)
  }
}
```

这保证了并发执行时 context 是只读的，修改只在批次结束后按确定顺序施加。

## 实际效果

```
模型输出：[Read A, Read B, Read C, Write D, Bash E]

分区结果：
批次 1: [Read A, Read B, Read C] → 并发安全，三个同时发起
批次 2: [Write D] → 非并发安全，单独串行
批次 3: [Bash E] → 非并发安全，等 Write D 完成后执行
```

并发读取可以把 3 次 IO 的时间压到约等于 1 次 IO。在模型频繁读多个文件的场景（代码分析、文档生成），这个优化非常显著。

## 并发上限

`getMaxToolUseConcurrency()` 读取环境变量 `CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY`，默认值是 10。这是安全上限，防止一次性发起太多并发 IO 造成系统负担。

## 面试指导

被问工具执行机制时，核心要点是三层：分区（partitionToolCalls）、批次执行（并发或串行）、延迟 context 修改。不要简单说"只读并发、写串行"——这个结论是对的，但面试官真正想听的是你知道 context 修改竞争的问题以及怎么解决。并发批次完成后才应用 context 修改是这套设计最精细的部分。
