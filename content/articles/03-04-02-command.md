# Command 到底统一了哪些公共身份

Claude Code 把不同来历的能力都收进 `Command`，并不是只给它们套一个共同名字。它真正统一的是一整套“公共身份”。一个命令至少要回答这些问题: 它叫什么，要怎样显示，什么时候该出现，怎么向人解释参数，从哪来，能不能让模型主动调用，在哪些账户或环境里可用，外表上属于哪一类命令。

这些东西看起来像界面字段，实际上是运行时秩序。补全列表靠它们排序，帮助界面靠它们解释，远端和 bridge 安全过滤要看它们，模型能不能看见、用户能不能直接敲，也都要先有这层共同身份。没有这些共通字段，系统就没法把技能、插件、MCP、workflow 和内建命令放进同一套选择、显示和治理流程里。

所以 `Command` 最有价值的地方，不是“把东西装进一个类型”，而是先发了一张统一身份证。它让异质能力在进入系统的第一步就变得可识别、可比较、可过滤。对普通读者来说，这其实非常关键: 一个优雅的 agent，不只是会做事，更会先把每一类能力整理成自己能管理的对象。

## 实现链
`Command` 统一的不只是名字，还包括 `type`、`userInvocable`、`context`、`model`、`effort`、`allowedTools`、`pluginInfo` 这些身份字段。也就是说，一条命令不只是一段代码，而是一份运行时政策对象。

## 普通做法
普通命令对象常常只有“命令名 + 回调函数 + 帮助文本”。

## 为什么不用
因为这里的命令会影响模型、工具权限、遥测、桥接安全和 UI 表达，只保留一个 handler 根本不够。

## 代价
命令 schema 更大，理解和维护都更费劲；但只有这样，入口层才能用统一方法处理策略和显示，而不是事后补字段。
