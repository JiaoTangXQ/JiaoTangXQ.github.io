# task id 为什么先解决谁是谁的问题

任务世界里最先要解决的，不是“它在做什么”，而是“它到底是谁”。

## 实现链

`generateTaskId()` 在 `src/Task.ts` 里按任务类型分前缀，`local_agent` 用 `a`，`in_process_teammate` 用 `t`，再拼八位小写字母数字。前缀让系统一眼知道任务大类，随机部分避免碰撞，整个设计还考虑了大小写不敏感文件系统和符号链接攻击。

这不是 UI 编号，而是贯穿输出文件、通知、杀任务、恢复和面板定位的主键。

## 普通做法

普通做法常常直接用数组下标、时间戳，或者把执行器返回的某个内部 ID 直接拿来用。够用时很省事。

## 为什么不用

Claude Code 不敢这么随便，是因为任务 ID 不是只在一个模块里用。它会进入文件路径、跨进程通知、任务列表、团队协作和后续恢复。如果 ID 不稳定，后面所有围绕它建立的结构都会发虚。

所以它先解决“谁是谁”，再解决“谁干了什么”。这是非常典型的工程取向。

## 代价

代价是每加一种新任务类型，都得考虑前缀、兼容性和外围系统影响。ID 生成看起来不像业务功能，却成了不能乱动的基础设施。
