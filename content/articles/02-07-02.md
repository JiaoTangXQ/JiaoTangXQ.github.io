# 首屏前和首屏后为什么要切成两套工作节奏

Claude Code 的启动并不是一口气冲到底，而是很清楚地切成了两段节奏。第一段的目标只有一个: 尽快把一个可信、可用、不会误导用户的首屏交出来。第二段才是趁用户刚准备开始工作时，在后台继续把世界悄悄预热到更顺手的状态。

所以有些事必须挡在首屏之前。比如 trust 相关确认、某些必须到位的文件下载、会直接影响当前界面的关键准备。如果这些东西没完成，系统就还没有资格把舞台亮出来。可另一些事情又明显不值得继续拦在前面，比如 SessionStart hooks 的完整等待、MCP 资源连接、设置和技能变更探测器、很多缓存式预热。Claude Code 对这些事的处理很有分寸: 能并行就并行，能先挂 promise 让界面出来就先挂，能放到首屏后就放到首屏后。

这背后其实是一种很朴素但很难坚持的工程判断: “快”不是指所有事情都越早越好，而是指该先出现的东西先出现，不该抢跑的东西别堵路。也正因为它把首屏前和首屏后切成了两套工作节奏，后面的 `deferred prefetches` 才不是补丁，而是整条启动链自然的下半场。

## 实现链
这页对应的实现链都落在 REPL 交棒和 deferred prefetch 这套双节奏上：首屏前只做那些不做就没法安全进入 REPL 的事，首屏后再继续刷新、预热、建议和补充装配。代码在管理的不是单纯步骤，而是用户感知到的节奏。

## 普通做法
更普通的启动要么全部做完再显示，要么一股脑都异步开跑，谁先完算谁。

## 为什么不用那种做法
Claude Code 两种都没选死，因为前者太慢，后者太乱。它要的是先给用户可工作的前台，再让后台继续补齐。

## 代价与回报
回报是等待被重新分配。代价是启动完成这件事在代码里不再是单一时刻，而是一段前台和后台交错的过程。
