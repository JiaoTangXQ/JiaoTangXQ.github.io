# 这种写法怎样承载几千个微型文档而不散架

一旦决定写到很细，新的风险就出现了: 文档会不会越长越碎，最后只剩一地卡片。Claude Code 这种项目，后面很可能真的会长出几千个叶子文档，所以结构必须先能承重。

这套承重方法靠的是分层而不是堆积。卷负责说明读者现在身处哪一片大世界，章负责说明这一段主线正在讨论哪类问题，节负责把问题族群先拢住，叶子文档则只处理一个很小但完整的设计点。这样每个叶子都可以很短，却不会失去坐标。

真正防止散架的，还有交叉索引和结构化元数据。因为很多好设计本来就会横跨多卷，像制度会碰流程，连续性会碰恢复，远端边界会碰权限和界面。只靠目录层级不够，必须再用显式关系把它们钉住。这样后面无论是人继续扩写，还是 AI 继续整理，都还能知道每一片碎叶子究竟长在整棵树的哪里。

## 实现链
从代码映射看，目录树首先服务开发，不直接服务理解。`main.tsx`、`QueryEngine.ts`、`commands.ts`、`Tool.ts`、状态存储这些文件分别落在不同目录，但它们回答的其实是入口、运行时骨架、制度和状态问题。文档如果只照目录平铺，就看不见这些跨目录的实现链。

## 普通做法
最直觉的写法是按目录讲文件，或者按产品功能讲菜单。这两种写法都很常见，也最省心。

## 为什么不用那种做法
Claude Code 这本书没有只选其中一种，因为这套系统的精彩之处大多发生在交叉处：流程走到哪里，为什么要用这种边界处理。只按目录会让普通读者迷路，只按功能又会把实现判断糊成大块。

## 代价与回报
回报是文档能真正贴着代码里的设计问题下钻。代价是文档结构和目录结构不再一一对应，维护时必须不断做导读、回收和索引。
