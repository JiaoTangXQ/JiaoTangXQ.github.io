# 第十三卷回收索引：连续性世界里的 `autocompact` 先保会话，不做漂亮整理

## 实现链
连续性世界里另一半更关键的，不是把桌面整理得多好看，  
而是会话本身在快撑不住的时候，还能不能先活下来。

这条线真正逼出来的，是第十三卷里关于“整理后真正要保的是继续工作，而不是表面整洁”的判断。

这页实际回收的实现链是：
- `10-03-02` 把 `autocompact` 解释成“保会话，不做漂亮整理”，这会在 `13-05-02` 里重新成立，说明系统先保的是会话可继续，而不是表面上的干净。

真正成熟的自动整理，不会先追求漂亮。  
它先保证会话还活着，再谈后面怎么继续工作。

## 普通做法
更普通的做法，是一发现上下文变长就立刻整理，或者把当前状态和长期记忆统统塞回主会话，省掉额外的插槽和外置存储。

## 为什么不用
Claude Code 不这么做，是因为整理太早会打断主回合，记忆不分层会把短期噪音和长期知识混在一起，最后让连续性系统同时失去可读性和可恢复性。

## 代价
代价是需要额外维护 compact boundary、回合后插槽、外置 memory 和记忆回灌链，系统会更厚。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
