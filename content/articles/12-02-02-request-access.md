# 面试题：为什么 request_access 比逐步确认更适合桌面操作授权？

## 权限设计的经典矛盾

给高风险能力设计权限系统时，有两种对立的哲学：

**最小权限原则**：每次操作都单独请求权限，确保用户对每一步都有感知和控制。

**会话级授权**：在会话开始时一次性处理所有授权，后续操作在授权范围内自动执行。

桌面控制能力是高风险的：它能操控鼠标、键盘、打开任意应用、截取屏幕。按照最小权限原则，应该每次点击都确认一遍。

Claude Code 没这么做。`setup.ts` 的注释直接说明了原因：

```ts
/**
 * The mcp__computer-use__* tools are added to allowedTools so they bypass 
 * the normal permission prompt — the package's request_access handles approval
 * for the whole session.
 */
```

## 逐步确认为什么不行

想象一个典型的桌面自动化任务：填写一份 20 个字段的表单，每个字段需要点击输入框（1次点击）、输入内容（1次 type）、可能还要从下拉菜单选择（2-3次点击）。一个简单的表单填写任务就是 40-80 个工具调用。

如果每次调用都弹确认框：
- 用户需要点击 40-80 次"确认"
- 每次确认都是打断，任务完成时间翻几十倍
- 用户会对确认框产生"确认疲劳"，开始习惯性点击而不是真正审查
- "确认了每一步"在心理上反而降低了用户对整体任务的把控感

逐步确认不只是不便利，还会产生"确认框剧场"——表面上每一步都有审批，实际上用户完全没有在真正审查。

## request_access 的设计逻辑

会话级授权把问题转化成：**在会话开始时，让用户明确回答"你授权这次会话里的所有桌面操作吗？"**

这个问题更清晰，用户做决定时有足够的心理空间去认真权衡，而不是被连续弹窗打断思维。

从实现角度，`wrapper.tsx` 维护了整个会话的授权状态：

```ts
// 会话级状态对象
onPermissionRequest()  // 触发 request_access 流程
onAllowedAppsChanged() // 更新本次授权可操作的应用列表
grantFlags             // 本次授权的权限标志位
allowedApps           // 本次授权可操作的应用白名单
```

这些状态贯穿整个会话生命周期。一旦 `request_access` 通过，`allowedTools` 里的工具就可以直接执行，不再需要逐步确认。

## 风险模型的转变

逐步确认解决的是"单次操作的风险"：用户能在每一步拒绝。

会话级授权解决的是"整体会话的风险"：用户在开始时对整个任务范围做判断。

哪个更合理？这取决于任务的特性。

**逐步确认适合**：操作之间相互独立，每步的结果用户都能立即判断，且任务规模小（5步以内）。

**会话级授权适合**：操作是连续序列，中间步骤的意义只有完成后才能评估，且任务规模大（10步以上）。

桌面自动化任务几乎都是连续序列：登录→导航→填写→提交，中间任何一步单独看都不构成完整意义。用逐步确认的方式授权这类任务，实际上是在要求用户用单步的眼光审查序列任务，这种错配会让确认流于形式。

## allowedApps 的精细化设计

会话级授权不是"一刀切全开"，而是允许精细化：`allowedApps` 可以只放行本次任务需要操作的应用。

用户在 `request_access` 阶段看到的问题是："这个任务需要操作以下应用：[Chrome, VS Code]，是否授权？"——而不是"是否允许操作所有应用？"

这种精细化把风险从"全局风险"降为"本次任务特定应用的风险"，在便利性和安全性之间找到了一个更好的平衡点。

## 会话级授权的代价

任何设计都有代价。会话级授权的代价是：

**权限状态要显式管理**。`allowedApps`、`grantFlags` 这些状态必须在会话生命周期内可靠维护，不能用全局变量随便存。

**授权边界要严格定义**。什么算"本次会话"？多长时间后需要重新授权？任务中途中断后如何处理？

**锁机制**。防止并发的桌面操作会话同时运行，`computerUseLock.ts` 专门处理这个问题。

这些复杂度是显式的、集中的，而不是分散在每次工具调用里。这反而更容易推理和测试。

## 面试考察点

这道题考察的是**权限系统设计中的用户体验 vs 安全感知的权衡**，以及**对"高风险连续操作"这类场景的特殊性理解**。

标准答案不是"会话级授权更安全"或"逐步确认更安全"，而是：**不同风险特性的任务，需要不同粒度的授权方案，关键是匹配任务的时间结构和用户的决策方式**。

Claude Code 选择了会话级授权，是因为它面向的是连续操作序列，而不是独立的单次动作。
