# 为什么只按产品功能写又一定会太粗

如果完全反过来，不按目录写，而是只写“启动”“命令”“工具”“任务”“权限”这种大章，问题也不会消失。这样当然比贴着文件树走更顺，可它很快会把 Claude Code 真正动人的地方抹平。

因为这套系统的优雅，往往不藏在大标题里，而藏在更小的设计问题里。比如，为什么显示帮助不值得付完整初始化成本；为什么未知 slash 命令不能立刻粗暴报错；为什么一次动作结束后必须补成 `tool_result` 闭环；为什么上下文治理不是一次压缩，而是一串轻重不同的修复。把这些都塞进“启动优化”“命令系统”“主循环恢复”这种大抽屉里，读者会知道它有这些功能，却看不见设计师到底在解决什么难题。

只按产品功能写，最后常常会变成一份体面的功能总览。它会显得完整，却不够细；会显得清楚，却讲不出那些真正体现工程品味的微小转折。对这本书来说，这种粗糙是不能接受的。

## 实现链
从代码映射看，目录树首先服务开发，不直接服务理解。`main.tsx`、`QueryEngine.ts`、`commands.ts`、`Tool.ts`、状态存储这些文件分别落在不同目录，但它们回答的其实是入口、运行时骨架、制度和状态问题。文档如果只照目录平铺，就看不见这些跨目录的实现链。

## 普通做法
最直觉的写法是按目录讲文件，或者按产品功能讲菜单。这两种写法都很常见，也最省心。

## 为什么不用那种做法
Claude Code 这本书没有只选其中一种，因为这套系统的精彩之处大多发生在交叉处：流程走到哪里，为什么要用这种边界处理。只按目录会让普通读者迷路，只按功能又会把实现判断糊成大块。

## 代价与回报
回报是文档能真正贴着代码里的设计问题下钻。代价是文档结构和目录结构不再一一对应，维护时必须不断做导读、回收和索引。
