# Claude Code 的文件历史检查点系统：AI 代码修改的"撤销"是怎么做的？

当 Claude 用 `FileEditTool` 修改了一个文件，你想撤销这个修改，你会怎么做？对于人类编辑器，Cmd+Z 就够了。但对于 AI 代理：它可能在一次对话里修改了几十个文件，你需要能精确回退到某一条消息之前的状态。

Claude Code 的文件历史系统（`utils/fileHistory.ts`）解决的就是这个问题。

## 数据结构设计

```typescript
type FileHistorySnapshot = {
  messageId: UUID        // 关联到哪条消息
  trackedFileBackups: Record<string, FileHistoryBackup>
  timestamp: Date
}

type FileHistoryBackup = {
  backupFileName: string | null  // null 表示该版本文件不存在
  version: number                // 单调递增的版本号
  backupTime: Date
}

type FileHistoryState = {
  snapshots: FileHistorySnapshot[]
  trackedFiles: Set<string>
  snapshotSequence: number  // 单调递增，即使 snapshot 被驱逐也继续增
}
```

每条对话消息对应一个 `FileHistorySnapshot`。快照里记录了每个被追踪文件在此时刻的备份文件名和版本号。

`backupFileName: null` 是一个很有意思的设计——它显式表示"这个文件在此版本不存在"。这样在回退时就能区分"文件被修改了"和"文件被新建了（可以删除）"以及"文件被删除了（可以恢复）"。

## 硬链接优化

文件备份如果用 `copyFile` 实现，每个快照都复制一份完整内容，大文件会消耗大量磁盘空间。Claude Code 用 **硬链接（hard link）** 优化这个问题：

如果文件内容没变（通过哈希比较），直接用 `link(existingBackup, newBackupPath)` 创建硬链接，不需要复制数据。硬链接指向同一个 inode，多个文件名共享同一份磁盘数据，删除一个不影响其他，直到最后一个引用删除时才真正释放空间。

这个优化对于"大量小改动"的场景特别有效——如果你修改了一个 1000 行文件的第 5 行，Claude Code 不需要为每个快照复制整个文件。只有文件内容真的变了，才会创建新的物理副本。

## 容量管理

```typescript
const MAX_SNAPSHOTS = 100
```

最多保留 100 个快照。当超出时，驱逐最旧的快照，同时清理对应的备份文件（如果没有其他快照引用同一个备份）。

`snapshotSequence` 是单调递增的计数器，**即使快照被驱逐也不重置**。这个设计让 `useGitDiffStats` 能用它作为"活跃度信号"——当 `snapshots.length` 因为达到上限而不再增长时，`snapshotSequence` 还在增长，反映了真实的修改频次。

## 开关控制

```typescript
export function fileHistoryEnabled(): boolean {
  if (getIsNonInteractiveSession()) {
    return fileHistoryEnabledSdk()
  }
  return (
    getGlobalConfig().fileCheckpointingEnabled !== false &&
    !isEnvTruthy(process.env.CLAUDE_CODE_DISABLE_FILE_CHECKPOINTING)
  )
}
```

文件历史在非交互会话（SDK 模式）和交互会话（REPL 模式）下有不同的默认行为。环境变量 `CLAUDE_CODE_DISABLE_FILE_CHECKPOINTING` 可以完全关闭，适用于 CI 环境——不需要撤销能力，只是跑自动化任务。

## diff 显示

`DiffStats` 类型用于在 UI 里显示"本次对话修改了多少代码"：

```typescript
type DiffStats = {
  filesChanged?: string[]
  insertions: number
  deletions: number
} | undefined
```

`diffLines` 来自 `diff` 库，对比快照前后的文件内容，计算插入/删除行数。这就是 Claude Code 状态栏里显示的 `+12 -3` 的来源。

## 面试角度

这个系统有几个值得深挖的设计决策：

1. **快照关联消息 ID 而不是时间戳**：让"回退到某条消息之前"成为精确操作，时间戳容易有并发问题
2. **null 作为有意义的值**：`backupFileName: null` 比用空字符串更清晰，表达了"不存在"这个状态
3. **硬链接降低存储成本**：内容寻址（通过哈希判断是否重复）+ 硬链接是 Git 内部存储 blob 对象的同款思路
4. **单调序列号和容量上限解耦**：`snapshotSequence` 和 `snapshots.length` 分别回答不同的问题，设计清晰
