# TaskState 的七种类型：Claude Code 为什么要把这么多东西建模成「任务」？

`tasks/types.ts` 定义了 `TaskState` 联合类型：

```typescript
export type TaskState =
  | LocalShellTaskState        // bash/shell 命令
  | LocalAgentTaskState        // 本地子 Agent
  | RemoteAgentTaskState       // 远程 Agent
  | InProcessTeammateTaskState // 进程内 teammate
  | LocalWorkflowTaskState     // 工作流脚本
  | MonitorMcpTaskState        // MCP 监视器
  | DreamTaskState             // Dream 模式任务
```

七种类型。为什么不就叫"后台进程"？因为这七种东西虽然形态不同，但它们共享同一套管理语义。

## 共同的 TaskStateBase

七种任务状态都继承自 `TaskStateBase`（`Task.ts` 里定义）：

```typescript
export type TaskStateBase = {
  id: string
  type: TaskType
  status: 'pending' | 'running' | 'completed' | 'failed' | 'killed'
  description: string
  toolUseId?: string    // 关联到哪个 tool_use 调用
  startTime: number
  endTime?: number
  totalPausedMs?: number
  outputFile: string    // 任务输出写到哪个文件
  outputOffset: number  // 已读取的输出偏移量
  notified: boolean     // 用户是否已被通知
}
```

不管是一个 bash 命令还是一个远程 Agent，它们都有：
- **唯一 ID** — 便于查找和关联
- **状态机** — `pending → running → completed/failed/killed`
- **输出文件** — 所有任务的输出都写到磁盘，不是存在内存里
- **notified 标志** — 防止重复通知用户

## 统一停止接口

`stopTask.ts` 是任务停止的统一入口——无论是模型通过 `TaskStopTool` 请求停止，还是 SDK 的 `stop_task` 控制请求：

```typescript
export async function stopTask(
  taskId: string,
  context: StopTaskContext,
): Promise<StopTaskResult> {
  const task = appState.tasks?.[taskId]
  if (!task) {
    throw new StopTaskError(`No task found with ID: ${taskId}`, 'not_found')
  }
  if (task.status !== 'running') {
    throw new StopTaskError(`Task is not running`, 'not_running')
  }
  const taskImpl = getTaskByType(task.type)
  await taskImpl.kill(taskId, setAppState)
  // ...
}
```

`getTaskByType(task.type)` 返回对应的任务实现，每种类型有自己的 `kill` 方法。这是标准的 Strategy 模式——停止接口统一，但停止行为按类型不同。

## shell 任务的特殊处理

停止 bash 任务时有一个细节：

```typescript
// Bash：抑制"exit code 137"的通知（噪音）
// Agent 任务：不抑制——AbortError 携带了有用的部分结果

if (isLocalShellTask(task)) {
  // 把 notified 设为 true，阻止后续通知
  // 但直接发送 SDK 事件，让 SDK 消费者知道任务已停止
}
```

`kill -9` 结束的 bash 进程退出码是 137。如果不处理，用户会看到"命令失败，退出码 137"——但用户自己停止的任务失败通知是无意义的噪音。所以主动把 `notified = true`，阻止这个通知出现。

但 SDK 消费者（通过 SDK 接口集成的外部系统）仍然需要知道任务停止了。所以直接发送 `emitTaskTerminatedSdk`，绕过 UI 通知路径但不绕过 SDK 通知路径。

## isBackgroundTask 的判断逻辑

UI 状态栏显示"后台任务数"，但不是所有任务都应该计入：

```typescript
export function isBackgroundTask(task: TaskState): task is BackgroundTaskState {
  if (task.status !== 'running' && task.status !== 'pending') {
    return false  // 已完成/失败/停止的任务不显示
  }
  if ('isBackgrounded' in task && task.isBackgrounded === false) {
    return false  // 前台任务不算"后台任务"
  }
  return true
}
```

`isBackgrounded` 字段区分了"正在后台跑"和"正在前台运行但还没完成"。用户正在等待的任务不应该出现在"后台任务"计数里。

## DreamTask：Claude Code 的环境感知模式

`DreamTaskState` 是个有意思的特殊类型。"Dream"模式让 Claude Code 在空闲时持续监控环境变化（文件变化、Git 状态、CI 结果），在有有趣变化时主动通知用户。这是一个持续运行的感知任务，不是单次执行。

## 面试角度

统一任务模型是系统设计里的"识别共性"技巧：

1. **先识别什么是共同的管理需求**：生命周期（pending→running→终态）、通知、输出存储、取消——这些需求在七种任务上完全相同
2. **差异通过 Strategy 表达**：`kill` 方法按类型不同，但调用接口统一
3. **状态机要有终态**：`isTerminalTaskStatus` 函数定义了哪些状态是终态——这防止了"已完成的任务又收到了新的状态更新"
4. **通知去重**：`notified` 字段比事件去重 ID 更简单，因为每个任务只需要被通知一次
