# AppState 是如何驱动 Claude Code 界面的？Selector 模式实战解析

Claude Code 的终端界面远比它看起来复杂：它要同时显示流式输出、权限对话框、后台任务列表、teammate 状态面板、搜索高亮……这些内容来自不同的运行时层级，如果用"需要更新就 re-render"的朴素思路，结果一定是满屏闪烁和状态不一致。

它的解法是：把所有运行时状态收进一个中央 `AppState`，组件通过 Selector 订阅所需切片，派生出"当前该显示什么"。

## AppState 的结构

`AppStateStore.ts` 里可以看到 `AppState` 的组成，关键字段包括：

- `messages: Message[]` — 当前对话的全量消息序列
- `tasks: Record<string, TaskState>` — 所有运行中任务（bash、agent、teammate）的状态
- `toolPermissionContext: ToolPermissionContext` — 当前权限上下文
- `viewingAgentTaskId?: string` — 用户当前正在查看的 Agent 任务 ID
- `speculationState: SpeculationState` — 预测性渲染状态（Speculation 系统）
- `mcp: { tools: Tools; connections: MCPServerConnection[] }` — MCP 工具状态
- 通知、Overlay、mailbox 等各种上下文状态

这些字段覆盖了从底层执行到顶层 UI 显示的完整状态链。

## Selector 的作用

`state/selectors.ts` 里只有两个函数，但这两个函数体现了整个设计的核心思路。

以 `getActiveAgentForInput` 为例：

```typescript
export type ActiveAgentForInput =
  | { type: 'leader' }
  | { type: 'viewed'; task: InProcessTeammateTaskState }
  | { type: 'named_agent'; task: LocalAgentTaskState }

export function getActiveAgentForInput(appState: AppState): ActiveAgentForInput {
  const viewedTask = getViewedTeammateTask(appState)
  if (viewedTask) {
    return { type: 'viewed', task: viewedTask }
  }
  const { viewingAgentTaskId, tasks } = appState
  if (viewingAgentTaskId) {
    const task = tasks[viewingAgentTaskId]
    if (task?.type === 'local_agent') {
      return { type: 'named_agent', task }
    }
  }
  return { type: 'leader' }
}
```

这个 Selector 做的事情很清晰：**把"当前用户输入应该发给谁"这个问题从 AppState 中派生出来**，返回一个带类型标签的判别联合体（discriminated union）。组件拿到这个结果就知道该渲染什么——不需要自己去判断 `viewingAgentTaskId` 是不是 teammate。

## 为什么不用局部 state？

当任务状态变化时（比如某个 teammate 完成了工作），所有关心这个任务的组件都需要同步更新——任务列表、状态指示器、输入框路由、通知面板。如果它们各自维护局部 state，同步逻辑会非常复杂，而且几乎必然会出现短暂的不一致。

中央 `AppState` + 订阅模式的优势：**状态只有一份**，每次更新所有订阅者同时收到最新值。`setAppState(f: (prev: AppState) => AppState)` 是一个纯函数式更新，可组合，可测试。

## Speculation 状态：预计算的 UI 状态

`AppState` 里有一个特别值得关注的字段——`speculationState`：

```typescript
export type SpeculationState =
  | { status: 'idle' }
  | {
      status: 'active'
      id: string
      abort: () => void
      messagesRef: { current: Message[] }
      writtenPathsRef: { current: Set<string> }
      boundary: CompletionBoundary | null
      // ...
    }
```

Speculation 系统在用户还没提交输入时，就预测性地开始生成下一个响应，把结果暂存在 `speculationState`。如果用户真的提交了和预测一致的输入，直接把预计算结果拿来用，节省延迟。这种"状态机里的预计算结果"的设计在面试中是个好话题：它本质上是 CPU 分支预测思想在应用层的体现。

## 面试角度

"前端如何管理复杂状态"是经典面试题。Claude Code 的 AppState 方案是一个有说服力的实例：

1. **单一可信来源（Single Source of Truth）**：AppState 是唯一的事实，没有"这里的 messages 和那里的 messages 到底哪个最新"的问题
2. **纯函数更新**：`setAppState(prev => next)` 使状态变更可测试、可回放
3. **Selector 作为抽象层**：组件不直接读 AppState 字段，通过 Selector 拿派生值，Selector 可以独立测试
4. **discriminated union 返回值**：Selector 返回带类型标签的联合类型，让组件侧的类型推断更安全
