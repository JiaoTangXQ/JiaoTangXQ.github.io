# AgentTool 怎么启动子 Agent？从 runAgent 看子代理的完整生命周期

当主线程的 Claude 觉得某个子任务可以委托出去，它调用 `AgentTool`。这会触发 `runAgent.ts` 里的完整子 Agent 生命周期：

## 启动：上下文克隆

子 Agent 不是从零开始的——它继承父上下文的很多状态：

```typescript
const subagentContext = createSubagentContext(parentContext, {
  agentId: newAgentId,
  // 克隆父的 fileStateCache（读文件记录）
  // 但子 Agent 的 setAppState 是 no-op（子不能直接修改父状态）
})
```

`createSubagentContext` 里有一个关键决策：子 Agent 的 `setAppState` 是个空操作（no-op）。这意味着子 Agent 不能直接修改父 App 的状态。所有状态变更必须通过工具调用的返回值流回父级，而不是直接写全局状态。

这个设计防止了"子 Agent 修改父 Agent 的消息历史"这样的危险情况。

## 工具集过滤

子 Agent 不一定能用所有工具：

```typescript
// constants/tools.ts 里定义了什么工具子 Agent 可以用
export const ALL_AGENT_DISALLOWED_TOOLS: string[] = [
  // 某些工具不应该递归（比如 AgentTool 本身，防止无限嵌套）
]

export const CUSTOM_AGENT_DISALLOWED_TOOLS: string[] = [
  // 自定义 Agent 额外禁用的工具
]
```

子 Agent 的工具集是父工具集的子集，经过 `ALL_AGENT_DISALLOWED_TOOLS` 过滤。防止 Agent 递归无限创建子 Agent 是最重要的限制之一。

## MCP 连接：子 Agent 可以有自己的 MCP 服务器

`runAgent.ts` 里有一段意想不到的逻辑：子 Agent 在启动时可以连接额外的 MCP 服务器：

```typescript
if (agentDefinition.mcpServers) {
  for (const serverConfig of agentDefinition.mcpServers) {
    const connection = await connectToServer(serverConfig)
    const tools = await fetchToolsForClient(connection)
    agentMcpTools.push(...tools)
  }
}
```

这让"代码审查 Agent"可以连接专门的代码审查 MCP 服务器，而"部署 Agent"可以连接 CI/CD MCP 服务器，主 Agent 不需要管理所有 MCP 连接。

## 钩子注册：每个 Agent 有独立的钩子生命周期

```typescript
// 注册 CLAUDE.md 里定义的 frontmatter hooks
await registerFrontmatterHooks(agentContext)

// 执行 agent 启动钩子
await executeSubagentStartHooks(agentContext)
```

每个子 Agent 有自己的钩子集合，独立于父 Agent 的钩子。当子 Agent 结束时，钩子被清理（`clearSessionHooks`），不会泄漏到父级。

## Transcript 记录：子 Agent 有自己的侧链

```typescript
await setAgentTranscriptSubdir(agentId, parentSessionId)
// ...子 Agent 运行...
await recordSidechainTranscript(agentId, messages)
await clearAgentTranscriptSubdir(agentId)
```

子 Agent 的对话历史存储在独立的子目录（"sidechain transcript"）。这有几个作用：
1. 调试时可以单独查看某个子 Agent 的完整对话
2. 恢复会话时可以重建子 Agent 的状态
3. 不污染主会话的 transcript

## 资源清理

子 Agent 结束时有明确的清理步骤：

```typescript
// 杀掉子 Agent 启动的所有 shell 任务
await killShellTasksForAgent(agentId, setAppState)

// 清理 prompt cache 追踪
cleanupAgentTracking(agentId)

// 注销 Perfetto 追踪（如果启用）
unregisterPerfettoAgent(agentId)
```

这些清理确保子 Agent 的资源（进程、缓存条目、监控注册）不会泄漏到父级。特别是 `killShellTasksForAgent`——如果子 Agent 启动了一个 bash 任务然后被取消，这个任务必须也被终止，不能继续在后台占用资源。

## 内置 Agent 定义

`built-in/` 目录下有几个预定义的 Agent：

- `generalPurposeAgent.ts` — 通用 Agent，用于委托一般性子任务
- `exploreAgent.ts` — 专门用于代码库探索
- `planAgent.ts` — 生成执行计划
- `verificationAgent.ts` — 验证工作结果
- `claudeCodeGuideAgent.ts` — 指导用户使用 Claude Code

这些内置 Agent 有预配置的系统提示和工具集，用户也可以通过 `loadAgentsDir.ts` 加载自定义 Agent 定义文件（从 `.claude/agents/` 目录）。

## 面试角度

子 Agent 系统的设计是分布式系统和递归分解的典型案例：

1. **上下文继承但状态隔离**：子 Agent 能看到父的文件缓存，但不能直接写父状态——继承是"读"权限，不是"写"权限
2. **工具集是权限边界**：过滤工具集比"运行时检查是否允许"更安全，因为模型压根看不到被过滤的工具
3. **资源所有权明确**：每个 Agent 启动的资源（shell 任务、MCP 连接、钩子）在 Agent 结束时由 Agent 自己清理
4. **sidechain transcript 隔离调试**：子 Agent 有自己的对话记录，调试时可以独立分析，不与主线程混淆
