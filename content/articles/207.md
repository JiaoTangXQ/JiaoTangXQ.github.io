# 先收口再开闸：主流程的自我约束从哪里来

面试题版本：**"Claude Code 的主循环在处理用户请求之前做了哪些前置检查？如果跳过这些检查直接进入主流程，最坏的情况是什么？这些检查是如何组织的，为什么不放到主流程里？"**

## 主流程解释不了自己为何不失控

一个没有前置约束的主循环：
```typescript
// 最简版本
while (true) {
  const input = await getUserInput()
  const response = await callModel(input, tools)
  const results = await executeTools(response.toolCalls)
  // 继续...
}
```

这个循环可以做任何事——执行任意 bash 命令、修改任意文件、调用任意 API。它没有任何内在的约束能说明"为什么它不会失控"。约束必须从外部施加。

## 收口动作的组织

Claude Code 把前置约束组织在几个明确的阶段：

**输入阶段的收口**：
- 用户输入超长时，主循环不是直接报错，而是先检查是否可以分割或摘要
- 工具参数通过 `inputSchema.safeParse()` 验证，非法参数在执行前被拦截
- MCP 工具调用的参数通过额外的安全过滤层

**能力阶段的收口**：
- 工具注册时声明了 `isConcurrencySafe`，主循环不能绕过这个声明
- 权限检查在工具执行前完成，`useCanUseTool()` 返回 false 时工具不执行
- `REMOTE_SAFE_COMMANDS` 和 `BRIDGE_SAFE_COMMANDS` 限制了不同入口的命令范围

**副作用阶段的收口**：
- `preAction` 钩子在工具执行前运行，可以检查或阻止特定操作
- `stop hooks` 在主循环停止前运行，可以注入额外的验证步骤
- `contextModifier` 在工具执行后但在下一轮开始前应用，是一个可控的副作用传播点

## 为什么不放到主流程里

把约束内嵌到主流程里的样子：

```typescript
while (true) {
  const input = await getUserInput()
  // 主流程里到处是检查
  if (input.length > MAX_INPUT) { truncate(input) }
  const response = await callModel(input, tools)
  for (const tool of response.toolCalls) {
    if (!canUseTool(tool)) continue  // 权限检查散落在循环里
    if (!validateInput(tool.input)) continue  // 验证也在这里
    await executeTool(tool)  // 执行和检查混在一起
  }
}
```

问题在于：这样的主流程自身承担了太多职责，而且每个约束之间没有明确的顺序关系——权限检查是应该在输入验证之前还是之后？如果在某个工具执行后发现需要重新检查权限，该如何处理？

把约束分阶段放到主流程外面（作为前置条件和后置条件），主流程内部可以假设约束已经满足，专注于"已经确认合法的操作如何执行"这个问题。

## 预取的对立面

`startDeferredPrefetches()` 说的是"能提前的提前"，而收口说的是"该限制的先限制"。这两者是一对：

- 预取：把不影响当前操作安全性的异步工作提前开始，减少后续等待
- 收口：把影响当前操作安全性的检查提前完成，确保主流程在安全边界内运行

两者都在主循环开始前运行，但方向相反：预取是"为了后面更快"，收口是"为了后面更安全"。

## 为什么是前置而不是后置

把约束做成后置（"做完了再检查有没有出问题"）在有些场景下有道理，但在 AI 执行工具这个场景里，后置检查来不及：

**文件修改是不可逆的**：bash 命令删除了一个文件，事后检查到权限问题，文件已经没了。

**副作用已经扩散**：一个网络请求已经发出去了，事后发现这个请求不应该发，请求的效果无法撤销。

**工具链中间状态**：多个工具按顺序执行，第三个工具的执行依赖前两个的结果。如果第一个工具的执行后来被判定为不应该发生，整个工具链的状态是不一致的。

所以收口必须在执行前，而不是执行后。

## 面试角度的总结

主流程的收口设计回答了"一个有自主执行能力的 AI 系统如何在不人工监控每一步的情况下保持安全边界"这个问题。答案是：把所有边界条件组织为明确的前置约束，让主流程在已知合法的空间内运行，而不是在运行中不断检查是否越界。这样主流程本身可以保持简洁，约束逻辑在边界层集中管理。
