# PaneBackend 抽象层：为什么 tmux 和 iTerm2 需要同一套接口

`TmuxBackend` 和 `ITermBackend` 都实现了 `PaneBackend` 接口：

```typescript
interface PaneBackend {
  type: 'tmux' | 'iterm2'
  displayName: string
  supportsHideShow: boolean
  
  isAvailable(): Promise<boolean>
  isRunningInside(): Promise<boolean>
  createTeammatePaneInSwarmView(name: string, color: AgentColorName): Promise<CreatePaneResult>
  sendCommandToPane(paneId: PaneId, command: string, useExternalSession?: boolean): Promise<void>
  setPaneBorderColor(paneId: PaneId, color: AgentColorName, useExternalSession?: boolean): Promise<void>
  setPaneTitle(paneId: PaneId, name: string, color: AgentColorName, useExternalSession?: boolean): Promise<void>
  enablePaneBorderStatus(windowTarget?: string, useExternalSession?: boolean): Promise<void>
  rebalancePanes(windowTarget: string, hasLeader: boolean): Promise<void>
  killPane(paneId: PaneId, useExternalSession?: boolean): Promise<boolean>
  hidePane(paneId: PaneId, useExternalSession?: boolean): Promise<boolean>
  showPane(paneId: PaneId, targetWindowOrPane: string, useExternalSession?: boolean): Promise<boolean>
}
```

这个抽象层解决的不是"未来可能加 Windows Terminal"的问题，而是当下就存在的问题。

## 两种后端的技术差异

**Tmux** 通过命令行接口操作：

```bash
tmux split-window -t leader-pane -h -l 70% -P -F '#{pane_id}'
tmux send-keys -t pane-id 'claude --agent-id researcher@team' Enter
tmux select-pane -t pane-id -P 'fg=blue'
```

**iTerm2** 通过 `iterm2_set` 协议（一套特殊的 escape sequence）和 AppleScript 操作，API 完全不同。

如果没有抽象层，代码里会充满：

```typescript
if (backend === 'tmux') {
  await execFileNoThrow('tmux', ['split-window', '-t', paneId, '-h', '-l', '70%'])
} else if (backend === 'iterm2') {
  await iterm2.createTab()  // iTerm2 的 API
}
```

这种条件分支会扩散到每个需要操作 pane 的地方：颜色设置、标题设置、布局调整、隐藏/显示……每个操作都要分叉一次，最终代码里有数十个 tmux/iTerm2 的条件分支。

## 抽象层让 PaneBackendExecutor 不需要知道后端类型

`PaneBackendExecutor` 是把 `PaneBackend` 包装成 `TeammateExecutor` 接口的适配器。它的代码里没有任何 `if tmux else iterm2` 的逻辑：

```typescript
async spawn(config: TeammateSpawnConfig): Promise<TeammateSpawnResult> {
  // 无论是 tmux 还是 iTerm2，都调用同样的方法
  const { paneId, isFirstTeammate } = 
    await this.backend.createTeammatePaneInSwarmView(config.name, config.color)
  
  await this.backend.sendCommandToPane(paneId, spawnCommand, !insideTmux)
  
  // ...后续操作同样不区分后端
}
```

`PaneBackendExecutor` 只知道"这个 backend 能做什么"，不知道"这个 backend 具体怎么做"。具体怎么做，交给各自的实现类。

这就是接口抽象的核心价值：**让上层逻辑面对稳定的"能做什么"，不面对不稳定的"怎么做"。**

## Registry 是选择后端的唯一地方

`registry.ts` 的 `detectAndGetBackend()` 函数是整个系统里唯一真正知道"用哪个后端"的地方：

```typescript
export async function detectAndGetBackend(): Promise<PaneBackend> {
  if (cachedDetectionResult) return cachedDetectionResult
  
  // 按优先级检查
  if (await isInsideTmux()) {
    const tmux = getTmuxBackend()
    if (await tmux.isAvailable()) {
      cachedDetectionResult = tmux
      return tmux
    }
  }
  
  if (await isInsideITerm2()) {
    const iterm2 = getITermBackend()
    if (await iterm2.isAvailable()) {
      cachedDetectionResult = iterm2
      return iterm2
    }
  }
  
  // 外部 tmux 作为最后选择
  // ...
}
```

选完之后，结果被缓存，全程只有一个后端实例。`PaneBackendExecutor` 拿到的是一个不透明的 `PaneBackend` 对象，不知道也不需要知道它的具体类型。

## 后端注册的自注册机制

`TmuxBackend.ts` 最后一行：

```typescript
// Register the backend with the registry when this module is imported.
// This side effect is intentional - the registry needs backends to self-register
// to avoid circular dependencies.
registerTmuxBackend(TmuxBackend)
```

后端类在被 import 时自动注册到 registry。这避免了 registry 需要显式知道所有后端类（否则会产生循环依赖：registry → TmuxBackend → registry）。

这是一个"自注册插件"模式——每个后端只需要保证自己被 import，就会自动出现在 registry 的可用列表里。`ensureBackendsRegistered()` 通过动态 import 触发这个注册：

```typescript
export async function ensureBackendsRegistered(): Promise<void> {
  if (backendsRegistered) return
  await import('./TmuxBackend.js')   // 触发 TmuxBackend 的自注册
  await import('./ITermBackend.js')  // 触发 ITermBackend 的自注册
  backendsRegistered = true
}
```

## 面试怎么答

如果面试题是"如何设计支持多种后端的 pane 管理系统"：

**正确思路**：把"能做什么"（接口）和"怎么做"（实现）分开。上层逻辑只依赖接口，不依赖实现细节。后端选择逻辑集中在一个地方（registry），而不是散落在各处的条件分支里。

**避免的错误**：把条件分支 `if tmux else iterm2` 分散到多个地方。这会让添加第三种后端变成噩梦——需要在所有条件分支处都加一个新的 else 分支。

**自注册的价值**：后端类自我注册避免了 registry 成为所有后端类的知识中心（知道越多就越难修改）。新增后端只需要两件事：实现接口、导入时自注册。注册的发现和后端的实现完全解耦。

**类比**：这类似于浏览器的内容协商机制——`Accept: image/png, image/jpeg` 列出能接受的格式，浏览器选择最合适的。上层不需要知道具体的图片解码器如何工作，只需要声明支持的能力，让底层自行匹配。
