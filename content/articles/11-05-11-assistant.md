# 面试题：KAIROS 模式下整条状态栏被隐藏的决策，揭示了什么关于"错误信息 vs 缺失信息"的设计哲学？

这道题以一个具体的技术决策为出发点，考察你对"信息质量"和"界面诚实性"的深层理解。它没有单一正确答案，但能说清楚权衡的候选人，才是真正思考过这类问题的人。

## 场景重建

在 KAIROS 激活时，Claude Code 的进程结构变成：

```
用户终端 → REPL 进程（外壳）→ Agent 子进程（真正执行）
             └─ model: claude-opus-4
             └─ cwd: /home/user
             └─ mode: default
                                   └─ model: ??? (可能不同)
                                   └─ cwd: ??? (可能不同)
                                   └─ mode: ??? (可能不同)
```

状态栏能读到的是左边外壳的状态。用户通过状态栏做操作判断时，期待看到的是右边真正执行现场的状态。

## 三种可能的处理方案

**方案 A：继续显示外壳状态**

什么都不改，继续显示 REPL 进程的 model、cwd、mode。

代价：用户会被外壳状态误导。特别是 permission mode——如果外壳是 default 但 agent 子进程是 auto-approve，用户以为需要确认的操作其实会被自动执行。

**方案 B：显示带注释的外壳状态**

在每个字段旁边加 "(outer)" 标注，或者在状态栏最前面加 "WARNING: showing outer process state"。

代价：占用空间，视觉噪音。更深的问题是：用户在快速操作时往往不会注意到细小的注释，误导风险仍然存在。而且这个"双层状态"的 UI 模型对普通用户来说是需要额外学习的认知负担。

**方案 C（当前实现）：整条隐藏**

没有状态栏，用户去 agent 特定的视图查状态。

代价：用户在这个模式下少了一个信息来源，需要主动去找状态。但至少没有被错误信息引导做出错误判断。

## 为什么选方案 C 而不是方案 A 或 B

核心论点是：**用户基于错误信息做出的行动，代价远大于用户因信息缺失而停下来主动查询**。

一个思想实验：

如果状态栏显示 mode=default，用户允许了一个"以为需要确认"的操作，但实际上 agent 子进程在 auto-approve 下早就执行完了——用户的"确认"是对真实已发生事件的事后批准，不是事前控制。这种误导是无法被用户察觉的。

如果状态栏什么都不显示，用户会注意到，会去问"为什么没有状态栏"，会主动找到正确的状态查询方式。这条路有摩擦，但用户始终知道自己不确定，不会在不确定的状态下做出自信的（但可能错误的）判断。

## 这个哲学的更广泛应用

**在数据库设计里**：用 NULL 而不是默认值 0，是同样的原则。NULL 告诉应用"我不知道这个值"，0 告诉应用"这个值是零"。对于财务数据，一个代表"未知"的 NULL 比一个意外的 0 更安全。

**在医疗记录里**：没有过敏记录（NULL）和"已确认无过敏"（false）是两种不同的状态。把 NULL 显示成"无过敏"是危险的——医生会根据这个信息做用药决策。

**在监控系统里**：指标不可用时，好的监控系统会显示"N/A"或者把图表标灰，而不是显示上一个已知值。显示过时数据可能比显示"未知"更危险，因为它给了虚假的精确感。

## Claude Code 中的模式

整条隐藏是 Claude Code 里处理"数据不可信"问题时的一贯选择：

| 场景 | 数据问题 | 处理方式 |
|------|----------|----------|
| KAIROS 激活 | 所有字段代表外壳，不代表执行现场 | 整条状态栏隐藏 |
| remote 模式 | permission mode 不代表远端 agent | 只隐藏 permission mode |
| agent 退场 | owner 字段代表已不在线的协作者 | 只隐藏 owner 字段 |
| task 超出 maxDisplay | 超出部分信息无法完整显示 | 用"…+N 个"摘要替代，不截断不显示 |

注意到模式了吗？**隐藏的范围精确等于不可信数据的范围**——不多不少。

## 面试指导

**哲学问法**："错误信息和缺失信息，哪个对用户危害更大？"

答：通常情况下，错误信息更危险。缺失信息让用户知道自己不知道某件事，会去寻找。错误信息让用户以为自己知道，会直接基于错误假设行动。特别在安全敏感的操作里（permission mode、权限边界），这种差异尤为突出。

**设计追问**："方案 B（显示带注释）你觉得为什么不好？"

答：注释在快速操作时容易被忽视（人在执行习惯性任务时会跳过"看起来正常"的信息）。更深的问题是它引入了"外层状态"的认知负担，要求用户理解进程架构才能解读状态栏，这对大多数用户来说是不合理的要求。

**代码细节追问**："如果 KAIROS 将来有办法从 agent 子进程同步状态，这段逻辑会怎么变？"

答：`statusLineShouldDisplay()` 的判断会改变——不再是简单的"KAIROS 激活就隐藏"，而是"KAIROS 激活 AND 没有可信的 agent 状态数据时才隐藏"。这说明当前的"整体隐藏"是一个保守的临时解法，等待基础设施具备后可以演进到"显示正确的 agent 状态"。

---

*核心考察点：错误信息 vs 缺失信息的哲学思考、界面诚实性原则的系统性理解、设计决策的保守性与未来演进空间。*
