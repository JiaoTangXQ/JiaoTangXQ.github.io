# 主流程无法解释自己为何不失控，所以需要先收口

面试题版本：**"Claude Code 是一个可以执行任意 bash 命令的 AI 工具。你怎么解释它为什么不会因为模型的错误判断而导致灾难性的结果？有哪些层次的约束？"**

## 主流程的能力边界

Claude Code 的主循环在技术上可以：
- 执行任意 bash 命令（通过 BashTool）
- 修改任意文件（通过 FileEditTool）
- 发起任意网络请求（通过 BashTool 或 MCPTool）
- 调用任意 MCP server（取决于已配置的服务器）

这些能力意味着，如果主循环不受约束，一个模型的错误判断（比如误以为用户想删除某个重要文件）可能导致不可逆的损失。

主流程代码本身无法解释"为什么它不会失控"——执行逻辑不包含安全约束，只包含"用户想做什么、调用哪个工具、处理结果"。

## 约束的层次

约束被放在主流程之外的多个层次：

**第一层：工具声明的副作用语义**

工具在定义时声明了自己的影响范围：
```typescript
// BashTool 的声明
isConcurrencySafe(): boolean  // 不并发安全（可能有副作用）
interruptBehavior(): string   // 如何处理中断
```

这个声明让主循环知道某些工具需要特殊处理（不并发、谨慎中断），而不是盲目地并发执行所有工具。

**第二层：权限系统**

`useCanUseTool()` 在每次工具执行前检查权限。权限不只是"允许/拒绝"，还包括：
- 这类操作是否需要用户确认（比如删除文件）
- 在当前上下文（只读 worktree）里是否允许写操作
- 用户是否已经批准了这个特定的操作

**第三层：hooks 系统**

`preAction` hooks 在工具执行前运行，可以：
- 记录操作日志（审计）
- 额外的验证（比如检查文件是否在项目目录内）
- 在某些条件下拒绝执行

**第四层：模型的训练**

模型本身被训练成会在需要权限确认时暂停、会在高风险操作时谨慎、会在用户明确授权时才执行破坏性操作。这是最软的一层约束，但也是最先被激活的。

## "先收口"的具体实现

"先收口"在 Claude Code 里的体现：

**初始化时的权限配置**：会话开始时确定工具权限范围（`PermissionMode`），这个配置在整个会话里是约束，不能被单次工具调用覆盖。

**输入验证在执行前**：`inputSchema.safeParse()` 在任何工具执行前都会运行，非法输入在执行前就被拒绝，不会进入工具的执行逻辑。

**preAction 在副作用前**：`preAction` hooks 在工具产生任何副作用之前运行，这是最后一道可以阻止执行的门。

## 为什么不把约束放进主流程

"把安全检查放在每个工具调用里"的反面教材：

```typescript
// 反面例子：安全逻辑散落在主流程里
if (tool.name === 'bash') {
  if (command.includes('rm -rf')) {
    // 危险命令检测
    if (!await confirmUser('确定要删除吗？')) return
  }
}
```

这种方式：
- 只覆盖了已知的危险模式，无法处理未知的危险情况
- 安全逻辑和执行逻辑耦合在一起，修改执行逻辑时容易破坏安全逻辑
- 很难复用和测试安全逻辑（它们藏在执行流程里）

把约束放在独立的层次（权限系统、hooks），让安全逻辑可以独立演进，覆盖范围不依赖于"已知危险模式的列举"。

## 面试角度的总结

"主流程无法解释自己为何不失控"这个认知，是 Claude Code 的安全设计的起点。正因为主流程本身不包含安全约束，才需要把约束建立在独立的、系统性的层次上：工具声明提供静态属性，权限系统提供动态检查，hooks 提供自定义约束，模型训练提供隐式引导。这四层约束共同构成了"Claude Code 不会轻易失控"的实际基础。
