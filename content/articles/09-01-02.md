# 团队世界比任务世界多的是身份和秩序

任务系统只要知道“这件事是谁在做、做到哪了”；团队系统还得知道“谁代表用户、谁只是 worker、谁能批权限、谁该先说话”。

## 实现链

`spawnInProcessTeammate()` 会先创建 `TeammateIdentity`，里面有 `agentId`、`agentName`、`teamName`、`color`、`planModeRequired`、`parentSessionId`。`InProcessTeammateTaskState` 再把这个身份放进任务状态。与此同时，通信要经过 mailbox，权限要过 leader bridge 或 permissionSync，布局还要分配颜色和 pane。

这说明团队不是在任务系统上多挂一个 `isTeam=true` 就完，而是多出一整层身份和制度字段。

## 普通做法

普通做法往往把多 agent 看成“共享任务系统的多个消费者”，大家只是都能拿任务、发结果。

## 为什么不用

Claude Code 认为这还远远不够。因为一旦进入团队，人面对的不只是任务，还面对角色。谁是 lead，谁是 worker，谁的消息默认可见，谁的权限请求要上收，这些都是任务字段解释不了的。

所以团队世界必须长出独立身份层和秩序层。

## 代价

代价是状态模型更大、更难看懂，也更容易让人觉得“为什么一个 teammate 要这么多额外字段”。答案很简单：因为它不再只是劳动者，而是组织中的一个职位。
