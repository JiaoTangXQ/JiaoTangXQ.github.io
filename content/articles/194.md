# 注册式桥接为什么承认打包现实：扩展接入链的工程诚实

面试题版本：**"Claude Code 的插件/MCP 接入系统里有 manifest 文件、注册表、目录扫描这类机制。这些看起来有些"笨重"的设计是怎么来的？直接动态加载不是更优雅吗？"**

## 什么是打包现实

"打包现实"指的是软件在真实系统里分发和安装时的物理事实：

- 一个 MCP server 是一个独立的程序，需要安装到某个路径
- 不同操作系统（macOS、Windows、Linux）的程序安装路径不同
- 用户可能在系统级安装，也可能在用户级安装，也可能在项目级安装
- 同一个工具可能有多个版本并存
- 程序需要知道自己被允许访问什么资源（manifest 声明）

这些是工程现实，不是设计失误。任何扩展系统如果不正视这些，接入链就会在用户真实环境里频繁出错。

## 注册式桥接做了什么

在 Claude Code 里，`src/services/mcp/config.ts` 维护 MCP server 的配置，支持多个配置层级：

```typescript
export const ConfigScopeSchema = lazySchema(() =>
  z.enum([
    'local',      // 项目本地 .claude/config
    'user',       // 用户全局 ~/.claude/config
    'project',    // 项目级配置
    'dynamic',    // 运行时动态注册
    'enterprise', // 企业管理配置
    'claudeai',   // Claude.ai 平台注入
    'managed',    // 管理员控制
  ]),
)
```

这七种 scope 不是理论上的分层，而是真实用户场景的直接映射：
- 开发者在项目里配置特定工具（`local`/`project`）
- 用户在自己的机器上配置常用工具（`user`）
- 企业管理员强制配置安全合规工具（`enterprise`/`managed`）
- 平台动态注入工具（`dynamic`/`claudeai`）

每种 scope 有不同的优先级、覆盖规则和持久化位置。这个层级结构看起来复杂，但它反映的是真实用户群体中配置管理的实际需求。

## 为什么动态加载不够

动态加载的极端形式：用户把一个 MCP server 的 URL 或路径传给 Claude Code，系统直接连接或加载，不需要任何预先配置。

这个方案的问题：

**无法表达意图**：用户说"我想用这个 server"，但系统不知道这个 server 提供什么能力，需要什么权限，是否安全。manifest 文件让 server 预先声明这些信息，用户在决定是否信任之前可以审查。

**无法跨会话持久**：动态连接只在当前会话里有效，下次重启 Claude Code 需要重新配置。注册到配置文件让工具成为持久的会话环境一部分。

**无法进行管理控制**：企业场景里，安全管理员需要控制哪些 MCP server 可以被使用，哪些被禁止。没有注册机制，就无法实现这种管理控制。

## 真实安装路径的处理

`src/services/mcp/client.ts` 里处理 MCP server 的启动，需要找到 server 的可执行文件路径。这涉及到：

```typescript

function resolveMcpServerPath(config: McpServerConfig): string {
  if (config.command.startsWith('/')) {
    return config.command  // 绝对路径
  }
  // 相对路径：在 PATH 里查找
  // 在项目 node_modules/.bin 里查找
  // 在用户 ~/.local/bin 里查找
  // Windows：处理 .exe 后缀
  // ...
}
```

这段路径查找逻辑不美观，但它处理的是用户真实安装环境里会遇到的情况。跳过这个逻辑，换成"用户自己保证路径正确"，会导致大量的安装失败和无法复现的 bug。

## 工程诚实的价值

"承认打包现实"的对立面是"假装打包现实不存在"——通过高度抽象的接口隐藏分发、安装、配置的复杂性，然后在边界情况下悄悄崩溃或报出难以理解的错误。

Claude Code 选择在接入层把这些复杂性显式处理掉，用更多的代码换取更可靠的用户体验。这是工程诚实：代码量多一些，但行为是可预期的。

## 面试角度的总结

注册式桥接和 manifest 文件看起来"笨重"，是因为它们在承认一个真实世界的约束：软件不是在理论真空里运行的，而是在有操作系统、有安装路径、有用户权限、有企业管控的真实环境里。优雅的动态加载在 demo 里够用，但在覆盖多平台、多角色、多部署场景时会在现实的细节里碎掉。接入层的"笨重"是系统把这些细节消化掉的代价。
