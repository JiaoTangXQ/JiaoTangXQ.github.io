# 终端里每秒 60 帧：Ink 的双缓冲渲染凭什么比 blessed 快 10 倍？

大部分 CLI 工具的"渲染"就是 `console.log`。状态变了？清屏重画。有动画？setInterval 加全量重绘。这对简单工具够用，但 Claude Code 的终端界面有权限对话框、任务列表、流式输出、搜索高亮、选区覆盖、多 Agent 状态面板——复杂度已经接近一个 GUI 应用。

如果还用"清屏重画"的路子，每次击键都会闪烁。

Claude Code 的 Ink 渲染器用了五个关键优化把这个问题解决到了"完全无感"的程度。每一个都值得在性能面试中展开讲。

## 优化一：双缓冲——为什么需要两个帧？

Ink 在内存中始终维护两个帧缓冲区：`frontFrame`（当前显示内容）和 `backFrame`（上一帧显示内容）。每次渲染：

```
1. React 产出新的虚拟 DOM
2. 布局引擎计算每个元素的位置和大小
3. 渲染器把虚拟 DOM 画到一个新的 Screen 缓冲区 → 这就是新的 frame
4. diff(frontFrame, frame) → 计算最小差异补丁
5. 把补丁写到 stdout
6. backFrame = frontFrame; frontFrame = frame（帧交换）
```

为什么要保留 backFrame？因为差异计算需要知道"上一帧画了什么"。如果只有一个缓冲区，你要么全量重绘（慢），要么在原缓冲区上修改后丢失了"修改前是什么"的信息（无法做差异优化）。

双缓冲在游戏引擎里是基础知识，但在终端渲染里很少有人用。大部分终端 UI 库（如 blessed）要么全量重绘，要么维护一个简陋的"脏区域"标记。Claude Code 直接把游戏引擎的双缓冲搬到了终端里，代价是内存翻倍（两个 Screen 缓冲区），但换来了极致的增量更新。

## 优化二：三个对象池——消灭 GC 压力

终端渲染的每一帧都会产生大量临时对象：每个字符的样式、每个单元格的内容、每个超链接的 URL。如果每帧都 new 这些对象，GC 压力会让渲染产生周期性卡顿（GC pause）。

Ink 用了三个池来消灭这个问题：

**CharPool**——字符串驻留池。把字符串映射成整数 ID，后续操作直接比较整数而不是字符串。ASCII 字符走快速路径：用一个 `Int32Array` 直接用 charCode 作为索引查找，连 Map.get 的开销都省了。非 ASCII 字符（中文、emoji）走 Map 路径。

```
'a' → charCode 97 → ascii[97] = 3 → ID 3
'中' → Map.get('中') → ID 847
```

**StylePool**——ANSI 样式驻留池。把 `[bold, red, underline]` 这样的样式组合映射成一个整数 ID。巧妙的是，ID 的最低位（bit 0）编码了"这个样式是否影响空格字符的显示"——有背景色或反色的样式会让空格变得可见。渲染时通过 `id & 1` 就能判断是否需要输出空格，省去了遍历样式数组的开销。

StylePool 还缓存了**样式转换序列**：从样式 A 切换到样式 B 需要输出的 ANSI 转义码。缓存 key 是 `fromId * 0x100000 + toId`（把两个 ID 打包成一个整数），第一次计算后就永远是 O(1) 查表。

**HyperlinkPool**——超链接 URL 驻留池。原理同上，把 URL 字符串映射成整数 ID。

三个池每 5 分钟重置一次，防止长会话中无限增长。重置有成本（需要迁移活跃的引用），但 5 分钟的周期让这个成本完全可忽略。

这个设计模式在面试中叫 **string interning**（字符串驻留），是 JVM（String.intern()）、Python（小整数缓存）、和游戏引擎中的经典优化。在终端渲染场景中用到它比较少见，这也是为什么 Claude Code 的终端体验比同类工具流畅得多。

## 优化三：queueMicrotask——为什么渲染要延迟一个微任务？

React 的 reconciler 在 `resetAfterCommit` 回调中通知 Ink "该渲染了"。但 Ink 没有在这里立即渲染，而是用 `queueMicrotask(this.onRender)` 延迟了一个微任务。

为什么？因为 React 的 commit 阶段有一个时序问题：

```
React commit 阶段：
  1. resetAfterCommit (reconciler 回调)  ← 这里通知渲染
  2. ref 绑定
  3. useLayoutEffect 回调执行            ← 这里设置光标位置
```

如果在步骤 1 立即渲染，光标位置还没被 `useLayoutEffect` 设置好——你会看到光标总是晚一帧跳到正确位置。用户在输入框打字时，光标会有一个肉眼可见的"追赶"效果。

`queueMicrotask` 把渲染推迟到当前事件循环的微任务队列末尾。此时 useLayoutEffect 已经执行完了，光标位置是最新的。而且因为微任务在同一个事件循环 tick 内执行，不会像 `setTimeout(fn, 0)` 那样等到下一个 tick，所以吞吐量完全不受影响。

这个知识点在面试中考的是**事件循环模型**：宏任务 vs 微任务 vs 同步代码的执行顺序。能说清楚 `queueMicrotask` 和 `setTimeout(fn, 0)` 的区别、以及为什么选微任务而不是宏任务，是前端性能面试的加分点。

Ink 还在外面包了一层 `throttle(deferredRender, FRAME_INTERVAL_MS)`——即使 React 一秒钟 commit 100 次，Ink 最多只渲染 60 帧。多余的 commit 会被合并，只有最新的状态被渲染。

## 优化四：帧污染追踪——选区覆盖后的自愈

当用户在终端中选择文本时，Ink 需要在选中区域上覆盖一层反色样式。这个覆盖是直接修改 `frontFrame` 的 Screen 缓冲区：

```
选择前: frontFrame.screen[row][col].style = normalStyle
选择中: frontFrame.screen[row][col].style = inverseStyle  ← 直接修改！
```

问题来了：下一帧做 diff 时，frontFrame 已经被选区覆盖污染了。diff 算法以为"上一帧这个位置是反色的"，会算出错误的差异。

Ink 用 `prevFrameContaminated` 标志解决这个问题。每次覆盖选区后，设置这个标志。下一帧渲染时检测到这个标志，就放弃增量 diff，改为**全量重绘一帧**。全量重绘后清除标志，后续帧恢复增量模式。

```
帧 N:     正常增量渲染
帧 N+1:   用户按下鼠标，选区覆盖污染了 frontFrame
帧 N+2:   prevFrameContaminated=true → 全量重绘（一次性开销）
帧 N+3:   prevFrameContaminated=false → 恢复增量渲染
```

这是一个**自愈机制**：污染是临时的，系统能自动恢复到最优路径。面试中这叫"degradation-then-recovery"模式——遇到异常先降级处理，异常过去后自动恢复正常路径，不需要人工干预。

## 优化五：Alt-Screen 光标自愈——tmux 不再能搞乱你的显示

在 Alt-Screen 模式（Claude Code 的全屏模式）下，有一个恶心的问题：tmux 可能在你不知道的情况下移动物理光标（比如状态栏刷新、pane 切换）。Ink 的 diff 算法基于相对光标移动——如果物理光标被偷偷移走了，所有后续的相对移动都会偏移，内容就会"飘"。

Ink 的解决方案简单粗暴：**每一帧开头先输出 CSI H（Cursor Home），把物理光标强制归零**。然后告诉 diff 算法"上一帧的光标在 (0,0)"。这样不管 tmux 把光标移到了哪里，下一帧都能从已知位置开始。

```
每帧输出序列:
  CSI H          → 光标归 (0,0)
  [差异补丁]      → 增量更新变化的单元格
  CSI row;1 H    → 光标停在底部（输入行所在位置）
```

最后一步也很关键：光标必须停在底部输入行，而不是最后一个 diff 写入的位置。否则 iTerm2 的光标指示线（cursor guide）会在每一帧跳到不同的行，产生严重的闪烁。

为什么主屏模式（非 Alt-Screen）不能用这个技巧？因为主屏的光标位置 y 值可能超过终端可见区域（滚动历史），而 CSI H 只能定位到可见区域内的位置。所以这个优化只在 Alt-Screen 下生效。

主屏模式用了另一套策略：维护光标 y 坐标的"相对移动不变式"——光标始终在内容末尾，通过相对移动（向上/向下 N 行）来定位 diff 写入位置。

## 面试怎么用这些知识

这五个优化覆盖了面试中最常考的几个性能话题：

| 优化 | 面试考点 | 适用题型 |
|------|---------|---------|
| 双缓冲 | 增量更新 vs 全量重绘 | "如何优化一个频繁更新的 UI？" |
| 对象池 | GC 压力、字符串驻留 | "长时间运行的应用内存持续增长怎么排查？" |
| 微任务延迟 | 事件循环模型 | "queueMicrotask 和 setTimeout 有什么区别？" |
| 帧污染追踪 | 降级-恢复模式 | "当缓存被外部操作污染后怎么处理？" |
| Alt-Screen 自愈 | 系统鲁棒性 | "如何让系统在外部干扰下保持正确？" |

如果面试官问"你做过哪些性能优化"，能从实际项目中拿出这种级别的技术细节，比泛泛而谈 memoization 和虚拟列表要有说服力得多。关键不是你用了什么技巧，而是你能不能解释清楚**为什么在这个具体场景下需要这个技巧**。
