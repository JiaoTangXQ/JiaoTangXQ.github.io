Claude Code 的另一个关键做法，是让新世界进门时先学旧世界的话。`setupComputerUseMCP()` 和 `setupClaudeInChrome()` 没把外延能力做成另一套专属协议，而是先把它们包装成动态 MCP 配置和统一命名的工具集合，再决定哪些工具因为整场会话授权而可以绕过单次确认。`remotePermissionBridge.ts` 也没有为远端权限单独发明一套本地审批 UI，而是把远端请求翻译成 synthetic assistant message 和本地可显示的 tool stub，让原来的权限链继续接得住。`LocalAgentTask.tsx` 则把 subagent 明确定义成任务对象，有进度、输出文件、回收时机和通知格式，而不是让它们以“后台线程”身份到处飘。

更常见的做法，是能力一扩张就另起炉灶：browser 有 browser 的规则，remote 有 remote 的审批，subagent 有自己的生命周期。那样每个新世界一开始都比较轻松，但最后系统会长出多套互不兼容的制度，用户也得学习多套心智模型。

Claude Code 明显是故意避开这条路的。它宁可多写一些翻译层、stub、包装器和任务外壳，也尽量不让扩张能力改写原有秩序。这个取舍很合理，但并不便宜：适配层会多，理解时也得多跨几个模块。只是相较于维护两三套并行运行时，这种代价还是小得多。

## 实现链
remote、computerUse、Claude in Chrome 这些新世界都先经过 permission bridge、task 托管或 setup 装配，再接回原主干。

## 普通做法
给每个新世界单独发明自己的审批和运行协议。

## 为什么不用
因为那样扩张越多，系统越像几个拼起来的产品。

## 代价
接入层会变厚，新增能力接入成本更高。
