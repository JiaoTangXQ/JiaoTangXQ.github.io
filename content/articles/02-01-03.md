# 为什么说"能并行的全并行"是个陷阱？

一个新工程师看完 `main.tsx` 的前几行，可能会产生一个很自然的想法：

> 既然提前启动 MDM 和 Keychain 能省 100ms，那我把 `init()` 里的其他操作也提前呢？把配置系统、mTLS 配置、代理设置都塞到 import 期间并行？

这个思路的问题，正是 Claude Code 通过克制展示出来的：**并行不是越多越好，而是要建立在正确的依赖顺序上**。

## 什么样的操作可以提前并行？

`startMdmRawRead()` 和 `startKeychainPrefetch()` 满足以下条件：

1. **无前置依赖**：不需要读取任何其他配置，不需要知道"这次启动是什么模式"
2. **结果稳定**：每次读取结果相同（MDM 策略和 keychain 存储不会在毫秒内变化）
3. **可安全重复**：如果后续代码再次读取，不会因为"已经在后台读过一次"而产生冲突

反例：能不能提前启动 `init()` 里的 `applySafeConfigEnvironmentVariables()`？

**不能**。因为这个函数的行为取决于"允许哪些设置来源"（由 `--setting-sources` 参数控制），而参数解析发生在 `preAction` 阶段，比 `init()` 更早。如果提前调用，就会用错误的设置来源配置环境变量，后面即使重新调用也无法修正已经发生的副作用。

## Bare 模式：并行的反例

`main.tsx` 里有一处很有说服力的设计：

```typescript
function startDeferredPrefetches(): void {
  if (isEnvTruthy(process.env.CLAUDE_CODE_EXIT_AFTER_FIRST_RENDER) ||
      isBareMode()) {
    return;  // 直接返回，跳过所有预取
  }
  // ...后续所有预取操作
}
```

`--bare` 模式（用于脚本化调用，没有交互 REPL）会完全跳过首屏后的所有 deferred prefetch。这包括：
- `initUser()`
- `getUserContext()`
- `getRelevantTips()`
- AWS/GCP 凭据预取
- 文件计数
- 分析门控初始化
- 官方 MCP URL 预取
- 模型能力刷新
- 设置/技能变更探测器

为什么？因为这些预热是为"用户在 REPL 里打字时，后台悄悄把世界准备好"设计的。`--bare` 模式没有用户打字这个时间窗口，预取变成了纯粹的额外开销。

## Trust 边界：另一个不能提前的原因

源码里有一段非常关键的注释：

```typescript
/**
 * Prefetch system context (including git status) only when it's safe to do so.
 * Git commands can execute arbitrary code via hooks and config (e.g., core.fsmonitor,
 * diff.external), so we must only run them after trust is established or in
 * non-interactive mode where trust is implicit.
 */
function prefetchSystemContextIfSafe(): void {
  const hasTrust = checkHasTrustDialogAccepted();
  if (hasTrust) {
    void getSystemContext();
  }
  // Otherwise, don't prefetch - wait for trust to be established first
}
```

`getSystemContext()` 会执行 `git status`、`git log` 等命令。问题在于 git 钩子（`core.fsmonitor`、`diff.external` 等）可以执行任意代码。如果在 trust 确认之前运行 git 命令，就等于在未知项目目录里执行了可能被恶意植入的代码。

这就是为什么这个预取不能无条件提前——它必须等 trust 建立后才能进行。这个设计把"安全性"和"性能优化"的边界切得非常清楚。

## 总结：三条提前并行的必要条件

| 条件 | MDM 读取 | Keychain 预取 | git status |
|------|---------|--------------|-----------|
| 无前置依赖 | ✓ | ✓ | ✗（需要 trust） |
| 结果不受后续配置影响 | ✓ | ✓ | ✗（受 cwd 影响） |
| 在所有启动路径中都有用 | ✓ | ✓ | ✗（bare 模式不需要） |

这张表展示了为什么只有极少数操作适合放进 import 阶段：满足所有三个条件的操作少之又少。

## 面试指导

系统设计题里经常出现"如何优化启动速度"这类问题。高分回答的关键不是"尽量并行"，而是能分析"每个操作的并行前提"。

**陷阱**："所有 I/O 都异步化，全部并行，用 Promise.all 等结果"——这会把有依赖关系的操作也并行了，导致顺序错误。

**正确框架**：
1. 识别哪些操作没有前置依赖（可以最早启动）
2. 识别哪些操作有安全要求（不能在某些条件下运行）
3. 识别哪些操作只在特定模式下有价值（不应无条件预取）

Claude Code 在这个问题上展示了很好的工程克制：快，但不乱；提前，但不盲目。
