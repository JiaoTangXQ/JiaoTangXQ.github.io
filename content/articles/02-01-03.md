# 抢跑式预取为什么不等于乱并行

很多系统一谈性能优化，就容易滑到另一种失控: 能并行的全并行，能预取的全预取。Claude Code 没这么粗糙。它虽然一开始就抢跑，但后面仍然会在真正进入 `init()` 前，用 `preAction` 把前面启动的 MDM 和 keychain 结果先等齐。也就是说，抢跑只是把等待提前隐藏，并没有把依赖顺序抹掉。

更重要的是，它不是在任何模式下都盲目这么做。像 `--bare` 这种极简模式，就会明确跳过大量预热；有些启动后缓存式预取还会看节流时间、看 trust、看是否值得打网络请求。这里透露出来的原则很清楚: 并行不是目的，减少用户真正感受到的阻塞才是目的。

所以 Claude Code 的快，不是靠一股脑把世界点燃，而是靠一种很克制的时间治理。能提前的提前，必须收口的收口，不该抢的就别抢。真正成熟的启动工程，快和稳从来不是二选一。

## 实现链
这页对应的实现链都发生在 `main.tsx` 模块刚开始执行的阶段：先记启动点，再提前打出 MDM 读取、keychain 预取或其他可安全重叠的等待项，真正依赖这些结果的地方再稍后收口。它优化的不是某个函数体，而是整段启动时间河流。

## 普通做法
更普通的做法是把模块加载当成不可控前奏，等进入主逻辑以后再优化；或者一见到慢操作就一股脑并发开跑。

## 为什么不用那种做法
Claude Code 两条路都没选死。它不愿意浪费模块求值阶段的等待，但也不把依赖关系抹平，所以才形成了先开跑、后会合的两段式启动。

## 代价与回报
回报是把等待藏进原本就存在的时间缝。代价是理解入口时要同时记住什么已经启动、什么还不能消费结果。
