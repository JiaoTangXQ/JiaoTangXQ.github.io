# MCP真正管理的是连接和状态，不只是远程调用

## 实现链
- `src/services/mcp/client.ts` 里最重的部分不是 `callTool`，而是连接建立、transport 选择、认证处理、`needs-auth`、session 过期和能力缓存。代码先决定“这条能力通道现在处于什么状态”，再决定能不能调用工具。
- 同一个 MCP server 还可能走 `stdio`、SSE、HTTP、WebSocket 等不同 transport，所以客户端要长期记住连接对象，而不是只在调用时临时发一个请求。

## 普通做法
- 普通理解会把 MCP 当成“远端工具调用协议”，像 HTTP API 一样需要时打一下。
- 这种做法最容易想到，因为表面上看，工具确实只是被调用。

## 为什么不用
- Claude Code 没停在“调用协议”这层，是因为 MCP server 往往带着认证、资源刷新、prompt 列表、长连接状态和会自己变化的能力表。
- 如果把它当一次性接口，失败恢复、会话过期和状态缓存都会无处安放，最后每次调用都像重新搭一遍临时脚手架。

## 代价
- 代价是 `client.ts` 变得很重，读起来不像一个简单 SDK 封装。
- 但这套重法是合理的，因为它对应的本来就是一条常驻能力总线，而不是几次零散 RPC。
