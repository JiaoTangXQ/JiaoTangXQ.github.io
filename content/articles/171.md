# 第十三卷回收索引：界面世界里界面先被承认为运行时一部分，状态世界才真正有地方落屏

## 实现链
界面世界真正要钉住的，不是某个组件长什么样，  
而是界面和状态先后应该站在哪一层。

但现在它已经自然分成两条更窄的线：

- 界面先被承认为运行时的一部分
- 跨子系统事实进入同一个 `AppState` 世界，状态才真正有地方落屏

如果继续把这两件事挤在同一页里，就会把“界面先成为什么”与“状态最后落在哪里”重新揉成一个判断。

所以更稳的做法，是把 `171` 收成父级入口，再往下拆成两页。

这页现在作为父级入口，继续分出去的实现分支是：
### 界面先被承认为运行时的一部分

### 跨子系统事实进入同一个 `AppState` 世界，状态才真正有地方落屏

- [214-第十三卷回收索引-界面世界-跨子系统事实进入同一个AppState世界状态才真正有地方落屏.md](/Users/xxx/tmp/cc/docs/claude-code-book/appendices/214-第十三卷回收索引-界面世界-跨子系统事实进入同一个AppState世界状态才真正有地方落屏.md)

它不再负责细讲两条回收线。  
它只负责保住一个入口判断：

界面和状态要先被当成运行时问题，后面的事实才有地方安放。  
不然屏幕上看起来是在展示，实际上只是把碎片挂在半空。

## 普通做法
更普通的做法，是把界面当展示壳，状态散落在各个组件或服务里，最后再临时拼起来给人看。 对父级入口页来说，更常见的写法就是继续把两三条不同链挤在一页里，读起来省页数，但判断会糊。

## 为什么不用
Claude Code 不这么做，是因为跨子系统事实如果没有同一个落屏世界，界面很快就会出现各说各话的假真相，长会话尤其容易失控。 这些父级页现在存在，就是为了先把分支关系讲清，再让更细页分别承担细节。

## 代价
代价是 AppState 会更大，显示层也得承担一部分制度、接缝和注意力管理逻辑。 作为父级入口，它还额外带来导航变深、读者要多跳几页的成本。
