# autoCompact 是怎么判断该不该压缩上下文的？

## 先看数字

打开 `autoCompact.ts`，首先看到一堆常量：

```typescript
export const AUTOCOMPACT_BUFFER_TOKENS = 13_000
export const WARNING_THRESHOLD_BUFFER_TOKENS = 20_000
export const ERROR_THRESHOLD_BUFFER_TOKENS = 20_000
export const MANUAL_COMPACT_BUFFER_TOKENS = 3_000
const MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3
const MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20_000
```

注释里还有一条很有意思的数据：
> BQ 2026-03-10: 1,279 sessions had 50+ consecutive failures (up to 3,272) in a single session, wasting ~250K API calls/day globally.

这就是 `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3` 存在的原因——这是真实生产数据驱动的工程决策。

## 四个阈值的层级结构

`calculateTokenWarningState()` 返回一个结构，对应四个不同的"警戒线"：

```
|---------|-------------|------------|---------|------------|
0      warning        error     autocompact   blocking  window
         -20k         -20k        -13k          -3k
```

- **warning**：token 快用完了，UI 开始显示警告色
- **error**：更严重的警告，颜色更红
- **autocompact**：触发自动压缩的阈值
- **blocking**：绝对上限，超过这里连手动 compact 都被阻止

注意 autocompact 和 blocking 之间还有 10k token 的缓冲区。这是刻意留给"compact 过程本身产生的 token"的——compact 调用模型生成摘要，这个摘要本身会占用 token。如果 autocompact 在离 blocking 只剩 1k 的地方触发，compact 过程中就可能撞墙。

## effective context window 的计算

并不是直接用模型官方的 context window。`getEffectiveContextWindowSize()` 会减去摘要输出的保留量：

```typescript
const reservedTokensForSummary = Math.min(
  getMaxOutputTokensForModel(model),
  MAX_OUTPUT_TOKENS_FOR_SUMMARY,  // 20,000
)
return contextWindow - reservedTokensForSummary
```

注释说明了这个 20,000 token 的来源：实测 p99.99 的 compact 摘要输出是 17,387 token，向上取了个整数安全边。

## 电路断路器

`AutoCompactTrackingState` 里有一个 `consecutiveFailures` 计数器：

```typescript
type AutoCompactTrackingState = {
  compacted: boolean
  turnCounter: number
  turnId: string
  consecutiveFailures?: number
}
```

每次 autocompact 失败（比如返回 `prompt_too_long`），计数器 +1。超过 `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3`，就停止继续触发 autocompact。

"prompt_too_long" 错误发生时，意味着上下文已经彻底超出限制，autocompact 本身也无法在这个 context 下成功运行。继续重试只会循环失败，每次都消耗一次 API 调用。这是典型的"检测到失效模式就早退"的电路断路器设计。

## 环境变量覆盖

这个系统还有两个测试用的逃生舱：

```typescript
const autoCompactWindow = process.env.CLAUDE_CODE_AUTO_COMPACT_WINDOW
const envPercent = process.env.CLAUDE_AUTOCOMPACT_PCT_OVERRIDE
```

前者直接覆盖 context window 大小，后者把阈值改为百分比。在集成测试中可以用极小的 window 快速触发 compact 逻辑，而不需要等真实的 200k token 累积。

## isAutoCompactEnabled() 的三重检查

```typescript
export function isAutoCompactEnabled(): boolean {
  if (isEnvTruthy(process.env.DISABLE_COMPACT)) return false
  if (isEnvTruthy(process.env.DISABLE_AUTO_COMPACT)) return false
  return getGlobalConfig().autoCompactEnabled
}
```

两个环境变量，一个用户配置。`DISABLE_COMPACT` 是"连手动 compact 也禁掉"，`DISABLE_AUTO_COMPACT` 是"只禁自动，保留手动"。用户配置在最后兜底。

## 面试指导

被问上下文管理时，这道题要说清楚三层：1）四阈值结构而不是单一阈值（warning/error/autocompact/blocking）；2）有效 context window 减掉了摘要输出保留量；3）电路断路器防止连续失败浪费 API 调用。最后一点最能体现工程成熟度——这是真实生产数据（25 万次/天被浪费）推动的设计，不是凭空臆测的安全措施。
