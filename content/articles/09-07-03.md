# 颜色的生命周期：为什么 teammate 的颜色要跟着团队的生命周期走

`teammateLayoutManager.ts` 里有两个简单的函数，但它们的设计值得认真看一眼：

```typescript
const colorAssignments = new Map<string, AgentColorName>()

export function assignTeammateColor(teammateId: string): AgentColorName {
  // 同一个 ID 永远返回同一个颜色
  const existing = colorAssignments.get(teammateId)
  if (existing) return existing
  // ... 分配新颜色并存储
}

export function clearTeammateColors(): void {
  colorAssignments.clear()
}
```

颜色分配是稳定的（同一个 teammate 始终一个颜色），但可以被完全重置。

这看起来很简单，但这两个函数一起定义了颜色的生命周期，而这个生命周期的选择有具体的工程理由。

## 为什么需要稳定颜色

在一个团队会话的生命周期里，researcher 的颜色从蓝色变成了绿色，会发生什么？

**tmux pane 层面**：pane 边框的颜色变了，但 pane 本身还在同一个位置。用户短暂困惑"这个绿色边框是新来的 teammate 还是原来的 researcher？"

**权限弹窗层面**：researcher 的权限请求在 Leader UI 里显示的 badge 颜色变了。用户在过去一小时里建立了"蓝色 = researcher"的记忆，绿色 badge 出现时需要重新识别。

**心智模型断裂**：颜色是用户在并发 agent 工作时维持"谁在做什么"心智模型的关键工具。频繁的颜色变化会让这个心智模型不停地被破坏和重建，增加认知负担。

**稳定颜色的结论**：在同一个团队会话里，每个 teammate 的颜色不应该变化。用户建立的视觉映射应该在整个会话里有效。

## 为什么需要会话结束时重置

如果颜色永远不重置，会有另一个问题：

第一个团队会话：researcher 是蓝色，reviewer 是绿色，tester 是红色。

第二个团队会话（可能在几天后）：用户希望 researcher 还是蓝色（符合上次的记忆），但如果这次 reviewer 先加入，它可能抢了蓝色……

实际上，颜色池的大小是有限的（8-10 种颜色）。如果颜色分配不重置，随着时间推移，颜色 Map 会越来越大，最终所有颜色都被占用，新 teammate 无法分配颜色。

但更重要的是：**每个团队会话是独立的，颜色映射不需要跨会话持久化**。用户不会在第二个团队会话里期望颜色和第一个一样——因为这是一个新团队，成员可能不同，任务不同，上下文不同。

`clearTeammateColors()` 在团队清理时被调用（`cleanupTeamDirectories` 之后），给下一个团队会话一个干净的颜色分配起点。

## 同一个颜色 Map 不能被两个团队共享

还有一个微妙的问题：同一时刻可能有两个团队同时运行（虽然不常见）。

如果颜色 Map 是模块级单例（如现在的实现），两个团队会共享同一个 Map，颜色分配可能产生冲突：团队 A 的 researcher 是蓝色，团队 B 也想要蓝色，但蓝色已经被团队 A 占了。

这是一个已知的设计局限。在当前实现里，颜色 Map 对整个进程是单例的，不区分团队。对于多团队并发（不常见的场景），颜色可能"被占用"而无法按预期分配。

解决这个问题的方案是把颜色 Map 改为按团队 key 存储：`Map<teamName, Map<teammateId, AgentColorName>>`。但这会增加复杂度，而多团队并发不是 Claude Code 的主要场景，所以接受了这个已知的局限。

这是一个工程取舍的例子：完美的方案存在，但在当前的使用场景下，简单方案的代价是可接受的。

## 颜色生命周期 = 团队生命周期

把上面的分析综合起来：

- **会话内稳定**：保证用户的视觉映射不被打断
- **会话结束重置**：避免颜色 Map 无限增长，给新会话干净起点
- **不跨团队持久化**：每个团队是独立的，颜色映射不需要跨团队延续

这三条规则一起定义了颜色的生命周期：**颜色随团队会话创建，随团队会话结束**。

这和 teammate 的身份信息（TeammateIdentity）的生命周期是一致的：身份随 teammate 加入团队创建，随 teammate 退出或团队清理而消失。颜色作为身份的视觉编码，生命周期应该和身份保持同步。

## 面试怎么答

如果面试题是"如何设计一个稳定的颜色分配系统"：

**核心设计**：稳定映射（相同输入 → 相同输出）+ 显式重置点（生命周期边界处清理）。

**不推荐的方案**：随机分配（每次可能不同），或者基于哈希分配（理论上稳定，但哈希冲突可能导致颜色碰撞）。

**生命周期思考**：任何持久化的状态（Map、数组、缓存）都需要明确它的生命周期——什么时候创建，什么时候销毁，什么时候需要重置。没有明确生命周期的状态，最终会变成内存泄漏或者陈旧数据的来源。

**加分项**：识别"简单方案有已知局限，但在当前场景下可接受"是工程成熟度的表现。颜色 Map 不区分团队是一个已知局限，但在单团队场景（主要使用场景）下不会产生问题。选择简单方案并清晰地知道它的局限在哪里，比盲目追求完美方案要好。
