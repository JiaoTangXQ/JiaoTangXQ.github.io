# Claude Code 的七个世界如何连成一台机器？

## 架构总结型问题的标准

面试中遇到这类问题，回答"有很多层"没有价值。真正有价值的回答，是能说清楚层与层之间靠什么连接，为什么不会散架。

Claude Code 的代码给出了一个具体答案：每一层只对一件事负责，不顺手替下一层做事。

## 七层的分工

**第一层：入口层**。`main.tsx` 负责启动时的世界判断。它先读模式（CLI、headless、remote、coordinator），再决定哪些能力该提前预热，把会话初始化之前的准备工作全部做完。它不持有对话，只决定进入哪个世界、带着哪些初始条件进去。

**第二层：会话壳**。`QueryEngine` 负责"同一场会话怎样持续存在"。它拥有一整个会话周期内的消息历史、文件状态快照、权限拒绝记录和 token 使用量。它不决定工具怎么跑，只保证会话不丢失状态。

**第三层：采样与工具执行**。`query.ts` 里的主循环负责实际的模型采样和工具调用。它从 `QueryEngine` 拿到会话状态，把结果持续产出成标准消息，再交给下游消费。它不管界面，也不直接改任务状态。

**第四层：工具协议**。`Tool.ts` 定义每个工具是什么，能做什么，需要哪些权限。它只是协议定义，不参与任务管理和显示逻辑。

**第五层：任务对象**。`Task.ts` 把劳动做成正式工作单元。有 `taskId`、`status`、`outputFile`、`startTime`、`endTime`，有 `isTerminalTaskStatus()` 判断终态。任务结束时，输出文件路径和通知格式都已经整理好，不需要消费者各自猜。

**第六层：事实状态**。`AppStateStore` 收拢界面当前相信什么是真的。设置、模型、任务列表、MCP 连接、bridge 状态、通知、伴侣状态——都在这里。界面不自己推断真实状态，只读这一份。

**第七层：制度层**。权限、settings、hooks 在能力上场之前先决定"哪些能力有资格进场"。这一层不在工具跑出去后再拦，而是在会话开始时先把可疑通道关掉。

## 为什么七层能连成一台机器

七层之所以能连在一起，靠的不是一个总控模块把所有人拧在一起，而是每一层都只通过约定好的接口交接。

`main.tsx` 把初始化好的世界交给 `QueryEngine`；`QueryEngine.submitMessage()` 把会话状态的积累和 transcript 记录完成后，把真正的采样交给 `query()`；`query()` 把标准消息和工具结果交给 `AppStateStore`、任务层和 UI；`LocalAgentTask` 把子 agent 的输出整理成统一格式再交出去。

每一棒都是标准件，不是口头约定。所以问题出在某一棒时，至少知道去哪一层找责任。

## 普通做法的问题

更常见的写法是把这七件事混进一个大 orchestrator：输入来了就拼 prompt，模型回了就顺手调工具，工具结束再顺手改 UI，权限也顺手内嵌在工具里。这样一开始写得快，但功能一多就出问题——谁都能顺手改一点，谁也没有真正对某件事负责。

Claude Code 写得更慢，但长大以后不散架。

## 面试指导

被问到"你能描述一下这个系统的架构吗"时，不要背层的名字，要说清楚：每层的责任边界在哪里，靠什么接口交接，为什么不互相踩踏。

能说清楚"层之间怎么连"的候选人，比只能说"有几层"的候选人，技术成熟度高出一个量级。
