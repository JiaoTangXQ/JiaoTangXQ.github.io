很多 agent 产品会把“很强”做成一个长清单：能搜、能改、能跑命令、能分派子 agent、能接远端、能接浏览器。Claude Code 的代码里更值钱的地方，不是这张清单够长，而是它一直在逼自己回答“这件事到底该由谁负责”。`main.tsx` 先决定进哪个世界、哪些能力该开、哪些前置读取要抢跑；`QueryEngine` 专门拥有一整个会话的消息、文件状态、权限拒绝和使用量；`Tool.ts` 只定义工具协议，不抢任务和界面的话语权；`Task.ts` 把劳动正式做成可跟踪、可终止、可通知的任务对象；`AppStateStore` 再把界面真正要展示的事实收在一个地方。

更普通也更容易想到的实现，是做一个大 orchestrator：用户输入来了就拼 prompt，模型回了就顺手调工具，工具结束再顺手改状态，UI 只是把结果打印出来。那样一开始写得快，但功能一多就会出现一个问题：谁都能顺手改一点，于是没有一层真的对某件事负责。Claude Code 没选这条路，所以你会看到很多“看似啰嗦”的边界：命令是一层，工具是一层，任务是一层，显示状态又是一层。

这不是为了把架构图画漂亮，而是为了让系统在能力继续长的时候还能知道自己在干什么。代价也很明确：理解门槛更高，很多简单需求都要跨几层才能看完。但如果目标是长期工作的 agent 工作台，而不是一段一次性魔术，这个代价是值得付的。

## 实现链
`main.tsx`、`QueryEngine`、`Tool`、`Task`、`AppStateStore` 各自只占一段责任链，不把入口、行动、任务和显示混成一个大 orchestrator。

## 普通做法
围着一个中心调度器不断加分支。

## 为什么不用
因为那种做法在功能多起来后会失去责任边界。

## 代价
代码层次更厚，简单问题也要跨层看。
