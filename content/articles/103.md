# Claude Code 退出 auto 模式时，为什么要"归还"用户的历史特权？

这是一个系统设计层面容易被忽视的问题：用户平时积累的操作习惯（比如"这个目录我总是允许写"），进入 auto 模式后会怎么处理？

## auto 模式的本质是临时冻结

Claude Code 进入 auto 模式（dangerously allow all）时，系统会做两件事：

1. 剥离那些即使 allow-all 也不应该放开的危险工具
2. 暂时冻结用户历史积累的逐条权限许可

第二件事才是关键。用户可能过去跑过很多 session，每次都手动批准了某些操作，这些批准记录存在权限层的 `permissions` 对象里。进入 auto 模式时，这些历史记录不是被删除，而是被冻结——系统知道它们在那里，只是在这个模式下它们变得冗余（因为 auto 已经放行了更多东西）。

## 为什么退出时必须归还

当用户决定退出 auto 模式，期望回到"我来手动决定每一步"的状态时，如果历史积累的逐条许可在退出时没有被恢复，会发生什么？

用户会发现：他之前已经批准过 100 次的操作，现在又开始问了。这是因为系统在进入/退出 auto 模式时，把用户积累的权限习惯给清空了。

这不只是体验问题，更是一个**信任问题**：用户相信系统在记录他的决定，结果发现模式切换会悄悄清除他的历史。

## 制度语境的借贷模型

Claude Code 的权限层把这个问题建模成"借贷"：

- 进入 auto 模式 = 系统借走了用户管理逐条权限的责任
- 退出 auto 模式 = 系统把这个责任归还给用户，**连同他之前积累的历史记录**

`ToolPermissionContext` 的设计允许权限对象知道自己的来源。每一条许可都有标注：它是从哪个层引入的？是项目配置？用户历史？还是 auto 模式的临时扩展？

这个溯源能力让"归还"变得可能：退出 auto 模式时，只需要移除 auto 模式层贡献的临时扩展，保留其他层的原始状态，用户历史习惯就自然恢复了。

## 暂时冻结危险特权的合理性

有人可能会问：如果 auto 模式是用户主动打开的，为什么还要保留对危险工具的限制？

这里有一个很重要的区分：

- **用户历史习惯**（比如"我总是允许写 /tmp 目录"）：进入 auto 模式后应该保留，因为这是用户明确表达的意愿
- **危险工具保底限制**（比如"运行系统命令时不管什么模式都要先显式授权"）：这是系统的安全底线，不会因为 auto 模式而取消

两者的区别在于**意图层次**：用户历史习惯是用户对具体操作的判断，危险工具限制是系统对风险类别的判断。auto 模式只是放松了"每次都手动确认"的要求，但没有改变"哪些东西本质上是危险的"这个判断。

## 权限层的成熟度标准

一个成熟的权限层应该能回答这几个问题：

1. 这条权限是从哪里来的？（溯源）
2. 在当前制度语境下，它是有效的还是被暂时冻结的？（语境感知）
3. 当语境切换时，我需要恢复什么、丢弃什么？（生命周期管理）

Claude Code 的 `ToolPermissionContext` 在设计上要能回答这三个问题。一个只能回答"这个操作现在允不允许"的权限层，是一等公民；只能回答"这个开关打开了没有"的，是三等公民。

## 面试答题策略

被问"你的权限系统如何处理模式切换"时，关键是要表现出对**权限状态生命周期**的理解：

权限状态不是静态配置，它在不同运行模式下有不同的有效性。成熟的设计会把权限变更做成带溯源的操作，而不是直接覆盖。这样在模式回退时，可以精确恢复到之前的状态，不会意外清除用户的历史积累。

追问"如果用户在 auto 模式里手动批准了一个操作，退出后这条批准还在吗"，这是个好问题：这取决于该批准是在"auto 模式临时扩展层"还是"用户历史层"。如果系统设计正确，用户在 auto 模式里的手动批准，应该直接写入用户历史层，这样退出 auto 模式后它仍然有效。
