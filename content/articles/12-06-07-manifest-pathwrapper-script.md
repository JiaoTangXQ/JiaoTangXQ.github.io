# 协议分析：Chrome Native Host manifest 的 path 字段为什么不能带参数？

## Chrome Native Messaging 协议的这条规定

Chrome 的 Native Messaging 规范要求 native host manifest 的格式：

```json
{
  "name": "com.anthropic.claude_code_browser_extension",
  "description": "Claude Code Browser Extension Native Host",
  "path": "/absolute/path/to/executable",
  "type": "stdio",
  "allowed_origins": ["chrome-extension://..."]
}
```

规范明确：`path` 必须是可执行文件的绝对路径，**不能包含参数**。

这是 Chrome 协议设计的结果，逻辑是：Chrome 直接 exec 这个路径，然后通过 stdin/stdout 建立通信。如果路径里带参数，Chrome 在解析时会把整个字符串当路径，找不到那个文件，启动失败。

（与之对比：其他协议，如 VS Code 的语言服务器协议，允许在 manifest 里指定命令和参数数组。Chrome 选择了更严格的格式。）

## Claude Code 为什么需要参数

Claude Code 是一个多功能二进制文件，不同的 `--flag` 参数决定它以哪种模式运行：

- 默认：Claude Code CLI
- `--chrome-native-host`：Chrome Native Host 模式，处理 Chrome 扩展和本地 socket 的通信
- `--claude-in-chrome-mcp`：Claude in Chrome 的 MCP server 模式
- `--computer-use-mcp`：Computer Use 的 MCP server 模式

如果没有 wrapper script，manifest 里的 `path` 只能指向裸的 Claude Code 二进制，没有参数，Claude Code 就会以默认 CLI 模式启动，而不是 Native Host 模式。这显然不行。

## createWrapperScript() 的实现

```ts
async function createWrapperScript(command: string): Promise<string> {
  const platform = getPlatform()
  const chromeDir = join(getClaudeConfigHomeDir(), 'chrome')
  
  // 平台特定的文件扩展名
  const wrapperPath = platform === 'windows'
    ? join(chromeDir, 'chrome-native-host.bat')
    : join(chromeDir, 'chrome-native-host')

  // 平台特定的脚本内容
  const scriptContent = platform === 'windows'
    ? `@echo off\nREM Chrome native host wrapper script\nREM Generated by Claude Code - do not edit manually\n${command}\n`
    : `#!/bin/sh\n# Chrome native host wrapper script\n# Generated by Claude Code - do not edit manually\nexec ${command}\n`

  // 内容不变则复用现有文件
  const existingContent = await readFile(wrapperPath, 'utf-8').catch(() => null)
  if (existingContent === scriptContent) { return wrapperPath }

  // 写入并设置权限（Unix）
  await mkdir(chromeDir, { recursive: true })
  await writeFile(wrapperPath, scriptContent)
  if (platform !== 'windows') { await chmod(wrapperPath, 0o755) }
  
  return wrapperPath
}
```

传入的 `command` 参数在不同场景下是：
```ts
// Bundled 版本（native binary）
const execCommand = `"${process.execPath}" --chrome-native-host`

// 非 bundled 版本（通过 Node.js 运行 cli.js）
const execCommand = `"${process.execPath}" "${cliPath}" --chrome-native-host`
```

生成的 Unix wrapper script 内容类似：
```sh
#!/bin/sh
# Chrome native host wrapper script
# Generated by Claude Code - do not edit manually
exec "/usr/local/bin/claude" --chrome-native-host
```

manifest 的 `path` 字段指向这个脚本，脚本里包含了 Claude Code 真正需要的参数。

## exec 而不是直接调用

注意 Unix 脚本里用的是 `exec ${command}` 而不是直接 `${command}`：

```sh
#!/bin/sh
exec "/path/to/claude" --chrome-native-host
```

`exec` 会用新程序替换当前 shell 进程（而不是 fork 一个子进程）。这意味着：
- 没有额外的 shell 进程在运行，只有 Claude Code 进程
- Claude Code 的 PID 就是 Chrome 启动的那个进程的 PID，不是 shell 的 PID
- 进程树更干净，资源开销更小

如果用普通调用（没有 `exec`），会有一个 shell 进程和一个 Claude Code 进程同时运行，Chrome 在 Native Host 协议里追踪的是 shell 进程的 PID，而实际处理通信的是 Claude Code 进程。这会让进程管理复杂化。

## 固定落点 ~/.claude/chrome/

wrapper script 固定放在 `~/.claude/chrome/` 目录，而不是系统临时目录：

```ts
const chromeDir = join(getClaudeConfigHomeDir(), 'chrome')
// → ~/.claude/chrome/
```

固定落点有几个好处：
1. manifest 里的 path 是稳定的，不会每次重启变化
2. 调试时容易找到这个文件
3. 不同版本的 Claude Code 升级后，wrapper 脚本会被更新（内容变化会触发重写），但路径不变，manifest 不需要更新

如果用随机临时目录，manifest 里的 path 每次都变，需要每次重写 manifest 并触发 reconnect。

## 面试考察点

这道题考察的是**在第三方协议约束下的接缝层设计**。

当第三方系统（这里是 Chrome）有一些固定的接口约定（`path` 字段不能带参数），而你的系统有自己的需求（需要传参数），中间就需要一个适配层。

这种情况在工程里很常见：
- CDN 只支持 GET 请求，而你的 API 用 POST → 在 CDN 前加一层转换
- 某 SDK 只接受文件路径，而你有内存数据 → 写临时文件，用完删除
- Chrome manifest path 不带参数，而你需要参数 → 写 wrapper script

解决思路总是类似：**在你控制的部分增加一层，在那层里做协议两侧的转换**。wrapper script 就是这层转换的具体实现。
