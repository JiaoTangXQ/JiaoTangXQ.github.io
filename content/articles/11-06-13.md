# stringWidth + truncateToWidth：任务行在紧守屏幕预算

`TaskListV2.tsx` 里渲染一行任务时：

```typescript
const ownerWidth = stringWidth(ownerLabel);
const truncatedSubject = truncateToWidth(subject, columns - 15 - ownerWidth);
const truncatedActivity = truncateToWidth(activity, columns - 15);
```

先量 owner 占了多少宽，再把 subject 压进剩余空间，activity 也单独限宽。

## 为什么要精确到列

任务面板的作用是让用户「扫一眼，判断现状，继续工作」。每一行都应该在一眼扫过去的时间里传达完它的信息，而不是要求用户停下来读完。

如果不限宽，一个长的 agent 名加一段长的 activity 描述，很可能把这行变成两行、三行甚至更多。任务列表就变成了活动日志，每次打开都要滚很长，「扫一眼」变成了「认真阅读」。

限宽是在主动保持信息密度在可接受范围内——每条任务就是一行，这一行里装的是最值得看的那些字。

## stringWidth 而不是 .length

`stringWidth` 而不是字符串 `.length`，是因为终端里的字符宽度不是固定的。中文、日文、韩文字符通常占两列宽，emoji 有时更宽。如果用字节长度或字符数来算，中文名字的 owner 就会显示不对，后面的 subject 可能被截断在奇怪的位置。

`stringWidth` 是在正确地量「这串字符实际占用多少列」，而不是简单地数「有多少个字符」。

## 状态用样式，不用额外行

`isInProgress`、`isCompleted`、`isBlocked` 三种状态用粗体、删除线和变灰来区分，而不是追加额外的状态行。这是同样的策略：把信息压缩进一行能表达的范围内，不为了信息更完整而扩展布局。

任务面板的节制感来自于这种持续的「我能用多少列就用多少」的约束意识。
