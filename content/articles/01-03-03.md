# 命令为什么更像入口，而不是行动本身

很多人第一次看到斜杠命令，会本能地把它理解成“系统能力本体”。Claude Code 的设计更克制。命令不是能力本身，而是用户进入系统的入口层，是把一句人话导向正确执行世界的分流装置。

这里最关键的不是命令名字，而是命令类型。Claude Code 明确把命令分成 `prompt`、`local`、`local-jsx` 等几种路线。有的命令会把一段新制度装进当前会话，让主循环继续往下跑；有的命令由本地程序立刻做完；还有的命令先拉起一段临时界面，让人参与选择。也就是说，命令层首先决定的是“这次该走哪条路”，而不是“用哪只手去碰世界”。

这就是它优雅的地方。用户看到的是统一的命令面，内部却保留了完全不同的执行方式。产品可以不断长出新入口，但真正行动的能力层不必跟着变乱。命令像前台接待，负责理解来意、安排入口、分派路线；它重要，但它并不是那个真正下场干活的人。

如果把命令和工具混成一层，系统很快就会变得既难解释，也难扩展。Claude Code 先把入口和行动拆开，后面整套命令系统、技能系统和插件系统才有了共同的落脚点。

## 实现链
这一章对应的实现链是角色拆分：命令负责把人带进正确入口，工具负责真正伸手做事，任务负责把劳动变成正式对象，子 agent 则站在 `AgentTool` 和任务生命周期之间。代码没有把这些角色混成一个万能 handler，而是沿 `commands.ts`、`Tool.ts`、`Task.ts` 和 AgentTool/LocalAgentTask 这几层拆开。

## 普通做法
更普通的做法是让命令既做入口又做动作，必要时再在某个回调里顺手记状态、起并发。这样局部很快，也很像脚本。

## 为什么不用那种做法
Claude Code 不这么写，因为它需要共享制度：权限、状态显示、恢复、后台托管、团队协作都要求这些角色各守边界。角色一旦混写，后面每加一种制度都只能去各处补洞。

## 代价与回报
回报是系统里谁负责入口、谁负责动作、谁负责生命周期更清楚。代价是层次比普通脚本工具更正式，理解时要接受“会做事”和“被系统正式管理”不是一回事。
