# 分层不稀奇，真正稀奇的是交棒不烂尾——Claude Code 怎么做到的？

## 分层的常见失败模式

几乎所有稍微认真写的系统都会说"我们有分层"。但分层本身不解决问题，真正决定系统质量的是层与层之间的交棒是否干净。

最常见的失败模式不是"没有分层"，而是"上游顺手把下游也做了"。输入层自己决定哪些消息该显示，查询层自己直接改 UI，远端模块自己发明一套审批流程。每个局部看起来都在省事，但接缝处最容易出错——特别是在恢复、中断和跨模式切换时。

## Claude Code 的交棒方式

**第一棒：main.tsx 到 QueryEngine**

`main.tsx` 不自己持有对话，它做完世界判断和能力预热之后，把初始化好的世界交给后续运行时。它交出去的是一个已经准备好的"舞台"，而不是"我顺便帮你做了前三步"。

**第二棒：QueryEngine 到 query()**

`QueryEngine.submitMessage()` 负责会话状态的积累和 transcript 更新，再把真正的采样与工具执行交给 `query()`。这两件事分开很重要：`QueryEngine` 是会话的持久容器，`query()` 是单轮的执行引擎。不混在一起，所以 `query()` 可以被中断、重试，而不影响会话历史。

**第三棒：query() 到 AppStateStore、任务层和 UI**

`query()` 持续产出标准消息和工具结果。它不自己改 `AppStateStore`，也不直接操作 UI，而是让消费者各自去读。这样当 UI 需要变更显示方式时，不需要改 `query()`；当任务层需要增加新字段时，也不需要改 `query()`。

**第四棒：LocalAgentTask 到通知和 UI**

任务结束时，`LocalAgentTask` 把状态、输出文件路径和通知格式整理成统一格式。它不假设消费者是谁，只保证交出去的是标准件。通知系统、UI 和日志层各自消费自己需要的部分。

## 为什么这种方式更稳

这种交棒方式对三种场景特别友好：

**恢复场景**：当连接断开需要从某个状态恢复时，因为每一层交出的都是标准件，恢复点很清晰——从 `QueryEngine` 里找到最后一条 transcript，从 `AppStateStore` 里找到最后一份事实状态，就可以重新从那一棒开始。

**中断场景**：`query()` 被中断时，`QueryEngine` 的 transcript 状态不受影响，因为两者的责任边界是分开的。中断的是执行，不是历史。

**跨模式切换场景**：从 CLI 模式切换到 remote 模式时，因为会话状态在 `QueryEngine` 里，切换只是换了一个显示层，不需要重建对话历史。

## 代价

链路更长是真实的代价。阅读时需要跨 `main.tsx`、`QueryEngine`、`query.ts`、`Task.ts`、`AppStateStore` 多个文件才能理解一次完整的消息流转。对于新读者来说，第一次进入这个代码库的门槛确实高。

但问题出在某一棒时，至少知道应该去哪一层找责任。比起"大家都在顺手替别人做事"的系统，这是一种更可维护的负担。

## 面试指导

被问"这套系统如何保证层与层之间不互相干扰"时，可以用"交棒标准件"这个概念来回答：不靠口头约定，靠每一层交出的都是下游真正听得懂的格式。

举具体例子：`QueryEngine` 和 `query()` 的职责分离，正是为了让恢复和中断有清晰的操作点。
