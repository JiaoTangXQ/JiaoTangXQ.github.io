分层本身不稀奇，真正稀奇的是交棒不烂尾。Claude Code 的代码里，很多关键设计都不是“这一层多聪明”，而是“这一层只把下游真正需要的东西交出去”。`main.tsx` 不自己持有整场对话，而是把初始化过的世界交给后续运行时；`QueryEngine.submitMessage()` 负责一轮会话的状态积累和 transcript 记录，再把真正的采样与工具执行交给 `query()`；`query.ts` 不自己承担界面展示，而是持续产出标准消息和工具结果，让 `AppStateStore`、任务层和 UI 去消费；任务结束时，`LocalAgentTask` 又把状态、输出文件和通知整理成统一格式，避免每个消费者各猜各的。

更容易想到的写法，是让上游顺手把下游也做了。比如输入层自己决定哪些消息该显示，查询层自己直接改 UI，remote 或 browser 模块自己发明一套审批流程。那样每个局部都看似更省事，但最后会变成谁都在替别人做半步，接缝处最容易出错。

Claude Code 现在这种交棒方式明显更稳，尤其对恢复、中断和跨模式切换更友好，因为每一层拿到的都是自己听得懂的“交接件”。缺点是链路更长，阅读时要多跳几层；好处是问题出在某一棒时，至少知道应该先去哪一层找责任。

## 实现链
`main.tsx -> QueryEngine -> query() -> Task/UI` 之间靠标准消息、任务状态和 transcript 交棒，而不是上游顺手替下游做事。

## 普通做法
哪层拿到信息就顺手把后面的工作也做了。

## 为什么不用
因为交棒不清楚时，恢复和跨模式切换最容易烂尾。

## 代价
链路会更长，阅读时需要多跳几层。
