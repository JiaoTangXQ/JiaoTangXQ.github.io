# Command 类型是怎么成为运行时通用语言的？翻译层的设计代价

## 问题

MCP 服务器能提供工具、资源、prompts、甚至 skills。本地技能是 Markdown 文件。插件有 manifest 和来源信息。这三种东西进入同一个系统时，是怎么"说同一种语言"的？

答案是 `Command` 类型。但这个翻译过程不是免费的。

---

## Command 是什么

`src/types/command.ts` 定义了几种变体。最核心的是 `PromptCommand`：

```typescript
type PromptCommand = CommandBase & {
  type: 'prompt'
  source: SettingSource | 'builtin' | 'mcp' | 'bundled'
  loadedFrom: LoadedFrom  // 'skills' | 'plugin' | 'bundled' | 'mcp' | 'commands_DEPRECATED' | 'managed'
  getPromptForCommand(args: string, context: ToolUseContext): Promise<ContentBlockParam[]>
  allowedTools: string[]
  whenToUse?: string
  disableModelInvocation?: boolean
  userInvocable?: boolean
  hooks?: HooksSettings
  paths?: string[]
  // ... 更多字段
}
```

`source` 字段记录来源，`loadedFrom` 字段记录装载方式，`getPromptForCommand` 是运行时调用的统一入口。不管原来是磁盘技能还是 MCP prompts，调用方都调这个函数。

---

## 三种来源是怎么翻译进来的

**技能 → Command**

`createSkillCommand()` 函数把解析好的 frontmatter 字段和 Markdown 正文一起打包：

```typescript
return {
  type: 'prompt',
  name: skillName,
  source,          // 'userSettings' | 'projectSettings' | 'policySettings'
  loadedFrom: 'skills',
  async getPromptForCommand(args, toolUseContext) {
    let finalContent = baseDir
      ? `Base directory for this skill: ${baseDir}\n\n${markdownContent}`
      : markdownContent
    finalContent = substituteArguments(finalContent, args, true, argumentNames)
    if (loadedFrom !== 'mcp') {
      finalContent = await executeShellCommandsInPrompt(finalContent, ...)
    }
    return [{ type: 'text', text: finalContent }]
  },
  // ...
}
```

**MCP 工具 → MCPTool**

MCP 的工具走另一条路，变成 `MCPTool` 而不是 `PromptCommand`。MCP 的 prompts 如果被翻译成 skills，则通过 `mcpSkillBuilders.ts` 的桥接，再调 `createSkillCommand()`。

**插件命令 → Command**

`loadPluginCommands.ts` 里的每个插件 Markdown 文件也走 `createSkillCommand()` 的变体，区别是 `loadedFrom: 'plugin'`，并且额外做了变量替换（`substitutePluginVariables`）。

---

## 翻译有什么代价

`formatDescriptionWithSource()` 函数暴露了一个细节——为了给用户显示来源信息，系统不得不在翻译完成后又反查原始字段：

```typescript
export function formatDescriptionWithSource(cmd: Command): string {
  if (cmd.source === 'plugin') {
    const pluginName = cmd.pluginInfo?.pluginManifest.name
    if (pluginName) return `(${pluginName}) ${cmd.description}`
    return `${cmd.description} (plugin)`
  }
  if (cmd.source === 'builtin' || cmd.source === 'mcp') {
    return cmd.description
  }
  if (cmd.source === 'bundled') {
    return `${cmd.description} (bundled)`
  }
  return `${cmd.description} (${getSettingSourceName(cmd.source)})`
}
```

这说明翻译是有损的——`Command` 对象虽然统一了，但为了在 UI 里还原来源信息，系统必须保留 `source`、`loadedFrom`、`pluginInfo` 这些来源字段作为"备注"。翻译层没有彻底抹掉来源，只是把它从"主信息"降级成了"辅助字段"。

---

## 更深的代价：MCP skills 的特殊处理

`getPromptForCommand` 里有一段值得注意的代码：

```typescript
// Security: MCP skills are remote and untrusted — never execute inline
// shell commands (!`…` / ```! … ```) from their markdown body.
if (loadedFrom !== 'mcp') {
  finalContent = await executeShellCommandsInPrompt(finalContent, ...)
}
```

MCP 来的技能和本地技能共用同一个 `createSkillCommand()`，但 shell 执行被条件屏蔽了。这是翻译层留下的安全豁口——同一个函数、不同的 `loadedFrom`、不同的行为分支。翻译越统一，这类隐式分支就越难发现。

---

## 面试指导

被问到"如何在系统中统一不同来源的能力"时，一个扎实的回答会提到：

1. 统一类型可以简化消费方，但翻译是有损的
2. 有损翻译需要保留来源字段作为辅助信息，供需要时反查
3. 同一类型内部可能存在来源相关的行为分支（如 MCP skill 禁止 shell 执行），这是最容易被忽视的隐患

"统一就是好的"是表面答案。说清楚"统一换来了什么、损耗了什么、以及哪些地方仍然必须感知来源"，才是架构级的回答。
