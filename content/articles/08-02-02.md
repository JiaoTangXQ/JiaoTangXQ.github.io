# 协议设计：Bridge 会话同步的不只是消息，还有元信息

## 最小化协议的诱惑

设计通信协议时，常见的直觉是"越小越好"：只传必要的内容，减少带宽和复杂度。

这个直觉在无状态 API 里很对。但在"会话同步"这个场景里，它会产生一个隐蔽的问题：**接收方拿到了内容，但没有上下文，无法正确解释内容**。

`replBridge.ts` 和 `remoteBridgeCore.ts` 的回调接口，说明了 Claude Code 认为哪些元信息是"必须同步"的：

```typescript
type EnvLessBridgeParams = {
  onInboundMessage?: (msg: SDKMessage) => void
  onUserMessage?: (text: string, sessionId: string) => boolean
  onPermissionResponse?: (response: SDKControlResponse) => void
  onInterrupt?: () => void
  onSetModel?: (model: string | undefined) => void
  onSetMaxThinkingTokens?: (maxTokens: number | null) => void
  onSetPermissionMode?: (mode: PermissionMode) => { ok: true } | { ok: false; error: string }
  onStateChange?: (state: BridgeState, detail?: string) => void
}
```

消息（`onInboundMessage`）只是一个回调。其他的七个回调，都是元信息。

## 三类元信息为什么不能省？

### 1. Title（会话标题）

在 claude.ai 的 Web 界面上，每个远端会话都显示为一张卡片，卡片上有标题。标题从哪来？

- 优先级最高：用户手动命名（`--name` 参数或 Web 界面重命名）
- 其次：从第一条有意义的用户消息推导（`extractTitleText` 函数）
- 兜底：用机器名 + 目录名生成

`bridgeMain.ts` 里的 `titledSessions` Set 记录了哪些 session 已经有了标题，防止 `onFirstUserMessage` 回调覆盖用户手动设置的标题：

```typescript
// Sessions that already have a title (server-set or bridge-derived) so
// onFirstUserMessage doesn't clobber a user-assigned --name / web rename.
const titledSessions = new Set<string>()
```

如果不同步标题，用户在 Web 看到的就是几十个无名会话，无法区分。

### 2. PermissionMode（权限模式）

Claude Code 有多种权限模式：`default`、`auto-edit`、`bypass-permissions`。这个模式影响远端 Agent 能否自主执行写文件、运行命令等操作，**必须由用户在本地明确设置，不能由服务端单方面改变**。

`onSetPermissionMode` 回调返回的是 `{ ok: true } | { ok: false; error: string }`，而不是 `void`。这意味着服务端发来的 permission mode 变更请求，客户端可以拒绝：

```typescript
onSetPermissionMode?: (mode: PermissionMode) => { ok: true } | { ok: false; error: string }
```

这是一个主动的安全设计：服务端不能通过发一条 `control_request` 就把客户端的权限模式提升到 `bypass-permissions`。客户端有拒绝权。

### 3. State（桥接状态）

`BridgeState` 的取值包括 `connecting`、`connected`、`reconnecting`、`failed`。这个状态同步到 Web 界面，让用户知道当前会话是在正常工作还是断线重连中。

如果不同步状态，用户会看到一个"卡死"的会话，不知道是在等待、断线还是崩溃了。

## 消息去重的工程细节

同步消息有一个容易忽视的问题：**回声消息（Echo）**。

当本地写一条消息到 transport，服务端可能把同一条消息广播给所有订阅者——包括发送方自己。如果不过滤，本地会看到自己发出的每条消息被重复显示。

`BoundedUUIDSet` 的双重去重：

```typescript
// recentPostedUUIDs：过滤自己发出去的消息被回声回来
// recentInboundUUIDs：过滤断线重连后服务端重放的重复消息
```

"Bounded"意味着这个 Set 有大小上限——记录最近 N 个 UUID，防止内存无限增长。这对于可能持续数小时的会话来说是必要的。

## 与普通终端镜像的对比

传统的远程终端（SSH）只传 stdin/stdout 字节流。屏幕上显示什么，完全由远端决定，本地只是个哑终端。

Claude Code 的设计是相反的：**本地界面是主权界面**。它有自己的权限策略、自己的历史滚动位置、自己的标题管理。远端传过来的元信息，是在告诉本地界面"这个会话应该呈现成什么样子"，但本地有权拒绝。

这种设计产生了更多的同步复杂度，但换来的是用户体验上的一致性——无论从哪个设备看同一个会话，看到的上下文是一致的。

## 面试指导

"你的系统需要同步哪些状态？"是分布式系统设计里一个开放性很强的问题。

优秀的回答需要区分：

- **业务数据**（消息内容）：通常是最容易想到的
- **元信息**（标题、状态、权限）：容易被忽视，但往往决定用户体验
- **安全相关元信息**（permissionMode）：需要特别处理，接收方应有拒绝权

Claude Code 的设计是一个很好的参考：它把"服务端能更改什么"和"服务端不能更改什么"区分得很清楚，这是一种防御性协议设计（defensive protocol design）的体现。
