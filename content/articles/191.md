# 扩展系统为什么要求所有来源汇回同一种运行时语言

面试题版本：**"Claude Code 的工具来源有内建工具、skills 目录、插件、MCP server 等多种。这些来源的工具在注册时都要经过规范化处理。规范化到底做了什么？为什么不能让每种来源保留自己的格式？"**

## 规范化做了什么

不同来源的工具原始格式差异很大：

**内建工具**（如 `BashTool`）：TypeScript 对象，`isConcurrencySafe()` 等方法直接实现在类里，输入 schema 用 Zod 定义。

**MCP server 工具**：JSON 格式，工具描述是字符串，没有 TypeScript 类型，`isConcurrencySafe` 等属性需要推断或用默认值。

**Skills 目录工具**：Markdown 文件，包含提示词文本，执行时需要特殊处理（把内容注入到系统 prompt，而不是直接调用函数）。

**插件工具**：外部加载，可能是 JavaScript 模块，接口格式由插件自己定义。

在 `src/services/mcp/normalization.ts` 里，所有 MCP 工具都被规范化：
```typescript
// 规范化后的统一 Tool 形状（概念描述）
{
  name: string,           // 经过前缀处理、冲突解决的规范名称
  description: string,    // 规范化后的描述文本
  inputSchema: JsonSchema, // 统一的 JSON Schema 格式
  // 以下属性可能是推断值或默认值
  isConcurrencySafe: () => boolean,
  interruptBehavior: () => string,
}
```

## 不统一会发生什么

假设系统决定"每种来源保留自己的格式，在调用时再做适配"：

```typescript
// 反面例子：分散的适配逻辑
async function callTool(tool: AnyTool, args: unknown) {
  if (tool instanceof BuiltinTool) {
    return await callBuiltinTool(tool, args)
  } else if (tool.source === 'mcp') {
    return await callMcpTool(tool as McpTool, args)
  } else if (tool.source === 'skill') {
    return await callSkillTool(tool as SkillTool, args)
  }
  // ...
}
```

这个 `if-else` 链会出现在：权限检查、提示词构建、工具执行、结果处理、日志记录、进度更新……每一个需要处理工具的地方。

每次这个链需要更新，要改多处。每次加一种新的工具来源，要在所有这些地方加一个新的 `else if`。这是工具来源数量和代码维护成本的乘法效应。

## 统一在哪个层做

Claude Code 选择在**接入层**做统一，而不是在消费层做适配。

接入层：`normalization.ts`、`mcpSkillBuilders.ts`、`loadSkillsDir.ts` 等，处理"把外部格式转成内部格式"这个问题。

消费层：`findToolByName()`、权限检查、提示词构建、`StreamingToolExecutor`，这些代码只见到统一的 Tool 形状，不需要知道工具来自哪里。

这个架构决策的效果：**添加新工具来源只需要写一个新的规范化适配器，消费层代码零修改**。

## 同一种"运行时语言"的含义

"运行时语言"不只是格式统一，还包括语义统一：

**并发安全性**：所有工具都要声明 `isConcurrencySafe()`，不管它本身来自哪里。MCP 工具如果没有声明，系统给一个保守的默认值（不并发安全）。

**中断行为**：所有工具都要声明 `interruptBehavior()`，告知系统当用户中断时应该怎么处理这个工具的运行中实例。

**输入 Schema**：所有工具的输入都用 JSON Schema 描述，不管内部用 Zod 还是其他工具定义。这个 Schema 会被用来生成提示词里的工具描述，帮助模型理解如何调用工具。

这些语义属性让系统可以对所有工具做统一的决策：是否并发执行、中断时如何处理、权限检查规则是什么。

## 面试角度的总结

工具来源统一规范化的核心价值在于：**把来源的多样性收口在接入层，让消费层代码保持简单**。这不是在假装所有工具都一样（它们确实不同），而是把差异转化为规范格式中的数据（比如不同的默认 `isConcurrencySafe` 值），而不是代码分支。消费层见到的是统一的语言，不需要知道工具来自哪里，就能对它做权限检查、提示词构建、执行管理。
