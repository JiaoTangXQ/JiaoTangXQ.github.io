Claude Code 真正的强，不是某个点爆发力特别大，而是每一层被要求只把自己的事做好。`main.tsx` 负责的是启动时的身份判断、能力预热和模式分流；`QueryEngine` 负责的是“同一场会话怎样持续存在”；`Task.ts` 负责的是“劳动怎样变成正式工作单元”；`AppStateStore` 负责的是“界面当前相信什么是真的”；权限和设置模块再负责“哪些能力现在有资格进场”。这些层都很重，但各自的重心很清楚。

更普通的实现，往往是把“先做成一个能跑的 agent”放在第一位，于是入口、模型循环、工具调用、任务显示、权限提示全混在一个中心模块里。这样一开始很方便，因为任何新功能都能“顺手塞进去”。问题是，等远端、MCP、subagent、browser、bridge 都来了之后，所有层都会互相踩踏：界面想自己推断真实状态，工具层想自己记住权限决定，任务层又顺手管消息显示。

Claude Code 通过分层避免了这种互踩。好处是每层都能继续长，而不必每长一块就把整套系统重新拧一遍。代价是很多地方需要显式翻译和交棒，文件数和状态对象都会变多。但这是“长大以后还不散架”的代价，不是多余装饰。

## 实现链
启动、会话壳、任务对象、事实状态、制度层各自有明确边界，不在同一层顺手改别人的事。

## 普通做法
让一个中心模块同时管输入、工具、任务、UI 和权限。

## 为什么不用
因为那样每加一个新能力都会牵动全身。

## 代价
交接点会更多，文件和状态对象都会变大。
