# 副作用为什么要挂到 preAction 之后：副作用的层次归属

面试题版本：**"Claude Code 的 hooks 系统里 preAction 在工具执行前运行。如果 preAction hook 失败了，工具执行会发生吗？这个机制保证了什么？有什么局限？"**

## preAction 的位置

Claude Code 工具执行的时序：

```
用户/模型请求工具调用
    ↓
输入验证（inputSchema.safeParse）
    ↓
权限检查（useCanUseTool）
    ↓
preAction hooks 执行  ← 最后一道门
    ↓
工具实际执行（副作用发生在这里）
    ↓
contextModifier 应用（上下文状态更新）
    ↓
工具结果返回
```

`preAction` 处于副作用之前、输入验证之后的位置。如果 `preAction` 失败（抛出异常或返回 error），工具执行被阻止，副作用不会发生。

## 为什么副作用必须在 preAction 之后

**副作用的不可逆性**：文件一旦被修改，删除，很难完美回滚。网络请求一旦发出，不能未发。数据库写操作一旦提交，难以撤销。

**preAction 是最后一道门**：preAction 之前的验证（input schema、权限检查）是系统自动的，preAction 是用户/开发者可以自定义的最后机会。这道门在副作用之前，意味着自定义约束可以在实际影响系统状态之前生效。

**副作用的层次归属**：帮助信息、输入解析、决策——这些都应该是无副作用的。只有"执行操作"这个层次才应该有副作用。如果副作用发生在更早的阶段（比如在权限检查时就开始写文件），系统状态会在约束条件还没全部满足时就被修改。

## preAction hook 的实际用途

**审计日志**：每次文件修改前记录到日志，如果 preAction 失败，说明记录写入失败，不执行工具以保证日志和实际操作的一致性。

**额外的路径验证**：确保被操作的文件在项目目录内（防止路径遍历攻击）。权限系统可能允许"修改文件"，但 preAction 可以进一步限制"只允许修改当前项目内的文件"。

**速率限制**：限制某个工具每分钟最多调用多少次，如果超过限制，preAction 拒绝执行。

**条件性确认**：对于某类高风险操作（比如删除超过 10 个文件），preAction 弹出额外确认对话框。

## 局限和边界情况

**preAction 本身的副作用**：如果 preAction hook 自己产生副作用（比如写日志），然后工具执行失败，日志里会有一条"执行了但实际没执行"的记录。这是 preAction 设计需要注意的地方——理想的 preAction 应该是幂等的或无副作用的。

**异步 preAction 的时序**：如果多个工具并发执行，各自的 preAction 也并发运行，可能出现 preAction A 通过了，但 preAction B 在 A 的副作用发生后才失败的情况。这需要 preAction 实现者考虑并发情况。

**不可阻止的失败**：preAction 可以阻止"已知危险"，但无法阻止"未预期的危险"。模型可能以一种 preAction 逻辑没有考虑到的方式造成问题。

## 与 postAction 的对比

Claude Code 也有 `postAction`/`stop hooks`，在工具执行后运行。和 preAction 的关键区别：

- preAction 可以**阻止**执行
- postAction 只能**响应**结果，无法回滚已发生的副作用

这意味着安全约束和验证应该在 preAction 里，结果记录、状态更新应该在 postAction 里。

## 面试角度的总结

副作用挂到 preAction 之后，是一个层次归属的设计决策：解析、验证、决策这些无副作用的操作先于执行，执行层面的副作用在所有前置条件满足后才发生。这让 preAction 成为了一个有意义的拦截点——自定义逻辑可以在"系统已经认为合法"和"副作用实际发生"之间插入，这是用户可以扩展系统安全策略的最合适位置。
