# 统一命名作为骨架接口：为什么名字能让整条链共认同一对象

面试题版本：**"Claude Code 里同一个工具的名字被后端执行逻辑、提示词构建、前端渲染同时使用。如果这三层的名字出现不一致（比如后端叫 bash_tool，提示词里叫 BashTool，前端显示叫 Bash），会出现什么具体问题？"**

## 三层共认的机制

在 Claude Code 的运行流程里，一个工具调用经历：

1. **模型推理**：模型在系统 prompt 里看到工具描述（名字 + schema），决定调用 `bash_tool`
2. **主循环处理**：`findToolByName(tools, "bash_tool")` 找到对应的工具对象
3. **权限检查**：检查当前上下文是否允许调用名为 `bash_tool` 的工具
4. **执行**：调用工具的 `call()` 方法
5. **结果处理**：把结果记录到 `tool_result`，关联到 `tool_use_id`
6. **UI 更新**：前端渲染器找到对应的工具状态，更新进度显示

这六步用的是同一个字符串 `"bash_tool"`。如果任意一步里的名字不一致，就会出现：

- 模型调用了 `bash_tool`，后端找不到工具（步骤 2 失败）
- 工具执行了，但权限记录里关联的是另一个名字（步骤 3 通过了但记录不对）
- 工具执行成功，但 UI 找不到对应的状态条目来更新（步骤 6 失败）

## 名字作为骨架接口的含义

"骨架接口"不是 TypeScript 里的 `interface`，而是一个更宏观的概念：**多个独立系统之间共同承认的最小公共锚点**。

名字作为骨架接口，意味着：
- 它不携带实现细节（名字不解释工具怎么工作）
- 它不携带层级信息（名字不说明工具属于哪个系统）
- 但它足以让每一层找到自己需要的东西

`findToolByName("bash_tool")` 在后端工具表里找到 `BashTool` 对象；提示词构建器用 `"bash_tool"` 生成 schema 描述；权限记录用 `"bash_tool"` 关联权限条目；UI 用 `"bash_tool"` 找到对应的渲染组件。

每一层做的事情完全不同，但都通过同一个名字找到了自己的目标。名字是这四条线汇聚的交叉点。

## 命名漂移的实际危害

"漂移"不是突然的改变，而是缓慢的偏差。常见的漂移场景：

**显示名称和内部名称分离**：开发者觉得 `bash_tool` 太技术，想在 UI 上显示"Bash 命令执行"。如果这个显示名和内部名分开维护，时间久了会变成两个不同步的东西——UI 更新了显示，后端忘了更新提示词里的说明。

**版本演进中的重命名**：工具 v1 叫 `file_read_tool`，v2 叫 `read_file_tool`。如果这个重命名没有在所有使用这个名字的地方同步更新，就会出现"历史 transcript 里的 `file_read_tool` 找不到对应工具"的情况。

**提示词和代码的漂移**：提示词里写的工具名是硬编码的字符串，代码里的工具名是从 TypeScript 常量来的。两者没有编译时链接，重构代码时可能忘了更新提示词。

Claude Code 在 `src/tools/BashTool/toolName.ts`（以及其他工具的同名文件）里把工具名字定义为导出的常量，然后在所有地方（提示词模板、工具定义、权限检查）都引用这个常量，而不是写字符串字面量。这是防止漂移的工程实践。

## 为什么不用枚举或 ID 而用字符串名字

工具用字符串名字而不是数字 ID 或枚举，有几个具体原因：

**提示词的自然语言需求**：模型的系统 prompt 里，工具名字是自然语言的一部分（"使用 `bash_tool` 来运行命令"）。字符串名字天然适合这个场景，数字 ID 不行。

**跨系统的人类可读性**：工具名字会出现在日志、调试输出、transcript 里，需要人类可读。`bash_tool` 比 `tool_42` 更容易理解。

**MCP 协议的兼容性**：MCP 协议本身用字符串名字标识工具，内部 Claude Code 工具用字符串名字保持了与 MCP 协议的一致性，减少了转换层的复杂度。

## 面试角度的总结

统一命名作为骨架接口，价值在于：**用一个简单的字符串，维持多个独立系统（后端、提示词、渲染）对同一对象的共识**。这比用不同层之间的显式 API 连接来协调更轻量，也比让每层自己维护一个"工具表"来管理更不容易漂移。代价是必须严格管理名字的生命周期（定义、重命名、废弃），把名字作为一等公民来对待，而不是随手写的字符串字面量。
