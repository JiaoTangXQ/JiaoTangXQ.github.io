# registerMCPSkillBuilders 的初始化顺序约束：为什么选择提前失败而不是默默降级？

## 代码的明确表态

```typescript
export function getMCPSkillBuilders(): MCPSkillBuilders {
  if (!builders) {
    throw new Error(
      'MCP skill builders not registered — loadSkillsDir.ts has not been evaluated yet',
    )
  }
  return builders
}
```

这个 `throw` 是一个设计选择，不是偶然。它明确地说：如果在 `registerMCPSkillBuilders` 被调用之前就调用了 `getMCPSkillBuilders`，系统宁可立即崩溃，也不要继续工作。

为什么？

---

## 备选方案：默默降级

如果改成：

```typescript
export function getMCPSkillBuilders(): MCPSkillBuilders | null {
  return builders  // null 如果没注册
}
```

调用方就要处理 `null`：

```typescript
// mcpSkills.ts 里
const b = getMCPSkillBuilders()
if (!b) {
  // 没有 builders，做什么？
  // 选项 1：返回空的 skills 列表（静默降级）
  return []
  // 选项 2：记日志（但谁会看日志？）
  // 选项 3：抛出错误（和现在一样，只是发生位置不同）
}
```

选项 1（返回空列表）看起来最温和。但问题是：

**用户触发了 MCP skill，什么也没发生，没有任何错误信息。**

这是最糟糕的一种失败：系统"看起来在运行"，但实际上什么也没做。用户会反复尝试、困惑，最终开 issue 说"MCP skill 不工作了"，而错误原因只是初始化顺序问题。

---

## 提前失败的价值

在 `getMCPSkillBuilders` 里 throw，有几个关键好处：

**好处一：错误在边界暴露，不渗入业务逻辑**

如果 `builders` 为 null 是一个 bug（初始化顺序错误），那么错误应该在"第一次检测到它"时就暴露，不应该悄悄传播到 `mcpSkills.ts` 的业务逻辑里，让后者产生更难理解的行为。

**好处二：错误信息精确**

```
'MCP skill builders not registered — loadSkillsDir.ts has not been evaluated yet'
```

这条错误信息告诉开发者：
- 发生了什么：builders 没注册
- 可能的原因：`loadSkillsDir.ts` 还没被加载

相比之下，如果返回 null 导致后续 NullPointerException，错误信息可能是"Cannot read property 'createSkillCommand' of null"，完全没有上下文。

**好处三：迫使调用方在正确的时机使用**

`throw` 创造了一个硬约束：调用方必须在 `registerMCPSkillBuilders` 之后才能调 `getMCPSkillBuilders`。这个约束由代码强制，而不是靠注释说"请在注册后使用"——后者很容易被忽视。

---

## 注册时机的保障

`loadSkillsDir.ts` 在模块加载时（不是在某个函数被调用时）执行注册：

```typescript
// loadSkillsDir.ts 末尾（模块初始化代码）
registerMCPSkillBuilders({ createSkillCommand, parseSkillFrontmatterFields })
```

而 `loadSkillsDir.ts` 通过 `commands.ts` 的静态 import 被加载：

```typescript
// commands.ts
import {
  getSkillDirCommands,
  clearSkillCaches,
  getDynamicSkills,
} from './skills/loadSkillsDir.js'
```

`commands.ts` 在应用启动时就被加载（它是主模块的依赖）。所以注册在启动时就完成了，远早于任何 MCP server 连接。

注释也确认了这一点：
```
Registration happens at loadSkillsDir.ts module init, which is eagerly
evaluated at startup via the static import from commands.ts — long before
any MCP server connects.
```

---

## 测试时的问题

注册式模式有一个测试难点：如果测试文件直接测 `getMCPSkillBuilders`，需要先调用 `registerMCPSkillBuilders`，否则 throw。

这意味着测试文件里需要 setup：

```typescript
beforeEach(() => {
  registerMCPSkillBuilders({
    createSkillCommand: mockCreateSkillCommand,
    parseSkillFrontmatterFields: mockParseSkillFrontmatterFields,
  })
})

afterEach(() => {
  // 重置，避免测试间污染
  // （需要 resetMCPSkillBuilders 这样的测试工具函数）
})
```

这是注册式模式引入的额外测试复杂度。但对于这种"只注册一次、应用全生命周期使用"的场景，这是可接受的代价。

## 面试指导

**"系统中某个关键组件没有初始化时，应该怎么处理后续请求？"**

三种策略：

1. **抛出错误（fail fast）**：立即暴露问题，防止错误扩散
2. **返回默认值（graceful degradation）**：系统继续工作，功能有限
3. **等待初始化完成（block/queue）**：把请求排队，等组件就绪后处理

选择哪种取决于：
- 这个组件是否是核心功能？（核心 → fail fast；非核心 → graceful degradation）
- 缺少这个组件时，继续执行会产生误导性结果吗？（会 → fail fast）
- 调用方有能力处理缺失情况吗？（有 → graceful degradation）

MCP skill builders 是 MCP skill 功能的核心组件。缺少它不是"部分降级"，而是"完全不工作但假装在工作"。所以 fail fast 是正确的选择。

**补充技巧**：在实际项目里，"系统级初始化顺序"最好用文档或 ADR（Architecture Decision Record）记录，说明"哪些模块必须在哪些模块之前初始化，以及为什么"。这比只靠代码里的 throw 更便于团队理解。
