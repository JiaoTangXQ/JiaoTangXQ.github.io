# Claude in Chrome 接的是用户真实浏览器，不是实验沙盒

Claude in Chrome 最关键的不是“Claude 也能点网页”，而是它默认面对的是用户自己那只真实浏览器。`prompt.ts` 一上来就要求每次会话先跑 `tabs_context_mcp`，先看当前已经开着哪些标签页，而不是默认新开一张干净空白页；同一个 prompt 里还反复强调要避免触发阻塞式浏览器对话框，因为那会让真实扩展会话直接卡死。`setup.ts` 另一头则负责接进用户真实的扩展、Native Host 和浏览器安装位置，而不是偷偷起一只自带状态的“测试浏览器”。

更常见的浏览器自动化方案，是自己起一个受控浏览器上下文，登录态、扩展、弹窗、页面脏状态都尽量置空。那样当然更可控，但它解决的是“脚本怎么跑”，不是“用户当前工作的真实现场怎么继续被接管”。Claude Code 没走那条路，所以它必须尊重现有标签页、现有账号、现有权限和现有混乱。

这条路明显更难，因为真实浏览器远比实验沙盒脏得多，也更容易掉链子。但如果目标是让工作台真的伸到用户已经在用的现场，而不是另外造一个实验环境，这反而是更诚实的实现。

## 实现链
`prompt.ts` 明说这套工具操作的是用户现有 Chrome session，先读当前 tabs，再决定要不要新开页。`createChromeContext()` 还会保存 `pairedDeviceId`、读取 OAuth token、处理断线提示和同账号约束，说明它接的是用户真实登录现场。

## 普通做法
普通浏览器自动化更喜欢开一个干净、可控的沙盒浏览器。

## 为什么不用
Claude Code 不走那条路，因为它想解决的是“替用户在他当前浏览器里做事”，包括已登录站点、OAuth 和真实网页状态。沙盒虽然稳定，但不是真实现场。

## 代价
代价是安装、授权、配对、掉线和账号一致性都会变成必须处理的问题。
