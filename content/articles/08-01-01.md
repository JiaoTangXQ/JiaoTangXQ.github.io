# 工程取舍：远端会话为什么不能用「远程工具调用」模型实现？

## 两种模型的差异

在设计远端执行系统时，存在两种截然不同的思路：

**模型 A：远程工具调用（Remote Tool Call）**

本地 Agent 决定执行某个工具，通过网络把调用发到远端，等待结果返回，继续下一步。链路是请求-响应的，每次调用独立。

**模型 B：远端会话（Remote Session）**

本地和远端共享一个持续存在的"会话"。消息历史、工具状态、审批上下文都在这个会话里累积。用户在本地看到的界面，是远端会话状态的投影。

Claude Code 选择了模型 B。`initReplBridge.ts` 在启动时做的事情，清楚地说明了这个选择：

```
entitlement 检查 → OAuth 验证 → 组织策略 → 标题推导 → git 上下文 →
创建 environment_id + environment_secret → 启动 poll 循环
```

这不是在"发一次请求"，而是在初始化一个持续的会话对象。

## 为什么不选模型 A？

### 问题一：历史消息的所有权

在模型 A 里，历史消息存在本地，每次请求带着 context 发过去。这在短任务里没问题，但 Claude Code 的远端场景不是短任务——用户在 claude.ai 网页上开启的任务，可能持续数小时，中途还会切换设备。

一旦历史存在本地，"切换设备"就无法实现——新设备没有历史。Teleport 功能（`src/utils/teleport/*`）的存在，正说明历史消息需要可迁移，这要求有一个中立的存储点，而不是散在发起方。

### 问题二：断线恢复的语义

在模型 A 里，网络断线等于任务丢失——重试只能从头开始。

在模型 B 里，断线是个临时状态。`bridgePointer.ts` 里的崩溃恢复指针（`bridge-pointer.json`）记录了当前的 `sessionId` 和 `environmentId`，进程重启后可以通过 `/bridge/reconnect` 重新接管。这个语义在模型 A 里根本无法表达。

### 问题三：权限审批的双向性

模型 A 的权限审批只需要"本地用户同意"——工具调用发出去之前，本地确认一下即可。

模型 B 里，权限请求可能从服务端发来（`control_request`），也可能需要从本地发回响应（通过 `sendPermissionResponseEvent` 写到 `/v1/sessions/{id}/events`）。审批流是双向的，且必须和会话状态绑定——一个过期的 `requestId` 不能被重放。

## 代价是什么？

选择模型 B，代价是显而易见的：

**初始化路径很长**。`initReplBridge.ts` 在真正连接之前，要过至少 6 个检查点。任何一个失败，都要给出清晰的错误信息，而不是模糊的"连接失败"。

**状态管理复杂**。`bridgeMain.ts` 里同时维护着 `activeSessions`、`sessionStartTimes`、`sessionWorkIds`、`sessionIngressTokens`、`timedOutSessions`、`titledSessions` 六张 Map，每张 Map 在不同的生命周期节点被写入和清理。

**错误类型需要精确区分**。`BridgeFatalError` 和普通 `Error` 不同，`BridgeFatalError` 不可重试（401/403/404/410），而 429 和 5xx 只是暂时性的，需要退避重试。

## 实际收益

收益也是对称的：

- 会话可以跨设备迁移（Teleport）
- 进程崩溃后可以恢复（bridge-pointer）
- 权限审批可以在任意客户端完成
- 多个观察者可以同时看同一个会话（viewerOnly 模式）

这些能力，都是模型 A 从架构上就无法提供的。选择"重"的模型 B，是在有意识地用复杂度换取能力边界。

## 面试指导

这类"为什么不用简单方案"的问题，是系统设计面试里最容易扣分的地方。

优秀的回答不是"因为简单方案不够好"，而是**具体说出简单方案在哪三个 edge case 上会失败**，然后说明当前方案如何处理这三个 case，最后承认当前方案的额外复杂度，并说明这个复杂度是有意为之的取舍。

Claude Code 的 bridge 层，是这类答题框架的一个很好的真实案例。
