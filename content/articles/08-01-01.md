# 远端能力改写的是工作台边界，不是工具清单

## 实现链
- `initReplBridge.ts` 和 `replBridge.ts` 不只是把某个工具暴露到远端，而是创建远端 session、维护 environment、同步标题和会话上下文；`RemoteSessionManager.ts` 再接管消息发送、订阅和审批流。
- 这说明远端能力一旦打开，变的不是“工具列表多了一项”，而是谁在执行、状态放在哪、消息从哪来、审批回哪裁决。

## 普通做法
- 更普通的做法是给本地 agent 多加一个 remote shell 工具，需要时调用一下。
- 那种方案里，本地工作台还是原来的工作台，远端只是附属工具。

## 为什么不用
- Claude Code 没选这个轻量路线，因为它想保留的是一个完整远端会话，而不是一把临时远程扳手。
- 一旦目标变成“远端也有会话生命”，那边界被改写就是必然结果：谁拥有状态、谁维护历史、谁承担恢复，都不能再沿用本地工具模式。

## 代价
- 代价是远端模式实现明显更重，也更难拿“只是一条网络请求”去解释。
- 但如果目标真的是跨设备、跨环境、可恢复的工作台，这种重法是合理的。
