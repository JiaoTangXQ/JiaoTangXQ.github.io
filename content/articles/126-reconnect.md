# 深度分析：为什么断线重连需要是"被设计进去的"，而不是靠 try-catch 处理

## 两种处理断线的方式

**方式一：异常路径**

连接正常时一切正常。连接断了抛异常，异常被捕获，显示一个通用错误提示"连接失败，请重试"。用户点重试，如果成功就继续，如果失败再次显示错误。

这种方式的代码量少，逻辑简单，90% 的情况下够用。

**方式二：产品路径**

连接断了不是"出了问题"，而是"进入了一个需要用户参与恢复的状态"。这个状态有自己的 UI，有清晰的用户引导，有自动检测重连成功的机制，有成功后的无缝续接。

Claude Code 选择了方式二。

**面试题：为什么花这么大力气把"失败"设计成产品路径？**

## 断线频率决定设计级别

Chrome 扩展的连接不是"偶尔会断"，而是"一定会断"。

以下情况都会导致连接断开：
- 用户手动刷新浏览器
- Chrome 更新后重启
- 用户进入休眠后唤醒
- 扩展被意外禁用（某些安全软件会这样做）
- 用户重启了电脑
- 系统资源不足时 Chrome 可能终止后台扩展

这些不是边缘情况，而是正常使用过程中必然会发生的事。如果用异常路径处理，用户会频繁看到错误提示，每次都要自己判断该怎么做，这是糟糕的体验。

**当某件事频率足够高，它就应该被设计成正常的用户旅程，而不是错误情况。**

## reconnect 页面需要回答的问题

一个好的 reconnect 页面不只是显示"连接断了"，它需要回答用户的几个问题：

**发生了什么？** 是 Claude Code 的问题、还是扩展的问题、还是浏览器的问题？不同原因对应不同的解决步骤。

**我需要做什么？** 有些断线可以自动恢复（等几秒重试），有些需要用户操作（重新安装扩展），有些需要重启服务（Claude Code 进程崩溃）。

**现在恢复了没有？** 用户按照引导操作后，如何知道是否成功？需要自动检测，而不是让用户自己点"测试连接"然后猜测结果。

**之前做了什么会丢失吗？** 用户可能在断线前给了 AI 指令，这些指令是否还在？Claude Code 是否知道要继续之前的任务？

## 自动重试的策略设计

自动重试需要策略，不是无脑循环 `while (true) { connect() }`。

合理的策略要考虑：

**指数退避**：第一次重试等 1 秒，第二次等 2 秒，第四次等 8 秒……避免在服务不可用时持续轰炸。

**重试上限**：重试次数超过上限后，停止自动重试，告知用户需要手动干预。

**区分可重试和不可重试的错误**：扩展不存在（404）是不可重试的，网络超时是可重试的。把它们混在一起处理，会在不可重试的情况下浪费时间。

**用户取消**：用户如果明确关闭了浏览器或禁用了扩展，不应该继续重试，应该切换到"告知用户如何重新启用"的界面。

## 缓存检测：避免多余的重试

每次 Claude Code 启动，需要检测 Chrome 扩展是否已安装。

这个检测可以每次都做完整检测（查文件系统、发握手包、等响应），也可以缓存上次的检测结果（如果上次成功了，先假设这次也可以，尝试连接，失败了再做完整检测）。

缓存检测的好处是：正常情况（扩展已安装）下的启动速度更快，不需要每次都等检测完成。

代价是：缓存可能过期（用户卸载了扩展但缓存里还显示已安装），这时会先尝试连接失败，再降级到完整检测。

这多了一次失败尝试，但对用户来说只是多了不到 1 秒的延迟，可以接受。

## reconnect 与任务恢复的关系

reconnect 不只是重新建立连接，还需要处理"连接断开期间任务的状态"。

如果 AI 正在操作浏览器时连接断了（比如 AI 刚发出"点击提交按钮"的指令），恢复连接后：
- 这个指令执行了没有？
- 需要重新发一遍吗？
- 如果不知道是否执行，应该告诉用户让他确认吗？

这些问题没有统一答案，取决于具体操作的幂等性。但好的 reconnect 设计需要考虑到这些场景，而不是假设"重连成功后万事大吉"。

## 面试要点

1. **高频事件必须设计成用户旅程**：Chrome 扩展连接断开不是偶然，而是必然，因此需要完整的产品流程处理，而不是 try-catch。
2. **reconnect 页需要回答用户的具体问题**：发生了什么、需要做什么、现在好了没、之前的工作丢失了吗。
3. **自动重试需要策略**：指数退避、重试上限、错误分类——不是简单的重试循环。
4. **缓存检测是性能与正确性的权衡**：缓存加速了正常路径，引入了过期缓存的可能性，这个取舍是可以接受的。
