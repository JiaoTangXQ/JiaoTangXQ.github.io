# 真正见功力的是知道什么绝不能被带上路

## 实现链
- `initReplBridge.ts` 和 `replBridge.ts` 的注释反复强调：不要把 `sessionStorage`、`growthbook`、`commands.ts`、整棵 React 组件树直接拉进 bridge core，需要什么就注入什么。
- 真正体现功力的地方，不是“我能把更多本地能力搬过去”，而是作者很清楚哪些依赖一旦带上路，就会让整个远端桥接变得臃肿而脆弱。

## 普通做法
- 普通做法常常是先把本地现成模块全复用起来，等 bundle 太大或循环依赖爆炸时再回头瘦身。
- 这是很常见的工程路径，因为前期推进最快。

## 为什么不用
- Claude Code 没把 bridge core 走成“先肥后瘦”，从代码注释看，它一开始就在防这种耦合扩散。
- 原因很简单：bridge core 既要跑在 REPL，也要跑在 daemon/SDK。把不该带的东西带上去，不只是大一点，而是入口边界会直接失真。

## 代价
- 代价是开发者写起来要更克制，很多地方得放弃“直接 import 最方便”的诱惑。
- 但这比事后大拆更省，因为边界一开始就没被打穿。
