# 深度分析：为什么统一消息世界比零损耗翻译更重要

## 问题场景

远端工具（在云端容器里运行）在执行过程中会产生实时进度更新：

- "已扫描 234 个文件"
- "正在分析第 5 个潜在问题"
- "已完成 60%"

这些更新对用户有价值——让用户知道长时间运行的工具在做什么，不是卡住了。

在本地工具里，这些进度更新直接写入 `outputFile`，主会话实时轮询读取，展示给用户。简单，直接，实时性好。

但远端工具在云端容器里，它的 `outputFile` 在云端。主会话（在本地 CLI）看不到。

**如何让本地 CLI 看到云端工具的进度更新？**

## 两种可能的方案

**方案一：维护两套消息格式**

创建一种新的消息类型 `RemoteToolProgress`，让本地 CLI 能接收这种消息，渲染时用特殊样式展示（比如实时滚动的进度条）。

好处：实时性最好，进度更新和远端工具的执行频率一样高。

代价：系统里现在有两种消息格式——正常的对话消息（`UserMessage`、`AssistantMessage`）和远端进度消息（`RemoteToolProgress`）。所有处理消息的地方都需要处理这两种格式。

**方案二：翻译成 system message**

在某些节点（比如工具执行完成或者每隔一段时间），把工具进度信息翻译成 `system message` 注入对话历史。

好处：不需要新的消息类型，所有现有的消息处理代码都能处理 `system message`，不需要修改。

代价：实时性降低。进度更新不是每次都立刻到本地，而是在特定节点才到。用户看到的是"快照"而不是"直播"。

Claude Code 选择了方案二。

## 翻译损耗是可接受的

"进度更新变成快照而不是直播"——这个损耗有多大？

对于长时间运行的任务（几分钟），每隔几秒收到一次进度更新，和实时收到更新，用户体验差异很小。

对于短时间任务（几秒钟），进度更新可能只有一次（"开始了"和"完成了"），实时性根本不重要。

真正会感觉到差距的，是需要非常高频进度更新的任务（比如每秒几十次更新）。这类任务在 Claude Code 的使用场景里很少见——大部分时间 AI 在思考和执行，不是在高频产生进度事件。

所以翻译损耗在实际使用中几乎不可感知，但维护代价非常真实。

## 消息世界分裂的长期代价

如果维护两套消息格式，长期会发生什么？

**每个新功能都要考虑两套格式**：新增一个"消息搜索"功能，要处理两种消息格式。新增一个"消息导出"功能，同样。每个功能的开发时间增加，每个 bug 要在两个地方修复的概率增加。

**测试覆盖翻倍**：每个处理消息的单元测试，需要分别测试普通消息和远端进度消息的情况。

**状态同步更复杂**：对话历史里混杂着两种格式，恢复会话时需要分别处理，迁移数据格式时需要分别迁移。

**新成员上手成本增加**：学习 Claude Code 的消息系统时，要理解"这里有两种消息格式，一种用于普通对话，一种用于远端进度"，增加了认知负担。

这些代价都是真实的，而且随着时间推移是累积的——代码库越复杂，每个功能的开发和维护成本越高。

## 长期产品在和现实谈判

"宁可接受翻译损耗，也要保住统一消息世界"——这是一种"和现实谈判"的产品思维。

现实是：远端工具产生的进度信息格式，和本地对话系统的消息格式，天然不兼容。

谈判的结果是：接受一些翻译损耗（进度实时性降低），换取不需要维护两套格式（长期维护成本降低）。

这不是技术上的最优解（最优解可能是某种零损耗翻译协议），而是**长期可维护性和当前功能完整性之间的最优平衡**。

## 面试要点

1. **翻译损耗 vs 消息世界分裂是真实的取舍**：不是一个明显更好，而是在不同维度上各有优劣，需要判断哪个维度更重要。
2. **消息世界统一是长期维护性的基础**：每个消息处理功能只需要处理一种格式，新功能开发和 bug 修复都更简单。
3. **实时性损耗在大多数场景不可感知**：Claude Code 的典型任务时间尺度，每几秒收到进度快照足够，不需要实时直播。
4. **长期产品持续和技术债谈判**：接受局部不完美换取系统整体的可维护性，是长期产品开发的常见模式。
