# 这行代码看起来多余，删掉会怎样？

代码审查时有一类问题特别能测试工程师的真实理解：找出一行"看起来多余"的代码，问删掉会怎样。

来看 `main.tsx` 开头的这几行：

```typescript
profileCheckpoint('main_tsx_entry');
import { startMdmRawRead } from './utils/settings/mdm/rawRead.js';
// eslint-disable-next-line custom-rules/no-top-level-side-effects
startMdmRawRead();
import { ensureKeychainPrefetchCompleted, startKeychainPrefetch } from './utils/secureStorage/keychainPrefetch.js';
// eslint-disable-next-line custom-rules/no-top-level-side-effects
startKeychainPrefetch();
```

**问题：把 `startMdmRawRead()` 和 `startKeychainPrefetch()` 这两行删掉，程序还能正常工作吗？**

## 答案是：能

MDM 配置和 keychain token 不是"只有这两个函数能读取"的独家能力。后续代码在真正需要这些数据时，照样会去读——通过 `applySafeConfigEnvironmentVariables()`、`isRemoteManagedSettingsEligible()` 等函数触发真正的读取。

删掉这两行，程序行为完全正确，测试全绿，功能没有任何损失。

## 那为什么不删？

删掉这两行之后，每次 macOS 用户启动 Claude Code，都会在某个关键路径上多等 65ms 到 135ms。

这个延迟不会消失，只是从"import 期间的并行等待"变成了"主流程上的串行阻塞"。

### MDM 的情况

`applySafeConfigEnvironmentVariables()` 是 `init()` 早期必须调用的函数，它负责把 MDM 下发的企业策略应用到环境变量。如果 `startMdmRawRead()` 没有提前预热，这里就会同步等待 MDM 子进程读取完毕。

注释里的描述是："fires MDM subprocesses (plutil/reg query) so they run in parallel with the remaining ~135ms of imports"。

### Keychain 的情况

更精确。注释说：`isRemoteManagedSettingsEligible()` 会通过 `applySafeConfigEnvironmentVariables()` 内部的同步子进程 spawn 串行读取 keychain，每次读取约 65ms，两个 key 就是 130ms。预取让两次读取并行，把总时间压到单次读取时间（~65ms）附近。

## 代价分析：多等 65ms 到底有多重要？

这取决于视角。

**从单次调用看**：65ms 几乎感知不到。人类对延迟的感知阈值大约在 100ms。

**从"冷启动体验"看**：用户敲下 `claude` 后，屏幕上什么都没有，这段时间每一毫秒都会被感知为"卡顿"。65ms 的额外等待意味着这段沉默多了 65ms。

**从高频使用看**：如果一天启动 Claude Code 20 次，每次多 65ms，一年下来多等了约 8 分钟。数字不大，但这是纯粹的浪费——能并行的等待被串行化了。

## 真正有价值的工程问题

删掉这两行，功能不变，但用户体验每次都会悄悄变差一点点。这类代码最难被识别，因为：

1. 它们在功能层面是"多余的"
2. 没有对应的测试会失败
3. 性能回归很细微，不会触发任何告警

这也是代码注释如此详细的原因。注释在说的不是"这里干了什么"，而是"如果你觉得这几行多余想删掉，请先读懂这段说明"。

## 面试指导

"删掉会怎样"系列问题的评分标准：

**及格**：能回答"删掉后功能还正常，但是……"——知道这类代码是性能优化，不是逻辑必要。

**良好**：能量化影响——"删掉后每次启动多等约 65-135ms，具体在 keychain 读取那一步"。

**优秀**：能说清楚为什么这个优化只在特定位置有效——"因为此处的等待时间和模块加载时间是可以重叠的，如果移到 main() 开头，重叠窗口就消失了"。

工程品味的一个标志，就是能区分"功能意义上的必要代码"和"性能意义上的必要代码"，并且在审查或重构时不把后者误删。
