# Trust dialog 和 onboarding 有什么本质区别？

## 一个看似无关紧要的分类问题

大多数 CLI 工具第一次运行时，会弹出类似这样的内容：

> 欢迎使用 MyTool！
> 在开始之前，我们需要您同意以下条款……
> 接下来让我们配置您的偏好……
> （中间夹着一个安全提示）
> 好了，开始使用吧！

这把三种性质完全不同的事情混在一起：欢迎（onboarding）、条款（trust）、配置（preferences）。

Claude Code 的 `showSetupScreens()` 有意把它们分开，并且在代码注释里明确标注了哪些是"trust"层，哪些是"onboarding"层：

```typescript
// interactiveHelpers.tsx（简化）
export async function showSetupScreens(...): Promise<void> {
  // 1. Onboarding（欢迎/功能介绍）
  await maybeShowOnboarding(...)
  
  // 2. Trust（工作区信任边界）
  const trustGranted = await showTrustDialog(...)
  if (!trustGranted) {
    process.exit(0);
  }
  
  // 3. 只有 trust 通过后才能做的事情
  await processMcpJsonApprovals(...)   // 审批 MCP 服务器
  void getSystemContext()              // 预取 system context（执行 git 命令）
  checkExternalReferencesInClaudeMd() // 检查 CLAUDE.md 外部引用
  applyConfigEnvironmentVariables()   // 应用完整环境变量
  initializeTelemetryAfterTrust()     // 初始化遥测
```

## 为什么 trust 必须独立于 onboarding？

Trust dialog 的源码注释非常直接：

```typescript
// Trust dialog is the workspace trust boundary.
// Not affected by permission modes.
```

这句话的关键词：**workspace trust boundary**。

Onboarding 是"欢迎新用户了解产品功能"——用户可以跳过，可以以后再看，不影响功能。

Trust 是"这个目录里的代码能不能被执行"——这是一个安全边界，不能跳过，不能"等会儿再确认"。

一旦 trust 被设计成 onboarding 的一部分（就像很多工具那样），它的可跳过性就变成了安全漏洞：用户可以在不了解信任含义的情况下快速点过所有"欢迎屏幕"，在没有真正理解的情况下就进入了完整能力模式。

## Trust 通过后才发生的事情说明了什么

Trust 通过后才执行的操作，揭示了"不信任"状态下为什么必须限制：

**`void getSystemContext()`**：执行 `git status`、`git log` 等命令。Git hooks（`core.fsmonitor`、`diff.external`）可以执行任意脚本。在不信任的目录里运行 git 命令，等于执行了潜在的恶意代码。

**`checkExternalReferencesInClaudeMd()`**：检查 `CLAUDE.md` 里是否有 `@import` 外部文件引用。不信任的项目可能通过 CLAUDE.md 引用恶意文件，欺骗 AI 模型执行恶意操作。

**`applyConfigEnvironmentVariables()`**：应用完整环境变量（包括项目级配置里的变量）。不信任的项目可能通过 `.claude/settings.json` 注入恶意环境变量。

这三个操作都有一个共同特征：**执行了来自工作目录的代码或配置**。这正是 trust 边界的意义——不是"用户是否同意条款"，而是"这个目录的内容是否可信"。

## 非交互模式的信任假设

```typescript
// showSetupScreens()
if (getIsNonInteractiveSession()) {
  return;  // 跳过所有界面，包括 trust dialog
}
```

非交互模式（`-p`/`--print`）直接跳过了 `showSetupScreens()`，包括 trust dialog。

这不是"降低了安全要求"，而是把信任假设从"界面确认"改成了"调用前提"：

> 使用 `-p` 模式的用户（脚本、CI/CD）默认对当前目录已信任。  
> 帮助文本明确说明：只在你信任的目录里使用。

信任仍然存在，只是形式从"弹窗确认"变成了"调用时的隐式前提"。

## 面试指导

"信任模型"是一个比较高级的系统设计话题，面试里不常被问到，但在安全相关职位或 platform 设计里很关键。

核心思路：
1. **把安全边界从 UX 流程里独立出来**：安全边界不应该因为 UX 调整而被绕过
2. **信任不是二元的**：Claude Code 区分了"全局信任"（有没有接受过 trust dialog）和"本次会话信任"（这个目录是否被信任）
3. **不同执行模式有不同的信任假设**：交互模式需要显式确认，自动化模式需要调用方保证

能把这三点说清楚，就比"加个弹窗问用户是否同意"这种浅层回答要深入得多。
