# microcompact 为什么站在更前面

`microcompact` 的位置特别能体现 Claude Code 的品味。它不是传统意义上的大总结，更像一次尽量温和的局部减肥。

这一层之所以站在前面，是因为它处理的是“还能保持原貌的大部分上下文，只把某些工具结果变薄一点”。尤其在 cached microcompact 路径里，它甚至尽量不去改本地消息内容，而是把删减动作推到 API 层和缓存层去做，尽量保住原来的会话结构和缓存前缀。

这和重型压缩是完全不同的思路。重型压缩承认“原样留着已经不现实了”，`microcompact` 则还在努力争取“能不能先少动一点”。Claude Code 把它放在前面，等于是在给连续性多争一线生机。

所以 `microcompact` 重要，不是因为它最强，而是因为它最克制。真正懂长会话的人，都会先珍惜这种克制。

代码里它之所以站得靠前，是因为 `query()` 把 `microCompact` 当成“局部修剪”而不是“总结重写”。更普通的实现是直接跳到 `autoCompact`：既然已经长了，就让系统生成一份摘要，把旧消息整体替换掉。这种方法实现统一，但副作用很大，尤其在刚刚还在密集使用工具、下一轮很可能继续追着同一细节工作的时候，过早摘要会把最有用的细纹理抹掉。

Claude Code 先上 `microcompact`，说明它更看重“尽量保住近处细节”，而不是“尽快把上下文做小”。这通常更合理，但也让主循环多了一层中间状态：你得同时理解“还没 compact，但已经不是原始历史”的半成品上下文，这比单纯摘要更难给人讲清楚。

## 实现链
`microCompact` 先做局部修剪，尽量不提前改写整段会话历史。

## 普通做法
直接跳到 `autoCompact` 做整段摘要。

## 为什么不用
因为很多时候只是近处太肥，不值得重写整个叙事。

## 代价
会多出一层“已修剪但未摘要”的中间状态。
