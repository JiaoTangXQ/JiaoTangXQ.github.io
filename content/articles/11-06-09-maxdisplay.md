# 小屏时 maxDisplay 直接归零，说明空间不够时任务面板可以整块退后

终端行数太少时，这个面板不是“少放几条”，而是直接不放。它承认有些屏幕条件下，任务区本身就不该抢主视线。

## 实现链

`TaskListV2.tsx` 用 `rows <= 10 ? 0 : Math.min(10, Math.max(3, rows - 14))` 算 `maxDisplay`。一旦 `maxDisplay` 变成 0，`visibleTasks` 就会是空数组，而 `hiddenTasks` 会吞下全部任务；同时 `hiddenSummary` 只有在 `maxDisplay > 0` 时才会渲染，所以小屏下这块区域实际上会整块退后，只剩 standalone 模式的总览行还在。

## 普通做法

更普通的做法是硬挤出一两行，让列表至少看起来没空着，或者把所有任务都缩成更小的字。

## 为什么不用

Claude Code 不想把任务面板压成一条难读的缝。底部空间已经太窄时，最好的做法往往不是继续争位，而是让出空间给真正正在输入和响应的主界面。

## 代价

代价是小屏上任务区会彻底消失，用户得去别的入口找任务信息。这个决定更保守，但它避免了在最紧的屏幕里制造半截内容。
