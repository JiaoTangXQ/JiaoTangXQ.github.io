# status 为什么不是装饰而是生命周期总闸门

在 Claude Code 里，`status` 不是给 UI 上色用的标签，而是决定系统还能不能对任务做事的总闸门。

## 实现链

`TaskStatus` 只有五种状态，而 `isTerminalTaskStatus()` 把 `completed`、`failed`、`killed` 统一标为终态。`LocalAgentTask` 在 `completeAgentTask()`、`failAgentTask()`、`killAsyncAgent()` 里都会先改状态，再清理控制器和清理函数。`InProcessTeammateTask` 也会在注入消息前先检查是不是终态。

所以 status 不只是“描述发生了什么”，而是在控制“接下来还能不能继续注入、显示、回收或通知”。

## 普通做法

更普通的写法，是把状态当展示字段，而真正的逻辑分散在各种布尔值里，比如 `isRunning`、`isCancelled`、`hasExited`。

## 为什么不用

Claude Code 没这么做，是因为一旦真正控制流不认状态，状态就会退化成装饰，最终 nobody trusts it。系统会出现“UI 说完成了，但实际上还能塞消息”“看起来 killed 了，但清理没做完”这种错位。

它选择让状态成为总闸门，等于强迫所有模块承认同一个生命周期事实。

## 代价

代价是状态切换要非常谨慎。谁先改、谁后清理、谁负责通知，都得按顺序做，否则很容易出现重复通知、僵尸任务或过早回收。
