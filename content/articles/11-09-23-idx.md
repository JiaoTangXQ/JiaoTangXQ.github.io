# StickyTracker 的 lastIdx 只用消息 idx 去重，不用 estimate

`StickyTracker` 跟踪当前最靠近视口顶部的 prompt 消息。当这个「当前 prompt」发生变化时，它通知 `FullscreenLayout` 更新 sticky header。

这里有一个去重机制：`lastIdx` 记录上一次推送的消息索引。只有当找到的消息 idx 和 `lastIdx` 不同，才算「prompt 变化了」，才触发更新。

## 为什么不用 estimate 去重

每次滚动，`StickyTracker` 都会：

1. 找出当前视口顶部附近的 prompt 消息及其 idx
2. 计算这条消息的当前 estimate（位置估算值，基于 scrollTop 和已知的行高）

estimate 是一个持续变化的浮动数字——每帧滚动都会让它略微变化。如果把 `{ idx, estimate }` 作为去重键，每次微小滚动都会产生一个「新的」状态（因为 estimate 变了），就会频繁触发 sticky header 更新，header 里的内容会不停地「更新」，即使内容实际上没有变化。

## 身份和位置是两回事

`idx` 是消息在列表里的**身份**——它是「第 42 条消息」这个稳定事实。除非消息列表变化，这个数字不会变。

`estimate` 是消息的**位置估算**——它随着滚动和行高变化而持续漂移，是一个不稳定的近似值。

Sticky header 显示的是「当前对应哪条 prompt」，这是一个关于**身份**的问题，不是关于**位置**的问题。当 idx 没变，就意味着「还是那条 prompt」，不需要更新 header，即使 estimate 已经变了一截。

用 idx 去重，是在正确地回答「这个 prompt 变了吗」这个问题，而不是错误地用位置漂移来代替身份变化。
