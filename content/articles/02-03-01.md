# 如果把 init() 移到 main() 最前面，会发生什么？

## 重构假设

假设我们做这个"简化"：

```typescript
// 重构后（错误版本）
export async function main() {
  await init();  // 移到最前面
  initializeTelemetryAfterTrust();
  
  // ...然后注册 Commander 命令...
  program.command('mcp')...
  program.parse();
}
```

移除 `preAction` 钩子，把 `init()` 提到最前面。代码更线性，更容易理解执行顺序。

会发生什么？

## 问题一：--help 变慢了

用户输入 `claude --help`，希望立刻看到帮助文本。

现在，它要先：
1. `enableConfigs()`：读磁盘，~5ms
2. `applySafeConfigEnvironmentVariables()`：可能触发 keychain 读取，~65ms
3. `configureGlobalMTLS()`：配置证书，~10ms
4. `configureGlobalAgents()`：配置代理，~5ms
5. `preconnectAnthropicApi()`：发起 TCP 预热，~100ms（异步，但占用事件循环）
6. 初始化远端设置 promise

然后才输出帮助文本。用户等了几百毫秒，看到了和之前完全相同的帮助页。

这不是假设性的问题。在 Claude Code 这种规模的工具里，`init()` 的执行时间很容易超过 200ms。

## 问题二：配置解析失败会阻断 --help

如果 `~/.claude/settings.json` 格式不正确，`enableConfigs()` 会抛出 `ConfigParseError`。

原来的设计通过 `preAction` 保证：`--help` 路径根本不会执行 `init()`，所以配置文件损坏不影响用户查看帮助。

移到 `main()` 最前面后，配置文件格式错误会导致 `--help` 也失败：

```
$ claude --help
Configuration error in ~/.claude/settings.json: Unexpected token 'x' at position 42
```

用户想看帮助来修复问题，却因为有问题才看不到帮助。这是一个循环依赖的错误处理场景。

## 问题三：--settings 参数不再有效

这是最微妙的问题。

`init()` 调用 `enableConfigs()` 时，`--settings` 参数还没有被 Commander 解析，`eagerLoadSettings()` 的结果可能还没有生效（取决于调用顺序）。

原来的设计保证顺序：
1. `eagerLoadSettings()`（main() 开头，手动解析）
2. Commander 结构注册
3. `program.parse()`
4. `preAction` 触发 → `init()` 执行

移到最前面后：
1. `init()` 执行 → `enableConfigs()` 用默认设置
2. Commander 结构注册
3. `program.parse()`
4. Commander 发现 `--settings` 参数，但 `init()` 已经跑完了

此时 `--settings` 参数来晚了，配置文件的影响无法在最初的 `init()` 里体现。

## 问题四：preAction 钩子的去重保证消失

`init()` 用 `memoize` 包裹，保证只执行一次：

```typescript
export const init = memoize(async (): Promise<void> => { ... })
```

如果在 `main()` 最前面调用一次，后来 `preAction` 里（如果还保留着）又调用一次，`memoize` 会阻止重复执行——但如果把 `preAction` 完全移除，`memoize` 的价值就只剩下防止意外重复调用，而失去了"统一初始化入口"的语义。

## 为什么当前设计正确

`preAction` 作为闸门，本质上是在说："只有当你真的要执行某个命令时，才需要付完整初始化成本。"

这个判断很简单，但影响很深：
- 帮助路径保持零成本
- 初始化失败不影响用户查看命令说明
- 设置参数有足够的时间在 `init()` 之前生效
- 初始化入口统一，不会因调用路径不同而行为不一致

## 面试指导

"把初始化提前"是一类常见的看似合理的重构建议。评估这类建议时，要问：

1. **哪些路径会因此付出不必要的代价？** （`--help`、`--version`、纯文本输出的轻量命令）
2. **初始化失败是否会影响不应该失败的路径？** （配置错误 vs 帮助显示）
3. **初始化是否依赖某些参数，而这些参数要更晚才解析完成？** （`--settings`、`--setting-sources`）
4. **去重逻辑是否还成立？** （`memoize` 在新位置是否有意义）

能系统性地问出这四个问题，基本上就能覆盖"提前初始化"这类重构的所有风险。
