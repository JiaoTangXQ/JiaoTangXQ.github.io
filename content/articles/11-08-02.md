# normalizeMessages 做完之后还要 reorderMessagesInUI，为什么顺序不是一次到位？

这是一道典型的"为什么要这么多层"的架构题。短的回答是：因为每一层解决的问题属于不同的语义域，合并会让某层既知道 API 结构又知道 UI 规则，两种知识不该住在同一个函数里。

## 六道工序各解决什么

**normalizeMessages**：把原始 API 消息（`Message[]`）转成渲染层用得上的 `RenderableMessage[]`。这里处理的是数据形态问题：streaming 中间状态、`tool_use` 块的 content array、server_tool_use 等，都在这里被压平或补充元数据。

**过滤**：去掉 `shouldHideMessage` 判断为真的条目——比如纯系统消息、invisible synthetic 行。这是可见性问题，和规范化逻辑不同。

**reorderMessagesInUI**：在 UI 层把某些消息挪到更合适的位置。最典型的例子是：agent 任务完成时的 `task_complete` 消息在 API 时序里可能出现在最后一条用户消息之前，但在界面上要紧跟在对应 agent 输出后面。这是展示逻辑，不是数据逻辑。

**applyGrouping**：把连续的同类工具调用（如多次 Read）折叠成一个 `grouped_tool_use`，把连续 search 折叠成 `collapsed_read_search`。这是视觉聚合，在重排之后做才不会把本来连续的工具调用拆开。

**collapse/截断**：对超长会话应用 transcript cap（200 条上限，详见 11-09-32），把早期消息变成 `compact_summary`。

**buildMessageLookups**：在上面所有变换都稳定之后，扫出 7 张关系表。lookup 必须在最后，因为它要看重排后的最终 messagesToShow，不能在中途某个状态上建关系。

## 为什么不把 reorder 放进 normalize

面试里这个问题很常见。答案是：normalize 的输入是 `Message[]`，它的任务是"让数据可用"；reorder 的输入已经是 `RenderableMessage[]`，它的任务是"让界面好看"。两者对"正确性"的定义不同。

normalize 里的 bug 是数据丢失或类型错误，用单元测试就能覆盖，输入输出都是纯数据。reorder 里的 bug 是"看起来位置不对"，它的正确性依赖用户的阅读期望，需要集成测试甚至截图对比。把两者放在一个函数里，debug 时就很难判断是数据问题还是展示问题。

## lookups 不能在中途建

这是这条流水线里最容易踩的坑。假设你在 applyGrouping 之前就建了 lookup，会发生什么？

grouping 把 5 条独立的 Bash 工具调用合并成一个 `grouped_tool_use`，但 lookup 里已经记录了 5 个独立的 `toolUseByToolUseID` 条目，`siblingToolUseIDs` 也已经按合并前的结构建出来。等渲染层真正访问的是合并后的 grouped 对象，lookup 里找不到对应的 ID，组件就会得到 undefined。

所以 lookup 必须在最后一道工序后才能建：只有这时候，`messagesToShow` 里的每一条 `RenderableMessage` 都已经是最终形态。

## 面试指导

这道题考的是"流水线设计"的意识，而不只是背 Claude Code 的实现细节。

**核心答案要点**：每道变换的输入应该是前一道的稳定输出，"稳定"的意思是不会再被后续步骤改变结构。如果某道变换 A 依赖的数据还会被后续变换 B 修改，那 A 必须排在 B 之后。

**延伸问题**：如果 session 要支持实时协作（两个用户同时看同一个 transcript），哪些层需要加锁或变成响应式？答：normalize 和 lookup 层——前者处理原始消息流，后者建关系图，两者都会因为远端消息到来而重建；reorder 和 group 只处理 UI 逻辑，可以在本地视图里独立运行。

**一个陷阱追问**：`shouldRenderStatically` 里用的 `siblingToolUseIDs` 是在 grouping 之前还是之后建的？答：之后。group 操作之后 sibling 关系才是最终的，之前建出来的 sibling map 可能把已经被 group 的工具调用算作各自独立的 sibling，导致"还有 sibling 在等"的判断永远为真，消息永远不静态化。
