# 搜索的第一跳选最近命中，不选全局第一条

用户在 transcript 中间某处按下 `/` 开始搜索。键入几个字母后，系统找到了多个命中位置。第一次预览应该跳到哪里？

直觉上是「最早出现的那条」——毕竟按 `n` 向后翻才是正常的搜索流程。但 Claude Code 选的是另一个标准：**最接近用户当前位置的那条**。

## searchAnchor 是什么

用户按下 `/` 开始搜索的那一刻，`setAnchor()` 会记录当前的 `scrollTop`，存入 `searchAnchor.current`。这是用户「出发时所在的位置」。

`setSearchQuery` 在计算第一次预览时，不是取 `matches[0]`（全局序号最小的命中），而是遍历所有命中，找到距离 `searchAnchor` 最近的那条消息，对它调用 `jump(idx, wantLast=true)`。

`wantLast=true` 还有一个细节：在这条消息里，先定位到**最后一次出现**的命中，而不是第一次。这样当消息比较长时，预览位置会接近视口底边，而不是跳到消息顶部然后让用户找关键词在哪里。

## 为什么要贴近当前位置

用户在当前位置开始搜索，通常有两种意图：

1. 找当前正在阅读的上下文里的某个词
2. 从当前位置附近开始，向前后扩散查找

这两种意图都意味着：第一跳应该尽量贴近「出发点」，而不是把用户送到 transcript 开头的第一个命中。

如果跳到 transcript 开头，用户就需要：理解自己被移到了哪里、重新找回当前阅读上下文、再决定要不要从这里开始搜。这是一次不必要的视线迁移。

## 第一跳设计的背后

这个细节说明搜索 UX 不只是「找到匹配项」，还关乎「从哪里开始找」。起点的选择影响用户的工作流：好的起点让搜索融入当前阅读流程，坏的起点让搜索打断它。
