# 一旦碰到真实设备，问题就不再只是多几个工具

一旦能力真的落到真实设备上，问题立刻会从“工具有没有”变成“谁来授权、怎样连接、断了怎么收、跨平台怎么装、日志怎么留”。`utils/computerUse/setup.ts` 里最醒目的地方，不是工具列表，而是它专门生成动态 MCP 配置和 `allowedTools`，并且明确让 `request_access` 负责整场会话授权。它的意思很清楚：设备能力不是零散 API，而是一整段要被制度接住的现实行动。

更直觉的做法，是给模型丢几个 `click`、`type`、`screenshot` 工具，然后每次调用再弹窗确认。那样实现很快，但用起来会非常碎，用户也会在每一步都被打断。Claude Code 没这么做，是因为它要处理的不是“多几个动作”，而是“一段真实操作会话”。既然是会话，授权、边界和恢复就都该按会话来设计。

这也是为什么第十二卷后面会出现 wrapper、manifest、socket、bridge、paired device 和 reconnect 这些词。真正难的从来不是把动作名列出来，而是让现实世界愿意被接进来，又不把原来的工作台秩序冲散。

## 实现链
`setupComputerUseMCP()` 生成的不只是工具名，还把整簇能力包装成动态 MCP server；`setupClaudeInChrome()` 还要处理 manifest、wrapper、扩展检测、bridge 和 reconnect。代码重点已经不是“有哪些动作”，而是“怎样把这条接入链真的接通”。

## 普通做法
最直觉的做法是先把 `click`、`type`、`screenshot` 之类动作加进去，其他问题以后再补。

## 为什么不用
Claude Code 没这么做，因为设备世界的难点本来就不在动作本身，而在授权、安装、连接、掉线恢复和跨平台差异。动作先上线、制度后补，最后只会到处打补丁。

## 代价
代价是第十二卷很多代码都在做基础设施，而不是直接做“智能动作”；看起来更重，但这是接真实世界必须付的前置成本。
