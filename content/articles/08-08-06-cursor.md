# 翻页失败先保留cursor说明阅读器姿态里重试比逞强更重要

## 实现链
- `useAssistantHistory.ts` 在 `fetchOlderEvents()` 失败时，不会把 cursor 清空，而是把 sentinel 改成“failed to load older messages — scroll up to retry”，让下一次滚到顶时继续重试。
- 这说明作者把阅读器看成长期交互界面，而不是一次失败就放弃的短操作。

## 普通做法
- 普通做法是请求失败就把“没有更多了”和“加载失败了”混成同一种终止状态，避免用户不断重试。
- 代码会更简单。

## 为什么不用
- Claude Code 没把失败当终局，是因为远端历史分页失败很可能只是临时网络问题。直接清 cursor 会让用户误以为会话历史真的到头了。
- 从代码看，它宁可保留可重试状态，也不愿为了少写点逻辑牺牲阅读连续性。

## 代价
- 代价是同一个失败可能被重复触发，用户和开发者都要接受“重试是正常行为”。
- 但对阅读器来说，这比过早宣判“没了”更合理。
