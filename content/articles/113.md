# Claude Code 源码导图：三个"世界"的代码落点

## 为什么要分成三个世界

Claude Code 的复杂性不是一开始就有的。早期版本就是一个"AI 帮你写代码"的 CLI 工具。

后来加了 MCP（Model Context Protocol）——工具可以动态加载了，系统开始有了"扩展世界"。

后来加了远端执行——代码跑在云端，权限审批仍在本地，系统开始有了"远端边界世界"。

后来加了多 agent 协作——一个团队的 AI 成员共同完成任务，系统开始有了"团队世界"。

这三个方向相互独立，但也相互影响。理解每个方向对应哪些源码，是读懂 Claude Code 架构的关键。

## 扩展世界：src/tools/ 和 src/plugins/

**核心文件**：`src/tools/` 目录下的各种工具实现。

扩展世界的核心问题是：外部能力（MCP 服务、浏览器扩展、IDE 插件）如何加入 Claude Code 的工具体系，同时不破坏已有的权限管理和工具发现机制。

`src/tools/AgentTool/` 是这条线上最重要的目录——它实现了"用 AI agent 作为工具"的能力，让工具的边界从"确定性函数"扩展到"自主执行的 agent"。

`src/plugins/` 处理插件的加载和生命周期。

关键的设计判断在 `src/Tool.ts`：所有外部能力最终都要实现这个接口，用同一种语言描述自己。

## 远端边界世界：src/remote/ 和 src/bridge/

**核心文件**：`src/remote/` 目录下的四个文件。

```
src/remote/
  RemoteSessionManager.ts    # 管理远端会话的生命周期
  SessionsWebSocket.ts       # WebSocket 连接层
  remotePermissionBridge.ts  # 权限请求的翻译层
  sdkMessageAdapter.ts       # 消息格式适配
```

远端边界世界的核心问题是：当工具执行跑在远端容器时，权限审批、消息传递、断线恢复这些原本在本地处理的事情，如何跨越网络边界继续工作？

`remotePermissionBridge.ts` 是这条线上最清晰的代码——它只有两个函数，但每个函数都在解决一个具体的跨界问题（参见 098 文章的分析）。

`RemoteSessionManager.ts` 处理整个远端会话的生命周期，包括连接建立、状态同步、断线检测和重连逻辑。

## 团队世界：src/utils/swarm/ 和 src/hooks/swarm/

**核心文件**：`src/utils/swarm/` 目录。

```
src/utils/swarm/
  reconnection.ts      # 团队成员的初始化和重连
  permissionSync.ts    # 跨 agent 的权限同步
  teamHelpers.ts       # 团队文件系统工具函数
```

加上散布在其他目录的团队相关文件：
- `src/utils/mailbox.ts`：agent 间通信的文件邮箱实现
- `src/utils/teammateMailbox.ts`：队友专用的邮箱工具
- `src/context/mailbox.tsx`：邮箱的 React context
- `src/hooks/useSwarmInitialization.ts`：团队初始化的 hook

团队世界的核心问题是：多个 AI agent 如何在一起工作，同时保持各自的身份和权限，以及在任何一个 agent 断线或重启时能正确恢复协作状态。

`permissionSync.ts` 里有详细的设计注释，解释了"worker 遇到权限请求 → 发给 leader → leader 用户确认 → 发回 worker → worker 继续执行"的完整流程。这是团队世界最复杂的一个问题的解法。

## 三个世界的共同语言

三个世界各有专属代码，但有两样东西是共同的：

**MCP 作为公共协议**：扩展工具、浏览器工具、IDE 工具都接成 MCP 服务；远端工具通过 MCP 命名约定融入工具体系；团队内部通信虽然不走 MCP，但工具接口仍遵循 MCP 的命名规范。

**任务对象作为公共载体**：无论是本地工具调用、远端 agent 执行，还是团队成员的任务，都落成 `TaskStateBase` 扩展的对象。这让主界面的任务面板能显示所有类型的任务，不需要为每种类型单独渲染。

## 面试要点

1. **扩展世界的落点是 Tool 接口**：所有外部能力统一实现 `Tool.ts` 定义的接口，这是扩展不改变骨架的技术基础。
2. **远端边界世界的核心是翻译层**：`remote/` 目录下的四个文件专门处理跨界翻译，让审批链和消息处理不需要感知"远端"的存在。
3. **团队世界的核心是文件邮箱和权限同步**：mailbox 提供可恢复的通信面，permissionSync 解决跨 agent 的权限裁决问题。
4. **三个世界共享 MCP 语言和任务对象**：不同的世界有不同的实现，但共同的协议让它们能用同一套管理基础设施。
