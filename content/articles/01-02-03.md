# 为什么行动层要分成命令、工具和任务

很多系统一谈“能力”，喜欢把所有会动的东西都堆在一起。用户输入一个斜杠命令，模型调一个工具，后台再起一个子 agent，最后全都叫“功能”。这样短期很省事，长期却一定会乱，因为这三种东西根本不是同一种角色。

Claude Code 在这里分得很克制。`Command` 更像入口，它决定的是“这次要用哪一种工作方式进入系统”。有的命令会把一段提示词装进当前会话，有的命令直接在本地完成事情，有的命令甚至要临时拉起一段界面。`Tool` 才是模型真正伸手做事的那层，它代表读文件、改文件、跑 Bash、查网络、发消息、创建任务这些行动能力。`Task` 则更像被正式托管的工作单元，它关心的是后台运行、状态变化、输出位置、能否恢复和能否接管。

这三层一旦混在一起，系统就会开始说不清楚：什么东西只是改变工作方式，什么东西真的会改世界，什么东西又会在后台持续存在。Claude Code 把它们拆开以后，很多复杂能力反而变简单了。斜杠命令不再假装自己就是行动本身；工具不需要背负会话入口的职责；子 agent 和后台工作也不再是漂浮在空中的神秘过程，而是能被纳入任务制度。

这个分法的巧思，在于它不是为了概念整齐，而是为了给复杂度找正确的容器。代价也很真实：普通读者第一次进来时，往往会觉得层次太多。但如果不这样分，后面命令卷、工具卷、任务卷就会全部缠在一起，系统越强，解释反而越糊。

## 实现链
代码里命令、工具、任务是三层硬边界。`commands.ts` 和 `types/command.ts` 管入口；`tools.ts` 与 `Tool.ts` 管能力定义、schema 和权限口径；`Task.ts` 管工作单元的登记、推进和回收。它们分别回答入口、行动、生命周期三个问题。

## 普通做法
更直觉的写法是把三层揉在一起：命令函数直接去读文件、跑 bash、改状态，必要时顺手记一点进度。小工具这样写最快。

## 为什么不用那种做法
Claude Code 没这么做，因为它面对的不只是一次性命令，而是模型、用户、工具、子 agent、UI 和恢复链一起参与的工作过程。命令如果既负责入口又负责动作又负责生命周期，后面就没法稳定插入权限、审计和任务管理。

## 代价与回报
回报是边界清楚，复杂能力能沿正确层级插进去。代价是源码阅读会多一跳，你看到命令时不一定马上看到它把事情做完。
