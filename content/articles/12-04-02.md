# 架构分析：对外统一、对内高效——Claude Code 外延能力的接口设计哲学

## 两条平行的设计决策

读完 `setupComputerUseMCP()` 和 `setupClaudeInChrome()`，会发现两者都遵循同一个模式：

**对外**：返回标准 `type: 'stdio'` MCP 配置、统一命名的工具名、追加的 system prompt。

**对内**：computerUse 走进程内绑定（不真起子进程）；Chrome 则用 `createChromeContext()` 复用同一份上下文，而不是每次重新初始化。

```ts
// computerUse 的返回值
return {
  mcpConfig: { [COMPUTER_USE_MCP_SERVER_NAME]: { type: 'stdio', ... } },
  allowedTools,
}

// Claude in Chrome 的返回值
return {
  mcpConfig: { [CLAUDE_IN_CHROME_MCP_SERVER_NAME]: { type: 'stdio', ... } },
  allowedTools,
  systemPrompt: getChromeSystemPrompt(),
}
```

两者同构，尽管它们连接的是完全不同的外部世界。

## 为什么不二选一

有人可能会说：既然如此，干脆选一个方向就好——要么追求接口纯粹（每个 MCP server 都真起子进程），要么追求实现简单（完全不用 MCP，直接内置函数）。

Claude Code 两个极端都没选，原因是两者各有不可接受的代价：

**纯粹子进程方案的问题**：
- 每个工具调用都要跨进程 IPC，桌面操作这种高频连续场景延迟会显著上升
- 子进程管理增加系统复杂度（启动时机、崩溃处理、进程间状态同步）
- 对 computerUse 这种需要低延迟响应的场景，成本不可接受

**完全放弃 MCP 外观的问题**：
- 工具发现、权限系统、system prompt 注入都依赖 MCP 配置格式
- `allowedTools` 的管理逻辑要重写
- 后端 API 的 `mcp__computer-use__*` 前缀检测会失效
- 每个消费方都要学新的接入方式

第三条路：保持 MCP 外观，内部实现不受限。消费方接口稳定，内部优化空间保留。

## createChromeContext() 的复用策略

Claude in Chrome 的内部实现展示了另一种"对内高效"的策略——不是进程内绑定，而是上下文复用。

`createChromeContext()` 会一次性初始化整个 Chrome 桥接所需的上下文：socket 路径、bridge 配置、logger、权限模式、身份认证、事件追踪等等。这个上下文既被 subprocess MCP server 路径使用，也被 in-process 路径使用：

```ts
export function createChromeContext(env?: Record<string, string>): ClaudeForChromeContext {
  const logger = new DebugLogger()
  const chromeBridgeUrl = getChromeBridgeUrl()
  // ... 初始化所有共享状态

  return {
    serverName: 'Claude in Chrome',
    logger,
    socketPath: getSecureSocketPath(),
    getSocketPaths: getAllSocketPaths,
    // ... 所有共享状态
  }
}
```

这样做的意义：无论 Chrome 能力通过哪条路径被调用（真实的 MCP 子进程，或者进程内模拟），它们看到的是同一个上下文，身份是同一个、日志走同一个 logger、socket 路径是同一个。

不同路径不会有不同的行为，这对测试和调试都极其重要。

## 「wrapper 缓存 dispatcher」是同一哲学的体现

在 `wrapper.tsx`（computerUse）里，dispatcher 也被缓存而不是每次重新创建：

```ts
// 伪代码示意
let cachedDispatcher: Dispatcher | null = null

function getDispatcher() {
  if (!cachedDispatcher) {
    cachedDispatcher = createDispatcher()
  }
  return cachedDispatcher
}
```

这个缓存策略和 `createChromeContext()` 的复用是同一种逻辑：初始化成本高的对象，建立一次，反复复用。

对外表现是每次工具调用都"看起来像"在用同一个 MCP server，对内实现是复用同一个上下文对象，不付重复初始化的成本。

## 对消费方意味着什么

这种"外部口音统一"的设计，最终的受益方是系统里所有消费 MCP 配置的代码。

它们不需要知道：
- 这个 MCP server 是真的在运行子进程，还是进程内绑定？
- 这个桥接是走本地 socket，还是走远程 bridge？
- 这个工具调用是真实的网络请求，还是直接函数调用？

它们只需要知道：这是一个 `type: 'stdio'` 的 MCP server，有一组工具，工具名以 `mcp__computer-use__` 或 `mcp__claude-in-chrome__` 开头，就这样。

这是"对外统一"带来的最大好处：**新能力接入不要求消费方了解接入的实现细节**。

## 面试考察点

这道题考察的是**大型系统里如何在"接口稳定性"和"实现效率"之间取得平衡**。

一个常见的面试问题是："如果你要重构这个设计，你会怎么做？"

思路：
1. **现状的好处**：消费方解耦，内部可以自由优化
2. **现状的代价**：代码层面的"诚实性"差（stdio 配置但不 spawn）
3. **理想改进**：引入 `type: 'in-process'` 配置类型，让意图更清晰；但需要协调所有消费方升级

这类题没有标准答案，考察的是对架构权衡的推理过程，而不是对某个设计的记忆。
