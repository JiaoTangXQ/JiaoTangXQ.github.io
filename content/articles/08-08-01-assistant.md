# 分布式系统分析：viewerOnly 历史拉取为什么是第一步而不是懒加载？

## viewerOnly 的使用场景

ViewerOnly 模式允许第三方客户端（或同一用户的另一个设备）以**只读方式**观看一个正在进行的远端会话。

常见场景：
- 用户在电脑上发起了一个远端任务，用手机查看进度
- 团队成员观看另一个人的 Agent 操作过程
- 调试时实时查看某个 session 的输出

这个场景和普通的消息流订阅有一个本质区别：**观看者不是从头开始的**。任务已经进行了一段时间，有历史消息，观看者需要先了解上下文，才能理解当前正在发生的事情。

## 为什么先拉历史？

直接订阅实时消息流，不拉历史，会发生什么：

- 用户看到的是"半截现场"：正在执行的工具、当前的输出，但不知道为什么在做这件事
- Claude 可能在回复用户之前消息的内容，但观看者不知道之前的消息是什么
- 工具调用的上下文丢失：为什么用 `Edit` 修改这个文件？这是任务的哪一步？

没有历史，实时流是无意义的碎片。

`useAssistantHistory.ts` 在挂载时立刻调用历史拉取（`anchor_to_latest`）：

```typescript
// 挂载时的效果，简化版
useEffect(() => {
  async function init() {
    await prepareApiRequest()      // 获取 OAuth + 组织信息
    const history = await fetchHistory({ anchor: 'latest' })
    setMessages(history)           // 先设置历史
    subscribeToRealtime()          // 再订阅实时流
  }
  init()
}, [])
```

**先建立历史上下文，再接入实时流**。这是有意的顺序——历史加载完成前，实时流的消息可能需要排队或者等待，确保不会出现"历史和实时消息交错"的显示混乱。

## anchor_to_latest 的设计意图

`anchor_to_latest` 不是拉取全部历史，而是拉取**最新的一页**（recent window），然后可以向上翻页加载更早的历史。

这解决了一个实际问题：长时间运行的会话可能有几百条消息，全部拉取既慢又占内存。大多数观看者只关心"现在是什么情况"，而不是"一开始发生了什么"。

`anchor_to_latest` 锚定到最近的消息，让观看者看到的是会话当前的上下文，而不是从创建之初开始的完整历史。

## 与懒加载的对比

**懒加载方案**：只显示实时流，用户滚动到消息列表顶部时，自动加载更早的历史。

这在社交应用（微博、Twitter 历史）里很常见，因为用户主要消费的是最新内容，历史是可选的。

viewerOnly 的场景不同：

- 社交应用：用户知道内容从当前时刻开始，看不看历史是可选的
- viewerOnly：用户不知道任务从什么时候开始，没有上下文就不知道在看什么

对于"理解当前状态"这个需求，历史是必须的，不是可选的。把它设计成懒加载（"我不想看就不加载"）会让大多数用户在默认情况下看到没有上下文的内容。

## 请求失败时的处理

历史拉取可能失败（网络问题、权限问题）。`useAssistantHistory.ts` 在失败时的设计选择，直接影响用户体验：

- **失败后静默降级**：只显示实时流，不报错（但用户不知道历史加载失败了）
- **失败后显示错误**：告知用户"历史加载失败，只能看到实时内容"
- **失败后重试**：自动重试，用户等待

Claude Code 的选择（基于注释推断）是让错误尽早暴露，而不是静默降级——和 `handleControlRequest` 里的"明确失败"原则一致。这让用户知道他们看到的不是完整的会话历史，而不是误以为会话就从当前时刻开始。

## 历史消息的去重问题

实时流订阅和历史拉取之间存在一个竞争窗口：

- t=0：开始历史拉取
- t=1：服务端发出一条新消息 M
- t=2：历史拉取完成，结果包含消息 M（刚好在拉取窗口内）
- t=3：实时流收到消息 M（重复了）

`BoundedUUIDSet`（`recentInboundUUIDs`）解决了这个问题：历史消息在被处理时，它们的 UUID 被加入 Set；当实时流收到相同 UUID 的消息时，识别为重复并过滤掉。

这个去重机制在 viewerOnly 场景下尤其重要，因为历史拉取和实时流订阅的时间重叠窗口是不可避免的。

## 面试指导

"历史消息加载策略"是移动应用和实时系统里的经典设计题。

判断"是否需要先加载历史"的框架：

1. **用户的目标是什么？**了解"当前状态"（需要历史上下文）还是"最新更新"（不需要历史）？
2. **没有历史，实时流是否有意义？**能孤立理解 vs 必须有上下文才能理解
3. **历史的量级？**全量历史 vs 锚定窗口（anchor_to_latest）

viewerOnly 属于"需要历史上下文才能理解当前状态"的场景，因此优先加载历史是正确的设计。社交 feed 属于"最新更新本身就有意义"的场景，懒加载历史是正确的。识别场景的性质，是设计策略选择的前提。
