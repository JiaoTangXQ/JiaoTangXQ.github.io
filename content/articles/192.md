# 新旧工具来源如何共用同一条装配流水线

面试题版本：**"Claude Code 从最初只有内建工具，到后来支持 MCP server、skills 目录、插件等多种来源。这个演进过程有没有破坏原有的工具处理流水线？为什么能做到向后兼容？"**

## 流水线的结构

Claude Code 的工具装配流水线大致分几个阶段：

1. **收集阶段**：从各个来源收集工具定义
2. **规范化阶段**：把各种格式统一成内部 Tool 类型
3. **注册阶段**：把规范化后的工具注册到工具表
4. **消费阶段**：按名称查找、权限检查、构建提示词、执行

MCP server 引入之前，步骤 1 只有内建工具一个来源。MCP 引入后，步骤 1 增加了一个新来源，但步骤 2-4 完全不变。

这是为什么向后兼容能保证：**步骤 1 是可扩展的插槽，步骤 2-4 是稳定的流水线**。

## 装配流水线的稳定性来自接口边界

`src/tools.ts` 里用 `getAllTools()` 收集所有工具：

```typescript
// getAllTools 是来源插槽的聚合点
export function getAllTools(): Tools {
  const builtinTools = getBuiltinTools()
  const mcpTools = getMcpTools()      // 新来源，不改原有代码
  const skillTools = getSkillTools()  // 另一个新来源
  // ...
  return [...builtinTools, ...mcpTools, ...skillTools]
}
```

这个函数是系统演进的接缝：每次加一种新来源，只需要在这里加一行，然后实现对应的 `get*Tools()` 函数（含规范化逻辑）。

步骤 2-4 的代码从不修改，因为它们只依赖 `getAllTools()` 的返回值类型（`Tools`），而不是具体的来源信息。

## 为什么不给 MCP 单独开一条快速通道

在一些早期系统设计里，新类型的来源会得到"快速通道"处理：因为担心规范化流水线太慢，或者为了保留 MCP 特有的一些元数据，所以在消费层加了一个 `if (tool.source === 'mcp') { specialMcpPath(tool) }`。

这种做法的短期代价不大，但长期问题是：
- MCP 工具的权限检查规则可能和内建工具不同
- MCP 工具的提示词格式可能单独维护一套
- MCP 工具的错误处理有特殊路径

每一条 `if-else` 都是系统分叉的开始。两套制度动作不可能永远同步演进，最终一定会出现"这个 feature 只有内建工具支持，MCP 还没支持"或者"这个 bug 只在 MCP 工具里有，内建工具没有"这类问题。

Claude Code 的选择是：**忍受接入层规范化的代价，换取消费层的统一**。把差异消化在接入点，而不是让差异渗透到所有消费代码里。

## 升级时的制度动作不分叉

"升级时也不让新旧生态各自长出第二套制度动作"这句话的工程含义：

**制度动作**指的是权限检查、工具执行、进度更新、结果处理这类系统级操作。

当 Claude Code 在主循环里加了新的能力——比如工具执行结果要同时更新 `outputFile`——这个改动只需要在消费层流水线里加一次，所有工具（内建的、MCP 的、技能的）自动获得这个新行为。

如果新旧工具有各自的流水线，这类系统级改动就要在两个地方同时改，而且得确保两个地方的改动语义一致——这是维护成本的双重收税。

## 面试角度的总结

新旧工具来源共用同一条装配流水线，依赖的是一个简单但关键的架构决定：**工具来源的多样性只在接入层可见，消费层只见到统一的 Tool 类型**。这让系统演进时，新来源的加入不会影响消费层的逻辑，系统级的改动也不需要在多条流水线里同步。代价是接入层需要认真实现每种来源的规范化适配，但这个代价是一次性的，消费层的统一是长期的回报。
