# 第十三卷回收索引：前四卷里的 Bash 失败会带崩兄弟工具，错误不能沿隐含依赖扩散

## 实现链
前四卷里还有一条线，专门回答“为什么一个 Bash 失败，不能只当成一个局部失败”。

这条线真正逼出来的，是第十三卷里关于复杂度换稳定的另一半判断。

这页实际回收的实现链是：
- `04-06-03` 把 Bash 失败写成会带崩兄弟工具，这会继续支撑 `13-05-02`，说明成熟系统宁可多付一些编排复杂度，也不让错误沿着隐含依赖链扩散。

Bash 失败最直白地说明：看起来只是一个脚本出错，实际可能把一串隐含依赖一起拖下水。  
系统要先做的，不是事后解释，而是先把扩散关住。

## 普通做法
更普通的做法，是把启动、输入、主循环里的这些小判断留在顺手逻辑里，不单独立规矩。

## 为什么不用
Claude Code 不这么做，是因为工作台一旦要长期运行，这些看似前期的小处理其实都是后面秩序能否站稳的根，不能等出事了再补。

## 代价
代价是前期代码和解释都会显得更重，也更早暴露出长期工程的野心。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
