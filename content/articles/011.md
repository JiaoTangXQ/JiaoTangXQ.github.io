# Command 和 Tool 在 Claude Code 里为什么是两个完全不同的概念？

在大多数 CLI 工具里，command 和 tool 几乎是同义词。但在 Claude Code 的源码里，这两个词是严格区分的——它们处于完全不同的运行时层级，承担完全不同的职责。

## Command 是什么

Command 是**用户**和系统交互的入口。当你输入 `/pr`、`/doctor`、`/compact` 时，你在调用 Command。

从类型签名看 Command 的本质：

```typescript
// commands.ts 中的核心思路
type Command = {
  name: string
  description: string
  // 用户侧触发的处理函数
  handler: (args: string[], context: CommandContext) => Promise<void>
  // slash 命令的补全建议
  completions?: string[]
}
```

Command 由人触发，在对话层工作，可以操作整个应用状态：改变模式、重置会话、展示 UI 面板。`/compact` 可以重写消息历史；`/clear` 可以清空整个对话。这些动作不是模型决定的，是用户主动发起的。

## Tool 是什么

Tool 是**模型**与外部世界交互的接口。当 Claude 调用 `BashTool`、`FileEditTool`、`WebFetchTool` 时，它在调用 Tool。

Tool 在 `Tool.ts` 里有完整定义，关键字段：

```typescript
export type Tool = {
  name: string
  call(args, context, canUseTool, parentMessage, onProgress): Promise<ToolResult>
  checkPermissions(input, context): Promise<PermissionResult>
  isConcurrencySafe(input): boolean
  isReadOnly(input): boolean
  isDestructive?(input): boolean
  // 渲染相关
  renderToolUseMessage(input, options): React.ReactNode
  renderToolResultMessage(content, progressMessages, options): React.ReactNode
  // ...
}
```

Tool 由模型决定调用时机，经过权限系统检查，每次调用都有完整的审计链。

## 核心区别总结

| 维度 | Command | Tool |
|------|---------|------|
| 触发方 | 用户（slash 命令或键盘快捷键） | 模型（tool_use block） |
| 权限检查 | 无（用户主动发起即视为授权） | `checkPermissions` + `ToolPermissionContext` |
| 并发安全 | 通常是串行用户操作 | `isConcurrencySafe` 标注 |
| 可中断 | 用户随时可输入新命令 | `interruptBehavior: 'cancel' | 'block'` |
| 渲染 | 应用级 UI 操作 | `renderToolUseMessage` + `renderToolResultMessage` |

## 容易混淆的边界：SkillTool

`SkillTool` 是一个有意思的例子——它的名字带 Tool，但它的本质更接近 Command 的封装。Skill 让模型能"调用用户定义的工作流"，背后是把预定义命令序列注入到工具调用上下文。这是系统里少有的"模型触发、但内容是用户预定义命令流"的混合物。

## 为什么 Agent 也不等于 Tool

`AgentTool` 让模型启动子 Agent，但 Agent（包括 subagent 和 teammate）都是运行主体，不是接口。`Task.ts` 里的 `TaskType` 列出了所有可能的工作主体：

```typescript
export type TaskType =
  | 'local_bash'
  | 'local_agent'
  | 'remote_agent'
  | 'in_process_teammate'
  | 'local_workflow'
  | 'monitor_mcp'
  | 'dream'
```

其中 `local_agent` 和 `in_process_teammate` 都是 Agent，但进入运行时的方式和生命周期管理完全不同。混淆 Tool 和 Agent 最大的代价是：会以为"调用一个 AgentTool 就是创建了一个有完整权限管理的 teammate"——实际上不是。

## 面试角度

这个区分在面试中的实际价值：如果你在设计一个 AI 助手系统，面试官问"模型应该怎么和外部世界交互"——能区分"模型主动触发的操作"和"用户主动触发的操作"，并解释为什么这两套要分开设计（权限模型不同、审计需求不同、中断语义不同），会比"全部用 function calling 实现"的回答深一个层次。
