# setup() 里为什么 setCwd() 必须是第一个调用？

## 代码里的显式约束

```typescript
// setup.ts
// IMPORTANT: setCwd() must be called before any other code that depends on the cwd
setCwd(cwd)

// Capture hooks configuration snapshot to avoid hidden hook modifications.
// IMPORTANT: Must be called AFTER setCwd() so hooks are loaded from the correct directory
const hooksStart = Date.now()
captureHooksConfigSnapshot()

// Initialize FileChanged hook watcher — sync, reads hook config snapshot
initializeFileChangedWatcher(cwd)
```

两个 `IMPORTANT` 注释，都和 `setCwd()` 有关。这不是一般的代码风格约定，而是在提醒维护者：这里的顺序有语义上的约束，不能随意调整。

## setCwd() 实际上在做什么？

`setCwd()` 不只是设置一个全局变量。它更新了 `bootstrap/state.ts` 里的 `cwd` 字段，后续所有调用 `getCwd()` 的代码都会返回新值。

在 Claude Code 里，`getCwd()` 被用于：
- 读取项目配置（`.claude/settings.json`）
- 读取 hooks 配置
- 读取 CLAUDE.md 文件
- 确定 git 根目录
- 解析相对路径
- 确定 session 存储位置

所有这些操作，必须在 cwd 被设置为正确值之后才能执行。

## captureHooksConfigSnapshot() 的依赖

```typescript
// hooksConfigSnapshot.ts（大致逻辑）
export function captureHooksConfigSnapshot(): void {
  const hooks = loadHooksFromConfig()  // 读取当前 cwd 下的 hooks
  snapshot = JSON.stringify(hooks)     // 存储快照
}
```

`captureHooksConfigSnapshot()` 从当前 cwd 读取 hooks 配置，存入快照。这个快照用于后续检测 hooks 是否被修改（防止恶意 hook 在运行中被替换）。

如果 `setCwd()` 在 `captureHooksConfigSnapshot()` 之后调用：快照是从错误目录读取的，后续的 hook 比较也是针对错误目录的 hooks。在 worktree 模式下，这意味着系统在整个会话里使用的是原始目录的 hooks，而不是 worktree 的 hooks。

## 竞争条件场景

设想一个安全相关的竞争条件：

1. `captureHooksConfigSnapshot()` 从目录 A 读取 hooks（A 是正常项目）
2. `setCwd(worktreePath)` 切换到目录 B（B 是用户指定的 worktree）
3. 整个会话期间，hook 比较用的是目录 A 的快照
4. 目录 B 里有一个恶意 hook，但快照检测看的是目录 A，检测不到

这不只是功能 bug，还是潜在的安全漏洞：hook 变化检测失效。

## 初始化 FileChanged watcher 的顺序依赖

```typescript
initializeFileChangedWatcher(cwd)
```

这个 watcher 监控 cwd 目录里的文件变化，触发 hooks 重载等功能。它必须在正确的 cwd 下初始化，否则监控的是错误目录。

但它还有一个更微妙的依赖：必须在 `captureHooksConfigSnapshot()` 之后，因为它的热更新逻辑依赖快照存在。

所以完整的顺序约束是：

```
setCwd(cwd)                     // 第一
  ↓
captureHooksConfigSnapshot()    // 第二（依赖正确的 cwd）
  ↓
initializeFileChangedWatcher()  // 第三（依赖快照存在）
```

## 为什么注释而不是类型系统来表达约束？

一个好问题：为什么不把 `captureHooksConfigSnapshot()` 设计成接受 `cwd` 参数，从而通过函数签名表达依赖？

```typescript
// 替代设计
captureHooksConfigSnapshot(cwd)  // 显式传入 cwd，不依赖全局状态
```

可能的原因：`getCwd()` 在代码库很多地方被调用，使用全局状态比满地传参更实际。权衡是：依赖关系变得隐式，需要注释来补充说明。

Claude Code 选择了"注释表达约束"而不是"类型系统表达约束"，这是一个真实的工程取舍，不是最优解，但在现有代码结构下是合理的选择。

## 面试指导

"函数调用顺序约束"是系统设计里一个常见但难以完美解决的问题。

三种表达方式：
1. **注释**：`// IMPORTANT: must be called before X`——简单，但不强制
2. **类型系统**：通过参数类型让编译器强制顺序——强制，但需要重构
3. **封装**：把必须按顺序执行的操作封装成一个函数——最强，但减少了灵活性

Claude Code 选择了第一种，并且通过大写 `IMPORTANT` 来增强可见性。能分析这个选择的权衡，是展示工程判断力的好机会：不是所有约束都值得重构来通过类型系统强制，有时候明确的注释 + 团队约定就足够了。
