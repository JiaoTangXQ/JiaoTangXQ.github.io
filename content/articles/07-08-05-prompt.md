# 资源变化为什么连 prompt 缓存也打掉？过期 prompt 比多一次网络请求更危险

## 成本分析：什么更贵——过期数据还是多一次请求？

这是一个具体的工程权衡题。

**选项 A：保守缓存（资源变化就失效 prompt）**
- 代价：多一次 `listPrompts` 请求（通常几十毫秒，网络开销）
- 好处：prompt 总是和最新资源状态一致

**选项 B：激进缓存（资源变化不影响 prompt）**
- 代价：可能用过期的 prompt，导致模型基于不一致的上下文工作
- 好处：节省一次 `listPrompts` 请求

Claude Code 选 A。原因不只是"保险起见"，而是有具体的失效场景支撑。

---

## 过期 prompt 会造成什么问题？

**场景一：资源被删除，prompt 里还在引用它**

```markdown
<!-- 旧 prompt 模板 -->
请分析以下配置文件：
{resource://config/production.yaml}
```

如果 `production.yaml` 已经被删除（`resources/list_changed` 通知），但 prompt 缓存没更新，模型会看到一个引用了不存在资源的 prompt。

模型可能会：
- 尝试读取这个资源，得到"资源不存在"错误
- 基于错误的假设继续工作（"配置文件在这里，只是我读不到"）
- 给出错误的分析结果

**场景二：资源结构变化，prompt 里的参数引用过时**

```markdown
<!-- 旧 prompt 模板 -->
分析 {resource://data/schema.json} 里的 `users.age` 字段
```

如果 schema 重构了，`users.age` 改成了 `profiles.age`，但 prompt 缓存没更新，模型收到的 prompt 里还是旧字段名。这种错误在调试时非常难发现，因为 prompt 看起来"正确"，错误在语义层面。

---

## 模型错误的特殊成本

在普通软件系统里，"读到过期缓存"通常只导致短暂的显示错误，用户刷新一下就好了。

但在 AI 系统里，"模型基于过期上下文工作"的后果更严重：

1. **错误不明显**：模型不会说"我拿到了过期数据"，它会基于过期数据给出一个看起来合理的答案
2. **用户可能信任这个答案**：如果模型的回答听起来很有道理，用户不会怀疑它用了过期数据
3. **修正成本高**：如果用户基于错误答案采取了行动，修正代价远高于多发一次网络请求

---

## 为什么不给 prompt 独立的 TTL？

另一种方案：给 prompt 缓存设置一个较短的 TTL（如 5 分钟），定期自动刷新，不依赖资源变化通知。

这个方案的问题：
- **延迟**：TTL 到期前，仍然可能用过期 prompt（最坏情况是 5 分钟）
- **不精准**：即使资源没变，5 分钟后也会触发无意义的刷新
- **无法感知"资源刚刚变化了"**：TTL 方案对变化不敏感，必须等定时器

通知驱动的失效（`resources/list_changed` → 立即失效 prompt 缓存）在延迟和精准性上都优于 TTL。

---

## 实际失效逻辑的传播

```
MCP server 推送 resources/list_changed
    ↓
client.ts 收到通知
    ↓
1. await client.listResources()  → 更新资源缓存
2. invalidatePromptCache(serverName)  → 清空该服务器的 prompt 缓存
    ↓
下次用户触发 prompt
    ↓
cache miss → await client.listPrompts()  → 重建 prompt 缓存
    ↓
用户看到最新的 prompt 列表
```

这个链条里有一个延迟：prompt 缓存不是在资源变化时立即重建（那样太激进），而是等到下次真正需要 prompt 时才重新拉取（lazy invalidation）。这是"立即失效，延迟重建"的标准策略。

## 面试指导

**"缓存失效策略中，如何权衡一致性和性能？"**

回答这道题要先区分两个维度：

1. **失效时机**：立即（事件驱动）vs 定时（TTL）vs 手动
2. **重建时机**：立即（失效时同步重建）vs 延迟（下次使用时重建）

Claude Code 的 prompt 缓存策略是：**立即失效 + 延迟重建**。

这个组合的逻辑：
- 立即失效保证"不会用到过期数据"
- 延迟重建避免"失效时可能没有人需要这份数据，重建是浪费"

什么时候用立即重建？当"数据一旦失效就会立刻被需要"时（比如关键配置缓存）。

强调：**"数据过期的代价"是关键因素**。对于 AI 系统，过期数据导致模型错误的代价比多一次 `listPrompts` 网络请求要贵得多。这个判断驱动了缓存策略的选择。
