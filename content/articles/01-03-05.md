# 面试追问：Task.ts 里的任务 ID 生成只有 8 个字符，够用吗？这里有个边界条件

## 先看这段注释

`Task.ts` 里 `generateTaskId` 函数上面有一段注释，通常会被快速读过去：

```
// Case-insensitive-safe alphabet (digits + lowercase) for task IDs.
// 36^8 ≈ 2.8 trillion combinations, sufficient to resist brute-force symlink attacks.
```

这里有两个值得停下来想的东西。

第一：为什么是"case-insensitive-safe"？因为任务 ID 会被用作文件路径（`outputFile: getTaskOutputPath(id)`），而某些文件系统（比如 macOS 的 HFS+ 默认配置）是大小写不敏感的，`A1b2C3d4` 和 `a1b2c3d4` 会指向同一个文件。用全小写字母 + 数字的字母表（`0-9a-z`）彻底规避这个问题。

第二：为什么安全需求是"抵抗暴力枚举 symlink 攻击"，而不是"避免碰撞"？

## symlink 攻击是怎么发生的

`outputFile` 是 `TaskStateBase` 的必选字段，每个任务在被创建时就分配了一个输出文件路径。任务运行过程中，子 agent 或 bash 命令的输出会被写入这个文件。

如果攻击者能在任务 ID 生成之前，提前在 `outputFile` 对应的路径创建一个符号链接，指向一个敏感文件（比如 `~/.ssh/authorized_keys`），那么后续对 `outputFile` 的写操作就会覆盖那个敏感文件。

2.8 兆种组合意味着：就算攻击者每秒猜 100 万个 ID，平均也需要 2800 多秒才能碰中。足够长的窗口让这种攻击在实践中不可行。

这说明 `outputFile` 不是普通的日志路径，而是有安全含义的资源。

## 7 种任务类型的前缀设计

`Task.ts` 里定义了 7 种 `TaskType`，每种有不同的前缀字符：

```
local_bash       → 'b'  (保留向后兼容)
local_agent      → 'a'
remote_agent     → 'r'
in_process_teammate → 't'
local_workflow   → 'w'
monitor_mcp      → 'm'
dream            → 'd'
```

`local_bash` 的注释说"保留 'b' 以向后兼容"——这意味着前缀字符已经成为了某种外部可见的接口契约，可能有日志分析工具、调试脚本或者 transcript 处理逻辑在依赖"以 b 开头的 ID 是 bash 任务"这个规律。一旦改掉就是破坏性变更。

前缀的实际价值：不用解析状态就能从 ID 判断任务类型，在日志里区分不同类别的任务，在调试时一眼知道某个 ID 对应什么。

## TaskStatus 的 5 个状态和一个注释

```typescript
type TaskStatus = 'pending' | 'running' | 'completed' | 'failed' | 'killed'
```

注意没有 `paused` 状态。Claude Code 确实有任务暂停机制（`totalPausedMs` 字段记录累计暂停时间），但暂停不是一个 TaskStatus，而是在 `running` 状态下叠加的一个时间计量。这个区别很重要：暂停的任务仍然算 running，只是累计了暂停时长用于统计。

`isTerminalTaskStatus` 函数的注释解释了设计意图：`True when a task is in a terminal state and will not transition further.` terminal 状态只有三种：`completed`、`failed`、`killed`。这三种状态下，任务不会再产生输出，不会接收新消息，不会被前台接管。在系统里这个函数用来做守卫——"不要向终态任务注入消息"、"从 AppState 里驱逐已完成任务"、"孤儿清理路径"。

`DreamTask.ts` 里有一个典型示例：`completeDreamTask` 函数把状态设为 `completed` 时，同时把 `notified: true` 写进去。注释解释：`notified: true immediately — dream has no model-facing notification path (it's UI-only), and eviction requires terminal + notified.` 也就是说，任务被从 AppState 里驱逐的条件是**同时满足** terminal 状态和 notified=true。`dream` 类型的任务不需要通知模型（它是纯 UI 表面），所以直接标记为已通知。

## outputFile 是任务系统的关键字段

`TaskStateBase` 里有一个很安静的字段：`outputFile: string`，和它配套的是 `outputOffset: number`。

这两个字段解决了一个真实问题：子 agent 在后台运行时，它的输出写到哪里？前台界面怎么知道有多少新内容？

答案是：`outputFile` 是每个任务专属的磁盘文件路径，`outputOffset` 记录当前已读位置。前台界面想查看任务输出时，从 `outputOffset` 开始读 `outputFile`。多个进程（任务写入进程和 UI 读取进程）通过这个文件协调，不需要共享内存或进程间通信。

这个设计的好处之一：任务即使崩溃或被 kill，输出历史仍然在磁盘上，可以事后读取。`isTerminalTaskStatus` 不会删掉 `outputFile`——那是任务系统的外部审计轨迹。

## 面试角度

**考官会怎么问**：给你看 `TaskStateBase` 的字段定义，问"为什么需要 outputFile，为什么需要 outputOffset"，或者"任务 ID 用随机字符，而不是自增 ID，有什么考虑"。

**满分答法**：从 outputFile 的安全性（symlink 攻击）和可观察性（跨进程读写历史输出）两个维度回答；对于任务 ID，说清楚文件系统路径安全性是比碰撞率更核心的考量。

**常见错误**：把任务 ID 的设计归因为"随机 ID 比自增 ID 更扩展性好"（这在分布式系统里对，但 Claude Code 的场景更关心的是文件系统安全），或者说不清楚 `outputFile` 和日志文件的区别（日志是追加的可选项，`outputFile` 是任务状态的必要组成部分，直接参与任务生命周期管理）。
