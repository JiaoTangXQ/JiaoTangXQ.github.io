# 为什么任务是被系统正式托管的工作单元

很多 agent 产品一谈到“后台执行”或“子 agent”，听起来都像魔法：仿佛只要一句话发出去，另一个聪明体就会自己在暗处忙完一切。Claude Code 比较成熟的一点，是它不肯把这种行为写成魔法，而是把它落成了一个工程对象：任务。

在 `Task.ts` 里，任务不是一团模糊的异步过程。它有明确的 `type`、`status`、`description`、`startTime`、`endTime`、`outputFile`、`outputOffset`，还有统一的 `taskId`。这意味着系统对后台工作关心的不是“它大概在跑”，而是“它是谁、现在到哪一步、输出落在哪里、能不能被清理、能不能被前台重新接管”。一旦这些事实被钉住，后台 agent、远端 agent、bash、workflow、监控任务就不再是五种互不相干的怪功能，而是同一家族里的不同任务类型。

这一步的价值非常大。它把“多 agent”从表演效果翻译成了可管理的劳动。`AppStateStore` 里专门保存 `tasks`、前台任务、查看中的 agent 任务、名称到 `AgentId` 的登记表，这说明系统不只是让任务存在，还让它们能被看见、被切换、被终止、被回收。对普通用户来说，你看到的是“后台有人在干活”；对这套系统来说，它看到的是“有一份正式登记过的工作正在运行”。

这正是优雅之处。真正好的 agent 工程，不会让能力越强就越像黑箱，而是会在能力变复杂时，把它们翻译成更稳定、更可追踪的对象。代价当然也有：一旦任务种类越来越多，理解成本就会上升，文档和界面都必须帮用户分清楚什么是任务、什么是工具、什么是会话里的普通消息。Claude Code 选择先把制度立起来，再慢慢优化理解成本，这个顺序是对的。

## 实现链
这一章对应的实现链是角色拆分：命令负责把人带进正确入口，工具负责真正伸手做事，任务负责把劳动变成正式对象，子 agent 则站在 `AgentTool` 和任务生命周期之间。代码没有把这些角色混成一个万能 handler，而是沿 `commands.ts`、`Tool.ts`、`Task.ts` 和 AgentTool/LocalAgentTask 这几层拆开。

## 普通做法
更普通的做法是让命令既做入口又做动作，必要时再在某个回调里顺手记状态、起并发。这样局部很快，也很像脚本。

## 为什么不用那种做法
Claude Code 不这么写，因为它需要共享制度：权限、状态显示、恢复、后台托管、团队协作都要求这些角色各守边界。角色一旦混写，后面每加一种制度都只能去各处补洞。

## 代价与回报
回报是系统里谁负责入口、谁负责动作、谁负责生命周期更清楚。代价是层次比普通脚本工具更正式，理解时要接受“会做事”和“被系统正式管理”不是一回事。
