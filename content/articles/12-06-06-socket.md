# 安全分析：Chrome Native Host 的 socket 为什么要先清陈旧、再收权限？

## Unix socket 作为 IPC 机制的安全特性

Chrome Native Host 需要接收来自 Chrome 扩展的工具请求，并把请求转发给 Claude Code 的 MCP 服务。这里使用的是 Unix domain socket，而不是 TCP 端口。

选择 Unix socket 而不是 TCP 的理由之一就是安全性：Unix socket 是文件系统上的一个文件，访问控制完全由文件系统权限决定。TCP 端口默认对本机所有进程可达，而 Unix socket 可以通过目录权限精确控制哪些进程可以连接。

这也解释了为什么 socket 的权限设置很重要。

## chromeNativeHost.ts 的启动顺序

```ts
async start(): Promise<void> {
  this.socketPath = getSecureSocketPath()

  if (platform() !== 'win32') {
    const socketDir = getSocketDir()

    // 第一步：处理 socket 目录可能是文件/socket 的遗留情况
    try {
      const dirStats = await stat(socketDir)
      if (!dirStats.isDirectory()) {
        await unlink(socketDir)  // 如果不是目录，删掉它
      }
    } catch { /* 不存在，正常 */ }

    // 第二步：创建目录，权限设为 0700
    await mkdir(socketDir, { recursive: true, mode: 0o700 })
    await chmod(socketDir, 0o700).catch(() => {})

    // 第三步：清理 stale sockets
    const files = await readdir(socketDir)
    for (const file of files) {
      if (!file.endsWith('.sock')) { continue }
      const pid = parseInt(file.replace('.sock', ''), 10)
      try {
        process.kill(pid, 0)  // 信号 0：检查进程是否存在
        // 进程还活着，留着
      } catch {
        // 进程已死，清理 socket 文件
        await unlink(join(socketDir, file)).catch(() => {})
        log(`Removed stale socket for PID ${pid}`)
      }
    }
  }

  // 第四步：监听 socket
  this.server = createServer(socket => this.handleMcpClient(socket))
  await new Promise<void>((resolve, reject) => {
    this.server!.listen(this.socketPath!, () => { this.running = true; resolve() })
  })

  // 第五步：设置 socket 文件权限为 0600
  if (platform() !== 'win32') {
    await chmod(this.socketPath!, 0o600)
  }
}
```

每一步都有其安全含义。

## 0700 目录权限的含义

`0700` = 所有者可读/写/执行，组和其他用户无权限。

这意味着：
- 只有当前用户可以进入这个目录（执行权限 = 可以列出目录内容和访问目录内的文件）
- 其他用户无法查看目录里有哪些 socket 文件，更无法连接

这防止了本机上其他用户（在多用户系统如服务器上）偷偷连接到 Claude Code 的 Chrome 桥。

对应的，socket 文件权限设为 `0600`（只有所有者可读写），进一步限制了连接权。即使有人能绕过目录权限看到文件名，也无法建立连接。

## process.kill(pid, 0) 的技巧

这是一个 Unix 编程里的经典技巧：

```ts
try {
  process.kill(pid, 0)  // 发信号 0
  // 不抛出 = 进程存在
} catch {
  // 抛出 ESRCH = 进程不存在
}
```

`kill(pid, 0)` 不发送任何信号，只检查进程是否存在且调用者有权限向其发信号。如果进程不存在，抛出 `ESRCH` 错误。

这是判断"某个 PID 的进程是否还活着"的标准 Unix 做法，不需要读 `/proc/PID/` 或其他平台相关的方式。

## 为什么需要清理 stale sockets

Socket 文件的生命周期和进程不绑定——进程死了，socket 文件还在。如果 Claude Code 进程崩溃（而不是正常退出），它来不及删除自己的 socket 文件。

下次启动时，如果不清理旧的 socket 文件，`getSocketDir()` 里会有一堆死进程留下的 `.sock` 文件。`getAllSocketPaths()` 会把这些路径都返回，Chrome 扩展会尝试连接这些死 socket，失败后才尝试新 socket，增加连接延迟。

PID-based 命名（`{pid}.sock`）使得清理逻辑非常简单：从文件名拿到 PID，检查进程是否存在，不存在就删文件。

## 「先迁移旧格式，再做当前操作」的模式

注意第一步：检查 `socketDir` 是否是目录，如果不是目录（而是文件或 socket）就删掉它。

这是在处理系统升级时的遗留情况。旧版本的 Claude Code 可能使用单一 socket 文件而不是目录+多文件的方案。如果升级后代码认为该路径是目录，但实际是文件，`mkdir` 会失败。

这种"先检查格式是否符合当前期望，再进行当前操作"的模式，是处理系统演进时的常见手法。

## 面试考察点

这道题考察的是**本地 IPC 安全设计**。

Unix socket 权限管理和 TCP 不同，但在多用户系统（服务器、共享开发环境）里同样重要。关键知识点：
- 目录执行权限控制是否可以 "看到" 目录里的内容
- 文件权限控制实际的读写访问
- socket 文件权限控制是否可以连接这个 socket
- `process.kill(pid, 0)` 检查进程存活的标准方法

在系统设计面试中，如果要设计一个只允许特定用户访问的本地 IPC 通道，Unix socket + 文件系统权限是比 TCP + 认证更简洁、更可靠的方案。
