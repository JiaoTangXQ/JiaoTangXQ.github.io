# 状态世界导读：AppState 先把当前现实钉成同一个世界

这一章要先钉住一个前提：`AppState` 不是页面数据仓库，而是整台工作台当前成立的事实世界。权限模式、任务、远端连接、团队上下文、通知、底栏选择，这些东西如果不先住进同一个现实里，界面后面再精致也会散。

所以这一章的七片叶子，要连起来读成“现实世界怎样被建立”。先看 [11-02-01-AppState-装的不是页面数据，而是当前世界事实.md](/Users/xxx/tmp/cc/docs/claude-code-book/11-终端界面怎样成为运行时/11-02-AppState 为什么是整台工作台的事实世界/11-02-01-AppState-装的不是页面数据，而是当前世界事实.md)，再顺着默认开局、产品立场、身份现实、副作用总闸门和外部元数据回灌往下读，这一章才会从“状态字段介绍”变成“界面运行时怎样先认清自己身处哪个世界”的入口。

## 实现链

这一组真正落在 `AppStateStore.ts`、`AppState.tsx` 和 `onChangeAppState.ts`：先用 `getDefaultAppState()` 定开局，再用外部 store 托管共同现实，再用 `onChangeAppState()` 集中处理副作用。第十一卷这里不是在讲某几个字段，而是在讲“整台工作台怎样先承认自己活在同一个现实里”。

## 普通做法

更普通的 UI 会按页面拆几份局部 state：输入一份、任务一份、通知一份、远端再来一份，谁需要谁自己记。

## 为什么不用

Claude Code 不走分散 state 的路，是因为它的现实会跨子系统连动。权限模式、远端状态、任务、通知和提示文案彼此会互相改写，如果每块都自己记账，界面很快就会出现“每个人都说自己那份是真的”。

## 代价

代价是 `AppState` 会显得很重，很多字段第一次看像一份小型操作系统；但好处是整台工作台先有了一份共同现实。
