# 团队问题首先是组织问题：为什么多 agent 系统不只是并发问题

面试题版本：**"从单个 agent 扩展到多个 agent 协同，主要的技术挑战是什么？只是并发控制和资源共享的问题，还是有更根本的设计挑战？"**

## 并发和组织的区别

**并发问题**：多个 agent 同时读写同一个文件怎么办？agent A 和 agent B 都要修改同一段代码，如何避免冲突？如何分配计算资源？

这些是工程问题，有成熟的解决方案：锁、乐观并发控制、工作队列、资源调度器。

**组织问题**：谁负责什么？agent A 做完了一个子任务，结果交给谁？当 agent B 的决策需要 agent A 的输出时，这个依赖怎么表达？如果某个 agent 失败了，整个任务的责任链怎么走？

这些是结构问题，没有通用的银弹，取决于任务的具体性质和 agent 的能力边界。

## 身份和秩序是多 agent 的核心

Claude Code 的 `src/tasks/LocalAgentTask/` 和 swarm 相关代码里，每个子 agent 有明确的：

**身份**：唯一的 `AgentId`，明确的任务描述（`description`），知道自己在整个任务结构里的位置。

**秩序**：子 agent 通过 `TaskCreateTool` 创建，通过 `AgentTool` 协调，任务之间的依赖关系在创建时就需要被明确表达，而不是靠运行时的隐式假设。

**角色边界**：每个 agent 的工具权限是在创建时配置的，一个专门负责搜索的子 agent 不应该有文件写入权限，即使技术上可以给它。

这些都是组织结构的问题，不是并发控制的问题。

## 为什么任务系统是基础，而不是 swarm 本身

"任务系统解劳动闭环，swarm 解组织秩序"这个分工很关键。

任务系统（`TaskCreateTool`、`TaskStateBase`、`outputFile`、通知机制）解决的是：一个工作单元从创建到完成的完整生命周期管理，包括监控、中断、结果收集。

swarm 解决的是：多个工作单元之间的协调关系——哪些可以并行、哪些有依赖顺序、如何聚合结果、失败时如何处理。

如果没有扎实的任务系统，swarm 就没有可以协调的基本单元。每个子 agent 是一个不透明的黑盒，无法监控状态、无法收集结果、无法处理失败。Swarm 的协调逻辑就无从下手。

## 比任务世界多出的是什么

从单个 agent 到多 agent，增加的不只是工作量，而是新的**协调维度**：

**结果整合**：多个 agent 各自完成了子任务，需要把结果合并成连贯的整体。这不只是文件合并，还包括决策的一致性——如果 agent A 决定用某种方法，agent B 的决策需要与之兼容。

**进度同步**：agent A 需要等 agent B 完成某步骤才能继续。这种依赖关系不是并发控制（不是竞争同一个资源），而是业务逻辑上的先后顺序。

**错误扩散控制**：单个 agent 失败如何影响其他 agent？是立即终止所有 agent，还是让独立的子任务继续，还是在父 agent 层面重试？这些决策需要对任务结构有整体理解。

**成本和时间预算**：多个并行 agent 意味着多倍的 token 消耗。何时值得并行、何时串行更经济，这是组织层面的权衡，不是技术层面的问题。

## 面试角度的总结

多 agent 系统的主要挑战是组织问题而非并发问题，这个判断的含义是：技术并发控制可以在已有的任务系统基础上解决，但组织结构（身份、角色、依赖关系、结果整合）需要在系统设计阶段就被作为一等公民对待。Claude Code 通过把任务系统和 swarm 组织层明确分离，让每一层专注于自己的问题，而不是把所有问题混在并发控制这一个维度里处理。
