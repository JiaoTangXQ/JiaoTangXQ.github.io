# 同一个命令，不同入口，为什么能力边界不一样？

## 问题从这里开始

`/compact` 命令：在本地终端里直接执行，立刻压缩上下文，用户看到终端输出。

同样是 `/compact`，从手机通过 bridge 发过来：能执行，但走的是 `BRIDGE_SAFE_COMMANDS` 白名单里的路径，没有终端 UI。

同样是 `/model`，从手机发过来：直接被阻断，因为它是 `local-jsx` 类型，会弹出一个 Ink 终端界面，在 bridge 场景里没有渲染目标。

**命令名字一样，入口不同，能做的事情就不一样。** 这不是设计失误，而是刻意的。

## Tool 接口里的副作用声明

在更底层的工具层，同样的模式出现得更明显。`src/Tool.ts` 里的 Tool 接口不只有「输入进来、字符串出去」：

```typescript
// contextModifier is only honored for tools that aren't concurrency safe.
contextModifier?: (context: ToolUseContext) => ToolUseContext

isConcurrencySafe(input: z.infer<Input>): boolean
```

`contextModifier` 的意思是：工具执行完以后，可以改写下一轮的工具执行上下文。比如一个切换工作目录的工具，执行完之后 `cwd` 就变了，后续的文件操作都要用新路径。

`isConcurrencySafe()` 的意思是：这个工具能不能和其他工具并发执行？读文件类的工具通常是并发安全的，改文件、改权限、改状态的工具通常不是。

这两个字段说明了一个重要的事实：**工具不是纯函数，它们的结果可能改写接下来能看到什么、能做什么。**

## 同名命令不是同一能力

回到命令层。为什么 `/compact` 在本地和 bridge 下表现不同？

本地 REPL 的 `/compact` 是 `local-jsx` 类型：它会渲染一个终端界面，展示压缩进度，允许用户中途取消。这需要一个活跃的终端会话。

Bridge 里的 `/compact` 走 `BRIDGE_SAFE_COMMANDS`，是一个只输出文本结果的版本。没有 UI，没有交互，就是压缩，然后把结果文本发回去。

同样叫 `/compact`，物理机制完全不同。`isBridgeSafeCommand()` 在处理时会先检查 `cmd.type`，再决定走哪条路：

```typescript
export function isBridgeSafeCommand(cmd: Command): boolean {
  if (cmd.type === 'local-jsx') return false
  if (cmd.type === 'prompt') return true
  return BRIDGE_SAFE_COMMANDS.has(cmd)
}
```

`type` 不是分类标签，而是执行路径的声明——它决定了这个命令在被 `getMessagesForSlashCommand()` 处理时走哪个 `case`。

## 副作用抽象被边界改写

「副作用型工具不能被当纯函数」——这是 Claude Code 源码里一个反复出现的判断。

当一个工具的 `contextModifier` 改写了 `toolUseContext`，下一个工具看到的上下文已经不同了。当一个 Bash 工具失败并触发了 `siblingAbortController.abort()`，正在并发执行的其他工具也会接到中止信号。

这种副作用的传播路径，在不同入口下表现不同：

- 本地 REPL 里，Bash 工具的失败可以弹出确认框让用户决策
- Bridge 里，只能按预设行为处理，没有交互余地

所以「同一个副作用抽象」（Bash 工具）在不同入口下，实际上是不同的能力。不是能力降级，而是能力的边界在进场时就按入口类型划定了。

## 工程上的代价

这种设计的代价是显而易见的：

命令系统需要维护多套白名单（`REMOTE_SAFE_COMMANDS`、`BRIDGE_SAFE_COMMANDS`），每次新增命令都要考虑它在各个入口下的行为。

工具接口因为需要声明 `isConcurrencySafe`、`contextModifier`、`interruptBehavior`，比「纯函数」的接口重得多。

测试工具时不再只是「给一个输入、检查输出」，还要测试各种并发场景和中断场景下的副作用传播。

但这些代价换来的是：**系统的能力边界在任意时刻都是明确的，不需要在运行时推断「在这个入口下这个操作到底能不能做」**。

## 面试指导

如果被问「你们怎么处理同一功能在不同场景下的行为差异」，可以用这个角度回答：

不同场景下的行为差异，本质上是「同一抽象在不同约束条件下的不同投影」。纯函数化让这种差异消失在隐式假设里，显式声明副作用则让差异变成可测试的接口。

Claude Code 选择了后者：每个工具和命令都声明自己的副作用边界，入口层按这些声明决定路由，而不是在运行时靠条件判断修补行为差异。
