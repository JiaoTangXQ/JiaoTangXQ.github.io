# 刷新重连和资源订阅说明它是常驻能力层

## 实现链
- `client.ts` 不只建立一次连接，它还要处理 session 过期后的重连、工具和资源列表的重新拉取，以及连上后的一些附带通知。能力表不是写死在第一次握手里，而是会跟着连接生命一起更新。
- 换句话说，MCP 在 Claude Code 里更像常驻能力层：连上后还要继续维护“我现在能提供什么”，而不是握手完就万事大吉。

## 普通做法
- 更普通的办法是连接一次、拉一次列表，然后整个会话期间都假定它不会变。
- 很多简单集成确实这么做，因为服务器能力往往被当成静态配置。

## 为什么不用
- Claude Code 不敢这么假设，因为 MCP server 可能因为认证、外部世界变化或 server 自己的状态改变而刷新能力。
- 如果客户端不重连、不重算，用户看到的就是一份已经过期的能力清单，后面再好的 UI 也只能展示假真相。

## 代价
- 代价是每次状态切换都要牵动缓存和刷新逻辑，调试难度比“只连一次”的客户端高很多。
- 但这也是把 MCP 当正式能力层的必然成本。
