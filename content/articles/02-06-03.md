# 为什么非交互会话会绕开这整套界面化确认

看到这里，很多人会下意识地问一句: 既然 trust 这么重要，为什么 `--print` 这类非交互会话反而会绕开整套界面确认？答案不是“这里安全要求变低了”，而是 Claude Code 明确换了一套信任假设。

交互模式面对的是一个坐在终端前、可以逐步确认风险的人，所以系统可以把 trust dialog、外部引用警告、权限提示这些东西一层层摆给你看。非交互模式面对的却往往是脚本、管道、CI 或别的程序调用，它没有空间停下来弹一串界面，也没有意义假装用户会在半路逐个点确认。于是 Claude Code 干脆把话说直: 这条路径默认把执行环境视为已信任，但帮助文本会明确提醒你，只该在自己信任的目录里使用。

源码里这个态度非常一致。非交互路径不会进入 `showSetupScreens()`；有些只有在 trust 之后才会做的动作，在这里会直接执行，比如应用完整环境变量、预取 system context。也就是说，它不是偷偷跳过安全，而是把“信任”从界面确认改成了调用前提。这样的设计很诚实，也很成熟: 自动化世界要的不是假装谨慎的弹窗，而是清楚、稳定、可预期的制度边界。

## 实现链
这页对应的实现链都属于 trust 边界：先判断当前会话有没有资格进入交互世界，再决定是展示哪一层确认、用哪种方式表达风险，还是在非交互路径里直接走另一套边界处理。

## 普通做法
更普通的做法是把风险确认当作 onboarding 的一部分，或者把同一套界面确认硬套给所有会话形态。

## 为什么不用那种做法
Claude Code 不这么做，因为它把 trust 看成运行时边界，而不是欢迎页。对交互和非交互会话来说，边界必须真实，不能只求界面统一。

## 代价与回报
回报是用户进入的是被认真确认过的能力世界。代价是首屏前会多出一层制度感很强的流程。
