# 为什么 -p 模式跳过 trust dialog 不是安全漏洞？

## 表面上的疑问

```typescript
export async function showSetupScreens(...): Promise<void> {
  if (getIsNonInteractiveSession()) {
    return;  // 直接返回，跳过一切，包括 trust dialog
  }
  // ...trust dialog...
}
```

如果 trust dialog 是"工作区安全边界"，为什么非交互模式可以直接跳过？

这不是在说"脚本不需要安全"——Claude Code 在非交互模式下依然会执行代码、读写文件、调用 API。如果 trust dialog 真的在保护什么，跳过它不应该造成安全问题吗？

## 关键区分：信任的来源不同

**交互模式**的信任来源：用户坐在终端前，看到对话框，主动点击"我理解并接受"。用户的实时确认是信任的来源。

**非交互模式**的信任来源：调用者——运行脚本的人或系统——对执行环境负责。

Claude Code 的帮助文本（`-p`/`--print` 的描述）明确说明：

```
-p, --print            Print response to stdout without interactive mode.
                       Only use in directories you trust.
```

"Only use in directories you trust"——这是把信任的职责转移给了调用方。这不是弱化安全，而是换了一种安全模型。

## 为什么这种转移是合理的？

想象 CI/CD pipeline 调用 Claude Code：

```bash
# .github/workflows/review.yml
- name: Code Review
  run: claude -p "review the changes in this PR" --output-format json
```

这个脚本由谁触发？由代码库的维护者触发，在受保护的 CI 环境里运行，对目录内容负责的是仓库的 owner 和 CI 系统的 admin。

如果在这里弹出 trust dialog：
1. CI 没有终端，无法显示对话框
2. 即使能显示，也没有人能实时点击
3. 脚本会永远挂在等待用户输入的状态

非交互模式的调用前提就是"没有真人实时监控"，trust dialog 的设计前提是"有真人实时确认"。两者从设计上就是互斥的。

## 非交互模式下仍然存在的安全机制

跳过 trust dialog 不等于无防护：

**1. 完整环境变量仍然被应用（但以"已信任"为前提）**

```typescript
// 非交互模式不走 showSetupScreens()
// 但在启动早期，initializeTelemetryAfterTrust() 会直接被调用
// 完整环境变量在 trust 确认后应用
```

非交互模式相当于"自动通过 trust 确认"，后续的能力解锁仍然完整执行。

**2. permission mode 仍然生效**

非交互模式里，操作级别的权限控制（auto mode、default mode）仍然有效。`--dangerously-skip-permissions` 在非交互模式里同样需要显式传入，不会自动启用。

**3. MCP 服务器审批**

非交互模式下，如果 `.mcp.json` 里有新的未审批服务器，行为取决于配置（可能拒绝连接，或者只连接已审批的服务器）。这不会因为跳过 trust dialog 就自动放行。

## 这是"隐式信任"还是"信任前提"？

一个有意义的语义区分：

**隐式信任**：系统不知道是否应该信任，但懒得询问，就当信任处理。这是危险的设计。

**信任前提**：系统明确知道某些调用路径需要调用者提前保证信任。这是一种合理的责任分配。

非交互模式属于后者。帮助文本明确说明了这个前提，API 文档也应该记录这个约束。信任的责任被明确转移了，而不是被悄悄忽略了。

## 面试指导

"绕过安全检查是不是漏洞"这类问题，答案通常是"要看安全模型是什么"。

安全检查的目的不一定是"绝对防止危险操作"，而可能是"确保操作是在正确的知情状态下发起的"。

Trust dialog 的目的是确保**有交互能力的用户**在充分知情的情况下确认信任。对于**没有交互能力的自动化调用**，这个检查本来就不适用——它的目标用户（实时确认的人类）根本就不在场。

能把这个分析说清楚，是对安全设计有深度理解的体现：安全不是对所有路径应用相同的最高保护，而是对每条路径应用适合该路径的正确保护机制。
