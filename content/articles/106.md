# Task.ts 为什么要把后台任务做成可治理对象，而不是 Promise？

"起一个异步任务"这件事，最简单的实现是：`const promise = doSomethingAsync(); spinner.show()`。

Claude Code 没有这么做，原因在 `Task.ts` 的类型定义里说得很清楚。

## TaskStateBase 里每个字段都在回答一个问题

```typescript
export type TaskStateBase = {
  id: string           // 谁在跑？
  type: TaskType       // 是什么类型的工作？
  status: TaskStatus   // 现在处于哪个生命周期阶段？
  description: string  // 这份工作是什么？
  toolUseId?: string   // 它是从哪个工具调用触发的？
  startTime: number    // 什么时候开始的？
  endTime?: number     // 什么时候结束的？
  totalPausedMs?: number  // 累计暂停了多久？
  outputFile: string   // 输出在哪里？
  outputOffset: number // 当前读到哪里了？
  notified: boolean    // 完成通知是否已经发出？
}
```

对比一下 Promise 版本能告诉你什么：你只知道"有一个异步操作在跑，结束了会 resolve 或 reject"。你不知道它叫什么，什么时候开始的，进行到哪里了，结果在哪里，有没有通知过任何人。

`TaskStateBase` 把这些问题每一个都显式地问了，并且要求在状态对象里维护答案。

## TaskType 和 TaskStatus 构成"小宪法"

```typescript
export type TaskType =
  | 'local_bash'
  | 'local_agent'
  | 'remote_agent'
  | 'in_process_teammate'
  | 'local_workflow'
  | 'monitor_mcp'
  | 'dream'

export type TaskStatus =
  | 'pending'
  | 'running'
  | 'completed'
  | 'failed'
  | 'killed'
```

这两个 union type 是系统对"任务是什么、任务生命周期有哪些阶段"的完整声明。任何新类型的任务，都必须往 `TaskType` 里加；任何新的终止状态，都必须往 `TaskStatus` 里加。

`isTerminalTaskStatus` 函数是对这个宪法的执行：

```typescript
export function isTerminalTaskStatus(status: TaskStatus): boolean {
  return status === 'completed' || status === 'failed' || status === 'killed'
}
```

只有三种终态，它们在代码里明确列举，而不是靠各个模块各自猜测"这个任务算不算结束了"。

## Task ID 的设计意图

```typescript
const TASK_ID_PREFIXES: Record<string, string> = {
  local_bash: 'b',
  local_agent: 'a',
  remote_agent: 'r',
  in_process_teammate: 't',
  local_workflow: 'w',
  monitor_mcp: 'm',
  dream: 'd',
}
```

Task ID 有类型前缀，不是纯随机 UUID。这让你在日志、调试输出、错误信息里看到任何 task ID，立刻就知道它是哪种类型的任务，不需要额外查询。

后面接的是 8 位 base-36 随机数（"36^8 ≈ 2.8 trillion combinations, sufficient to resist brute-force symlink attacks"——这条注释说明 outputFile 路径用 task ID 构建，需要防止路径预测攻击）。

## outputFile 让过程痕迹可接手

`outputFile: string` 和 `outputOffset: number` 这两个字段，是让后台任务"可续读"的关键机制。

任务在运行过程中把输出写入 `outputFile`，`outputOffset` 跟踪当前消费到哪里了。当任务被切换到后台，前台不再关注它时，新的进度依然写入文件。当用户切换回来要看这个任务的进度，系统从 `outputOffset` 开始读取，而不是丢失这期间的所有输出。

这等于给每个后台任务都开了一个"外部痕迹"——任务在运行，痕迹在累积，不管前台是否在关注。

## generateTaskId 的对称性

```typescript
export function generateTaskId(type: TaskType): string {
  const prefix = getTaskIdPrefix(type)
  const bytes = randomBytes(8)
  let id = prefix
  for (const byte of bytes) {
    id += TASK_ID_ALPHABET[byte % TASK_ID_ALPHABET.length]
  }
  return id
}
```

使用 `randomBytes`（密码学安全随机）而不是 `Math.random`，原因在注释里已经说明了：task ID 用来构建 outputFile 路径，如果可被预测，攻击者可以预先创建同名文件实现路径劫持。这是安全设计，不是过度工程。

## 面试答题要点

被问到"你们的后台任务是怎么管理的"，这个问题其实在问：

- 你们的任务有没有标准化的生命周期状态？
- 任务的输出怎么做到可追溯？
- 任务完成时怎么通知主会话？
- 前后台切换时，任务状态怎么保持一致？

能说出"任务是以状态对象存在于 AppState 里，而不是作为独立的 Promise 存在"，说明理解了核心设计。任务对象的生命周期由 `status` 字段管理，所有模块在做涉及任务的判断时，都通过这个字段读取，不会各自维护独立的状态。

追问"任务完成了，何时从 AppState 里清理"：这由 `evictAfter`（LocalAgentTaskState 里有这个字段）控制，任务进入终态后不是立刻从状态树里删除，而是保留一段时间让用户能看到刚完成的任务。这是"体面退场"的机制。
