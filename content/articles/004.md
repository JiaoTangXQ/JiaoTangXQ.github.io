# ToolPermissionContext 里有哪些字段，每个字段管什么？

## 类型分析：逐字段解读

在 `Tool.ts` 里，`ToolPermissionContext` 被定义为一个用 `DeepImmutable` 包裹的对象类型。这个 `DeepImmutable` 包装说明了一件事：这个结构是不可变的——它只能被整体替换，不能被局部修改。这是"权限状态是时刻点快照"这个设计决策的直接体现。

```typescript
export type ToolPermissionContext = DeepImmutable<{
  mode: PermissionMode
  additionalWorkingDirectories: Map<string, AdditionalWorkingDirectory>
  alwaysAllowRules: ToolPermissionRulesBySource
  alwaysDenyRules: ToolPermissionRulesBySource
  alwaysAskRules: ToolPermissionRulesBySource
  isBypassPermissionsModeAvailable: boolean
  isAutoModeAvailable?: boolean
  strippedDangerousRules?: ToolPermissionRulesBySource
  shouldAvoidPermissionPrompts?: boolean
  awaitAutomatedChecksBeforeDialog?: boolean
  prePlanMode?: PermissionMode
}>
```

逐个拆解：

**`mode: PermissionMode`**
当前的安全姿态。可取值来自 `PermissionMode.ts`：`default`、`plan`、`acceptEdits`、`bypassPermissions`、`dontAsk`，以及内部的 `auto`（仅 Anthropic 员工版本有效）。这个字段决定了"拦截门槛在哪里"。

**`additionalWorkingDirectories`**
允许工具操作的额外目录集合。不只有主 CWD，还有用户通过配置追加的目录。工具在请求文件操作权限时，会检查目标路径是否落在这些目录内。

**`alwaysAllowRules / alwaysDenyRules / alwaysAskRules`**
三套规则集，类型是 `ToolPermissionRulesBySource`——意味着规则本身也有来源（用户规则、项目规则、企业策略规则）。三套规则不互斥，优先级由外部逻辑处理。`alwaysDenyRules` 的优先级高于 `alwaysAllowRules`，安全边不能被允许规则绕过。

**`isBypassPermissionsModeAvailable`**
布尔值。`bypassPermissions` 模式（完全跳过所有权限检查）是否对当前用户可用。这个能力不是默认开放的，需要特定条件才能解锁，比如信任的工作区或 CLI flag。

**`shouldAvoidPermissionPrompts`**
当为 `true` 时，所有权限请求直接自动拒绝，不弹 UI。注释里说明了场景：后台无 UI 的 agent。如果一个子 agent 在后台跑，没有终端可以展示权限确认框，那么遇到需要确认的操作就应该直接拒绝，而不是永远等待一个永远不会来的用户输入。

**`awaitAutomatedChecksBeforeDialog`**
当为 `true` 时，在弹出人工确认框之前，先等 classifier 等自动检查完成。这是给 swarm coordinator worker 场景设计的——worker 的权限请求先经过自动化分类器，如果能自动批准就不打扰用户。

**`prePlanMode`**
进入 Plan Mode 之前的权限模式备份。Plan Mode 是"只规划不执行"的安全状态，退出时需要恢复到之前的模式。

## 工厂函数和默认值

```typescript
export const getEmptyToolPermissionContext: () => ToolPermissionContext = () => ({
  mode: 'default',
  additionalWorkingDirectories: new Map(),
  alwaysAllowRules: {},
  alwaysDenyRules: {},
  alwaysAskRules: {},
  isBypassPermissionsModeAvailable: false,
})
```

默认值是最严格的起点：`default` 模式，没有额外目录，三套规则为空，bypass 能力关闭。新会话从这个最小权限状态开始，再通过配置和用户交互逐步展开。

## 为什么不用扁平的几个 boolean？

这个类型的设计暗示了一个重要判断：权限不是一个全局开关，而是多维的实时状态。它要同时建模"我在什么模式下"、"哪些目录允许访问"、"哪些工具有规则豁免"、"现在能不能弹 UI"等多个相互独立的维度。

把所有维度压成一个对象，然后用 `DeepImmutable` 确保这个快照的一致性，是这套设计最关键的工程决策。

## 面试指导

被问权限系统时，不要只说"有 allowlist 和 denylist"。说清楚五个维度：mode（全局安全姿态）、directory scope（目录边界）、rule sets（三套来源感知规则）、UI availability（是否能弹框）、mode transitions（Plan Mode 进出的状态备份）。`DeepImmutable` 包装是点睛之笔——它说明这个对象在传递时不会被意外修改，权限状态的一致性由类型系统保证，而不只是靠约定。
