# 面试题：函数返回 updater 而不是直接调用 setState，这个设计有什么好处？

通常的状态管理是单向的：本地状态变化 → 推送到远端。

但 Claude Code 需要处理一个更复杂的场景：远端会话元数据也可以是真相来源，本地状态需要服从远端的某些变化。

```typescript
export function externalMetadataToAppState(
  metadata: SessionExternalMetadata,
): (prev: AppState) => AppState {
  return prev => ({
    ...prev,
    ...(typeof metadata.permission_mode === 'string'
      ? {
          toolPermissionContext: {
            ...prev.toolPermissionContext,
            mode: permissionModeFromString(metadata.permission_mode),
          },
        }
      : {}),
    ...(typeof metadata.is_ultraplan_mode === 'boolean'
      ? { isUltraplanMode: metadata.is_ultraplan_mode }
      : {}),
  })
}
```

这个函数接受外部元数据，返回一个 `AppState` 更新函数（`(prev: AppState) => AppState` 的形式正好符合 `store.setState` 的参数）。

---

## 什么是 SessionExternalMetadata

从 `onChangeAppState` 里可以看到另一个方向的同步：

```typescript
notifySessionMetadataChanged({
  permission_mode: newExternal,
  is_ultraplan_mode: isUltraplan,
})
```

这里把本地的权限模式「外部化」（通过 `toExternalPermissionMode` 映射到外部的名称），然后推送给 CCR（远端协调器）。

`SessionExternalMetadata` 就是这个推送的数据结构：

```typescript
type SessionExternalMetadata = {
  permission_mode?: string  // 外部化的权限模式名称
  is_ultraplan_mode?: boolean
  // 可能还有其他字段
}
```

这是一个「翻译边界」——本地用内部模式名（可能是 `bubble`、`ungated_auto` 等），外部只暴露公开的模式名（`default`、`plan`、`auto`）。

---

## 双向同步的流向

```
本地 AppState
     ↓ onChangeAppState（出去方向）
     ↓ toExternalPermissionMode（翻译：内部 → 外部）
     ↓ notifySessionMetadataChanged
     ↓
  CCR / 远端
     ↓
     ↓ 外部元数据变化
     ↓
externalMetadataToAppState（回来方向）
     ↓ permissionModeFromString（翻译：外部 → 内部）
     ↓ store.setState
     ↓
本地 AppState
```

两个方向都有翻译层：
- 出去时：`toExternalPermissionMode` 把内部模式映射为外部模式
- 回来时：`permissionModeFromString` 把外部模式字符串解析回内部类型

翻译层的存在说明两个系统对权限模式有不同的表示：本地有更细粒度的内部模式，外部只有简化的公开模式。

---

## 函数返回 updater 而不是直接更新

```typescript
export function externalMetadataToAppState(
  metadata: SessionExternalMetadata,
): (prev: AppState) => AppState {
  return prev => ({ ... })
}
```

这个函数不接受 `store`，不直接调用 `store.setState`。它返回一个 `(prev: AppState) => AppState` 类型的函数。

这是一个有意的设计选择，让这个函数变成可组合的：

```typescript
// 调用方可以这样用
store.setState(externalMetadataToAppState(metadata))

// 或者和其他更新组合
store.setState(prev =>
  externalMetadataToAppState(metadata)(
    someOtherUpdate(prev)
  )
)
```

返回 updater 函数而不是直接更新，让这个函数更纯粹（不依赖 store 实例），也更容易测试（只需要传入测试数据，检查返回的新状态）。

---

## 可选字段的处理

```typescript
...(typeof metadata.permission_mode === 'string'
  ? { toolPermissionContext: { ... } }
  : {})
```

外部元数据的字段是可选的，不一定每次都包含 `permission_mode`。这个条件展开（conditional spread）确保：只有字段真的存在时才更新对应的状态，不存在时不改变当前值。

这比 `if (metadata.permission_mode)` 更严格：`typeof x === 'string'` 明确检查类型，避免空字符串（`''`）被当作「字段不存在」处理。

---

## 不是所有远端变化都能回灌

注意 `externalMetadataToAppState` 只处理了两个字段：`permission_mode` 和 `is_ultraplan_mode`。

不是所有远端的状态变化都会回灌本地。比如，远端 viewer 的「哪个消息被展开了」不会回灌，因为那是远端 viewer 的本地 UI 状态，和本地工作台无关。

这个函数处理的是「对本地运行有实质影响」的元数据：权限模式直接影响工具执行，ultraplan 标记影响特定功能的触发条件。其他的展示性信息不需要双向同步。

---

## 面试指导

**这个模式叫什么名字？**

这是 CQRS（Command Query Responsibility Segregation）的变体，但更接近「事件驱动的状态同步」：远端的状态变化被视为一个「命令」（修改本地权限模式），通过一个纯函数翻译为本地状态更新。

**双向同步的主要风险是什么？**

**竞态条件**：本地修改了权限模式，推送给远端；远端同时也修改了权限模式，推送回来。两个变化到达的顺序不确定，可能导致状态翻转。

Claude Code 的处理方式是「最后写入者胜出」（last-write-wins）：不管哪个变化先到，最后一次 setState 的结果就是最终状态。对权限模式来说，这个策略合理——如果用户在本地刚切到 `plan` 模式，远端推来了一个 `default` 模式，这个竞态说明某处协调逻辑有问题，让最后一个写入者胜出是合理的降级策略。

**如何测试 externalMetadataToAppState？**

```typescript
test('外部 plan 模式回灌本地', () => {
  const initialState = getDefaultAppState()
  const updater = externalMetadataToAppState({ permission_mode: 'plan' })
  const newState = updater(initialState)
  expect(newState.toolPermissionContext.mode).toBe('plan')
})
```

因为函数是纯的（返回 updater 而不是直接操作 store），测试非常简单。这是「返回 updater」而不是「直接更新 store」的测试收益。
