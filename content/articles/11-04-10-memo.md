# useMemo 不是性能优化，是在保住消息的身份稳定性

`PromptInputQueuedCommands.tsx` 里有一段 `useMemo`：

```typescript
const previewMessages = useMemo(() => {
  return queuedCommands.map(cmd => createUserMessage(cmd.text));
}, [queuedCommands]);
```

初看是常规的性能 memo。但原因其实更深。

## createUserMessage 的副作用

`createUserMessage()` 不是纯转换函数。它在生成消息时会调用 `uuid()`，每次调用都产生一个全新的唯一 ID。

这意味着：哪怕排队命令的文本完全没有变化，只要 `queuedCommands` 引用更新触发了重新生成，每个预览消息就会得到全新的 UUID，从 React 的角度看变成了「新消息」。

## UUID 变化带来的视觉后果

React 用 key 来追踪列表元素的身份。当一个列表项的 key 变了（哪怕内容相同），React 会认为旧节点消失、新节点出现，触发完整的 unmount → mount 周期。

对于排队预览区域，这意味着：消息视觉上会「闪」一下，动画状态重置，滚动位置可能跳动。用户看到的效果是：明明什么都没变，但排队区在不停地抖动、重建，像是在反复「出现」同样的内容。

## memo 的真正作用

加上 `useMemo` 之后，只要 `queuedCommands` 数组引用没变，`previewMessages` 就不会重新计算，UUID 也就保持稳定。

这不是在省计算量——`createUserMessage` 本身很轻。这是在保住身份稳定性：让「同一条排队消息」从 React 的角度看也是「同一条排队消息」，而不是每次渲染都变成陌生的新节点。

流式界面里，身份稳定和内容正确同等重要。内容不变但身份每帧都变，用户感受到的是一个不断在「重启」的界面，而不是一个平稳显示状态的驾驶舱。
