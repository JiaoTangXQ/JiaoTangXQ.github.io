# 排队预览要 memo 住，说明流式输入区最怕身份抖动

排队区里的预览消息不是每次重绘都重新现造，而是被刻意 memo 住。原因并不花哨，只是因为这些消息一旦反复重建，就会带着新的身份出现，随后引发闪烁、抖动和阅读不稳。系统在这里优先保的不是性能数字，而是人眼前“还是同一条东西”的连续感。

这正是流式界面最见功力的地方。真正伤人的往往不是大错误，而是界面一会儿像同一条消息，一会儿又像另一条消息。Claude Code 很清楚，驾驶舱最怕身份抖动，所以先把口音稳住，再谈实时刷新。

## 实现链

`PromptInputQueuedCommands.tsx` 用 `useMemo()` 包住消息列表生成，因为里面的 `createUserMessage()` 每次都会生成新 UUID。要是不 memo，哪怕内容没变，排队预览也会因为身份抖动被当成全新消息重建。

## 普通做法

更随手的实现会在每次 render 时直接重新构造预览项，反正 React 会自己 diff。

## 为什么不用

Claude Code 不敢这么随手，是因为这里的预览项不是静态字符串，消息身份会影响动画、渲染稳定性和列表抖动。对流式输入区来说，身份稳定和内容正确一样重要。

## 代价

代价是要额外维护 memo 依赖和构造路径；但换来的是底部排队区不会因为无意义重建而抖来抖去。
