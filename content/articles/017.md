# Claude Code 上下文连续性地图

## 先说结论：连续性不是“别忘太多”，而是分层保存工作能力

Claude Code 没把上下文管理理解成“token 不够时压一把”。代码里能看到的真实态度是：长会话迟早会溢出，所以系统要提前决定什么留在眼前、什么局部减重、什么外置成工作记忆、什么沉成长期记忆。

## 第一层：`microcompact` 先做轻量减重

### 实现链

`query.ts` 会先跑 `microcompact` 这一层，把某些更容易压缩的结果先局部处理，而不是一上来就做重型总结。

### 普通做法

更普通的做法是：上下文一长就直接整体总结，或者索性截断。

### 为什么不用

Claude Code 想先保住细节，只对最容易压薄的部分下手。这样可以推迟大手术，避免会话过早失去颗粒度。

### 代价

连续性链变得更长，调试时也更难区分是哪一层在起作用；但历史保真度更高。

## 第二层：`autocompact` 是预算驱动的重型压缩

### 实现链

`autocompact` 会结合模型上下文窗口、warning/error/blocking 阈值和当前消息体量来判断是否触发自动压缩。

### 普通做法

更直觉的实现是到某个长度就机械压缩。

### 为什么不用

Claude Code 不把压缩只看成“长度到了”。它还要考虑别的机制是否正在介入，比如 context collapse、reactive compact、某些 querySource 特殊情况。也就是说，它在问“现在压缩是不是当前最合适的动作”。

### 代价

预算逻辑和例外分支会更多；但这让系统不至于在错误时机做错误压缩。

## 第三层：手动 `/compact` 不是自动压缩的备用开关

### 实现链

系统同时保留手动 compact 路径，让用户显式触发整理。

### 普通做法

很多产品要么只保留手动整理，要么只保留自动整理。

### 为什么不用

Claude Code 区分了两种权力：系统为了自救自动整理，和用户为了接下来更好工作主动整理。两者不是同一回事。

### 代价

系统存在多种整理入口，概念上更复杂；但用户主导权也被保留了下来。

## 第四层：`session memory` 把当前工作状态外置出来

### 实现链

`SessionMemory` 相关逻辑会维护一份关于当前会话工作的外置笔记，并不把所有连续性都强塞回主上下文。

### 普通做法

更直觉的设计是：上下文都留在会话里，压缩时顺便总结一下。

### 为什么不用

Claude Code 发现“当前工作状态”这种东西，既重要，又不适合一直占着主 prompt。外置成 session memory，可以让主会话继续前进，又不至于完全忘掉当前在做什么。

### 代价

系统多了一层外置记忆文件和同步逻辑；但会话更稳。

## 第五层：`auto-memory` 处理的是长期沉淀，不是眼前工作状态

### 实现链

`auto-memory` 会抽取值得长期保留的知识，写到更长期的记忆位置，而不是继续维护当前会话工作流。

### 普通做法

很多系统只有一种“记忆”，把当前状态和长期知识混在一起。

### 为什么不用

Claude Code 把“现在这段工作记住什么”和“以后长期都该记住什么”拆开，是因为这两类记忆的时间尺度和错误代价完全不同。

### 代价

记忆体系会多层化，理解成本更高；但更符合真实使用。

## 第六层：回合后的 hooks 说明整理也要守时间制度

### 实现链

stop hooks、post-sampling hooks 和相关整理逻辑会把很多后处理动作放到“这轮已经结束之后”去做。

### 普通做法

更简单的做法，是在主采样过程中顺手做完所有整理。

### 为什么不用

Claude Code 不想让整理动作打断主输出节奏。它把很多后处理挪到回合结束后，是在维护“先完成当前工作，再整理后事”的时间制度。

### 代价

连续性管理跨越主循环前后两个时段，理解路径更长；但主回合更干净。
