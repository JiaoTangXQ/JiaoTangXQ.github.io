# 被系统接受的一条输入会带着哪些额外世界一起进入

普通人看到的输入，通常只有一句文字。Claude Code 接受的一条输入，却常常背着一整个小世界一起进来。除了文字本身，它还可能带着前置内容块、粘贴图片、图片元信息、附件消息、agent 提及、工具许可变化、hooks 产出的附加上下文，甚至还会带着“这条输入是不是隐藏给人看、但仍然要给模型看”的制度标签。

这件事很重要，因为它说明会话不是靠一条字符串长大的，而是靠一组被组织好的消息共同长大的。源码里普通文本会进入 `processTextPrompt()`，但在那之前，系统已经可能替它拼上图片块、附件世界、hook 附件和一条 `isMeta` 的图片说明。也就是说，真正被接受进来的从来不是一句孤零零的话，而是一组已经被整理好的会话材料。

这正是 Claude Code 和很多“把输入框内容直接丢给模型”的程序差别所在。它承认一轮输入不是文本本身，而是文本连同周边世界一起进入。模型真正接住的，也因此不是“用户敲了什么”，而是“系统已经确认这轮对话现在拥有了什么”。

## 实现链
被接受的不只是文本。`processUserInput()` 还会把 `attachmentMessages`、图片元信息对应的 `isMeta` 消息、hook 附加上下文、bridge 安全条件、以及可能的 `allowedTools / model / effort` 一起打包到结果里。

## 普通做法
更普通的聊天入口只保留“用户说了什么”，顶多再附一份附件文本摘要。

## 为什么不用
因为后面主循环需要知道的不只是原句，还要知道这轮有哪些结构化资源、哪些内容只给模型看、哪些命令改写了工具权限和模型选择。

## 代价
消息体系会变得更杂，特别是新手第一次看到 `isMeta` 或 attachment message 会觉得绕；但不把这些世界一起带进去，后面的推理现场就会一直缺上下文。
