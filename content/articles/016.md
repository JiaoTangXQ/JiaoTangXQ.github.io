# Claude Code 结构决策记录

这份附录现在不再是开放问题清单，而是解释这本书为什么按现在这套结构收口。

## 实现链

全书最终采用的是“按运行时主线和设计问题下钻”，而不是按源码目录平铺。原因很直接：核心代码真实上是跨目录协作的。启动链要跨 `main.tsx`、命令装配和设置；查询链要跨 `QueryEngine.ts`、`query.ts`、压缩和工具执行；任务与团队又要跨 `Task.ts`、`AgentTool`、swarm、mailbox 和 UI。

所以卷和章的边界最终按运行时职责来划，而不是按文件夹名来划。

## 普通做法

更普通的源码书写法，是按目录切卷：工具卷、UI 卷、utils 卷、services 卷。或者按用户旅程切：打开程序、输入问题、看到结果。

## 为什么不用

Claude Code 没这么分，是因为两种直觉切法都不够。按目录切，会把跨目录设计掰断；按用户旅程切，又容易把同一机制散在很多地方。最终正文采用的是混合策略：先沿运行时主线保证不迷路，再在章级拆设计问题。

这也是为什么任务卷和团队卷分开、治理卷不再硬拆、UI 卷按结构机制而不是按旅程分层。它不是在追求最整齐的目录，而是在追求最接近代码真实工作的骨架。

## 代价

代价是整本书不会像“按目录讲源码”那样特别规整。很多模块会在不同卷里反复出现，因为它们本来就在不同运行链里扮演不同角色。但这比做一套看起来整齐、实际失真的分卷更合理。
