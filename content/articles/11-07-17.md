# 消息源头识别不能只看最后一个块，说明拆块后也要认出同一轮

一条消息进了 UI 之后, 很可能会被拆成多个块、多个行, 甚至前后夹进 divider 和折叠组。Claude Code 在这种情况下仍然坚持“同一轮还是同一轮”, 说明它认的是源头身份, 不是屏幕上最后露出来的那个块。

## 实现链

`Messages.tsx` 里 `dividerBeforeIndex` 的计算不是盯着某个最后块, 而是拿 `unseenDivider.firstUnseenUuid` 的前 24 个字符去和 `renderableMessages` 里的 `uuid` 前缀比对。注释直接写了原因: `deriveUUID()` 会保留源消息 uuid 的前 24 位, 所以同一条消息就算被拆成多个 block, 仍能靠这段前缀认回同一个源头。这样 unseen divider、展开状态和后续定位都不会因为拆块而认错人。

## 普通做法

更容易想到的做法是只盯着当前最后一个 block 或当前数组位置, 谁现在排在那儿就认谁。

## 为什么不用

Claude Code 不这么做, 因为消息一旦经过 `normalizeMessages()`、重排、分组和折叠, 屏幕上的最后一个块不一定代表真正的源头。要是只认最后块, 同一轮被拆开以后就会丢身份, divider 会插错, 展开组会断, 搜索和点击锚点也会飘。

## 代价

代价是 UI 层要依赖一套“派生 uuid 仍保留源头前缀”的约定。这个约定很有效, 但也让消息身份不再是完全显眼的直觉逻辑, 新读者得先理解 `deriveUUID()` 和前缀匹配这层隐含契约。
