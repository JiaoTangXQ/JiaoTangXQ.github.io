# compact boundary：整理不是抹去历史，而是承认它被整理过

面试题版本：**"Claude Code 压缩会话历史后，如何让后续的模型推理知道历史被压缩过？compact boundary 具体存储了什么信息，起到了什么作用？**

## compact boundary 是什么

`src/utils/messages.ts` 里的 `createMicrocompactBoundaryMessage()` 创建一种特殊类型的消息，作为压缩历史和当前对话之间的边界标记。

这个 boundary 消息包含：
- 压缩发生的时间戳
- 被压缩的消息范围（从第几条到第几条）
- 摘要的 token 数量（帮助评估压缩效果）
- 压缩摘要本身（或对摘要的引用）

`getMessagesAfterCompactBoundary()` 函数在需要时找到最近的 compact boundary，返回 boundary 之后的"新鲜历史"。

## 为什么需要显式承认

一种简单的压缩方式：把旧消息删掉，换成摘要，假装没发生过。模型只看到一段历史，不知道这段历史里有些是原始的，有些是摘要的。

这种"静默压缩"的问题：

**模型可能产生错误的信心**：模型看到的对话看起来很连贯，但它不知道中间有大量内容被压缩了。如果用户问"你还记得我们之前讨论的细节吗？"，模型可能过于自信地回应，即使那些细节已经被压缩掉了。

**多次压缩后边界丢失**：第一次压缩后，如果再次需要压缩，系统不知道历史里哪些部分已经是摘要了，可能会重复压缩（把摘要再次摘要），信息损失加剧。

**恢复困难**：如果会话需要从某个历史点恢复，没有 boundary 标记就不知道哪里是安全的恢复点，哪些消息是有效的原始历史。

## compact boundary 的认识论意义

"整理后先承认历史被整理过"这句话有具体的工程含义：

**承认给模型**：compact boundary 出现在消息历史里，模型可以看到它，知道"boundary 之前的内容是摘要，boundary 之后是原始对话"。这让模型在回答问题时可以有正确的认识——对摘要部分谦虚，对原始部分有信心。

**承认给系统**：系统代码可以通过查找 compact boundary 来决定：
- 哪些消息应该被包含在下一次 API 调用里（boundary 之后的）
- 哪些消息是安全的恢复起点
- 压缩了多少次、每次压缩的范围是什么

**承认给用户**：如果用户查看对话历史，compact boundary 是一个可见的分隔，告知"这里发生了一次上下文整理"。

## 与事务日志的类比

数据库的事务日志记录每次数据变更，不只是最终状态。这让数据库可以：
- 回滚到任意历史点
- 确定某条记录在某个时间点的状态
- 理解数据"为什么"是当前这个状态

compact boundary 有类似的作用：它不是数据本身（历史消息），而是关于历史的元数据——"历史在这里发生了什么改变"。这种"关于历史的历史"让系统可以正确理解当前状态的来源。

## 新的接缝，而不是遮掩

"compact boundary 把压缩变成新的历史接缝"这个表述说明了压缩的正确心态：不是把过去抹掉，而是把过去收敛成一个新的起点。

压缩前的对话 + compact boundary + 压缩后的新对话，共同构成一条可追溯的历史线。任何时刻，你都知道系统处于哪个历史阶段，这个阶段是从什么地方开始的，之前有没有压缩过。

这和"假装历史从来没那么长"完全不同——它是诚实地记录了历史被整理的事实，让这个整理本身成为可被理解、可被利用的信息。

## 面试角度的总结

compact boundary 的核心价值是**让压缩这个动作本身成为可见的历史事件**，而不是悄悄发生的信息删减。显式的 boundary 标记让模型有正确的认识（知道什么是摘要，什么是原始），让系统有明确的操作点（从哪里开始是新鲜历史），让用户有可理解的历史视图（这里曾经发生过一次整理）。这种"承认整理"的设计，比"假装没整理"更诚实，也更可维护。
