# 第十三卷回收索引：前四卷里的 `tool_result` 与 `transition`

## 实现链
前四卷里还有一条线，专门回答“动作做完以后，为什么还得有结案和继续理由”。

这页现在只保留入口判断：`tool_result` 先结案，`transition` 再把继续理由制度化。

这页现在作为父级入口，继续分出去的实现分支是：
### `tool_result` 让每次动作都正式结案

### `transition` 把继续理由制度化

- [252-第十三卷回收索引-前四卷-transition把继续理由制度化.md](/Users/xxx/tmp/cc/docs/claude-code-book/appendices/252-第十三卷回收索引-前四卷-transition把继续理由制度化.md)

它不再负责把结案和继续理由揉在一起讲完。  
它只负责保住一个入口判断：

动作结束以后，必须先结案，再把继续理由写进制度里，回合才算真正收住。

## 普通做法
更普通的做法，是把启动、输入、主循环里的这些小判断留在顺手逻辑里，不单独立规矩。 对父级入口页来说，更常见的写法就是继续把两三条不同链挤在一页里，读起来省页数，但判断会糊。

## 为什么不用
Claude Code 不这么做，是因为工作台一旦要长期运行，这些看似前期的小处理其实都是后面秩序能否站稳的根，不能等出事了再补。 这些父级页现在存在，就是为了先把分支关系讲清，再让更细页分别承担细节。

## 代价
代价是前期代码和解释都会显得更重，也更早暴露出长期工程的野心。 作为父级入口，它还额外带来导航变深、读者要多跳几页的成本。
