# 第十三卷回收索引：界面世界里的 sticky prompt 把局部上下文钉回现场说明界面也在做接缝锚点

## 实现链
界面世界的另一半，不是“信息有没有少”，而是“线索有没有被钉回现场”。

这条线真正逼出来的，是第十三卷里关于接缝锚点和局部上下文的判断。

这页实际回收的实现链是：
- `11-09-02` 把 `sticky prompt` 写成局部上下文锚点，这会在 `13-03-02` 里再次出现，成为最典型的接缝设计案例。

好的界面不会假装长会话没有漂移。  
它会在关键位置钉回一些局部上下文，让人重新回到现场。

## 普通做法
更普通的做法，是把界面当展示壳，状态散落在各个组件或服务里，最后再临时拼起来给人看。

## 为什么不用
Claude Code 不这么做，是因为跨子系统事实如果没有同一个落屏世界，界面很快就会出现各说各话的假真相，长会话尤其容易失控。

## 代价
代价是 AppState 会更大，显示层也得承担一部分制度、接缝和注意力管理逻辑。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
