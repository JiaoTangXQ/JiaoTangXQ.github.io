# 面试题：RECENT_COMPLETED_TTL_MS 为什么是 30 秒？completionTimestampsRef 的生命周期管理

这道题的核心是两个问题：30 秒这个数值是怎么定出来的？计时器是如何被精确管理的，既不会意外续命，也不会提前关闭？

## 常量定义和语义

```typescript
const RECENT_COMPLETED_TTL_MS = 30_000;
```

**为什么是 30 秒？**

30 秒是一个设计权衡，基于以下考量：

- **太短（< 5s）**：用户可能正在看其他地方，一眨眼任务就消失了，完成反馈失去意义
- **太长（> 60s）**：在密集的多 agent 任务里，任务区会被已完成项堆满，看不到进行中的任务
- **30 秒**：大约是一个人"完成一件事，处理完成反馈，决定下一步"的自然节奏

这是一个人机交互领域的经验值，不是精确计算出来的。类似的"反馈可见时间"在通知系统（Toast）里通常是 3-5 秒（操作级），在状态指示里是 30-60 秒（任务级）。

## completionTimestampsRef 的生命周期

```typescript
const completionTimestampsRef = React.useRef(new Map<string, number>());
const previousCompletedIdsRef = React.useRef<Set<string> | null>(null);

// 初始化
if (previousCompletedIdsRef.current === null) {
  previousCompletedIdsRef.current = new Set(
    tasks.filter(t => t.status === 'completed').map(t => t.id)
  );
}

// 每次渲染时更新
const currentCompletedIds = new Set(
  tasks.filter(t => t.status === 'completed').map(t => t.id)
);
const now = Date.now();

// 新完成的任务：记录时间戳
for (const id of currentCompletedIds) {
  if (!previousCompletedIdsRef.current.has(id)) {
    completionTimestampsRef.current.set(id, now);
  }
}

// 不再完成的任务：清理时间戳（任务被删除或重置）
for (const id of completionTimestampsRef.current.keys()) {
  if (!currentCompletedIds.has(id)) {
    completionTimestampsRef.current.delete(id);
  }
}

previousCompletedIdsRef.current = currentCompletedIds;
```

这段逻辑在渲染时（不在 effect 里）同步执行，有三个关键行为：

**新完成时**：记录当前时间戳，开始计时

**不再是 completed 时**：删除时间戳。任务被删除或状态回退时，之前的时间戳清理掉，避免"幽灵时间戳"

**保持跟踪**：`previousCompletedIdsRef` 记录上次完成集合，用于 diff 出"新完成"的任务

## 定时器：精确触发、不续命

```typescript
React.useEffect(() => {
  if (completionTimestampsRef.current.size === 0) {
    return;
  }
  const currentNow = Date.now();
  let earliestExpiry = Infinity;
  for (const ts of completionTimestampsRef.current.values()) {
    const expiry = ts + RECENT_COMPLETED_TTL_MS;
    if (expiry > currentNow && expiry < earliestExpiry) {
      earliestExpiry = expiry;
    }
  }
  if (earliestExpiry === Infinity) {
    return;
  }
  const timer = setTimeout(
    forceUpdate => forceUpdate((n: number) => n + 1),
    earliestExpiry - currentNow,
    forceUpdate
  );
  return () => clearTimeout(timer);
}, [tasks]);  // 只依赖 tasks
```

关键设计决策：

**只挂最早到期的定时器**

不是为每个完成任务各挂一个定时器，而是找出最早到期的那个，只挂一个定时器。当这个最早的到期、触发重渲染后，effect 会重新跑，找到下一个最早的，再挂一个定时器。

这避免了 N 个定时器同时运行，减少了内存和 CPU 的消耗。

**依赖是 `[tasks]`，不是 `[]`**

如果依赖是空数组，定时器只在 mount 时挂一次，但 tasks 变化（新任务完成）后不会重新计算最早到期时间，可能错过新完成的任务的定时器设置。

如果依赖是 `[completionTimestampsRef]`，ref 对象的引用永远不变，定时器永远不会重设。

`[tasks]` 是正确的：tasks 变化时，重新计算最早到期时间，重新挂定时器。

**不续命的保证**

"续命"指的是每次重渲染都重置计时器，让任务始终处于"刚刚完成"的状态。这里不会续命，因为：

- `completionTimestampsRef.set(id, now)` 只在 `!previousCompletedIdsRef.current.has(id)` 时执行
- 也就是说，时间戳只在"第一次完成"时记录，后续渲染不会覆盖

## forceUpdate 触发的机制

```typescript
const [, forceUpdate] = React.useState(0);
```

```typescript
const timer = setTimeout(
  forceUpdate => forceUpdate((n: number) => n + 1),
  earliestExpiry - currentNow,
  forceUpdate  // 把 forceUpdate 作为参数传入，避免 closure
);
```

注意这里的技巧：`setTimeout` 回调把 `forceUpdate` 作为参数（`setTimeout(callback, delay, ...args)`），而不是在 closure 里捕获。这避免了 stale closure 问题——即使 effect cleanup 后 forceUpdate 引用变了，传进去的参数版本也是挂定时器时的版本。

## 面试指导

**直接问法**："30 秒 TTL 是怎么定的？"

答：经验值，对应"完成一件事、处理反馈、决定下一步"的自然节奏。太短（< 5s）用户可能来不及看，太长（> 60s）密集任务里完成项会堆满面板。30 秒是任务级反馈可见性的合理范围。

**技术追问**："为什么只挂最早到期的定时器而不是每个任务一个？"

答：N 个任务就挂 N 个定时器，资源消耗 O(N)。只挂最早的一个，每次触发后重新找最早，资源消耗 O(1)（任意时刻只有一个定时器）。逻辑上，只需要知道"什么时候要重渲染"，而最早到期时间就是下次必须重渲染的时间点。

**精确性追问**："这套机制能保证 30 秒窗口不会被续命或提前关闭吗？"

答：不续命靠的是"时间戳只在首次完成时记录"（`!previousCompletedIdsRef.current.has(id)` 守卫）。不提前关闭靠的是"定时器只在 `earliestExpiry - currentNow > 0` 时挂"（已过期的不挂新定时器）。cleanup 函数 `clearTimeout` 防止组件卸载时定时器继续触发。三个机制共同保证了 30 秒窗口的精确性。

*核心考察点：定时器管理的精确性、"N 个 vs 1 个"定时器的权衡、stale closure 的规避技巧、ref 和 state 的协同。*
