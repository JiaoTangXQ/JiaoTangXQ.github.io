# ProgressTracker 的 input_tokens 为什么不累加而是取最新值？

`LocalAgentTask.tsx` 里有一个 `ProgressTracker` 类型：

```typescript
export type ProgressTracker = {
  toolUseCount: number
  // Track input and output separately to avoid double-counting.
  // input_tokens in Claude API is cumulative per turn (includes all previous context),
  // so we keep the latest value. output_tokens is per-turn, so we sum those.
  latestInputTokens: number
  cumulativeOutputTokens: number
  recentActivities: ToolActivity[]
}
```

注释里藏着一个重要细节：`input_tokens` 和 `output_tokens` 的累计方式**不一样**。

## Claude API 的 token 计数方式

理解这个设计，需要先理解 Claude API 的 usage 字段行为。

每轮对话，API 响应里的 `usage` 大概长这样：

```json
{
  "input_tokens": 1547,
  "cache_creation_input_tokens": 0,
  "cache_read_input_tokens": 1200,
  "output_tokens": 312
}
```

**`input_tokens`（含 cache 相关字段）是累计值**：它包含了整个对话历史的所有 token，每轮都重新计算一次。如果第一轮的对话历史是 1000 个 token，第二轮加上新回复后变成 1500 个 token，那么第二轮的 `input_tokens` 是 1500，不是 500（增量）。

**`output_tokens` 是单轮值**：每轮 Agent 生成的 output token 数，每轮独立计算，不累计。

如果对 `input_tokens` 也做累加，会严重高估 token 使用量：

```
第 1 轮：input = 1000，cumulative input = 1000
第 2 轮：input = 1500，累加后 cumulative input = 2500（错误！实际才消耗了 500 个新 token）
第 3 轮：input = 2200，累加后 cumulative input = 4700（严重高估）
```

正确做法是：input token 只保留最新值（`latestInputTokens = latest input_tokens`），output token 才做累加（`cumulativeOutputTokens += output_tokens`）。

## updateProgressFromMessage 的实现

```typescript
export function updateProgressFromMessage(
  tracker: ProgressTracker,
  message: Message,
  resolveActivityDescription?: ActivityDescriptionResolver,
  tools?: Tools,
): void {
  if (message.type !== 'assistant') return

  const usage = message.message.usage

  // Keep latest input (it's cumulative in the API), sum outputs
  tracker.latestInputTokens = usage.input_tokens
    + (usage.cache_creation_input_tokens ?? 0)
    + (usage.cache_read_input_tokens ?? 0)
  tracker.cumulativeOutputTokens += usage.output_tokens

  for (const content of message.message.content) {
    if (content.type === 'tool_use') {
      tracker.toolUseCount++
      // 跳过内部工具（SyntheticOutput）
      if (content.name !== SYNTHETIC_OUTPUT_TOOL_NAME) {
        const input = content.input as Record<string, unknown>
        const classification = tools
          ? getToolSearchOrReadInfo(content.name, input, tools)
          : undefined
        tracker.recentActivities.push({
          toolName: content.name,
          input,
          activityDescription: resolveActivityDescription?.(content.name, input),
          isSearch: classification?.isSearch,
          isRead: classification?.isRead,
        })
      }
    }
  }

  // 只保留最近 5 条活动
  while (tracker.recentActivities.length > MAX_RECENT_ACTIVITIES) {
    tracker.recentActivities.shift()
  }
}
```

注意 `cache_creation_input_tokens` 和 `cache_read_input_tokens` 也加进了 `latestInputTokens`。这两个字段是 Claude 的 prompt cache 相关计数，有时代表缓存命中节省的 token（写缓存和读缓存的成本不同），需要一起统计才能反映实际的 token 消耗。

## getTokenCountFromTracker：两个字段合并为一个数字

```typescript
export function getTokenCountFromTracker(tracker: ProgressTracker): number {
  return tracker.latestInputTokens + tracker.cumulativeOutputTokens
}
```

这个函数把两个不同语义的字段合并成一个"总 token 数"用于展示。用户看到的进度里只有一个数字，背后的两种计算方式对用户透明。

## recentActivities：滑动窗口设计

```typescript
const MAX_RECENT_ACTIVITIES = 5
```

`recentActivities` 只保留最近 5 条工具调用活动，用 `while + shift()` 维护一个长度上限为 5 的滑动窗口。

为什么不保留全部？

一个长时间运行的 Agent 可能调用数百次工具。把所有活动都保存在内存里，不仅占用空间，对于"显示当前在干什么"这个用途也没有必要——用户只需要看最近几条。

`recentActivities` 的作用是给 UI 展示"最近在干什么"（比如"刚刚读了 src/Task.ts"），而不是记录完整的执行历史（那是 transcript 文件的工作）。

## ActivityDescriptionResolver：预计算活动描述

```typescript
export type ActivityDescriptionResolver = (
  toolName: string,
  input: Record<string, unknown>,
) => string | undefined

export function createActivityDescriptionResolver(tools: Tools): ActivityDescriptionResolver {
  return (toolName, input) => {
    const tool = findToolByName(tools, toolName)
    return tool?.getActivityDescription?.(input) ?? undefined
  }
}
```

每个 Tool 可以实现 `getActivityDescription()` 方法，返回人类可读的活动描述。比如 `FileReadTool.getActivityDescription({ file_path: 'src/Task.ts' })` 可能返回 `"Reading src/Task.ts"`。

这个描述在**工具调用发生时**就被计算好，存入 `ToolActivity.activityDescription` 字段，而不是在 UI 渲染时再查找工具并计算。这是一个"预计算换渲染性能"的优化：工具调用时计算一次，之后每次 UI 刷新直接读字段即可。

## 面试要点

这个设计揭示了一个实践中常见的问题：**对外部 API 行为的错误假设会导致数据统计 bug**。

`input_tokens` 是累计值，`output_tokens` 是单轮值——这两个字段的语义完全不同，但从字段名上完全看不出来。只有读了 API 文档（或者踩了坑）才能知道。

代码里把这个知识显式地写进了注释和字段名（`latestInputTokens` vs `cumulativeOutputTokens`），让后来的维护者不需要再去查 API 文档就能理解这里的逻辑。

这是一个好注释的典型案例：不是解释代码在做什么（显而易见），而是解释为什么这样做（外部约束）。
