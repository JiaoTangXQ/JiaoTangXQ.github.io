# 同样是 runAgent()，普通子 agent 和 teammate 有什么不同？

`runAgent()` 是 Claude Code 里执行 AI agent 的核心函数，普通子 agent 和 in-process teammate 都调用它。

从执行引擎的角度，两者几乎相同：发 API 请求、处理工具调用、管理对话历史。

差异在于外壳，不在于核心。

## 普通子 agent 的外壳

普通子 agent（通过 AgentTool 启动）在任务系统里运行：

```
LocalAgentTask
  ├── 注册任务 (registerTask)
  ├── 追踪进度 (progressTracker)
  ├── runAgent() 执行核心
  └── 完成后清理 (evictTerminalTask)
```

它需要的"外壳"：任务注册、进度追踪、结果通知、清理。这些都是"一个工作单元"所需要的管理设施。

## Teammate 的额外外壳

Teammate 同样调用 `runAgent()`，但外层多了 Swarm 系统的制度外壳：

```
InProcessTeammateTask
  ├── TeammateIdentity (agentId, teamName, color, planModeRequired, parentSessionId)
  ├── runWithTeammateContext(teammateContext, () => {
  │     runWithAgentContext(agentContext, () => {
  │       runAgent() 执行核心    ← 和普通 agent 一样
  │     })
  │   })
  ├── 轮询 mailbox（接收来自 Leader 和其他 teammate 的消息）
  ├── 处理 shutdown_request（优雅退出协议）
  ├── 处理 permission_request（通过 leader 批权限）
  ├── 发送 idle notification（完成时通知 leader）
  └── 同步到 team file（setMemberActive）
```

多出来的每一层都在回答一个"组织成员需要回答的问题"：

- **TeammateIdentity**：我是谁？我在哪个团队？
- **runWithTeammateContext**：我的身份信息如何沿着 async 调用链传播？
- **mailbox 轮询**：Leader 和队友有没有给我发消息？
- **shutdown_request 处理**：Leader 叫我退出时，我要优雅完成当前工作再退出，不是直接崩溃。
- **permission_request 处理**：我需要权限时，找 Leader 协调，不是自己弹窗。
- **idle notification**：我完成工作了，Leader 需要知道我空了。
- **team file 同步**：我的状态变化（active/idle）需要同步到团队配置文件，让 Leader 能随时看到成员状态。

## 为什么 runAgent() 不感知这些

一个关键的架构决策：`runAgent()` 本身不知道自己在团队里还是独立运行。这些团队制度全部在 `runAgent()` 的外层处理。

这样做的原因：

**单一职责**：`runAgent()` 的职责是"运行一个 AI agent 对话循环"，不是"参与团队协作"。如果把 mailbox 轮询、idle notification 等逻辑塞进 `runAgent()`，它就变成了一个知道太多事情的大函数，职责边界模糊。

**可复用性**：`runAgent()` 可以在任何上下文里使用——独立会话、子 agent、teammate、单次调用。如果它内置了团队协作逻辑，在非团队场景里使用就会带着不需要的负担。

**可测试性**：测试 `runAgent()` 不需要 mock mailbox、leaderPermissionBridge、team file。这些都在外层，可以独立测试。

这是依赖倒转原则（DIP）的体现：高层策略（团队协作制度）不应该依赖低层机制（AI 对话循环），而是高层策略在低层机制之上加外壳。

## 制度外壳让 agent 有秩序地共存

`inProcessRunner.ts` 开头的注释写得很清楚：

> Wraps runAgent() for in-process teammates, providing:
> - AsyncLocalStorage-based context isolation via runWithTeammateContext()
> - Progress tracking and AppState updates
> - Idle notification to leader when complete
> - Plan mode approval flow support
> - Cleanup on completion or abort

"制度外壳"的价值不是让单个 agent 更强，而是让多个 agent 能有秩序地共存。

没有这个外壳，5 个 agent 各自运行 `runAgent()`，权限请求各自弹窗，完成通知各自随机，Leader 不知道哪个 agent 完成了，用户不知道在看哪个 agent 的输出。

有了这个外壳，每个 agent 的行为都被纳入了团队制度框架：身份明确、通信有规矩、权限有归口、退场有协议。

## 面试怎么答

如果面试题是"如何在不修改核心执行逻辑的情况下，为 agent 增加团队协作能力"：

**设计原则**：把团队协作制度做成"外壳"，核心执行逻辑（runAgent）保持不变。外壳负责所有组织相关的事务，内核负责 AI 对话循环。

**具体机制**：AsyncLocalStorage 在外层设置身份上下文，内层代码可以读取但不需要知道来源；mailbox 轮询在外层并行运行，收到消息后注入到 agent 的输入里；idle notification 在外层捕获 runAgent 的完成事件后发送。

**价值**：核心执行逻辑（runAgent）保持纯粹和可测试；团队制度（外壳）可以独立演进，比如更换后端、改变通信协议、调整权限模型，都不需要修改 runAgent。这是"开放封闭原则"在多 agent 架构里的应用：核心对修改封闭，制度外壳对扩展开放。
