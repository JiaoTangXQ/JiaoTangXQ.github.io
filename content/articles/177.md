# 第十三卷回收索引：治理世界里的设置变化先拦截扇出再落地

## 实现链
治理世界里有一条很清楚的线，不是“改了值界面就跟着变”，  
而是设置变化本身也要先经过程序。

这条线真正逼出来的，是第十三卷里关于强而不乱和制度先行的判断。

这页实际回收的实现链是：
- `06-07-02` 把设置变化写成“先拦截、再扇出、再落地”，这会在 `13-04-01` 里继续支撑“强能力先过程序”。

一个系统真把设置当制度装配中心，  
就不会把变化直接当成界面刷新事件。  
它会先拦截，再扇出，再落地。

## 普通做法
更普通的做法，是每个能力各带一点本地校验，或者干脆给用户一个粗糙的总开关，把细节风险留给人自己判断。

## 为什么不用
Claude Code 不这么做，是因为强能力一旦缺少统一制度快照和变更流程，就会在最危险的地方出现口径不一致，最后连改规则本身都没有共同程序。

## 代价
代价是 permission context、settings 扇出、hooks 和模式契约都更重，读起来也更像制度系统。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
