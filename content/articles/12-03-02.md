# 系统设计：工具名、提示词和渲染层如何保持三方对齐？

## 多层系统对齐的经典挑战

在多层软件架构里，有一类常见问题：同一个业务概念在不同层面有不同的表达，这些表达之间没有强绑定，随时间推移会悄悄漂移。

用户管理系统里，数据库叫 `user_id`，API 返回 `userId`，前端存 `currentUser.id`，错误日志里打 `uid`——同一个概念，四种名字，哪天改字段时一定有地方漏掉。

Claude Code 的 computer-use 功能在这个问题上做了一个有意识的选择：让同一套 `mcp__computer-use__*` 身份在三个独立系统里保持一致。

## 三层的具体对齐方式

**第一层：工具名称**

`setupComputerUseMCP()` 生成的工具名全部通过 `buildMcpToolName` 产生：

```ts
buildMcpToolName(COMPUTER_USE_MCP_SERVER_NAME, t.name)
// mcp__computer-use__screenshot
// mcp__computer-use__click
// mcp__computer-use__type
```

**第二层：system prompt 注入**

后端 API 检测到 `mcp__computer-use__*` 前缀时，会自动注入 `COMPUTER_USE_MCP_AVAILABILITY_HINT`。这是一段给模型看的、说明"你现在有桌面操作能力"的提示词。模型需要这段提示才能正确地决定何时、如何使用这些工具。

**第三层：UI 渲染**

`toolRendering.tsx` 会识别 computer-use 工具调用，并做特殊的终端界面渲染——比如把截图工具调用显示成带图标的 UI 元素，而不是普通的函数调用文本。

这三层里，没有任何一层是靠"约好了记住"来维持一致的——它们都依赖同一个常量 `COMPUTER_USE_MCP_SERVER_NAME`，而工具名都通过同一个 `buildMcpToolName` 函数生成。

## 为什么不分层各用各的名字？

更常见的架构是：后端用内部 ID，API 层做映射，前端用显示名，日志用另一套缩写。

这种分层命名看起来更灵活，但在实践中有几个问题：

**问题一：映射层的维护成本**。每次加新工具，要在多个地方同步添加映射。

**问题二：调试时的认知跳跃**。看日志时看到 `cu_screenshot`，不知道对应 API 里的哪个 endpoint，不知道对应 UI 里的哪个工具项。

**问题三：文档和测试的分裂**。写文档时要解释"这个功能的内部名字是 X，API 名字是 Y，UI 显示是 Z"。

Claude Code 用统一命名消除了这些问题：无论在哪一层看到 `mcp__computer-use__screenshot`，你都知道它指向的是完全相同的那个操作。

## 这种设计的局限

统一命名有其代价：**命名方案的修改成本极高**。

如果将来想重命名工具（比如把 `screenshot` 改成 `capture_screen`），需要同时修改：
- 工具定义处
- `buildMcpToolName` 的调用处
- 后端 API 的前缀检测逻辑
- `allowedTools` 的维护代码
- `toolRendering.tsx` 的识别逻辑
- 可能还有 Cowork 等其他使用同名工具的产品

这就是"命名即协议"的真实代价：统一带来了对齐，也带来了刚性。

好的做法是：一旦确定了命名约定，把它文档化，让所有参与方知道"这个字符串是跨系统约定，不要随意改"。Claude Code 通过代码注释做了这件事：

```ts
// The API backend detects mcp__computer-use__* tool names and emits a CU 
// availability hint into the system prompt. Built-in tools with different 
// names wouldn't trigger it.
```

这个注释不只是解释了"为什么要这样命名"，还隐含地说明了"如果你想改这个名字，你需要同时改多少地方"。

## 扩展到其他能力簇

这套三层对齐的模式在 Claude in Chrome 里有完全相同的镜像：

- 工具名：`mcp__claude-in-chrome__*`
- system prompt：`BASE_CHROME_PROMPT` + `getChromeSystemPrompt()`
- UI 渲染：`toolRendering.tsx` 里的 chrome 特殊处理（`tabId` 追踪、`[View Tab]` 链接）

两套外延能力，两套统一命名，同样的三层对齐模式。这说明这不是 computer-use 特有的设计，而是 Claude Code 处理"外延能力接入"的通用模式。

## 面试考察点

这道题考察的是**跨层系统设计中的一致性维护策略**。

面试官问到"如何保证多个独立服务/层对同一个业务概念的理解保持一致"时，可以展开的方向：

1. **强绑定**：通过共享类型定义、接口规范，让编译器强制一致
2. **软绑定**：通过命名约定 + 集中常量 + 代码注释，降低漂移风险
3. **契约测试**：用测试验证多系统之间的约定在运行时仍然成立

Claude Code 采用了第二种方案（软绑定），因为第一种方案（强绑定）需要后端 API 服务和前端代码共享类型定义，在当前架构下实现成本较高。

这不是最理想的，但是务实的。资深工程师能区分"理想架构"和"在当前约束下最合理的架构"。
