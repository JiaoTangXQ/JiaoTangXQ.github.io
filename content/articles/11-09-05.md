# setSearchQuery 搜不到时为什么要 scrollTo(searchAnchor)？incsearch 的试探回退机制

大多数搜索实现在无匹配时只是清空结果列表。Claude Code 还要把视图滚回到搜索开始的位置，为什么？

## incsearch 的用户体验模型

incsearch（incremental search，即时搜索）的体验承诺是：用户每打一个字，界面立刻跳到最近的匹配，用户看着结果实时变化，找到想要的就停下来按 Enter 确认。

这个过程中，每次键入都可能触发视图跳转。用户在连续输入 "error handling" 时：

```
e → 跳到最近的含 "e" 的消息（可能离当前很远）
er → 跳到最近的含 "er" 的消息
err → ...
```

每次都跳来跳去，但用户**不想迷路**。如果输错了（"errror" 多打一个 r），没有匹配，视图应该回到哪里？

答案应该是："回到搜索开始前我在看的位置"，而不是"停在最后一次匹配的地方"。

## searchAnchor 的设定和使用

`searchAnchor.current` 是一个 `number`（scrollTop 值），在用户按下 `/` 开始搜索时通过 `JumpHandle.setAnchor()` 记录：

```typescript
setAnchor: () => {
  searchAnchor.current = scrollRef.current?.getScrollTop() ?? -1
}
```

按 `/` 时记录的 scrollTop，就是"用户开始搜索时的视口位置"。

在 `setSearchQuery` 里，每次 query 更新后：

```typescript
function setSearchQuery(q: string) {
  // 扫描所有消息找匹配
  const matches = scanAllMessages(q)
  
  if (matches.length === 0) {
    // 没有匹配：回到锚点
    if (searchAnchor.current >= 0) {
      scrollRef.current?.scrollTo(searchAnchor.current)
    }
    return
  }
  
  // 有匹配：跳到最近的一个
  jump(findClosestMatch(matches, scrollRef.current?.getScrollTop()))
}
```

没有匹配时 `scrollTo(searchAnchor.current)`，把用户送回搜索前的位置。

## n/N 不重置 anchor

按 `n`（下一个匹配）和 `N`（上一个匹配）时，锚点不更新。锚点只在：
1. 用户按 `/` 开始新搜索时设置
2. 搜索完全退出时清空（设为 -1）

这意味着整个 incsearch 过程中（不断输入、修改、找到匹配后 n/N 导航），"搜索开始点"始终不变。无论中途跳到多远的地方，只要 query 变成无匹配，就回到搜索起点，而不是最后一次导航的位置。

## pending step 机制

键盘操作的时序问题：用户快速按 `n` 时，`jump()` 还没完成（seek effect 还没运行），下一个 `n` 就来了。如果直接处理，会触发两次 jump，先跳 A 再跳 B，看起来是两次跳跃。

`pendingStepRef.current` 解决这个问题：

```typescript
// 在 seek effect 里
if (pendingStepRef.current !== 0) {
  const dir = pendingStepRef.current
  pendingStepRef.current = 0
  stepRef.current(dir)
}
```

seek effect 完成时检查是否有 pending step，有的话顺序执行下一步。多次 `n` 会覆盖 `pendingStepRef.current`（每次写入都覆盖之前的值），只保留最后一次的方向。

这实现了"按住 n 快速浏览"的效果：每个 seek 完成后执行一步，不堆积，不丢失方向。

## disarmSearch 和 incsearch 退出

用户手动滚动时（j/k/PageUp/鼠标滚轮），搜索"解除"：

```typescript
// ScrollKeybindingHandler 的 onScroll 回调
jumpRef.current?.disarmSearch()

// disarmSearch 实现
disarmSearch: () => {
  searchState.current.matches = []
  searchState.current.ptr = 0
  elementPositions.current = { msgIdx: -1, positions: [] }
  setPositions?.(null)  // 清除高亮层
  // 注意：不清除 searchAnchor（anchor 在下次 / 时才覆盖）
}
```

解除后黄色当前匹配高亮消失（positions 清空），但之前的搜索结果高亮仍保留（inverse highlight，从 render-to-screen 层写入），下次 n/N 会重新建立搜索上下文。

## 面试指导

这道题考察搜索体验中的"状态恢复"设计，实际上是一道 UX + 状态机的综合题。

**核心决策**：搜索状态有两层——"当前位置"（动态，随每次 incsearch 跳转）和"搜索起点"（静态，搜索开始时记录）。分开维护两层，让"出错后回到起点"成为可能。如果只有一层状态，出错时只能"停在原地"，用户不知道自己跳了多远。

**对比 vim 的 /search**：vim 的 incsearch 在输入过程中预览匹配，取消时（Esc）回到搜索开始时的位置，确认时（Enter）停在最后一次匹配。Claude Code 的 searchAnchor 实现了类似的"取消回起点"语义。

**一个边界情况**：如果用户 `/` 然后立刻 n/N 不输入 query，会怎样？答：matches 是空数组，n/N 的 step 函数对空 matches 是 noop，不跳转。searchAnchor 已经在按 `/` 时设置，Esc 或手动滚动可以回到起点。
