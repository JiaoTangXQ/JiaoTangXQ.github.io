# Session Hooks：Claude Code 里的 pre/post 钩子系统是怎么设计的？

Claude Code 支持在关键执行点插入自定义逻辑：工具调用前（PreToolUse）、工具调用后（PostToolUse）、会话开始（SessionStart）、compact 前后（PreCompact/PostCompact）。

这套系统的设计细节在 `utils/hooks/sessionHooks.ts` 里，有几个值得深挖的工程决策。

## 两种钩子类型

**命令钩子（HookCommand）**：执行外部命令或脚本，可以持久化到 settings.json：

```typescript
type HookCommand = {
  type: 'command'
  command: string
  timeout?: number
  // ...
}
```

**函数钩子（FunctionHook）**：在进程内执行 JavaScript 回调函数，会话级临时存在：

```typescript
export type FunctionHookCallback = (
  messages: Message[],
  signal?: AbortSignal,
) => boolean | Promise<boolean>

export type FunctionHook = {
  type: 'function'
  id?: string
  callback: FunctionHookCallback
  errorMessage: string  // 钩子失败时的错误消息
  statusMessage?: string // 钩子执行期间的状态消息
}
```

函数钩子的 `callback` 返回 `boolean`——`true` 表示通过，`false` 表示阻止当前操作。这让函数钩子可以实现"不允许写入特定文件"、"检查代码风格"这类程序化的权限检查。

## 为什么用 Map 而不是 Record？

这是整个实现里最有意思的细节，注释写得非常详细：

```typescript
// Map（不是 Record），这样 .set/.delete 不改变容器身份。
// 修改函数会 mutate Map 但返回不变的 prev，
// 让 store.ts 的 Object.is(next, prev) 检查短路，跳过监听者通知。
//
// 这在高并发 workflow 下很重要：parallel() 里的 N 个 schema-mode agent
// 在同一个同步 tick 里触发 N 次 addFunctionHook。
// 用 Record + spread：每次调用 O(N) 复制增长中的 map（总 O(N²)），
// 同时触发所有 ~30 个 store 监听者。
// 用 Map：.set() 是 O(1)，返回 prev 意味着零监听者触发。
export type SessionHooksState = Map<string, SessionStore>
```

关键洞察：Session hooks 是**运行时临时状态**，不是响应式 UI 状态。它们只在 query loop 里通过 `getAppState()` 快照读取，不需要响应式更新。

所以：mutation in-place + 返回原引用 = 状态改变了但 `Object.is` 比较为 true = 不触发 store listeners。这个模式把 N 个并发钩子注册从 O(N²) + O(30N) 的监听触发降到 O(N) + 0 个监听触发。

## 钩子事件的类型系统

钩子系统使用 `HookEvent` 类型枚举所有支持的事件点，来自 `HOOK_EVENTS` 常量：

```typescript
type SessionStore = {
  hooks: {
    [event in HookEvent]?: SessionHookMatcher[]
  }
}
```

钩子注册时必须指定事件类型，不能注册"全局钩子"。这让权限检查更精确——"在 PostToolUse 之前阻止"和"在 PreToolUse 之前阻止"是不同的语义。

## 钩子和 CLAUDE.md frontmatter

`utils/hooks/registerFrontmatterHooks.ts` 负责从 CLAUDE.md 文件的 frontmatter 里读取钩子配置：

```yaml
---
hooks:
  PreToolUse:
    - command: "./scripts/check-before-edit.sh"
      timeout: 5000
---
```

这让项目级的工具调用前检查可以通过 CLAUDE.md 配置，不需要修改全局设置。每次子 Agent 启动时会重新注册这些钩子（`registerFrontmatterHooks`），子 Agent 结束时清理（`clearSessionHooks`）。

## 面试角度

这套钩子系统有几个值得讨论的设计点：

1. **命令钩子 vs 函数钩子**：一个可以持久化（settings.json），一个只在进程内有效。持久化钩子适合"总是检查代码格式"，函数钩子适合"这次会话里临时限制某个操作"
2. **Map 替代 Record 的 O(N²) 优化**：当状态不需要触发响应式更新时，mutation-in-place 是合理选择，但前提是了解状态管理框架的更新检测机制
3. **钩子的返回值协议**：`boolean` 比抛出异常更简洁，但需要 `errorMessage` 字段来传递错误信息
4. **生命周期绑定**：钩子的注册和清理与 Agent 生命周期绑定，不会泄漏到父 Agent
