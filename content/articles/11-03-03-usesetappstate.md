# useSetAppState 把写入和订阅故意拆开

Claude Code 还专门把 `useSetAppState()` 单独留出来，让组件在只需要写状态时，不必顺手订阅任何东西。这看起来是小技巧，实际上是在切断一种很常见的性能浪费。

因为很多控制按钮、命令入口、事件桥接点，本来就只负责改世界，不需要每次世界变化都重新画自己。把写和读拆开，界面才能继续长大而不越来越重。

## 实现链

`useSetAppState()` 直接返回稳定的 `store.setState`，不顺带订阅任何状态。也就是说，写入入口和读取订阅被故意拆开：想改状态的组件可以只拿写句柄，不必因为顺手写一下就把自己挂到整份状态变化上。

## 普通做法

更普通的 context API 往往会把 `[state, setState]` 一起给出去，谁拿到写入口，顺手也就把整份 state 订阅了。

## 为什么不用

Claude Code 不把读写绑在一起，是因为有不少组件只是触发动作，不需要跟着状态变化一起重渲染。把写入口单独拆出来，能少制造很多无意义订阅。

## 代价

代价是 API 比常见 React 模式多分了一层，读者需要先理解为什么这里连写句柄都要单独包一个 hook；但性能和边界都更稳。
