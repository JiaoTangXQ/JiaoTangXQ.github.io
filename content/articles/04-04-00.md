# 分层修复链导读：上下文治理不是一个压缩功能

这一章最容易被误读成“Claude Code 有很多压缩技巧”。真正更重要的判断，是这些技巧根本不是并列招式，而是一条从小修到大修的分层修复链。

所以这一章的四片叶子，要当成同一条链来读。先看 [04-04-01-为什么工具结果预算和snip要先于真正的压缩.md](/Users/xxx/tmp/cc/docs/claude-code-book/04-主循环怎样思考与执行/04-04-为什么上下文整形不是一次大手术而是一串小修复/04-04-01-为什么工具结果预算和snip要先于真正的压缩.md)，再顺着 `microcompact`、`collapse/autocompact` 和最后这页“分层修复链”收口往下走，读者才会明白：主循环里的上下文治理不是一个功能，而是一套按严重程度接力的修复制度。

代码里这条链是显式展开的，而不是一个叫 `compressEverything()` 的黑盒。`query()` 先在进入模型前检查 blocking limit、tool result snip 和 cache-safe 条件，再在真实错误出现后按 `collapse_drain_retry`、`reactive_compact_retry` 等 `transition.reason` 逐层补救。更容易想到的办法，是只留一个“上下文太长就总结一遍”的统一压缩器，因为 API 表面上只暴露了一个问题：token 不够。

Claude Code 没这么做，是因为“上下文太长”在代码里并不是一种单一病因。工具结果太胖、最近几轮对话太啰嗦、媒体内容过大、停钩子补出来的新信息太多，这些问题的最便宜修法并不一样。这样设计更符合真实故障，但代价就是主循环会长出一套恢复状态机，不熟的人第一次读会觉得分支很多。

## 实现链
`query()` 先走 snip / microcompact，再按 `collapse_drain_retry`、`reactive_compact_retry` 等路径救场。

## 普通做法
只保留一个统一 compact 功能。

## 为什么不用
因为“上下文太长”在代码里是多种病因，不是一种病。

## 代价
主循环会长成恢复状态机。
