# 设计题：给 LLM Agent 设计一套上下文管理系统

这是一道在大型 AI 公司面试里越来越常见的系统设计题：**给一个长期运行的 LLM Agent 设计上下文管理系统，要求在 token 超限时能自动处理，同时尽量保留对话连续性。**

大多数候选人的第一反应是：设计一个"压缩器"——token 接近上限时，调用 LLM 把当前对话总结成摘要，用摘要替换原来的历史。这个方案能工作，但面试官通常会追问：每次都全量总结，成本太高怎么办？有些内容总结后会丢失关键细节怎么办？

这时候，你需要一个分层的回答。

## Claude Code 的答案：四级修复链

Claude Code 把上下文管理设计成一条**按代价从低到高排列的修复链**，而不是一个统一的压缩器。每一级处理不同的病因，代价和破坏性各不相同。

### 第一级：工具结果预算 + Snip（轻量裁剪）

**病因**：某几次工具调用返回了超大结果（比如读了一个 10000 行的文件）。

**治法**：`applyToolResultBudget()` 对单个工具结果的大小设上限；`snipCompact` 对消息列表做局部裁剪，只删最胖的那几条。

**代价**：极低。不调用 API，不改变对话结构，只是截断或清空特定工具结果的内容。

**适用条件**：token 压力来自最近几次"超大工具结果"，而不是整段历史太长。

### 第二级：Micro-compact（温和减重）

**病因**：历史工具结果积累太多，但大多数是可预测的"低信息密度"内容（文件读取、grep 输出、shell 命令结果）。

**治法**：`microcompactMessages()` 识别出这些可安全删除的工具结果，通过 cache editing API 在服务端删除，**不修改本地消息结构**，缓存前缀依然有效。

**代价**：低。不调用摘要 API，利用 prompt cache 机制，成本主要是一次 cache edit 操作。

**关键设计**：识别"可 compact 工具"（`COMPACTABLE_TOOLS`）：`Read`, `Bash`, `Grep`, `Glob`, `WebSearch`, `WebFetch`, `Edit`, `Write`。这些工具的历史结果通常不需要保留，因为文件可以重读、命令可以重跑。

### 第三级：Context Collapse（结构性折叠）

**病因**：某些对话片段虽然重要，但可以被压缩成更紧凑的表示形式（比如一段很长的规划讨论，可以折叠成最终决策）。

**治法**：`applyCollapsesIfNeeded()` 在本地维护一个折叠提交日志，把已折叠的片段替换为摘要，**优先于 autocompact 执行**——如果 collapse 后上下文已经够了，就不需要 autocompact。

**代价**：中等。需要维护折叠状态，恢复时需要 `projectView()` 重放提交日志。

### 第四级：Auto-compact（全量压缩）

**病因**：上下文已经接近 `autoCompactThreshold`（通常是 context window 减去 20k 的缓冲），前面几级都没能解决问题。

**治法**：`compactConversation()` 调用一次独立的 LLM，用整段历史生成摘要，用摘要替换原来的消息。如果是 session memory 模式，还会把历史持久化存储。

**代价**：高。需要一次完整的 API 调用（通常用 Haiku 这类快速模型），会丢失历史细节，缓存完全失效。

## token 警告层级系统

为了知道什么时候启动哪一级修复，Claude Code 维护了一套 token 警告状态：

```typescript
function calculateTokenWarningState(tokenUsage: number, model: string) {
  const warningThreshold = threshold - 20000   // 显示警告
  const errorThreshold = threshold - 20000     // 显示错误提示
  const autoCompactThreshold = effectiveWindow - 13000  // 触发 autocompact
  const blockingLimit = effectiveWindow - 3000 // 完全阻止（保留手动 /compact 空间）
}
```

这几个阈值不是随意设的：`13000` 的缓冲是为 auto-compact 的摘要输出预留空间（p99.99 的摘要在 17387 tokens 以内）；`3000` 是为用户手动运行 `/compact` 预留的最后保障。

## 失败时的兜底：Reactive Compact

上面四级是"预防性"的。但如果还是发生了 API 的 413（prompt too long）错误，Claude Code 还有一个"应急"修复层：

```
收到 413 错误
  ↓
withheld（不立刻暴露给外层）
  ↓
尝试 collapse_drain_retry（把暂存的折叠提交掉，再试一次）
  ↓
尝试 reactive_compact_retry（触发一次应急全量压缩）
  ↓
恢复失败 → 把错误暴露出来，返回 { reason: 'prompt_too_long' }
```

"withholding"（暂扣错误）是这里最精妙的设计：收到 413 时不立刻把错误 yield 给外层，而是悄悄尝试自救。如果自救成功，外层看到的是一次成功的 response，完全感知不到曾经有过一次 413。这个模式对 SDK 集成者非常友好——他们不需要在客户端写上下文管理逻辑。

## 面试指导

回答这道题时，一个好的结构是按"代价从低到高"组织你的设计：

1. **先做最便宜的**：截断超大工具结果（无 API 成本）
2. **再做局部优化**：清除历史低信息密度内容（利用 cache 机制）
3. **然后结构性压缩**：折叠特定对话片段（有状态，可恢复）
4. **最后全量摘要**：调用小模型生成压缩摘要（有 API 成本，丢失细节）
5. **应急层**：遇到 API 413 时的 reactive compact（withholding 模式）

能说出这五层、能解释每层的代价和适用条件，这道题就已经回答得比大多数候选人深了。如果还能说出"为什么 collapse 要先于 autocompact 执行"（答案：collapse 是可逆的投影，autocompact 会不可逆地丢失历史），面试官会很满意。
