# 第三卷导读：一句话不是直接去聊天，而是先被翻译成系统入口

## 这卷在回答什么

第三卷要回答的，不是用户一句话怎样最快送到模型面前，而是一条输入为什么必须先被系统整理、分流、翻译，最后才变成真正的会话入口。原始文本里可能混着图片、附件、路径、slash 命令、桥接来源和制度边界，它从来都不是一句可以原样直送的“纯提示词”。

所以这卷真正要讲的是：入口不是洞口，而是一条翻译链。Claude Code 真正成熟的地方，不在于它敢把一切都交给模型，而在于它先替系统把入口整理成了可继续执行的对象。

## 这页真正想补的东西

这一卷前半段在讲“先把用户说的话整理成系统认得的事件”，后半段在讲“再把这些事件送进命令和边界世界”。把这两段接起来以后，第三卷的判断才完整：一句话不是直接去聊天，而是先被翻译成系统入口。

也正因为如此，这一页不需要变成长附录。它只负责把读者送进正确入口，并先把整卷主判断钉住一次：真正成熟的 agent，不会把入口治理偷懒外包给模型。

## 实现链
入口不是直接把字符串塞给模型，而是先过 `processUserInput()`。这条链先处理图片缩放和落盘、再抽附件、再看 bash / slash / bridge 安全边界，最后才决定走 `processTextPrompt()` 还是命令分支，再由 `QueryEngine.submitMessage()` 接到主循环。

## 普通做法
更普通的 CLI agent 会把输入当成一段 prompt 文本，只在最外层做一个“是不是 `/command`”的分支。

## 为什么不用
因为这里的入口已经混进了图片、附件、桥接来源、隐藏 meta 提示和命令执行模型。先粗暴分支，后面就得不断补丁。

## 代价
入口层明显更重，读起来也更不像一个简单聊天程序；但换来的是后面的主循环接到的是已经被系统承认过的正式事件，而不是一团原始字符串。
