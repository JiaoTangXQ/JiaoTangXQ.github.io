# 面试题：用户输入一句话，Claude Code 为什么不直接发给模型？

**面试官**：假设你在设计一个 AI 编程助手，用户在输入框里打了一段话，最简单的实现是什么？

**候选人**：直接拿字符串，加个 system prompt，发 API 请求。

**面试官**：那 Claude Code 为什么没这么做？

---

这是第三卷要回答的核心问题。不是「怎么最快把话转给模型」，而是「一条输入为什么必须先被系统整理、分流、翻译，才能成为真正的会话入口」。

## 一条输入里藏了多少东西

用户眼里的输入框是一行文字。系统眼里，同一条输入可能混着：

- 粘贴进来的截图（还没缩放，还没落盘）
- 路径引用、@提及、IDE 选中的代码段
- 以 `/` 开头的 slash 命令（可能是内建命令、skill、插件或 MCP 能力）
- 来自 bridge 的远端触发（移动端发来的命令，未必能在本地执行）
- `ultraplan` 关键词（需要先被替换再路由到子命令）
- hooks 产出的附加上下文（外部制度可能在这里阻断或补充输入）

把这团东西原样丢给模型，等于把「运行时该做的分拣工作」外包给语言模型去猜。Claude Code 拒绝这样做。

## 这卷的地图

入口函数是 `processUserInput()`，位于 `src/utils/processUserInput/processUserInput.ts`。它的执行顺序大致是：

```
原始输入
  ↓ 图片块规范化 + 缩放
  ↓ 粘贴图片落盘 + 并行 resize
  ↓ bridge 来源判断（isBridgeSafeCommand）
  ↓ ultraplan 关键词检测（preExpansionInput）
  ↓ 附件提取（getAttachmentMessages）
  ↓ bash / slash / 普通 prompt 分流
  ↓ UserPromptSubmit hooks 执行
  ↓ 返回 { messages, shouldQuery, allowedTools, model, effort }
```

这条链之后，`QueryEngine.submitMessage()` 才接手，决定要不要真正发起本轮 query。

## 各章导览

前半段（03-01 到 03-03）讲输入如何先被整理成系统认识的事件：

- **03-01**：`processUserInput` 是分拣台，不是转发器。输入在这里被翻译成系统可接手的会话材料。
- **03-02**：图片的特殊待遇——为什么要先存盘、缩放，再生成 `isMeta` 元信息消息。附件为什么独立出来而不是塞进原文。
- **03-03**：slash 命令为什么不是第一道门，未知命令为什么先要判断它是不是路径，进入 slash 分支后又发生哪一类翻译。

后半段（03-04 到 03-07）讲整理完的输入如何进入命令世界：

- **03-04**：`Command` 统一抽象——内建命令、skills、插件、MCP 为什么要先变成同一种语言。
- **03-05**：三种执行模型（`prompt`、`local`、`local-jsx`）为什么在统一外观下保持物理分离。
- **03-06**：`prompt` 命令如何组装一轮新的会话工作条件——正文、工具许可、hooks、附件一起装进去。
- **03-07**：`context: fork` 通往任务系统，`REMOTE_SAFE_COMMANDS` 和 `BRIDGE_SAFE_COMMANDS` 如何守住不同入口边界。

## 真正的问题

这卷的判断用一句话说完：**一句话不是直接去聊天，而是先被翻译成系统入口。**

真正成熟的 agent，不会把入口治理外包给模型。`processUserInput()` 的存在，正是在让系统替自己的入口负责。

---

*面试指导：被问到「Claude Code 的输入处理」时，不要直接跳到 API 调用。从「原始输入的多样性」和「系统入口的职责」两个角度打开，再顺着 processUserInput → 分流 → QueryEngine 这条链往下讲，才是答对方向。*
