# 六层规则来源怎么汇成一份 ToolPermissionContext？

面试官追问：”你说权限规则来自多个来源。具体是哪些来源？它们的优先级怎么排？如果 userSettings 说允许，policySettings 说拒绝，谁赢？”

## 六层来源的全景

`PERMISSION_RULE_SOURCES` 常量定义了权限规则的所有合法来源：

```typescript
// permissions.ts
const PERMISSION_RULE_SOURCES = [
  ...SETTING_SOURCES,    // userSettings, projectSettings, localSettings, policySettings, flagSettings
  'cliArg',              // --allowed-tools 命令行参数
  'command',             // /tools 等交互式命令临时授权
  'session',             // SDK session 级别的临时规则
] as const satisfies readonly PermissionRuleSource[]
```

`SETTING_SOURCES` 本身是另一个常量，展开后包含：`userSettings`（`~/.claude/settings.json`）、`projectSettings`（`.claude.json`）、`localSettings`（`.claude.local.json`）、`policySettings`（MDM/企业管控）、`flagSettings`（运行时 flag 传入的临时 settings 文件）。

## ToolPermissionContext 的结构

六层来源的规则最终都归并进一个对象：

```typescript
export type ToolPermissionContext = DeepImmutable<{
  mode: PermissionMode
  alwaysAllowRules: ToolPermissionRulesBySource   // 允许规则，按来源分组
  alwaysDenyRules: ToolPermissionRulesBySource    // 拒绝规则，按来源分组
  alwaysAskRules: ToolPermissionRulesBySource     // 强制询问规则，按来源分组
  isBypassPermissionsModeAvailable: boolean
  isAutoModeAvailable?: boolean
  strippedDangerousRules?: ToolPermissionRulesBySource  // auto 模式剥离的危险规则暂存
  // ...
}>
```

`ToolPermissionRulesBySource` 是一个以 `PermissionRuleSource` 为 key、字符串数组为 value 的 Record。每条规则字符串形如 `”Bash(ls *)”` 或 `”Read”`。

**关键点**：规则是**按来源分组存储**的，不是扁平合并。这意味着系统在查询时可以精确知道每条规则的来源，这对”这条规则能不能被删除/覆盖”的判断至关重要。

## 规则加载：loadAllPermissionRulesFromDisk

```typescript
// permissionsLoader.ts
export function loadAllPermissionRulesFromDisk(): PermissionRule[] {
  const rules: PermissionRule[] = []
  for (const source of getEnabledSettingSources()) {
    const settings = getSettingsForSource(source)
    rules.push(...settingsJsonToRules(settings, source))
  }
  return rules
}
```

`settingsJsonToRules` 把 settings JSON 里的 `permissions.allow`、`permissions.deny`、`permissions.ask` 数组转成 `PermissionRule` 对象，每个对象包含 `source`、`ruleBehavior`、`ruleValue` 三个字段。

## 规则查询：从 context 里读，而不是现场读磁盘

关键设计决策：**所有运行时权限查询都从 `ToolPermissionContext` 这个快照里读，不再回头访问磁盘或设置缓存**。

```typescript
export function getAllowRules(context: ToolPermissionContext): PermissionRule[] {
  return PERMISSION_RULE_SOURCES.flatMap(source =>
    (context.alwaysAllowRules[source] || []).map(ruleString => ({
      source,
      ruleBehavior: 'allow',
      ruleValue: permissionRuleValueFromString(ruleString),
    })),
  )
}
```

`getAllowRules`、`getDenyRules`、`getAskRules` 都是这个模式：遍历所有来源，从 context 对象里展开规则字符串，解析成结构化对象返回。

## 优先级怎么排

六层来源没有简单的线性优先级。优先级规则分情况：

**deny 规则优先于 allow 规则**（在同一来源内）。如果 `userSettings` 同时有 allow 和 deny 匹配同一个工具，deny 赢。

**policySettings 可以锁定其他来源**。`shouldAllowManagedPermissionRulesOnly()` 为 true 时，只有 `policySettings` 的规则生效，其他来源的 allow 规则全部忽略。

**cliArg 的危险规则在 auto 模式下被剥离**。这意味着 `--allowed-tools Bash(*)` 这样的 CLI 参数在 auto 模式下失效——系统会主动扫描 CLI 规则并标记危险的。

**session 规则是临时的**，会话结束后消失，不会持久化。

## 为什么要用快照而不是实时查询

如果每次权限检查都读最新的 settings 文件，会有两个问题：

1. **同一次工具调用的不同阶段看到不同规则**：比如工具装配时读到 allow，运行时文件被改了又读到 deny，行为不一致。

2. **并发问题**：多个子 agent 同时调用工具时，如果它们都去竞争读磁盘，竞态条件会导致规则状态不稳定。

快照模型把”规则是什么”和”现在该不该执行”解耦：前者在会话初始化时确定，后者在每次调用时针对快照做判断。

## 这道题考什么

**考点一**：能否准确说出六层来源的名称和对应的存储位置（`~/.claude/settings.json`、`.claude.json` 等）。

**考点二**：理解规则按来源分组存储（而不是扁平合并）的意义——这是后续”删除规则”和”覆盖规则”功能的基础。

**考点三**：能解释快照模型的取舍：更一致的读取，代价是快照可能短暂过期（settings 变化后 changeDetector 有 1-1.5 秒的稳定等待窗口才触发刷新）。
