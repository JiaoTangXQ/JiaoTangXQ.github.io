# autocompact 的触发阈值和整理节奏是怎么设计的？

面试常见变体：「你怎么判断什么时候该压缩上下文？压缩太早浪费，压缩太晚会失控。」

Claude Code 的答案藏在 `compactWarningState.ts` 和 `compactWarningHook.ts` 两个文件里——压缩不是单一阈值触发的，而是分成「警告期」和「触发期」两个阶段。

## 警告状态机

`compactWarningState.ts` 维护一个紧凑的状态：当前上下文的 token 使用量是否已经进入危险区域。这个状态会在每次 API 响应返回时更新，因为模型的响应里包含了 `usage` 字段：

```typescript
// usage 字段来自 API 响应
type NonNullableUsage = {
  input_tokens: number
  output_tokens: number
  cache_creation_input_tokens: number
  cache_read_input_tokens: number
}
```

系统把这些数字累加，和模型的上下文窗口上限对比，计算出「已用百分比」。

## 克制的整理时机

`compactWarningHook.ts` 是真正做决策的地方。它不会在上下文超过某个百分比时立刻触发 compact，而是按照这个逻辑：

1. **先警告，不打断**：在上下文达到警告线时，在状态栏显示提示，但不打断当前回合
2. **回合结束才检查**：只在主循环的 `stopHooks` 阶段（一个回合完全结束后）才评估是否需要 compact
3. **连续性优先于 token 效率**：宁可让上下文稍微超一点，也不要在工具调用进行中突然插入整理操作

这和「快满了就截断」的策略形成鲜明对比。截断的危险在于：如果恰好截断的那段消息是当前任务的关键上下文（比如「用户让我修改的是文件 X 的第 Y 行」），之后的所有操作都会偏向。

## 整理的「自动化克制」

有一个有趣的设计细节：即使 autocompact 被启用，Claude Code 也不会在没有用户输入的情况下无限制地自动触发整理。这个「自动化克制」背后的逻辑是：

- compact 会消耗额外的 token（调用模型生成摘要）
- 频繁 compact 意味着频繁的 API 费用，用户可能没意识到
- compact 质量取决于摘要的好坏，自动触发的 compact 不如用户主动触发的 compact 可控

因此，autocompact 的「auto」不是「随时自动」，而是「在适当时机自动判断是否需要」。用户可以用 `/compact` 命令主动触发，也可以在设置里配置上下文压缩策略。

## 整理后的上下文连续性检测

compact 执行后，`compact.ts` 里有一段 `analyzeContext` 调用：

```typescript
const analysis = analyzeContext(messages)
const metrics = tokenStatsToStatsigMetrics(analysis)
// 上报整理后的上下文质量指标
```

这个分析的目的是检测「compact 后的上下文是否仍然能支撑继续工作」。如果摘要后关键信息损失太多（比如文件路径、变量名、当前任务状态），compact 实际上伤害了工作质量而不是帮助。

## 上下文使用量的实时展示

用户在 Claude Code 界面上能看到当前上下文的使用情况（token 计数）。这个数据来自 `cost-tracker.ts` 对每次 API 调用的 usage 累计，再通过状态栏组件展示。

设计意图是让用户有感知：「我现在用了多少上下文，还剩多少余量，是否应该主动 compact 一下」。这把整理的控制权部分还给了用户，而不是完全由系统自动决策。

## 面试答题要点

上下文管理的阈值设计问题，关键考察点：

1. **警告 ≠ 行动**：检测到上下文压力时，先警告再决策，不要立刻触发侵入性操作
2. **时机先于策略**：再好的整理策略，时机不对也会造成伤害
3. **费用可见性**：压缩本身有成本（token），不能对用户隐藏
4. **质量反馈循环**：整理后要验证上下文质量，确认「整理没有造成关键信息丢失」
5. **用户主控 vs 自动化**：把部分控制权还给用户，特别是在副作用显著的操作上

追问「什么情况下 compact 会失败」：forked agent 调用失败（网络、模型响应异常）；摘要质量太差（模型对长文本摘要效果不稳定）；compact 后 token 并没有减少多少（高度压缩的代码/数据）。
