## 教程的局限

教程告诉你"怎么做"。文档告诉你"是什么"。

但它们都不告诉你"为什么这么做而不是那么做" —— 而这恰恰是工程能力的核心。

源码告诉你一切。每一行代码都是一个决策的痕迹。为什么选择这个数据结构？为什么这里有个 try-catch？为什么这个函数是这样拆分的？

## 如何读源码

读源码不是从第一行读到最后一行。有效的读法：

1. **带着问题去读** —— "React 的 useState 是怎么保持顺序的？"比"我要读 React 源码"有效一百倍。
2. **从入口开始** —— 找到你关心的功能的入口点，然后顺着调用链走。
3. **忽略你不关心的部分** —— 源码里大量是错误处理、兼容性代码、性能优化。第一遍只看主路径。
4. **做笔记** —— 画调用图，记录你的理解和困惑。

## 推荐的阅读对象

对于前端开发者：

- **Vite 的 dev server** —— 理解现代构建工具的核心设计
- **Zustand** —— 极简状态管理，整个库只有几百行
- **SWR** —— 数据获取的优雅抽象，缓存策略很精巧
- **Radix UI** —— 无头组件的典范实现

选小而精的库开始。不要一上来就读 React 核心 —— 那是等你有了足够经验之后的事。

## 一个意外收获

定期读源码有一个意外的好处：你会开始在自己的代码里做更好的决策。

不是因为你记住了某个实现细节，而是因为你见过足够多的好代码，形成了直觉。这和品味训练是一样的道理 —— 见得多了，自然知道什么是好的。
