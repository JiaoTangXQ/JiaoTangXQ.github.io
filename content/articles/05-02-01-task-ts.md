# 为什么 Task.ts 先统一任务种类和终态

`Task.ts` 像一部小宪法，因为它先规定哪些任务算同一个世界里的公民，以及什么状态算真正结束。

## 实现链

`src/Task.ts` 先列出 `TaskType`，包括 `local_agent`、`remote_agent`、`in_process_teammate` 等，再定义 `TaskStatus` 为 `pending/running/completed/failed/killed`。同时还给出 `isTerminalTaskStatus()`，明确只有 `completed`、`failed`、`killed` 才是终态。

这个函数后面不是摆设。任务回收、禁止继续注入消息、终态清理、面板驱逐都依赖它，说明“什么叫结束”在系统里是被中心化定义的。

## 普通做法

普通做法常见两种。一种是每类任务自己定义状态，比如某个模块有 `done`，另一个模块有 `cancelled`；另一种是先只看布尔值，如 `isRunning`、`isFinished`。

早期这么做很快，但系统一大就会开始互相翻译。

## 为什么不用

Claude Code 没让每个任务自己发明状态，因为任务系统的价值就在于跨任务统一治理。只要终态不统一，回收逻辑、通知逻辑、UI 逻辑和消息注入保护就得各写一遍。

它先把类型和终态钉死，等于先保证“虽然你们干的活不同，但什么时候算开始、什么时候算结束，大家必须说同一种话”。

## 代价

代价是扩展新任务类型时要先进宪法。开发者不能随手在某个子模块里临时加一套状态名词，否则很快会和中心框架打架。这让扩展速度稍慢，但换来一致性。
