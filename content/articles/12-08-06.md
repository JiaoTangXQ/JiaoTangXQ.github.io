# 权限模式先验校验后再注入，说明浏览器能力也不绕开原制度

浏览器侧的权限模式不是随便从环境变量里读个字符串就塞进去。Claude Code 先拿 `CLAUDE_CHROME_PERMISSION_MODE` 做窄枚举校验，只有落在少数合法模式里，才会变成 `initialPermissionMode` 注入浏览器运行时；不合法就警告并忽略。

这说明浏览器能力虽然长在系统边缘，但并没有绕开原制度偷偷长大。它还是要先承认“哪些模式是真的、哪些只是胡说”，再允许自己开工。优雅的外延能力，不是边界一远就随便，而是边界越远越先收口。

## 实现链
`createChromeContext()` 会先用 `isPermissionMode()` 校验 `CLAUDE_CHROME_PERMISSION_MODE`，只有合法值才注入初始权限模式；`usePromptsFromClaudeInChrome()` 又把宿主的 tool permission mode 同步翻译成扩展侧 `ask` 或 `skip_all_permission_checks`。

## 普通做法
普通做法是让浏览器扩展自己维护一套独立权限状态，CLI 不同步也无所谓。

## 为什么不用
Claude Code 不让两边各玩各的，因为浏览器能力也是工具能力，不能在宿主说“要问一遍”时，扩展侧偷偷处于跳过检查状态。

## 代价
代价是两边都要接受一层权限术语翻译，调试时要同时理解宿主名词和扩展名词。
