# 为什么技能正文、工具许可和 hooks 会一起被送进去

如果把一个 prompt 命令理解成“往会话里塞一段提示词”，那它当然也能工作，但会很浅。Claude Code 对 prompt 命令的理解明显更深一些：它送进当前会话的，不只是技能正文，还包括这次技能附带的工具许可、相关附件，甚至可能连 hooks 都一起注册进去。

`getMessagesForPromptSlashCommand()` 里这件事写得很完整。系统先拿到 `getPromptForCommand()` 生成的正文，再根据技能来源决定是否注册 hooks，记录这次技能调用，以便后面压缩和恢复时还认得它；随后解析 `allowedTools`，把额外许可做成 `command_permissions` 附件；再从技能正文里提取附件世界，最后把这些东西一起组装成会话消息。换句话说，这不是“加一段提示词”，而是在当前会话里临时装入一种新的工作方式。

这正是 prompt 命令为什么会成为整套命令系统灵魂的原因。它能改变的不是一句回复，而是这一轮之后系统与模型共同面对的行动空间。技能正文告诉模型该怎样想，工具许可告诉系统这次能怎样做，hooks 则让外部制度有机会一起介入。三者放在一起，才是一种完整的工作模式。

很多人容易把技能想成“更长的提示词模板”。Claude Code 在这里的厉害之处，是它把技能提升成了会话级制度装配。这样做当然更复杂，但也正因为复杂，它才配得上“工作方式”这个名字。

## 实现链
`processSlashCommand.tsx` 不只加载技能正文，还会把 `allowedTools`、权限设定、hook 产出的附加上下文、附件消息一起并入 prompt 命令结果，再交给后面的 query。

## 普通做法
很多 skill 系统只把正文 prompt 给模型，至于工具权限和 hook 环境由外层默认继承。

## 为什么不用
因为这里的 skill 不是“只换话术”，而是要临时改写行动空间。如果正文和权限、hooks 不同步，模型就会在错误的制度环境里工作。

## 代价
一次 prompt 命令会牵动比想象中更多的环境变量和状态，调试也更难；但这样至少不会出现“文案说能做，工具其实被禁了”的假现场。
