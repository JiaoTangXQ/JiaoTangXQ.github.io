# 面试题：同一个命令名，在本地、远端、bridge 三个入口下分别是什么？

## 三组对比

选三个典型命令，看它们在不同入口下的命运：

### `/config`（local-jsx 命令，配置面板）

**本地终端**：
1. `processUserInput` → `effectiveSkipSlash = false`（本地输入，无 bridge 标记）
2. 进入 slash 分支
3. `getMessagesForSlashCommand('config', ...)` → `command.type === 'local-jsx'`
4. 加载配置面板 JSX，挂到终端，等用户交互完成

结果：配置面板出现，用户可以操作

**远端模式（CCR）**：
1. `REPL.tsx` 在 `handleRemoteInit` 时过滤命令列表
2. `config` 不在 `REMOTE_SAFE_COMMANDS` 里，从命令列表移除
3. 用户在远端 REPL 的补全列表里看不到 `/config`
4. 即使强行输入，`hasCommand('config', context.options.commands)` 返回 false（命令已被移除）

结果：命令不可见，强行输入当普通文本处理

**Bridge 输入（手机/web 发来的）**：
1. `processUserInput` → `bridgeOrigin = true`，`skipSlashCommands = true`
2. bridge 来的 `/config`：`isBridgeSafeCommand(configCmd)` → `cmd.type === 'local-jsx'` → `return false`
3. 返回错误消息：`/config isn't available over Remote Control.`

结果：被拦截，bridge 客户端看到友好的错误提示

---

### `/review`（prompt 命令，代码 review skill）

**本地终端**：
1. 进入 slash 分支
2. `getMessagesForSlashCommand('review', ...)` → `command.type === 'prompt'`
3. 技能内容注入，`shouldQuery: true`，主循环推进
4. 模型按 review 指南分析代码

结果：完整的代码 review 推理

**远端模式**：
1. `/review` 不在 `REMOTE_SAFE_COMMANDS` 里
2. 如果远端 CCR 没有包含这个命令，它在远端不可见

注意：这里有微妙之处。`REMOTE_SAFE_COMMANDS` 只管本地 TUI 命令的可见性，远端会话本身可能有自己的命令集（通过 CCR 协议传来）。`/review` 作为 skill，如果 CCR 服务端允许，可能仍然可用。

**Bridge 输入**：
1. `bridgeOrigin = true`，输入 `/review`
2. `isBridgeSafeCommand(reviewCmd)` → `cmd.type === 'prompt'` → `return true`
3. 命令正常执行：技能内容注入，模型推理，结果流回 bridge 客户端

结果：完全正常执行——`prompt` 命令在 bridge 里一律允许

---

### `/compact`（local 命令，压缩历史）

**本地终端**：
1. 进入 slash 分支
2. `command.type === 'local'`，直接本地执行
3. 压缩消息历史，返回 compact 结果

**远端模式**：
1. `compact` 不在 `REMOTE_SAFE_COMMANDS` 里，在远端 REPL 不可见

**Bridge 输入**：
1. `bridgeOrigin = true`，输入 `/compact`
2. `isBridgeSafeCommand(compactCmd)` → `cmd.type === 'local'` → 检查 `BRIDGE_SAFE_COMMANDS`
3. `compact` 在 `BRIDGE_SAFE_COMMANDS` 里 → `return true`
4. 命令正常执行：本地压缩，结果（displayText）流回 bridge 客户端

结果：Bridge 允许 `/compact`，但远端 REPL 不显示它

这是一个有趣的不对称：`/compact` 在远端 REPL 不可见（因为它不影响 TUI 状态，放在远端显示没有意义），但 bridge 来的 `/compact` 可以执行（因为它产生文本输出，可以流回 bridge 客户端）。

## 判断链总结

```
输入字符串（以 / 开头）
  ↓
是否来自 bridge？
  是 → isBridgeSafeCommand？
      local-jsx → 拦截，返回错误
      prompt → 允许执行
      local → BRIDGE_SAFE_COMMANDS? 是→允许 否→拦截
  否 → 命令在当前模式的 commands 列表里？
      是（本地模式）→ 正常执行
      否（远端模式已过滤）→ 当普通文本处理
```

## 为什么要三层判断

只有一层判断（比如统一白名单）会出现什么问题？

- 如果用 `REMOTE_SAFE_COMMANDS` 来决定 bridge 安全，`/files` 就会被错误排除（它不在 `REMOTE_SAFE_COMMANDS` 里，但 bridge 可以执行）
- 如果用 `BRIDGE_SAFE_COMMANDS` 来决定远端 REPL 显示，`/theme` 就会被错误排除（它在 REMOTE_SAFE_COMMANDS 里但不在 BRIDGE_SAFE_COMMANDS 里）

三种入口对「什么是可接受的」有不同的标准，就必须有三套独立的判断。强行统一只会产生漏判和误判。

---

*面试指导：被问到「你们系统在不同环境下命令行为一致吗」时，不要直接说「一致」或「不一致」。用「入口语境不同导致同一命令名被映射到不同能力」来回答，然后举 /compact 在远端不可见但在 bridge 可执行这个不对称的例子，说明这是有意为之的，不是设计缺陷。*
