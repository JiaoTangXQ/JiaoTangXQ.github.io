# stopTask：Claude Code 如何统一停止七种不同类型的任务？

Claude Code 的任务系统有七种任务类型，每种类型的生命周期管理方式不同。但停止任务这个操作被统一封装在 `tasks/stopTask.ts` 里——`TaskStopTool`（LLM 调用）和 SDK 的 `stop_task` 控制请求都调用同一个函数。

## TaskState 的七种类型

```typescript
export type TaskState =
  | LocalShellTaskState      // 本地 bash/shell 命令
  | LocalAgentTaskState      // 本地 sub-agent
  | RemoteAgentTaskState     // 远程 agent（通过网络）
  | InProcessTeammateTaskState // 队友（in-process）
  | LocalWorkflowTaskState   // 本地工作流
  | MonitorMcpTaskState      // MCP 监控任务
  | DreamTaskState           // Dream 任务
```

七种类型来自不同的执行场景：
- Shell 任务：用户发起的 bash 命令，最简单的一种
- 本地 Agent 任务：由 `AgentTool` 启动的子代理
- 远程 Agent 任务：通过网络连接的远程子代理（如 CCR）
- 队友任务：Coordinator 模式下的 Worker
- 工作流任务：预定义步骤序列
- MCP 监控任务：监控 MCP 服务器状态
- Dream 任务：自动化/计划任务

## StopTaskError：结构化错误码

```typescript
export class StopTaskError extends Error {
  constructor(
    message: string,
    public readonly code: 'not_found' | 'not_running' | 'unsupported_type',
  ) {
    super(message)
    this.name = 'StopTaskError'
  }
}
```

三种失败情况：
- `not_found`：任务 ID 不存在（可能已经清理了）
- `not_running`：任务存在但不在运行状态
- `unsupported_type`：任务类型没有对应的 `kill` 实现

有 `code` 字段而不是多个不同的 Error 类，让调用者可以用 `instanceof` + `error.code` 做精确的条件处理，比多个错误类型更简洁。

## stopTask 的执行流程

```typescript
export async function stopTask(
  taskId: string,
  context: StopTaskContext,
): Promise<StopTaskResult> {
  const task = appState.tasks?.[taskId] as TaskStateBase | undefined

  if (!task) throw new StopTaskError(`No task found with ID: ${taskId}`, 'not_found')
  if (task.status !== 'running') throw new StopTaskError(`...`, 'not_running')

  const taskImpl = getTaskByType(task.type)
  if (!taskImpl) throw new StopTaskError(`...`, 'unsupported_type')

  await taskImpl.kill(taskId, setAppState)
  // ...
}
```

`getTaskByType` 返回任务类型的实现对象——每种 `TaskState` 类型都有对应的实现，包含 `kill` 方法。`kill` 是 async 的，因为不同类型的停止操作有不同的时机（有些需要等网络响应，有些是本地进程信号）。

## Bash 任务 vs. Agent 任务：通知抑制的差异

停止后的处理是两种任务类型最大的区别：

```typescript
if (isLocalShellTask(task)) {
  let suppressed = false
  setAppState(prev => {
    const prevTask = prev.tasks[taskId]
    if (!prevTask || prevTask.notified) return prev
    suppressed = true
    return { ...prev, tasks: { ...prev.tasks, [taskId]: { ...prevTask, notified: true } } }
  })
  if (suppressed) {
    emitTaskTerminatedSdk(taskId, 'stopped', {
      toolUseId: task.toolUseId,
      summary: task.description,
    })
  }
}
```

**Bash 任务**：停止后抑制 "exit code 137" 的通知（137 = 128 + 9，是 SIGKILL 的标准退出码）。这是噪声——用户自己停的，不需要看到"命令被信号终止"的报错。但 SDK 消费者还是需要知道任务结束了，所以通过 `emitTaskTerminatedSdk` 直接发出事件（绕过常规的 XML 通知路径）。

**Agent 任务**：不抑制通知。原因在注释里解释得很清楚：Agent 任务停止时，`AbortError` catch 块会发送包含 `extractPartialResult(agentMessages)` 的通知——这里面有子 agent 已经完成的工作成果，是有价值的内容，不应该被抑制。

这个差异体现了一个设计原则：**通知的价值取决于内容，不是类型**。停止 bash 命令没什么有用信息，停止 agent 有部分结果值得传递。

## isBackgroundTask：区分前台和后台

```typescript
export function isBackgroundTask(task: TaskState): task is BackgroundTaskState {
  if (task.status !== 'running' && task.status !== 'pending') {
    return false
  }
  if ('isBackgrounded' in task && task.isBackgrounded === false) {
    return false
  }
  return true
}
```

这个 type guard 用于决定任务是否显示在"后台任务指示器"里。任务满足后台条件需要：
1. 状态是 `running` 或 `pending`（已完成/失败的不算）
2. 没有明确标记为"前台"（`isBackgrounded === false`）

`'isBackgrounded' in task` 的检查说明不是所有任务类型都有这个字段——这是 TypeScript 里处理可选字段的标准方式，通过 `in` 操作符先检查字段存在性。

## StopTaskContext 的依赖注入

```typescript
type StopTaskContext = {
  getAppState: () => AppState
  setAppState: (f: (prev: AppState) => AppState) => void
}
```

`stopTask` 通过 Context 对象注入 `getAppState`/`setAppState`，而不是直接访问全局状态。这让函数可以在测试中注入 mock 状态，不依赖全局单例。

`setAppState` 的类型是函数而不是直接值——这是不可变状态更新的标准模式，防止并发修改产生竞争条件。

## 面试角度

stopTask 的设计展示了多态性管理的工程权衡：

1. **统一接口掩盖多态实现**：七种任务类型有七种不同的 `kill` 方法，但调用者只需要知道 `stopTask(taskId, context)`
2. **错误码而不是异常类型**：`StopTaskError.code` 比三个不同的 Error 子类更简洁，足够表达区分逻辑
3. **副作用的条件性**：通知抑制不是无条件的，而是根据任务类型决定——这个判断逻辑是业务规则，不是实现细节
4. **SDK 事件和 UI 通知的解耦**：`emitTaskTerminatedSdk` 直接给 SDK 消费者发事件，和 UI 的 XML 通知路径是独立的两条管道
