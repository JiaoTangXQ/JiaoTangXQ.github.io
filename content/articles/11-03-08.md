# 工程设计：useAppStateMaybeOutsideOfProvider 为什么存在？什么时候该用它？

两个 hook，一个严格，一个宽松：

```typescript
// 严格版本：Provider 外抛错
function useAppStore(): AppStateStore {
  const store = useContext(AppStoreContext)
  if (!store) {
    throw new ReferenceError(
      'useAppState/useSetAppState cannot be called outside of an <AppStateProvider />'
    )
  }
  return store
}

// 宽松版本：Provider 外返回 undefined
export function useAppStateMaybeOutsideOfProvider<T>(selector: (state: AppState) => T) {
  const store = useContext(AppStoreContext)
  const get = () => store ? selector(store.getState()) : undefined
  return useSyncExternalStore(
    store ? store.subscribe : NOOP_SUBSCRIBE,
    get,
    get
  )
}
```

为什么同一件事有两个版本？

---

## 严格版本的价值

`useAppState` 背后的 `useAppStore` 会检查 context，没有就抛 `ReferenceError`。

这是有意为之的严格性：对于应该在 Provider 里运行的组件，如果它在 Provider 外被渲染了，立刻暴露问题，不要静默失败。

静默失败的问题在于，组件会用 `undefined` 渲染，可能展示错误内容，也可能产生 `NaN`、空字符串等难以调试的二次问题。调试时，你看到的错误现象（某个按钮显示异常）距离真实原因（组件在 Provider 外渲染了）可能有好几层间接性。

抛错把问题提前到「发生的那一刻」，错误信息直接告诉你发生了什么，消除了间接性。

---

## 宽松版本的场景

什么情况下需要在可能没有 `AppStateProvider` 的地方读取状态？

**错误边界**

错误边界（Error Boundary）通常被放在组件树的外层，甚至在 `AppStateProvider` 之外，因为它的职责是捕获整棵树里的异常。如果错误边界里想展示一些有用的上下文信息（比如当前模型、当前任务 ID），但它所在的位置可能比 `AppStateProvider` 更外层，就需要宽松版本。

**可复用的 UI 组件库**

某些 UI 组件需要读取 AppState 来做样式或行为决定，但这些组件可能被用在 Storybook、测试环境、或者某个没有完整 Provider 树的独立渲染上下文里。宽松版本让它们在缺少 Provider 时优雅降级，而不是崩溃。

**测试和 Storybook**

在单元测试里单独测一个组件时，通常不想搭建完整的 Provider 树。如果组件用的是宽松版本，不加 Provider 也能渲染（只是 AppState 相关的状态是 `undefined`），测试的 setup 代码可以更简单。

**DevBar 和调试工具**

开发时的调试面板可能需要在各种上下文里渲染，包括没有完整 Provider 的 iframe 或独立窗口。宽松版本允许调试工具在这些环境里正常工作（降级显示）而不是崩溃。

## NOOP_SUBSCRIBE 的设计

```typescript
const NOOP_SUBSCRIBE = () => () => {}
```

当 store 不存在时，传给 `useSyncExternalStore` 的 subscribe 是 `NOOP_SUBSCRIBE`。

为什么不直接传 `null` 或 `undefined`？

`useSyncExternalStore` 的类型签名要求 subscribe 必须是一个函数，且必须返回一个取消订阅函数：

```typescript
subscribe: (onStoreChange: () => void) => () => void
```

`NOOP_SUBSCRIBE` 满足这个契约：
- 它是一个函数（满足「必须是函数」）
- 它返回 `() => {}`（满足「必须返回取消订阅函数」）
- 调用时什么都不做（「Noop」的含义）
- 返回的取消订阅函数调用时也什么都不做

这是一个「安全占位符」：在没有实际 store 可以订阅的情况下，满足 API 契约，让 `useSyncExternalStore` 正常运行，只是不会收到任何更新通知。

---

## 用 undefined 代表「在 Provider 外」

宽松版本的返回类型是 `T | undefined`，其中 `undefined` 专门表示「当前不在 Provider 里，没有状态可读」。

这是一个重要的设计选择，和「返回默认值」不同：

假设某个字段是布尔型，「在 Provider 外」时如果返回 `false`，调用方无法区分两种情况：
- 「Provider 里的值确实是 false」
- 「根本没有 Provider」

这两种情况在业务上含义完全不同。前者是真实的业务状态，后者是「不在正常的运行上下文里」。

返回 `undefined` 清晰地表达了第二种情况，让调用方可以做出正确的决策：

```typescript
const model = useAppStateMaybeOutsideOfProvider(s => s.mainLoopModel)

if (model === undefined) {
  // 我们在 Provider 外，没有模型信息
  return <span>unknown</span>
}
return <span>{model}</span>
```

## 选择哪个版本的判断标准

一个实用的判断方式：

**问题：如果这个组件被渲染在 Provider 外，是 bug 还是合理的场景？**

- 是 bug（这个组件就应该在 Provider 里，在 Provider 外就是配置错误） → 用严格版本 `useAppState`，让错误尽早暴露
- 是合理的场景（这个组件可能在多种上下文里使用，有些没有 Provider） → 用宽松版本 `useAppStateMaybeOutsideOfProvider`，优雅降级

大多数业务组件属于前者。只有少数需要跨多种渲染上下文工作的组件属于后者。

## 面试指导

**为什么不在所有地方都用宽松版本？**

宽松版本的代价是调用方必须处理 `undefined`。每个用了宽松版本的组件都要有「没有 Provider 时怎么渲染」的逻辑，这增加了代码量和测试范围。更重要的是，宽松版本掩盖了配置错误——如果某个组件本来应该在 Provider 里但被误放到了外面，宽松版本会让它静默显示错误状态，而不是立刻暴露问题。

**NOOP_SUBSCRIBE 和 useRef 不更新的关系**

`NOOP_SUBSCRIBE` 返回的是一个不订阅任何东西的 subscribe 函数，这意味着当 store 不存在时，组件永远不会收到「store 变化了」的通知，也不会重渲染。这是正确的：如果没有 store，当然没有 store 变化，组件的状态（`undefined`）是稳定的。

**如果 Provider 在运行时被卸载了，宽松版本会怎样？**

如果 `AppStateProvider` 被动态卸载（虽然正常不应该这样做），`AppStoreContext` 的值会变回默认值 `null`，`useContext(AppStoreContext)` 返回 `null`，宽松版本会开始返回 `undefined`，subscribe 换成 `NOOP_SUBSCRIBE`。组件会重渲染（因为 context 值变了），但不会崩溃，显示的是「无 Provider 状态」。
