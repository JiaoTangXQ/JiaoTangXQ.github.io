# 架构分析：Claude in Chrome 为什么选择用 StdioServerTransport 而不是自定义协议？

## 设计浏览器桥时的协议选择

如果你要设计 Claude Code CLI 和 Chrome 扩展之间的通信协议，最直觉的想法可能是：

**方案一：定义一套私有 RPC 协议**
```
CLI ← JSON RPC over stdio ← Chrome Native Host ← Chrome Extension
```

优点：灵活，可以完全根据需求定制。
缺点：这套 RPC 协议的消费方（工具发现、权限、UI 渲染）都得适配这套新协议。

**方案二：直接用 MCP 协议**
```
CLI → MCP Client → MCP Server (Chrome tools) → Chrome Native Host → Chrome Extension
```

优点：复用所有已有的 MCP 基础设施。
缺点：需要把浏览器能力包装成 MCP 工具的形式。

Claude Code 选了方案二。`mcpServer.ts` 里：

```ts
export async function runClaudeInChromeMcpServer(): Promise<void> {
  enableConfigs()
  initializeAnalyticsSink()
  const context = createChromeContext()

  // 直接使用 MCP SDK 的标准 server
  const server = createClaudeForChromeMcpServer(context)
  const transport = new StdioServerTransport()

  // ...

  await server.connect(transport)
}
```

`StdioServerTransport` 是 `@modelcontextprotocol/sdk/server/stdio.js` 提供的标准实现——这是 MCP 协议的官方 Node.js SDK，和系统里其他 MCP server 使用完全相同的传输层。

## 复用 MCP 基础设施的具体好处

**工具发现不需要改变**：Claude Code 的工具发现逻辑扫描所有已注册的 MCP server，调用它们的 `ListTools`，把工具名加入可用工具列表。Chrome 工具通过同样的路径被发现，工具发现代码不需要知道这些工具来自浏览器。

**权限系统复用**：`allowedTools` 里放的是 `mcp__claude-in-chrome__*` 工具名，和 `mcp__computer-use__*` 完全相同的格式。权限系统不需要区分"这个工具是本地的还是浏览器的"。

**system prompt 注入机制复用**：`getChromeSystemPrompt()` 返回的 system prompt 通过 `setupClaudeInChrome()` 的返回值进入主循环，和其他 MCP server 的 system prompt 注入方式完全一样。

**UI 渲染框架复用**：`toolRendering.tsx` 处理 Chrome 工具的 UI 展示，但使用的是通用的工具调用 UI 框架，只是加了 Chrome 特有的标签（`[View Tab]` 链接等）。

## 私有协议方案会带来什么问题

如果选了方案一（私有 RPC 协议），需要额外构建：
- 一套 Chrome 工具的发现机制（不能用现有的 MCP 工具发现）
- Chrome 工具的权限系统（不能用现有的 `allowedTools` 机制）
- Chrome 工具的 system prompt 注入（不能用现有的 `setupXxx()` 约定）
- Chrome 工具调用的 UI 渲染（不能复用现有的工具 UI 框架）

每一项都是新的代码，每一项都是新的维护负担。

## StdioServerTransport：MCP 的基础传输层

`StdioServerTransport` 通过进程的 stdin/stdout 传输 JSON-RPC 消息，这正是 MCP 协议为 "subprocess MCP server" 场景设计的标准方式。

```
主进程
  └─ 启动 claude --claude-in-chrome-mcp（子进程）
       ├─ stdin: 接收 JSON-RPC 请求（ListTools、CallTool 等）
       └─ stdout: 发送 JSON-RPC 响应
```

这个子进程运行 `runClaudeInChromeMcpServer()`，通过 `StdioServerTransport` 和父进程（主 Claude Code 进程）通信。

父进程那边有 `MCP Client`，用 `StdioClientTransport` 和这个子进程通信——这是 MCP 协议里 client 和 stdio server 之间的标准连接方式。

整条通信链：
```
主进程 MCP Client ←→ stdio ←→ 子进程 runClaudeInChromeMcpServer()
                                         ↓
                               createChromeContext() 
                                         ↓
                               Chrome Native Host ←→ Chrome Extension
```

## 面试考察点

这道题考察的是**在系统扩展时如何最大化复用现有基础设施**，以及**对协议统一价值的判断**。

核心问题：**什么时候应该定义新协议，什么时候应该复用现有协议？**

定义新协议合理的场景：
- 现有协议的性能不满足要求
- 现有协议缺少必须的特性
- 现有协议的学习成本太高，会造成混乱

复用现有协议合理的场景：
- 现有协议功能足够
- 消费方已经懂这个协议
- 复用可以减少大量重复建设

Claude Code 的浏览器桥接满足后者：MCP 协议功能足够，所有消费方（工具发现、权限、UI）已经支持 MCP，复用可以避免大量重复建设。选择自定义协议在这种情况下是对现有积累的浪费。
