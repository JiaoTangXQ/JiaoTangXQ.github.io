# 自动启用和缓存检测：接入链如何认真对待真实用户现场

面试题版本：**"Claude Code 检测到浏览器扩展可用时，会自动建立连接吗？还是需要用户手动启用？自动检测的边界条件是什么——什么情况下自动连接是正确的，什么情况下不是？"**

## 自动启用的前提

完全自动的逻辑："每次启动时检测到扩展可用，自动连接"。这在大多数情况下正确，但有几种边界情况会出问题：

**用户故意禁用了 bridge**：用户上次明确选择"不使用浏览器集成"，这个选择应该被记住，不应该在每次启动时被自动覆盖。

**浏览器正在启动中**：Claude Code 和浏览器同时打开时，浏览器可能还没完全加载扩展，这时尝试连接会失败。应该等一段时间再重试，不是立即失败并提示用户"连接失败"。

**多用户或多会话环境**：终端服务器或多用户系统里，可能有多个 Claude Code 实例，不是每个都应该连接浏览器扩展。

**扩展版本不兼容**：检测到扩展存在，但版本太旧，不支持当前 Claude Code 所需的能力，自动连接会导致功能异常。

## 缓存能力列表的问题

浏览器 bridge 连接成功后，Claude Code 知道了扩展支持哪些能力（截图、DOM 操作、标签管理等）。这个能力列表可以缓存，避免每次都重新查询。

但缓存可能变得不准确：
- 用户在浏览器里更新了扩展，新版本有新能力或废弃了旧能力
- 不同标签页可能有不同的权限（某些页面的 DOM 不允许被读取）
- 浏览器更新可能改变某些 API 的行为

`bridgeDebug.ts` 里的缓存检测逻辑：
- 记录上次能力查询的时间戳
- 在连接重建时重新查询（而不是使用缓存）
- 在能力调用失败时标记对应能力为不可用，触发重新查询

## 启用时机的判断

即使决定自动连接，也需要选择合适的时机。`initReplBridge.ts` 里的初始化不是在应用启动的第一步，而是：

1. 主 UI 已经渲染（用户可以看到 Claude Code 界面）
2. 基本配置已加载（知道用户是否禁用了 bridge）
3. 主循环准备好了（避免 bridge 连接产生的消息在系统还没准备好时到达）

在这个时机之后才尝试 bridge 连接，是在系统稳定后才引入新的异步状态，而不是在启动过程中增加更多的不确定性。

## "认真对待真实用户现场"的含义

真实用户现场不是理想的 demo 环境：
- 用户可能有多个 Chrome profile，扩展只在某个 profile 里安装
- 用户可能有公司代理，影响 bridge 通信
- 用户可能在 VM 里运行 Claude Code，浏览器在宿主机里
- 用户的防火墙可能阻止 Native Messaging

"认真对待"意味着：
1. 检测而不是假设：不假设安装了就能用，要实际验证
2. 降级而不是失败：检测失败时，不影响 Claude Code 的核心功能，只是浏览器集成不可用
3. 诊断而不是沉默：出问题时，提供有意义的错误信息，而不是静默失败
4. 记住而不是重复：用户的选择（启用/禁用）被记录，不要每次都重新问

## 面试角度的总结

自动启用和缓存检测的设计细节，体现了接入链对真实用户环境的尊重。用户的现场不是一个可控的实验室，而是有各种边界情况的真实系统。Claude Code 的 bridge 实现通过细致的检测、合理的缓存失效、合适的启用时机，把"检测到扩展存在"这个简单事实转化为"可靠地建立并维护 bridge 连接"这个产品承诺。
