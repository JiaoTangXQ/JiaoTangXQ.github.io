# 第十三卷回收索引：外延执行里自动启用与缓存检测认真对待真实用户现场

## 实现链
外延执行里最先该被承认的，不是成功启用本身，  
而是真实用户现场里会不会出现缓存、检测和启用要一起处理的复杂情况。

这条线真正逼出来的，是第十三卷里关于工程现实的判断。  
真正成熟的产品接入，不会把自动启用和缓存检测当成附带步骤。

这页实际回收的实现链是：
- `12-06-02` 把自动启用、缓存检测写成认真对待真实用户现场的形状，这会在 `13-06-01` 里回收到“失败、不同步和重新接回来的过程也要一起被设计进去”。

很多接入链只盯着“能不能启用”。  
Claude Code 在这里把自动启用和缓存检测放在一起，等于承认接入链首先要替真实用户把现场接稳。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
