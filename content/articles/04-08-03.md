# 工具执行后的世界为什么要被重新讲给模型听？

一次工具执行批次完成后，`query()` 不会直接进入下一轮 API 调用。在继续循环之前，它会做一件「整理现场」的工作。

## 什么需要在工具执行后被重新注入？

看 `QueryEngine.submitMessage()` 里构建下一轮消息时的逻辑：

```typescript
// 获取下一轮需要注入的附件消息
const attachmentMessages = await getAttachmentMessages({
  messages: this.mutableMessages,
  processUserInputContext,
  // 包括：
  // - 排队中的用户消息（slash 命令、任务通知）
  // - 文件状态变化
  // - 预取好的记忆
  // - 技能发现结果
  // ...
})
```

这些「附件」来自不同来源：

### 1. 异步预取的记忆

```typescript
// 在 query() 进入前就开始预取
using pendingMemoryPrefetch = startRelevantMemoryPrefetch(
  state.messages,
  state.toolUseContext,
)

// 工具执行完后，等待预取结果
const attachments = await getAttachmentMessages(...)
```

记忆预取在整个工具执行期间（5-30 秒）在后台运行。工具执行完成、准备进入下一轮时，才等待预取结果。这隐藏了记忆检索的延迟——不是"等记忆准备好再做工具"，而是"工具执行的同时记忆在准备"。

### 2. 技能发现结果

```typescript
const pendingSkillPrefetch = skillPrefetch?.startSkillDiscoveryPrefetch(
  null, messages, toolUseContext,
)
// ... 工具执行 ...
// 使用 pendingSkillPrefetch 的结果构建附件
```

如果在当前会话里发现了新的可用 skill（比如 `.claude/skills/` 目录里的自定义技能），这些发现会作为附件注入，让模型在下一轮知道"现在有这些额外能力可以用"。

### 3. 队列中的任务通知

如果有其他任务（比如 background agent）完成了工作，通知消息会在工具执行完后被注入，让主循环知道"后台有新信息"。

### 4. 文件状态同步

工具执行可能修改了文件（EditFile、WriteFile、Bash 命令）。下一轮开始前，需要把最新的文件状态同步进 readFileCache，确保下一轮读取文件时用的是最新内容，而不是工具执行前的缓存。

## 为什么不能只靠 tool_result？

一个合理的问题：工具的执行结果已经在 tool_result 里了，为什么还需要额外的附件注入？

因为 **tool_result 只包含工具的直接输出**，不包含工具执行产生的系统状态变化。

举个例子：
- `WriteFile` 工具写入了 `src/main.ts`
- tool_result 可能是：`"File written successfully"`
- 但 `readFileCache` 需要被更新，记录 `src/main.ts` 的最新内容
- 如果下一轮有工具需要读取 `src/main.ts`，它应该得到刚写入的内容，而不是写入前的缓存

这种系统状态更新不能靠 tool_result 传递，需要在应用层显式同步。附件注入就是这个同步机制的载体之一。

## 不同来源的附件有不同的生命周期

Claude Code 在处理附件时有一个重要的区分：

```typescript
function filterDuplicateMemoryAttachments(
  attachments: AttachmentMessage[],
  messages: Message[],
): AttachmentMessage[] {
  // 过滤掉已经出现在消息链里的记忆附件
  // 防止同一条记忆被重复注入（比如在多次工具执行轮次后）
}
```

记忆附件需要去重。如果第一轮工具执行后注入了一条关于"用户偏好使用 TypeScript"的记忆，第二轮工具执行后这条记忆不应该再次出现——模型已经知道了，重复注入会浪费 token，更糟糕的是会让模型对"我是第一次知道这件事还是已经知道了"产生困惑。

技能发现的结果有不同的处理：它们可能每轮都需要重新检查（因为工具执行可能创建了新的 skills 目录或修改了技能文件）。

## 附件注入的失败处理

附件注入不在关键路径上：

```typescript
void startRelevantMemoryPrefetch(...)  // fire-and-forget
  .then(result => { /* 注入 */ })
  .catch(e => { /* 静默失败，记录日志 */ })
```

如果记忆预取失败（网络问题、文件读取失败），这一轮不会有记忆附件，但会话继续。模型可能不知道某条相关记忆，但这比让整个会话崩溃要好得多。

这是一个明确的可靠性选择：附件是增强性的，不是必须的。核心会话不因为附件失败而中断。

## 面试指导

这个设计解决的是一个常见的分布式系统问题：**如何让一个有状态的系统在异步操作完成后保持视图一致性？**

Claude Code 的方案：
1. **异步预计算**（在工具执行期间后台进行）
2. **同步注入点**（工具执行批次完成后，下一轮 API 调用前）
3. **类型化附件**（区分记忆、技能、通知，不同类型有不同的注入策略）
4. **去重处理**（防止同一信息多次注入）

能说出"附件注入是在「工具完成」和「下一轮 API」之间的一个显式同步点"的候选人，面试官会认为他理解了 agent 状态管理的关键挑战：工具执行会改变世界，模型需要看到被改变后的世界。
