# expandKey + buildMessageLookups 合起来构成了什么？一个跨消息的因果视图层

到这里，`buildMessageLookups` 的各个细节都讲完了。现在从更高的层次看：这整套机制为什么值得这么复杂？

## transcript 和因果视图的区别

Claude Code 的 `transcript` 是一个追加式日志，按时间顺序存储每次交互的原始消息。这个格式服务于"历史记录"——你能知道发生了什么，但需要自己推断"为什么"和"这些事之间有什么关系"。

`buildMessageLookups` + `expandKey` 构成的是一个**因果视图层**，它在 transcript 之上建立了另一套组织原则：

- `toolUseByToolUseID`：任意位置的工具结果都能找到它的来源请求
- `siblingToolUseIDs`：一次并发工具块里的所有工具互相可见
- `progressMessagesByToolUseID`：一个工具调用期间的所有中间状态都聚合在它名下
- `expandKey`：工具请求和结果在展开状态上是同一个实体

这不是 transcript 有的，是界面层额外建立的。

## 两条逻辑线如何汇合

**线 1：buildMessageLookups → 关系图**

一次 O(n) 扫描，输出 ID → 数据的映射。任何组件需要"这个工具调用的结果在哪"都查这张图，不自己扫数组。

**线 2：expandKey → 展开单元**

展开状态用 `tool_use_id` 做键，不用 `uuid`（消息级 ID）。这让"展开"的语义从"展开这行消息"变成"展开这次工具调用"。

两条线使用相同的 `tool_use_id` 作为连接键，这是有意为之的共识，不是巧合。

## 为什么这样设计

假设用最朴素的方式构建 transcript 界面：每条消息按时间顺序渲染，工具调用和结果是两行，展开各自管各自。用户想知道"这个 grep 结果是哪条命令产生的"，就得靠肉眼往上找，或者自己脑子里推断。

Claude Code 的因果视图层把这个推断工作做进了代码里。用户点击一条工具结果，展开之后同时看到请求和结果，因为 expandKey 让它们共用展开键；`toolUseByToolUseID` 让 `UserToolResultMessage` 能显示"这是 Bash 命令的输出"而不只是"这是一条工具结果"。

这个设计决策的代价是：每次 transcript 更新都要重建 7 张查找表，界面层维护了超过数据层的知识。好处是：界面不是 transcript 的被动回放者，而是主动对因果关系进行了诠释。

## lookup 的完整生命周期

```
transcript 更新
   ↓
normalizeMessages → normalizedMessages
   ↓
reorderMessagesInUI + applyGrouping + collapse
   ↓
buildMessageLookups → lookups（7 张表）
   ↓
lookupsRef.current = lookups（写入稳定回调的通道）
   ↓
VirtualMessageList 渲染可视窗口
   ↓
每个 MessageRow 读 lookups，决定样式、展开状态、静态化时机
   ↓
用户操作 → 读 lookupsRef.current（事件时刻的最新 lookups）
```

这条链上的每一步都有明确的职责，都在 lookup 机制成立之前完成各自的任务。

## 面试指导

这是这一章节的总结题，考察你是否能把分散的细节整合成一个完整的心智模型。

**一句话概括**：Claude Code 的消息渲染层在 transcript（时间序列）之上维护了一个因果视图（ID 关系图），`buildMessageLookups` 负责建图，`expandKey` 负责让 UI 操作尊重因果边界，两者用相同的 `tool_use_id` 作为语言。

**扩展到设计原则**：这是 CQRS（Command Query Responsibility Segregation）思想在前端的体现。transcript 是只写的事件日志（追加操作），lookups 是只读的查询视图（读操作）。把两者分开，让各自都能以最优化的方式工作——日志不用考虑查询效率，查询不用考虑日志写入的时序。

**面试里的反问机会**：如果面试官问"这么做有什么缺点"，好的答案是：**lookups 和 transcript 可能短暂不一致**。比如 transcript 已经更新了，但 buildMessageLookups 还没跑完（异步情况下），这时 lookups 是旧的，读到的关系图是上一次的快照。Claude Code 通过在渲染函数体里同步建 lookups 来保证一致性，但如果引入异步 buildMessageLookups（比如 Web Worker），这个不一致窗口就需要额外处理。
