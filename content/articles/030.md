# BashTool 的 bash 命令安全解析：为什么要 parse AST 而不是匹配字符串？

BashTool 允许 Claude 执行任意 shell 命令，这是整个系统里风险最高的能力。在这种情况下，权限检查如果做得不够严格，攻击者可以绕过任何基于字符串匹配的检查。

`tools/BashTool/` 目录下有大量安全相关代码，核心思路是：**对 bash 命令做语法解析，检查 AST 结构，而不是匹配字符串**。

## 为什么字符串匹配不够

考虑这个权限规则：禁止向 `/etc/` 写入。

字符串匹配方案可能这样实现：
```
command.includes('> /etc/')  →  危险
```

但这会被绕过：
```bash
cat passwd>/etc/passwd         # 没有空格
cat passwd 1>/etc/passwd       # 重定向符前有 fd 号
cat passwd | tee /etc/passwd   # 用管道+tee
bash -c "cat passwd > /etc/x"  # 嵌套执行
```

字符串层面的模式匹配是打地鼠——你堵上一个，攻击者找到另一个变体。

## 规则匹配的固定点计算

`shouldUseSandbox.ts` 里有一段很有意思的逻辑——对 bash 命令的"stripped"版本做迭代匹配：

```typescript
// 问题：FOO=bar bazel run ... 和 timeout 30 bazel run ... 都应该匹配 "bazel:*"
const candidates = [trimmed]
const seen = new Set(candidates)
let startIdx = 0

// 迭代剥离环境变量前缀和安全包装命令
while (startIdx < candidates.length) {
  const endIdx = candidates.length
  for (let i = startIdx; i < endIdx; i++) {
    const cmd = candidates[i]!
    const envStripped = stripAllLeadingEnvVars(cmd, BINARY_HIJACK_VARS)
    if (!seen.has(envStripped)) {
      candidates.push(envStripped)
      seen.add(envStripped)
    }
    const wrapperStripped = stripSafeWrappers(cmd)  // 去掉 timeout, nice 等
    if (!seen.has(wrapperStripped)) {
      candidates.push(wrapperStripped)
      seen.add(wrapperStripped)
    }
  }
  startIdx = endIdx
}
```

这是一个固定点迭代（fixed-point iteration）：不断剥离环境变量和安全包装命令，直到不再产生新的候选形式。然后对所有候选形式都做规则匹配。

注释里明确说明了这为什么是必要的：`timeout 300 FOO=bar bazel run` 单次剥离只会得到要么 `FOO=bar bazel run`（剥离了 timeout）要么 `timeout 300 bazel run`（剥离了 env var），需要多次迭代才能得到最简形式 `bazel run`。

## 重要的免责声明

代码里有一段关键注释值得引用：

```
// NOTE: excludedCommands is a user-facing convenience feature, not a security boundary.
// It is not a security bug to be able to bypass excludedCommands — the sandbox permission
// system (which prompts users) is the actual security control.
```

"可以绕过 excludedCommands 不是安全漏洞"——这是设计意图，不是疏忽。排除命令列表是用户配置的便利功能（"我不想让 Claude 用 npm，让它用 yarn"），真正的安全控制是沙箱+用户确认机制。

这种"明确说清楚哪层是安全边界"的做法很值得学习——它防止了"以为这里是安全边界，实际不是"的误解。

## 面试角度

bash 命令安全分析是一个有深度的安全工程话题：

1. **语法解析 > 字符串匹配**：任何基于字符串的安全检查都可以通过语法变体绕过
2. **固定点迭代处理命令变形**：剥离包装命令的迭代直到稳定，比单次剥离更完整
3. **明确安全边界很重要**：注释说清楚"这不是安全边界，那才是"，防止混淆
4. **防御深度（Defense in Depth）**：AST 解析 + 沙箱 + 用户确认多层保护，不依赖单一机制
