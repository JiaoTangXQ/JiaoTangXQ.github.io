# getCommands 最后把新旧来源收回同一条装配流水线

## 实现链
- `src/commands.ts` 的职责就是把内置命令、旧 command、skills、plugins 甚至部分 MCP 暴露能力统筹起来，让调用方最终只面对一个命令列表。
- 也就是说，迁移期虽然有多种来源，但最终出口仍然是一条统一流水线，而不是谁来谁自己挂 UI。

## 普通做法
- 普通做法是不同来源维护不同注册表，本地命令看一个数组，插件再看另一个数组，远端功能再走第三套查找逻辑。
- 这种拆法短期开发很快，因为每条线只管自己。

## 为什么不用
- Claude Code 不愿让上层感知这么多来源，因为补全、帮助、权限和调度都希望面对统一接口。
- 如果 `getCommands` 不收口，命令系统之外的每个模块都得学会“这条命令到底是旧的、新的、插件的还是远端翻译过来的”。

## 代价
- 代价是这条总装配线会越来越像交通枢纽，里面难免充满特例和兼容分支。
- 但这比把来源复杂度扩散到全系统要好，因为脏活至少被集中在了一个地方。
