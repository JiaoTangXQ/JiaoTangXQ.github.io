# 条件能力为什么会改写用户所处的世界

面试题版本：**"Claude Code 里激活 computerUse 能力前后，用户体验有什么本质变化？为什么说这种能力'改写了用户所处的世界'？这对权限设计有什么含义？"**

## 普通工具 vs 条件能力

**普通工具**（如 file_read）：用户请求读取文件，Claude 读取并返回内容。用户的世界没有变化，只是 Claude 的行动空间多了一种操作。

**条件能力**（如 computerUse）：Claude 可以操控用户正在使用的浏览器，点击按钮，填写表单，截图观察，导航到任意网页。用户的世界发生了变化——他们的浏览器行为现在可以被 Claude 影响，这改变了"浏览器是我单独控制的工具"这个基本假设。

"改写用户所处的世界"的含义不是戏剧性的，而是工程性的：**激活条件能力后，用户模型（用户对系统边界的理解）需要更新**。

## 条件能力改变了什么边界

**文件系统边界**：file_read/file_edit 工具改变了 Claude 可以访问哪些文件（从"不能访问文件系统"到"可以访问项目文件"）。这是最常见的边界扩张。

**浏览器边界**：computerUse 改变了 Claude 可以影响哪些应用程序（从"不能影响浏览器"到"可以控制浏览器"）。这个边界扩张的影响范围更大——浏览器访问的是整个互联网，而不只是项目文件。

**认证边界**：如果 Claude 可以操控浏览器，它可以访问用户已经登录的网站（用已有的认证 cookie），这意味着它能做的事情受限于用户的网络权限，而不只是 Claude 自身的权限。

**可观察性边界**：截图能力让 Claude 可以看到用户屏幕上的任何内容，包括用户可能没有意识到在展示给 Claude 的敏感信息。

## 为什么边界判断先于能力讨论

在能力技术层面之前，用户需要理解"启用这个能力意味着什么"。

如果先讨论"computerUse 很强大，可以自动化很多浏览器操作"，然后才问用户"你允许吗"，用户可能在没有充分理解边界变化的情况下授权。

正确的顺序是：
1. 解释这个能力会如何改变 Claude 的行动边界
2. 解释这对用户意味着什么（Claude 可以看到/操控什么）
3. 用户在理解边界后做出授权决策
4. 激活能力

这个顺序要求权限系统把"边界说明"作为授权流程的一部分，而不是技术细节的附录。

## 优雅必须正视真实边界

"优雅必须正视真实边界和真实人"这句话在 computerUse 场景里的含义：

**真实边界**：computerUse 给了 Claude 一种新的行动空间，这个空间和用户的私人数据、工作流程、认证状态有交集。边界不是抽象的概念，而是"Claude 现在可以做 X"。

**真实人**：用户不是系统管理员，可能不理解 Native Messaging、截图权限、OAuth scope 这些概念。权限说明必须用用户能理解的语言描述边界，而不是技术术语。

优雅的权限设计不是"把权限请求做成最小化、最不显眼的"，而是"清楚地帮助用户理解他们在授权什么"。

## 面试角度的总结

条件能力"改写用户所处世界"，在工程上的含义是：这类能力的激活改变了系统的行动边界，用户对系统的心智模型需要随之更新。权限设计的职责不只是"是/否授权"，而是帮助用户理解边界变化，在充分知情的情况下做决策。这要求把边界说明作为授权流程的核心，而不是隐藏在技术文档里的细节。
