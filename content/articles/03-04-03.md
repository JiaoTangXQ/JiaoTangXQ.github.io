# 统一入口之后，来源差异又被保留在什么地方

如果一个抽象只会统一，不会保留差异，它很快就会变成假抽象。Claude Code 没犯这个错。`Command` 把所有能力收进同一入口以后，仍然把那些会改变现实后果的不同郑重留下来了。

最明显的差异有几类。第一类是执行世界不同，`prompt`、`local`、`local-jsx` 仍然明确分开。第二类是来源不同，技能、插件、bundled、MCP、workflow 这些来历会继续带着自己的标记、插件信息和 kind。第三类是使用边界不同，有的命令只在特定账户环境里出现，有的不给模型调用，有的不能在非交互里用，有的在远端或 bridge 世界里还要再过一层安全筛选。

这说明 Claude Code 的统一不是“大家以后都一样”，而是“大家先用同一种语言报到，再按真实物理差异各就各位”。这种分寸感非常重要。因为用户看到的是同一排命令，系统看到的却是一组出处、执行方式、权限边界都不完全相同的能力。真正成熟的设计，不会为了整齐把这些差异抹掉，而是会把它们保留在最该保留的位置上。

## 实现链
统一入口之后，来源差异没有消失，而是留在 `command.type` 和 `pluginInfo / loadedFrom / kind` 这些字段里，再由 `getMessagesForSlashCommand()` 的 switch 和遥测逻辑去消费。

## 普通做法
一种极端是来源完全各走各路，另一种极端是全抹平，假装所有命令内部机制都一样。

## 为什么不用
前者会让入口层失控，后者又会掩盖 `local`、`local-jsx`、`prompt`、`fork` 这些真实差异。它选的是“入口统一，执行分流”。

## 代价
你得同时理解“表面统一”和“内部保留差异”这两层；但这比要么完全分裂、要么假装没差异更诚实。
