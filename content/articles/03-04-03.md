# 面试题：Command 统一后，来源差异去哪了？

## 面试官追问

「你说 Command 统一了不同来源。那如果我需要知道一个命令是来自插件还是内建的，或者来自 skills 目录的，怎么拿到这个信息？」

## loadedFrom：来源存档

```typescript
loadedFrom?: 'commands_DEPRECATED' | 'skills' | 'plugin' | 'managed' | 'bundled' | 'mcp'
```

每个从外部加载的命令都会在加载时被打上 `loadedFrom` 标记。在遥测系统里：

```typescript
// processSlashCommand.tsx 里的遥测代码
...("external" === 'ant' && {
  skill_name: commandName,
  ...(returnedCommand.type === 'prompt' && {
    skill_source: returnedCommand.source,
  }),
  ...(returnedCommand.loadedFrom && {
    skill_loaded_from: returnedCommand.loadedFrom,
  }),
  ...(returnedCommand.kind && {
    skill_kind: returnedCommand.kind,
  }),
})
```

遥测代码区分来源，以便在 analytics 里分析内建命令和用户自定义 skills 的使用分布，而不需要修改入口层的通用逻辑。

## pluginInfo：插件的完整身份

对于来自插件的命令，`PromptCommand` 里有：

```typescript
pluginInfo?: {
  pluginManifest: PluginManifest
  repository: string
}
```

这比 `loadedFrom: 'plugin'` 精细得多——它包含了插件名、版本、marketplace 信息。安全策略层（`isSourceAdminTrusted`、`isRestrictedToPluginOnly`）、遥测系统、更新通知都会读这个字段。

一条来自官方 marketplace 插件的 skill，和来自用户本地目录的 skill，`loadedFrom` 都是 `'plugin'`，但 `pluginInfo.repository` 不同，`isOfficialMarketplaceName(marketplace)` 的结果不同，最终的信任级别也不同。

## command.type：执行世界的差异

来源差异主要由 `loadedFrom` 保存，执行世界的差异则主要由 `command.type` 保存：

```typescript
switch (command.type) {
  case 'local-jsx': ...  // 临时 UI
  case 'local': ...      // 本地程序执行
  case 'prompt': ...     // 会话模式装入
}
```

这个 switch 在 `getMessagesForSlashCommand` 里，是所有差异最终展开的地方。入口层（补全、过滤、显示）不感知这个 switch，只有真正执行时才分叉。

## context: 'fork' 是 prompt 里的第二层差异

`PromptCommand` 里还有：

```typescript
context?: 'inline' | 'fork'
```

`prompt` 命令里，`context: 'fork'` 意味着这个命令要在独立子 agent 里运行，而不是把内容内联进当前会话。这是执行层面的差异，不影响入口层的统一展示。

在 `getMessagesForSlashCommand` 里：
```typescript
case 'prompt': {
  if (command.context === 'fork') {
    return await executeForkedSlashCommand(...)  // 开子 agent
  }
  return await getMessagesForPromptSlashCommand(...)  // 内联展开
}
```

## 差异的层次结构

把三类差异放在一起看：

```
入口层（补全、过滤、显示）
  ← 不看差异，统一处理 Command

执行层（getMessagesForSlashCommand）
  ← 看 command.type 分流（local-jsx / local / prompt）
    ← prompt 里再看 command.context 分流（inline / fork）

遥测/调试层
  ← 看 loadedFrom、pluginInfo 做来源分析

安全/策略层
  ← 看 pluginInfo.repository、source、availability 做权限判断
```

差异没有消失，而是被有序地分配到了「最需要它的层」。入口层的简洁，是靠把差异向后推实现的。

## 为什么不在 CommandBase 里暴露更多差异字段

一个常见的设计诱惑：把执行方式的差异也提升到 `CommandBase` 层，比如加一个 `runMode: 'local' | 'prompt' | 'fork' | 'local-jsx'`，在入口层就能看到所有差异。

问题在于：入口层不需要这个信息，而需要这个信息的执行层已经有了 `command.type` 和 `command.context`。提升到 `CommandBase` 只会让每个新来源都必须声明一个不属于它的字段，增加了约束而没有增加能力。

好的抽象暴露的信息是「这一层需要的信息」，不多也不少。

---

*面试指导：被问到「统一抽象会不会丢失信息」时，用「信息被分层保存，在需要的层再展开」来回答，举 loadedFrom 在遥测层使用、command.type 在执行层分流这两个具体例子，说明信息没有丢失，只是被合理地推到了需要它的地方。*
