# 面试题：processUserInput 的返回值为什么有 5 个字段而不是 1 个？

## 场景推演

面试官给你看这个类型定义：

```typescript
export type ProcessUserInputBaseResult = {
  messages: (UserMessage | AssistantMessage | AttachmentMessage | SystemMessage | ProgressMessage)[]
  shouldQuery: boolean
  allowedTools?: string[]
  model?: string
  effort?: EffortValue
  resultText?: string
  nextInput?: string
  submitNextInput?: boolean
}
```

然后问：「为什么不直接返回一个 `messages` 数组就够了？」

## 逐字段拆解

**`messages`**：当然要有。但它放着的不一定都是用户说过的话。里面可能有图片元信息（`isMeta: true`，用户看不见但模型要知道）、附件消息、hooks 附加上下文、系统级警告。这已经是一个处理过的消息列表，不是原始输入的镜像。

**`shouldQuery`**：不是每次 `processUserInput` 完成后都要继续问模型。`/compact`、`/clear`、`/config` 这类本地命令处理完就结束，`shouldQuery: false`。只有普通 prompt 和 `prompt` 型命令才需要 `shouldQuery: true`，告诉外层的 `QueryEngine.submitMessage()` 这一轮需要继续走主循环。少了这个字段，外层就不知道该不该触发推理。

**`allowedTools`**：slash 命令（尤其是 `prompt` 型 skill）可以在 frontmatter 里声明 `allowedTools`，临时放开某些工具权限。`parseToolListFromCLI(command.allowedTools ?? [])` 在 `getMessagesForPromptSlashCommand` 里完成解析，结果要传到外层才能让这一轮推理使用正确的工具集。如果把这个信息压进 messages 里，外层就没法在调用 API 前设置对应的工具许可。

**`model`** 和 **`effort`**：`prompt` 型命令可以在 frontmatter 里指定 `model: claude-3-haiku` 或 `effort: low`。这些信息需要在外层发起 API 请求前被读取，写进 messages 里没有意义，因为 API 调用参数不从 messages 里读。

**`resultText`**：给 `-p`（非交互 headless）模式用。fork 类子 agent 完成后，结果文本要回填到父 session。这个字段独立传递，不用从 messages 里反向解析。

**`nextInput` / `submitNextInput`**：用于命令链——某些命令完成后需要自动触发下一条输入（比如 `/discover` 选中功能后自动发起命令）。这两个字段控制下一步行为，也不适合嵌进 messages。

## 为什么这个设计是对的

这组返回值的形状，其实是「这一轮输入处理完以后，系统需要知道哪些事情」的精确描述：

- 会话里发生了什么（messages）
- 下一步做什么（shouldQuery、nextInput、submitNextInput）
- 下一步的参数是什么（allowedTools、model、effort）
- headless 模式要输出什么（resultText）

把这些东西全压成一个 `messages` 数组，外层就得从消息里反向推断「这次要不要推理、用什么模型、有没有额外工具权限」——这正是本该由入口层负责的事情被偷懒甩锅给下游处理了。

## 源码位置

`ProcessUserInputBaseResult` 定义在 `src/utils/processUserInput/processUserInput.ts`。`QueryEngine.submitMessage()` 在调用 `processUserInput()` 后检查 `shouldQuery` 决定是否继续推进，同时把 `allowedTools`、`model`、`effort` 传入实际的 query 调用。

---

*面试指导：被问到「为什么返回值有这么多字段」时，从「不同字段的消费者不同」这个角度切入——messages 给 transcript，shouldQuery 给主循环，allowedTools/model/effort 给 API 参数。这比说「功能多所以字段多」更说明架构判断。*
