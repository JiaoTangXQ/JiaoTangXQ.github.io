# REMOTE_SAFE_COMMANDS 为什么是一层独立边界

远端模式最容易犯的错，就是把本地命令表整个搬过去，再祈祷别出事。Claude Code 没这么做，它专门有一层 `REMOTE_SAFE_COMMANDS`，先把哪些命令能在 remote 世界里出现这件事单独定下来。

这层边界的重点不是“有没有危险”，而是“这个命令在远端现场到底还有没有意义”。像退出、清屏、帮助、主题、键位、状态线、用量、复制这类命令，在远端 REPL 里仍然有稳定语义；但很多依赖本地 UI、本地文件状态或本地交互前提的命令，就不该被短暂地展示出来，再等系统过几秒告诉你其实不能用。源码里甚至明确写了，过滤要尽量发生在 REPL 渲染前，避免本地命令在 remote 模式下先闪现一遍。

所以 `REMOTE_SAFE_COMMANDS` 真正解决的是远端世界的入口整洁度。它不是对本地命令的一次简单删减，而是在认真回答: 当会话发生在另一端时，哪些命令仍然属于这个世界。把这件事单独拉成一层边界，本身就说明 Claude Code 对“入口即语义”这件事非常敏感。

## 实现链
`commands.ts` 里单独维护了 `REMOTE_SAFE_COMMANDS`，远端会话只暴露这部分命令。也就是说，远端不是“本地命令全集的一个皮肤”，而是有自己被允许的入口面。

## 普通做法
一种偷懒做法是本地能跑的命令远端也全开，出了问题再在命令内部补判断。

## 为什么不用
因为远端缺少本地终端、文件环境和交互前提。有些命令在本地成立，在远端根本没有执行现场。

## 代价
用户会感到能力不完全对齐；但这比给出一个看似一致、实际到处报错的远端命令集更诚实。
