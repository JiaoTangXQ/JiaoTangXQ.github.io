# setup() 和 init() 都叫初始化，但 setup() 需要 cwd 参数——这个差异意味着什么？

## 两个函数签名的对比

```typescript
// init() 的签名
export const init = memoize(async (): Promise<void> => { ... })
// 无参数

// setup() 的签名
export async function setup(
  cwd: string,
  permissionMode: PermissionMode,
  allowDangerouslySkipPermissions: boolean,
  worktreeEnabled: boolean,
  worktreeName: string | undefined,
  tmuxEnabled: boolean,
  customSessionId?: string | null,
  worktreePRNumber?: number,
  messagingSocketPath?: string,
): Promise<void>
// 9 个参数
```

`init()` 是无参数的纯初始化函数。`setup()` 有 9 个参数，每一个都描述"这次会话的具体现场"。

## 参数来源分析

这 9 个参数从哪里来？

```typescript
// main.tsx 里的调用示例（简化）
await setup(
  getCwd(),                      // 当前工作目录，来自命令行或进程环境
  parsedPermissionMode,          // 来自 --permission-mode 参数
  allowDangerouslySkipPermissions, // 来自 --dangerously-skip-permissions
  worktreeEnabled,               // 来自 --worktree 参数
  worktreeName,                  // 来自 --worktree-name
  tmuxEnabled,                   // 来自 --tmux 参数
  customSessionId,               // 来自 --session-id
  worktreePRNumber,              // 来自 --worktree-pr
  messagingSocketPath,           // 来自 --messaging-socket-path
)
```

所有参数都来自 Commander 解析出来的用户输入。这意味着：在 Commander 解析完成、action handler 运行之前，无法调用 `setup()`。

这不是偶然的——`setup()` 本来就应该这时候运行，因为它需要"这次用户想要什么样的会话"的完整信息。

## setup() 做了什么会改变后续依赖？

```typescript
// setup.ts（部分）
setCwd(cwd)                         // 全局 cwd 变更
captureHooksConfigSnapshot()         // 从新 cwd 读取 hooks 配置
initializeFileChangedWatcher(cwd)    // 监控新 cwd 的文件变化

// worktree 模式下
process.chdir(worktreeSession.worktreePath)   // 改变进程工作目录！
setCwd(worktreeSession.worktreePath)
setOriginalCwd(getCwd())
setProjectRoot(getCwd())
clearMemoryFileCaches()              // 清除旧目录的缓存
updateHooksConfigSnapshot()          // 重新读取 hooks 配置
```

`setup()` 做的最关键的事是：**把全局 cwd 切换到正确的工作现场**。

在 `setup()` 执行之前，`getCwd()` 返回的是进程启动时的目录（或 `--cwd` 参数）。在 `setup()` 执行之后，特别是在 worktree 模式下，`getCwd()` 返回的是新创建的 worktree 路径。

这意味着：任何依赖 `getCwd()` 的代码，只有在 `setup()` 之后调用才是安全的。

## 提前调用 setup() 有什么风险？

假设把 `setup()` 提前到 `init()` 之前。

问题一：`setup()` 内部会调用 `getIsGit()`、`findCanonicalGitRoot()` 等依赖 cwd 的 git 操作。如果 `init()` 的一些配置影响了这些操作（比如 git 可执行文件路径的配置），顺序颠倒会导致错误。

问题二：`setup()` 需要 Commander 解析的结果，而 Commander 解析在 `preAction` 阶段。`init()` 在 `preAction` 里执行，比命令 action handler 更早。`setup()` 在 action handler 里。无法在 `init()` 阶段就拿到 `setup()` 需要的参数。

问题三：worktree 模式下 `process.chdir()` 改变进程工作目录。如果在 `init()` 的网络配置（`configureGlobalAgents()`）之前就 `chdir`，可能影响某些依赖路径的代理配置解析。

## "可以运行" vs "在正确地方运行"

这两个函数的本质区别：

`init()`：让这个进程"能运行"——基础设施就绪，可以安全地发 HTTPS 请求、读写配置、注册信号处理器。

`setup()`：让这个进程"在正确地方运行"——切换到用户想要的工作目录，创建需要的 worktree，建立正确的 session 上下文。

前者是机器层面的就绪，后者是任务层面的定位。

## 面试指导

这道题的核心是识别"两阶段初始化"的架构原因。

常见的面试问法是："你会把所有初始化放在一个 init 函数里，还是分开？"

好的答案不是"分开更整洁"，而是：
1. 如果不同初始化阶段的前置条件不同（一个无参数就能运行，一个需要完整的用户输入），就必须分开
2. 如果不同阶段有不同的可重入性要求（一个 memoize 只跑一次，一个可以用不同参数多次运行），就应该分开
3. 如果不同阶段的副作用级别不同（一个是全局基础设施，一个是会话级状态），分开可以让错误边界更清晰

Claude Code 的 `init()`/`setup()` 分离，同时满足了这三个理由。
