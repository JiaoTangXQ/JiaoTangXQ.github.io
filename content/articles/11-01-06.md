# 性能分析：为什么观测指标不能实时写盘？process.on('exit') flush 的权衡与风险

「每次记录就同步写盘」是最直觉的实现方案，但在高频场景里这是一个隐患。

Claude Code 的统计系统选择了「内存缓存 + 退出时 flush」，这是一个有明确权衡的决定，值得拆开来看。

---

## 实时写盘有什么问题

假设每次工具调用完成都要记录统计：

```ts
// 如果每次都同步写盘
function recordToolCall(toolName: string, duration: number) {
  const stats = readStatsFromDisk()   // 读磁盘
  stats.toolCalls[toolName] = (stats.toolCalls[toolName] ?? 0) + 1
  writeStatsToDisk(stats)             // 写磁盘
}
```

Claude Code 在一次复杂任务里可能调用几十上百次工具。每次都读写磁盘：

1. **增加延迟**：即使是 SSD，一次同步读写也需要几毫秒。工具调用本身可能只需要 100ms，但如果每次都额外写盘，累积延迟就不可忽视
2. **阻塞事件循环**：如果用同步 I/O（`writeFileSync`），会直接阻塞 Node.js 的事件循环，期间所有输入处理和 UI 更新都暂停
3. **频繁触发文件系统事件**：如果有 watch 机制监听配置目录，频繁写统计文件可能触发不必要的 watch 回调

## Claude Code 的实际做法

```tsx
// StatsProvider 内部（概念还原版）
useEffect(() => {
  const flush = () => {
    const metrics = store.getAll()
    if (Object.keys(metrics).length > 0) {
      saveCurrentProjectConfig(current => ({
        ...current,
        lastSessionMetrics: metrics,
      }))
    }
  }
  process.on('exit', flush)
  return () => process.off('exit', flush)
}, [store])
```

做法是：**在内存里积累，进程退出时统一写盘**。

整个会话里，统计数据只存在于 `createStatsStore()` 创建的内存结构里（三个 Map）。所有的 `increment`、`recordHistogram` 操作都是纯内存操作，没有 I/O。

只有当进程正常退出时（`exit` 事件），才把所有数据 flush 到磁盘。

---

## 这个设计的代价：数据丢失窗口

`process.on('exit')` 的触发条件是进程**正常退出**：

- `process.exit()` 主动退出
- 事件循环自然耗尽
- 未捕获的异常（这个有时也能触发 exit，但不保证）

**不触发的情况：**

- `SIGKILL`（`kill -9 <pid>`）：操作系统直接杀死进程，没有机会执行任何 JS 代码
- OOM（内存耗尽）：OS 强制终止
- 系统崩溃

在这些情况下，这次会话的统计数据就丢了。

---

## 为什么接受这个代价

这是一个有意的权衡，逻辑是：

**统计数据不是业务关键数据**。丢了一次会话的指标不会导致功能损坏，不会丢用户数据，不会有任何对用户可见的影响。

**读取频次不对称**。统计数据在会话里持续写入（每次工具调用），但只在会话结束后被读取一次（下次启动时读取 `lastSessionMetrics`）。为了这一次读取，实时写盘的成本完全不成比例。

**容忍性高**。指标用于分析和调优，本身就是统计意义上的数据，偶尔丢一次会话不影响大局。

---

## 和异步批量写入的对比

还有一种中间方案：不是实时写盘，也不是退出时 flush，而是定期批量写入：

```ts
// 每 30 秒批量 flush 一次
setInterval(() => {
  const metrics = store.getAll()
  saveMetricsToDb(metrics)
}, 30_000)
```

这个方案减少了数据丢失窗口（最多丢 30 秒的数据），但代价是：

1. 30 秒 timer 的存在会让 Node.js 进程不能自然退出（需要 `timer.unref()` 或手动清除）
2. 在高频写入场景（每秒多次记录），30 秒积累的数据量需要批量序列化，可能有 CPU 峰值
3. 如果应用想在「特定时机」保存数据（比如用户手动触发导出），这个 timer 逻辑和按需 flush 逻辑会有干扰

`process.on('exit')` 的方案更简单，代价更明确，适合对数据丢失有较高容忍度的观测场景。

## StatsProvider 挂 useEffect 的设计考量

```tsx
useEffect(() => {
  process.on('exit', flush)
  return () => process.off('exit', flush)
}, [store])
```

依赖数组是 `[store]`。这意味着如果 `store` 实例发生变化，旧的 flush 函数会被清除，新的 flush 函数会被注册。

在正常使用里，`store` 不会变化（`useMemo` 保证只创建一次）。但这个写法保证了正确性：如果 store 因某种原因被替换，不会有两个 flush 函数同时注册在同一个 exit 事件上，导致双重写盘。

`return () => process.off('exit', flush)` 是清理函数。当 `StatsProvider` 卸载时（理论上这在 Claude Code 里不会发生，因为 Provider 的生命周期和进程一样长），cleanup 会取消 exit 监听。这是一个「以防万一」的良好实践。

## 面试指导

**写盘策略的选择标准是什么？**

关键问题：
- 数据丢失的代价是什么？（用户数据 vs 观测指标）
- 写入频率是多少？（每秒一次 vs 每次操作）
- 读取时机是什么？（实时读取 vs 事后分析）
- 进程被强杀的概率有多高？（服务器进程 vs 用户终端工具）

**什么场景应该用 WAL（Write-Ahead Log）而不是批量 flush？**

当数据是业务关键型时（比如数据库事务、支付记录），必须用 WAL 或类似机制确保每次写入都持久化。牺牲性能换可靠性。Claude Code 的统计数据不是这类，所以可以接受简单的批量 flush。

**SIGKILL 不可捕获是 Node.js 的问题还是操作系统设计？**

这是 POSIX 的设计：SIGKILL 是强制终止信号，任何进程都不能捕获或忽略它。这是操作系统保留的最后手段，确保任何进程都能被终止，即使它的代码有 bug 导致无法正常响应其他信号。应用层代码永远无法防御 SIGKILL。
