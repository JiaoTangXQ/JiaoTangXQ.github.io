# session memory 的更新节奏：为什么不实时同步，而是定期写入？

这是一道性能和工程取舍题："如果你要实现一个跟踪 AI Agent 工作状态的外置记忆系统，每轮都写入和定期写入有什么不同？为什么选后者？"

---

## 两种极端方案

**每轮实时写入**：
- 优点：状态最新，随时可以用最精确的记录
- 缺点：每轮都需要一次文件 IO 或 API 调用（如果写入需要模型参与），增加每轮延迟；如果写入本身需要"总结当前状态"，还需要一次额外的模型调用

**完全不外置，状态全在上下文里**：
- 优点：无额外开销
- 缺点：工作状态持续占用上下文空间；compact 之后状态可能被简化丢失；重启后无法恢复

Claude Code 选了中间路径：**有条件的定期写入**。

---

## 两个写入时机

session memory 的更新发生在两种情况：

**时机 1：compact 触发时**

当 `trySessionMemoryCompaction` 被调用，系统会检查是否有有效的 session memory 文件可用。如果 session memory 文件存在且有效，系统可以用它来做"轻量 compact"（保留最近消息 + session memory 附件）而不是全量 compact。

在这个路径里，session memory 是被**消费的**，不是被创建的。创建是在其他地方发生的。

**时机 2：stop hooks 里的后台更新**

在 `handleStopHooks` 的执行序列里，session memory 相关的更新被安排在回合结束后执行。这个更新以"fire-and-forget"方式运行，不阻塞对用户的响应。

---

## 为什么不在主回合里更新

核心原因：**写入 session memory 可能本身需要一次模型调用**（让模型总结当前工作状态，写成结构化的笔记格式）。这种操作：

1. 会消耗额外的 token（输入 + 输出）
2. 会增加每轮的延迟（等模型生成状态文件内容）
3. 可能和主任务的内容混淆（用户在等主任务的回应，不应该看到"正在写笔记"的中间过程）

把这个操作推到回合结束后，让主回合先落稳，用户先看到回应，然后再在后台写入状态——这是"主线程优先"的工程哲学。

---

## 滞后性的接受

session memory 的定期更新意味着它**不是实时镜像**。如果 compact 在最后一次 session memory 写入之后很快触发，用到的 session memory 内容可能已经有一到两轮的滞后。

这个滞后是被接受的，原因是：

1. **大多数情况下影响不大**：session memory 描述的是工作台状态，不是具体的代码细节。"正在做 auth 模块重构，已完成登录流程，下一步是 JWT 验证"这类状态不会在一两轮里发生根本变化。

2. **最近轮次有原始消息保底**：session memory compact 路径里，最近几轮的原始消息是保留的（`messagesToKeep`）。即使 session memory 文件滞后，最近的细节仍然在上下文里。

3. **比没有 session memory 好**：和"compact 丢掉所有历史只剩摘要"相比，有一个稍微滞后的 session memory 仍然比什么都没有强。

---

## setLastSummarizedMessageId 的游标追踪

session memory 系统维护一个游标 `lastSummarizedMessageId`，记录"上次写入 session memory 时处理到哪条消息了"：

```typescript
// compact.ts 里多处出现
setLastSummarizedMessageId(undefined)
```

这个 `undefined` 重置表示：compact 之后，上次的游标失效了（因为消息 uuid 已经变化），下次写入 session memory 时要从头开始处理新的消息序列。

游标机制的作用是避免重复处理：如果上次 session memory 写入处理到了消息 X，下次只需要从 X 之后的消息开始增量处理，不需要重新分析全部历史。

---

## setSDKStatus 通知用户

compact 发生时，有一个状态通知：

```typescript
context.setSDKStatus?.('compacting')
```

session memory 的更新是后台的，不需要这个通知。compact（全量摘要）是前台阻塞的，需要让用户知道"现在在做压缩，稍等"。这个差异也体现了两者的时机设计——一个是前台显式操作，一个是后台安静写入。

---

## 面试指导

这道题的核心是理解"实时性" vs "开销"的权衡。

在回答时可以引出这几个权衡：

1. **写入代价**：更新频率越高，每轮延迟越大（如果写入需要模型参与）
2. **主线程优先**：整理工作不应该干扰主任务的执行，推到回合后是合理的
3. **滞后可接受**：工作状态变化相对缓慢，一两轮的滞后不影响大局
4. **最近消息保底**：session memory compact 保留最近原始消息，弥补了滞后的代价

能从"为什么选这个时机"而不只是"是这个时机"来分析，说明你真正理解了这个设计的权衡逻辑。
