# 登记任务时为什么要把用户正在看的现场保留下来

`registerTask()` 看起来像一次简单入表，实际上它在保护用户眼前那块现场不要被任务替换冲掉。

## 实现链

`src/utils/task/framework.ts` 的 `registerTask()` 在发现同 ID 任务是“替换注册”时，不会直接覆盖一切，而是刻意继承 `retain`、`startTime`、`messages`、`diskLoaded`、`pendingMessages`。代码注释写得很直白：这是为了 `resumeAgentBackground` 之类的替换场景，不要把用户正在看的 transcript 和保留状态冲掉。

这说明任务登记不仅服务数据正确，也服务“用户此刻正盯着什么”。

## 普通做法

普通做法往往是新状态覆盖旧状态。只要任务重新注册，就把对象整体替换，谁还在看它、内存里刚追加过什么，统统不管。

## 为什么不用

Claude Code 不敢这么粗暴，因为这里有一个很产品化的问题：用户看到的现场和内部任务对象不一定同一时刻变化。特别是后台恢复、继续追读、刚打进一条消息但还没落盘时，直接替换会让用户感觉“刚才看的东西突然没了”。

所以它把 `retain` 和 `messages` 这类字段视为现场资产，登记时优先保住。

## 代价

代价是 `registerTask()` 不再只是数据层函数，而带有明显 UI 语义。框架层因此更重，也更难做成纯粹的底层库。
