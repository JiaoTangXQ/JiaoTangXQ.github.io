# 第十三卷回收索引：任务世界里 `pendingMessages` 先排队再过轮次

## 实现链
结果回到主会话以前，消息本身也要先排好秩序。  
`pendingMessages` 不是随到随发的散点，而是要先排队再过轮次。

这页实际回收的实现链是：
- `05-08-02` 把 `pendingMessages` 先排队再说清，这会继续支撑 `13-02-02`，说明主会话接住结果以前，消息也要先排好自己的轮次。

主会话接结果并不是先看到一个孤立的“任务完成”弹窗。  
它先接到的是有顺序的消息流，`pendingMessages` 先把秩序排好，后面的结果才不会打乱会话节奏。

## 普通做法
更普通的做法，是把后台动作当成松散的异步句柄，跑起来就算开始，结束了再临时补状态和通知。

## 为什么不用
Claude Code 不这么做，是因为没有统一任务对象，就没有稳定的生命周期、输出、通知和回看链，前后台切换也会变成一串彼此不认账的临时处理。

## 代价
代价是任务状态字段、输出文件、通知协议和前后台转场逻辑都会变厚，维护者要记住更多接缝。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
