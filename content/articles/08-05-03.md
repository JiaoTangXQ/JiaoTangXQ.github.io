# 协议设计：远端消息为什么不能直接渲染，需要一层翻译？

## 两种消息格式的设计目标不同

`SDKMessage`（`src/entrypoints/agentSdkTypes.ts`）是 Claude Code 的**对外通信格式**。它的设计目标是：

- 跨系统（可以被第三方 SDK 解析）
- 跨版本（字段有向后兼容保证）
- 机器可读（JSON 结构，不含 UI 上下文）

本地 `Message`（`src/types/message.ts`）是**内部状态格式**。它的设计目标是：

- 高效渲染（带有 UI 所需的计算字段）
- 状态联动（消息类型和本地状态机直接对应）
- 带本地语义（`isMeta`、`isVirtual`、`isCompactSummary` 等字段是本地概念，远端没有）

两种格式的设计目标不同，因此结构也不同。直接渲染 `SDKMessage`，等于让本地 UI 去解析一种不属于它的格式——不是不可能，但会把两个系统的耦合混在一起。

## 翻译层需要处理的四类特殊消息

### 1. compact_boundary

上下文压缩（Compact）是 Claude 维持长会话的关键机制：当对话历史太长，自动把旧历史压缩成摘要，插入 `compact_boundary` 标记。

远端会话的压缩发生在 CCR 容器里，但本地 UI 需要在正确的位置显示分割线——"以下内容是在压缩后继续的"。

翻译层需要把 `compact_boundary` 类型的 `SDKMessage` 转成本地的 `isCompactSummary: true` 消息，让 UI 能正确渲染分割线。

### 2. tool_result 闭环

工具执行有个 UI 周期：

- 工具开始执行 → 显示进度指示器
- 工具执行完成 → 进度指示器消失，显示结果

在本地模式下，这个"开始-结束"周期由同一段代码直接管理。在远端模式下，工具执行在 CCR 里，本地只收到消息。翻译层需要识别 `tool_result` 消息，把对应工具的进度指示器清除掉。

如果不处理这个，用户会看到一直转的进度指示器——即使工具早就执行完了。

### 3. task_started / task_notification

后台任务（通过 slash command 发起的 Agent 任务）在远端执行时，会发出 `task_started` 和 `task_notification` 消息。本地 UI 维护着任务计数器（"3 tasks running"这样的状态）。

翻译层需要把这些消息映射到本地任务计数器的增减。如果不翻译，本地计数器永远是 0，UI 显示的状态和实际不符。

### 4. init 消息里的 slash commands

初始化消息（`init`）里可能包含历史会话里执行过的 slash commands（如 `/model`、`/editor`）。翻译层需要重放这些 commands 的效果——因为它们改变了会话的状态（选用的模型、编辑器设置等）——而不是把它们显示为普通文本消息。

## 不翻译会发生什么

假设翻译层缺失，直接把 `SDKMessage` 流投喂给本地 UI：

```
收到 compact_boundary → UI 不认识这个类型 → 忽略 → 没有分割线
收到 tool_result      → UI 不知道进度该停 → 进度一直转
收到 task_started     → 任务计数器不更新 → 显示 0 个任务
收到 init with cmd    → 会话状态没有重放 → 模型选择丢失
```

每一个"不翻译"都导致本地 UI 和真实状态的静默漂移——数据没有丢，但显示是错的。

## `isEligibleBridgeMessage` 的反向过滤

翻译不只是"把 SDKMessage 转成 Message"，还有一个反向的过滤：**哪些本地消息应该发给远端**。

`bridgeMessaging.ts` 里的 `isEligibleBridgeMessage`：

```typescript
export function isEligibleBridgeMessage(m: Message): boolean {
  // Virtual messages (REPL inner calls) are display-only
  if ((m.type === 'user' || m.type === 'assistant') && m.isVirtual) {
    return false
  }
  return (
    m.type === 'user' ||
    m.type === 'assistant' ||
    (m.type === 'system' && m.subtype === 'local_command')
  )
}
```

`isVirtual: true` 的消息是 REPL 内部的展示消息，不是真实的对话轮次，不应该发给远端。Tool result、progress 消息、compact 摘要，也都是内部的 REPL 状态，远端不需要。

这个过滤层和入方向的翻译层是对称的：入方向把 SDKMessage 转成 Message，出方向把 Message 过滤成可发的子集。

## 面试指导

"协议翻译层是否必要"是微服务架构里的经典讨论。

反对派的论点：翻译层增加了复杂度，也是 bug 的温床（如果翻译逻辑有误，很难排查）。

支持派的论点：如果两个系统的内部模型不同，强行让它们共享格式，会导致两者都无法自由演进——改一个字段，两边都要同步改。

Claude Code 的解法是分层隔离：`SDKMessage` 是稳定的对外接口，`Message` 是可以自由演进的内部格式，翻译层承担两者之间的映射。这个设计让 SDK 格式的变更不影响 UI，UI 的重构不影响 SDK 格式。
