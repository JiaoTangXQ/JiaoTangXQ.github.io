# 三次之后不再提醒：排队消息可编辑提示的退场设计

Claude Code 有一个不起眼但很有用的功能：当系统在处理任务时，用户提交的下一条消息会进入排队状态，而这条排队消息其实可以编辑——按上箭头就能回去改。

问题是：这个能力不直观。用户第一次排队，很可能完全不知道可以回头改。所以系统会在底部给一条提示：「按上箭头可以编辑排队中的消息」。

但它只给三次。

## 为什么要有终止条件

知道这条提示的人，第一次、第二次会真的用到它。第三次还给一遍，算是确保记住了。第四次，那就是把用户当成永远的新手。

Claude Code 里驾驶舱底部区域本来就紧张。每一行提示都在竞争同一块有限空间。如果每一个功能的引导都不肯退场，结果就是底部变成一排永久公告，真正需要注意的状态被淹没在反复教育里。

这类提示的正确生命周期是：**出现 → 完成教学 → 退场**。退场不是放弃，而是把空间让给更重要的事。

## 与语音 hint 的对比

这个模式和 `MAX_VOICE_HINT_SHOWS = 3` 使用了完全一样的结构：一个全局配置里的计数器，每次提示都递增，超过上限就不再渲染。

两者的共同点是：系统相信用户会学习，不打算把同一条操作提醒永远挂着。这是对用户的一种尊重——我教过你，你学会了，我就不重复。

## 提示的本质

真正好的操作提示不追求永久存在，而是追求「被恰好需要时才出现」。新手第一次排队消息时，提示有价值。熟手第十次排队时，同一条提示只是噪音。

有退场设计的提示，才是真正想帮人的提示。
