# selector 订阅不是语法细节，而是性能纪律

`useAppState(selector)` 最有价值的地方，不是 API 好不好看，而是它强迫组件承认一件事：你到底只依赖哪一小块事实。拿整棵状态树，就等于把自己绑到每一次无关变化上。

长会话、多面板、频繁任务更新时，这种纪律非常重要。Claude Code 不是等界面慢了再补救，而是一开始就用选择器限制重渲染传播。

## 实现链

`useAppState(selector)` 不是语法糖，而是订阅纪律：组件必须声明自己到底订阅哪一小片状态，`useSyncExternalStore` 再只在这一片结果变化时重渲染。`selectors.ts` 还专门要求 selector 保持纯净，只做数据提取，不带副作用。

## 普通做法

更常见的写法，是组件一次把整份 context 拿进来，再在组件内部随手解构自己想要的字段。

## 为什么不用

Claude Code 不接受整份订阅，是因为这会让底部提示、输入区和状态栏都被无关字段拖着一起重渲染。这里的性能纪律靠的就是“每个组件只承认自己真正在看哪一片现实”。

## 代价

代价是写组件时要更克制，selector 写坏了会直接伤性能；但这也逼着组件边界更清楚。
