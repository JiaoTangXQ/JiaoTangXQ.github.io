# 深度分析：统一转场协议为什么比多个入口各自实现更重要

## 两种进入后台的方式

**手动后台化**：用户正在前台看着一个任务，主动按下快捷键把它发到后台。这个操作是明确的、有意图的。

**自动后台化**：某些任务在创建时就被标记为"应该在后台运行"，或者在某些条件触发时自动转入后台（比如主会话需要处理另一件更紧急的事，当前任务自动退到后台）。这个操作是系统决定的，用户可能没有感知。

从触发来源看，这两种方式很不一样。

但从"任务的状态应该变成什么"看，它们是一样的：任务的 `isBackgrounded` 从 `false`（或不存在）变为 `true`，任务从主会话的主展示区移到后台面板，任务的输出继续写入 `outputFile`，`backgroundSignal` 机制开始负责通知主会话。

**面试题：如果结果一样，为什么不让两种入口各自实现这个状态变化？**

## 两个入口各自实现的问题

想象这样的代码：

```typescript
// 手动后台化路径
function handleManualBackground(taskId: string) {
  updateTaskState(taskId, { isBackgrounded: true })
  dismissFromMainSession(taskId)
  addToBackgroundPanel(taskId)
  // 手动路径的额外逻辑：记录用户操作日志
}

// 自动后台化路径
function handleAutoBackground(taskId: string) {
  updateTaskState(taskId, { isBackgrounded: true })
  dismissFromMainSession(taskId)
  addToBackgroundPanel(taskId)
  // 自动路径的额外逻辑：发出通知告知用户任务被自动后台化了
}
```

看起来代码重复不多，但随着功能演进，问题会出现：

**状态同步的遗漏**：有人在手动路径里加了"后台化时保存任务的当前滚动位置"，但忘了在自动路径里加相同逻辑。结果：手动后台化后调回来滚动位置还在，自动后台化后调回来滚动位置重置了。

**顺序问题**：后来发现 `addToBackgroundPanel` 必须在 `updateTaskState` 完成后才能调用（否则面板读取到的是旧状态）。在手动路径里修复了顺序，但没有意识到自动路径也需要同样的修复。

**测试覆盖**：测试了手动后台化，但没有测试自动后台化，两者的测试用例是分开写的，对应的行为可能悄悄地出现了分歧。

## 统一转场协议的实现

统一协议的意思是：不管触发来源，后台化的核心操作只在一个地方实现。

```typescript
// 统一的后台化协议
function backgroundTask(taskId: string, options: BackgroundOptions) {
  // 所有路径都必须做的事
  updateTaskState(taskId, { isBackgrounded: true })
  dismissFromMainSession(taskId)
  addToBackgroundPanel(taskId)
}

// 手动后台化
function handleManualBackground(taskId: string) {
  backgroundTask(taskId, { source: 'user' })
  logUserAction(...)  // 只在手动路径才做的事
}

// 自动后台化
function handleAutoBackground(taskId: string) {
  backgroundTask(taskId, { source: 'system' })
  notifyUserOfAutoBackground(...)  // 只在自动路径才做的事
}
```

现在，对"后台化必须做什么"的修改只需要在一个地方进行，手动和自动路径自动受益。测试核心逻辑只需要测试 `backgroundTask` 函数，不需要分别测试两条路径。

## 制度动作和入口动作的区分

这个设计背后有一个值得提炼的原则：**制度动作（系统对状态做什么）和入口动作（谁触发的）是两件不同的事，应该分开实现**。

"后台化"是一个制度动作：有明确的前置条件、明确的状态变化、明确的后置保证。

"用户按了快捷键"和"系统判断需要让路"是入口动作：决定了触发时机和触发来源，但不决定后台化本身怎么进行。

混在一起实现，会让"怎么触发"和"触发后做什么"耦合起来，任何一个的变化都可能影响另一个。

分开实现，入口可以自由演化（增加新的触发方式、改变触发条件），制度动作也可以自由演化（增加后台化时需要做的新步骤），两者互不干扰。

## 面试要点

1. **相同的结果应该有相同的路径**：手动和自动后台化产生的状态变化相同，核心操作应该在同一个地方实现，不是各自复制一份。
2. **制度动作和入口动作分开**：后台化的"做什么"（制度）和"谁触发"（入口）是不同关注点，混在一起会让两者相互干扰演化。
3. **分开测试的代价**：两个入口各自实现，需要各自测试，且可能出现行为分歧；统一实现，核心逻辑测试一次，两条路径都覆盖。
4. **统一协议让演进更安全**：新增后台化时必须做的步骤，只在一个地方加，不需要寻找并更新所有触发点。
