# 系统设计：如果让你设计 AI Agent 团队的 Leader 角色，职责边界在哪里？

面试官问：你在设计多 Agent 协作系统时，Leader 和 Worker 的职责边界怎么划？

这个问题没有唯一答案，但 Claude Code 的回答很清晰，而且背后有完整的工程推导。

## Claude Code 的主张：Leader = 唯一对外接口

Claude Code 给 Leader 的核心定位不是"能力最强的 Agent"，而是**团队对外的唯一接口**。

具体体现在三个方面：

**用户交互收口**：系统提示明确写：`The user interacts primarily with the team lead`。Worker 的输出默认对用户不可见，Worker 想让用户知道某件事，必须通过 SendMessage 发给 Leader，由 Leader 决定是否转告用户。

**权限审批收口**：所有 Worker 的权限请求（不管是 in-process 走 leaderPermissionBridge，还是 tmux 走 permissionSync）最终都在 Leader 的 UI 队列里排队，由用户通过 Leader 界面统一处理。

**消息可见性收口**：Worker 之间的 SendMessage 对用户不可见，只有发给 Leader 或者 Leader 主动查询的内容，才出现在用户视野里。

## 为什么不是"能力最强的 Worker"

"Leader 是最强的 Worker" 这个定位会导致什么？

**用户界面分裂**：如果 Leader 只是能力强，它还是在执行任务，不是在统一接口。5 个 Agent 都可以随时给用户发消息，用户面对的是 5 个来源的并发输出。

**权限审批分散**：每个 Agent 都有一定能力的 Worker 是平等的，权限应该向谁申请？如果 Worker 们直接找用户，用户同样面对并发打断。如果 Worker 们找 Leader，Leader 是以"权威者"身份还是以"代理人"身份处理？

**仲裁无依据**：当多个 Worker 产生冲突意见时（比如 researcher 说应该用方案 A，reviewer 说应该用方案 B），谁来做最终决定？如果 Leader 只是更强的 Worker，它凭什么有仲裁权？

"唯一对外接口"的定位解决了这些问题——Leader 不是"最强执行者"，而是"唯一代理人"，代表团队对外的所有正式接口。

## 接口收口的具体实现

**leaderPermissionBridge.ts**：暴露两个 setter，让非 React 的执行循环能往 Leader 的确认队列里推请求。

**permissionSync.ts**：`pending/` 目录里积累的权限请求，Leader 通过轮询发现并呈现给用户。

**inProcessRunner.ts**：完成时只发 idle notification，不把完整内容自动灌给 Leader。Leader 需要内容时，靠 Worker 的 SendMessage 主动发过来。

这三个机制共同保证了：**用户面对的 UI 只有一个（Leader 的 UI），不管后端有多少 Worker 在并发工作**。

## 代价：Leader 是软瓶颈

这个设计有明显的代价：Leader 成为团队的串行瓶颈点。

所有需要用户参与的操作（权限确认、结果反馈、方向调整）都通过 Leader。如果 5 个 Worker 同时需要权限，它们会在 Leader 的队列里排队，第 2 个 Worker 需要等第 1 个被处理后才能得到响应。

但这个"瓶颈"在实践中是可接受的，原因是：

1. 权限请求的频率通常不高（每几分钟一次），队列很少真正积压。
2. 用户的处理速度本身就是限速因素——你一次只能思考一个请求，无论是否有队列，你实际都是串行处理的。
3. 相比"5 个并发弹窗"，排队等候反而让用户能更专注地处理每一个请求。

## 这个设计回答了什么问题

如果用一句话概括：**Leader 的存在是为了保护用户不被多 Agent 并发的复杂性淹没**。

Worker 们可以尽情并发——并发执行是它们的价值所在。但这些并发的"触手"，对用户都是通过 Leader 这个单一界面感知的。Leader 把"内部有多少 Agent 在并发工作"这个事实，翻译成用户可以线性理解和响应的单一序列。

这不是技术上的中心化，而是**体验上的中心化**——执行并行，接口串行。

## 面试怎么答

面试题是"如何设计 Leader 角色的职责边界"：

**低分答案**：Leader 执行最难的任务，协调其他 Worker，汇总结果。这是"能力最强的 Worker"定位，没有回答接口问题。

**中等答案**：Leader 负责与用户通信，Worker 专注执行。这正确了，但还不完整。

**高分答案**：Leader 是团队对外的唯一接口，具体包括：统一承接用户交互（用户主要面对 Leader）、统一承接权限审批（所有 Worker 的权限请求都通过 Leader 排队呈现）、统一控制消息可见性（Worker 的输出默认不直接可见）。这三个收口保护了用户不被并发复杂性淹没。

**加分项**：能说清楚这个设计的代价（Leader 是软瓶颈）以及为什么这个代价是可接受的（用户本身就是串行处理者，队列比并发弹窗实际上更高效）。好的架构设计不是没有代价，而是代价可以被清晰量化和接受。
