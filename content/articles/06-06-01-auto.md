# 进入 auto 前为什么要主动剥离 Bash(*) 这样的放行规则？

这是一个工程取舍题：用户明确设置的规则，系统为什么有权修改？

## 场景重现

假设用户在 `~/.claude/settings.json` 里配了：

```json
{
  “permissions”: {
    “allow”: [“Bash”]
  }
}
```

这条规则在 default 模式下意味着”允许所有 bash 命令，不弹窗”。用户这样配置，通常是因为信任自己的工作环境，觉得每次 git 操作都要确认太烦。

现在用户按 Shift+Tab 切换到 auto 模式。如果这条规则原封不动保留，会发生什么？

## 保留规则的危险

auto 模式的核心保证是：**所有不在白名单里的工具调用都要经过分类器评估**。但 `Bash` 的全量 allow 规则会绕过这个流程。

具体来说，运行时权限检查的逻辑是：先看 allow 规则，如果匹配就直接放行，不走分类器。所以 `Bash(*)` 的 allow 规则实质上等于”所有 bash 命令不需要分类器审批”——这破坏了 auto 模式存在的意义。

从用户角度看，auto 模式的承诺是”我会智能判断什么是安全的”。如果实际上某些操作因为旧规则绕过了分类器，这个承诺就是假的。

## stripDangerousPermissionsForAutoMode 的实现

```typescript
export function stripDangerousPermissionsForAutoMode(
  context: ToolPermissionContext,
): ToolPermissionContext {
  const rules: PermissionRule[] = []
  for (const [source, ruleStrings] of Object.entries(context.alwaysAllowRules)) {
    if (!ruleStrings) continue
    for (const ruleString of ruleStrings) {
      const ruleValue = permissionRuleValueFromString(ruleString)
      rules.push({ source: source as PermissionRuleSource, ruleBehavior: 'allow', ruleValue })
    }
  }

  const dangerousPermissions = findDangerousClassifierPermissions(rules, [])
  if (dangerousPermissions.length === 0) {
    return { ...context, strippedDangerousRules: context.strippedDangerousRules ?? {} }
  }

  // 记录被剥离的规则到 strippedDangerousRules（用于离开 auto 时恢复）
  const stripped: ToolPermissionRulesBySource = {}
  for (const perm of dangerousPermissions) {
    if (!isPermissionUpdateDestination(perm.source)) continue
    ;(stripped[perm.source] ??= []).push(permissionRuleValueToString(perm.ruleValue))
  }

  return {
    ...removeDangerousPermissions(context, dangerousPermissions),
    strippedDangerousRules: stripped,  // 暂存被剥离的规则
  }
}
```

被剥离的规则不是删除，而是存在 `strippedDangerousRules` 字段里。这是为了离开 auto 模式时能精确恢复。

## isDangerousBashPermission：危险判断的粒度

```typescript
export function isDangerousBashPermission(
  toolName: string,
  ruleContent: string | undefined,
): boolean {
  if (toolName !== BASH_TOOL_NAME) return false

  // 工具级 allow（Bash 无参数，或 Bash(*)）：允许所有命令
  if (ruleContent === undefined || ruleContent === '') return true

  const content = ruleContent.trim().toLowerCase()
  if (content === '*') return true

  // 解释器前缀模式
  for (const pattern of DANGEROUS_BASH_PATTERNS) {
    const lowerPattern = pattern.toLowerCase()
    if (content === lowerPattern) return true
    if (content === `${lowerPattern}:*`) return true  // python:*
    if (content === `${lowerPattern}*`) return true   // python*
    if (content === `${lowerPattern} *`) return true  // python *
    if (content.startsWith(`${lowerPattern} -`) && content.endsWith('*')) return true
  }
  return false
}
```

`DANGEROUS_BASH_PATTERNS` 包含了所有能执行任意代码的解释器：python、node、ruby、perl、php、java 等。

**关键细节**：`Bash(git *)` 这样的规则不会被剥离——前缀匹配的 git 命令是安全的（`isDangerousBashPermission` 检测不到 “git” 在危险列表里）。只有”能执行任意代码”的模式才会被标记为危险。

## Agent allow 规则也是危险的

除了 Bash，Agent 类型的 allow 规则在 auto 模式下也是危险的：

```typescript
export function isDangerousTaskPermission(
  toolName: string,
  _ruleContent: string | undefined,
): boolean {
  return normalizeLegacyToolName(toolName) === AGENT_TOOL_NAME
}
```

任何 `Agent` 的 allow 规则都会被剥离。原因是：如果子 agent 被预先授权，分类器在评估主线程的委托指令时，无法知道子 agent 实际上会执行什么——委托攻击（delegation attack）的防御就失效了。

## 剥离时不修改磁盘文件

重要的安全保证：`stripDangerousPermissionsForAutoMode` 只修改内存里的 `ToolPermissionContext`，**不修改磁盘上的 settings 文件**。用户离开 auto 模式后，之前在 `~/.claude/settings.json` 里设的规则一字不动地还在。

`removeDangerousPermissions` 里的 `isPermissionUpdateDestination` 检查确保了只有能持久化的来源（`userSettings`、`projectSettings` 等）的规则会进入 `strippedDangerousRules` 暂存，CLI 参数和 policySettings 来源的规则直接忽略（它们不能被恢复到磁盘，也不需要恢复）。

## 这道题考什么

**核心考点**：能否解释”为什么系统要在用户不知情的情况下修改权限”——这不是 bug，而是 auto 模式的安全契约。用户选择 auto 模式，等于同意”让分类器决定安全边界”，旧的宽松规则和这个契约冲突，必须暂时搁置。

**细节考点**：剥离是内存操作，不影响磁盘。`strippedDangerousRules` 暂存确保了可逆性。这个”剥离-暂存-恢复”的三段式设计，是一个值得讨论的工程模式。
