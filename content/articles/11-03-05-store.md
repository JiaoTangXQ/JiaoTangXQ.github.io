# 源码解读：50 行的 createStore，为什么足够承载整个 Claude Code 的状态管理？

Redux 的 `createStore` 实现是 200+ 行。Zustand 的核心也有 100+ 行。

Claude Code 的 `createStore` 是 35 行。

更少的代码不一定是更好的代码，但在这里，35 行确实是完整且足够的。理解这 35 行，就理解了整套订阅机制的工作方式。

---

## 完整源码

```typescript
type Listener = () => void
type OnChange<T> = (args: { newState: T; oldState: T }) => void

export type Store<T> = {
  getState: () => T
  setState: (updater: (prev: T) => T) => void
  subscribe: (listener: Listener) => () => void
}

export function createStore<T>(
  initialState: T,
  onChange?: OnChange<T>,
): Store<T> {
  let state = initialState
  const listeners = new Set<Listener>()

  return {
    getState: () => state,

    setState: (updater: (prev: T) => T) => {
      const prev = state
      const next = updater(prev)
      if (Object.is(next, prev)) return
      state = next
      onChange?.({ newState: next, oldState: prev })
      for (const listener of listeners) listener()
    },

    subscribe: (listener: Listener) => {
      listeners.add(listener)
      return () => listeners.delete(listener)
    },
  }
}
```

---

## 每一行都有理由

**`let state = initialState`**

状态用普通变量持有，不是 class 属性，不是 ref，就是一个闭包里的 `let`。简单有效。

**`const listeners = new Set<Listener>()`**

监听者用 `Set` 而不是 `Array`。`Set` 保证每个监听函数只能注册一次（虽然实际上每次 `subscribe` 都传入新的函数引用，不会真正重复），更重要的是 `Set` 的迭代和删除都是安全的——可以在迭代过程中删除元素而不出问题。

**`if (Object.is(next, prev)) return`**

这是这 35 行里最重要的一行。

`Object.is` 做的是引用相等比较（对对象类型而言），不是深比较。如果 `updater` 返回了 `prev`（没有创建新对象），store 会认为状态没有变化，跳过所有通知，不触发任何重渲染。

这有两个含义：

1. **效率**：如果状态实际上没变，什么都不做
2. **约束**：调用者必须在有真实变化时返回新对象，不能修改 prev 后原样返回

**`onChange?.({ newState: next, oldState: prev })`**

`onChange` 是可选的副作用回调，在状态变化之后、通知监听者之前调用。`onChangeAppState` 就在这里被触发。

注意顺序：`state = next` 先执行，然后调用 `onChange`，然后通知监听者。这意味着 `onChange` 里如果调用 `store.getState()`，已经能拿到新状态。

**`for (const listener of listeners) listener()`**

通知所有监听者。监听者是什么？是 `useSyncExternalStore` 注册的回调，React 用它来知道「store 变化了，需要重新读取状态」。

监听者本身只是一个空函数（`() => void`），不携带任何信息。React 收到通知后自己调用 `getState` 读取最新状态，然后调用 selector，比较结果。

**`return () => listeners.delete(listener)`**

`subscribe` 返回一个取消订阅函数，这是 React 的 `useSyncExternalStore` API 要求的。组件卸载时，React 会调用这个清理函数，避免内存泄漏。

## useState 保证 store 只创建一次

```typescript
// AppStateProvider 内部
const [store] = useState(
  () => createStore<AppState>(
    initialState ?? getDefaultAppState(),
    onChangeAppState,
  )
)
```

`useState` 接受初始化函数（lazy initialization），函数只在第一次渲染时调用。这确保：

- `store` 引用在整个组件生命周期内保持稳定
- Context value（`AppStoreContext.Provider value={store}`）永远是同一个对象，不触发 Context 消费者的重渲染

如果用普通变量 `const store = createStore(...)` 而不是 `useState`，每次组件重渲染都会创建新的 store 实例，所有订阅者的连接都会断开。

## 和 Redux createStore 的对比

Redux 的 `createStore` 多了什么：

| 功能 | Claude Code store.ts | Redux createStore |
|------|---------------------|-------------------|
| 状态读取 | `getState()` | `getState()` |
| 状态更新 | `setState(updater)` | `dispatch(action)` + reducer |
| 订阅/取消 | `subscribe(listener)` | `subscribe(listener)` |
| 变更检测 | `Object.is` | `Object.is` |
| 中间件 | 无 | `applyMiddleware` |
| 增强器 | 无 | `enhancer` |
| 时间旅行 | 无 | Redux DevTools |

Claude Code 不需要中间件（`onChangeAppState` 直接处理副作用）、不需要增强器、不需要时间旅行（终端工具没有浏览器 DevTools）。这些功能不需要，就不引入它们的复杂度。

---

## `setState(updater)` 而不是 `setState(value)`

Redux 的 `dispatch` 接受 action 对象，reducer 决定怎么处理。Claude Code 的 `setState` 接受 updater 函数，调用方直接描述「如何从旧状态变成新状态」。

```typescript
// Redux 风格
dispatch({ type: 'PERMISSION_MODE_CHANGED', payload: { mode: 'plan' } })
// 然后在 reducer 里处理 PERMISSION_MODE_CHANGED

// Claude Code 风格
setState(prev => ({
  ...prev,
  toolPermissionContext: {
    ...prev.toolPermissionContext,
    mode: 'plan',
  }
}))
```

Redux 风格的好处是 action 有名字，可以用 DevTools 追踪。Claude Code 风格更直接，不需要 action creator 和 reducer 样板。在这个场景里，直接性更有价值。

## 面试指导

**为什么 `setState` 接受 updater 函数而不是直接接受新状态值？**

updater 函数确保更新基于最新状态。如果有两个更新同时到达（比如来自不同的事件处理器），updater 保证第二次更新看到的是第一次更新后的状态，不会丢失第一次更新。这是避免竞态条件的标准模式。

**Object.is 和 === 在状态比较里的区别？**

`Object.is(NaN, NaN)` 为 true（`NaN === NaN` 为 false），`Object.is(+0, -0)` 为 false（`+0 === -0` 为 true）。在状态比较里，`Object.is` 更符合「是不是真的同一个值」的语义。对对象引用来说，两者没有区别。

**监听者 Set 在迭代时删除元素安全吗？**

对 JavaScript 的 `Set` 而言，规范明确说明：在 `forEach` 迭代中删除已经被访问的元素不影响迭代，删除未被访问的元素也不影响（被删除的元素不会被访问到）。但添加新元素可能会导致它们被访问。对 `for...of` 迭代也有类似保证。
