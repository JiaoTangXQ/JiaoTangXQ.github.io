# 对比分析：观测系统和业务系统为什么不能混住？从 Provider 嵌套顺序说起

这是一道「看起来简单但能区分深度」的设计题。

题目：`FpsMetricsProvider` 和 `StatsProvider` 为什么放在 `AppStateProvider` 外面，而不是里面？如果换个位置，会有什么问题？

---

## 先建立两个概念

**业务现实（Business Reality）**：驱动应用行为的事实。权限模式是 default 还是 auto、当前运行了哪些任务、用户的 MCP 连接状态——这些事实变化会直接影响界面行为和用户操作。

**观测数据（Observability Data）**：描述应用运行状况的数据。帧率是 58fps、这次会话里工具调用了多少次、内存用了多少——这些数据用于监控和诊断，不驱动业务逻辑。

Claude Code 的 Provider 嵌套清楚地反映了这个区分：

```
FpsMetricsProvider   ← 观测层（最外）
  StatsProvider      ← 观测层
    AppStateProvider ← 业务现实层（最内）
```

---

## 如果把 FPS 放进 AppState，会发生什么

假设我们这样做：

```ts
// 如果 AppState 里有 FPS
type AppState = {
  // ... 业务字段
  currentFps: number    // 坏主意
  avgFps: number        // 坏主意
}
```

然后在某个地方定期更新它：

```ts
useEffect(() => {
  const timer = setInterval(() => {
    const metrics = getFpsMetrics()
    setAppState(prev => ({ ...prev, currentFps: metrics?.currentFps ?? 0 }))
  }, 100)
  return () => clearInterval(timer)
}, [])
```

问题一：**每秒 10 次 setState，每次触发所有 AppState 订阅者重渲染**。

Claude Code 的 `AppState` 有几十个字段，有些组件订阅的是 `toolPermissionContext`，有些订阅 `tasks`，有些订阅 `footerSelection`。它们和 FPS 完全无关，但如果 FPS 住在 AppState 里，每次 FPS 更新就会触发所有人重渲染。

即使用了 selector——`useAppState(s => s.toolPermissionContext)`——`useSyncExternalStore` 仍然会在每次 AppState 变化时调用 `store.subscribe` 的所有监听者，然后调用 selector 检查结果是否变化。每秒 10 次 FPS 更新 = 每秒调用所有 selector 10 次。这不是灾难，但是无意义的开销。

问题二：**观测数据污染了业务状态的语义**。

当新团队成员打开 `AppStateStore.ts` 理解系统时，他应该能从里面读出「这台机器当前的业务现实是什么」。如果里面有 `currentFps`，他就会困惑：我们需要在业务逻辑里用到 FPS？是不是有哪个功能依赖帧率？

## FpsMetrics 的实际实现：getter 模式

```tsx
// fpsMetrics.tsx（概念还原版）
const FpsMetricsContext = createContext<(() => FpsMetrics | undefined) | undefined>(undefined)

export function FpsMetricsProvider({ getFpsMetrics, children }) {
  return (
    <FpsMetricsContext.Provider value={getFpsMetrics}>
      {children}
    </FpsMetricsContext.Provider>
  )
}

export function useFpsMetrics() {
  return useContext(FpsMetricsContext)  // 返回 getter 函数，不是数值
}
```

Context 存的是 **getter 函数**，不是 FPS 数值。这意味着：

- Context 值永远不变（函数引用稳定），不触发任何重渲染
- 消费者需要 FPS 时主动调用 getter 取数据
- FPS 的更新完全不经过 React 的更新机制

这是「拉取」（pull）而不是「推送」（push）模式。观测数据只在被需要时才被读取，不主动推入 React 的更新队列。

## Stats 为什么也在外层

`StatsProvider` 的情况稍有不同。Stats 不像 FPS 那样高频更新，但它同样是观测数据。

关键区别在于：Stats 需要在进程退出时落盘（`process.on('exit', flush)`），而这个副作用与任何 AppState 字段无关。把 Stats 放在 AppStateProvider 外层，意味着即使 AppState 树被卸载，Stats 也仍然存在，仍然能在退出时完成落盘。

另一个原因：Stats 需要观测整棵应用树的行为，包括 AppState 子树里发生的事情。如果 Stats 在 AppState 里面，它就无法观测到自己的父级，也无法统计某些跨层级的指标。

---

## 观测在外 vs 观测在内的架构对比

**观测在外（Claude Code 的做法）**

```
观测层包裹业务层
观测层 → 业务层（单向依赖）
业务层无法引用观测层
观测失效不影响业务运行
```

**观测在内（混住的问题）**

```
业务层包含观测数据
任何观测更新 → 触发业务订阅者
业务语义被观测细节污染
调试时难以区分「业务状态」和「仪表读数」
```

这不是 React 特有的问题。后端的可观测性设计（日志、metrics、tracing）也遵循同样的原则：telemetry 系统应该是旁路（sidecar），不应该侵入业务代码路径。

## 面试指导

**追问：如果 AppState 里的某个字段确实需要 FPS 来做判断怎么办？**

一种做法：不直接把 FPS 存进 AppState，而是在需要这个判断的地方临时调用 `useFpsMetrics()` 的 getter 读取当前值，做一次性的判断，不订阅它。

另一种做法：如果确实有业务逻辑依赖 FPS（比如「FPS 低于阈值时降低动画精度」），那应该有一个专门的系统订阅 FPS getter、做判断、然后把判断结果（比如 `shouldReduceAnimation: boolean`）放进 AppState——把观测转化为业务结论再进入业务层。

**追问：这种分层在纯后端服务里有类比吗？**

有。Prometheus metrics 是观测层，不应该混进业务的数据库里。业务数据库存的是「用户有多少积分」，Prometheus 存的是「这个接口的 p99 延迟是多少」。前者驱动业务，后者支持诊断。
