# 观测层包在事实层外说明观察现实不等于决定现实

这几个 provider 的顺序很有意思。最外层是 `FpsMetricsProvider`，其次是 `StatsProvider`，最里面才是 `AppStateProvider`。也就是说，观测层包着事实层，而不是反过来。

Claude Code 的态度很清楚：系统可以被观察，但观察本身不该混成共同现实。性能和统计是旁路视角，不是世界本体。

## 实现链

`FpsMetricsProvider` 和 `StatsProvider` 都包在 `AppStateProvider` 外面，说明观测层先于事实层装配，但并不拥有事实本身。FPS 和统计可以观察整棵树怎样运行，却不直接写进 `AppState` 这份共同现实，观察现实和决定现实被故意拆开了。

## 普通做法

更容易想到的做法，是把统计、性能和业务状态都塞进同一个 store，反正读起来方便，哪里要显示就从同一处拿。

## 为什么不用

Claude Code 不这么合并，是因为观测数据和业务事实不是一回事。把它们混住在同一个状态世界里，订阅会变脏，组件也会被一堆只用于观测的更新拖着跑。

## 代价

代价是层次更多，观测链和事实链要分开理解；好处则是业务现实不会被监控数据污染，性能和状态职责更清楚。
