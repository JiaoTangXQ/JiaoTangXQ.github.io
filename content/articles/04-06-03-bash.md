# 边界条件：为什么 Bash 失败会取消同批兄弟工具，但 ReadFile 失败不会？

直觉上，并发执行的工具应该是相互独立的——其中一个失败，其他工具继续跑完，收集所有结果，再把失败和成功一起报给模型。这是最常见的"fail-partial"策略，大多数并发任务框架默认就是这样。

但 `StreamingToolExecutor` 在处理 Bash 错误时做的不是这个。

## 观察：Bash 的错误会传播

在 `StreamingToolExecutor` 内部，当一个 Bash 工具执行失败（返回了错误结果），执行器会触发 `siblingAbortController.abort()`，取消同一批里还在执行中的其他工具：

```typescript
// StreamingToolExecutor 内部（简化逻辑）
if (isErrorResult(result) && tool.name === BASH_TOOL_NAME) {
  // 取消同批兄弟
  this.siblingAbortController.abort(`Cancelled: parallel tool call ${siblingId} errored`)
}
```

被取消的兄弟工具会收到一个合成的 `tool_result`，内容类似：

```
Cancelled: parallel tool call bash_tool_abc123 errored
```

## 为什么 Bash 比 ReadFile 特殊？

关键在于 Bash 的**语义**。

ReadFile 是一个声明性的读操作：给一个路径，返回内容。如果文件不存在，报错很直接：`File not found`。这个错误不改变文件系统，也不影响其他文件的可读性。并发执行的 ReadFile B 失败了，ReadFile A 的结果依然是有效的——它们的执行互不依赖。

Bash 是一个命令性的执行操作：它运行真实的 shell 命令，可以有任意副作用。更重要的是，Bash 命令在实际工作中经常存在**隐含的依赖链**：

```bash
# 假设模型调用了两个 Bash 工具
Bash A: mkdir -p /tmp/workspace && cd /tmp/workspace
Bash B: git clone https://... repo
```

如果 Bash A 失败了（比如权限不足，`/tmp/workspace` 没有创建成功），Bash B 的 `git clone` 会在错误的工作目录里执行。结果可能是：
- clone 成功（但放错了地方）
- clone 失败（工作目录不存在）
- 更糟的情况：clone 把文件放到了意想不到的地方

这种情况下，让 Bash B 继续跑完，反而会把环境弄得更乱，后续的清理和恢复会更复杂。

## 这是一个「fail-fast」vs「fail-partial」的选择

这两种策略的适用场景不同：

**Fail-partial（各自独立，不传播）**：
- 适用于工具之间没有隐含依赖
- 适用于"部分成功总比全部失败好"的场景
- 例：同时搜索 5 个不同的 API，有些超时，有些成功，成功的结果依然有用

**Fail-fast（传播失败，取消兄弟）**：
- 适用于工具之间有隐含的顺序或状态依赖
- 适用于"部分执行比不执行更危险"的场景
- 例：一系列 Bash 命令组成了一个操作序列，中间断了后面的都没意义

Claude Code 的判断是：Bash 工具在大多数使用场景下属于第二类。

## 这个判断是否过于保守？

可能是。有些 Bash 命令确实是独立的读操作：

```bash
# 这两个 Bash 命令完全独立
Bash A: cat /etc/hosts
Bash B: git log --oneline -10
```

如果 A 失败了，B 的结果依然是有效的。但 `StreamingToolExecutor` 还是会取消 B。

这是一个**保守策略的副作用**：为了避免"部分执行后现场混乱"，牺牲了"部分独立 Bash 命令的并行成果"。

Claude Code 接受了这个代价，原因是：失败恢复比失败预防更难。如果允许 Bash 兄弟继续执行，主循环会得到一组"部分成功，部分失败，但世界状态不确定"的工具结果。模型要基于这种不确定的状态做决策，下一步行动可能是基于一个已经不一致的前提。

相比之下，取消兄弟工具，让主循环收到一个"这批工具全部失败"的确定结果，模型可以从一个清晰的失败状态重新规划，代价是损失了那些本可成功的兄弟任务的结果。

## 与 ReadFile 的实际对比

```
场景：并发执行 [ReadFile A, Bash B]

ReadFile A 失败 → Bash B 继续跑完（fail-partial）
Bash B 失败 → ReadFile A ??? 

答案取决于实现。如果 Bash B 使用了 siblingAbortController，
且 ReadFile A 也响应这个 abort signal，那么 ReadFile A 会被取消。
```

这里有一个值得注意的细节：sibling abort 的传播方向不一定是对称的。`StreamingToolExecutor` 里的实现需要工具在执行时检查 `siblingAbortController.signal`，而不同工具对这个信号的响应可能不同。

## 面试指导

这道题考察的是**错误隔离策略的选择**。面试答案的框架：

1. **先问工具之间是否有隐含依赖**：如果有，fail-fast 更安全；如果没有，fail-partial 效率更高
2. **分析不同失败的后果**：
   - 失败后环境状态是否确定？（Bash 失败后状态可能模糊）
   - 部分执行是否比不执行更危险？（Bash 的部分执行往往更危险）
3. **选择默认行为**：Claude Code 选择了 Bash 失败时保守地取消兄弟，接受了损失部分独立 Bash 结果的代价

能说出"针对不同工具的风险模型选择不同的错误传播策略"的候选人，面试官会认为他理解了并发错误处理的核心难点。
