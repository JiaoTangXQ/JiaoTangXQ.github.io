# 为什么 Bash 错误会带崩兄弟工具

从表面上看，一个并发工具批次里的每个工具都像彼此独立的任务。既然独立，那其中一个失败，其他兄弟继续跑完，似乎也很自然。但 Claude Code 在这里做了一个不那么“教科书式”的决定：如果出错的是 Bash，它会主动中止同批里还在跑的兄弟工具。

这个决定一点也不武断。Bash 和很多读型工具不一样，它经常处在隐含的依赖链里。前一个命令负责创建目录，后一个命令才往里写文件；前一个命令切了分支，后一个命令才继续操作仓库。只要链条前面断了，后面那些命令继续跑下去，不仅没有意义，还可能把现场弄得更脏。`StreamingToolExecutor` 因此专门保留了一个 sibling abort controller，一旦 Bash 产出错误结果，就会触发同批兄弟的取消。

这里最见功力的地方，是它没有把所有错误都一视同仁。读文件、抓网页、搜代码这类能力，即使一个失败，别的工具往往仍然有价值；而 Bash 的失败更接近“现场条件已经变了”。Claude Code 不是追求形式统一，而是在不同工具背后看见了不同的现实含义。

这也是成熟 agent 和玩具 agent 的区别。玩具系统会把所有工具抽象成同一种盒子，成熟系统会承认不同工具对应不同风险模型。Claude Code 在这里的选择很像一个经验很足的工程师：宁可少一点表面上的并发优雅，也不要让错误沿着隐含依赖链悄悄扩散。

代码里的做法是专门给兄弟工具挂了一个 `siblingAbortController`。一旦某个并行工具，尤其是 `Bash` 这种会大范围改动现场的工具报错，执行器就会为剩余工具合成 `tool_result` 错误块，原因写成 `Cancelled: parallel tool call ... errored`，而不是放任其他兄弟默默跑到底。更直觉的实现是“谁失败谁负责，其他已经在跑的任务继续跑完”，因为这样吞吐更高，逻辑也更局部。

Claude Code 保守地一锅端掉兄弟工具，是因为它更怕“部分成功但世界已经不再一致”。对只读工具来说，这种取消有时显得偏重；但对 Bash 这种副作用工具来说，这种保守比让主循环在一堆半旧半新的事实之上继续推理更稳。代价当然是可能损失部分本可保留的并行成果。

## 实现链
Bash 类错误会触发 `siblingAbortController`，给兄弟工具补取消型 `tool_result`。

## 普通做法
谁失败谁结束，兄弟任务继续跑。

## 为什么不用
因为副作用世界一旦被某个兄弟改坏，其他并行结果就未必可信。

## 代价
可能会取消掉本可保留的并行成果。
