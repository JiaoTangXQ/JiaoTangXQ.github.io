# 会话持久化：Claude Code 如何把一次对话保存到磁盘又能无缝恢复？

`/resume` 命令可以恢复最近的会话，`--resume` 参数可以指定会话 ID 恢复。这意味着每次对话的完整状态都被持久化到了磁盘。

`utils/sessionStorage.ts` 是这套持久化系统的实现，它有几个值得深挖的设计决策。

## 会话日志的格式

会话数据以结构化日志形式保存。每一条 `Entry` 是一个 JSON 对象：

```typescript
type Entry =
  | SerializedMessage         // 对话消息
  | FileHistorySnapshotMessage // 文件历史快照
  | AttributionSnapshotMessage // 代码归因快照
  | ContentReplacementEntry   // 工具结果替换记录
  | TranscriptMessage         // 完整 transcript
  | QueueOperationMessage     // 消息队列操作
  | ContextCollapseCommitEntry
  | ContextCollapseSnapshotEntry
```

日志文件不是整个会话的 JSON dump，而是**增量追加**的多行 JSON（每行一条 Entry）。这有几个好处：
1. 不需要在每次更新时重写整个文件
2. 可以用 `readFileTailSync` 只读最后 N 行，恢复最近状态
3. 进程崩溃时已写入的部分不会丢失

## 尾部读取的同步实现

会话恢复时需要读取文件末尾（最近的状态通常在文件末尾）。有趣的是这里用了**同步读取**：

```typescript
import { closeSync, fstatSync, openSync, readSync } from 'fs'

function readFileTailSync(path: string, bytes: number): string {
  const fd = openSync(path, 'r')
  const stats = fstatSync(fd)
  const readFrom = Math.max(0, stats.size - bytes)
  const buffer = Buffer.alloc(Math.min(bytes, stats.size))
  readSync(fd, buffer, 0, buffer.length, readFrom)
  closeSync(fd)
  return buffer.toString()
}
```

为什么不用 async？因为这个读取发生在启动阶段，在第一次 render 之前。如果用 async，就需要等待 Promise 解析，而 React/Ink 的初始化是同步的——混用异步会让启动流程更复杂。

注释里明确说明：`// Sync fs primitives for readFileTailSync — separate from fs/promises imports`，特意把同步和异步的 fs 函数分开 import，防止混淆。

## 会话恢复的关键状态

恢复一个会话需要重建哪些状态？从 `sessionStorage.ts` 的 Entry 类型可以看到：

1. **消息历史**：`SerializedMessage` — 完整的对话记录
2. **文件历史**：`FileHistorySnapshotMessage` — Claude 修改了哪些文件，可以用来恢复撤销功能
3. **团队上下文**：存储在 transcript 里，`reconnection.ts` 负责从 transcript 恢复团队身份
4. **工具结果替换**：`ContentReplacementEntry` — 大型工具结果被替换成文件引用，恢复时需要知道映射关系


在生成会话文件名时，`planSlugCache` 缓存了从 Claude 的计划描述生成的 slug（比如"refactor-authentication-module"）。这个 slug 被用作会话的人类可读标识符。

缓存的原因：生成 slug 需要调用模型，这不应该在每次会话数据更新时都发生。缓存保证了一个会话的 slug 只生成一次，后续写入都复用同一个值。

## 面试角度

会话持久化是 stateful 应用的核心问题。Claude Code 的方案有几个值得讨论的点：

1. **追加日志 vs 全量快照**：追加写入更高效，但需要在恢复时重放。权衡是什么？
2. **同步 IO 在启动阶段的使用**：不是所有 IO 都应该异步，启动初始化有时用同步更简单可靠
3. **进程崩溃安全**：append-only 日志天然支持崩溃安全，不需要事务机制
4. **UUID 隔离多实例**：会话 ID 作为隔离边界，比锁机制更简单
