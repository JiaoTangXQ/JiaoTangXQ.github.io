# 浏览器配对会写回持久身份，说明外延能力也要有长期记忆

浏览器扩展一旦配对成功，Claude Code 不把它当成一次性握手就算了，而是把 `pairedDeviceId` 和 `pairedDeviceName` 写回全局配置。这样做的意思很明确：浏览器不是一阵风吹来的临时外设，而是这台工作台长期协作的一个现实端点。

更妙的是，如果配对身份没变，它连配置都不会重写。这里既有“长期记忆”，也有“不乱惊动”的克制。一个系统要想把手真正伸出终端，就不能只会临时连上去，还得记得自己究竟是和谁在长期合作。

## 实现链
`createChromeContext()` 的 `onExtensionPaired()` 会把 `pairedDeviceId/pairedDeviceName` 写回全局配置，`getPersistedDeviceId()` 再把这份持久身份读出来用于后续连接。浏览器接入不是一次性匿名会话。

## 普通做法
普通做法是每次启动都重新配对，把设备身份当临时变量。

## 为什么不用
Claude Code 不这么做，因为浏览器桥本来就是长期设施。没有稳定身份，重连和恢复都只能从零开始。

## 代价
代价是要承担持久状态失效的风险，比如设备换名或旧配对信息过期。
