# sticky header 点击后为什么要先置 'clicked' 再 scrollTo？两段式动作的设计原因

一个 1px（或 1 行）的像素对位问题，在终端界面里可以是很明显的错位。

## sticky header 的布局结构

在 `FullscreenLayout` 里，sticky prompt header 显示时会占用顶部的 1 行空间（`paddingTop={1}`）。这意味着内容区域从屏幕行 1 开始（行 0 是 header），而不是行 0。

当 sticky header 不显示时（`padding={0}`），内容区域从屏幕行 0 开始。

这 1 行的差异在视觉上很细微，但在"跳转到 prompt 位置"时会产生明显的错位：

```
[ sticky header：User: 如何处理大文件？        ]  ← 行 0
[                                               ]  ← 行 1（内容区起始）
[ ❯ 如何处理大文件？                           ]  ← 行 2（prompt 的实际位置）
```

如果用户点击 header 想"跳回这个 prompt"，目标是让 prompt 的 `❯` 出现在屏幕顶部（行 0），但 header 还在的时候，内容区顶部是行 1，prompt 最多只能到行 1。

## 'clicked' 状态的机制

`StickyPrompt` 类型有三种形态：

```typescript
type StickyPrompt = {
  text: string
  scrollTo: () => void
} | 'clicked' | null
```

点击 header 时的动作：

```typescript
// StickyPromptHeader 的 onClick
const handleClick = () => {
  setSticky('clicked')  // ← 先设为 clicked
  sticky.scrollTo()     // ← 再执行 scrollTo
}
```

`FullscreenLayout` 检测到 `sticky === 'clicked'` 时，把 `paddingTop` 设为 `0`（不是 `1`），header 内容隐藏，但布局空间也收缩。这样 scrollTo 之后内容区从行 0 开始，prompt 的 `❯` 正好落在行 0。

## 为什么不直接 scrollTo 然后清除 header

反过来做（先 scrollTo 再隐藏 header）会产生一帧的错位：

1. scrollTo 执行，内容区顶部是行 1（header 还在）
2. 渲染提交，用户看到：prompt 在行 1（而不是行 0）
3. 隐藏 header，内容区变成行 0，prompt 上移到行 0

用户会看到 prompt 从行 1 跳到行 0 的一帧闪烁。

先设 clicked 再 scrollTo：

1. 设 clicked，下次渲染 header 隐藏，padding 收缩
2. scrollTo 在同一个 React batch 里执行（或在同一帧里）
3. 渲染提交，内容区已经是行 0，prompt 直接落在行 0

没有中间状态被渲染出来。

## clicked 状态的自动清除

`'clicked'` 不是永久状态。下次 StickyTracker 运行（用户重新滚动）时，会重新计算 sticky prompt：

```typescript
// StickyTracker 在每次滚动时
const newSticky = findStickyPrompt(messages, scrollTop)
if (newSticky !== null || prevSticky === 'clicked') {
  setSticky(newSticky)  // 覆盖 'clicked' 状态
}
```

只要用户稍微滚动，`'clicked'` 就被真实的 sticky prompt 值（或 null）替换。这个"用完就清"的机制让 `'clicked'` 状态不需要额外的清理逻辑——它的生命周期由用户的下一次滚动操作结束。

## 面试指导

这是一道"时序和像素对齐"的细节题，考察你是否关注 UI 的精确性。

**核心设计原则**：动画或跳转的目标终态应该在用户操作发生时立即确定，不应该需要后续的修正。先把终态设好（header 隐藏 → padding 为 0），再执行跳转（scrollTo），跳转的落点就是终态的正确位置，不会有中间状态的闪烁。

**React batching 的作用**：React 18 的 automatic batching 让同一个事件处理器里的多个 setState 合并成一次渲染。`setSticky('clicked')` 和 `scrollTo()` 在同一个 onClick 里，如果 scrollTo 是同步的（没有 await），两个操作会在同一次 commit 里完成，避免中间帧。

**加分追问**：如果用户快速点击 sticky header 两次，第二次点击时 sticky 已经是 `'clicked'`，`sticky.scrollTo` 不存在，会报错吗？答：需要在点击处理里检查 `typeof sticky === 'object'`，如果是 `'clicked'` 或 `null`，不执行 scrollTo。实际实现里通常有这个守卫。

**延伸设计**：这种"先设终态再执行动作"的模式在动画系统里很常见，叫做"commit-then-animate"。先把 CSS class 切换到目标状态，让浏览器知道终态，然后触发 transition，浏览器会自动补间从当前状态到目标状态的动画。Claude Code 在终端里用类似的思路处理布局状态和滚动。
