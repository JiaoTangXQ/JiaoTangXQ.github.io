# 多 agent 团队里的制度层：leader 收口和权限秩序怎么在代码里体现？

多 agent 系统最常见的问题不是「跑不起来」，而是「跑起来但失控」。Claude Code 的 swarm 架构把这个问题提前解决，方法是在团队里设置显式的制度层，而不是让 agent 在共享状态里自由竞争。

## 身份先于执行

`computeInitialTeamContext()` 在 `main.tsx` 里是**同步调用**，在第一次 React 渲染之前执行：

```typescript
// main.tsx 里的时序
const initialTeamContext = computeInitialTeamContext()  // 同步
const initialState = {
  ...coreInitialState,
  teamContext: initialTeamContext,  // 注入初始状态
}
// 然后才有 ReactDOM.render(...)
```

这个时序保证了：agent 在开始工作的第一帧就知道自己是 leader 还是 teammate。如果是 teammate，它知道 leader 的 agent ID，知道自己应该把权限请求发给谁。

如果改成「先渲染，异步获取身份」，会发生什么？第一次渲染期间 agent 可能已经开始处理消息，这时它还不知道自己该不该直接执行某些操作，还是应该先问 leader。

## leader 的角色定义

leader 不执行额外的业务逻辑，它的核心职责只有两件事：

1. **权限收口**：teammate 的工具权限请求汇到 leader，由 leader 的 UI 展示给用户决策
2. **视野整合**：用户在 leader 的界面里看到所有 teammate 的状态（通过 pane 布局）

`leaderPermissionBridge.ts` 的设计体现了这个定位：

```typescript
// leader 启动时注册两个 setter
registerLeaderToolUseConfirmQueue(setToolUseConfirmQueue)
registerLeaderSetToolPermissionContext(setToolPermissionContext)

// in-process teammate 需要权限时调用
const setter = getLeaderToolUseConfirmQueue()
if (setter) {
  setter(prev => [...prev, newConfirmRequest])
}
```

注意这里的反转：不是 teammate 主动「发消息给 leader」，而是 teammate 调用 leader 注册的函数，直接把确认请求推入 leader 的 React 状态。这比消息传递更直接，也只适用于同进程的 teammate；跨进程 teammate 走的是 mailbox 路径。

## 权限同步的防竞争设计

`permissionSync.ts` 里的 `createPermissionRequestMessage` 和 `createPermissionResponseMessage` 对应权限请求和响应的消息格式。这两个函数的关键在于它们包含的 `id` 字段：每个权限请求都有唯一 ID，响应必须带上对应的 ID。

这解决了一个经典的竞争问题：如果两个 teammate 同时等待权限，leader 如何把响应路由回正确的请求方？答案是通过请求 ID 匹配——leader 收到用户的决策后，在响应消息里带上原始请求 ID，写入原始请求者的 mailbox。

## 团队制度的另一面：避免权力过度集中

一个容易走偏的设计方向是让 leader 有更多权力：比如 leader 可以代替 teammate 决定「这个操作安全，直接批准」。Claude Code 没走这条路，原因是：

- 用户委托给 Claude Code 的信任是直接的，不是通过 leader agent 中转的
- 如果 leader 能自动批准 teammate 的请求，等于绕过了用户监督
- 一旦 leader 被攻击（比如 prompt injection），它的「代理批准」能力会成为系统漏洞

所以 leader 的作用是**把权限请求传递给用户**，而不是**代表用户做决策**。

## 团队颜色和 pane：视觉层的制度

`cli/print.ts` 里为每个 teammate 分配的颜色，和 `teammateLayoutManager.ts` 管理的 pane 布局，看起来是纯 UI 问题，实际上是制度层的视觉化。

用户在同时看多个 agent 工作时，颜色是「这条输出属于哪个 agent」的唯一快速识别信号。如果所有 agent 的输出混在一起没有颜色区分，用户根本无法监督谁在做什么——监督能力消失，制度等于失效。

## 面试答题思路

「设计多 agent 系统时如何避免失控」：

1. **身份先于执行**：agent 必须在开始工作前就知道自己在团队里的角色
2. **权限收口点必须单一**：所有需要人类决策的操作汇到同一个地方展示，不能分散
3. **leader 是传递者不是决策者**：leader 的职责是把请求传给人，而不是代替人决策
4. **视觉层是制度的一部分**：如果用户看不清谁在做什么，监督就等于没有

追问「如果 leader 崩溃了怎么办」：teammate 的权限请求会进入 mailbox 等待，teammate 会暂停执行等待响应超时。系统设计上 leader 崩溃不会导致 teammate 继续执行未授权的操作，而是等待或超时退出。
