# 深度分析：折叠不是隐藏信息，而是把信息压成人能处理的形状

## 信息量 vs 信息可处理性

Claude Code 执行一个"重构这个模块"的任务，可能会产生：

- 50+ 次文件读取（Read 工具）
- 20+ 次文件编辑（Edit 工具）
- 10+ 次 bash 命令（Bash 工具）
- 若干次搜索（Grep、Glob 工具）

如果每个工具调用都展开显示完整内容，用户屏幕上会有几百行输出。要找到"AI 在哪里做了重要决策"这个信息，需要在几百行里搜索。

信息是完整的，但信息是**不可处理的**——要找到有意义的内容，认知成本太高。

## 折叠做了什么

折叠不是删除信息，而是把信息组织成层次结构：

**层次一（默认显示）**：工具调用的摘要信息
- "读取了 src/components/ 目录下的 12 个文件"
- "修改了 3 个文件（src/utils/helper.ts, src/components/Button.tsx, ...）"
- "执行了 npm test，测试通过"

**层次二（展开后显示）**：具体的工具调用列表
- 每次 Read 的文件路径
- 每次 Edit 的 diff 摘要

**层次三（再次展开）**：完整的工具调用内容
- 完整的文件内容
- 完整的 diff

用户在大部分时候只需要看**层次一**——了解 AI 做了什么，是否符合预期。只有当用户想深入某个操作时，才需要展开到层次二或三。

这把"几百行信息"变成了"几行摘要 + 按需展开的详情"。

## 分组的语义

分组不只是把相关的折叠在一起，而是**把同一"意图"下的操作归组**。

例如：
- "读取相关文件"组：AI 在理解问题时的读取操作
- "实施修改"组：AI 在执行修改时的写入操作
- "验证结果"组：AI 在确认修改正确时的测试命令

这个分组反映了 AI 解决问题的思路：先理解，再修改，再验证。用户看到这个结构，能快速判断 AI 的工作方式是否合理，不需要逐条分析每个工具调用。

## 为什么是"替人压缩"而不是"为人隐藏"

"替人压缩"和"为人隐藏"的区别：

**隐藏**：信息永久不可见，用户如果想知道就无法获取。

**压缩**：信息仍然完整，但默认以摘要形式展示，用户需要时可以展开。

Claude Code 的折叠是压缩：所有信息都在，只是默认压缩了。用户可以展开任何分组查看详情，可以展开任何工具调用查看完整输出。

这是"不剥夺用户的知情权，只减少不必要的认知负担"。

## 长会话里的累积效应

在短会话里（10 轮对话，总共 30 个工具调用），折叠的价值不那么明显——就算全部展开，也还可读。

在长会话里（50 轮对话，总共 300 个工具调用），折叠变得必不可少：

没有折叠：用户要在 300 个工具调用里找到"第 40 轮时 AI 做了什么修改"，几乎是不可能的。

有折叠：每轮对话的操作被分组折叠，用户可以快速定位到第 40 轮，展开查看那一轮的操作，找到目标。

折叠的价值随会话长度增长而增长——这也是为什么它是"长会话才不至于失控"的关键功能。

## 和 sticky prompt 的配合

sticky prompt（174 文章的主题）在输入框附近钉住当前的上下文信息；分组折叠在历史记录区域管理信息密度。

两者从不同方向解决同一个问题：长会话里的"在哪里、做了什么、现在要怎么处理"的认知定向问题。

sticky prompt 解决"现在的上下文是什么"，分组折叠解决"历史上发生了什么"。用户在长会话里能保持方向感，这两个机制缺一不可。

## 面试要点

1. **折叠是压缩而不是隐藏**：信息仍然完整，只是默认以摘要形式展示，用户按需展开，不剥夺知情权。
2. **分组反映操作的语义层次**：按"意图"（理解/修改/验证）分组，而不只是按工具类型分组，让用户能理解 AI 的工作逻辑。
3. **折叠的价值随会话长度增长**：短会话里不那么关键，长会话里是防止失控的必要机制。
4. **和 sticky prompt 互补**：折叠管理历史，sticky prompt 管理当前上下文，两者配合让长会话维持认知可控性。
