# RemoteSessionManager 为什么要在本地重建一套权限判断而不是信任远端？

一个容易忽视的设计问题：Claude Code 连接到远端 CCR（Claude Code Runner）容器时，工具实际上在远端执行，那为什么本地还要维护一套权限判断？直接信任远端不是更简单？

## 远端模式的现实

`RemoteSessionManager` 管理着本地客户端与远端会话之间的全部通信。它通过 `SessionsWebSocket` 收发消息，把远端产生的 `SDKMessage` 注入本地的消息流——用户看到的对话界面和本地运行时在消息格式上是一致的。

但这里有一个关键不对称：工具在远端运行，权限确认在本地发生。

```typescript
// RemoteSessionManager 的核心回调结构
export type RemoteSessionCallbacks = {
  onMessage: (message: SDKMessage) => void
  onPermissionRequest: (
    request: SDKControlPermissionRequest,
    requestId: string,
  ) => void
  onPermissionCancelled?: (requestId: string, toolUseId: string | undefined) => void
  onConnected?: () => void
  onDisconnected?: () => void
}
```

当远端容器执行工具遇到需要授权的动作时，它通过 `SDKControlPermissionRequest` 消息把请求发回来，本地的 `onPermissionRequest` 回调处理它，弹出确认对话框，用户点击后再把结果发回远端。

## remotePermissionBridge 的角色

`remotePermissionBridge.ts` 解决的是一个具体的技术难题：本地的权限确认 UI（`ToolUseConfirm`）要求有一个 `AssistantMessage` 对象，但远端场景下工具调用发生在 CCR，本地根本没有这条原始消息。

```typescript
// 合成一条本地不存在的 AssistantMessage
export function createSyntheticAssistantMessage(
  request: SDKControlPermissionRequest,
  requestId: string,
): AssistantMessage {
  return {
    type: 'assistant',
    uuid: randomUUID(),
    message: {
      id: `remote-${requestId}`,
      content: [{
        type: 'tool_use',
        id: request.tool_use_id,
        name: request.tool_name,
        input: request.input,
      }],
      ...
    }
  }
}
```

这个合成消息让远端的权限请求能复用本地已有的确认 UI，而不是另写一套远端专用的弹窗逻辑。

同样的问题出现在工具定义上：CCR 可能加载了本地没有的 MCP 工具。`createToolStub` 为这些未知工具创建最小化的 stub，让它们也能通过本地的 `FallbackPermissionRequest` 路径处理，而不是直接崩溃。

## viewerOnly 模式的边界

`RemoteSessionConfig.viewerOnly` 是一个有趣的字段：

```typescript
/**
 * When true, this client is a pure viewer.
 * Ctrl+C/Escape do NOT send interrupt to the remote agent;
 * 60s reconnect timeout is disabled;
 * session title is never updated.
 * Used by `claude assistant`.
 */
viewerOnly?: boolean
```

普通远端会话里，本地客户端可以发中断信号（Ctrl+C）打断远端执行。`viewerOnly` 把这个能力拿掉——客户端只能看，不能干预。这对「观察另一个 Claude 在做什么」的场景（比如 `claude assistant` 命令）是正确的行为，但代价是用户在 viewerOnly 状态下发现任务跑偏了，无法直接停止它。

## 为什么不信任远端

直觉上，远端容器是受控环境，应该可以信任它的权限判断。但 Claude Code 的设计前提是：**用户对工具的授权决定必须发生在用户自己的机器上**。

理由有两个：

1. **责任归属**：如果权限在远端判断，用户事后无法审计「这个动作是谁授权的、什么时候授权的」。本地确认产生的记录留在本地日志里。

2. **攻击面**：一个被攻击的远端容器可以伪造「用户已授权」的信号。让授权流程必须经过本地客户端，等于给这条链路加了一层不受远端控制的验证节点。

## 断线恢复的设计

`RemoteSessionManager` 还处理断线重连。它维护一个重试逻辑，在连接丢失后按指数退避尝试重连，直到 `onDisconnected` 被触发（viewerOnly 模式不启用 60 秒超时）。

这个设计说明远端会话被当成「长期连接」而非「一次性请求」来处理——一旦连接，系统期望它能持续存在，中间的网络抖动不应该让整个会话崩溃。

## 面试答题思路

被问「远端 AI agent 的权限系统怎么设计」时，核心点是：

1. **确认必须在用户侧发生**：即使执行在远端，授权决定不能在远端做
2. **消息格式要有本地和远端的翻译层**：远端格式和本地格式不会完全一致，翻译层是必要的
3. **存根（stub）是处理未知能力的务实选择**：不知道的工具不能直接拒绝，要有降级路径
4. **viewerOnly 是一个有价值的最小权限原则应用**：只读权限不等于无用，但要明确边界
