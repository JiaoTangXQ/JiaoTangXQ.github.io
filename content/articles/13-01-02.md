Claude Code 的“优雅”不是发生在 demo 最顺的那条路径上，而是发生在它承认世界会出错、人会犹豫、系统会超长之后还尽量保持秩序。`permissionSetup.ts` 里不是简单做一个“允许/拒绝”开关，而是专门把危险的 Bash、PowerShell 和 Agent 自动放行规则挑出来，不让 auto mode 在最危险的地方偷懒。`query.ts` 里不是默认所有流式调用都会完整收口，而是提前准备好中断、缺失 `tool_result`、重试和恢复的收尾逻辑。`compact.ts` 也不是一刀把旧消息删掉，而是先去掉图片这种对总结无益却很占 token 的内容，再把技能、附件、计划等关键线索重新接回新边界。`claudeInChrome/setup.ts` 甚至默认在非交互场景禁用浏览器能力，只有显式开关或受控场景才打开。

更常见的做法，是把这些现实问题都留给用户自己兜底：权限太危险就靠用户小心一点，流式中断了就重试一下，context 太长了就手动清空，浏览器桥接连不上就让用户自己猜发生了什么。这样写起来轻，但系统一旦进入真实使用环境就会不断把脏活甩给人。

Claude Code 没有假装风险不存在，而是把一部分风险成本前置到代码结构里。这让它更厚、更难读，也让很多文件显得不够“纯”。但如果所谓优雅只能在一切顺利时成立，那就不是产品级优雅，只是演示级顺滑。

## 实现链
权限预设、stop hooks、compact、Chrome setup 都把风险、失败和用户犹豫当成代码里的正式场景。

## 普通做法
把风险都留给用户自己兜底。

## 为什么不用
因为真实使用环境里最先出问题的恰恰是这些不浪漫的地方。

## 代价
系统会更重，也更难读。
