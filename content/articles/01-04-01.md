# 为什么主流程只能解释它怎样动起来，不能解释它为何不失控

流程图很迷人，因为它总能给人一种“我已经看懂了”的感觉。启动，然后进主循环，然后调工具，然后继续下一轮。只看这条线，Claude Code 确实像一台会自己往前跑的机器。但真正难的部分恰恰不在“它怎样动”，而在“它为什么没有动坏”。

主流程解释的是推进顺序，却解释不了边界是谁定的。为什么有些工具能直接做，有些必须先问？为什么自动模式不能因为一条宽松规则就获得无限执行能力？为什么同一个会话到了不同环境里，权限、目录、工作区和可见工具都会变化？这些问题都不属于流程，它们属于制度。

Claude Code 对这一点很清醒。`toolPermissionContext` 被长期放在状态里，`permissionSetup` 会从不同来源加载规则、合成上下文，还会主动剥掉自动模式下危险的放行规则；settings 决定哪些制度来自哪里；hooks 允许外部逻辑在关键时刻插手；连续性机制又决定哪些记忆该留下、哪些上下文该压缩。也就是说，真正约束这套 agent 的，不只是“下一步该干什么”，而是“什么能干、什么不能干、什么即使能干也要换一种方式干”。

这正是成熟系统和演示系统的分水岭。演示系统只要能跑起来就行，成熟系统必须先回答自己怎样在现实世界里活下去。Claude Code 最值得尊敬的一点，是它没有把治理写成一堆补丁，而是承认制度和主流程一样，都是主结构。没有这层承认，所谓优雅只会停留在表面。

## 实现链
这页落到的实现链都属于制度世界：权限装配、设置来源、hooks 介入、连续性设施或条件能力 gating。代码的共同特点是，它们不是在主循环失败以后补救，而是在主循环真正推进前后就提前改写资格和边界。

## 普通做法
更普通的做法是先把 happy path 主流程跑通，再把权限、外部规则和跨回合记忆当辅助功能补上。很多系统都是这样长出来的。

## 为什么不用那种做法
Claude Code 没这么长，是因为对 agent 来说，真正危险的往往不是下一步怎么走，而是这一步有没有资格走、走完后有没有留下可继续工作的状态、外部制度有没有权利插手。它把这些问题提前写进了运行时。

## 代价与回报
回报是系统更不容易越权和失忆。代价是相关逻辑分散，阅读时不能只盯着一条主流程。
