# 第十三卷导读：总结不是回顾，而是把判断重新钉住

## 这卷真正要回答什么

第十三卷不是把前面十二卷换个顺序再说一遍，而是把一个更硬的问题说清楚：Claude Code 这套设计，到底是代码里真的成立，还是只是文案上看着顺。

要判断这件事，不能只看“它功能很多”。得回到源码里看几件具体的事：`main.tsx` 为什么先决定模式、权限和入口，再把会话交出去；`QueryEngine` 为什么把一整个会话的消息、拒绝记录、工具和文件状态都揽在自己手里；`Task.ts` 为什么把工作正式做成任务对象；`AppStateStore` 为什么宁可显得很重，也要把远端、bridge、MCP、任务、通知和界面选择都放进同一份事实世界。

更常见、也更容易想到的做法，是把这些东西揉进一个“大而全的 agent 循环”里：输入来了就调模型，模型想调用工具就直接调，UI 只是结果展示层。Claude Code 没这么做。它选择的是更慢理解、但更抗增长的路子: 入口、行动、制度、显示分层，各层之间靠明确的交棒和翻译衔接。第十三卷就是专门评这件事值不值。

## 为什么这卷必须这样收口

如果只顺着正文一路读下去，读者很容易产生两种误判。第一种误判是把 Claude Code 神化成“设计总是对的”；第二种误判是把它看成“只是代码多、功能杂，但恰好没散”。这两种看法都不对。

从代码看，它很多地方确实比常见实现更讲秩序，比如用 `Task` 和 `AppStateStore` 把任务与显示事实固定住，用 `permissionSetup` 把危险动作拦在制度层，用 `remotePermissionBridge` 和 MCP 包装层让新世界先学旧世界的话。但从代码看，它也真的付出了代价：状态厚、接缝多、feature gate 多、兼容路径长、理解门槛高。

所以这一卷的职责不是礼赞，而是把判断钉死：哪些地方是真的合理，哪些地方只是现实工程下的体面处理，哪些经验值得迁移，哪些只是 Anthropic 这个产品阶段不得不背的重量。

## 实现链
第十三卷会回看 `main.tsx`、`QueryEngine`、`Task`、`AppStateStore`、权限与外延能力接入链，检查这些判断是否真的写进了代码结构。
