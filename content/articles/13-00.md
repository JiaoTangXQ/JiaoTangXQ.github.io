# 面试官问：读完 Claude Code 全书，你能说清楚它到底是什么吗？

## 这道题要考什么

面试官问完所有细节之后，最后往往会来一道收口题：你能用一句话说清楚这个系统到底是什么吗？

读完前十二卷，你知道 `main.tsx` 怎么分流、`QueryEngine` 怎么管会话、`Task.ts` 怎么正式化劳动、`AppStateStore` 怎么收拢事实、权限层怎么卡位。但"说清楚是什么"这件事比"说清楚怎么做"要难得多。它要求你站出来判断：这套设计，到底是代码里真的成立，还是只是文案上看着顺？

这正是第十三卷存在的理由。

## 两种常见误判

读完系统性架构资料，读者最容易掉进两个坑。

第一个坑是神化。觉得"它设计得真好，每个决定都是对的"。这个看法太轻，因为它把工程美学和实现现实混在一起了。Claude Code 确实比很多 agent 产品更讲秩序，用 `Task` 和 `AppStateStore` 把任务与显示事实固定住，用 `permissionSetup` 把危险动作拦在制度层前面，用 `remotePermissionBridge` 和 MCP 包装层让新世界先学旧世界的话。但这些决定都是有代价的：状态厚、接缝多、feature gate 多、兼容路径长、理解门槛高。它不是一台没有摩擦的机器。

第二个坑是降级。觉得"功能多，但恰好没散，运气好而已"。这个看法也偏了。从代码看，很多地方的秩序是刻意选择的结果，不是堆砌碰出来的。入口、行动、制度、显示分层；各层之间靠明确的交棒和翻译衔接；新世界进来时先通过已有的制度链——这些不是运气，是反复抵抗"顺手塞进去"冲动的产物。

## 这卷要做的判断

第十三卷的目标，是把这个判断钉死：哪些地方是真的合理，哪些地方只是现实工程下的体面处理，哪些经验值得迁移，哪些只是 Anthropic 这个产品阶段不得不背的重量。

判断必须回到代码。`main.tsx` 为什么先决定模式、权限和入口，再把会话交出去；`QueryEngine` 为什么把一整个会话的消息、拒绝记录、工具和文件状态都揽在自己手里；`Task.ts` 为什么把工作正式做成任务对象；`AppStateStore` 为什么宁可显得很重，也要把远端、bridge、MCP、任务、通知和界面选择都放进同一份事实世界。

这些选择值不值，得从代码里看，不能只从架构图看。

## 怎么读这一卷

- 想先看"它为什么算优雅"：从 13-01 进，顺着到 13-04，看分层、交棒、接缝零件和制度先行怎么在代码里落地。
- 想先看"它重在哪里、丑在哪里"：从 13-05 进，顺着到 13-07，看复杂度、历史兼容和普通读者真正值得学走的设计习惯。

## 面试指导

这道收口题的标准答案不是一句话总结，而是能说清楚"这套系统选择了什么权衡，代价是什么，值不值得"。

说不清楚权衡的答案，哪怕细节再熟，也只是背了答案，没有真正理解这个系统。
