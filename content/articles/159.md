# 深度分析：swarm 比单 agent 多了什么，多出来的这些为什么不可少

## 单 agent 不需要的东西

单 agent 模式下，Claude Code 有：
- 一个会话（conversation）
- 一套工具（tools）
- 一个权限上下文（ToolPermissionContext）
- 一个输出流（渲染到终端）

这一套东西在单 agent 下工作得很好。

当你把这套系统复制三份（起三个 agent 同时工作），会发现少了三样东西：

**缺少一：身份**

三个 agent 各自有自己的会话，但谁是谁？当用户看到三个输出窗口，他需要知道哪个是处理前端任务的 agent，哪个是处理后端的，哪个是负责协调的。

单 agent 不需要身份，因为只有一个。多 agent 必须有身份，否则人无法区分谁在做什么。

**缺少二：秩序**

三个 agent 各自有权限请求时，同时向用户弹出三个确认对话框。用户不知道怎么处理，很可能随机点一个。

单 agent 有一个权限上下文，权限决策很自然。多 agent 的权限决策需要一个秩序：谁负责向用户呈现权限请求，谁负责把用户的决定传递回各个 agent。

**缺少三：可恢复通信面**

三个 agent 通过进程内调用互相传递信息。某个 agent 崩溃了，和它通信的信息也丢了，其他 agent 不知道崩溃前的状态。

单 agent 的状态全在自己进程里，崩溃就是全部重新来过。多 agent 需要通信面在外部持久化，让崩溃的 agent 重启后能恢复协作上下文。

## 身份系统的具体形态

在 `src/utils/swarm/reconnection.ts` 里，团队 context 包含：

```typescript
return {
  teamName,           // 这个 agent 在哪个团队
  teamFilePath,       // 团队配置在哪里
  leadAgentId: teamFile.leadAgentId,  // 谁是 leader
  // ...
}
```

身份不只是"我叫什么"，还包括"我在什么组织里"和"组织里谁更高一级"。

`agentName` 是 agent 对人类可见的名字（可能是"前端专家"、"测试工程师"这样的角色名）。

`agentId` 是系统内部的唯一标识（用于消息路由、状态存储、日志追踪）。

两者分开，是因为人类可读的名字和系统内部路由需要不同的特性：人类可读的名字可以重复（两个团队可以各有一个"前端专家"），系统内部 ID 必须全局唯一。

## 秩序层的工程实现

秩序层的核心是 `permissionSync.ts` 里的权限同步协议：worker 的权限请求不直接展示给用户，而是路由到 leader，由 leader 的 UI 统一展示。

这个设计的关键判断是：**用户看到的权限请求数量应该和团队数量无关，而应该和"需要用户参与决策的数量"有关**。

大部分时候，worker 的权限请求可以被 leader 自动批准（如果这类操作已经被授权过）。只有真正需要用户判断的请求才展示给用户。

这是秩序层的价值：不是减少请求数量（该有多少就有多少），而是**筛选哪些请求真的需要用户参与**，让用户不被可以自动处理的请求淹没。

## 可恢复通信面的基础设施

可恢复通信面由三个组件合作实现：

`Mailbox`（内存部分）：在进程内，消息通过内存队列快速传递。

`teammateMailbox`（文件部分）：跨进程，消息写入文件系统，持久化等待被读取。

`swarm/reconnection.ts`（恢复部分）：重连时读取文件邮箱里的未处理消息，重建 agent 在团队里的协作上下文。

这三层配合，让通信面既高效（进程内走内存）又可靠（跨进程和崩溃恢复走文件）。

## 为什么这三样东西不可少

**缺身份**：用户无法区分多个 agent，swarm 对用户来说是一个黑盒，用户失去了理解和干预协作过程的能力。

**缺秩序**：权限请求混乱，用户体验崩溃。更深层的问题是，没有秩序意味着没有权威——不知道谁的决定是有效的，系统可能进入冲突状态。

**缺可恢复通信面**：swarm 变得脆弱。任何 agent 的崩溃或重启都会让整个团队的协作失去连续性，实际上无法用于需要长时间运行的任务。

这三样东西是 swarm 从"看起来可以用"到"真的可以用"的差距。

## 面试要点

1. **身份系统是人机协作的基础**：agentId 用于系统路由，agentName 用于人类识别，两者分开是因为需求不同。
2. **秩序层筛选而不是减少请求**：把可以自动处理的请求和需要用户参与的请求分开，是秩序层的核心价值。
3. **可恢复通信面分三层**：内存队列（高效）+ 文件持久化（可靠）+ 重连恢复（可续），三层配合实现完整的可靠通信。
4. **三样东西缺任何一个 swarm 就会降级**：这不是锦上添花的特性，而是 swarm 能够在真实环境里长期运行的必要条件。
