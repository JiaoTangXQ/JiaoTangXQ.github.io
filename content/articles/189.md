# 自动后台化为什么不是特殊逻辑：统一转场协议的价值

面试题版本：**"Claude Code 里有哪些情况会触发任务的后台化？不同触发源的后台化走的是同一套代码路径吗？为什么要统一？"**

## 触发后台化的多种场景

任务从前台切换到后台，在 Claude Code 里可以由多种来源触发：
1. 用户主动点击"后台运行"
2. 任务等待时间过长，系统自动建议后台化
3. 用户发起新任务，系统把旧任务自动推到后台
4. 批处理场景下多个任务同时运行，多数跑在后台

这些触发源的上下文完全不同：有的有用户交互，有的没有；有的知道"为什么"后台化，有的只是被系统调度的结果。

## 朴素方案的问题

如果为每种触发源写单独的处理逻辑：

```typescript
// 用户手动后台化
function handleUserBackgroundRequest(taskId: string) {
  notifyUser('任务已移到后台')
  setTaskBackground(taskId)
  updateUI()
}

// 超时自动后台化
function handleTimeoutBackground(taskId: string) {
  // 要不要通知用户？
  // 用不用同样的 setTaskBackground？
  // UI 更新逻辑是否一样？
  backgroundTaskSilently(taskId)
  maybeNotify(taskId)
}

// 批量调度后台化
function handleSchedulerBackground(taskIds: string[]) {
  // 又一套逻辑...
}
```

这三套逻辑随时间推移会产生漂移：一个 bug 只在某些触发路径下修了，另几条路径没修；一个新特性（比如"后台化后自动发通知"）要同时改三处。

## 统一转场协议的实现思路

Claude Code 的做法是把"后台化"本身抽象为一个动作，无论触发源是什么，最终都要走同一个转场函数：

```typescript
// 概念上的统一入口（不是源码直译，是设计思路的体现）
function transitionToBackground(
  taskId: string,
  reason: BackgroundReason,
  context: TransitionContext
): void {
  // 1. 发送 backgroundSignal
  sendBackgroundSignal(taskId)
  // 2. 更新 AppState（显示姿态变化）
  setAppState(s => markTaskBackground(s, taskId))
  // 3. 根据 reason 决定是否通知用户
  if (shouldNotify(reason, context)) {
    scheduleBackgroundNotification(taskId)
  }
}
```

`reason` 字段记录为什么后台化，但不影响核心制度动作（signal + state update）的执行。通知逻辑根据 reason 判断，但核心路径是统一的。

## 统一的好处：新的触发源不需要写新逻辑

当一个新的触发场景出现（比如，远端 webhook 触发某任务后台化），只需要：
1. 在 `BackgroundReason` 里加一个新的枚举值
2. 在 `shouldNotify()` 里决定这个新 reason 是否通知用户
3. 调用同一个 `transitionToBackground()`

不需要重新实现 signal 发送、状态更新、UI 刷新的逻辑。这种"只扩展 reason，不扩展核心路径"的模式让系统保持了线性复杂度增长，而不是每新增一种触发源就新增一套代码。

## 制度动作和触发上下文的分离

"入口可以不同，制度动作只有一套"这个原则的工程解释是：

**触发上下文**（谁触发的、为什么、在什么情况下）是多样的，应该保留多样性。

**制度动作**（一旦决定后台化，系统做什么）应该是确定的、唯一的。

如果把这两层混在一起，就会变成"每种情况都有自己的特殊处理"，系统行为变得难以预测——用户想知道"任务后台化后系统会做什么"，答案变成了"看是哪种后台化"。

统一后，行为是可预期的：任务一旦进入后台化流程，无论通过哪个入口，制度动作是相同的。

## 面试角度的总结

自动后台化走统一转场协议，核心是把**触发来源的多样性**和**制度动作的确定性**分开。前者允许多样，后者要求统一。这个分离保证了系统在增加新触发场景时不需要重复实现核心逻辑，也让后台化行为对用户来说是可预期的，而不是"看触发来源决定"的黑盒。
