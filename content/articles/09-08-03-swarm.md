# Swarm 为什么同时管理执行后端和空间布局？这不是职责不清，是团队运行时的完整性

打开 `utils/swarm/` 目录，你会发现它同时包含：

- `backends/registry.ts`：决定用哪种后端（in-process / tmux / iTerm2）
- `teammateLayoutManager.ts`：颜色分配和 pane 布局管理
- `inProcessRunner.ts`：同进程执行循环
- `permissionSync.ts`：跨进程权限协调
- `leaderPermissionBridge.ts`：执行层到 UI 层的桥接

从"关注点分离"的角度看，这里混放了：后端选择（基础设施）、空间布局（视觉/UX）、执行逻辑（核心功能）、权限协调（安全/控制）。

这是设计混乱吗？

## 为什么执行方式和空间布局不可分离

在单 agent 场景里，"在哪里运行"和"在哪里显示"确实是可以独立的：agent 在进程内运行，在哪个终端窗口显示是 UI 层的事，两者没有强耦合。

但在多 agent 团队场景里，执行方式直接决定了空间组织的可能性：

**如果选了 tmux 执行方式**：每个 teammate 在独立的 tmux pane 里，空间布局意味着：pane 怎么分割、leader 在哪里、teammate 的 pane 边框颜色是什么。执行和空间是同一件事的两个方面——你不能说"用 tmux 跑，但在 iTerm2 里显示"。

**如果选了 in-process 执行方式**：没有物理上的 pane 分割，空间组织通过 UI overlay（React 组件里的任务面板）展示。执行方式决定了空间展示的整体形态。

**如果 auto 模式决策了用哪种后端**：这个决策会立刻传播到：用什么方式创建 pane（或者不创建）、颜色分配是否需要映射到 pane 边框颜色、用户在哪里看到 teammate 的输出。

**执行方式和空间组织是同一个决策的两个维度，不是两个独立决策**。把它们放在同一个模块里，是承认了这个现实，而不是混淆职责。

## 团队运行时（Team Runtime）作为完整概念

如果给这个目录的职责下一个定义：`utils/swarm/` 管理的是**团队运行时**。

"团队运行时"是一个完整概念，包含：

1. **在哪里跑**（后端选择：registry.ts）
2. **以什么身份跑**（TeammateContext：teammateContext.ts）
3. **在哪个空间里呈现**（空间布局：TmuxBackend.ts、teammateLayoutManager.ts）
4. **怎么通信**（mailbox：teammateMailbox.ts）
5. **怎么协调权限**（permissionSync.ts、leaderPermissionBridge.ts）
6. **怎么退场**（shutdown request/response 协议）

这 6 件事缺少任何一件，团队就不完整。用一个比喻：**这是 AI Agent 团队的操作系统**——它不只管"代码在哪里跑"，还管"团队成员在哪里存在、怎么沟通、怎么获得权限、怎么有序退出"。

## 对比：如果按传统职责分离来组织

如果强行按"关注点分离"把这些代码分到不同目录：

```
core/
  runAgent.ts
  
execution/
  inProcessRunner.ts
  
infrastructure/
  backends/
    tmux.ts
    iterm2.ts
    in-process.ts
    registry.ts
    
communication/
  mailbox.ts
  permissionSync.ts
  
security/
  leaderPermissionBridge.ts
  
ui/
  layoutManager.ts
  colorAssignment.ts
```

这在目录结构上更"干净"。但问题是：当你需要理解"一个 teammate 是如何被创建的，它在哪里运行，它在哪里显示"，你需要在 6 个不同的目录里跳来跳去。

`utils/swarm/` 把这 6 件事放在一起，是按**概念完整性**而不是按**技术类型**组织的。当你需要理解团队运行时，你只需要看这一个目录。

这是两种不同的模块化方式：
- **按技术类型分**：所有后端逻辑在一起，所有通信逻辑在一起——对单独维护某个技术栈友好
- **按概念域分**：所有团队运行时逻辑在一起——对理解完整功能友好

Claude Code 选择了按概念域分，因为"团队运行时"是一个内聚的概念，各部分之间有大量的调用关系（registry 调用 layoutManager，inProcessRunner 调用 mailbox，permissionSync 调用 mailbox……）。把这些内聚代码拆散到不同目录，会增加导航成本而不是降低理解成本。

## 面试怎么答

如果面试题是"如何评估一个模块的边界是否合理"：

**评估维度**：

1. **内聚性**：模块内部的代码相互调用频繁吗？如果内部代码关系紧密，合在一起是合理的。
2. **独立修改频率**：模块里的代码，修改 A 是否通常需要同时修改 B？如果是，它们应该在一起。
3. **理解完整性**：理解这个模块的核心功能，需要跨越到其他模块去看多少细节？跨越越少，边界越合理。

**Claude Code 的判断**：`utils/swarm/` 内部各个文件相互调用频繁（高内聚）；修改执行后端通常需要同步修改布局管理（高修改相关性）；理解"teammate 是如何运行的"只需要看这一个目录（高理解完整性）。三个维度都支持把这些代码放在一起。

**反直觉的结论**：有时候"职责分离"反而会降低代码质量——当相关的代码被强行分到不同目录时，它们之间的耦合没有消失，只是变成了隐性的跨目录依赖，更难发现，更难维护。好的模块边界是让耦合显式化（放在同一个目录里），而不是用目录结构掩盖耦合。
