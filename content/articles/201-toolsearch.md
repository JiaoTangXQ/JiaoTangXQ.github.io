# ToolSearch 和工具前缀：发现机制如何成为骨架的一部分

面试题版本：**"当工具数量很多时，Claude Code 怎么让模型知道哪些工具可用？不可能把所有工具都放进系统 prompt，那么工具发现机制是怎么工作的？"**

## 工具发现的规模问题

Claude Code 支持内建工具、MCP 工具、技能工具（来自 `~/.claude/skills/` 目录）。一个重度用户可能有几十个 MCP server，每个 server 有多个工具，再加上大量技能，工具总数可能超过一百个。

把所有工具的完整描述都放进系统 prompt，会带来：
- Token 成本爆炸：每个工具描述几十到几百 token，一百个工具几千到几万 token 的固定成本
- 模型注意力稀释：工具太多，模型在推理时可能忽略重要工具，或者在不合适的时候选择了不合适的工具
- 上下文窗口压力：相关工具占的空间少了，会话历史能保留的就少了

## ToolSearch 的工作原理

`src/services/skillSearch/` 目录实现了技能工具的按需搜索。不是把所有技能一次性加载到工具表，而是：

1. 系统 prompt 里告诉模型"如果需要特定技能工具，可以使用 `skill_search` 工具按关键词搜索"
2. 模型推理时，先判断有没有现成工具满足需求
3. 如果没有，调用 `skill_search("关键词")` 查询可用技能
4. `skill_search` 返回匹配的技能列表
5. 模型选择并请求加载特定技能
6. 技能加载后，成为当前会话里的可用工具

这是"按需加载"而不是"一次性加载"。大多数技能在单次会话里不会被用到，按需加载避免了无谓的上下文占用。

## 工具前缀的双重作用

当工具来源变多（内建 + MCP server A + MCP server B + 技能），命名冲突成为现实问题。`read_file` 在 server A 里是读取远程文件，在内建工具里是读取本地文件，名字相同但语义不同。

工具前缀解决了两个问题：

**命名空间隔离**：`mcp__server-a__read_file` 和 `read_file` 是不同的工具，不会混淆。

**语义提示**：前缀 `mcp__server-a__` 告诉模型这个工具来自特定 MCP server，模型可以据此推断工具的能力边界（比如，`mcp__database__` 前缀的工具可能都与数据库操作相关）。

`src/services/mcp/mcpStringUtils.ts` 处理工具名称的前缀生成和解析逻辑。

## 技能提示的引导作用

技能工具（`~/.claude/skills/*.md`）是用户自定义的命令，用 Markdown 写提示词，可以包含内联工具调用逻辑。这类工具的发现不能靠系统 prompt 列举（用户可能随时添加新技能），而是靠：

1. 系统 prompt 里有一个"你可以通过 `/` 触发技能"的说明
2. 用户输入 `/` 时，UI 提示自动补全显示可用技能列表
3. 用户选择技能后，系统加载对应的 Markdown 文件，把提示词注入当前会话

技能提示（skill hints）是 UI 层的发现辅助：不是让模型记住所有技能名，而是在用户需要时提供可视化的技能列表。

## 发现机制是骨架的一部分

"发现方式本身就是骨架一部分"这句话的含义：工具能不能被模型正确发现和使用，和工具本身的功能一样重要。

一个功能强大但无法被发现的工具等于不存在。设计工具系统时，必须同时设计：
- 工具在提示词里如何被描述（什么时候用完整描述，什么时候用摘要）
- 工具如何被搜索和按需加载（ToolSearch）
- 工具名如何在命名空间里保持唯一（前缀）
- 工具如何对用户可见（UI 补全提示）

这四件事不是工具功能的附属，而是工具系统的核心基础设施。

## 面试角度的总结

ToolSearch + 前缀 + 技能提示共同解决了"工具多了之后如何保持可发现性"这个问题。核心设计思路是：不是无差别地把所有工具推给模型，而是建立一套分层的发现机制——常用核心工具始终可见，专门工具按需搜索加载，命名空间保证无歧义，UI 辅助降低用户发现成本。这套机制本身需要作为系统骨架来设计，而不是在工具多了之后再临时解决。
