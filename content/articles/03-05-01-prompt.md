# 为什么 prompt 命令为什么更像会话工作方式

`prompt` 命令最容易被误解成“一个会展开成大段文本的快捷方式”。Claude Code 的实现明显不是这么浅。对它来说，`prompt` 命令真正改变的是当前会话接下来如何工作。

在命令分派里，`prompt` 分支不会像 `local` 那样直接把事情做完，也不会像 `local-jsx` 那样先拉起一段界面。它会进入 `getMessagesForPromptSlashCommand()`，生成要送进会话的正文，再附带工具许可、附件、可能的 hooks，最后以 `shouldQuery: true` 的方式把这些内容继续交给主循环。也就是说，它不是本地动作的替代品，而是在重写这一轮之后模型与系统共同面对的工作条件。

这就是为什么 `prompt` 更像一种工作方式。你不是在本地立刻执行一个命令，而是在对当前会话说：接下来请按这一套规则、这一套背景和这一套能力边界来思考与行动。技能、插件和很多扩展命令之所以最后都落在这里，就是因为它们真正要改变的是会话制度，而不是终端立即发生的一次动作。

从这个角度看，`prompt` 命令才是 Claude Code 命令系统最有灵魂的一层。它不是快捷键，而是临时装入当前会话的一种模式。

## 实现链
`prompt` 命令最终走 `getMessagesForPromptSlashCommand()`，产出一组新消息、可用工具、模型和 effort 设定，再把整轮重新送回主循环。它不是执行一个小函数，而是在改写接下来这段会话怎么工作。

## 普通做法
直觉上更像命令的东西，通常会被做成本地 handler，跑完输出一段文本就结束。

## 为什么不用
因为很多 skill 本质上不是“立刻做一件事”，而是“暂时把这场会话切成另一种工作模式”。这件事只能通过主循环完成，不能靠本地 handler 冒充。

## 代价
命令质量会强依赖 prompt 设计和主循环稳定性，不像本地命令那样边界分明；但它也因此成为这套命令系统里最像 agent 的部分。
