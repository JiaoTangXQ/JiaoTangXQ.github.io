# postSamplingHooks 凭什么成为整理时机的协调者？

整理（compression/compaction）操作本身不复杂，复杂的是**它应该在什么时机发生**。

## 整理时机为什么是难题

LLM 的推理过程（sampling）结束后，系统有一批需要做的事情：

- 更新 token 计数，检查是否需要触发 autocompact
- 如果会话历史超过阈值，执行 microcompact
- 判断是否需要更新 SessionMemory
- 记录进度信息（工具使用次数、token 消耗）
- 触发用户配置的 hooks

这些事情的共同特点是：它们都必须**在当前主回合落稳之后**才能进行。如果在 LLM 还在推理的过程中触发 autocompact，你会强行截断正在进行的 sampling，导致输出被破坏。

但如果没有统一的"回合落稳"信号，每个组件只能自己轮询、自己猜测什么时候是合适的时机——这是典型的"条件竞争"温床。

## 统一插槽的价值

`postSamplingHooks` 解决的核心问题是：**提供一个统一的、明确的回合后执行点**，让所有"需要等回合结束再做"的操作都有地方注册。

这是一个调度协调问题，不是功能实现问题。各个组件（compact、memory、progress tracking）自己知道怎么做自己的事，但它们不知道"现在是不是合适的时机"。`postSamplingHooks` 承担的就是这个时序信号的职责。

当 LLM 完成一轮 sampling 并且工具调用序列结束后（主回合落稳），`postSamplingHooks` 框架按顺序触发注册在里面的所有 hook。这时每个 hook 都知道：自己运行的时候，主回合已经结束了，当前状态是稳定的。

## 为什么顺序也重要

不是所有 post-sampling hook 都彼此独立。比如：

1. 先检查 token 用量
2. 如果需要 compact，先做 compact
3. compact 完成后，更新 session memory（因为要基于压缩后的状态做记忆）

如果把这三步分散到三个独立的位置，每个组件各自在"回合后某个时间"触发，执行顺序就没有保证。autocompact 可能在 session memory 更新之后才运行，导致记忆是基于压缩前状态生成的，与实际的上下文不一致。

`postSamplingHooks` 的插槽设计允许按序注册，保证有依赖关系的操作按正确顺序执行。

## 整理是正式时机，不是临时补丁

这一点是设计哲学层面的：整理动作不应该在"发现上下文快满了就赶紧做"的压力下运行。

在压力触发模式下，整理会和正在进行的工作竞争，可能在当前工具调用的中间截断上下文。这种情况下生成的压缩摘要，很可能是不完整的——因为它是在"当前轮次还没结束"时产生的。

`postSamplingHooks` 把整理从"响应压力"变成了"在正式插槽内执行"：整理总是在回合边界上发生，总是在完整的、稳定的状态下生成摘要。摘要质量因此更稳定，系统的连续性保证也更强。

## 回合后插槽的可扩展性

这个设计还有一个好处：未来如果需要在回合后做更多事情（比如检查是否需要触发某种通知、是否需要向外部系统推送进度），只需要注册新的 hook，不需要修改任何已有组件的内部逻辑。

这种可扩展性来自于"时序协调职责"和"具体操作职责"的分离。`postSamplingHooks` 只管时序，各个 hook 只管自己的操作。

## 面试答题要点

被问到"你们的回合后处理是怎么设计的"时，关键是说清楚两件不同的事：

1. **"需要在回合后做什么"**（这是功能问题：compact、memory update、progress tracking）
2. **"怎么保证这些事情在正确的时机做"**（这是时序协调问题）

`postSamplingHooks` 解决的是第二个问题。很多系统把这两个问题混在一起，导致各个组件里散布着"检查是否完成了回合"的条件判断。

追问"如果有 hook 抛出异常怎么办"：这取决于 hook 的类型。有些 hook（比如 compact）的失败是可以接受的（下一轮再试），有些 hook 的失败需要让整个回合标记为失败。`postSamplingHooks` 的设计应该支持 hook 声明自己的错误处理策略。
