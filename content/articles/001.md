# Claude Code 源码全景地图

这张地图回答的是一个最基础的问题：为什么 Claude Code 看起来像很多目录，实际却像一台完整工作台。

## 实现链

这套代码真正的骨架，不是目录名，而是运行链。启动侧主要看 `main.tsx`、`interactiveHelpers.tsx`、`commands.ts`。会话侧看 `QueryEngine.ts` 和 `query.ts`。行动侧看 `Tool.ts`、`tools.ts`、`services/tools/*`。任务与子 agent 看 `Task.ts`、`tasks.ts`、`tasks/*`、`tools/AgentTool/*`。团队机制看 `utils/swarm/*`、`utils/teammateMailbox.ts`。界面与状态看 `state/*`、`components/*`、`context/*`。

这条分法对应的是代码真实协作方式。一次启动会跨入口、设置、命令装配；一轮查询会跨 QueryEngine、主循环、工具调度、权限和消息显示；一个子 agent 会跨 AgentTool、任务框架、通知与回收。

## 普通做法

更普通的源码导览，会按目录平铺：`utils` 一章、`components` 一章、`services` 一章、`tools` 一章。这样省事，也最符合文件系统直觉。

业内很多中小项目按目录讲没有问题，因为目录边界和运行边界差不多。

## 为什么不用

Claude Code 不能这么讲，因为它的重要设计大多跨目录。比如工具不是只在 `tools/` 里成立，还要经过 `Tool.ts` 的协议、`services/tools/*` 的执行、权限系统的拦截和 UI 的展示。再比如任务不是 `tasks/*` 自己的事，还要接上 `Task.ts`、`framework.ts`、`AgentTool` 和消息通知。

如果按目录平铺，新手会看到“有很多文件夹”；如果按运行链拆，他才会看懂“这些文件夹怎样合成同一台机器”。

## 代价

这种地图的代价是阅读门槛更高。你不能只扫目录名，而要顺着调用链和状态链读代码。同一个目录也会在不同地图里反复出现，看起来不像传统手册那样整齐。

但它更真实。Claude Code 的复杂度本来就不在单个目录里，而在这些目录如何被组织成工作台。
