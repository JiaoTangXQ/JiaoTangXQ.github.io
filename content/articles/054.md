# Claude Code 的长会话整理为什么不在主回合里做 compact？

面试题：「你的 AI agent 上下文窗口快满了，怎么处理？」

最直觉的答案是：检测到上下文超过阈值，立刻摘要压缩旧消息。Claude Code 没这么做，因为这个「立刻」会踩进当前正在执行的工作里。

## 问题的本质

想象用户正在让 Claude 修改一个复杂的 bug，Claude 已经读了 10 个文件、执行了 3 个工具调用，现在上下文快满了。这时候如果系统打断执行做 compact，会发生什么？

- compact 本身需要调用模型生成摘要，又消耗 token
- compact 完成后，之前那 10 个文件的读取结果可能在摘要里丢失细节
- 更糟的是，compact 中途如果失败，整个会话状态变得不一致

Claude Code 的选择是：**compact 不抢进主回合，只在回合边界后触发**。

## compact.ts 的分层设计

`compact.ts` 里有几个不同粒度的整理机制：

**microcompact**：最轻量的整理，只压缩明确可以丢弃的内容（比如工具调用的中间输出），不尝试做语义摘要。在 token 压力还不严峻时作为预防性操作。

**autocompact**：触发完整的摘要压缩。它的核心逻辑是通过 `runForkedAgent` 起一个独立的 forked agent 来生成摘要：

```typescript
// compact.ts 的核心路径
const result = await runForkedAgent({
  messages: messagesToCompact,
  systemPrompt: buildCompactSystemPrompt(),
  maxTokens: COMPACT_MAX_OUTPUT_TOKENS,
  ...
})
```

注意：是 forked agent，不是在当前 agent 上下文里直接调用模型。这样即使 compact 失败，主 agent 的状态不受影响。

**compact boundary marker**：`SDKCompactBoundaryMessage` 是一个特殊消息类型，标记历史里的某个位置「这之前已经被整理过了」。重新加载会话时，系统看到这个标记，知道边界之前的内容已经是摘要形式。

## 时机：stopHooks 和回合后插槽

整理触发的时机是关键。`compact.ts` 里有 `executePreCompactHooks` 和 `executePostCompactHooks` 两个 hook 点，但真正控制触发时机的是主循环里的 `stopHooks`。

一轮完整的对话是这样的：
1. 用户输入 → 主 agent 执行
2. 主 agent 完成一轮（遇到 stop_reason 或工具全部完成）
3. **此时才检查是否需要 compact**
4. 如果需要，异步触发 compact（不阻塞下一个用户输入的处理）

这个「回合后插槽」的设计意味着 compact 和工作是并行关系：compact 在后台整理历史，主线程继续响应用户。用户感受到的是稍慢一点的响应（compact 争抢资源），而不是「系统暂停了」。

## CompactBoundary 与连续性

compact 之后，系统需要保证 agent 能继续工作，而不是忘掉之前做了什么。`compact.ts` 里有一段逻辑处理「重带回线索」：

```typescript
// compact 后重新附上必要的上下文
const attachments = [
  await getAgentListingDeltaAttachment(...),
  await getDeferredToolsDeltaAttachment(...),
  await getMcpInstructionsDeltaAttachment(...),
]
```

这些 attachments 把「当前有哪些可用工具」「有哪些延迟工具」这类元信息重新注入到 compact 后的上下文里。没有这一步，compact 后的 agent 可能不知道自己有哪些能力。

## SessionMemory：compact 之外的长期记忆

`compact` 处理的是当前会话内的上下文压缩。跨会话的知识积累是另一套机制：`SessionMemory`（对应 `~/.claude/MEMORY.md`）。

`memdir.ts` 在每次会话开始时把 MEMORY.md 的内容注入 system prompt 的一部分。这个文件有两个上限：200 行和 25KB（`MAX_ENTRYPOINT_LINES = 200`，`MAX_ENTRYPOINT_BYTES = 25_000`）。超过这个大小会被截断并给出警告。

两者的分工是：compact 负责「当前会话还能继续工作」，MEMORY.md 负责「下次会话还记得重要事实」。

## 面试答题思路

「AI agent 上下文管理策略」的答题要点：

1. **整理时机比整理方法更重要**：在错误的时机做整理比不整理更糟糕
2. **用 forked agent 做摘要，不在主 agent 上下文里摘要**：保护主状态的完整性
3. **boundary marker 解决恢复问题**：不是把摘要当成历史的第一条消息，而是明确标记边界
4. **分层整理**：轻量的 microcompact 用于日常，完整的 autocompact 用于极端情况
5. **工作记忆和长期记忆分开**：会话内 compact ≠ 跨会话记忆，两套机制解决两个问题
