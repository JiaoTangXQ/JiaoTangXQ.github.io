# attachments 为什么在每轮 query 末尾才注入而不是在开始时一次性加载？

## 附件的来源

Claude Code 里的 attachments 是一类特殊消息，它们不是用户输入，也不是模型输出，而是系统在合适时机注入到对话里的上下文材料：

- 记忆文件（从 `~/.claude/memory/` 读取的相关记忆片段）
- 文件状态（当前工作目录里的文件变化）
- 技能发现（新发现的可用技能）
- 任务完成通知（后台 agent 任务完成的结果）
- 挂起命令（队列里等待的命令）

## 为什么在轮次末尾而不是开始时注入？

`query.ts` 里 `getAttachmentMessages()` 的调用位置很靠后：

```typescript
// query.ts — 在一轮 tool_results 处理之后
// Instrumentation: Track message count before attachments
logEvent('tengu_query_before_attachments', {
  messagesForQueryCount: messagesForQuery.length,
  // ...
})

// Get queued commands snapshot before processing attachments.
for await (const attachment of getAttachmentMessages(
  null,
  updatedToolUseContext,
  null,
  queuedCommandsSnapshot,
  [...messagesForQuery, ...assistantMessages, ...toolResults],
  querySource,
)) {
```

这段代码在 `assistantMessages` 和 `toolResults` 都处理完之后才运行。原因有几个：

**1. 任务完成通知是动态生成的**

后台 agent 任务在跑，主会话在继续对话。任务什么时候完成是不确定的——如果在轮次开始时就采集一次 attachments，那轮次进行中完成的任务就会被错过，只能等到下一轮才能被主会话看到。

在末尾采集，能捕获到这一轮执行过程中完成的任务通知。

**2. 队列管理的精确性**

注释里明确说了：

> Slash commands are excluded from mid-turn drain — they must go through processSlashCommand after the turn ends

slash 命令不能在 turn 过程中被当作 attachment 处理，只能在 turn 结束后走 `processSlashCommand`。如果在开始时就采集队列，这个区分就会混淆。

**3. 避免重复记忆注入**

`filterDuplicateMemoryAttachments()` 是一个去重函数，它会对比当前要注入的记忆和已有消息历史里已经出现过的记忆，防止同一段记忆被反复注入。

这个去重需要知道「已有消息历史是什么」——用 `[...messagesForQuery, ...assistantMessages, ...toolResults]` 作为参考。如果在轮次开始时采集，assistantMessages 还是空的，去重就不准确。

## startRelevantMemoryPrefetch 的异步预热

记忆检索是可以提前开始的——用户输入后就可以开始检索相关记忆，不需要等整轮工具调用完成：

```typescript
// query.ts 靠前的位置
using pendingMemoryPrefetch = startRelevantMemoryPrefetch(...)
```

注意 `using` 关键字（TypeScript 5.2 的 Explicit Resource Management）——这个 prefetch 会在当前作用域结束时自动清理。

这是「提前开始，但延迟消费」的模式：记忆检索提前并发执行（不阻塞主流程），等到 `getAttachmentMessages()` 需要用记忆时，prefetch 大概率已经完成，可以直接拿结果。

## 队列分层的细节

```typescript
const queuedCommandsSnapshot = getCommandsByMaxPriority(
  sleepRan ? 'later' : 'next',
).filter(cmd => {
  if (isSlashCommand(cmd)) return false
  if (isMainThread) return cmd.agentId === undefined
  return cmd.mode === 'task-notification' && cmd.agentId === currentAgentId
})
```

优先级分为 `'next'` 和 `'later'`：
- Sleep 工具跑过的情况下，取 `'later'` 级别（Sleep 相当于一个时间边界，之后的任务可以被纳入）
- 否则只取 `'next'` 级别（更紧迫的）

Subagent 和主线程的队列是分开管理的：主线程只看 `agentId === undefined` 的命令，subagent 只看发给自己 agentId 的任务通知。

## 面试延伸

这个设计回答了一个更一般的问题：「如何在异步系统里管理动态上下文注入的时机？」

关键点是：上下文材料的有效性和生产时机是动态的，不能在流程开始时一次性采集完。需要在合适的时间点（末尾或特定检查点）重新采集，并且要有去重机制防止重复注入。
