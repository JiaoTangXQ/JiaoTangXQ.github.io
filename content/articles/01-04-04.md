# 系统设计题：如果你来设计 agent 的跨会话记忆，你会选择哪种方案，为什么 Claude Code 用了四种？

## 面试题里的陷阱

"请设计一个 agent 的跨会话记忆系统"。

大多数候选人会给出三类方案之一：

1. **永久保留**：每次会话都带上完整历史记录。简单，但上下文窗口撑不住长会话。
2. **周期压缩**：达到一定长度就压缩一次，用摘要替换原始内容。可行，但压缩时机和策略难定。
3. **向量数据库**：把历史存到 RAG 系统里，每次按相关性检索。扩展性好，但引入了检索质量问题，而且对上下文里"当前会话的完整信息"这个需求不适合。

Claude Code 用了四种，每种对应一个不同的问题。把这四种机制的设计目标拆开来看，能看清楚"跨会话记忆"其实是若干不同问题的集合，不是一个统一的问题。

## 第一种：microcompact

`microcompact` 是最轻量的那种。它在会话还没到临界状态之前就开始工作：把消息历史里"最不重要的部分"先压掉——比如重复出现的文件读取结果、冗长的工具输出（你让模型读了同一个文件三次，前两次可以压缩）。

它解决的问题是：**防止上下文在达到临界点之前就开始降解**。如果等到快满了才压缩，那次压缩必须很激进，信息损失大。如果持续地把最不可能需要再引用的内容轻轻压掉，就能让真正重要的内容保持在高质量状态。

这里有一个设计选择：什么是"最不重要"？这个判断不是随机的，它基于内容类型（工具输出通常比用户消息更可压缩）和引用频率（从未被后续对话引用过的内容更可压缩）。

## 第二种：autocompact

`autocompact` 是保命机制。当会话历史快撑满上下文窗口时触发，它做的是一次完整的历史摘要：把过去所有的对话浓缩成一段结构化摘要，从这段摘要开始新的上下文。

注释里的描述很直接：`When 会话快撑不住时保命`。

它和 `microcompact` 的区别是时机和侵略性：`microcompact` 是持续性的轻度清理，`autocompact` 是到达阈值时的一次性大清理。两者都是损失性操作，但 `autocompact` 的损失更大——所有消息都变成了摘要，细节不可恢复。

`autocompact` 触发时，Claude Code 会先运行 hooks（`pre_compact` 和 `post_compact`），让外部系统有机会在压缩前后做额外处理（比如把原始历史存档到外部存储）。

## 第三种：session memory

`session memory` 是当前工作的"外置笔记本"。它不是历史压缩，而是模型可以主动写入的结构化存储——模型决定把什么东西写进去（比如"用户偏好的代码风格"、"这个项目的命名约定"、"我们已经确认的技术方向"），写进去的内容会在后续对话里作为附件附上。

和前两种的区别：`microcompact` 和 `autocompact` 是系统自动管理的，`session memory` 是模型主动管理的。模型知道自己在写笔记，写什么、何时写，是模型的决策。

这解决的是一个不同的问题：有些信息不是历史记录的压缩，而是从对话中提炼出来的结论。如果用历史压缩来保留这些内容，它们会和其他内容混在一起，不够突出；作为独立的笔记存储，它们在每次对话开始时就被注入，始终可见。

## 第四种：auto-memory

`auto-memory` 是最长时间尺度的机制。它在 dream 任务（`DreamTask`，后台记忆整合子 agent）里运行，对多次历史会话做回顾，提炼出跨会话的长期知识：用户的工作习惯、常见的项目模式、以前解决过的问题类型。

它和 session memory 的区别是：session memory 是当前会话内的笔记，auto-memory 是跨会话的沉淀。两者的粒度和持久性不同。

`DreamTask` 在 `DreamTask.ts` 里被实现为一种后台任务，有自己的 `taskId`（`d` 前缀）、`phase`（`starting`/`updating`）、`filesTouched` 列表。它是一个真正的 agent：有自己的提示词、工具集、执行周期，只是在后台运行，不打扰前台对话。

## 四种机制的关系

```
时间尺度 → 短 ←————————————————→ 长
          microcompact  autocompact  session memory  auto-memory
主动/被动 → 系统自动 ←—————→ 系统自动  模型主动       系统自动
侵略性   → 低       ←—————→ 高        N/A（写入）   低（整合）
触发条件 → 持续     ←—————→ 阈值      模型决策       会话后台
```

这四种不互相替代，而是处理不同的子问题：
- **microcompact**：在上下文满之前的持续清理
- **autocompact**：上下文快满时的紧急压缩
- **session memory**：重要结论的显式存储
- **auto-memory**：跨会话知识的后台整合

如果只用其中一种，其他三个问题就没有解决。

## 面试指导

**考法**：`Claude Code 怎么处理长上下文？` 或 `agent 的记忆系统应该怎么设计？`

**满分结构**：说明记忆问题本身是多层次的，然后用时间尺度 × 主动性 × 侵略性三个维度区分四种机制，每种机制说清楚它解决了哪个具体问题、为什么其他机制替代不了它。

**常见低分答案**：只说"压缩历史"或"向量检索"。这说明候选人把多个不同问题合并成了一个，没有分析出问题的层次。

**加分点**：提到 `autocompact` 里有 hooks（`pre_compact`/`post_compact`），说明连压缩这个"系统内部事务"也被设计成可被外部制度介入的节点。以及提到 `DreamTask` 是一个真正的后台 agent，不是一个简单的定时脚本，这说明你理解了 Claude Code 的"记忆整合"和"任务执行"是同一套架构在不同场景的应用。
