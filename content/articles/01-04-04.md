# 连续性机制为什么说明系统不只活在这一轮

如果把 Claude Code 只看成“一问一答”的机器，很多设计都会显得过重。可一旦你承认它在处理的是持续工作，而不是单轮聊天，很多东西就突然合理了。连续性机制正是在说明这一点: 这套系统从来不只活在当前这一轮。

它准备的不是一个单独的“记忆功能”，而是一整套时间制度。轻量的 `microcompact` 负责先减掉最容易变薄的部分，`autocompact` 负责在会话快撑不住时保命，`session memory` 像当前工作的外置笔记，`auto-memory` 则把更值得长期保留的东西沉到底层。它们解决的不是同一件事，却都在回答同一个问题: 这一轮结束以后，什么应该继续留在系统里。

这说明 Claude Code 对连续性的理解很成熟。真正重要的不是“永远不忘”，而是知道哪些东西该留在眼前，哪些该压缩，哪些该外置保存。也正因为它把连续性做成了多层制度，后面的恢复、记忆、附件回流和长会话治理才不会互相打架。

所以连续性不是附属优化，而是这台系统的时间观。

## 实现链
这页落到的实现链都属于制度世界：权限装配、设置来源、hooks 介入、连续性设施或条件能力 gating。代码的共同特点是，它们不是在主循环失败以后补救，而是在主循环真正推进前后就提前改写资格和边界。

## 普通做法
更普通的做法是先把 happy path 主流程跑通，再把权限、外部规则和跨回合记忆当辅助功能补上。很多系统都是这样长出来的。

## 为什么不用那种做法
Claude Code 没这么长，是因为对 agent 来说，真正危险的往往不是下一步怎么走，而是这一步有没有资格走、走完后有没有留下可继续工作的状态、外部制度有没有权利插手。它把这些问题提前写进了运行时。

## 代价与回报
回报是系统更不容易越权和失忆。代价是相关逻辑分散，阅读时不能只盯着一条主流程。
