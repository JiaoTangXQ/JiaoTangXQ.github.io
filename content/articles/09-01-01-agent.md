# 多几个 agent 只会多几个声音，不会自己长出组织

并发会带来更多声音，但不会自动带来领导、秩序和边界。

## 实现链

`AgentTool.tsx` 里只有在同时给出 `team_name` 和 `name` 时，才走 teammate 生成路径。并且代码明确禁止 teammate 再生成 teammate，理由是 `TeamFile.members` 是扁平 roster，嵌套 teammate 会让归属关系混乱。这已经很能说明问题：团队不是随便多开一层 agent 树，而是先承认一个明确 roster。

## 普通做法

普通做法会觉得多开几个 agent 自然就是团队。谁有结果谁说，谁先抢到任务谁做，最后再合并。

## 为什么不用

Claude Code 不敢把团队建立在“谁先出声谁算数”上，因为一旦没有显式组织，用户就不知道该听谁，worker 也不知道该向谁汇报。系统里只会多出并行噪音。

所以它先限制结构，再允许协作。先承认团队 roster 是平的、lead 是单点、通信要显式，团队才算成立。

## 代价

代价是组织能力看起来没有那么自由。你不能无限套娃 teammate，也不能临时搞一堆临时上下级，但换来的是整个团队结构更可解释。
