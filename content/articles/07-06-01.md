# 命名空间冲突如何解决？plugin:namespace:command 的生成逻辑

## 面试题：如果 100 个不同作者的插件都提供了一个叫 "review" 的命令，怎么解决冲突？

最简单的回答：后加载的覆盖先加载的（Last Writer Wins）。

但这有什么问题？不确定性。用户不知道执行的是哪个 "review"，调试时无法复现问题，日志里的命令名也没有意义。

Claude Code 的方案：在命名阶段就消灭歧义——命令名本身就携带来源信息。

---

## buildNamespace：把目录结构变成冒号分隔的层级

```typescript
function buildNamespace(targetDir: string, baseDir: string): string {
  const normalizedBaseDir = baseDir.endsWith(pathSep)
    ? baseDir.slice(0, -1)
    : baseDir

  if (targetDir === normalizedBaseDir) {
    return ''  // 在根目录，没有命名空间
  }

  const relativePath = targetDir.slice(normalizedBaseDir.length + 1)
  return relativePath ? relativePath.split(pathSep).join(':') : ''
}
```

这个函数把目录路径里的层级结构翻译成冒号分隔的命名空间：

```
/plugins/my-plugin/code-review/strict/ → namespace: 'code-review:strict'
/plugins/my-plugin/code-review/      → namespace: 'code-review'
/plugins/my-plugin/                  → namespace: ''（根级，没有命名空间）
```

---

## getSkillCommandName：技能目录格式的命名

```typescript
function getSkillCommandName(filePath: string, baseDir: string): string {
  const skillDirectory = dirname(filePath)        // .../code-review/SKILL.md → .../code-review
  const parentOfSkillDir = dirname(skillDirectory) // → .../my-plugin
  const commandBaseName = basename(skillDirectory) // → 'code-review'

  const namespace = buildNamespace(parentOfSkillDir, baseDir)
  return namespace ? `${namespace}:${commandBaseName}` : commandBaseName
}
```

对于技能目录格式（`skill-name/SKILL.md`），命令名取的是**目录名**，不是文件名。命名空间是目录的**父路径**相对于根的部分。

结果示例：
```
plugins/my-plugin/SKILL.md         → 'my-plugin'
plugins/my-plugin/review/SKILL.md  → 'my-plugin:review' (如果 baseDir 是 plugins/)
```

## 插件级别的全限定名

`loadPluginCommands.ts` 在技能名前面还要加插件名前缀：

```typescript
// 简化示意
function getPluginCommandName(pluginName: string, relativeCommandName: string): string {
  return relativeCommandName ? `${pluginName}:${relativeCommandName}` : pluginName
}
```

最终形态：
```
my-plugin:code-review:strict
my-plugin:review
my-plugin
```

这样两个不同插件各自有一个叫 "review" 的命令，全限定名分别是 `plugin-a:review` 和 `plugin-b:review`，不存在冲突。

---

## 与 Java 包名的对比

这个设计和 Java 的全限定类名（FQCN）类似：

| Java | Claude Code |
|---|---|
| `com.example.utils.StringUtils` | `my-plugin:utils:string-helper` |
| 包名唯一性由域名反转保证 | 插件名唯一性由 marketplace 注册保证 |
| 类名在包内唯一 | 命令名在插件内唯一 |
| FQCN 全局唯一 | 全限定命令名全局唯一 |

区别是：Java 的包名是编译时固定的，而 Claude Code 的命令名是在加载时动态生成的——插件作者不需要手动填写全限定名，系统根据目录结构自动派生。

---

## 代价：命令名变长

```
/review    vs    /my-plugin:code-review:strict
```

用户直接输入命令时，后者明显更难打。这是一个真实的 UX 代价。

Claude Code 的缓解策略是：
1. 补全（Tab）——用户不需要完整输入全名
2. `displayName` 字段——技能可以声明一个更短的显示名，而全限定名只用于内部查找
3. 搜索——`SearchPalette` 支持模糊匹配，用户可以输入 "review" 找到所有包含这个词的命令

## 面试指导

**"如何处理分布式系统中的命名冲突？"**

这道题的经典回答是"全局唯一 ID"（UUID、自增 ID）。但 Claude Code 的方案更有趣：用**语义化的层级命名空间**而不是无意义的 ID。

优点：
- 名字本身携带位置信息（来自哪个插件、哪个目录）
- 调试时一眼能看出来源
- 冲突解析不依赖外部注册服务（直接从目录结构派生）

缺点：
- 插件/目录重命名会破坏现有引用
- 名字可能很长

什么时候用语义化命名空间比 UUID 更合适？当命名的"位置信息"本身就有价值，且命名空间的树形结构是稳定的时候。插件系统就是这个场景。
