# 面试题：让你给 Claude Code 写技术文档，你会选择按目录结构组织还是按功能组织？

## 先复现一下迷路过程

拿到 Claude Code 源码，打开根目录，尝试理解这个系统。

目录树的第一层有 `src/`，进去之后有 `entrypoints/`、`tools/`、`commands/`、`tasks/`、`services/`、`utils/`……几十个子目录。先打开 `entrypoints/cli.tsx`，这是程序的入口。读了一半，发现它调用了 `main.js`，但 `main.js` 不在 `entrypoints/` 里，在上一层。找到了，读到一半，发现关键的会话逻辑在 `QueryEngine.ts`，同样在上一层目录。`QueryEngine.ts` 里调用了 `processUserInput`，这个函数在 `utils/processUserInput/processUserInput.ts`。在那里找到了工具分发逻辑，但具体工具的定义在 `tools/BashTool/BashTool.ts`……

这是一个真实会发生的迷路过程。目录树把代码组织成开发者视角的文件管理结构，但读者想要的是"这个系统是怎么活起来的"这个问题的答案，而这个答案横跨了至少六个不同目录。

## 按目录写的问题

"按目录写"的技术文档会变成这样：

```
第 1 章：entrypoints/ 目录
  - cli.tsx：入口文件
  - agentSdkTypes.ts：SDK 类型
  
第 2 章：tools/ 目录
  - BashTool/
  - FileReadTool/
  ...
  
第 3 章：utils/ 目录
  - processUserInput/
  - queryContext.ts
  ...
```

读者在第 2 章读到 BashTool 的实现，但不知道 BashTool 是怎么被调用到的——因为那部分逻辑在第 1 章的 QueryEngine.ts 里，而 QueryEngine.ts 在第 1 章只是被"介绍了一下功能"。读者在第 3 章读到 `processUserInput`，但不知道它为什么要做那些事——因为这个函数解决的设计问题（用户输入可能是斜杠命令/普通提示词/工具结果，需要统一处理入口）从来没有被明确说出来。

目录树服务于代码的存放，不服务于读者的理解。这是它作为文档组织结构的根本局限。

## 按功能写的问题

反过来，只按功能写（"启动"、"命令系统"、"工具调用"、"任务管理"、"权限系统"）会把所有细节糊成大块。

Claude Code 里有一个很典型的小设计点：`cli.tsx` 里的 `--version` 处理路径是零模块加载的快速路径，因为 version 字符串在构建时被内联进了 `MACRO.VERSION`，不需要初始化任何模块就能返回。这个设计解决的是"用户只是想看个版本号，不应该为此等待整个 CLI 初始化"的问题。

如果这个设计点被塞进"启动"这个大功能章节里，它就变成了"还有一个启动优化"，而不是"工程师在用户体验和初始化成本之间做了一个精确权衡"。大章节让读者知道系统有这个功能，却看不见设计师在解决什么问题。

## Claude Code 源码里有哪些横跨多目录的实现链

真正重要的设计理解，往往来自跨目录的视角：

**从用户输入到模型响应的完整链路**：
```
cli.tsx → main.tsx → QueryEngine.submitMessage 
  → processUserInput（utils/processUserInput/）
  → query（query.ts）
  → 工具执行（tools/XXXTool/）
  → 权限检查（utils/permissions/）
  → AppState 更新（state/AppStateStore.ts）
```

这条链路至少经过 6 个目录。如果按目录讲，每个目录只讲自己的部分，读者永远不会理解这条链路是怎么运作的。

**上下文装配链路**：
```
QueryEngine.fetchSystemPromptParts
  → context.ts 的 getSystemContext（git 状态）
  → context.ts 的 getUserContext（CLAUDE.md 发现）
  → constants/prompts.ts 的 getSystemPrompt（工具描述注入）
```

这三个来源最终合成一个系统提示，影响模型的所有后续行为。按目录写，这三个部分在三个不同地方，读者看不见它们是同一件事的三个组成部分。

## 什么叫"沿流程 + 设计问题下钻"

好的文档组织方式是：**沿真实执行流程推进，在每个关键节点停下来说清楚这里在解决什么设计问题**。

流程给读者方向感：知道现在在看的代码在整个执行链路的什么位置。设计问题给读者深度：知道为什么这里要这样做，而不是用更简单的方式做。

两个缺一不可。只有流程，读者知道"发生了什么"，不知道"为什么这样设计"。只有设计问题，读者知道"解决了什么问题"，不知道"这个问题在执行过程的哪个环节出现"。

## 面试指导

**这道题的考点**：技术写作能力和系统理解能力是否同时具备。光会说"要有可读性"不够，要能说出具体的组织策略以及为什么那个策略在这个系统上有效。

**满分答法**：先指出按目录和按功能各自的局限，然后提出"沿执行流程 + 在关键节点深入设计问题"的组织方式，举出一条具体的跨目录执行链路作为例子，说明为什么这条链路必须被作为一个整体来讲而不是分散在不同章节。

**容易被追问的地方**：什么叫"关键节点"？不是随便一个函数调用，而是"如果不在这里做这个特殊处理，会出现什么具体问题"的地方。比如 `--version` 的零模块加载、`getCommands` 的非 memoize、`mutableMessages` 的跨轮保持，这些都是值得停下来深入的节点。
