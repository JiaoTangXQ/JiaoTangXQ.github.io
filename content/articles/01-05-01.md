# 为什么按目录平铺会让普通读者迷路

源码目录告诉我们文件放在哪里，却不会直接告诉我们系统怎样活起来。`main.tsx`、`QueryEngine.ts`、`commands.ts`、`Tool.ts`、`AppStateStore.ts` 分别躺在不同位置，但真正工作时，它们并不是排着队让人参观，而是在同一条运行链里彼此接力。

如果按目录一路平铺去讲，读者很容易产生错觉: 好像每个目录都在单独负责一块完整世界。可 Claude Code 最精彩的地方，恰恰不是某个目录自己有多完整，而是很多关键设计都横穿多个地方。一次用户输入，既会碰到输入整理、命令分流、权限边界、主循环推进，也会在最后回到消息显示和状态更新。目录树把这些东西拆散了，运行过程却把它们重新编成一条生命线。

对普通读者来说，理解“这一句话后来经历了什么”，比记住“它先路过哪个文件夹”重要得多。按目录写最容易把书写成源码导览，而不是系统解释。那样读者会看见很多文件名，却仍然不知道 Claude Code 为什么会被设计成今天这个样子。

## 实现链
从代码映射看，目录树首先服务开发，不直接服务理解。`main.tsx`、`QueryEngine.ts`、`commands.ts`、`Tool.ts`、状态存储这些文件分别落在不同目录，但它们回答的其实是入口、运行时骨架、制度和状态问题。文档如果只照目录平铺，就看不见这些跨目录的实现链。

## 普通做法
最直觉的写法是按目录讲文件，或者按产品功能讲菜单。这两种写法都很常见，也最省心。

## 为什么不用那种做法
Claude Code 这本书没有只选其中一种，因为这套系统的精彩之处大多发生在交叉处：流程走到哪里，为什么要用这种边界处理。只按目录会让普通读者迷路，只按功能又会把实现判断糊成大块。

## 代价与回报
回报是文档能真正贴着代码里的设计问题下钻。代价是文档结构和目录结构不再一一对应，维护时必须不断做导读、回收和索引。
