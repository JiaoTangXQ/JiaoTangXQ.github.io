# 为什么越成功的长会话，上下文压力越大？

这个问题乍一看有点反直觉：成功的会话不应该更好吗？怎么成功反而是问题的来源？

答案要从 LLM 的工作方式说起。

---

## 上下文是"工作台"，不是"档案柜"

在传统软件里，完成的任务可以归档，只保留当前要用的。LLM 没有这个机制——它每次生成回应都要"看到"完整的对话历史。这意味着：

- 你三小时前读的那份文件，现在还在上下文里
- 你中间让它跑的 50 条 bash 命令，输出结果全在里面
- 每次 tool_use 调用加 tool_result 返回，都是一对消息
- 每次它输出的思考块（thinking block），也占 token

一个成功的会话——真正在做事的那种——会产生大量这类"工作痕迹"。这不是 bug，这是它在认真工作的证明。

---

## 成功本身是上下文压力的来源

把这个过程量化一下。假设每轮工具使用平均消耗 2000 token（read 一个大文件很容易 5000 token），每轮模型回应 1000 token，每轮用户输入 200 token。那么：

- 10 轮之后：约 32,000 token
- 50 轮之后：约 160,000 token
- 80 轮之后：超出 200K 上下文窗口

这是一个**做得越多、积累越重**的正反馈循环。越是真实的长期任务，越会遇到上下文压力。把一个大功能从头做到尾，或者调试一个复杂问题，都属于这类。

---

## query.ts 怎么看待这件事

从 `query.ts` 的主循环结构可以看出，它一开始就把"上下文可能随时装满"当成常态来处理，而不是异常情况：

```typescript
// query.ts - 每次进入循环都执行 token 估算和压缩判断
let messagesForQuery = [...getMessagesAfterCompactBoundary(messages)]
// ... (snip, microcompact, autocompact 依次执行)
```

`getMessagesAfterCompactBoundary` 本身就说明了问题：它只取上一次 compact 之后的消息，因为系统默认会话可能已经经历过一次或多次压缩。这是把"会话被压缩过"当成正常状态而不是异常情况来处理。

---

## 三类积累：哪些最沉重

**工具结果（tool results）**是最主的重量来源。一次 `read_file` 读了一个 500 行的文件，那 500 行就全在上下文里，哪怕模型早就处理完了。microcompact 专门针对这类内容，把已经用完的工具结果清除掉。

**思考块（thinking blocks）**是第二重。扩展思考模式下，模型会在每次回应前输出思考过程，这部分可以很长。从 `apiMicrocompact.ts` 可以看到 `clear_thinking_20251015` 这个策略，专门处理旧的思考块（但保留最近的，因为思考块有"必须连贯"的协议约束）。

**附件和 delta 消息**是第三类。每次 MCP server 连接、plan 更新、工具列表变更，都会产生 attachment message。这些在 compact 后会被重新注入（`getDeferredToolsDeltaAttachment`），但在压缩前它们也都占着位置。

---

## 为什么不能只靠更大的窗口？

有人会说：等 Claude 支持 1M 上下文了，这个问题不就消失了？

不完全是。更大的窗口推迟了问题，但没有解决问题。而且：

1. **成本**：token 数量和推理成本近似线性甚至超线性。200K token 的请求比 20K token 贵得多。
2. **质量**：研究表明，当上下文非常长时，模型对远距离内容的注意力会下降（"lost in the middle"问题）。
3. **速度**：更长的上下文意味着更长的 TTFT（首 token 延迟）。

所以即使窗口更大，主动管理上下文仍然是高质量长会话的必要条件。

---

## 面试指导

如果被问到"如何设计 LLM Agent 的上下文管理"，这是一个很好的开场框架：

**承认长会话的上下文增长是结构性的，不是偶发的**。这说明你理解问题的根因，不会提出"等上下文爆了再处理"这种被动方案。

然后自然地引出分层管理：局部清理（microcompact）、系统级摘要（autocompact）、外置记忆（session/auto memory）。每一层解决不同粒度的问题，组合成一个持续运作的时间制度。
