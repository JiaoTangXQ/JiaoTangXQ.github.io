# 面试题：设计一个多任务管理框架，你的 base type 里应该放什么？

**面试题：你要设计一个多任务管理系统，支持 shell 任务、AI Agent 任务、远程任务等多种类型。请设计 base type。**

大多数候选人的第一反应是：把所有任务都会用到的执行字段放进去。`command` 或 `prompt`，`result` 或 `output`，也许加上 `timeout`。

这个思路有问题。`local_bash` 需要 `command`，`local_agent` 需要 `prompt`，`remote_agent` 可能需要 `endpoint`。这些字段的"共性"是假的——它们形态不同，含义不同，硬塞进 base type 只会产生大量 optional 字段和防御性代码。

Claude Code 的 `TaskStateBase` 展示了另一种思路。

## 真正的公共字段：管理接口

```typescript
export type TaskStateBase = {
  id: string           // 唯一标识，贯穿整个生命周期
  type: TaskType       // 类型判别字段，支持类型收窄
  status: TaskStatus   // 生命周期总闸门
  description: string  // 人类可读描述
  toolUseId?: string   // 关联到触发它的工具调用
  startTime: number    // 排序和耗时计算
  endTime?: number     // 终态时间戳
  totalPausedMs?: number
  outputFile: string   // 输出在哪里——统一的可观察接口
  outputOffset: number // 增量读取游标
  notified: boolean    // 通知是否已发——防重复的幂等标志
}
```

这 11 个字段没有一个是"执行细节"。它们是**管理接口**：让系统能识别任务、追踪状态、读取输出、发送通知、决定何时回收。

## 为什么 `outputFile` 和 `outputOffset` 在 base type 里

这是 Claude Code 设计中最巧妙的部分之一。

`outputFile` 是文件路径，`outputOffset` 是已读取的字节偏移量。有了这两个字段，框架层的轮询逻辑可以对所有任务类型使用完全相同的代码：

```typescript
// framework.ts 里的轮询——完全不区分任务类型
if (taskState.status === 'running') {
  const delta = await getTaskOutputDelta(
    taskState.id,
    taskState.outputOffset,  // 从上次读到的位置继续
  )
  if (delta.content) {
    updatedTaskOffsets[taskState.id] = delta.newOffset
  }
}
```

`local_bash` 的输出文件直接是 shell 进程的 stdout；`local_agent` 的输出文件是一个软链接，指向 Agent 的 transcript JSONL 文件。框架不需要知道这些差别——它只管从 offset 读新内容。

## 为什么 `notified` 是 base type 字段

`notified: boolean` 看起来像是通知模块的私有状态，为什么放进公共骨架？

因为"是否已通知"是一个**全局协调问题**，不是单个模块的内部问题。

Claude Code 里，任务完成通知可能来自多个路径：
- `completeAgentTask()` 成功完成时
- `failAgentTask()` 出错时
- `killAsyncAgent()` 被强制终止时
- `TaskStopTool` 从外部调用停止时

如果 `notified` 不在公共骨架里，每条路径都需要额外的协调机制来防止重复通知。放进 base type 后，任何路径发通知前都做同一个检查：

```typescript
updateTaskState<LocalAgentTaskState>(taskId, setAppState, task => {
  if (task.notified) return task  // 已通知，跳过
  shouldEnqueue = true
  return { ...task, notified: true }
})
// 状态更新和 shouldEnqueue 标志在同一个原子操作里
if (!shouldEnqueue) return
// 真正发送通知
```

这是 check-then-act 的原子化版本：状态更新（`notified: true`）和发通知的决定绑定在同一个 `updateTaskState` 回调里，避免并发时的重复通知。

## 类型系统的配合

把 `type` 字段放进 base type，并配合各子类型的字面量类型覆盖，TypeScript 能自动做类型收窄：

```typescript
function handleTask(task: TaskState) {
  switch (task.type) {
    case 'local_agent':
      // 这里 task 自动被收窄为 LocalAgentTaskState
      console.log(task.prompt)  // 编译通过
      console.log(task.pendingMessages)  // 编译通过
      break
    case 'local_bash':
      // 这里 task 自动被收窄为 LocalShellTaskState
      console.log(task.command)  // 编译通过
      break
  }
}
```

如果 `type` 不在 base type 里，这个自动类型收窄就不存在，所有操作都需要手动断言。

## 面试答题框架

设计多任务 base type 时，用这个问题检验每个候选字段：

> **"如果没有这个字段，框架层（管理层、调度层）还能统一处理所有任务类型吗？"**

- `id`：没有 id，就没办法在字典里找到任务。必须放。
- `status`：没有状态，就不知道任务是否结束。必须放。
- `outputFile`：没有这个，框架没法统一读取输出。必须放。
- `prompt`：没有 prompt，框架该怎么办？框架根本不需要知道 prompt。不要放。
- `model`：同上。不要放。

这个问题能帮你区分"真正的公共字段"和"某类任务的执行字段被错误地提升到 base type"。
