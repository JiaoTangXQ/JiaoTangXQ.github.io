# 公共字段为什么要比执行细节更早出现

Claude Code 先写公共字段，不是因为它不关心执行，而是因为它知道治理比执行更容易失控。

## 实现链

`TaskStateBase` 里先放的是 `id`、`type`、`status`、`description`、`startTime`、`outputFile`、`outputOffset`、`notified` 这些字段，而不是某类任务特有的 prompt、模型或 UI 数据。`createTaskStateBase()` 也是先生成这层骨架，再由具体任务去追加自己的执行细节。

这说明 Claude Code 先问的是“这件事怎样被系统管理”，然后才问“这件事具体怎么执行”。

## 普通做法

普通做法更喜欢先把执行字段堆出来。比如子 agent 模块先写 prompt、model、messages、result，等真的需要统一任务面板时，再补 `status` 和 `id`。

这种顺序贴近“功能先行”的开发习惯。

## 为什么不用

Claude Code 没这么排，是因为执行字段往往只服务本任务，而公共字段决定的是跨任务协作。`outputFile` 决定日志怎么看，`notified` 决定消息会不会重复发，`status` 决定能不能继续注入消息，`startTime` 甚至影响列表排序和保留策略。

也就是说，公共字段看起来朴素，实际却是整个任务世界能否统一运作的地基。

## 代价

这样设计会让基础类型看起来比较“官僚”。一开始就有很多每个任务都得带上的字段，读代码的人会感觉系统很重。但这正是它换来的能力：后面各任务不用再各自发明一套最小治理结构。
