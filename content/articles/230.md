# 清理器和退出路径为什么要在启动时就定义

面试题版本：**"Claude Code 进程退出时需要做什么清理工作？这些清理逻辑是在什么时候注册的，为什么不等到真正需要退出时再决定？"**

## 退出时需要做什么

Claude Code 正常退出或异常退出时，有一系列清理工作：

- 终止所有运行中的后台任务（发送 kill 信号）
- 关闭所有 MCP server 连接
- 关闭 bridge 连接
- 刷新 session memory 到磁盘（保存本次会话的重要内容）
- 关闭日志文件句柄
- 清理临时文件
- 发送分析事件（如果配置了）

这些操作如果没做完，会导致：子进程变成僵尸进程、临时文件积累、session memory 部分丢失、分析数据不完整。

## 在启动时注册清理器

`src/setup.ts` 里（或类似的启动入口），进程清理器在应用启动阶段就被注册：

```typescript
// 启动时就注册
process.on('SIGTERM', async () => {
  await cleanup()
  process.exit(0)
})

process.on('SIGINT', async () => {
  await cleanup()
  process.exit(0)
})

process.on('uncaughtException', async (error) => {
  logError(error)
  await cleanup()
  process.exit(1)
})
```

`cleanup()` 函数包含了所有清理逻辑，在所有可能的退出路径（正常退出、用户 Ctrl+C、未捕获异常、SIGTERM）上都被注册。

## 为什么在启动时，不等到退出时

**等到退出时的问题**：如果在需要退出时才临时决定怎么清理，会遇到：
- 退出路径有多种（信号、异常、正常结束），每种都要单独处理
- 退出时的代码不知道当前系统状态（有哪些任务在跑、哪些连接还开着），可能清理不完整
- 某些退出路径可能没有机会执行清理代码（比如 kill -9 无法被拦截，但其他信号应该能被处理）

**在启动时注册的好处**：
- 所有退出路径都有相同的清理行为，不会遗漏某种退出方式
- 清理逻辑和系统初始化逻辑对称——启动时申请的资源，退出时释放
- 清理逻辑作为一等公民被设计，而不是边角处理

## 平稳退出 vs 崩溃退出

清理器还需要区分两种退出情况：

**平稳退出**（用户主动退出、正常完成）：
- 等待正在进行的任务完成（或超时后 kill）
- 完整刷新 session memory
- 发送正常退出的分析事件

**崩溃退出**（未捕获异常、严重错误）：
- 快速 kill 所有子进程（不等完成）
- 尝试刷新 session memory（但可能部分失败）
- 记录错误，发送异常退出的分析事件

这两种情况的清理策略不同，都需要在启动时就计划好。

## 对称性原则

"启动时定义退出路径"体现了一个工程对称性原则：**申请和释放是对称的，初始化和清理是对称的**。

进程在启动时：建立 MCP 连接、注册 bridge、启动后台任务监控。

进程在退出时：关闭 MCP 连接、关闭 bridge、停止后台任务监控。

如果退出时的清理代码散落在各个模块里（每个模块自己决定什么时候清理），很容易出现：某个模块清理了两次（冲突）、某个模块没有清理（遗漏）、清理顺序不对（依赖关系问题）。

集中在启动时注册的清理器，能看到整体清理顺序，能保证所有资源都被处理，能控制依赖关系（先关闭 MCP server 连接，再停止依赖 MCP 的后台任务）。

## 面试角度的总结

在启动时定义清理路径，是"把退出当作系统设计一等公民"的体现。这让退出不再是"程序结束的时候顺手做点清理"，而是"有明确设计的生命周期管理的终点"。对称的申请和释放、集中的清理逻辑、对所有退出路径的覆盖——这些让 Claude Code 可以在任何情况下安全退出，而不是留下各种资源泄漏和不一致状态。
