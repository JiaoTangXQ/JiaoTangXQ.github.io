# 面试题：任务完成后为什么不立即从 AppState 里删除？

## 直觉上的困惑

操作系统里，进程退出后内核会回收它的资源。数据库里，一个事务提交后它的锁就释放了。

为什么在 Claude Code 里，一个任务完成后，还要在 AppState 里留一段时间？

## 任务在 AppState 里是 UI 的数据源

任务面板显示的内容，来源于 AppState 里的任务列表。

当一个后台任务完成时，面板需要告诉用户：
- 任务完成了
- 完成的结果是什么
- 是否有需要用户关注的事项

如果任务一完成就从 AppState 删除，面板里的任务条目会瞬间消失。用户可能看到一个"闪烁"——任务还在，然后突然不在了，而用户还没来得及看到完成的通知。

更糟糕的情况：如果用户恰好在任务完成的那一刻在看任务详情，面板内容瞬间消失，用户不知道发生了什么。

`retain` 标志解决的就是这个问题：任务完成后，在 AppState 里继续保留，让 UI 有时间展示完成状态。

## PANEL_GRACE_MS：宽限期的量化

在 `src/utils/task/framework.ts` 里有 `PANEL_GRACE_MS` 常量，代表任务完成后在面板里保留的毫秒数。

这个数字需要平衡两个需求：
- 足够长：用户能看到任务完成的通知，面板不会瞬间消失
- 足够短：已完成的任务不会长期占用 AppState，不影响系统内存和性能

合理的取值可能在 3-10 秒之间（具体值在源码里）。

## evictAfter：精确控制退场时间

`evictAfter` 是一个时间戳，表示"在这个时刻之后才可以驱逐这个任务"。

与 `PANEL_GRACE_MS`（全局宽限期）相比，`evictAfter` 是任务级别的精确控制。

为什么需要任务级别的精确控制？

有些任务完成后需要更长的保留时间：
- 需要用户确认的任务（必须保留直到用户看到结果并做出响应）
- 有重要副作用的任务（比如修改了配置文件，用户需要时间查看变更）

有些任务完成后可以更快清理：
- 后台的周期性检查任务（不需要告知用户，清理越快越好）
- 临时的一次性工具调用（结果已经注入会话，对象本身可以释放）

## 驱逐逻辑的协调问题

在一个有多个任务的系统里，驱逐逻辑需要协调：

不能所有任务都在同一时刻驱逐——如果在一帧里删除 10 个任务，React 会触发大量重渲染，UI 可能卡顿。

应该以任务为单位，每个任务在自己的 `evictAfter` 时间点被单独处理。

还需要处理：一个任务被驱逐时，如果用户恰好在展开它的详情，应该先关闭详情再驱逐，还是强制驱逐让 UI 处理悬空状态？

Claude Code 用 `PANEL_GRACE_MS` 给这个协调留了充足的时间：在宽限期内，UI 有机会完成它自己的动画和过渡，然后任务才真正被清除。

## orphan-cleanup 路径

`isTerminalTaskStatus` 的注释里提到了"orphan-cleanup paths"——孤儿清理路径。

孤儿任务是那些处于终态（completed/failed/killed），但没有被正常驱逐的任务。可能是因为：
- 驱逐逻辑出了 bug
- 系统突然重启，清理逻辑没有来得及运行
- 某些边缘情况下 `evictAfter` 没有被正确设置

孤儿清理路径是兜底机制：定期扫描 AppState，找到处于终态且已经超过 `evictAfter` 的任务，强制清除。

这类"定期扫描+强制清理"的机制在长期运行的系统里非常重要：它保证即使正常清理路径失效，系统也不会慢慢积累越来越多的垃圾状态。

## 面试要点

1. **retain 保证 UI 有时间展示完成状态**：任务完成不等于 UI 更新完成，需要给面板时间展示通知。
2. **evictAfter 是任务级别的精确控制**：比全局宽限期更灵活，可以根据任务类型设置不同的保留时长。
3. **驱逐是有协调成本的操作**：批量驱逐会导致 UI 卡顿，PANEL_GRACE_MS 给协调留出时间。
4. **孤儿清理是必要的兜底机制**：正常清理路径可能失效，定期扫描确保系统不积累垃圾状态。
