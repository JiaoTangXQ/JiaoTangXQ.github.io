# 面试题：LocalAgentTask 为什么要把阶段摘要和实时进度分开记账？

## 从源码里的两种字段开始

在 `src/tasks/LocalAgentTask/LocalAgentTask.tsx` 里，AgentProgress 的结构是：

```typescript
export type AgentProgress = {
  toolUseCount: number      // 累计工具调用次数
  tokenCount: number        // 累计 token 消耗
  lastActivity?: ToolActivity   // 最后一次工具活动
  recentActivities?: ToolActivity[]  // 最近几次活动
  summary?: string          // 阶段摘要
}
```

`summary` 字段是可选的，而且和其他字段的更新节奏不同：`toolUseCount`、`tokenCount`、`lastActivity` 在每次 AI 消息处理后更新（由 `updateProgressFromMessage` 驱动），而 `summary` 是由外部显式设置的，不是实时计算的。

**面试题：这两类信息为什么不能合并成一个字段？或者，为什么不用一个字符串描述就足够了？**

## 两类信息服务不同的读者需求

实时进度（`toolUseCount`、`tokenCount`、`lastActivity`）回答的问题是：**现在正在发生什么？**

用户盯着一个长时间运行的任务，需要知道 AI 还活着、还在工作。"已使用 47 个工具调用，最后一次操作：读取 src/main.ts"这样的信息，让用户确认任务没有卡住。

阶段摘要（`summary`）回答的问题是：**到目前为止做了什么，完成了哪些关键步骤？**

当用户切回来查看一个后台任务的进度时，他们不想看到原始的工具调用列表，而是想看到"已分析问题，发现了三个潜在的根因，目前在验证第二个假设"这样的阶段性总结。

这两类信息的时间粒度不同：实时进度是秒级更新的，阶段摘要是分钟级甚至更长时间才更新一次的。它们的格式不同：实时进度是结构化数据（数字、工具名），阶段摘要是自然语言描述。它们的来源不同：实时进度是从 AI 响应里解析的，阶段摘要是 AI 主动生成的总结。

## updateProgressFromMessage 的设计

```typescript
export function updateProgressFromMessage(
  tracker: ProgressTracker,
  message: Message,
  resolveActivityDescription?: ActivityDescriptionResolver,
  tools?: Tools
): void {
  if (message.type !== 'assistant') {
    return  // 只处理 AI 消息
  }
  const usage = message.message.usage
  // input_tokens 在 API 里是累计值，所以取最新值覆盖
  tracker.latestInputTokens = usage.input_tokens + ...
  // output_tokens 是每轮的，所以累加
  tracker.cumulativeOutputTokens += usage.output_tokens
  
  for (const content of message.message.content) {
    if (content.type === 'tool_use') {
      tracker.toolUseCount++
      // 记录工具活动，最多保留 MAX_RECENT_ACTIVITIES 条
      tracker.recentActivities.push({ ... })
    }
  }
  // 超出限制的旧活动被丢弃
  while (tracker.recentActivities.length > MAX_RECENT_ACTIVITIES) {
    tracker.recentActivities.shift()
  }
}
```

这个函数有几个值得注意的设计：

**input_tokens 取最新值而不是累加**：因为 Claude API 里 `input_tokens` 是当前轮次的输入（包含所有历史），每轮都在增长，如果累加会双重计算。这是一个容易踩的坑，注释里明确说明了原因。

**只保留最近 5 条活动记录**：不是无限追加。这样做的原因是内存控制——一个长时间运行的任务可能有几百次工具调用，全部保存在内存里没有意义，用户界面也不会展示那么多。

**Synthetic output tool 被排除**：`SYNTHETIC_OUTPUT_TOOL_NAME` 对应的工具调用不计入活动记录。这是内部工具，不应该展示给用户。

## 摘要的更新时机

`summary` 不由 `updateProgressFromMessage` 更新，它有自己的更新路径。

根据 `getSdkAgentProgressSummariesEnabled()` 的存在，可以推断摘要功能有一个特性开关。当这个开关打开时，AI 在某些节点（比如完成一个阶段、使用特定工具后）会生成摘要，通过单独的路径（可能是 `SUMMARY_TAG` 对应的 XML 标签）写入任务状态。

这个设计的深层逻辑是：**实时进度是"被动观察"，摘要是"主动汇报"**。被动观察的逻辑由框架驱动（每次消息处理后调用），主动汇报的逻辑由 AI 自己控制（决定什么时候汇报）。把这两种逻辑的更新路径分开，是让两种信息各自保持清晰的关键。

## 当两者混在一起会怎样

想象一个把两者合并的设计：每次工具调用后更新一个统一的描述字符串，每次 AI 写摘要时覆盖这个字符串。

结果：用户查看任务状态时，看到的可能是"读取 src/utils/helper.ts"这样的低级操作描述，而不是任务的阶段进展。或者，摘要被最新的工具活动覆盖了，用户无法看到之前积累的阶段总结。

分开记账，两种信息可以独立更新，独立展示，各自服务它们的读者。这是信息架构而不是技术问题。

## 面试要点

1. **两类信息服务不同的需求**：实时进度回答"现在发生什么"，阶段摘要回答"到目前做了什么"，合并会让两者都展示不好。
2. **input_tokens 不能累加**：因为 Claude API 的计量规则，input_tokens 是累计值，每轮取最新值覆盖，output_tokens 才是累加。
3. **recentActivities 有长度限制**：不是无限追加，内存控制是有意的设计，不是疏忽。
4. **被动观察 vs 主动汇报**：框架驱动实时进度更新，AI 主动控制摘要更新时机，分开路径是让两种逻辑各自清晰的方式。
