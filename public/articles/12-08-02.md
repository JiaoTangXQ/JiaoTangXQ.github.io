# 连得更远以后，系统仍然像一个整体——这才是真正的优雅

「连得更远」本身不难。给模型加一个 computer use 工具，写几个 AppleScript 命令，就能控制桌面。给模型加一个浏览器扩展 API，就能读取页面内容。技术上，「连接」本身是最容易的部分。

难的是：连上之后，系统还是同一个系统吗？

## 碎片化是怎么发生的

每次为新能力开一条特权旁路，就在系统里加了一个「这里不一样」的地方。

- 这里不需要权限确认（因为是浏览器能力，我们信任扩展）
- 这里用自己的日志（因为 MCP 库有自己的 logger）
- 这里的遥测字段不走白名单（因为调试时需要更多信息）
- 这里的配对是临时的（因为每次启动都重新握手就好了）

每一个「这里不一样」单独看都有理由，合在一起就是一个分裂的系统。出问题时，工程师不知道该在哪里找线索；新人加入，不知道有多少个「不一样」需要记住。

## 整体性的具体体现

Claude Code 在每个接缝上都在对抗碎片化：

**配对写回持久身份**：浏览器不是临时外设，配对结果存进配置，下次启动认识同一个设备。

**断线文案说真实原因**：`onToolCallDisconnected()` 告诉用户检查账号、检查扩展、试试重启 Chrome，而不是一句「连接失败」。

**遥测只放行白名单**：bridge 层的分析字段受到和主系统同样的隐私约束，不因为「这里是新能力」就放宽。

**桥上推理翻回文本**：模型在浏览器侧产生的输出，最终过滤成宿主能执行的文本命令，不把内部 block 类型扩散到更多层。

这些细节不让人惊叹，但它们在一起，是整体性的真正含义。
