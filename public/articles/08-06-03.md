# 安全边界分析：本地不认识的远端工具，审批制度能否继续成立？

## 问题的来源

Claude Code 支持 MCP（Model Context Protocol）工具扩展。远端 CCR 容器可能安装了本地没有的 MCP 服务器，因此有本地不认识的工具。

当远端发来一个 `control_request`（权限请求），请求执行一个叫 `my-company/deploy` 的工具，而本地根本没有加载这个 MCP 服务器，本地的工具注册表里没有 `my-company/deploy`——怎么办？

三种选择：

1. **自动放行**：本地不认识，默认允许
2. **自动拒绝**：本地不认识，默认拒绝
3. **最小化 stub + 人工审批**：为本地不认识的工具创建一个存根（stub），保持审批流正常运行

## 自动放行的安全漏洞

自动放行看起来"用户友好"——用户不会看到陌生的报错，任务继续运行。

但它打开了一个安全漏洞：**"工具是否需要审批"变成了"本地是否加载了这个工具"的函数**。

设想一个场景：攻击者在服务端配置了一个名为 `safe-reader` 的 MCP 工具，但这个工具实际上会往服务器发送数据。由于本地没有加载这个工具，自动放行让它在没有用户意识到的情况下执行了危险操作。

这不是合理的安全设计。**权限审批的边界应该是"工具是否危险"，不应该是"工具是否被本地认识"**。

## 自动拒绝的可用性问题

自动拒绝看起来"安全"——不认识的工具一律不允许。

但它的问题是可用性折断：用户在远端安装了一个新的 MCP 工具，在本地 claude 还没来得及更新的情况下，所有使用这个工具的任务都会失败，而且失败信息非常不友好（"权限被拒绝"，原因是"工具未知"）。

更重要的是：`createToolStub` 路由到的是 `FallbackPermissionRequest`——这是设计中明确存在的"兜底审批 UI"，专门处理无法精细显示的情况。自动拒绝跳过了这个设计，直接将"工具未知"等同于"工具危险"，混淆了两个不同的概念。

## createToolStub 的设计

```typescript
export function createToolStub(toolName: string): Tool {
  return {
    name: toolName,
    isEnabled: () => true,
    userFacingName: () => toolName,
    renderToolUseMessage: (input: Record<string, unknown>) => {
      // 取前三个参数显示，让用户有最低限度的信息
      return Object.entries(input).slice(0, 3)
        .map(([k, v]) => `${k}: ${v}`).join(', ')
    },
    needsPermissions: () => true,  // 总是需要权限 — 未知工具默认需要审批
    call: async () => ({ data: '' }),
    // ...
  }
}
```

关键字段：`needsPermissions: () => true`。

stub 被设计成**总是需要审批**，不管工具实际上是什么。这个保守设计保证了：哪怕本地不认识这个工具，用户也会被询问是否允许执行。

`renderToolUseMessage` 展示前三个参数——这是在"完全不显示"（用户没有任何信息）和"完整显示"（本地有完整工具定义才能实现）之间的折中。三个参数通常足够让用户判断"这个操作在做什么"。

## 与审批链的接口

stub 被传入 `createSyntheticAssistantMessage`，然后进入本地的 `ToolUseConfirm` 流程：

```
createToolStub("my-company/deploy")
  ↓
createSyntheticAssistantMessage(request, requestId)
  ↓
ToolUseConfirm（本地审批 UI）
  ↓
用户点击 Allow / Deny
  ↓
sendPermissionResponseEvent（发回服务端）
```

整条链路和本地工具执行时的审批流完全相同。stub 的存在，让 `ToolUseConfirm` 不需要知道"这个工具是不是本地认识的"——它只关心"这是一个 Tool 对象，有 `needsPermissions()`，我需要问用户"。

## 制度边界的维持

这个设计体现了一个重要的安全原则：**权限检查应该是无条件的，而不是有条件的**。

"条件是本地认识这个工具"就是一个有条件的权限检查，它的安全性依赖于"本地工具注册表的完整性"。只要注册表不完整（比如 MCP 工具未加载），安全屏障就有漏洞。

无条件的权限检查：任何工具调用（不管来源、不管是否认识）都走审批流，差别只是审批 UI 的信息丰富程度。

stub 的存在，把"工具认知"和"权限审批"解耦了：

- 不认识 → stub（保证审批信息不为零）
- 认识 → 完整 Tool 对象（审批信息丰富）

但无论哪种，都必须经过审批。

## 面试指导

"如何处理系统边界的未知输入"是一个很考验安全思维的问题。

三种处理策略（放行/拒绝/降级审批）的权衡：

| 策略 | 安全性 | 可用性 | 适用场景 |
|------|--------|--------|----------|
| 自动放行 | 低（有漏洞） | 高 | 不适用于权限相关 |
| 自动拒绝 | 高（但过于保守） | 低 | 严格受控环境 |
| 降级审批 | 高（无条件审批） | 中 | 权限场景的正确选择 |

"降级审批"的关键在于：**把"无法精细展示"和"不需要审批"区分开**。这两件事没有任何关系，但在实现上很容易被混为一谈。
