# 第十三卷回收索引：任务世界里同步子 agent 也先注册成前台任务，说明系统先认对象，再认姿态

## 实现链
任务世界里有些工作一开始就不该被当成“已经退到后台的附属物”。  
同步子 agent 即使最后要切换姿态，也先得作为前台任务被系统认出来。

这条线真正逼出来的，是第十三卷里关于正式工作单元与显示姿态先后顺序的判断。  
对象身份先站稳，后面的前台后台才只是显示姿态。

这页实际回收的实现链是：
- `05-06-01` 把同步子 agent 也先注册成前台任务讲清，这会在 `13-07-01` 里回收到“系统先认对象，再认显示姿态”，说明正式工作单元先成立，后面才谈它怎么被显示。

很多系统会先把工作推去后台，再回头补对象身份。  
Claude Code 更克制，它先把子 agent 当成前台任务认出来，再让姿态随现场变化。

## 普通做法
更普通的做法，是把后台动作当成松散的异步句柄，跑起来就算开始，结束了再临时补状态和通知。

## 为什么不用
Claude Code 不这么做，是因为没有统一任务对象，就没有稳定的生命周期、输出、通知和回看链，前后台切换也会变成一串彼此不认账的临时处理。

## 代价
代价是任务状态字段、输出文件、通知协议和前后台转场逻辑都会变厚，维护者要记住更多接缝。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
