# 架构问题：PromptInput 里的八个对话框，为什么不放在 App 顶层？

打开 `PromptInput.tsx`，渲染函数底部有一组很长的条件渲染：

```tsx
{showHistorySearch && <HistorySearchDialog ... />}
{showModelPicker && <ModelPicker ... />}
{showBackgroundTasks && <BackgroundTasksDialog ... />}
{showTeamsDialog && <TeamsDialog ... />}
{showAutoModeOptIn && <AutoModeOptInDialog ... />}
{showBridgeDialog && <BridgeDialog ... />}
{showGlobalSearch && <GlobalSearchDialog ... />}
{showQuickOpen && <QuickOpenDialog ... />}
```

八个对话框，全部在 `PromptInput` 里。这看起来像膨胀，但背后有一个清晰的原则。

---

## 「谁打开谁管理」原则

这八个对话框有一个共同点：它们都是从输入区的操作发起的。

- 按 `Ctrl+R`（历史搜索快捷键）→ 打开 `HistorySearchDialog`
- 按 `Alt+M`（模型选择器快捷键）→ 打开 `ModelPicker`
- 按 `Ctrl+T`（任务快捷键）→ 打开 `BackgroundTasksDialog`
- 按 `Ctrl+\`（Bridge 快捷键）→ 打开 `BridgeDialog`

这些快捷键都在 `PromptInput` 的键盘事件处理器里监听。`PromptInput` 拥有「打开」这个动作——它知道什么时候该打开对话框，也知道打开时需要传什么参数（比如当前输入内容、当前光标位置）。

如果把对话框提升到 App 顶层，打开逻辑就变成：`PromptInput` 捕获快捷键 → 通知某个顶层状态 → App 层根据状态渲染对话框。这是一个信号绕了一大圈才到达渲染的路径。

更重要的是，这些对话框在使用时需要访问输入区的上下文：

```tsx
<HistorySearchDialog
  onSelect={(text) => {
    // 选中历史记录后要设置输入框内容
    setInput(text)
    setCursorOffset(text.length)
  }}
  currentInput={input}
/>
```

`HistorySearchDialog` 的 `onSelect` 回调需要直接调用 `setInput`——这是 `PromptInput` 内部的状态设置函数，不能轻易暴露到外层。如果对话框在 App 顶层，这个回调就需要通过 props 或 context 层层传递，或者把 `input` 状态也提升到顶层，引发更多耦合。

---

## 生命周期与输入区紧密绑定

从生命周期的角度看，这些对话框只在输入区存在时才有意义：

**HistorySearchDialog**：帮用户从历史记录里找到内容填入输入框。输入框关闭（比如 Agent 正在运行中锁定输入），历史搜索也应该关闭。

**ModelPicker**：让用户切换 AI 模型。这个操作的直接影响是下一次提交用什么模型，和输入区的提交动作强关联。

**PromptSuggestion 相关**：预测性输入建议，逻辑上就是输入框功能的延伸。

把这些对话框管理在 `PromptInput` 里，它们的打开/关闭状态跟随输入区的存在自然地初始化和清理：

```tsx
const [showHistorySearch, setShowHistorySearch] = useState(false)
// 当 PromptInput 卸载时，所有这些 useState 自然清理
```

如果放在 App 顶层，需要手动管理「当 PromptInput 不可用时关闭所有输入区对话框」的逻辑。

---

## 对话框的输入上下文依赖

更实际的问题是数据传递。`ModelPicker` 在打开时需要知道当前选中的模型，关闭时要把选中结果写回：

```tsx
<ModelPicker
  currentModel={mainLoopModel}
  onSelect={(model) => {
    setAppState(prev => ({ ...prev, mainLoopModel: model }))
    setShowModelPicker(false)
  }}
/>
```

`GlobalSearchDialog` 需要在搜索结果选中后触发相机飞行（跳到对应星球/内容）。`BackgroundTasksDialog` 选中任务后需要更新 `viewingAgentTaskId`。

这些不是「纯展示对话框」，而是需要和 `PromptInput` 的本地状态以及全局 AppState 双向交互。`PromptInput` 作为中介，持有本地状态引用，同时通过 hooks 连接全局状态，是这些对话框的天然归属地。

---

## 如果放在 App 顶层，会发生什么

假设把 `HistorySearchDialog` 提升到 `App.tsx`：

**数据流变化：**

```
PromptInput 按键 → 触发 AppState 更新（showHistorySearch: true）
App.tsx 订阅 AppState → 渲染 HistorySearchDialog
HistorySearchDialog 选择结果 → 触发 AppState 更新（historicalInput: '...'）
PromptInput 订阅 AppState → 消费 historicalInput → 清空 historicalInput
```

整个流程从「本地 setState + callback 直接调用」变成了「通过 AppState 绕一圈」，引入了更多的 AppState 字段、更多的订阅、更多的时序问题（比如 HistorySearch 结果写入 AppState 和 PromptInput 读取之间的竞争）。

**更关键的问题：**`PromptInput` 有 `pastedContents`（粘贴内容映射）、`cursorOffset`（光标位置）等本地状态，这些不应该进入 AppState（它们是瞬态的输入状态，不需要跨组件共享）。但历史搜索选中后需要设置这些本地状态，这意味着 `HistorySearchDialog` 需要一个指向 `PromptInput` 内部的回调——无论如何都需要 `PromptInput` 参与。

---

## 「重量」和「就近原则」

React 组件架构里有一个判断拆分的经验法则：**如果两个东西需要频繁传递数据，就让它们靠近**。

对话框和输入区之间有高密度的双向数据传递，把它们放在同一个组件里，是「就近原则」的自然体现，代价是 `PromptInput.tsx` 看起来重。

更好的问题不是「这些对话框该不该在这里」，而是「PromptInput 里哪些东西真正可以抽离」。

可以抽离的是：不需要访问 PromptInput 本地状态的、有独立生命周期的功能。目前的对话框都不满足这两个条件——它们要么需要设置输入框内容（`HistorySearchDialog`），要么需要感知当前输入状态（`ModelPicker`），要么是输入操作的直接延伸（`PromptSuggestion`）。

## 面试指导

**为什么不用 Portal 把对话框渲染到 App 顶层但保持逻辑在 PromptInput**

React Portal 可以把对话框渲染到 DOM 树的指定位置，但组件的 props 和 state 仍然属于原始父组件。这个方案可以解决「样式层级/z-index 问题」，但不能解决「PromptInput 被卸载时如何清理对话框」的问题——因为 Portal 的渲染生命周期仍然跟随父组件。Claude Code 是终端应用，不存在网页里的 z-index 或遮罩层问题，Portal 没有额外价值。

**什么时候对话框应该提升到 App 顶层**

如果一个对话框可以被多个不相关的组件打开（比如全局的「确认删除」对话框），且它的数据来源和处理不依赖任何特定组件的本地状态，提升是合理的。Claude Code 里没有这种对话框——每个对话框都有特定的发起者（通常是 PromptInput 的快捷键）和特定的上下文要求。

**Dialog 状态用 useState 而不是 AppState 管理，意味着什么**

`showHistorySearch` 这类 boolean 不进入 AppState，是因为它们是瞬态 UI 状态，不需要跨组件共享，不需要在状态变化时触发其他地方的副作用，也不需要序列化或持久化。用 `useState` 管理它们更轻量，不污染全局状态树，PromptInput 卸载时自然清理。
