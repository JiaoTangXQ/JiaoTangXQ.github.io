# MCP Server 进来的不只是工具：新世界如何先过原制度

面试题版本：**"用户配置一个新的 MCP server 后，Claude Code 会做哪些检查和设置，然后才让这个 server 的工具可用？这个流程说明了什么设计思路？"**

## MCP server 进来时系统做了什么

在 `src/services/mcp/` 里，`MCPConnectionManager` 管理 MCP server 的生命周期。一个新的 MCP server 被配置后，系统并不是直接把它的工具列表加入到工具表里。它经历了：

**第一步：建立连接**
选择合适的 transport（stdio、sse、http、in-process），建立连接，处理连接失败和重试逻辑。

**第二步：能力协商**
调用 `listTools()`、`listResources()` 等，了解这个 server 提供什么能力。

**第三步：规范化**
把 server 报告的工具描述规范化成内部 Tool 格式，处理命名冲突（用前缀区分不同 server 的同名工具），设置默认的安全属性。

**第四步：权限检查点**
MCP server 需要用户批准才能使用（`src/services/mcpServerApproval.tsx`）。第一次连接时，系统会显示这个 server 的信息、它请求的能力，让用户决定是否信任。

**第五步：注册到工具表**
只有通过了权限批准，规范化后的工具才被加入工具表，变成可调用的状态。

## 进来的不只是工具

为什么说"进来的不只是工具"？

**边界影响**：一个 MCP server 可能提供文件系统访问、网络访问、数据库访问等能力。这些能力会改变 Claude 的行动边界——原来的"代码编辑助手"变成了"可以操作数据库的 AI"。这个边界变化需要用户知情并同意。

**信任问题**：MCP server 的代码由第三方提供，其行为不受 Anthropic 控制。如果工具直接进入工具表，用户没有机会评估信任关系。

**命名空间**：MCP server 引入的工具名可能和已有工具冲突，需要命名空间隔离，否则模型在调用工具时可能搞混。

**任务托管**：如果这个 server 的工具需要异步执行或长时间运行，需要和任务系统集成，而不是直接同步调用。

## "先过原制度"的含义

"原制度"是 Claude Code 已有的：
- 工具权限检查（`useCanUseTool`）
- 输入校验（Zod schema）
- 并发安全约束（`isConcurrencySafe`）
- 工具执行沙箱和超时控制

MCP server 的工具接入后，需要服从这套制度，而不是绕过它。这意味着：

- MCP 工具在执行前会经过和内建工具相同的权限检查
- MCP 工具的输入会经过 schema 验证
- 被标记为不并发安全的 MCP 工具不会和其他工具并发执行

这些不是 MCP 特有的规则，而是所有工具必须遵守的制度。新来的 MCP server 先把它的工具纳入这个制度，才能开始工作。

## 为什么不让 MCP server 自己管理权限

一些扩展系统的设计是"插件自管"：插件有自己的权限声明格式，自己的执行沙箱，自己的安全策略。这样可以让每个插件做精细控制，也减少了核心系统的复杂度。

问题是：当系统有多个 MCP server，每个有自己的权限模型，用户要学会理解多套不同的权限概念，工具调用的安全性变成了"哪个 server 的安全策略最弱"决定的。

Claude Code 选择统一权限制度的代价：每个新 MCP server 都要适配标准权限模型，不能自定义。但收益是：用户只需要理解一套权限逻辑，系统的安全边界是整体可推理的。

## 面试角度的总结

MCP server 的接入流程——连接、协商、规范化、权限批准、注册——说明了 Claude Code 的扩展哲学：**新能力在展开之前先过原制度的每一道门**。这不是为了阻碍扩展，而是为了让扩展后的系统仍然是一个整体：有统一的权限模型、统一的工具语义、统一的执行保障。每跳过一道门，系统就多一个无法整体推理的例外。
