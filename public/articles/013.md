# Claude Code 怎么把工具调用、权限对话、任务进度显示在同一个消息流里？

打开 Claude Code，你看到的消息流不只是"人说一句、AI 说一句"的对话。工具执行进度、权限弹窗、后台任务通知、消息压缩边界、teammate 状态切换——这些内容都要塞进同一个 UI 列表里，还要保持视觉连贯性。

要实现这一点，消息类型系统必须足够丰富，同时渲染层必须知道如何处理每种类型。

## 消息类型体系

Claude Code 的消息系统不只是 `{ role: 'user' | 'assistant', content: string }`。从 `types/message.ts` 可以看到完整的消息联合类型，大概可以分成几类：

**对话消息**：`UserMessage`、`AssistantMessage` — 人机交互的基础单元

**工具消息**：
- `ToolUseBlock`（嵌在 AssistantMessage 里）— 模型决定调用某个工具
- `ToolResultBlockParam`（嵌在 UserMessage 里）— 工具执行结果回传
- `ProgressMessage<ToolProgressData>` — 工具运行期间的进度推送

**系统消息**：`SystemMessage`，包括多种子类型：
- `SystemLocalCommandMessage` — 本地命令输出
- 其他系统级通知

**辅助消息**：
- `AttachmentMessage` — 注入内存文件（CLAUDE.md）的附件
- `TombstoneMessage` — 被删除/清除的消息占位符
- `CompactBoundaryMessage` — 消息压缩的分界线
- `ToolUseSummaryMessage` — 工具使用的摘要视图

## ProgressMessage 是怎么工作的

工具进度是最复杂的部分。当 BashTool 执行一个需要几秒钟的命令时，需要实时推送进度给 UI：

```typescript
export type ToolProgressData =
  | BashProgress      // { type: 'bash', output: string, ... }
  | AgentToolProgress // { type: 'agent', status: ..., ... }
  | MCPProgress       // { type: 'mcp', ... }
  | REPLToolProgress
  | SkillToolProgress
  | TaskOutputProgress
  | WebSearchProgress

export type ProgressMessage<P extends ToolProgressData = ToolProgressData> = {
  toolUseID: string  // 关联到哪个 tool_use 调用
  data: P
}
```

`ProgressMessage` 通过 `toolUseID` 关联到具体的工具调用。渲染时，`filterToolProgressMessages` 把某个工具调用的所有进度消息聚合在一起，传给该工具的 `renderToolUseProgressMessage` 方法。

这意味着每个工具可以完全自定义进度 UI——BashTool 显示命令输出，AgentTool 显示子 Agent 的工作状态，WebSearchTool 显示搜索结果摘要，互不干扰。

## 权限消息不是普通消息

权限请求（"Claude 要执行这个 bash 命令，是否允许？"）在 UI 上看起来像是插入在消息流中的对话框，但它实际上**不是** `Message` 类型的实例。

它来自 `ToolPermissionContext` 和权限队列系统，通过 `context/notifications.js` 的通知机制出现在界面上。这就是为什么权限对话框可以在工具执行暂停期间出现，又可以在用户操作后精确地关闭并继续执行——它的生命周期由权限系统管理，不是消息队列管理。

## compact 边界的作用

当对话消息太长导致 context window 快满时，Claude Code 会触发 compact（压缩）流程，把历史消息浓缩成摘要。`CompactBoundaryMessage` 是压缩前后的分界线，`getMessagesAfterCompactBoundary` 函数只返回最新 compact 边界之后的消息：

```typescript
// query.ts 里的用法
export function getMessagesAfterCompactBoundary(messages: Message[]): Message[] {
  // 找到最后一个 compact 边界，只把之后的消息发给 API
  const lastBoundary = findLast(messages, m => m.type === 'compact_boundary')
  if (!lastBoundary) return messages
  return messages.slice(messages.indexOf(lastBoundary) + 1)
}
```

这个 `CompactBoundaryMessage` 同时也是 UI 渲染的分界点——界面上会显示一条"--- 会话已压缩 ---"的提示线，让用户知道这里有一次历史浓缩。

## 面试角度

"如何设计一个复杂 UI 里的消息/事件流"是前端系统设计的经典题型。Claude Code 的方案体现了几个值得借鉴的思路：

1. **消息类型是完整的业务语义，不是格式描述**：`TombstoneMessage`、`CompactBoundaryMessage` 都是有实际语义的类型，不是"元数据"
2. **进度更新通过关联 ID 而不是嵌套结构组织**：`ProgressMessage.toolUseID` 让进度和工具调用解耦，互不污染消息历史
3. **渲染权重分离**：不同类型的消息可以对应完全不同的渲染逻辑，而不需要一个巨型 if-else 判断
4. **系统消息在 API 边界被剥离**：`SystemLocalCommandMessage` 在 `normalizeMessagesForAPI` 时会被过滤掉，它只服务于界面展示，不送给模型
