# 第十三卷回收索引：外延执行里 in-process server 表面仍像 stdio，接口层先替实现复杂度挡风

## 实现链
外延执行里有一条更偏接口层骨架的线，不在“多做了什么”，而在“实现复杂度先被挡在了哪一层后面”。

这条线真正逼出来的，是第十三卷里关于接口层克制空间的判断。  
成熟接口不会急着把真实实现的临时形状直接端出来。

这页实际回收的实现链是：
- `12-04-01` 把 `in-process server` 写成表面像 `stdio server`、内部却不必每次真起子进程，这会在 `13-03-01` 里回收到“接口层先替真实实现留出克制空间”，说明外观先稳住，复杂度先别外溢。

接口层最见功力的时候，不是它把所有内部细节都摊开。  
而是它先让表面保持熟悉，再替真实实现留出克制空间。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
