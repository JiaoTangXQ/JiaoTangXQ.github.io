# ResolveOnce.claim() 为什么需要原子性？权限决定的竞态问题

`PermissionContext.ts` 里有一个值得细读的数据结构：`createResolveOnce`。它解决的问题是：多个异步路径（hook、分类器、用户输入）可能同时产生权限决定，但最终只能有一个决定生效。

## 问题的来源

权限决定的产生路径至少有三条，它们可能**并发运行**：

1. **Hook 检查**：`executePermissionRequestHooks` 是异步的，可能需要运行外部脚本
2. **分类器检查**：`awaitClassifierAutoApproval` 调用远程 API，有网络延迟
3. **用户输入**：弹窗等待用户点击，时间不确定

这三条路径都可能返回一个 `PermissionDecision`。如果多条路径同时返回，而且都调用了同一个 Promise 的 resolve，会发生什么？

```typescript
// 假设有一个简单版本
let resolved = false
function tryResolve(decision: PermissionDecision) {
  if (!resolved) {
    resolved = true
    resolve(decision)  // 多条路径可能同时到达这里
  }
}
```

在 JavaScript 里，事件循环是单线程的，不存在真正的并发写同一个变量的问题。但是在 `await` 之后，代码会让出控制权，让其他回调执行。所以：

```
路径A: if (!resolved)  → 条件为 true
路径A: await 某个操作  → 让出控制权
路径B: if (!resolved)  → 条件仍为 true（还没被改）
路径B: resolved = true → 现在改了
路径B: resolve(B决定)  → B的决定生效
路径A: resolved = true → 再改一次（已无意义）
路径A: resolve(A决定)  → 尝试再次 resolve，但 Promise 已经 resolved
```

实际上 Promise resolve 是幂等的——第二次 resolve 会被忽略。所以上面的简单版本不会产生”两个决定都生效”的问题，但会产生”哪个决定赢是不确定的”问题：取决于哪条路径先从 await 恢复。

## createResolveOnce 的实现

```typescript
type ResolveOnce<T> = {
  resolve(value: T): void
  isResolved(): boolean
  /**
   * 原子性地检查并标记为已解决。
   * 返回 true 说明此调用者赢得了竞争（没有其他人先解决）
   * 在 await 之前调用，关闭 isResolved() 检查和 resolve() 调用之间的窗口
   */
  claim(): boolean
}

function createResolveOnce<T>(resolve: (value: T) => void): ResolveOnce<T> {
  let claimed = false
  let delivered = false
  return {
    resolve(value: T) {
      if (delivered) return
      delivered = true
      claimed = true
      resolve(value)
    },
    isResolved() {
      return claimed
    },
    claim() {
      if (claimed) return false
      claimed = true
      return true
    },
  }
}
```

关键在于 `claim()` 方法。它在 await 之前调用，把”是我先来的”这个事实记录下来。即使之后在 await 里让出了控制权，其他路径来检查时也会发现 `claimed = true`，知道已经有人赢了。

## 使用模式

```typescript
// 在 hook 处理里的典型用法
async function handleWithClaim(resolveOnce: ResolveOnce<PermissionDecision>) {
  // 1. 先检查是否已经有人赢了
  if (resolveOnce.isResolved()) return

  // 2. 做一些异步操作
  const hookResult = await runSomeHook()

  // 3. 在 await 之后，不能再用 isResolved()，必须用 claim()
  // 因为在 await 等待期间，可能有其他路径已经 claim 了
  if (!resolveOnce.claim()) return  // 我们输了竞争

  // 4. 只有 claim() 返回 true 的调用者才能 resolve
  resolveOnce.resolve(makeDecision(hookResult))
}
```

注释里解释了为什么需要 `claim()` 而不能直接用两次 `isResolved()` 检查：

> “Use this in async callbacks BEFORE awaiting, to close the window between the `isResolved()` check and the actual `resolve()` call.”

但实际上 `claim()` 通常在 await 之后使用，因为 await 之前还没有竞争窗口。真正的用途是：**在 await 之后，用 claim() 代替 isResolved() + resolve() 的组合，防止在 claim 和 resolve 之间再次让出控制权**。

## 两个标志位：claimed 和 delivered

有意思的是，`createResolveOnce` 维护了两个独立的标志：

- `claimed`：有没有人宣告”我来处理这个决定”
- `delivered`：决定有没有真正被传递出去

`claim()` 只改 `claimed`。`resolve()` 改两者。

这个设计的作用是：`claim()` 成功后，如果由于某种原因 `resolve()` 没被调用（比如代码提前 return 了），`claimed = true` 仍然防止其他路径重新竞争。这避免了一类 bug：某个路径 claim 了但没有 resolve，导致 Promise 永远挂起。

## 这道题考什么

这道题的表面是并发控制，实质是对 JavaScript 异步模型的理解。

**关键洞察**：JavaScript 不是真正并发的，但 await 会让出控制权，允许其他回调在这期间执行。在 await 之后，不能假设 await 之前的状态仍然有效——需要重新检查，或者用 claim() 这样的原子操作在 await 之前就”锁定”状态。

**设计意图**：`createResolveOnce` 的存在说明权限决定的异步竞争是一个真实场景，而不是理论问题。hook、分类器、用户 UI 三条路径确实会并发运行，都有可能产生决定，需要一个明确的机制来保证”只有一个决定生效”。
