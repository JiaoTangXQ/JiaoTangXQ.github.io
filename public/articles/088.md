# 第十三卷回收索引：扩展世界里的统一语言与装配骨架

## 实现链

这页具体回收的判断是“扩展世界—统一语言与装配骨架”。
这页背后真正回收的是 `skills/loadSkillsDir.ts`、plugin manifest 处理、MCP 连接管理、命令和工具装配这条实现链。Claude Code 让 skills、plugins、MCP 最终尽量回到同一组命令、工具、资源语言里，再挂回同一条主运行时装配线，所以它真正守的是统一语言和统一装配骨架，而不是“扩展很多”。

## 普通做法

更普通的扩展系统会给不同来源各做一套入口：提示词型扩展是一套、插件是一套、远端连接又是一套，各自维护自己的生命周期和调用方式。这样短期最省事，因为每种扩展都按自己的天然形状接进来。

## 为什么不用

Claude Code 没沿这个方向走，是因为扩展一旦多起来，最先崩的不是能力本身，而是主运行时的制度。要是不同来源各自带一套装配方式，权限、调度、日志、恢复和发现都会分裂。它宁愿多写桥接层，也要把新来源翻译回旧骨架能听懂的语言。

## 代价

这样做的好处是长期整洁，主运行时始终只需维护一套核心制度。代价则是桥接层很多，第一次读代码时很容易觉得它“为什么不直接接进来”，而要接受多一层翻译和装配的复杂度。

## 继续下钻

- 再多来源也先汇回同一种运行时语言
- 新旧来源重新收回同一条装配流水线
