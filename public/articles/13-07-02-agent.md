Claude Code 第二个最值得学走的地方，是它几乎所有大决定都在假设“这不是一次性问答，而是一段会继续下去的工作关系”。`QueryEngine` 不是一轮 prompt 的薄包装，而是会话对象；`query.ts` 不是只在 happy path 上跑一遍，而是一直考虑中断、工具结案、重试和下一轮继续；`compact.ts` 的目标也不是把旧上下文清空，而是把会话压薄后还能继续干活；`AppStateStore` 里那些 remote、bridge、task、notification、companion 状态也都在说明：系统默认会继续活着，不是吐完一段字就散场。

更容易想到的产品方式，是把 agent 做成一次性魔术：你说一句，它回一段；上下文太长了就重新开始；失败了就让用户再试一次。这种东西很容易炫，但很难承担真正的工作。

Claude Code 值得学的不是具体哪一个模块，而是这种“默认长期合作”的设计姿态。它当然更重、更复杂，也因此不适合所有产品。但如果你的目标是工作台，而不是演示台，那就应该尽早在代码里把这种长期性做出来，而不是等问题堆出来以后再补。

## 实现链
会话壳、主循环、compact、状态托管和恢复链都在告诉你：它被做成了可长期工作的工作台。

## 普通做法
只优化一次演示里最顺的那条路。

## 为什么不用
因为真正有价值的是长期可继续、可恢复、可治理，而不是一次惊艳。

## 代价
做成工作台意味着系统更重，也更难写出短平快版本。
