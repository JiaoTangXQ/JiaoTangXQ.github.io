# 把 agent 做成长久工作台，比做成一次性魔术更难在哪里？

## 工程反思：演示台和工作台的区别

在大多数 agent 产品的 demo 里，有一个隐含假设：用户说一句话，AI 给出惊人的答案，然后这个对话就结束了。

这是**演示台**的设计假设。

Claude Code 的设计假设不一样：用户会继续说话，会话会持续进行，工具会被多次调用，上下文会越来越长，有些操作会失败，用户会改变想法，系统需要从中断中恢复。

这是**工作台**的设计假设。两者之间的差距，具体体现在代码的每一个大决定里。

## 六个"默认长期"的设计决定

**决定一：QueryEngine 是会话对象，不是请求处理器**

每次请求完成后丢弃上下文，是演示台设计。`QueryEngine` 持续维护消息历史、文件快照和拒绝记录，是工作台设计。用户能说"把上次改的那个函数还原"，因为"上次"就在那里。

**决定二：query.ts 默认会被中断**

演示台的主循环只在 happy path 上跑一遍。`query.ts` 里的主循环为流式中断、缺失 tool_result、工具超时和重试都有专门的处理路径。不是因为这些情况常见，而是因为工作台里这些情况迟早会发生。

**决定三：compact 目标是"压薄后能继续干活"**

演示台的上下文管理是清空重来。`compact.ts` 的目标是在压缩后保持会话的连续性——关键线索接回新边界，技能附件重新注入，不是从零开始，而是从"一份更薄的过去"继续。

**决定四：AppStateStore 假设系统会继续活着**

`AppStateStore` 里有 remote 连接状态、bridge 状态、任务列表、通知队列、伴侣状态——这些字段的存在本身就是在说：系统默认会继续活着，不是吐完一段字就散场。

**决定五：LocalAgentTask 把子任务做成正式对象**

演示台里 subagent 结束了就结束了，结果丢到日志里。`LocalAgentTask` 有 `outputFile`、`diskLoaded`、`evictAfter`——子任务结束后，输出文件仍然可以回看，回收有明确的触发点，不靠 UI 碰运气。

**决定六：权限制度先行，假设会话会变长**

演示台的权限管理是每次弹窗确认一下。`permissionSetup.ts` 在会话开始时建立整场的权限框架，因为工作台里用户不可能每个操作都手动确认——需要一套能长期工作的信任结构。

## 从演示台到工作台的三个成本

把 agent 做成工作台，比做成演示台要付三个额外成本：

**成本一：状态更重**。工作台需要记住更多东西，`QueryEngine`、`AppStateStore`、`LocalAgentTask` 都是这个选择的产物。状态越多，内存占用越高，一致性越难保证。

**成本二：失败路径更多**。工作台假设自己会遇到中断、超时、网络错误、工具失败。演示台可以假设 happy path。处理失败路径需要更多代码，也更难测试。

**成本三：用户期望更高**。当你把 agent 做成工作台，用户会自然地期望它能记住上次的对话，能从失败中恢复，能处理长时间的任务。演示台可以规避这些期望，工作台不行。

## 这不适合所有产品

工作台设计意味着系统更重，也更难写出短平快版本。

如果目标是一个单轮问答工具，演示台设计完全合理。不应该为了"看起来像 Claude Code"就把所有状态都做成持久化的。

工作台设计在以下场景值得投资：
- 会话需要持续多轮，而且上下文很重要
- 用户的工作流需要从中断中恢复
- 系统会处理长时间运行的任务
- 工具调用结果需要被后续步骤引用

如果这些都不是你的场景，工作台的复杂度就是不必要的成本。

## 面试指导

被问"你会怎么设计一个 agent"时，先问自己：这是演示台还是工作台？

然后根据答案决定要不要投资状态持久化、恢复路径、长会话管理和权限制度。

能在面试里清晰说出"这个设计假设用户只会用一次"和"这个设计假设用户会长期使用"之间的区别，并且说清楚各自的成本，是工程成熟度的重要标志。
