# 一旦跨边界，消息权限和界面都得重算

## 实现链
- `useRemoteSession.ts` 会把远端 SDK 消息翻译成本地 REPL 能显示的消息，权限请求则走 `remotePermissionBridge.ts` 合成回本地审批对象；`useAssistantHistory.ts` 还会在 viewerOnly 场景下单独补拉历史并保持滚动姿态。
- 也就是说，跨边界后不仅执行位置变了，消息怎么显示、权限在哪裁决、界面要不要分页历史，都要重新判断。

## 普通做法
- 普通远端终端做法通常只传 stdin/stdout，界面逻辑几乎原样沿用，审批也让远端自己处理。
- 这样实现最省，因为 UI 不用太懂远端世界。

## 为什么不用
- Claude Code 没这么偷懒，是因为它的本地 UI 不只是显示文本，还承担审批、任务计数、工具进度和阅读姿态。远端如果不翻译回本地制度，本地界面就会开始说假话。
- 从代码结构看，它优先保住的是本地工作台的一致性，而不是最短通信链。

## 代价
- 代价是 remote 相关代码会渗到 hook、状态和界面层，条件分支比单机模式多很多。
- 但这就是“边界被改写”的真实代价，不是作者平白把系统写复杂。
