# 跨边界后，消息、权限、界面为什么都要重新翻译？

## 分布式系统视角：边界两侧的语言不同

在分布式系统设计中，有一个经常被忽视的问题：**两个服务之间的通信协议，和各自内部的数据模型，往往不是同一种语言**。

Claude Code 的远端架构里，这个问题体现得很具体：

- 远端 CCR 容器发出的是 `SDKMessage`（`src/entrypoints/agentSdkTypes.ts` 里定义的类型）
- 本地 REPL 界面消费的是 `Message`（`src/types/message.ts` 里定义的类型）

这两个类型虽然有交叉，但结构不同。如果不做翻译层，本地界面直接消费 SDKMessage，会出现两种问题：要么界面崩溃，要么界面显示"假数据"（缺少本地字段的默认值）。

## 权限审批的跨边界问题

更复杂的是权限请求（`control_request`）的处理。

在本地模式下，权限请求很简单：Claude 决定执行某个有风险的工具 → 本地 UI 弹出确认框 → 用户点击 → 工具执行。工具和审批在同一进程里，直接用 callback 连接。

远端模式下，执行链被物理分割：

```
工具执行: 在 CCR 容器里（远端）
权限请求发出: 从 CCR 容器经 WebSocket 到本地
用户审批: 在本地 UI 里
审批结果发回: 从本地经 HTTP POST 到 /v1/sessions/{id}/events
工具继续执行: 在 CCR 容器里（远端）
```

这意味着 `remotePermissionBridge.ts` 需要做一件非常特殊的事：**为一个本地不存在的工具调用，构造一个合成的 `AssistantMessage`**。

```typescript
export function createSyntheticAssistantMessage(
  request: SDKControlPermissionRequest,
  requestId: string,
): AssistantMessage {
  return {
    type: 'assistant',
    uuid: randomUUID(),
    message: {
      id: `remote-${requestId}`,
      content: [{
        type: 'tool_use',
        id: request.tool_use_id,
        name: request.tool_name,
        input: request.input,
      }],
      // ... 其他字段填默认值
    },
  }
}
```

本地的审批 UI（`ToolUseConfirm`）要求一个真实的 `AssistantMessage`，但这个消息从来没有在本地产生过——工具调用发生在远端。所以需要用远端传来的 `tool_name`、`input`、`tool_use_id` 字段，在本地重建一个"假的"但格式合法的消息对象。

## 工具 stub 的问题

还有一个更细的问题：本地可能根本不知道远端使用了哪些工具。

比如远端安装了 MCP 工具 `my-company/deploy`，本地没有加载这个 MCP 服务器。`createToolStub` 函数解决了这个问题：

```typescript
export function createToolStub(toolName: string): Tool {
  return {
    name: toolName,
    isEnabled: () => true,
    userFacingName: () => toolName,
    renderToolUseMessage: (input) => {
      // 取前三个字段显示，让审批 UI 有内容可渲
      return Object.entries(input).slice(0, 3)
        .map(([k, v]) => `${k}: ${v}`).join(', ')
    },
    needsPermissions: () => true,
    // ... 其他方法返回空实现
  }
}
```

这个 stub 路由到 `FallbackPermissionRequest`，让用户能看到"有个叫 `my-company/deploy` 的工具请求执行，参数是..."，而不是程序崩溃。

## 界面层的影响

`viewerOnly` 模式（只读观看另一个会话）还需要处理历史消息的加载顺序。服务端的 `from_sequence_num` 机制用于断线重连后的消息补全——不是从头重放整个历史，而是从上次收到的 seq-num 开始补。

这个机制在 `BoundedUUIDSet` 的双重去重逻辑里体现：

- `recentPostedUUIDs`：过滤自己发出去的消息被回声回来的情况
- `recentInboundUUIDs`：过滤服务端在重连后重放历史时的重复消息

## 面试指导

"如何处理跨服务边界的数据格式转换"是分布式系统面试的经典题。

关键在于：不要用 `any` 类型糊弄，不要假设两侧的数据结构恰好一致。好的做法是：

1. 显式定义边界两侧的数据类型（`SDKMessage` vs `Message`）
2. 写一个明确的转换层，不让转换逻辑散落在业务代码里
3. 处理"对方有我没有的字段"的情况（工具 stub）
4. 处理"格式合法但语义上不存在"的情况（合成 AssistantMessage）

Claude Code 的 `remotePermissionBridge.ts` 是一个很好的参考实现。
