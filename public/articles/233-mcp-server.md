# MCP server 进来的是一整套需要被制度接住的能力

面试题版本：**"一个 MCP server 提供了 10 个工具。在 Claude Code 里接入这个 server，系统需要做哪些工作，不只是把这 10 个工具加入工具表？"**

## 接入不只是加工具

一个 MCP server 进来时，系统要处理的远不止"把工具加进去"：

**连接和认证**：
- 根据 server 配置选择传输方式（stdio/sse/http）
- 建立连接，处理认证（OAuth、API key 等）
- 处理连接失败和重连逻辑

**能力协商**：
- 调用 `initialize()` 了解 server 的 MCP 协议版本和能力
- 调用 `listTools()` 获取工具列表
- 调用 `listResources()` 了解 server 提供哪些资源
- 调用 `listPrompts()` 了解 server 提供哪些预定义提示词

**命名空间管理**：
- 为这个 server 的所有工具添加前缀（避免命名冲突）
- 检测和解决与现有工具的命名冲突
- 注册 server 的工具到统一工具表

**权限决策**：
- 展示 server 的信息给用户（来自哪里、提供什么能力）
- 等待用户的信任决策
- 记录信任决策（下次不需要重复询问）

**整合到任务系统**：
- 如果 server 有需要异步执行的工具，注册对应的任务处理逻辑
- 把 server 的生命周期和 Claude Code 进程的生命周期绑定（退出时关闭连接）

## "一整套能力"的含义

把这个过程和"安装一批新工具"对比：

**安装工具**：工具有 name、schema、实现函数。系统把它们加入工具表，就完成了。

**接入 MCP server**：server 有自己的协议版本、自己的认证要求、自己的能力声明格式、自己的传输机制、自己的生命周期。这些不是可选的元数据，而是正确使用这个 server 的必要上下文。

比如，一个提供数据库操作的 MCP server：
- 工具名字不能和本地文件操作工具重名（命名空间问题）
- 工具执行需要建立并维护数据库连接（连接生命周期问题）
- 某些工具需要事务支持（状态管理问题）
- 用户需要知道他们在允许 AI 操作数据库（权限信息问题）

这些都不是"工具安装"能覆盖的，它们需要系统级别的制度接管。

## 边界翻译的必要性

`src/services/mcp/normalization.ts` 做的"规范化"，不只是格式转换，还是边界翻译：

MCP 协议的工具描述格式和 Claude Code 内部的 Tool 格式有差异，但更重要的是语义差距——MCP server 的工具描述不携带"并发安全性"、"中断行为"等 Claude Code 内部需要的属性。

规范化过程需要：
- 格式转换：MCP JSON 格式 → 内部 Tool 类型
- 属性推断：根据工具类型推断并发安全性（数据库写操作 → 不并发安全）
- 语义对齐：把 MCP 的能力声明翻译成 Claude Code 权限系统能理解的语言

## "先过原制度"的工程含义

"新世界进来时也先过原制度"说的是：MCP server 的能力在进入系统前，必须经过 Claude Code 已有的每一道门：

1. **权限批准门**：用户信任了吗？
2. **命名空间门**：名字唯一了吗？
3. **规范化门**：格式符合内部标准吗？
4. **并发安全门**：工具的并发属性声明了吗？
5. **生命周期门**：连接如何管理、退出时如何清理？

只有通过了所有这些门，MCP server 的工具才正式成为 Claude Code 可用工具的一部分。这不是阻碍扩展，而是确保扩展后的系统仍然是一个整体，而不是几个子系统的拼凑。

## 面试角度的总结

"MCP server 进来的不只是工具"这个判断的工程实质是：一个外部能力来源需要被系统完整接管，而不只是把它的功能添加到列表里。完整接管意味着：连接管理、权限决策、命名空间隔离、规范化、生命周期绑定——这些是制度性的接入，不是功能性的插件安装。
