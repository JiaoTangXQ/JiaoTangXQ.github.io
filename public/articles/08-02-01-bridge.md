# Bridge ID 验证为什么需要正则白名单？安全边界分析

## 攻击面在哪里

通常我们讲安全时，会假设客户端是攻击方，服务端是受害方。但 Bridge 架构里有一个方向相反的威胁：**服务端响应被篡改，导致客户端把恶意字符串插入 URL 路径**。

`bridgeApi.ts` 里有这样一段代码：

```typescript
/** Allowlist pattern for server-provided IDs used in URL path segments. */
const SAFE_ID_PATTERN = /^[a-zA-Z0-9_-]+$/

export function validateBridgeId(id: string, label: string): string {
  if (!id || !SAFE_ID_PATTERN.test(id)) {
    throw new Error(`Invalid ${label}: contains unsafe characters`)
  }
  return id
}
```

几乎每个需要把 ID 插入 URL 的操作，都先调用这个验证：

```typescript
async pollForWork(environmentId, ...) {
  validateBridgeId(environmentId, 'environmentId')
  // 然后才构造 URL: /v1/environments/${environmentId}/work/poll
}

async acknowledgeWork(environmentId, workId, ...) {
  validateBridgeId(environmentId, 'environmentId')
  validateBridgeId(workId, 'workId')
  // 构造 URL: /v1/environments/${environmentId}/work/${workId}/ack
}
```

## 为什么这是必要的？

想象 `environmentId` 的值是 `../../admin`。

那么构造出的 URL 就变成了：

```
/v1/environments/../../admin/work/poll
```

经过服务器的路径规范化，这可能等价于：

```
/admin/work/poll
```

这是经典的**路径穿越攻击（Path Traversal）**。在客户端-服务端通信中，如果客户端完全信任服务端返回的 ID，服务端（或中间人劫持后的伪服务端）就能操纵客户端访问它本不应该访问的端点。

## 白名单而不是黑名单

注意这里用的是白名单（`/^[a-zA-Z0-9_-]+$/`），不是黑名单（过滤掉 `/`、`.`、`..`）。

白名单的优势是：

- 明确声明"只有这些字符合法"
- 新的特殊字符自动被拒绝，不需要持续维护黑名单
- 正则本身是文档：任何人读到这行，立刻知道合法 ID 的格式

黑名单的问题是漏洞枚举的完备性难以保证。URL 编码（`%2F` = `/`）、Unicode 同形字符、空字节注入，黑名单很容易漏掉新的变体。

## 注释里说了什么

源码注释写得很明白：

> Validate that a server-provided ID is safe to interpolate into a URL path. Prevents path traversal (e.g. `../../admin`) and injection via IDs that contain slashes, dots, or other special characters.

"server-provided ID"——这里的关键词是**服务端提供的**。不是用户输入，不是配置文件，而是 API 响应里的字段。

这说明代码作者已经意识到：即使是可信的服务端，返回的数据也不应该无验证地插入 URL。在零信任架构的语境下，这是正确的实践——信任来自验证，不来自来源。

## 与其他安全机制的配合

`validateBridgeId` 不是孤立的。配合它工作的还有：

**`BridgeFatalError` 的 status 精确区分**：

```typescript
case 401:
  throw new BridgeFatalError(`Authentication failed`, 401, errorType)
case 403:
  // 检查是否是 session/environment 过期，还是权限不足
  throw new BridgeFatalError(
    isExpiredErrorType(errorType) ? 'Session expired...' : 'Access denied...',
    403, errorType
  )
case 410:
  // 410 明确表示 environment 已被 GC，不可恢复
  throw new BridgeFatalError('Session has expired', 410, 'environment_expired')
```

410 和 403 的语义不同：403 可能是临时权限问题，410 是永久性的"资源已不存在"。不做区分，重试逻辑会一直无效地重试一个已经消失的 environment。

**`isSuppressible403` 的静默处理**：

某些 403 是设计内的——用户没有 `environments:manage` 权限时，`StopWork` 会返回 403，但这不影响核心功能。强行展示给用户只会造成混乱：

```typescript
export function isSuppressible403(err: BridgeFatalError): boolean {
  if (err.status !== 403) return false
  return (
    err.message.includes('external_poll_sessions') ||
    err.message.includes('environments:manage')
  )
}
```

## 面试指导

"如何防止客户端被服务端的恶意响应攻击"是一个不常见但很考验深度的面试问题。

好的答案需要覆盖：

1. **识别攻击面**：服务端返回的哪些字段会被插入敏感位置（URL、SQL、命令行）？
2. **选择验证策略**：白名单优于黑名单，原因是完备性更强
3. **失败处理**：验证失败时是抛异常还是返回默认值？Bridge 选择抛异常（快速失败，不掩盖错误）
4. **分级处理**：不是所有错误都一样严重，BridgeFatalError vs 普通 Error 的区分，体现了错误分级的工程思维
