# bridge桥接的是本地会话，不是单个指令

## 实现链
- `initReplBridge.ts` 在桥接启动时会先做 entitlement、OAuth、组织策略、标题推导和 git 上下文准备，然后把这些显式参数交给 `replBridge.ts` 的 `initBridgeCore()` 去创建远端 session。
- 整个桥接对象持有的是会话级句柄：能写消息、发 control request、接 ingress、更新标题、最终 teardown，而不是一次性“发条命令等结果”的函数。

## 普通做法
- 普通远端执行更像 `ssh host cmd` 或一次 HTTP POST，发完指令、拿回结果、链路就结束。
- 这种方式特别适合短任务和无状态工具调用。

## 为什么不用
- Claude Code 没把 bridge 做成单指令模式，因为它要搬的是持续对话、审批交互和远端会话卡片，而不是一次性计算。
- 只要目标是“远端继续当一个工作台”，桥接对象就必须围绕 session 设计，而不是围绕 command 设计。

## 代价
- 代价是初始化阶段很长，代码里也会出现大量“会话创建失败怎么办、重连怎么办、标题更新怎么办”的分支。
- 但这比把长会话硬塞进短调用模型里要诚实。
