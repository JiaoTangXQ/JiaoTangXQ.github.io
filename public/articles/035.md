# CacheSafeParams：Claude Code 如何让 forked Agent 共享父级的 prompt cache？

Anthropic API 的 prompt cache 让相同前缀的重复请求只计算一次——第一次请求的 prefill 被缓存，后续请求直接命中，大幅降低延迟和成本（约 90% 的 cache read 价格 vs 全量 input 价格）。

当 Claude Code 启动子 Agent（compact、verification、子任务 Agent）时，这些 Agent 的系统提示往往和父 Agent 相同。`utils/forkedAgent.ts` 里的 `CacheSafeParams` 是专门为这个场景设计的：

## 什么决定 prompt cache 的 cache key？

Anthropic API 的 cache key 由以下部分组成：
1. System prompt（系统提示）
2. Tools（工具列表及其 schema）
3. Model（使用的模型）
4. Messages prefix（消息历史前缀）
5. Thinking config（扩展思考配置）

这五个维度只要有一个不同，就会 cache miss。`CacheSafeParams` 就是这五个维度的容器：

```typescript
export type CacheSafeParams = {
  systemPrompt: SystemPrompt        // 必须和父级相同
  userContext: { [k: string]: string }  // 影响消息前缀
  systemContext: { [k: string]: string } // 追加到系统提示
  toolUseContext: ToolUseContext     // 包含 tools、model、thinkingConfig
  forkContextMessages: Message[]    // 父级的消息前缀（cache 共享的关键）
}
```

`forkContextMessages` 是关键——这是父 Agent 的消息历史，作为 fork 请求的消息前缀。因为 API 的 cache 是前缀匹配，子 Agent 请求从同一个前缀开始，就能命中父级已经缓存的 prefill。

## 为什么不直接传 parentContext？

如果把父级的 `ToolUseContext` 直接传给子 Agent 用，会有一个问题：子 Agent 对 `appState` 的修改会直接影响父 Agent。

`createSubagentContext` 做了关键的隔离：

```typescript
// forkedAgent.ts 里的模式
const subagentContext = {
  ...parentContext,
  // 子 Agent 不能直接修改父的 AppState
  setAppState: (fn) => {
    // 对非交互子 Agent：这是 no-op
    // 对进程内 teammate：路由到正确的 setAppState
  },
  // 克隆文件缓存，不共享引用
  readFileState: cloneFileStateCache(parentContext.readFileState),
  // 独立的拒绝追踪状态
  localDenialTracking: createDenialTrackingState(),
  // 克隆工具结果替换状态
  contentReplacementState: cloneContentReplacementState(parentContext.contentReplacementState),
}
```

注意 `cloneFileStateCache`——子 Agent 看到的是父 Agent 文件缓存的**快照**，而不是共享引用。这防止了子 Agent 的文件操作影响父 Agent 的缓存状态。

## maxOutputTokens 和 thinking budget 的交互

注释里有一个隐藏的陷阱：

> `maxOutputTokens` 可能会意外改变 thinking config，因为 `claude.ts` 在老版模型上会用 `maxOutputTokens` 来限制 `budget_tokens`。
>
> 对于 compact 任务：需要限制输出大小（不需要超长摘要），但设置 `maxOutputTokens` 会压缩思考预算，导致 cache miss。

这个细节在实际工程里很容易忽略：改了 `maxOutputTokens` 以为只是控制输出长度，但实际上影响了 thinking config，导致 cache key 变了，cache miss 了，成本上升。

解法是对支持 adaptive thinking 的新模型，`maxOutputTokens` 不影响 thinking config，所以新模型可以放心设置。老模型需要意识到这个副作用。

## 全链路 usage 追踪

```typescript
// forked agent 使用量累加到父 Agent
const usage: NonNullableUsage = EMPTY_USAGE
// 在 query loop 里每次请求后更新 usage
updateUsage(usage, turnUsage)
// 结束后累加到全局
accumulateUsage(usage)
// 日志事件
logEvent('tengu_fork_agent_query', { /* usage 统计 */ })
```

每个 forked agent 的 token 使用量被单独追踪，最后累加到全局。这让"compact 操作用了多少 token"可以单独统计，而不是混在主对话的使用量里。

## 面试角度

prompt cache 优化是 LLM 应用工程的重要话题：

1. **cache 命中需要精确的前缀匹配**：对 cache key 的每个维度都要有意识地控制
2. **fork 请求携带父级消息前缀**：这是最重要的 cache 共享技巧——子任务从父任务的上下文开始
3. **参数副作用要在文档里说清楚**：`maxOutputTokens` 影响 thinking budget 这种非直觉的副作用必须注释
4. **隔离 vs 共享**：fileStateCache 克隆（隔离写操作），消息前缀共享（复用 cache）——不同的字段有不同的隔离策略
