# isVisible 在高度未测量时为什么返回 true 而不是 false？乐观可见性的设计

高度未知时，不知道消息是否被 virtual scroll 过滤掉，但 `isVisible` 还是返回 true。这个"乐观假设"带来什么好处，又有什么代价？

## isVisible 的实现

```typescript
const isVisible = useCallback((i: number) => {
  const h = getItemHeight(i)
  if (h === 0) return false      // 高度为 0：确定不可见（折叠、隐藏）
  // h === undefined：未测量，按可见处理
  return isNavigableMessage(messages[i]!)
}, [getItemHeight, messages])
```

`getItemHeight(i)` 的返回值：
- `undefined`：这条消息从未挂载过，高度未知
- `0`：高度为 0（可能是折叠状态、纯 padding 消息等）
- `n > 0`：测量到的实际高度

`isVisible` 对 `undefined` 和 `n > 0` 的处理相同——都进一步检查 `isNavigableMessage`（是否是可以键盘导航的消息类型）。只有 `0` 被排除。

## 为什么不把 undefined 也排除

理想状态下，只有被渲染到屏幕上的消息才算"可见"，应该严格检查高度。但虚拟列表的问题在于：消息只有在出现在视口里才会被测量，测量值才会进入 `heightCache`。

**测量时序的问题**：

1. 用户按 `j` 跳到下一条消息
2. `cursorNavRef.navigateNext()` 调用 `isVisible` 找下一条可见消息
3. 如果下一条消息刚好在视口边缘刚被挂载，其 `yogaNode` 可能还没运行 layout pass
4. `getItemHeight` 返回 `undefined`（缓存还没写入）

如果 `undefined → false`（不可见），导航会跳过这条消息，直接找下一条。用户会看到光标莫名跳过一条消息，列表感觉"有洞"。

如果 `undefined → true`（假设可见），这条消息被包含在导航候选中，光标可以落到这里。等下次 layout pass 完成，高度缓存更新，这条消息的真实高度确认，一切正常。

## 乐观假设的成本

乐观假设 `undefined → visible` 的成本：导航有可能停在一条高度为 0 的未测量消息上（比如一个空消息），此时光标停在一条用户看不到的行，逻辑上可见但视觉上不存在。

Claude Code 认为这个成本可以接受，因为：
1. 高度为 0 的情况专门用 `h === 0 → false` 排除（已测量的空消息不可导航）
2. `isNavigableMessage` 的过滤进一步排除了非用户交互的消息类型
3. 完全未测量的消息在正常流程里很快就会被测量（layout pass 是自动触发的）

真正有问题的情况（未测量 + 实际不可见）很罕见，而"确定不可见"的情况（高度 = 0）已经被精确排除了。

## cursorNavRef 的 scan 函数

`isVisible` 的主要使用者是 `cursorNavRef` 里的线性扫描：

```typescript
const scan = (from: number, dir: 1 | -1, pred = isVisible) => {
  for (let i = from; i >= 0 && i < messages.length; i += dir) {
    if (pred(i)) {
      select(messages[i]!)
      return true
    }
  }
  return false
}
```

这个 O(n) 扫描在每次键盘导航时执行，`isVisible` 的判断直接影响导航结果。快速连续按 `j` 时，scan 会被调用多次，每次都需要快速的 `isVisible` 判断。`getItemHeight` 查的是 Map，是 O(1)，加上 `isNavigableMessage` 的 switch 判断，整个 `isVisible` 是 O(1)。

## 面试指导

这道题是"严格 vs 乐观"在工程中的权衡，属于防御式编程 vs 乐观假设的讨论。

**严格方案的问题**：`undefined → false` 看起来更安全（未知就视为不可见），但在这个系统里，未知不等于不可见，只是还没测量。把未知当不可见会产生假阴性（实际可见但被排除），导致导航跳过消息。假阴性比假阳性危害更大——用户会觉得消息丢了。

**乐观方案的问题**：`undefined → true` 可能导致假阳性（实际不可见但被包含），但因为 `undefined` 的消息几乎都会在下次 layout 后被测量并得到正确高度，假阳性的持续时间极短（1-2 帧），用户通常感知不到。

**类似的工程决策**：很多系统在"数据不确定时"选择乐观假设：
- 乐观更新（optimistic update）：假设 API 调用会成功，先更新 UI，失败了再回滚
- 服务降级：外部服务不可用时假设数据有效，继续提供服务，而不是直接报错

**加分追问**：如果消息被折叠（grouped_tool_use collapsed），`getItemHeight` 返回的是展开状态还是折叠状态的高度？答：只有最后一次实际测量的高度。如果用户展开后再折叠，高度缓存里存的是上次测量的值（可能是展开时的高度）。这时候会有短暂的高度不一致，spacer 算错一点，但下次 layout pass 后缓存更新，恢复正确。
