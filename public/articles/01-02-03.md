# 场景推演：一个斜杠命令、一次工具调用、一个后台 agent，各自经过了哪些代码？

假设三件事同时发生在一个 Claude Code 会话里：用户输入了 `/compact`，模型调用了 `BashTool`，以及主 agent 发起了一个子 agent。这三件事都叫"系统在做某件事"，但它们走的代码路径完全不同。

## 路径一：`/compact` 命令

`/compact` 在 `commands.ts` 里登记为一个 `Command` 对象，类型是 `local`（本地命令）或者触发一段界面交互的 `local-jsx`。

`commands.ts` 的 `COMMANDS()` 函数（用 `memoize` 包住，只生成一次）返回一个 `Command[]` 数组，包含 `compact`、`clear`、`config`、`help` 等几十个命令。`getCommands` 函数在每次调用时会重新过一遍 `meetsAvailabilityRequirement` 和 `isEnabled` 检查——不做 memoize，因为认证状态可能在会话中途变化（比如 `/login` 之后）。

命令到了 `processUserInput` 里被识别为斜杠命令后，不会去调模型，而是直接在本地执行。它负责的问题是：**这一次，用什么工作方式进入系统？** 对于 `compact`，答案是"切换到压缩历史的工作模式"。命令执行完，消息数组和 AppState 被更新，然后退出或继续循环。它不产生工具结果，不等模型响应，不需要 taskId。

## 路径二：模型调用 `BashTool`

模型决定调用 `BashTool` 时，发过来的是一个 `tool_use` 块，包含 `name: "Bash"` 和 `input: { command: "..." }`。

这里走的是 `tools.ts` 里的工具分发路径。`getAllBaseTools()` 函数返回一个有序工具数组，`BashTool` 在靠前位置。每个工具是一个实现了 `Tool` 接口的对象，这个接口有 `call`、`description`、`inputSchema`、`isConcurrencySafe`、`isReadOnly`、`isDestructive`、`validateInput`、`userPermissionDecision` 等十几个字段。

工具被调用前，`canUseTool` 会先跑一遍——这就是 `QueryEngine` 里那个 `wrappedCanUseTool`。如果返回 `deny`，拒绝记录进 `permissionDenials`，工具不执行。如果通过，`BashTool.call()` 启动 shell 进程，流式返回 `BashProgress` 类型的进度消息，最终产出 `ToolResult<string>`。

工具关心的问题是：**这只手伸出去，应该怎么动，动完返回什么？** 它的边界就是这次操作本身。它不持有跨轮状态，不登记任务，不管理生命周期。

## 路径三：子 agent 通过 `AgentTool` 被派出去

主 agent 调用 `AgentTool` 时，这次"工具调用"立刻在两个维度上展开。

从工具维度看，`AgentTool` 和 `BashTool` 一样是 `Tool` 接口的实现，走同一套权限检查和调用分发。

但从任务维度看，`AgentTool.call()` 内部会调用 `generateTaskId('local_agent')`，生成一个以 `a` 开头的 8 字符 ID（字母表是 `0-9a-z`，36^8 约 2.8 兆种组合）。然后 `createTaskStateBase` 会建出一个 `LocalAgentTaskState`，包含 `id`、`type: 'local_agent'`、`status: 'pending'`、`startTime`、`outputFile`（路径由 `getTaskOutputPath(id)` 算出）、`outputOffset: 0`。这个状态对象会通过 `registerTask` 写进 `AppStateStore` 里的 `tasks` 字段。

一旦进入 `AppState.tasks`，这个子 agent 就是可见的、可查询的、可终止的。界面的底部状态栏会显示它，`isBackgroundTask` 函数会在它 `status === 'running'` 且 `isBackgrounded === true` 时把它列为后台任务。杀掉它要走 `Task.kill` 接口，不是简单地 `abort` 一个 Promise。

## 三层分法解决的是三个不同问题

这条推演的结论不是"分三层很整齐"，而是：

- **Command 层**解决的是"用户意图如何被导向正确的工作方式"，它不做事，它分流。
- **Tool 层**解决的是"模型能力如何被安全执行"，它有输入约束、权限检查、并发属性，但没有生命周期。
- **Task 层**解决的是"持续运行的工作单元如何被管理"，它有 ID、状态、输出路径、可中止能力。

把三层揉成一层的代价是具体的：如果 `BashTool` 还要负责登记 taskId，它就无法做简单的并发推理；如果命令系统直接执行动作，就没有统一的权限检查点可插；如果子 agent 不变成 Task，用户就无法在 UI 里看到它的状态。

## 这道题的面试价值

考官拿这道题看的不是你会不会背"命令-工具-任务三层架构"，而是你能不能**从具体执行路径推导出分层的必要性**。

高分答法：选一个具体场景（比如子 agent），把从工具调用到 task 注册再到 AppState 更新的链条说清楚，然后解释为什么少任何一环都会有具体的问题。

低分陷阱：把三层描述成"分工明确"、"职责清晰"这类形容词，没有代码层面的支撑。或者反过来，背了一堆类名但说不清楚为什么 AgentTool 的 call 要做两件事（工具执行 + 任务注册）。
