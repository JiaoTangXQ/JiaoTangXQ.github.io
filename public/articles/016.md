# main.tsx 启动的前 200ms：Claude Code 为什么把副作用放在 import 顶部？

打开 `main.tsx`，第一眼就能看到三段不寻常的注释和代码交错结构：

```typescript
// 这些副作用必须在所有其他 import 之前运行
import { profileCheckpoint } from './utils/startupProfiler.js';
profileCheckpoint('main_tsx_entry');  // 立即调用！

import { startMdmRawRead } from './utils/settings/mdm/rawRead.js';
startMdmRawRead();  // 立即调用！

import { startKeychainPrefetch } from './utils/secureStorage/keychainPrefetch.js';
startKeychainPrefetch();  // 立即调用！

// 然后才是正常的 import 列表...
import React from 'react';
import { feature } from 'bun:bundle';
// ...
```

这个结构违反了"import 放顶部、副作用代码放在主体里"的常见实践。但注释已经解释了原因，而且注释非常具体。

## 为什么这三个副作用要尽早启动

### 1. profileCheckpoint：基准时间戳

`profileCheckpoint('main_tsx_entry')` 记录 main.tsx 入口时间。Claude Code 有一套启动性能分析系统（`startupProfiler`），通过在关键位置打时间戳来测量各阶段耗时。

这个必须在所有其他 import 之前运行，否则后续模块的求值时间会被计入"入口到某阶段"的测量里，导致数据失真。

### 2. startMdmRawRead：并行启动 MDM 子进程

注释说明：`startMdmRawRead` 会触发 `plutil`（macOS）和 `reg query`（Windows）的子进程，读取 MDM（移动设备管理）企业策略配置。

这些子进程的启动和读取需要约 **135ms** 的时间（注释原文的数据）。如果等到主模块加载完再启动它们，这 135ms 就只能串行等待。把它放在 import 顶部，让它和后续 ~135ms 的模块求值时间**并发执行**，实际用户感知的延迟是两者的最大值而不是之和。

### 3. startKeychainPrefetch：macOS Keychain 双读并发

这是最有技术含量的优化。注释说明：

> `startKeychainPrefetch` 并发触发两次 macOS Keychain 读取（OAuth token + 旧版 API key）。如果不提前启动，`isRemoteManagedSettingsEligible()` 会在 `applySafeConfigEnvironmentVariables()` 内部串行 spawn 这两个读取，每次约 65ms，两次合计约 130ms，发生在每次 macOS 启动时。

macOS Keychain 访问是一个同步 spawn 调用，每次约 65ms，两次必须读——OAuth token 和旧版 API key。如果串行读：65 + 65 = 130ms。如果在模块加载期间并发启动：max(65, 65) = ~65ms。启动时间缩短约 65ms，代价是把副作用提前到 import 阶段。

## 并行预热的代价

这种设计的代价是显而易见的：把副作用放在 import 层级让代码变得难以测试，也更难分析模块的实际依赖关系。这就是为什么代码里有 `// eslint-disable-next-line custom-rules/no-top-level-side-effects` 的注释——这是一个明确知道在违反规范、但有理由例外的地方。

## 启动链的其余部分

`main.tsx` 剩余的启动逻辑非常复杂，涉及：
- `init()` 初始化（遥测、全局配置、Trust Dialog 检查）
- GrowthBook feature flag 初始化
- MDM 策略加载和等待
- 远端托管设置加载
- 凭证加载（OAuth、AWS、GCP）
- REPL 启动 vs 非交互模式分叉

整个启动过程有精确的 `profileCheckpoint` 打点，可以分析每一段的耗时。这种对启动性能的精细关注，反映了 Claude Code 作为频繁启动的 CLI 工具对首次响应时间的高要求。

## 面试角度

"如何优化 CLI 工具的启动时间"是工具链性能面试的好题目。Claude Code 展示的思路：

1. **并行化 IO，不要串行等待**：把必须做但耗时的 IO（Keychain、MDM、进程 spawn）尽早并发触发
2. **基准测量先于一切**：`profileCheckpoint` 要在模块求值之前就开始，否则测量基准会漂移
3. **明确记录性能假设**：注释里写的 `~65ms`、`~135ms` 不是随便估的，是实际测量值，便于回归
4. **用例外需要注释说明**：`eslint-disable` 出现的地方，要有注释解释为什么这是合理的例外
