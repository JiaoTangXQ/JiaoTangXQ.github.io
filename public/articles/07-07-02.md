# MCP 连接不是一次性握手——能力列表为什么需要持续更新？

## 对比：REST API vs MCP 长连接

REST API 的能力是静态的：你调 `GET /api/v1/tools`，拿回工具列表，列表不会自己变。如果 API 更新了，你要主动重新请求。

MCP 长连接不一样：服务器可以主动推送 `list_changed` 通知，告诉客户端"我的能力变了，请重新拉取"。

这个差异改变了客户端的设计哲学——从"拿一次用到底"变成"保持同步"。

---

## session 过期的处理

```typescript
class McpSessionExpiredError extends Error {
  constructor(serverName: string) {
    super(`MCP server "${serverName}" session expired`)
    this.name = 'McpSessionExpiredError'
  }
}
```

当工具调用返回 HTTP 404 + JSON-RPC `-32001` 时，`client.ts` 抛出 `McpSessionExpiredError`。调用方捕获这个错误后：

1. 清除当前连接缓存
2. 调用 `ensureConnectedClient`（重新建立连接）
3. 使用新的 client 实例重试工具调用

这个重连流程是透明的——模型的 `callTool` 请求最终会成功，不需要感知"中间有一次 session 过期"。

---

## 重连有上限

`PendingMCPServer` 里的字段：

```typescript
export type PendingMCPServer = {
  name: string
  type: 'pending'
  config: ScopedMcpServerConfig
  reconnectAttempt?: number
  maxReconnectAttempts?: number
}
```

`reconnectAttempt` 和 `maxReconnectAttempts` 说明重连不是无限循环。超过最大次数后，服务器状态变为 `failed`，不再尝试。这防止了"一个坏掉的服务器不停消耗资源"的情况。

---

## 为什么不能缓存首次连接的能力列表？

一个常见的优化直觉：第一次连接时拿到工具列表，存起来，会话期间一直用。这样可以避免重复网络请求。

但这个优化在 MCP 场景里可能是错的：

**场景一：服务器热更新**

MCP 服务器可能在运行时部署新版本，新版本新增或删除了工具。如果客户端一直用旧列表，用户调用的工具可能已经不存在了，或者新工具对用户不可见。

**场景二：认证状态变化**

某些工具只对已认证用户可见。OAuth 认证后，服务器推送 `list_changed` 通知，客户端需要重新拉取才能看到那些受保护的工具。

**场景三：权限变化**

管理员可能在运行时给用户授权或撤销某些工具的访问权。不刷新列表，权限变化就不生效。

---

## `list_changed` 的精准失效

相比"定时轮询"，`list_changed` 的优势是精准：

```
定时轮询（每 30 秒）：
  - 最坏情况：用户等 30 秒才看到新工具
  - 每隔 30 秒有一次无效请求（服务器没变化但还是拉取）
  - 无法知道变化的是工具列表还是资源列表

list_changed 通知：
  - 变化发生后立即刷新（延迟 ≈ 网络延迟）
  - 只在真正有变化时发请求
  - 通知类型指定了变化的是哪类列表（tools/resources/prompts）
```

第三点特别重要：`client.ts` 可以根据通知类型只刷新对应的缓存，不需要把所有缓存全部丢弃。

---

## 实际含义：能力是"状态"，不是"配置"

这个设计的深层含义是：MCP 服务器的工具列表不是静态配置，而是运行时状态。它可以变化，变化依赖于服务器的内部状态（认证、权限、版本）。

`client.ts` 接受这个现实，把能力管理做成了主动同步而不是被动拉取。这让 Claude Code 可以在 MCP 服务器的世界里准确感知"现在能做什么"，而不是基于一份可能已经过期的快照做决策。

---

## 面试指导

**"分布式系统中，客户端如何保持与服务端状态同步？"**

三种主流策略：

1. **客户端轮询**：简单，但延迟高、浪费带宽
2. **服务端推送（WebSocket/SSE）**：低延迟，但需要持久连接和状态管理
3. **事件溯源**：客户端重放事件历史，但实现复杂

MCP 用的是第二种——服务端推送通知 + 客户端按需拉取。这是一种混合策略：推送只告诉"哪类东西变了"（低带宽），拉取获取完整的新列表（高可靠）。

说出"推送通知的价值不只是低延迟，更是精准失效——它告诉客户端哪部分缓存需要刷新，而不是全部重来"，这是对缓存失效策略的深度理解。
