# 副作用层次与制度化装配：为什么工具和钩子要在会话开始时就位

面试题版本：**"Claude Code 里 hooks 是在什么时候注册的？一个 hook 如果在会话进行中途被添加，和在会话开始前就已经注册，行为上有什么不同？这个差别有什么实际意义？"**

## 会话开始时的装配窗口

当 Claude Code 启动一个新会话（或通过 `prompt` 命令重新配置会话），有一个明确的装配阶段：

```typescript
// 概念描述：会话初始化时的装配
async function initSession(config: SessionConfig) {
  // 1. 加载 skill 正文（自定义提示词）
  const skills = await loadSkills(config.skillPaths)
  
  // 2. 配置工具许可（哪些工具这次可用）
  const toolPermissions = resolvePermissions(config.allowedTools)
  
  // 3. 注册 hooks
  const hooks = loadHooks(config.hooksConfig)
  
  // 把这些一起带进会话状态
  return createSessionState({ skills, toolPermissions, hooks })
}
```

这个装配阶段是显式的、有顺序的、在第一次模型调用之前完成的。

## 副作用层次的含义

"副作用故意挂到 preAction 之后"这个设计说明：在 Claude Code 里，副作用是有层次的：

**零副作用阶段**：输入解析、命令分发、决策判断。这些操作只读取状态，不修改任何东西。

**受控副作用阶段**（preAction 之后）：工具执行、文件修改、外部 API 调用。这些操作有副作用，但只在经过 preAction 钩子验证后才执行。

**后置副作用**：工具结果的上下文传播（通过 contextModifier）、session memory 更新、日志记录。这些在工具执行后发生，但有明确的触发时机。

把副作用挂到特定阶段后面，不只是顺序问题，而是让每个阶段的职责更清晰：输入处理不应该有副作用，决策不应该有副作用，只有执行阶段才应该修改世界状态。

## hooks 为什么要制度化装配

考虑两种注册 hooks 的方式：

**运行时动态注册**：
```typescript
// 反面例子：hooks 可以在任意时刻被添加
conversation.onToolCall((call) => {
  if (call.name === 'bash') {
    console.log('bash called')
  }
})
```

**制度化装配**：
```typescript
// Claude Code 的方式：hooks 是会话配置的一部分
const session = await loadSession({
  hooks: {
    preToolCall: [bashLogger, permissionChecker],
    postToolCall: [resultValidator],
    // ...
  }
})
```

运行时动态注册的问题：
- 任何代码在任意时刻都可以添加 hooks，会话行为变得不可预测
- 压缩和恢复会话时，动态注册的 hooks 无法被序列化和恢复
- 用户调试时，无法清晰看到"当前有哪些 hooks 在生效"

制度化装配的好处：
- 会话开始时所有 hooks 已就位，行为是可预期的
- 会话状态可以完整序列化（包括 hooks 配置），支持恢复
- 用户可以审查 hooks 配置，了解会话的完整行为

## 工具许可的前置配置

`prompt` 命令在重新配置会话时，会重新设置工具许可。这不是"临时允许某个工具用一次"，而是"重新定义这次会话的工具边界"。

区别在于：临时允许可以随时收回，会话中途改变行为不一致；重新配置是一个明确的边界切换，新的配置生效后，所有后续操作都在新配置下运行。

这和 Linux 的进程权限模型类似：进程的文件权限在启动时确定，不能在运行中途被另一个进程偷偷修改。Claude Code 的工具许可也是这样：在会话的装配阶段确定，不能在主循环运行中途被任意修改。

## 面试角度的总结

副作用分层和制度化装配解决的是同一个问题：**让会话行为在整个生命周期内保持可预期性**。副作用挂到特定阶段后面，让每个阶段的职责清晰；工具和 hooks 在会话开始时就位，让整个会话的行为约束是已知的、可审查的。这种"先把规则定好，再开始工作"的顺序，让系统的行为是可推理的，出了问题也能快速找到是哪个配置导致的。
