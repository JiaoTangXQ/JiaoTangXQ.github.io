# cwd 为什么用 AsyncLocalStorage 而不是全局变量？

## 对比分析：两种实现的区别

假设有两个设计方案：

**方案 A：全局变量**
```typescript
let globalCwd = process.cwd()
export function getCwd() { return globalCwd }
export function setCwd(path: string) { globalCwd = path }
```

**方案 B：AsyncLocalStorage（Claude Code 的实际实现）**
```typescript
const cwdOverrideStorage = new AsyncLocalStorage<string>()

export function runWithCwdOverride<T>(cwd: string, fn: () => T): T {
  return cwdOverrideStorage.run(cwd, fn)
}

export function getCwd(): string {
  return cwdOverrideStorage.getStore() ?? getCwdState()
}
```

在单线程单 agent 场景下，两者行为完全一样。但 Claude Code 支持 swarm——多个 agent 跑在同一个 Node/Bun 进程里。这时候差异就出来了。

## 问题场景

```
主进程
├── agent_alice（CWD: /projects/frontend）
│   └── 调用 BashTool → 期望在 /projects/frontend 下执行
└── agent_bob（CWD: /projects/backend）
    └── 调用 BashTool → 期望在 /projects/backend 下执行
```

如果用全局变量，agent_alice 在执行之前调用 `setCwd('/projects/frontend')`，然后 agent_bob 立刻调用 `setCwd('/projects/backend')`，等 alice 的 BashTool 真正执行时读到的 cwd 可能已经是 `/projects/backend`。

这是经典的"写了一半被覆盖"竞态条件，在单线程异步环境（await 之间存在交叉执行）下完全可能发生。

## AsyncLocalStorage 的工作原理

`AsyncLocalStorage` 是 Node.js 的原生 API（Bun 也支持），它给每条独立的 async 调用链提供了自己的"局部存储"。不同 `run()` 调用之间的存储完全隔离，即使它们在同一个事件循环线程上交错执行。

```
runWithCwdOverride('/projects/frontend', async () => {
  // 在这整个 async 树里，getCwd() 都返回 /projects/frontend
  await someLongOperation()   // 期间其他 agent 可以安全改他们的 cwd
  return getCwd()             // 仍然是 /projects/frontend
})
```

关键是：`cwdOverrideStorage.run(cwd, fn)` 注册了一个上下文，在 `fn` 内部（包括 fn 触发的所有 await 链）调用 `getStore()` 都返回这个 cwd，无论中间有没有其他代码修改了别的 async 链的存储。

## 回退机制

```typescript
export function getCwd(): string {
  try {
    return pwd()
  } catch {
    return getOriginalCwd()
  }
}

function pwd(): string {
  return cwdOverrideStorage.getStore() ?? getCwdState()
}
```

`cwdOverrideStorage.getStore()` 返回 `undefined` 的情况：当前代码不在任何 `run()` 上下文里（比如顶层代码）。这时候回退到 `getCwdState()`，它读取全局的初始化 cwd 状态，最后再 fallback 到 `getOriginalCwd()`。

三层 fallback：async local → global state → original cwd。保证了任何情况下都有合理的 cwd 返回值。

## 反常识的地方

直觉上，Node.js 是单线程的，既然不会真正并发执行，为什么需要 AsyncLocalStorage？

关键词是"交叉执行"而不是"并发执行"。单线程异步代码在 await 暂停时，事件循环会执行其他任务。如果两个 agent 各自有 await，它们的执行片段会交替出现。全局变量在这个模型下仍然有竞态。AsyncLocalStorage 用"调用链上下文"而不是"时间序"来隔离状态，这才是正确的思路。

## 面试指导

这道题考的是对 Node 异步模型的理解深度。说"单线程没有竞态"是错的——await 暂停期间会执行其他任务。AsyncLocalStorage 的正确理解是"给每条 async 调用链提供独立的上下文存储"，不是"给每个线程"。展开说 Claude Code 的三层 fallback 链（async local → global state → original cwd）展示了你真的读了代码而不是在推断。
