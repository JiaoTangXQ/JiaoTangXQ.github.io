# Claude Code 外延执行能力地图

## 先说结论：这些能力不是第二运行时，而是主工具系统的外延边界

`teleport`、`computerUse`、`claudeInChrome` 看起来像三种完全不同的能力。Claude Code 的做法不是为它们分别造三套平行系统，而是尽量把它们继续翻译进已有运行时。

## 第一层：`teleport` 解决的是远端工作台管理，不是普通联网

### 实现链

`teleport` 相关逻辑在处理：

1. code sessions
2. environments
3. 远端消息发送
4. 标题更新
5. git bundle 上传等远端现场准备

### 普通做法

更容易想到的做法是：远端能力就是多几个 HTTP API。

### 为什么不用

Claude Code 不只是要调用远端接口，而是要管理远端工作台会话。环境选择、标题、代码种子、消息延续都属于同一个会话世界，不是散接口。

### 代价

远端能力会显得很重，也更像平台底座；但更贴近真实使用。

## 第二层：`computerUse` 没被粗暴塞成几把本地工具

### 实现链

它通过专门 server、固定 MCP 命名和宿主适配，尽量以能力簇的形式进入主系统。

### 普通做法

更直觉的做法是直接给系统多加几个“鼠标/键盘/截图”内建工具。

### 为什么不用

Claude Code 想让桌面能力继续服从工具系统、命名体系和提示词识别规则，所以它尽量把这套能力包装成可被系统认出的 MCP 能力簇。

### 代价

实现更绕，也需要宿主适配；但主系统的一致性更强。

## 第三层：`claudeInChrome` 走的是产品级浏览器接入链

### 实现链

这条线会处理：

1. 扩展安装和检测
2. Native Host
3. 动态 MCP config
4. 启用策略
5. 浏览器世界专门的提示词和工具呈现

### 普通做法

更容易想到的浏览器能力，是另做一套自动化协议或外挂脚本。

### 为什么不用

Claude Code 想把浏览器世界继续纳入已有运行时，而不是再造一套浏览器专用小宇宙。所以它尽量仍然走 MCP、工具命名和主系统已有约束。

### 代价

平台差异和安装链会明显更复杂；但系统语言仍然统一。

## 第四层：共同点不是“都能操作外部世界”，而是“都被制度化接入”

### 实现链

这三条线共同服从：

1. settings
2. permission / enablement 边界
3. 命名和工具识别
4. 会话或 MCP 入口

### 普通做法

很多产品到了外部能力这里，就让每种能力自己搞一套接入规则。

### 为什么不用

Claude Code 想保住主运行时的一致性。外部能力再特别，也尽量不要让内部世界为它们一一裂开。

### 代价

接入层会变厚，也需要更多翻译；但整套系统更像同一台机器，而不是拼装怪。
