# 工程细节：echo 消息为什么先当心跳处理，再做去重过滤？

## 一个反直觉的处理顺序

初看消息处理逻辑时，会觉得应该这样写：

```
收到 WebSocket 消息
→ 检查是否是 echo（自己发出去被回声回来的）
→ 如果是 echo，直接丢弃
→ 否则，处理消息
```

但 `bridgeMessaging.ts` 里的 `handleIngressMessage` 处理顺序不是这样的。echo 检测发生在消息路由之后，而不是之前。

而在 `remoteIO.ts` 的上一层（`RemoteSessionManager`），还有另一个处理：

```
收到任何 WebSocket 消息
→ 立刻清除超时计时器
→ 然后才进入消息处理
```

这两个顺序决定都有明确的理由。

## 为什么任何消息都先清超时计时器？

超时计时器是一个"远端是否还活着"的守卫：如果一段时间内没有收到任何消息（包括 echo），判断为连接已经失效，触发重连。

echo 消息（自己发出去的消息被服务端广播回来）从网络层看，和普通的业务消息没有区别——它到达本地，证明了这一时刻的网络链路是通的，服务端也在运行。

如果先判断"这是 echo，丢弃"，再考虑"是否要清超时计时器"，就破坏了这个语义：

**场景：远端正在冷启动或慢流式输出**

假设远端会话刚启动，或者正在输出长文本（streaming），这个阶段可能有大量 echo 消息，但几乎没有新的业务消息到达本地。

如果先丢 echo，超时计时器看不到任何消息，会触发"远端卡死"判断，启动重连——即使远端其实完全正常，只是在处理任务没有向本地发新消息。

先清超时计时器，就保留了"echo 作为心跳"的语义：echo 到达 = 链路通 = 不需要重连。

## BoundedUUIDSet：为什么不用普通 Set？

去重使用的是 `BoundedUUIDSet` 而不是 `Set<string>`。"Bounded"（有界）的意义：

**内存安全**：一个长时间运行的会话（数小时）会产生大量消息，每个消息都有 UUID。如果用无界的 `Set`，随着时间积累，Set 会无限增长。`BoundedUUIDSet` 只保留最近 N 个 UUID，防止内存泄漏。

**但为什么 N 要足够大？**

注释提到：

> Defensive dedup: drop inbound prompts we've already forwarded. The SSE seq-num carryover is the primary fix for history-replay; this catches edge cases where that negotiation fails.

断线重连时，服务端可能从历史某个点开始重放消息（seq-num 机制）。如果本地的 `BoundedUUIDSet` 太小，之前处理过的 UUID 被淘汰出 Set 了，重放的历史消息就无法被识别为重复，会被重复处理。

N 的选择是一个权衡：太小 → 历史消息无法去重；太大 → 内存占用高。

## 双重去重集合的语义差异

`handleIngressMessage` 里有两个 Set：

```typescript
function handleIngressMessage(
  data: string,
  recentPostedUUIDs: BoundedUUIDSet,   // 我发出去的
  recentInboundUUIDs: BoundedUUIDSet,  // 我收到并已处理的
  ...
)
```

**`recentPostedUUIDs`**：记录本地发出去的消息 UUID。用于过滤 echo——自己发的消息被服务端广播回来。

**`recentInboundUUIDs`**：记录已经处理过的入站消息 UUID。用于过滤重放——断线重连后服务端重发历史消息。

这两个集合的内容不同（出站 vs 入站），触发填充的时机不同，因此是两个独立的 Set，而不是合并成一个。

## 消息顺序的最终全貌

完整的处理顺序：

```
WebSocket 消息到达
  ↓
清除超时计时器（echo 也算心跳）
  ↓
JSON 解析 + control message key 规范化
  ↓
是 control_response？→ 触发权限审批回调
是 control_request？→ 触发控制请求处理
不是 SDKMessage？→ 丢弃
  ↓
UUID 存在？
  → 在 recentPostedUUIDs 里？→ 是 echo，丢弃（但心跳已清）
  → 在 recentInboundUUIDs 里？→ 是重放，丢弃
  ↓
是 user 消息？→ 加入 recentInboundUUIDs，触发 onInboundMessage
其他？→ 触发对应处理
```

每一个箭头都是一个判断，每个判断的顺序都不是随机的。

## 面试指导

"稳定性靠细节"是一个在面试中很难回答好的点——因为细节很难用宏观的设计理念来解释。

比较有说服力的表达方式：用**反例**来说明顺序的重要性。

"如果先去重再清超时计时器，会发生什么？"——这比直接说"应该先清超时计时器"更有说服力，因为它展示了你已经考虑过错误情况，而不是碰巧选对了顺序。

在分布式系统设计中，消息处理的顺序是正确性的核心之一。能主动说出"如果 X 和 Y 的顺序颠倒，在 Z 场景下会出现什么问题"，是对系统理解深度的体现。
