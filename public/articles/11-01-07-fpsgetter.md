# FPS 只暴露 getter 说明性能观测不该写进共同状态世界

`FpsMetricsProvider` 对外只给一个 `getter`，而不是把 FPS 数字塞进 `AppState`。这说明性能观测是一条旁路，不应该反过来变成共同状态世界的一部分。

这样既能看性能，又不会让“为了观察界面”这件事再次触发界面自己的重排。好的观测，先避免把自己变成噪音。

## 实现链

`FpsMetricsProvider` 暴露的是 `getFpsMetrics()` 这种读取入口，而不是把 FPS 不断推送进 `AppState`。性能数据需要时再取，不需要时就不惊动共同状态世界，说明 FPS 观测被故意做成旁路。

## 普通做法

更普通的做法，是把帧率、耗时之类的性能数字直接塞进全局 state，然后让哪里想看就去订阅。

## 为什么不用

Claude Code 不这么做，是因为 FPS 变化很频繁，但多数业务组件并不需要知道它。把这类高速观测写进共同状态，只会制造无意义订阅和重渲染。

## 代价

代价是性能数据不像普通 state 那样随手可拿，调用方得显式走 getter；但换来的是性能观测不会把整个状态世界拖成高频更新源。
