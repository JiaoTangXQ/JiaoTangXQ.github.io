# Claude Code 的扩展系统为什么不给每种来源单独做一套入口？

面试高频题：「你设计一个支持 skill/plugin/MCP 三种扩展的 AI 工具，会怎么做架构？」

大多数候选人的第一反应是——三种来源天然不同，各做一套 loader 最清晰。Claude Code 的答案恰恰相反：三种来源全部汇入同一组运行时语言（Command/Tool/Resource），再挂回同一条装配流水线。

## 三条来源长什么样

**Skills（`.claude/commands/` 目录下的 Markdown 文件）**

`loadSkillsDir.ts` 扫描项目和全局两个目录，把每个 `.md` 文件解析成 `PromptCommand`。解析时会读取 YAML frontmatter（`description`、`allowed_tools`、`model`、`effort`），然后把正文当成 prompt 模板注入，最终装入 `Command[]`。

```typescript
// loadSkillsDir.ts 核心路径
const markdownFiles = await loadMarkdownFilesForSubdir(dir, 'commands')
for (const file of markdownFiles) {
  const frontmatter = parseFrontmatter(file.content)
  const cmd: PromptCommand = {
    type: 'prompt',
    name: basename(file.path, '.md'),
    description: coerceDescriptionToString(frontmatter.description),
    prompt: substituteArguments(body, args),
    ...
  }
}
```

**Plugins（JSON manifest 描述的工具集合）**

`pluginLoader.ts` 读取 `~/.claude/plugins/` 下的 manifest，每个 plugin 可以带一组 tools、resources 和 prompts。加载后统一转成 Claude Code 内部的 `Tool` 类型，再挂入 `tools` 数组——这和内置工具用的是同一个接口。

**MCP（通过 WebSocket/stdio 连接的外部 server）**

`mcpSkillBuilders.ts` 负责把 MCP server 暴露的 tools 和 resources 翻译成 `Tool` 和 `Command`。关键是翻译，而不是透传——MCP 工具的 `inputSchema` 被映射成 Claude Code 的 `ToolInputSchema`，权限检查也走 `ToolPermissionContext` 而不是另起炉灶。

## 统一装配的真正含义

三条路最终都进入 `QueryEngine` 的工具列表。`getSlashCommandToolSkills()` 负责把所有 skills 转成可以被调度的形态，`loadAllPluginsCacheOnly()` 做 plugin 的缓存装配，MCP 则通过 `MCPServerConnection` 接入。

这里有一个容易误解的细节：「统一」不是说三种扩展长得一样，而是说**运行时看到的接口一样**。`Tool` 类型要求实现：
- `name: string`
- `isEnabled(ctx): boolean`
- `call(input, ctx): AsyncGenerator<ToolResultBlockParam>`
- `renderToolUseMessage(input): ReactNode`

不管来自 skill、plugin 还是 MCP，到了调度层都是同一张脸。

## 为什么不各做一套

如果三种来源各有一套生命周期，权限管理会立刻分裂。比如：skill 走 `ToolPermissionContext`，MCP 工具自己管权限，plugin 弹自己的对话框——结果是用户根本搞不清「哪些工具现在受控，哪些不受控」。

Claude Code 的 `ToolPermissionContext` 随身携带 `permissionMode`、`rules`、`additionalDeniedTools` 等字段。所有工具（包括 MCP 来的）在 `isEnabled()` 时都要过这一道。统一装配保住的不是代码整洁，而是制度完整性。

另一个理由：发现机制。用户用 `/help` 查看可用命令，系统需要从同一个来源列表里遍历。如果三种来源分开存储，要么 `/help` 漏掉某类扩展，要么需要三次合并——而合并逻辑本身就是新的 bug 温床。

## 桥接层的代价

统一语言不是免费的。MCP 的 `tool_result` 格式和 Claude Code 内部的 `ToolResultBlockParam` 存在差异，`mcpSkillBuilders.ts` 里有一段不短的转换代码。Plugin manifest 里的 `allowed_tools` 字段格式和 skill frontmatter 的 `allowed_tools` 也不完全一致，`parseSlashCommandToolsFromFrontmatter` 要统一处理。

这些桥接件加起来，让新人第一次读代码时会产生「为什么不直接用原格式」的困惑。答案是：直接用原格式，你省的是今天的一个 adapter，换来的是将来每个消费方都要单独适配三种格式。

## 面试答题思路

被问「扩展系统怎么设计」时，可以说：

1. **先定义运行时语言**：系统里什么是「工具」、什么是「命令」、什么是「资源」，把接口固定下来
2. **各来源做翻译，不做透传**：每种扩展格式都通过一个 adapter 映射成内部接口，adapter 承担格式差异
3. **装配线只走一条**：权限检查、发现列表、日志追踪都只需要面对统一接口
4. **成本在桥接层**：承认桥接层的存在，把它写清楚，而不是分散到所有消费方

追问「这样做会有什么问题」时，诚实地说：桥接层是技术债的集中点，MCP 协议升级时 adapter 需要同步更新；如果某种来源有非常特殊的能力，强行塞进统一接口可能造成信息损失。
