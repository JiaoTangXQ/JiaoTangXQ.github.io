# 为什么 init() 搭的是环境，而不是会话

很多人一看到 `init()`，直觉就会把它理解成“开始进入这次对话”。Claude Code 的 `init()` 不是。它做的事情虽然很多，但核心都指向同一个目标: 先把一块能安全、稳定、可持续运行的地面铺出来，而不是立刻把用户推进某一轮会话。

所以它先做的是启用配置系统、应用安全环境变量、补 CA 证书、建立 graceful shutdown、异步补 OAuth 账户信息、初始化 IDE 探测、埋下远端设置和 policy limit 的加载 promise、配置 mTLS 和全局代理、预连接 Anthropic API、处理 Windows shell、注册清理器、准备 scratchpad。这些动作看起来分散，实质上都属于“运行底座”。

这一步最重要的地方就在于，它把“会话开始”往后推了一层。Claude Code 明显知道，一台真正会在现实世界里跑很久的 agent，先要能可靠地活，再谈怎么说话、怎么行动、怎么继续下一轮。`init()` 不是在开始聊天，它是在确保后面不管走到哪条路径，系统脚下至少先有一块不会塌的地。

## 实现链
这页对应的实现链都落在 `init.ts`：配置系统、安全环境变量、CA 证书、代理、平稳退出、预连接和远端 promise 会在真正会话启动前先被装起来。它关注的是运行环境，而不是马上给用户一段对话。

## 普通做法
更常见的写法是把环境准备和会话准备混在一起，等需要网络、证书或清理时再临时处理。

## 为什么不用那种做法
Claude Code 没这么拖后，是因为这些基础设施一旦错位，后面的会话层每走一步都可能踩雷。它宁可先把地基打厚。

## 代价与回报
回报是会话代码不必反复兜底底层问题。代价是 init 阶段要先处理大量用户暂时看不见的东西。
