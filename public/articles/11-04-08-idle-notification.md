# idle notification 会被静默过滤，说明排队区不是每种后台声响都值得上屏

排队区并不会忠实转播后台发生的每一件小事。像 `idle_notification` 这一类系统心跳，会被直接静默过滤掉。原因很朴素：用户眼前的排队区应该服务下一步动作，而不是把后台自言自语也搬到驾驶舱中央。

这类克制很重要。很多系统一接入后台能力，就忍不住把每个脉冲都亮出来，结果界面很快被“其实不用管”的消息占满。Claude Code 在这里选的是另一条路：不是所有真实都要入屏，只有和操作相关的真实才值得占位置。

## 实现链

`PromptInputQueuedCommands.tsx` 会先用 `isIdleNotification()` 从 JSON 内容里识别 `idle_notification`，再在处理队列时直接把这类命令过滤掉。排队区只让和下一步操作有关的通知上屏，不替后台心跳做转播。

## 普通做法

更直觉的做法，是后端发来什么前端就照单全收地 render，免得漏消息。

## 为什么不用

Claude Code 不接受这种全量转播，是因为驾驶舱区域服务的是“下一步该做什么”，不是后台自言自语的日志窗口。idle 心跳如果也上屏，只会稀释真正需要处理的通知。

## 代价

代价是渲染层得懂一点通知语义，不能完全做成无脑展示；但界面因此更克制，不容易被后台噪声淹没。
