# 面试题：summarizeRecentActivities 是如何把连续 read/search 操作折叠成一句摘要的？

这道题要求你理解"活动聚合"的逻辑——为什么相似的操作要合并，以及合并的标准是什么。

## 调用上下文

```typescript
// TaskListV2.tsx
const activities = bgTask.progress?.recentActivities;
const desc = (activities && summarizeRecentActivities(activities)) ??
             bgTask.progress?.lastActivity?.activityDescription;
if (desc) {
  teammateActivity[bgTask.identity.agentName] = desc;
  teammateActivity[bgTask.identity.agentId] = desc;
}
```

有两级降级：
1. 优先用 `summarizeRecentActivities(activities)`（智能摘要）
2. 如果没有 recentActivities，退化到 `lastActivity.activityDescription`（最后一次活动的描述）

## summarizeRecentActivities 的折叠逻辑

函数位于 `src/utils/collapseReadSearch.ts`（从注释推测）。核心思路：

如果最近的活动序列里，有多个连续的 `read_file` 或 `grep`/`search` 类操作，把它们合并成一句：

- 3 次连续读文件 → `"reading files"`
- 5 次连续搜索 → `"searching codebase"`
- 混合 → `"reading and searching"`

这不是简单的"最后一条活动"，而是对最近一段活动序列的语义摘要。

**为什么折叠而不是显示每一步？**

一个正在分析代码库的 agent，可能在几十秒内读了 20 个文件，搜了 10 次。如果逐条显示：

```
task-3: reading src/auth/login.ts
task-3: reading src/auth/logout.ts
task-3: reading src/auth/session.ts
task-3: searching "token validation"
task-3: searching "session expiry"
...
```

这不是"进度"信息，而是"活动日志"。对于协调者来说，有用的信息是"这个 agent 现在在搜集认证相关的信息"，而不是"它在第 17 步读了这个文件"。

折叠后：`"reading and searching authentication"` 或类似的一句话，传递了方向感，不是步骤序列。

## 为什么活动摘要只给 in_progress 且未阻塞的任务

```typescript
// TaskItem
const showActivity = isInProgress && !isBlocked && activity;
```

三个条件的逻辑：

- `isInProgress`：只有当前在执行的任务才有意义展示活动
- `!isBlocked`：被阻塞的任务展示"正在忙"的活动感会误导用户（任务其实被卡住了）
- `activity`：有摘要才显示

**被阻塞时的视觉处理**

```typescript
// 阻塞时的显示（注意 showActivity 为 false）
<Box>
  <Text color={color}>{icon} </Text>
  <Text bold={isInProgress} strikethrough={isCompleted} dimColor={isBlocked}>
    {displaySubject}
  </Text>
  {/* showOwner 的显示 */}
  {isBlocked && (
    <Text dimColor>
      {" "}{figures.pointerSmall} blocked by{" "}{blockerList}
    </Text>
  )}
</Box>
```

被阻塞时：subject 变灰，阻塞原因出现，活动摘要消失。这是三种视觉信号同时调整，统一传递"这个任务现在无法推进"的语义。

## 活动宽度的截断

```typescript
const maxActivityWidth = Math.max(15, columns - 15);
const displayActivity = activity ? truncateToWidth(activity, maxActivityWidth) : undefined;
```

活动摘要被截断到 `columns - 15` 的宽度。活动摘要比 subject 允许更宽（不减去 ownerWidth），因为活动摘要行通常在 subject 行下面单独显示，有整行宽度可用。

## 面试指导

**直接问法**："为什么不直接显示 teammate 的最后一次操作？"

答：最后一次操作是"步骤日志"，不是"进度摘要"。对于协调者，有价值的是"这个 agent 现在在做什么类型的工作"，而不是"它刚才的第 N 步是什么"。summarizeRecentActivities 把连续同类操作折叠成一句话，传递方向感而不是步骤序列。

**追问**："为什么被阻塞的任务不显示活动摘要？"

答：被阻塞的任务无法推进，显示"正在搜索文件"这样的活动感会给用户错误印象（任务其实卡住了）。活动摘要是"热度信号"，热度只给真正能向前走的任务。阻塞任务用变灰+阻塞原因替代活动摘要，传递"需要解除阻塞"而不是"正在工作"。

**系统追问**："agentName 和 agentId 都写入活动映射，有什么实际影响？"

答：任务的 owner 字段可能用短名也可能用完整 ID，双键映射保证了不管哪种格式都能查到活动摘要。如果只写一个键，某些任务行可能查不到活动描述，即使对应的 teammate 确实在运行。

---

*核心考察点：活动聚合 vs 日志转播的语义差异、多条件显示逻辑、"热度信号"只给可推进任务的设计动机。*
