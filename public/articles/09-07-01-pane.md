# 颜色、位置、边框：tmux pane 布局为什么是团队协作的基础设施而不是视觉装饰

`TmuxBackend.ts` 里的布局逻辑很具体：

- Leader 在左侧，固定 30% 宽度
- 第一个 Worker 来的时候，从 Leader pane 水平分割出 70% 给 Worker
- 后续 Worker 在 Worker 区域里继续分割（奇数个时垂直，偶数个时水平）
- 每个 Worker pane 有颜色编码的边框和标题

当用户在终端窗口里看到这个布局时，有一种直觉上的"明白感"——蓝色边框那个是 researcher，绿色那个是 reviewer，左边那个是我在和它说话的 leader。

这种直觉不是偶然的，是刻意设计的。

## 视觉稳定性 = 认知负担降低

想象一个场景：5 个 agent 同时工作，没有颜色编码，没有固定位置，界面上只有 5 个大小差不多的 pane，标题显示的是随机的 session ID。

你怎么知道哪个是 researcher？哪个是 leader？哪条输出是谁产生的？

每次需要这个信息，你都要：扫一遍所有 pane 的标题 → 找到目标 pane → 然后才能关注它的内容。这个额外的认知步骤，在长时间监督多个 agent 工作时会积累成显著的认知疲劳。

颜色 + 位置是人脑的"空间记忆"机制的直接利用。**你记住"蓝色左侧是 researcher"比记住"session-abc-123 是 researcher" 容易得多**，因为颜色和位置是视觉属性，不是抽象文本，调用的是不同的记忆系统。

## 颜色分配的稳定性

`teammateLayoutManager.ts` 的颜色分配策略：

```typescript
const colorAssignments = new Map<string, AgentColorName>()

export function assignTeammateColor(teammateId: string): AgentColorName {
  // 如果已经分配过，返回同一个颜色
  const existing = colorAssignments.get(teammateId)
  if (existing) return existing

  // 从未使用的颜色里挑一个
  const usedColors = new Set(colorAssignments.values())
  const availableColors = COLOR_POOL.filter(c => !usedColors.has(c))
  const color = availableColors[0] ?? COLOR_POOL[colorAssignments.size % COLOR_POOL.length]!

  colorAssignments.set(teammateId, color)
  return color
}
```

关键点：`colorAssignments` 是一个 Map，同一个 `teammateId` 永远返回同一个颜色，直到 `clearTeammateColors()` 被调用（团队清理时）。

为什么要稳定？

想象在工作了 30 分钟后，突然 researcher 的颜色从蓝色变成了绿色。你之前建立的"蓝色 = researcher"的视觉映射瞬间失效，需要重新学习新的映射。这是一种无意义的认知重置。

稳定颜色让视觉映射可以累积——越用越熟悉，不会被意外打断。

## tmux 布局里 30% 是怎么来的

```typescript
// 第一个 teammate 来时，Leader 占 30%，Worker 占 70%
splitResult = await execFileNoThrow(TMUX_COMMAND, [
  'split-window',
  '-t', currentPaneId,
  '-h',        // 水平分割
  '-l', '70%', // 新 pane 占 70%
  ...
])
```

为什么是 30/70 而不是 50/50？

Leader 的主要作用是**被用户看到和操作**：用户和 AI 的对话在 Leader 这边，权限审批弹窗在这边，进度汇报也在这边。Leader 不需要很大的空间来显示输出，它需要的是可见性。

Worker 的主要作用是**执行任务**：它们会输出大量的执行日志、代码变更、错误信息。它们需要足够的空间来显示这些内容，用户才能在需要时瞥一眼看看执行是否正常。

30/70 的分割，**把宝贵的左侧位置给了用户交互的焦点（Leader），把更大的空间给了需要可读性的执行输出（Worker）**。

## pane 隐藏和显示

`TmuxBackend.ts` 实现了 `hidePane` 和 `showPane`：

```typescript
// 隐藏：把 pane 移到隐藏的 detached session 里
async hidePane(paneId: PaneId, useExternalSession = false): Promise<boolean> {
  const runTmux = useExternalSession ? runTmuxInSwarm : runTmuxInUserSession
  
  // 创建隐藏 session（如果不存在）
  await runTmux(['new-session', '-d', '-s', HIDDEN_SESSION_NAME])
  
  // 把 pane 移过去
  const result = await runTmux([
    'break-pane', '-d',
    '-s', paneId,
    '-t', `${HIDDEN_SESSION_NAME}:`,
  ])
  
  return result.code === 0
}
```

这解决了一个实用问题：有时候 Worker 完成了任务，进入 idle 状态，它的 pane 就变成了无用的终端窗口占用空间。隐藏 pane 不是杀掉它（Worker 进程还在，还可以接收新任务），而是把它移出视野，减少视觉噪音。

需要的时候，`showPane` 可以把它移回来，继续在 Leader 旁边显示。

这是"空间是团队资源"思维的具体体现——不只是创建 pane，还要管理 pane 的可见性，让用户在任何时刻看到的都是当前最相关的 Worker，而不是所有已经启动过的 Worker。

## 面试怎么答

如果面试题是"多 agent 系统的可视化界面应该如何设计"：

**低分答案**：做一个仪表盘，列出所有 agent 的状态，每个 agent 一行。这是监控视图，不是协作视图——用户是旁观者，不是参与者。

**中等答案**：让每个 agent 有独立的视图窗口，用颜色区分。这是正确方向，但缺少"空间关系编码信息"。

**高分答案**：空间位置和颜色都是信息载体，不是装饰。Leader 固定在某个位置，Worker 围绕 Leader 展开，颜色稳定不变。用户建立的空间记忆（"蓝色左侧的 researcher"）在整个会话里有效，不会被界面变化打断。

**额外思考**：人类团队的协作空间（会议室里的座位、大屏幕上的看板）也是刻意设计的——不是随机分配的，是为了让所有人在视觉上快速建立"谁在哪里，状态是什么"的认知。AI Agent 团队的界面设计面临相同的人因工程问题，借鉴人类协作空间的设计经验是合理的。
