# 深度分析：文件邮箱为什么是多 agent 通信的最好选择

## Mailbox 的实现

在 `src/utils/mailbox.ts` 里，`Mailbox` 类实现了一个内存内的消息队列：

```typescript
export class Mailbox {
  private queue: Message[] = []
  private waiters: Waiter[] = []
  
  send(msg: Message): void { /* 发消息到队列或直接触发等待者 */ }
  poll(fn): Message | undefined { /* 非阻塞地检查是否有匹配的消息 */ }
  receive(fn): Promise<Message> { /* 等待直到有匹配的消息 */ }
}
```

但 agent 间的实际通信，还需要 `src/utils/teammateMailbox.ts` 和 `src/utils/swarm/permissionSync.ts` 里的文件系统层：

消息通过 `writeToMailbox` 写到文件，通过轮询读取。这是文件邮箱（file-based mailbox）的含义：消息的持久化在文件系统上，而不只是内存队列。

**面试题：为什么不用更"现代"的通信方式（WebSocket、消息总线）？**

## 不用 WebSocket 的理由

WebSocket 是高效的实时通信方式，适合服务器-客户端或服务器-服务器通信。

但 Claude Code 的 swarm 场景有一个特殊约束：**所有 agent 可能在同一台机器上，也可能在不同机器上**。

如果是同机 agent：WebSocket 需要一个监听端口，所有 agent 都连到这个端口。问题是：谁负责监听？如果 leader 监听，leader 重启时所有 worker 的连接都断了。如果有独立的消息服务器，就需要部署和维护一个额外的服务。

如果是跨机 agent：WebSocket 需要网络配置，端口开放，防火墙规则。这在企业内网和受限环境里可能很难配置。

文件邮箱没有这些问题：文件系统是最基本的共享资源，所有进程都能访问，不需要额外服务，不需要网络配置。在同机场景下，这就是进程间通信的最简单方式。

## 不用共享内存的理由

共享内存是性能最好的进程间通信方式，延迟几乎为零。

问题是：

**进程崩溃时共享内存可能处于不一致状态**：如果一个 agent 在写共享内存的中途崩溃，另一个 agent 读到的可能是部分写入的数据。检测和恢复这种不一致需要复杂的锁机制和检查点协议。

**共享内存需要进程协调访问**：所有 agent 需要协商谁可以写哪部分内存，避免并发写入导致数据损坏。这本质上是在内存层面重新实现一套协议，反而更复杂。

**不支持跨机通信**：共享内存天然是单机的，如果 swarm 需要扩展到多机，共享内存方案必须完全重写。

## 文件邮箱的真正优势

**持久性**：消息写到文件，进程重启后还在。这是文件邮箱最重要的特性——**可恢复性**。

某个 worker 崩溃了，它还没处理的消息仍然在文件邮箱里。重启后，worker 先读取邮箱里的未处理消息，恢复到崩溃前的状态，继续工作。这在 WebSocket 和共享内存方案里需要额外的机制才能实现，文件邮箱是天然支持的。

**不依赖持续连接**：文件系统读写不需要维持连接。agent 可以批量写消息，然后去做自己的事，再批量读响应。不需要像 WebSocket 那样随时保持连接在线。

**可调试性**：邮箱里的消息是文件，可以直接用文本编辑器查看，用 `grep` 搜索，用 `tail -f` 实时观察。这对调试多 agent 协作问题非常有价值。

**最大兼容性**：文件系统操作在所有操作系统上都有标准实现，不需要特殊权限（除了读写访问），不受网络限制。

## 锁文件的协调作用

文件邮箱需要处理并发写入的问题（两个 agent 同时写同一个邮箱文件会损坏数据）。

`permissionSync.ts` 里引用了 `lockfile` 工具：

```typescript
import * as lockfile from '../lockfile.js'
```

文件锁（file lock）是协调并发文件访问的标准机制：写入前先获取锁，写完释放锁，其他进程在锁被持有时等待。

文件锁的可靠性比代码里实现的锁更高：操作系统保证在进程崩溃时锁被自动释放（不会出现死锁）。

这是"利用操作系统已有机制"的设计原则：不重新发明锁，而是用经过验证的系统原语。

## 面试要点

1. **文件邮箱的核心价值是持久性**：消息写到文件，进程重启后仍然存在，可恢复性是主要优势。
2. **比 WebSocket 简单**：不需要额外服务，不需要网络配置，同机场景直接用文件系统。
3. **比共享内存更安全**：文件锁比进程锁更可靠（系统保证崩溃时自动释放），并发写入有完善的保护机制。
4. **可调试性是额外红利**：邮箱文件可以直接用标准工具查看和分析，多 agent 调试的门槛更低。
