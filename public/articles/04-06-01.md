# 状态机图：工具调度器如何把「并发」和「串行」动态切换？

当模型一次回复里调用了多个工具，工具调度器（`toolOrchestration.ts`）面临一个经典的调度问题：有些工具可以同时跑，有些不行。

## 核心抽象：isConcurrencySafe

每个工具都声明自己是否并发安全：

```typescript
// 在 Tool 类型定义里
isConcurrencySafe(input: I): boolean
```

什么算"并发安全"？源码里的语义是：这个工具的执行**不会改变其他工具执行所依赖的状态**。具体来说：

- `ReadFile`（读文件）：并发安全 —— 多个读操作互不干扰
- `Glob`（文件列表）：并发安全 —— 只读
- `Grep`（搜索）：并发安全 —— 只读
- `WebSearch`（搜索）：并发安全 —— 只读网络
- `Bash`（执行命令）：**不并发安全** —— 可能修改文件、修改环境变量、改变工作目录
- `WriteFile`（写文件）：**不并发安全** —— 会改变文件系统
- `EditFile`（编辑文件）：**不并发安全** —— 会改变文件系统

注意：`Bash` 的 `isConcurrencySafe` 不是固定值，它会解析命令内容来判断：

```typescript
// BashTool 的实现（简化）
isConcurrencySafe(input: BashToolInput): boolean {
  // 解析命令，判断是否只读
  // 如果解析失败，返回 false（保守策略）
  try {
    return isReadOnlyBashCommand(input.command)
  } catch {
    return false  // 解析失败时视为不安全
  }
}
```

## partitionToolCalls：动态切批次

```typescript
function partitionToolCalls(
  toolUseMessages: ToolUseBlock[],
  toolUseContext: ToolUseContext,
): Batch[] {
  return toolUseMessages.reduce((acc: Batch[], toolUse) => {
    const tool = findToolByName(toolUseContext.options.tools, toolUse.name)
    const parsedInput = tool?.inputSchema.safeParse(toolUse.input)
    const isConcurrencySafe = parsedInput?.success
      ? (() => {
          try {
            return Boolean(tool?.isConcurrencySafe(parsedInput.data))
          } catch {
            return false  // ← 保守策略：解析失败视为不安全
          }
        })()
      : false
    
    // 关键逻辑：与前一个 batch 合并，还是新开一个
    if (isConcurrencySafe && acc[acc.length - 1]?.isConcurrencySafe) {
      acc[acc.length - 1]!.blocks.push(toolUse)
    } else {
      acc.push({ isConcurrencySafe, blocks: [toolUse] })
    }
    return acc
  }, [])
}
```

这是一个**贪心分批算法**：从左到右扫描工具列表，相邻的并发安全工具合并成一批，遇到不安全工具就单独成一批。

## 状态机图：一次工具调度的完整流程

假设模型调用了这 5 个工具：

```
[ReadFile A] [ReadFile B] [Bash cmd] [Grep X] [Grep Y]
```

分批结果：

```
Batch 1: [ReadFile A, ReadFile B]  — isConcurrencySafe: true
Batch 2: [Bash cmd]                — isConcurrencySafe: false
Batch 3: [Grep X, Grep Y]          — isConcurrencySafe: true
```

执行流程：

```
runTools() 开始
  │
  ▼
处理 Batch 1 (并发安全批次)
  ├─ runToolsConcurrently([ReadFile A, ReadFile B])
  │   ├─ 同时启动 ReadFile A 和 ReadFile B
  │   ├─ 收集 contextModifier (排队，不立刻应用)
  │   └─ yield 消息（按完成顺序，不强制按请求顺序）
  │
  ├─ 批次结束后，按请求顺序应用 contextModifier
  │   for (block of Batch1.blocks):
  │     apply queuedContextModifiers[block.id]
  │
  ├─ yield { newContext }  ← 上下文已更新
  │
  ▼
处理 Batch 2 (非并发安全)
  ├─ runToolsSerially([Bash cmd])
  │   ├─ 单独执行 Bash
  │   ├─ 立刻应用 contextModifier
  │   └─ yield 消息
  │
  ▼
处理 Batch 3 (并发安全批次)
  ├─ runToolsConcurrently([Grep X, Grep Y])
  │   ├─ 同时启动 Grep X 和 Grep Y
  │   └─ ...
```

## 关键设计：上下文修改的延迟应用

并发批次里有一个微妙的细节：**contextModifier 不在工具执行时立刻应用，而是在整批完成后按序应用**。

```typescript
// 并发批次的处理
const queuedContextModifiers: Record<string, ((c: ToolUseContext) => ToolUseContext)[]> = {}

for await (const update of runToolsConcurrently(blocks, ...)) {
  if (update.contextModifier) {
    const { toolUseID, modifyContext } = update.contextModifier
    queuedContextModifiers[toolUseID] ??= []
    queuedContextModifiers[toolUseID].push(modifyContext)
  }
  yield { message: update.message, newContext: currentContext }  // ← 上下文还没更新
}

// 批次结束后，按请求顺序（blocks 的顺序）应用修改
for (const block of blocks) {
  const modifiers = queuedContextModifiers[block.id]
  for (const modifier of modifiers) {
    currentContext = modifier(currentContext)
  }
}
```

为什么要这样做？假设 ReadFile A 和 ReadFile B 都会修改上下文（比如记录"这个文件我读过了"），如果 B 先完成、先修改了上下文，然后 A 完成时基于已经被 B 修改的上下文再修改一次——结果取决于完成顺序，是不确定的。

延迟应用 + 按请求顺序应用，确保上下文修改的结果是**确定的**，不受实际完成顺序影响。

## 并发数量限制

```typescript
function getMaxToolUseConcurrency(): number {
  return (
    parseInt(process.env.CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY || '', 10) || 10
  )
}

// 使用 all() 函数控制并发数
yield* all(
  toolUseMessages.map(async function* (toolUse) { ... }),
  getMaxToolUseConcurrency(),  // ← 最多同时 10 个
)
```

默认最多 10 个并发。可以通过环境变量调整。这个限制防止了"一次回复里模型调用了 100 个工具，同时发起 100 个文件读取"这种资源耗尽场景。

## 面试指导

这道题考察的是**调度器设计**。一个好的调度器需要回答三个问题：
1. **谁可以并行**：`isConcurrencySafe` 的语义
2. **如何分批**：`partitionToolCalls` 的贪心算法
3. **副作用如何应用**：contextModifier 的延迟应用保证确定性

能说出"并发执行但延迟应用上下文修改，以确保确定性"这个点的候选人，面试官会认为他理解了并发设计的核心难点——不是"如何并发执行"，而是"如何安全地处理并发执行的副作用"。
