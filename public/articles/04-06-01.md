# 为什么并发安全和非并发安全工具必须被区别对待

一提到 agent 调工具，很多人最先想到的优化就是并发。既然多个动作可以同时做，那就一起做，越多越快。问题是，现实里的工具根本不共享同一种物理性质。有的工具只是读，有的工具会写文件，有的工具会改状态，有的工具还会把上下文本身一起改掉。把这些东西全都当成“可以并发的函数”，很容易把系统搞乱。

Claude Code 的处理很务实。无论是普通工具编排，还是流式工具执行，都会先判断一个工具是不是 `isConcurrencySafe`。能安全并发的，会被放进同一批里一起跑；不能安全并发的，则要单独串行执行。更细的一点是，即使并发工具带来了上下文修改，这些修改也不会立刻胡乱写回，而是先排队，等这一批都结束以后再按原顺序应用。

这背后其实是在承认一个很朴素的事实：速度从来不是唯一目标，可解释性和一致性同样重要。Claude Code 并没有迷信“更多并发一定更高级”，它是在问另一个问题: 这次并发会不会破坏世界的一致性？如果会，那宁可慢一点，也不要把后面的上下文和结果顺序搅乱。

这是一种工程上的克制。真正优雅的地方，不是把并发堆到极限，而是知道哪里该收手。也正因为它先把工具分成不同物种，后面那些看上去很复杂的恢复、继续和任务衔接，才有稳定地基。

代码上，这个判断落在 `toolOrchestration.partitionToolCalls()` 和 `StreamingToolExecutor.canExecuteTool()` 里。工具先通过 `inputSchema.safeParse()` 拿到合法输入，再调用各自的 `isConcurrencySafe()` 决定能不能跟兄弟并跑；能并跑的会被合成只读 batch，不能并跑的就单独占用执行窗口。更普通的做法要么是“一律串行，最安全”，要么是“只要是多个 tool_use 就一律并发，先快再说”。

Claude Code 没选任何一个极端，因为这套系统里有读文件、搜代码这种近似只读的动作，也有 Bash、编辑、权限修改这种会改变现场的动作。全串行会平白牺牲速度，全并发会让世界状态互相踩踏。代价是每个工具作者都得认真声明并发安全性，一旦判断错，bug 会比纯串行更隐蔽。

## 实现链
`partitionToolCalls()` 和 `isConcurrencySafe()` 先把工具划成只读 batch 与独占 batch。

## 普通做法
要么全串行，要么全并发。

## 为什么不用
因为工具里同时存在近似只读和强副作用动作。

## 代价
工具作者得认真声明并发安全性。
