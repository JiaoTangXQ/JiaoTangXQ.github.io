# 扩展再多也得汇回同一种运行时语言

## 实现链
- `src/commands.ts` 会把内置命令、旧 command、skills、plugin commands 以及 MCP 暴露出来的命令能力收回同一条装配线，最后都用 `Command` 这类本地运行时对象说话。
- MCP 侧虽然来源最复杂，但 `client.ts` 仍然会把工具包装成 `MCPTool`，把资源包装成读写资源工具，再把 prompts、skills 通过桥接层翻译成 Claude Code 自己认得的对象。

## 普通做法
- 普通做法是给每种来源各留一套入口：插件自己一套 UI，远端 server 自己一套面板，本地技能再单独一套命令。
- 这样写最省翻译层，因为谁提供能力，谁就自己负责展示和调用。

## 为什么不用
- Claude Code 没走这条路，因为模型、权限、UI 和帮助系统都更适合面对统一对象，而不是面对五种不同来源的半成品。
- 如果不先翻译成同一种运行时语言，后面的权限判断、提示生成、工具渲染和 slash command 搜索都要按来源分支，复杂度会从扩展层一路渗到全系统。

## 代价
- 代价是翻译层会损失一些来源特有的细节，尤其是远端能力常常得先被压扁成 Claude Code 本地能理解的形状。
- 但这套取舍总体是合理的，因为它把复杂度集中在接入边缘，而不是把全系统都变成“来源感知”的状态。
