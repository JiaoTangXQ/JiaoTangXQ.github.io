# local 命令：有些事就不该绕模型一圈

面试题版本：**"Claude Code 里的 `/clear`、`/compact`、`/status` 这类命令和 `/ask` 这类命令有什么本质区别？为什么有些命令不需要调用模型？这个区分有什么意义？"**

## local 命令的执行路径

`getMessagesForSlashCommand()` 处理 type 为 `'local'` 的命令时：

```typescript
case 'local': {
  // 懒加载命令的本地实现模块
  const module = await import(command.modulePath)
  // 直接调用实现
  const result = await module.execute(args, context)
  // 把结果作为系统消息写回会话
  return {
    type: 'local_output',
    content: result.output,
  }
}
```

整个流程：解析命令 → 加载模块 → 执行 → 写回结果。没有 LLM API 调用，没有 token 消耗，没有不确定性。

## 什么操作适合 local

**具有确定性答案的操作**：
- `/status`：显示当前会话的状态信息（token 使用量、活跃任务数等）。这个答案在程序里是确定的，不需要模型推理。
- `/list`：列出可用的技能或命令。这是一个查询操作，结果是确定的。
- `/compact`：执行上下文压缩。压缩操作有确定的步骤，程序直接执行。

**副作用明确的操作**：
- `/clear`：清空对话历史。明确的操作，不需要模型理解"清空"是什么意思。
- `/cost`：显示本次会话的费用估算。纯计算，不需要模型。

**需要稳定输出格式的操作**：
- 状态显示需要固定格式（token 用了多少、任务几个）。如果让模型输出，格式会不确定，可能每次不一样。

## 绕模型一圈的代价

对于 `/clear`、`/status` 这类命令，如果让模型处理：

1. 发 API 请求（几百毫秒到几秒的延迟）
2. 消耗 token（成本）
3. 模型可能不完全理解意图（不确定性）
4. 输出格式可能变化（不稳定）

对于一个确定性的本地操作，这四个代价没有任何必要。

更糟糕的是，让模型处理会话管理命令（`/clear`、`/compact`）意味着这些操作本身的结果会出现在对话历史里，影响下一次模型的推理。一个"清空历史"的操作如果经过模型处理，可能产生一段"我已经清空了我们的对话历史"的模型回复，这本身就是矛盾的。

## 分层的本质

`'local'` vs `'prompt'` vs `'local-jsx'` 这三种命令类型，是在做一个分层判断：

**local**：这个操作在程序层面完全可以确定，不需要模型参与。程序做。

**prompt**：这个操作需要模型推理或者需要在对话上下文里处理。模型做。

**local-jsx**：这个操作需要一个临时的交互界面（比如配置向导、复杂的选择对话框）。程序控制界面，用户交互完成后回到会话。

分层的价值不只是性能，而是**每件事由最合适的层来做**。让程序做程序能做的事，让模型做需要推理和语言理解的事，让界面做需要交互的事。

## 面试角度的总结

local 命令类型的本质是：认识到不是所有用户命令都需要经过模型推理，程序可以直接、稳定、高效地完成很多操作。这个区分避免了把确定性操作变成不确定性的 LLM 调用，保证了关键操作（清空历史、显示状态、执行压缩）的稳定性和速度，也避免了这些操作在对话历史里产生不必要的干扰。
