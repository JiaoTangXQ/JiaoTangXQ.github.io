# Bug 猎人：QueryEngine.submitMessage 里有一处设计，如果用普通函数替换会出现竞态

## 发现这个地方

`QueryEngine.submitMessage` 函数签名：

```typescript
async *submitMessage(
  prompt: string | ContentBlockParam[],
  options?: { uuid?: string; isMeta?: boolean }
): AsyncGenerator<SDKMessage, void, unknown>
```

这是一个 AsyncGenerator——它不返回一个 `Promise<result>`，而是用 `yield` 逐条吐出 `SDKMessage`。

这个选择有什么代价和收益？如果把它改成一个普通的 `async function`，返回 `Promise<SDKMessage[]>`，会发生什么？

## 找到这段注释

`submitMessage` 内部有一段很长的注释，解释了一个具体的时序问题：

```
// Persist the user's message(s) to transcript BEFORE entering the query
// loop. The for-await below only calls recordTranscript when ask() yields
// an assistant/user/compact_boundary message — which doesn't happen until
// the API responds. If the process is killed before that (e.g. user clicks
// Stop in cowork seconds after send), the transcript is left with only
// queue-operation entries; getLastSessionLog filters those out, returns
// null, and --resume fails with "No conversation found".
```

翻译过来：

1. 用户发送一条消息
2. `submitMessage` 开始执行，调用 API 等待模型响应
3. 如果在 API 响应回来之前进程被 kill（用户点了 Stop）
4. 如果用 `for-await` 在 yield 时才写 transcript，此时 transcript 里只有队列操作记录，没有用户消息
5. 下次用户用 `--resume` 恢复会话，`getLastSessionLog` 过滤掉那些记录，返回 null
6. `--resume` 失败：`No conversation found`

注释里还有一段对 `--bare` 模式的处理：

```
// --bare / SIMPLE: fire-and-forget. Scripted calls don't --resume after
// kill-mid-request. The await is ~4ms on SSD, ~30ms under disk contention
// — the single largest controllable critical-path cost after module eval.
// Transcript is still written (for post-hoc debugging); just not blocking.
```

在 bare 模式下，transcript 写入是 fire-and-forget（`void transcriptPromise`），不阻塞关键路径，因为脚本调用不会 `--resume`。在普通交互模式下，写入是 `await` 的，等写完再进 query 循环。

这就是 AsyncGenerator 比普通函数解决问题的地方：它允许在**开始 yield 消息之前**先完成关键的副作用（transcript 写入），而不需要等到所有消息都 yield 完。普通函数要么全部 await 完再返回（延迟太高），要么全部返回再写（丢失先写的保障）。

## 另一个竞态：permissionDenials 的跟踪

`submitMessage` 里有一个 `wrappedCanUseTool` 闭包：

```typescript
const wrappedCanUseTool: CanUseToolFn = async (tool, input, ...) => {
  const result = await canUseTool(tool, input, ...)
  if (result.behavior !== 'allow') {
    this.permissionDenials.push({
      tool_name: sdkCompatToolName(tool.name),
      tool_use_id: toolUseID,
      tool_input: input,
    })
  }
  return result
}
```

这个 wrapper 在每次 `submitMessage` 调用时被重新创建，但 `this.permissionDenials` 跨多次 `submitMessage` 累积。这意味着：

- 第一轮对话：工具调用被拒绝 → `permissionDenials` 有 1 条记录
- 第二轮对话：又有工具调用被拒绝 → `permissionDenials` 有 2 条记录

这两条记录都会在 SDK 报告里出现。如果调用方只关心"本次调用"的拒绝情况，需要在每次 `submitMessage` 之前手动清空，或者知道 `permissionDenials` 是会话级别的累积而不是单轮级别的。

这不是 bug，是设计决策——但它是一个需要明确知道的"边界条件"。

## 主流程不能解释的第三个问题

上面两个都是"主流程图画不出来的逻辑"。还有一个更隐晦的：

`submitMessage` 开头有这样一行：`this.discoveredSkillNames.clear()`。

`discoveredSkillNames` 是在会话级别累积的技能发现记录（用于 telemetry：某个技能是否是这次对话里首次被发现的）。但它在每次 `submitMessage` 开始时被清空，而不是跨轮积累。

注释解释：`Must persist across the two processUserInputContext rebuilds inside submitMessage, but is cleared at the start of each submitMessage to avoid unbounded growth across many turns in SDK mode.`

这是一个精确的"哪个粒度上清空"的决策：回合内保留（因为一次 submitMessage 里 processUserInputContext 会被重建两次，第二次不应该重新 discover 已经 discovered 的技能），但回合间清空（防止 SDK 模式下长会话里集合无限增长）。

## 面试指导

**考官的问法**：`submitMessage` 为什么是 AsyncGenerator，直接用 Promise 有什么问题？

**满分答法**：说出 transcript 时序问题（进程被 kill 时 resume 失败）、说明 AsyncGenerator 如何让"先写 transcript，再开始 yield"成为可能、并指出 `--bare` 模式下的 fire-and-forget 优化为什么成立（脚本调用不 resume，4ms 的等待不值得）。

**加分点**：主动提到 `permissionDenials` 的跨轮累积是会话级别的，以及 `discoveredSkillNames` 的清空时机——这两个都是"主流程图里看不见的状态管理"。

**常见错误**：说"AsyncGenerator 是为了流式输出"——这是表象，不是原因。流式输出可以用 callback 或者 EventEmitter 做，选 AsyncGenerator 的核心原因是它允许在第一次 yield 之前执行有副作用的初始化逻辑，同时保持整个调用过程的可取消性（AbortController 可以打断 for-await 循环）。
