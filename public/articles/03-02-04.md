# 面试题：为什么 processUserInput 先整理输入，再决定分流方向？

## 工程决策

面试官问：「你来设计 processUserInput，会先判断命令类型（`/` 开头 = slash 命令），再处理图片和附件，还是反过来？」

大多数人的直觉：先判断类型，再做对应处理。这看起来更「高效」——如果是 slash 命令，可能根本不需要处理图片。

Claude Code 的选择：先整理，后分流。

## 为什么先整理

翻开 `processUserInputBase`，执行顺序是：

```
1. 图片块规范化（非 string 输入时 resize image blocks）
2. 粘贴图片落盘 + 并行 resize
3. bridge 安全检查（isBridgeSafeCommand 逻辑）
4. ultraplan 关键词检测（基于 preExpansionInput）
5. 附件提取（shouldExtractAttachments 条件）
6. ——这里才——
7. if bash → processBashCommand
8. if slash → processSlashCommand
9. else → processTextPrompt
```

为什么图片处理和 bridge 检查要发生在分流之前？

**首先，slash 命令也需要图片和附件**

slash 命令不是纯文本操作。`processSlashCommand` 接受 `imageContentBlocks` 和 `attachmentMessages` 两个参数：

```typescript
const slashResult = await processSlashCommand(
  inputString,
  precedingInputBlocks,
  imageContentBlocks,     // ← 处理后的图片块
  attachmentMessages,     // ← 已提取的附件
  ...
)
```

如果先走 slash 分支，再处理图片，slash 处理器就得自己再做一遍图片 resize。或者把图片处理代码复制到每个分支里——这正是先分流后整理的代价。

**其次，bridge 检查会影响 slash 是否允许执行**

```typescript
let effectiveSkipSlash = skipSlashCommands
if (bridgeOrigin && inputString !== null && inputString.startsWith('/')) {
  const parsed = parseSlashCommand(inputString)
  const cmd = parsed ? findCommand(parsed.commandName, ...) : undefined
  if (cmd) {
    if (isBridgeSafeCommand(cmd)) {
      effectiveSkipSlash = false  // 允许执行
    } else {
      return { messages: [...], shouldQuery: false, ... }  // 直接拦截
    }
  }
}
```

bridge 检查发生在分流之前，是因为它的结果会决定 slash 分支要不要走。先分流就没法做这个检查了。

**第三，ultraplan 关键词会劫持整条路径**

```typescript
if (feature('ULTRAPLAN') && mode === 'prompt' && !inputString.startsWith('/') && ...) {
  const rewritten = replaceUltraplanKeyword(inputString).trim()
  return processSlashCommand(`/ultraplan ${rewritten}`, ...)
}
```

用户说了「ultraplan 这个代码」（不以 `/` 开头），系统检测到 ultraplan 关键词，把它转换成 `/ultraplan ...` 然后走 slash 路径。这个转换必须在分流决策之前发生，否则 ultraplan 逻辑就得被复制到普通 prompt 分支里。

注意 `preExpansionInput` 参数的作用：ultraplan 检测用的是「粘贴展开之前」的原始输入，防止粘贴内容里偶然包含「ultraplan」这个词触发误判。

## 「整理 → 分流」带来的一致性

这个顺序保证了一件事：**每个分支接手的，都是已经被整理过的输入**。

slash 分支拿到的 `imageContentBlocks` 已经是 resize 后的合规 image block，不需要再次处理。`attachmentMessages` 是已经从字符串里提取出来的结构化附件，不需要再次解析。bridge 检查的结果已经反映在 `effectiveSkipSlash` 里，slash 分支直接用这个值，不需要再判断环境。

每个分支都收到「干净的输入对象」，而不是一团还没拆包的混合物。

## 代价

入口热路径更长。即使是一条最简单的文本 prompt，也要经过图片处理（快速 no-op）、bridge 检查（快速 no-op）、ultraplan 检测（快速字符串检查）才能到达 `processTextPrompt`。

这些检查极快，但理论上不是零成本。Claude Code 的判断是：这点成本远小于「每个分支各自维护重复的前置处理逻辑」带来的维护负担。

---

*面试指导：被问到「为什么要这个执行顺序」时，核心论点是「后续分支共享前置处理结果」。具体举 slash 命令需要已处理好的 imageContentBlocks 这个例子，最直接说明为什么先整理后分流比先分流后整理更合理。*
