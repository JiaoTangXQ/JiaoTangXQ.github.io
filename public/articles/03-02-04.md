# 为什么输入链要先把这些东西整理好，再决定怎么分流

很多系统会先看到斜杠命令就立刻分支，再慢慢想办法把图片、附件、路径这些东西补进去。Claude Code 反过来做。它会先把图片处理好，把附件候选提出来，把 bridge 情况、ultraplan 关键词之类的特殊条件看清，再决定这一条输入究竟该走普通提示、Bash，还是 slash 命令分支。

这背后的原因很简单: 分流之前，系统还不知道自己面对的到底是一句什么样的话。要是过早分流，很容易把本来属于这轮输入的世界丢在门外。比如 slash 命令其实也可能需要带着图片块和附件消息一起进来；bridge 来源还可能临时改写 slash 是否允许执行；ultraplan 关键词甚至会把普通句子改写成另一条内部命令。先分流，后补世界，最后只会让每条分支都得各自再做一遍脏活。

Claude Code 选的是更稳的顺序: 先把这一轮输入身上的材料登记清楚，再决定要进哪一扇门。这样每个分支拿到的，都是已经被整理过的合格对象，而不是一团还没拆包的混合物。这种顺序感，正是它后面能同时支持普通提示、命令、图片和远端来源，而流程又不显得混乱的原因。

## 实现链
顺序上是先标准化图片和 pasted contents，再判断 bridge 安全和 ultraplan 关键字，再抽附件，最后才决定走 bash、slash 还是普通 prompt。也就是说，分流用的是“整理后的输入”，不是原始输入。

## 普通做法
直觉上更容易先看首字符或模式位，比如看到 `/` 就先扔进 slash parser，别的以后再补。

## 为什么不用
因为 slash、bridge 和 prompt 命令也需要看到已经规范化的图片块、附件和上下文；先分流再整理，后面每个分支都得各自补一套前置处理。

## 代价
入口热路径更长，读起来也不像简单 if/else；但这避免了同一套整理逻辑在三个分支里复制粘贴。
