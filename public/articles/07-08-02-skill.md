# MCP skills 和本地技能共用同一套解析器——血缘关系与安全隔离的平衡

## 问题：MCP skills 长得像本地技能，但它们可信吗？

从用户的角度看，MCP skills 和本地技能（`~/.claude/skills/`）的使用方式完全一样：都是 slash commands，都有 description 和 arguments，都可以在聊天里直接 `/skill-name` 触发。

但来源完全不同：本地技能是用户自己写的文件，MCP skills 来自远端服务器，服务器可能由第三方运营。

Claude Code 的解决方案是：共用解析器，但用 `loadedFrom` 标记做运行时差异处理。

---

## 共用解析器的实现

`mcpSkillBuilders.ts` 保存了对 `loadSkillsDir.ts` 里两个函数的引用：

```typescript
export type MCPSkillBuilders = {
  createSkillCommand: typeof createSkillCommand
  parseSkillFrontmatterFields: typeof parseSkillFrontmatterFields
}

let builders: MCPSkillBuilders | null = null

export function registerMCPSkillBuilders(b: MCPSkillBuilders): void {
  builders = b
}
```

`mcpSkills.ts` 里的 `fetchMcpSkillsForClient` 通过 `getMCPSkillBuilders()` 取出这些函数，用于把服务器返回的 skill 定义转换成 `Command` 对象：

```typescript
const { createSkillCommand, parseSkillFrontmatterFields } = getMCPSkillBuilders()

// 把服务器侧的 skill 描述翻译成 Command
const command = createSkillCommand({
  skillName,
  // ...来自服务器的 frontmatter 数据
  loadedFrom: 'mcp',     // ← 关键：标记为 mcp 来源
  source: 'mcp',
  baseDir: undefined,    // 没有本地目录
})
```

同一个 `createSkillCommand` 函数，但 `loadedFrom` 不同。

---

## loadedFrom: 'mcp' 的安全隔离

在 `createSkillCommand` 返回的对象里，`getPromptForCommand` 是一个闭包，捕获了 `loadedFrom`：

```typescript
async getPromptForCommand(args, toolUseContext) {
  // ...参数替换...

  // Security: MCP skills are remote and untrusted — never execute inline
  // shell commands (!`…` / ```! … ```) from their markdown body.
  // ${CLAUDE_SKILL_DIR} is meaningless for MCP skills anyway.
  if (loadedFrom !== 'mcp') {
    finalContent = await executeShellCommandsInPrompt(finalContent, ...)
  }

  return [{ type: 'text', text: finalContent }]
},
```

这个 `if (loadedFrom !== 'mcp')` 是一个硬编码的安全边界：MCP skills 的 Markdown 正文里，即使写了 `` !`shell command` `` 这样的 shell 执行语法，也不会被执行。

为什么？因为允许远端服务器在用户机器上执行任意 shell 命令是一个严重的安全漏洞。用户使用了一个 MCP 服务，服务的维护者更新了 skill 定义，加入了恶意 shell 命令——如果没有这个边界，用户每次使用这个 skill 都会在自己机器上执行那个命令。

## 资源、技能、资源工具的三角关系

MCP skills 的存在意义之一是"让服务器能力可以被模型以 slash command 方式调用"。但 skill 里可能引用该服务器提供的资源：

```markdown
<!-- MCP skill 正文（来自远端服务器）-->
分析这份报告：
1. 读取资源 report://latest
2. 提取关键指标
```

如果资源 `report://latest` 变了（`resources/list_changed`），这个 skill 的内容可能就不再准确了。

这就是为什么 `client.ts` 在处理 `resources/list_changed` 时，不只失效资源缓存，还要失效依赖这些资源的 skill 缓存（通过 getMcpSkillCommands 间接失效）。

---

## `getSkillToolCommands` 里的 MCP skills 过滤

```typescript
export const getSkillToolCommands = memoize(
  async (cwd: string): Promise<Command[]> => {
    const allCommands = await getCommands(cwd)
    return allCommands.filter(
      cmd =>
        cmd.type === 'prompt' &&
        !cmd.disableModelInvocation &&
        cmd.source !== 'builtin' &&
        (cmd.loadedFrom === 'bundled' ||
          cmd.loadedFrom === 'skills' ||
          cmd.loadedFrom === 'commands_DEPRECATED' ||
          cmd.hasUserSpecifiedDescription ||  // plugin/MCP 需要显式 description
          cmd.whenToUse),
    )
  },
)
```

注意：`loadedFrom === 'mcp'` 不在直接允许列表里。MCP skills 要出现在 `getSkillToolCommands` 的结果里，需要有 `hasUserSpecifiedDescription` 或 `whenToUse`——这意味着服务器提供的 skill 定义必须包含描述，否则不会暴露给模型。

这是一个隐性的质量门槛：没有描述的 MCP skill 不被模型看见。

## 面试指导

**"如何设计一个'信任分级'的插件系统，让不同来源的插件有不同的权限？"**

Claude Code 的方案是标记（tagging）而不是分层：

1. 所有来源的 skill 都经过同一套解析器（代码复用）
2. `loadedFrom` 标记记录来源，是对象本身的属性
3. 执行层根据 `loadedFrom` 决定允许哪些操作（shell 执行只对 non-mcp）

这比"为每种来源写独立的执行器"更灵活：新增来源只需要：a）给 `LoadedFrom` 类型加一个枚举值，b）在需要差异化处理的地方加一个 `if` 判断。

缺点：`if (loadedFrom !== 'mcp')` 这类判断分散在各处，修改时需要全局搜索。这是"标记方案"的通病——比起"运行时分层"，更难做全局的策略修改。

"用标记 + 条件判断"还是"用类继承 + 方法重写"来实现信任分级，是架构层面的真实取舍，面试时说出来会很加分。
