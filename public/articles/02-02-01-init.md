# 为什么有些设置必须在 init() 之前先读出来

很多程序会把“读取设置”当成一件随时都能补做的小事。Claude Code 不是。它在真正进入 `init()` 之前，就先把 `--settings` 和 `--setting-sources` 读出来，因为它很清楚: 设置不是给已经成形的系统换皮肤，而是在决定这套系统从一开始能看见什么、信什么、允许什么。

这背后的工程判断很重要。`init()` 里很早就会碰到安全环境变量、配置系统、远端受管设置、凭据与代理这些东西。如果前半段初始化是在“旧设置世界”里展开，后半段才突然知道真正该读哪份设置、允许哪类来源，那整条链就会变成一堆补丁。前面已经做出的判断，后面还得回头改。

所以 Claude Code 选择了更干净的做法: 在启动最早期先把会改写初始化视野的设置读出来，再让 `init()` 在这个已经确定的世界里工作。这不是早优化，而是先把世界边界画对。真正成熟的 agent，不会先盲跑一段，再问自己刚才到底该按哪套制度行事。

## 实现链
这页落到的实现链都是“身份先于环境”：设置来源、特殊入口、工作目录、模式或连接方式会先决定当前到底处在哪个世界，然后 `init()` 和后续装配才有正确前提。代码不是先搭一台通用机器再改身份，而是先把身份定准。

## 普通做法
更常见的写法是先进统一初始化，等大部分系统都起来以后，再根据参数或环境决定最后走哪条模式。

## 为什么不用那种做法
Claude Code 没这么做，因为对它来说，身份不是界面皮肤，而是会反过来改写环境、权限和能力世界的前提。先按错身份初始化，后面很多步骤都会偷偷站错坐标。

## 代价与回报
回报是环境基准统一。代价是入口看起来更像大路由表，而不像单线启动脚本。
