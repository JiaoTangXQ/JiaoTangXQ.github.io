# 第十三卷回收索引：连续性世界里 `compact boundary` 先承认历史已经被整理过

## 实现链
连续性世界里更关键的，不是整理本身。  
而是整理之后，系统要先承认历史已经经过整理，新的接缝才站得住。

这条线真正逼出来的，是第十三卷里关于 `compact boundary` 的判断。

这页实际回收的实现链是：
- `10-05-01` 把 `compact boundary` 写成新的历史接缝，这会继续支撑 `13-07-02`，说明整理不是抹掉历史，而是先承认历史已经被整理过。

好的连续性设计，不会假装历史没被整理。  
它先承认历史已经经过一次整理，再让 `compact boundary` 把这次整理变成新的可继续接手的边界。

## 普通做法
更普通的做法，是一发现上下文变长就立刻整理，或者把当前状态和长期记忆统统塞回主会话，省掉额外的插槽和外置存储。

## 为什么不用
Claude Code 不这么做，是因为整理太早会打断主回合，记忆不分层会把短期噪音和长期知识混在一起，最后让连续性系统同时失去可读性和可恢复性。

## 代价
代价是需要额外维护 compact boundary、回合后插槽、外置 memory 和记忆回灌链，系统会更厚。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
