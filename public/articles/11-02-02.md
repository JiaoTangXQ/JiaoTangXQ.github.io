# 架构对比：多个子 Store vs 单一 AppState，「同步地狱」究竟有多难解决？

## 一个假设的「分散 store」架构

先设计一个「干净分层」的替代方案，看它会产生什么问题。

```
PermissionStore { mode, toolContext }
TaskStore { tasks, foregroundedTaskId }
MCPStore { clients, tools, commands }
NotificationStore { current, queue }
RemoteStore { connectionStatus, sessionUrl, ... }
```

每个子 store 独立维护自己的状态，子系统之间通过事件总线通信：

```
PermissionStore.emit('mode-changed', newMode)
  → TaskStore.onPermissionModeChanged(newMode)
  → NotificationStore.clearPermissionRelatedNotifications()
  → RemoteStore.syncPermissionMode(newMode)
```

表面上很清晰，每个 store 职责单一，互不耦合。

---

## 同步地狱的五个维度

**维度一：顺序依赖**

`TaskStore.onPermissionModeChanged` 和 `RemoteStore.syncPermissionMode` 哪个先执行？如果 Remote 先收到通知，而 Task 还没更新，这个窗口期里的状态是不一致的。

**维度二：错误处理**

`RemoteStore.syncPermissionMode` 失败了（网络超时），应该回滚 PermissionStore 吗？还是等重试？这时候 UI 显示的是哪个状态？

**维度三：订阅管理**

哪个 store 应该订阅哪个 store？如果 NotificationStore 需要同时响应权限变化和任务变化，它需要订阅两个 store。订阅的生命周期怎么管理？

**维度四：派生状态**

「当前用户能做什么」这个信息，同时依赖 PermissionStore（当前模式）和 MCPStore（可用工具）。这个派生信息应该住在哪里？

**维度五：调试难度**

一个用户操作导致了 5 个子 store 的更新，在时间线上分散。你在 DevTools 里看到某个组件渲染异常，需要追溯所有相关的 store 变化，而且这些变化可能不是原子的——你看到的某个时间点的「快照」可能是半同步的中间态。

---

## AppState 怎么规避这五个问题

**单一 setState 入口**

```typescript
setState: (updater: (prev: T) => T) => void
```

权限模式的变化、任务的更新、MCP 工具的变化，全部通过同一个 `setState` 函数进入系统。更新是原子的——要么全部发生，要么全部不发生，不存在「半更新」状态。

**`onChangeAppState` 统一处理副作用**

所有副作用在状态变化之后，在同一个 diff 函数里按顺序执行：

```typescript
function onChangeAppState({ newState, oldState }) {
  // 权限模式同步
  if (prevMode !== newMode) {
    notifySessionMetadataChanged(...)
    notifyPermissionModeChanged(newMode)
  }
  // 模型持久化
  if (newState.mainLoopModel !== oldState.mainLoopModel) {
    saveGlobalConfig(...)
  }
  // 认证缓存清理
  if (newState.settings !== oldState.settings) {
    clearApiKeyHelperCache()
    // ...
  }
}
```

执行顺序确定，副作用不会因为「哪个 store 先收到事件」而有差异。

**派生状态靠 selector，不靠新 store**

需要「当前用户能做什么」这个派生信息？

```typescript
const canRunTools = useAppState(s =>
  s.toolPermissionContext.mode === 'default' &&
  s.mcp.tools.length > 0
)
```

selector 是纯函数，从同一份 AppState 里派生。不需要一个新的 `CapabilityStore`，不需要维护额外的同步逻辑。

## 什么情况下分散 store 更好

Claude Code 的方案不是放之四海皆准的。

**分散 store 更适合的场景：**

1. 模块之间真的高度独立，没有跨模块联动需求
2. 团队规模很大，不同团队负责不同模块，需要真正的隔离
3. 模块的状态需要在不同生命周期挂载和卸载（比如路由层级的局部状态）

Claude Code 的各个子系统之间的联动密度非常高，而且整个应用是单进程的（在终端里运行），团队不需要跨边界隔离，这让单一 AppState 的代价（「状态对象很重」）小于收益（「不需要同步」）。

## 面试指导

**CQRS（命令查询职责分离）在这里的体现是什么？**

写入通过 `setState` 发生，读取通过 `useAppState(selector)` 发生。两条路径完全分离。`useSetAppState()` 返回的是纯粹的写句柄，不附带任何读操作。

**事件源（Event Sourcing）和这个模式有什么区别？**

Event Sourcing 把「发生了什么」存储为不可变的事件序列，当前状态从事件序列派生。AppState 存储的是「当前状态」，不保留历史。事件源对需要完整审计链的场景更合适，但实现复杂度更高。

**如果 AppState 将来需要支持「撤销/重做」，应该怎么改造？**

把 `setState` 改成先把旧状态推进历史栈，再执行更新。结合 selector 的按需读取，大多数组件不需要修改。困难点在于：带函数类型的字段（比如 `tasks`）的历史状态可能持有已经失效的闭包。这是一个真实的工程难题，没有完美解法。
