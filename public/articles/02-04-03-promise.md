# 预连接和远端设置 promise 为什么要提前埋下

`init()` 里有一类动作特别能体现 Claude Code 的时间观: 它不是等某项能力真正被用到时才临时起步，而是提前把等待埋进启动过程里。最典型的两种就是 API 预连接，以及远端设置、policy limit 这类加载 promise 的提前初始化。

前者很好理解。真正发出第一次 API 请求之前，TCP 和 TLS 握手往往就要吃掉一大截时间，所以 Claude Code 先把连接热起来，让后面真正进入动作链时少一次肉眼可感的停顿。后者更成熟。远端设置和 policy limit 并不是马上就要展示给用户的东西，但很多后续系统可能会等它们，因此 `init()` 先把 promise 建好，还专门保证即使后面某条路径没有真的去加载，也不会把别的系统永远卡死。

这说明 Claude Code 对“提前”这件事的理解很细。它提前埋的不是结果，而是等待点。真正好的启动工程，不是把所有事提前做完，而是把未来必然要遇到的等待，尽量提前藏进系统已经在运转的时间里。

## 实现链
这页对应的实现链都落在 `init.ts`：配置系统、安全环境变量、CA 证书、代理、平稳退出、预连接和远端 promise 会在真正会话启动前先被装起来。它关注的是运行环境，而不是马上给用户一段对话。

## 普通做法
更常见的写法是把环境准备和会话准备混在一起，等需要网络、证书或清理时再临时处理。

## 为什么不用那种做法
Claude Code 没这么拖后，是因为这些基础设施一旦错位，后面的会话层每走一步都可能踩雷。它宁可先把地基打厚。

## 代价与回报
回报是会话代码不必反复兜底底层问题。代价是 init 阶段要先处理大量用户暂时看不见的东西。
