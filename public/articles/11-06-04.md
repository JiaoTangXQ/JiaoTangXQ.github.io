# 面试题：TaskListV2 如何从 teamContext 和 appStateTasks 两个数据源建出 teammate 状态映射？

这道题测试你能否追踪跨数据源的信息流，以及理解为什么需要这种合并逻辑。

## 两个数据源的职责

**teamContext**：静态配置层

```typescript
const teamContext = useAppState(s => s.teamContext);
```

包含 teammates 的配置信息：名字、颜色、角色等。这是相对静态的，在会话开始时确定，期间不会频繁变化。

**appStateTasks**：动态运行层

```typescript
const appStateTasks = useAppState(s_0 => s_0.tasks);
```

包含后台任务的实时状态：每个 agent 当前在做什么、是否仍在运行。这是高度动态的，每次工具调用都可能更新。

这两个数据源的**颗粒度不同**：teamContext 是关于"这个 agent 是什么"（静态身份），appStateTasks 是关于"这个 agent 现在在做什么"（动态状态）。

## 第一步：从 teamContext 建颜色映射

```typescript
const teammateColors: Record<string, keyof Theme> = {};
if (isAgentSwarmsEnabled() && teamContext?.teammates) {
  for (const teammate of Object.values(teamContext.teammates)) {
    if (teammate.color) {
      const themeColor = AGENT_COLOR_TO_THEME_COLOR[teammate.color as AgentColorName];
      if (themeColor) {
        teammateColors[teammate.name] = themeColor;
      }
    }
  }
}
```

`AGENT_COLOR_TO_THEME_COLOR` 是一个颜色名到主题色的映射（把 agent 定义的颜色名转成 Ink/Yoga 能渲染的主题颜色）。这一步建出的是：`{ "researcher": "cyan", "coder": "green", ... }`

**为什么用 `teammate.name` 作为键，而不是 ID？**

因为任务的 `owner` 字段可能只写短名（`"researcher"`）不写完整 ID（`"researcher@team"`）。颜色映射要能匹配任务 owner，所以用短名作为键。

## 第二步：从 appStateTasks 建活动和在线状态映射

```typescript
const teammateActivity: Record<string, string> = {};
const activeTeammates = new Set<string>();

if (isAgentSwarmsEnabled()) {
  for (const bgTask of Object.values(appStateTasks)) {
    if (isInProcessTeammateTask(bgTask) && bgTask.status === 'running') {
      activeTeammates.add(bgTask.identity.agentName);
      activeTeammates.add(bgTask.identity.agentId);  // 双键！

      const activities = bgTask.progress?.recentActivities;
      const desc = (activities && summarizeRecentActivities(activities)) ??
                   bgTask.progress?.lastActivity?.activityDescription;
      if (desc) {
        teammateActivity[bgTask.identity.agentName] = desc;
        teammateActivity[bgTask.identity.agentId] = desc;  // 双键！
      }
    }
  }
}
```

关键点：**同时用 `agentName` 和 `agentId` 作为键**（双键映射）。

这是处理"同一个 agent 两种写法"问题的核心。下一篇会详细分析，但这里先理解：任务 `owner` 字段的值可能是 `"researcher"`（短名）也可能是 `"researcher@team"`（完整 ID），双键映射让两种写法都能查到。

## 第三步：渲染时的条件组合

```typescript
<TaskItem
  task={task}
  ownerColor={task.owner ? teammateColors[task.owner] : undefined}
  openBlockers={task.blockedBy.filter(id => unresolvedTaskIds.has(id))}
  activity={task.owner ? teammateActivity[task.owner] : undefined}
  ownerActive={task.owner ? activeTeammates.has(task.owner) : false}
/>
```

每个 TaskItem 接收：
- `ownerColor`：该 owner 的颜色（来自颜色映射）
- `activity`：该 owner 的当前活动摘要（来自活动映射）
- `ownerActive`：该 owner 是否仍在线（来自在线集合）

这三个数据来自两个不同的数据源（teamContext + appStateTasks），但被统一成 TaskItem 能消费的简单接口。

## 数据合并的时机

注意这个合并逻辑发生在组件的**渲染阶段**（不在 useEffect、不在 useMemo），每次渲染都重新计算。

为什么可以接受？因为：
1. `teamContext.teammates` 是静态的，重新迭代开销小
2. `appStateTasks` 的数量有限（通常是几个 agent，不是几千个）
3. 这些映射是在渲染时按需计算的"派生数据"，不需要缓存

如果未来 agent 数量极大，可以考虑用 `useMemo` 缓存颜色映射（它依赖静态 teamContext），但活动映射仍然需要每次渲染重算（它依赖动态 appStateTasks）。

## standalone 模式的总览层

```typescript
if (isStandalone) {
  return (
    <Box flexDirection="column" marginTop={1} marginLeft={2}>
      <Box>
        <Text dimColor>
          <Text bold>{tasks.length}</Text>{' tasks ('}
          <Text bold>{completedCount}</Text>{' done, '}
          {inProgressCount > 0 && <>
            <Text bold>{inProgressCount}</Text>{' in progress, '}
          </>}
          <Text bold>{pendingCount}</Text>{' open)'}
        </Text>
      </Box>
      {content}
    </Box>
  );
}
```

standalone 模式先给一行汇总，再列具体任务。注意 `inProgressCount` 的条件渲染——只有 `> 0` 时才显示。这避免了"0 in progress"的噪音（用户不需要被告知"没有进行中的任务"）。

## 面试指导

**直接问法**："TaskListV2 从哪里得到 teammate 的颜色和活动信息？"

答：两个数据源。颜色来自 teamContext（静态配置），活动和在线状态来自 appStateTasks（动态运行状态）。渲染时合并成 `teammateColors`、`teammateActivity` 和 `activeTeammates` 三个映射，供 TaskItem 使用。

**追问**："为什么要用双键映射（agentName + agentId）？"

答：任务的 owner 字段可能用短名也可能用完整 ID，来自不同的创建路径。双键映射保证两种格式都能查到对应的颜色和状态，避免"同一个 teammate 被识别成两个不同的人"的问题。

**架构追问**："为什么不把颜色和状态放在同一个数据源里？"

答：因为它们的更新频率和关注点不同。颜色是配置（相对静态），活动是运行时状态（高度动态）。分开存储符合"把不同变化速率的数据分开"的数据设计原则。合并是在展示层按需做的，不需要在数据层维护一个"merged state"。

**边界情况**："如果 teamContext 里有 teammate 但 appStateTasks 里没有对应的运行任务，会发生什么？"

答：颜色映射（来自 teamContext）仍然存在，活动映射和 activeTeammates（来自 appStateTasks）里没有这个 teammate。TaskItem 收到的 `ownerActive` 是 false，会触发 owner 名字不显示的逻辑（`showOwner` 条件要求 `ownerActive` 为 true）。这是正确行为：已经退场的 teammate 不应该显示"在线"标志。

---

*核心考察点：跨数据源的信息合并逻辑、双键映射的设计动机、渲染时派生 vs useMemo 缓存的权衡。*
