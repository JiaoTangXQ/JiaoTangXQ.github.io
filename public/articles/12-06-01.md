# 系统设计：Chrome Native Messaging 接入链为什么这么重？

## Chrome Native Messaging 的协议约束

Chrome Native Messaging 是 Chrome 提供的一种让扩展和本地程序通信的机制。它的设计比较严格：

1. 本地程序必须以一个 JSON 文件（manifest）注册，告诉 Chrome 在哪里找到它
2. manifest 必须放在特定目录（或 Windows 上的特定 registry key）
3. manifest 里的 `path` 字段必须是一个可执行文件路径，且**不能包含参数**
4. Chrome 通过 stdin/stdout 和 Native Host 通信
5. 每种 Chromium 浏览器（Chrome、Brave、Edge 等）有自己的 manifest 目录

这些约束是协议设计的结果，不是 Claude Code 自己造的麻烦。`setup.ts` 里的工程量，是这些约束的直接产物。

## setup.ts 的主要工作清单

`setupClaudeInChrome()` 函数的主体逻辑：

```ts
export function setupClaudeInChrome(): { mcpConfig, allowedTools, systemPrompt } {
  const isNativeBuild = isInBundledMode()

  // 1. 创建 wrapper script（因为 manifest path 不能带参数）
  void createWrapperScript(execCommand)
    .then(manifestBinaryPath =>
      // 2. 把 manifest 安装到各浏览器的 NativeMessagingHosts 目录
      installChromeNativeHostManifest(manifestBinaryPath)
    )
    .catch(e => logForDebugging(`Failed to install native host: ${e}`, { level: 'error' }))

  // 3. 返回 MCP 配置（这才是"工具 API"的部分）
  return { mcpConfig: { ... }, allowedTools, systemPrompt: getChromeSystemPrompt() }
}
```

注意这个函数的结构：工具 API 的配置（第 3 步）只是结尾的一小块，大头是前两步的基础设施准备。

`installChromeNativeHostManifest()` 内部还会处理：
- 对每个浏览器目录尝试写入 manifest
- Windows 下写 registry
- 检测 manifest 是否真的改变了
- 如果改变了且扩展已安装，触发 reconnect 页面

## 为什么要 wrapper script

浏览器的 native host manifest 格式里，`path` 字段是这样的：

```json
{
  "name": "com.anthropic.claude_code_browser_extension",
  "path": "/usr/local/bin/claude",
  "type": "stdio",
  "allowed_origins": ["chrome-extension://..."]
}
```

`path` 只能是一个可执行文件路径，**不能包含参数**。但 Claude Code 需要用参数区分不同的运行模式：普通 Claude Code、`--chrome-native-host` 模式、`--claude-in-chrome-mcp` 模式。

解决方案：创建一个 wrapper script，manifest 指向这个 wrapper：

```ts
async function createWrapperScript(command: string): Promise<string> {
  const platform = getPlatform()
  const chromeDir = join(getClaudeConfigHomeDir(), 'chrome')
  const wrapperPath = platform === 'windows'
    ? join(chromeDir, 'chrome-native-host.bat')
    : join(chromeDir, 'chrome-native-host')

  const scriptContent = platform === 'windows'
    ? `@echo off\n${command}\n`
    : `#!/bin/sh\nexec ${command}\n`

  // 只有内容变了才重写
  const existingContent = await readFile(wrapperPath, 'utf-8').catch(() => null)
  if (existingContent === scriptContent) { return wrapperPath }

  await writeFile(wrapperPath, scriptContent)
  if (platform !== 'windows') { await chmod(wrapperPath, 0o755) }
  return wrapperPath
}
```

wrapper script 的内容是 `exec /path/to/claude --chrome-native-host`（或带 cli.js 的非 bundled 版本）。manifest 指向这个脚本，脚本再调用真正的 Claude Code 二进制，传上所需的参数。

## 多浏览器目录铺设

`getAllNativeMessagingHostsDirs()` 返回所有需要铺 manifest 的目录：

```ts
// macOS 示例
~/Library/Application Support/Google/Chrome/NativeMessagingHosts/
~/Library/Application Support/BraveSoftware/Brave-Browser/NativeMessagingHosts/
~/Library/Application Support/Arc/User Data/NativeMessagingHosts/
~/Library/Application Support/Microsoft Edge/NativeMessagingHosts/
~/Library/Application Support/Chromium/NativeMessagingHosts/
~/Library/Application Support/Vivaldi/NativeMessagingHosts/
~/Library/Application Support/com.operasoftware.Opera/NativeMessagingHosts/
```

每个 Chromium 浏览器都有自己的 NativeMessagingHosts 目录，互相不共享。要支持所有这些浏览器，就必须给每个目录都写一份 manifest。

写入时的错误处理：

```ts
try {
  await mkdir(manifestDir, { recursive: true })
  await writeFile(manifestPath, manifestContent)
  anyManifestUpdated = true
} catch (error) {
  // 只记录日志，不失败 — 浏览器可能没安装
  logForDebugging(`Failed to install manifest at ${manifestPath}: ${error}`)
}
```

对每个目录的写入失败都只记录日志不抛出，因为很多浏览器用户不会全装。某个浏览器没安装，它的目录不存在，写入失败是正常的。

## anyManifestUpdated 的触发逻辑

```ts
let anyManifestUpdated = false

for (const manifestDir of manifestDirs) {
  const existingContent = await readFile(manifestPath, 'utf-8').catch(() => null)
  if (existingContent === manifestContent) { continue }  // 内容没变，跳过
  
  // 写入
  await writeFile(...)
  anyManifestUpdated = true
}

// 如果有任何 manifest 真的改写过
if (anyManifestUpdated) {
  void isChromeExtensionInstalled().then(isInstalled => {
    if (isInstalled) {
      void openInChrome(CHROME_EXTENSION_RECONNECT_URL)  // 触发 reconnect
    }
  })
}
```

这段逻辑有几个精心设计的地方：
1. 只有内容真的变了才算"更新"，不变就不算
2. 只有真的更新过，才触发 reconnect
3. 还要检查扩展是否安装，没装的话 reconnect 没意义

## 面试考察点

这道题考察的是对**协议约束驱动系统复杂度**的理解。

很多工程问题的复杂度不是"设计失误"，而是外部协议约束的必然结果。Chrome Native Messaging 的 `path` 不能带参数、每个浏览器有独立目录、Windows 用 registry——这些都是协议规定的，不是可以绕开的。

面对这类约束，工程师的选择是：
1. 只支持最简单的情况，其他情况让用户自己处理
2. 老实承受全部约束，把每种情况都处理到

Claude Code 选了第二条路，代价是 `setup.ts` 里大量看似"非核心"的基础设施代码。但这些代码正是让"真实用户的机器上能正常运行"的保证。
