# 真正的难点不在工具，而在让浏览器愿意接这条线

Claude in Chrome 这条链最费工程量的地方，不是工具 API，而是“怎么让 Chrome、Brave、Arc、Windows registry、Native Host manifest 和本地二进制都承认这条连接”。`setup.ts` 里能看到它在做的都是这种脏活：先决定是否 bundled，再生成 wrapper script，因为浏览器原生 host manifest 的 `path` 字段不能带参数；再把 manifest 安装到各浏览器的 NativeMessagingHosts 目录；Windows 下还得额外写 registry；如果 manifest 真变了，再视情况打开 reconnect 页面，提醒扩展重新接线。

更直觉的做法，是把责任全扔给浏览器扩展或者只支持单平台单浏览器。那样写起来轻，但一旦你想接进真实用户机器，就会立刻撞上平台碎片、路径差异和安装现实。Claude Code 明显不是没想到这些问题，而是从一开始就在认真清理这条接入链。

所以这里真正难的不是“能不能发一个点击命令”，而是“产品能不能在各种现实机器上稳定把桥搭起来”。这也是为什么第十二卷要花整整一章讲 manifest、wrapper、registry 和 socket: 真正的产品级能力，往往死在接入链，不死在工具函数。

## 实现链
`setup.ts` 的主体工作不是列工具，而是造 wrapper、写 native host manifest、扫描浏览器数据目录、处理 registry、再决定要不要打开 reconnect 页面。`chromeNativeHost.ts` 还要建 socket server、清理旧 socket、设权限。真正难的是把链路接通。

## 普通做法
普通做法是先把工具接口写出来，安装和连接问题放到文档里让用户自己解决。

## 为什么不用
Claude Code 不这么做，因为浏览器接入一旦靠人工拼装，真实用户现场会大面积掉线。

## 代价
代价是接入代码比业务动作更长，很多看似“不是 agent 核心”的平台细节都要自己背。
