# 灵魂持久化，骨架每次重新生成：防止配置编辑改写稀有度

`buddy/companion.ts` 里的 `getCompanion()` 有一个有趣的设计：

```typescript
export function getCompanion(): Companion | undefined {
  const stored = getGlobalConfig().companion;  // 读「灵魂」
  if (!stored) return undefined;
  const { bones } = roll(companionUserId());   // 重新生成骨架
  return { ...stored, ...bones };              // 合并，骨架覆盖 stored 里的骨架字段
}
```

每次调用，骨架都是从 `userId` 重新确定性生成的，不是从配置文件里读出来的。

## 灵魂和骨架的区别

**灵魂（soul）**：companion 的个性、名字、玩家对它的定制化选项——这些是用户选择的、应该跨 session 持久化的部分。

**骨架（bones）**：companion 的外形属性——稀有度（rarity）、物种（species）、眼睛样式、帽子、是否闪亮（shiny）、属性值……这些是根据 userId 确定性随机生成的。

`StoredCompanion` 类型（在 `buddy/types.ts` 里）只包含灵魂字段，不包含骨架字段。持久化存储的是灵魂，骨架每次运行时重新算出来。

## 为什么骨架不能持久化

如果骨架也存在配置文件里，用户可以直接编辑配置，把 `rarity: "common"` 改成 `rarity: "legendary"`，伪造一个本来不属于他们的稀有 companion。

骨架通过 `roll(userId)` 生成，`userId` 是绑定账号的，只要 userId 不变，每次 `roll` 的结果都一样（Mulberry32 伪随机数，确定性算法）。配置文件里没有骨架字段，所以编辑配置无法改变骨架。

这是一种简单有效的防作弊机制：持久化「应该由用户决定的部分」，在运行时重新计算「不应该由用户修改的部分」。合并时骨架后置覆盖，确保配置里哪怕有残留的旧骨架字段，也会被正确的当前骨架覆盖。
