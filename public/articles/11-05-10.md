# 面试题：statusLineShouldDisplay 的判断逻辑是什么？什么情况下整条状态栏会被关掉？

这道题不只是问"什么时候不显示"，更在考察你对"一个组件何时应该主动关闭自己"的设计直觉。

## 完整判断逻辑

```typescript
export function statusLineShouldDisplay(settings: ReadonlySettings): boolean {
  // Assistant mode: statusline fields (model, permission mode, cwd) reflect the
  // REPL/daemon process, not what the agent child is actually running. Hide it.
  if (feature('KAIROS') && getKairosActive()) return false;
  return settings?.statusLine !== undefined;
}
```

两个条件：
1. KAIROS（助手/proactive 模式）激活时直接返回 false
2. 其他情况下，只有用户在 settings 里配置了 `statusLine` 才显示

## KAIROS 激活时为什么整条隐藏

注释写得非常明确：**状态栏里的字段（model、permission mode、cwd）反映的是 REPL/daemon 进程，而不是实际运行的 agent 子进程**。

在 KAIROS 模式下，Claude Code 进入了一种"协调者-执行者"的双层架构：
- 外层：REPL 进程（用户交互层）
- 内层：实际执行任务的 agent 子进程

这时本地 REPL 进程的状态（model=claude-opus-4、cwd=/home/user、mode=default）可能和内层 agent 的实际状态完全不同。状态栏如果继续显示，用户看到的是外层进程的元信息，不是实际执行现场的状态。

**为什么不显示"外层 + 内层"两套信息？**

这当然可以做，但工程成本很高（需要 IPC 从 agent 子进程读状态），而且界面复杂度会大幅上升（用户需要理解"外层状态"和"内层状态"的区别）。在权衡之后，选择了更保守的方案：**整条隐藏，让用户去 agent 特定的视图里查看状态**。

## settings?.statusLine 的守卫

第二个条件是 `settings?.statusLine !== undefined`。这意味着状态栏是一个**可选的 opt-in 功能**：

- 默认情况下，`settings.statusLine` 未设置，状态栏不显示
- 用户在 `.claude/settings.json` 里配置了 `statusLine: { command: "..." }` 后才启用

这个设计的好处是：没有配置 hook 命令的用户，不会看到一条空白的状态栏行，也不会因为 hook 执行失败而看到错误信息。功能的存在对用户来说是透明的，直到他们主动选择启用。

## 关掉整条状态栏 vs 隐藏部分字段

这里的决策和前面"remote 模式隐藏 permission mode"是不同级别的操作：

| 级别 | 例子 | 原因 |
|------|------|------|
| 隐藏单个字段 | remote 模式不显示 permission mode | 该字段的数据不可信，其他字段仍然有效 |
| 整条隐藏 | KAIROS 模式下关掉整条状态栏 | 所有字段都代表了错误的执行现场 |

前者是"选择性剔除错误信息"，后者是"整层信息都失去了代表资格"。

## 关于 disableAllHooks 的边缘处理

在 `StatusLine.tsx` 的 mount 时还有一个关联处理：

```typescript
if (settings.disableAllHooks === true) {
  logForDebugging('Status line is configured but disableAllHooks is true', {
    level: 'warn'
  });
}
```

即使 `statusLine` 配置了，如果 `disableAllHooks` 是 true，hook 命令也不会执行。这里的处理是记录 warning 但不崩溃——用户可能不知道 `disableAllHooks` 会影响状态栏，warning log 给了一个诊断入口。

## 进度条的类似处理

终端进度条（`ProgressBar`）有类似的显示闸门：在 remote 模式和 proactive 模式下，本地终端的进度条会被关闭，因为本地进程不再是执行节奏的权威来源。

这形成了一个 pattern：**每个"状态显示组件"都要检查自己是否有资格代表当前执行现场**。如果没有，宁可整块隐藏，而不是继续显示可能误导的信息。

## 面试指导

**直接问法**："状态栏什么时候会整条不显示？"

答：两种情况。KAIROS/助手模式激活时：所有状态栏字段代表的是外层 REPL 进程而不是真正执行的 agent，整条隐藏。用户没配置 `settings.statusLine` 时：状态栏是 opt-in 功能，未配置时默认不显示。

**原理追问**："为什么 KAIROS 模式下不显示两套信息（外层+内层）？"

答：工程成本高（需要 IPC），界面复杂度高（用户需要理解两层状态的区别）。更保守的选择是整条隐藏，让用户去 agent 特定视图查状态。这是在"工程可行性"和"界面诚实性"之间做了保守但正确的选择。

**设计追问**："你认为什么时候应该隐藏单个字段，什么时候应该隐藏整个组件？"

答：如果某个字段的数据不可信，但其他字段仍然有效，隐藏单字段。如果整个组件所代表的执行现场都已经失效（比如外层进程 vs 内层 agent），隐藏整个组件。关键判断是"还有多少字段是有效的"——如果有效字段比无效字段多，局部隐藏；如果大部分字段都失效，整体隐藏更诚实。

**陷阱题**："statusLineShouldDisplay 里只有两行逻辑，太简单了吧？"

答：简单的接口背后通常是经过权衡后的精炼。这两行覆盖了两个完全不同的关闭原因（模式约束 + 配置 opt-in），把它们放在同一个函数里保证了调用方只需要一个判断点。复杂度在于判断这两种情况的**设计动机**，而不是代码行数。

---

*核心考察点：组件自我关闭的设计动机、opt-in vs 默认开启的取舍、执行现场代表资格的判断逻辑。*
