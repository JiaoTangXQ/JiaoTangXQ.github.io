# 真正进入 slash 分支之后，会发生哪一类翻译

真正进入 slash 分支以后，Claude Code 做的第一件事不是“执行命令”，而是翻译。它要先把用户眼里的 `/命令`，翻译成系统能接手的一类正式会话事件。

这层翻译至少有几步。先解析命令名和参数，确认这到底是不是系统认识的命令；再根据命令类型决定它究竟要变成哪一种东西: 是 `prompt` 型的会话装配，`local` 型的本地动作，还是 `local-jsx` 型的临时界面；随后把结果包成消息、系统输出、附件、权限变化，必要时再决定这一轮是不是要继续推给模型。也就是说，slash 分支的任务不是立刻做事，而是把“命令语言”转译成“会话语言”。

这也是为什么 `processSlashCommand()` 很重要。它站在用户入口和运行时之间，负责告诉整套系统: 这一条斜杠输入接下来该被理解成什么。好的命令系统从来不是靠命令名字本身生效，而是靠这层翻译把名字变成真实运行中的动作、材料和边界。Claude Code 的成熟，恰恰就在于它把这一步做成了一门正式工种，而不是随手写几个 `if/else` 就算了。

## 实现链
真正进 slash 分支后，系统不会直接执行一个回调，而是进入 `getMessagesForSlashCommand()`，再按 `command.type` 分成 `local`、`local-jsx`、`prompt`、`context:fork` 等不同路径，统一返回 `messages / shouldQuery / resultText / nextInput` 这组结果。

## 普通做法
更普通的命令系统会给每个命令一个 handler，handle 完直接打印或退出。

## 为什么不用
因为这里的命令有的只改本地状态，有的要临时渲染 UI，有的要生成一轮新的模型输入，有的甚至要开 forked agent。只靠一个简单 handler 返回字符串不够描述这些差异。

## 代价
命令层变得像第二个小运行时；但这正是统一外观下还能保留不同执行模型的原因。
