# 面试题：getMessagesForSlashCommand 返回的东西，是命令结果还是会话材料？

## 面试官追问

「slash 命令执行完，`getMessagesForSlashCommand` 返回什么？」

错误答案：「返回命令的执行结果字符串。」

正确答案：「返回一组会话材料，包括 messages、shouldQuery，以及可能的 allowedTools、model、effort。」

这个区别很重要。

## 函数签名

```typescript
async function getMessagesForSlashCommand(
  commandName: string,
  args: string,
  setToolJSX: SetToolJSXFn,
  context: ProcessUserInputContext,
  precedingInputBlocks: ContentBlockParam[],
  imageContentBlocks: ContentBlockParam[],
  _isAlreadyProcessing?: boolean,
  canUseTool?: CanUseToolFn,
  uuid?: string,
): Promise<SlashCommandResult>
```

`SlashCommandResult` 是 `ProcessUserInputBaseResult & { command: Command }`，包含：

```typescript
{
  messages: (UserMessage | AssistantMessage | AttachmentMessage | ...)[]
  shouldQuery: boolean
  allowedTools?: string[]
  model?: string
  effort?: EffortValue
  resultText?: string
  nextInput?: string
  submitNextInput?: boolean
  command: Command  // 是哪个命令产生了这个结果
}
```

返回值的形状和 `processUserInput` 本身的返回值几乎一样（多了 `command` 字段）。这不是巧合——slash 命令处理后，结果被直接传递给 `processUserInput` 的调用者，不需要再做转换。

## 三种 type，三种翻译

### `local-jsx`：翻译成一个等待用户交互的 Promise

```typescript
case 'local-jsx': {
  return new Promise<SlashCommandResult>(resolve => {
    const onDone = (result?, options?) => {
      // 等到用户完成 UI 交互后才 resolve
      void resolve({ messages: [...], shouldQuery: ..., command })
    }
    void command.load().then(mod =>
      mod.call(onDone, context, args)
    ).then(jsx => {
      if (jsx == null) return
      setToolJSX({ jsx, shouldHidePromptInput: true, isLocalJSXCommand: true })
    })
  })
}
```

`local-jsx` 不立刻返回，而是先把 JSX 挂到界面上，等 `onDone` 被调用（用户完成选择或关闭面板）才 resolve。从外层看，这和其他命令一样是 `await getMessagesForSlashCommand()`，但实际上它是一个等待用户交互完成的 Promise。

### `local`：翻译成一次本地执行的文本结果

```typescript
case 'local': {
  const mod = await command.load()
  const result = await mod.call(args, context)
  if (result.type === 'skip') {
    return { messages: [], shouldQuery: false, command }
  }
  if (result.type === 'compact') {
    return { messages: buildPostCompactMessages(...), shouldQuery: false, command }
  }
  // text result
  return {
    messages: [userMessage, createCommandInputMessage(`<local-command-stdout>${result.value}</local-command-stdout>`)],
    shouldQuery: false,
    command,
    resultText: result.value
  }
}
```

`local` 命令同步执行，结果写进 `local-command-stdout` 标签里，不触发后续推理（`shouldQuery: false`）。注意 `compact` 类型的特殊处理——`/compact` 命令会重写整个消息历史，`buildPostCompactMessages` 产出的是压缩后的新消息集合。

### `prompt`：翻译成一轮新的会话工作条件

```typescript
case 'prompt': {
  if (command.context === 'fork') {
    return await executeForkedSlashCommand(...)
  }
  return await getMessagesForPromptSlashCommand(command, args, ...)
}
```

`prompt` 命令不直接执行，而是进入 `getMessagesForPromptSlashCommand`，产出一组要注入当前会话的消息（技能正文、工具许可、附件），然后 `shouldQuery: true`，让主循环继续推进。

如果命令有 `context: 'fork'`，则进入 `executeForkedSlashCommand`，拉起一个独立子 agent。

## 为什么不直接执行然后打印

命令系统之所以不直接「执行然后打印」，是因为命令的结果有很多种形态需要被会话系统理解：

- `shouldQuery: true`：这次命令执行完，主循环还要继续
- `allowedTools`：这次命令激活了额外工具权限
- `model`：这次命令切换了推理模型
- `messages: []`：这次命令（`skip` 类型）不留任何 transcript

如果命令只返回字符串，这些信息就丢失了，外层就不知道接下来该怎么做。把命令结果包装成「会话材料」，才能让命令系统和会话系统无缝衔接。

---

*面试指导：被问到「slash 命令是怎么影响后续会话的」时，核心回答是「通过 shouldQuery 和 allowedTools 控制下一步，而不是通过返回值字符串」。这个区别说明了 Claude Code 的命令系统不是简单的 CLI，而是会话级别的状态管理。*
