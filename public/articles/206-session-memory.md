# 第十三卷回收索引：连续性世界里 session memory 只在阈值和自然停顿点更新

## 实现链
连续性世界里另一半更容易被忽略的，不是整理本身，  
而是记忆更新会不会抢在主回合秩序前面。

这条线真正逼出来的，是第十三卷里关于连续性机制也要服从主回合节奏的判断。  
session memory 不是随时乱写的旁路，它得跟着主回合的停顿点走。

这页实际回收的实现链是：
- `10-06-02` 把 `session memory` 限定在阈值和自然停顿点更新，这会在 `13-04-01` 里继续回收到“连续性机制也得先服从主回合秩序，而不是反客为主”。

成熟的连续性系统，不会把记忆更新当成随手就能插入的动作。  
它只在阈值和自然停顿点更新，让记忆跟着主回合节奏前进。

## 普通做法
更普通的做法，是一发现上下文变长就立刻整理，或者把当前状态和长期记忆统统塞回主会话，省掉额外的插槽和外置存储。

## 为什么不用
Claude Code 不这么做，是因为整理太早会打断主回合，记忆不分层会把短期噪音和长期知识混在一起，最后让连续性系统同时失去可读性和可恢复性。

## 代价
代价是需要额外维护 compact boundary、回合后插槽、外置 memory 和记忆回灌链，系统会更厚。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
