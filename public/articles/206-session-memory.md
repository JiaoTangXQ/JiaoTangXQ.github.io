# session memory 为什么只在阈值和停顿点更新

面试题版本：**"Claude Code 的 session memory 用于在会话间保留重要信息。为什么不在每轮对话结束后都更新 session memory，而要等阈值触发或停顿点？频繁更新有什么问题？"**

## session memory 是什么

`src/services/SessionMemory/sessionMemory.ts` 实现了跨会话的记忆系统。session memory 保存的是：
- 用户偏好和工作习惯
- 项目特定的背景信息
- 上次会话未完成的任务状态
- 模型需要记住的长期事实

这些信息在每次新会话开始时被加载到系统 prompt 里，让模型"记住"上次的上下文。

## 每轮都更新的问题

直觉上，每轮结束时都把新学到的信息写入 session memory 看起来更完整。问题是：

**写入成本不低**：session memory 的写入通常需要一次 LLM 调用（来决定什么值得写入、如何表述）。每轮都触发意味着每轮都有额外的 LLM 调用，这在活跃会话里会明显增加成本和延迟。

**噪音污染**：每轮对话都包含大量瞬时信息——当前正在讨论的代码片段、临时的报错信息、一次性的决策背景。如果每轮都写入，session memory 会很快被这些瞬时信息占满，真正的长期知识被稀释。

**写入时机不对**：主回合刚结束时，哪些信息"值得长期记住"并不总是明确的。可能要多轮之后，某个信息的重要性才变得清晰。过于频繁的写入容易把临时重要的信息错误地标记为长期重要。

## 阈值触发的逻辑

token 阈值触发 session memory 更新的逻辑：当上下文窗口使用量超过某个百分比（比如 80%），系统认为当前会话已经积累了足够多的对话，值得把其中重要的内容提炼出来写入 session memory。

这个时机的选择不是任意的：
- 阈值保证了只有会话积累了足够信息后才触发，避免了频繁的无效写入
- 发生在即将需要压缩之前，恰好是一个"提炼当前会话精华"的自然时机
- 与压缩流程协同：session memory 更新后，旧的对话历史可以安全压缩，因为重要信息已经保存

## 自然停顿点的逻辑

用户明确要求保存（比如"记住这个设计决定"）是一个自然停顿点。这类请求说明用户自己判断当前信息值得长期保存，这是一个比算法判断更可靠的信号。

另一个自然停顿点是会话结束前：如果用户主动结束会话（不是意外断开），系统可以在关闭前做一次最终的 session memory 更新，确保本次会话的重要内容不会丢失。

## "连续性机制服从主回合节奏"

这个原则和整理等回合结束的逻辑是同一类：**辅助机制的执行时机服从主工作的节奏，不能反客为主**。

session memory 更新是辅助机制，主回合（用户和 AI 的交互）是主工作。辅助机制的存在是为了让主工作跨会话保持连续性，而不是为了让辅助机制自身尽可能完整。

如果辅助机制的执行成本（每轮额外 LLM 调用、延迟）开始影响主工作的流畅性，那就违背了辅助机制的初衷。

## session memory 更新的内容选择

什么信息值得写入 session memory，什么不值得？

**值得写入**：用户的代码风格偏好（"prefer explicit type annotations"）、项目的架构决策（"所有 API 调用都要通过 service 层"）、反复需要的上下文（"测试数据库在 localhost:5432/test"）

**不值得写入**：当前 bug 的具体报错信息（下次会话可能已经解决了）、一次性的临时决策（"这次临时用 console.log 调试"）、过于细节的代码片段（context 里有更完整的版本）

阈值和停顿点触发的限制，让这个过滤判断有足够的信息积累才做，而不是每轮都仓促决定。

## 面试角度的总结

session memory 只在阈值和停顿点更新，体现的是"辅助机制不抢占主回合资源"的设计原则。频繁更新的代价（成本、延迟、噪音污染）会让辅助机制本身成为负担。阈值和停顿点的限制，在保证重要信息被保留的同时，把更新行为的时机让给了主工作流程来决定。
