# 连续性本身就是时间制度：为什么工作台需要记忆

面试题版本：**"Claude Code 有 session memory、compact boundary、task outputFile 等多种跨会话/跨回合的持久化机制。这些机制各自解决什么问题？它们共同构成了什么？"**

## 时间制度的含义

"时间制度"不是指管理时间本身，而是指**系统对时间维度的工作是有规划的**：不只是"现在能工作"，还要保证"明天接着能工作"、"从上次断点继续"、"追溯过去的决策"。

一个没有时间制度的工具：每次对话都从零开始，没有上下文，没有历史，每次都要重新建立背景。这样的工具虽然可以完成单个任务，但无法成为长期工作台。

Claude Code 选择成为长期工作台，这意味着连续性不是一个可选特性，而是基础设施。

## 三种连续性机制的分工

**Session memory**：跨会话的长期知识。用户偏好、项目背景、重要决策——这些需要在多次会话间持久保留。session memory 解决的是"下次启动时，系统还认识这个用户的工作上下文"的问题。

**Compact boundary**：单次会话内的历史压缩边界。当对话太长，需要压缩但不能丢失关键信息，compact boundary 标记了"这里之前的历史已经被摘要处理"，让后续推理知道自己站在哪个历史层次上。

**Task outputFile**：后台任务的持久化过程记录。任务在运行中，它的输出被写到磁盘，即使主会话断开，任务结果不丢失，下次可以接着读取。

这三个机制覆盖了三种不同的时间跨度：
- outputFile：任务级别（小时到天）
- compact boundary：单次会话内（分钟到小时）
- session memory：用户级别（天到月）

## 没有时间制度会怎样

**用户每次都要重新建立背景**："我们在这个项目里用 TypeScript strict mode，所有 API 调用要经过 service 层，测试用 Jest"——如果没有 session memory，用户每次会话都要重复这段说明。

**任务无法跨会话接续**：启动了一个长时间运行的分析任务，网络断了，重连后不知道任务跑到哪里了，只能重新跑。没有 outputFile，任务进度对用户不透明。

**压缩后失去方向感**：会话太长被压缩，但压缩后模型不知道"这是一个被压缩过的历史"，可能误以为对话从一开始就是摘要形式，对历史信息过于自信或过于谨慎。没有 compact boundary，连续性在压缩时悄悄断裂。

## 长期工作台的设计目标

"时间制度"这个框架的含义是：系统在设计时就假设工作不是一次性的，而是持续的、有历史的、需要接续的。

这个假设影响了很多设计选择：
- 为什么任务对象要有 `outputFile`（工作结果需要持久化）
- 为什么压缩后要留 `compact boundary`（历史需要可理解的层次）
- 为什么 session memory 要有主动触发机制而不是自动触发（控制连续性的质量）

这些机制合在一起，构成了 Claude Code 作为长期工作台的基础设施：不只是今天可以帮你工作，而是积累理解、建立信任、持续协作。

## 面试角度的总结

session memory、compact boundary、outputFile 这三个机制共同回答了"Claude Code 如何成为长期工作台"这个问题。它们从不同时间跨度上处理连续性：任务级（outputFile）、会话内（compact boundary）、跨会话（session memory）。这不是三个独立特性的堆砌，而是一套时间管理基础设施的三个组成部分，让工具可以积累上下文、可以接续工作、可以追溯历史。
