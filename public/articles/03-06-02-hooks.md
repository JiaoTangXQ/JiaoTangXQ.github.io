# 面试题：一个 skill 怎么在被调用后持续影响整个 session？

## 面试官追问

「一个 skill 被调用后，它的 prompt 内容注入当前轮次，之后还会影响后续的会话吗？」

答案取决于这个 skill 有没有声明 hooks。有 hooks 的 skill，影响力远超「当前轮次」。

## Skill 的 hooks 配置

`PromptCommand` 类型里有：

```typescript
export type PromptCommand = {
  type: 'prompt'
  // ...
  hooks?: HooksSettings  // ← skill 可以声明 hooks
  skillRoot?: string     // ← hook 脚本的根目录
  ...
}
```

`HooksSettings` 是和 settings 里的 hooks 配置相同的结构：

```typescript
type HooksSettings = {
  PreToolUse?: HookMatcher[]
  PostToolUse?: HookMatcher[]
  UserPromptSubmit?: HookMatcher[]
  // ...
}
```

一个 skill 的 frontmatter 可以这样写：

```yaml
---
name: security-review
description: Run a security-focused code review
hooks:
  PostToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./check-no-secrets.sh"
          once: true  # ← 触发一次后自动移除
---
```

## registerSkillHooks 的执行时机

在 `getMessagesForPromptSlashCommand` 里：

```typescript
const hooksAllowedForThisSkill =
  !isRestrictedToPluginOnly('hooks') || isSourceAdminTrusted(command.source)

if (command.hooks && hooksAllowedForThisSkill) {
  const sessionId = getSessionId()
  registerSkillHooks(
    context.setAppState,
    sessionId,
    command.hooks,
    command.name,
    command.type === 'prompt' ? command.skillRoot : undefined
  )
}
```

skill 被调用的瞬间（`/security-review` 被执行），hooks 被注册到 session 状态。

## registerSkillHooks 做了什么

```typescript
export function registerSkillHooks(
  setAppState,
  sessionId,
  hooks,
  skillName,
  skillRoot?,
) {
  for (const eventName of HOOK_EVENTS) {
    const matchers = hooks[eventName]
    if (!matchers) continue

    for (const matcher of matchers) {
      for (const hook of matcher.hooks) {
        const onHookSuccess = hook.once
          ? () => removeSessionHook(setAppState, sessionId, eventName, hook)
          : undefined

        addSessionHook(
          setAppState, sessionId, eventName,
          matcher.matcher || '',
          hook,
          onHookSuccess,
          skillRoot,
        )
      }
    }
  }
}
```

`addSessionHook` 把 hook 添加到 AppState 里。AppState 是 React 状态，所有依赖它的组件会响应更新。

从这一刻开始，`PostToolUse(Bash)` 事件触发时，`./check-no-secrets.sh` 就会被执行。这和 settings 里配置的全局 hooks 行为完全一样，只是作用域是当前 session，来源是 skill 调用。

## once: true 的一次性 hook

```typescript
const onHookSuccess = hook.once
  ? () => {
      logForDebugging(`Removing one-shot hook for event ${eventName} in skill '${skillName}'`)
      removeSessionHook(setAppState, sessionId, eventName, hook)
    }
  : undefined
```

`once: true` 的 hook 在成功执行一次后会通过 `onHookSuccess` 回调自动从 session 中移除。这对「需要在 session 启动时做一次检查，之后不再重复」的场景很有用。

## 权限门控

注意这一行：

```typescript
const hooksAllowedForThisSkill =
  !isRestrictedToPluginOnly('hooks') || isSourceAdminTrusted(command.source)
```

如果系统策略把 hooks 限制为「只允许 admin 信任的插件来源」（`isRestrictedToPluginOnly('hooks')` 为 true），那么普通用户 skill 即使声明了 hooks，也不会被注册。

这是防止不受信任的第三方 skill 通过 hooks 机制执行任意代码的安全边界。

## 压缩时的 hooks 恢复

`addInvokedSkill` 记录了技能调用，在 `/compact` 压缩会话时，`buildPostCompactMessages` 会检查哪些技能被调用过，重新注入对应的技能内容。

但 hooks 的恢复逻辑更复杂——`once: true` 的 hook 如果已经执行过，就不应该在压缩后再次注册。这部分细节在 hooks 恢复链里处理。

## 影响时间线

```
session 开始
  ↓
用户: /security-review
  ↓ skill 被调用
  ↓ registerSkillHooks 注册 PostToolUse(Bash) hook
  ↓ 技能正文注入当前轮次
  ↓ 主循环推进，模型开始推理
  ↓
  ↓（后续每次 bash 命令执行后）
  ↓ PostToolUse hook 触发，运行 ./check-no-secrets.sh
  ↓
  ↓（直到 session 结束，或 once: true hook 被触发一次后移除）
session 结束
```

一次 `/security-review` 调用，为后续整个 session 装入了监控机制。

---

*面试指导：被问到「skill 和 hook 的关系」时，核心是「skill 调用时可以注册 session 级 hooks，hooks 在 skill 调用后持续激活」。这是把 skill 从「一次性指令注入」升级为「会话级制度装入」的关键区别。*
