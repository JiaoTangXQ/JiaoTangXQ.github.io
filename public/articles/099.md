# Claude Code 的远端 bridge 为什么不做透明代理？

面试时被问到"你们的远端执行是怎么做的"，很多人的第一反应是：透明代理，让本地和远端看起来尽量一样。

Claude Code 做了相反的选择——不只是工程取舍，而是对"远端是另一层现实"的认识。

## 透明代理会让错误没有落点

透明代理的逻辑是：把边界藏起来，用户最好感觉不到远端和本地有什么区别。这样心智模型简单，工程上也像是在"把连接打通"。

问题在于，一旦跨边界，有四件事立刻成为新的问题域：

1. **权限谁裁**：本地权限模型在远端是否适用？谁有权批准远端执行的操作？
2. **消息怎么翻**：远端事件流和本地消息格式不一样时，谁来做翻译？
3. **连接断了怎么办**：session 中途断线，已经在跑的工作怎么处理？
4. **哪些东西不能带上路**：有些本地状态是不能透传到远端的，谁来把关？

这四件事如果被透明代理藏起来，就没有处理位置。错误会在最难收拾的时刻突然爆发，而不是在边界上被捕获。

## SpawnMode 是承认差异的第一步

`bridge/types.ts` 里的 `SpawnMode` 类型就三个值：

```typescript
type SpawnMode = 'single-session' | 'worktree' | 'same-dir'
```

注释里直接写了 `same-dir` 的风险："can stomp each other"（共享目录，session 之间会互相踩踏）。

把这个差异写进类型系统，而不是藏在实现里，是一个明确的设计信号：**远端不同模式之间有本质差异，不能假装它们是同一件事**。`worktree` 模式每个 session 独立 git worktree，`single-session` 模式一次只跑一个，`same-dir` 模式则主动接受了并发踩踏的风险。

三种模式，三种风险模型，不能统一成一个"透明远端"。

## 权限边界的处理位置

`bridge/bridgePermissionCallbacks.ts` 里的 `BridgePermissionCallbacks` 类型有一个字段特别说明问题：

```typescript
type BridgePermissionCallbacks = {
  sendRequest(requestId, toolName, input, ...): void
  sendResponse(requestId, response): void
  cancelRequest(requestId: string): void  // 这个
  onResponse(requestId, handler): () => void
}
```

`cancelRequest` 的存在说明了一个场景：bridge 连接断开，或者 session 提前结束，web app 那边已经弹出了权限提示框，必须有人去收回它。

如果做透明代理，这种"权限提示的生命周期和 session 生命周期不匹配"的情况根本没有处理位置。

## 消息验证在边界发生

`isBridgePermissionResponse` 是一个 type predicate，而不是 `as` 强转：

```typescript
function isBridgePermissionResponse(value: unknown): value is BridgePermissionResponse {
  if (!value || typeof value !== 'object') return false
  return 'behavior' in value &&
    (value.behavior === 'allow' || value.behavior === 'deny')
}
```

这个细节说明：**边界不只是物理连接的两端，更是信任断层的位置**。从远端传回来的数据是不可信的，必须在边界显式验证，不能用 `as` 假装它是已知类型。

## SessionHandle 是边界责任的具体化

`SessionHandle` 类型是远端 session 管理层的核心对象：

```typescript
type SessionHandle = {
  sessionId: string
  done: Promise<SessionDoneStatus>
  kill(): void
  forceKill(): void          // 正常退出和强制清除是两件事
  accessToken: string
  lastStderr: string[]       // ring buffer
  writeStdin(data: string): void
  updateAccessToken(token: string): void  // token 可以在 session 运行中刷新
}
```

`forceKill` 和 `kill` 分开，因为远端 session 的正常退出（等子进程完成清理）和强制清除（直接杀掉）是完全不同的操作，不能用同一个接口。

`updateAccessToken` 说明 token 会在 session 运行中刷新——连接不能依赖初始化时的静态凭据。如果做透明代理，这种"连接活着，但凭据已经换了"的状态没有处理机制。

## 面试答题要点

被问到"为什么不做透明远端"时，核心答案是：

**透明代理把问题藏起来，不代表问题消失了**。权限生命周期、消息验证、token 刷新、不同 session 模式之间的隔离——这些问题在边界上处理，比等到它们以奇怪的方式爆发出来要好得多。

桥接层厚，是因为边界上需要处理的事情本来就多。不是设计失败，而是工程诚实。

追问"怎么处理断线重连"时：`BridgeConfig.reuseEnvironmentId` 字段让重连时可以复用后端已有的 environment，而不是每次都新建。`BridgeApiClient` 接口里的 `reconnectSession` 方法在原来的 bridge 进程死掉后，还能把 session 重新排队到 environment 上——这是从一开始就设计进接口的能力，不是后来补的。
