# 为什么 deferred prefetches 更能暴露工程品味

很多系统在首屏出来以后就像松了一口气，仿佛启动已经结束了。Claude Code 不这么看。它在首屏之后还会继续做一大串 deferred prefetches：用户信息、用户上下文、system context、提示信息、云厂商凭据、文件统计、分析开关、官方 MCP 列表、模型能力、设置变更探测器、技能变更探测器，甚至事件循环卡顿探测。

这件事最精彩的地方，不是“它做了很多事”，而是它把这些事精确地放到了首屏之后。`startDeferredPrefetches()` 的注释写得很诚实：这些工作不会阻塞第一次渲染，但依然会争用 CPU 和事件循环，所以必须和真正的首屏路径切开。也就是说，它追求的不是盲目提前，而是把重活塞进用户正在打字的那段时间里。

这其实暴露了 Claude Code 对“启动”二字的理解。启动不是屏幕亮起来那一瞬间，而是系统从冷到热、从能用到顺手的整个过程。首屏前它克制，首屏后它继续替你预热世界。这样当第一轮真正开始时，很多本来会打断体验的等待，已经被悄悄消化掉了。

工程品味往往就藏在这种地方。差的系统只会追求一个漂亮的“已启动”时刻，好的系统会继续追问：用户下一秒最可能需要什么，我能不能先替他准备好。

## 实现链
这页对应的实现链都落在 REPL 交棒和 deferred prefetch 这套双节奏上：首屏前只做那些不做就没法安全进入 REPL 的事，首屏后再继续刷新、预热、建议和补充装配。代码在管理的不是单纯步骤，而是用户感知到的节奏。

## 普通做法
更普通的启动要么全部做完再显示，要么一股脑都异步开跑，谁先完算谁。

## 为什么不用那种做法
Claude Code 两种都没选死，因为前者太慢，后者太乱。它要的是先给用户可工作的前台，再让后台继续补齐。

## 代价与回报
回报是等待被重新分配。代价是启动完成这件事在代码里不再是单一时刻，而是一段前台和后台交错的过程。
