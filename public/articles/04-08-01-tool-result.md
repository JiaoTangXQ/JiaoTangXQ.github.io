# 为什么 tool_result 必须把一次动作真正闭环

在 Claude Code 里，`tool_use` 不是一句说完就算的旁白，而是一项已经正式发生的动作声明。既然动作已经发生，系统就必须给它一个结果。这个结果就是 `tool_result`。

如果没有这一步，后面的整轮思考都会悬空。模型已经伸手去做事了，却没人告诉它事情到底成功、失败，还是被拒绝；主循环也没法稳定地把这一轮接到下一轮，因为现场里还挂着一笔没有结案的动作。Claude Code 连内部注释都在反复强调：一条 assistant 轨迹如果出现了 `tool_use`，后面就必须有对应的 `tool_result`，否则整条轨迹都不完整。

所以 `tool_result` 的意义不只是“返回工具输出”。它更像一次正式结案。读文件也好，跑命令也好，开子任务也好，只要系统承认这次动作真的发出去了，就必须再回来把这件事说清楚。没有这一步，后面那些附件、恢复、继续理由都会失去地基。

这是一种很硬的运行时纪律。也正因为 Claude Code 守住了这条纪律，它的主循环才不会越跑越像一堆半截动作。

代码里，`query()` 会显式检查 `tool_use` 后面有没有对应的 `tool_result`，中断、fallback 或异常路径还会主动补 `yieldMissingToolResultBlocks()`，就是为了不让一条动作链悬空。更普通的实现是把工具结果只存在内存状态里，或者直接把结果拼回 assistant 文本里，省掉一层结构化消息。

Claude Code 坚持保留正式的 `tool_result`，是因为它面对的不是一次性网页对话，而是能恢复、能 compact、能继续下一轮的长期 transcript。只靠内存或自然语言摘要都太脆弱。代价是 transcript 更重，消息类型也更多，但这份重量换来的就是“动作真的闭环了”。

## 实现链
系统要求 `tool_use` 后面必须有对应 `tool_result`，否则会主动补齐。

## 普通做法
只在内存里记工具执行结果，或把结果揉回自然语言。

## 为什么不用
因为 transcript、resume 和下一轮推理都需要正式动作记录。

## 代价
消息更重、类型更多。
