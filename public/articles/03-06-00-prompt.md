# 面试题：一条 prompt 命令能改变当前会话的哪些维度？

## 导读

这一章（03-06）要先钉住一个判断：`prompt` 型命令改变的不是「模型收到了什么文本」，而是「这一段会话的工作条件是什么」。

具体涉及三件事：
1. `getPromptForCommand` 生成的不只是文本（03-06-01）
2. 技能正文、工具许可和 hooks 为什么要同时装进去（03-06-02）
3. prompt 命令为什么是命令系统里最像 agent 的部分（03-06-03）

## 五个维度

一次 `prompt` 命令调用可以同时改变：

**维度 1：会话内容（`messages`）**

技能正文被包装成 `isMeta: true` 的用户消息注入。模型在当前轮次会按技能正文的指导思考，用户 transcript 不显示这段指令。

**维度 2：工具权限（`allowedTools`）**

```typescript
const additionalAllowedTools = parseToolListFromCLI(command.allowedTools ?? [])
```

技能 frontmatter 里声明的 `allowedTools` 被解析，返回给 QueryEngine 用于本轮推理。某个 skill 可以临时开放 `computer_use` 或某个特定 MCP 工具，而不需要永久修改全局权限设置。

**维度 3：模型选择（`model`）**

```typescript
return {
  ...,
  model: command.model,  // ← 这轮推理用哪个模型
  ...
}
```

某些 skill 指定了专门适合的模型（比如快速任务用 `haiku`，复杂推理用 `sonnet`）。主循环会用这个字段覆盖默认模型选择，只影响这一轮。

**维度 4：推理深度（`effort`）**

```typescript
effort: command.effort
```

`effort: 'low' | 'medium' | 'high'` 控制推理深度。某些 skill 明确指定了适合的 effort 级别，避免简单任务浪费算力，或复杂任务被低 effort 处理。

**维度 5：Session hooks（持续影响）**

```typescript
if (command.hooks && hooksAllowedForThisSkill) {
  registerSkillHooks(context.setAppState, sessionId, command.hooks, ...)
}
```

hooks 一旦注册，在 session 结束之前都处于激活状态（`once: true` 的 hook 触发一次后自动移除）。一个技能调用可以为接下来的整个 session 装入额外的监控或增强行为。

## 为什么这五个维度同时改变是合理的

一个 skill（比如 `/security-review`）的完整语义是：

「用这个推理模型，在这个工具权限范围内，按这套 review 标准，对当前代码做安全审查，并且在每次 bash 命令执行后触发合规检查 hook。」

如果这五个维度要分五个独立步骤设置，用户体验会很碎片化。把它们打包进一个 `prompt` 命令，一次调用就完成了所有配置，这才是「技能是会话级工作方式」的体现。

## 与 `local` 命令的对比

`local` 命令执行完：

```
messages: [用户消息, 系统输出消息]
shouldQuery: false
allowedTools: 未变
model: 未变
hooks: 未变
```

`prompt` 命令执行完：

```
messages: [技能正文(isMeta), ...附件消息, command_permissions消息]
shouldQuery: true
allowedTools: 可能扩展
model: 可能切换
hooks: 可能新增
```

这不是同一件事的两种实现，而是两种本质不同的操作。`local` 命令做完一件事然后停下，`prompt` 命令装入一套工作条件然后让会话继续。

---

*面试指导：被问到「AI agent 的 skill 系统是怎么设计的」时，从「skill 同时改变五个维度」这个角度切入，说明这是「工作方式切换」而不是「功能调用」。这个区别体现了 agent 系统设计的深度。*
