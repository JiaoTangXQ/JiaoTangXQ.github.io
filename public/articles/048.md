# SkillTool：技能（Skill）在 Claude Code 里是怎么加载和执行的？

Skill 是 Claude Code 的可复用执行单元——本质上是一个带 frontmatter 的 Markdown 文件，可以包含提示词、shell 命令模板、参数声明。用户可以通过 `/skill-name` 调用，Claude 也可以通过 `SkillTool` 工具调用。

## 技能来源：多来源合并

```typescript
async function getAllCommands(context: ToolUseContext): Promise<Command[]> {
  const mcpSkills = context.getAppState().mcp.commands.filter(
    cmd => cmd.type === 'prompt' && cmd.loadedFrom === 'mcp',
  )
  if (mcpSkills.length === 0) return getCommands(getProjectRoot())
  const localCommands = await getCommands(getProjectRoot())
  return uniqBy([...localCommands, ...mcpSkills], 'name')
}
```

技能有多个来源：
- 本地 `~/.claude/skills/` 目录
- 项目级 `.claude/skills/` 目录
- bundled 内置技能
- MCP 服务器提供的 prompts（但只有 `loadedFrom === 'mcp'` 的，普通 MCP prompts 被过滤掉）
- Plugin 插件来的技能

来自 MCP 的技能需要特殊过滤：只包含明确作为"MCP skills"注册的（`loadedFrom === 'mcp'`），不包含普通 MCP prompts——注释解释了这样做的原因：在加这个过滤前，模型可以通过猜测名字（`mcp__server__prompt`）来直接调用本不可被发现的 MCP prompts。

`uniqBy` 去重：多来源可能有同名技能，以名字为 key 去重，本地优先于 MCP。

## executeForkedSkill：用 forked sub-agent 隔离执行

```typescript

```

技能执行是在 forked sub-agent 里完成的——不是在主对话流里直接注入提示词，而是启动一个独立的 agent，这个 agent 完成后把结果汇报回来。

这样设计的好处：技能的执行历史（中间思考、工具调用）不会出现在主对话的消息历史里，主 context window 只看到技能的最终输出。

## feature gate 的条件 require 模式

```typescript
const remoteSkillModules = feature('EXPERIMENTAL_SKILL_SEARCH')
  ? {
      ...(require('../../services/skillSearch/remoteSkillState.js') as typeof import(...)),
      // ...
    }
  : null
```

这里用了 `require()` 而不是 `import`——注释解释了原因：

> 静态 import 会把 `akiBackend.ts` 拉进来（通过 `remoteSkillLoader → akiBackend`），而 `akiBackend.ts` 有模块级的 `memoize()`/`lazySchema()` 常量，这些常量作为有副作用的初始化器会"幸存"过 tree shaking。

简单说：静态 import 无法被 tree shaker 完全消除（因为模块有副作用），用 `require()` 则只在需要时（`feature('EXPERIMENTAL_SKILL_SEARCH')` 为 true 时）才执行，完全避免了这个问题。

## 技能名的隐私分层

```typescript
logEvent('tengu_skill_tool_invocation', {
  command_name: forkedSanitizedName as AnalyticsMetadata_I_VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS,
  // _PROTO_skill_name 路由到 BigQuery 的特权列（不经过 redaction）
  // command_name 在 additional_metadata 里是 redacted 变体
  _PROTO_skill_name: commandName as AnalyticsMetadata_I_VERIFIED_THIS_IS_PII_TAGGED,
  // ...
})
```

对于自定义技能（非内置、非 official），`command_name` 字段被替换为 `'custom'`（`forkedSanitizedName`）——用户的自定义技能名可能包含敏感信息（项目名、功能描述），不应该出现在通用分析日志里。

`_PROTO_skill_name` 字段带有 `I_VERIFIED_THIS_IS_PII_TAGGED` 类型注释，路由到 BigQuery 的特权列，只有有相应权限的分析师才能访问。

## loadSkillsDir：Markdown 技能文件的解析

技能文件是 Markdown 格式，支持通过 frontmatter 声明：
- **参数**：`allowed_tools`、`model` 等
- **Shell 命令**：`!` 前缀的命令在技能执行前运行
- **Effort 值**：控制子 agent 的 token 预算

```typescript
export type LoadedFrom =
  | 'commands_DEPRECATED'
  | 'skills'
  | 'plugin'
  | 'managed'
  | 'bundled'
  | 'mcp'
```

`commands_DEPRECATED` 的存在说明了历史演进：之前叫"commands"，现在叫"skills"，旧的加载路径被标记为 deprecated 但还保留着做兼容。

## 参数替换和 shell 执行

技能的 Markdown 内容可以有模板参数：

```typescript
import { parseArgumentNames, substituteArguments } from '../utils/argumentSubstitution.js'
import { executeShellCommandsInPrompt } from '../utils/promptShellExecution.js'
```

`parseArgumentNames` 从技能内容里提取 `{{argument_name}}` 形式的参数，`substituteArguments` 在执行时把实际值替换进去。`executeShellCommandsInPrompt` 处理技能里用 `!cmd` 标记的 shell 命令（在把技能内容发给 model 之前就执行，把输出嵌入提示词）。

## 面试角度

SkillTool 揭示了 Claude Code 插件生态的工程逻辑：

1. **多来源的 uniq 合并**：本地技能优先于 MCP，这是一个明确的优先级决定，不是随机的
2. **forked 执行隔离**：技能执行不污染主对话历史，是保持主 context 干净的核心机制
3. **静态 import vs. 动态 require**：当模块有副作用初始化时，feature gate 必须用 `require()` 而不是 `import`，才能真正实现 dead code elimination
4. **隐私分层的分析**：`VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS` / `VERIFIED_THIS_IS_PII_TAGGED` 这些类型注释是强制性的，帮助防止 PII 泄漏到通用日志
5. **技能名的 sanitization**：用户自定义内容（技能名）不应该原样进入日志，这是 PII 处理的基本原则
