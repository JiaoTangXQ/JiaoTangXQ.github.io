# 面试题：Claude Code 远端权限桥接为何要合成假消息，而不是直接透传？

## 问题背景

Claude Code 支持远端执行模式（CCR，Cloud Code Runner）。在这个模式下，工具实际运行在云端容器里，但权限审批（比如"是否允许执行这个 bash 命令"）仍需要回到本地 UI 展示给用户确认。

一个初级的实现思路是：远端容器发来一个权限请求结构，本地直接把它塞给 `PermissionRequest` 组件渲染就行了。

Claude Code 没有这么做。在 `src/remote/remotePermissionBridge.ts` 里，它专门写了两个函数：

```typescript
export function createSyntheticAssistantMessage(
  request: SDKControlPermissionRequest,
  requestId: string,
): AssistantMessage {
  return {
    type: 'assistant',
    uuid: randomUUID(),
    message: {
      id: `remote-${requestId}`,
      type: 'message',
      role: 'assistant',
      content: [
        {
          type: 'tool_use',
          id: request.tool_use_id,
          name: request.tool_name,
          input: request.input,
        },
      ],
      // ...省略字段
    },
  }
}

export function createToolStub(toolName: string): Tool {
  return {
    name: toolName,
    isEnabled: () => true,
    userFacingName: () => toolName,
    // ...
  }
}
```

**面试题：为什么要多造这两个"合成对象"？不合成直接传不行吗？**

## 直接透传会发生什么

`ToolUseConfirm` 类型要求一个真实的 `AssistantMessage`。这不是偶然的——本地审批链的整个工作流都假设它在处理一条"AI 已经提出了工具调用请求"的正常消息。

如果直接透传远端容器的原始 `SDKControlPermissionRequest`，会有以下问题：

**问题一：类型体系崩了。** 本地 `ToolUseConfirm` 接口声明了 `AssistantMessage` 字段，如果传入一个远端结构，TypeScript 会在编译期就拒绝。这不是技术问题，是架构信号——审批链的契约不接受非标输入。

**问题二：审批链假设工具是本地已知的。** 本地的权限处理器（`PermissionContext.ts`）在判断是否允许一个工具调用时，会查找工具定义、读取工具的 `userFacingName()`、记录 analytics 日志。远端 MCP 工具在本地根本不存在定义，如果不造一个 stub，整个查询链会在中途 `undefined`。

**问题三：消息格式是跨层的通用语言。** Claude Code 的整个消息体系（`AssistantMessage` / `UserMessage`）是前后端、本地远端之间的公共协议。保持这个协议的完整性，意味着任何位置的代码都能用同一套逻辑处理消息，不需要"这是远端来的，走特殊路径"。

## 合成对象的设计意图

`createSyntheticAssistantMessage` 做的事情，不是欺骗系统，而是**把远端世界的请求翻译成本地世界能理解的语言**。这和口译员的工作一样：输入是一种语言，输出必须是听众能处理的另一种语言，但意义不能丢。

关键字段的映射很直接：
- `request.tool_use_id` → `content[0].id`（工具调用 ID，用于后续回调路由）
- `request.tool_name` → `content[0].name`（工具名，用于权限查找）
- `request.input` → `content[0].input`（输入参数，用于展示给用户确认）

`model`、`stop_reason`、`usage` 等字段设为空值——这些在审批链里不需要，合成时也没有真实值可填。这是刻意的：不硬造假数据，只填必须有值的字段。

`createToolStub` 同理。它的目的是让工具查找不 crash，同时把"这是一个远端工具，本地没有完整定义"的现实诚实地表达出来——`renderToolUseMessage` 会把 input 的 key-value 直接展示，而不是调用一个不存在的自定义渲染器。

## 跨边界设计的核心判断

这两个函数背后隐藏着一个设计判断：**远端不是本地的透明延伸，跨过边界的代价必须在边界层显式承担**。

如果透传，这个代价被推给了审批链的每个环节：每个地方都要判断"这条消息是本地的还是远端的"，每个地方都要处理工具可能不存在的情况。维护成本随时间累积，每次改审批逻辑都要想着"远端情况下会不会有问题"。

如果在 bridge 层合成，代价集中在一处：远端请求进来 → bridge 翻译 → 本地审批链拿到标准格式 → 一切照常运行。以后改审批链，不需要考虑远端；以后改远端通信，不需要了解审批链细节。

这个"翻译层集中承担跨界代价"的模式，在 Claude Code 里反复出现：`remotePermissionBridge.ts` 是其中最小但也最干净的一个例子。

## 面试要点

如果被问到这道题，核心答案是：

1. **类型契约**：本地审批链有强类型假设，远端结构不满足，必须适配。
2. **工具存在性**：远端工具本地可能没定义，stub 让查找链不 crash，同时诚实地降级展示。
3. **职责集中**：跨界翻译工作集中在 bridge，其他层保持简单，不需要处理"这是远端来的"特殊情况。
4. **协议统一性**：保持 AssistantMessage 作为全系统通用语言，任何地方拿到的都是同一种结构。

一句话总结：**bridge 不是在造假，而是在履行翻译职责，让两个不同语言的世界能对话，代价由 bridge 集中承担。**
