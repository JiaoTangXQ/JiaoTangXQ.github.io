# 为什么 getCommands() 在 worktree 模式下要等 setup() 完成后才能调用？

## 有条件的并行优化

在 `main.tsx` 里，有这样一段逻辑（简化）：

```typescript
// 非 worktree 模式：提前并行调用
if (!worktreeEnabled) {
  void getCommands(getProjectRoot())   // 提前启动，不等待
  void getAgentDefinitions()
}

await setup(cwd, permissionMode, worktreeEnabled, ...)

// worktree 模式下，只有在 setup() 完成后才能安全调用
if (worktreeEnabled) {
  void getCommands(getProjectRoot())   // setup 改变了 projectRoot
}
```

同一个 `getCommands()` 调用，在 worktree 和非 worktree 场景下的时机完全不同。为什么？

## getCommands() 依赖的是什么？

`getCommands(projectRoot)` 接受一个 `projectRoot` 参数，从这个目录加载命令定义：
- 读取 `${projectRoot}/.claude/commands/` 目录
- 加载 bundled skills（内置技能集合）
- 加载插件命令

在非 worktree 模式下，`projectRoot` 在 `setup()` 执行前后是相同的——用户在某个目录启动 Claude Code，这个目录就是 projectRoot，`setup()` 不会改变它。

在 worktree 模式下，`setup()` 会做这件事：

```typescript
setProjectRoot(getCwd())  // getCwd() 此时是 worktree 路径
```

`projectRoot` 从原始目录变成了 worktree 目录。如果在 `setup()` 之前调用 `getCommands(getProjectRoot())`，读取的是原始目录的命令定义；`setup()` 完成后，`projectRoot` 已经变了，之前缓存的结果指向了错误的目录。

## memoize 的问题：缓存了错误的结果

`getCommands()` 是 memoized 的——多次调用同一个 `projectRoot`，只执行一次实际加载。

这意味着：如果在 `setup()` 之前调用了 `getCommands('/original/path')`，缓存了这个结果，后来 `setup()` 把 `projectRoot` 改成了 `/worktree/path`，再调用 `getCommands('/worktree/path')` 会触发新的加载（因为参数不同）——但如果调用时传的还是旧路径，就会从缓存里拿到错误数据。

代码注释里有一处类似的警告：

```typescript
// Pre-load plugin hooks (consumed by processSessionStartHooks before render)
// Note: this must be done AFTER setup() in worktree mode to avoid
// caching the bundled skills from the wrong directory
```

## 为什么非 worktree 模式可以提前调用？

在非 worktree 模式下，`setup()` 的 `setProjectRoot()` 调用：

```typescript
// setup.ts 非 worktree 路径
setOriginalCwd(getCwd())
setProjectRoot(getCwd())
```

此处的 `getCwd()` 是在 `setCwd()` 之后调用的，返回的是 `cwd` 参数本身——和 `setup()` 调用之前 `getProjectRoot()` 返回的值相同（假设 `--cwd` 没有改变默认值）。

所以在非 worktree 模式下，`getCommands(getProjectRoot())` 在 `setup()` 前后传入的参数是相同的，提前调用没有问题。

## 这个优化节省了多少时间？

`getCommands()` 会加载：
- 所有 bundled skills（内置技能，定义在源码里）
- 项目自定义命令（从 `.claude/commands/`）
- 插件命令

这个操作在 SSD 上大约需要几十毫秒，在 HDD 或 NFS 上可能超过 100ms。通过提前并行调用，这段加载时间可以和 `setup()` 的其他操作（worktree 检查、session ID 生成等）重叠执行。

在 worktree 模式下，这个优化消失了。但 worktree 创建本身就是昂贵操作（git worktree add，可能几百毫秒），所以总体影响不大。

## 面试指导

这道题考查的是"并行优化的条件性"——不是所有情况下都能并行，并行的可行性依赖于特定前提。

**识别并行优化的三个前置问题**：
1. 操作 A 依赖的状态，在操作 B 执行前后会变吗？（如果会变，不能提前并行）
2. 操作 A 的缓存 key 是否包含了可能变化的状态？（如果包含，提前缓存的结果可能失效）
3. 操作 A 的结果是否会被多处消费？（如果是，任何一处用到错误缓存都是 bug）

能把这三个问题应用到具体场景（比如 worktree 切换），就能准确判断"什么时候并行是安全的"。Claude Code 的这段代码是一个很好的例证：同一个操作，在不同条件下有不同的安全性，需要有条件的并行策略。
