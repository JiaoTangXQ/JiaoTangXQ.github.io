# 为什么某些并行优化必须在 setup() 之后才能安全成立

Claude Code 明明很爱抢时间，却偏偏在 `setup()` 这里表现得格外老实。这不是保守，而是它很清楚: 有些并行优化只有在工作现场已经固定之后才安全，否则省下来的几十毫秒，最后会变成更难察觉的整条链路跑偏。

源码里这种克制写得很明白。主程序会在非 worktree 场景下，提前并行去拉 `getCommands()` 和 agent 定义；但只要 `--worktree` 可能触发 `process.chdir()`，这组并行就立刻让位，等 `setup()` 完成后再按新的 `cwd` 重新读取。`setup()` 自己也把一些预热工作挪到了后面，比如把插件 hooks 预加载和 `getCommands(getProjectRoot())` 放在工作现场确定之后再做，还专门注明早一点做会把 bundled skills 的结果缓存错。

这说明 Claude Code 追求的从来不是“并行越多越好”，而是“并行必须建立在正确世界上”。快只是结果，准才是前提。真正优雅的启动工程，不会为了漂亮的启动时间去赌后面所有路径都恰好没撞上现场变化。

## 实现链
这页对应的实现链都属于 `setup.ts` 那把刀：把前面抽象可运行的环境，真正切进当前 cwd、worktree、session、repo 和连接现实里。代码在这里开始回答“这一次到底在哪干活”。

## 普通做法
更常见的写法是把当前进程环境直接当现场，不再专门处理理论环境和真实工作现场之间的差别。

## 为什么不用那种做法
Claude Code 不敢这么偷懒，因为它经常要面对 worktree、resume、远端和 session 恢复这些现实差异。没有 setup，这些差异只会在后面各处偷偷出现。

## 代价与回报
回报是系统真的站到用户当前世界里。代价是 setup 对顺序和现场细节都很敏感，看起来更脏也更重。
