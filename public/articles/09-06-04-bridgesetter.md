# 接口能开多宽就开多宽？Claude Code 为什么刻意把权限桥限制在两个 setter

`leaderPermissionBridge.ts` 里有两个 setter：

1. `setToolUseConfirmQueue`：往权限确认队列里推请求
2. `setToolPermissionContext`：更新权限上下文

就这两个。不多。

在技术上，同样的注册机制可以把更多东西暴露出去：`setMessages`（往 Leader 的消息历史里插入消息）、`setCurrentModel`（切换 Leader 使用的模型）、`getAppState`（读取当前完整状态）……

为什么不暴露？

## "能做"到"随便做"的距离很短

软件工程里有一个经典观察：一旦某个接口开放了某种能力，它早晚会被使用，而且往往以设计者没预料到的方式被使用。

假设桥暴露了 `setMessages`。`inProcessRunner.ts` 最初只是想"在 Worker 完成时往 Leader 消息历史里记录一条日志"，这是合理的。但接下来：

- 开发者 B 想在 Worker 报错时往 Leader 消息里插入一条错误消息。
- 开发者 C 想在 Worker 完成时把整个工作摘要插入到 Leader 的对话历史，让 Leader 的 AI 直接看到。
- 开发者 D 想在 Worker 收到新任务时更新 Leader 界面的消息显示。

几个月后，`inProcessRunner.ts` 里充满了各种直接修改 Leader 消息历史的代码。执行循环和 UI 状态紧密耦合在一起。每次修改消息历史的逻辑都要同时考虑 Runner 里的影响。

**这不是恶意的滥用，而是"能做"自然演变成"这样做最简单"。** 只要接口存在，就会被使用；只要使用是方便的，就会被频繁使用。

## 最小权限原则的架构应用

最小权限原则（Principle of Least Privilege）在安全系统里常被引用：进程、用户、组件应该只拥有完成其任务所必需的权限，不多不少。

Claude Code 在接口设计上应用了同样的原则。执行循环需要做什么？

1. 推权限请求（需要 `setToolUseConfirmQueue`）
2. 同步新的权限规则（需要 `setToolPermissionContext`）

这两件事，全了。其他事情（修改消息历史、切换模型、读取 AppState）不是执行循环的职责，不暴露，防止未来被依赖。

这种约束是主动的架构决策，不是技术上做不到。

## 窄接口如何防止"耦合蔓延"

"耦合蔓延"是大型系统慢慢腐化的常见路径：两个模块之间的接口开始时是精简的，但随着需求增长，接口逐渐宽化，最后两个模块通过接口互相依赖了大量内部细节，变成了事实上的一个模块。

窄接口是防止耦合蔓延的最有效方式：**接口只开放 A 模块真正需要 B 模块提供的东西**，超出这个范围的需求，必须通过重新设计接口来满足（而不是顺手用现有接口搞定）。

这个"必须重新设计接口"的门槛，是有意识的摩擦力。不是阻止功能开发，而是强制每次接口扩展都要显式地思考：这是两层之间真正需要传递的信息，还是只是因为方便所以这么做？

## 当未来真的需要更多能力时怎么办

这是一个合理的问题。如果未来 Worker 需要往 Leader 的消息历史里插入一条汇报，怎么办？

不是扩展 leaderPermissionBridge 的接口。

正确的做法是：Worker 用 `SendMessage` 工具发一条消息给 Leader，由 Leader 的执行循环接收并决定是否要更新自己的消息历史。

这个流程更长，但结构更清晰：执行层的职责是执行，UI 层的职责是渲染；两层之间通过定义好的消息通道（mailbox）通信，而不是通过执行层直接修改 UI 状态。

桥的存在是必要的（权限确认必须实时打通两层），但桥的宽度应该是最小的（只传确实需要实时打通的信号）。

## 面试怎么答

如果面试题是"如何设计模块间的接口边界"：

**错误方向**：接口尽量全面，提供的能力越多，调用方越方便，接口越有价值。

**正确方向**：接口应该只暴露模块间真正需要传递的内容。每个额外暴露的能力都是一个潜在的耦合点，时间久了会成为实际的依赖。

**实践方法**：设计接口时，问"调用方必须通过这个接口做什么"，而不是问"调用方可能想通过这个接口做什么"。前者给出的是最小接口，后者给出的是宽泛接口。

**长期视角**：窄接口在短期看起来"不够用"，经常需要修改；宽接口在短期看起来"很方便"，但在长期会成为耦合的来源。权衡这两种代价，通常窄接口的长期成本更低——修改接口的成本（有限的地方需要更新）远低于解耦深度耦合的成本（可能影响整个系统）。
