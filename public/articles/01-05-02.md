# 代码发现：Claude Code 里有几十个"琐碎"的设计细节，把它们归进功能大类会丢失什么？

## 先找几个例子

在 `context.ts` 里：

```typescript
const shouldDisableClaudeMd =
  isEnvTruthy(process.env.CLAUDE_CODE_DISABLE_CLAUDE_MDS) ||
  (isBareMode() && getAdditionalDirectoriesForClaudeMd().length === 0)
```

这段逻辑说的是：在 `--bare` 模式下，如果用户没有用 `--add-dir` 显式指定任何目录，就跳过 CLAUDE.md 的自动发现。但如果用户显式指定了目录，照样发现。

紧跟着有一行注释：`--bare means "skip what I didn't ask for", not "ignore what I asked for".`

这个区分非常精确，而且对用户体验有直接影响：用 `--bare` 的用户通常是在做脚本调用，他们不想要任何他们没明确要求的东西，但如果他们明确指定了一个 CLAUDE.md 目录，他们当然还是想要那个目录被读取。

如果把这个细节塞进"命令行参数"功能章节，它就变成"`--bare` 模式下 CLAUDE.md 行为"这个无聊的功能描述。它真正包含的设计思考——"bare 的语义边界是什么"——就消失了。

## 另一个例子：2000 字符的截断

`context.ts` 里：

```typescript
const MAX_STATUS_CHARS = 2000
```

Git status 输出超过 2000 字符会被截断，截断后附上：`\n... (truncated because it exceeds 2k characters. If you need more information, run "git status" using BashTool)`

为什么是 2000 字符？这个数字对应的是"能让模型理解当前工作状态的合理信息量"和"不让 git status 占用过多上下文 token"之间的一个判断。太少（比如 500 字符），大型项目的 git status 会被严重截断，模型看不到全貌；太多（比如 10000 字符），一个 unstaged changes 很多的 repo 会把系统提示的一大块占满。

更重要的是截断消息的设计：不是静默截断，而是告诉模型"这里被截断了，如果你需要完整信息，用 BashTool 去查"。这把控制权交给了模型，让它自己决定是否需要更多信息，而不是由系统强行决定"2000 字符就够了"。

这个设计细节属于"上下文管理"功能章节，但如果它只是功能描述的一行（`git status 输出超过 2000 字符会被截断`），读者不会知道"截断消息会告诉模型如何获取更多信息"这个设计决策的存在，更不会理解这个决策背后的"把控制权给模型"的思路。

## 再一个例子：getGitStatus 的并行执行

```typescript
const [branch, mainBranch, status, log, userName] = await Promise.all([
  getBranch(),
  getDefaultBranch(),
  execFileNoThrow(gitExe(), ['--no-optional-locks', 'status', '--short'], ...),
  execFileNoThrow(gitExe(), ['--no-optional-locks', 'log', '--oneline', '-n', '5'], ...),
  execFileNoThrow(gitExe(), ['config', 'user.name'], ...),
])
```

这五个 git 命令被并行执行，而不是串行。`--no-optional-locks` 标志被加到 `git status` 和 `git log` 上，这个标志告诉 git 不要获取索引锁，避免在高并发 I/O 场景下因为锁等待而超时。

这是一个很小的性能细节，但它背后是"context.ts 的 getGitStatus 在每次会话开始时都会跑，是关键路径上的一个操作，应该被尽量优化"的判断。

按功能组织，这个细节可能出现在"性能优化"章节作为一个列表项，或者根本不出现。它真正展示的东西——"在关键路径上的每一行代码都应该被认真对待"这个工程品质——被功能大类吸收后就不见了。

## 功能章节遮住的是什么

当一本技术书只按功能组织时，它能告诉读者：

- 系统有哪些功能
- 每个功能的输入输出是什么
- 功能的配置选项有哪些

它不能告诉读者：

- 设计师在这里面对了什么具体挑战
- 这个实现方式和其他可能的方式相比，有什么具体优劣
- 为什么要在这里做这个看起来"多余"的处理
- 代码里的哪些选择是深思熟虑的，哪些只是默认值

这后面这些东西，恰恰是能帮助读者发展自己工程判断力的内容。知道"Claude Code 有 CLAUDE.md 发现功能"，和理解"bare 模式下 CLAUDE.md 的语义边界是如何被精确设计的"，是完全不同的认知层级。

## 面试指导

**考法**：写一段关于 Claude Code 上下文装配机制的文档（测试你能不能把实现细节转化成有价值的叙述）。

**满分策略**：不只描述功能，要描述设计决策。每说一个细节，紧跟一个"为什么这样设计，而不是更简单的方式"。比如 2000 字符截断不只是"有截断"，而是"截断并告知模型如何获取更多"是为了把控制权交给模型，这是整个系统"让模型有能力做出更好决策"哲学的一个局部体现。

**判断标准**：好的技术文档写完之后，读者不只知道"这个系统怎么用"，还知道"这个系统的工程师在做每个决策时的思考逻辑是什么"。这种理解才能迁移到读者自己的工程工作中。
