# MAX_VOICE_HINT_SHOWS = 3：新手引导该有终止条件

`PromptInputFooterLeftSide.tsx` 里有一行常量：

```typescript
const MAX_VOICE_HINT_SHOWS = 3;
```

语音功能的 footer hint 最多出现三次。每次显示后，`voiceFooterHintSeenCount` 写回全局配置并递增。等到第四次进入相关模式，提示已经不在了。

## 帮助的生命周期

新手引导的经典错误是没有终止条件。「为了确保用户看见」而永久保留提示，结果是：熟手用户每次打开都要看一遍他们已经学会的东西。帮助变成了打扰，打扰变成了噪音，噪音变成了视而不见的背景。

三次这个数字没有什么神圣之处——重要的是**存在上限**。系统在承认：用户会学习，学会以后不需要再被教。帮助的职责是完成传授，然后退场。

## 实现细节

`voiceHintUnderCap` 在 `ModeIndicator` 挂载时用 `useState(() => ...)` 初始化一次，避免提示在同一个 session 里因为另一个 Claude Code 实例增加计数而中途消失。

`useEffect` 里，在语音功能被激活的那一次，计数写入配置：

```typescript
saveGlobalConfig({ voiceFooterHintSeenCount: count + 1 });
```

下次打开时，如果 `voiceFooterHintSeenCount >= 3`，`voiceHintUnderCap` 初始化为 `false`，提示从一开始就不渲染。

## 一个更大的模式

这类「计数后退场」的设计在 Claude Code 里不止一处。排队消息可编辑的提示也有类似上限。可编辑提示也只出现有限次。

背后的原则是一致的：**驾驶舱空间有限，提示应该服从预算。** 新手引导占用的是稀缺的底部区域，它应该尽快完成自己的使命，把空间还给更需要它的状态信息。

一个会「主动消失」的帮助，比永远在场的帮助更有用。
