# 为什么副作用型工具永远不能被当作纯函数

很多系统一谈工具调用，脑子里默认的模型都是函数：给输入，得输出，最好还能并发。Claude Code 很明确地承认，真实工具世界不是这样。尤其是那些带副作用的工具，它们改的不只是结果，还可能改文件、改状态、改后续上下文，甚至改下一轮还能不能继续。

这就是为什么它对 `contextModifier` 特别谨慎。上下文修改不是谁先跑完谁先写回，而是被当成一种会改变世界的动作来处理。流式执行里，只有非并发安全工具的上下文修改才会被正式应用；普通工具编排里，也会把修改按批次和顺序收束回来。这里的潜台词很重要: 世界一旦被改了，后面的判断就不是在同一个世界里做出的。

所以副作用型工具永远不能被当作纯函数。把它们当纯函数，快是快了，但上下文会悄悄漂移，权限判断会变得暧昧，恢复路径也会越来越难讲清。Claude Code 宁可承认工具有“物理性质”，也不愿意用过度抽象把这种危险抹平。

优雅的系统不只是抽象得漂亮，还要在该承认现实脏乱的时候承认现实。

从代码看，这个判断不是只写在某篇设计文里，而是写进了 `Tool` 定义、`isConcurrencySafe()`、`interruptBehavior()` 和执行后的 `contextModifier` 机制里。也就是说，工具的结果不只是回一段文本，还可能改变后续可见工具、权限状态、消息上下文甚至 UI 进度。更普通的系统常把工具当 RPC：输入进去，字符串出来，主循环除了拼接文本什么都不用知道。

Claude Code 不敢这么乐观，因为它这里的工具早就不是“纯查询 API”了，而是会编辑文件、开子 agent、调整权限、接入真实设备的动作。把这种东西假装成纯函数，只会让恢复、并发和重试越来越假。代价则是工具协议更重，连“工具做完了”都可能意味着上下文要重写，不再是简单地收个返回值。

## 实现链
工具协议里有 `interruptBehavior`、`contextModifier` 和并发安全判断，承认工具会改世界。

## 普通做法
把工具当纯 RPC，只回字符串结果。

## 为什么不用
因为这里的工具会编辑文件、开子 agent、改权限和接真实设备。

## 代价
工具协议更重，测试也更难。
