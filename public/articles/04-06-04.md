# 工程取舍：工具协议为什么不能只是「输入→输出」？

最简单的工具协议是：

```typescript
type Tool = {
  name: string
  description: string
  inputSchema: z.ZodSchema
  execute(input: I): Promise<string>  // 返回字符串就行了
}
```

给输入，得到字符串输出，交给模型。很多 demo 系统就是这样的。

但 Claude Code 的 `Tool` 类型里有这些额外字段：

```typescript
type Tool<I> = {
  name: string
  inputSchema: z.ZodSchema<I>
  execute(input: I, context: ToolUseContext): AsyncGenerator<ToolProgress | ToolResult>
  
  isConcurrencySafe(input: I): boolean    // 这次调用可以并发吗？
  interruptBehavior?: 'cancel' | 'wait'   // 中断时怎么办？
  maxResultSizeChars?: number             // 结果大小限制
  backfillObservableInput?: (input: Record<string, unknown>) => void  // 外部可见输入
  
  // 执行后的上下文修改
  // 通过 ToolProgress 或 ToolResult 里的 contextModifier 字段传递
}
```

这些字段不是冗余的，每一个都在处理"纯函数抽象失效"的一种情况。

## contextModifier：工具会改变世界

`contextModifier` 是工具执行后返回的一个函数，用来修改 `ToolUseContext`：

```typescript
type ToolProgress = {
  message?: Message
  contextModifier?: {
    toolUseID: string
    modifyContext: (context: ToolUseContext) => ToolUseContext
  }
}
```

什么时候工具需要 contextModifier？当它的执行改变了后续工具执行所依赖的状态时：

- Agent 子工具（AgentTool）：启动一个子 agent，需要更新 `toolUseContext` 里的 agent 追踪信息
- 权限修改工具：改变了后续工具的权限决策依据
- CWD 变更：某些 Bash 命令会改变工作目录，影响后续文件路径解析

在**串行批次**里，contextModifier 在工具执行后立刻应用：

```typescript
// runToolsSerially
for await (const update of runToolUse(toolUse, ...)) {
  if (update.contextModifier) {
    currentContext = update.contextModifier.modifyContext(currentContext)  // ← 立刻应用
  }
  yield { message: update.message, newContext: currentContext }
}
```

在**并发批次**里，contextModifier 被排队，整批完成后按顺序应用：

```typescript
// runToolsConcurrently
for await (const update of runToolsConcurrently(blocks, ...)) {
  if (update.contextModifier) {
    queuedContextModifiers[toolUseID].push(modifyContext)  // ← 先排队
  }
  yield { message: update.message, newContext: currentContext }  // ← 上下文暂不更新
}

// 批次结束后按请求顺序应用
for (const block of blocks) {
  for (const modifier of queuedContextModifiers[block.id]) {
    currentContext = modifier(currentContext)  // ← 按序应用
  }
}
```

为什么并发批次不立刻应用？因为并发执行的工具完成顺序是不确定的，如果 B 的 contextModifier 在 A 完成前就应用了，A 的 contextModifier 再应用时，基于的是一个"已经被 B 修改过"的上下文。结果依赖于完成顺序，这是不可接受的不确定性。

## isConcurrencySafe：工具声明自己对并发的影响

前面已经分析过这个字段。关键点是：**这是一个工具对系统的承诺**，不是运行时检查。

工具作者必须认真考虑"这个工具的执行是否会影响其他工具的执行环境"，然后诚实地声明：

```typescript
// ReadFile：只读，不改状态
isConcurrencySafe(input: ReadFileInput): boolean {
  return true
}

// WriteFile：写操作，可能影响后续读
isConcurrencySafe(input: WriteFileInput): boolean {
  return false
}

// Bash：动态判断
isConcurrencySafe(input: BashInput): boolean {
  try {
    return isReadOnlyBashCommand(input.command)
  } catch {
    return false  // 解析失败保守处理
  }
}
```

如果工具作者声明错了（比如一个实际上有副作用的工具声明了 `true`），并发执行会产生不确定的结果，但系统不会在运行时发现这个错误——它只是会出现难以复现的 bug。

## maxResultSizeChars：工具对 token 预算的承诺

```typescript
maxResultSizeChars?: number
```

工具可以声明自己的结果大小上限。在工具结果预算系统里：

```typescript
// applyToolResultBudget 里
const excludedTools = new Set(
  toolUseContext.options.tools
    .filter(t => !Number.isFinite(t.maxResultSizeChars))
    .map(t => t.name),
)
```

没有声明 `maxResultSizeChars` 的工具（`!Number.isFinite`）被排除在预算控制外——系统对它们的结果大小没有任何限制。声明了的工具，结果会在 `applyToolResultBudget` 里被截断。

## interruptBehavior：工具如何响应中断

```typescript
interruptBehavior?: 'cancel' | 'wait'
```

当用户按 Ctrl+C（或系统触发 abort），正在执行中的工具会收到 abort signal。`interruptBehavior` 声明工具应该怎么处理：

- `'cancel'`：立刻停止，返回取消状态
- `'wait'`：等当前操作完成再停止（适用于不能安全中途停止的操作）

默认行为（未声明时）通常是 `'cancel'`。

## 代价：工具协议更重，测试更难

这套协议比"输入→输出字符串"的纯函数抽象要复杂得多。每个新工具的作者需要：

1. 认真考虑并发安全性并诚实声明
2. 如果工具会改变执行上下文，需要实现 `contextModifier`
3. 考虑中断行为（是否可以安全中途取消）
4. 可选地声明结果大小上限

这些额外的复杂度不是系统为了看起来高级而添加的——它们处理的是真实存在的问题。但对工具作者来说，入门成本确实更高。

## 面试指导

这道题在面试里的通用形式是：**"什么时候纯函数抽象会失效？"**

Claude Code 工具协议的答案是三类失效：
1. **状态副作用**：工具执行后改变了系统状态（contextModifier）
2. **并发干扰**：工具执行可能影响其他并发工具的结果（isConcurrencySafe）
3. **中断语义**：工具不能安全地在任意时刻停止（interruptBehavior）

能识别出这三类失效，并说出 Claude Code 的具体应对机制，是一个深度的回答。
