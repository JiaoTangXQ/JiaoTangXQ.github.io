# Claude Code 的团队架构：Leader 和 Teammate 在代码层面是什么关系？

Claude Code 允许多个 Claude 实例组成一个"团队"协同工作。从用户视角这很神奇，但从代码视角，这是一套有明确协议的分布式系统——有身份注册、通信信道、权限同步、断线恢复。

## 团队身份的建立

当一个 Claude 实例作为 teammate 启动时，`utils/swarm/reconnection.ts` 里的 `computeInitialTeamContext` 负责建立它的身份：

```typescript
export function computeInitialTeamContext(): AppState['teamContext'] | undefined {
  const context = getDynamicTeamContext()
  if (!context?.teamName || !context?.agentName) {
    return undefined  // 不是 teammate，返回 undefined
  }

  const { teamName, agentId, agentName } = context
  const teamFile = readTeamFile(teamName)
  
  const isLeader = !agentId  // 没有 agentId 意味着是 leader

  return {
    teamName,
    teamFilePath,
    leadAgentId: teamFile.leadAgentId,
    selfAgentId: agentId,
    selfAgentName: agentName,
    isLeader,
    teammates: {},
  }
}
```

注意 `isLeader = !agentId` 这个判断——Leader 没有 `agentId`（它本身就是 lead），Teammate 有分配给自己的 `agentId`。

这个函数被设计为**在第一次 render 之前同步调用**（注释里明确说明）——这样就不需要用 `useEffect` 在挂载后再初始化状态，避免了"初始状态为 undefined 然后一次 render 后变成真实值"的闪烁和竞态。

## 三种 Backend 支持多种终端环境

Teammate 需要在用户的终端里可见——这需要知道用户在用什么终端：

```typescript
// utils/swarm/backends/ 目录下
backends/
  InProcessBackend.ts    // 同进程运行，开发和测试用
  TmuxBackend.ts         // tmux 多窗口
  ITermBackend.ts        // iTerm2 多 tab/pane
  PaneBackendExecutor.ts // 通用 pane 抽象
  detection.ts           // 检测当前使用哪个终端
  registry.ts            // 后端注册表
```

`detection.ts` 在启动时检测终端类型，`registry.ts` 根据检测结果选择合适的 Backend。这是典型的策略模式（Strategy Pattern）——核心逻辑不依赖具体终端，Backend 提供统一接口。

## 权限同步：为什么 Teammate 需要 Leader 的权限决定

当 Leader 在主会话里接受了"允许这个工具的全部请求"，Teammate 执行同样工具时不应该再询问一次用户。`utils/swarm/permissionSync.ts` 处理这个同步：

```typescript
// permissionSync.ts 的职责
// Leader 做了权限决定 → 广播给所有 Teammates
// Teammate 收到广播 → 更新自己的 ToolPermissionContext
```

这里有一个微妙的安全考量：同步的是**允许规则**，而不是绕过所有检查。Teammate 仍然有完整的权限检查机制，只是它继承了 Leader 已经做过的决定，不需要重复询问用户。

## Mailbox：Teammate 之间的消息通道

```typescript
// context/mailbox.tsx
type TeammateMailbox = {
  agentId: AgentId
  messages: TeammateMessage[]
  unread: number
}
```

每个 Teammate 有一个 mailbox，Leader 可以向任何 Teammate 发送消息（比如任务分配），Teammate 也可以向 Leader 报告状态。这是异步的消息传递，不是同步的函数调用。

为什么不用同步调用？因为 Teammate 可能运行在独立的进程里（`TmuxBackend`），或者处于等待用户确认的暂停状态。消息传递比函数调用更健壮，支持断线重连。

## 断线恢复

`reconnection.ts` 里有一段关键注释：

```
- Fresh spawns: Initialize from CLI args (set in main.tsx via dynamicTeamContext)
- Resumed sessions: Initialize from teamName/agentName stored in the transcript
```

如果 Claude Code 崩溃后重启，它可以从会话记录里恢复团队上下文——知道自己叫什么、属于哪个团队、谁是 Leader。这让团队协作不会因为单个实例崩溃而整体失败。

## 面试角度

多 Agent 协作系统的设计是高级面试话题。Claude Code 的方案揭示了几个关键决策：

1. **Leader 收口而不是民主决策**：权限决定在 Leader 处做，广播给 Teammate。这避免了"多个 Agent 各自问用户"的混乱
2. **身份先于能力**：Teammate 启动时先确立身份（`isLeader`、`agentId`、`teamName`），再加载能力。身份决定权限边界
3. **消息传递 > 共享状态**：Mailbox 而不是共享内存，更适合跨进程、支持中断恢复的场景
4. **Backend 抽象屏蔽终端差异**：核心逻辑不知道用户在用 tmux 还是 iTerm2，Backend 层处理这个差异
