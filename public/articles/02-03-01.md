# 为什么显示帮助不该付出完整初始化成本

很多命令行工具有个常见坏习惯：哪怕用户只是想看一眼 `--help`，程序也会先把完整初始化跑一遍，网络、配置、插件、日志、环境检查一个不少。对作者来说这很省事，因为所有路径都走同一套逻辑；对用户来说却很粗暴，因为“看帮助”和“真正执行”根本不是同一种重量。

Claude Code 在这里用了一个很聪明但很朴素的闸门：Commander 外壳先把命令行结构搭起来，而真正昂贵的初始化被挂进 `preAction`。源码里的注释说得很直接，这样做就是为了只在真正执行命令时才跑初始化，而不是在显示帮助时也把整套世界都点亮。

这一步带来的好处远不只是快。它顺便把启动世界分成了两层：外层负责识别用户到底想干什么，内层负责在“确实要行动”时再去加载设置、初始化 sink、跑迁移、拉远端设置。于是帮助信息、参数说明和命令骨架可以保持很轻，而运行时副作用则被牢牢关在闸门后面。

这类设计很容易被低估，因为它不产生炫目的新功能。但如果没有这道闸门，Claude Code 这种体量的工具很快就会显得臃肿。真正好的启动工程，首先要学会区分谁只是来看地图，谁是真的要开车上路。

## 实现链
这页对应的实现链都落在 `preAction` 前后的职责切分上。代码把必须早做但很轻的判断留在闸门前，把完整环境、交互世界和重副作用留到闸门后，借这道线把不同成本的路径分开。

## 普通做法
更直觉的做法是把所有前置工作都塞进同一个大 handler，先后顺序只靠注释维持。

## 为什么不用那种做法
Claude Code 不这么混着写，是因为它既要处理帮助/版本这类轻路径，又要处理完整会话这类重路径。如果没有这道闸门，轻路径会一直给重路径买单。

## 代价与回报
回报是不同路径各付各的成本。代价是控制流被切成多阶段，入口阅读会更费心。
