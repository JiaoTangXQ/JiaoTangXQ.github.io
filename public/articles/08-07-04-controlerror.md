# 协议设计：遇到不认识的 control 请求，为什么必须立刻回 error？

## 两种处理未知消息的方式

当协议接收方收到一条不认识的消息时，有两种策略：

**静默忽略（Silent Drop）**：直接丢弃，不做任何响应。接收方继续运行，好像什么都没发生。

**明确失败（Explicit Failure）**：立刻发回一个错误响应，告知发送方"我不认识这条消息/无法处理"。

Claude Code 的 `handleControlRequest` 选择了明确失败：对于未知 subtype 的 `control_request`，立刻回复一个 `control_response` 错误包。

## 为什么静默忽略在这里是错的

`control_request` 是服务端主动发起的问询：

> 我想执行一个操作，你（客户端）是否批准？

服务端发出这个请求后，**会等待响应**。如果没有响应，服务端有超时机制（约 10-14 秒），超时后会关闭 WebSocket 连接。

静默忽略的问题：
- 服务端不知道是"客户端在处理中但还没回"，还是"客户端根本不认识这个请求"
- 服务端必须等满超时才能关闭，这段时间资源被占用
- 关闭连接后，错误原因是超时，而不是"不支持的请求类型"——调试时难以定位

明确失败的好处：
- 服务端立刻知道失败原因："unsupported subtype: xyz"
- 服务端可以立刻做出反应（优雅降级、报错、换方案）
- 不浪费超时等待时间
- 日志里有明确的错误记录，而不是神秘的超时

## "明确失败"是分布式系统的基本礼仪

在分布式系统里，有一个常被引用的原则：**让错误尽快、尽早地传播到关心它的地方**。

这和"fail fast"的哲学一致：与其让错误在系统内部静默累积，不如在发现时立刻暴露。

对于协议设计，这意味着：**不认识的消息，立刻告诉对方"不认识"，不要假装没收到**。

原因是：发送方最清楚"没有响应"意味着什么。如果接收方静默丢弃，发送方无法区分：
1. 消息在网络中丢失（超时重发）
2. 接收方正在处理中（继续等待）
3. 接收方不支持这个操作（应该报错）

立刻回 error，发送方立刻进入情况 3 的处理路径，不会在情况 1/2 上浪费时间。

## 前向兼容的代价

静默忽略的支持者可能会说：这样更有利于"前向兼容"（Forward Compatibility）——新版本服务端发出的新 subtype，老版本客户端只是忽略，不会报错，系统继续运行。

但在 `control_request` 场景里，前向兼容是一个虚假的安全感：

即使客户端静默忽略了未知的 `control_request`，服务端还是会等超时，等完超时还是会断连。表面上"兼容"了（没有立刻报错），实际上功能完全失效（服务端的操作没有得到授权响应）。

真正的前向兼容应该是：服务端在设计新 subtype 时，为不支持该 subtype 的老客户端准备降级方案，而不是依赖老客户端的静默忽略。

## 版本协商的正确做法

如果需要前向兼容，正确的做法是**能力协商**：

1. 客户端在连接时声明自己支持哪些 control 类型（版本/能力列表）
2. 服务端根据客户端的能力，只发送客户端能处理的 control 请求
3. 如果客户端不支持某个 subtype，服务端在连接建立时就知道，而不是在运行时才发现

这比"发出去看反应"更可靠，也更可预测。

（`initReplBridge.ts` 里确实有 `initialize` 类型的 `control_request`，用于在连接建立时同步能力信息，这就是能力协商机制的雏形。）

## 面试指导

"协议设计中，如何处理未知/不支持的消息类型"是一个涉及多个权衡的设计题。

答题框架：

1. **识别消息是否需要响应**：如果发送方会等待响应（问询模式），静默忽略是不可接受的；如果发送方是单向推送（广播模式），静默忽略通常没问题
2. **明确失败的信息价值**：告知"不支持"，比超时后无解释关闭连接，给调试提供了更多信息
3. **前向兼容的正确实现**：通过能力协商（capabilities negotiation）在建连时解决，而不是在运行时依赖静默忽略

Claude Code 的选择是：问询模式（`control_request`）下，立刻明确失败。这不是"不友好"，而是对发送方的尊重——让它尽快知道真实情况。
