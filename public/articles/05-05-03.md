# 为什么完成、失败和终止都先改状态再善后

任务收尾最怕顺序错乱。Claude Code 统一采用“先改状态，再做外围善后”的手法，是为了防止后续模块看见半真半假的现场。

## 实现链

`completeAgentTask()`、`failAgentTask()`、`killAsyncAgent()` 都先在 `updateTaskState()` 里把 `status` 改到终态，顺手写 `endTime`、清空控制器和 cleanup，然后才去 `evictTaskOutput()` 或交给通知层发消息。`InProcessTeammateTask` 的 kill 流程也一样，先把任务从运行态切出去，再去做团队文件和 SDK 事件清理。

也就是说，Claude Code 优先保证“系统视角下它已经结束”这件事成立，再去做外围扫尾。

## 普通做法

普通做法经常是先杀进程、先删文件、先发通知，最后才改状态。因为开发者脑子里“真正的动作”是清理，不是状态同步。

## 为什么不用

Claude Code 不敢这么排，是因为别的模块可能随时读状态。你如果先清理、后改状态，别的地方就可能在这段时间里把它当成还活着的任务，继续塞消息、继续展示、继续轮询。

先改状态，相当于先把门关上，再收拾屋子。

## 代价

代价是状态切换后的一小段时间里，外围清理可能还没完全完成。也就是说，系统会短暂处于“逻辑已结束、物理善后仍在进行”的阶段。但 Claude Code 显然更怕反过来的错位。
