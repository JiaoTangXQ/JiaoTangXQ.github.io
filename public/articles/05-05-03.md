# 先改状态再善后：completeAgentTask 的操作顺序为什么不能颠倒？

`LocalAgentTask.tsx` 里三个终止函数的结构几乎完全一样：

```typescript
export function completeAgentTask(result: AgentToolResult, setAppState: SetAppState): void {
  const taskId = result.agentId
  updateTaskState<LocalAgentTaskState>(taskId, setAppState, task => {
    if (task.status !== 'running') return task     // 幂等检查

    task.unregisterCleanup?.()                     // 注销清理钩子

    return {
      ...task,
      status: 'completed',                         // 1. 先改状态
      result,
      endTime: Date.now(),
      evictAfter: task.retain                      // 2. 设驱逐时间
        ? undefined
        : Date.now() + PANEL_GRACE_MS,
      abortController: undefined,                  // 3. 清空执行引用
      unregisterCleanup: undefined,
      selectedAgent: undefined,
    }
  })
  void evictTaskOutput(taskId)                     // 4. 最后清理磁盘输出
}
```

操作顺序：**先改状态 → 设驱逐时间 → 清空引用 → 清理磁盘**。这个顺序是刻意的，颠倒任何两步都可能造成问题。

## 为什么状态必须第一个改

`updateTaskState()` 是一个原子操作（React `setState` 回调内），整个函数里的状态变更一次性提交到 `AppState`。

在这个原子提交完成之前，其他任何地方读取 `task.status`，看到的都还是 `'running'`。一旦提交完成，所有读取 `task.status` 的地方立刻看到 `'completed'`。

这是"先改状态"的意义：**把"任务结束"这个事实，原子性地广播给整个系统**。

如果先做清理（先清空 `abortController`，先删磁盘文件），再改状态，就会有一段时间：任务对象说自己是 `'running'`，但执行资源已经没了。在这段时间里：

- 消息注入保护（检查 `status !== 'running'`）认为任务还活着，允许注入消息，但 `AbortController` 已经是 `undefined`
- 进度更新尝试写回任务状态，但任务在磁盘上已经没有输出文件了
- 框架轮询认为任务在跑，尝试读取输出，但文件已经被删

这些都是"半真半假的现场"——系统的物理状态和逻辑状态短暂不一致。

## 幂等检查的作用

```typescript
if (task.status !== 'running') return task
```

这一行让所有三个终止函数变成**幂等操作**。多次调用 `killAsyncAgent(taskId, setAppState)` 是安全的，第二次调用时任务已经是 `'killed'`，检查失败直接返回原对象。

这个幂等性在实践中很重要。多个路径可能同时尝试终止同一个任务：

- 用户点击 ESC
- 父任务被 kill，触发子任务连带 kill
- `cleanupRegistry` 里的全局清理钩子触发

没有幂等保护的话，这三个路径可能依次执行，任务的 `status` 被改了三次，`endTime` 被记了三次，`evictTaskOutput` 被调了三次。不幂等的副作用（特别是异步磁盘操作）在多次执行时会产生竞争。

## unregisterCleanup 的时机

```typescript
task.unregisterCleanup?.()
```

`unregisterCleanup` 是任务注册时从 `registerCleanup()` 拿到的注销函数。`registerCleanup` 把任务的清理钩子（kill 函数）注册到全局清理注册表，进程退出时统一触发。

在任务正常结束时，需要从注册表里注销这个钩子，否则进程退出时会再触发一次 kill（对一个已经完成的任务执行 kill，是无害的，但是冗余）。

这个注销在 `updateTaskState` 回调内部、状态提交之前执行。顺序上是：先注销钩子（防止进程退出时重复 kill），然后作为状态更新的一部分提交新状态。

## evictTaskOutput 为什么在最后

```typescript
void evictTaskOutput(taskId)   // 在 updateTaskState 之外调用
```

`evictTaskOutput()` 的作用是等待磁盘写入队列 flush 完毕，然后从内存 map 里删除这个任务的 `DiskTaskOutput` 对象。它是异步的（`Promise`），所以用 `void` 忽略返回值（fire-and-forget）。

为什么放在最后？因为在状态改变（通知通过 `enqueueAgentNotification` 发出）之后，框架读取 `outputFile` 路径，生成通知消息。这个读取必须在文件还存在的时候完成。

如果先 evict（从内存 map 删除 + flush），再改状态，再发通知，通知里的 `output_file` 路径就可能指向一个已经被清理的文件。

## fail 和 kill 的对比

三个函数的结构几乎相同，区别只在状态值和额外字段：

- `completeAgentTask`：`status: 'completed'`，有 `result` 字段
- `failAgentTask`：`status: 'failed'`，有 `error` 字段
- `killAsyncAgent`：`status: 'killed'`，还会调用 `task.abortController?.abort()` 中止正在进行的网络请求

`killAsyncAgent` 多了一步 `abortController.abort()`，这是唯一的差异——正常完成和失败时，执行本身已经结束了，AbortController 不需要再触发；kill 时，可能还有网络请求在进行，需要主动中止。

## 面试要点

**"先改状态"原则**在并发和分布式系统里很常见，叫法不同但本质一样：

- 数据库里叫 WAL（Write-Ahead Log）：先写日志再改数据
- 分布式系统里叫 two-phase commit 的 prepare 阶段：先宣布意图再执行
- 这里叫：先把逻辑状态改成终态，再做物理清理

核心思想相同：**逻辑终态一旦确认，就不应该再被推翻**。物理清理可能失败（磁盘 I/O 错误），可能延迟（异步），但逻辑状态的确认是原子的、立刻的。先确认逻辑终态，再容忍物理清理的不完美，是更健壮的选择。
