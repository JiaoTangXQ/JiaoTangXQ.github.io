# 第十三卷回收索引：连续性世界里整理推迟到回合结束后

## 实现链
连续性世界里最容易弄乱的，不是有没有整理动作，  
而是整理是不是抢在主回合前面发生了。

这条线真正逼出来的，是第十三卷里关于制度先于便利的判断。  
整理如果太早进场，主回合就会被迫一边做事一边收拾场面。

这页实际回收的实现链是：
- `10-09-01` 把整理动作推迟到回合结束之后，这会在 `13-04-01` 里继续变成“主回合没落稳前，任何整理都不该进场”。

成熟的连续性系统，不会把整理塞进主回合里抢节奏。  
它先让主回合落稳，再让整理在回合结束后进入。

## 普通做法
更普通的做法，是一发现上下文变长就立刻整理，或者把当前状态和长期记忆统统塞回主会话，省掉额外的插槽和外置存储。

## 为什么不用
Claude Code 不这么做，是因为整理太早会打断主回合，记忆不分层会把短期噪音和长期知识混在一起，最后让连续性系统同时失去可读性和可恢复性。

## 代价
代价是需要额外维护 compact boundary、回合后插槽、外置 memory 和记忆回灌链，系统会更厚。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
