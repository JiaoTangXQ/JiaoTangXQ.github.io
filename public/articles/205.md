# 整理为什么要等回合结束：主回合优先于上下文管理

面试题版本：**"Claude Code 的自动压缩（autocompact）会在什么时机触发？为什么不在检测到上下文快满时立即触发，而要等当前回合结束？中途触发会出现什么问题？"**

## 压缩时机的选择

`src/services/compact/autoCompact.ts` 里的 `calculateTokenWarningState()` 函数会在每次 token 计数更新时评估是否需要压缩，但即使评估结果是"需要压缩"，实际的压缩动作不会立即触发。

压缩触发的条件是：**当前回合的所有工具调用都已完成，模型的响应已经完整输出**。也就是说，压缩发生在两个回合之间的边界，而不是回合内部。

## 中途触发会出现什么问题

一个典型的回合可能包含：
1. 模型决定调用 3 个工具
2. 第一个工具（读文件）执行完成
3. 第二个工具（搜索代码）执行完成
4. **此时检测到上下文超限，如果立即压缩...**
5. 第三个工具（编辑文件）还没执行

如果在步骤 4 触发压缩：

**工具结果对不上**：压缩会重写消息历史，但第三个工具的 `tool_use` block 已经在消息里了，压缩后如果这个 block 被裁掉，后续的 `tool_result` 就找不到对应的 `tool_use`，违反了 API 协议（每个 `tool_result` 必须对应一个 `tool_use`）。

**模型推理中断**：压缩后的消息历史可能不包含前两个工具的结果，模型在第三个工具执行完成后无法基于完整的工具调用历史进行下一步推理。

**并发安全问题**：`StreamingToolExecutor` 正在管理工具的并发执行状态，中途修改消息历史会破坏执行器的内部状态。

## 回合边界作为安全压缩点

回合结束时，系统处于一个干净的状态：
- 所有 `tool_use` 都有对应的 `tool_result`
- 模型的响应已经完整
- `StreamingToolExecutor` 的执行账本已清空
- `transition` 机制已经决定下一步该做什么

在这个边界点触发压缩，不会破坏任何进行中的操作。`src/query.ts` 里的主循环在每次循环开始时会检查是否需要执行压缩（来自上一轮设置的标志），然后再开始新的模型调用。

## 整理和主回合的关系

"整理推迟到回合结束后"这个规则不是工程实现的限制，而是一个设计原则：**上下文管理操作服从于主回合的节奏，而不是打断它**。

如果整理可以在任意时刻抢占主回合，主回合就变成了"随时可能被打断的操作"，每一步都要考虑"这一步如果被压缩打断了怎么办"。这会显著增加主回合逻辑的复杂度。

把整理限制在回合边界，主回合内部可以放心假设"在当前回合里，消息历史不会被修改"。这个假设简化了工具执行、结果处理、状态管理的所有逻辑。

## 什么时候这个边界会被突破

有一种情况会打破这个规则：context collapse 和 reactive compact。

如果 token 超限非常严重（比如一个工具返回了极大的输出），主回合根本无法完成（模型调用会因为 prompt too long 失败）。这时，`contextCollapse` 会介入，尝试在当前回合内把消息历史裁减到可用范围，然后重试模型调用。

这是一种紧急补救，和正常的回合后压缩不同。它优先级更高，处理的是"主回合本身无法执行"的崩溃情况，而不是主动的维护性压缩。

## 面试角度的总结

整理等回合结束是一个优先级规则：主回合的完整性优先于上下文空间的实时优化。这让主回合内部的逻辑可以做简单假设，不需要防御中途被修改的消息历史。代价是压缩不能做到"刚好在超限时立即触发"，而是要等到当前回合安全结束，可能会出现一段时间的 token 消耗接近上限但还没触发压缩的窗口期。
