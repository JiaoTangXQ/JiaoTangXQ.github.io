# 对外统一、对内高效：MCP 接口层的语义守恒策略

面试题版本：**"MCP 的 Transport 接口对外统一，但内部有 stdio、sse、http、in-process 多种实现。这种设计的核心张力是什么？Claude Code 怎么解决的？"**

## 张力在哪里

统一接口和高效实现通常是对立的。完全统一的接口意味着最小公分母——所有实现都能支持的功能集，这往往比某种特定实现能提供的要少。完全为效率优化意味着每种实现都走特殊路径，调用方必须知道自己在用哪种。

Claude Code 在 MCP 层的解法：**接口层守语义，实现层选机制**。

## 接口层的内容

`src/services/mcp/types.ts` 里定义了 Transport 的枚举类型：

```typescript
export const TransportSchema = lazySchema(() =>
  z.enum(['stdio', 'sse', 'sse-ide', 'http', 'ws', 'sdk']),
)
```

但使用这些 transport 的 MCP client 代码只依赖 `@modelcontextprotocol/sdk/client` 的 `Transport` 接口，而不依赖具体类型。这意味着：

- 换 transport 类型不需要改 client 代码
- 测试可以用 in-process transport 替换真实的 stdio server
- 运行时可以根据配置动态选择传输机制

## 语义守恒的具体体现

"对外统一"不只是"接口一样"，更重要的是**语义一样**。这体现在几个地方：

**工具发现语义**：不管 MCP server 跑在哪里、通过什么传输，`listTools()` 返回的是同一种格式的工具描述。`src/services/mcp/normalization.ts` 负责把各种来源的工具描述规范化。

**调用语义**：`callTool(name, args)` 不管后端是子进程还是 in-process 对象，都以相同方式发起，等待 JSON-RPC 响应。

**错误语义**：连接断开、工具不存在、参数校验失败，在所有传输类型下产生相同形状的错误，上层可以统一处理。

这种语义守恒让 Claude Code 在扩展新的传输类型时，只需要实现 Transport 接口，不需要修改任何上层代码。当 `sse-ide` 这种特殊传输类型被加入时，所有工具调用、权限检查、日志记录代码都自动适用。

## 为什么不用更简单的方案

最简单的方案是只支持一种传输：比如全部用 stdio。这样代码最简单，不需要 normalization 层，不需要 transport 抽象。

问题是这个选择会锁死未来的演进空间：
- 内建工具想用 in-process 来减少延迟，做不到
- 远端 MCP server 想用 http，做不到
- IDE 插件想注入特殊的 sse 通道，做不到

更糟糕的是，一旦多个场景都有需求，你要么破坏原来的接口，要么在外部加很多 if-else 分支，每处调用都要先判断"我现在在用哪种传输"。

接口抽象的价值正是让这些扩展点**在接入层收口，而不是在所有调用点扩散**。

## 成熟工程里的常见取舍

这个模式在工程里很常见，但不总是正确选择：

**适合抽象的**：传输协议（各种实现本质相同）、存储后端（SQL/NoSQL/内存 cache）、通知渠道（email/webhook/push）。

**不适合过度抽象的**：性能临界路径上的代码（抽象层的虚拟分派有开销）、语义真正不同的操作（强行统一会丢失重要差别）。

Claude Code 在 MCP transport 这里的判断是：语义足够相同（都是 JSON-RPC 工具调用），性能开销不在临界路径（工具调用本身就是慢操作），值得抽象。

## 面试角度的总结

"对外统一、对内高效"不是矛盾，而是**在合适的层做合适的事**。接口层的职责是守住长期稳定的语义契约，实现层的职责是在当前约束下找到最高效的物理路径。Claude Code 通过 Transport 接口分离了这两个关切，让系统在保持接口稳定的同时，可以独立演进每种传输的内部实现。
