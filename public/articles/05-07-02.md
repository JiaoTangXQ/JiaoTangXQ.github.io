# 面试题：如何设计一个支持"超时自动降级"的前后台任务切换？

**系统设计题：** 你在设计一个 AI Agent 执行界面。当用户启动一个子 Agent 时，它先在前台显示（用户可以看到实时输出）。如果 Agent 运行超过 30 秒，自动转到后台继续运行（不打断 Agent，但用户界面不再独占显示）。用户也可以随时手动转后台。请设计这个机制。

这是 Claude Code 里 `autoBackgroundMs` 参数解决的真实问题。

## 需求分析

**功能需求：**
- 支持手动触发（用户按按钮）
- 支持超时触发（N 秒后自动）
- 两种触发方式效果必须相同
- 切换时不中断 Agent 的执行
- 前台切后台后，Agent 继续在后台跑，输出继续写入

**约束：**
- Agent 执行是异步的，可能跑几秒或几分钟
- 状态存在 React 风格的不可变状态树里
- Promise resolver 不能存进 React 状态

## Claude Code 的设计

**数据结构：**

```typescript
// 模块级 Map，存放所有前台 Agent 的 Promise resolver
const backgroundSignalResolvers = new Map<string, () => void>()

// LocalAgentTaskState 里只存一个布尔位，不存函数
isBackgrounded: boolean
```

**注册阶段（任务创建时）：**

```typescript
function registerAgentForeground({ agentId, autoBackgroundMs, ... }) {
  // 创建信号 Promise
  let resolveBackgroundSignal: () => void
  const backgroundSignal = new Promise<void>(resolve => {
    resolveBackgroundSignal = resolve
  })
  // resolver 外化到 Map
  backgroundSignalResolvers.set(agentId, resolveBackgroundSignal!)

  // 超时触发器
  let cancelAutoBackground: (() => void) | undefined
  if (autoBackgroundMs) {
    const timer = setTimeout(() => {
      triggerBackground(agentId, setAppState)  // 和手动触发走同一个函数
    }, autoBackgroundMs)
    cancelAutoBackground = () => clearTimeout(timer)
  }

  return { taskId: agentId, backgroundSignal, cancelAutoBackground }
}
```

**触发函数（手动和超时共用）：**

```typescript
function triggerBackground(taskId, setAppState) {
  // 1. 改状态
  setAppState(prev => ({
    ...prev,
    tasks: {
      ...prev.tasks,
      [taskId]: {
        ...prev.tasks[taskId],
        isBackgrounded: true,
      },
    },
  }))

  // 2. 触发信号
  const resolver = backgroundSignalResolvers.get(taskId)
  if (resolver) {
    resolver()
    backgroundSignalResolvers.delete(taskId)
  }
}
```

**执行循环（感知转场）：**

```typescript
const result = await Promise.race([
  runAgent(...),
  backgroundSignal.then(() => BACKGROUND_SENTINEL),
])

if (result === BACKGROUND_SENTINEL) {
  // 信号触发：释放前台控制，Agent 继续在后台运行
  return handleBackgrounded()
}
// 正常完成
return handleCompleted(result)
```

## 为什么手动和超时走同一条路径

这是这个设计最重要的原则：**两种触发方式共用同一个触发函数**。

在代码里，`autoBackgroundMs` 的 `setTimeout` 回调和手动调用 `backgroundAgentTask()` 做的事情完全一样：调用 `triggerBackground()` 函数，没有任何区别。

为什么这很重要？

假设手动触发和超时触发走不同路径：

```typescript
// 手动路径
function backgroundAgentTask() {
  setIsBackgrounded(true)
  resolver()
  cleanupResolver()
}

// 超时路径（如果单独写）
setTimeout(() => {
  setIsBackgrounded(true)  // 忘了 cleanupResolver？
  resolver()
  // 漏了 cleanupResolver()！
}, autoBackgroundMs)
```

两条路径，两个地方可能引入 bug，两个地方可能行为不一致。用户会遇到"手动转后台正常，自动转后台有 bug"的问题。

共用路径的好处不只是减少代码重复，更重要的是**结构性地保证一致性**——不是靠人肉 review 保证两段代码一致，而是靠同一段代码物理上不可能不一致。

## 取消超时触发

`cancelAutoBackground` 的存在是必要的：

```typescript
const cancelAutoBackground = () => clearTimeout(timer)
```

如果 Agent 在超时之前就正常完成了，需要取消这个超时器，否则它在未来某个时间点仍然会触发，对一个已经完成的任务调用 `triggerBackground`（虽然幂等，但不必要）。

更重要的是：如果用户在超时之前手动转了后台，resolver 已经被调用并从 Map 里删掉了；如果不取消超时，超时器触发时会再次尝试从 Map 里取 resolver，得到 `undefined`，无害但冗余。

## 面试答题框架

回答"超时自动降级"类设计题，可以用这个框架：

**1. 识别触发类型**：手动触发、超时触发，可能还有其他（比如系统资源紧张时自动降级）。

**2. 统一触发接口**：所有触发方式最终调用同一个函数，确保行为一致。

**3. 分离信号和执行**：转场信号（resolver）和执行循环（runAgent）通过 Promise 解耦，不需要直接调用执行循环的内部方法。

**4. 考虑生命周期**：触发后的清理（Map 删除、timer 取消）要显式处理。

**5. 考虑幂等性**：多次触发（手动 + 超时都触发了）不应该造成问题。

Claude Code 的这个设计，在代码量极少（一个 Map，一个 Promise，几十行代码）的情况下，满足了以上所有要点。这是"简单但正确"的好设计，不是"复杂但全面"的过度设计。
