# 自动启用、缓存检测和重连页说明它在认真对待真实用户现场

`setup.ts` 里有一堆细节，专门说明 Claude in Chrome 不是“装上就行”的玩具功能。自动启用并不是无条件开启，而是要求交互式会话、检测到扩展已装，并且还要么是 ant 用户、要么过了 feature flag；默认非交互场景直接关掉。manifest 改写后也不是假装一切已经生效，而是去检查扩展是否真的安装，如果装了再主动打开 reconnect 页面。连扩展检测都不是简单缓存一次了事，而是专门保留“正向检测可缓存、负向检测别乱缓存”的倾向，避免因为一时误判把能力永久藏起来。

更普通的实现，是要么把功能全自动打开，要么每次都让用户自己重新判断“现在到底连上没有”。前者容易越权，后者会把现场维护成本一直推给用户。Claude Code 选的是中间路线：尽量自动，但只在足够确定时自动；一旦环境变化，就主动做重连和提醒。

这套做法更像产品，而不是 demo。代价是代码显得啰嗦，里面全是缓存、检测、flag、交互模式和 reconnect 页面这类杂事。但这些杂事恰恰决定了真实用户第一次装、第二次用、半路升级时会不会觉得系统一直在和自己作对。

## 实现链
`shouldAutoEnableClaudeInChrome()` 依赖同步缓存 `isChromeExtensionInstalled_CACHED_MAY_BE_STALE()`，而这个缓存又会后台刷新；manifest 有改动时，`setup.ts` 还会根据是否已装扩展决定要不要打开 reconnect 页。整套链路都在降低“装好了但没接上”的真实现场摩擦。

## 普通做法
普通做法要么每次启动都全量扫描磁盘，要么完全不做缓存和重连辅助。

## 为什么不用
Claude Code 不走两头极端，因为冷启动时全量扫描太慢，而完全不辅助又会让首次接入非常脆。

## 代价
代价是会接受一定程度的 stale 状态，比如缓存命中 `true` 时可能先尝试一次无声连接。
