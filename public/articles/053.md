# Claude Code 的多 agent 团队为什么需要 leader、mailbox 和权限桥而不是共享状态？

「多 agent 系统怎么协调」是架构面试里最容易陷入泛泛而谈的问题。Claude Code 的 swarm 实现给出了一个具体的答案：不用共享内存，用文件邮箱 + leader 裁决 + 显式权限同步。

## 团队的基本结构

一个 team 由一个 leader 和若干 teammate 组成。`AppState.teamContext` 存储当前 agent 在团队中的身份：

```typescript
// 从 reconnection.ts 推断的 teamContext 结构
type TeamContext = {
  teamName: string
  teamFilePath: string
  leadAgentId: string
  agentId?: string    // 有则为 teammate，无则为 leader
  agentName: string
  color?: string
}
```

`computeInitialTeamContext()` 在 `main.tsx` 渲染之前同步计算这个初始状态——这个时序很重要，保证 React 第一次渲染时就已经知道自己是 leader 还是 teammate，不需要异步等待。

## 文件邮箱：最低公分母通信

`permissionSync.ts` 的注释直接说明了通信机制：

> 系统使用 teammate mailbox 做消息传递：worker 把权限请求发到 leader 的 mailbox，leader 把权限响应发到 worker 的 mailbox。

为什么不用进程间通信（IPC）或共享内存？

1. **跨进程边界**：在进程内 teammate（`inProcessRunner`）和真正独立进程的 teammate 都存在，文件邮箱是两种形态都能用的通信基础
2. **可恢复性**：如果 leader 崩溃重启，它可以重读 mailbox 里未处理的消息。共享内存在进程崩溃时会丢失所有状态
3. **人可读**：调试时可以直接查看邮箱文件，不需要特殊工具

邮箱使用锁文件（lockfile）保护写操作，用「已读标记」避免同一条消息被处理两次。

## 权限同步的流程

`SwarmPermissionRequestSchema` 定义了 worker 向 leader 发送权限请求的格式：

```typescript
z.object({
  id: z.string(),           // 请求的唯一 ID
  workerId: z.string(),     // 发请求的 worker agent ID
  workerName: z.string(),
  workerColor: z.string().optional(),
  teamName: z.string(),
  // ... 工具调用详情
})
```

流程是：
1. Worker 需要权限 → 写入 leader mailbox
2. Leader 轮询 mailbox → 检测到 permission_request
3. 用户在 leader UI 上确认/拒绝
4. Leader 写入 worker mailbox（permission_response）
5. Worker 轮询自己的 mailbox → 读取结果 → 继续执行

注意：这是一个**异步轮询**机制，不是事件推送。Worker 在等待期间会暂停执行，占着一个 agent slot。

## leaderPermissionBridge：连接 React 和非 React 代码

`leaderPermissionBridge.ts` 解决了一个有趣的架构问题：in-process teammate 是在非 React 代码里运行的，但权限确认 UI（`ToolUseConfirm`）是 React 组件。

```typescript
// 模块级的注册表，让非 React 代码能调用 React 的 setState
let registeredSetter: SetToolUseConfirmQueueFn | null = null

export function registerLeaderToolUseConfirmQueue(setter: SetToolUseConfirmQueueFn): void {
  registeredSetter = setter
}

export function getLeaderToolUseConfirmQueue(): SetToolUseConfirmQueueFn | null {
  return registeredSetter
}
```

Leader 在 React 初始化时调用 `registerLeaderToolUseConfirmQueue`，把 `setToolUseConfirmQueue` 注册进来。当 in-process teammate 需要权限时，它通过 `getLeaderToolUseConfirmQueue()` 拿到这个 setter，然后把确认请求推入 React 的状态队列。

这是一种「模块级单例作为跨层通信桥」的模式——不优雅，但有效，而且比引入全局 EventEmitter 更容易追踪谁注册了什么。

## 为什么 leader 不直接代理所有权限

一个关键设计决定：leader 不自动批准 teammate 的权限请求，而是把它展示给用户决定。这意味着用户在多 agent 场景下的权力没有被削减——每一个敏感操作都还是需要人来批准。

代价是效率：如果一个 teammate 需要频繁执行需要授权的操作，每次都要经过 leader-用户-leader-worker 这条链路，会显著拖慢团队工作速度。这也是为什么 `DANGEROUS_BASH_PATTERNS` 这类黑名单很重要——把已知危险模式预先拦截，减少运行时的权限请求频率。

## 面试答题思路

「多 agent 系统如何维持秩序」的答题框架：

1. **先确定通信基础设施**：共享内存 vs 消息传递，后者更适合需要恢复性的场景
2. **明确权限归属**：敏感操作的授权决定属于谁？不能让 agent 自授权
3. **身份先于并发**：agent 在开始工作之前必须知道自己是谁、在哪个团队、谁是 leader
4. **跨层通信要显式**：React 和非 React 代码的通信不应该通过隐式全局状态，要有注册/注销机制

追问「文件邮箱会不会太慢」：对于需要频繁通信的场景确实是问题；Claude Code 的设计假设是 permission request 相对低频，高频的数据交换通过 `outputFile` 而不是 mailbox 传递。
