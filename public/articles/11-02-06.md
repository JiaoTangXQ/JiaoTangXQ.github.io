# 设计模式：8 个入口都能修改权限模式，副作用怎么保证不漏不重？

## 问题的起点：状态修改的入口有多少

Claude Code 里，`AppState.toolPermissionContext.mode`（权限模式）可能被哪些地方修改？

1. 用户按 Shift+Tab 在 UI 里切换模式
2. 用命令行 `/plan` 或 `/auto` slash command 切换
3. 退出 plan 模式时（`ExitPlanModePermissionRequest` 对话框的选项）
4. 收到远端会话元数据推送（`externalMetadataToAppState`）
5. `set_permission_mode` 控制请求处理器
6. REPL bridge 的 `onSetPermissionMode` 事件
7. Rewind 操作恢复历史状态
8. 作为 teammate 启动时的初始化

**八个入口**，都会修改同一个字段。

如果副作用分散在每个入口里，你需要确保这八个地方都正确做了：
- 通知 CCR 外部元数据已更新（`notifySessionMetadataChanged`）
- 触发 SDK 状态流（`notifyPermissionModeChanged`）

漏了任何一个，就会出现「本地权限模式变了，但远端还以为是旧模式」的不一致。

## onChangeAppState 的 diff 逻辑

```typescript
export function onChangeAppState({ newState, oldState }) {
  const prevMode = oldState.toolPermissionContext.mode
  const newMode = newState.toolPermissionContext.mode

  if (prevMode !== newMode) {
    const prevExternal = toExternalPermissionMode(prevMode)
    const newExternal = toExternalPermissionMode(newMode)

    if (prevExternal !== newExternal) {
      notifySessionMetadataChanged({
        permission_mode: newExternal,
        is_ultraplan_mode: isUltraplan,
      })
    }

    notifyPermissionModeChanged(newMode)
  }
  // ...
}
```

这段代码有几个值得注意的细节：

**双层比较**

先比较「内部模式」（`prevMode !== newMode`），再比较「外部化模式」（`prevExternal !== newExternal`）。

内部有一些模式（比如 `bubble`、`ungated auto`）在外部化时会映射到 `default`。如果只是从 `default` 切换到 `bubble`，外部化结果没变，就没必要通知 CCR——通知了只是噪音。

所以 CCR 通知只在外部化模式真正变化时才发，而 SDK 通知（`notifyPermissionModeChanged`）传递原始模式，让接收方自己决定要不要响应。

**ultraplan 标记的处理**

```typescript
const isUltraplan =
  newExternal === 'plan' &&
  newState.isUltraplanMode &&
  !oldState.isUltraplanMode
    ? true
    : null
```

`null` 在 RFC 7396 里表示「删除这个键」。`true` 表示「设置为 true」。这是一种精巧的 JSON Merge Patch 用法：只在第一次进入 ultraplan 模式时设置标记，之后不再发送（避免重复触发远端侧逻辑）。

## 模型和配置的持久化

```typescript
if (newState.mainLoopModel !== oldState.mainLoopModel) {
  if (newState.mainLoopModel === null) {
    updateSettingsForSource('userSettings', { model: undefined })
    setMainLoopModelOverride(null)
  } else {
    updateSettingsForSource('userSettings', { model: newState.mainLoopModel })
    setMainLoopModelOverride(newState.mainLoopModel)
  }
}
```

模型变化时，写回用户设置文件。这意味着用户用 `/model claude-opus-4` 切换了模型，下次启动会记住这个选择。

把这个副作用放在 `onChangeAppState` 里，而不是 `/model` 命令的 handler 里，意味着：通过任何方式切换模型都会持久化——包括可能的 API 路径、测试场景、或未来新增的切换入口。

## 认证缓存的清理

```typescript
if (newState.settings !== oldState.settings) {
  try {
    clearApiKeyHelperCache()
    clearAwsCredentialsCache()
    clearGcpCredentialsCache()

    if (newState.settings.env !== oldState.settings.env) {
      applyConfigEnvironmentVariables()
    }
  } catch (error) {
    logError(toError(error))
  }
}
```

设置变化时，清理所有认证相关的缓存。这是因为设置里可能包含 API keys 或认证配置，设置一变，缓存的旧认证信息就可能失效。

注意这里用了 `try-catch`，但其他副作用逻辑没有。这暗示认证缓存清理是相对独立的操作，失败了可以记录错误但不影响整体流程。

## 副作用集中的代价

`onChangeAppState` 随着系统功能增加会越来越长。每次需要「某种状态变化时触发某种副作用」，都要在这里加代码。

这个函数本质上是一个「状态变化到副作用」的路由表。长远来看，如果副作用类型越来越多，可能需要把它拆成多个专门的 onChange 处理器，然后通过组合传入 AppStateProvider。

但目前的长度是可管理的。注释非常详尽——每个副作用块都有注释说明为什么这样做，以及曾经试过哪些方案、为什么被放弃。这是一种「把知识写进代码旁边」的实践，让维护者不需要去翻 git log 就能理解背景。

## 面试指导

**中介者模式（Mediator Pattern）在这里的体现**

`onChangeAppState` 就是一个中介者：它接收状态变化，决定哪些外部系统需要被通知，处理通知的顺序和条件。各个外部系统（CCR、SDK、配置文件、认证缓存）不需要互相了解，只需要和中介者打交道。

**纯 Observer 和 Mediator 的区别**

Observer 里，观察者直接订阅被观察者的变化。Mediator 里，所有通信经过中间人。对于多对多的通信，Mediator 减少了直接连接数。`onChangeAppState` 是 Mediator，不是 Observer：它不是「每个副作用系统订阅 AppState」，而是「一个中介者集中处理所有副作用」。

**为什么不用响应式（Rx）流来处理这些副作用？** RxJS 的操作符可以优雅地处理 distinctUntilChanged（跳过无变化的状态）、debounce（防抖）等场景。但引入 RxJS 会增加依赖和学习成本，而 `onChangeAppState` 的逻辑相对直接，手动实现 diff 就够了。
