# 深度分析：为什么强能力需要先被制度容纳，而不是先上线再补制度

## 两种加入强能力的顺序

**顺序一：能力先，制度后**

先实现功能（比如 bash 执行），让它能跑，能演示，能给用户带来价值。然后在使用过程中发现问题，逐步加权限检查、加审计日志、加用户确认。

这是大多数产品的实际路径：快速上线，迭代补制度。

**顺序二：制度先，能力后**

在实现 bash 执行之前，先确定这个能力的风险分类（高风险），确定它的权限边界（需要用户在会话开始时授权），确定它的审批路径（每次执行前向用户展示命令），确定它的审计要求（记录所有执行）。制度确定后，再实现能力。

Claude Code 选择了顺序二。

**面试题：为什么顺序如此重要？**

## 能力先上线的问题

**制度补不上去**

一个已经在生产环境运行、已经有用户依赖的能力，加制度约束会破坏已有的使用方式。

比如，bash 执行已经在生产了，现在要加"每次执行前用户必须确认"。很多依赖自动化脚本的用户会立刻抱怨——他们的工作流需要非交互式执行。

为了向后兼容，只能加一个"可以关闭确认"的选项。于是制度有了一个大洞：用户可以直接绕过，制度形同虚设。

**制度的一致性无法保证**

如果能力先上线，不同的能力在不同时期加制度，每次加的方式可能不同，结果是制度七拼八凑：bash 执行的权限在这里检查，文件编辑的权限在那里检查，两者用不同的接口，用不同的存储格式。

审计系统要同时支持两种格式，权限管理界面要同时展示两种方式，这是持续的维护成本。

## ToolPermissionContext 是"可承重对象"的体现

在 `src/hooks/toolPermission/PermissionContext.ts` 里，`ToolPermissionContext` 是随每次工具调用携带的制度快照。

它包含：
- 当前会话的权限级别（从 request_access 或设置里读取）
- 已批准的操作记录
- 等待审批的队列
- 审批历史

这个"制度快照"是在工具调用机制设计阶段就被设计进去的，不是事后补丁。

**重要的含义**：任何新的工具，只要遵循 Tool 接口，就自动进入这套制度——权限检查、审计日志、用户确认都自动生效，不需要工具自己实现。

这就是"强能力先被制度容纳"的工程体现：制度是系统的基础设施，新能力进来不是"绕过制度然后补上"，而是"进来就被制度承接"。

## 优雅是能够承载强能力

这里有一个关于"优雅"的深层理解。

普通意义上的优雅是指"简洁"、"没有多余的东西"。

但 Claude Code 的治理设计表明，有一种更深层的优雅：**能够承载强能力而不失控**。

一个没有制度、只有能力的系统，可能表面上"简洁"（没有那么多权限逻辑），但它是脆弱的——任何能力的滥用都没有防线。

一个能制度先行、制度承载能力的系统，看起来"更重"（有权限检查、审批路径、审计日志），但它是稳固的——能力可以很强，系统不会因为强能力而失控。

这种稳固，才是更高层次的优雅。

## 强能力的风险标注

每个工具在 `Tool.ts` 的接口定义里，可以有风险标注字段（比如是否需要用户确认、风险级别是什么）。

这个标注在工具定义时就写好，不是在权限检查时动态判断。

这意味着：工具的风险属性是工具自身声明的一部分，不是外部系统猜测出来的。工具的作者在设计工具时，就需要思考"这个工具是高风险还是低风险"，并把这个判断写进工具定义。

这是"制度先于能力"的另一个面：工具作者必须在实现功能的同时，声明功能的制度属性。

## 面试要点

1. **能力先上线后补制度会留下大洞**：为了向后兼容，制度约束往往无法真正落实，最终形同虚设。
2. **ToolPermissionContext 是系统级的制度基础设施**：所有工具自动进入制度体系，不需要各自实现权限逻辑。
3. **优雅是承载能力而不失控**：能让强能力存在且系统不乱，比"简洁无制度"是更高层次的优雅。
4. **工具声明自己的风险属性**：风险标注在工具定义时写好，不是外部猜测，要求工具作者在设计时就思考风险。
