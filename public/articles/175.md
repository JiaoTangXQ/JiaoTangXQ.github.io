# 第十三卷回收索引：治理世界里的强能力先被写成可承重对象制度骨架才有意义

## 实现链
治理世界里最先要讲清的，不是“要不要能力”，  
而是强能力到底怎样才敢被写出来、被放进系统里。

这条线真正逼出来的，是第十三卷里关于优雅定义和制度先行的判断。

这页实际回收的实现链是：
- `06-01-02` 先把治理解释成让强能力敢存在的前提，后面才有 `13-01-01` 对“优雅不是功能多”的定义。

很多系统把制度理解成能力上线之后的刹车。  
Claude Code 更进一步，它先把能力写成可承重对象，再让制度去承载它。

## 普通做法
更普通的做法，是每个能力各带一点本地校验，或者干脆给用户一个粗糙的总开关，把细节风险留给人自己判断。

## 为什么不用
Claude Code 不这么做，是因为强能力一旦缺少统一制度快照和变更流程，就会在最危险的地方出现口径不一致，最后连改规则本身都没有共同程序。

## 代价
代价是 permission context、settings 扇出、hooks 和模式契约都更重，读起来也更像制度系统。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
