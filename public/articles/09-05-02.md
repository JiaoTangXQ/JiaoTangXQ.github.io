# writeToMailbox 里的锁后重读：一个很容易忽略但至关重要的并发细节

`writeToMailbox()` 的执行流程，很多人第一次读到时会觉得有点奇怪：

```
1. 确保 inbox 文件存在（用 wx flag 原子创建）
2. 获取文件锁
3. 重新读取 inbox 文件          ← 这一步是重点
4. 把新消息追加到消息数组
5. 把完整数组写回文件
6. 释放锁（finally 块里）
```

第 3 步为什么要"重新读取"？加锁之前不是已经知道文件内容了吗？

## 竞态条件的时间窗口

假设流程里没有第 3 步（加锁后直接用之前读到的内容）：

```
时间线：
  Agent A                           Agent B
  ───────────────────────────────   ────────────────────────────
  t=0: 读取 inbox.json              t=0: 读取 inbox.json
       → messages = [msg1, msg2]         → messages = [msg1, msg2]
  t=1: 获取锁（成功）
  t=2: 往 messages 里追加 msgA
  t=3: 写回 [msg1, msg2, msgA]
  t=4: 释放锁
                                    t=5: 获取锁（成功）
                                    t=6: 往 messages 里追加 msgB
                                         （但 messages 还是旧的 [msg1, msg2]！）
                                    t=7: 写回 [msg1, msg2, msgB]
                                         ← msgA 丢失了！
```

问题出在哪？Agent B 在 t=0 读取了文件，但直到 t=5 才拿到锁。在这 5 个时间单位里，Agent A 已经写入了 `msgA`。但 B 拿到锁后用的仍然是 t=0 时读取的旧内容，B 的写入覆盖了 A 的写入。

**这就是"读取和加锁之间有时间窗口"的竞态条件**。加锁保证了写入的互斥，但如果用的是锁之前读取的旧数据，竞态窗口只是被缩短了，而不是消除了。

## 锁后重读消除竞态窗口

加入第 3 步后：

```
时间线（Agent B 的行为）：
  t=0: 读取 inbox.json（旧内容）
  t=5: 获取锁（成功）
  t=6: 再次读取 inbox.json          ← 锁后重读
       → messages = [msg1, msg2, msgA]  ← 看到了 A 写入的内容！
  t=7: 往 messages 里追加 msgB
  t=8: 写回 [msg1, msg2, msgA, msgB]
  t=9: 释放锁
```

锁后重读确保了：**拿到锁时，立刻获取最新状态**。无论在等待锁的过程中发生了什么变更，都能在重读时捕获到。

竞态窗口从"读取 → 写入"缩小到了零（锁保护了"读取 → 写入"整个操作）。

## 为什么说这是"最容易漏掉的步骤"

大多数人在第一次实现文件锁时，会这么写：

```typescript
async function writeToMailbox(agentName: string, message: TeammateMessage) {
  const inboxPath = getInboxPath(agentName)
  const lockPath = `${inboxPath}.lock`

  const release = await lockfile.lock(lockPath)
  try {
    // 直接用调用前读到的或者内存里的 messages
    const messages = [...currentMessages, message]
    await writeFile(inboxPath, JSON.stringify(messages))
  } finally {
    await release()
  }
}
```

这段代码看起来有锁，但如果 `currentMessages` 是在调用 `writeToMailbox` 之前读取的，拿到锁时这份数据可能已经过时。

正确的写法必须在锁内重读：

```typescript
const release = await lockfile.lock(lockPath)
try {
  // 锁内重读，确保是最新状态
  const content = await readFile(inboxPath, 'utf-8')
  const currentMessages = JSON.parse(content) as TeammateMessage[]
  const updatedMessages = [...currentMessages, message]
  await writeFile(inboxPath, JSON.stringify(updatedMessages))
} finally {
  await release()
}
```

这个细节在代码审查中经常被忽略，因为"有锁了，应该安全了"是一个直觉上合理但逻辑上不完整的判断。

## markMessageAsReadByIndex 也要锁后重读

同样的问题也出现在 `markMessageAsReadByIndex()` 里：

```typescript
export async function markMessageAsReadByIndex(
  agentName: string,
  index: number,
  teamName?: string,
): Promise<void> {
  const inboxPath = getInboxPath(agentName, teamName)
  const release = await lockfile.lock(...)
  try {
    // 锁内重读
    const messages = await readMessages(inboxPath)
    if (messages[index]) {
      messages[index].read = true
      await writeFile(inboxPath, JSON.stringify(messages))
    }
  } finally {
    await release()
  }
}
```

标记已读也是"读取 → 修改 → 写回"的操作，同样需要锁后重读。如果在锁外读取了 messages，等到拿到锁时可能有新消息进来，标记已读时用旧的 messages 写回，会丢掉那些新消息。

## LOCK_OPTIONS 的参数选择

```typescript
const LOCK_OPTIONS = {
  retries: {
    retries: 10,
    minTimeout: 5,
    maxTimeout: 100,
  },
}
```

这些数字不是随意选的：

**retries: 10**：在极端情况下（多个 agent 密集写入），最多等 10 次重试，给约 500ms 的容忍窗口。

**minTimeout: 5ms**：最短等待 5ms，避免 spin lock（频繁无间隔重试会浪费 CPU，在 Node.js 的事件循环模型里尤其低效）。

**maxTimeout: 100ms**：最长单次等待 100ms，避免锁等待让 agent 卡住太久。

指数退避（backoff）确保重试间隔从 5ms 开始逐步增大，多个 agent 竞争时自然错开，避免同时重试造成的"惊群效应"。

## 面试怎么答

如果面试题是"文件锁能保证并发安全吗"：

**完整答案**：文件锁只保证写入互斥，不保证"用于写入的数据是最新的"。要真正保证并发安全，必须在拿到锁之后重新读取数据，而不是用锁之前读取的数据。这个"锁后重读"步骤是整套并发安全的关键，也是最容易遗漏的地方。

**实际影响**：在读请求频繁、写操作相对少的场景（比如 mailbox 主要是读，偶尔有多个 agent 同时写），锁后重读几乎不会带来性能开销，因为锁竞争本来就不多。代价可以忽略，安全收益显著。

**加分项**：能把这个模式类比到数据库的 select for update（先锁定行，再读取，再修改，在同一个事务里完成），或者 compare-and-swap（原子地读取并更新）。锁后重读是文件系统版本的 "select for update"——在锁的保护下确保数据一致性。
