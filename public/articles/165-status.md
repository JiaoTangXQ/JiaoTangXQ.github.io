# 第十三卷回收索引：任务世界里 status 先变成生命周期总闸门再谈别的模块

## 实现链
任务世界里制度先行的另一半，不是先写规则本身，而是先把 status 变成生命周期总闸门，让所有模块都得看它。

这条线真正逼出来的，是第十三卷里关于终态收口的判断。  
复杂系统之所以还能继续长，是因为它先让终态落位，再让别的模块继续猜。

这页实际回收的实现链是：
- `05-03-03` 把 `status` 写成生命周期总闸门，这会在 `13-04-01` 里继续成立，说明复杂系统必须先让终态收口，再让别的模块继续猜。

很多系统一上来就让别的模块继续读状态。  
Claude Code 更稳的地方，是先把 `status` 变成总闸门，再让后面的判断沿着同一条终态线往前走。

## 普通做法
更普通的做法，是把后台动作当成松散的异步句柄，跑起来就算开始，结束了再临时补状态和通知。

## 为什么不用
Claude Code 不这么做，是因为没有统一任务对象，就没有稳定的生命周期、输出、通知和回看链，前后台切换也会变成一串彼此不认账的临时处理。

## 代价
代价是任务状态字段、输出文件、通知协议和前后台转场逻辑都会变厚，维护者要记住更多接缝。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
