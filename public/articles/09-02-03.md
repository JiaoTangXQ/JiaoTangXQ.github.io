# 为什么群发消息被要求"sparingly"？Claude Code 的团队通信熵控制

`teammatePromptAddendum.ts` 里有一条经常被忽视的系统提示规定：

> You can broadcast to all teammates using `to: "*"`, but use this sparingly.

"sparingly" 这个词被刻意放进了系统提示，这不是措辞习惯，而是一个明确的制度约束。

## 广播太便宜时会发生什么

先做一个思想实验：5 个 teammate 都可以随时对全体广播，没有任何成本限制。

每个 teammate 在工作过程中都会产生大量中间状态：发现了一个有趣的代码模式，遇到了一个不确定的情况，想分享一个潜在的优化思路……如果广播是默认姿势，这些都会发出去。

5 个成员，每人每分钟广播 3 条消息，每分钟团队收到 15 条消息。每条消息每个人都要看，判断是否需要响应。

结果是：**信息密度超过了团队的处理能力，没有成员能区分哪条消息是真正需要响应的信号，哪条只是别人的过程噪音**。

这不是多 agent 系统独有的问题。在人类团队里，"所有决策都在群里发"的公司往往比"只在必要时发群消息"的公司效率更低——即使前者的信息总量更多。

## 定向消息作为默认姿势

Claude Code 的设计选择是：**定向消息是默认，广播是例外**。

`SendMessage` 的 `to` 参数默认应该指向一个具体的 agent 名字（比如 `researcher` 或 `team-lead`），而不是 `"*"`。

这带来了几个有意思的效果：

**发送方思考路由**。当你必须写出 `to: "researcher"` 而不是 `to: "*"`，你就在做一个判断：这条消息是给研究员的，还是给所有人的？这个判断本身就是一次过滤。

**接收方知道为什么收到**。如果你收到一条消息，它发给你（`to: "you"`），你就知道这是专门给你的，需要响应。如果是广播，你要先判断"这和我有关系吗"，再决定要不要响应。定向消息减少了这一步不确定性。

**噪音天然可控**。广播消息数量会被"稀缺感"自然控制——因为广播成本高（你得确定这条信息真的对所有人都有价值），真正的广播就只发最值得共享的内容。

## 什么情况下应该广播

"sparingly" 不是"永不广播"。真正值得广播的场景：

1. **进度里程碑**：某个关键子任务完成，所有人都需要知道可以解锁依赖的下一步了。
2. **阻塞发现**：某个共享资源被锁，所有人需要暂停相关操作。
3. **方向变更**：leader 发现整体策略需要调整，需要所有 worker 停下来重新对齐。

这三种场景的共同点：**信息对所有接收方都可操作**（actionable），而不只是"有趣但不需要响应"。

## Claude Code 如何在底层支持这个约束

只在系统提示里说"sparingly"是不够的，因为 AI 模型可能不遵守。

Claude Code 在底层做了什么来强化这个约束？

首先，`SendMessage` 工具本身是显式工具调用，不是自动触发的。agent 必须主动决定"我要发消息了"，这个主动决策就是一道天然门槛。

其次，mailbox 的消息有 `summary` 字段——这暗示接收方会看摘要决定是否要展开。如果广播的消息摘要不清晰，接收方可能直接跳过。发送方为了让自己的消息被处理，必须用摘要准确传达重要性。

第三，agent 的工具调用次数和 token 消耗都有成本。频繁广播会让 agent 很快达到上下文限制，这是一个隐形的经济约束。

## 面试怎么答

如果面试题是"多 agent 系统怎么控制消息通信的复杂度"：

**低分答案**：限制消息长度，或者加一个消息速率限制器。这是技术层面的强制截断，不是真正的控制。

**中等答案**：区分"点对点消息"和"广播消息"，把广播做成明确的特殊操作。这已经是正确方向。

**高分答案**：从信息熵角度分析。多 agent 系统的通信问题不是"消息能不能发出去"，而是"团队成员能不能有效处理收到的所有消息"。控制信息熵的核心是：让每条消息的发送都是一个有成本的决策，而不是零成本的广播。具体实现上：默认定向消息、广播需要显式选择、消息有摘要字段辅助优先级判断。

**加分项**：能类比到实际工程中的类似问题——比如 Slack 频道里"@channel" vs "@specific-person" 的使用规范，或者 pub/sub 系统里如何避免 topic 粒度太粗导致消费方收到大量无关消息。AI Agent 团队通信面临的信息熵问题，和人类团队通信是同构的。
