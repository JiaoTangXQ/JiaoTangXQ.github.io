# autocompact 的熔断器：为什么连续失败三次后就不再重试？

这是一道故障恢复设计题。面试问题是："如果自动压缩本身也一直失败，系统怎么防止陷入无限重试？"

---

## 问题背景

autocompact 有可能失败。常见原因：

1. **compress 请求本身 prompt_too_long**：要摘要的内容太大，即使经过 PTL retry 也无法通过
2. **网络超时**：摘要请求是一次独立的 API 调用，可能因网络问题失败
3. **API 错误**：服务端返回了不可重试的错误
4. **用户中断**：用户按 Esc 取消了正在进行的 compact

---

## 熔断常量

```typescript
const MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3
```

BQ 数据（2026-03-10）中有一段注释：
> BQ 2026-03-10: 1,279 sessions had 50+ consecutive failures (up to 3,272) in a single session, wasting ~250K API calls/day globally.

这是一个真实数据。在引入熔断之前，某些上下文不可恢复的会话会在每个 turn 都触发 autocompact，每次都失败，然后继续下一轮又触发，产生了大量无效的 API 调用（全局每天 ~25 万次）。

---

## 熔断逻辑

```typescript
export async function autoCompactIfNeeded(...) {
  // 熔断检查：在 shouldAutoCompact 之前
  if (
    tracking?.consecutiveFailures !== undefined &&
    tracking.consecutiveFailures >= MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES
  ) {
    return { wasCompacted: false }  // 直接跳过，不尝试
  }

  // ... shouldAutoCompact 判断 ...

  try {
    const compactionResult = await compactConversation(...)
    return {
      wasCompacted: true,
      compactionResult,
      consecutiveFailures: 0,  // 成功重置
    }
  } catch (error) {
    const prevFailures = tracking?.consecutiveFailures ?? 0
    const nextFailures = prevFailures + 1
    if (nextFailures >= MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES) {
      logForDebugging(`autocompact: circuit breaker tripped...`, { level: 'warn' })
    }
    return { wasCompacted: false, consecutiveFailures: nextFailures }
  }
}
```

失败计数存储在 `AutoCompactTrackingState.consecutiveFailures` 里，通过 `state.autoCompactTracking` 在 query loop 的迭代之间传递。成功时重置为 0，失败时递增。

---

## 跳过条件：比熔断更早的保护

在熔断之前，`shouldAutoCompact()` 已经有多个跳过条件：

**递归保护**：
```typescript
if (querySource === 'session_memory' || querySource === 'compact') {
  return false
}
```
compact agent 自己不会触发 compact，session_memory agent 也不会。这防止了"compact 请求本身触发 compact"的无限递归。

**context collapse 保护**：
```typescript
if (feature('CONTEXT_COLLAPSE')) {
  if (isContextCollapseEnabled()) {
    return false
  }
}
```
当 context collapse 模式启用时，autocompact 让位给 collapse 机制。两者不能同时运行，因为 autocompact 在 collapse 正在工作的区间触发，会"抢走"正在收集的 granular context。

**marble_origami（ctx-agent）保护**：
```typescript
if (feature('CONTEXT_COLLAPSE')) {
  if (querySource === 'marble_origami') {
    return false
  }
}
```
context collapse 的内部子 agent。如果它触发了 autocompact，`runPostCompactCleanup` 会调用 `resetContextCollapse()`，破坏主线程的 commit log（模块级全局状态）。

---

## 熔断后系统的状态

熔断触发后，autocompact 停止重试，但系统不会崩溃。接下来会发生什么取决于 token 数是否继续增长：

**如果 token 数低于 blocking limit**：会话继续正常工作，但不再自动压缩。用户会看到上下文警告，可以手动运行 `/compact`。

**如果 token 数达到 blocking limit**：
```typescript
if (isAtBlockingLimit) {
  yield createAssistantAPIErrorMessage({
    content: PROMPT_TOO_LONG_ERROR_MESSAGE,
    error: 'invalid_request',
  })
  return { reason: 'blocking_limit' }
}
```
系统在下一轮 API 调用之前检测到这个情况，返回一个"合成的"错误消息，告知用户需要手动 compact。

---

## 用户中断的特殊处理

```typescript
if (!hasExactErrorMessage(error, ERROR_MESSAGE_USER_ABORT)) {
  logError(error)
}
```

用户中断（`ERROR_MESSAGE_USER_ABORT`）不记录为错误，但仍然会递增失败计数。这意味着如果用户连续三次中断 autocompact（每轮都在 compact 时按 Esc），熔断也会触发。这是合理的——如果用户持续拒绝 compact，系统也不应该继续强行尝试。

---

## 工程启示

这个熔断设计体现了一个重要原则：**自动化系统的容错边界要明确，不能让"失败的自动化"比"没有自动化"更差**。

一个每轮都尝试失败的 compact 机制，比完全没有 compact 机制更糟糕，因为它：
1. 浪费 API 调用（成本）
2. 增加每轮的延迟（用户体验）
3. 给用户看到大量错误消息（困惑）

熔断器让系统在无法自愈的情况下优雅地退化，而不是持续无效尝试。

---

## 面试指导

"系统级自动化功能一直失败怎么处理"是个很好的故障恢复设计题。

关键要点：
1. **量化**：多少次失败算"需要停止"——3 次是基于真实 BQ 数据推导的
2. **状态持久化**：失败计数要跨迭代传递，不能每轮重置
3. **优雅降级**：熔断后系统不崩溃，继续工作，但用户需要手动处理
4. **区分可重试和不可重试**：用户中断、API 错误、PTL 超限，原因不同但最终都导致失败计数递增

能说清楚"为什么是 3 次"（真实数据驱动）而不只是"防止无限重试"，说明你理解了这个设计不只是通用熔断模式，而是对具体问题的量化响应。
