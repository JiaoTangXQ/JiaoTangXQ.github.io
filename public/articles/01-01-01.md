# 面试题：如果把 Claude Code 说成"CLI 聊天壳"，面试官会在哪里追问你？

## 第一追问：入口文件为什么要做十几件事？

假设你说"Claude Code 就是在终端里和模型聊天"。面试官会立刻问：那 `cli.tsx` 为什么要在最前面检查 `--version`、`--daemon-worker`、`remote-control`、`--tmux`、`--bare`、`daemon`、`ps/logs/attach/kill` 这十几种路径，而不是直接初始化一个对话循环？

源码里的答案很直接：这不是一个聊天程序，而是一个需要按使用场景分发执行路径的工作台系统。`cli.tsx` 里的 `main` 函数是一条长长的 if-else 链。`--version` 是零模块加载的快速路径，直接打印构建时内联的版本字符串就退出。`--daemon-worker=<kind>` 是监督进程在内部 spawn 用的，进来后直接走 `runDaemonWorker`，根本不需要认证、不需要 analytics。`remote-control` 走 `bridgeMain` 之前还要先检查 OAuth token、等 GrowthBook policy 加载、确认 `allow_remote_control` 没被企业策略禁用。每条路径对应一种真实的部署场景：本地 REPL、无头 SDK、后台进程、远端 bridge、worker 节点。

聊天壳不会有这些分叉。工作台必须有。

## 第二追问：QueryEngine 为什么不是个函数，而是个有状态的类？

"它是一个 agent 运行时引擎"——很多人会背这句话，但背不出为什么。

看 `QueryEngine.ts` 的构造函数就知道了。它接收的 `QueryEngineConfig` 包含 `tools`、`commands`、`mcpClients`、`agents`、`canUseTool`、`readFileCache`，还有 `customSystemPrompt`、`maxTurns`、`maxBudgetUsd`、`jsonSchema`。构造完以后，实例会持有 `mutableMessages`（整个会话的消息历史）、`permissionDenials`（这个类型是 `SDKPermissionDenial[]`，记录每次工具请求被拒的工具名、use ID、输入参数）、`totalUsage`（聚合的 API 消耗数据）、`discoveredSkillNames`（这一轮发现的技能，每次 `submitMessage` 开始时清空）、`loadedNestedMemoryPaths`（已经注入的嵌套 CLAUDE.md 路径，防止同一个文件被反复注入）。

这些字段跨多轮对话共享。你问一句话，`submitMessage` 执行完，这些状态留着，下一句话进来时继续接上。这不是聊天，这是有长期工作记忆的会话引擎。

## 第三追问：权限拒绝为什么要被专门跟踪？

聊天壳不需要跟踪权限。工作台需要，因为工作台里有真正会改变世界的操作。

`QueryEngine` 里有一个 `wrappedCanUseTool` 函数，它把传进来的 `canUseTool` 包了一层。每次工具请求被拒绝（`result.behavior !== 'allow'`），就往 `permissionDenials` 数组里 push 一条记录，包含 `tool_name`（经过 `sdkCompatToolName` 标准化）、`tool_use_id`、`tool_input`。这些拒绝记录最终会通过 `SDKCompactBoundaryMessage` 等结构暴露给 SDK 调用方——外部系统可以精确看到"这次执行里哪些操作被拒了、被拒的输入是什么"。

这是治理能力，不是聊天功能。

## 第四追问：context.ts 里的 getGitStatus 和 getUserContext 有什么意思？

上下文装配是工作台和聊天壳最显眼的差异之一。`context.ts` 里的 `getGitStatus` 用 `memoize` 包住，每次会话只跑一次：并行调用 `getBranch`、`getDefaultBranch`、`git status --short`、`git log --oneline -n 5`、`git config user.name`，然后拼成一段文字注入系统提示。Git 状态超过 2000 字符会被截断，截断时会附上提示让模型用 `BashTool` 去拿完整状态。`getUserContext` 同样被 `memoize`，会发现并读取 `CLAUDE.md` 文件——但有个隐藏条件：`--bare` 模式下如果没有显式 `--add-dir`，就跳过自动发现，因为 `--bare` 的语义是"不执行我没要求的事"，不是"彻底忽略我要求的事"。

聊天程序不关心你在哪个分支上工作。工作台从会话开始第一秒就关心。

## 这道题考什么，怎么拿高分

面试官问"Claude Code 是什么"，真正想考的是：你能不能把架构定位和具体实现绑在一起说，而不是只背概念。

**满分答法：** 先说定位（工作台 = 命令调度 + 运行时引擎 + 行动层 + 制度层），再用具体机制佐证（`cli.tsx` 的多路径分发、`QueryEngine` 的跨轮状态、`permissionDenials` 跟踪、`getGitStatus` 上下文装配），最后解释为什么聊天壳做不了这些。

**常见错误：** 说"它支持工具调用所以是 agent"——这只是能力描述，不是架构解释。说"它有 React 组件所以是工作台"——完全跑偏，React 是 Ink 渲染器，不是工作台的本质。说"因为有 MCP 接口"——MCP 是扩展点，不是工作台定义本身。

**容易被追问死的点：** 有人会说"QueryEngine 是会话管理"——面试官会追问"会话管理和消息列表有什么区别"。区别在于权限跟踪、预算约束、文件缓存、多轮状态。如果你只背了"会话管理"这四个字，到这里就答不下去了。
