# 为什么把 Cladue Code 看成 CLI 聊天壳会一路误读下去

很多人第一次见到 Claude Code，都会先把它理解成“在终端里和模型聊天”。这个直觉不算错，但只够看见最表面的那一层。聊天壳的职责很简单：把人的一句话送进去，再把模型吐出来的一段话显示出来。可 Claude Code 从一开始就不是按这个目标长出来的。

它真正搭起来的是一台终端里的工作台。`main.tsx` 不是只负责把界面点亮，它先抢时间做预取、判断入口、建立环境、选择模式，再把会话交给后面的运行时。`QueryEngine` 也不是只替模型转发文本，它长期保存消息、权限拒绝、文件缓存、技能发现和会话状态。再往下看，`Tool`、`Task`、`AppStateStore` 这些名字已经把答案写得很明显了：这里被正式管理的是行动、任务、状态、权限、插件、远端连接和后台工作，而不只是文本往返。

一旦把它误读成聊天壳，后面很多设计都会显得多余。为什么要有任务状态？为什么要有权限制度？为什么要有插件、MCP、bridge、后台 agent、transcript、恢复和界面状态？如果它只是聊天，这些东西都会像后来硬贴上去的“功能”。但如果把它看成工作台，这些反而会回到正常位置：它们不是装饰，而是让一个 agent 真能在现实世界里持续工作的骨架。

这套设计最漂亮的地方，在于它没有把“模型很聪明”误当成“系统已经完整”。模型负责组织语言、判断下一步和调用能力，但真正让事情发生的，是工具层、任务层、制度层和状态层一起配合。它的不足也正在这里：因为外表仍然是一块终端，初看的人很容易低估内部架构，直到读到中段才意识到自己面对的并不是一个会说话的壳，而是一套被精心收束起来的运行时。

## 实现链
代码里的实现链并不是读一行输入、回一段文本这么薄。`main.tsx` 先决定入口、模式、预取和环境，再把真正的会话交给 `QueryEngine.ts`；`QueryEngine` 继续托管消息、权限拒绝、文件上下文、工具结果和会话状态；真正落地动作时又会走到 `Tool.ts`、`Task.ts` 和状态存储上。整个入口从一开始就在管理工作台。

## 普通做法
更普通的 CLI 聊天工具会把主流程写成：读 prompt，调用模型，流式输出文本，必要时补几条命令。它关心的是把文字进出模型这条链做顺。

## 为什么不用那种做法
Claude Code 没按聊天壳来写，是因为它从代码结构上就在准备长期工作：工具要过权限，任务要有生命周期，状态要跨回合保持，远端和扩展能力要接进同一运行时。如果只按聊天壳组织，这些能力最后都会像后贴上去的例外。

## 代价与回报
回报是系统更稳，很多能力不是外挂而是骨架。代价也明显：入口更重、层次更多，新读者第一次看源码会觉得它不像一个普通 CLI。
