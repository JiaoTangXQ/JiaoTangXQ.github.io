# setupGracefulShutdown() 为什么是初始化的一部分，而不是最后一步？

## 典型的错误模式

很多程序的结构是这样的：

```typescript
async function main() {
  await initNetwork();       // 建立连接
  await initDatabase();      // 连接数据库
  await startServer();       // 启动服务
  
  // 最后注册清理
  process.on('SIGTERM', cleanup);
  process.on('SIGINT', cleanup);
}
```

这有一个竞争条件：如果在 `initNetwork()` 和 `process.on('SIGTERM', cleanup)` 之间进程收到了 SIGTERM，`cleanup()` 永远不会被调用，网络连接就悬挂了。

Claude Code 是怎么做的？

## init() 里的顺序

```typescript
enableConfigs()                      // 第一步
applySafeConfigEnvironmentVariables() // 第二步
applyExtraCACertsFromConfig()         // 第三步
setupGracefulShutdown()               // 第四步  ← 在任何网络操作之前
// ...
configureGlobalMTLS()
configureGlobalAgents()
preconnectAnthropicApi()
```

`setupGracefulShutdown()` 在所有网络相关初始化之前。这意味着：从注册退出处理的那一刻起，不管进程因什么原因退出，清理逻辑都能执行。

## registerCleanup() 的设计

后续代码里，每次创建需要清理的资源，都会立即注册对应的清理器：

```typescript
// LSP manager cleanup
registerCleanup(shutdownLspServerManager)

// Team cleanup
registerCleanup(async () => {
  const { cleanupSessionTeams } = await import(
    '../utils/swarm/teamHelpers.js'
  )
  await cleanupSessionTeams()
})
```

`registerCleanup()` 把清理函数添加到一个列表，`gracefulShutdown()` 会按注册顺序（或反序）依次执行它们。

关键设计：`registerCleanup()` 和资源创建**紧挨着**，不是分开在不同地方。这防止了"创建了资源忘记注册清理"的经典 bug。

## gracefulShutdown vs gracefulShutdownSync

```typescript
export async function gracefulShutdown(exitCode: number): Promise<void>
export function gracefulShutdownSync(exitCode: number): void
```

两个版本满足不同场景：
- `gracefulShutdown`：异步版本，允许清理器做异步操作（如等待 HTTP 请求完成）
- `gracefulShutdownSync`：同步版本，用于 SIGINT 处理器里（信号处理器不能是 async）

```typescript
// init.ts 里的错误处理
if (getIsNonInteractiveSession()) {
  process.stderr.write(
    `Configuration error in ${error.filePath}: ${error.message}\n`,
  )
  gracefulShutdownSync(1)  // 同步版本，因为这里在 catch 块里
  return
}
```

## 会话级清理 vs 进程级清理

Claude Code 区分了两种清理：

**进程级清理**（通过 `registerCleanup()`）：进程退出时必须执行的操作，如关闭 LSP 服务器、清理会话级别创建的 team。

**会话级清理**：在会话结束时（不一定是进程退出），需要做的清理。

这个区别很重要。Claude Code 支持多种会话模式（worktree 切换、会话恢复、远端会话），有些"会话结束"不等于"进程退出"。把所有清理都放在进程退出处理里，可能错过会话切换时需要的清理。

## 为什么这是"成熟系统"的标志？

**演示代码**：不在意退出清理，因为进程死了就死了，没有持久资源需要维护。

**生产级 agent**：需要维护的资源遍布各处——LSP 语言服务器、团队配置、会话文件、临时目录、网络连接。不正确退出会留下：
- 孤儿进程（已经停止的 Claude Code，但 LSP 还在跑）
- 脏配置（会话创建的 team 没有清理）
- 临时文件（worktree 相关的临时状态）

注释里有一段很能说明问题：

```typescript
// gh-32730: teams created by subagents (or main agent without
// explicit TeamDelete) were left on disk forever. Register cleanup
// for all teams created this session.
```

这是从一个真实 bug（`gh-32730`）里学到的。团队对象如果不清理，会永远留在磁盘上，积累垃圾数据。正是因为有了 `registerCleanup()` 机制，修复这类 bug 只需要找到资源创建的地方，添加一行 `registerCleanup()` 调用。

## 面试指导

资源管理和退出清理是很多初级工程师忽视的区域，也是系统可靠性的关键。

面试评估要点：
1. **资源创建和清理的配对**：清理注册应该紧跟资源创建，不应分开管理
2. **退出路径的覆盖**：正常退出、SIGTERM、SIGINT、uncaughtException——所有路径都需要走清理流程
3. **同步 vs 异步**：信号处理器必须同步返回，但清理操作可能是异步的——需要两套机制
4. **时机**：清理注册应该在资源创建之前或同时，防止竞争条件

能把这四点说清楚，就是对资源生命周期管理有扎实理解的体现。
