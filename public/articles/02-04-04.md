# 清理器和平稳退出为什么也是启动的一部分

很多程序谈启动时，只想着“怎么活起来”，很少在这一刻就认真考虑“等会儿怎么收场”。Claude Code 在 `init()` 里不是这样。它一上来就先建立 graceful shutdown，再注册 LSP manager 清理、team 清理这类 cleanup。换句话说，这台系统还没正式开始干活，就已经在为未来的结束方式留位置了。

这其实是成熟运行时和演示程序的分水岭。只会启动的系统，出事时往往会留下半开的连接、没清掉的子进程、脏掉的状态和永远挂着的临时资源。Claude Code 明显知道自己不是一次性的脚本，而是一台会跨很多模式、很多会话、很多后台能力长期运行的机器，所以退出策略必须从一开始就进入设计。

这类设计不会直接给用户带来“新功能”的惊喜，却会决定这套系统长期用起来到底干不干净。真正好的 agent 工程，不会把收尾当善后，而会把它当成启动时就该一起定义好的制度。

## 实现链
这页对应的实现链都落在 `init.ts`：配置系统、安全环境变量、CA 证书、代理、平稳退出、预连接和远端 promise 会在真正会话启动前先被装起来。它关注的是运行环境，而不是马上给用户一段对话。

## 普通做法
更常见的写法是把环境准备和会话准备混在一起，等需要网络、证书或清理时再临时处理。

## 为什么不用那种做法
Claude Code 没这么拖后，是因为这些基础设施一旦错位，后面的会话层每走一步都可能踩雷。它宁可先把地基打厚。

## 代价与回报
回报是会话代码不必反复兜底底层问题。代价是 init 阶段要先处理大量用户暂时看不见的东西。
