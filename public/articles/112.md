# Claude Code 源码导图：内核层与连续性的代码簇在哪里

## 读大型项目源码的困境

`claude-code-sourcemap-extracted/src/` 下有 30 多个顶层目录，几百个文件。

你可以按文件名猜功能，但猜出来的是"这个文件做什么"，而不是"这一组文件在一起说明什么设计判断"。

更有价值的阅读方式是：找到功能主题 → 定位对应的代码簇 → 读懂这个簇在整体架构里的位置。

这篇文章是内核层与连续性这条线的源码导图。

## 内核层：负责"系统是什么"的代码

Claude Code 的内核层不是一个目录，而是几个相互依赖的核心文件：

**`src/Task.ts`**：任务的类型定义和生命周期协议。七种 `TaskType`、五种 `TaskStatus`、`TaskStateBase` 公共底座、`generateTaskId` 和 `createTaskStateBase` 工厂函数都在这里。这是所有后台能力的合同。

**`src/Tool.ts`**：工具的定义协议。每个工具需要实现什么接口——`name`、`inputSchema`、`isEnabled()`、`call()`、`renderToolUseMessage()` 等——都在这里定义。这是所有能力的入口契约。

**`src/QueryEngine.ts`**：主循环的核心。它负责把用户输入、历史消息、工具调用和 AI 响应串联成一个对话轮次。读这个文件能看清 Claude Code 的"一轮对话"究竟经历了哪些步骤。

**`src/state/AppState.ts`**（推测路径）：系统的全局状态结构。`teamContext`、任务列表、权限状态、会话信息都在这里聚合。这是所有组件的共同现实。

## 连续性：负责"系统跨越时间保持自己"的代码

**`src/history.ts`**：会话历史的存储和读取。不只是记录对话，而是在进程重启后能恢复到上次的状态。

**`src/migrations/`**：历史数据格式的演进处理。每次 Claude Code 更新改变了存储格式，migration 确保旧会话仍然可以被正确读取。这是系统对历史负责的代码。

**`src/services/`** 目录下的压缩相关服务：长会话的上下文压缩。当对话历史超过模型上下文窗口时，这里的代码决定如何摘要和压缩，同时保留足够的信息让对话继续有意义。

**`src/utils/sessionStorage.ts`**：会话数据的持久化工具。`getAgentTranscriptPath` 这样的函数是连续性基础设施里最低层的砖块——确保每个 agent 的执行记录都有稳定的文件系统位置。

## 两条线的交叉点

内核层和连续性在一个地方汇聚：**任务对象的输出文件（outputFile）**。

`TaskStateBase` 里有 `outputFile: string` 和 `outputOffset: number`。这两个字段让任务输出写到磁盘，而不是只活在内存里。进程重启后，系统可以从 offset 续读，而不是丢失所有进度。

这个设计把内核层（任务协议）和连续性（跨时间保持状态）连接在一起：**任务不只是当前进程里的对象，而是外部化的、可持续的工作记录。**

## 如何阅读这个代码簇

建议的阅读顺序：

1. 先读 `Task.ts`，建立任务类型和生命周期的基本概念
2. 读 `Tool.ts`，理解每个"能力"的接口约定
3. 读 `tasks/types.ts`，看七种任务类型如何组织成 union
4. 读一个具体任务实现（比如 `tasks/LocalAgentTask/LocalAgentTask.tsx`），看公共底座如何被扩展
5. 最后读 `history.ts` 和 migration，理解跨时间的连续性如何被工程化

这个顺序从"协议"到"实现"再到"跨时间"，能看清内核层的结构逻辑。

## 面试要点

1. **内核层不是一个目录，而是一组协议文件**：Task.ts、Tool.ts、QueryEngine.ts 定义了系统是什么，其他代码在这个定义上扩展。
2. **连续性在文件系统上落地**：outputFile、sessionStorage、migrations——连续性的工程基础是磁盘，不是内存。
3. **两条线的交叉是 outputFile**：任务的外部化输出让内核层的任务对象具备了跨时间的连续性。
4. **阅读源码先找协议文件**：大型项目里，先读类型定义和接口协议，再读具体实现，比按文件名猜功能更有效率。
