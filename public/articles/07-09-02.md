# 注册式桥接是在承认打包现实，不是追求理论最美

## 实现链
- `mcpSkillBuilders.ts` 注释写得很直白：变量动态 import 在 Bun 打包产物里会失效，字面量动态 import 又会把依赖图重新拖回大循环里。于是作者改成启动时注册、后面再取用。
- 这不是“理论上最漂亮”的架构，而是面对 Bun、依赖检查和 bundle 现实后做出的务实桥接。

## 普通做法
- 普通软件工程教材更喜欢讲“按需 import”“延迟加载”“谁用谁依赖”。
- 如果只看抽象原则，这些说法都比注册式桥接更干净。

## 为什么不用
- Claude Code 没坚持理论最美，是因为这里的约束不是黑板上的模块图，而是实际打包器和运行时会不会炸。
- 从代码注释看，这一层完全是在承认现实：只要理论漂亮会换来 bunfs 下的找不到模块或新一轮循环依赖，那就不值得。

## 代价
- 代价是实现带一点“工程补丁味”，新读者会觉得不如直接 import 直观。
- 但这种折中通常比执着抽象纯度更成熟，因为它真的能活在当前工具链里。
