# Bun 打包约束下的注册式桥接：当理论最美遇到工具链现实

## 面试题变体：在工具链有限制时，如何做架构决策？

这道题考的不是"书上的最优解"，而是"在真实约束下的最优解"。

`mcpSkillBuilders.ts` 的存在是一个典型案例：理论上有三种解法，但两种在 Bun 打包环境下不可行，所以选了第三种。

---

## 问题的三种解法

**解法 A：变量动态 import**

```typescript
// mcpSkills.ts 里
const modulePath = './loadSkillsDir.js'
const { createSkillCommand } = await import(modulePath)
```

逻辑上简洁——需要的时候才加载，不需要担心循环依赖，因为运行时 `import` 的时机已经晚于可能的循环点。

**为什么不可行：**

```
// mcpSkillBuilders.ts 注释：
The non-literal dynamic-import approach ("await import(variable)") fails at
runtime in Bun-bundled binaries — the specifier is resolved against the
chunk's /$bunfs/root/… path, not the original source tree, yielding "Cannot
find module './loadSkillsDir.js'".
```

Bun 在打包时会把所有模块路径编译成打包格式（`/$bunfs/root/...`）。对于字面量路径（如 `import('./loadSkillsDir.js')`），Bun 知道要做什么。但对于变量路径，Bun 不知道这个变量在运行时会是什么值，所以保持原样——于是运行时找不到模块。

**解法 B：字面量动态 import**

```typescript
// mcpSkills.ts 里
const { createSkillCommand } = await import('./loadSkillsDir.js')
```

字面量路径，Bun 可以正确处理。

**为什么不可行：**

依赖分析工具（dependency-cruiser）会追踪这个 import，发现 `mcpSkills.ts → loadSkillsDir.ts → ... → mcpSkills.ts` 的循环，在 CI 的依赖检查步骤里报错。整个项目的循环依赖规则会被这一个 import 打破，可能带来级联的合规问题。

**解法 C：注册式桥接（现在的做法）**

```typescript
// mcpSkillBuilders.ts：只导入类型，不导入实现
import type { createSkillCommand, parseSkillFrontmatterFields } from './loadSkillsDir.js'

// loadSkillsDir.ts 初始化时注册
registerMCPSkillBuilders({ createSkillCommand, parseSkillFrontmatterFields })

// mcpSkills.ts 使用时取出
const { createSkillCommand } = getMCPSkillBuilders()
```

---

## 为什么解法 C 绕过了两个约束？

**绕过 Bun 打包约束：**

没有任何动态 import。所有 import 都是静态的字面量（`import type`），Bun 可以正确处理。

**绕过循环依赖约束：**

`mcpSkillBuilders.ts` 只 `import type`，类型在编译后消失，不产生运行时依赖边。所以 dependency-cruiser 看到的依赖图里，这条边不存在。

代价是：多了一个"看起来什么都没干"的模块，增加了理解成本。

---

## 注释是第一公民

`mcpSkillBuilders.ts` 的注释相当详细：

```typescript
/**
 * Write-once registry for the two loadSkillsDir functions that MCP skill
 * discovery needs. This module is a dependency-graph leaf: it imports nothing
 * but types, so both mcpSkills.ts and loadSkillsDir.ts can depend on it
 * without forming a cycle (client.ts → mcpSkills.ts → loadSkillsDir.ts → …
 * → client.ts).
 *
 * The non-literal dynamic-import approach ("await import(variable)") fails at
 * runtime in Bun-bundled binaries — the specifier is resolved against the
 * chunk's /$bunfs/root/… path, not the original source tree, yielding "Cannot
 * find module './loadSkillsDir.js'". A literal dynamic import works in bunfs
 * but dependency-cruiser tracks it, and because loadSkillsDir transitively
 * reaches almost everything, the single new edge fans out into many new cycle
 * violations in the diff check.
 *
 * Registration happens at loadSkillsDir.ts module init, which is eagerly
 * evaluated at startup via the static import from commands.ts — long before
 * any MCP server connects.
 */
```

这 9 行注释解释了：
1. 这个模块的目的（dependency-graph leaf）
2. 为什么不用方案 A（Bun 打包路径问题）
3. 为什么不用方案 B（dependency-cruiser 循环检查）
4. 注册时机（`loadSkillsDir.ts` 模块初始化时，通过 `commands.ts` 的静态 import 触发）

这种注释不是在解释代码"做了什么"（代码本身就说了），而是在解释"为什么这样写，以及放弃了哪些备选方案"。这是真正有价值的注释。

## 工具链约束是真实的架构输入

这个案例的更大意义是：工具链约束（Bun 的打包行为、dependency-cruiser 的规则）是真实的架构约束，不能被"理论最优解"忽视。

在纸上画架构图时，"动态 import 解决循环依赖"是完全合理的方案。但在特定工具链下，这个方案可能不可行。

**好的工程师和差的工程师的区别之一**：好的工程师知道他们的方案在哪些工具链下有限制，并能说明备选方案和选择理由。

## 面试指导

**"你遇到过因为工具链限制而不得不改变架构决策的情况吗？"**

这道题考的是工程判断力和诚实程度。

好的回答模式：
1. 描述原本想用的方案（理论最优）
2. 解释工具链限制（具体是什么限制，为什么）
3. 描述最终采用的方案（现实最优）
4. 说明这个方案的代价是什么（比如可读性降低）

`mcpSkillBuilders.ts` 这个案例非常典型，可以用来回答这类问题：
- 想用变量动态 import（简洁）
- Bun 打包不支持
- 想用字面量动态 import（次优）
- dependency-cruiser 规则不允许
- 最终用注册式桥接（代价：额外的模块，初始化顺序约束）

说出"每种解法的代价"比只说"选了这种方案"更成熟。
