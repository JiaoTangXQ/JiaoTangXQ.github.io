# 哪些昂贵工作被故意提前打了出去

Claude Code 在模块刚开始求值时，并没有把所有事情一股脑提前，而是只挑了少数几种“又慢、又大概率会用到、而且可以安全独立启动”的工作先打出去。

最典型的就是两类。第一类是 MDM 读取，这类操作往往要起子进程去读系统级配置；第二类是 keychain 预取，要提前把 OAuth 和旧 API key 的读取跑起来。源码注释写得很直白: 如果不这么做，后面第一次真正读设置时，就会在更关键的路径上同步卡住几十毫秒。

这正说明 Claude Code 的抢跑很有节制。它抢的不是“看起来有点慢的东西”，而是那些后面几乎肯定会踩到、而且一旦晚读就会把真正启动路径拖慢的东西。换句话说，它不是单纯提前做事，而是在提前清理未来最容易挡路的石头。

## 实现链
这页对应的实现链都发生在 `main.tsx` 模块刚开始执行的阶段：先记启动点，再提前打出 MDM 读取、keychain 预取或其他可安全重叠的等待项，真正依赖这些结果的地方再稍后收口。它优化的不是某个函数体，而是整段启动时间河流。

## 普通做法
更普通的做法是把模块加载当成不可控前奏，等进入主逻辑以后再优化；或者一见到慢操作就一股脑并发开跑。

## 为什么不用那种做法
Claude Code 两条路都没选死。它不愿意浪费模块求值阶段的等待，但也不把依赖关系抹平，所以才形成了先开跑、后会合的两段式启动。

## 代价与回报
回报是把等待藏进原本就存在的时间缝。代价是理解入口时要同时记住什么已经启动、什么还不能消费结果。
