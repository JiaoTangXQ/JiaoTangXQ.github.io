# prompt 命令如何临时重摆人、模型、系统、工具的关系

面试题版本：**"用户执行 `/code-review` 这个技能命令后，当前会话发生了什么变化？这和用户直接在对话里说'帮我做代码审查'有什么本质区别？"**

## 直接说话 vs prompt 命令

**直接说话**：用户说"帮我做代码审查"，这句话进入对话历史，模型在当前会话的上下文里理解这个请求，用现有的工具权限和行为规则来响应。

**prompt 命令**（`/code-review`）：这个命令加载了一个预定义的技能，技能包含：
- 专门的系统提示词（"你是一个严格的代码审查专家，重点关注..."）
- 特定的工具许可（可能需要允许文件读取、不允许文件修改）
- 特定的 hooks 配置（审查结束后自动生成报告）
- 可能的初始附件（代码审查标准文档）

执行 prompt 命令后，后续的对话是在这个新配置的框架下运行的，不是在默认的通用框架下运行。

## prompt 命令的实现

`getMessagesForSlashCommand()` 处理 `'prompt'` 类型命令时：

```typescript
case 'prompt': {
  const skill = await loadSkill(command.skillPath)
  
  // 组装技能带来的所有材料
  const sessionConfig = {
    systemPrompt: skill.systemPrompt,
    toolPermissions: skill.allowedTools,
    hooks: skill.hooks,
    attachments: skill.initialAttachments,
    resumeState: skill.resumeConfig,
  }
  
  // 重新配置当前会话
  setAppState(s => applySessionConfig(s, sessionConfig))
  
  // 把技能激活消息注入会话
  return {
    type: 'session_reconfigured',
    content: skill.activationMessage,
  }
}
```

注意：这不是"发送一段消息给模型"，而是"重新配置整个会话的运行框架，然后让新框架生效"。

## 工具许可的重要性

技能命令里最关键的部分之一是工具许可的变更。

**默认会话**：模型有所有通用工具可用（读文件、写文件、bash、搜索等）。

**代码审查技能**：可能只允许读文件和搜索，不允许修改任何文件——因为审查只应该观察，不应该修改代码。

这个工具许可的限制不只是安全措施，也是行为约束：在"只能读、不能写"的工具配置下，模型自然会把注意力放在分析和报告上，而不是修改代码。工具许可在塑造模型的行为边界。

## 为什么要一起带进来

"技能正文、工具许可、hooks、附件和恢复一起带进当前会话"这里强调"一起"的原因：

**它们互相依赖**：技能的系统提示词可能说"你可以使用 file_read 工具来分析文件"，但如果工具许可里没有 file_read，这句话变成了模型的错误认知（它以为可以用但实际不能用）。技能正文和工具许可必须一起配置，不能分开。

**压缩和恢复需要完整上下文**：当会话被压缩后，新的会话需要知道"这次会话从哪个技能开始的"。如果技能配置只是一段文本注入，压缩后可能找不到完整的技能定义，恢复时会丢失部分配置。一起带进来的完整配置让压缩和恢复有可靠的参考点。

**hooks 依赖提示词的语义**：`postToolCall` hooks 可能依赖系统提示词定义的行为规则（"每次文件操作后都要更新审查日志"）。如果 hooks 和提示词分开加载，它们可能不匹配。

## 面试角度的总结

prompt 命令和直接说话的区别是：前者是在**配置工作框架**，后者是在**框架内工作**。prompt 命令把这个会话从"通用框架"切换到"特定技能框架"，不只是多了一段文字，而是改变了整个会话的能力边界、行为规则、工具配置。这种"重摆关系"的能力让 Claude Code 可以在同一个会话里切换工作模式，而不需要为每种工作模式开启单独的会话。
