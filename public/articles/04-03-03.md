# 工具可见性和权限边界怎样在动模型前先决定

很多人会想当然地以为，模型先开始想，想到需要什么工具时，系统再临时决定给不给。Claude Code 没这么做。它会先把这一轮可见的行动空间整理好，再把模型放进去思考。

这里其实有两层边界。第一层是“看得见什么”。工具池会先按当前权限上下文装配出来，连 MCP 工具也一起纳入，然后把那些被整条禁止的工具先过滤掉。也就是说，有些工具不是等模型调用时才被拒绝，而是从一开始就不出现在它眼前。第二层是“即使看得见，实际能不能做”。到了真正调用时，系统还会再按当前权限模式、规则和工具自身逻辑做一次判定。

这两层同时存在，非常见功力。只做后一层，模型会反复朝着根本不可能的方向思考；只做前一层，又会把实时边界变化处理得过于粗糙。Claude Code 先决定行动空间，再决定单次行动是否放行，于是模型面对的是一个比较真实、又不会过早僵死的世界。

这说明它理解的不是“怎么把工具挂给模型”，而是“怎么给模型一个可信的工作现场”。优雅并不在工具多，而在行动空间先被整理成了可以推理的样子。

更具体地说，`query()` 在真正调用模型前就会把 `toolUseContext.options.tools`、当前 `toolPermissionContext`、MCP 工具、agent 定义和 allowedTools 一起整理成这一轮可见的行动空间；等模型真的发出 `tool_use`，还会再经过 `canUseTool()` 和工具自身校验。更普通的系统常常只做后一层：先把所有工具都暴露给模型，等它调用了再弹权限框或者报错。这种写法实现简单，但模型会在一块虚假的行动地形上思考，反复规划本来就不可能执行的动作。

Claude Code 先收窄可见空间、再处理单次放行，本质上是在减少“无效推理”。它没有把权限只当拦截器，而是当成模型推理现场的一部分。代价是规则会分散在装配和执行两头，新手读代码时会觉得“为什么明明前面过滤过了，后面还要再判一遍”，但这正是静态边界和动态边界分开承担职责的结果。

## 实现链
模型看到的工具集合先按权限和当前上下文装配，真正执行时再过一次 `canUseTool()`。

## 普通做法
把所有工具都先暴露给模型，调用时再拒绝。

## 为什么不用
因为那样模型会在虚假的行动空间上白白推理。

## 代价
策略分散在装配和执行两层。
