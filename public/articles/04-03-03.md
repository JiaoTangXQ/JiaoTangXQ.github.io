# 工具权限为什么要分「可见性」和「执行权限」两层来管？

一个常见的直觉是：工具权限只需要一层拦截器。模型想调用某个工具，执行前检查权限，不允许就返回错误消息。这个设计简单、直接，很多系统就是这么做的。

Claude Code 的做法不一样。它把工具访问控制显式拆成了两层，在不同时机起作用。理解为什么要这样拆，能帮你回答一类更通用的系统设计问题：什么时候应该在入口过滤，什么时候应该在执行时判断。

## 第一层：可见性过滤（模型调用前）

`query()` 在构建 `toolUseContext` 时，就已经把这一轮可见的工具集合确定下来了。这个集合来自几个来源的组合：

- 基础工具（`getAllBaseTools()`）
- MCP 工具（`appState.mcp.tools`）
- Agent 工具（`toolUseContext.options.agentDefinitions`）
- 按权限模式过滤掉不允许的工具

这个集合会作为 `tools` 参数传给 API 调用：

```typescript
for await (const message of deps.callModel({
  messages: ...,
  systemPrompt: ...,
  tools: toolUseContext.options.tools,  // ← 已过滤的工具集
  ...
}))
```

模型在生成 `tool_use` 时，只能从这个集合里选。**看不见的工具，它不会尝试调用。**

## 第二层：执行权限（每次调用时）

当模型真的生成了一个 `tool_use` 块，执行前还会经过 `canUseTool()` 的判断：

```typescript
const result = await canUseTool(
  tool,
  input,
  toolUseContext,
  assistantMessage,
  toolUseID,
  forceDecision,
)
```

这个函数检查的是**实时**权限：当前工作目录是否在允许范围内？文件路径是否符合规则？当前权限模式（plan / autoApprove / interactive）是否允许这个操作？有没有用户配置的拒绝规则？

返回 `behavior !== 'allow'` 时，`QueryEngine` 会把这次拒绝记入 `permissionDenials`，用于 SDK 层的汇报。

## 为什么需要两层？

**如果只有执行层（无入口过滤）：**

模型会在一个虚假的工具集合上推理。比如在 `plan` 模式下，很多写操作工具实际上不会被允许。如果这些工具依然出现在模型的视野里，模型会制定包含写操作的计划，然后反复遇到"Permission denied"，不得不重新规划。这是真实的无效推理成本。

**如果只有入口过滤（无执行层）：**

静态过滤无法处理动态条件。工作目录可以在工具执行过程中改变；用户在对话中途可以更新权限设置；MCP 服务器的状态可能随时变化。这些变化都需要在执行时重新评估，而不是在会话开始时静态决定。

## 两层边界的实际位置

```
模型可见空间 (静态):
  buildQueryConfig() 时确定 streamingToolExecution 等 gate
  toolUseContext.options.tools 列表
  MCP/Agent tool 注册状态
  ↓
模型生成 tool_use
  ↓
执行边界 (动态):
  canUseTool() — 权限模式、路径规则、实时配置
  tool.inputSchema.safeParse() — 输入合法性
  tool 自身逻辑 — isConcurrencySafe 等判断
```

两层各司其职：静态边界减少无效推理，动态边界保证实时安全。

## 代价与工程取舍

这个设计的代价是**策略分散**。新读者看到执行层的 `canUseTool()` 时，常常会问：明明前面已经过滤过了，为什么还要再判一次？这会让代码多了一层概念负担。

另一个代价是**两层需要保持一致**。如果静态过滤了某个工具，但执行层还留着允许逻辑，会引入概念混乱。如果两层策略不同步，会出现"看得见但调不了"的状态——这对模型来说比"完全看不见"更糟糕，因为模型会在一个已经调用失败过的工具上反复尝试。

## 面试指导

这道题的通用版本是："访问控制应该在哪一层做？"

一个成熟的回答会区分两类控制的目标不同：
- **可见性控制（白名单入口）**：减少推理噪音，让 LLM 只在真实可用的行动空间里工作
- **执行控制（动态判断）**：应对实时变化，处理无法在入口静态决定的边界条件

能把这两点说清楚，并且能说出"只做一层"的代价是什么，就是一个完整的回答。Claude Code 的两层实现可以作为这个框架的具体例证。
