# 面试题：状态栏的信息排列顺序是怎么决定的？为什么 mode 必须排在任务标签前面？

这是一道看起来是 UI 细节的题，实际上考察的是你对"当屏幕空间有限时，谁有优先权"的系统性思考。

## 问题的上下文

终端底部一行是有限资源。当内容超出终端宽度，靠后的部分会被截断。所以排列顺序不只是美学选择，而是一个隐式的**优先级声明**：写在前面的，是遇到空间紧张时最后被牺牲的。

## 源码里的排列逻辑

在 `PromptInputFooterLeftSide.tsx` 里，拼接各部分时有明确的顺序注释：

```typescript
// modePart is placed first so it won't be displaced by long task pills
const parts = [
  modePart,    // 1. permission mode — 最先
  tasksPart,   // 2. 任务提示
  remotePart,  // 3. 远端状态
  prPart,      // 4. PR 状态（条件显示）
  // ...
];
```

注释本身就是答案：**modePart 排第一是为了防止长任务标签把 mode indicator 挤掉**。

## 为什么这个顺序重要

**Permission mode 直接决定下一步能做什么**

当前 permission mode 有三个主要状态：
- `default`：需要逐项确认工具操作
- `acceptEdits`：文件编辑不需要确认
- `autoApprove` / `--dangerously-skip-permissions`：所有操作自动批准

这个状态决定了用户接下来能不能放心让 agent 执行某个操作。如果你以为当前是 default 模式，但实际上是 autoApprove，做出的操作判断会完全不同。

**任务 pills 是"当前热闹"，mode 是"当前制度"**

任务 pill 告诉你"现在有几件事在进行"，这是动态的工作状态。Permission mode 告诉你"这些事在什么安全框架下进行"，这是更底层的约束条件。

一个类比：知道会议室里有三个项目组在开会（任务），和知道这栋楼现在是访客管控区（制度），后者对你的行为约束更根本。

**任务 pill 更容易很长**

实际使用中，任务描述可能很长，比如"researching authentication in legacy codebase for PR review"。Mode 指示器则是极短的几个字。把长的放前面，短的放后面，短的在 wrap 时更容易被吃掉。

## 截断策略

这个顺序配合 `Text wrap="truncate"` 的行为，形成了一套隐式的截断策略：

```
[mode: default] [task: research auth...] [remote: session-xyz] [PR: #123]
                                                                 ↑ 首先被截断
                                                 ↑ 其次被截断
                     ↑ 然后被截断
↑ 最后被截断
```

也就是说，在越来越窄的终端里，用户丢失信息的顺序是：先丢 PR 状态，再丢远端链接，再丢任务细节，最后才丢 permission mode。这个丢失顺序是对用户最友好的降级策略。

## 类比：操作室仪表盘的设计

专业操作室（飞机驾驶舱、核电站控制室）的仪表布局有一个原则：最关键的系统状态指示器必须在视觉中心，而不是角落。不是因为那些指示器最常看，而是因为一旦出错，被第一眼注意到最重要。

Claude Code 的状态栏在一维空间里做了同样的事：把"安全制度状态"放在最不容易被挤掉的位置。

## 面试指导

**入门问法**："底部区域信息是按什么顺序排的？"

答：permission mode 排最前，然后是任务提示、远端状态、PR 状态。这个顺序是信息优先级的声明——越靠前的，越是遇到空间不足时最后被截断的。

**进阶问法**："如果你来设计这个顺序，依据是什么？"

答：看每个字段对用户行为判断的影响程度。Permission mode 直接决定安全边界，是最基础的约束条件，必须先保。任务状态是动态工作信息，PR 状态是补充信息，优先级更低。同时要考虑字段长度——短的放前面，窄屏时更容易保住。

**系统性追问**："Claude Code 里还有哪些地方体现了类似的优先级排序？"

答：`shouldShowPrStatus` 的条件（`primaryItemCount < 2`），明确要求主字段不满时 PR 才能出现；任务面板的 `maxDisplay` 优先级顺序（最近完成 → 进行中 → 待处理 → 旧的完成）；`filterForBriefTool` 里 system message 必须保留但 api_metrics 可以丢弃。

**陷阱问题**："你会用 flexbox justify-content: space-between 来实现这个布局吗？"

答：终端里 Ink 框架用的是 Yoga（CSS Flexbox 的 C 实现），而不是真正的 CSS。`wrap="truncate"` 是 Ink/Yoga 特有的属性，意思是超出宽度时截断而不是换行。这和浏览器里的 `text-overflow: ellipsis` 类似但不完全一样——理解这个环境差异，才能正确实现终端 UI 里的截断优先级控制。

---

*核心考察点：信息优先级的系统性思考、截断策略的设计动机、从 UI 细节推导设计原则的能力。*
