# 点击未挂载的 sticky 目标：估算 + pending + 校正，三步完成虚拟化跳转

这里的问题比看起来复杂：用户点击了 sticky header，要跳回对应的 prompt 消息，但那条消息在虚拟列表里还没挂载——它在 topSpacer 里，没有对应的 DOM 节点。

## 三步策略

**第一步（点击时）**：把 `capturedEstimate`（当前已知的位置估算）存下来，立即 `scrollTo(capturedEstimate)` 把视口移到大概位置。这会触发虚拟列表重算，把目标消息附近的区域挂载到 DOM 里。

**第二步（并行）**：把目标消息的 `idx` 存入 `pending.current`。这是一个「等目标出现后要做的事」的标记。

**第三步（目标挂载后）**：虚拟列表检测到 `pending.current` 有值，且目标消息的 DOM 元素已经出现，调用 `scrollToElement` 精确校正位置，然后清空 `pending.current`。

## 「pending」的作用

`pending.current` 是一个跨渲染周期的意图传递机制。第一步只是让目标消息「可能」在视口范围内，第三步才能精确定位。两步之间可能跨越多个渲染周期——虚拟列表重算、消息挂载、DOM 更新……

用 ref 而不是 state 存储 `pending`，是因为这个意图不需要触发渲染，只需要在下一次合适的时机被读取和执行。

## 接受断层，而不是假装没有

这套三步策略的前提是：**虚拟化断层是真实存在的**，必须显式处理，不能假装目标消息总是已挂载的。

很多跳转逻辑会假设「目标元素肯定在 DOM 里」，在普通列表里这完全正确，但在虚拟列表里是错的。认清断层，设计出容纳断层的处理流程，这是虚拟化环境下实现可靠跳转的关键思维转换。
