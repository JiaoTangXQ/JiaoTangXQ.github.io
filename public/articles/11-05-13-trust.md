# 面试题：状态栏命令被 trust 拦住时，Claude Code 为什么不静默跳过而是发通知？

这是一道关于"静默失败"危害性的设计题。答案直接，但能展开讲清楚"为什么"的候选人，比只说"因为这样更好"的候选人更有说服力。

## 场景还原

```typescript
// StatusLine.tsx mount 时
useEffect(() => {
  const statusLine = settings?.statusLine;
  if (statusLine) {
    // ...
    if (!checkHasTrustDialogAccepted()) {
      addNotification({
        key: 'statusline-trust-blocked',
        text: 'statusline skipped · restart to fix',
        color: 'warning',
        priority: 'low',
      });
      logForDebugging('Status line command skipped: workspace trust not accepted', {
        level: 'warn',
      });
    }
  }
}, []);
```

两个操作：
1. 在 UI 里添加一条 warning 通知（`addNotification`）
2. 在 debug 日志里记录 warn（`logForDebugging`）

注意：这不是在 hook 命令失败时触发，而是在**检测到 trust 没通过**时触发——命令甚至还没有被尝试执行。

## 静默失败的危害分析

如果选择静默跳过（什么通知都不发），会发生什么？

**场景 1：用户第一次配置状态栏**

用户在 `.claude/settings.json` 里配置了一个自定义状态栏脚本，重启后期待看到状态栏显示。但界面上什么都没有。用户会以为：
- 脚本格式写错了？
- 命令路径不对？
- 这个功能只在特定环境下工作？

他们会花时间排查一个根本不是语法或路径问题的 bug，因为真正的原因（trust 没通过）从来没被告知过他们。

**场景 2：状态栏突然"消失"**

在某次 workspace trust 状态变化后，状态栏停止显示。用户发现状态栏不见了，但界面完全正常，没有任何错误提示。用户会开始怀疑是否是终端渲染问题，或者是某次更新改变了行为。

**静默失败的本质问题**：它让两种完全不同的状态看起来一样——"从未配置状态栏"和"配置了但被 trust 拦截"。用户无法区分这两种情况，可诊断性为零。

## 为什么是 low 优先级的 warning

通知有优先级（`priority: 'low'`），这是个有意的设计决策。

**不是高优先级的理由**：这个问题不是立即需要用户处理的紧急问题。状态栏不显示不会导致操作失败，只是缺少了一个便利功能。把它定为高优先级 warning 会制造不必要的焦虑。

**不是完全忽略的理由**：用户配置了功能却看不到效果，这是一个值得知道的状态，特别是在排查问题时。低优先级通知出现在合适的位置，不打断当前任务，但在用户觉得"好像少了什么"时提供了线索。

**`text: 'statusline skipped · restart to fix'`**

注意通知文字里还给了解决方向："restart to fix"。这不只是告知问题，还指向了解决路径。诊断信息的好坏，不只是"有没有通知"，还包括"通知是否有行动意义"。

## logForDebugging 和 addNotification 的区别

两个操作服务于不同的受众：

**`addNotification`**：对话框式通知，用户能看到。适合"用户需要知道"的信息。

**`logForDebugging`**：写入 debug 日志文件，用户通常看不到，除非主动去查 debug 日志。适合"开发者或高级用户排查问题时需要"的详细信息。

把同一个事件同时写到两个地方，覆盖了两种不同程度的调查需求。普通用户看通知，高级用户看日志。

## 更广泛的原则：缺席本身是信息

这里的设计体现了一个重要原则：**一个本来应该存在的东西不在了，这个"缺席"本身是需要传达的信息**。

这在软件设计的很多地方都有应用：

- 数据库：NULL 和 0 的区别不能模糊——0 表示"我知道，值是零"，NULL 表示"我不知道这个值"
- HTTP 状态码：404 Not Found 和 204 No Content 是不同的——前者说"你找的东西不在"，后者说"操作成功，但没有内容返回"
- 监控系统：指标缺失（missing）和指标为零（zero）应该触发不同的告警策略

在 Claude Code 的状态栏里，`statusLineText === undefined`（命令还没返回）和 `statusLineText === ''`（命令返回了空字符串）和"命令被 trust 跳过所以从未执行"，是三种不同的状态，应该有不同的视觉/通知响应。

## 面试指导

**直接问法**："trust 挡住状态栏命令时，为什么不静默跳过？"

答：因为用户配置了状态栏但看不到效果，界面上没有任何提示，会让他们花时间排查错误的方向。静默失败让"配置了但被拦截"和"从未配置"看起来一样，可诊断性为零。发一条 low 优先级通知，并写 debug 日志，给了不同程度的诊断入口。

**追问**："为什么是 low 优先级而不是高优先级？"

答：状态栏不显示不影响核心操作，不是紧急问题。低优先级通知不打断当前任务，只在用户注意到"缺少什么"时提供线索。如果用高优先级，会制造不必要的焦虑，也会让通知系统因为"次要问题"而降低对"真正紧急问题"的可信度。

**设计追问**："通知文字 'statusline skipped · restart to fix' 的设计有什么值得注意的地方？"

答：它不只是说"发生了什么"（skipped），还说了"怎么修"（restart to fix）。好的错误/警告信息应该包含行动指引，而不只是状态描述。用户看到这条通知，知道下一步是什么，而不是停在"知道有问题但不知道怎么办"的状态里。

**类比追问**："你能举个类似的'静默失败有害'的例子吗？"

答：很多。比如后台任务失败但不通知用户（用户以为任务还在进行中）；配置文件解析失败但用了默认值（用户以为自定义配置生效了）；认证 token 过期但显示旧数据（用户以为看到的是实时数据）。共同点是：应该出现的东西没出现，界面没有告知，用户在错误的假设下继续操作。

---

*核心考察点：静默失败的危害、缺席状态的诊断可见性、错误通知的优先级和行动指引设计。*
