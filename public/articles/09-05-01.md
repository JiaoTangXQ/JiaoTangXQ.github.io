# 5 个 AI Agent 同时写同一个 JSON 文件，Claude Code 怎么保证消息不丢？

我第一次看到 Claude Code 的多 Agent 通信方案时，以为自己看错了。

没有 Redis。没有 RabbitMQ。没有 gRPC stream。没有 WebSocket 推送。5 个 AI Agent 之间的通信，靠的是往 `~/.claude/teams/{team}/inboxes/{agent}.json` 文件里写 JSON。

这看起来像是大学课程作业的方案，但它跑在生产环境里，而且**真的不丢消息**。怎么做到的？

## 先看一个会丢消息的版本

假设你要实现一个简单的文件信箱：Agent A 给 Agent B 发消息，往 B 的信箱文件追加一条记录。最直觉的写法：

```
1. 读取 inbox.json，得到消息数组
2. 往数组里 push 新消息
3. 把整个数组写回 inbox.json
```

如果只有一个发送者，这毫无问题。但当 Agent A 和 Agent C 同时给 Agent B 发消息：

```
时间线：
  Agent A                    Agent C
  ─────                      ─────
  读取 inbox.json (3条消息)    读取 inbox.json (3条消息)
  push 第4条                  push 第4条 (不同内容)
  写回 inbox.json (4条)       写回 inbox.json (4条)
```

Agent C 的写入覆盖了 Agent A 的写入。Agent A 的消息丢了。这就是经典的 **read-modify-write 竞态条件**。

面试中如果有人说"加个锁就行了"，那只答对了三分之一。Claude Code 确实加了锁，但锁只是第一层防护。

## 第一层：文件锁 + 指数退避

Claude Code 使用 `proper-lockfile` 库对信箱文件加排他锁。锁的配置是：

```
重试次数: 10
最小等待: 5ms
最大等待: 100ms
退避策略: 指数退避
```

为什么是 10 次、5-100ms？这不是随便填的数字。Claude Code 的典型 swarm 规模是 2-10 个 Agent，每个 Agent 发消息的频率在秒级。在这个负载下：

- **5ms 最小等待**：足够短，不会让发送者感觉到延迟
- **100ms 最大等待**：即使 9 个 Agent 同时抢锁，指数退避也能在 100ms 内错开
- **10 次重试**：在极端情况下（比如磁盘 I/O 突然变慢），10 次重试提供了约 500ms 的容忍窗口

这组参数的设计思路是：**为常见场景优化延迟，为极端场景保留容错**。

## 第二层：锁后重读——最容易忽略的关键步骤

拿到锁之后，Claude Code 做的第一件事不是写入，而是**重新读取信箱文件**。

为什么？因为在你等待锁的那几十毫秒里，其他 Agent 可能已经修改了文件。如果你还用锁前读到的旧数据来写入，就会丢掉那些修改。

这就是"锁后重读"（read-after-lock）模式，也叫"乐观读 + 悲观写"。完整流程：

```
1. 确保信箱文件存在（用 flag:'wx' 原子创建，EEXIST 则跳过）
2. 获取文件锁（最多重试 10 次，指数退避）
3. 重新读取信箱文件  ← 关键！拿到锁后才读
4. push 新消息
5. 写回整个文件
6. 释放锁（在 finally 块里，保证即使出错也释放）
```

第 3 步是整个方案的灵魂。没有它，即使有锁，也只是把竞态窗口从"读写之间"缩小到了"锁获取到读取之间"。有了它，竞态窗口缩小到零。

## 第三层：原子创建防止初始化竞态

信箱文件不存在时需要创建。如果两个 Agent 同时检测到文件不存在：

```
Agent A: fs.existsSync('inbox.json') → false → fs.writeFileSync('inbox.json', '[]')
Agent C: fs.existsSync('inbox.json') → false → fs.writeFileSync('inbox.json', '[]')
```

这又是一个竞态。Claude Code 的做法是用 `flag: 'wx'` 模式写文件——这是 POSIX 标准的原子操作：如果文件不存在则创建，如果已存在则抛出 `EEXIST` 错误。代码捕获 EEXIST 后静默忽略，因为这意味着其他 Agent 已经帮我们创建好了。

这个细节很小，但**恰恰是初级和高级工程师的分水岭**。初级工程师会写 if-exists-then-create（两步操作，中间有竞态窗口），高级工程师知道用原子文件操作一步到位。

## 为什么不用"更好"的方案？

面试官可能会追问：为什么不用 Redis、数据库、或者进程间 IPC？

这个问题的答案不在技术层面，而在**部署约束**层面。Claude Code 的多 Agent 协作要支持：

- **tmux pane 模式**：每个 Agent 是独立进程，可能在不同 tmux pane 中
- **进程内模式**：多个 Agent 在同一进程中，通过 AsyncLocalStorage 隔离
- **跨设备模式**：通过远端桥接连接的 Agent

文件系统是这三种模式的**最低共同通信面**。tmux pane 里的进程不共享内存，进程内的 Agent 可以访问文件系统但不一定有 Redis，远端 Agent 可以通过挂载共享文件系统。

如果选 Redis，tmux 模式需要每台机器启动 Redis 服务。如果选 IPC，进程内模式和远端模式需要不同的 IPC 实现。如果选数据库，部署复杂度直接上一个台阶。

文件系统是唯一一个**所有部署形态都天然支持**的通信层。选它不是因为"最优"，而是因为"最通用"。

## 还有一个你可能没注意到的细节

`proper-lockfile` 的导入用了懒加载：

```
let _lockfile: Lockfile | undefined

function getLockfile(): Lockfile {
  if (!_lockfile) {
    _lockfile = require('proper-lockfile')
  }
  return _lockfile
}
```

为什么？因为 `proper-lockfile` 依赖 `graceful-fs`，后者在首次 require 时会 monkey-patch 所有 fs 方法，耗时约 8ms。如果在模块顶层 import，这 8ms 会加到每次启动的耗时里——即使用户只是跑了个 `claude --help`。

这种"延迟到首次使用时才加载"的模式，在 CLI 工具中极其常见。Claude Code 的启动路径上充满了这类优化，每个省几毫秒，累计起来就是冷启动从"能感觉到延迟"变成"秒开"。

## 消息结构：颜色和摘要不是装饰

每条消息的类型定义：

```
TeammateMessage:
  from:       发送者名称
  text:       消息内容
  timestamp:  ISO 时间戳
  read:       是否已读
  color?:     发送者颜色（red, blue, green...）
  summary?:   5-10 词的预览
```

`color` 和 `summary` 看起来是 UI 装饰，但它们实际上解决了一个真实的 UX 问题：当 5 个 Agent 同时在工作时，lead Agent 的通知栏里会堆满消息。颜色让你一眼区分谁发的，摘要让你不用展开就能判断优先级。

而 `read` 字段的更新也走了完整的锁流程——`markMessageAsReadByIndex()` 同样是锁后重读再写入，防止标记已读时覆盖了新到的消息。

## 面试怎么答

如果面试题是"设计一个多进程间的消息传递系统"：

**错误答案**：直接上 Redis/Kafka，讲一堆分布式消息队列的知识。面试官想看的不是你会用中间件，而是你理解并发问题的本质。

**及格答案**：文件 + 文件锁。说明为什么需要锁，锁如何防止竞态。

**优秀答案**：文件 + 文件锁 + 锁后重读 + 原子创建 + 指数退避。每一层防护解决什么问题，为什么单独的锁不够，为什么指数退避比固定等待好，为什么用 `wx` flag 而不是 if-exists-then-create。

**满分答案**：在优秀答案的基础上，还能解释**为什么选择文件系统而不是更"高级"的方案**——不是技术限制，而是部署约束决定了通信层的选择。最好的工程决策不是选最强的技术，而是选约束条件下的最优解。

这道题考的不是你知道多少中间件，而是你对**并发原语的理解深度**，以及在真实约束下做工程权衡的能力。
