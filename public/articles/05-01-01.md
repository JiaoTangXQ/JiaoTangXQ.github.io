# 会做事还不等于进入任务世界

“能跑”只是执行能力，“进入任务世界”则意味着系统已经能认出这次劳动、盯住它、干预它、回收它。

## 实现链

在 Claude Code 里，真正进入任务世界的标志不是 `runAgent()` 被调用，而是 `createTaskStateBase()` 生成了统一骨架，并且 `registerTask()` 把它放进 `AppState`。`src/Task.ts` 给出最小共同字段，`src/utils/task/framework.ts` 负责登记，之后各种 UI、通知、回收逻辑才有地方接上。

换句话说，系统不是靠“某段代码现在正在跑”来认定任务存在，而是靠“这段劳动已经有正式任务状态”来认定它存在。

## 普通做法

普通做法是把执行过程本身当主角。只要子进程已经启动、模型请求已经发出，开发者就会觉得任务已经存在；其他东西以后再说。

这样做最像人类临时派活：先干，记账以后补。

## 为什么不用

Claude Code 不这么做，是因为执行过程天然不稳定。一个调用可能很快结束，可能进入后台，可能中途被终止，可能被父任务接管，可能需要把输出挂到单独文件上。如果系统只是盯着“正在跑的调用”，那一旦执行方式变化，管理能力就会断。

它先建任务对象，等于先给这次劳动一个稳定身份。后面无论执行换多少层壳，这个身份都还在。

## 代价

这意味着很多看似简单的操作都不能直接写成“函数跑完就算完”。开发者必须接受一个更正式的世界：先建对象，再跑执行，再同步状态，再决定怎么退场。写起来更繁琐，但换来的是长期稳定。
