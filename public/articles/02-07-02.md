# processSessionStartHooks() 在 launchRepl() 之前还是之后？为什么？

## 答案：之前，但有条件地等待

```typescript
// main.tsx（简化，交互模式）
await showSetupScreens(...)

// SessionStart hooks
await processSessionStartHooks(...)  // ← launchRepl 之前

// 准备初始状态
const initialState = getDefaultAppState(...)

// 启动 REPL
await launchRepl(root, { initialState, ... }, replProps, renderAndRun)

startDeferredPrefetches()
```

`processSessionStartHooks()` 在 `launchRepl()` 之前调用，也就是在首屏出现之前执行。

但"在首屏之前"不意味着"等 hook 全部完成才渲染"——`processSessionStartHooks()` 有自己的并发处理逻辑。

## SessionStart hook 是什么？

用户可以在 `.claude/settings.json` 里配置 hooks：

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'Session started at $(date)' >> ~/.claude/sessions.log"
          }
        ]
      }
    ]
  }
}
```

每次 Claude Code 启动时，这些命令会在会话开始时执行。用途包括：记录会话日志、设置环境变量、检查前置条件等。

## 为什么在首屏之前执行？

**原因一：hook 可能影响 initialState**

某些 SessionStart hook 可能修改环境变量，而 `initialState` 包含从环境变量读取的信息。如果 hook 在 `getDefaultAppState()` 之后执行，那些环境变量修改就无法反映到初始状态里。

**原因二：hook 可能执行关键的前置检查**

用户可能用 SessionStart hook 来验证前置条件：

```bash
# 检查必要工具是否安装
if ! command -v gh &> /dev/null; then
  echo "Error: gh CLI not installed" >&2
  exit 1
fi
```

如果 hook 执行失败（exit code != 0），会话可能需要中止。这个判断必须在首屏渲染之前完成，否则用户会看到 REPL 界面但实际上前置条件不满足。

**原因三：hook 结果需要在首屏前可见**

某些 hook 向 stdout 输出信息，用户期望在进入 REPL 之前看到这些输出（比如欢迎信息、环境状态报告）。

## 但是：并不总是等待 hook 完成

```typescript
// processSessionStartHooks 的实现（简化）
export async function processSessionStartHooks(...): Promise<void> {
  const hooks = getSessionStartHooks();
  if (hooks.length === 0) return;
  
  // 对于超时或后台类型的 hook，不等待
  const results = await Promise.allSettled(
    hooks.map(hook => executeHookWithTimeout(hook, HOOK_TIMEOUT))
  );
  
  // 记录失败，但不中止（取决于 hook 的 error mode）
}
```

`processSessionStartHooks()` 会等待所有 hook 完成（或超时），但超时不阻塞——如果 hook 超过了配置的超时时间，它会被视为"已完成（超时）"，主流程继续。

这是一个合理的权衡：Hook 系统应该对用户体验有保障，用户写了一个慢 hook 不应该导致 Claude Code 永远无法启动。

## 与 setup hooks 的区别

Claude Code 还有另一类 hook：`processSetupHooks()`（在 `main.tsx` 里更早的位置调用）。

```typescript
// 很早，在 init() 之前
await processSetupHooks(...)

// ...init()...setup()...showSetupScreens()...

// SessionStart
await processSessionStartHooks(...)

// launchRepl()
```

`SetupHooks` 在连配置系统都还没初始化的阶段运行，用于非常基础的环境准备（比如设置某些必要的环境变量，这些变量需要在 `init()` 之前生效）。

`SessionStartHooks` 在 trust 确认之后运行，可以访问完整的会话上下文。

这个分层说明：不是所有 hooks 都在同一个时机运行，hook 的类型决定了它的执行时机，执行时机决定了 hook 能看到什么环境状态。

## 面试指导

"hooks 在哪个时机执行"是插件系统设计里的核心问题。

面试评估要点：
1. **执行时机的语义**：这个 hook 类型在什么时候运行？运行时系统状态是什么？
2. **失败处理**：hook 失败是阻塞还是警告？超时怎么处理？
3. **顺序保证**：多个 hook 是串行执行还是并行执行？
4. **副作用范围**：hook 能修改什么系统状态？有什么隔离？

Claude Code 的 hook 系统展示了"不同类型 hook 对应不同执行时机"的设计——SetupHooks 极早运行（基础环境准备），SessionStartHooks 在 trust 后运行（可访问完整上下文）。这种分层比"只有一类 hook，在某个固定时机运行"要灵活得多，但也更复杂。能分析这个权衡，是系统设计能力的体现。
