# 任务世界前半段导读：先把劳动翻译成可管理对象

这一章前半段聚焦的是“任务对象为什么先于执行细节出现”。这里的主角不是模型，不是 prompt，而是 `Task.ts`、任务框架和那几个看起来很朴素的公共字段。

## 实现链

代码上的顺序很明确。`src/Task.ts` 先把任务种类、终态判断、基础字段和 ID 生成固定下来；`src/tasks.ts` 再把不同任务类型接成统一注册表；`src/utils/task/framework.ts` 负责统一登记和更新。也就是说，在 `LocalAgentTask` 或 teammate 这些具体角色真正出场前，系统已经先把“任务对象长什么样”定义好了。

这一层是第五卷的入口，因为它回答的是最基础的问题：为什么 Claude Code 不把 agent 看成“会做事的黑盒”，而要先把它翻译成“可登记、可观察、可回收的对象”。

## 普通做法

很多系统会反过来做。先把执行跑通，再慢慢补状态。比如先让 agent 能调用模型、能返回结果，后面再额外加一张任务表，或者给 UI 临时挂几条状态字段。

这种做法在早期 demo 很常见，因为它能最快展示“AI 已经会工作了”。

## 为什么不用

Claude Code 没走这条路，是因为一旦任务不是第一类对象，后面所有能力都会变成补丁。切后台要补一套状态，回前台要补一套状态，消息回流要补一套状态，团队协作再补一套状态。补到最后，你会发现系统里已经到处都是“像任务但不是真任务”的影子结构。

它选择先把劳动翻译成任务对象，本质上是在提前统一语言。后面无论是本地 shell、子 agent、远端 agent 还是 teammate，只要进入任务世界，就得先说同一种话。

## 代价

代价是前半段文法会显得“过度设计”。对小白来说，看见这么早就出现 `TaskType`、`TaskStatus`、`outputFile`、`notified` 这类字段，会觉得系统还没干活，怎么先在管档案。

但这正是 Claude Code 的取舍：它宁可一开始重，也不愿后面在每条执行路径上都重新发明一遍任务语义。
