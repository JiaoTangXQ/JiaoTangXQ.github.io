# 同一种名字让后端、提示词和渲染认出同一簇能力

Claude Code 在这里明显不是局部最优思路。它不是只让工具能跑起来，而是故意让工具名、系统提示词暗示、用户可见渲染一起围绕同一套 `computer-use` 身份展开。

这样做的好处是，全链路都知道自己面对的是哪一簇能力。好的架构经常这样：表面像命名统一，实际是在照顾多层系统同时对齐。

## 实现链
`setupComputerUseMCP()` 产出的名字会被后端识别，`toolRendering.tsx` 又用同一组工具名做 UI 覆盖，`main.tsx` 则把这些名字并进 `allowedTools`。同一前缀同时服务了模型提示、权限判定和终端显示。

## 普通做法
更常见的做法是后端只认内部 ID，前端再自己起一套人类可读名字。

## 为什么不用
Claude Code 不用这种分裂命名，因为这类外延能力跨过了后端、运行时和 UI。名字一分裂，读者和系统都会更难追踪“现在说的是不是同一簇能力”。

## 代价
代价是 UI 命名自由度更低，很多地方都得围着同一前缀做适配。
