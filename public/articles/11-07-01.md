# Messages.tsx 不是渲染器，是编排器

拿到 `messages` 数组，直接 `map(render)` 就行了——这是聊天界面最直觉的实现方式。Claude Code 没有这么做。

## 渲染前的变换链

`Messages.tsx` 在把消息交给渲染层之前，会先经历一条长链：

```
messages
  → normalizeMessages()
  → reorderMessagesInUI()
  → applyGrouping()
  → collapseBackgroundBashNotifications()
  → collapseHookSummaries()
  → collapseReadSearchGroups()
  → collapseTeammateShutdowns()
  → computeUnseenDivider()
  → computeSliceStart()
  → VirtualMessageList 或 非虚拟路径
```

每一步都在做一件事：把「计算机产生的事件序列」变成「人应该看到的叙事结构」。

## 为什么需要这么多步骤

Claude Code 里的一次对话包含：助手回复（可能有多个内容块）、工具调用、工具结果、前后置 hooks、thinking 块、system 消息、进度更新、teammate 关闭通知……

这些东西按时间顺序排列，对计算机来说是完整的记录，对人来说是碎片的混乱。`tool_use` 和它对应的 `tool_result` 可能被其他事件隔开。Hooks 会插在工具调用前后。进度更新会频繁穿插。

如果原样显示，用户需要自己在屏幕上重新拼出「这次工具调用完成了什么」的故事，在大量后台噪声里找到真正推进工作的那几行。

编排层替用户完成了这个工作：把碎片重组为连续的叙事，把噪声折叠起来，让最重要的信息自然浮现。

## 编排的代价

这条变换链让消息区承载了相当重的「政策」。调试时，如果屏幕上看到的顺序和原始 transcript 不一样，或者某条消息消失了，你需要理解整条变换链才能找到原因。

这是真实的复杂性成本。换来的是：用户看到的不是原始日志的回放，而是一个经过思考的、为阅读设计的叙事。
