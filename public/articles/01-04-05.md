# 面试追问：Claude Code 的 bridge 模式打开后，为什么命令列表会变，权限方向也会反过来？

## 从 cli.tsx 的一段判断开始

`cli.tsx` 里有这样一段：

```typescript
if (feature('BRIDGE_MODE') && (
  args[0] === 'remote-control' || 
  args[0] === 'rc' || 
  args[0] === 'remote' || 
  args[0] === 'sync' || 
  args[0] === 'bridge'
)) {
  // ...
  const { getClaudeAIOAuthTokens } = await import('../utils/auth.js')
  if (!getClaudeAIOAuthTokens()?.accessToken) {
    exitWithError(BRIDGE_LOGIN_ERROR)
  }
  const disabledReason = await getBridgeDisabledReason()
  if (disabledReason) {
    exitWithError(`Error: ${disabledReason}`)
  }
  const versionError = checkBridgeMinVersion()
  if (versionError) {
    exitWithError(versionError)
  }
  // 还要等 GrowthBook policy 加载，检查 allow_remote_control
  await waitForPolicyLimitsToLoad()
  if (!isPolicyAllowed('allow_remote_control')) {
    exitWithError("Error: Remote Control is disabled by your organization's policy.")
  }
  await bridgeMain(args.slice(1))
```

这段代码在进入 `bridgeMain` 之前做了四件事：OAuth 检查、GrowthBook feature gate 检查、版本兼容性检查、企业策略检查。

普通的"功能开关"不需要这么多前置检查。这说明 bridge 模式打开的不只是一个功能，而是一种不同的工作场景——这个场景有自己的权限前提、版本要求、企业策略约束。

## 命令集为什么变

进入 remote 连接模式后，Claude Code 会过滤命令列表，只保留"remote-safe"子集。

什么是 remote-safe？本地命令（类型 `local` 或 `local-jsx`）需要在本地终端执行 UI 操作或本地文件系统操作，在远端连接的上下文里，这些操作发生在哪台机器上会产生歧义——是发送操作请求到远端机器，还是在本地执行？

remote-safe 的命令是那些不依赖本地 UI 渲染、不假设当前工作目录是本地文件系统的命令。它们可以通过消息协议在远端机器和本地界面之间被正确路由。

过滤不是删除命令本身，而是改变 `getCommands()` 的返回结果——`isCommandEnabled` 的判断会根据当前是否处于 remote 模式返回不同结果。这意味着相同的命令定义，在不同工作场景下的"可用性"是不同的。

## 权限方向为什么反过来

正常模式下，权限请求是本地弹框：模型想执行一个需要用户确认的操作，本地 UI 弹出权限对话框，用户点击允许/拒绝，结果返回给模型。

remote 模式下，工作场景是：远端机器上跑着一个 Claude Code 实例（比如 CCR 容器），本地设备上是用户的界面。权限请求从远端机器产生，需要翻越网络，在本地设备的界面上弹出，用户在本地确认，结果再翻回远端机器。

这不只是网络传输问题，而是改变了系统里"谁负责 UI"的假设。本地实例在这种场景下可能变成 `viewerOnly` 模式——它只展示远端的状态，不直接控制执行。它看到的是远端发过来的消息流，它发出的是用户在本地的交互结果。

## ToolPermissionContext 里的 shouldAvoidPermissionPrompts

`ToolPermissionContext` 有一个字段：`shouldAvoidPermissionPrompts?: boolean`。注释是：`When true, permission prompts are auto-denied (e.g., background agents that can't show UI)`。

这个字段是为 headless 场景设计的——后台 agent 或者无 UI 的 worker 不能弹权限框。当 `shouldAvoidPermissionPrompts` 为 true 时，需要用户确认的工具调用会被自动拒绝，不会挂起等待。

remote 连接的某些配置下，如果本地界面不可用（比如纯 API 调用场景），这个字段会被设为 true。这保证了即使工作场景从"有本地 UI"变成"无本地 UI"，系统行为仍然是确定的——而不是挂起等待一个永远不会来的确认。

## 模式切换是规则迁移，不是功能开关

这整个设计背后有一个核心思想：Claude Code 的工作场景不是单一的。本地 REPL、远端连接、headless SDK、CCR 容器、后台 agent——这些场景的"规则世界"是不同的。

- 本地 REPL：可以弹 UI，命令在本地执行，权限框本地响应
- remote 连接：UI 可能在另一端，命令需要路由，权限请求需要跨网络
- headless SDK：没有 UI，工具调用不能等待用户，权限请求自动拒绝
- 后台 agent：没有前台，UI 资源是独立的，输出写 outputFile

把这些差异用"功能开关"处理，会导致越来越多的 `if (remote) { ... } else { ... }` 散布在代码各处。Claude Code 的做法是把场景规则集中在 `toolPermissionContext`（权限世界）、命令过滤函数（命令世界）、以及连接配置对象（消息路由世界）里——切换场景就是更换这些对象，不是在所有地方加判断。

## 面试指导

**考法**：Claude Code 支持 remote 模式，这会影响哪些系统行为？或者：为什么打开 bridge 模式前需要检查那么多东西？

**满分答法**：说清楚三个维度的变化：命令集过滤（remote-safe 子集）、权限方向变化（本地弹框 vs 跨网络路由）、UI 角色变化（主控 vs 观察者）。然后解释这不是功能开关，而是整个工作现场规则的迁移，举出 `shouldAvoidPermissionPrompts` 作为"场景变化导致系统行为系统性变化"的具体例子。

**容易被追问的点**：什么叫"remote-safe 命令"？答：不依赖本地 UI 渲染、不假设当前目录是本地文件系统、能通过消息协议被正确路由的命令。和是否"安全"（security）无关，是和"在远端场景下能否被正确执行"有关。
