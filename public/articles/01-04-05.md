# 条件能力为什么会改写用户所处的世界

有些能力看起来像“可选功能”，其实一旦打开，就不只是多一个按钮，而是把整台系统带进了另一个世界。Claude Code 里的 `bridge`、`remote`、`assistant`，大体都属于这一类。

它们改写的不是某个局部实现，而是用户所处的工作现场。消息不一定只在本地来回，权限请求可能要从远端再翻回本地，命令集会被预先筛成 remote-safe 版本，某些模式下你甚至只是一个 viewer，而不再是原来那种本地 REPL 主控者。也就是说，系统并不是在同一个世界里多长了几个功能，而是在切换世界规则。

这就是为什么这些能力不能轻写成“附加模式”。它们碰到的是会话边界、权限边界、消息边界和界面边界。Claude Code 很成熟的一点，是它没有把这些变化藏在深处，而是让远端会话配置、命令过滤、viewerOnly、session URL、模式标记这些东西显式进入运行时。

所以条件能力真正厉害的地方，不是它们更炫，而是它们承认: 一旦工作现场变了，系统规则也必须跟着变。

## 实现链
这页落到的实现链都属于制度世界：权限装配、设置来源、hooks 介入、连续性设施或条件能力 gating。代码的共同特点是，它们不是在主循环失败以后补救，而是在主循环真正推进前后就提前改写资格和边界。

## 普通做法
更普通的做法是先把 happy path 主流程跑通，再把权限、外部规则和跨回合记忆当辅助功能补上。很多系统都是这样长出来的。

## 为什么不用那种做法
Claude Code 没这么长，是因为对 agent 来说，真正危险的往往不是下一步怎么走，而是这一步有没有资格走、走完后有没有留下可继续工作的状态、外部制度有没有权利插手。它把这些问题提前写进了运行时。

## 代价与回报
回报是系统更不容易越权和失忆。代价是相关逻辑分散，阅读时不能只盯着一条主流程。
