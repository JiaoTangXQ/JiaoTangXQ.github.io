# 扩展世界不是一个洞口，而是三种增长方式

## 实现链
- `loadSkillsDir.ts` 先从项目、用户、托管目录读取 Markdown 技能；`pluginLoader.ts` 再从 marketplace、会话目录和内联目录发现插件；`services/mcp/client.ts` 最后维护 MCP server 的连接和能力表。
- 这三条线进入系统的时间点也不同：skills 在本地启动时就能解析，plugins 先过安装与启用判断，MCP 要先连上 transport 才知道服务器今天到底提供什么。

## 普通做法
- 更普通的做法是只提供一个 `extensions/` 目录，所有扩展都按“放个包进去”处理。
- 另一种更直觉的做法是只分“本地扩展”和“远端扩展”两类，不再细看中间差别。

## 为什么不用
- Claude Code 没这么分，因为代码里的约束不是“本地 vs 远端”这么粗。skills 重点是元数据和工作方式，plugins 重点是生态治理和安装来源，MCP 重点是连接状态和动态能力。
- 把它们合成一个洞口，会逼系统为最复杂的那一类付出所有成本，最后连最简单的本地技能也被讲得像一个在线平台接入。

## 代价
- 代价是学习成本上升，新手要先接受“扩展世界并不统一”这个事实。
- 但这比先给一个好记、却与代码不符的大框架更值，因为后面每一页都能直接落回真实模块。
