# 搜索文本缓存在 WeakMap 而不是 Map——为什么消息对象比消息 ID 更适合做 key？

两种缓存方案对比：`WeakMap<RenderableMessage, string>` vs `Map<string, string>`（以 uuid 为 key）。两者功能等价，WeakMap 为什么更好？

## WeakMap 的自动释放

`WeakMap` 的 key 必须是对象。当 key 对象没有其他引用时（垃圾回收可以回收它），WeakMap 里对应的 entry 自动消失，不需要手动删除。

`Map<string, string>` 的情况：transcript 被 compact 后，早期消息的 `RenderableMessage` 对象被 GC 回收了，但 Map 里以 uuid 为 key 的 toLowerCase 结果没有自动删除，Map 持续增长。要清理就需要在 compact 操作里手动遍历 Map 删掉旧条目，或者全量重置（清空整个 Map，下次搜索重新预热）。

`WeakMap<RenderableMessage, string>` 的情况：早期消息对象被 GC 回收时，WeakMap 里以该对象为 key 的 entry 自动释放。无需手动清理。

## 消息对象是稳定 key 的条件

WeakMap 以对象引用为 key，这意味着**同一个消息，如果被重新创建成新的对象**（即使内容相同），缓存就失效了。

Claude Code 的消息对象在什么情况下是稳定的（引用不变）？

```typescript
// normalizeMessages 的行为
// append-only：新消息追加时，旧消息的对象引用不变
// rewrite：/clear 或 compact 时，所有消息对象被替换
```

关键保证：append-only 模式下，`normalizedMessages[i]` 的对象引用在消息创建后不变。新消息追加到数组末尾，不影响前面消息的引用。

这意味着 WeakMap 缓存在以下情况下始终有效：
- 消息只是追加（新消息进来）
- 消息被 group 或 reorder（但原始 RenderableMessage 对象不变，只是视图层的排列变化）

在以下情况下需要重新计算：
- `/clear`：所有消息对象被替换
- compact：早期消息对象被新的 summary 对象替换（旧对象 GC，WeakMap entry 自动释放）

两种情况下的重计算是正确的：clear 后内容全变了，需要重新预热；compact 后被压缩的消息不再显示，不需要缓存它们的搜索文本了。

## fallbackLowerCache 和 Messages.tsx 的缓存分工

注意源码里有两个独立的搜索文本缓存：

**1. `fallbackLowerCache`（VirtualMessageList.tsx 里）**：

```typescript
const fallbackLowerCache = new WeakMap<RenderableMessage, string>()
function defaultExtractSearchText(msg: RenderableMessage): string {
  const cached = fallbackLowerCache.get(msg)
  if (cached !== undefined) return cached
  const lowered = renderableSearchText(msg)
  fallbackLowerCache.set(msg, lowered)
  return lowered
}
```

这是 fallback：用于测试或没有 Messages.tsx 提供自定义缓存的场景。

**2. `Messages.tsx` 的自定义缓存**：

`Messages.tsx` 在 `extractSearchText` prop 里提供自己的缓存实现，这个实现还处理工具调用的 `extractSearchText`（不只是消息文本 toLowerCase，还包含工具输入的格式化）。

两个缓存同时存在，fallback 是安全网，Messages.tsx 的实现是主路径。

## 热路径的性能目标

`setSearchQuery` 被触发的时机是用户每次按键（incsearch）。假设用户输入 "error handling" 这 13 个字符，每个字符触发一次 `setSearchQuery`，每次需要对所有 700 条消息运行 `extractSearchText`。

没有缓存：700 × 13 = 9100 次 toLowerCase 操作，9100 次字符串分配
有缓存：700 次缓存写入（预热时），9100 次 Map.get（O(1) 查找），0 次额外字符串分配

对于 700 条平均 500 字符的消息：
- 无缓存：每次 setSearchQuery 约 350KB 字符串分配
- 有缓存：每次 setSearchQuery 约 0 分配（全部缓存命中）

## 面试指导

这道题是 JavaScript 内存管理和缓存设计的交叉题。

**WeakMap vs Map 的选择标准**：
1. key 是对象（不是 primitive）→ 可以用 WeakMap
2. key 的生命周期和你的 value 的生命周期一致（key 消失时 value 也应该消失）→ 选 WeakMap
3. key 的生命周期独立于 value（你需要显式控制缓存失效）→ 选 Map

**WeakMap 的限制**：
- key 必须是对象（不能是 string/number/symbol）
- 不可迭代（没有 .keys()、.values()、.size）
- 不保证回收时机（GC 是惰性的）

**追问**：如果两个不同的消息内容相同但是对象引用不同，WeakMap 会缓存几份？答：两份（以两个不同对象为 key）。这是 WeakMap 的正确行为——对象同一性（reference equality）决定是不是同一个 key，而不是内容相等（deep equality）。在 Claude Code 里，同样内容的消息不会出现两个不同对象（消息对象在 normalizeMessages 里被惰性创建，不会重复），所以这不是问题。
