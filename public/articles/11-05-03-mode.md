# 面试题：为什么远端模式下要隐藏本地 permission mode？这个决策背后的设计思路是什么？

这道题表面上在问一个 UI 细节，深处是在考察你对"界面的诚实义务"是怎么理解的。

## 先搞清楚"远端模式"是什么

Claude Code 有一个 remote session 的概念：本地终端作为 UI 前端，实际的 agent 运行在远端容器或进程里。这时本地这层"壳子"能读到的 permission mode，是本地进程的状态，而远端执行环境可能运行在完全不同的安全制度下。

这里有个关键的认知差异：

- **用户期待看到的**："当前正在执行的 agent 处于什么权限模式"
- **本地能提供的**："本地进程配置的 permission mode"

在单机模式下，这两者是同一件事。但在远端模式下，它们可能完全不同。

## 代码层面的决策

```typescript
// PromptInputFooterLeftSide.tsx
const modePart = !getIsRemoteMode() ? buildModePart(permissionMode) : null;
```

一行代码，一个明确的条件：只要是远端模式，`modePart` 直接返回 null，不管本地的 `permissionMode` 是什么值。

这不是"我们不知道怎么显示远端 mode"的回避，而是主动的架构选择：**当你没有资格代表某个状态时，就不要代表它**。

## 为什么这个决策是对的

**错误信息比缺少信息更危险**

想象一个场景：远端 agent 运行在 auto-approve 模式下（危险），但本地进程配置的是 default 模式（较安全）。如果状态栏继续显示本地的 default mode，用户会误以为当前 agent 在安全模式下运行，做出错误的操作判断——比如放心地让它去执行一些本来需要确认的操作。

**"不确定"本身是信息**

界面少了一个字段，用户会注意到，会去追问原因，或者通过别的途径了解远端状态。但如果界面显示了一个错误的状态，用户很可能就直接信了，不再追问。前者迫使用户主动寻求真相，后者让用户停留在假象里。

**信息权限的边界**

`getIsRemoteMode()` 的判断实际上是在说：**这条信息的所有权不属于本地进程**。只有真正拥有某个状态的主体，才有资格在界面上代表它。本地进程的 permission mode 只代表本地，不代表远端执行现场。

## 对比：如果不这样做会怎样

**方案 A：始终显示本地 mode**

用户看到 mode 标签，以为它代表当前正在执行的 agent，做出错误判断。这是静默的误导。

**方案 B：显示但加免责声明**

比如显示"local: default"而不是"default"。这比什么都不加好，但占用了更多空间，而且用户在快速扫读时还是容易忽略那个"local:"前缀。

**方案 C（当前选择）：直接不显示**

用户注意到状态栏少了一个字段，会去找原因或者查远端 dashboard。这是更诚实的界面行为。

## 延伸到更大的设计原则

这个决策体现的是一种**界面诚实性原则（UI Honesty Principle）**：

> 界面显示的每一个字段，都对用户做出了隐式承诺——"这个信息代表你现在的真实状态"。如果这个承诺做不到，就不该有这个字段。

这个原则在 Claude Code 里多处出现：
- assistant/KAIROS 模式下整条状态栏都隐藏（`statusLineShouldDisplay`）
- 任务 owner 只在对方仍在线时显示
- 远端进度条在某些模式下关闭

共同逻辑都是：**没资格代表某个事实的时候，不代表它**。

## 面试指导

**基础问法**："远端模式下为什么不显示 permission mode？"

答：因为本地的 permission mode 代表的是本地进程状态，不代表远端执行环境的状态。显示错误状态比不显示危害更大，所以用 `!getIsRemoteMode()` 条件直接把这个字段关掉。

**深层追问**："这个决策体现了什么设计原则？"

答：界面对用户有隐式承诺——显示的字段代表真实状态。没有资格代表某个状态时，不代表它。错信息比少信息危险，"不确定"本身就是一种诚实的信息。

**类比考察**："你还能举出 Claude Code 里类似的例子吗？"

答：`statusLineShouldDisplay()` 在 KAIROS 激活时整条隐藏；任务 owner 用 `ownerActive` 条件控制是否显示；远端进度条在某些模式下关闭。这些都是同一个原则的应用——**只汇报自己有资格代表的状态**。

**代码考察**："条件是 `!getIsRemoteMode()` 还是 `getIsRemoteMode() ? null : buildModePart()`，有什么区别？"

答：实际上逻辑一样，前者更简洁。关键是这个条件放在 `modePart` 的生成处，不是放在渲染处——这确保即使渲染逻辑复用，数据层面也是干净的 null，不会出现"有数据但被隐藏"的歧义。

---

*核心考察点：界面诚实性原则、错误信息 vs 缺失信息的危害比较、设计决策的系统性理解（能从一个细节推出整套逻辑）。*
