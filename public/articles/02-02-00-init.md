# 为什么 init() 之前要先决定"这次启动是谁"？

想象一个工厂流水线的设计问题：

**方案 A**：先启动所有机器（通用初始化），再看今天生产什么产品，再调整机器参数。

**方案 B**：先看订单（判断本次用途），再只启动需要的机器，按产品要求配置参数。

大多数软件选了方案 A，因为更简单。Claude Code 选了方案 B，并且这个选择渗透到了启动链的每一层。

## 问题的本质

`init()` 函数（位于 `entrypoints/init.ts`）做了很多事：启用配置系统、应用环境变量、配置 mTLS、设置代理、预连接 API……每一步的正确性，都隐含依赖于"这次启动到底是什么模式、什么身份、读什么设置"。

如果这些身份判断来得太晚，就会出现这样的情况：
- 配置系统按默认来源初始化，但 `--setting-sources` 参数说要限制来源
- 环境变量按完整模式应用，但这次是非交互脚本调用，有些变量不应该生效
- 连接模式按本地假设配置，但这次是 `claude ssh` 远端会话

晚一步重新配置？可以，但代价是前面已经发生的副作用没法回滚。

## 哪些事情必须在 init() 之前做？

看 `main()` 函数里 `init()` 调用之前的代码，可以找到几类"必须先定"的判断：

**1. 设置来源**

```typescript
function eagerLoadSettings(): void {
  const settingsFile = eagerParseCliFlag('--settings');
  if (settingsFile) {
    loadSettingsFromFlag(settingsFile);
  }
  const settingSourcesArg = eagerParseCliFlag('--setting-sources');
  if (settingSourcesArg !== undefined) {
    loadSettingSourcesFromFlag(settingSourcesArg);
  }
}
```

`eagerParseCliFlag` 是不走 Commander 正式解析的轻量版参数读取——在 Commander 注册命令结构之前就生效。因为 `init()` 里的配置系统必须从正确的来源读，来源不能等到 Commander 解析完才知道。

**2. 交互模式判断**

```typescript
const cliArgs = process.argv.slice(2);
const isNonInteractive = cliArgs.includes('-p') || cliArgs.includes('--print');
setIsInteractive(!isNonInteractive);
initializeEntrypoint(isNonInteractive);
```

是否交互，会影响 telemetry 初始化、trust 处理方式、entrypoint 记录，以及很多后台预热是否值得执行。

**3. 特殊入口识别**

`cc://` URL、`claude ssh`、`claude assistant`、深链接——这些特殊入口需要在进入正常命令解析之前被识别并"重写" argv，否则 Commander 会按普通命令处理，导致参数含义错误。

## 这是"大路由表"还是"先定身份"？

**面试官视角**：你在 `main()` 函数开头写了一大堆判断，然后才调用 `init()`。这不是把 main 函数变成一个大路由表了吗？代码读起来很乱。

**辩护**：这不是路由表，而是"世界边界"的前置声明。

路由表的特征是：在多个平等的分支里选一个，然后进入那个分支执行。

Claude Code 的前置判断是：确认"这个世界的参数"，然后让后续所有代码都在这个已知参数的世界里执行。差别在于，前置判断不分支，而是设定全局上下文。

后续的 `init()`、`setup()`、`showSetupScreens()`、`launchRepl()`——全部都在"已经知道是谁"的前提下工作，而不是一边执行一边猜测当前身份。

## 面试指导

这类"初始化顺序"问题在系统设计面试中经常出现，特别是在讨论多租户系统、多模式应用或插件化架构时。

**核心判断标准**：某个初始化步骤 A 是否依赖另一个判断 B 的结果？如果是，B 必须先于 A 完成。不能因为 B "看起来是小事"就推迟。

**Claude Code 的启示**：与其设计一个"先通用初始化，遇到特殊情况再处理"的系统，不如识别出哪些是"影响一切的前提判断"，把它们提到最前面。看起来入口变复杂了，但后续每一层代码都因此变得更干净——它们不需要再到处做"当前是什么模式"的防御性检查。
