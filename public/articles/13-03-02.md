很多小连接件的作用，说穿了就是两件事：少失控一次，少重造一层。`compact.ts` 里的 `stripImagesFromMessages()` 会在压缩前把图片和文档先换成占位文本，因为总结器真正需要的是“这里出现过图片”，不是把整张图再喂一遍；`stripReinjectedAttachments()` 又会把下一轮本来就会重新补回来的技能附件先剥掉，避免总结里塞满过期提示。`permissionSetup.ts` 里对 Bash、PowerShell 和 Agent 自动放行规则的危险检测也是同一个思路：不是等风险动作已经跑起来再补救，而是尽量不让它以“便利”的名义先溜过去。

更常见的实现，是把这些小地方都当成可以后补的“边缘问题”。图片太长了再报错，技能附件重复了再忍一下，危险规则误放过了再靠日志排查。这些问题单看都不大，但它们会持续把系统往失控那边推。

Claude Code 的处理方式明显更成熟，因为它在这些小地方优先保的是秩序，而不是一时的省代码。代价是逻辑被拆得更细，看起来不够简洁；但对长期运行的 agent 来说，这些细碎限制往往比再多一个大功能更值钱。

## 实现链
像 `FullscreenLayout`、compact 边界拼装、permission setup 这种小件，都是为了让现有主干继续工作，而不是重造一层。

## 普通做法
每遇到特殊场景就另开一套新框架。

## 为什么不用
因为重造一层看似干净，长期却会制造双轨系统。

## 代价
小连接件多了以后，局部理解门槛会上升。
