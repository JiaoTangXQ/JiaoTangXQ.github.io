# 第十三卷回收索引：外延执行里 reconnect 页说明失败与重接也要一起被设计进去

## 实现链
外延执行里接入链真正难的，不是把门打开一次，而是失败以后还能不能顺手接回来。

这条线真正逼出来的，是第十三卷里关于工程现实的另一半判断。  
真正成熟的产品接入，不会只对“成功启用”做设计。

这页实际回收的实现链是：
- `12-06-02` 也把 `reconnect` 页写成接入链的一部分，这会继续支撑 `13-06-01`，说明真正的产品接入不是装完就结束，而是要把回来的路也一并设计好。

很多接入链只把成功当终点。  
Claude Code 在这里保留 `reconnect` 页，等于承认失败与重接也必须一起被设计进去。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
