# reconnect 设计说明：失败不是尾声，而是下一步的前提

面试题版本：**"Claude Code 的浏览器 bridge 连接断开后，重连策略是什么？无限重试好不好？指数退避的背后逻辑是什么？什么情况下应该停止重试？"**

## 连接断开的分类

不是所有的连接断开都应该重试：

**可恢复的断开**：
- 浏览器被锁屏/休眠后唤醒（连接超时断开，但扩展仍然可用）
- Claude Code 或浏览器短暂崩溃后重启
- 网络抖动（如果用的是 http/ws transport）

对这类断开，应该重试。

**不可恢复的断开**：
- 用户卸载了浏览器扩展（连接断开是因为扩展不存在了）
- manifest 文件被删除或 Native Host 路径变了
- 用户主动点击"断开连接"

对这类断开，不应该无限重试，应该通知用户。

## 指数退避策略

`src/bridge/createSession.ts` 里的重连逻辑使用指数退避：

```typescript
const RETRY_DELAYS = [1000, 2000, 4000, 8000, 16000, 30000]  // 毫秒
let retryCount = 0

async function scheduleReconnect() {
  const delay = RETRY_DELAYS[Math.min(retryCount, RETRY_DELAYS.length - 1)]
  retryCount++
  await sleep(delay)
  await attemptReconnect()
}
```

为什么指数退避：

**避免雪崩**：如果大量连接同时断开并立即重试，会造成重连风暴。指数退避让重试分散在时间轴上。

**给系统恢复时间**：连接断开的原因可能需要一段时间才能自然恢复（浏览器在重启中、用户在切换 profile）。立即重试可能在系统还没准备好时就失败，白白消耗资源。

**节约资源**：如果断开是持久性的（用户不打算用 bridge 了），无限快速重试会持续消耗 CPU 和 I/O。指数退避在确认不能连后，会降低到很低的重试频率。

## 最大重试次数和退出条件

重试不是无限的，有几个退出条件：

**连接失败类型变化**：初始连接失败（timeout）和重连中途失败（connection refused）是不同的。如果失败类型从"超时"变成"明确拒绝"，说明问题升级了，应该停止重试并报告给用户。

**总等待时间上限**：如果累计等待时间超过某个阈值（比如 5 分钟），即使还有重试次数，也应该放弃并告知用户"bridge 连接持续失败，请检查扩展状态"。

**用户明确放弃**：提供界面让用户可以手动取消重连尝试。

## 重连时的状态恢复

重连成功后，不只是"连上了就完事了"，还需要：

**重新同步状态**：重连期间 Claude Code 可能有新的工具调用请求，这些请求在重连后需要被处理，而不是被丢弃。

**重新发现能力**：浏览器可能更新了扩展，能力列表可能变化，重连后要重新查询，不用缓存的旧数据。

**通知用户**：如果重连发生在用户可见的活动期间（不是后台静默重连），应该通知用户"bridge 重新连接成功"。

## 用户感知的分级

重连策略的用户感知应该是分级的：

**透明的重连**（立即重试，用户不感知）：连接断开时间短，立即重连成功。

**有指示的重连**（状态栏显示"重新连接中..."）：重连需要几秒，用户看到状态但不需要操作。

**需要用户参与的重连**（显示错误和操作按钮）：重连多次失败，提示用户检查扩展状态、重启浏览器等。

这种分级避免了两个极端：要么一切都悄悄进行（用户看不到问题）；要么一切都报错（用户被不重要的通知轰炸）。

## 面试角度的总结

reconnect 的设计质量体现在：把失败路径当作需要设计的功能，而不是"先做成功路径，失败了再说"。指数退避、分类的退出条件、重连后的状态恢复、分级的用户感知——这些是对"连接会失败"这个现实的认真对待，让 bridge 连接的整个生命周期（建立、维护、失败、恢复）都是设计的一部分。
