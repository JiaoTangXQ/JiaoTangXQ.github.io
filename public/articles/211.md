# 压缩不是清空：附件、计划和技能线索如何在整理后重回工作台

面试题版本：**"Claude Code 压缩会话历史时，哪些内容会被保留，哪些会被压缩成摘要？判断标准是什么？如果用户压缩前正在进行一个多步骤任务，压缩后这个任务的进度还能恢复吗？"**

## 压缩的信息损失问题

会话历史越长，每次压缩丢失的细节越多。理想情况下，压缩后：
- 对话的核心脉络被保留
- 当前任务的进度和状态被保留
- 可能再次需要的资源引用被保留
- 重复的、已过时的、不再需要的内容被丢弃

难点在于：如何区分"还需要"和"不再需要"？

## 附件在压缩中的处理

当用户在会话里添加了文件附件（比如上传了一个代码文件让 Claude 分析），这个附件的内容可能很长，直接包含在每次 API 调用里会消耗大量 token。

`src/utils/attachments.ts` 里的 `filterDuplicateMemoryAttachments()` 处理了一个常见情况：相同的附件在会话里被重复引用。只需要保留最后一次引用的附件内容，之前的可以去掉（减少 token 消耗，但不丢失信息）。

`createAttachmentMessage()` 创建的附件消息有特殊格式，让系统知道这是可以被引用、但不必每次都完整展开的内容。压缩时，这类附件消息会被转换成引用形式（"文件 X 的内容可以通过工具重新读取"），而不是被完整摘要成自然语言。

## 进行中的计划和任务

如果用户在会话里确立了一个多步骤计划（"我们要按这个顺序：先 A、再 B、然后 C"），这个计划的结构是接续工作的关键。

`buildPostCompactMessages()` 函数（`src/services/compact/compact.ts`）在生成压缩摘要时，会特别处理：
- 明确的任务列表或计划
- 已完成的步骤
- 当前正在进行的步骤
- 还未开始的步骤

压缩摘要里会有一个明确的"当前任务状态"部分，格式化地描述任务进度，而不是把它稀释在自然语言摘要里。这让压缩后的模型可以快速重建"我们在哪里"的认知。

## 技能配置的保留

通过 `prompt` 命令加载的技能（skill）配置包含：
- 系统提示词（告知模型这次会话的角色和规则）
- 工具许可（哪些工具可以用）
- 特定的行为指引

这些配置在会话里是基础性的——它们定义了这次会话的基本工作框架。如果压缩后这些配置丢失，模型就不知道自己是在什么上下文里工作，可能开始使用错误的工作方式。

压缩后，这些技能配置不是通过摘要保留（自然语言摘要可能不够精确），而是通过重新注入（把原始配置再次放进系统 prompt）来保证完整性。

## "桌面重新摆好"的含义

"压缩后把真正要继续工作的线索重新摆回现场"这个表述有具体的工程含义：

**可以恢复的引用** vs **需要原地保留的内容**：
- 文件内容：可以通过工具重新读取，不必在历史里保留完整内容
- 当前工作状态：必须在压缩摘要里明确表示，否则模型不知道从哪里继续

**摘要的结构 vs 自然语言描述**：
- 结构化的任务状态（"步骤 2/5 已完成，当前在步骤 3"）比自然语言摘要（"我们已经完成了前两个步骤"）更容易被后续推理利用

**技能配置的精确性**：
- 技能提示词需要原样保留，不能被摘要（摘要可能改变语义）

## 面试角度的总结

压缩不是清空，是有选择的精简。Claude Code 的压缩策略对不同类型的内容有不同处理：可以通过工具重新获取的内容（文件、代码）变成引用；需要保持精确的配置（技能、工具许可）重新注入；任务进度结构化保留；重复和过时的对话内容被摘要。这种分类处理让压缩后的会话保持了"可以继续工作"的状态，而不只是"有一个关于过去的摘要"。
