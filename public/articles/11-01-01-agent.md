# 终端里的 agent 不是打印机，而是一台应用

把 Claude Code 理解成“输入一句，打印一段”，后面很多设计都会显得过度。因为它真正维护的不是一条输出流，而是消息区、任务面板、权限请求、状态栏、通知和对话框共同存在的一整块界面世界。

这就是第十一卷的起点。终端没有把它变回脚本，反而逼它更像一台应用，因为这么多状态必须在同一时刻被组织起来。

## 实现链

`App.tsx` 一上来就把 `FpsMetricsProvider`、`StatsProvider` 和 `AppStateProvider` 包在组件树外面，再把真正的 UI 子树交进去。顶层先装配观测、统计和事实世界，再谈消息、输入和面板怎么显示，说明它起手就把终端当成一台长期运行的应用。

## 普通做法

更普通的终端工具会把顶层写成“读输入、刷输出”的薄壳，状态最多散在几个局部变量里，统计和性能数据要么没有，要么顺手记一下。

## 为什么不用

Claude Code 不这么写，是因为权限、远端、任务、通知和输入控制都要共享同一份运行时现实。要是顶层只是打印壳，这些能力最后都会各自长一套状态，前端会越补越乱。

## 代价

代价是入口虽然只有几层 provider，看起来却已经带着系统分层；新读者必须顺着 provider 和 store 读下去，理解成本比普通 CLI 高。
