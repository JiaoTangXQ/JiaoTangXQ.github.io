# 导读：四种「失败」为什么需要四种不同的恢复路径？

在多数系统里，失败处理的逻辑很简单：`try-catch`，打印错误，告诉用户重试。这对无状态的 API 来说完全够用。

但 Claude Code 的 `query()` 主循环不是无状态的。它管理着上下文历史、工具执行状态、权限决策、多轮 transcript——任何一次失败，都发生在一个有历史、有状态的会话里。粗暴的 catch-and-report 会丢弃这些状态，用户不得不重新开始。

更糟的是，某些失败如果用错了恢复方式，会造成**死循环**。比如：

```
场景：stop hook 阻塞 + reactive compact
- 模型输出了某段代码
- stop hook 检查后报错："代码违反了规范，需要修正"
- 系统把 stop hook 错误写回消息流，继续下一轮
- 模型重新生成，但上下文已经很长，触发了 prompt too long
- reactive compact 压缩了上下文，但压缩过程又触发了 stop hook...
```

如果 `hasAttemptedReactiveCompact` 在 stop hook blocking 后被重置，就会进入这个循环。Claude Code 的代码注释里明确提到了这个反例，并专门保留 `hasAttemptedReactiveCompact` 的值跨越 stop hook 重试。

## 四类失败，四套恢复逻辑

### 类型 1：模型级失败 — fallback model

**触发条件**：主模型不可用（高负载、服务降级），流式输出中途触发 `FallbackTriggeredError`。

**恢复逻辑**：
1. 打 tombstone，清除已输出的半截消息
2. 丢弃并重建 StreamingToolExecutor（防止旧 tool_use_id 污染）
3. 切换到 fallbackModel
4. 剥掉与旧模型绑定的 thinking signature
5. 在**同一轮**用新模型重试

**为什么不能用通用 catch**：通用 catch 会把整轮工具历史都丢掉，用户需要重新发消息。fallback 的目标是**对用户透明**——他们感知不到模型切换了。

### 类型 2：上下文级失败 — reactive compact / collapse drain

**触发条件**：API 返回 413（prompt too long），或媒体文件过大。

**恢复逻辑**：
1. 先 withhold 错误（不立刻暴露给外层）
2. 尝试 `collapse_drain_retry`：把暂存的 context collapse 提交，再重试
3. 如果 drain 后还是 413，尝试 `reactive_compact_retry`：全量压缩历史，再重试
4. 如果两次都失败，把错误暴露出来，返回 `{ reason: 'prompt_too_long' }`

**为什么 withholding 很重要**：如果立刻暴露 413 给外层（比如 SDK 集成方），SDK 可能会终止这个会话。但 413 在大多数情况下是可以自救的——先 withhold，给恢复机制一次机会。

### 类型 3：续写级失败 — max output tokens recovery

**触发条件**：模型输出被截断（hit max_output_tokens 限制）。

**恢复逻辑**：
1. 先 withhold 截断错误
2. 如果可用，尝试升级 max_output_tokens（escalate to 64k），同消息重试
3. 如果还是被截断，注入恢复提示（"直接接着说，不要道歉，不要重述"）
4. 最多恢复 3 次（`MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3`）
5. 超过 3 次，暴露截断错误

**为什么不是 API 级别的 retry**：这不是服务端错误，是模型输出本来就长。解决方法不是重发请求，而是用人类语言告诉模型"你被打断了，直接继续"。这要求在消息层做处理，而不是在 HTTP 层。

### 类型 4：制度级失败 — stop hook blocking

**触发条件**：stop hooks（用户或系统配置的外部校验钩子）检查模型输出后报错，认为输出不符合要求。

**恢复逻辑**：
1. 把阻塞错误写入消息流（`isMeta: true`，对用户隐藏但对模型可见）
2. 标记 `stopHookActive: true`
3. 带着 `{ reason: 'stop_hook_blocking' }` 进入下一轮
4. 模型在"刚才的输出被制度层拒绝"的上下文下重新生成

**为什么这是"继续"而不是"报错"**：stop hook 的阻塞本质上是一个反馈——"你的输出有问题，请重来"。这和"命令执行失败，请用户重新输入"是不同的语义。前者应该继续当前会话，后者才是报错给用户。

## 各恢复路径的互动保护

四类恢复不是完全独立的，它们之间有互斥和保护机制：

```
hasAttemptedReactiveCompact 跨越 stop_hook_blocking 保留
  ↑
防止：compact → stop hook 阻塞 → reset → compact → ...

collapse_drain_retry 只尝试一次
  ↑
防止：drain → 413 → drain → 413 → ...

max_output_tokens_recovery 有次数限制（3次）
  ↑
防止：截断 → 恢复 → 截断 → 恢复 → ...

fallback model 在同一轮内只发生一次
  ↑
防止：fallback → 再次失败 → 再次 fallback → ...
```

## 面试指导

这一章的系统设计价值在于：**失败处理是系统设计的一等公民，不是事后补丁**。

如果面试官问"你会怎么设计 LLM Agent 的错误恢复"，一个好的回答框架是：
1. **先分类错误**：不同来源的失败需要不同的恢复策略
2. **区分"报错给用户"和"系统自救"**：很多失败用户不应该感知到
3. **防止恢复路径互相干扰**：恢复本身也可能触发新的失败，需要保护机制
4. **显式的 transition 类型**：让每次循环继续的原因可被观测和测试
