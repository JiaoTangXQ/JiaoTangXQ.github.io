# 恢复簇导读：失败不是报错收尾，而是决定下一轮怎么继续

这一章真正关心的，不是“报错之后怎么补一把”，而是系统在不同失败来源下，凭什么还能继续往前走。模型失败、上下文失败、续写失败、制度失败，都会改写下一轮的理由。

所以这一章的四片叶子，不该分开当作几个错误专题。先看 [04-07-01-为什么fallback-model不该被当成普通报错分支.md](/Users/xxx/tmp/cc/docs/claude-code-book/04-主循环怎样思考与执行/04-07-失败时系统怎样继续往前走/04-07-01-为什么fallback-model不该被当成普通报错分支.md)，再顺着 reactive compact、两种 continuation，最后读到 stop hooks 阻塞这页，整个恢复簇才会完整：Claude Code 的恢复世界处理的不是“怎么把错误吞掉”，而是“系统现在凭什么还能继续下一轮”。

代码证据就是 `query()` 的状态对象和一整排 `transition.reason`。`fallback_model`、`collapse_drain_retry`、`reactive_compact_retry`、`max_output_tokens_recovery`、`stop_hook_blocking`、`token_budget_continuation` 都不是日志标签，而是下一轮真的会据此改动作。更常见的写法是把错误都收进一个大 `catch`，打印说明，让用户自己重新发一遍。

Claude Code 不满足于那种“报错收尾”，因为它要支持长会话、工具调用和恢复链，很多失败其实仍然有便宜的继续方式。代价是主循环不再像线性故事，而像一个有明确理由码的恢复状态机。

## 实现链
失败会被翻译成 `transition.reason`，再决定下一轮如何继续。

## 普通做法
统一进一个 catch，打印错误后收尾。

## 为什么不用
因为很多失败在这里仍然有便宜且安全的继续方式。

## 代价
主循环会越来越像恢复状态机。
