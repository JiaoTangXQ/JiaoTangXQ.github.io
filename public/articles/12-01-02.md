# 面试题：真实设备接入为什么比写几个工具函数难一个量级？

## 一道典型的低估题

面试中有一类经典问题：给候选人一个看起来简单的需求，观察他们有没有看到隐藏的复杂度。

"我们想让 AI Agent 能操控桌面，你觉得工程量大概多少？"

初级回答：加几个调用 OS API 的工具函数，一两天搞定。

资深回答：这取决于你想接的是"演示能力"还是"真实用户现场"。如果是后者，真正的工程量在工具之外。

Claude Code 的 `utils/computerUse/` 和 `utils/claudeInChrome/` 两个目录，正好是第二种答案的具体展开。

## 工具函数只是冰山一角

以 `computerUse` 为例，实际的工具动作（鼠标点击、键盘输入、截图）不是主要工程量。真正的文件分布是这样的：

```
utils/computerUse/
  common.ts          — 常量、能力定义、MCP server 名
  setup.ts           — 动态 MCP 配置生成、工具命名
  gates.ts           — 坐标模式判断（Chicago mode 等）
  computerUseLock.ts — 并发锁，防止多个操作序列同时跑
  wrapper.tsx        — 会话级状态：allowedApps、grantFlags、截图尺寸、中断
  executor.ts        — 实际动作执行
  inputLoader.ts     — 输入设备加载
  swiftLoader.ts     — macOS Swift 绑定加载
  hostAdapter.ts     — 宿主适配器
  cleanup.ts         — 清理逻辑
  drainRunLoop.ts    — 事件循环排水
  escHotkey.ts       — ESC 键中断
  toolRendering.tsx  — 工具调用的 UI 渲染
  mcpServer.ts       — in-process MCP server
```

工具动作本身（executor）只是其中一个文件。其余大部分都在处理**制度问题**：谁能用、用什么条件、用完怎么收拾、出了问题怎么中断。

## setup.ts 暴露的真实工程清单

`setup.ts` 里的注释已经把问题讲清楚了：

```ts
/**
 * The mcp__computer-use__* tools are added to allowedTools so they bypass 
 * the normal permission prompt — the package's request_access handles approval
 * for the whole session.
 *
 * The MCP layer isn't ceremony: the API backend detects mcp__computer-use__*
 * tool names and emits a CU availability hint into the system prompt.
 * Built-in tools with different names wouldn't trigger it.
 */
export function setupComputerUseMCP(): {
  mcpConfig: Record<string, ScopedMcpServerConfig>
  allowedTools: string[]
}
```

这段注释揭示了三层"非工具"工作量：

1. **命名约定**：`mcp__computer-use__*` 这个前缀不只是展示用的，后端 API 靠它来判断是否注入 CU availability hint。名字错了，整个能力链路都会断。

2. **授权前置**：工具进 `allowedTools` 不是因为它安全，而是因为 `request_access` 已经在会话级处理了授权。这意味着你必须先实现一套会话级授权系统。

3. **进程策略**：`command/args` 实际上不会被 spawn——客户端会按 server name 走 in-process 分支。这是一种刻意的"外观 MCP，实现进程内"的双层设计。

## Chrome 接入：真正的平台级脏活

如果说桌面控制是"工具 + 制度"，Chrome 接入则是"工具 + 制度 + 平台碎片"。

`setupClaudeInChrome()` 的主体逻辑：

```ts
export function setupClaudeInChrome(): { mcpConfig, allowedTools, systemPrompt } {
  const isNativeBuild = isInBundledMode()

  // 必须先创建 wrapper script，因为 Chrome native host manifest 的 path 字段不能带参数
  void createWrapperScript(execCommand)
    .then(manifestBinaryPath => installChromeNativeHostManifest(manifestBinaryPath))
    .catch(...)

  return { mcpConfig: { [CLAUDE_IN_CHROME_MCP_SERVER_NAME]: { type: 'stdio', ... } }, ... }
}
```

注意，这个函数的主要工作不是构建 MCP 配置，而是：
- 创建 wrapper script（因为 manifest path 字段不能带参数）
- 安装 native host manifest 到多个浏览器目录
- Windows 下还要写 registry

这些都发生在 `setupClaudeInChrome()` 返回之前（通过异步 fire-and-forget）。工具还没开始用，安装层的脏活已经在后台跑了。

## 真实设备接入的六类隐藏工程量

从 Claude Code 的源码可以归纳出，接真实设备时必须额外处理的六类问题：

| 类别 | computerUse 示例 | Chrome 示例 |
|------|-----------------|------------|
| 授权 | `request_access` 会话级 | feature flag + subscription 检查 |
| 安装 | - | wrapper script + manifest + registry |
| 连接管理 | in-process MCP | native socket + bridge |
| 平台差异 | macOS Swift binding | 7 种 Chromium 浏览器 × 3 个平台 |
| 断线恢复 | `computerUseLock` | reconnect URL + legacy fallback |
| 日志/遥测 | `toolRendering.tsx` | `DebugLogger` + 白名单遥测 |

这六类问题在演示 demo 里都可以忽略，但在真实用户的机器上，每一类都会出问题。

## 面试考察点

这道题考察的核心判断是：**在设计阶段能不能把"工具可用"和"系统可用"区分开来。**

工具可用：在受控环境下，调用动作 API 能得到正确结果。

系统可用：在用户的真实机器上，无论什么浏览器组合、什么 OS 版本、无论是第一次安装还是第 N 次重连，系统都能稳定工作并优雅失败。

Claude Code 选择的是第二条路，代价是接入代码明显比业务逻辑更重。但这不是"过度工程"，而是接真实设备必须付的前置成本。等出了问题再补，只会到处打补丁，而且每一个补丁都会和其他补丁打架。
