# 两阶段搜索跳转（jump + seek effect）为什么不合并成一步？

Claude Code 的搜索跳转不是"算出位置→直接 scrollTo"这样的单步操作，而是分成了 `jump()` 函数和 `seekEffect` 两个阶段。为什么？

## 问题：元素必须先挂载才能扫描

搜索跳转的目标是高亮某条消息里的匹配文本。高亮的实现需要 `scanElement`——扫描目标消息的 DOM 元素，找到匹配文本的行列坐标，然后在屏幕上叠加高亮层。

问题在于：虚拟列表只挂载可视窗口附近的消息。如果跳转目标在当前视口外（比如搜索跳转到消息 #500，但当前视口在 #200），目标元素根本没有挂载，`scanElement` 没有元素可扫。

简单的想法：先 `scrollTo` 到目标位置，滚动后元素挂载，然后扫描。但 `scrollTo` 之后新元素的挂载发生在下一次渲染 + commit 之后，是异步的。在 `scrollTo` 调用的同一个帧里，元素还没挂载，`scanElement` 还是拿到空数组。

## jump() 做什么

```typescript
function jump(idx: number, wantLast: boolean) {
  // 1. 设置 scanRequest，记录要跳转到哪
  scanRequestRef.current = { idx, wantLast, tries: 0 }
  // 2. 滚动到目标位置（虚拟列表会挂载该位置附近的消息）
  scrollToIndex(idx)
  // 3. bump seekGen，触发 seek effect 在下次 commit 后运行
  bumpSeek()
}
```

`scrollToIndex` 调整 scrollTop + 更新 offsets，让虚拟列表在下次渲染时在视口里包含第 `idx` 条消息。`bumpSeek` 自增 `seekGen` 状态，让 seek effect 重新执行。

注意：`jump()` 调用之后，目标消息还没挂载，这是设计上允许的。

## seek effect 做什么

```typescript
useEffect(() => {
  const req = scanRequestRef.current
  if (!req) return
  
  const el = getItemElement(req.idx)
  if (!el || el.yogaNode?.getComputedHeight() === 0) {
    // 目标消息还没挂载（或高度为 0）
    if (req.tries > 1) {
      // 重试超过一次，放弃，自动前进到下一条
      stepRef.current(req.wantLast ? -1 : 1)
      return
    }
    // 重试一次
    scanRequestRef.current = { ...req, tries: req.tries + 1 }
    scrollToIndex(req.idx)
    bumpSeek()
    return
  }
  
  // 元素已挂载，精确 scrollTo 并扫描
  scanRequestRef.current = null
  s.scrollTo(Math.max(0, getItemTop(req.idx) - HEADROOM))
  const positions = scanElement?.(el) ?? []
  // 高亮处理...
}, [seekGen])  // 只依赖 seekGen，不依赖 offsets/messages
```

seek effect 的 deps 只有 `seekGen`，不包含 offsets 或 messages。原因：offsets 在每次渲染后重建（高频变化），如果放进 deps，每次 VirtualMessageList 渲染后 effect 都会重跑，造成不必要的扫描。seekGen 只在显式跳转时增加，effect 只在真正需要时运行。

## 两阶段之间的时序

```
jump() 调用
   ↓ scrollToIndex（调整 scrollTop + range）
   ↓ bumpSeek（seekGen++ → 触发重渲）
React 渲染
   ↓ 新的 range 包含目标消息
   ↓ VirtualItem(idx) 被渲染
DOM commit
   ↓ 目标消息元素挂载到 DOM
seek effect（passive effect，在 commit 之后运行）
   ↓ 读取目标消息的 DOM 元素（此时已挂载）
   ↓ 精确 scrollTo + scanElement
   ↓ setPositions → 叠加高亮
```

两阶段的必要间隔是"DOM commit"。React 保证 passive effect（`useEffect`，不是 `useLayoutEffect`）在 commit 之后运行，这时 DOM 已经更新，新挂载的元素可以访问。

## phantomBurstRef 防护

有时候搜索引擎（字符串 indexOf）匹配了某条消息，但 scanElement 扫描该消息的实际渲染时找不到高亮位置（"幽灵匹配"——可能是搜索文本在渲染后被折叠、截断或 ANSI 代码干扰了）。

这时自动前进到下一条匹配：`stepRef.current(1)`。但如果 20 条连续匹配都是幽灵，就会无限自动前进，形成无限循环。

`phantomBurstRef` 计数连续幽灵匹配次数，超过 20 次时停止自动前进，防止死循环。

## 面试指导

这道题是"React effect 时序"的进阶考察，考你是否理解 commit 和 effect 的先后关系。

**核心知识点**：
- `useEffect`（passive effect）在 DOM commit 之后、浏览器 paint 之后运行
- `useLayoutEffect` 在 DOM commit 之后、浏览器 paint 之前运行
- 需要"读取 DOM 之后再操作滚动"的场景，两者都可以，但 `useEffect` 更安全（不阻塞 paint）

**为什么用 useEffect 而不是 useLayoutEffect**？seek effect 先 scrollTo 后 scanElement，scanElement 读 DOM 测量值。如果用 useLayoutEffect，这个 scrollTo 会在 paint 之前发生，可能导致用户看到一帧闪烁（旧位置 → 新位置）。用 useEffect，scrollTo 在 paint 之后，用户看到的是直接的最终位置，更流畅。

**重试机制的合理性**：tries 最多 1 次（代码里 `tries > 1` 才放弃）。在极端情况下（如目标消息在极其靠近 transcript 末尾，offsets 估算严重偏差），第一次 scrollToIndex 的位置不精确，目标消息没有被包含在新的 range 里。第二次 seek effect 里再 scrollToIndex 一次，用更新后的 offsets 重算位置，通常能命中。
