# cwd、worktree 和 session 为什么会重写后面的依赖顺序

很多依赖关系之所以成立，不是因为代码作者喜欢讲顺序，而是因为“现场一变，后面的答案就全变”。Claude Code 在 `setup()` 之后才敢把很多东西真正接上，就是因为 `cwd`、worktree 和 session 一旦定下来，后面的世界坐标也跟着改写了。

最典型的就是 worktree。只要启用 `--worktree`，`setup()` 就可能 `process.chdir()` 到新的工作树，把 `projectRoot` 改到那里，清掉原目录下的记忆文件缓存，再重新抓 hooks 配置。这样一来，后面的技能、命令、hooks、cron、session 存储，实际都已经不该再对着原来那个目录工作了。如果还按旧顺序提前读取，就会把整个会话建立在错误现场上。

这正是 Claude Code 很成熟的一点。它没有把 `cwd` 当成一个不起眼的参数，也没有把 worktree 当成一个后面再补的附属功能。它承认: 只要工作现场变了，依赖顺序也必须跟着重排。真正的工程严谨，不是永远并行，而是知道什么时候必须先把地面找准。

## 实现链
这页对应的实现链都属于 `setup.ts` 那把刀：把前面抽象可运行的环境，真正切进当前 cwd、worktree、session、repo 和连接现实里。代码在这里开始回答“这一次到底在哪干活”。

## 普通做法
更常见的写法是把当前进程环境直接当现场，不再专门处理理论环境和真实工作现场之间的差别。

## 为什么不用那种做法
Claude Code 不敢这么偷懒，因为它经常要面对 worktree、resume、远端和 session 恢复这些现实差异。没有 setup，这些差异只会在后面各处偷偷出现。

## 代价与回报
回报是系统真的站到用户当前世界里。代价是 setup 对顺序和现场细节都很敏感，看起来更脏也更重。
