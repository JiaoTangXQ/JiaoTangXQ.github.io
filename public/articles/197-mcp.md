# 第十三卷回收索引：外延执行里浏览器世界也先说成 MCP 语言

## 实现链
外延执行里还有一类问题，不在能力能不能跑，  
而在新世界进来以后，系统先用什么语言把它接住。

这条线真正逼出来的，是第十三卷里关于语言统一的判断。  
接入新世界时，第一步不是另起一套说法，而是先让它说对原来的语言。

这页实际回收的实现链是：
- `12-07-01` 把浏览器世界也尽量说成 `MCP` 语言，这会在 `13-04-02` 里回收到“接入新世界时先做语言统一”，说明系统先认得出同一种说法。

真正稳的接入，不是先把东西接进来，再想办法翻译。  
而是先让它说对原语言，再谈边界往外长。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
