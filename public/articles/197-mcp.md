# 浏览器能力为什么要先说成 MCP 语言：语言统一优先于能力接入

面试题版本：**"Claude Code 接入浏览器能力（比如截图、DOM 查询）时，为什么要把这些能力包装成 MCP 工具，而不是直接在系统里加一套浏览器 API？这个包装有什么实际价值？"**

## 不包装的方案会是什么样

假设 Claude Code 直接给浏览器能力加一个专用 API 路径：

```typescript
// 专用路径方案
class BrowserCapabilities {
  async screenshot(selector?: string): Promise<Buffer>
  async queryDOM(selector: string): Promise<Element[]>
  async navigateTo(url: string): Promise<void>
  // ...
}

// 在主循环里单独处理
if (tool.isBrowserTool) {
  return await browserCapabilities.execute(tool)
}
```

这个方案的问题：
- 权限检查需要单独为 BrowserCapabilities 实现
- 提示词里的工具描述需要单独生成
- 日志、监控、工具执行状态追踪需要单独实现
- 并发安全规则需要单独声明和处理
- 未来加入更多浏览器工具时，每次都要在所有这些地方加适配

## MCP 包装带来的复用

把浏览器能力包装成 MCP 工具后：

```typescript
// MCP 工具形状（规范化后）
{
  name: "browser_screenshot",
  description: "截取当前页面或指定元素的截图",
  inputSchema: { type: "object", properties: { selector: { type: "string" } } },
  isConcurrencySafe: () => false,  // 截图会受页面状态影响
  // ...
}
```

这个工具通过统一的工具注册系统进入工具表后，**自动获得了所有内建工具已有的基础设施支持**：

- 权限检查：用户对 MCP 工具统一决策是否允许
- 提示词生成：工具描述自动被包含在系统 prompt 里
- 执行管理：`StreamingToolExecutor` 处理并发、超时、中断
- 进度追踪：工具执行状态在任务系统里统一显示
- 结果记录：`tool_result` 格式的统一处理

这是"先说同一种语言"的价值：进入语言后，所有语言的能力（基础设施）都能被使用。

## 语言统一让模型能理解工具

模型在推理时通过提示词了解可用工具。如果浏览器能力是一个特殊 API，需要特殊的提示词描述，模型需要学习一套新的接口语义。

如果包装成 MCP 工具，模型看到的是它已经熟悉的工具调用格式：`name`、`description`、`inputSchema`。模型不需要知道这个工具背后是浏览器能力还是文件操作，它只需要理解工具的语义（截图是做什么的），然后用标准的工具调用格式触发。

这降低了模型正确使用新能力的认知门槛：接口语言统一了，学习成本降低了。

## 前缀的命名空间作用

当浏览器工具通过 MCP 进入系统，它们通常有 server 前缀，比如 `browser/screenshot`、`browser/navigate`。这个前缀做了两件事：

1. **避免命名冲突**：`screenshot` 可能和其他工具重名，`browser/screenshot` 是唯一的
2. **帮助模型理解来源**：模型通过前缀可以推断工具的大致能力边界——`browser/` 前缀的工具都与浏览器相关

在 `src/services/mcp/mcpStringUtils.ts` 里有前缀处理的逻辑，确保来自不同 server 的工具即使同名也不冲突。

## "先语言，再能力"的顺序

"浏览器世界也先说成 MCP 语言"这个选择的深层逻辑是：**接入顺序应该是先成为系统的一部分，再展开能力**。

如果反过来——先展开浏览器能力，再考虑怎么集成进系统——往往会发现能力和系统的整合需要很多特殊处理，最终代码里充满了 `if (tool.isBrowser) { ... }` 这样的分支。

先说 MCP 语言，意味着浏览器能力在进入系统时就已经是系统语言的一部分，不需要后续的特殊适配。

## 面试角度的总结

浏览器能力包装成 MCP 工具，本质是**用接入标准化换取实现复用**。标准化的代价是需要写规范化适配器（把浏览器 API 映射到 MCP 工具格式），收益是所有的工具基础设施（权限、执行、监控、提示词生成）可以直接复用。这是 Claude Code 在接入新能力时的一致选择：先融入语言体系，再展开能力，而不是为每种新能力单独开一条特殊路径。
