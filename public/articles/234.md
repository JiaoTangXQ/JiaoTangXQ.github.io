# 边界翻译和任务托管：新世界如何被原制度正式接管

面试题版本：**"Claude Code 如何处理 MCP 工具的异步执行？MCP 工具和内建工具在执行模型上有什么区别？这个区别如何在不影响主循环的情况下被消化？"**

## 边界翻译解决的问题

MCP 协议和 Claude Code 内部协议在几个关键语义上有不对齐：

**工具执行模型**：Claude Code 内部的工具调用是"调用 → 等待结果 → 继续"的同步模型（即使内部是异步的，主循环视角是同步的）。MCP 工具的执行可能更复杂——server 可能在处理过程中发送多个进度更新，最终才给出结果。

**错误处理**：Claude Code 内部有标准的错误类型（权限错误、超时、工具不存在等），MCP 的错误响应格式不同，需要翻译成内部标准格式。

**资源引用**：MCP server 可能返回"资源引用"（`resource://...`）而不是直接内容，Claude Code 需要知道如何解引用这个资源。

边界翻译层（`src/tools/MCPTool/`）处理这些不对齐：把 MCP 的执行模型适配成主循环期望的同步视角，把 MCP 的错误翻译成内部错误类型，把资源引用转换成实际内容。

## 任务托管的必要性

某些 MCP 工具的执行时间很长（几分钟到几小时）——比如一个触发长时间 CI 任务的 MCP 工具。如果这类工具走普通的同步调用路径，主循环会一直等待，阻塞所有其他操作。

任务托管的做法：把这类长时间 MCP 工具的执行封装成一个任务对象（`local_agent` 或类似类型），通过任务系统管理：

1. 工具调用触发任务创建
2. 任务在后台执行 MCP 调用，把进度写入 outputFile
3. 主循环不等待，继续处理其他工作
4. 任务完成后，通过 `pendingMessages` 把结果通知主循环

这样 MCP 工具的异步执行被自然地纳入了任务系统的管理框架，不需要为 MCP 特别设计一套异步处理机制。

## "先过原制度"的两层含义

**制度一：权限和信任**
MCP 工具在被使用前，必须通过 Claude Code 的权限系统批准。不管这个工具来自多么可信的 MCP server，每次执行都要经过 `useCanUseTool()` 的检查。这让 MCP 工具和内建工具有相同的权限保障。

**制度二：生命周期管理**
通过任务托管，MCP 工具的长时间执行被纳入任务系统的生命周期管理。用户可以查看 MCP 任务的进度，可以中断它，可以在重连后继续查看结果——这些能力不是 MCP 特有的，而是任务系统的通用能力被 MCP 任务复用了。

## 不走原制度会怎样

如果 MCP 工具不经过边界翻译，直接在主循环里"特殊处理"MCP 响应：

```typescript
// 反面例子：MCP 特殊路径
if (tool.source === 'mcp') {
  // MCP 特殊的执行和等待逻辑
  const mcpResult = await executeMcpTool(tool, args)
  // MCP 特殊的结果处理
  return formatMcpResult(mcpResult)
}
```

这段代码意味着：
- 权限检查可能绕过（MCP 工具自己决定是否需要权限）
- 异步长时间任务阻塞主循环
- MCP 的错误格式和内建工具的错误格式不同，需要在上层做额外适配
- MCP 工具无法被监控、中断、或在后台管理

每一个这样的"特殊路径"都是系统统一性的一个破口。

## 面试角度的总结

边界翻译和任务托管是"让 MCP 能力先过原制度"的两个具体机制：翻译层把 MCP 的协议差异消化掉，让主循环不需要知道工具来自 MCP；托管机制把 MCP 的异步执行纳入任务系统，让后台 MCP 操作和后台 agent 操作有相同的管理方式。这两个机制让 MCP 在进入系统后，行为表现和内建能力一致，而不是一个有特殊待遇的外来者。
