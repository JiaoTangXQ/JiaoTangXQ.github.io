# 第十三卷回收索引：扩展世界里边界翻译和任务托管让新世界进来时也先过原制度

## 实现链
新世界真正能被接住，不是因为它名字够大，  
而是因为边界翻译和任务托管先把它放回原来的制度动作里。

这条线真正逼出来的，是第十三卷里关于“先过原制度，再谈扩张”的判断。

这页实际回收的实现链是：
- `13-04-02` 把边界翻译和任务托管写成“新世界进来时也先过原制度”，这会继续支撑 `07-08-01`，说明接进来的不是孤立工具，而是必须被原制度接住的一整套能力。

真正稳的扩张，不是让新世界平行生长。  
它是先把边界翻译好，再把任务托管进原制度里。

## 普通做法
更普通的做法，是每种来源保留自己的装配方式和运行时语言，等真的冲突了再临时做桥接。

## 为什么不用
Claude Code 不这么做，是因为来源越多，越要先学旧骨架的语言；否则系统会被多套协议、多套入口和多套制度动作拖散。

## 代价
代价是注册、翻译和回收线都会变多，很多地方看起来像是在重复做统一。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
