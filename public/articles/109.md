# 面试题：TaskStateBase 是什么，它为什么要被所有任务类型共享？

## 从源码出发

在 `src/Task.ts` 里有这样一段定义：

```typescript
export type TaskStateBase = {
  id: string
  type: TaskType
  status: TaskStatus
  description: string
  toolUseId?: string
  startTime: number
  endTime?: number
  totalPausedMs?: number
  outputFile: string
  outputOffset: number
  notified: boolean
}
```

这个结构被所有具体任务类型继承。`LocalShellTaskState`、`LocalAgentTaskState`、`RemoteAgentTaskState`、`InProcessTeammateTaskState`、`LocalWorkflowTaskState`、`MonitorMcpTaskState`、`DreamTaskState` 都包含它的全部字段。

**面试题：为什么要抽出这个公共底座？每种任务单独定义自己的字段不是更灵活吗？**

## 七种 TaskType 的差异

先看看这七种任务类型在做什么：

```typescript
export type TaskType =
  | 'local_bash'        // 本地 shell 命令
  | 'local_agent'       // 本地子 agent
  | 'remote_agent'      // 远端容器里运行的 agent
  | 'in_process_teammate' // 进程内队友
  | 'local_workflow'    // 本地工作流
  | 'monitor_mcp'       // MCP 服务监控
  | 'dream'             // Dream 任务（AI 自主探索）
```

这些任务的运行方式天差地别：有的开 shell 子进程，有的用 WebSocket 连云端，有的在同一进程内用 React 状态管理。但它们有一个共同的需要：**主界面、后台面板、主会话都需要能认出"这是哪一份工作"**。

如果没有公共底座，这个"认出来"就会很困难。面板组件渲染任务列表时，要为每种类型写不同的字段读取逻辑；`isTerminalTaskStatus` 这样的判断函数要对每种任务单独实现；通知系统在说"某任务完成了"时，要知道去哪里找 `notified` 标志。

## 公共底座保证了什么

**`id` 和 `type` 字段**：任务 ID 不是随机字符串，而是带类型前缀的有意义标识：

```typescript
const TASK_ID_PREFIXES: Record<string, string> = {
  local_bash: 'b',
  local_agent: 'a',
  remote_agent: 'r',
  in_process_teammate: 't',
  // ...
}
```

这让日志里出现 `a3f2b8c1` 时，你一眼能知道它是 local_agent，不需要查表。前缀是编码在 ID 里的类型信息。

**`status` 和 `isTerminalTaskStatus`**：状态机是所有任务共用的：

```typescript
export type TaskStatus = 'pending' | 'running' | 'completed' | 'failed' | 'killed'

export function isTerminalTaskStatus(status: TaskStatus): boolean {
  return status === 'completed' || status === 'failed' || status === 'killed'
}
```

注释里明确说了这个函数的用途："guard against injecting messages into dead teammates, evicting finished tasks from AppState, and orphan-cleanup paths"。换句话说，这个判断在三个不同的地方被复用，如果每种任务类型单独定义终态，这三处都要分别处理。

**`outputFile` 和 `outputOffset`**：每个任务都有一个磁盘上的输出文件，而且记录了"已经读到哪里了"（offset）。这让任务输出变成了可以随时续读的流，不管任务是在运行还是已完成，其他组件都能以同样的方式读取进度。

**`notified` 标志**：任务完成后的通知逻辑只需要检查这一个字段，不用知道任务类型。

## 公共底座 vs 每种类型单独定义

如果选择"每种任务单独定义字段"：

面板组件在渲染任务列表时：
```typescript
// 没有公共底座时，可能变成这样
if (task.type === 'local_bash') {
  return <div>{(task as LocalShellTaskState).shellStatus}</div>
} else if (task.type === 'local_agent') {
  return <div>{(task as LocalAgentTaskState).agentStatus}</div>
} // ...
```

有了公共底座后：
```typescript
// 任何类型的任务都能这样处理
return <div>{task.status} - {task.description}</div>
```

这不是代码品味的问题，而是维护成本的问题。每次新增任务类型，不共享底座就意味着要找到所有处理任务的地方，逐一添加新的分支。

## createTaskStateBase 的保守设计

初始化函数同样值得注意：

```typescript
export function createTaskStateBase(
  id: string,
  type: TaskType,
  description: string,
  toolUseId?: string,
): TaskStateBase {
  return {
    id,
    type,
    status: 'pending',       // 始终从 pending 开始
    description,
    toolUseId,
    startTime: Date.now(),
    outputFile: getTaskOutputPath(id),
    outputOffset: 0,
    notified: false,
  }
}
```

注意它没有 `endTime`——这个字段在任务结束时才填。没有 `totalPausedMs`——这只在任务被暂停过时才有意义。

这种"只填必须有的字段，其余等到实际发生时再填"的设计，让任务状态的变化过程是可追溯的：从只有 `startTime` 到出现 `endTime`，从没有 `totalPausedMs` 到出现暂停记录，字段的有无本身就在说明任务的生命历程。

## 面试要点

1. **公共底座是系统共识的载体**：七种不同任务类型通过 `TaskStateBase` 表达"我是一份可观察的工作"，这个承诺让主界面、通知系统、清理逻辑都能用同一套接口处理任何类型的任务。
2. **类型前缀 ID 是编码的类型信息**：不只是 UUID，而是携带语义的标识符，日志和调试时直接可读。
3. **outputFile + outputOffset 是外部化的进度**：任务输出写到磁盘，其他组件通过 offset 续读，这让任务状态不依赖任何进程内的生命周期。
4. **notified 字段是通知协议的接口点**：统一的通知状态字段，让通知逻辑不需要了解任务类型就能工作。
