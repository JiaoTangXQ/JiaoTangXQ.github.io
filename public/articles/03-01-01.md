# 为什么一条输入不能原样直接扔给模型

从用户视角看，输入框里只是一句话；从系统视角看，它往往是一团混合物。里面可能夹着粘贴图片、路径、附件、@提及、slash 命令、桥接来源、制度变化，甚至还要先经过 hooks 的审查。Claude Code 如果把这整团东西原样丢给模型，表面上会很直接，实际上是在把本该由系统承担的整理工作偷懒甩锅。

`processUserInput()` 的存在，正是在替系统履行这份责任。它会先处理图片和粘贴内容，必要时缩放、落盘、补元信息；再决定要不要提取附件；再判断这是不是 Bash、是不是 slash 命令、是不是桥接来源的特殊消息；只有这些事情都整理完，普通文本提示才会作为“已经被系统理解过的一条输入”进入后面的会话循环。

这一步的价值，不只是为了兼容更多输入形态，更是为了把“用户说了什么”和“系统现在知道了什么”区分开。模型看到的从来不该只是原始字符串，而应该是被整理成合适会话事件之后的结果。否则模型会被迫同时处理太多本该由运行时负责的脏活，整套系统也会越来越像一个什么都往语言模型里塞的草台班子。

Claude Code 在这里表现得很像一个成熟前台。它不急着把话转出去，而是先确认来的人带了什么、要去哪一扇门、哪些东西该登记、哪些东西该改写。这种输入治理，正是后面一切优雅流程的起点。

## 实现链
`processUserInput()` 会先把数组型输入里的图片块规范化，再把粘贴图片落盘、缩放成 API 可接受的 image block，之后抽附件、判断 bash / slash / bridge 条件，最后才把普通文本交给 `processTextPrompt()` 生成消息。

## 普通做法
最直觉的写法是把输入框当前字符串原样塞给模型，图片和路径都当普通文本处理。

## 为什么不用
因为模型并不适合替运行时分辨“这是一张图片、一个附件、一个命令还是一条需要隐藏给用户的 meta 提示”。这些是系统级判断，不该外包给模型猜。

## 代价
入口处理会更慢，也更复杂；但这份复杂度换来的是后面每一层都能收到已经分好类的输入，而不是继续在主循环里拆炸弹。
