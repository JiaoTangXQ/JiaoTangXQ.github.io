# 先过滤噪声行，再截断：渲染预算要分给真正有内容的消息

`Messages.tsx` 的非虚拟路径在到达 `MAX_MESSAGES_WITHOUT_VIRTUALIZATION` 这个截断线之前，消息列表已经经历了多次过滤和变换。截断面对的不是原始的消息队列。

## 截断的实际时序

```
原始 messages
  → normalizeMessages()
  → reorderMessagesInUI()
  → 过滤 isNullRenderingAttachment 的行（纯后台附件，不面向用户）
  → 过滤进度消息（不占正文预算）
  → applyGrouping() + 各类折叠（压缩重复内容）
  → 到这里才应用 200 条上限
```

截断计数的起点，是已经去过噪的列表。

## 为什么顺序重要

如果先截断再过滤，效果是：截断会从原始队列的第 201 条开始，但原始队列里可能前 50 条都是进度消息和空附件，只有后 150 条是真正有内容的消息。截断在第 200 条打住，但用户看到的实际内容只有 150 条。

先过滤再截断，效果是：200 条的预算完整地分配给了真正会被用户看到的内容行——它们经过了分组和折叠，已经是「为阅读组织过」的形态。预算用在了刀刃上。

## isNullRenderingAttachment 是什么

这类附件是纯后台数据，没有对应的 UI 渲染——比如某些系统内部的状态附件。它们在消息列表里存在，但从用户角度看是隐形的。让它们占据截断预算，就是用不可见的数据消耗了用户可见内容的配额。

过滤掉它们，截断才真正意味着「用户会看到的最多 200 条内容」，而不是「所有消息里最多 200 条（包括隐形的）」。
