# Anthropic API 的 Beta 机制：Claude Code 怎么启用扩展功能？

Anthropic API 的某些功能不在标准接口里，而是通过 `anthropic-beta` 请求头声明使用。Claude Code 在每次 API 调用时需要合并多个来源的 beta 标志，确保启用了所有需要的能力。

`utils/betas.ts` 处理这个聚合逻辑。

## Beta 来源的三个层级

**1. 模型级 beta（`getModelBetas`）**：某些 beta 功能只对特定模型有效。比如 `token-counting-2024-01-01` 只有某些模型支持。模型级 beta 根据当前使用的模型名自动包含。

**2. SDK beta（`getSdkBetas`）**：通过 bootstrap state 传入，来自 SDK 配置（比如通过 API 接入 Claude Code 时可以声明需要的 beta 集合）。

**3. Bedrock 额外 beta（`getBedrockExtraBodyParamsBetas`）**：通过 AWS Bedrock 调用时有特殊的参数格式，需要单独处理。

最终 API 请求里的 beta 列表是这三者的合并：

```typescript
export function getMergedBetas(
  model: string,
  sdkBetas?: string[],
): string[] {
  const modelBetas = getModelBetas(model)
  const sdkBetaList = sdkBetas ?? []
  return [...new Set([...modelBetas, ...sdkBetaList])]
}
```

用 `Set` 去重，防止同一个 beta 标志被重复声明（重复声明不会报错，但是无意义的冗余）。

## 工具到 API Schema 的转换

Claude Code 的 `Tool` 类型和 API 需要的 `BetaToolUnion` 格式不完全相同，`toolToAPISchema` 负责这个转换：

```typescript
// utils/api.ts
export function toolToAPISchema(tool: Tool): BetaToolUnion {
  // 把 Tool 的 inputSchema（Zod schema）转成 JSON Schema
  // 加上 tool name 和 description
  // 处理 MCP 工具的特殊格式
  // 处理 shouldDefer（ToolSearch 延迟加载）
}
```

这个转换很关键——`Tool` 类型用的是 Zod schema 来描述输入结构，但 API 要的是 JSON Schema 格式。而且有些工具有 `inputJSONSchema` 字段（MCP 工具直接提供 JSON Schema），需要优先用这个格式。

## Effort 值和 API 参数的映射

`resolveAppliedEffort` 把 effort 值（用户可以设置"我希望 Claude 在这个任务上花多少精力"）转成具体的 API 参数：

```typescript
// 可能影响的 API 参数：
// - thinking.budget_tokens（扩展思考的 token 预算）
// - max_tokens（限制输出长度）
// - temperature（控制创造性 vs 确定性）
```

Effort 是一个用户友好的抽象层，用户不需要知道 `budget_tokens=8000` 意味着什么，只需要说"高 effort"或"低 effort"。

## Tool deferred loading（ToolSearch 系统）

```typescript
// tools.ts
export function isToolSearchEnabledOptimistic(): boolean {
  // 检查是否启用 ToolSearch（工具延迟加载）
}
```

当工具太多时，把所有工具的完整 schema 放在每次 API 请求里会消耗大量 token（工具 schema 可能很长）。ToolSearch 是一个优化：

- 首次请求只发送少数"总是需要"的工具（`alwaysLoad: true`）
- 其他工具被"延迟"（`shouldDefer: true`），只发送工具名和一行简介
- 模型可以调用 `ToolSearch` 工具来查找需要的工具，然后工具的完整 schema 才被加载

这是一个 lazy-loading 思想在 LLM 工具系统里的应用。

## 系统提示的缓存前缀

`getCLISyspromptPrefix` 和 `logAPIPrefix` 处理系统提示的缓存分割：

```typescript
// 系统提示分为两部分：
// 1. 缓存前缀（static，长期不变，可以被 prompt cache 缓存）
// 2. 动态后缀（随 session 变化，每次请求可能不同）
```

Anthropic API 的 prompt cache 以前缀匹配——前缀越长、越稳定，缓存命中率越高。把不变的系统提示内容（工具描述、固定指令）放在前面，把变化的内容（当前目录、用户设置）放在后面，最大化缓存利用。

## 面试角度

与 LLM API 集成的工程实践：

1. **Beta 功能聚合而不是分散管理**：每个地方自己维护 beta 列表容易遗漏，集中聚合更安全
2. **schema 转换层要显式**：Zod → JSON Schema 的转换不应该在每个调用点手动做
3. **lazy-loading 减少 prompt token**：工具太多时，按需加载工具 schema 是实际可行的优化
4. **缓存前缀要稳定**：把不变的内容放在系统提示前面，变化的内容放后面，这是使用 prompt cache 的核心技巧
