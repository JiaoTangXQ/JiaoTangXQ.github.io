# stub 和兼容路径说明什么？一个成熟产品在和过去谈判

## 工程反思：兼容性是责任，不是懒惰

代码里看到 stub、兼容路径和历史残留，第一反应通常是"这里没有清理干净"。但在成熟产品里，这个第一反应经常是错的。

stub 和兼容路径存在，往往是因为系统已经在跑，已经有真实用户依赖旧行为。直接删掉旧路径，伤害的不是代码洁癖，而是真实用户的工作流。

## remotePermissionBridge 的 stub 设计

`remotePermissionBridge.ts` 里，当远端工具请求权限时，如果这个工具在本地没有对应的注册工具，会为它生成一个最小化的 `Tool` stub：

```typescript
// 最小可显示、可审批的本地 Tool
const stubTool = {
  name: toolName,
  description: `Remote tool: ${toolName}`,
  // 足够让权限确认 UI 渲染的最小字段
}
```

这个 stub 不是漂亮的设计，是务实的选择。没有它，当远端运行一个本地未注册的工具时，权限确认链就会直接崩溃——因为它找不到工具的描述信息来渲染界面。

有了 stub，权限链继续工作，用户能看到"有一个叫 `xxx` 的远端工具在请求权限"，而不是一条意义不明的错误。

## QueryEngine 里的兼容边界

`QueryEngine` 和 `query.ts` 里有不少"照顾旧消息格式"的处理：补合成 tool_result、处理旧格式的消息边界、兼容不同版本客户端的输出格式。

这些处理的存在理由是：Claude Code 已经有大量保存在磁盘上的 transcript 文件，以及仍在使用旧版本客户端的用户。如果新版本直接拒绝旧格式，这些 transcript 就会无法加载，这些用户就会遇到莫名其妙的错误。

更干脆的做法是直接删掉兼容逻辑：旧格式就报错，旧路径就移除，历史字段一律不认。代码会更干净。代价是老会话断掉，旧入口失效，用户现场受伤。

Claude Code 的选择是继续谈判：兼容可以保，但尽量压在边缘，不让它扩散到主干结构。

## 谈判的边界在哪里

和过去谈判不是无限妥协。一个清晰的原则是：**兼容旧行为，但不让旧行为改写新的制度**。

具体来说：旧格式的消息可以被解析和处理，但新的权限检查逻辑不会因为"这是旧消息"而绕过。旧路径可以继续存在，但新功能不会为了不惊扰旧路径而降低自己的设计质量。

这个边界画得清不清楚，是区分"健康的兼容"和"被历史拖着走"的关键。

## 代价

遗留分支拉高了阅读和测试负担。读代码时遇到兼容逻辑，需要先弄清楚"这是为了哪个旧行为"，才能判断它能不能删。测试时需要覆盖旧格式和新格式两条路径。

这个代价是真实的，但它是成熟产品的成熟代价，不是设计失误的标志。

## 面试指导

被问"你会怎么处理遗留代码"时，一个有深度的答案不是"全部重写"，而是：

先判断这段遗留代码是否有真实的依赖方（用户、数据、外部系统），再决定是删除、替换、还是用 stub + 翻译层包起来让新系统继续工作。

能做出这个判断的人，比只会说"旧代码都应该清理掉"的人，在真实工程场景里更有价值。
