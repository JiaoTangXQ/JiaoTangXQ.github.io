# 面试题：状态栏用 wrap='truncate' 而非 wrap='wrap'，这个选择背后的终端 UX 原则是什么？

这是一道深入终端渲染语义的题目。它的答案不复杂，但能说清楚"为什么"的候选人，说明真正理解了终端界面和 Web 界面的本质差异。

## 问题设定

在 Ink 框架里，`Text` 组件有几种 `wrap` 选项：
- `wrap="wrap"`：内容超出宽度时自动换行
- `wrap="truncate"`：内容超出宽度时截断（末尾加省略号或直接截断）
- `wrap="truncate-start"`：从开头截断
- `wrap="truncate-middle"`：从中间截断

底部区域的 parts 文本使用的是 `wrap="truncate"`。为什么不用 `wrap="wrap"`？

## 终端高度是刚性预算

在 Web 里，页面高度是弹性的：内容撑开多少就有多少高度，用户可以往下滚动。底部导航栏用 `position: fixed`，不占据文档流空间。

终端完全不同：终端窗口的高度是固定的（N 行），所有内容必须在这 N 行里分配空间。没有"往下滚动看页面内容"这个操作——终端的滚动是看历史，不是看更多内容。

在这个约束下，如果底部区域使用 `wrap="wrap"`：

1. 某条补充信息的文本很长
2. 文本自动换行，底部从 1 行变成 2 行
3. 消息区被压缩 1 行
4. 用户正在阅读的内容消失一行

这和前面讨论的"高度 0→1 抖动"问题是一样的，但这次更糟：不是临时的，而是内容本身驱动的高度增长。

## truncate 是一种信息压缩策略

`wrap="truncate"` 的选择等于做了一个决定：**当底部内容超出一行宽度时，牺牲信息完整性，保护布局高度稳定**。

这是一个刻意的信息取舍：用户看到的底部信息可能是不完整的（被截断的）。但这个不完整是可预期的——用户知道"底部就是一行"，知道信息可能被截断，会主动去别的地方查完整信息。

比起高度不稳定造成的阅读中断，这种"信息可能被截断"的预期是用户更容易适应的。

## wrap 和 truncate 在 UX 层面的语义差异

**`wrap="wrap"` 的隐式承诺**：所有内容都会显示，但占用不可预测数量的行。

**`wrap="truncate"` 的隐式承诺**：只占固定数量的行，但不保证所有内容都可见。

对于底部区域这种"固定行高度的摘要区域"，第二个承诺更合适。它把底部定义为一个**有固定容量的摘要空间**，而不是一个**能无限扩展的内容区域**。

## 实际截断的优先级效应

```
[mode: autoApprove] [2 tasks running]  [remote: https://session-a2f3...
                                                              ↑ 被截断
```

被截断的总是最后面的内容（远端 URL 的后半段、PR 状态的详细描述等）。最前面的内容（mode、任务数量、远端的存在）总是被保留。

这和前面讲的排列顺序原则是一体的：**排列顺序 + 截断策略**共同实现了信息的优先级保障。不管终端有多窄，`mode` 总是可见的；`remote` 的 URL 可能被截断，但用户至少知道"有一个远端连接"；PR 在极窄时可能完全不出现（由前面的条件判断决定）。

## 边界情况的处理

如果内容极端宽，连 modePart 也开始被挤压怎么办？

这时候 `wrap="truncate"` 只作用于 parts，modePart 和 tasksPart 有自己的宽度保护逻辑：

```typescript
const maxSubjectWidth = Math.max(15, columns - 15 - ownerWidth);
const displaySubject = truncateToWidth(task.subject, maxSubjectWidth);
```

任务 subject 会被手动截断到 `columns - 15` 的宽度，保留至少 15 列给状态指示器和图标。这是一个在 `wrap="truncate"` 之前的手动截断逻辑，更精确地控制了每个部分的最大宽度。

## 面试指导

**直接问法**："为什么底部文本用 truncate 而不是 wrap？"

答：因为终端高度是刚性预算。底部一旦用 wrap 换行，就会偷走消息区的显示空间，导致阅读内容被压缩。truncate 把底部定义为"固定高度的摘要空间"，信息可能被截断，但布局高度始终稳定。

**追问**："truncate 意味着用户看不到完整信息，这可以吗？"

答：可以，因为底部本来就是摘要层，不是完整信息层。被截断的通常是 URL 末尾、详细描述等次要部分，核心信息（远端连接的存在、任务状态）仍然在。用户知道"底部只有一行"，知道可能有截断，会去别的地方查完整信息。这比"底部无限增长"对用户体验的伤害小得多。

**系统性追问**："wrap 策略在终端 UI 和 Web UI 里的根本差异是什么？"

答：Web 里页面高度弹性，内容驱动高度，用户通过滚动访问所有内容。终端里窗口高度固定，所有内容在有限行里竞争，没有弹性增长空间。这导致终端 UI 对"布局高度稳定"的需求远高于 Web，也导致截断比换行更常用——因为截断是"我知道自己的空间是固定的"，换行是"让内容来决定我占多少空间"。

**代码能力考察**："你如何在终端里实现一个'溢出时显示+N more'的提示而不是截断？"

答：手动计算可显示宽度，截断文本，再在末尾追加"+N more"文字（用 stringWidth 函数量取实际显示宽度，因为有些字符宽度是 2）。这不能直接用 `wrap="truncate"` 自动实现，需要应用层自己计算。TaskListV2 里的 `hiddenSummary` 逻辑（"+N in progress, M pending"）就是类似的手动实现。

---

*核心考察点：终端高度刚性预算理解、truncate vs wrap 的 UX 语义差异、终端界面和 Web 界面设计约束的根本差异。*
