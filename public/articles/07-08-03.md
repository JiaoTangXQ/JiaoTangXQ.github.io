# resources/list_changed 触发时，为什么 prompt 缓存也要一起失效？

## 从一个 bug 说起

假设一个 MCP 服务器提供了以下能力：
- 资源：`file://project/config.yaml`（项目配置文件）
- Prompt：`Code Review`（模板里引用了上述资源：`参考配置：{resource://file://project/config.yaml}`）

用户修改了 `config.yaml`，服务器推送了 `resources/list_changed`。

如果只刷新资源缓存，不失效 prompt 缓存：
- 资源列表是最新的（`config.yaml` 已更新）
- Prompt 模板是旧的（模板里的配置引用可能指向旧格式）
- 模型用旧 prompt 去理解新资源，产生错误的上下文解读

这就是为什么资源变化必须传播到 prompt 缓存。

---

## 依赖关系的有向图

```
資源 (Resources)
    ↓ 被引用
Prompts（可能引用资源内容）
    ↓ 被引用
Skills（可能基于 prompt 构建）
    ↓ 被暴露
模型看到的能力集合
```

在这个依赖链里，下游缓存的有效性依赖上游数据的新鲜度。当上游变化时，所有依赖它的下游缓存都应该失效。

---

## 为什么不连 tools 一起失效？

注意：`resources/list_changed` **不**失效工具缓存。因为工具和资源没有依赖关系：

- 工具是"函数"——给它输入，它返回输出。工具的定义（名称、参数、描述）不依赖资源内容
- 资源是"数据"——工具可以读取资源，但工具的定义本身不引用资源

所以：
- `tools/list_changed` 只失效工具缓存
- `resources/list_changed` 失效资源缓存 + prompt 缓存（因为 prompt 可能引用资源）
- `prompts/list_changed` 只失效 prompt 缓存

这是精准失效，不是全量重置。

---

## 对比：过失效 vs 欠失效

**过失效（过于激进）：**
任何 `list_changed` 都重置所有缓存。

优点：绝对不会用到过期数据。
缺点：频繁的无效缓存重建，工具列表很长时（有些服务器有几十个工具）重建开销大。

**欠失效（过于保守）：**
`resources/list_changed` 只刷新资源缓存，不失效 prompt 缓存。

优点：性能好，缓存命中率高。
缺点：模型可能用过期的 prompt 模板，导致上下文不一致。

Claude Code 的选择是**按依赖关系失效**：既不过于激进，也不欠失效。

---

## 实际代码：失效的触发点

`client.ts` 里注册资源变化通知：

```typescript
client.setNotificationHandler('notifications/resources/list_changed', async () => {
  // 1. 重拉资源列表
  const resources = await client.listResources()
  updateResourcesCache(serverName, resources)

  // 2. 连带失效 prompt 缓存（因为 prompt 可能引用这些资源）
  invalidatePromptCache(serverName)

  // 3. 如果 MCP_SKILLS 开启，连带失效 skill 缓存
  if (feature('MCP_SKILLS')) {
    invalidateMcpSkillCache(serverName)
  }
})
```

三步失效对应三层依赖。

## 更广的含义：分布式缓存一致性

这个问题在更大的系统里会变得更复杂。假设有三个微服务：

```
UserService (数据源)
    ↓ 被依赖
ProfileService (缓存用户数据)
    ↓ 被依赖
RecommendationService (缓存用户偏好，基于 profile)
```

当 UserService 的数据变化时，应该怎么通知？

常见方案：
1. **直接通知**：UserService 通知 ProfileService，ProfileService 再通知 RecommendationService
   - 优点：精准
   - 缺点：UserService 需要知道下游依赖关系（耦合）
2. **发布-订阅**：UserService 发布变化事件，各服务自己决定是否失效
   - 优点：解耦
   - 缺点：下游需要自己维护依赖关系图
3. **版本号/时间戳**：所有缓存都带版本，查询时验证版本是否最新
   - 优点：简单
   - 缺点：频繁的版本检查增加延迟

Claude Code 的 MCP 场景相对简单（单客户端，少数几种对象类型），直接通知链是合理的。

---

## 面试指导

**"如何在分布式系统中维护缓存一致性？"**

先澄清约束：
- 有多少级缓存？（一级 vs 多级）
- 数据的依赖关系是什么？（是否有下游依赖）
- 对一致性的要求是什么？（强一致 vs 最终一致）

Claude Code 的 MCP 场景是：多类对象（tools/resources/prompts/skills）有单向依赖关系，需要最终一致性（通知到达后立即刷新）。这适合"通知 + 精准失效"方案。

如果依赖关系复杂（多对多），考虑引入依赖图（如 React Query 的 `invalidateQueries`，可以根据 queryKey 的匹配模式批量失效）。
