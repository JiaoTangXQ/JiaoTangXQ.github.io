# PermissionUpdate：权限规则是怎么被修改的？五种操作的不可变更新链

Claude Code 的权限状态不是直接被修改的——每次改变权限规则，都通过 `PermissionUpdate` 对象描述意图，再通过 `applyPermissionUpdate` 生成新的 `ToolPermissionContext`。这是经典的不可变状态更新模式。

## PermissionUpdate 的五种操作类型

```typescript
type PermissionUpdate =
  | { type: 'setMode'; mode: PermissionMode }
  | { type: 'addRules'; behavior: 'allow' | 'deny' | 'ask'; destination: PermissionUpdateDestination; rules: PermissionRuleValue[] }
  | { type: 'replaceRules'; behavior: ...; destination: ...; rules: PermissionRuleValue[] }
  | { type: 'removeRules'; behavior: ...; destination: ...; rules: PermissionRuleValue[] }
  | { type: 'addDirectories'; destination: ...; directories: string[] }
  | { type: 'removeDirectories'; directories: string[] }
```

每种操作的语义：
- `setMode`：切换权限模式（default/plan/acceptEdits 等）
- `addRules`：在已有规则列表里追加新规则
- `replaceRules`：替换某个 destination 下的所有规则
- `removeRules`：从规则列表里删除指定规则
- `addDirectories`：添加额外的工作目录（允许访问非 cwd 下的目录）
- `removeDirectories`：移除工作目录

## applyPermissionUpdate：纯函数的状态转换

```typescript
export function applyPermissionUpdate(
  context: ToolPermissionContext,
  update: PermissionUpdate,
): ToolPermissionContext {
  switch (update.type) {
    case 'setMode':
      return { ...context, mode: update.mode }

    case 'addRules': {
      const ruleKind = update.behavior === 'allow' ? 'alwaysAllowRules'
                     : update.behavior === 'deny'  ? 'alwaysDenyRules'
                                                   : 'alwaysAskRules'
      return {
        ...context,
        [ruleKind]: {
          ...context[ruleKind],
          [update.destination]: [
            ...(context[ruleKind][update.destination] || []),
            ...ruleStrings,
          ],
        },
      }
    }
    // ...
  }
}
```

所有 case 都是 spread 操作——从不直接修改 `context` 对象，总是返回新对象。`behavior → ruleKind` 的映射把逻辑和数据结构解耦：调用方说"添加一个 allow 规则"，函数内部知道要写到 `alwaysAllowRules` 字段。

## additionalWorkingDirectories：用 Map 管理目录权限

```typescript
case 'addDirectories': {
  const newAdditionalDirs = new Map(context.additionalWorkingDirectories)
  for (const directory of update.directories) {
    newAdditionalDirs.set(directory, {
      path: directory,
      source: update.destination,  // 记录这个目录是从哪里授权的
    })
  }
  return { ...context, additionalWorkingDirectories: newAdditionalDirs }
}
```

`additionalWorkingDirectories` 是 Map 而不是 Set——不只是存目录路径，还存 `source`（谁授权了这个目录）。`source` 字段用于审计和撤销：如果某个配置源被移除，可以根据 `source` 找到并清理它添加的所有目录。

不可变更新时先复制 Map（`new Map(context.additionalWorkingDirectories)`），再修改副本，保持了纯函数语义。

## removeRules：Set 去重 + filter

```typescript
case 'removeRules': {
  const existingRules = context[ruleKind][update.destination] || []
  const rulesToRemove = new Set(ruleStrings)
  const filteredRules = existingRules.filter(rule => !rulesToRemove.has(rule))

  return {
    ...context,
    [ruleKind]: { ...context[ruleKind], [update.destination]: filteredRules },
  }
}
```

要删除的规则先放进 Set（O(1) 查找），再用 filter 遍历现有规则。比双重循环（O(N*M)）性能好，也比 `includes` 调用（O(N)）更清晰。

## destination：规则属于哪里

`PermissionUpdateDestination` 是权限规则的"归属"——规则是临时的（只对当前会话）还是持久的（写入配置文件）？属于哪个配置源（用户全局设置、项目设置、企业策略）？

```typescript
type PermissionUpdateDestination =
  | 'localProject'    // 写入项目级 .claude/settings.json
  | 'userSettings'    // 写入用户级 ~/.claude/settings.json
  | 'session'         // 只在当前会话内有效
  | 'policySettings'  // 企业策略（只读，不能通过 UI 修改）
```

`persistPermissionUpdates` 函数负责把 `destination` 不是 `session` 的更新写入对应的配置文件。

## extractRules：从更新列表提取规则

```typescript
export function extractRules(
  updates: PermissionUpdate[] | undefined,
): PermissionRuleValue[] {
  if (!updates) return []
  return updates.flatMap(update => {
    switch (update.type) {
      case 'addRules':
        return update.rules
      default:
        return []
    }
  })
}
```

只提取 `addRules` 类型的规则（不包括 `replaceRules`、`removeRules`）。这个函数用于权限请求对话里展示"这次操作需要什么权限"——用户看到的是需要新增的规则，而不是整个更新操作列表。

## 面试角度

PermissionUpdate 的设计是不可变状态管理的典型应用：

1. **操作描述而不是直接修改**：把"要做什么"和"怎么做"分离，PermissionUpdate 是意图，applyPermissionUpdate 是执行
2. **五种操作覆盖全部变更场景**：add/replace/remove 三种规则操作 + setMode + 目录管理，没有多余的 case
3. **destination 把规则的归属编码进更新对象**：不需要在调用方判断"这个规则应该写到哪个文件"，更新对象自带这个信息
4. **不可变让调试变简单**：每个权限状态变更都有一个对应的 PermissionUpdate 对象，可以序列化、记录、重放
