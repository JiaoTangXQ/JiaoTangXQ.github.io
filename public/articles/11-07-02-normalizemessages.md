# normalizeMessages：让每个消息块有且只有一个身份

消息编排流水线的第一步是 `normalizeMessages()`，它解决的问题很基础但很关键：输入的消息格式不统一。

## 原始消息的问题

API 返回的消息可以包含多个内容块。一条助手消息里可能同时有 thinking 块、文本块和多个工具调用块。一条用户消息里可能有文本和工具结果。

如果把这些多块消息直接传给后续处理，每个处理步骤都需要知道「消息可能有多块内容」，每个步骤都需要处理块的拆分和关联。这会让后续逻辑复杂度成倍增加。

## normalize 做了什么

`normalizeMessages()` 把每个多块消息拆开，每块变成独立的消息单元。关键的额外处理：

**`deriveUUID()`**：为拆出来的每个块派生一个稳定的 UUID。这不是随机生成的，而是基于原始消息 UUID + 块位置派生的——同一条原始消息拆出来的第二块，永远派生出同一个 UUID。这保证了身份稳定性：即使消息列表重新渲染，同一个块仍然有同一个 ID。

**`isNewChain`**：标记一条原始消息拆出的各块中，哪些是后续块（不是第一块）。这让后续的重排和分组逻辑知道「这些块来自同一条原始消息，应该保持顺序」。

**不需要拆的直接通过**：system 消息、progress 消息、attachment 保持原样。normalize 只处理需要统一格式的那部分。

## 建立统一形状的世界

`normalizeMessages()` 之后，后续每一步处理面对的都是「单内容块消息」的世界。重排可以按 tool_use_id 配对，分组可以按类型聚类，折叠可以按消息边界决策。没有人需要再去处理「这条消息里可能有几个块」这个问题。

这是流水线设计的经典做法：在最开始统一输入格式，让后续每个步骤都在一个干净的假设上工作。
