# 为什么 setup() 是理论世界切进真实现场的那一刀

在 `init()` 之前，Claude Code 更像是在搭一套“可以运行”的环境；到了 `setup()`，它才真正开始回答另一个更具体的问题: 这次会话到底要在哪个现场里活。两者差别很大。前者是把机器准备好，后者是决定这台机器此刻究竟站在哪里、替谁工作、用哪一套会话设施说话。

所以 `setup()` 做的事情看起来很杂，骨子里却很统一。它会设定真正的 `cwd`，捕获 hooks 配置快照，启动文件变更 watcher，按需要创建 worktree、tmux session 和消息 socket，切换 session 身份，更新 `projectRoot`，甚至在工作现场变化后重新抓一次 hooks 配置。也就是说，它不是在做零散初始化，而是在把“抽象启动”落成“这一次具体会话”。

这一步为什么这么关键？因为后面的很多能力根本不是对着“一个理论上的项目”运行的，而是对着当前这个真实现场运行的。技能从哪里找，hooks 从哪里读，命令对哪个目录生效，session 记忆挂在哪个项目根上，都要等这一步落地以后才真正有坐标。`setup()` 的价值，恰恰就在于它让后面整条链第一次踩到了地。

## 实现链
这页对应的实现链都属于 `setup.ts` 那把刀：把前面抽象可运行的环境，真正切进当前 cwd、worktree、session、repo 和连接现实里。代码在这里开始回答“这一次到底在哪干活”。

## 普通做法
更常见的写法是把当前进程环境直接当现场，不再专门处理理论环境和真实工作现场之间的差别。

## 为什么不用那种做法
Claude Code 不敢这么偷懒，因为它经常要面对 worktree、resume、远端和 session 恢复这些现实差异。没有 setup，这些差异只会在后面各处偷偷出现。

## 代价与回报
回报是系统真的站到用户当前世界里。代价是 setup 对顺序和现场细节都很敏感，看起来更脏也更重。
