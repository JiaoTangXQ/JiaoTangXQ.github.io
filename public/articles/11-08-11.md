# CollapsedReadSearchContent 和 AssistantToolUseMessage 直接消费 lookups，不是 props——设计上意味着什么？

在传统的 React 组件设计里，父组件计算状态，通过 props 传给子组件。Claude Code 这里反过来：父组件把整个 lookups 对象传给子组件，子组件自己查。这有什么区别？

## 两种传递方式的区别

**方式 A：父组件计算好再传**

```typescript
// Messages.tsx 里
const toolResult = lookups.toolResultByToolUseID.get(toolUseId)
const isResolved = lookups.resolvedToolUseIDs.has(toolUseId)
const siblings = lookups.siblingToolUseIDs.get(toolUseId)

return <AssistantToolUseMessage
  toolResult={toolResult}
  isResolved={isResolved}
  siblings={siblings}
/>
```

**方式 B：子组件自己查（实际实现）**

```typescript
// Messages.tsx 里
return <AssistantToolUseMessage
  msg={msg}
  lookups={lookups}
/>

// AssistantToolUseMessage 里
function AssistantToolUseMessage({ msg, lookups }) {
  const toolUseId = msg.message.content[0].id
  const toolResult = lookups.toolResultByToolUseID.get(toolUseId)
  const isResolved = lookups.resolvedToolUseIDs.has(toolUseId)
  // ...
}
```

## 方式 B 的优势：组合查询

`CollapsedReadSearchContent` 在渲染一个折叠的 Read/Search 组时，需要对**组内每个工具调用**做状态查询：

```typescript
function CollapsedReadSearchContent({ group, lookups }) {
  return group.messages.map(msg => {
    const toolUseId = getToolUseId(msg)
    const result = lookups.toolResultByToolUseID.get(toolUseId)
    const isResolved = lookups.resolvedToolUseIDs.has(toolUseId)
    const isErrored = lookups.erroredToolUseIDs.has(toolUseId)
    // ...
  })
}
```

如果用方式 A，父组件需要预先知道 group 里有多少条消息，并为每条消息提取好状态，组装成数组传进去。这把"组的结构"和"状态查询"的逻辑都推到了父组件，父组件变成了知道得太多的"上帝组件"。

方式 B 让 `CollapsedReadSearchContent` 自己决定查哪些 ID，父组件只负责传 group 结构和 lookups 原始数据，不负责预提取。

## 方式 A 的优势：props 更精确，更容易测试

方式 A 有一个明确的优势：子组件的 props 类型完全描述了它依赖的数据，单元测试时不需要构造整个 lookups 对象，只需要传入精确的 `toolResult`、`isResolved` 等值。

Claude Code 选择方式 B 意味着做了取舍：测试时需要构造 lookups mock，但换来了组件内部的查询灵活性和父组件的简洁。

## lookups 的引用稳定性

`lookups` 每次 `buildMessageLookups` 调用都是新对象（引用变化），这意味着使用方式 B 的组件在 lookups 更新时总会接受新的 prop，会重渲。

这在 Claude Code 的架构里是预期的：lookups 更新代表数据状态变化，消费 lookups 的组件应该重渲以反映新状态。想避免不必要的重渲，应该用 `React.memo` + 比较 lookups 里真正影响该组件的键，而不是减少 lookups 的传递。

## erroredToolUseIDs 的渲染影响

`AssistantToolUseMessage` 会检查 `lookups.erroredToolUseIDs.has(toolUseId)`，如果为真，会渲染错误样式（红色边框、错误图标）。这个信息不走独立的 props，也不从本地 state 来，就是从 lookups 里读的一个 Set 成员检查。

设计上的意义：工具调用是否出错，是一个**跨消息的状态**——错误信息在 `tool_result` 里，工具定义在 `tool_use` 里，它们在不同的消息行。组件自己无法从手边的单条消息判断，必须通过 lookup 查跨行关系。

## 面试指导

这道题的核心是"数据在组件层次里应该流到哪个层级才计算"。

**两种模式的适用场景**：
- 方式 A（提前计算 → props）：子组件的数据需求可以在父组件静态地确定，子组件的 props 类型应该精确表达依赖
- 方式 B（传原始数据 → 子组件查）：子组件的数据需求依赖于它自己的内部结构（比如组内有多少条消息），父组件无法预先枚举

**一个好的判断标准**：如果给子组件单独写测试时，构造 mock props 的代价（方式 A）和构造 lookups mock 的代价（方式 B）差不多，优先选方式 A（更精确的 props 类型）。如果方式 A 的 mock 需要枚举所有可能的子结构（动态数量的工具调用），选方式 B。

**进阶追问**：lookups 里有 9 张表，但 `AssistantToolUseMessage` 只用了其中 3 张，传整个 lookups 会不会导致过度依赖？答：是的，这是方式 B 的代价。解决方案是定义一个 `ToolUseLookups` 类型，只包含子组件实际需要的字段，让父组件提取后传入。这是 TypeScript 里 Pick 类型的典型用场。
