# 深度分析：StatusLine 被 hook 编排意味着什么

## 普通状态栏 vs Claude Code 的状态栏

普通的状态栏是一个**展示端点**：有一些数据（当前分支、行号、文件名），状态栏把这些数据格式化展示出来。状态栏是被动的，它不影响任何逻辑。

Claude Code 的 StatusLine 允许用 hook 命令生成文本：

```json
{
  "hooks": {
    "StatusLine": "echo 'Working on: $(cat .current-task)'"
  }
}
```

hook 命令是可以执行任意逻辑的：读文件、调接口、检查 git 状态、甚至触发其他动作。

**当状态栏的内容由 hook 命令决定时，状态栏就不再只是展示壳了。**

## hook 系统是什么

Claude Code 的 hook 系统允许在特定事件点执行用户定义的命令：

- `PreToolUse`：工具调用前
- `PostToolUse`：工具调用后
- `Notification`：通知触发时
- `StatusLine`：状态栏需要刷新时
- `ConfigChange`：配置变化时

hook 命令的执行结果会影响对应事件的处理：
- `PreToolUse` hook 可以拒绝或修改工具调用
- `StatusLine` hook 的输出就是状态栏的内容

在 `src/hooks/useSettingsChange.ts` 里，设置变化会触发 hook 系统重新评估：

```typescript
export function useSettingsChange(
  onChange: (source: SettingSource, settings: SettingsJson) => void,
): void {
  // ...
  useEffect(
    () => settingsChangeDetector.subscribe(handleChange),
    [handleChange],
  )
}
```

`settingsChangeDetector` 会通知所有订阅者配置发生了变化，其中包括需要重新执行 StatusLine hook 的逻辑。

## 制度进入显示层的含义

"制度进入显示层"是一个听起来很抽象的说法，但有具体的含义：

**示例：使用 hook 控制状态栏**

用户可以配置一个 hook，让状态栏显示"当前任务：[从某个文件读取的任务名]"。这个 hook 在每次 Claude Code 界面刷新时执行，读取最新的任务信息。

这意味着状态栏的内容**不是 Claude Code 系统决定的**，而是用户通过 hook 注入进来的外部信息。

**更重要的是**：这个机制可以用来显示任何制度性信息——当前会话的权限级别、合规状态、团队配置、当前执行的任务阶段。

状态栏变成了"制度可见性"的一个表面：制度里的重要信息，可以通过 hook 把它展示给用户，而不需要修改 Claude Code 的源码。

## 为什么不直接在源码里加状态栏字段

一种替代方案：每次需要在状态栏展示新信息，就修改 `StatusLine` 组件的代码，加一个新字段。

这个方案的问题：

**需求无法预见**：每个用户对状态栏的需求不同——有的想看 git 状态，有的想看当前任务，有的想看系统资源。不可能在源码里预先支持所有可能的信息。

**企业定制需求**：企业用户可能需要在状态栏展示内部系统的信息（比如"当前连接的数据库"、"本次会话的审计 ID"）。这些信息 Claude Code 根本不知道存在，没法预先实现。

hook 机制把这个扩展权力交给用户：你想在状态栏展示什么，写一个 hook 命令，Claude Code 执行这个命令并展示结果。

## 状态栏作为制度最前沿的摘要

状态栏通常是屏幕上最持续可见的信息区域，用户的眼睛习惯于扫一眼状态栏来了解系统当前的状态。

如果状态栏只展示"行号"这样的无关信息，这个注意力入口就浪费了。

如果状态栏能展示"制度最前沿的摘要"——当前任务是什么、权限级别是什么、是否有待处理的事项——用户就能以很低的认知成本了解系统状态，不需要去翻日志或者打开详情面板。

这是"界面也是运行时的一部分"的具体体现：界面不只是显示结果，而是主动管理用户对系统状态的认知。

## 面试要点

1. **StatusLine hook 让状态栏从展示端点变成信息入口**：不只展示预设的数据，还能展示用户通过 hook 注入的任意信息。
2. **制度进入显示层意味着界面参与信息治理**：用户可以把关键的制度信息（权限状态、合规状态、任务阶段）放进状态栏，降低认知成本。
3. **hook 机制是可扩展性的关键**：无法预见所有用户需求，用 hook 把扩展权力交给用户，比在源码里穷举所有可能更可持续。
4. **状态栏是制度可见性的表面**：持续可见的 UI 元素是展示"系统现在在哪里"的自然位置，hook 机制让这个位置被充分利用。
