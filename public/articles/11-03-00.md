# 性能分析：useContext 和 useSyncExternalStore+selector，重渲染次数差多少？

React Context 有一个著名的性能陷阱：只要 context value 变化，所有消费者都会重渲染，不管它们是否真的关心这次变化。

这在 `AppState` 里是不可接受的。`AppState` 有 60+ 个字段，每次任何一个字段变化都重渲染所有订阅组件，一个终端界面会卡到无法使用。

Claude Code 的解决方案是完整的「外部 store + selector」模式：

```typescript
// 组件只在 verbose 字段变化时才重渲染
const verbose = useAppState(s => s.verbose)

// 组件只在 mainLoopModel 字段变化时才重渲染
const model = useAppState(s => s.mainLoopModel)
```

这一章讲这个机制从底层到上层的完整实现。

---

## 为什么不用 useContext

先看 `useContext` 的问题：

```tsx
// 如果用 Context 直接传 AppState
const AppContext = createContext<AppState>(defaultState)

function StatusBar() {
  const state = useContext(AppContext)
  return <span>{state.verbose ? 'verbose' : 'normal'}</span>
}
```

每次 `AppState` 里任何字段变化（比如某个任务的状态更新），`AppContext` 的值就是一个新对象，`StatusBar` 就会重渲染，即使 `verbose` 字段根本没变。

在 Claude Code 的使用场景里：
- 任务状态每秒可能更新多次
- 消息列表持续增长
- Bridge 连接状态偶尔变化

如果每次变化都触发所有订阅组件重渲染，一次活跃会话里每秒可能发生几十次不必要的重渲染。

## 外部 store 的核心机制

解决方案是把状态移到 React 树之外，然后用 `useSyncExternalStore` 以精细粒度订阅：

```typescript
// store.ts
export function createStore<T>(initialState: T, onChange?: OnChange<T>): Store<T> {
  let state = initialState
  const listeners = new Set<Listener>()

  return {
    getState: () => state,
    setState: (updater) => {
      const prev = state
      const next = updater(prev)
      if (Object.is(next, prev)) return  // 无变化就不通知
      state = next
      onChange?.({ newState: next, oldState: prev })
      for (const listener of listeners) listener()
    },
    subscribe: (listener) => {
      listeners.add(listener)
      return () => listeners.delete(listener)
    },
  }
}
```

```typescript
// AppState.tsx
export function useAppState(selector) {
  const store = useAppStore()
  const get = () => selector(store.getState())
  return useSyncExternalStore(store.subscribe, get, get)
}
```

`useSyncExternalStore` 的工作方式：
1. 调用 `store.subscribe` 注册监听
2. 每次 store 通知变化时，重新调用 `get`（即 `selector(store.getState())`）
3. 用 `Object.is` 比较新旧 selector 结果
4. 只有结果变化时才触发组件重渲染

## 这一章覆盖什么

`11-03` 系列拆解这套订阅机制的每个组成部分：

**AppStateProvider 的克制**：provider 本身很薄，只建立 store 和接入 settings，复杂度都被压进 store 和 selector 里

**selector 作为性能纪律**：不只是 API，而是「你必须声明自己依赖哪一小片状态」的约束

**useSetAppState 的分离**：只需要写状态的组件不应该顺带订阅状态

**Store 的 50 行实现**：createStore 里 Object.is 比较、Set 监听者、onChange 回调的组合

**useSyncExternalStore 的选择**：为什么这个 React 18 API 正好适合 Claude Code 的场景

**安全读取版本**：`useAppStateMaybeOutsideOfProvider` 处理「可能在 Provider 外」的场景

**纯 selector**：selectors.ts 里派生逻辑和副作用的分离原则

**settings 的统一入口**：配置变化怎样经过单一入口进入 store

## 面试指导

这一章有一道很好的面试题可以贯穿始终：

**题目：一个大型应用有 100 个字段的全局状态，10 个组件各自订阅其中几个字段。如果不用 selector，每次状态更新会触发多少次重渲染？用了 selector 之后呢？**

不用 selector（useContext）：每次任何字段变化，10 个组件全部重渲染。如果每秒 10 次状态变化，就是每秒 100 次重渲染。

用 selector（useSyncExternalStore + Object.is）：每次状态变化，只有「订阅了变化字段」的组件重渲染。如果 10 个组件各订阅 5 个字段，1 次状态变化平均只影响 0.5 个组件（假设字段变化分布均匀）。

这道题的答案引出了 selector 和 `Object.is` 比较的重要性，也引出了「selector 不能返回新对象」的约束（因为 `Object.is({}, {})` 永远是 `false`）。
