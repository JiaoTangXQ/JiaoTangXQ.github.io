# 订阅纪律导读：唯一现实要靠选择器和统一入口守住

如果说 `11-02` 先把现实钉成同一个世界，`11-03` 讲的就是这个世界怎样不被读写方式弄散。这里真正重要的，不是某个 React API，而是唯一现实、稳定身份、订阅纪律和统一写入口怎样一起守住长期可维护性。

所以这一章的十片叶子，不该拆成零散性能技巧。先看 [11-03-01-AppStateProvider-故意做薄，说明复杂度被压进了状态系统.md](</Users/xxx/tmp/cc/docs/claude-code-book/11-终端界面怎样成为运行时/11-03-useAppState(selector) 为什么是性能和可维护性的关键/11-03-01-AppStateProvider-故意做薄，说明复杂度被压进了状态系统.md>)，再顺着 selector 订阅、唯一 store、外部现实订阅、安全读取版本、纯派生 selector 和统一设置入口往下读，这一章才会从“状态管理细节”变成“事实世界怎样在长期会话里保持唯一和稳定”的纪律层。

## 实现链

这一组主要落在 `AppState.tsx` 和 `selectors.ts`：provider 保持很薄，store 只创建一次，读通过 `useAppState(selector)`，写通过 `useSetAppState()`，配置变化也经统一入口进来。唯一现实不是靠口号守住，而是靠订阅纪律和统一写入口守住。

## 普通做法

更普通的 React 应用会同时混用 context 值、局部 state、派生对象和散落的 effect 监听，能跑就行。

## 为什么不用

Claude Code 不愿意这么随手混，是因为它的状态来源里本来就有很多非 React 世界的输入。越是外部现实多，越得先把“怎么订阅”和“从哪里写入”讲清楚。

## 代价

代价是 hooks 约束更强，组件作者不能想怎么取就怎么取；但换来的是整台应用更不容易被无序订阅拖垮。
