# 面试题：画出 Claude Code 的主流程图，然后说出它最大的谎言是什么

## 主流程图长什么样

如果让候选人快速画一张 Claude Code 的运行流程图，大多数人会画成这样：

```
启动
  └→ 初始化（配置、认证、MCP 连接）
       └→ 装配上下文（系统提示、用户上下文、Git 状态）
            └→ 主循环
                 ├→ 接收用户输入
                 ├→ 调用模型
                 ├→ 模型返回工具调用 → 执行工具 → 追加结果
                 └→ 模型返回文本 → 显示给用户 → 等待下一轮
```

这张图描述的都是真实发生的事，没有错误内容。问题在于它的**遮挡**。

## 主流程图遮住了什么

一个标准追问：在主循环里，为什么有些工具调用会被自动允许，有些需要弹权限框，有些直接被拒绝，还有些被预先从工具列表里去掉了？

主流程图里找不到答案，因为这些决策发生在"调用模型"和"执行工具"这两个方框的**外面**。

`toolPermissionContext` 是一个存在于 `AppState` 里的长期状态对象，包含 `mode`（default/auto/plan 等）、`alwaysAllowRules`、`alwaysDenyRules`、`alwaysAskRules`、`additionalWorkingDirectories`。它在主循环开始之前就被初始化，在整个会话里持续被更新（比如用户批准了某个工具的"记住这个决定"时），影响每一次工具调用的判断。

这个权限上下文不在主流程图的任何一个方框里。但它决定了主流程图里那个"执行工具"方框能不能真正执行。

## 制度层的四条线

制度层不是主流程之外的补充，而是主流程能在现实里运作的前提。它有四条主要的线：

**权限与设置线**：`permissionSetup` 从多个来源（settings 文件、企业 MDM 策略、命令行参数、环境变量）加载规则，合成一个 `toolPermissionContext`。在自动模式（`--auto-approve`）下，系统还会主动检查并剥除其中的危险宽松规则，防止"一个宽泛的允许规则意外地放行了不该被放行的操作"。

**Hooks 线**：在工具调用前后（`PreToolUse`、`PostToolUse`）、停止前后（`Stop`）、会话开始结束（`SessionStart`、`SessionStop`）有 hook 触发点。hooks 的定义有完整的 schema（在 `schemas/hooks.ts` 里），支持 `command` 类型（执行 shell 命令）和 `prompt` 类型（让 LLM 评估）。Hook 不是简单的"跑一个脚本"，它有 `if` 条件（用权限规则语法过滤触发条件）、`timeout`、`async/asyncRewake`（后台运行，可在 exit code 2 时唤醒模型）、`once`（只触发一次然后自动移除）。

**连续性线**：会话结束不等于记忆消失。`microcompact`（轻量压缩）、`autocompact`（会话快撑不住时触发的深度压缩）、`session memory`（当前工作的外置笔记）、`auto-memory`（跨会话的长期记忆沉淀）四个机制分别处理不同时间尺度上的上下文保持问题。它们不是"记忆功能"，而是回答"下一轮开始时，这一轮的什么东西应该还在"。

**条件能力线**：bridge 模式、remote 模式、assistant 模式会改变整个会话的工作规则——命令集被过滤成 remote-safe 子集、权限请求从远端翻回本地、消息路由改变。这不是功能开关，而是"工作现场的切换"，系统规则随之整体迁移。

## 为什么制度层和主流程地位相等

一个成熟 agent 系统和演示系统的分水岭，不在于能不能调工具，而在于：

- 工具被调之前，是否有明确的授权依据？
- 工具调完之后，上下文是否被合理维护？
- 外部规则（企业政策、团队标准）能否以可审计的方式接入？
- 工作现场切换时，规则是否一致地跟着变？

这四个问题，主流程图一个都回答不了。制度层才能回答。

## 面试指导

**考官的真实意图**：不是让你画流程图，而是看你有没有意识到"推进顺序"和"边界约束"是两种不同的系统关切。

**满分策略**：画出主流程图（展示你理解正常执行路径），然后主动说"但这张图遗漏了制度层"，举出一到两个具体例子（比如权限上下文如何改变工具执行、hooks 如何在工具边界插入外部规则），解释为什么这些不是附属功能而是主结构。

**常见误区**：把制度层理解为"错误处理"或"安全功能"。错误处理关心的是主流程失败后怎么恢复，制度层关心的是主流程推进的前提条件是否满足——这是两件不同的事情。
