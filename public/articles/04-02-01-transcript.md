# 为什么用户消息被接受后要先写进 transcript

很多系统在持久化这件事上都有一种本能：等模型成功回了话、这一轮看起来“完成了”，再把内容记下来。这个直觉对简单聊天没太大问题，但对一个要支持恢复、继续和长会话的 agent 来说，已经太晚了。

`QueryEngine.submitMessage()` 里有一段很值得注意的处理。用户输入经过 `processUserInput()` 整理完之后，系统会先把这些新消息写进 transcript，然后才进入真正的 `query()` 主循环。代码里的注释把原因说得很直接：如果用户消息已经被接受，但进 API 之前进程就被杀掉，transcript 里只剩排队痕迹，没有真正的用户消息，`--resume` 会找不到可恢复的会话。

这背后的想法很成熟。Claude Code 把“用户消息已经被系统接受”视为一个足够重要的事实，它值得被立刻持久化，而不是等幸福路径全部走通以后再补记。换句话说，这套系统真正保护的不是一次完美回合，而是一次真实发生过的会话事件。只要这一点成立，后面的恢复、重放、继续和调试才有根。

这也解释了它为什么在不同模式下用了不同策略。交互式会话里，恢复价值很高，所以这里宁可承受一点点写盘成本；`--bare` 这类脚本场景里，它又改成 fire-and-forget，避免把关键路径拖慢。优雅之处不只是“先写再说”，而是它知道什么时候必须认真写，什么时候可以把代价压低。

代码里的做法非常直接：`submitMessage()` 先调用 `processUserInput()`，把用户消息、附件和可能的系统消息都放进 `mutableMessages`，然后在进入 `for await (const message of query(...))` 之前就执行 `recordTranscript(messages)`。注释里把原因写死了：如果 API 还没回，进程就被杀掉，transcript 里只剩排队痕迹，`--resume` 会认为这场会话根本没正式开始。更常见的实现，是等模型先回第一段内容，或者整轮结束后再统一落盘，因为那样磁盘 IO 更少，主路径也更短。

Claude Code 不用那条路，是因为它把“用户这句话已经被系统接受”视为一个必须可恢复的事实，而不是“等模型有回应之后才算发生”。这对长会话、恢复和调试都更稳，但代价是每轮都会把几毫秒到几十毫秒的写盘成本放到关键路径里，所以它才对 `--bare` 这种脚本场景改成 fire-and-forget，不把同一种坚持硬套到所有模式上。

## 实现链
`submitMessage()` 在进入 `query()` 前先 `recordTranscript(messages)`，脚本模式才改成 fire-and-forget。

## 普通做法
等模型回完或整轮结束再统一持久化。

## 为什么不用
因为用户消息一旦被系统接受，就必须可被 `/resume` 找回。

## 代价
关键路径上增加了磁盘 IO。
