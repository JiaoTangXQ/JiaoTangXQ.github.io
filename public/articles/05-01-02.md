# 任务把后台能力先翻译成可管理劳动

后台能力本身不稀奇，难的是把后台能力变成产品里可被看见、可被接管、可被通知的一段劳动。

## 实现链

`LocalAgentTask` 在创建时不会只保存一个 `prompt`。它还会写入 `description`、`outputFile`、`notified`、`progress`、`pendingMessages`、`retain`、`evictAfter` 等字段。`framework.ts` 会给新任务发 `task_started` 事件，完成时再由 `enqueueAgentNotification()` 把结果重新投回主消息队列。

这说明 Claude Code 理解“后台任务”的方式不是“静静跑在旁边”，而是“被系统正式托管的一段劳动”。

## 普通做法

普通做法一般只做两件事：一是把任务丢到后台线程或子进程，二是等它结束时拿结果。中间如果要展示一点进度，就再加个 spinner，或者轮询一下日志。

这种方式重的是执行，轻的是治理。

## 为什么不用

Claude Code 不满足于“后台能跑”。它要的是后台任务仍然属于同一个工作台，仍然能被主会话追踪，仍然有机会切回前台或把结果纳入下一轮推理。为此，后台能力必须先被翻译成任务对象，而不是留在执行器私有内存里。

小白可以把这理解成：不是把活塞进仓库就算管理，而是要让仓库系统里真的有这张工单。

## 代价

代价是后台功能会连带牵出很多配套机制。你会看到输出文件、通知 XML、任务回收、面板保留、消息排队这些看似分散的东西全跟上来了。系统明显更重，但也明显更像一个能长期运行的工作台。
