# 设计题：同一进程里的多个 AI Agent，通信该用共享内存还是消息传递？

这是一道经典系统设计题的 AI Agent 版本。

传统答案是：同进程用共享内存，跨进程用消息传递。共享内存快，消息传递有边界。

Claude Code 的答案不同：**哪怕同进程，也要用消息传递**。

## 源码里的强制规定

`teammatePromptAddendum.ts` 把这条规则直接写进了 teammate 的系统提示：

> To communicate with teammates, you MUST use the SendMessage tool. Simply outputting text is NOT visible to teammates.

这不是建议，是约束。teammate 如果直接输出文字，别的成员根本看不到。

为什么输出文字不可见？因为 teammate 的输出最终是给 leader 界面看的，不是自动广播给所有成员。真正能被团队成员读取的只有 `teammateMailbox.ts` 里的收件箱文件。

## 共享内存方案的问题

先认真考虑共享内存方案是什么样的：

在同一个 Node.js 进程里跑 5 个 teammate，它们共享一块内存区域——比如一个全局的 `sharedContext` 对象。任何 teammate 可以往里写，任何 teammate 可以从里读。

这看起来很方便，但实现下去会遇到几个问题：

**上下文污染**。Researcher 在探索一个方向，把中间结果写进 sharedContext。Code-reviewer 因为也监听这块内存，把这些不成熟的探索当成正式结论，基于此给出了错误的审查意见。这在多人协作里叫"过早共享"，在 AI Agent 里一样会发生。

**边界失控**。你怎么知道哪些信息应该共享？Agent 在思考时会产生大量中间状态，全部写入共享内存会迅速产生噪音；只写部分，又需要 agent 自己判断哪些值得分享——这个判断本身是有成本的。

**调试困难**。当输出结果错误时，无法追溯"是谁写了这条数据、什么时候写的"。共享内存没有消息的概念，没有发送者、没有时间戳、没有"已读"状态。

**跨后端不一致**。同进程 teammate 有共享内存，但 tmux pane teammate 不在同一进程。如果你为同进程 teammate 设计了基于共享内存的通信协议，tmux teammate 就需要另一套协议——团队系统因此分裂成两种实现。

## SendMessage 的真正成本

"强制用 SendMessage"听起来像是人为制造门槛。但这个"门槛"是刻意设计的。

每次 teammate 调用 `SendMessage`，它都在做一个显式判断：**我现在有什么值得告诉其他人？** 这个判断本身就是有价值的过滤器。

如果通信是零成本的（共享内存），agent 就没有理由过滤——把所有想法都丢出去，让接收方自己筛选。这会产生团队噪音。

如果通信需要显式发送一条消息，agent 就自然会思考：这条信息完整吗？现在发合适吗？对方需要吗？这种隐性过滤让团队通信信噪比更高。

`teammateMailbox.ts` 里每条消息都有 `summary` 字段（5-10 词的预览），这说明系统假设消息是有重量的——重到需要一句摘要来帮接收方决定是否要展开读。如果通信是零成本的，根本不需要摘要。

## 跨后端一致性的工程价值

从架构角度看，强制使用 `SendMessage` 还带来了另一个好处：同进程 teammate 和 tmux teammate 在通信协议上完全一致。

无论你是在哪种后端运行，发消息就是调用 `writeToMailbox()`，读消息就是轮询 mailbox 文件。这使得：

- 代码不需要区分"这是同进程通信还是跨进程通信"
- 测试只需要测一套通信逻辑
- 混合部署（有些 teammate 是 in-process，有些是 tmux pane）自然工作，无需特殊处理

这是一个用"略低的性能"换来"大幅降低的复杂度"的典型取舍。Node.js 进程内写文件再读文件，确实比共享内存慢；但比起维护两套通信协议的工程成本，这个性能损耗几乎不值一提。

## 面试怎么答

如果面试题是"同进程 agent 通信用什么方案"：

**表面答案**：共享内存更快，消息传递有边界。视场景选择。

**深入答案**：取决于你更怕什么。如果更怕性能瓶颈，选共享内存。如果更怕调试困难和跨后端不一致，选消息传递。Claude Code 选了消息传递，因为它的团队系统同时支持同进程和跨进程，统一的通信协议比几毫秒的延迟优化更重要。

**加分项**：消息传递的真正价值不在于"能通信"，而在于**它创造了一个边界，让发送方必须主动决定要共享什么**。这个边界在多 agent 系统中是控制信息熵的关键机制。没有这个边界，agent 要么什么都共享（噪音），要么靠约定决定共享什么（隐性规则，难以维护）。
