# 面试题：状态栏底部有哪几个信息层级？modePart、tasksPart 和 parts 是怎么协同工作的？

这道题要求你能从信息架构的角度分析底部区域的实现，而不只是描述它"显示了什么"。

## 三个层级的职责

**modePart**（最高优先级）

Permission mode 的视觉指示器。它独占一个位置，排在所有其他内容之前。

为什么是最高优先级？因为 permission mode 是操作安全边界的声明，是用户在执行任何操作前最需要知道的一件事。它放在最前面，意味着在宽度最紧的情况下它是最后被截断的。

```typescript
const modePart = !getIsRemoteMode() ? buildModePart(permissionMode) : null;
```

条件性生成——远端模式下为 null（前面分析过的诚实性原则）。

**tasksPart**（中等优先级）

任务相关的提示区：`esc`、`stop agents`、`show tasks`、teammate pills 等。这些是"当前工作状态"的操作入口，使用频率高，但比 mode 低一级。

```typescript
const tasksPart = buildTasksPart({
  hasTeams, hasBackgroundTasks, isViewingTeammate, hasTeammatePills,
  isLoading, columns, ...
});
```

这是条件构建的——根据团队状态、加载状态、屏幕宽度等动态决定内容。

**parts**（补充信息）

远端链接、PR 状态、其他补充信息。这些是"有了更好，没有也行"的附加上下文。

```typescript
const parts: (React.ReactNode | null)[] = [
  remotePart,   // 远端会话链接（有远端时才有）
  prPart,       // PR 状态（满足条件时才有）
  // ...
];
```

所有 null 项会被过滤掉，只渲染实际有内容的部分。

## 拼接和截断策略

这三个层级不是简单地从左到右拼接，而是有更精细的布局：

```typescript
// 大概的渲染结构
<Box flexDirection="row">
  {modePart}          {/* 始终在最前 */}
  {tasksPart}         {/* 任务相关，排第二 */}
  <Text wrap="truncate">
    {parts.join('  ')} {/* 补充信息，可被截断 */}
  </Text>
</Box>
```

`wrap="truncate"` 只应用于 parts，不应用于 modePart 和 tasksPart——这保证了最重要的信息不会被截断，截断只发生在补充信息上。

**Byline 组件的作用**

某些情况下，底部会用 `Byline` 组件来处理单行显示：

```typescript
// 大概逻辑
<Box>
  {modePart && <Byline>{modePart}</Byline>}
  {tasksPart}
  {parts.filter(Boolean).map((p, i) => (
    <React.Fragment key={i}>{p}</React.Fragment>
  ))}
</Box>
```

`Byline` 是一个 helper 组件，确保其包含内容在一行内显示，不会因为内容太多而换行。

## 信息的"明面"和"背面"

这套三层结构实际上实现了信息的分层展示：

- **立即可见（明面）**：modePart（始终可见），tasksPart（通常可见）
- **条件可见（背面）**：parts 里的内容，根据空间和条件决定是否显示

当底部空间不足时，截断顺序是从右到左：parts 先被截断，然后是 tasksPart 的部分内容，modePart 最后被保护。

这不是"隐藏信息"，而是"按重要性决定谁先被看见"。那些在窄屏下消失的内容，在宽屏下还会出现。信息没有消失，只是在当前约束下无法显示。

## 为什么要三层而不是一层

**可维护性**

三个层级有各自独立的构建逻辑（`buildModePart`、`buildTasksPart` 等），修改某一层的逻辑不需要理解其他层。如果所有内容都在一个大的字符串拼接里，改动任何一部分都需要理解整体结构。

**可测试性**

每个 buildXxxPart 函数可以独立测试，不依赖完整的底部渲染上下文。

**扩展性**

要加一种新的补充信息？往 parts 数组里加一项。要增加新的操作提示？修改 `buildTasksPart`。不需要重构整个底部渲染逻辑。

**信息优先级的编码**

把信息优先级直接编码进数据结构（层级），而不是隐含在渲染顺序里。这让"modePart 最重要"这个决策变得显式可读，而不是隐藏在"它恰好写在最前面"这个事实里。

## 面试指导

**结构性问法**："底部区域的信息是怎么组织的？"

答：三个层级：modePart（permission mode，最高优先级），tasksPart（任务和操作提示，中等优先级），parts（补充信息如远端链接和 PR 状态，可被截断）。modePart 和 tasksPart 不参与截断，只有 parts 在空间不足时会被截断。

**追问**："为什么不用一个数组装所有内容？"

答：三个层级有不同的优先级、不同的截断规则、不同的条件逻辑。分层使每层的逻辑清晰，可独立修改和测试。一个数组的做法会把这些不同的关注点混在一起，难以维护。

**系统追问**："如果要给底部加一个'当前 git branch'的显示，你会放在哪一层？"

答：放在 parts 里（补充信息层）。git branch 是有用的上下文，但不是操作安全边界（不像 permission mode），也不是当前工作流的操作入口（不像任务提示）。在空间充足时显示，空间不足时是可以被截断的。同时，它应该加一个条件检查：只有在 git 仓库里才显示，类似 PR 状态的条件检查。

**设计推演**："这套三层结构能处理多少条目的补充信息？有没有上限？"

答：parts 数组没有硬编码上限，但 `wrap="truncate"` 的截断行为意味着超出终端宽度的内容会被切掉。如果 parts 变得很多，用户在窄屏下可能始终看不到后面的内容。所以在往 parts 里加内容时，需要审慎地考虑是否真的有必要，以及在宽度有限时用户会不会错过重要信息。

---

*核心考察点：信息分层架构设计、截断策略的优先级编码、三层结构的可维护性和扩展性。*
