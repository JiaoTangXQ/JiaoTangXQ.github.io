# brief 过滤后仍要保留真实用户输入和必要系统提示，说明删掉的是冗余文本不是世界

`brief` 模式最容易做错的一点, 就是把“压缩文字”误做成“压缩世界”。Claude Code 没这么干。它删的是重复助手文本, 但真实用户输入、必要系统反馈和真正承担解释职责的工具结果仍然得留着。

## 实现链

`filterForBriefTool()` 的代码写得很直白。它会保留 `system` 消息, 只排除 `api_metrics` 这种调试噪声; 对 `assistant` 只保留 Brief 工具的 `tool_use`; 对 `user` 则保留对应 `tool_result` 和真实用户输入, 明确丢掉 `isMeta` 的伪用户消息; 对 `attachment` 只保留 `queued_command` 且 `commandMode === 'prompt'` 的人类输入。这套规则说明 brief 过滤不是把整轮洗成只剩工具壳, 而是在保住“这一轮到底发生了什么”所必需的世界信息。

## 普通做法

更普通的做法要么是保留所有文本, 结果 brief 失去意义; 要么是除了工具调用什么都砍, 让用户只看到一堆动作残骸。

## 为什么不用

Claude Code 不选后者, 因为人理解一轮会话靠的不只是工具名, 还靠用户原话、系统提示和必要的环境反馈。要是把这些一并删掉, 屏幕会更短, 但世界也被删空了。它不选前者, 是因为那又回到原始冗余状态。现在这套规则是在“少说”和“别失真”之间找平衡。

## 代价

代价是过滤逻辑需要知道很多上下文细节: 什么算真实用户输入, 什么是 meta, 哪些 attachment 其实代表人类命令, 哪些 system message 不能删。规则一多, 维护成本就会高于那种“一刀切”的简化实现。
