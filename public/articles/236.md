# 检测和装配是接入链的一部分，不是安装后的附带步骤

面试题版本：**"Claude Code 如何检测浏览器扩展是否正确安装并可用？如果扩展安装了但无法连接，系统是怎么诊断和处理的？"**

## 安装 ≠ 可用

一个典型的安装问题清单：

1. 扩展安装了，但 Native Host 没安装
2. Native Host 安装了，但路径写错了（指向不存在的可执行文件）
3. manifest 文件存在，但 `allowed_origins` 不包含当前扩展的 ID
4. 所有文件都对，但浏览器需要重启才能识别新的 Native Host 配置
5. 所有配置正确，但用户的安全软件阻止了 Native Messaging 连接

每一种情况的症状可能相同（"扩展连接不上"），但解决方法完全不同。

## 检测的实现

`src/bridge/bridgeEnabled.ts` 和 `bridgeDebug.ts` 负责检测 bridge 的各个组件是否正确：

**检测 Native Host 可执行文件**：
- 读取 manifest 文件
- 检查 manifest 里的 `path` 字段是否指向实际存在的可执行文件
- 检查该文件是否有执行权限

**检测 manifest 格式**：
- 验证 manifest JSON 是否格式正确
- 验证 `allowed_origins` 是否包含当前使用的扩展 ID
- 验证 `type` 是否为 `stdio`

**检测连接可用性**：
- 尝试建立一次 Native Messaging 连接
- 检查是否能收到正确的握手响应
- 记录连接成功还是超时

这三级检测让系统知道："是安装问题、还是配置问题、还是运行时连接问题"，而不是一个模糊的"出了什么问题"。

## 缓存检测的逻辑

Claude Code 不是每次都重新做完整检测（开销太大），而是缓存检测结果。但缓存的结果可能变得过时（用户重装了浏览器、Native Host 被更新了）。

缓存失效的判断：
- Native Host 可执行文件的修改时间变了（被更新）
- manifest 文件被修改（配置变了）
- 扩展版本变了（扩展更新可能改变 extension ID）
- 上次检测失败后过了足够长时间（可能是临时问题）

`bridgeEnabled.ts` 维护检测结果的缓存，并在上述情况下让缓存失效，触发重新检测。

## 装配确认

检测通过后，连接建立时还需要"装配确认"——验证双方（Claude Code 和浏览器扩展）理解的协议版本、能力集合是否一致：

```typescript
// 连接建立后的握手
const handshake = {
  version: BRIDGE_PROTOCOL_VERSION,
  capabilities: getSupportedCapabilities(),
}
await bridge.send({ type: 'hello', data: handshake })
const response = await bridge.receive({ type: 'hello_response', timeout: 5000 })
// 检查版本兼容性
if (!isCompatible(handshake.version, response.version)) {
  throw new BridgeVersionMismatchError(...)
}
```

这个握手步骤确认了"连接成功"不只是物理层面的连接，还包括协议层面的兼容——双方能理解对方说什么。

## 为什么这些步骤不能省

如果把检测和装配省掉，假设安装成功就代表可用：

**调试噩梦**：连接失败时，用户和开发者都不知道是哪一步出了问题。是 manifest 的问题？是可执行文件路径的问题？是协议版本的问题？没有检测数据，无从判断。

**静默失败**：扩展无法连接，用户可能不知道是 bridge 出了问题，以为是 Claude Code 功能本身的问题，花时间在错误的方向调试。

**难以恢复**：不知道哪里出了问题，就不知道怎么恢复。检测数据提供了"下一步应该检查什么"的明确指引。

## 面试角度的总结

检测和装配是接入链的组成部分，不是安装后的可选验证。它们把"能用"从一个祈祷的状态变成一个可验证的状态：每个组件（manifest、可执行文件、连接、协议）都有明确的健康检查，任何一个组件出问题，系统可以快速定位并给出有意义的错误信息。这是产品质量的体现：不只是在正常路径上工作，也在失败路径上提供良好的诊断和引导。
