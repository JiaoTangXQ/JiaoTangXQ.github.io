# 第十三卷回收索引：外延执行里检测与装配也属于接入链，笨重不是多余而是替真实用户把现场接稳的代价

## 实现链
外延执行里另一半更实际的难点，不在“能不能接进来”，  
而在检测、装配和重连这些动作能不能一起把现场接稳。

这条线真正逼出来的，是第十三卷里关于工程现实的另一种判断。  
笨重不是多余，而是系统替真实用户把现场接稳的代价。

这页实际回收的实现链是：
- `12-06-01` 也把检测和装配写成接入链的一部分，这会继续支撑 `13-06-01`，说明笨重不是多余，而是系统替真实用户把现场接稳的代价。

真正成熟的产品接入，不会只在“成功点亮”时设计。  
它也会把检测、装配和把现场接稳的代价一起算进去。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
