# Claude Code 的 bridge 层为什么知道什么不能被带上路？

面试官可能这样问："你们系统里有远端执行能力，为什么不做透明代理，让本地和远端看起来一模一样？"

答对这个问题，需要先把一件事说清楚：**Claude Code 的 bridge 层不是被动的连接件，而是主动的边界守卫。**

## 透明代理的诱惑

最直觉的做法是：远端环境和本地环境尽量长一样，用户感觉不到边界，工程上只是"把连接打通"。这样上手快，心智模型简单。

但 Claude Code 没这么做。原因写在 `bridge/types.ts` 里，一读就清楚。

## BridgeConfig 承认了边界的存在

`bridge/types.ts` 里定义的 `BridgeConfig` 有这几个字段特别值得注意：

```typescript
export type BridgeConfig = {
  spawnMode: SpawnMode  // 'single-session' | 'worktree' | 'same-dir'
  reuseEnvironmentId?: string  // 重连时复用后端 environment
  sessionIngressUrl: string    // 可能与 apiBaseUrl 不同
  sessionTimeoutMs?: number    // 每 session 独立超时
  sandbox: boolean
}
```

`spawnMode` 三种值本身就是在说：每次远端连接不一定是同一种模型。`worktree` 模式每个 session 独立 git worktree，`same-dir` 则共享目录（注释里直接写了 "can stomp each other"）。桥接层不假装这些差异不存在，它在配置里就明确分类了。

## 权限不能透传

`bridge/bridgePermissionCallbacks.ts` 里定义了权限交互的完整类型：

```typescript
type BridgePermissionCallbacks = {
  sendRequest(requestId, toolName, input, toolUseId, description, ...): void
  sendResponse(requestId, response: BridgePermissionResponse): void
  cancelRequest(requestId): void
  onResponse(requestId, handler): () => void
}
```

这里有一个 `cancelRequest`，大多数简单远端系统不会有这个。为什么要 cancel？因为当 bridge 连接出问题，或者 session 提前结束，web app 那边已经弹出了权限提示框，必须有人去收回它。如果做透明代理，这种生命周期不匹配的情况根本没有处理位置。

`isBridgePermissionResponse` 是 type predicate 而不是 `as` 强转，说明这个响应是外部传来的不可信数据，必须在边界显式验证。这一个细节说明了：**边界不只是物理连接的两端，更是信任断层的位置。**

## SessionHandle 为什么那么重

`bridge/types.ts` 中的 `SessionHandle` 值得逐字读：

```typescript
export type SessionHandle = {
  sessionId: string
  done: Promise<SessionDoneStatus>
  kill(): void
  forceKill(): void
  activities: SessionActivity[]
  currentActivity: SessionActivity | null
  accessToken: string
  lastStderr: string[]
  writeStdin(data: string): void
  updateAccessToken(token: string): void  // token 刷新后也要更新
}
```

`forceKill` 和 `kill` 分开，因为远端 session 的正常退出和强制清除是两件事。`updateAccessToken` 说明 token 会在 session 运行中刷新，连接不能依赖初始化时的静态凭据。`lastStderr` 是 ring buffer，在 session 失败诊断时可以看到最后几行错误输出。

这些字段每一个都在回答一个问题：当远端 session 和本地主会话之间发生任何状态不同步时，谁来承担？答案是 `SessionHandle` 承担，而不是让两端各自猜。

## env-less 是另一个层级的决策

`bridge/envLessBridgeConfig.ts` 定义的 `EnvLessBridgeConfig` 揭示了一个更深的设计决策。它有单独的版本检查函数 `checkEnvLessBridgeMinVersion`，有独立于 v1 的 `min_version` 配置字段，说明 v1（env-based）和 v2（env-less）是两套独立的接入路径，可以独立推版本。

注释里直接写："v2 analogue of checkBridgeMinVersion() — reads from tengu_bridge_repl_v2_config instead of tengu_bridge_min_version so the two implementations can enforce independent floors."

两套路径有不同的 token 刷新策略（`token_refresh_buffer_ms`，有详细的 floor/cap 注释说明为什么 30s 是 floor，30min 是 cap），有不同的重连超时（`connect_timeout_ms` p99 注释写了观测值是 2-3s，给了 5× 余量）。

这不是在追求整洁，而是在承认：**env-less 路径真的比 env-based 路径有不同的失败模式**，把它们强行统一只会让两套错误处理都变残缺。

## 面试答题策略

被问到"为什么不做透明远端"时，回答的核心是：

1. **边界不是工程的妥协，是设计的一等公民**。权限谁裁、token 怎么刷、session 怎么 kill，都是边界问题，不是实现细节。

2. **透明代理的代价是隐性的**。当远端和本地状态不同步时，透明代理没有处理位置，错误会在最难收拾的时刻爆出来。

3. **桥接层厚，是因为边界上需要处理的事情本来就多**。不是设计失败，是工程诚实。

追问"你们的 bridge 怎么处理重连"时，可以说：`BridgeConfig.reuseEnvironmentId` 字段让重连时可以复用后端已有的 environment，而不是每次都新建。`reconnectSession` API 在原来的 bridge 进程死掉后，还能把 session 重新排队到 environment 上。这不是附加功能，而是从一开始就设计进 `BridgeApiClient` 接口的能力。
