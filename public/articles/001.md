# Claude Code 启动的第一行代码做了什么？

## 面试官：你看过 Claude Code 的 main.tsx 吗，说说它做的第一件事

大多数人会说"导入依赖"。但实际上，`main.tsx` 在任何 import 语句生效之前，就已经触发了三个并行的副作用：

```
profileCheckpoint('main_tsx_entry')   // 计时桩
startMdmRawRead()                      // 后台读取 MDM/企业策略
startKeychainPrefetch()                // 后台读取 keychain 凭证
```

这三行夹在 `import` 语句之间，而且每一行上面都有 `// eslint-disable-next-line custom-rules/no-top-level-side-effects` 注释——说明团队知道这是例外，刻意保留了它。

## 为什么要在模块顶层触发副作用？

Node 的模块加载本身是串行的，import 链走完需要时间。Claude Code 的做法是：在模块开始加载的同时，把那些"必须等待的慢操作"提前发射出去。

`startMdmRawRead()` 会触发平台级的 MDM 策略读取——macOS 上是 `plutil`，Windows 上是 `reg query`。这些都是子进程调用，自身需要数十毫秒。如果等 import 全部完成再启动，用户会多等那么多时间。

`startKeychainPrefetch()` 更明显。注释里说，macOS 的 keychain 需要两次 sync spawn 调用（OAuth token + 旧版 API key），加起来约 65ms。如果等 `init()` 时才读，就是串行 65ms；把它提前到模块顶层，这 65ms 就和 135ms 的 import 链并行运行，实际感知延迟可能只加几毫秒。

## 这说明了一种架构思维

Claude Code 的启动优化是显式的、有注释的、被 lint 规则跟踪的。团队并不是随手加了个异步调用，而是清楚地知道"这里要发射什么，等待什么，代价是什么"。

用 ASCII 图来看这条启动时间线：

```
t=0  main.tsx 开始解析
     ├── profileCheckpoint() 打桩
     ├── startMdmRawRead() → 子进程启动（~50ms）
     └── startKeychainPrefetch() → keychain 读取（~65ms）
     
t=~135ms  import 链加载完毕
           init() 开始执行
           └── ensureKeychainPrefetchCompleted() ← 等待之前的 prefetch 结果
```

如果没有 prefetch，init() 到这里要同步等 65ms；有了 prefetch，这步通常是 0 等待，因为并行的 65ms 已经消化掉了。

## 面试官追问：这样写有什么风险？

模块顶层副作用有一个明显代价：顺序变得不可控。`startMdmRawRead()` 在 `bootstrap/state.js` 完全初始化之前就运行了。如果状态模块还没加载完，读 MDM 时内部用到的某个函数就可能报错。

Claude Code 的处理方式是：这三个函数设计成不依赖其他未初始化的模块，且它们只是"发射"，不是"使用结果"。结果的消费被延迟到 `init()` 里面的显式等待点。

这是一种精确控制的"先射箭再画靶"策略：箭已经在路上，等到需要结果时再接住它。

## 面试指导

被问到 Claude Code 启动性能时，不要只说"异步"。具体说：三个顶层副作用、并行化 MDM 和 keychain 读取、`profileCheckpoint` 打桩用于测量、用 `ensureKeychainPrefetchCompleted()` 在需要点等待结果。这套设计的核心是把"必须等的慢操作"从关键路径里挤出去，让它在 import 链加载期间完成。
