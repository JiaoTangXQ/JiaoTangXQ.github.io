# 模式、目录和连接方式如何决定这次 Claude 是谁

面试题版本：**"同一个 Claude Code 工具，在不同情况下运行时行为有所不同：本地 REPL 模式、远端 API 模式、IDE 插件模式。这些模式的差别体现在哪些地方？为什么需要这种区分？"**

## "这次 Claude 是谁"的含义

这不是身份认证的问题（Claude 始终是同一个 AI），而是**能力边界的问题**：在这个会话里，Claude 被授权做哪些事？

不同的运行上下文赋予了不同的能力边界：

**本地 REPL**（用户在终端里直接运行）：
- 可以读写当前工作目录的文件
- 可以执行 bash 命令
- 可以使用所有配置的工具
- 高信任度：用户直接控制，在自己的机器上

**远端 API 模式**（被其他程序调用）：
- 只能使用 `REMOTE_SAFE_COMMANDS` 里的命令
- 某些可能有副作用的工具被禁用
- 低信任度：调用者可能是自动化脚本，不是直接的用户操作

**IDE 插件/bridge 模式**：
- 受限于 `BRIDGE_SAFE_COMMANDS`
- 操作范围可能限制在当前打开的项目
- 中等信任度：IDE 里的用户操作，有 UI 反馈

## PermissionMode 的实现

`src/types/permissions.ts` 里的 `PermissionMode` 枚举定义了主要模式：

```typescript
export type PermissionMode = 
  | 'default'      // 本地 REPL，用户直接使用
  | 'acceptEdits'  // 自动接受编辑建议
  | 'bypassPermissions'  // 跳过某些权限检查（测试/CI）
  | 'plan'         // 只规划，不执行
```

加上连接来源（local/remote/bridge）和工作目录（项目根目录），这三个维度共同决定了工具权限的实际范围。

## 工作目录的安全边界

工作目录是 Claude Code 最重要的隐式边界。`file_edit` 工具通常只允许修改工作目录内的文件——不管用户是否明确设置了这个限制，这是一个合理的默认安全假设：

- 用户在 `/home/user/projects/myapp` 里启动 Claude Code
- Claude 的文件操作应该限制在这个目录内
- 修改 `/home/user/other-project/` 里的文件不应该被允许，除非用户明确授权

这个约束通过检查文件路径是否在工作目录的子路径里来实现，不需要每次都询问用户。

## 连接方式改变信任假设

**REPL 连接**：用户直接在终端里输入，所有操作都经过用户的眼睛和手指。信任假设：用户知道自己在做什么。

**Bridge 连接**（来自 IDE 插件或浏览器扩展）：操作来自特定的 UI 上下文，用户在 IDE/浏览器里操作。信任假设：操作在已知的应用上下文里，但可能没有像 REPL 那样的完整可见性。

**SDK 调用**（来自其他程序）：没有直接的用户界面，操作由程序自动发起。信任假设：可能没有实时的用户监督，需要更保守的权限配置。

`isBridgeSafeCommand()` 和 `REMOTE_SAFE_COMMANDS` 这类过滤机制，正是基于"连接方式改变信任假设"这个逻辑来限制可用命令范围的。

## "进场前就认清"的实际意义

"真实边界不是出事后再补，而是进场前就认清"的含义：

在会话开始时，基于运行模式、工作目录、连接方式三个维度，系统就已经确定了这次会话的权限边界。后续的工具调用都在这个边界内运行，不需要在每次调用时重新判断"这是本地操作还是远端操作"。

这让权限系统是静态的（在会话开始时配置），而不是动态的（在每次调用时判断），让系统行为可预期，也让安全边界更难被绕过。

## 面试角度的总结

Claude 在不同上下文里的身份差异，是由运行模式、工作目录、连接方式三个维度共同决定的，而不是一个单一的"权限级别"。这三个维度在会话开始时就确定了能力边界，让整个会话的行为在这个边界内保持一致。这种"进场前认清边界"的设计，比"运行中动态判断权限"更可预期，也更难出现权限混乱的情况。
