# isNonInteractive 这个布尔值为什么会影响后面几十个函数的行为？

## 一个简单的参数，复杂的影响

```typescript
const cliArgs = process.argv.slice(2);
const isNonInteractive = cliArgs.includes('-p') || cliArgs.includes('--print');
setIsInteractive(!isNonInteractive);
initializeEntrypoint(isNonInteractive);
```

这四行代码在 Commander 解析之前执行，把"是否交互"这个全局状态写入 `bootstrap/state.ts`。从这里开始，`getIsNonInteractiveSession()` 这个函数会在代码库里被调用几十次。

为什么这么多地方都要检查它？

## 两种模式下的"用户"是不同的

交互模式的用户：坐在终端前的人，能看到提示，能做出响应，可以等待。

非交互模式的"用户"：脚本、CI/CD pipeline、别的程序——它们通过管道接收输出，不能等待，不能响应弹窗，只需要标准化的机器可读结果。

这两种"用户"对系统的几乎每个行为都有不同期待。

## 影响盘点

### Trust 处理方式

```typescript
// interactiveHelpers.tsx
export async function showSetupScreens(...) {
  if (getIsNonInteractiveSession()) {
    // 跳过所有 setup 屏幕，直接返回
    return;
  }
  // 显示 onboarding、trust dialog 等
}
```

非交互模式默认信任——帮助文本明确说明"只在你信任的目录里使用"，这是把责任转移给调用者。

### 遥测初始化时机

```typescript
// entrypoints/init.ts
if (getIsNonInteractiveSession() && isBetaTracingEnabled()) {
  // 非交互 + beta tracing：立即初始化遥测（需要在第一个 query 前就绪）
  void doInitializeTelemetry().catch(...)
}
```

非交互模式下 beta tracing 必须在第一次 query 前就初始化好，因为没有 REPL 首屏这段"窗口期"可以利用。

### System context 预取

```typescript
function prefetchSystemContextIfSafe(): void {
  const isNonInteractiveSession = getIsNonInteractiveSession();
  if (isNonInteractiveSession) {
    // 非交互模式，直接预取（默认信任）
    void getSystemContext();
    return;
  }
  // 交互模式，只有已建立 trust 才预取
  const hasTrust = checkHasTrustDialogAccepted();
  if (hasTrust) {
    void getSystemContext();
  }
}
```

### Deferred prefetch 的范围

```typescript
function startDeferredPrefetches(): void {
  if (isBareMode()) {
    return; // bare 模式完全跳过
  }
  // 非 bare 的非交互模式也会执行部分 prefetch
  // ...
}
```

### ConfigParseError 的处理方式

```typescript
// entrypoints/init.ts
} catch (error) {
  if (error instanceof ConfigParseError) {
    if (getIsNonInteractiveSession()) {
      // 非交互：直接写到 stderr，程序退出
      process.stderr.write(`Configuration error in ${error.filePath}: ${error.message}\n`);
      gracefulShutdownSync(1);
      return;
    }
    // 交互：显示 Ink 对话框（需要终端渲染）
    return import('../components/InvalidConfigDialog.js').then(m =>
      m.showInvalidConfigDialog({ error }),
    )
  }
}
```

同一个配置解析错误，交互和非交互模式下的处理方式完全不同——一个弹对话框，一个写 stderr 然后退出。

## 为什么不能晚一点设置？

`setIsInteractive(!isNonInteractive)` 在 `main()` 函数非常靠前的位置执行，是有原因的。

**依赖链**：`initializeEntrypoint(isNonInteractive)` 设置 `CLAUDE_CODE_ENTRYPOINT` 环境变量（`sdk-cli` 或 `cli`），这个变量会被后续的 telemetry 事件携带。如果 telemetry 初始化先于这个设置，记录的 entrypoint 就是错的。

**Trust 依赖**：`init()` 里的 `initializeTelemetryAfterTrust()` 会调用 `getIsNonInteractiveSession()` 来决定是否立即初始化还是等待 trust。如果这时 `isInteractive` 还没设置，就会用错误的默认值。

## 模式，而不是分支

从工程角度看，`isNonInteractive` 不是一个普通的参数，而是一个"模式声明"。

普通参数改变某个具体行为，模式声明改变系统从根本上如何理解它的职责。

就像 HTTP 服务的"开发模式"vs"生产模式"——不是单个开关，而是一整套假设的集合。Claude Code 的交互/非交互区分也是如此：不是某个地方的行为不同，而是系统对"用户是谁、信任假设是什么、错误如何呈现"这些根本问题有完全不同的答案。

## 面试指导

设计 CLI 工具时，一个常见失误是把"模式差异"分散在代码各处——每个函数都零散地检查"当前是不是 CI 环境"、"当前有没有 TTY"。

更好的做法：
1. 在启动时明确声明当前是什么模式
2. 把模式信息写入一个集中的全局状态
3. 各处通过读这个状态来做决策，而不是每次重新推导

这样的好处是：模式切换时的行为变化是可枚举的（只需要搜索 `getIsNonInteractiveSession()` 的调用），而不是散落在代码的隐性角落里。Claude Code 展示了这种集中式模式管理的具体实现方式。
