# 模式、目录和连接方式为什么不能走到半路再决定

在启动最前面，Claude Code 很早就判断 `-p`、`--init-only`、`--sdk-url`、终端是不是 TTY，也会尽早看清这次有没有 `ssh` 主机、远端目录、特殊连接来源。这样做的核心，不是参数解析写得勤快，而是系统必须先决定“这一次我到底是谁”，后面很多东西才有一致答案。

因为交互还是非交互，会改写输入捕获、trust 呈现、entrypoint 记录、会话持久化和很多后台工作；本地还是远端，会改写目录语义、连接语义、权限语义，甚至改写这次会话到底算不算真正跑在当前机器上。要是这些事走到半路才定，前面的代码就只能默认一种身份，后面再不断补救另一种身份。

Claude Code 更像一个先定角色、再进场的系统。它不愿意让“我是谁”变成后面到处渗漏的隐性条件，而是尽量在最开始就把模式、现场和连接方式钉住。对普通读者来说，这其实很能说明它为什么像一台成熟工作系统，而不像一个临时拼起来的聊天程序: 它先认清自己要成为什么，再开始做事。

## 实现链
这页落到的实现链都是“身份先于环境”：设置来源、特殊入口、工作目录、模式或连接方式会先决定当前到底处在哪个世界，然后 `init()` 和后续装配才有正确前提。代码不是先搭一台通用机器再改身份，而是先把身份定准。

## 普通做法
更常见的写法是先进统一初始化，等大部分系统都起来以后，再根据参数或环境决定最后走哪条模式。

## 为什么不用那种做法
Claude Code 没这么做，因为对它来说，身份不是界面皮肤，而是会反过来改写环境、权限和能力世界的前提。先按错身份初始化，后面很多步骤都会偷偷站错坐标。

## 代价与回报
回报是环境基准统一。代价是入口看起来更像大路由表，而不像单线启动脚本。
