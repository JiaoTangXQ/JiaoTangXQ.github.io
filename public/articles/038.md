# normalizeMessagesForAPI：什么消息不能发给 Anthropic API？

Claude Code 的内部消息列表里有很多 Anthropic API 不认识的类型——`SystemLocalCommandMessage`（本地命令输出）、`SystemInformationalMessage`（系统通知）、`TombstoneMessage`（已删除消息占位符）、`AttachmentMessage`（注入的文件内容）……

在把消息历史发给 API 之前，`normalizeMessagesForAPI` 函数负责清理这些"仅供界面使用"的消息。

## 需要剔除的消息类型

从 `utils/messages.ts` 的导入可以看到内部消息类型的完整列表：

```typescript
type SystemMessage =
  | SystemLocalCommandMessage    // 本地 slash 命令的输出，只给用户看
  | SystemInformationalMessage   // 系统通知，只给用户看
  | SystemAPIErrorMessage        // API 错误通知，只给用户看
  | SystemCompactBoundaryMessage // compact 边界标记
  | SystemMicrocompactBoundaryMessage
  | SystemMemorySavedMessage     // 内存保存通知
  | SystemPermissionRetryMessage // 权限重试通知
  | SystemTurnDurationMessage    // 每轮时长统计
  | SystemApiMetricsMessage      // API 指标
  | SystemBridgeStatusMessage    // Bridge 状态
  | SystemAgentsKilledMessage    // Agent 被杀掉的通知
  | SystemAwaySummaryMessage     // Away 模式摘要
  | SystemScheduledTaskFireMessage
  | SystemStopHookSummaryMessage
```

这些 `System*` 消息类型都是 UI 专用的——它们告诉用户"发生了什么"，但不应该出现在发给模型的消息历史里。模型看到"系统：本次对话用了 12 秒"是没有意义的信息。

## normalizeMessagesForAPI 的转换规则

转换逻辑的核心原则：**只保留 API 能理解的消息，把内部扩展字段剥离，保证 API 格式合规**。

```typescript
export function normalizeMessagesForAPI(messages: Message[]): NormalizedMessage[] {
  return messages.flatMap(msg => {
    // SystemLocalCommandMessage 被 Exclude<> 类型保证不会出现在 normalizeMessagesForAPI 的输入里
    // 如果出现了，flatMap 返回空数组（过滤掉）
    
    if (msg.type === 'system') {
      return []  // 所有 system 消息都过滤掉
    }
    
    if (msg.type === 'attachment') {
      // AttachmentMessage 展开成标准的 user content block
    }
    
    // UserMessage 和 AssistantMessage 去掉内部字段
    // 比如 toolUseResult 字段（只在 transcript 显示，API 不需要）
  })
}
```

注意注释里的细节：`SystemLocalCommandMessage` 被 TypeScript 的 `Exclude<>` 类型**在类型层面就阻止了**进入 `normalizeMessagesForAPI` 的参数。这是类型系统强制执行的约束，不是运行时检查。

## AttachmentMessage 的展开

`AttachmentMessage` 是一种特殊情况——它在内部是一种独立消息类型，但在 API 层要展开成 `user` 角色消息里的 content block：

```typescript
// 内部格式
{ type: 'attachment', content: [{ type: 'text', text: '...CLAUDE.md 内容...' }] }

// API 格式（展开后合并进前一条 user 消息的 content 里）
{ role: 'user', content: [{ type: 'text', text: '...CLAUDE.md 内容...' }, ...其他内容] }
```

这个展开操作不是简单地转换类型，而是把 attachment 的内容合并进相邻 user 消息的 content 数组里。

## TombstoneMessage：被删除消息的占位符

`TombstoneMessage` 标记一条已经从对话历史里删除的消息位置。它在发给 API 时被过滤掉——API 看到的消息序列里不会有空洞。但在 UI 里保留占位符是为了让"消息 N 被删了"这个事实对用户可见（而不是消息列表莫名其妙变短）。

## thinking block 的处理

扩展思考（Extended Thinking）产生的 `ThinkingBlock` 在默认模式下也会被过滤：

```typescript
// 某些情况下 thinking blocks 需要在 API 请求里保留
// 某些情况下需要被替换成 RedactedThinkingBlock
// 这取决于模型版本和 session 配置
```

`RedactedThinkingBlock` 是一种特殊的 thinking block——内容被加密/混淆，但结构保留。这让 API 知道"这里有一段思考过程"但看不到具体内容。

## 面试角度

内部模型和 API 模型的转换层是大型应用的通用问题：

1. **内部消息类型 > API 消息类型**：内部系统有更丰富的语义，API 只需要足够的信息
2. **类型系统强制约束优于运行时检查**：`Exclude<SystemLocalCommandMessage, ...>` 让错误在编译时就被发现
3. **过滤逻辑集中在一个函数里**：分散的过滤比集中过滤更难维护和测试
4. **UI 状态和 API 状态要分离**：渲染用的信息（时长、指标、通知）不应该污染 API 请求
