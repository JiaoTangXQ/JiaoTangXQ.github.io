# 面试题：为什么把 feature flag 判断放在 getDefaultAppState 而不是组件里？

```typescript
return {
  // ...
  thinkingEnabled: shouldEnableThinkingByDefault(),
  promptSuggestionEnabled: shouldEnablePromptSuggestion(),
  // ...
}
```

这两行代码很有意思。它们不是技术配置，而是**产品体验的默认值**。

「思考模式默认开不开」「自动提示默认有没有」——这两个决定会直接影响用户打开 Claude Code 后的第一印象。把它们放在 `getDefaultAppState()` 里，意味着这些判断在状态创建时就完成了，不等任何组件来问。

---

## shouldEnableThinkingByDefault 做了什么

`shouldEnableThinkingByDefault()` 从名字可以推断，它会检查：

- 当前模型是否支持思考模式（不是所有模型都支持）
- 用户设置里是否有明确的偏好
- 可能还有实验性 feature flag

这个判断的结果被放进 AppState，然后通过 `useAppState(s => s.thinkingEnabled)` 被思考开关按钮、输入区提示、以及实际的请求构造逻辑读取。

如果这个判断散落在各个组件里（每个组件自己调用 `shouldEnableThinkingByDefault()`），会有两个问题：

1. 判断结果可能因调用时机不同而不一致（比如 A 组件先调用，用的是旧配置；B 组件后调用，配置已更新）
2. 当用户手动切换思考模式时，需要更新某个「全局状态」，但这个状态在哪里没有统一的答案

把它放在 AppState 里，这两个问题都消失了：统一的真相来源，变化通过 `setState` 传播。

---

## promptSuggestion 的完整状态

`promptSuggestionEnabled` 只是一个布尔值，但 AppState 里还有一个更完整的 `promptSuggestion` 字段：

```typescript
promptSuggestion: {
  text: string | null
  promptId: 'user_intent' | 'stated_intent' | null
  shownAt: number
  acceptedAt: number
  generationRequestId: string | null
}
```

这不只是「提示存不存在」，而是完整的提示生命周期状态：当前显示的文本、是什么类型的提示、什么时候显示的、什么时候被接受的。

把这些状态放在 AppState 里而不是组件本地，意味着：
- 提示的显示状态可以被多个组件读取（输入区显示提示文字，数据统计记录接受率）
- 提示的接受/拒绝可以从任何地方触发（包括键盘快捷键）

## 初始化时的 feature flag 决策

`shouldEnableThinkingByDefault()` 和 `shouldEnablePromptSuggestion()` 可能都依赖 GrowthBook（一个 feature flag 系统）。这里有一个微妙的时序问题：

如果 feature flag 数据是异步加载的（比如从网络拉取），`getDefaultAppState()` 在调用时 flag 可能还没到。

Claude Code 的应对策略是：先用「已知的本地配置」决定初始值，如果 flag 数据后来到了，通过 `useSettingsChange` → `applySettingsChange` 更新 AppState。这就是为什么 `AppStateProvider` 里有：

```typescript
useSettingsChange(onSettingsChange)
```

初始值是基于已知信息的最佳猜测，后续可以被远端配置修正。

---

## 「默认体验」集中管理的好处

对比两种方式：

**分散管理**：每个组件在渲染时判断「这个功能默认开不开」

```tsx
function ThinkingToggle() {
  const defaultEnabled = shouldEnableThinkingByDefault()  // 每次渲染都调用
  const [enabled, setEnabled] = useState(defaultEnabled)
  // ...
}
```

**集中管理**：在 AppState 创建时决定，组件只读状态

```tsx
function ThinkingToggle() {
  const enabled = useAppState(s => s.thinkingEnabled)  // 读现成的值
  // ...
}
```

分散管理的问题：`shouldEnableThinkingByDefault()` 可能是有副作用的操作（读配置文件、查 feature flag），在每次渲染里调用是浪费，而且如果配置变化了，已经渲染的组件不会重新判断。

集中管理的好处：判断只做一次（或在 settings 变化时重做），结果存在 AppState 里，所有组件通过订阅自动响应变化。

## 面试指导

「默认值里应该包含什么？」是一个设计问题，有几个考察点：

**为什么不把默认体验写成枚举常量？** 常量无法响应运行时信息（用户配置、feature flag、进程身份）。函数调用可以在创建时查询这些信息，返回「当前最合适的默认值」。

**feature flag 和 AppState 的关系是什么？** Feature flag 是「这个功能对这个用户是否可用」的决策，AppState 是「当前用户配置是什么」的事实。Flag 通过 `applySettingsChange` 写进 AppState，AppState 是单一真相来源。组件不直接读 flag，只读 AppState。

**如果一个 feature flag 只影响一个组件，应该放进 AppState 还是组件本地？** 放 AppState 的好处是统一管理，可以被其他组件读取（比如遥测统计）。放本地的好处是更简单。判断标准：如果多个地方需要知道这个值，放 AppState；如果真的只有一个地方用，放本地。
