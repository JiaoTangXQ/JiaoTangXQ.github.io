# 对外统一、对内高效，是成熟工程常见的取舍

`computerUse` 和 Claude in Chrome 这两条线都在做同一件事：对系统其它部分继续说 MCP 的话，对内部实现则尽量少付 ceremony 成本。`setupComputerUseMCP()` 明明在本进程里，却还是生成标准 MCP 配置；Claude in Chrome 的 `setup.ts` 也一样，对外提供的是 `mcpConfig + allowedTools + systemPrompt`，而不是一组浏览器私有指令。这说明 Claude Code 看重的不是“每一层都绝对原生”，而是“系统别的部分能不能继续把你当成熟悉的对象来接”。

更普通的选择，要么是协议纯度优先，要么是实现方便优先。前者会坚持“只要叫 server 就必须真是外部 server”，后者会说“既然都在本地，就直接写死成内部函数”。Claude Code 选的是第三条路：外面看到的口音稳定，里面根据现实情况节省成本。

这类设计通常更合理，因为大型系统最怕的是每扩一个世界就换一套接法。但它也有副作用：文档和代码都更难一眼看懂，必须接受“表面像 A，内里其实按 B 实现”这种双层现实。只不过这恰恰是成熟产品常见的形状，不是疏忽。

## 实现链
这套取舍在代码里很统一：对外都注册成动态 MCP server，让工具发现、权限、UI 和后端提示不用分支；对内则尽量走缓存、进程内绑定和轻量上下文，比如 `wrapper.tsx` 缓存 dispatcher、Chrome 复用同一 `createChromeContext()`。

## 普通做法
普通系统常常只能二选一，要么追求接口干净，要么追求内部快。

## 为什么不用
Claude Code 不愿意在这里二选一，因为它的扩展能力需要长期长在主干上，接口一旦分裂，后面每接一种能力都得重来一遍。

## 代价
代价是适配层会显得比业务动作本身还重，工程复杂度提前支付了。
