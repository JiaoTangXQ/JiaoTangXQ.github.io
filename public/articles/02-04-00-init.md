# init() 和 setup() 都是初始化，为什么要分成两个函数？

## 问题的起点

很多 CLI 工具有一个大的 `initialize()` 函数，把所有准备工作都放进去。Claude Code 有两个：

- `init()`（`entrypoints/init.ts`）
- `setup()`（`setup.ts`）

调用顺序是 `init()` 先，`setup()` 后。但这两个函数都在做"初始化"，为什么不合并？

## 关键区别：抽象世界 vs 具体现场

`init()` 的所有操作都不依赖"这次会话具体在哪个目录、哪个 worktree、恢复哪个 session"：

- `enableConfigs()`：读全局配置（`~/.claude/settings.json`），不依赖当前工作目录
- `configureGlobalMTLS()`：证书配置，全局性的
- `configureGlobalAgents()`：代理配置，全局性的
- `preconnectAnthropicApi()`：TCP 预热，不依赖当前目录
- `setupGracefulShutdown()`：进程退出处理，全局性的

这些东西在"这次会话是在 `/home/user/project-a` 还是 `/tmp/worktree-123`"确定之前就可以安全执行。

`setup()` 的操作则完全不同，它们都高度依赖"当前具体会话是什么"：

```typescript
export async function setup(
  cwd: string,                    // 当前工作目录
  permissionMode: PermissionMode,  // 权限模式
  worktreeEnabled: boolean,        // 是否启用 worktree
  worktreeName: string | undefined,
  tmuxEnabled: boolean,
  customSessionId?: string | null,
  ...
)
```

`setup()` 函数签名本身就说明了它的依赖：必须知道 `cwd`、权限模式、是否 worktree 等具体参数才能运行。

## 如果把它们合并会怎样？

```typescript
// 假设的合并版本（错误设计）
async function initAndSetup(cwd: string, ...) {
  // init() 的内容
  enableConfigs();
  configureGlobalMTLS();
  configureGlobalAgents();
  preconnectAnthropicApi();
  
  // setup() 的内容
  setCwd(cwd);
  captureHooksConfigSnapshot();
  // ...worktree 创建...
}
```

表面上更简单，但丢失了一个重要能力：**`init()` 可以在 `cwd` 确定之前调用**。

在 Claude Code 的实际流程里，`init()` 在 `preAction` 里调用，而 `cwd` 来自 Commander 解析的参数，在 action 函数里才能拿到。`setup()` 在 action 函数里调用，拿到了正确的 `cwd`。

如果合并，合并后的函数就必须等所有参数都就绪才能开始，`preconnectAnthropicApi()` 这类不依赖 `cwd` 的操作就会被迫推迟，失去并行优化的机会。

## 另一个关键：setup() 可以多次调用

`init()` 用 `memoize` 包裹，设计成只执行一次。

`setup()` 没有这个约束。在某些特殊流程里（比如 worktree 切换、会话恢复），可能需要用新参数重新调用 `setup()`。

如果合并，要么整个初始化变成不可重复的，要么连证书、代理这些全局配置也被重复执行（可能有副作用）。

## 分层的好处

把"机器层面的基础设施"（`init()`）和"这次会话的具体现场"（`setup()`）分开，带来的好处不只是代码组织更清晰：

**测试隔离**：可以单独测试 `init()` 的基础设施配置，不需要构造完整的会话上下文。

**错误边界清晰**：`init()` 失败意味着环境配置有问题（证书错误、配置文件损坏），`setup()` 失败意味着当前目录或会话配置有问题。不同类型的错误，对应不同的错误处理策略。

**并行优化窗口**：两层之间的时间可以被利用。`init()` 启动了 API 预热、远端设置加载等异步操作，`setup()` 执行时这些操作已经在后台跑着了。

## 面试指导

"为什么要分两个初始化函数"这类架构问题，答案的核心不是"为了代码整洁"，而是能说出具体的约束和好处：

1. **时序约束**：A 可以在某些参数可用之前执行，B 不行——所以必须分开
2. **可重入性**：A 只能执行一次，B 可以用不同参数多次执行——所以必须分开
3. **错误语义**：A 的失败和 B 的失败含义不同——分开可以给出更精确的错误信息
4. **并行机会**：A 和 B 之间有可被利用的等待时间——分开可以在这个窗口做优化

把这四个维度说出来，比"分开更清晰"要有说服力得多。
