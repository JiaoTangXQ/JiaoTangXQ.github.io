# init 环境导读：先把运行环境搭起来再谈会话

这一章真正关心的，不是某个初始化函数做了几件事，而是 Claude Code 为什么先搭运行环境，再谈用户真正要开始哪段会话。

所以这一章的四片叶子，要连起来读成“环境先于会话”。先看 [02-04-01-为什么init搭的是环境而不是会话.md](/Users/xxx/tmp/cc/docs/claude-code-book/02-程序怎样启动成一个agent/02-04-init怎样先把运行环境搭起来/02-04-01-为什么init搭的是环境而不是会话.md)，再顺着安全环境、远端 promise 和清理器往下读，这一章才会从“初始化细项说明”变成“启动怎样先替后面一切搭地基”的入口。

## 实现链
这一章主看 `entrypoints/init.ts`。`init()` 在这里先做配置系统、安全环境、mTLS/代理、清理器、预连接、远端设置 promise、scratchpad 等环境基础设施，重点不是打开一个会话，而是把这台机器能工作的地基搭起来。

## 普通做法
很多系统里的 init 直接和用户会话绑定，环境与会话混着启动。

## 为什么不用那种做法
Claude Code 先把环境单独拿出来，是因为网络、证书、清理、代理和遥测这些东西不该散在真正会话逻辑里临时出现。

## 代价与回报
回报是后面的会话层更干净。代价是 `init()` 本身很像一台机器的开机脚本。
