# 从启动到退出，Claude Code 经历了多少个状态转换？

## 这道综合题在考什么

面试官让你"把一次完整的会话走一遍"，表面上是在考你知不知道流程，实际上是在考你知不知道每个状态转换点由谁负责——以及为什么不能由同一个模块全部负责。

## 第一阶段：启动与世界判断

`main.tsx` 启动时做的第一件事，是确定这次进入的是哪个世界：CLI 交互模式、headless 非交互模式、remote 查看器模式，还是 coordinator 模式。这不是普通的 if/else，而是真正的世界分流——不同世界后续的能力集合、权限配置和入口路径都不一样。

紧接着是能力预热：MCP 服务器发现、插件加载、settings 读取、OAuth 账户检查。这些工作在会话开始前全部完成，不在第一轮对话时才去做。

**状态转换：**从"进程启动"到"世界已确定 + 能力已就绪"。

## 第二阶段：制度层卡位

能力就绪之后，`permissionSetup.ts` 先把权限规则从 settings、rules 来源和 auto mode 里读出来，筛掉危险的自动放行规则，建立初始权限上下文。这一步完成后，后续所有工具调用才有了合法的前置检查。

**状态转换：**从"能力已就绪"到"权限上下文已建立"。

## 第三阶段：会话持续

用户每发一条消息，`QueryEngine.submitMessage()` 先把消息追加到 transcript，更新文件状态快照，记录本轮的 token 使用量。然后把实际的采样与工具执行交给 `query()`。

`query()` 的主循环负责流式采样、工具调用、工具结果收集，以及中断处理和重试。这一层持续产出标准消息，不自己改界面状态。

**状态转换：**从"消息已提交"到"本轮工具链已结案 + transcript 已更新"。

## 第四阶段：任务运行

当 `query()` 调用了 agent 工具时，`LocalAgentTask` 把这次子任务正式做成任务对象：分配 `taskId`、绑定 `outputFile`、记录 `startTime`，开始追踪状态。

`TaskStatus` 只有五种：`pending`、`running`、`completed`、`failed`、`killed`。`isTerminalTaskStatus()` 是统一判断，避免死掉的任务还被继续塞消息。

**状态转换：**从"工具调用已触发"到"任务对象已创建 + 状态可追踪"。

## 第五阶段：事实聚合

`AppStateStore` 持续收拢界面应该相信什么。任务状态变了、MCP 连接断了、bridge 重连了、companion 有新反应了——所有这些都写进同一份事实世界，界面组件只读这一份，不自己推断真实状态。

**状态转换：**从"各层分散的状态变化"到"界面可消费的统一事实"。

## 第六阶段：Context 超长时的压缩

当会话 token 接近上限，`compact.ts` 先用 `stripImagesFromMessages()` 去掉图片，再用 `stripReinjectedAttachments()` 剥掉本轮会重新补回来的技能附件，最后生成总结消息，把关键线索接回新边界。

**状态转换：**从"长会话"到"压缩后可继续工作的短会话"。

## 第七阶段：退出

用户退出时，终态任务被标记，持久化数据写回，进程退出。

**总计**：一次完整会话里，真正的状态转换点有七个，由六个不同的模块负责，每个模块只管自己那一棒。

## 面试指导

回答这类"把流程走一遍"的问题时，最常见的失误是只说"做了哪些事"，没有说清楚"每件事由谁负责、靠什么交接"。

把状态转换点说清楚，比泛泛描述流程更有价值——因为这恰好展示了你是否真的理解了分层架构，而不只是知道有这些功能。
