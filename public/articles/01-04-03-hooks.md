# 场景推演：一个 PreToolUse hook 从配置到触发经过了哪些代码？

## 起点：一条 hook 配置

假设用户在 `~/.claude/settings.json` 里配置了：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "type": "command",
        "command": "~/.claude/hooks/validate-bash.sh $CLAUDE_TOOL_INPUT",
        "if": "Bash(rm -rf *)",
        "timeout": 10,
        "statusMessage": "验证删除命令安全性"
      }
    ]
  }
}
```

这条配置的意思是：在任何 `Bash` 工具调用匹配 `rm -rf *` 模式时，先运行 `validate-bash.sh`，给它 10 秒，如果脚本超时或者返回非零退出码，工具调用被阻止。

## 第一站：配置解析和 schema 验证

`schemas/hooks.ts` 里用 Zod 定义了 hook 的完整结构。`BashCommandHookSchema` 包含：

- `type: 'command'`
- `command: string`
- `if`：一个可选的权限规则语法字符串（`"Bash(rm -rf *)"` 就是权限规则语法），用于过滤触发条件
- `timeout`：秒数
- `statusMessage`：在 spinner 里显示的自定义状态消息
- `once`：只触发一次然后自动移除
- `async`：在后台异步执行，不阻塞工具调用
- `asyncRewake`：后台执行，但 exit code 2 时会唤醒模型（视为阻塞性错误）

这个 schema 里有一个很有意思的字段：`asyncRewake`。它解决的是一个中间状态：有些验证你不想让它阻塞正常流程（所以要 async），但如果验证失败，你又希望模型知道（所以要 rewake）。`exit code 2` 在这里被约定为"这是一个需要模型关注的问题"的信号。

## 第二站：if 条件过滤

`if` 字段用的是**权限规则语法**，和 `alwaysAllowRules` 里的规则语法相同。这意味着你可以写 `"Bash(git *)"` 来只对 git 命令触发，写 `"Read(*.env)"` 来只在读取 env 文件时触发。

hook 被触发之前，系统会先用这个 `if` 条件匹配当前工具调用的 `tool_name` 和 `tool_input`。不匹配的 hook 根本不会被启动，而不是启动后在脚本里自己判断退出。这个前置过滤的目的是：避免为不相关的工具调用 spawn 子进程，减少性能开销。

## 第三站：trust 边界

hook 不是随便什么脚本都能跑的。Claude Code 有一个"managed-only hooks"的治理开关——当这个开关打开时，只有被显式信任的 hook 来源才能生效。这个设计针对的场景是：企业管理员不希望员工的本地 settings 文件里的 hook 覆盖或绕过企业标准。

在 headless 模式或后台 agent 里，还有另一层约束：`shouldAvoidPermissionPrompts` 为 true 时，agent 不能弹出需要用户交互的界面。这时，如果一个 PermissionRequest hook 没有明确表态（允许或拒绝），系统的处理不是"默认允许"，而是"默认拒绝"——因为在无人值守的 agent 里，不确定比错误拒绝更危险。

## 第四站：退出码语义

hook 脚本的退出码决定工具调用的命运：

- `exit 0`：允许工具调用继续
- `exit 1` 或任何非零（除了 2）：阻止工具调用，将 stderr 输出作为错误信息返回给模型
- `exit 2`：特殊含义——如果是 `asyncRewake` hook，exit 2 会唤醒模型并把 stdout 作为一条消息注入会话

exit 2 的语义让 hook 系统变成了一个异步的"模型唤醒通道"：你可以在后台跑一个监控任务，当检测到某个条件时（代码覆盖率下降了、API 响应时间超阈值、有新的 PR 需要关注），通过 exit 2 把信息注入当前 agent 会话，让模型做出响应。

## 第五站：session 作用域

hook 有两个注册来源：持久配置文件（`settings.json`）和**会话级动态注册**。`/hooks` 命令允许用户在当前会话里临时添加或移除 hook，这些 hook 只在当前会话有效，不写回 settings 文件。

会话级 hook 的注册也进 AppState——和权限规则一样，运行时动态产生的 hook 需要一个地方存，而那个地方就是状态系统。

## 这不是"运行一个脚本"

普通的"脚本扩展接口"会在某个事件发生时跑一个 shell 命令，仅此而已。Claude Code 的 hooks 系统在这个基础上叠加了：

- 前置过滤（if 条件），只为匹配的调用 spawn 进程
- trust 边界，防止不受控脚本绕过企业策略
- 多种执行模式（同步/异步/asyncRewake），对应不同的控制需求
- 退出码语义化，尤其是 exit 2 的唤醒语义
- 会话级动态注册，支持临时规则
- headless 模式降级，无 UI 时的安全默认行为

## 面试指导

**考法**：给你看 hooks 的配置 schema，问"一个 PreToolUse hook 是怎么影响工具调用的"。

**满分答法**：从 if 过滤（避免不必要 spawn）→ trust 检查（企业治理边界）→ 脚本执行 + timeout → 退出码映射（0/非零/2）→ headless 降级这条链路完整说下来。特别提一下 exit 2 的唤醒语义，因为这是这套系统里最不直觉但最有工程价值的设计。

**常见误区**：把 hooks 说成"在工具调用前后运行脚本"——这是功能描述，不是系统解释。面试官想看到的是你理解 hooks 是"外部制度的接入点"：它让团队规范、验证逻辑、监控系统能以可审计的方式参与 agent 的工具决策，而不是绕过系统。
