# hooks 为什么意味着外部制度也能插手

如果说权限和设置是系统内部写好的制度，那么 hooks 代表的就是另一件更厉害的事: Claude Code 允许外部制度，在受控前提下正式进入运行时。

这不是普通的“脚本扩展接口”。它会在很多生命周期节点触发，比如工具前后、停止前后、权限请求、会话开始结束。更重要的是，这些 hooks 并不是在灰色地带偷偷运行。Claude Code 给它们准备了专门的输入结构、信任边界、会话级注册、甚至“只允许受管 hooks”这样的治理开关。连 headless 或后台 agent 不能弹权限框时，系统也会先问 PermissionRequest hooks，看看外部制度有没有明确意见。

这件事说明 Claude Code 很清醒。它知道一套真正会落地的 agent，不可能永远只活在自己写死的规则里。公司政策、团队习惯、验证流程、插件能力，都可能需要在关键节点插手。但它又没有把这种插手做成任意脚本狂欢，而是先用 trust、managed-only、session 作用域这些边界把门装好。

所以 hooks 的意义，不是“用户可以改更多东西”，而是“系统承认制度本身也需要扩展”。这让 Claude Code 更像一个可治理的工作平台，而不是一个封闭聪明体。

## 实现链
这页落到的实现链都属于制度世界：权限装配、设置来源、hooks 介入、连续性设施或条件能力 gating。代码的共同特点是，它们不是在主循环失败以后补救，而是在主循环真正推进前后就提前改写资格和边界。

## 普通做法
更普通的做法是先把 happy path 主流程跑通，再把权限、外部规则和跨回合记忆当辅助功能补上。很多系统都是这样长出来的。

## 为什么不用那种做法
Claude Code 没这么长，是因为对 agent 来说，真正危险的往往不是下一步怎么走，而是这一步有没有资格走、走完后有没有留下可继续工作的状态、外部制度有没有权利插手。它把这些问题提前写进了运行时。

## 代价与回报
回报是系统更不容易越权和失忆。代价是相关逻辑分散，阅读时不能只盯着一条主流程。
