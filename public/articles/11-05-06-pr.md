# 面试题：PR 状态什么时候显示、什么时候不显示？shouldShowPrStatus 的完整判断逻辑

这道题在面试里通常是"小题"，但能答出完整逻辑的人会给面试官留下深刻印象，因为这需要你同时理解数据条件、空间条件和优先级逻辑三个层面。

## 完整判断条件

```typescript
const shouldShowPrStatus =
  prStatus !== null &&           // 有 PR 数据
  primaryItemCount < 2 &&        // 主内容不太多
  (primaryItemCount === 0 || columns >= 80);  // 有主内容时宽度足够
```

三个条件，每个都有独立的设计动机。

## 条件一：`prStatus !== null`

这是最基础的数据守卫。`usePrStatus()` hook 会异步拉取 PR 状态，在数据返回之前是 `null`。没有数据就不显示，这没什么特别的——重要的是另外两个条件。

## 条件二：`primaryItemCount < 2`

`primaryItemCount` 统计的是底部区域里"主要内容项"的数量，通常包括 mode pill、任务 pills、teammate 提示等。

为什么主内容超过 2 个就不显示 PR？

**底部不是无限展台**

终端底部一行大约能放 80-120 个字符。每个"主内容项"可能消耗 15-30 个字符。当主内容已经有 2 个时，底部空间已经相当拥挤，再加 PR badge 会让整条底部变成一串密密麻麻的标签，反而谁都看不清。

**PR 状态是补充信息**

PR 状态告诉你当前工作目录关联的 PR 是什么状态（open/draft/merged/closed）。这是有用的上下文，但它不是做操作判断的必要信息——你正在执行的权限模式、正在运行的任务才是必要的。把补充信息限制在主信息允许的空间里，是正确的信息优先级排序。

## 条件三：`(primaryItemCount === 0 || columns >= 80)`

这个复合条件更有趣：
- 如果完全没有主内容（`primaryItemCount === 0`），PR 状态可以不管宽度就显示
- 如果有主内容，必须宽度 >= 80 才显示

**为什么 80 是阈值？**

80 列是终端的经典最小宽度。小于 80 的终端通常是被用户压缩过的，或者是嵌入在 IDE 里的小终端。这类终端的用户已经接受了有限的显示空间，强行塞入 PR badge 只会让本来就窄的底部更拥挤。

**为什么 primaryItemCount === 0 时可以无视宽度？**

如果底部没有任何其他主内容，PR 状态实际上就成了"最重要的当前状态"，此时不管多窄都值得显示。这是一个边界情况的合理处理：不能因为有宽度检查就在底部什么都没有时也不显示 PR。

## 这套逻辑体现的设计模式

**分级显示**

不是二元的"显示/不显示"，而是根据当前上下文动态决定谁有显示资格。这类逻辑在 UI 设计里叫做 **progressive disclosure**——根据可用空间和上下文重要性，决定哪些信息可以渐进地露出来。

**预算思维**

底部空间是固定预算，主内容先花，补充内容用剩余部分。这是信息架构里的资源分配问题，不是简单的"有没有数据"问题。

**条件的顺序也有意义**

最便宜的检查（`prStatus !== null`）放最前面，最后才是相对复杂的 `columns >= 80` 检查。这是短路求值的自然延伸：先排除最明显的"不显示"情况，再做更复杂的计算。

## 面试指导

**直接问法**："PR 状态的显示条件是什么？"

答：三个条件都满足才显示：有 PR 数据（prStatus !== null）、主内容少于 2 个（primaryItemCount < 2）、要么没有主内容要么终端宽度 >= 80。

**追问**："为什么不直接判断是否有空间？"

答：因为"是否有空间"本身是个模糊的判断，需要结合主内容数量和终端宽度综合评估。单纯看宽度不够，因为即使很宽，主内容太多也会挤满。单纯看主内容数量不够，因为窄屏上添加任何内容都会很挤。两个条件的组合才能准确表达"PR 有空间显示"这个语义。

**设计追问**："如果 PR 信息很重要，为什么还要这么多条件限制它？"

答：这不是在限制 PR 信息的重要性，而是承认底部区域是有预算的资源。PR 状态是补充信息，它的价值在于"额外提供上下文"，而不是"阻止主要状态被看见"。当空间紧张时，补充信息必须让位给主要状态，这是正确的优先级判断。

**扩展追问**："这套逻辑对你自己设计 UI 有什么启示？"

答：在资源有限的界面里（终端、移动端、嵌入式界面），"是否显示某个元素"不应该只是数据是否存在的问题，还要考虑当前上下文里它的相对优先级，以及显示它是否会挤压更重要的信息。这是"空间预算"思维，比"能显示就显示"的贪心思路更成熟。

*核心考察点：多条件逻辑的设计动机、progressive disclosure 模式、底部区域信息预算思维。*
