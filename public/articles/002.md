# Claude Code 模块总表

这页不是简单罗列目录，而是说明每个模块簇在运行时扮演什么角色。

## 实现链

最值得优先记住的模块簇是：

- `main.tsx`、`interactiveHelpers.tsx`、`commands.ts`：入口装配层，决定程序怎样启动、有哪些命令、先做哪些 setup。
- `QueryEngine.ts`、`query.ts`、`query/*`：会话与回合层，负责长期会话状态和单轮执行。
- `Tool.ts`、`tools.ts`、`services/tools/*`：工具协议与工具执行层。
- `Task.ts`、`tasks.ts`、`utils/task/*`、`tasks/*`：任务对象与生命周期框架。
- `tools/AgentTool/*`、`utils/swarm/*`、`utils/teammateMailbox.ts`：子 agent 与团队协作层。
- `state/*`、`components/*`、`context/*`：终端界面和状态显示层。
- `utils/permissions/*`、`components/permissions/*`：权限制度层。
- `services/mcp/*`、`skills/*`、`plugins/*`、`remote/*`：扩展和外部世界接入层。

## 普通做法

普通做法会把模块按“通用工具”“业务逻辑”“UI 组件”“第三方集成”粗分。这样适合典型前后端项目。

## 为什么不用

Claude Code 没这么粗分，是因为它的很多模块不是“静态分层”，而是“运行时职责分工”。例如 `AgentTool` 既属于工具，又牵出任务和团队；`components/permissions/*` 虽然在 UI 目录里，实际上又直接影响执行能否落地。

如果按最直觉的目录分类，新手会误以为 UI 只是皮肤、utils 只是杂物、tools 只是 API 包装。源码实际不是这样。

## 代价

这种模块总表不如目录树那么一眼明白，因为它要求读者接受“同一个目录在不同运行链里有不同角色”。代价是认知更抽象，价值是更贴近真实代码结构。
