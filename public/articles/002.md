# QueryEngine 和 query() 有什么区别，为什么要拆成两层？

## 从代码发现问题

打开 `QueryEngine.ts`，这个类的字段是一组跨回合存活的状态：

```typescript
private mutableMessages: Message[]          // 消息历史，跨 turn 累积
private abortController: AbortController    // 会话级取消控制
private permissionDenials: SDKPermissionDenial[]  // 记录所有被拒绝的权限
private totalUsage: NonNullableUsage        // 累计 API usage
private readFileState: FileStateCache       // 文件读取缓存
private discoveredSkillNames = new Set<string>()   // turn 级别，每轮清空
private loadedNestedMemoryPaths = new Set<string>() // 内存路径
```

再看 `submitMessage()` 方法的签名——它是 `async *`，一个 async generator。每次调用代表一轮对话，内部调用 `query()`，最终 yield 出 `SDKMessage` 事件流。

`query.ts` 里的 `query()` 函数则不持有状态。它接受"这一轮需要的一切"——消息列表、工具、模型、abort controller——然后运行，结束后返回。它不关心上一轮发生了什么，也不关心下一轮会怎样。

## 两层的职责边界

```
QueryEngine（会话层）
├── 跨轮状态：消息历史、usage、权限拒绝
├── 生命周期：连接 → 多轮对话 → 关闭
└── submitMessage() ─ 每轮入口，委托给 query()

query()（回合层）
├── 无状态接收：每次调用都是干净开始
├── 执行全程：模型调用、工具执行、compact、stop hooks
└── 一轮跑完即返回，不保留记忆
```

这个边界并不是随意划的。回答"为什么拆"需要看 QueryEngine 的字段设计。

`permissionDenials` 被设计为只增不减——它记录这段会话中所有被拒绝的权限请求。SDK 消费方想知道"整段会话拒绝过什么"，但 query() 执行的是单轮逻辑，它没有上帝视角。必须由 QueryEngine 跨轮积累。

`readFileState` 是文件缓存。读过的文件内容在轮间共享，避免重复 IO。如果每轮都是无状态，这个缓存就没有意义。

`mutableMessages` 最关键——它是整个会话的消息历史。每轮 `submitMessage()` 都把新的用户消息加进去，query() 执行完后的消息（包括工具调用和 assistant 回复）也追加进来。下一轮就在这个历史上继续。

## 单层设计会怎么崩

如果把两层揉成一个大函数，常见的写法是：一个函数接受消息历史引用，在内部既处理单轮逻辑，又维护累积状态。

问题：compact 触发时，query() 会修改消息列表（裁剪旧消息，保留摘要）。如果消息列表是直接共享的，compact 就会和外部状态形成耦合——compact 改了历史，外部的权限追踪、文件缓存可能引用了已经被清掉的消息 ID。

两层设计里，query() 拿到的是 `mutableMessages` 的引用，可以就地修改（compact 在这层发生），但 QueryEngine 仍然持有这份列表的所有权，知道什么时候能安全读取。

## 面试指导

回答"会话管理"相关问题时，强调两个关键词：跨轮状态和单轮状态。跨轮的放 QueryEngine——消息历史、usage、权限拒绝、文件缓存；单轮的放 query()——模型调用、工具执行、compact 触发、stop hooks。两者通过 `submitMessage()` 衔接，`discoveredSkillNames` 是个有趣的例外：字段放在 QueryEngine，但每次 `submitMessage()` 开头清空，属于"turn 作用域但需要被持有"的特殊情况。
