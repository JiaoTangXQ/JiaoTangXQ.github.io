# AsyncLocalStorage 凭什么能隔离同一进程里的多个 Agent？

面试官问：同一个 Node.js 进程里运行 5 个 AI Agent，怎么保证它们各自的"身份信息"（名字、颜色、所属团队）不会互相污染？

初级答案：全局变量 + 命名前缀，比如 `global.agent1_name`、`global.agent2_name`。

高级答案：`AsyncLocalStorage`。

Claude Code 用的是高级答案，而且用得很彻底。

## 全局变量方案的问题

先弄清楚全局变量方案到底坏在哪里。

假设有两个 teammate 同时运行，一个叫 `researcher`，一个叫 `reviewer`。它们都需要知道"我是谁"，所以都会调用 `getAgentName()` 这个函数。

如果 `getAgentName()` 读的是全局变量 `global.currentAgentName`：

```
t=0: researcher 开始跑，设置 global.currentAgentName = 'researcher'
t=1: reviewer 也开始跑，设置 global.currentAgentName = 'reviewer'
t=2: researcher 调用 getAgentName()，得到 'reviewer'  ← 错了
```

这就是经典的 race condition。进程内并发执行时，全局状态天然是竞争资源。

## AsyncLocalStorage 的工作原理

`AsyncLocalStorage` 是 Node.js 18+ 的内置 API（在 `async_hooks` 模块里）。它的核心特性：**存储在 AsyncLocalStorage 里的值，会沿着 async 调用链自动传播，但不会跨越 async 上下文边界**。

类比：想象每个异步调用链有一个隐形的"随身背包"，`AsyncLocalStorage` 就是往这个背包里放东西。子调用（await 下面的代码、回调函数）能看到背包里的内容，但同时运行的其他调用链有自己的背包，互不干扰。

`teammateContext.ts` 里的实现：

```typescript
const teammateContextStorage = new AsyncLocalStorage<TeammateContext>()

export function runWithTeammateContext<T>(
  context: TeammateContext,
  fn: () => T,
): T {
  return teammateContextStorage.run(context, fn)
}

export function getTeammateContext(): TeammateContext | undefined {
  return teammateContextStorage.getStore()
}
```

`AsyncLocalStorage.run(store, fn)` 的语义：在 `fn` 执行的整个异步调用链里，`getStore()` 都返回这个 `store`，其他调用链里的 `getStore()` 不受影响。

## Claude Code 怎么用它

`spawnInProcessTeammate()` 创建 teammate 时，生成 `TeammateContext`：

```typescript
const teammateContext = createTeammateContext({
  agentId: 'researcher@my-team',
  agentName: 'researcher',
  teamName: 'my-team',
  color: 'blue',
  planModeRequired: false,
  parentSessionId: leaderSessionId,
  abortController: new AbortController(),
})
```

然后 `inProcessRunner.ts` 用 `runWithTeammateContext` 包住整个 agent 执行循环：

```typescript
runWithTeammateContext(teammateContext, () => {
  // 在这里面执行 runAgent()
  // 所有 await、回调、子调用都能通过 getTeammateContext() 拿到 researcher 的上下文
})
```

与此同时，另一个 teammate `reviewer` 有自己的 `runWithTeammateContext` 调用，有自己的背包。

当 `researcher` 的执行链里调用 `getAgentName()`：

```typescript
export function getAgentName(): string | undefined {
  const inProcessCtx = getTeammateContext()
  if (inProcessCtx) return inProcessCtx.agentName  // 返回 'researcher'
  return dynamicTeamContext?.agentName
}
```

同时 `reviewer` 的执行链里调用同一个函数，返回的是 `reviewer`，不是 `researcher`。同一个函数，同一时刻，两个不同的返回值。

这就是 AsyncLocalStorage 的魔法：**用 async 调用链的传播边界来代替进程边界**。

## 两层隔离：TeammateContext + AgentContext

Claude Code 给同进程 teammate 套了两层 AsyncLocalStorage：

**外层：TeammateContext**（`teammateContext.ts`）
- 包含：agentId、agentName、teamName、color、planModeRequired、parentSessionId
- 用途：团队协作相关的身份信息，用于消息路由、权限归因、UI 显示

**内层：AgentContext**（`agentContext.ts`）
- 包含：agent 调用级别的分析元数据
- 用途：日志归因、分析追踪、工具调用记录

为什么要两层？因为这是两种不同维度的"谁"：TeammateContext 回答"我是团队里的哪个成员"，AgentContext 回答"这次执行是以什么 agent 身份发起的"。

一个比喻：TeammateContext 是员工证（你是哪个部门的、叫什么名字），AgentContext 是工作签核（这次执行是在哪个项目下发起的、要记到谁的工时上）。两者相关，但不是同一件事。

## 独立 AbortController 的设计

`spawnInProcessTeammate()` 还为每个 teammate 创建了**独立的** `AbortController`：

```typescript
// Create independent AbortController for this teammate
// Teammates should not be aborted when the leader's query is interrupted
const abortController = createAbortController()
```

注意注释：teammate 不应该在 leader 的 query 被中断时跟着中断。

这是另一个隔离维度：生命周期隔离。Leader 可能因为用户按 Esc 而中断当前 query，但正在工作的 teammate 不应该因此被杀掉——它们是独立任务的执行者，有自己的生命周期。

独立 AbortController + AsyncLocalStorage，一起构成了同进程 teammate 的完整隔离层：状态隔离（AsyncLocalStorage）+ 生命周期隔离（独立 AbortController）。

## 面试怎么答

如果面试题是"同进程并发执行时如何防止状态污染"：

**初级答案**：锁和互斥量。这能防止写冲突，但不能提供每个执行链"自己的"状态副本。

**中级答案**：每次函数调用都传入上下文参数。这可行，但对代码侵入性极大——所有函数签名都要带上一个 context 参数，很快变得不可维护。

**高级答案**：AsyncLocalStorage（或其他语言的类似机制：Java 的 ThreadLocal，Python 的 contextvars）。这提供了"隐式传参"能力：上下文沿着调用链传播，但不需要每个函数都显式接收它。

**架构理解**：AsyncLocalStorage 把"我是谁"的问题从"每次都要传参"变成了"在正确的地方建立上下文，后续自动传播"。这不只是技术便利，而是真正模拟了进程隔离的语义——只是隔离边界从进程内存变成了 async 调用链。
