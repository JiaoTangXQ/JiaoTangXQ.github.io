# 从单 Agent 到 Team Member：身份层为什么要单独建模？

一个 agent 独自工作时，系统只需要知道三件事：任务是什么、当前状态是什么、结果去哪里。这是任务系统的全部。

但当同一个 agent 成为团队成员，立刻需要回答一批新问题：它叫什么名字？它属于哪个团队？它的颜色是什么（因为 UI 要区分不同成员）？它是否需要先进入 plan mode 才能动手？它的 leader 是谁？

这些问题不是任务系统能回答的，它们构成了**身份层**。

## TeammateIdentity：团队成员的基础类型

`spawnInProcessTeammate()` 在创建 teammate 时，第一步是构建 `TeammateIdentity`：

```typescript
const identity: TeammateIdentity = {
  agentId,        // "researcher@my-team" — 唯一标识符
  agentName,      // "researcher" — 显示名
  teamName,       // "my-team" — 所属团队
  color,          // "blue" — UI 颜色标识
  planModeRequired, // true/false — 是否必须先 plan 再执行
  parentSessionId, // leader 的 session UUID，用于 transcript 关联
}
```

这个结构随后被放入 `InProcessTeammateTaskState`，伴随 teammate 的整个生命周期。

为什么每个字段都存在？逐个看：

- `agentId` 是 `agentName@teamName` 格式，在同一进程里可能同时跑多个团队，这个格式保证全局唯一。
- `color` 不是装饰——权限弹窗、日志输出、pane 边框都要用它来在视觉上区分"谁发出的"。
- `planModeRequired` 影响 teammate 的执行策略，高风险操作的 teammate 可以被要求"先规划后执行"。
- `parentSessionId` 让 Perfetto 追踪、日志分析工具能把 worker 的行为归到 leader 的 session 树里。

## 身份层和任务层的分工

在 `InProcessTeammateTaskState` 里，两层清晰并存：

```
任务层字段：
  status         — running / killed / completed
  messages       — 消息历史
  isIdle         — 是否在等待
  shutdownRequested — 是否已被请求退出

身份层字段：
  identity       — TeammateIdentity（上面那个结构）
  permissionMode — plan / default（当前执行模式）
  color          — 与 identity.color 一致，供状态查询
```

任务层回答"这件工作的当前状态"，身份层回答"做这件工作的是谁、它的角色是什么"。两层分离，各自可以独立变化——任务可以完成，但身份结构在 cleanup 之前不会消失。

## 为什么不能复用任务系统的字段

一个常见的工程懒惰是：往任务对象上加一个 `isTeamMember: true` 加一个 `teammateColor: 'blue'`，就算支持团队了。

这行不通的原因：

**查询语义不同**。任务系统经常按状态查询（"给我所有 running 的任务"）。团队系统经常按身份查询（"给我 researcher 这个成员的最新状态"）。这两种查询如果都压在同一个字段平面上，索引和 filter 的逻辑会互相干扰。

**生命周期不同**。任务完成后，任务字段可以清理。但成员身份在某些场景下需要在任务完成后还保留一段时间——比如 leader UI 要显示"researcher 刚刚完成了任务"的状态，这需要保留 identity 而不是清空。

**权限归属不同**。权限系统需要知道"是谁请求了这个权限"，这是身份信息；任务系统需要知道"这个权限请求对应哪个执行上下文"，这是任务信息。两种信息都需要，但来源不同。

`InProcessTeammateTask.tsx` 因此提供了专门按身份查找的函数，比如 `findTeammateTaskByAgentId(agentId, tasks)`，而不是用通用的任务 filter。

## 面试怎么答

如果面试题是"你的多 agent 系统如何管理 agent 的身份信息"：

**常见误区**：把身份信息散落在各个地方——颜色在 UI 层、名字在任务名里、权限在一个独立配置里。这样的系统在查询"这个请求是哪个 agent 发出的"时，需要跨多个地方拼凑。

**正确思路**：把 agent 的身份字段集中建模为一个类型（相当于 `TeammateIdentity`），然后让所有需要"知道这是谁"的地方都引用这个类型，而不是各自存一份碎片化的字段。

**进阶思路**：区分"身份信息"和"运行时状态"的生命周期。身份信息（名字、颜色、角色、所属团队）是配置型数据，生命周期长；运行时状态（当前正在干什么、是否 idle）是实时数据，生命周期短。两者不应该混在同一个对象里，否则更新运行时状态时会误触身份字段，反之亦然。

Claude Code 的 `TeammateIdentity` 和 `InProcessTeammateTaskState` 的拆分，正是这个原则的直接体现。
