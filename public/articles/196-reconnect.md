# 第十三卷回收索引：外延执行里自动启用与缓存检测认真对待真实用户现场

## 实现链
外延执行里这一支原来还能放在一页里，因为它在回答同一个问题：  
产品接入链愿不愿替真实用户把现场照顾完整。

但现在它已经自然分成两条更窄的线：

- 自动启用与缓存检测怎样认真对待真实用户现场
- `reconnect` 页怎样说明失败与重接也要一起被设计进去

如果继续把这两件事挤在同一页里，就会把“现场接入”与“失败重接”重新揉成一个判断。

所以更稳的做法，是把 `196` 收成父级入口，再往下拆成两页。

这页现在作为父级入口，继续分出去的实现分支是：
### 自动启用与缓存检测认真对待真实用户现场

- [237-第十三卷回收索引-外延执行-自动启用与缓存检测认真对待真实用户现场.md](/Users/xxx/tmp/cc/docs/claude-code-book/appendices/237-第十三卷回收索引-外延执行-自动启用与缓存检测认真对待真实用户现场.md)

### `reconnect` 页说明失败与重接也要一起被设计进去

- [238-第十三卷回收索引-外延执行-reconnect页说明失败与重接也要一起被设计进去.md](/Users/xxx/tmp/cc/docs/claude-code-book/appendices/238-第十三卷回收索引-外延执行-reconnect页说明失败与重接也要一起被设计进去.md)

它不再负责细讲产品接入链里的两个判断。  
它只负责保住一个入口判断：

产品级接入链不能只把门打开一次。  
它要替真实用户把接入、失败和重新接回来这整段过程都做完整。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。 对父级入口页来说，更常见的写法就是继续把两三条不同链挤在一页里，读起来省页数，但判断会糊。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。 这些父级页现在存在，就是为了先把分支关系讲清，再让更细页分别承担细节。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 作为父级入口，它还额外带来导航变深、读者要多跳几页的成本。
