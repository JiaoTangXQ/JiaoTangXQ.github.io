# 设计一个 LLM 上下文管理系统：为什么分层留存比硬记一切更优？

**面试题原型**："如果你要设计一个能支持长期工作的 LLM Agent，上下文管理系统应该怎么设计？"

这道题没有标准答案，但有层次之分。最初级的回答是"尽量保留所有历史"，更成熟的回答是"分层留存"。

---

## 为什么"硬记一切"是个陷阱

直觉上，保留所有历史是最安全的——不会丢信息。但这个直觉在 LLM 的工程约束下会碰壁：

**约束 1：上下文窗口有硬限制**。200K token 是真实的边界。超过了，API 会报 `prompt_too_long` 错误，整个会话崩溃。

**约束 2：长上下文的质量下降**。研究反复证明，当相关内容被大量无关内容淹没时，模型的有效注意力会降低。保留越多，有时候反而让模型"记不住"关键内容。

**约束 3：成本和延迟**。token 数和费用近似正比。一个 180K token 的请求可能比 40K token 的请求贵 4.5 倍，TTFT 也会更长。

所以"硬记一切"不只是"不够优雅"，而是会让系统越工作越慢、越贵、越不可靠。

---

## 分层留存的核心思路

Claude Code 的架构可以用三层模型来理解：

```
Layer 1：热窗口（Hot Window）
  - 当前 compact boundary 之后的原始消息
  - 模型能完整看到的最近历史
  - 这里的内容是"实时工作台"

Layer 2：压缩历史（Compressed History）
  - compact 之后的摘要消息（summaryMessages）
  - 比原始历史小 10x-20x
  - 保留了语义，丢弃了细节

Layer 3：外置记忆（External Memory）
  - ~/.claude/projects/<path>/memory/ 目录
  - auto-memory 和 session memory
  - 跨会话持久化，主动管理
```

每一层的特性不同，适合存放不同类型的内容。

---

## 什么应该在哪一层

**热窗口**存最近的工作上下文：刚读的文件、正在讨论的代码、上几轮的决策。这些内容"新鲜度"很高，模型需要直接引用它们。

**压缩历史**存的是"发生过什么"的概要：前面几十轮做了什么、遇到了什么问题、结论是什么。细节可以丢，但轨迹要在。

**外置记忆**存的是"以后还会用到的知识"：用户的偏好（总是喜欢先写测试再写实现）、项目的特殊约定（这个 API 不要用 mock）、长期的工作状态（这个重构还没做完，下次继续）。

从 `memoryTypes.ts` 可以看到 Claude Code 对 auto-memory 的分类：

```typescript
export const MEMORY_TYPES = ['user', 'feedback', 'project', 'reference'] as const
```

这四类记忆的共同特点是：**无法从当前代码状态派生**。代码模式、架构、git 历史，这些"可以 grep 到"的信息不应该进记忆，因为记忆有成本，而且可能过时。

---

## autoCompact.ts 里的阈值设计

Claude Code 的阈值设计是分层管理在数字上的体现：

```typescript
export const AUTOCOMPACT_BUFFER_TOKENS = 13_000
export const WARNING_THRESHOLD_BUFFER_TOKENS = 20_000
export const ERROR_THRESHOLD_BUFFER_TOKENS = 20_000
export const MANUAL_COMPACT_BUFFER_TOKENS = 3_000
```

解读：
- **autocompact 阈值** = effectiveContextWindow - 13K：还有 13K 余量时开始自动压缩，给摘要请求留空间
- **warning 阈值** = threshold - 20K：提前警告用户
- **blocking limit** = contextWindow - 3K：到这里就不让继续，强制 compact

这种分级让系统有足够的提前量来处理，而不是等到真正撞墙才反应。

---

## 分层设计的取舍

**好处**：每一层可以独立优化，不同粒度的内容用不同的策略处理。hot window 里的工具结果可以被 microcompact 清除，而不需要整个 compact。

**代价**：系统必须公开承认"不是所有历史都会一直留在主窗口里"，用户要接受连续性是分层管理的，某些远历史的细节只保留在摘要层甚至只在外置记忆里。

这个代价是真实的。如果你在第 150 轮问"你之前第 30 轮读的那个函数叫什么名字"，系统可能已经 compact 过两次了，精确的原始内容找不到了，只有摘要层的概述。这是设计上的妥协，不是 bug。

---

## 面试答题框架

回答"设计 LLM 上下文管理系统"这道题时，可以这样展开：

1. **承认约束**：上下文有限、长上下文质量下降、成本正比于 token 数
2. **提出分层**：热窗口（实时工作台）、压缩历史（语义摘要）、外置记忆（持久知识）
3. **讨论各层的清除/写入策略**：microcompact 清工具结果、autocompact 生成摘要、stop hooks 写记忆
4. **说明熔断和保护**：递归保护（compact agent 不触发 autocompact）、连续失败熔断、task_budget 跨 compact 延续

能把这四点说清楚，已经比大多数候选人的答案深得多。
