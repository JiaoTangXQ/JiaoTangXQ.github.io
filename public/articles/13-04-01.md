Claude Code 不太容易把自己做乱，靠的不是“模型懂事”，而是制度层先卡位。`permissionSetup.ts` 不是在工具已经跑出去后才问一句“要不要继续”，而是先从 settings、规则来源、auto mode、危险模式里把可疑通道筛一遍，尤其专门拦截那种会让 Bash、PowerShell 或 subagent 获得过大自由度的规则。`Task.ts` 里连任务终态都被正式定义成 `completed / failed / killed`，并且有 `isTerminalTaskStatus()` 这种统一判断，避免死掉的任务还被继续塞消息。settings 和 hooks 模块也不是插件式锦上添花，而是制度的一部分：哪些规则来自哪里、什么时候可以改、什么时候只允许观察，都是先定好再放能力进场。

更普通的 agent 产品，常见做法是把制度塞进 prompt，或者在每个工具按钮旁边各自长一段局部校验。那种办法短期内能跑，但规则一多就会出问题：不同入口的判断口径不一致，auto mode 和手动 mode 各有各的漏洞，危险动作往往是在“顺手放过一次”里溜出来的。

Claude Code 现在这种制度先行的做法明显更靠谱，因为它把“强能力是否有资格上场”变成了程序结构问题，而不是善意假设。代价也很明确：规则来源多、配置层厚、理解曲线陡。但只要目标不是一次性演示，而是长期可控，制度厚一点通常比能力乱一点更划算。

## 实现链
权限、settings、hooks 和任务协议都先把“能不能做”定清，再让模型和工具进场。

## 普通做法
默认相信模型和用户都只会走好路。

## 为什么不用
因为能力一旦变强，不先上制度就会先上事故。

## 代价
制度层会让很多路径显得更慢、更重。
