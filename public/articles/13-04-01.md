# permissionSetup.ts 为什么在会话开始前就把危险规则筛掉？

## 设计哲学问题

面试官问"你怎么设计一个 agent 的权限系统"，最常见的错误答案是"在工具调用前弹窗让用户确认"。这个答案不是错的，但它只解决了半个问题。

真正的权限问题不是"这次调用要不要确认"，而是"什么样的规则配置会让危险调用绕过确认"。

## 什么是制度先行

`permissionSetup.ts` 的核心逻辑不是拦截单次工具调用，而是在会话开始时扫描整个规则配置，把那些"会让 Bash、PowerShell 或 subagent 获得过大自由度"的自动放行规则找出来，阻止它们生效。

这一步发生在用户的第一条消息之前。不是等工具跑出去再问，而是在工具有资格上场之前，先把那条能让它偷偷溜过去的通道关掉。

**这就是制度先行**：不假设所有人都只走好路，而是把"强能力是否有资格上场"变成程序结构问题。

## 为什么不用 prompt 里的规则来替代

更普通的 agent 产品，常见做法是把权限管理塞进 system prompt：在系统提示里写"你不能执行危险命令"或者"每次文件修改前都要确认"。

这个做法的问题是：prompt 里的规则是给模型看的，而不是给程序结构看的。模型在特定场景下可能忽略、误解，或者被上下文引导绕过这些规则。Auto mode 下如果没有程序结构层面的卡位，最危险的操作往往是在"顺手放过一次"里溜出来的。

`permissionSetup.ts` 做的事情，是把这个决定从"模型的自觉"移到"程序的结构"——危险规则检测不依赖模型是否懂事，而是在代码层面先把那条通道关掉。

## Task.ts 的终态协议

制度先行的另一个体现是 `Task.ts` 里的任务终态设计。

`TaskStatus` 有且仅有五种状态：`pending`、`running`、`completed`、`failed`、`killed`。`isTerminalTaskStatus()` 是统一的终态判断函数：

```typescript
export function isTerminalTaskStatus(status: TaskStatus): boolean {
  return status === 'completed' || status === 'failed' || status === 'killed'
}
```

这个函数存在的意义是：避免每个消费者各自判断"这个任务是不是死了"。如果没有这个统一协议，就可能出现"死掉的任务还被继续塞消息"的情况——不是因为有人刻意做错，而是因为判断口径不一致。

## settings 和 hooks 也是制度的一部分

Settings 和 hooks 模块不是用户可选的便利功能，而是制度的一部分。它们定义了：哪些规则来自哪里，什么时候可以改，什么时候只允许观察。

项目级 settings 可以覆盖用户级 settings 的某些字段，但有些字段只能在用户级配置。这个分层不是为了复杂而复杂，而是在制度层面保证了"谁有权改什么"——在系统层面，而不是靠约定俗成。

## 代价

制度层让很多路径显得更慢、更重。在 headless 模式下，permission setup 会读取更多配置；在 auto mode 下，危险规则检测会拦截一些看起来无害的配置。

新用户第一次遇到这些拦截时，可能觉得系统太保守。但制度的成本通常只在事故发生之前才显得值不值，等出了事再来补，代价就不只是代码了。

## 面试指导

被问到"如何设计 agent 的安全边界"时，可以用"制度先行"来展开：

不只是拦截单次危险调用，而是在会话开始前就把"可能让危险调用绕过确认"的规则配置本身挡掉。前者是事后控制，后者是结构性预防。产品级的安全设计需要两者兼备，但很多人只想到前者。
