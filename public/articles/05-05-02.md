# 摘要服务为什么和实时进度分开记账

实时进度和摘要都在讲“任务进行到哪了”，但 Claude Code 故意把它们拆成两条账。

## 实现链

`updateAgentProgress()` 在更新进度时会刻意保留已有的 `summary`，避免实时消息把后台摘要覆盖掉。反过来，`updateAgentSummary()` 又只更新 `progress.summary`，并在开启相关选项时把摘要发给 SDK 消费者。代码注释已经点明：摘要是周期性服务写入的，实时进度是从消息流算出来的，两者不能互相踩。

这说明 Claude Code 把“当下发生了什么”和“此刻怎么向人概括”当成两种不同责任。

## 普通做法

很多系统会只保留一份进度文本。谁最后写入，界面就显示谁的版本。实现简单，但经常互相覆盖。

## 为什么不用

Claude Code 不这么偷懒，是因为实时活动和摘要天然不是同一粒度。前者适合告诉你“刚刚读了哪个文件、用了哪个工具”，后者适合告诉你“它大致已经查到哪一步”。混成一份很容易两头都不好用。

所以它明确分账：一个追事件，一个追概括。

## 代价

代价是状态模型更复杂，也更容易让读代码的人疑惑“为什么同一个 progress 里还有两套来源”。但这种复杂度换来了更稳定的人类可读性。
