# 深度分析：sticky prompt 解决了什么问题，为什么它是接缝锚点

## 长会话的漂移问题

你打开了 Claude Code，开始处理一个 bug。你发了第一条消息："这个函数的逻辑有问题，帮我看看"。

20 分钟后，会话有了 30 轮对话，你和 AI 一起修复了 bug、优化了几个相关函数、写了测试、修复了测试发现的另一个 bug、顺手重构了一个工具函数……

现在，你想继续，但不太确定：
- 当前在哪个分支？
- 当前修改了哪些文件？
- 最初的问题解决了吗，还是还有待处理？
- AI 现在在处理什么具体的任务？

这些问题不是记忆问题（你可以通过翻会话历史找到答案），而是**认知成本问题**：每次想做下一步时，需要花时间重新建立当前的上下文，而不是直接思考下一步应该做什么。

这就是"上下文漂移"：随着会话进行，用户对当前状态的认知逐渐模糊，需要不断重新定位。

## sticky prompt 的机制

sticky prompt 在输入框附近固定显示一段用户定义的上下文信息。这段信息在会话进行过程中持续可见，不随对话滚动而消失。

典型的 sticky prompt 内容：
- 当前 git 分支名：`feature/fix-auth-bug`
- 当前任务描述：`修复用户登录后跳转失败的问题`
- 当前工作状态：`[WIP] 已修复主流程，待处理边缘情况`

用户每次看向输入框时，都会瞥到这些信息，不需要主动去找。

## 为什么说这是接缝锚点

"接缝"是系统不同部分之间的边界。在长会话里，每次用户重新开始思考"下一步做什么"，都是一个接缝点：之前的思路在这里暂停，下一步的思路在这里开始。

没有锚点时，这个接缝是脆弱的：用户从"之前的思路"切换到"下一步思路"时，需要重建完整的上下文，这个重建过程容易出错（记错了当前状态）或者耗时（需要翻历史）。

sticky prompt 是这个接缝的锚点：它把局部上下文（当前分支、当前任务）钉在接缝处，用户从一个思路切换到下一个时，上下文信息还在那里，接缝变得稳固。

这是"界面也是运行时的一部分"的另一个例子：界面不只是显示结果，而是主动支持用户的认知流程，让接缝处的上下文切换更可靠。

## sticky prompt 的用户配置

sticky prompt 的内容可以用户自定义，这是一个重要的设计决策。

不同用户在不同场景下需要不同的上下文信息：

- 处理 bug 时：当前分支、相关 issue 编号
- 做代码审查时：PR 编号、审查要点
- 做重构时：当前重构目标、已完成的模块

系统无法预知用户需要什么上下文，所以把配置权给用户。

和 StatusLine hook 一样，这里体现了同一个设计原则：**无法预见需求，给用户工具让他们自己定制**。

## 与分组折叠的分工

在 173 文章里分析了分组折叠：管理历史操作的信息密度。

sticky prompt 管理的是另一个维度：当前状态的可见性。

两者配合：
- 折叠：让用户能快速扫描历史，找到"之前发生了什么"
- sticky prompt：让用户时刻能看到"现在在哪里"

在长会话里，这两个维度都很重要。如果只有折叠没有 sticky prompt，用户知道历史但容易忘记当前状态。如果只有 sticky prompt 没有折叠，用户知道当前状态但在历史里容易迷失。

## 实现上的考量

sticky prompt 的实现要解决几个问题：

**更新时机**：sticky prompt 应该随每次对话自动更新（如果它展示动态信息，比如当前 git 分支），还是用户手动更新（如果它是用户手写的任务描述）？

Claude Code 的做法是两者都支持：可以是静态文本（用户写好后不变），也可以是 hook 命令的输出（每次刷新时重新执行，获取最新状态）。

**显示位置**：sticky prompt 需要在最重要的操作区域附近——输入框附近。不在屏幕角落，不在标题栏（太远，用户注意力不在那里），而是在用户输入下一条消息之前必然会看到的地方。

## 面试要点

1. **sticky prompt 解决上下文漂移问题**：长会话里用户对当前状态的认知会模糊，fixed 显示当前上下文减少重新定位的认知成本。
2. **接缝锚点的含义**：思路切换时需要稳定的上下文参照，sticky prompt 把这个参照钉在切换点，降低接缝处的信息损失。
3. **用户自定义是可扩展性的体现**：系统无法预知每个用户在不同任务里需要什么上下文，把配置权交给用户是正确的选择。
4. **与折叠互补**：折叠管理历史维度，sticky prompt 管理当前状态维度，两者配合保持长会话的认知可控性。
