# 位置锚点要同时存 UUID 和 index，两个坐标互相校正

`computeSliceStart()` 维护一个锚点：

```typescript
sliceAnchorRef: { uuid: string; idx: number } | null
```

当需要计算可见窗口的起始位置时：
1. 先用 `uuid` 在当前列表里找，找到了就用找到的位置
2. 找不到（消息被折叠或移除了），用 `idx` 作为备用坐标
3. 两者都不合适时，退回保守的默认位置

## 为什么一个坐标不够

**只用 UUID 的问题**：消息可能因为折叠、compact、brief 过滤等原因从列表里消失。如果锚定的消息消失了，用 UUID 找不到任何东西，整个视口会重置到一个未定义的位置。

**只用 index 的问题**：消息列表被重组（重排、新消息插入、折叠后展开）后，同一条消息的下标会变化。用 idx 找到的可能完全不是原来那条消息。

## 两坐标的互补

UUID 在消息存在时是精确的——如果我要找的消息还在，UUID 能精确定位到它。当消息因为某些变换不在了，UUID 失效，idx 作为「大致还在哪里」的近似值接管：即使找不到原来那条消息，也能把视口保持在大致相同的位置区域。

这是一种渐进降级：先尝试精确的身份匹配，退而求其次使用位置近似，再退则使用保守默认。用户在操作时感受到的是：视口比较稳定，不会因为消息列表内部的重组而剧烈跳动。

## 锚点在何时更新

锚点在用户滚动时随着当前可见窗口的起点更新。用户阅读哪段，锚点就记录哪段的身份和位置。这样即使后续消息列表变化，也能尽力把视口保持在用户上次停留的地方。
