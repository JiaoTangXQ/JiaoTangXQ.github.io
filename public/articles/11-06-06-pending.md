# 被阻塞的 pending 任务要往后站，说明列表先帮助行动再帮助记账

`pending` 不是按编号平铺的。能动的先站前面，被未完成依赖卡住的后站，还会把阻塞原因直接说出来。

## 实现链

`TaskListV2.tsx` 先把所有未完成任务收进 `unresolvedTaskIds`，然后在 `pending` 排序里检查每个任务的 `blockedBy`。只要某个依赖还在这张未完成集合里，这个 `pending` 就会被往后放；渲染时如果 `openBlockers.length > 0`，还会在行尾补上 `blocked by #...`，把阻塞对象显式写出来。

## 普通做法

更普通的任务列表会直接按创建时间或 ID 排序，阻塞只是详情页里的字段，不会影响主列表的先后。

## 为什么不用

Claude Code 更想让用户一眼看出现在还能推进什么。卡住的任务当然不能被藏掉，但它们也不该抢走比真正可行动条目更靠前的位置。

## 代价

代价是顺序不再是纯时间线，而是依赖图和状态图一起参与排序。要读懂某条 `pending` 为什么被往后挪，就得同时看它的依赖和别的任务的完成状态。
