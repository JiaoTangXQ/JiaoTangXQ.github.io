# 安全分析：如果权限检查可以被绕过，会发生什么？

假设你是一个安全审计员，被要求评估 Claude Code 的权限模型。你会从哪里找突破口？

## 攻击面一：bypassPermissions 的准入门槛

最明显的风险点是 `bypassPermissions` 模式。按名字理解，它跳过所有权限检查。如果任何人都能随意进入这个模式，整个权限体系形同虚设。

实际的门控逻辑：

```typescript
export function isBypassPermissionsModeDisabled(): boolean {
  return (
    checkSecurityRestrictionGate('disableBypassPermissionsMode') ||
    isInProtectedNamespace()
  )
}
```

`checkSecurityRestrictionGate` 对接 GrowthBook，可以在运行时远程禁用这个模式。`isInProtectedNamespace()` 检测企业 MDM 或 namespace 保护环境。

当 `isBypassPermissionsModeDisabled()` 返回 true 时，即使代码路径试图设置 bypassPermissions，`applySettingsChange` 也会调用 `createDisabledBypassPermissionsContext()` 把上下文强制退回受限版本。

**风险评估**：这层保护是基础设施级的，不依赖应用逻辑。前提是 GrowthBook 的 feature gate 配置是可信的。

## 攻击面二：dontAsk 语义误解

一个更微妙的风险来自 `dontAsk` 的名称歧义。

如果开发者或用户认为 `dontAsk` 等于"不问就放行"，就可能在 headless/pipeline 场景里主动开启它来"避免打断流程"。但源码里的实际行为是 fail-closed：

```typescript
// permissions.ts 中对 dontAsk 的处理
case 'dontAsk':
  return {
    behavior: 'deny',
    message: DONT_ASK_REJECT_MESSAGE,
    decisionReason: { type: 'mode', mode: 'dontAsk' },
  }
```

`DONT_ASK_REJECT_MESSAGE` 是一条固定的拒绝消息，不是放行。这个模式的真实含义是：**操作没有预先授权？直接拒绝，不要问我**。

**风险评估**：名称本身是一个 UI/UX 风险。如果用户以为开了 dontAsk 就是"免打扰自动运行"，而实际上系统在静默拒绝操作，会产生隐蔽的功能失效。

## 攻击面三：规则注入

权限规则来自多个来源：`userSettings`、`projectSettings`、`localSettings`、`policySettings`、`cliArg`、`session`、`command`。

如果攻击者能在项目的 `projectSettings`（即 `.claude.json`）里注入 `alwaysAllow: ["Bash"]`，就能在目标的工作目录里为自己授权所有 bash 命令。

Claude Code 的对策是：
1. `shouldAllowManagedPermissionRulesOnly()` 检测 policySettings 是否强制要求只用管理员规则
2. 进入 auto 模式时，`findDangerousClassifierPermissions()` 主动扫描现有规则，找出危险的宽松授权并剥离

```typescript
export function isDangerousBashPermission(
  toolName: string,
  ruleContent: string | undefined,
): boolean {
  if (toolName !== BASH_TOOL_NAME) return false
  // 工具级别的 allow（无 ruleContent）意味着允许所有命令
  if (ruleContent === undefined || ruleContent === '') return true
  const content = ruleContent.trim().toLowerCase()
  if (content === '*') return true
  // 检测 python:*, node:* 等解释器前缀模式
  for (const pattern of DANGEROUS_BASH_PATTERNS) {
    if (content === `${pattern}:*`) return true
    if (content === `${pattern}*`) return true
    // ...
  }
  return false
}
```

**风险评估**：规则注入是真实攻击向量（尤其是通过 PR 污染 `.claude.json`）。auto 模式的危险规则剥离提供了一层防御，但 default 模式下，宽松规则仍然生效。

## 攻击面四：权限快照的时效性

`ToolPermissionContext` 是快照，不是实时查询。这意味着：在快照被创建之后、工具调用发生之前，如果规则发生了变化，系统用的仍是旧快照。

这是一个已知的设计取舍。`permissionSetup.ts` 里的 `applySettingsChange` 在检测到 settings 文件变化后会刷新快照：

```typescript
export function applySettingsChange(
  source: SettingSource,
  setAppState: (f: (prev: AppState) => AppState) => void,
): void {
  const updatedRules = loadAllPermissionRulesFromDisk()
  setAppState(prev => {
    let newContext = syncPermissionRulesFromDisk(prev.toolPermissionContext, updatedRules)
    // ...
    return { ...prev, toolPermissionContext: newContext }
  })
}
```

但刷新是异步的。`changeDetector.ts` 用 chokidar 监听文件系统，配了 `stabilityThreshold: 1000ms` 和 `pollInterval: 500ms` 的稳定等待——意味着文件变化后最多要等 1.5 秒快照才更新。

**风险评估**：这个窗口期很短，通常不构成实际风险。但在高频自动化场景里，如果 settings 被程序性修改，短暂的快照过期可能导致规则不一致。

## 治理层为什么不能只靠装配层兜底

从安全审计角度，有人可能建议：把所有 deny 规则都在工具装配时处理，运行时不再检查。这样代码更简单，权限边界也更清晰。

问题在于，装配层只能处理**静态**的权限配置（这个工具根本不该存在于工具池），但运行时的危险来自**动态**上下文——同一条 `Bash(ls *)` 在不同路径下风险完全不同，在工具池装配时无法判断。

双层结构（装配层 + 运行时层）是必须的，而不是冗余设计。这也是为什么 `permissions.ts` 和 `tools.ts` 要分开处理权限，两者解决的是不同维度的问题。

## 这道题考什么

**正确理解 dontAsk**：它是 fail-closed，不是 fail-open。能清楚描述这个语义的候选人说明真正读过代码，而不是靠名字猜测。

**双层权限模型**：能区分装配层（工具可见性）和运行时层（行为合法性），并解释为什么两层都需要存在。

**防御深度**：bypassPermissions 有 GrowthBook 远程门控；auto 模式主动剥离危险规则；settings 变化有稳定窗口再生效。能把这几层防御串起来讲，说明理解了系统的整体安全策略而不是单点机制。
