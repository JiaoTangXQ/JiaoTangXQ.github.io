# 治理不是刹车而是让强能力敢存在

## 实现链

这一页重点看“治理不是刹车而是让强能力敢存在”。对应源码主要是 `src/utils/permissions/permissions.ts`、`src/utils/settings/settings.ts`。
这一章真正落在 `PermissionMode.ts`、`permissionSetup.ts` 和设置变更链上。Claude Code 先把“当前会话处在什么安全姿态”做成正式模式，再让设置变化和工具上下文围着这层姿态重排，而不是等强能力真的要出手时，再靠零散弹窗临时补锅。

## 普通做法

更普通的做法，是把权限做成分散的确认框：危险动作来了就弹一次，没有动作就什么也不说。这样实现最短，也最符合很多命令行工具的直觉。

## 为什么不用

Claude Code 没采用这种临场拍板路线，因为它要管理的是一套会持续调用工具、连续推进任务的 agent 运行时。要是只靠零散弹窗，系统永远拿不到一个稳定的“当前安全姿态”，后面的工具筛选、模式切换和设置传播都会各说各话。

## 代价

这种制度化姿态更稳，能让强能力先有可承重的骨架再出现。代价是模式、限制和变更程序会比普通 CLI 厚很多，用户第一次接触时也更容易觉得系统在“小题大做”。
