# teammateModeSnapshot.ts：为什么用快照而不是实时读取 teammateMode？

`teammateModeSnapshot.ts` 的内容非常简单：

```typescript
let capturedTeammateMode: TeammateMode | undefined

export function captureTeammateModeSnapshot(mode: TeammateMode): void {
  capturedTeammateMode = mode
}

export function getTeammateModeFromSnapshot(): TeammateMode | undefined {
  return capturedTeammateMode
}
```

这是一个捕获-读取模式。启动时调用 `captureTeammateModeSnapshot`，之后所有查询调用 `getTeammateModeFromSnapshot`，返回同一个值。

为什么不直接在需要时实时计算 `teammateMode`？

## teammateMode 是什么，影响什么

`teammateMode` 决定了多 agent 功能使用哪种执行方式：

- `'tmux'`：每个 teammate 在独立的 tmux pane 里运行
- `'iterm2'`：每个 teammate 在 iTerm2 的标签页/pane 里运行
- `'in-process'`：所有 teammate 在同一个 Node.js 进程里，通过 AsyncLocalStorage 隔离
- `'auto'`：自动探测，优先选 pane 方式，不可用时回退到 in-process

这个值影响：

1. **UI 菜单显示**：哪些选项对用户可见（在 tmux 模式下显示 tmux 相关的说明，在 in-process 模式下显示进程内协作说明）
2. **功能可用性**：`supportsHideShow`（pane 隐藏/显示）只在 pane 模式下可用
3. **系统提示生成**：`teammatePromptAddendum.ts` 可能根据模式生成不同的 agent 系统提示

## 如果实时计算 teammateMode 会怎样

实时计算意味着每次调用都重新检测环境：检查环境变量、检查 tmux 是否可用、检查是否在 iTerm2 里。

这本身没问题，检测逻辑可以被实现得很快。问题在于：**同一个会话里，先后两次调用 `getTeammateMode()` 可能返回不同的值**。

什么时候会发生？

- 用户在会话中途关闭了 tmux，打开了 iTerm2
- 用户通过 `claude config set teammateMode in-process` 修改了配置
- 环境变量被外部脚本修改了

如果这发生了，系统里的不同部分会看到不同的 `teammateMode`：

- UI 组件（在某个时间点读取）：认为是 tmux 模式，显示了 tmux 相关选项
- 后端选择逻辑（在另一个时间点读取）：发现是 in-process 模式，不再创建 tmux pane
- 权限处理逻辑（又一个时间点）：认为是 tmux 模式，走了 pane 相关的权限路径

这种"系统的不同部分在不同时间点对同一个状态有不同认知"，是多组件系统里最难调试的 bug 来源之一。表现是：某些功能时而可用时而不可用，某些 UI 选项莫名出现或消失，权限请求路径随机走错。

## 快照的核心价值：把不确定性封闭在启动阶段

`captureTeammateModeSnapshot` 在进程启动时被调用一次，之后 `capturedTeammateMode` 就是不变的。

系统里所有依赖 `teammateMode` 的地方都调用 `getTeammateModeFromSnapshot()`，拿到的是同一个值，永远是同一个值。

这把"团队模式是什么"这个不确定性，从"可能在任何时间点改变"变成了"确定地是启动时的那个值"。

不确定性没有消除（启动时用什么模式仍然取决于环境），但它被封闭在了一个单一时刻（进程启动），而不是散布在整个会话里。

## 同样的模式在 AWS Lambda 里很常见

快照模式在 serverless 函数里极为常见：

```javascript
// Lambda 函数的初始化区域（容器重用时只执行一次）
const dbConnection = createDatabaseConnection(process.env.DB_URL)
const config = loadConfig()

// 处理函数（每次请求都执行）
exports.handler = async (event) => {
  // 复用 dbConnection 和 config，不重新初始化
  const result = await dbConnection.query(config.table, event.id)
  return result
}
```

`dbConnection` 和 `config` 在容器初始化时捕获一次，后续所有请求复用。这不只是性能优化（避免每次请求都重新连接数据库），也是稳定性保证（所有请求都看到相同的配置）。

Claude Code 的 `teammateModeSnapshot` 解决的是同一类问题，只是"请求"换成了"会话期间的各种操作"。

## 快照模式的局限

快照的代价是：**会话中途无法切换模式**。

如果用户在启动 Claude Code 时选择了 tmux 模式，但后来发现 tmux pane 布局不适合当前任务，希望切换到 in-process 模式——无法做到，需要重启整个会话。

这是刻意接受的代价。Claude Code 认为：一个会话里模式稳定一致，比能随时切换模式更重要。用户更能理解"重启后生效"的规则，而不是"某些时候可以切换，某些时候不行，切换后会有哪些影响不确定"。

## 面试怎么答

如果面试题是"什么时候应该用快照而不是实时计算"：

**快照适合的场景**：
1. 多个系统组件需要在同一时间点看到一致的值
2. 值的变化会导致系统状态的全局重置（代价高昂）
3. 值的获取有副作用或代价（检测环境需要 I/O）

**实时计算适合的场景**：
1. 只有一个地方需要这个值，没有多组件一致性问题
2. 值变化时系统能平滑过渡，不产生不一致状态
3. 值的变化是主要功能的一部分（比如"热更新"）

**工程原则**：把系统的不确定性封闭在边界点（启动、事务开始、请求开始），而不是让不确定性散布在系统的整个生命周期里。快照是实现这个原则的一种具体机制。
