# 故障恢复推演：历史分页失败时，cursor 为什么不能清空？

## 两种终止条件

历史分页加载可以有两种终止条件：

1. **正常终止**：服务端返回"没有更多了"（`has_more: false`），这意味着历史真的到头了，所有消息都已经加载
2. **异常终止**：网络请求失败（超时、连接断开、服务端 5xx），这意味着加载被打断，历史可能还有更多

这两种情况，用户需要看到的提示和行为是完全不同的：

- 正常终止 → 显示"会话起点"，不再触发翻页加载
- 异常终止 → 显示"加载失败，滚动到顶部重试"，保留重试能力

如果把两种情况用同一个状态表示（比如都清空 cursor，都显示"没有更多了"），用户会在网络临时故障后，误以为会话历史只有这么多——这是信息的失真。

## cursor 的状态语义

`cursor` 是历史分页的翻页凭证：服务端返回一个 cursor 值，下次请求带上这个值就能获取前一页的内容。

cursor 的状态语义：

- `cursor` 有值 → 还有更多历史，可以继续翻页
- `cursor` 为 null（服务端返回 `has_more: false`）→ 已到会话起点

如果请求失败时清空 cursor，cursor 变成 null，组件会认为"没有更多历史了"，停止尝试加载。这是用错误状态（网络失败）覆盖了原本的有效状态（还有更多内容）。

## 失败时的正确处理

`useAssistantHistory.ts` 在请求失败时的处理：

```typescript
async function fetchOlderEvents() {
  try {
    const result = await loadHistoryPage(cursor)
    if (result.has_more) {
      setCursor(result.next_cursor)  // 更新 cursor
      setSentinel('loading')          // 可以继续加载
    } else {
      setCursor(null)                 // 真的到头了
      setSentinel('session_start')    // 显示会话起点
    }
  } catch (error) {
    // 关键：不清空 cursor
    // cursor 保持原值，代表"还有更多内容，只是这次没加载到"
    setSentinel('failed')             // 显示失败状态，允许重试
  }
}
```

失败后：
- sentinel 变成"failed to load older messages — scroll up to retry"
- cursor 保持不变（还有更多内容的凭证还在）
- 当用户再次滚动到顶部，触发 `fetchOlderEvents`，用原来的 cursor 重试

这个设计让失败是可以恢复的（retryable），而不是终局（terminal）。

## 网络失败的频率

这个设计决策的合理性，还要考虑失败的预期频率。

历史分页的请求失败原因通常是：
- 短暂的网络抖动
- 服务端临时不可用（部署、维护）
- 用户切换网络（WiFi 到 4G）

这些都是临时性的失败，大概率在用户再次尝试时（重新滚到顶部）已经恢复了。

如果把临时性失败当成"永久性到头"处理，用户每次网络抖动后都需要刷新页面才能重试——这是对用户的惩罚，而不是对错误的正确处理。

## 与服务端重试的对比

这个模式和 HTTP 请求的重试策略有类似的结构：

- 5xx 错误（服务端问题）→ 临时失败，可重试
- 4xx 错误（客户端请求问题）→ 永久失败，不应重试
- 200 但 `has_more: false` → 正常终止，不需要重试

cursor 失败时保留，对应的是"5xx → 临时失败，可重试"这个逻辑——我们判断这次失败是临时的，所以保留重试能力。

## 「sentinel」承担了多重语义

sentinel 在不同状态下承担了不同的语义：

| 状态 | sentinel 显示 | 用户操作 |
|------|-------------|--------|
| 有更多历史 | "正在加载..." | 等待（自动触发） |
| 加载失败 | "加载失败，上滑重试" | 主动上滑触发重试 |
| 已到起点 | "会话于 [时间] 开始" | 没有更多操作 |

这三种状态覆盖了历史加载的全部情况，用户在任何时候都能从 sentinel 的显示知道当前处于哪种情况、下一步能做什么。

## 面试指导

"如何区分可恢复错误和不可恢复错误"是错误处理设计的核心问题。

判断框架：

1. **错误的根本原因是什么？**外部临时问题（网络）vs 内在不变问题（数据不存在）
2. **重试能解决吗？**如果错误的根本原因是暂时的，重试有意义；如果是永久的（404），重试没有意义
3. **如何区分两种情况？**用不同的状态标记，而不是用同一个终止状态掩盖所有错误

"cursor 失败时保留"是一个具体的设计决策，体现了"不要用错误状态覆盖有效状态"的原则——保留了重试能力，没有假装问题不存在。
