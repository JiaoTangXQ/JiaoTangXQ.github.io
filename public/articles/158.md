# 深度分析：TaskStateBase 的 10 个字段为什么是最小完备集

## 重新审视这 10 个字段

```typescript
export type TaskStateBase = {
  id: string
  type: TaskType
  status: TaskStatus
  description: string
  toolUseId?: string
  startTime: number
  endTime?: number
  totalPausedMs?: number
  outputFile: string
  outputOffset: number
  notified: boolean
}
```

这些字段可以分成四组，每组服务一个不同的系统关注点：

**身份组**：`id`、`type`
- 系统需要能唯一识别每个任务（`id`）
- 系统需要能知道这个任务是什么种类（`type`），以便分发到对应的处理器

**生命周期组**：`status`、`startTime`、`endTime`、`totalPausedMs`
- 系统需要知道任务现在在哪个阶段（`status`）
- 系统需要知道任务开始了多久（`startTime`、`endTime`）
- 长时间任务可能被暂停，需要正确计算实际运行时间（`totalPausedMs`）

**输出组**：`outputFile`、`outputOffset`
- 系统需要知道任务的输出在哪里（`outputFile`）
- 系统需要知道已经读取到哪里了（`outputOffset`）

**元数据组**：`description`、`toolUseId`、`notified`
- 系统需要能展示人可读的任务描述（`description`）
- 系统需要能把任务和发起它的工具调用关联起来（`toolUseId`）
- 系统需要知道是否已通知相关方（`notified`）

## 为什么这是最小完备集

**缺少任何一个字段会发生什么：**

没有 `id`：无法在 AppState 里唯一定位一个任务，更新和删除操作要遍历所有任务比较其他字段。

没有 `status`：无法知道任务在哪个阶段，清理逻辑不知道哪些任务可以删除，通知逻辑不知道什么时候发通知。

没有 `outputFile`：任务的输出只能通过内存中的引用访问，进程重启后输出丢失，无法跨进程共享任务进度。

没有 `notified`：任务完成后可能发送重复通知，或者遗漏通知——没有标志来标记"已经通知过了"。

**多出一些字段会有什么问题：**

如果把 `AgentProgress`（进度追踪）也放进 `TaskStateBase`，那么不需要进度追踪的任务（比如简单的 bash 命令）也要携带这些字段，增加内存占用。更重要的是，`AgentProgress` 是 agent 类型任务的专属特征，放进公共底座会让公共底座承载了非公共的概念。

## toolUseId 的角色

`toolUseId` 字段有点特殊——它是可选的（`?`），因为并非所有任务都由工具调用发起。

但当它存在时，它建立了一个非常重要的关联：**这个任务是由哪个工具调用发起的**。

这个关联让任务完成时能正确路由结果：当 `AgentTool` 的工具调用触发了一个 `local_agent` 类型的任务，任务完成后需要把结果作为这个工具调用的返回值注入对话历史。`toolUseId` 就是这个路由的依据。

没有这个字段，任务的结果无法和发起它的工具调用关联，AI 在等待工具调用结果时会超时或收到错误。

## endTime 和 totalPausedMs 的精确性

为什么需要两个时间字段来描述任务结束？

`endTime - startTime` 是**挂钟时间**（wall clock time）：任务从开始到结束的实际时间，包括可能的暂停时间。

`(endTime - startTime) - totalPausedMs` 是**实际执行时间**：扣除暂停后的净执行时间。

对于监控和计费场景，需要精确的执行时间，不是挂钟时间。如果一个任务运行了 10 分钟，中间有 7 分钟在等待用户输入（暂停），实际执行时间是 3 分钟。按 10 分钟计费对用户不公平，按 3 分钟计费才准确。

`totalPausedMs` 让这个精确计算成为可能，这是"公共字段里的工程诚实"：宁愿多一个字段，也不让时间计算不准确。

## 面试要点

1. **10 个字段是最小完备集**：每个字段服务一个特定的系统关注点（身份、生命周期、输出、元数据），少一个就会有功能缺口。
2. **toolUseId 建立任务和调用的关联**：这是结果路由的依据，任务完成后能找到"把结果给谁"的关键。
3. **totalPausedMs 保证时间计算精确**：挂钟时间和执行时间的区别在计费和监控里是重要的，公共底座直接支持这个区分。
4. **公共字段不承载专属特征**：AgentProgress 这样的专属特征不放进公共底座，保持公共底座的纯粹性。
