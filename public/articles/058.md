# 跨越本地/远端边界时，Claude Code 为什么要重算权限而不是复用本地权限？

这是一个容易被忽略的安全问题：用户在本地设置了「允许读取 `~/projects/` 下的文件」，现在连接到远端 CCR 容器，这条规则应该直接适用于远端吗？

Claude Code 的答案是：不应该，要重算。

## 权限规则的上下文依赖

本地的权限规则是在本地上下文里定义的：
- 文件路径规则（`~/projects/`）在远端机器上可能指向完全不同的目录
- 本地允许的网络访问在远端容器里可能有完全不同的风险等级
- 本地的 bash 命令规则在远端可能触发不同的系统资源

直接把本地规则「复用」到远端，等于用了错误的上下文做安全判断。

## bridge 的作用：而不只是转发

`SessionsWebSocket` 处理本地客户端和远端会话之间的 WebSocket 通信。它不是简单的消息转发器——它维护连接状态、处理重连、并且在消息类型上做严格的类型判断：

```typescript
function isSDKMessage(message: SDKMessage | SDKControlRequest | ...): message is SDKMessage {
  return (
    message.type !== 'control_request' &&
    message.type !== 'control_response' &&
    message.type !== 'control_cancel_request'
  )
}
```

这个类型守卫很重要：控制类消息（权限请求/响应）和普通 SDK 消息走不同的处理路径。控制消息不会被当作普通对话消息处理，避免了「权限请求被注入到对话流里欺骗用户」的攻击面。

## 远端权限请求的结构

当远端容器需要权限时，它发送 `SDKControlPermissionRequest`：

```typescript
// 远端权限请求包含的信息
{
  type: 'control_request',
  tool_use_id: string,     // 在远端执行的工具调用 ID
  tool_name: string,       // 工具名称
  input: Record<string, unknown>,  // 工具的输入参数
  requestId: string        // 请求 ID，用于匹配响应
}
```

本地客户端收到这个请求后，不能直接应用本地的 `ToolPermissionContext`（因为本地上下文里根本没有这个工具调用记录），而是通过 `createSyntheticAssistantMessage` 创建一条「合成的」助手消息，然后走本地的 `ToolUseConfirm` 流程。

这个「合成」的关键意义是：用户看到的确认界面和本地工具调用完全一样的界面。用户不需要知道这个工具在远端执行——他们需要知道的是「AI 想要做什么操作，我是否批准」。

## 环境敏感信息的隔离

`RemoteSessionConfig` 里有一个不显眼但重要的设计：本地客户端不向远端传递本地的 `ToolPermissionContext`。这意味着：

1. 本地的权限规则（哪些工具被允许、哪些被禁止）**不会**自动同步到远端
2. 远端容器的权限规则由远端环境决定
3. 本地客户端只做「展示给用户，收集用户决定」这一件事

这是一种「最小传输」原则：本地只传递真正需要的信息（用户输入、控制信号），不传递安全敏感的配置。

## viewerOnly 的边界

`viewerOnly` 模式下，本地客户端放弃了中断能力：Ctrl+C 不再发送 interrupt 信号给远端 agent。这看似降低了用户的控制权，但有其合理性：

- `claude assistant` 场景下，用户连接到一个正在执行任务的 agent，打断它可能造成任务数据损坏
- viewerOnly 明确传达了「你来这里是观察，不是控制」的语义
- 如果需要停止，应该通过显式的「终止会话」操作，而不是 Ctrl+C

这是一个「降低操作风险」的设计：把危险操作从容易误触的快捷键移除，放到需要明确意图的操作路径上。

## 面试答题思路

「系统如何处理跨边界的权限问题」：

1. **上下文决定规则的有效性**：一条权限规则只在它被定义的上下文里有意义
2. **重算 > 复用**：跨越边界时宁可多算一次，不要假设规则可以迁移
3. **用户决策界面保持一致**：无论工具在哪里执行，用户看到的确认界面应该一样
4. **最小传输原则**：不传递超出必要的安全配置到远端
5. **操作风险分级**：高风险操作（如中断执行）在某些模式下需要降低可及性

追问「如果本地和远端的权限规则冲突怎么办」：远端容器有自己的权限规则，本地客户端有自己的。实际上它们是独立的判断层次：远端先判断操作是否被容器政策允许，然后把需要用户确认的请求发到本地，本地再让用户做最终决定。两层都通过，操作才执行。
