# preAction 前后各自承担什么职责

`preAction` 最妙的地方，不只是把重活往后推了一步，而是顺手把“命令世界”和“运行时世界”切成了两层。闸门之前，Commander 负责的是让程序先像个像样的命令行工具: 它要把命令名、参数、帮助文本、隐藏命令、选项结构先搭出来，让用户能问路、能看地图、能知道这台机器大概会做什么。

闸门之后，程序才正式承认: 现在不是在介绍自己，而是真的要开始行动了。于是前面顶层抢跑的设置来源先收口，`init()` 真正启动，进程标题开始设置，日志 sink 接上，内联插件目录被注入，迁移被执行，远端设置和 policy limit 被异步拉起，设置同步也在这里挂上后台。

这样分工有个非常大的好处: 外层可以保持轻，内层可以保持真。帮助信息不必背着一整台运行时前进，真正的副作用也不会偷偷混进“只是看看命令说明”的路径里。Claude Code 在这里体现出来的不是小技巧，而是一种很成熟的系统边界感: 先让用户看懂门牌，再决定要不要真的开门进去。

## 实现链
这页对应的实现链都落在 `preAction` 前后的职责切分上。代码把必须早做但很轻的判断留在闸门前，把完整环境、交互世界和重副作用留到闸门后，借这道线把不同成本的路径分开。

## 普通做法
更直觉的做法是把所有前置工作都塞进同一个大 handler，先后顺序只靠注释维持。

## 为什么不用那种做法
Claude Code 不这么混着写，是因为它既要处理帮助/版本这类轻路径，又要处理完整会话这类重路径。如果没有这道闸门，轻路径会一直给重路径买单。

## 代价与回报
回报是不同路径各付各的成本。代价是控制流被切成多阶段，入口阅读会更费心。
