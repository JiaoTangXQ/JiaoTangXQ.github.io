# tool progress 为什么会被压成 system message？

在 Claude Code 的远端执行层，有一个让人困惑的设计：远端 session 发来的 `tool progress` 事件，不会以原始格式进入主会话的消息流，而是被翻译成 `system message`。

这个翻译损耗是主动设计，不是懒惰的快捷方式。

## 两种消息世界

远端执行时，系统里存在两套消息世界：

1. **远端 session 的事件流**：工具执行进度、中间输出、stream 事件
2. **主会话的消息流**：对话历史、tool use、tool result、system message

这两套消息世界不是同一件事。远端 session 产生的事件是流式的、高频的、格式多样的；主会话的消息流是结构化的、有序的、每一条消息都有明确语义。

如果把远端事件流直接注入主会话消息流，两件事会同时出问题：

- **主会话的历史会被污染**：大量的进度事件和流式输出会破坏 LLM 读取历史时的结构清晰度
- **语义会丢失**：`tool progress` 和 `tool result` 是两个不同的语义，混在一起会让系统分不清"工具还在跑"和"工具跑完了"

## 翻译的代价和收益

把 `tool progress` 压成 `system message` 是有代价的：进度事件的详细内容会被折叠，一些实时性信息会丢失。这是真实的翻译损耗。

但保住的是更重要的东西：**统一消息世界**。

统一消息世界意味着系统中任何一个消费消息的组件——无论是 LLM、UI 层、历史记录还是重放逻辑——都可以用同一套类型定义来理解消息。`system message` 是主会话消息流里已经存在的类型，每个组件都知道怎么处理它。

如果为远端 session 引入一种新的消息类型 `remote_progress`，那么每个消费消息的地方都需要多处理一种类型。规模小的时候这不是问题，但一旦系统里有多种远端 session，就会出现多套口音各说各话的局面。

## 接缝零件的功力

"宁可接受翻译损耗，也要保住统一消息世界"这个判断，体现了一种很具体的工程分寸感：**接缝零件的功力不是让两边无缝对接，而是让两边即使有差距也能继续工作**。

桥接层翻译 `tool progress` 的地方，本质上是在问：这份信息，哪些部分是必须进入主会话的，哪些部分只是过程噪音？

进度事件的"什么时候发生、发生了什么"可能是过程噪音；"工具已经产生了某种输出"是需要被记录的事实。翻译时保留事实，丢弃噪音，这不是懒，是判断。

## bundle 边界的另一面

参数注入的另一端也是类似的逻辑：远端层不能随手把原始参数透传进 bundle，不是因为实现困难，而是因为一旦 bundle 边界被穿透，统一体验就会先破。

bundle 边界保住的是：无论参数从哪里来，进入 bundle 之后的处理逻辑是一致的。这让 bundle 可以独立测试、独立升级，不用担心远端参数带来的意外副作用。

这两个地方——消息翻译和参数注入——都在同一个地方做判断：**跨边界时，不是透传，而是翻译；不是统一，而是承认差异后再重新统一**。

## 面试答题要点

被问到"你们的远端事件怎么进入主会话"时，不好的答案是："我们直接把事件流接进去。"

好的答案是："我们在边界做了翻译。远端的进度事件不适合直接进入主会话的消息结构，翻译成 system message 会损失一些细节，但保住了消息世界的统一性。消费消息的所有组件不需要知道这条 system message 来自远端——它只需要能正确处理 system message 这个类型。"

追问"为什么不为远端引入新的消息类型"时：每引入一种新类型，就要在系统的每个消费点增加一个处理分支。翻译损耗是局部的，类型蔓延是系统性的。
