# buddy 接进了三条线：UI、提示词、上下文附件

`buddy` 要正常工作，需要三件事同时发生：

1. **用户能看到它**（UI 层）
2. **模型知道它存在**（提示词层）
3. **会话记住它**（上下文附件层）

这三件事各自由不同的代码处理，各司其职。

## UI 层：PromptInput.tsx

当用户输入 `/buddy` 时，输入区给这个命令做彩虹高亮（渐变色），把它和普通文本区分开来。

同时，`companionReservedColumns()` 计算 buddy 精灵需要占用多少列，`PromptInput.tsx` 在布局时把这部分列数从输入区宽度里减掉。这样精灵出现时，输入框不会被挤压变形。

## 提示词层：buddy/prompt.ts

`companionIntroText()` 生成一段说明，告诉主模型：「有一个叫 [名字] 的伙伴在旁边，但它不是你，你是 Claude，当用户直接叫它名字时简短回应就好」。

这段说明通过 `getCompanionIntroAttachment()` 包装，准备注入。

## 上下文附件层：utils/attachments.ts + utils/messages.ts

`companion_intro` 附件通过消息系统的附件机制进入会话上下文。`utils/messages.ts` 把它处理成一条系统级别的补充消息，在每次模型调用时带进去。

但只带一次——`getCompanionIntroAttachment(messages)` 会先扫描 messages 里是否已经有同名的 `companion_intro`，如果已经存在，就不再重复注入。

## 三条线是一个整体

只有 UI 而没有提示词：用户看到了精灵，但模型不知道它是谁，用户叫它名字时模型不知道怎么回应。

只有提示词而没有 UI：模型知道有个伙伴，但用户看不到，整体体验是虚空的。

只有 UI 和提示词而没有附件机制：每次新对话模型都「忘了」伙伴的存在，每次都需要重新介绍。

三条线同时工作，才构成一个完整的、持续的 buddy 体验。
