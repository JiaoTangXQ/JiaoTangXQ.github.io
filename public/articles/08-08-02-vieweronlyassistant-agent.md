# 对比分析：viewerOnly 模式关掉了哪些能力，为什么是刻意的？

## 三项被关闭的能力

`RemoteSessionConfig` 里的 `viewerOnly: true` 标志，会触发三个与普通远端控制模式不同的行为：

### 1. 关闭本地 interrupt

在普通远端控制模式下，本地用户可以发送 interrupt 信号（Ctrl+C 等价）来中止远端 Agent 的执行。

viewerOnly 模式关闭了这个能力。原因很直接：**观看者没有权力中止会话**。这个会话可能是另一个用户发起的，或者是自动化任务在运行，观看者只有观看权限。

如果不关闭，任何连接 viewerOnly 的客户端都可以中止正在运行的会话，这会把"只读访问"变成隐蔽的"写入访问"——一个严重的权限越界。

### 2. 关闭 60 秒不响应重连超时

普通远端控制有一个保护机制：如果 60 秒内没有收到任何消息（包括心跳），判断远端可能卡死，触发重连。这个机制是为了应对"远端 Agent 挂死但 WebSocket 没断"的情况。

viewerOnly 关闭了这个超时。原因：**被观看的会话可能处于等待用户输入的空闲状态**，几分钟甚至更长时间不产生任何消息是正常的。

如果 viewerOnly 保留 60 秒超时，观看者会在会话空闲时被迫断开，需要重新连接——这破坏了"持续观看"的体验。更糟的是，每次重连都会触发历史重新加载，消息列表会出现抖动。

### 3. 不更新远端标题

普通远端控制模式下，本地会在检测到第一条有意义的用户消息时，自动把会话标题更新到服务端（PATCH `/v1/sessions/{id}`）。

viewerOnly 不做这个更新。原因：**观看者不应该修改被观看会话的元数据**。标题更新是一个写操作，会影响在 claude.ai 上看到的会话卡片展示。观看者更新了标题，会话发起者可能感到困惑。

## 角色定义的技术实现

这三项关闭，合在一起，构成了"观看者"这个角色的技术实现：

| 能力 | 普通远端控制 | viewerOnly |
|------|------------|------------|
| 发送 interrupt | 有 | 无（角色不允许） |
| 60s 无响应超时 | 有（保护机制） | 无（空闲是正常的） |
| 更新标题 | 有 | 无（不能写元数据） |
| 拉取历史 | 可选 | 有（主要功能） |
| 订阅实时流 | 有 | 有 |

"观看者有观看权但没有控制权"——这个角色边界在代码层面被精确地实现为三项关闭。

## 为什么不靠服务端权限控制？

有人可能会问：为什么不让客户端发 interrupt，由服务端来检查权限并拒绝？

这是一个值得讨论的设计选择。客户端侧关闭的理由：

1. **减少无效网络请求**：客户端已经知道自己是观看者，发 interrupt 只会被服务端拒绝，白白浪费一次网络往返。
2. **UI 的诚实性**：如果 interrupt 按钮对观看者显示出来，但点击后被服务端拒绝，用户体验是困惑的。客户端侧关闭意味着 UI 根本不显示这个选项，角色边界在 UI 上是清晰的。
3. **防御性设计**：两层检查（客户端 + 服务端）比只有一层更安全。即使客户端出 bug，服务端还有拦截。

## 与 cursor 模式的关联

`08-08-06-cursor.md` 里的 cursor 模式（`cursor: true`）是 viewerOnly 的一个扩展：不只是观看，还显示对方的光标位置（实时协作的感觉）。

cursor 模式在 viewerOnly 的基础上，增加了"追随另一个用户的视角"的功能，但仍然保持了"不能控制"的约束。这进一步说明 viewerOnly 是一个有意设计的基础模式，cursor 是基于它的扩展。

## 面试指导

"如何在多用户系统里实现不同的权限角色"是系统设计面试的常见题。

viewerOnly 的实现给出了一个很好的参考模式：

1. **用配置（标志位）而不是代码分叉来实现角色差异**：`viewerOnly: true` 是一个条件，不是一套独立的代码路径
2. **角色权限在多个维度上体现**：能否 interrupt、能否更新元数据、超时行为——每个维度单独处理
3. **客户端和服务端双重控制**：客户端不显示不应该有的能力（UX），服务端拒绝越权请求（安全）

把"角色"翻译成具体的"禁用某些操作+调整某些行为"，是实现角色系统时的核心工作，比直接在代码里硬编码角色判断更灵活。
