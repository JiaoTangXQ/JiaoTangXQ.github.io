# 面试题：UI 组件可以直接修改状态吗？Claude Code 的答案和 MVC 有什么不同？

这个问题的答案不是「可以」或「不可以」，而是「它取决于你对 UI 这一层的定位」。

Claude Code 对这个问题有一个非常清晰的立场，值得仔细读。

---

## 传统 MVC 的立场

经典 MVC 里，View 层是被动的：

- Model 保存数据，View 只读取并呈现
- 用户操作触发 Controller，Controller 修改 Model
- View 通过数据绑定或事件订阅更新

View 不「拥有」数据，它只是数据的镜像。

这个模型在桌面应用时代运作良好，因为数据源头是确定的——就是 Model 层。

---

## Claude Code 里 UI 承担了什么

看 `AppState` 里的这些字段：

```ts
// AppStateStore.ts
expandedView: 'none' | 'tasks' | 'teammates'
footerSelection: FooterItem | null
coordinatorTaskIndex: number
viewSelectionMode: 'none' | 'selecting-agent' | 'viewing-agent'
spinnerTip?: string
```

这些字段——展开哪个视图、底栏当前聚焦哪个 pill、协调者任务的选中索引——明显是 UI 状态。它们描述的是「界面现在长什么样」，不是「业务发生了什么」。

但它们和这些字段住在同一个 `AppState` 里：

```ts
toolPermissionContext: ToolPermissionContext   // 权限配置
tasks: { [taskId: string]: TaskState }         // 运行中任务
remoteConnectionStatus: 'connecting' | 'connected' | ...  // 远端连接
mcp: { clients, tools, commands, resources }   // MCP 连接状态
```

为什么 UI 状态和业务状态混住在一起？

---

## 因为「UI 状态」本身就是业务事实

`footerSelection` 这个字段决定键盘输入该路由给谁。如果底栏当前选中的是 bridge pill，那么方向键操作就是在操作 bridge 相关的东西，而不是任务列表。

这不是 UI 的样式问题，而是**当前操作上下文**的事实。

`coordinatorTaskIndex` 在注释里说明了为什么要住进 AppState：

```ts
// AppState (not local) so the panel can read it directly without prop-drilling
// through PromptInput → PromptInputFooter.
```

如果放在本地 state 里，就需要从 `PromptInput` 一级一级向下 prop drill，而渲染在别处的组件就无法读取这个值。住进 AppState，所有订阅者都能直接取。

---

## 那 UI 层能直接修改 AppState 吗？

能，而且应该。

`useSetAppState()` 直接返回 store 的 `setState`：

```ts
export function useSetAppState() {
  return useAppStore().setState
}
```

用户按下箭头键切换底栏 pill，组件直接调用：

```ts
const setAppState = useSetAppState()
setAppState(prev => ({ ...prev, footerSelection: nextPill }))
```

没有 action creator，没有 dispatch，没有 reducer。组件直接写。

---

## 这不是「UI 拥有状态」，而是「UI 是运行时的参与者」

两者的区别在于：

**UI 拥有状态**的模式里，UI 层维护自己的本地副本，状态的真相散落在各处。

**UI 是参与者**的模式里，所有人——UI 组件、后台任务、远端推送、命令处理器——都通过同一个接口写入同一份现实。UI 不是状态的主人，而是许多修改者之一。

```ts
// 任何来源都能修改 AppState
store.setState(prev => newState)  // UI 组件
store.setState(prev => newState)  // 后台任务完成回调
store.setState(prev => newState)  // 远端 WebSocket 推送
store.setState(prev => newState)  // 命令处理器
```

修改接口是统一的，修改者是多元的，状态的真相是唯一的。

---

## 和 Redux 的对比

Redux 里，修改状态需要：`dispatch(action)` → reducer 处理 → 新 state

好处是：每次状态变化都有明确的 action 类型，可以追踪、回放、做时间旅行。

代价是：每次读写都要写 action creator 和 reducer 样板代码，对于「当前底栏选中哪个 pill」这种纯 UI 状态来说是过度设计。

Claude Code 的方案更直接：`setState(updater)` 一步到位，副作用通过 `onChangeAppState` 统一处理，不需要 action 类型。

---

## 面试指导

**UI 状态应该放在哪里？**

判断标准：如果这个状态只有一个组件关心，放本地 state。如果多个组件需要读取同一个值（尤其是跨越 prop drill 层级的），放 AppState。如果这个状态变化需要触发副作用（写配置、通知远端），也放 AppState——`onChangeAppState` 会统一处理。

**为什么不用 useState 管 UI 状态，用 Context 向下传？**

Context 值一变，所有消费者都重渲染，不管他们是否关心这个变化。AppState + selector 模式让每个消费者只订阅自己关心的切片，避免无谓重渲染。这对一个状态字段很多的大型应用来说非常重要。
