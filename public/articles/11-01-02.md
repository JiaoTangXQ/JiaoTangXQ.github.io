# 界面不是外壳，而是运行时的一部分

Claude Code 的界面不是把后台结果翻译给人看那么简单。很多运行时事实只有进了界面才真正成立，比如当前谁在等权限、哪个任务被前置、现在看的是本地还是远端、团队里谁在动。

所以它的 UI 不是贴在外面的一层皮，而是运行时本身的一部分。没有这层状态组织，后面的能力就算存在，人也无法真正驾驶它。

## 实现链

从 `App.tsx` 的装配顺序就能看出来，界面不是最后一层皮，而是运行时的一部分：事实先住进 `AppStateProvider`，统计和 FPS 观测又故意放在外层，后面的 `PromptInput`、状态栏、面板和通知都靠这套共享现实工作。UI 在这里不是“把结果画出来”，而是在组织整个会话现场。

## 普通做法

更常见的 CLI UI 写法，是让界面只消费结果，真正的运行时活在命令循环或后端对象里，前端只是打印和接收按键。

## 为什么不用

Claude Code 不走那条路，是因为它的很多制度根本离不开界面层配合：权限模式要被看见，任务要被跟踪，远端事实要被承认，输入区还要承担控制面职责。把 UI 降成外壳，这些机制就会断层。

## 代价

代价是前端代码不再是“薄展示层”，而是直接背负运行时概念；读者必须同时理解状态、观测和交互三层，心智负担更重。
