# Claude Code 启动的前 200ms 里到底发生了什么？

时间戳 T=0：用户按下回车键。

以下是 Claude Code 在你看到任何输出之前，悄悄发生的事情。

## T=0ms：进程启动

Bun 运行时加载单文件可执行包。这是不可缩短的系统开销，几乎为零（Bun 的启动比 Node.js 快得多）。

## T=1ms：`profileCheckpoint('main_tsx_entry')` 触发

这是 `main.tsx` 文件中最早执行的一行代码，在所有 import 语句之前。它记录一个时间戳，用于后续计算各阶段耗时。

```typescript
profileCheckpoint('main_tsx_entry');
```

这不是简单的 `console.time()`。`startupProfiler` 模块会把这些检查点串联起来，最终可以输出类似"从进程启动到 init 完成耗时 183ms，其中模块加载 91ms，MDM 读取 47ms..."这样的细粒度报告。

## T=1ms：`startMdmRawRead()` 触发

在导入下一个 import 之前，MDM（移动设备管理）子进程就已经启动了：

```typescript
import { startMdmRawRead } from './utils/settings/mdm/rawRead.js';
startMdmRawRead();
```

在 macOS 上，这会执行 `plutil -extract` 来读取企业管理配置。这个系统调用需要经过 macOS 的 MDM 守护进程，延迟不可控，通常在 20-100ms 之间。

关键在于：这个调用是 fire-and-forget 的。主线程不等它，继续加载后面的模块。

## T=2ms：`startKeychainPrefetch()` 触发

紧接着，keychain 预取也启动了：

```typescript
import { ensureKeychainPrefetchCompleted, startKeychainPrefetch } from './utils/secureStorage/keychainPrefetch.js';
startKeychainPrefetch();
```

这同时发射两个 keychain 读取请求：
- OAuth token 路径
- 旧版 API key 路径

macOS Keychain 访问需要通过 Security framework 的 IPC 调用，每次大约 30-65ms。如果串行读取，就是 60-130ms 的死等。并行启动后，总时间接近单次读取时间。

## T=2ms 到 T=~130ms：主线程在做什么？

加载模块依赖树。`main.tsx` 直接导入了约 80+ 个模块，每个模块又有自己的依赖。JavaScript 模块系统会按照依赖顺序串行求值（同一个模块只求值一次）。

在这 130ms 里：
- MDM 子进程在后台读取企业配置
- Keychain 在后台返回凭据
- 主线程忙于模块初始化，CPU 基本被占满

**这是一个精心设计的时间重叠**：把 I/O 等待时间（MDM、Keychain）藏进了 CPU 工作时间（模块加载）里。

## T=~130ms：`profileCheckpoint('main_tsx_imports_loaded')`

模块加载完成，这是第二个重要检查点。从 `main_tsx_entry` 到这里的时间，就是模块求值耗时。

## T=~130ms：`main()` 函数开始执行

现在才进入我们通常理解的"程序主逻辑"。但此时：
- MDM 读取结果可能已经缓存完毕
- Keychain token 大概率已经预热好了
- 进程基础环境已就绪

## 为什么这个时间线很重要？

对于网页应用，300ms 是用户能接受的响应阈值。对于终端工具，这个阈值要苛刻得多——用户期望命令行响应"接近即时"。

如果不做 import 期间的并行预取，每次启动 Claude Code 都要额外支付 65-135ms 的 Keychain 等待和 MDM 等待，这些时间会加在每次敲下 `claude` 之后的"黑屏期"上。

## 面试指导

这类"前 N 毫秒发生了什么"的问题，考查的是对系统初始化的分层理解能力：

**第一层**：进程启动 → 运行时初始化 → 模块加载 → 主函数执行（大多数人只想到这个）

**第二层**：在上述各阶段内部，什么操作是可以并行的，什么必须串行

**第三层**：从用户感知角度出发，哪些延迟"可以被隐藏"，哪些必须直接出现在关键路径上

Claude Code 的设计之所以值得研究，不是因为它用了多高深的技术，而是因为它非常彻底地从"第三层"思考出发，一路向下做了具体的实现决策。能把这个思路讲清楚，就是一个很有说服力的系统设计答案。
