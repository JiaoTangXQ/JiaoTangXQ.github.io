# suppress 状态机：点击后先挡住 sticky 的自我刷新

当用户点击 sticky header 跳回某个 prompt 时，会发生一次视口滚动。这个滚动本身会触发 `StickyTracker` 的更新——「哦，视口位置变了，我需要重新计算当前 sticky prompt 是谁」。

问题是：这次重新计算可能在跳转还没完成时就发生，结果 sticky header 在跳转中途被更新成了「当前不完整位置对应的 prompt」，而不是「目标 prompt」。Sticky header 在自己修正过程中「反咬」了自己。

## 三态状态机

`suppress` 有三个状态：

- **`none`**：正常状态，sticky 更新自由运行
- **`armed`**：点击刚发生，准备进入抑制
- **`force`**：强制抑制，跳过 sticky 计算

点击时：`suppress = 'armed'`

下一帧的 effect 检测到 `armed`：切换到 `force`，跳过这次 sticky 计算

再下一帧，修正 effect 完成跳转：`suppress = 'none'`，恢复正常

## 为什么需要 armed 这个中间态

如果直接从 `none` 跳到 `force`，suppression 窗口开始的时机是「点击时」，但 React effect 的执行是异步的（下一次渲染）。

`armed` 作为中间态，让点击事件只是设置意图（「即将进入抑制」），由下一次渲染时的 effect 真正启动抑制。这确保了抑制窗口的开始和修正 effect 的执行在相同的渲染周期里对齐。

## 修正动作防止自我打架的普遍性

这类「修正动作本身触发了自己要防止的更新」的问题，在任何有滚动 + 状态追踪的界面里都会出现。

通用解法模式：**先抑制 → 执行修正 → 解除抑制**。关键是抑制窗口的粒度要精确——不能太宽（会错过真实的更新），不能太窄（会被抢先触发）。`armed → force → none` 这个三态机在这里提供了精确的窗口控制。
