# 特殊入口为什么必须尽早分流

`cc://`、深链接、`assistant`、`ssh`、direct connect 这些入口，看起来像是几种少数人才会用到的边角命令，但在 Claude Code 眼里，它们根本不是“普通启动以后再拐个弯”的小分支。它们代表的是不同的会话世界: 有的是协议唤起，有的是远端会话附着，有的是把本地终端接到另一台机器上，还有的是把界面变成纯查看器。

如果这种分流做得太晚，问题就会立刻出现。程序前面已经按“本地交互 REPL”假设走了一段，后面才发现其实这次是远端、是 viewer、是协议跳转，那前面那些关于权限、输入、会话来源、界面形态的默认前提都可能不再成立。晚分流的代价，不是多写几行判断，而是整条主链都变脏。

所以 Claude Code 的做法很果断: 尽早把这些特殊入口从 `argv` 里认出来，必要时重写参数，必要时直接提前退出，必要时把关键目标暂存在 pending 结构里，等主路径接手。这样后面的启动管线始终知道自己正在服务哪一种世界，而不是先假装人人都一样，再在半路把世界掰弯。

## 实现链
这页落到的实现链都是“身份先于环境”：设置来源、特殊入口、工作目录、模式或连接方式会先决定当前到底处在哪个世界，然后 `init()` 和后续装配才有正确前提。代码不是先搭一台通用机器再改身份，而是先把身份定准。

## 普通做法
更常见的写法是先进统一初始化，等大部分系统都起来以后，再根据参数或环境决定最后走哪条模式。

## 为什么不用那种做法
Claude Code 没这么做，因为对它来说，身份不是界面皮肤，而是会反过来改写环境、权限和能力世界的前提。先按错身份初始化，后面很多步骤都会偷偷站错坐标。

## 代价与回报
回报是环境基准统一。代价是入口看起来更像大路由表，而不像单线启动脚本。
