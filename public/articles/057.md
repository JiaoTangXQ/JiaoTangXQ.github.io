# inProcessRunner 和独立进程 teammate 的区别是什么？Claude Code 为什么需要两种形式？

Claude Code 的多 agent 团队支持两种 teammate：在同一进程内运行的（`inProcessRunner`），和作为独立进程运行的（通过 `spawnUtils` 启动）。这不是历史遗留，而是权衡不同场景的有意设计。

## inProcessRunner 的特征

`utils/swarm/inProcessRunner.ts` 启动的 teammate 运行在 leader 的同一个 Node.js 进程里。它们共享：
- 同一个 V8 堆（内存）
- 同一个文件系统句柄
- 但有**独立的 AppState**（通过 React 状态隔离）

权限处理走 `leaderPermissionBridge`：teammate 直接调用 leader 注册的函数，把确认请求推入 leader 的 React 状态队列。这比文件邮箱快得多，因为没有文件 I/O。

适用场景：短生命周期的子任务，需要快速启动、快速结束，不需要跨终端会话。

## 独立进程 teammate 的特征

通过 `spawnUtils` 启动的 teammate 是真正独立的进程，有自己的：
- 独立内存空间
- 独立的 claude-code 进程实例
- 独立的终端窗口（通过 tmux 或类似工具）

通信完全通过文件邮箱（`teammateMailbox`）进行，包括权限请求/响应、状态通知等。

适用场景：长生命周期的并行工作，需要用户能在多个独立终端间切换观察。

## teammateLayoutManager：空间布局的制度化

`utils/swarm/teammateLayoutManager.ts` 管理多个 pane 的布局逻辑。当一个团队启动时，系统需要决定每个 teammate 显示在哪个 pane，如何分配屏幕空间。

这个「空间分配」问题看起来是纯 UI 事务，但它实际上是团队认知负担的一部分。如果用户看不到某个 teammate 在做什么，等于那个 teammate 的工作对用户不可见——这违反了「用户应该能监督所有 agent 活动」的原则。

## teammateInit：启动时的身份初始化

`utils/swarm/teammateInit.ts` 处理 teammate 的身份初始化。它包含几个关键操作：
1. 把 team 信息写入 agent 的环境变量（`CLAUDE_CODE_AGENT_ID`、`CLAUDE_CODE_AGENT_NAME` 等）
2. 生成 teammate 的专属颜色（从颜色池里分配一个尚未使用的颜色）
3. 在 team 文件里注册新 teammate

颜色分配有一个小细节：系统维护一个颜色池，每次新增 teammate 时从未使用的颜色里分配。如果颜色池耗尽（团队人数超过颜色数量），会循环使用颜色。这避免了不同 teammate 颜色冲突的问题，但超大团队时可能出现两个 teammate 颜色相同——这是意识到的 trade-off，不是 bug。

## 两种形式的选择逻辑

```
需要快速、轻量、共享资源 → inProcessRunner
需要独立、长期、用户可见 → 独立进程
```

但在实际使用中，很多团队场景两者都会用到：leader 用独立进程形式让用户可以看到整体进展，但 leader 内部又会用 inProcessRunner 启动一些辅助性的短期子任务。

## 心智模型：团队是组织问题，不是并发问题

Claude Code 的 swarm 设计最重要的一个心智模型是：**多 agent 团队首先是一个组织问题，然后才是技术问题**。

组织问题的核心是：
- 谁有权做什么决定？（leader/teammate 层级）
- 信息如何在成员间流动？（mailbox）
- 成员之间如何识别彼此？（颜色、pane 布局）
- 成员离开或崩溃后，工作如何延续？（reconnection）

这些问题先解决清楚，技术实现（进程内 vs 独立进程）才是次要选择。

## 面试答题思路

「多进程 agent 和单进程 agent 各有什么优劣」：

1. **进程隔离 = 故障隔离**：独立进程的 teammate 崩溃不影响 leader 进程
2. **同进程 = 通信效率**：inProcessRunner 的权限请求通过函数调用，比文件 I/O 快几个数量级
3. **用户可见性要求决定架构**：如果 agent 的工作需要用户在独立窗口观察，必须是独立进程
4. **混合使用是常态**：大型系统里通常两种形式都有，关键是明确每种形式的适用场景

追问「如何保证两种形式在权限处理上行为一致」：通过抽象层（mailbox 协议）保证最终结果一致，但实现路径不同——同进程走函数调用，跨进程走文件邮箱。用户看到的确认 UI 是同一个组件，只是数据来源不同。
