# isItemClickable 用 useCallback + lookupsRef 而不是把 lookups 放进 deps，为什么？

这道题考察 React 中"数据新鲜度"和"函数稳定性"的矛盾，以及如何用 ref 解决它。

## 矛盾的来源

`isItemClickable` 的逻辑是：给定一条 `RenderableMessage`，判断点击它是否有意义（即展开能看到更多信息）。这个判断依赖 `lookups`：

```typescript
function isItemClickable(msg: RenderableMessage): boolean {
  const tu = lookupsRef.current.toolUseByToolUseID.get(toolUseIdOf(msg))
  if (!tu) return false
  return findToolByName(tu.name, tools)?.isResultTruncated?.(msg) ?? false
}
```

函数需要读最新的 `lookups`（因为工具调用的结果状态在流式过程中随时更新），但函数本身作为 prop 传给 `VirtualMessageList`，如果每次 lookups 更新都产生新的函数引用，`VirtualMessageList` 的所有行都会因为 prop 变化重渲。

流式期间每收到一个 `input_json_delta` 就更新 lookups → isItemClickable 变 → VirtualMessageList 重渲 → 60 个挂载的条目全部重渲 = 每秒几十次不必要的全量重渲。

## useCallback + ref 的解法

```typescript
// lookupsRef 始终指向最新的 lookups，但引用变化不触发重渲
const lookupsRef = useRef<Lookups>(emptyLookups)
lookupsRef.current = lookups  // 每次渲染时同步更新 ref

const isItemClickable = useCallback((msg: RenderableMessage) => {
  // 读取 ref.current 而不是闭包里的 lookups
  return computeClickable(msg, lookupsRef.current, tools)
}, [tools])  // deps 里只有 tools，不包含 lookups
```

`useCallback` 的 deps 里没有 `lookups`，所以函数引用在 `tools` 不变的情况下永远稳定。但函数体内读 `lookupsRef.current` 而不是闭包变量，所以每次调用时用的都是最新的 lookups，不会读到过期快照。

## ref 在渲染周期里的位置

注意 `lookupsRef.current = lookups` 在组件函数体里直接赋值，不在 `useEffect` 里。这是一个有意识的写法：

- 在 `useEffect` 里赋值：赋值发生在下次渲染提交之后，也就是说当前渲染里触发的事件（hover、click）读到的还是旧的 lookups
- 在函数体里直接赋值：赋值发生在渲染函数执行阶段，和 DOM 提交同步，事件处理器在 commit 之后触发，读到的是本次渲染产生的 lookups

这是 React 里"在渲染阶段写 ref"的惯用模式，和"状态派生值"不同，不需要 `useMemo`，因为目的不是缓存计算结果，而是让一个回调能访问最新的数据。

## 工具名查找链

`isItemClickable` 的最终决策依赖工具的 `isResultTruncated` 方法：

```typescript
findToolByName(toolName, tools)?.isResultTruncated?.(msg)
```

不是所有工具都有这个方法。Read 工具如果返回的文件超过截断阈值，会有 `isResultTruncated` 返回 true，点击可以展开查看完整内容；Bash 工具的输出不受此逻辑控制（Bash 的结果截断用不同机制处理）。

这个设计把"这个工具的结果是否值得点开"的判断下放给工具自己，而不是在 `isItemClickable` 里硬编码每个工具名。`findToolByName` 只是定位到工具对象，调用它的协议方法。

## 面试指导

这是一道关于"stale closure"和"over-rendering"权衡的经典题。

**两个问题，两个解法**：
1. 函数读到过期数据（stale closure）→ 用 ref 保存最新值，函数读 ref 而不是闭包变量
2. 函数引用不稳定导致不必要的重渲 → 用 useCallback 稳定引用，把不必要的变量移出 deps

两个问题同时出现时，两个解法可以叠加：useCallback 固定函数 + ref 保存数据。

**面试陷阱**：说"用 useMemo 缓存 isItemClickable"。useMemo 缓存的是函数的**计算结果**，但 `isItemClickable` 本身是一个函数，你要缓存的是函数引用，所以应该用 useCallback 而不是 useMemo。（实际上 `useCallback(fn, deps)` 就是 `useMemo(() => fn, deps)`，但在语义上 useCallback 更能表达"我要稳定这个函数引用"的意图。）

**进阶追问**：如果在 ref 写入和事件触发之间有一个渲染周期被跳过（React batching 或 concurrent mode），会不会读到错误的 lookups？答：在 React 的 concurrent mode 下，渲染可能被中断后重新执行，如果 ref 赋值在渲染函数体里，被中断的渲染写入的 ref 可能被正式提交的渲染再次覆盖。但 `lookupsRef.current = lookups` 的幂等性保证了这不是问题——同一次输入产生的 lookups 写多次等于写一次。
