# pairedDeviceId 不写进配置，下次启动就不认识你了

`onExtensionPaired()` 里有一段不起眼的条件判断：

```typescript
onExtensionPaired: (deviceId: string, name: string) => {
  saveGlobalConfig(config => {
    if (
      config.chromeExtension?.pairedDeviceId === deviceId &&
      config.chromeExtension?.pairedDeviceName === name
    ) {
      return config   // 没变就不写
    }
    return {
      ...config,
      chromeExtension: { pairedDeviceId: deviceId, pairedDeviceName: name },
    }
  })
}
```

两件事同时在这里发生：**写回**和**去重**。

配对成功后，`deviceId` 和 `name` 落进全局配置文件。下次 Claude Code 启动，`getPersistedDeviceId()` 从那里读出来，连接浏览器扩展时就不再是陌生人握手，而是认出了"上次见过"的那个设备。

但写回的动作有个前提：只有身份真的变化了才执行。如果 `pairedDeviceId` 和 `pairedDeviceName` 都没变，回调直接返回原来的 `config`，`saveGlobalConfig` 什么也不写。这避免了每次配对都触发一次配置文件 I/O，也避免了不必要的文件修改时间戳更新。

## 身份持久化意味着什么

浏览器桥是一个跨进程的长期设施。用户打开 Claude Code，建立浏览器连接，做了一些工作，关掉。第二天再打开，如果配对信息没有持久化，整个重连流程就得从头来：扩展重新发现，CLI 重新确认，用户重新等待。

把 `deviceId` 写进全局配置，把这个成本压缩到只发生一次。之后的每次启动，`getPersistedDeviceId()` 带着已知身份去连，扩展认出来，握手过程可以更直接。

这和 SSH 的 `known_hosts` 在概念上是同一回事：第一次见面验证，之后信任缓存，不再每次都重头建立关系。浏览器不是一次性外设，是这台工作台长期的现实延伸端。
