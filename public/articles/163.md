# 第十三卷回收索引：连续性世界里整理必须等主回合落稳才有资格进场

## 实现链
连续性世界里最容易弄乱的，不是有没有整理动作，  
而是整理是不是抢在主回合前面发生了。

但现在它已经自然分成两条更窄的线：

- 整理推迟到回合结束之后，主回合没落稳前任何整理都不该进场
- `session memory` 只在阈值和自然停顿点更新，连续性机制也先服从主回合秩序

如果继续把这两件事挤在同一页里，就会把“整理何时进场”与“记忆何时更新”揉成一个判断。

所以更稳的做法，是把 `163` 收成父级入口，再往下拆成两页。

这页现在作为父级入口，继续分出去的实现分支是：
### 整理推迟到回合结束后

- [205-第十三卷回收索引-连续性世界-整理推迟到回合结束后说明主回合没落稳前任何整理都不该进场.md](/Users/xxx/tmp/cc/docs/claude-code-book/appendices/205-第十三卷回收索引-连续性世界-整理推迟到回合结束后说明主回合没落稳前任何整理都不该进场.md)

### `session memory` 只在阈值和自然停顿点更新

- [206-第十三卷回收索引-连续性世界-session-memory只在阈值和自然停顿点更新说明连续性机制也先服从主回合秩序.md](/Users/xxx/tmp/cc/docs/claude-code-book/appendices/206-第十三卷回收索引-连续性世界-session-memory只在阈值和自然停顿点更新说明连续性机制也先服从主回合秩序.md)

它不再负责细讲每一种连续性整理时机。  
它只负责保住一个入口判断：

整理和记忆更新都不该抢在主回合前面。  
它们必须等主回合落稳后再进场。

## 普通做法
更普通的做法，是一发现上下文变长就立刻整理，或者把当前状态和长期记忆统统塞回主会话，省掉额外的插槽和外置存储。 对父级入口页来说，更常见的写法就是继续把两三条不同链挤在一页里，读起来省页数，但判断会糊。

## 为什么不用
Claude Code 不这么做，是因为整理太早会打断主回合，记忆不分层会把短期噪音和长期知识混在一起，最后让连续性系统同时失去可读性和可恢复性。 这些父级页现在存在，就是为了先把分支关系讲清，再让更细页分别承担细节。

## 代价
代价是需要额外维护 compact boundary、回合后插槽、外置 memory 和记忆回灌链，系统会更厚。 作为父级入口，它还额外带来导航变深、读者要多跳几页的成本。
