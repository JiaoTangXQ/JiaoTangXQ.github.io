# postSamplingHooks 与 session memory：Claude Code 的回合后整理机制全景

**面试题：Claude Code 的上下文管理为什么不在 AI 回复结束后立刻整理？它的整理时机是如何设计的？**

## 为什么整理时机是一个严肃问题

大多数 AI 系统的上下文管理是被动的：token 快满了，压缩；对话结束了，清空。Claude Code 在 v1 迭代中发现这种做法会产生一个隐患——如果整理动作被插入主回合中间，模型看到的消息列表在一次 turn 里发生了两次突变，会导致 tool call 结果匹配失败、附件重复注入、session memory 读到过期基准。

解决方案是把整理完全移出主回合，统一放进 **postSamplingHooks** 插槽。

## postSamplingHooks 是什么

`postSamplingHooks` 是 `registerPostSamplingHook()` 注册的回调集合，在每次 AI 采样（model API call）返回之后、用户下一条输入处理之前触发。

```ts
// src/utils/hooks/postSamplingHooks.ts
export function registerPostSamplingHook(
  hook: (ctx: REPLHookContext) => Promise<void>,
): () => void
```

session memory 的触发正是通过这个机制挂载的：

```ts
// src/services/SessionMemory/sessionMemory.ts
registerPostSamplingHook(async (ctx) => {
  if (!shouldUpdateSessionMemory(ctx.messages, ctx.agentId)) return
  await maybeUpdateSessionMemory(ctx.messages, ctx.agentId)
})
```

这意味着 session memory 的提取永远不会出现在 AI 正在生成回复的过程中，也不会出现在用户刚提交新消息之后（那时 main loop 还没采样完毕）。

## session memory 的触发条件

`shouldUpdateSessionMemory()` 内部有三道门：

**1. 初始化阈值（minimumMessageTokensToInit）**

默认 10,000 tokens。在会话体量不足时，提取 session memory 本身的成本（调用 forked subagent）比收益更大，所以系统先等会话有足够厚度。

**2. 更新间隔（minimumTokensBetweenUpdate + toolCallsBetweenUpdates）**

两个维度同时检查：上次提取后 token 增长量 ≥ 5,000，**且** 工具调用次数 ≥ 3。这防止在短对话中过于频繁地更新——每次更新都要 fork 一个子 agent，本身会消耗 token。

**3. 提取锁（extractionStartedAt）**

`markExtractionStarted()` 被调用后，后续的 postSamplingHook 检查会直接跳过，直到 `markExtractionCompleted()` 被调用。这是一个简单的并发锁，防止两次采样之间同时触发两个 memory 提取 agent。

## session memory 更新只在自然停顿点

除了 token/tool call 阈值，session memory 还有一个"自然停顿点"检测：

```ts
// 只在 assistant 消息没有 pending tool calls 时更新
if (hasToolCallsInLastAssistantTurn(messages)) return
```

如果 AI 的最后一条消息里还有未完成的 tool use（比如正在运行 Bash 命令），当前并不是一个安全的停顿点，session memory 的提取会延后到 tool results 都回来之后。

## 三条路径的优先级

当会话 token 触达 autocompact 阈值时，系统有三条路径可选，按优先级排列：

**路径 1：session memory compaction（轻量）**

`trySessionMemoryCompaction()` 先检查 session memory 文件是否已有足够内容。如果有，它直接截断消息列表，把 session memory 文件内容作为摘要注入，不需要重新跑一遍摘要生成 agent。这是"保会话"优先的最轻量做法。

**路径 2：全量 compactConversation（重量）**

如果 session memory 为空或内容不足，才触发 `compactConversation()`——fork 一个专门的摘要 agent，生成完整对话摘要，替换整个消息列表。

**路径 3：熔断（不压缩）**

连续失败 3 次（`MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3`）后，停止尝试，避免在不可恢复的上下文溢出场景中浪费 API 调用。

## 整理完成后的清理

无论是哪条压缩路径，结束后都会调用 `runPostCompactCleanup(querySource)`：

```ts
// 清除 memoize 缓存
getUserContext.cache.clear?.()
resetGetMemoryFilesCache('compact')
// 清除工具权限审批缓存
clearClassifierApprovals()
clearSpeculativeChecks()
// 清除 session memory 的基准 ID
setLastSummarizedMessageId(undefined)
```

最后一行是关键：`lastSummarizedMessageId` 记录着 session memory 上次提取到哪条消息。压缩后消息 UUID 全部重新排，如果不清空，下一次 session memory 触发会试图从一个已经不存在的 UUID 开始计数。

## 面试要点

**Q：为什么 session memory 用 forked subagent 而不是直接调用 API？**

因为 session memory 提取是一个长期运行的后台操作，它需要读取整个对话历史、调用 FileReadTool 读取当前 memory 文件、写入更新后的内容。用独立的 subagent context 可以避免和主 context 共享 token 计数，也可以在 subagent 里独立触发 tool calls 而不影响主 loop 的消息列表。

**Q：postSamplingHook 和 preAction hook 有什么区别？**

preAction 在 tool use 执行前触发，用于权限检查和副作用拦截；postSamplingHook 在 AI 采样完成后触发，用于上下文整理和记忆更新。两者在时序上不重叠，分别守住了主回合的前后两端。
