# 深度分析：任务对象替换时为什么要保住"正在使用的现场"

## 任务为什么需要被替换

正常情况下，一个任务对象从创建到终止，都是同一个对象（在 AppState 里同一个 id 对应的那个条目）。

但有些情况需要替换：

**子 agent 重连**：一个远端 agent 断线后重新连接，需要重新登记自己的状态。旧的任务对象里可能有过时的连接信息，新对象要替换进来。

**任务恢复**：进程重启后，从磁盘读取之前的任务记录，重建任务对象。这时的"重建"实际上是用磁盘上的数据替换内存里可能不存在的或旧的对象。

**状态修复**：发现任务对象的某些字段不一致，需要用正确的版本替换。

## 替换操作的危险之处

替换听起来简单：`AppState.tasks[id] = newTaskObject`。

但这行代码有一个隐藏的破坏性：如果用户正在看任务面板，此刻面板里展示的正是这个任务的 UI，那么替换操作会让 UI 突然改变。

更具体地说：

用户正在查看任务的进度详情——已完成了 23 个工具调用，最后一次操作是读取了 `src/main.ts`。这些信息来自 `AgentProgress`，存在任务对象里。

如果新任务对象没有携带这些进度信息（因为它是新建的对象，进度从零开始），替换后用户看到的进度消失了，变成了一个空的任务。

用户会感到困惑：进度哪去了？是任务出了问题还是界面 bug？

## 保住现场的技术要求

"不洗掉用户正在看的状态"转换成代码要求是：

替换任务对象时，必须把旧对象里"用户正在查看的"字段迁移到新对象。

哪些字段是"用户正在查看的"？

- `AgentProgress`（进度信息：工具调用次数、token、最近活动）
- `outputOffset`（已读取的输出位置，用于进度条）
- UI 状态相关的字段（是否展开、是否在焦点中）

哪些字段不需要迁移？

- 连接相关的字段（比如 WebSocket 连接对象，这正是需要替换的）
- 已过时的状态字段

## updateTaskState 的合并语义

在 `src/utils/task/framework.ts` 里，`updateTaskState` 函数不是"替换整个任务对象"，而是"合并指定字段"：

```typescript
// 推断的行为（不是直接引用源码）
function updateTaskState(
  taskId: string,
  patch: Partial<TaskState>,
  setAppState: SetAppState
): void {
  setAppState(prev => ({
    ...prev,
    tasks: prev.tasks.map(task =>
      task.id === taskId
        ? { ...task, ...patch }  // 合并，保留未指定的字段
        : task
    )
  }))
}
```

这个合并语义是保住协作现场的关键：只传入需要更新的字段，其余字段保持不变。

调用方不需要知道"哪些字段是用户在看的"，也不需要手动迁移。只要在替换时只传入真正需要更新的字段，现有的状态自然不会被清除。

## 一个反模式：全量替换

与之相对的反模式是全量替换：

```typescript
// 不好的做法
setAppState(prev => ({
  ...prev,
  tasks: prev.tasks.map(task =>
    task.id === taskId ? newTaskObject : task  // 全量替换
  )
}))
```

这种写法在 `newTaskObject` 没有正确初始化所有字段时，会静默地清除用户正在查看的状态。

静默地——这是最危险的地方。没有错误，没有警告，UI 改变了，用户困惑了。

## 对象身份 vs 对象内容

这个问题揭示了一个对象系统里的基本区分：**对象身份（identity）和对象内容（content）是两件不同的事**。

任务的"身份"是它的 `id`——在整个生命周期里不变的标识符。

任务的"内容"是它当前的字段值——会随着执行进展而变化。

替换操作的问题是：我们需要更新某些内容字段（连接信息），但不应该影响其他内容字段（进度信息）。

如果把对象当成不可区分的整体（要么全换，要么不换），这个区分就消失了，带来的就是"替换会洗掉进度"的问题。

通过 patch 语义，我们在 API 层面上明确表达了：**这次更新只涉及这几个字段，其他字段不动**。

## 面试要点

1. **任务替换有隐性风险**：全量替换看起来简单，但会静默清除用户正在查看的状态。
2. **合并语义（patch）是安全的默认行为**：只传入需要更新的字段，未指定字段自动保留，调用方不需要记住所有字段。
3. **对象身份和对象内容是不同的关注点**：id 不变，但内容字段可以部分更新，这不是一次"替换"而是"修补"。
4. **"协作现场"是具体字段**：进度信息、输出偏移量、UI 状态——这些是用户正在使用的内容，任何修改都要对这些字段格外小心。
