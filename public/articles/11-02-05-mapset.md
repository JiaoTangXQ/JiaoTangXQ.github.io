# 工程取舍：AppState 里放 Map 和 Set，你为什么不怕 JSON 序列化失败？

在 Redux 风格的状态管理里，有一条默认规则：state 必须可序列化为 JSON。这个约束让时间旅行调试、持久化、undo/redo 都变得可行。

Claude Code 的 AppState 打破了这条规则。

```typescript
agentNameRegistry: Map<string, AgentId>
fileHistory: FileHistoryState  // 包含 trackedFiles: Set<string>
sessionHooks: SessionHooksState  // 实际上是 Map
activeOverlays: ReadonlySet<string>
```

`Map`、`Set`、`ReadonlySet`——这些结构都不能直接 `JSON.stringify`。

这是设计缺陷，还是有意为之？

---

## 为什么用 Map 而不是 Record

`agentNameRegistry: Map<string, AgentId>` 是「agent 名字到 agent ID 的映射」。

注释说：`Latest-wins on collision. Used by SendMessage to route by name.`

用 `Map` 而不是 `Record<string, AgentId>` 的具体理由：

**1. Map 对「名字注册表」这个语义更准确**

注册表的操作是：注册（set）、查找（get）、注销（delete）、检查存在性（has）。Map 的方法名直接对应这些操作，而 `Record` 的操作要写成 `obj[key]`、`delete obj[key]`、`key in obj`，语义没那么清晰。

**2. Map 不会被原型链污染**

`{ __proto__: ... }` 是一个合法的对象键，如果 agent 名字恰好是 JavaScript 的内置属性名（虽然不太可能），用普通对象会有意外行为。`Map` 没有原型链污染问题。

**3. Map 的键可以是任意类型**

这里键是字符串，但 Map 的灵活性是真实的——如果以后需要用复合键，迁移成本更低。

---

## 为什么用 Set 而不是 Array

`activeOverlays: ReadonlySet<string>` 是当前活跃的覆盖层（Select 对话框等）的 ID 集合。

注释说：`Active overlays (Select dialogs, etc.) for Escape key coordination.`

用 `Set` 而不是 `string[]` 的原因：

**Set 保证元素唯一**

同一个对话框不能被打开两次。`Set` 自动去重，`Array` 需要手动检查重复。

**Set 的 has 操作是 O(1)**

`set.has(id)` 是常数时间，`array.includes(id)` 是线性时间。对于「某个 overlay 现在是否活跃」这种频繁查询，`Set` 更高效。

**ReadonlySet 提供不可变性约束**

`ReadonlySet<string>` 不暴露 `add`、`delete` 等修改方法，只暴露 `has`、`forEach`、`size` 等只读方法。这是一种类型层面的不可变性约束（虽然在运行时你仍然可以通过 `as Set<string>` 绕过）。

---

## Map 和 Set 的不可变更新模式

由于 `Map` 和 `Set` 没有不可变的原生 API，更新时需要创建新实例：

```typescript
// 添加一个 agent 注册
store.setState(prev => ({
  ...prev,
  agentNameRegistry: new Map([...prev.agentNameRegistry, [name, agentId]])
}))

// 添加一个 overlay
store.setState(prev => ({
  ...prev,
  activeOverlays: new Set([...prev.activeOverlays, overlayId])
}))

// 删除一个 overlay
store.setState(prev => {
  const newOverlays = new Set(prev.activeOverlays)
  newOverlays.delete(overlayId)
  return { ...prev, activeOverlays: newOverlays }
})
```

每次操作都创建新的 `Map` 或 `Set` 实例，`Object.is` 比较时新旧实例不同，订阅了这个字段的组件会收到更新通知。

这比 `Array` 操作更啰嗦一点，但语义更准确。

---

## 序列化问题：持久化用什么存

AppState 里的 `Map` 和 `Set` 不能直接 `JSON.stringify`，但某些字段需要持久化（比如 `fileHistory`）。

解决方案是在持久化时做转换：

```typescript
// 持久化前转换
const serialized = {
  ...appState,
  agentNameRegistry: [...appState.agentNameRegistry.entries()],
  fileHistory: {
    ...appState.fileHistory,
    trackedFiles: [...appState.fileHistory.trackedFiles],
  }
}

// 读取后还原
const restored = {
  ...serialized,
  agentNameRegistry: new Map(serialized.agentNameRegistry),
  fileHistory: {
    ...serialized.fileHistory,
    trackedFiles: new Set(serialized.fileHistory.trackedFiles),
  }
}
```

Claude Code 在 `onChangeAppState` 里处理持久化，这种转换逻辑就在那里。

---

## 为什么接受这些复杂性

更简单的选择是把所有结构都用 `Record` 和 `string[]`，保持 JSON 可序列化。很多 Redux 项目就这么做。

但这会带来另一层成本：

- 每次查找 agent 都要 `Object.keys(registry).find(key => registry[key] === id)`，而不是 `registry.get(id)`
- 每次检查 overlay 都要 `activeOverlays.includes(id)`，不如 `activeOverlays.has(id)` 直接
- 手动去重、手动保证唯一性

Claude Code 选择了「用正确的数据结构表示真实的关系，在需要持久化时做显式转换」，而不是「用可序列化的替代结构，在每次操作时做隐式的逻辑补丁」。

## 面试指导

**Map 和 Object 的性能差异在什么量级下才有意义？**

对于 `agentNameRegistry`，通常只有几个 agent，Map vs Object 的性能差异微乎其微。这里选 Map 是语义考虑，不是性能考虑。大量数据（比如 10 万条）的场景下，Map 的 get/has 相比 Object 才有可测量的优势。

**ReadonlySet vs ReadonlyArray，怎么选？**

如果元素需要唯一（去重），用 Set。如果需要保持插入顺序并允许重复，用 Array。`activeOverlays` 需要唯一（同一个 overlay 只能活跃一次），所以用 Set。

**Immer 怎么处理 Map 和 Set？** Immer 从 8.0 开始原生支持 Map 和 Set，用 Immer 的 `produce` 可以直接对 draft 上的 Map/Set 做 `set`、`delete`、`add`，Immer 会追踪这些操作并生成新的不可变实例。Claude Code 没用 Immer，所以要手动处理。
