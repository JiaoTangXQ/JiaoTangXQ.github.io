# 状态里出现 Map 和 Set 说明它在管身份现实不是表单数据

`AppState` 里放着 `Map`、`Set`、队列和注册表，这很说明问题。它管的不是几个输入框值，而是名字到 agent 的映射、活跃 overlay、文件追踪、通知队列、等待中请求这种运行时关系。

这样的现实本来就不只是平铺表格。Claude Code 宁可让状态世界更像一台真实运行中的机器，也不把复杂关系硬压扁成好看但失真的数据。

## 实现链

`AppStateStore.ts` 里有 `agentNameRegistry: Map<string, AgentId>`、`registeredTools: Map<...>`、`activeOverlays: Set<string>`、`trackedFiles: Set()` 这类结构。它装的是身份、索引和成员资格，而不是一串适合 JSON 表单的简单字段，说明这份 state 在管运行时现实。

## 普通做法

普通页面 state 往往只用对象和数组，方便序列化、比较和表单驱动；一见到 `Map`、`Set` 就会尽量绕开。

## 为什么不用

Claude Code 没为了“像普通前端 state”而把这些结构拍扁，是因为运行时里确实存在名字注册、工具登记、活跃覆盖层这类集合现实。拍平成数组只会把查找和去重逻辑重新散回各处。

## 代价

代价是这份 state 更不像典型 Redux 表单数据，序列化和调试都更讲究；但换来的是它更忠于源码里真实存在的身份关系。
