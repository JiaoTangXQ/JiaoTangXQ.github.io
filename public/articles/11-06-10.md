# 任务完成后的 30 秒确认期：一个有纪律的计时器

`TaskListV2.tsx` 里，任务完成后不会立刻从列表里消失。它有一个确认窗口——大约 30 秒——在这段时间里仍然显示在任务列表上，之后才淡出。

这个确认窗口是有意设计的，而且它的维护方式很有纪律。

## 两个 ref，一套机制

```typescript
completionTimestampsRef   // 记录每个任务「第一次进入 completed」的时间
previousCompletedIdsRef   // 记录上一轮渲染时哪些任务是完成状态
```

每次 `tasks` 变化时，`useEffect` 会扫描：哪些任务是新完成的（出现在当前 completed 列表里，但不在 `previousCompletedIdsRef` 里），为它们记录完成时间，找出所有完成任务里最早到期的时间，挂一个 `setTimeout` 到那个时间点触发重绘。

关键是：**`useEffect` 的依赖只有 `tasks`**。这个计时器不会因为其他状态变化触发，也不会因为父组件无关的重绘被偷偷续命。

## 为什么这个纪律重要

如果确认期靠每次重绘偷偷刷新，就不是「30 秒的确认期」，而是「每次组件刷新时自动延长的模糊缓冲」。用户点了什么、输入了什么、父组件因为别的原因重渲染了——都会悄悄把这个窗口延长。

Claude Code 想要的是一个制度性的确认期，不是一个副作用产物。任务完成了，用户有 30 秒时间看到结果，然后消失。这条规则应该明确、可预测，不该因为别的操作被无声无息地改变。

把计时器绑定到「任务变化」而不是「任何重渲染」，是在把意图写进代码里：这个计时器只受任务驱动，其他事情不应该影响它。
