# 面试追问：getDefaultAppState() 为什么不是一个静态常量？

一道简单但很能说明问题的面试题：

为什么 `getDefaultAppState()` 是一个函数，而不是一个直接 `export` 的常量对象？

```typescript
// 为什么不是这样？
export const DEFAULT_APP_STATE: AppState = {
  settings: getInitialSettings(),
  verbose: false,
  toolPermissionContext: {
    ...getEmptyToolPermissionContext(),
    mode: 'default',  // 写死了
  },
  // ...
}
```

答案不是「因为 `getInitialSettings()` 需要在运行时调用」，虽然这也是对的。真正的原因在权限模式的初始化逻辑里。

---

## 身份先于设置

```typescript
export function getDefaultAppState(): AppState {
  const teammateUtils = require('../utils/teammate.js')

  const initialMode: PermissionMode =
    teammateUtils.isTeammate() && teammateUtils.isPlanModeRequired()
      ? 'plan'
      : 'default'

  return {
    // ...
    toolPermissionContext: {
      ...getEmptyToolPermissionContext(),
      mode: initialMode,
    },
    // ...
  }
}
```

这段代码在说：**你是谁，决定了你以什么姿态出现。**

Claude Code 支持多 agent 协作（Swarm 模式），一个 teammate 进程被 leader 生成时，可以带着 `plan_mode_required` 标记启动。这意味着这个 teammate 必须在 `plan` 模式下运行——它不能擅自执行工具，只能制定计划让 leader 审批。

如果 `DEFAULT_APP_STATE` 是静态常量，所有进程都以 `default` 模式启动，然后再被修正成 `plan`——即使修正发生在第一帧之前，这个「修正」的存在本身就意味着曾经有一个短暂的错误状态。

把身份检查放在 `getDefaultAppState()` 里，进程从第一帧开始就在正确模式下，不存在任何中间态。

---

## 为什么用 require 而不是 import

```typescript
const teammateUtils = require('../utils/teammate.js') as typeof import('../utils/teammate.js')
```

注释里写了：`Use lazy require to avoid circular dependency with teammate.ts`。

`teammate.ts` 可能间接依赖 `AppStateStore.ts`，如果 `AppStateStore.ts` 顶层就 `import teammate.ts`，会形成循环依赖。

用 `require`（CommonJS 风格的懒加载）可以把依赖推迟到函数调用时，此时两个模块都已经初始化完成，不会有循环依赖问题。这是一个常见的打破循环依赖的技巧，在 TypeScript + Node.js 项目里时有出现。

`as typeof import(...)` 是告诉 TypeScript：这个 require 的结果和静态 import 的类型相同，恢复类型检查。

---

## getDefaultAppState 里的其他初始化逻辑

除了权限模式，`getDefaultAppState` 还有两个值得注意的初始化：

```typescript
thinkingEnabled: shouldEnableThinkingByDefault(),
promptSuggestionEnabled: shouldEnablePromptSuggestion(),
```

这两个值也是在运行时调用函数计算的，而不是写死的。

`shouldEnableThinkingByDefault()` 可能检查用户配置、实验性标志（feature flag）或 GrowthBook 开关。`shouldEnablePromptSuggestion()` 同理。

把这些判断集中在 `getDefaultAppState()` 里，有一个好处：**默认状态反映了当前用户的完整配置，而不是一个通用的「白板」状态。** 用户打开 Claude Code，他们看到的第一帧已经是根据他们的配置定制过的了。

---

## 另一个设计选择：init action

Redux 风格的初始化方式是：先创建带通用默认值的 store，然后 dispatch 一个 `INIT` action 来覆盖初始状态：

```
dispatch({ type: 'INIT', payload: { mode: 'plan', ... } })
```

这种方式把「初始化逻辑」和「状态创建」分开了，更符合 Redux 的模式，但它也意味着应用有一个短暂的「通用默认状态」——在 INIT action 被 dispatch 之前，组件读到的权限模式可能是错的。

`getDefaultAppState()` 直接在创建状态时就做好了所有判断，没有这个中间态。代价是初始化函数不再纯粹（它有副作用：调用了 `require`、读了环境变量），收益是状态从第一帧开始就是对的。

---

## sessionHooks 的特殊初始化

```typescript
sessionHooks: new Map(),
```

`sessionHooks` 是 `SessionHooksState`，实际上是一个 `Map`。这个字段被专门排除在 `DeepImmutable` 之外（因为 `SessionHooksState` 包含函数），初始化为空 Map。

Map 作为初始值的一个微妙之处：`getDefaultAppState()` 每次调用都会创建一个**新的** `new Map()`，而不是共享一个全局空 Map。这很重要——如果共享同一个 Map 实例，某个会话对 sessionHooks 的修改会意外影响另一个会话。对于需要「隔离」的状态，每次都创建新实例是必要的。

## 面试指导

这道「为什么不是静态常量」的问题，背后考察的是：

**「先错后改」vs「一开始就对」的取舍**：多数情况下，先错后改会有窗口期问题。越重要的状态（比如权限模式），越值得在初始化时就设置正确。

**副作用在初始化函数里合不合适？** `getDefaultAppState()` 里调用了 `require`、`getInitialSettings()`，这都是有副作用的操作。这违反了「纯函数」的原则，但在「初始化函数」的语义下是合理的——初始化本来就是有副作用的操作。

**循环依赖的解决思路有哪些？** 除了懒加载 require，还可以把共享的类型定义提取到第三个模块（让两个模块都只依赖第三者），或者重新设计模块边界来消除循环。懒加载是最不侵入的短期解法，但最好配合注释说明为什么需要这么做。
