# 默认状态会先看你是谁再决定开局

`getDefaultAppState()` 不是生成一张白纸，而是先看当前进程是不是带着 `plan_mode_required` 出生的 teammate，再决定默认权限模式。Claude Code 承认的不是“任何会话都一样”，而是“你一进场，就已经处在某种组织关系里”。

这件事很见功力。很多系统把身份差异拖到后面补丁式修正，它却把开局现实直接写进默认状态。好的工作台，默认值也得先认人。

## 实现链

`getDefaultAppState()` 不会直接返回一份静态常量，而是先检查当前是否是带 `plan_mode_required` 的 teammate，再决定初始 `permissionMode` 用 `plan` 还是 `default`。开局现实先看身份，再决定制度姿态。

## 普通做法

更普通的做法，是把默认 state 写成固定对象，等后面身份信息到了再局部修正。

## 为什么不用

Claude Code 不这么拖后处理，是因为权限模式一旦开局就错，后面的工具可见性、输入提示和执行约束都会短暂站在错现实里。它宁可在初始化时多做一步判断，也不愿意先错后补。

## 代价

代价是默认值不再纯粹，初始化函数得知道一点组织关系；但换来的是系统从第一帧开始就更像在正确世界里启动。
