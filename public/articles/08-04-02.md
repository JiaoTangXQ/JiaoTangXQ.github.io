# 安全边界分析：Bridge 的信任边界在哪里划定？

## 三层 entitlement 检查

`bridgeEnabled.ts` 的 `getBridgeDisabledReason()` 函数，把 bridge 模式的开启条件写得很明确：

```typescript
export async function getBridgeDisabledReason(): Promise<string | null> {
  if (feature('BRIDGE_MODE')) {
    if (!isClaudeAISubscriber()) {
      return 'Remote Control requires a claude.ai subscription...'
    }
    if (!hasProfileScope()) {
      return 'Remote Control requires a full-scope login token. Long-lived tokens...'
    }
    if (!getOauthAccountInfo()?.organizationUuid) {
      return 'Unable to determine your organization...'
    }
    if (!(await checkGate_CACHED_OR_BLOCKING('tengu_ccr_bridge'))) {
      return 'Remote Control is not yet enabled for your account.'
    }
    return null  // null = 没有禁用原因 = 允许
  }
  return 'Remote Control is not available in this build.'
}
```

四个条件，缺一不可：构建时特性标志（`feature('BRIDGE_MODE')`）、运行时订阅状态、token scope、GrowthBook 功能门控。

## 为什么是这四个，而不是一个？

### 1. 构建时特性标志（feature）

`feature('BRIDGE_MODE')` 是 Bun 打包时的条件编译。如果构建时不启用 bridge 模式，整个 bridge 相关代码在产物里完全不存在——不是运行时返回 false，而是物理上不存在。

这解决了一个问题：外部发布版本不需要包含 bridge 相关的字符串常量和逻辑，减少攻击面，也减少对最终用户的混淆。

### 2. claude.ai 订阅状态（isClaudeAISubscriber）

Bridge 连接到 CCR 时，用的是 claude.ai OAuth token（`getBridgeAccessToken()`）。Bedrock、Vertex、API Key 等访问方式没有这个 token，物理上就无法连接 CCR。

注释里的说明：

> Remote Control requires a claude.ai subscription. isClaudeAISubscriber() excludes Bedrock/Vertex/Foundry, apiKeyHelper/gateway deployments, env-var API keys, and Console API logins.

这不是"我们不想支持这些用户"，而是"这些用户根本没有 bridge 需要的凭证"。

### 3. Profile Scope（hasProfileScope）

这一条最容易被忽视，但有实际 bug 历史：

```typescript
if (!hasProfileScope()) {
  return 'Remote Control requires a full-scope login token. Long-lived tokens
  (from `claude setup-token` or CLAUDE_CODE_OAUTH_TOKEN) are limited to
  inference-only for security reasons. Run `claude auth login`...'
}
```

`claude setup-token` 生成的是**推理专用 token**（inference-only），不包含 `user:profile` scope。这类 token 无法调用 `/api/oauth/profile` 获取组织 UUID，导致 GrowthBook gate 因为 target 字段（`organizationUUID`）为空而总是返回 false。

用户看到的现象是"Remote Control is not yet enabled for your account"，但实际原因是 token 类型不对。

添加这一层检查，让错误信息从令人困惑的 GrowthBook gate 失败，变成明确的"你需要重新 login 以获取完整 scope"。

### 4. GrowthBook 功能门控（tengu_ccr_bridge）

GrowthBook 按 `organizationUUID` 进行目标定向（targeting）。这是**逐步灰度**的机制：先向特定组织开放，然后扩大范围，不用在代码里硬编码允许名单。

`checkGate_CACHED_OR_BLOCKING` 的设计值得注意：

- **cached 路径**：读本地磁盘缓存，即时返回（不等网络）
- **blocking 路径**：当缓存显示 false 或不存在时，等待服务器响应（最多 ~5 秒）

为什么需要 blocking？注释解释了：

> Returns cached `true` immediately (fast path). If the disk cache says `false` or is missing, awaits GrowthBook init and fetches the fresh server value (slow path), then writes it to disk. Use at entitlement gates where a stale `false` would unfairly block access.

如果只用缓存，新注册了 bridge 权限的组织，在磁盘缓存过期前，会一直被拒绝访问。blocking check 保证了缓存 false 时主动去服务器确认，避免正当用户被挡在门外。

## 信任来源的分层

这四个检查来自不同的信任来源：

| 检查 | 信任来源 | 失效场景 |
|------|----------|----------|
| feature flag | 构建时静态 | 旧版本二进制 |
| isClaudeAISubscriber | 本地 OAuth token 类型 | API key / Bedrock 用户 |
| hasProfileScope | token 的 scope 字段 | setup-token / env-var token |
| GrowthBook gate | 服务端实时配置 | 组织未开通 / 网络断开 |

**每一层的失效场景不同**，所以不能合并成一个检查。即使服务端说"你可以用 bridge"，如果本地 token 类型不对，也无法建立连接。

## 循环依赖的处理

`bridgeEnabled.ts` 有一段特殊的注释：

```typescript
// Namespace import breaks the bridgeEnabled → auth → config → bridgeEnabled
// cycle — authModule.foo is a live binding, so by the time the helpers below
// call it, auth.js is fully loaded.
import * as authModule from '../utils/auth.js'
```

用 namespace import 而不是具名 import，是为了打破模块循环依赖：`bridgeEnabled` → `auth` → `config` → `bridgeEnabled`。namespace import 产生 live binding，延迟到实际调用时才解析，绕过了循环。

这是 JavaScript ESM 模块处理循环依赖的一个常见技巧。

## 面试指导

"如何设计认证/授权系统"的面试中，把握"多层信任来源"是一个加分项。

不要把所有权限检查汇总成一个 `isAuthorized()` 函数。按照**信任来源分层**：

- 静态（构建时）：决定功能是否存在
- 本地（token 类型）：决定凭证是否合法
- 动态（服务端）：决定权限是否开通

每层独立失败，独立报错，给出不同的修复建议——这比一个统一的"无权限"错误要有用得多。
