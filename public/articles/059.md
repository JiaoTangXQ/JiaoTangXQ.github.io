# 远端会话的消息翻译层：tool_progress 为什么要被压成 system message？

远端 CCR 容器执行工具时，会产生 `tool_progress` 类型的消息——这是工具执行中间状态的通知，不是最终结果。本地消息系统理解 `tool_result`，但不一定理解 `tool_progress`。Claude Code 的解决方案是：把 `tool_progress` 压成本地能理解的 `system` 消息，接受信息损耗，保住统一消息世界。

## 翻译接缝的基本形态

`remote/sdkMessageAdapter.ts` 是远端到本地的消息翻译器。它的工作是把远端 CCR 产生的 `SDKMessage` 流翻译成本地 React 渲染能理解的格式。

```typescript
// 翻译层处理的消息类型分类
// 1. 直接透传：文本消息、工具调用请求、工具结果
// 2. 降级翻译：tool_progress → system message
// 3. 丢弃：纯状态同步消息（本地不需要展示的）
```

透传的消息类型不需要处理，直接进入本地的消息队列。降级翻译是有损的——工具执行进度的详细信息可能在压成 `system` 消息后丢失，用户看到的只是一条简单的状态提示，而不是实时的执行输出。

## 为什么要有翻译损耗

接受翻译损耗的理由是：**本地消息系统不能为了远端场景无限扩展**。

如果本地系统要完全理解 `tool_progress`，它需要：
1. 知道哪个工具在执行（关联到远端的工具调用）
2. 知道进度消息的格式（每种工具可能不同）
3. 在 UI 上正确渲染进度（还要和本地工具的进度展示保持一致）

这等于把远端工具的全部细节都带入本地。成本极高，而且远端工具会持续演化，本地的渲染代码要跟着维护。

选择降级翻译的代价是明确的：用户在远端执行期间看到的进度信息比本地执行期间更简单。这是接受了的损耗。

## 参数注入与 bundle 边界保护

远端会话还有另一个翻译问题：本地客户端需要向 CCR 传递一些初始化参数（比如 `sessionId`、`orgUuid`），但这些参数不能通过修改打包好的 JS bundle 来注入——bundle 是不可变的交付物。

解决方案是**运行时参数注入**：把参数通过 WebSocket 的初始化握手传递，CCR 在启动时从连接上下文里读取，而不是从 bundle 里读取。这保护了 bundle 的不可变性，也避免了「用参数污染打包产物」的问题。

## 统一消息世界的意义

Claude Code 在整个系统里维护一个「统一消息世界」：无论消息来自本地工具调用、远端执行、agent 输出还是系统事件，最终都被翻译成同一套 `Message` 类型结构，存入同一个消息队列，由同一套渲染逻辑处理。

这个统一性的价值在于：

1. **历史可回溯**：所有操作都留在同一条历史流里，不用在「本地历史」和「远端历史」间切换
2. **compact 的可操作性**：上下文压缩时只需要处理一种格式
3. **搜索和过滤**：用户可以在统一的历史里搜索任何操作记录

代价是：翻译层的存在意味着某些原始信息永久丢失。这是明确接受的 trade-off。

## 断线重连的消息连续性

当 WebSocket 断线后重连，`RemoteSessionManager` 面临一个消息连续性问题：断线期间远端产生的消息应该如何处理？

实际上，断线期间的消息会在重连后通过 CCR 的消息回放机制补发。本地客户端通过某个「已接收到的最后一条消息 ID」来告诉 CCR「从这条消息之后开始补」。这个机制保证了消息的连续性，但对消息的实时性有影响——用户可能在重连后突然看到一批之前「消失了」的消息。

## 面试答题思路

「分布式系统的消息格式统一问题」：

1. **统一接口的边界在哪里**：不是所有地方都能统一，先确定统一的范围和不统一的代价
2. **翻译损耗要显式承认**：不要假装翻译是无损的，把损耗文档化
3. **bundle 不可变性是重要的工程约束**：通过运行时注入解决初始化参数问题
4. **消息连续性设计要考虑断线场景**：「发送成功」不代表「对方收到了」，要有补偿机制

追问「如何减少翻译损耗」：可以在本地系统里扩展消息类型，原生支持 `tool_progress`。但这会增加本地系统的复杂度，让本地渲染层耦合远端工具的实现细节。对于 Claude Code 来说，这个代价不值得——接受损耗是更务实的选择。
