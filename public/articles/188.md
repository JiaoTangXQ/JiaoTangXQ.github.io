# 第十三卷回收索引：任务世界里先有同一任务对象，后有前后台之分，说明身份不能跟着姿态散

## 实现链
任务世界里更深的一层，不是“前台还是后台”，而是“那是不是同一份劳动”。  
如果对象身份先散了，后面的姿态切换只会让系统更乱。

这条线真正逼出来的，是第十三卷里关于层间交棒与分层习惯的判断。  
姿态可变，但对象不能跟着散。

这页实际回收的实现链是：
- `05-06-02` 把“先有同一任务对象，后有前台后台之分”讲清，这会在 `13-02-02` 里回收到“层与层之间之所以能交棒，是因为对象身份先被守住了”，说明身份先站稳，显示姿态才有切换空间。

很多系统只盯着前台和后台，没先认出那是同一份劳动。  
Claude Code 更克制，它先守住对象身份，再让姿态跟着现场变化。

## 普通做法
更普通的做法，是把后台动作当成松散的异步句柄，跑起来就算开始，结束了再临时补状态和通知。

## 为什么不用
Claude Code 不这么做，是因为没有统一任务对象，就没有稳定的生命周期、输出、通知和回看链，前后台切换也会变成一串彼此不认账的临时处理。

## 代价
代价是任务状态字段、输出文件、通知协议和前后台转场逻辑都会变厚，维护者要记住更多接缝。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
