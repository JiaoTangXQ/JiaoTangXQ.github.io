# 产品设计：/chrome 状态面板为什么要把「当前连接」和「怎么管理」放在同一块？

## 「功能开关」和「现场面板」的区别

很多功能的设置界面是这样的：
- 一个开/关切换
- 也许有一个"了解更多"链接
- 出了问题去文档找

Claude Code 的 `/chrome` 命令是另一种设计方向——一个现场面板，把以下内容都放在一起：

- 当前连接状态（已连接/未连接/扩展未安装）
- 扩展安装状态（是否检测到）
- 命令行开关的使用说明（`--chrome` 参数、`CLAUDE_CODE_ENABLE_CFC` 环境变量）
- 站点权限管理的指引（去哪里管理哪些网站有权限）
- 默认开关的当前值和修改方法

这些信息来自不同的技术层面，但它们都和同一个操作目标有关：**让 Claude in Chrome 在当前机器上正常运行，并且知道如何控制它的行为**。

## 为什么连接状态和使用说明要放在一起？

如果用户不知道 `--chrome` 这个 CLI flag，他们会怎么找？
1. 记得有这个功能，但忘了怎么开
2. 打 `claude --help`，可能找到，可能找不到
3. 去文档里搜索
4. 或者直接打 `/chrome`，在这里看到说明

对于用户来说，最重要的时机是**他们正在思考"我要用浏览器功能"的时候**。在这个时候同时告诉他们"现在是什么状态"和"怎么控制它"，比把这两个信息放在不同地方更自然。

这种设计减少了用户的心智跳跃：他们不需要先查当前状态，再查命令行用法，再把两个信息在脑子里合并。

## toolRendering.tsx 里的 [View Tab] 链接

在 `toolRendering.tsx` 里，有一段当 Claude 正在操作浏览器标签时的 UI 渲染：

```ts
// 追踪 Agent 操作的标签 ID
trackClaudeInChromeTabId(tabId)

// 如果终端支持超链接，渲染 [View Tab] 链接
if (supportsHyperlinks) {
  return `[View Tab](chrome-extension://...?tabId=${tabId})`
}
```

这个 `[View Tab]` 链接是把"工具调用状态"和"用户可操作入口"连在一起的设计。

用户在终端里看到 Agent 说"我在标签页 ABC 上操作"，如果好奇或者想监督，可以直接点击 `[View Tab]` 跳到那个标签页。这不只是信息展示，而是直接给了用户一个进入现场的途径。

## 现场面板的信息密度设计

现场面板和普通设置页面的区别，在于信息密度和目标的匹配：

**普通设置页**：信息组织清晰，每个设置项独立，适合用户想"了解所有选项"时用。

**现场面板**：信息围绕"此刻的目标"组织，把完成一个任务需要的所有信息聚合，减少跳转。

`/chrome` 面板更接近现场面板。用户打开它，通常是因为：
1. 想确认浏览器功能是否在工作
2. 出了问题想快速找到修复方法
3. 想知道如何在当前任务里开启或关闭浏览器功能

这三个场景需要的信息都在同一个地方，而不是分散在三个不同的文档页面里。

## 连接状态检测的时机

`ClaudeInChromeOnboarding.tsx` 会在渲染时调用扩展检测逻辑，而不是从全局状态读取一个静态值。这意味着每次用户打开 `/chrome` 面板时，都会做一次新鲜的扩展检测。

这个设计选择背后的判断：**扩展安装状态是一个可能随时变化的外部状态，不适合长时间缓存**。用户可能刚刚安装了扩展，或者刚刚卸载了扩展。每次查看状态时做新鲜检测，显示的是真实的当前状态，而不是上次检测时的状态。

相比之下，`shouldAutoEnableClaudeInChrome()` 里的缓存逻辑（只缓存正向检测）是为了启动时不阻塞，不是为了 UI 展示。UI 应该尽量展示最新状态。

## 面试考察点

这道题考察的是**产品信息架构**和**操作型界面设计**。

一个关键的设计问题是：**什么时候应该把信息分散到多个页面，什么时候应该集中在一个面板？**

判断原则：如果用户在完成一个任务时需要访问多处信息，而这些信息都属于同一个上下文，就应该把它们放在一起。

Claude in Chrome 的安装、连接、权限、开关这些信息，都属于"我要用浏览器功能，怎么确保它工作正常"这个上下文。集中展示比分散展示让用户少花时间，少犯错。
