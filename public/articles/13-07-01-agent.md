普通读者最该学走的，不是某个 prompt 句式，而是 Claude Code 在代码里反复坚持的四分法。`commands.ts` 这类入口层解决的是“用户和系统怎么开始说话”；`tools.ts` 和 `Tool.ts` 解决的是“系统能伸哪些手、这些手的协议是什么”；`Task.ts` 解决的是“劳动怎样变成正式工作单元”；`AppStateStore` 解决的是“界面当前认定什么是真的”。把这四件事分开以后，模型能力才有稳定的落脚点。

更常见的失败写法，是把这四件事混在一个 agent 主循环里：输入一来就顺手决定权限、直接调工具、直接改 UI、顺便记一点历史。这样一开始当然快，但功能一长、场景一变，整套系统会迅速变成一团难以追责的泥。

Claude Code 给普通读者最有价值的启发，不是“它有哪些招”，而是“它为什么先分清楚谁管入口、谁管行动、谁管制度、谁管显示”。这套分工习惯比任何单独技巧都更可迁移。

## 实现链
`commands.ts`、`tools.ts`、`Task.ts`、`AppStateStore` 展示了入口、行动、制度和显示四层应当分开。

## 普通做法
一上来就先堆能力清单。

## 为什么不用
因为不分层时，能力越多越像一团线。

## 代价
学这套方法要接受前期结构设计更重。
