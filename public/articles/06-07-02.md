# fanOut 为什么必须先重置缓存再通知？N 个监听者的磁盘读取问题

这是一个性能 bug 的修复历史，背后的工程逻辑值得细读。

## 问题背景

settings 变化传播的架构是：文件监听 → `fanOut` → 多个监听者。监听者包括 `useSettingsChange`（React hook）、`applySettingsChange`（AppState 更新函数）等。

每个监听者需要拿到最新的 settings，通常通过 `getSettingsWithErrors()` 或 `getInitialSettings()` 实现，这些函数会读 settings 缓存，缓存未命中时读磁盘。

## 旧版本的问题

之前每个监听者都在自己的处理函数里重置缓存：

```typescript
// 旧版本：每个监听者各自重置
function handleSettingsChange() {
  resetSettingsCache()    // 各自重置
  const settings = getSettingsWithErrors()  // 读磁盘
  // ...
}
```

当有 N 个监听者时，触发顺序是：
1. 监听者 1：重置缓存 → 读磁盘（miss）→ 填充缓存
2. 监听者 2：重置缓存（清空了 1 填充的！）→ 读磁盘（miss）→ 填充缓存
3. 监听者 3：重置缓存 → 读磁盘 → 填充缓存
4. ...

N 个监听者 = N 次磁盘读取。注释里记录的实测数据：

> “Profile showed 5 loadSettingsFromDisk calls in 12ms when remote managed settings resolved at startup.”

5 个监听者，12ms 内 5 次磁盘读取。大部分时间浪费在重复 I/O 上。

## 修复后的 fanOut

```typescript
// changeDetector.ts
function fanOut(source: SettingSource): void {
  resetSettingsCache()       // 只重置一次，在通知之前
  settingsChanged.emit(source)  // 通知所有监听者
}
```

现在的顺序：
1. 缓存重置（一次）
2. 通知监听者 1：读 settings → 缓存 miss → 读磁盘 → 填充缓存
3. 通知监听者 2：读 settings → **缓存 hit** → 直接返回
4. 通知监听者 3：读 settings → **缓存 hit** → 直接返回

N 个监听者，只有 1 次磁盘读取（第一个监听者触发）。

## 为什么顺序很关键

如果顺序颠倒——先 emit 再 reset：

```typescript
// 错误顺序
function fanOut(source: SettingSource): void {
  settingsChanged.emit(source)  // 先通知
  resetSettingsCache()           // 后重置（缓存已被填充，然后被清空！）
}
```

监听者 1 在 emit 里读磁盘，填充缓存。emit 返回后，resetSettingsCache 把缓存清空。下次有人读 settings 时又要重新读磁盘。

## INTERNAL_WRITE_WINDOW_MS：自己写的文件不通知

```typescript
const INTERNAL_WRITE_WINDOW_MS = 5000  // 5 秒

function handleChange(path: string): void {
  if (consumeInternalWrite(path, INTERNAL_WRITE_WINDOW_MS)) {
    return  // 内部写入，忽略
  }
  // ...
  void executeConfigChangeHooks(...).then(results => {
    if (hasBlockingResult(results)) return
    fanOut(source)
  })
}
```

当 Claude Code 自己修改 settings 文件（比如用户点了”总是允许”按钮），代码会先调用 `markInternalWrite(path)`，然后写文件。文件系统事件触发时，`consumeInternalWrite` 检测到这是内部写入，跳过通知。

5 秒窗口的设计意图：写文件后 5 秒内，如果有同路径的文件变化事件，都当作是自己写的忽略。5 秒是经验值，足够覆盖文件系统的延迟。

## MDM 轮询：无法用 chokidar 监听的配置源

```typescript
const MDM_POLL_INTERVAL_MS = 30 * 60 * 1000  // 30 分钟

function startMdmPoll(): void {
  const initial = getMdmSettings()
  lastMdmSnapshot = jsonStringify({ mdm: initial.settings, ... })

  mdmPollTimer = setInterval(async () => {
    const { mdm: current, ... } = await refreshMdmSettings()
    const currentSnapshot = jsonStringify({ mdm: current.settings, ... })

    if (currentSnapshot !== lastMdmSnapshot) {
      lastMdmSnapshot = currentSnapshot
      setMdmSettingsCache(current, ...)
      fanOut('policySettings')
    }
  }, MDM_POLL_INTERVAL_MS)

  mdmPollTimer.unref()  // 不阻止进程退出
}
```

MDM 设置来自注册表或 plist，不是文件系统，chokidar 无法监听。唯一的方式是轮询：每 30 分钟快照一次 MDM 状态，如果变化了就触发 fanOut。

`mdmPollTimer.unref()` 确保 MDM 轮询不会让进程一直活着——这是 Node.js 的 timer ref/unref 机制，unref 的 timer 在没有其他活跃任务时不阻止进程退出。

## 这道题考什么

**缓存与通知的顺序**是一个考察”是否考虑过资源竞争”的好题目。

正确回答需要：
1. 解释为什么”N 个监听者各自重置缓存”是 N 次磁盘读取
2. 解释 fanOut 的集中重置如何变成 1 次磁盘读取（第一个命中 miss，后续命中 hit）
3. 解释为什么”先 emit 再 reset”是错误的（填充又清空）

加分项：能提到 MDM 轮询和 `unref()` 的意义，说明对 Node.js 事件循环和进程生命周期有具体理解。
