# 为什么 computerUse 和 browser 能力不能有自己的一套审批流程？

## 架构总结：新世界怎么进场

Claude Code 能力扩张的历史可以从代码里读出来：`setupComputerUseMCP()`、`setupClaudeInChrome()`、`remotePermissionBridge`、`LocalAgentTask`——每一个都是一个"新世界"的进场。

问题是：新世界进来时，它的审批、生命周期和状态管理要怎么处理？是给它单独发明一套新协议，还是让它先学旧世界的话？

Claude Code 的选择是后者，而且这不只是风格偏好，而是有具体的工程理由。

## setupComputerUseMCP 和 setupClaudeInChrome 的翻译层

`setupComputerUseMCP()` 没有为 computerUse 能力建立专属的权限系统。它做的是：把 computerUse 相关的工具包装成标准 MCP 配置，用统一的工具命名方式注册，再决定哪些工具因为整场会话授权可以绕过单次确认。

这样，已有的权限链不需要知道"这是 computerUse 工具还是普通工具"，它只需要看到一个符合格式的工具定义就可以继续工作。

`setupClaudeInChrome()` 同理，并且增加了一个关键决定：默认在非交互场景禁用浏览器能力。不是"把浏览器能力开放，再靠用户小心"，而是"先关掉，显式打开"。这是制度先行在新能力接入时的具体体现。

## remotePermissionBridge 的翻译逻辑

前面已经提到过：远端权限请求通过 synthetic assistant message 和 stub 工具翻译成本地审批链能处理的格式。

这里要强调的是翻译的方向：是远端工具**学**本地审批链的语言，而不是本地审批链为了远端工具改造自己。

这个方向很重要。如果让本地审批链为远端工具改造，那每来一种新的远端工具类型，审批链就要改一次。如果要求新工具先翻译成本地格式，那审批链可以保持稳定，只有翻译层随新需求变化。

## LocalAgentTask 的任务托管

Subagent 是 Claude Code 里另一个"新世界"——它是在当前进程内异步运行的另一个 Claude 实例。它的进场方式是：被做成 `LocalAgentTask` 对象，有 `taskId`、`outputFile`、`status`、`notified`，有明确的进场时机和回收时机。

不是让 subagent "以后台线程身份到处飘"，而是第一天进来就得是个任务对象。这样 `AppStateStore` 里的任务面板能追踪它，`isTerminalTaskStatus()` 能判断它死没死，回收逻辑有固定触发点。

## 为什么宁可写翻译层

多写翻译层、stub、包装器和任务外壳，看起来比"给每个新世界单独发明一套协议"更麻烦。但从长期看，这是更小的代价：

- 翻译层出问题，只影响那个新能力的接入，不影响主干
- 主干制度升级时，翻译层按新接口更新，不需要改每个新世界各自的逻辑
- 用户只需要学一套心智模型，不是给 computerUse 一套、给 remote 一套、给 subagent 一套

这不是因为 Anthropic 的工程师特别自律，而是因为不这样做的代价是"扩张越多，系统越像几个拼起来的产品"。

## 代价

接入层会变厚。新能力要进场，先得搞清楚"旧世界的话"是什么语法，再写翻译层。这提高了每个新能力的接入成本。

不过相较于维护两三套并行运行时，这种代价还是小得多。

## 面试指导

被问"你会怎么把一个新能力（比如浏览器控制）接进一个已有的 agent 系统"时，这套思路是一个有深度的答案：

先问"旧系统的哪些结构可以复用"，而不是"新能力需要什么新协议"。把新能力包装成旧系统能处理的格式，比改造旧系统来适配新能力，通常是更低成本、更可维护的选择。
