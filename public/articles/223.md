# Claude Code 的启动预热为什么分成「首屏前」和「首屏后」两段？

## 性能的本质矛盾

启动时要做的事情太多：加载插件、预热 API key、检查更新、注册 hooks、初始化 MCP 服务器……

如果全部串行完成再渲染第一帧，用户会等很久。

如果全部并发，CPU 和 I/O 竞争会拖慢启动，而且有些任务在首屏渲染前必须完成（比如 API key 验证），否则渲染出来的界面就是错的。

解法：**把任务分成首屏必需和首屏可延迟两类**，分别处理。

## setup.ts 里的两段式结构

`src/setup.ts` 里有一个明显的 checkpoint 分界：

```typescript
profileCheckpoint('setup_before_prefetch')
// Pre-fetch promises - only items needed before render
```

注释「only items needed before render」是关键。首屏前的预热只包含渲染第一帧所必需的内容。

跨过这个 checkpoint 后，大量工作是「fire and forget」式的：

```typescript
// 命令加载 — void，不等完成
void getCommands(getProjectRoot())

// 插件 hooks — 动态 import + void
void import('./utils/plugins/loadPluginHooks.js').then(m => {
  void m.loadPluginHooks()
  m.setupPluginHookHotReload()
})

// attribution hooks — setImmediate，推到下一个 tick
setImmediate(() => {
  void import('./utils/attributionHooks.js').then(({ registerAttributionHooks }) => {
    registerAttributionHooks()
  })
})
```

注意 `setImmediate`——这不只是 `void`，而是显式推到当前微任务队列执行完之后。注释解释了原因：

> Defer to next tick so the git subprocess spawn runs after first render rather than during the setup() microtask window.

git subprocess spawn 必须在第一次渲染**之后**，不能在 setup 的 microtask 窗口内，否则会和渲染争夺事件循环。

## 跳过条件：bare 模式和同步安装模式

有些预热在特定模式下完全跳过：

```typescript
const skipPluginPrefetch =
  (getIsNonInteractiveSession() &&
    isEnvTruthy(process.env.CLAUDE_CODE_SYNC_PLUGIN_INSTALL)) ||
  isBareMode()
```

两种跳过情况：
1. `CLAUDE_CODE_SYNC_PLUGIN_INSTALL`：同步安装插件时，异步预热会和安装过程产生 race condition
2. `isBareMode()`：脚本调用模式下插件 hooks 永远不会被执行，预热是纯浪费

这是务实的判断：预热只在「预热收益大于零」的情况下才做。

## StreamingToolExecutor 的类似思路

`StreamingToolExecutor` 的并发控制也体现了类似的分段思维：

并发安全的工具（只读类）可以立即开始执行，不需要等其他工具完成。非并发安全的工具（写操作类）必须等前面的工具全部完成后才能启动。

这是「能提前的提前，必须等的等」原则的直接应用：不是「全部串行」，也不是「全部并发」，而是按安全性分类后分别处理。

## 工程启示

这种分段式预热的思路适用于任何有首屏体验要求的系统：

1. 列出所有启动时要做的工作
2. 标记哪些是「用户看到第一帧之前必须完成的」
3. 其余全部异步化，能 `setImmediate` 就 `setImmediate`
4. 对有竞态风险的任务，显式设计 skip 条件

启动时间 = 首屏必需任务的串行时间。其他所有工作都应该从这条路径上挪开。

## 面试场景

「你们的 CLI 工具启动很慢，怎么优化？」

这个模式是一个直接可用的框架：把启动任务按首屏必需/可延迟分类，延迟任务用 setImmediate 推出首屏窗口，对有竞态的场景加 skip 条件。Claude Code 的 `setup.ts` 就是这个框架的实际应用。
