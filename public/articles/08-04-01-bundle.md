# 参数注入是在保护bundle边界

## 实现链
- `initReplBridge.ts` 把 `cwd`、`sessionId`、`branch`、`gitRepoUrl`、`getAccessToken()`、`createSession()`、`toSDKMessages()` 这些依赖全部显式塞给 `initBridgeCore()`，而不是让 core 自己去 import bootstrap 状态。
- 注释写得很明白：这样拆是为了不把 `sessionStorage`、`commands.ts` 乃至整棵 React 树拖进 Agent SDK bundle。

## 普通做法
- 更直觉的写法是让 core 自己 import 所需模块，哪里需要数据就去哪里拿。
- 对单一 REPL 进程来说，这么写往往最省事。

## 为什么不用
- Claude Code 没这样写，因为 bridge core 还要服务 daemon/SDK 场景。只要它直接碰 bootstrap 状态，打包边界就会被 REPL 专用依赖打穿。
- 所以这里的参数注入不是形式主义，而是在保护 bundle 边界。

## 代价
- 代价是函数签名很长，读起来不像“优雅 API”。
- 但这笔丑成本换来了清楚的依赖边界，尤其对多入口产物来说很值。
