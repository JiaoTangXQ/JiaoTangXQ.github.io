# 附件注入为什么会把文件、记忆和技能发现重新带回下一轮

工具跑完以后，Claude Code 并不会立刻把这些结果草草塞回模型，然后开始下一次思考。它会先做一件更像运行时整理现场的事: 把这一轮刚刚发生过、但模型下一轮还需要知道的系统事实，重新装配成附件带回去。

这些附件的来源很多。排队中的任务通知和用户提示会被转成可继续响应的附件；文件改动、任务输出、模式切换等事实会被补回会话；异步预取好的相关记忆会在合适时机注入；技能发现也会作为附件重新出现。换句话说，Claude Code 不是只把“工具结果文本”接回下一轮，而是在把这台系统刚刚变化过的世界状态重新讲给模型听。

这个设计特别高明，因为很多重要信息并不天然长在 `tool_result` 里。比如新发现的技能、应该注意的记忆、后台任务刚刚传回来的通知，它们都属于“系统事实”，却不属于某个单独工具的直接输出。Claude Code 用附件层把这些事实补齐，等于替下一轮思考把舞台重新布置了一遍。

所以附件不是装饰，也不是附件上传功能的延伸。它更像主循环的缝合线。没有这一层，系统虽然还在运行，但每一轮之间会丢掉很多刚刚获得的现场感。

代码上，这不是一句泛泛的“保留上下文”，而是 `query()` 在组下一轮消息时把附件消息、文件记忆、技能发现之类的结构化对象一并挂回消息链；`attachments.ts` 也专门区分普通用户文本和附件消息，不把它们混成一坨 prompt 文本。更简单的做法是把所有上一轮结果折成一段自然语言摘要，再让模型自己从摘要里领会“有哪些文件、资源和技能值得记住”。

Claude Code 没走纯摘要路线，是因为附件和发现结果本来就有类型差异。把它们压扁成文字，模型还能不能稳定认出“这是一条文件记忆，不是一句闲聊提示”，就要靠运气了。代价是消息体系更复杂，读 transcript 时也需要知道哪些块属于附件层，不是普通会话句子。

## 实现链
下一轮继续时会把附件消息、文件记忆、技能发现等结构化对象重新带回消息链。

## 普通做法
把这些都压成自然语言摘要。

## 为什么不用
因为它们本来就有类型差异，压扁后很难稳定识别。

## 代价
消息体系和渲染都更复杂。
