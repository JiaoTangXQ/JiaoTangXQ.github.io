# 为什么 launchRepl 更像交棒，而不是高潮

很多人看到启动链，直觉上会以为真正进入 `REPL` 的那一刻才是高潮，好像所有复杂度都会在这里汇总爆发。但 Claude Code 恰恰反着做。`launchRepl()` 本身非常薄: 动态加载 `App`，动态加载 `REPL`，然后把控制权交给 `renderAndRun()`。它不像一场大演出，更像接力赛里那一下正式交棒。

这不是偷懒，而是一种很成熟的克制。因为真正决定这台系统能不能安全、稳定、顺滑地活起来的事情，大多已经在前面做过了: 模式分流、环境准备、工作现场确定、trust 边界确认、必要文件下载、首轮消息准备。等走到 `launchRepl()`，系统需要的不是再来一个巨大的总控函数，而是尽快把已经搭好的世界交给真正负责长期运行的界面和会话层。

这样的设计有个很明显的好处: 启动和运行不会互相糊成一团。前面的代码负责把舞台搭稳，后面的 `REPL` 自己负责长期表演。中间这一下越薄，边界越清，后面整本书也越容易讲清楚 Claude Code 到底是在哪里准备世界，在哪里真正开始长期工作。

## 实现链
这页对应的实现链都落在 REPL 交棒和 deferred prefetch 这套双节奏上：首屏前只做那些不做就没法安全进入 REPL 的事，首屏后再继续刷新、预热、建议和补充装配。代码在管理的不是单纯步骤，而是用户感知到的节奏。

## 普通做法
更普通的启动要么全部做完再显示，要么一股脑都异步开跑，谁先完算谁。

## 为什么不用那种做法
Claude Code 两种都没选死，因为前者太慢，后者太乱。它要的是先给用户可工作的前台，再让后台继续补齐。

## 代价与回报
回报是等待被重新分配。代价是启动完成这件事在代码里不再是单一时刻，而是一段前台和后台交错的过程。
