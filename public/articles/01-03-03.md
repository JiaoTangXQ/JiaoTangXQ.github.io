# 代码发现：commands.ts 里有一段你可能以为多余的过滤逻辑，删掉会怎样？

## 发现这段代码

`commands.ts` 里有两个相关但行为不同的东西。

第一个是 `COMMANDS()` 函数，它用 `memoize` 包住，返回一个完整的 `Command[]` 数组。这个数组只生成一次，后面每次调用都返回同一个引用。命令定义本身（名字、描述、类型）确实是静态的，用 memoize 合理。

第二个是 `getCommands()` 函数（注意不是 `COMMANDS()`）。它不被 memoize。每次调用，它都会重新过一遍两个检查：`meetsAvailabilityRequirement(cmd)` 和 `cmd.isEnabled?.()`.

注释里直接写明了原因：`Not memoized — auth state can change mid-session (e.g. after /login), so this must be re-evaluated on every getCommands() call.`

## 如果删掉这个设计

假设你认为这是性能浪费，把 `getCommands` 也用 `memoize` 包上：

1. 用户以匿名状态启动 Claude Code，此时 `isClaudeAISubscriber()` 返回 false，claude.ai 专属命令（`availability: ['claude-ai']`）不出现在命令列表里。
2. 用户运行 `/login`，完成 OAuth 认证。
3. 此后调用 `getCommands()`，因为结果已被缓存，仍然返回没有 claude.ai 命令的旧列表。
4. 用户刷新不了，只能重启整个 Claude Code 才能看到新命令。

`meetsAvailabilityRequirement` 里有一段值得看的逻辑：它的三种可用性标签（`claude-ai`、`console`、其他），每种对应一组不同的运行时检查——是不是 claude.ai 订阅用户、是不是直接 API 用户且不走第三方服务、URL 是不是第一方 Anthropic 域名。这些判断依赖认证状态，认证状态是可变的。

## 命令层的真正职责

这个设计细节揭示了命令层的角色：它不是一个静态的功能目录，而是一个**根据当前状态动态决定哪些入口可用**的分流系统。

命令的类型系统也能说明这一点。`Command` 接口（定义在 `types/command.ts`）有几种不同的 `type`：
- `prompt` 类型：把一段提示词装进当前会话，继续走模型
- `local` 类型：在本地直接执行，不经过模型
- `local-jsx` 类型：临时拉起一段 Ink/React 界面

这三种类型的执行路径完全不同。`/help` 是 `local` 类型，直接在终端打印一段文字，不占用 token。`/compact` 会触发压缩逻辑。`/login` 会拉起一段 OAuth 流程界面。

命令层问的问题是：**这次用户的意图，应该用哪种方式导入系统？** 而不是"这件事怎么做"。

## 和工具层的边界

命令不应该也不能做工具做的事——它们没有 Zod `inputSchema`、没有 `isConcurrencySafe`、没有 `validateInput`、没有 `userPermissionDecision`。这不是遗漏，而是命令根本不需要这些。

工具是模型调用的行动能力，需要输入校验、权限检查、并发属性、结果格式。命令是用户直接触发的入口，需要可用性过滤、认证感知、执行方式选择。

如果把命令和工具混成一层，最直接的问题是：权限系统去哪里插？命令不过权限检查，工具要过——合并之后无法建立统一的检查点。

## 面试角度

这道题的考点是：能不能通过一个"看起来多余的设计细节"，反推出系统的架构意图。

**高分答法：** 先说"这里不用 memoize 是有意为之"，然后解释可用性和认证状态耦合的原因，最后说明这个动态性是命令层作为"入口"而非"功能"的具体体现。

**常见错误：** 说"memoize 更好，应该优化"，或者说"这是为了灵活性"而不能说清楚灵活性体现在哪个具体场景。

**追问陷阱：** 如果面试官问"那 `COMMANDS()` 为什么可以 memoize？"——答案是命令定义本身（名字、描述、类型结构）是静态的，可变的只是"哪些命令现在可用"这个判断，两者分开处理。
