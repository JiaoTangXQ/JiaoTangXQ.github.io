# 为什么长期会话和单轮执行必须先分层

如果把一个 agent 的全部逻辑都塞进一条大循环里，短时间内看起来会很省事：收到一句话，拼上下文，调模型，跑工具，再继续下一轮。但只要会话变长，这种写法很快就会失控，因为“这一整段会话长期要记住什么”和“这一轮眼下要推进什么”其实不是同一种问题。

Claude Code 在这里做了一个很关键的分层。`QueryEngine` 托管的是长期事实：已有消息、累计使用量、权限拒绝记录、文件缓存、技能发现、嵌套记忆路径、会话级 abort controller。它像这场对话的长期外壳，负责让同一场会话可以一轮接一轮地活下去。`query()` 则只管回合推进：这一轮要用什么上下文、工具如何调度、压缩是否触发、失败后用什么理由继续、这一轮什么时候真的结束。

这不是形式上的拆文件，而是时间尺度上的拆层。长期会话关心的是“哪些事实要跨轮保留”，单轮执行关心的是“当前这一轮怎样往前推”。如果把两者混在一起，重试、压缩、回退、继续生成这些动作就会不断碰撞会话级状态；如果把两者切得太开，外层又会失去对内层的持续托管能力。Claude Code 把会话外壳和回合引擎分开，正好避开了这两个极端。

这个设计像一家餐厅的前厅和后厨。前厅负责维持这桌客人的持续关系，知道他们是谁、点过什么、还有哪些偏好；后厨负责眼前这一道菜怎样做完、出了问题怎样补救、什么时候继续下一道。真正优雅的地方，不在于它把结构拆得多漂亮，而在于它承认“长期持续”与“单轮推进”本来就是两种不同的工程节奏。

代码上，这种分层不是概念口号，而是 `QueryEngine.submitMessage()` 和 `query()` 的硬分工。`QueryEngine` 维护 `mutableMessages`、`permissionDenials`、`readFileState`、`totalUsage`、会话级 `abortController`，先把用户输入整理好、落进 transcript，再把已经成形的会话状态交给 `query()`；`query()` 自己则维护 `State.transition`、工具执行器、fallback、compact、max token recovery 和 stop hooks 这些“这一轮怎么继续”的临时推进逻辑。更普通也更容易想到的写法，是在一个大循环里一边改会话历史、一边做当前回合判断，看起来文件更少、参数更少。

Claude Code 没这么写，核心原因是恢复和续跑。只要有 `/resume`、compact、fallback、tool_result 补写和 stop hook 重试，这条链就不再是“这一轮跑完就算完”的一次性函数，而是“会话壳持续托管，回合引擎反复进出”的结构。这样做更合理，但代价也明确：状态在两层之间来回传，`messages`、`toolUseContext`、`transition` 的 handoff 比单循环难读，理解成本确实更高。

## 实现链
`QueryEngine.submitMessage()` 托管会话级状态，`query()` 托管单轮推进和 `transition` 状态机。

## 普通做法
把长期状态和单轮推进写进一个大循环。

## 为什么不用
因为 `/resume`、compact、fallback 和 tool_result 补写都要求长期壳与单轮引擎分开负责。

## 代价
状态交接更多，理解成本更高。
