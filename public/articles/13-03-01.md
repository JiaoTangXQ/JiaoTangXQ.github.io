# remotePermissionBridge 为什么要造一个假的 AssistantMessage？

## 彩蛋级设计：你知道吗

这个问题在面试中出现的概率不高，但如果出现了，它是在考你有没有真正读懂 Claude Code 里最有意思的那些接缝设计——那些名字不响、但专门替两层世界说人话的小连接件。

`remotePermissionBridge.ts` 就是这类连接件的典型。

## 问题在哪里

远端 session 里，模型在另一台机器上跑，工具调用在远端进程里发生。但用户的权限确认 UI 在本地——用户需要在本地看到"这个工具要做什么"然后决定允不允许。

问题来了：本地的权限确认链设计上依赖 `AssistantMessage` 对象——它需要从消息里找到工具调用的上下文，才能渲染出"你确定要允许这个操作吗"的界面。但远端发来的只是一个权限请求事件，没有真正的本地 `AssistantMessage`。

## synthetic message 的解法

`remotePermissionBridge.ts` 的解法是：**造一个**。

它在本地构造一个 synthetic assistant message，包含足够的工具调用信息让确认链能正常工作；同时为未知远端工具造一个最小的 `Tool` stub，让本地 UI 能渲染这个工具的名称和描述。

这样，本地的整套权限确认 UI 不需要知道"这是远端工具还是本地工具"，它只需要看到一个符合格式的 `AssistantMessage` 就可以继续工作。

**接缝件的价值就在这里**：它不试图改造权限确认链，也不为远端工具发明一套新的审批 UI，而是专门替两个世界之间的缝隙补平。

## mcpSkillBuilders.ts 的同类逻辑

`mcpSkillBuilders.ts` 也是同一类设计，只是更不起眼。它做的事情只是注册两段 builder，但这正好避开了 Bun 打包下动态 import 和依赖环同时爆炸的问题。没有它，MCP 技能发现就需要重造一套加载器；有了它，原来的加载系统继续工作。

## LocalAgentTask 的接缝字段

`LocalAgentTask.tsx` 里有几个看起来不华丽的字段：`pendingMessages`、`retain`、`diskLoaded`、`evictAfter`。这些字段在悄悄保证一件事："子 agent 的输出何时进场、何时可回看、何时可回收"，不靠 UI 碰运气，而是有明确的触发条件。

这就是接缝零件做对了时的样子：不抢主干的话语权，只把主干覆盖不到的角落补平。

## 更普通的做法

更普通的实现，是在每个调用点各写一点特殊判断：远端工具本地不认识就临时 if 一下，MCP 技能加载有环就多写个旁路脚本，子 agent 输出先糊到界面上再说。

短期内更快。长期会把接缝逻辑撒得到处都是，最后谁也说不清哪一层在兜底。

## 接缝零件的代价

接缝零件多了，系统会显得碎。读代码时不容易一眼看出为什么存在这个文件，因为它的价值不在于它自己多聪明，而在于它让别的东西继续工作。

维护接缝零件需要的是系统性记忆：你得记住"这个连接件是为了哪两层之间的哪个问题而存在的"。

但大系统真正可用，靠的往往就是这些不起眼的补缝件。

## 面试指导

当面试官问"你印象最深的是哪个设计细节"时，`remotePermissionBridge` 的 synthetic message 是一个很好的答案——它展示了你理解了"接缝件不发明新框架，而是让旧框架继续工作"这个设计原则。

这比泛泛说"代码分层清晰"要有具体说服力得多。
