真正见功力的地方，常常不是那些名字很响的大模块，而是专门替两层世界说人话的小连接件。`remotePermissionBridge.ts` 里，远端权限请求并不会因为本地没有真实 `AssistantMessage` 就直接跳过现有审批链，而是先造一个 synthetic assistant message，再给未知远端工具造一个最小 `stub`，这样本地的确认 UI 还能继续工作。`mcpSkillBuilders.ts` 看起来更不起眼，它做的事情只是注册两段 builder，但这正好避开了 Bun 打包下动态 import 和依赖环同时爆炸的问题，让 MCP 技能发现不必为了一点接缝重造一套加载器。`LocalAgentTask.tsx` 也一样：`pendingMessages`、`retain`、`diskLoaded`、`evictAfter`、任务通知 XML 这些字段都不华丽，却在悄悄保证“子 agent 的输出何时进场、何时可回看、何时可回收”不会靠 UI 碰运气。

更普通的做法，是在每个调用点各写一点特殊判断：远端工具本地不认识就临时 if 一下，MCP 技能加载有环就多写个旁路脚本，子 agent 输出先糊到界面上再说。短期内这样更快，但长期会把接缝逻辑撒得到处都是，最后谁也说不清哪一层在兜底。

Claude Code 这些“小零件”的价值就在这里：它们不试图发明第二套主干，而是专门替主干补缝。代价当然有，接缝零件一多，系统会显得碎，也更考验维护者记忆力。但大系统真正可用，靠的往往就是这些不起眼的补缝件。

## 实现链
`remotePermissionBridge`、`mcpSkillBuilders`、`LocalAgentTask` 这类小零件负责把不同世界的接缝补平。

## 普通做法
在每个调用点临时补 if/else 特判。

## 为什么不用
因为接缝逻辑一旦到处散落，久了就没人知道哪层在兜底。

## 代价
零件会多，系统看起来更碎。
