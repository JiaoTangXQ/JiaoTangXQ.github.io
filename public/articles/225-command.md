# 不同来源的命令如何收编成同一种 Command 语言

面试题版本：**"Claude Code 里 `/compact`（内建命令）、`/myskill`（技能目录命令）、`/mcp-tool`（MCP 命令）的底层实现完全不同，但用户体验是一致的。这个一致性是怎么实现的？"**

## 命令的统一外观

从用户角度看，所有 slash 命令都一样：输入 `/命令名 参数`，得到结果。不需要知道这个命令是在本地执行还是调用模型还是触发 MCP 操作。

这个一致的外观背后，是一个统一的 `Command` 数据结构。不管命令来自哪里，最终都要被表示为：

```typescript
type Command = {
  name: string           // 命令名称，用于查找和显示
  type: CommandType      // 'prompt' | 'local' | 'local-jsx'
  source: CommandSource  // 'builtin' | 'skill' | 'mcp' | 'plugin'
  // 权限和可见性
  visibleInHelp: boolean
  effectiveSkipSlash?: boolean
  // 执行所需的内容
  // (根据 type 不同，结构不同)
}
```

`findCommand()` 在这个统一的命令表里查找，不需要知道命令来自哪个来源。权限检查、帮助文本生成、自动补全都基于这个统一结构。

## 内部机制的分离

统一的外观不意味着统一的执行机制。`getMessagesForSlashCommand()` 在拿到 Command 对象后，按 `command.type` 分流：

- `'prompt'`：把命令内容注入到消息里，推进主循环（模型参与处理）
- `'local'`：直接调用本地模块，把结果写回会话（不经过模型）
- `'local-jsx'`：启动一个 Ink 组件作为临时界面，完成后回到会话

三种类型，三种执行路径，但对用户来说都是"输入命令，得结果"。

## 为什么要统一

**调用方不需要知道来源**：命令自动补全、帮助文档生成、权限检查——这些系统只需要遍历命令表，不需要为不同来源的命令写不同的逻辑。

**来源可以继续扩展**：加入一种新的命令来源（比如 GitHub Actions 触发的命令），只需要：
1. 实现把新来源的命令转成 `Command` 结构的适配器
2. 在命令加载时把新来源加入命令表

所有消费命令的代码（自动补全、权限检查、执行分发）不需要修改。

**用户心智模型一致**：用户只需要学一种方式使用命令——输入 `/`，看到自动补全，选择命令。来源的差别是实现细节，不应该泄露给用户。

## 面试角度的总结

命令系统的统一外观和分离内部机制，是"接入层规范化"在命令领域的应用。Command 结构是统一的接口，不同来源的命令在接入时被规范化成这种结构，消费层不需要知道来源差异。执行时按 `type` 分流，让不同机制的命令可以有各自最适合的执行方式。
