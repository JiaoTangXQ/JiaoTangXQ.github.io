# 深度分析：外延执行如何在扩张能力的同时保住既有秩序

## 扩张的两种路径

当一个系统需要增加新能力时，有两种基本路径：

**路径一：建立新秩序**——为新能力设计新的协议、新的权限体系、新的 UI 范式。好处是可以为新能力量身定制，不受旧系统的约束；坏处是系统变成了两个系统并排运行，交互点上充满了"这边的概念在那边叫什么"的转换问题。

**路径二：延伸既有秩序**——把新能力翻译成已有协议能描述的形式，让新能力服从已有的权限体系，用已有的 UI 范式展示。好处是系统仍然是一个整体，已有的工具（搜索、权限管理、日志记录）自动适用；坏处是翻译有代价，新能力的某些特性可能在翻译中丢失。

Claude Code 的外延执行选择了路径二。

## "伸出终端"的具体实现

Claude Code 原本是一个终端工具，工具调用的结果以文本形式返回。

把能力伸出终端意味着：
- 截图（返回图片，不是文本）
- 点击浏览器元素（副作用是改变 DOM，不是返回值）
- 输入键盘内容（副作用是改变表单状态）

这些操作和终端里的文本工具有本质不同。

Claude Code 的做法是把这些操作包装成 MCP 工具，让返回值和副作用都以 MCP 能表达的格式返回：
- 截图返回 base64 编码的图片，作为工具结果的一部分
- 点击操作返回操作成功/失败的状态，以及操作后的页面状态
- 键盘输入返回输入完成后的表单状态

这样，从主会话的角度看，调用一个浏览器工具和调用 `BashTool` 没有本质区别：都是"发出工具调用请求，等待结果，把结果注入对话历史"。

## 秩序延伸的具体体现

**权限体系的延伸**

`BashTool` 的权限走 `ToolPermissionContext` 和 `request_access`。

浏览器工具的权限同样走这条路径：`tabs_context_mcp` 的权限声明、`screenshot_mcp` 的权限声明，都通过同一套权限机制管理。

用户在设置里配置"允许 AI 访问浏览器"，这个配置走的是权限系统的标准路径，不是浏览器工具特有的设置入口。

**工具发现的延伸**

Claude Code 的 AI 通过提示词了解可用的工具，通过 `ToolSearch` 功能帮助 AI 找到最合适的工具。

浏览器工具注册为 MCP 工具后，自动进入这套发现机制。AI 生成响应时，能在上下文里看到"这些是可用的浏览器工具，它们能做什么"。

如果浏览器工具用独立协议，AI 就需要单独了解"浏览器工具怎么用"，提示词里要多出一段专门描述，维护成本增加。

**日志和追踪的延伸**

所有工具调用都经过相同的日志路径，记录"谁发出了调用、用了什么工具、输入是什么、输出是什么"。

浏览器工具的调用日志和 bash 工具的日志格式一样，在调试时不需要切换思维，查日志的工具也是同一套。

## 放弃了什么

延伸既有秩序，必然放弃了"为浏览器能力量身定制"的可能性。

**失去的专用优化**：浏览器操作有独特的特性，比如等待异步 DOM 更新。MCP 工具接口不是为这种等待设计的，需要通过轮询或特殊机制来模拟，比专用协议的原生等待机制更笨拙。

**失去的深度集成**：如果浏览器工具有专用协议，可以设计更丰富的交互——比如 AI 可以直接在 Chrome 的开发者工具面板里展示它的操作轨迹。用 MCP 工具，这种深度集成不自然。

**失去的性能极致**：MCP 的序列化开销比进程内直接调用更高，对于需要高频操作（每秒多次截图）的场景，这个开销是真实的。

这些代价被接受，是因为判断"系统整体性"比"单一能力的极致优化"更重要。

## 面试要点

1. **延伸秩序而不是新建秩序**：新能力翻译成 MCP 工具后，权限管理、工具发现、日志记录全部复用，不需要为浏览器能力单独建立管理基础设施。
2. **返回值统一化**：截图（图片）、点击（状态变更）都被翻译成 MCP 结果格式，从主会话角度看与其他工具无异。
3. **翻译有代价**：专用优化、深度集成、性能极致——这些在翻译过程中损失了，这是有意识的取舍，不是疏忽。
4. **系统整体性优于单一能力极致**：在一个复杂系统里，"到处都能用同一套工具"的价值，通常超过"某个功能能跑快 20%"的价值。
