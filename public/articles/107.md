# stopTask 为什么先改状态，再做 kill？能不能反过来？

`src/tasks/stopTask.ts` 里的 `stopTask` 函数，流程看起来很简单：找到任务、验证状态、kill 掉它、标记通知。

但里面有一个顺序决定值得细看：它在 kill 之前，有哪些检查是必须先做的？

## 代码里的顺序

```typescript
export async function stopTask(taskId, context): Promise<StopTaskResult> {
  const task = appState.tasks?.[taskId]
  
  // 1. 先验证任务存在
  if (!task) throw new StopTaskError(`No task found`, 'not_found')
  
  // 2. 先验证状态是 running
  if (task.status !== 'running') {
    throw new StopTaskError(`Task ${taskId} is not running`, 'not_running')
  }
  
  // 3. 找到对应的 task 实现
  const taskImpl = getTaskByType(task.type)
  if (!taskImpl) throw new StopTaskError(`Unsupported type`, 'unsupported_type')
  
  // 4. 执行 kill
  await taskImpl.kill(taskId, setAppState)
  
  // 5. 对 shell 任务特殊处理：抑制噪声通知
  if (isLocalShellTask(task)) { ... }
  
  return { taskId, taskType: task.type, command }
}
```

这里有一个微妙的问题：**kill 操作是异步的**（`await taskImpl.kill`），kill 完成前，任务状态还是 `running`。如果在 kill 进行中，另一个调用路径也尝试 stop 同一个任务，它会发现状态仍然是 `running`，然后也去 kill。双重 kill 可能导致资源泄漏或错误。

真正的"先改状态再善后"发生在 kill 的内部实现里。各个任务类型的 kill 函数（通过 `setAppState`）会**先把 status 改成 killed，再做清理工作**。这是顺序的关键：一旦状态改成 killed，其他任何路径再检查 `task.status !== 'running'` 时，就会得到正确的答案，不会重复触发 kill 逻辑。

## Shell 任务的特殊通知逻辑

```typescript
if (isLocalShellTask(task)) {
  let suppressed = false
  setAppState(prev => {
    const prevTask = prev.tasks[taskId]
    if (!prevTask || prevTask.notified) return prev
    suppressed = true
    return {
      ...prev,
      tasks: { ...prev.tasks, [taskId]: { ...prevTask, notified: true } },
    }
  })
  if (suppressed) {
    emitTaskTerminatedSdk(taskId, 'stopped', { ... })
  }
}
```

这段代码处理了 shell 任务（bash 命令）的通知抑制问题。普通情况下，bash 任务被 kill 时会收到 exit code 137（SIGKILL），这会触发一条"退出码 137"的通知，对用户来说是噪音（因为是用户主动 stop 的，他已经知道任务停了）。

但对 agent 类型的任务（`local_agent`），stop 时恰恰需要发通知，因为 agent 被终止时还带有部分执行结果（`extractPartialResult(agentMessages)`），这个部分结果是有价值的，需要让主会话知道。

所以 stop 逻辑在两类任务上的行为是不同的：shell 任务抑制 XML 通知，agent 任务保留通知。同时，为了让 SDK 消费者（外部集成者）仍然能得到 shell 任务关闭的事件，代码在抑制了 XML 通知之后，会手动 emit 一个 SDK 事件：

```typescript
emitTaskTerminatedSdk(taskId, 'stopped', {
  toolUseId: task.toolUseId,
  summary: task.description,
})
```

这是"两套通知路径"共存的证明：一条是 XML 通知（给主会话的文本界面），一条是 SDK 事件（给程序化 API 消费者）。

## StopTaskError 的错误码

```typescript
export class StopTaskError extends Error {
  constructor(
    message: string,
    public readonly code: 'not_found' | 'not_running' | 'unsupported_type',
  ) { ... }
}
```

三种错误码不是随手加的，它们对应了三种不同的处理策略：

- `not_found`：任务已经不在 AppState 里了，可能是被 GC 清除了，需要通知调用者"这个任务已经不存在了"
- `not_running`：任务存在但不是 running 状态，可能已经 completed 或 failed，stop 请求是无效操作
- `unsupported_type`：这种类型的任务没有 kill 实现，需要走错误处理流程

调用者通过 `error.code` 可以区分这三种情况并做出不同响应，而不是把所有失败都当成同一类错误处理。

## 面试答题要点

"先改状态再善后"不是规范建议，而是并发安全的必要条件。在多个调用路径可能同时操作同一个任务的系统里（比如：用户手动 stop、超时自动 stop、父任务取消子任务），如果每个路径都先执行破坏性操作（kill）再改状态，就无法阻止另一个路径在 kill 执行过程中进入同样的逻辑。

**状态是并发控制的闸门**，先把闸门关上，再做后面的清理，是防止多路径竞争的标准做法。被问到"你们的任务停止逻辑是怎么实现的"，这个顺序的理由是加分项。
