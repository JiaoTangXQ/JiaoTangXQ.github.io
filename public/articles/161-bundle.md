# 深度分析：参数注入为什么首先是边界问题，其次才是传值问题

## bundle 边界是什么

当 Vite 构建 Claude Code 时，它把源码打包成一个 bundle。bundle 里的代码在运行时形成一个边界：bundle 内部可以自由互相引用，bundle 外部的东西需要通过明确的"入口"进来。

常见的入口：
- 环境变量（`process.env.*`）
- 命令行参数（`process.argv`）
- 配置文件（`settings.json`）
- 通过 IPC 接收的消息（远端模式）

**bundle 边界被穿透的意思**：外部的值直接在 bundle 内部的随机位置被读取，没有统一的入口。

比如，远端容器发来一个包含配置的消息，bundle 内部有 10 个不同的组件分别读取这条消息里的不同部分，每个组件各自处理。

这不是 bundle 边界被穿透了，而是 10 个地方都有"外部入口"，维护起来极复杂。

## 参数注入的问题

在远端模式里，Claude Code 运行在两个环境：
1. 本地 CLI（用户端）：有用户的配置、权限上下文、UI
2. 远端容器（CCR）：运行工具，有云端的配置、权限需求

某些参数需要从远端容器传到本地 CLI（比如权限请求），某些参数需要从本地 CLI 传到远端容器（比如用户的权限决定）。

如果这些参数随意穿越边界，会产生：

**参数来源不明确**：一个组件在读取某个配置值，这个值是本地的还是远端传来的？不清楚，需要追踪整个调用链才能搞清楚。

**注入点漂移**：每次需要从远端接收新参数，就在某个地方加一个新的读取点。随着功能演进，注入点遍布各处，难以找到"哪些数据是从远端来的"。

**安全审计困难**：不知道哪些参数可能被远端控制，无法对这些参数做集中的安全检查（比如防注入攻击）。

## 集中注入点的设计

解决方案是：**只在明确的注入点允许外部参数进入 bundle，其他地方不直接读取外部参数**。

注入点通常在初始化阶段（`main.tsx` 里的 teleport/remote 分流逻辑）：

```typescript
// 推断的代码结构
const remoteConfig = parseRemoteArgs(process.argv)  // 解析远端配置
const initialState = buildInitialState(remoteConfig)  // 转换成内部格式
startApp(initialState)  // 启动应用，之后只用 initialState，不再直接读 remoteConfig
```

这个模式的好处：

1. **明确的注入点**：只有 `parseRemoteArgs` 和 `buildInitialState` 读取外部参数，其他代码读取 `initialState`（内部格式）。
2. **格式转换在注入时发生**：外部参数的格式（CLI args、JSON 消息）在注入点被转换成系统内部格式，之后所有代码只处理内部格式。
3. **安全检查集中**：对外部参数的验证（值范围检查、格式验证、安全过滤）在注入点做一次，而不是分散在每个使用点。

## 和统一体验的关系

`161-bundle` 文章的标题说的是"参数注入先保护 bundle 边界，再谈统一体验"。

这个顺序很重要：

如果 bundle 边界不保护，远端参数到处渗入，系统内不同组件读到的可能是同一参数的不同版本（一个读到了旧的，另一个读到了更新的），导致界面出现不一致的状态。

保护了 bundle 边界，参数在注入点被处理成一致的 `initialState`，系统内所有组件读同一份 `initialState`，状态一致性有了基础。

"统一体验"建立在"状态一致"之上，"状态一致"建立在"参数从统一入口进入"之上，所以参数注入的质量直接影响最终的体验一致性。

## 面试要点

1. **bundle 边界穿透是维护性问题**：参数入口越多，追踪数据来源越难，安全审计越复杂。
2. **集中注入点让格式转换只发生一次**：外部格式在边界处转换成内部格式，之后所有代码用内部格式，不需要各自处理外部格式的差异。
3. **安全检查应该在注入点集中做**：而不是分散在每个使用点，减少遗漏的可能性。
4. **参数注入质量影响体验一致性**：状态一致性的基础是参数从统一入口进入并被一致转换，bundle 边界保护是这个链条的起点。
