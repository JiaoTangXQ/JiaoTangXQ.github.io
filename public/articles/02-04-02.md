# 安全环境、证书和代理为什么必须先行

这类准备最容易被写成“底层细节”，但 Claude Code 明显不是这么看。它在 `init()` 里先应用安全环境变量，再补 `NODE_EXTRA_CA_CERTS`，然后配置 mTLS 和全局代理，最后才去做 API 预连接。这个顺序说明了一件很重要的事: 连接外部世界之前，先得把“通过什么规则去连接”定好。

原因其实很朴素。证书如果晚配，第一次 TLS 握手可能就已经走错；代理如果晚配，预热出来的连接就不是后面真正会复用的那套 transport；环境变量如果不先分成安全部分和完整部分，系统就会在 trust 之前提前吃进不该吃的外部影响。也就是说，这些准备并不是网络优化的小配件，而是在决定后面所有联网和外部调用到底能不能被信任。

Claude Code 在这里体现出来的不是谨小慎微，而是很硬的因果意识。真正安全的系统，不是等连接失败了再补配置，而是在第一次伸手碰外部世界之前，就先把手套戴好。

## 实现链
这页对应的实现链都落在 `init.ts`：配置系统、安全环境变量、CA 证书、代理、平稳退出、预连接和远端 promise 会在真正会话启动前先被装起来。它关注的是运行环境，而不是马上给用户一段对话。

## 普通做法
更常见的写法是把环境准备和会话准备混在一起，等需要网络、证书或清理时再临时处理。

## 为什么不用那种做法
Claude Code 没这么拖后，是因为这些基础设施一旦错位，后面的会话层每走一步都可能踩雷。它宁可先把地基打厚。

## 代价与回报
回报是会话代码不必反复兜底底层问题。代价是 init 阶段要先处理大量用户暂时看不见的东西。
