# MCP、Skill、Plugin 三套外部能力接入机制有什么本质区别？

Claude Code 同时存在三套把外部能力引入系统的机制：MCP（Model Context Protocol）、Skill、Plugin。面试中常被问到"为什么不统一成一套"——理解它们进入运行时的不同方式，是回答这个问题的关键。

## 三套机制各自做什么

**MCP** 是协议层接入。外部工具服务器通过标准化的 JSON-RPC 协议暴露工具，Claude Code 通过 `services/mcp/*` 把这些工具翻译成内部 `Tool` 对象。MCP 工具和内置工具在运行时几乎没有区别——它们都有 `name`、`inputSchema`、`call`、`checkPermissions` 等接口，都被放进同一个工具池交给模型使用。

关键实现在 `tools.ts` 的 `assembleToolPool` 函数：

```typescript
export function assembleToolPool(
  permissionContext: ToolPermissionContext,
  mcpTools: Tools,
): Tools {
  const builtInTools = getTools(permissionContext)
  const allowedMcpTools = filterToolsByDenyRules(mcpTools, permissionContext)
  const byName = (a: Tool, b: Tool) => a.name.localeCompare(b.name)
  return uniqBy(
    [...builtInTools].sort(byName).concat(allowedMcpTools.sort(byName)),
    'name',
  )
}
```

注意两点：内置工具和 MCP 工具分开排序后再合并——这是为了不破坏系统提示的 prompt cache 命中率，因为服务端的缓存断点放在最后一个内置工具之后。

**Skill** 是命令层接入。Skill 的本质是"可被触发的命令模板"，通过 `SkillTool` 和 `skills/*` 进入系统。用户可以用 `/skill` 斜杠命令来调用它们。Skill 不是工具调用，而是把预定义的提示词和工作流注入到对话上下文。

**Plugin** 是配置层和 UI 层接入。`plugins/*` 和 `utils/plugins/*` 处理插件的发现、加载和生命周期。插件可以扩展命令列表、注入系统提示组件、注册新的 slash 命令处理函数。

## 为什么不统一成一套？

三套机制覆盖的是不同的扩展维度：
- MCP：**工具执行能力**——让模型能调用外部 API、读取外部资源
- Skill：**工作流模板**——封装常用任务序列，降低重复提示成本
- Plugin：**系统行为**——修改 Claude Code 本身的命令处理和界面逻辑

强行统一的代价是：每种扩展都必须遵守最严格的那套契约。如果 Plugin 必须实现完整的 Tool 接口，那给一个"注册新 slash 命令"的简单插件写 `checkPermissions`、`renderToolResultMessage` 就是纯粹的噪音。

## 统一的部分：权限和工具池

虽然接入方式不同，这三套机制最终都要经过同一套权限体系（`ToolPermissionContext`）的过滤，都要被模型通过同一个工具调用接口使用，都要被 UI 系统渲染。这是设计的关键——**接入多样，使用统一**。

```typescript
export type ToolPermissionContext = DeepImmutable<{
  mode: PermissionMode
  alwaysAllowRules: ToolPermissionRulesBySource
  alwaysDenyRules: ToolPermissionRulesBySource
  alwaysAskRules: ToolPermissionRulesBySource
  isBypassPermissionsModeAvailable: boolean
  // ...
}>
```

无论工具来自内置、MCP、Skill 还是 Plugin，在运行时都要通过这个上下文过滤，都要调用 `canUseTool`，都要走 `checkPermissions` 流程。

## 面试延伸

面试中这个问题的变形很多：
- "你设计一个插件系统，怎么平衡灵活性和安全性？" → 答案核心是：能力接入多样，但执行时的权限检查必须统一
- "为什么 MCP 工具要和内置工具放在同一个工具池？" → 因为模型不应该知道工具来自哪里，统一接口降低模型使用复杂度
- "prompt cache 和工具顺序有什么关系？" → 排序稳定性直接影响 cache key，`assembleToolPool` 的排序逻辑就是在解这道题
