# WeakRef 在 AbortController 传播链里解决了什么内存问题？

`AbortController` 是现代 JavaScript 异步取消的标准机制。但当你需要**父子层级**的取消传播时——父 controller abort 时所有子 controller 也跟着 abort，子 abort 不影响父——一个看起来很简单的需求里藏着内存泄漏陷阱。

Claude Code 的 `utils/abortController.ts` 给出了一个用 `WeakRef` 解决这个问题的教科书级实现。

## 问题：父持有子的强引用

朴素实现是这样的：

```typescript
// 直觉实现
function createChildAbortController(parent: AbortController): AbortController {
  const child = new AbortController()
  parent.signal.addEventListener('abort', () => {
    child.abort(parent.signal.reason)
  })
  return child
}
```

问题在于这个闭包：`addEventListener` 的回调函数捕获了 `child`，而 `parent.signal` 持有这个回调函数。也就是说，只要 `parent` 活着，`child` 就不会被 GC——即使业务代码已经彻底不需要这个 `child` 了。

在 Claude Code 这样有大量嵌套工具调用、子 Agent、并发任务的系统里，这种父持有子的强引用会在长会话中积累成可观的内存泄漏。

## Claude Code 的解法：WeakRef

```typescript
export function createChildAbortController(parent: AbortController): AbortController {
  const child = createAbortController()

  // 快速路径：父已经 abort 了，直接传播，不需要设置监听
  if (parent.signal.aborted) {
    child.abort(parent.signal.reason)
    return child
  }

  // WeakRef 防止父持有子的强引用
  const weakChild = new WeakRef(child)
  const weakParent = new WeakRef(parent)
  const handler = propagateAbort.bind(weakParent, weakChild)

  parent.signal.addEventListener('abort', handler, { once: true })

  // 自动清理：当子被 abort 时，从父上移除监听
  child.signal.addEventListener(
    'abort',
    removeAbortHandler.bind(weakParent, new WeakRef(handler)),
    { once: true },
  )

  return child
}
```

关键设计决策：

**1. `weakChild = new WeakRef(child)`**

`parent.signal` 的监听回调里用的是 `weakChild` 而不是 `child`。这意味着父对子只有**弱引用**——如果业务代码丢掉了对 `child` 的所有强引用，`child` 就可以被 GC，即使父还活着。

`propagateAbort` 函数里会调用 `weakChild.deref()`，如果子已经被 GC 了，`deref()` 返回 `undefined`，安全跳过，什么都不发生。

**2. `{ once: true }`**

父的监听回调只需要触发一次（因为 abort 是一次性事件），加 `once: true` 让回调在触发后自动移除，进一步避免悬挂引用。

**3. 子 abort 时清理父监听**

如果子被**提前 abort**（不是因为父 abort，而是业务提前取消），它的监听会自动从父上移除：

```typescript
function removeAbortHandler(
  this: WeakRef<AbortController>,
  weakHandler: WeakRef<(...args: unknown[]) => void>,
): void {
  const parent = this.deref()
  const handler = weakHandler.deref()
  if (parent && handler) {
    parent.signal.removeEventListener('abort', handler)
  }
}
```

父和 handler 都用 `WeakRef` 持有，防止清理逻辑本身又制造强引用。

## 为什么是模块级函数而不是箭头函数

注释里有一句值得注意：

> "Module-scope function avoids per-call closure allocation."

`propagateAbort` 和 `removeAbortHandler` 都定义在模块顶层，用 `.bind()` 注入上下文，而不是在 `createChildAbortController` 内部创建箭头函数。这样每次调用只分配 `bind` 的结果（一个 bound function），而不是分配一个新的闭包对象加上它捕获的变量。在高频调用场景下这个差异是可测量的。

## 面试角度

这段代码是"用 WeakRef 解决什么实际问题"的绝佳示例。面试里这个话题通常很抽象，但有了这个具体案例就好说多了：

- **WeakRef 不是"弱化的引用"，而是"不阻止 GC 的引用"** — 这个定义上的区分很重要
- **WeakRef.deref() 可能返回 undefined** — 任何用到 WeakRef 的代码都必须处理这个情况
- **WeakRef 不能和 FinalizationRegistry 混用来保证回调时机** — GC 时机不可预测，不要把程序正确性建立在 GC 时机上
- **WeakRef 的适用场景**：缓存（不阻止被缓存对象 GC）、观察者解注册（不阻止订阅者 GC）、正是这种父子信号传播场景
