# 记忆代理只能编辑记忆目录——这个限制为什么是安全设计的核心？

面试题："如果你要设计一个 AI Agent 的自动记忆提炼系统，权限边界怎么划？"

Claude Code 的答案是：**能写什么文件，要精确限制到目录级别，不是工具级别**。

---

## createAutoMemCanUseTool：权限白名单

```typescript
export function createAutoMemCanUseTool(memoryDir: string): CanUseToolFn {
  return async (tool: Tool, input: Record<string, unknown>) => {
    // REPL 工具：允许（它只是一个薄包装层，实际工具仍受下面的规则约束）
    if (tool.name === REPL_TOOL_NAME) {
      return { behavior: 'allow', updatedInput: input }
    }

    // 读类工具：无限制允许（本身是只读的）
    if (tool.name === FILE_READ_TOOL_NAME ||
        tool.name === GREP_TOOL_NAME ||
        tool.name === GLOB_TOOL_NAME) {
      return { behavior: 'allow', updatedInput: input }
    }

    // Bash：只允许只读命令
    if (tool.name === BASH_TOOL_NAME) {
      if (tool.isReadOnly(parsed.data)) {
        return { behavior: 'allow', updatedInput: input }
      }
      return denyAutoMemTool(tool, 'Only read-only shell commands...')
    }

    // 写类工具：只允许写入 auto-memory 目录
    if ((tool.name === FILE_EDIT_TOOL_NAME || tool.name === FILE_WRITE_TOOL_NAME)
        && 'file_path' in input) {
      const filePath = input.file_path
      if (typeof filePath === 'string' && isAutoMemPath(filePath)) {
        return { behavior: 'allow', updatedInput: input }
      }
      // 路径不在 memory 目录：拒绝
    }

    return denyAutoMemTool(tool, `only ... within ${memoryDir} are allowed`)
  }
}
```

这个函数被 `extractMemories` 和 `autoDream` 共享，是所有自动记忆操作的权限门卫。

---

## 为什么不能只限制工具，要精确到路径

仅仅限制"不能用 FileWrite"是不够的，因为：
1. FileEdit（编辑现有文件）也是写操作
2. 允许写任何路径的 FileWrite，意味着记忆代理可以修改代码文件、配置文件、甚至 `.env`

精确到路径的限制，确保了即使记忆代理行为异常（比如 prompt 被注入，让它"把所有记忆清空然后写入 README"），它的写操作也被限制在 `~/.claude/projects/<path>/memory/` 这一个目录里，不会影响到项目代码。

---

## isAutoMemPath 的边界判断

```typescript
// memdir/paths.ts
export function isAutoMemPath(filePath: string): boolean {
  const autoMemPath = getAutoMemPath()
  return filePath.startsWith(autoMemPath)
}
```

这是一个路径前缀检查。`getAutoMemPath()` 返回的是记忆目录的绝对路径，确保路径遍历攻击（比如 `../../project/src/main.ts`）也能被拦截——展开后的路径不以 autoMemPath 开头，被拒绝。

---

## Bash 只允许只读命令

对 Bash 工具，判断是"这条命令是否只读"，而不是"是否允许 Bash"：

```typescript
if (tool.name === BASH_TOOL_NAME) {
  const parsed = tool.inputSchema.safeParse(input)
  if (parsed.success && tool.isReadOnly(parsed.data)) {
    return { behavior: 'allow', updatedInput: input }
  }
  return denyAutoMemTool(tool, 'Only read-only shell commands...')
}
```

`isReadOnly` 方法检查命令是否属于已知的只读命令列表（`ls`、`find`、`grep`、`cat`、`stat`、`wc`、`head`、`tail` 等）。记忆代理需要这些命令来扫描现有记忆文件（`ls ~/.claude/projects/...`），但不能运行 `rm`、`mv` 或者 `git push`。

## REPL 工具的特殊处理

```typescript
if (tool.name === REPL_TOOL_NAME) {
  return { behavior: 'allow', updatedInput: input }
}
```

乍看奇怪——为什么允许 REPL？

原因是技术实现细节：在 Ant 内部，原始工具（FileRead、Bash、FileEdit 等）被包装在 REPL 工具里，模型不直接调用 `file_read`，而是调用 `REPL`，再由 REPL 内部分发。REPL 是一个透传层，`createToolWrapper` 会在 REPL 内部再次调用 `canUseTool`，对实际的内层工具执行权限检查。

允许 REPL 本身，不等于允许 REPL 内部的所有操作——内部操作还要过一次同样的权限门。这是"透传层要放行，实际操作层要检查"的双重门模式。

---

## 这个设计解决的真实威胁

记忆代理是一个有完整对话历史访问权的 AI agent，它"知道"很多东西，包括项目代码、用户行为、敏感路径。

如果给它宽权限：
- 它可能错误地认为"应该把某段代码变更记录下来"，然后直接修改了代码文件
- 如果 prompt 被污染（比如有恶意的 tool_result 内容），攻击者可能诱导它写入任意位置
- 哪怕只是一个无害的误操作，写错了文件路径，也可能破坏项目

窄权限是第一道防线，不是最后一道。它的存在不意味着"其他地方不需要安全检查"，而是"即使其他地方出了问题，损害也被限制在可控范围内"。

## 面试指导

"自动化 AI Agent 的权限设计"是一道考察安全意识和防御性工程的题。

关键回答框架：
1. **最小权限原则**：只给完成任务所必需的权限，不多给
2. **写权限精确到路径**：不是"能不能写文件"，而是"能写哪个目录"
3. **路径遍历防护**：用 `isAutoMemPath` 而不是字符串包含检查
4. **透传层的双重检查**：REPL 放行，内层操作再检查

能说清楚"为什么仅限制工具名不够，还需要限制路径"，这说明你真正理解了权限边界的粒度问题，而不只是知道"要有权限检查"这个结论。
