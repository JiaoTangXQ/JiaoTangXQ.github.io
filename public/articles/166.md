# 面试题：为什么子 agent 收尾时必须先改状态，不能先做善后？

## 问题设置

子 agent 任务完成了。下面的工作需要做：

1. 把任务状态从 `running` 改为 `completed`
2. 发送完成通知给主会话（写入 `pendingMessages`）
3. 清理临时文件
4. 写最终日志
5. 释放 AbortController

**面试题：这五件事的顺序为什么很重要？先做哪些，后做哪些？**

## 先改状态：为什么是第一步

正确的顺序是：**第 1 步（改状态）必须是第一步**，其余步骤可以之后做。

原因：**`isTerminalTaskStatus` 是很多系统防卫的门卫**。

在 `Task.ts` 里，`isTerminalTaskStatus` 的注释说：

```
True when a task is in a terminal state and will not transition further.
Used to guard against injecting messages into dead teammates, 
evicting finished tasks from AppState, and orphan-cleanup paths.
```

这意味着：

- 如果有代码试图向这个任务注入消息，它会先检查 `isTerminalTaskStatus`
- 如果任务状态还是 `running`，注入消息的操作会被允许
- 注入了一条消息到一个"已经完成但还没改状态"的任务，这条消息永远不会被处理

如果任务实际上完成了，但状态还是 `running`，系统其他部分会以为这个任务仍然在运行，继续向它路由消息，继续为它分配资源，继续在 UI 上展示"正在运行"。

## 善后为什么可以在后

通知、清理资源、写日志——这些操作的顺序相对灵活，因为它们不影响系统对这个任务"是否还在运行"的判断。

状态已经是 `completed`，其他系统组件知道这个任务结束了，不会再尝试向它发消息。这时可以安全地进行各种善后工作。

即使善后过程中某个步骤失败了（比如写日志失败），任务的状态已经是 `completed`，系统的一致性没有受损。

## 不按顺序会发生什么

**场景：先发通知，再改状态**

```
子 agent 完成工作
↓
发送完成通知给主会话（pendingMessages）
  主会话开始处理通知...
    查询任务状态，任务还在 running
    发现和通知矛盾，不知道该怎么处理
↓
（主会话可能等待状态变化，或者忽略通知，或者出现 bug）
改状态为 completed
```

主会话在收到"任务完成"通知的同时，看到任务还在 `running` 状态，这是矛盾的信息。主会话可能：
- 忽略这条通知（以为是误发的）
- 进入等待循环，等状态变成 `completed`
- 抛出异常

**场景：先清理 AbortController，再改状态**

AbortController 用于中止任务。如果先释放了 AbortController，而状态还是 `running`，此时如果有其他代码检测到状态是 `running` 并尝试调用 `kill()`，会发现 AbortController 已经不存在，`kill()` 会出错。

## 原子性的期望

理想情况下，"任务的实际工作完成" + "状态改为 completed" 应该是原子的：不存在中间状态（实际完成了但状态还是 running）。

但在异步的、可能失败的真实系统里，纯粹的原子性很难保证。

退而求其次的做法是：**让状态变化成为第一步，让"实际完成"和"状态 = completed"之间的窗口尽可能短**。

状态变化之后，即使善后步骤失败了（系统崩溃了，网络断了），状态已经是正确的。下次恢复时，系统知道这个任务已经完成，不会重试或重路由。

如果状态变化不是第一步，善后失败后系统状态不一致，恢复时不知道该怎么处理。

## 面试要点

1. **先改状态是为了关闭防卫门**：isTerminalTaskStatus 是很多操作的前置检查，状态不变其他系统继续以为任务在运行。
2. **状态一致性是善后正确执行的前提**：状态已经是终态，善后失败了也不影响系统一致性；反之，善后失败但状态还是 running，系统会进入矛盾状态。
3. **原子性不可能时，先保证最重要的那一步**：实际完成和状态变化之间有时间差，让状态变化尽可能早，减少不一致窗口。
4. **善后失败是可接受的**：日志没写成、临时文件没清理——这些是不完美但可接受的结果；状态不一致是不可接受的，因为会影响其他系统组件的正确性。
