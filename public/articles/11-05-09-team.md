# 面试题：底部区域如何根据团队状态动态切换提示？hasTeams 和 hasBackgroundTasks 分别控制什么？

这道题测试你对"上下文感知 UI"的理解——同一个底部区域，在不同的工作场景下应该展示完全不同的行动入口。

## 问题背景

Claude Code 支持两种主要工作模式：
1. **单 agent 模式**：你和一个 agent 交互，它依次执行你的指令
2. **多 agent 团队模式（agent swarms）**：多个 agent 并行工作，你是协调者

这两种模式下，用户关心的问题和需要的快速操作入口完全不同：
- 单 agent：当前是否在加载？我能不能中断？
- 多 agent：哪些 agent 在跑？我能不能批量停止？有没有任务需要我处理？

## 核心条件变量

```typescript
const hasTeams = isAgentSwarmsEnabled() && !!teamContext?.teammates;
const hasBackgroundTasks = Object.values(appStateTasks).some(
  t => t.status === 'running'
);
const isViewingTeammate = screen === 'teammate-view';
const isLoading = /* agent is currently processing */;
```

四个变量组合出不同的底部状态。

## 提示的条件分发逻辑

**没有团队、没有后台任务时**（单 agent 标准模式）

```
[esc to cancel]  [ctrl+r memory]  ...
```

显示中断 agent 的快捷键和常用功能入口。这是用户最熟悉的状态。

**有团队但 agent 仍在跑时**

```
[stop agents]  [show tasks]  ...
```

中断入口变成了"停止所有 agent"而不是"取消当前操作"，因为现在可能有多个 agent 在并发执行，需要一个批量控制入口。"show tasks"让用户能跳转查看任务分配状态。

**正在查看某个 teammate 的视图时**

```
[esc to return]  [task: ...]  ...
```

提示切换成"返回"操作，因为用户现在在一个子视图里，主要的行动是返回到协调者视图。

**有后台任务但当前 agent 不在加载时**

这时通常不显示 esc（因为没有东西在跑需要中断），但可能显示任务状态和 teammate 指示器。

## 为什么不用同一套固定提示

**多余的提示比没有提示更糟**

如果在没有团队的单 agent 场景里显示"stop agents"，用户点击后会发现什么都没停（或者触发了意外操作）。这会建立错误的用户心理模型，并且造成不信任。

**缺少的提示会让用户迷路**

在多 agent 场景里，如果没有"show tasks"的快速入口，用户需要记住命令或者去菜单里找——这打断了协作节奏，特别是在需要快速决策的场景里。

**底部是"当前最高优先级操作"的快速通道**

底部提示不是功能列表，而是"在当前上下文下，最可能需要的操作"的快速入口。把当前上下文无关的操作塞进这里，是在用噪音稀释信号。

## hasTeammatesPills 的逻辑

```typescript
const hasTeammatePills = hasTeams && teamContext?.teammates &&
  Object.values(teamContext.teammates).some(t => t.status !== 'inactive');
```

这个条件控制是否在底部渲染 teammate 状态指示器（每个 agent 的颜色 pill）。

只有当 teammate 不处于 inactive 状态时才显示。这和前面讲的"owner 只在对方仍在线时显示"是同一个原则：**显示一个活跃的行动者，不显示已经退场的历史记录**。

## 底部作为"编排者调度台"的类比

在多 agent 协作场景下，用户扮演的角色是协调者，而不是操作者。底部区域需要为这个角色服务：
- 一眼看到有哪些 agent 在工作（teammate pills）
- 快速访问任务分配情况（show tasks）
- 能批量控制 agent（stop agents）

这和单 agent 场景里"一个人做一件事"的界面需求完全不同。同一块底部区域服务两种角色，通过条件分发做到了角色切换时的界面自适应。

## 面试指导

**直接问法**："底部提示是怎么根据团队状态变化的？"

答：有四个核心条件变量：hasTeams、hasBackgroundTasks、isViewingTeammate、isLoading。这些变量的不同组合决定了底部显示 esc、stop agents、show tasks 还是其他提示。单 agent 和多 agent 场景需要完全不同的快速操作入口，所以底部是条件分发的，而不是固定模板。

**追问**："为什么不用一套通用提示来兼容所有场景？"

答：因为不同场景下"最可能需要的操作"根本不同。单 agent 场景需要"中断当前"，多 agent 场景需要"批量控制"和"查看任务分配"。用通用提示要么显示太多噪音，要么缺少关键入口，两者都比条件分发更差。

**设计追问**："你自己设计这块时，怎么确定哪些条件值得做条件分发？"

答：分发条件应该是"改写用户当前角色或目标"的状态变化。比如从单 agent 切到多 agent 协作，用户的角色从"操作者"变成了"协调者"，界面入口也应该随之改变。如果只是参数值的变化（比如当前模型是哪个），不改变用户角色，就不值得在底部做条件分发。

**系统性追问**："这套条件分发和 TaskListV2 里任务的优先级排序有什么共同点？"

答：都是"当前上下文决定展示策略"的思路。TaskListV2 根据任务状态和阻塞关系决定哪些任务先站前面；底部提示根据协作模式和加载状态决定哪些快速操作先露出来。本质上都是在问"现在用户最需要看/做什么"，而不是"把所有信息平铺出来"。
