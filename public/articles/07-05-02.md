# 插件的装配门槛：manifest 校验、来源检查、启用状态——为什么缺一不可？

## 安全分析：什么是"半加载状态"，为什么它比"不加载"更危险？

想象一个插件：manifest 文件损坏，只有一半字段能读到。如果系统直接加载它会发生什么？

- 命令名可能是空的或重复的
- 描述可能缺失
- 权限声明可能不完整

这不是"插件加载失败"，而是一个带着缺陷的对象进入了命令注册表。帮助文本出现空条目，权限判断结果错误，甚至可能影响其他正常命令的行为。

Claude Code 的 `pluginLoader.ts` 通过多道门槛防止这种情况：宁可整包拒绝，也不接受半成品。

---

## 门槛一：manifest 校验

插件必须有一个合法的 `manifest.json`，包含：
- `name`：唯一的插件名
- `version`：版本号
- `description`：描述（影响 `hasUserSpecifiedDescription`）
- `commands` 或 `skills`：能力列表

Zod schema 验证在这一步发生。如果 manifest 不符合 schema，插件整包拒绝。这比"读到哪算哪"的宽松解析更严格，但也更可预测。

**为什么用 Zod 而不是手写校验？**

Zod 的 `safeParse` 返回 `{ success: boolean, data | error }`，不抛异常，调用方可以明确处理成功和失败两条路径。手写校验容易遗漏边缘情况（比如字段存在但类型错误），Zod 的类型系统会在编译期帮你检查校验是否完整。

---

## 门槛二：来源合法性检查

插件可以来自多个来源：
- Anthropic 官方 marketplace
- 第三方 marketplace（如果配置允许）
- 本地路径安装
- 会话临时安装（session-only）

`pluginLoader.ts` 的来源检查不只是"这个插件从哪里来"，还涉及当前策略是否允许这个来源。比如：

- 企业配置可能禁止第三方 marketplace 插件
- `isRestrictedToPluginOnly()` 检查当前环境是否只允许通过插件访问技能

这意味着同一个插件包，在不同的策略配置下可能被允许或拒绝——来源检查是运行时策略的执行点。

---

## 门槛三：重复名检测

```typescript
// 简化示意
const seen = new Set<string>()
for (const plugin of candidates) {
  if (seen.has(plugin.manifest.name)) {
    logForDebugging(`Duplicate plugin name: ${plugin.manifest.name}, skipping`)
    continue
  }
  seen.add(plugin.manifest.name)
  enabledPlugins.push(plugin)
}
```

插件名是全局命名空间里的占位符。两个名字相同的插件不能共存，因为后面的命名空间生成（`plugin:namespace:command`）依赖插件名的唯一性。

去重策略是"先到先得"——第一个成功通过所有检查的同名插件获得权利，后来者被静默跳过。这个策略简单可预测，虽然不是最优（可能丢弃"更好的版本"），但避免了冲突解析的歧义。

---

## 门槛四：启用状态判断

通过前三关不等于插件会被加载。还要检查：

```typescript
// getBuiltinPluginSkillCommands 里的类似逻辑
const enabledPlugins = loadedPlugins.filter(plugin => plugin.enabled)
```

`enabled` 状态来自用户配置或策略配置。用户可以安装插件但不启用它，或者管理员策略可以强制禁用某些插件。

这个分离很重要：**安装** 和 **启用** 是两个独立的状态。"已安装但未启用"的插件不进入命令装配，不影响帮助文本，不参与权限判断——它在系统里几乎不存在。

---

## 为什么这四道门槛都必要？

| 门槛 | 如果跳过会发生什么 |
|---|---|
| manifest 校验 | 带有缺陷字段的命令对象进入注册表，行为不可预测 |
| 来源检查 | 企业策略绕过，安全边界失效 |
| 重复名检测 | 命令名冲突，行为取决于"哪个先/后被调用"这种运气因素 |
| 启用状态 | 用户禁用的插件仍然影响系统行为 |

每道门槛都在防止一种具体的"污染"。四者缺一，都可能让坏状态渗入本来应该干净的命令注册表。

---

## 面试指导

**"如何在插件系统中保证运行时安全？"**

这道题的关键不是列举"我们做了 N 个检查"，而是解释检查的顺序和每个检查的边界：

1. **先验证 schema**：确保后续处理不会因为空指针或类型错误崩溃
2. **再检查策略**：来源合法性是外部约束，必须在内部处理前做
3. **再检查冲突**：名字空间的唯一性是系统级不变量，要在单个插件通过后立即检查
4. **最后检查启用状态**：这是用户/管理员控制层，放在最后确保最终决策权在正确的手里

强调"宁可整包拒绝，也不接受半成品"的理念——这比"尽量加载，运行时再处理错误"更符合"防御性编程"原则。
