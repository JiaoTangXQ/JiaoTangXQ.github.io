# registerTask 里的 5 个保留字段：UI 现场保护是怎么实现的？

`framework.ts` 里的 `registerTask()` 有一段不起眼但很关键的代码：

```typescript
export function registerTask(task: TaskState, setAppState: SetAppState): void {
  let isReplacement = false
  setAppState(prev => {
    const existing = prev.tasks[task.id]
    isReplacement = existing !== undefined

    // 注释原文：
    // Carry forward UI-held state on re-register (resumeAgentBackground
    // replaces the task; user's retain shouldn't reset). startTime keeps
    // the panel sort stable; messages + diskLoaded preserve the viewed
    // transcript across the replace (the user's just-appended prompt lives
    // in messages and isn't on disk yet).
    const merged =
      existing && 'retain' in existing
        ? {
            ...task,
            retain: existing.retain,
            startTime: existing.startTime,
            messages: existing.messages,
            diskLoaded: existing.diskLoaded,
            pendingMessages: existing.pendingMessages,
          }
        : task

    return { ...prev, tasks: { ...prev.tasks, [task.id]: merged } }
  })
  // ...
}
```

当同一个任务 ID 再次注册时（替换注册），5 个字段会从旧状态保留：`retain`、`startTime`、`messages`、`diskLoaded`、`pendingMessages`。

为什么是这 5 个？每个背后都有一个真实场景。

## retain：用户正在看这个任务

`retain: boolean` 由 `enterTeammateView()` 设置为 `true`，表示"UI 正在持有这个任务"。

当用户打开某个 Agent 任务的详细面板查看 transcript 时，`retain` 变成 `true`。此时即使任务已完成，也不应该被驱逐——因为用户还没读完。

如果替换注册时不保留 `retain`，就会出现这个场景：

1. 用户打开了一个 Agent 任务的面板，`retain = true`
2. 这个 Agent 被 `resumeAgentBackground()` 重新注册（比如用户给它发了条消息）
3. 新注册的任务对象 `retain = false`（默认值）
4. 系统认为用户没在看它，任务完成后立刻驱逐
5. 用户面板里的 transcript 消失了

保留 `retain` 就是为了防止这个场景。

## startTime：面板排序稳定性

`startTime` 影响面板的排序——通常任务按创建时间从早到晚排列。

如果替换注册时不保留 `startTime`，就会出现：

1. 任务 A 在 10:00 创建，排在面板第一位
2. `resumeAgentBackground()` 在 10:05 重新注册这个任务
3. 新的 `startTime = Date.now()` = 10:05
4. 任务 A 突然跑到面板靠后的位置，其他任务按时间排在前面
5. 用户视角：任务顺序突然乱了

注释说的"keeps the panel sort stable"就是这个意思。

## messages：用户刚打的消息还没落盘

这是 5 个字段里最微妙的一个。

`messages: Message[]` 是当前在内存中展示的消息列表。用户通过 `SendMessage` 向一个后台 Agent 发送消息时：

1. 消息立刻追加到内存里的 `messages` 数组（通过 `appendMessageToLocalAgent()`）
2. 消息被放进 `pendingMessages` 队列（等待下一轮注入 Agent）
3. 这条消息**还没写到磁盘**（transcript JSONL 文件）

此时如果发生替换注册，而新注册不保留 `messages`，用户在 UI 里看到的那条刚刚发出的消息就会消失（因为磁盘上还没有这条消息，`diskLoaded` 从磁盘加载时也找不到它）。

注释里这句话点得很准："the user's just-appended prompt lives in messages and isn't on disk yet"。

## diskLoaded：已读磁盘数据不要重新加载

`diskLoaded: boolean` 标记"已经从磁盘读过 transcript 了"。

`enterTeammateView()` 会触发一次磁盘读取，把 transcript 从 JSONL 文件加载到内存的 `messages` 数组里。这个操作只应该发生一次（后续通过流式 append 保持同步）。

如果替换注册时不保留 `diskLoaded`，就会：

1. 用户打开面板，`diskLoaded = true`，transcript 已加载
2. 任务被重新注册，`diskLoaded = false`
3. 系统认为"还没从磁盘加载过"，触发第二次全量读取
4. 用户看到 transcript 闪烁或者出现重复消息

## pendingMessages：排队中的消息不要丢失

`pendingMessages: string[]` 是等待在下一轮被注入 Agent 的消息列表。

如果替换注册时不保留 `pendingMessages`，这些排队中的消息就会丢失，Agent 收不到它们。

## `'retain' in existing` 的类型守卫意义

注释里有一句："'retain' in task narrows to LocalAgentTaskState (the only type with that field)"。

`'retain' in existing` 不只是一个存在性检查，它同时也是 TypeScript 的类型守卫。只有 `LocalAgentTaskState` 有 `retain` 字段，所以这个检查同时把类型收窄到了正确的子类型。`local_bash` 任务没有 `retain` 字段，替换注册时就直接覆盖（shell 任务没有这些 UI 现场保护需求）。

## 面试要点

这 5 个字段保留逻辑揭示了一个框架层设计原则：**状态层不应该对 UI 层一无所知**。

传统分层架构里，数据层不应该知道 UI 的事。但在任务框架这个场景里，"用户正在看哪个任务"（`retain`）、"显示哪些消息"（`messages`）、"面板排序"（`startTime`）这些 UI 语义，需要在数据层的状态操作里被尊重。

这不是层次污染，而是对"任务状态的完整性"的一种更宽泛的定义：任务状态不只是执行数据，也包括用户当前对任务的持有关系。
