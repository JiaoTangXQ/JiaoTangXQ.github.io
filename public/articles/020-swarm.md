# Claude Code swarm 团队机制地图

## 先说结论：swarm 不是多开几个 agent，而是组织世界

很多多 agent 系统真正做到的只有“并发更多”。Claude Code 的 swarm 想做的是更难的事：让多个 agent 之间长出身份、通信、审批和空间秩序。

## 第一层：teammate 先被当成组织成员，不是共享脑分身

### 实现链

从 teammate 提示和相关逻辑看，worker 并不默认共享一切，而是要通过显式消息和任务机制沟通。

### 普通做法

很多多 agent 演示会假设多个 agent 自然共享上下文，天然会协作。

### 为什么不用

Claude Code 承认协作不等于共享记忆，真正的团队要有通信制度。否则多 agent 很快就会变成互相抢麦的并发噪音。

### 代价

通信和角色概念会更多；但秩序更稳。

## 第二层：`inProcessRunner` 说明同进程也要维持隔离

### 实现链

即使 teammate 跑在同一进程里，系统也会继续维护上下文隔离、进度跟踪、完成通知和权限流。

### 普通做法

更直觉的做法是：既然同进程，就共享一堆状态，少做一层包装。

### 为什么不用

Claude Code 不想因为“同进程”就牺牲组织边界。否则 teammate 很容易重新把主线程状态搅乱。

### 代价

同进程实现也显得偏重；但边界更稳定。

## 第三层：权限必须集中到 leader，不许各批各的

### 实现链

`permissionSync` 相关代码说明：worker 的权限请求会通过邮箱/权限目录送到 leader，再由 leader 统一批准或拒绝。

### 普通做法

更省事的做法是每个 worker 自己弹权限框，自己得到批准。

### 为什么不用

Claude Code 不想让用户在多 agent 场景下被多个审批入口撕碎。只要是团队，就必须有一个审批中心。

### 代价

要维护 leader/worker 权限桥和同步协议；但人类操作入口更集中。

## 第四层：邮箱和目录通信说明它在为更复杂后端做准备

### 实现链

swarm 不只靠内存通信，还会用邮箱、pending/resolved 文件、锁文件等机制。

### 普通做法

同机多 agent 最容易想到的就是内存传消息。

### 为什么不用

Claude Code 在为不同进程、不同后端、可恢复和可观测场景做准备。文件和邮箱虽然不如纯内存优雅，但作为最低共同通信面更稳。

### 代价

表面上看更笨重；但鲁棒性更高。

## 第五层：swarm 还管“怎么被看见”

### 实现链

颜色、pane、布局管理这些逻辑说明 swarm 不只管后端调度，也在认真组织团队的空间呈现。

### 普通做法

很多系统把多 agent 仅仅当成后台并发，不认真对待显示和辨识。

### 为什么不用

Claude Code 知道多 agent 一旦超过一个，最先崩的常常是人的辨识能力。所以颜色、pane、标题、布局不是装饰，而是维持团队心智模型的基础设施。

### 代价

swarm 和 UI 深度耦合，整体更像一套小型团队运行时；但这正是它和普通任务系统的区别。
