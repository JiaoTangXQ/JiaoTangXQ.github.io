# 第七卷导读：扩展不是外挂堆，而是被翻译进统一运行时

## 实现链
- `src/skills/loadSkillsDir.ts` 负责把本地和托管目录里的技能 Markdown 解析成 `Command`，`src/utils/plugins/pluginLoader.ts` 和 `src/utils/plugins/loadPluginCommands.ts` 再把插件世界装进同一套命令骨架，`src/services/mcp/client.ts` 则长期维护 MCP 连接、资源和认证状态。
- 所以这一卷不是按“目录名”展开，而是按代码里三种不同的增长路径展开：本地技能先落成命令对象，插件先过安装与启用制度，MCP 先管连接与刷新，再把工具、资源、prompt 和 skill 一起接回本地运行时。

## 普通做法
- 更常见的讲法会把扩展世界统称成“插件系统”，然后把技能、第三方命令、远端能力全都塞进一个箱子里。
- 这种写法目录上很省事，但它默认所有扩展都只是“多一个接口”或“多一个脚本”。

## 为什么不用
- Claude Code 的代码根本不是这样长的。`loadSkillsDir.ts` 处理的是前置元数据、路径约束和 hooks；`pluginLoader.ts` 处理的是市场、版本、启用状态和变量替换；`client.ts` 处理的是常驻连接、认证缓存、刷新和失效。
- 如果把三者混成一种“插件”，新手会看不懂为什么有的扩展靠文件加载，有的要安装启用，有的还要处理断线、`needs-auth` 和 `list_changed`。

## 代价
- 代价是这一卷的概念比普通 CLI 教程重：同样叫“扩展”，背后其实是三套来源、三套边界、三种失败恢复方式。
- 但这个代价换来的好处也明确：读者能直接对上代码里的真实分工，而不是被一个看似统一、其实会误导人的大词糊过去。
