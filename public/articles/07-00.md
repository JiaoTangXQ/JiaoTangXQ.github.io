# 三条扩展路径、一个统一运行时——第七卷在说什么？

## 面试热身：如果让你设计一个可扩展的 CLI，你会怎么分层？

这是系统设计题里的常见开头。大多数候选人会画三个框：插件目录、远程工具、内置命令，然后说"通过统一接口调用"。

答案方向没错，但 Claude Code 的实际代码把这道题做得更深：它面对的不是三个同构的"扩展入口"，而是三种生命周期完全不同的增长路径。理解这一点，是读懂第七卷的前提。

---

## 三种路径，三种约束

**路径一：本地技能（skills）**

入口在 `src/skills/loadSkillsDir.ts`。技能是本地 Markdown 文件，启动时就能解析。它的核心约束是元数据前置——`description`、`allowed-tools`、`paths`、`hooks`、`agent`、`effort`、`shell` 这些字段必须在 frontmatter 里声明好，运行时才能根据它们做权限判断、帮助文本、调度决策。技能不需要网络，不需要安装，加载失败也不影响主进程。

**路径二：插件（plugins）**

入口在 `src/utils/plugins/pluginLoader.ts` 和 `loadPluginCommands.ts`。插件来自 marketplace 或本地安装目录，有 manifest、版本缓存、来源校验、启用状态。它的核心约束是治理优先——插件不是"放进去就能用"，而是要先过一道正式的装配和合法性检查，再进入命令系统。

**路径三：MCP 服务器**

入口在 `src/services/mcp/client.ts`。MCP 不是一次性 API 调用，而是一条常驻能力总线。它的核心约束是状态管理——连接、认证、`needs-auth` 缓存、`list_changed` 通知、资源刷新、挂起调用的 abort，全都是 MCP 层独有的问题，本地技能和插件根本不需要面对。

---

## 统一出口：Command 对象

三条路径最终都汇入 `src/commands.ts` 里的 `getCommands(cwd)` 函数，产出一个 `Command[]` 数组。调用方——补全、帮助、权限、调度——只需要面对这一个数组，不需要知道某条命令是从磁盘技能翻译来的、还是插件装配来的、还是 MCP 服务器动态暴露来的。

```typescript
export async function getCommands(cwd: string): Promise<Command[]> {
  const allCommands = await loadAllCommands(cwd)
  const dynamicSkills = getDynamicSkills()
  const baseCommands = allCommands.filter(
    _ => meetsAvailabilityRequirement(_) && isCommandEnabled(_),
  )
  // 动态技能去重后插入特定位置
  ...
}
```

翻译层确实有损耗——远程能力被压扁成本地对象时会丢一些来源特有细节。但 Claude Code 接受这个代价，因为它把复杂度集中在接入边缘，而不是把"来源感知"散布到全系统的每个模块里。

## 本卷结构

按代码里真实的分工展开，而不是按"目录名"展开：

- **07-02 ~ 07-04**：技能路径——frontmatter 解析、createSkillCommand 编译、getCommands 装配、新旧兼容
- **07-05 ~ 07-06**：插件路径——生态层治理、装配门槛、命名空间、变量替换
- **07-07 ~ 07-08**：MCP 路径——连接状态机、list_changed、needs-auth 缓存并发写、elicitation、transport 选型、能力簇收编
- **07-09**：桥接层——mcpSkillBuilders.ts 为什么要存在，注册式模式如何解决 Bun 打包问题

## 面试指导

**问"你熟悉插件系统吗"时，一个好答案会区分：**

- 扩展的来源（本地文件 vs 市场安装 vs 远程协议）
- 扩展的生命周期（启动时加载 vs 安装后启用 vs 连接后发现）
- 扩展失败的边界（单个技能失败不影响其他技能；MCP 连接失败有 needs-auth 缓存）

说"所有扩展都通过统一接口"是正确的，但说不出每条路径的约束不同，面试官会认为你只理解了接口层，没有理解系统层。
