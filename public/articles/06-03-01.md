# 工具级 deny 和命令级 deny：装配层和运行时层各管什么？

面试追问：”你说权限有装配层和运行时层。具体哪些判断在装配层做，哪些在运行时做？如果都放在运行时不行吗？”

## 装配层：工具池筛选

在工具池被构建时，`getDenyRuleForTool()` 会被调用来检查每个工具是否在 deny 名单里：

```typescript
// permissions.ts
export function getDenyRuleForTool(
  context: ToolPermissionContext,
  tool: Pick<Tool, 'name' | 'mcpInfo'>,
): PermissionRule | null {
  return getDenyRules(context).find(rule => toolMatchesRule(tool, rule)) || null
}
```

`toolMatchesRule` 只匹配**工具级**规则——规则没有 `ruleContent`（如 `”Bash”` 而不是 `”Bash(git *)”`）。这意味着装配层的 deny 是粗粒度的：整个工具要么出现在工具池里，要么被裁掉。

**装配层的语义**：这个工具在当前会话的”法律现实”里根本不存在。Claude 不会看到它，也不会尝试调用它。

## 运行时层：命令级审批

运行时的权限检查在 `PermissionContext.ts` 的 `createPermissionContext` 里构建，处理的是具体的工具调用请求：

```typescript
// PermissionContext.ts
async runHooks(permissionMode, suggestions, updatedInput, ...): Promise<PermissionDecision | null> {
  for await (const hookResult of executePermissionRequestHooks(
    tool.name, toolUseID, input, toolUseContext, ...
  )) {
    if (hookResult.permissionRequestResult) {
      const decision = hookResult.permissionRequestResult
      if (decision.behavior === 'allow') { return handleHookAllow(...) }
      else if (decision.behavior === 'deny') { return buildDeny(...) }
    }
  }
  return null
}
```

运行时层处理的判断包括：
- 当前命令是否匹配 allow 规则（`Bash(git *)`）
- 当前命令是否匹配 deny 规则（`Bash(rm *)`)
- 是否需要弹窗询问
- hook 有没有拦截决定
- 分类器（在 auto 模式下）是否批准

**运行时层的语义**：这个工具存在，但这次调用是否合法，要结合具体参数和当前上下文判断。

## 为什么两层缺一不可

**只保留装配层的问题**：装配层只能做工具级的”存在/不存在”判断，无法处理命令级的细粒度控制。`Bash` 这个工具不可能同时从工具池里裁掉（用户需要 git 命令），又允许（只允许安全的子集）。

**只保留运行时层的问题**：如果工具全部保留在工具池，Claude 看到所有工具，它的规划和决策就会把所有工具都纳入候选。即使运行时会拒绝，Claude 也可能在计划阶段浪费大量 token 规划一个注定失败的方案。装配层裁掉工具的作用是**缩小 Claude 的认知世界**，让它的规划更接地气。

## 规则匹配的精确语义

`toolMatchesRule` 里有一个微妙的细节：

```typescript
function toolMatchesRule(
  tool: Pick<Tool, 'name' | 'mcpInfo'>,
  rule: PermissionRule,
): boolean {
  // 规则必须没有 ruleContent 才能匹配整个工具
  if (rule.ruleValue.ruleContent !== undefined) {
    return false
  }
  // MCP 工具支持服务器级别的规则：mcp__server1 匹配 mcp__server1__tool1
  const ruleInfo = mcpInfoFromString(rule.ruleValue.toolName)
  const toolInfo = mcpInfoFromString(nameForRuleMatch)
  return (
    ruleInfo !== null &&
    toolInfo !== null &&
    (ruleInfo.toolName === undefined || ruleInfo.toolName === '*') &&
    ruleInfo.serverName === toolInfo.serverName
  )
}
```

`ruleContent` 的有无是区分”工具级规则”和”命令级规则”的分界线：
- `”Bash”` → ruleContent 为 undefined → 工具级，匹配整个 Bash 工具
- `”Bash(git *)”` → ruleContent 为 `”git *”` → 命令级，不被 toolMatchesRule 匹配

同时，MCP 工具支持服务器级别的 deny：`mcp__server1` 这条规则会 deny `mcp__server1__tool1`、`mcp__server1__tool2` 等所有来自该服务器的工具，不需要逐个列举。

## Agent 级别的 deny

除了工具级和命令级，还有 Agent 类型级别的 deny：

```typescript
export function getDenyRuleForAgent(
  context: ToolPermissionContext,
  agentToolName: string,
  agentType: string,
): PermissionRule | null {
  return getDenyRules(context).find(
    rule =>
      rule.ruleValue.toolName === agentToolName &&
      rule.ruleValue.ruleContent === agentType,
  ) || null
}
```

`Agent(Explore)` 这样的规则只 deny 特定类型的子 agent，而不是整个 Agent 工具。这是一个三级粒度：工具级 → 命令级 → Agent 类型级。

## 这道题考什么

考察对分层权限模型的理解是否到位。表面上看装配层和运行时层都在”判断权限”，但它们解决的问题不同：前者管”这个能力在 Claude 的认知世界里存不存在”，后者管”这次具体操作在制度上合不合法”。

能说清楚 `ruleContent` 是区分两层规则的分界线，并且能举出 MCP 服务器级 deny 这个具体例子的候选人，说明真正读过代码而不是在抽象层面理解。
