# 第十三卷回收索引：任务世界里自动后台化也落回同一转场协议

## 实现链
任务世界里有些入口一开始就不是人工点按，而是自动触发后台化。  
但这并不意味着系统要为每种入口都长出一套新的制度动作。

这条线真正逼出来的，是第十三卷里关于统一转场协议的判断。  
入口可以不同，最后的交棒动作最好还是收成同一套。

这页实际回收的实现链是：
- `05-07-02` 把自动后台化也写成同一条转场协议，这会在 `13-04-02` 里回收到“入口可以不同，真正关键的制度动作尽量只有一套”。

很多系统会把自动后台化当成另一套例外流程。  
Claude Code 更克制，它让自动入口最后也回到同一条正式转场协议。

## 普通做法
更普通的做法，是把后台动作当成松散的异步句柄，跑起来就算开始，结束了再临时补状态和通知。

## 为什么不用
Claude Code 不这么做，是因为没有统一任务对象，就没有稳定的生命周期、输出、通知和回看链，前后台切换也会变成一串彼此不认账的临时处理。

## 代价
代价是任务状态字段、输出文件、通知协议和前后台转场逻辑都会变厚，维护者要记住更多接缝。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
