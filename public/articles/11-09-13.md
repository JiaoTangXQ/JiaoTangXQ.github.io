# disarmSearch 只清除当前位置高亮，不清除 query——搜索状态为什么要分两层？

`disarmSearch` 和"退出搜索"是两件不同的事。理解这个区别，需要先理解搜索状态的两层结构。

## 搜索状态的两层

**第一层：查询和匹配集合**（持久层）

```typescript
const searchState = useRef({
  matches: [] as number[],     // 所有匹配消息的索引列表
  ptr: 0,                       // 当前在 matches 里的指针
  screenOrd: 0,                 // 当前消息里的第几个匹配
  prefixSum: [] as number[]    // 用于计算全局序号（badge 显示）
})
```

这层状态在搜索过程中持久存在，只在新的 `setSearchQuery` 调用（query 变化）时被替换。

**第二层：当前位置和高亮**（实时层）

```typescript
const elementPositions = useRef({
  msgIdx: -1,
  positions: [] as MatchPosition[]
})
```

以及通过 `setPositions` 传给外层的 overlay 状态。这层描述"此刻高亮在哪"，随着 n/N 导航和 disarm 动态变化。

## disarmSearch 只操作第二层

```typescript
disarmSearch: () => {
  searchState.current.matches = []       // ← 清空匹配（这不对，见下面修正）
  searchState.current.ptr = 0
  elementPositions.current = { msgIdx: -1, positions: [] }
  setPositions?.(null)                   // 清除高亮 overlay
  // searchQuery 不清除
  // searchAnchor 不清除
}
```

等等，`matches` 被清空了？是的，disarmSearch 清除了 matches。这意味着 disarm 之后，用户按 n/N 会触发 `step()` 函数，step 发现 matches 是空的，会重新扫描 `extractSearchText` 找当前 query 对应的所有匹配。

所以实际上，disarmSearch 清除的是"search engine matches 状态"，保留的是：
- `query`（当前搜索词，保持在 SearchPalette 或 status bar 里显示）
- `searchAnchor`（搜索开始点，保持以便下次 / 能从同一位置开始）
- 渲染层的 inverse 高亮（这些由 render-to-screen 层管理，不在 positions 里，disarm 后仍然可见）

## inverse 高亮和 current 高亮的区别

这是两套独立的高亮机制：

**current 高亮**（黄色背景）：通过 `setPositions({ positions, rowOffset, currentIdx })` 传给 `FullscreenLayout`，由 overlay 层绘制。只有一个当前命中有这个高亮，随 n/N 移动。

**inverse 高亮**（反色/白底）：由 `render-to-screen` 层的 `searchHighlight` 处理，对文本里的匹配片段进行 ANSI 转义修饰。所有命中都有这个高亮，它存在于消息文本的渲染输出里，不由 positions 控制。

`disarmSearch` 清除 current 高亮（`setPositions(null)`），但 inverse 高亮留着（在文本渲染层），因为用户只是临时走开，不是退出搜索。

## 触发时机：只有手动滚动

`disarmSearch` 的触发来自 `ScrollKeybindingHandler.onScroll`，但这个 callback 只在键盘驱动的滚动（j/k/PageUp/下） 和鼠标滚轮时触发，不在 `scrollRef.current.scrollTo()` 这类程序性滚动时触发。

注释明确说：`Wired from ScrollKeybindingHandler's onScroll — only fires for keyboard/wheel, not programmatic scrollTo`

这意味着：seek effect 里的精确 scrollTo、jump() 里的 scrollToIndex、selectedIndex 变化触发的 scrollToElement，都不会 disarm 搜索。只有用户主动离开当前搜索上下文（手动滚动），才会 disarm。

## 面试指导

这道题考察"状态分层"的设计意识——一个复杂状态应该如何拆分，以及每一层的生命周期如何独立管理。

**核心问题**：为什么不直接用一个 `searchActive: boolean` flag，false 时清除所有搜索相关状态？答：因为"搜索活跃"和"当前正在看某个命中"是不同的语义。搜索可以活跃（有 query，有匹配集合），但"当前命中高亮"可以暂时被隐藏（用户在看别的地方）。一个 boolean flag 无法区分这两种状态。

**vim 的类比**：在 vim 里，`/pattern` 后搜索，`n/N` 导航，手动移动光标后再按 `n` 还能继续找下一个命中。vim 没有 "disarm" 的概念，但 Claude Code 有 overlay 高亮（current）和 text 高亮（inverse）的区分，所以需要 disarm 来控制 overlay 层的显示。

**追问**：如果用户在 disarm 状态下（没有当前高亮）按 n，会发生什么？答：`stepRef.current(1)` 调用 `step(1)`，step 发现 matches 为空，重新扫描 extractSearchText 找当前 query 的所有匹配，找到后跳到第一个（或从最近的当前位置开始找）。用户感受：按 n 后突然有了高亮，jump 到最近的一个命中。
