# 子 agent 为什么站在 agent、工具和任务的交界处

子 agent 这个概念最容易被写成神秘色彩很重的东西，好像系统里忽然又冒出一个会思考的小人。Claude Code 的高明之处，在于它既承认子 agent 真的是一个新的 agent，又拒绝把它当成脱离制度的魔法。

从模型眼里看，发起子 agent 是一次工具调用，因为它是通过 `AgentTool` 触发的；从运行时眼里看，它又必须立刻变成任务，因为它要有 `taskId`、输出文件、前后台状态、可查看和可终止的生命周期；从系统分工上看，它当然还是一个 agent，因为它会拿到自己的提示词、能力边界，甚至可能有自己的类型、远端隔离方式和团队身份。

这就是它为什么站在三层交界处。少一层都不行。只把它当 agent，会重新掉回“暗处有个东西自己在忙”的黑箱叙事；只把它当工具，会低估它持续运行、回传进度、参与协作的复杂性；只把它当任务，又会看不见它其实保留了一整套 agent 的思考结构。Claude Code 把三种身份同时保留下来，于是委派既像能力调用，又像劳动托管，还像会话世界的延伸。

这种设计非常优雅，因为它让“多 agent”没有脱离主系统。代价也很明显：理解门槛会高一些，文档必须不断提醒读者，子 agent 不是第四种陌生生物，而是前三层在一个点上的重叠。

## 实现链
这一章对应的实现链是角色拆分：命令负责把人带进正确入口，工具负责真正伸手做事，任务负责把劳动变成正式对象，子 agent 则站在 `AgentTool` 和任务生命周期之间。代码没有把这些角色混成一个万能 handler，而是沿 `commands.ts`、`Tool.ts`、`Task.ts` 和 AgentTool/LocalAgentTask 这几层拆开。

## 普通做法
更普通的做法是让命令既做入口又做动作，必要时再在某个回调里顺手记状态、起并发。这样局部很快，也很像脚本。

## 为什么不用那种做法
Claude Code 不这么写，因为它需要共享制度：权限、状态显示、恢复、后台托管、团队协作都要求这些角色各守边界。角色一旦混写，后面每加一种制度都只能去各处补洞。

## 代价与回报
回报是系统里谁负责入口、谁负责动作、谁负责生命周期更清楚。代价是层次比普通脚本工具更正式，理解时要接受“会做事”和“被系统正式管理”不是一回事。
