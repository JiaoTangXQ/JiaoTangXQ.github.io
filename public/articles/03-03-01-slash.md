# 为什么 /命令 不是最早、最粗暴的第一分支

很多系统一见输入以 `/` 开头，就立刻跳去命令解析。这种做法简单，但它默认“命令识别”比“理解输入”更重要。Claude Code 的顺序正好相反。

在 `processUserInput()` 里，slash 分支之前已经发生了不少事情。系统会先处理图片和粘贴内容，必要时缩放、存盘、补元信息；再判断桥接来源是不是要特殊放行或特殊拦截；还会决定这一轮要不要先提取附件。只有这些前置整理完成以后，输入才会真正进入 slash command 分支。

这说明 Claude Code 对 slash 的理解很克制。它不是“第一道大门”，而是“已经被整理过的一条输入，接下来可能走向的第二入口”。这样做的好处很大：命令系统看到的不是一串未经处理的原始字符，而是已经带着图片、附件、来源语境和制度条件的输入对象。

这也是为什么它不容易误伤。越早抢着分流，看起来越果断，实际上越容易把还没被整理清楚的东西误判成命令。Claude Code 在这里选择先理解输入，再解释斜杠，这个顺序很对。

## 实现链
即使输入以 `/` 开头，系统也不会立刻认定它就是本地 slash 命令。它先看 `bridgeOrigin`、`skipSlashCommands`、图片和附件整理结果，再用 `effectiveSkipSlash` 决定这条消息有没有资格进入 slash 分支。

## 普通做法
很多 CLI 会把“首字符是 `/`”直接当命令，剩下的交给命令处理器自己收拾。

## 为什么不用
因为远端 bridge、隐藏 meta prompt 和某些移动端输入根本不应该触发本地 slash 命令；如果一上来就硬分支，`/config` 这种词在远端就会变成莫名其妙的本地控制指令。

## 代价
slash 的触发条件不再一眼看穿；但换来的是不同入口可以共享同一套输入语法，而不互相越权。
