# FileEditTool 为什么要做引号归一化？LLM 输出的格式陷阱

Claude 无法在输出中精确控制所有 Unicode 字符。特别是弯引号（`'`、`'`、`"`、`"`），模型在生成文本时往往会输出直引号（`'`、`"`）来替代。

这在文件编辑场景里是一个实际问题：如果用户的原始文件里有弯引号（在 macOS 上自动输入时很常见），Claude 输出的搜索字符串用直引号，就会匹配失败。

`tools/FileEditTool/utils.ts` 里专门处理了这个问题：

## 引号归一化

```typescript
// 定义弯引号常量——因为 Claude 无法输出弯引号，这些常量供系统内部使用
export const LEFT_SINGLE_CURLY_QUOTE = '\u2018'   // '
export const RIGHT_SINGLE_CURLY_QUOTE = '\u2019'  // '
export const LEFT_DOUBLE_CURLY_QUOTE = '\u201C'   // "
export const RIGHT_DOUBLE_CURLY_QUOTE = '\u201D'  // "

export function normalizeQuotes(str: string): string {
  return str
    .replaceAll(LEFT_SINGLE_CURLY_QUOTE, "'")
    .replaceAll(RIGHT_SINGLE_CURLY_QUOTE, "'")
    .replaceAll(LEFT_DOUBLE_CURLY_QUOTE, '"')
    .replaceAll(RIGHT_DOUBLE_CURLY_QUOTE, '"')
}
```

在搜索文件内容之前，先对**搜索字符串**和**文件内容**都做引号归一化。这样不管原始文件用了哪种引号风格，匹配都能成功。

## 行尾空格归一化

```typescript
export function stripTrailingWhitespace(str: string): string {
  // 处理三种行尾：CRLF、LF、CR
  const lines = str.split(/(\r\n|\n|\r)/)
  let result = ''
  for (let i = 0; i < lines.length; i++) {
    const part = lines[i]
    if (i % 2 === 0) {
      result += part.replace(/\s+$/, '')  // 偶数索引：行内容，去除行尾空格
    } else {
      result += part  // 奇数索引：行尾符，保留原样
    }
  }
  return result
}
```

这个实现值得注意：用 `split(/(\r\n|\n|\r)/)` 同时拆分内容和行尾符，通过奇偶索引区分两类片段。这样可以在去除行尾空格的同时精确保留原始的行尾符（不把 CRLF 变成 LF）。

## diff 生成和展示

`FileEditTool` 的编辑不是直接替换，而是要生成人类可读的 diff 展示给用户：

```typescript
import { structuredPatch } from 'diff'

// 生成结构化 patch
const patch = structuredPatch(
  oldPath, newPath,
  oldContent, newContent,
)

// 然后格式化成终端友好的 diff 视图
const display = getPatchForDisplay(patch)
```

`structuredPatch` 来自 `diff` 库，生成标准的 unified diff 格式。`getPatchForDisplay` 把它格式化成带颜色的终端输出（红色删除、绿色新增、灰色上下文）。

这个展示在权限请求阶段使用——当 Claude 想编辑一个文件，用户看到的不是"Claude 要修改 foo.ts"，而是精确的 diff，用户知道哪些行会被改动。

## 超时保护

```typescript
import { DIFF_TIMEOUT_MS } from '../../utils/diff.js'
```

diff 算法的时间复杂度在某些极端情况下会退化（比如两个长度相近、改动分散的大文件）。`DIFF_TIMEOUT_MS` 设置了一个超时上限，防止 diff 计算把 UI 卡死。超时后退化为显示"文件已修改，差异过大无法展示"的简化提示。

## 面试角度

这些"格式陷阱"处理是 LLM 应用工程的实际挑战：

1. **LLM 输出不是纯净代码**：模型可能输出弯引号、不一致的缩进、多余的空格。应用层需要处理这些"污染"
2. **归一化要双向**：不只是处理模型输出（搜索字符串），也要处理目标内容（文件内容）
3. **保留原始格式**：`stripTrailingWhitespace` 去掉空格但保留行尾符，是"修正错误但不改变原意"的归一化原则
4. **用 diff 显示而不是直接执行**：让用户看到改动内容，比"Claude 想修改文件，是否允许"更有信息量
