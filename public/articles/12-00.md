# 面试题：Claude Code 的外延执行为什么没有长出第二套运行时？

## 这道题真正在问什么

假设你在设计一个 AI Agent，现在需要给它加上"操控桌面"和"操控用户真实浏览器"两种外延能力。你会怎么架构？

大多数人的第一反应是：分别建子系统。桌面能力有自己的连接协议，浏览器能力有自己的权限逻辑，互相之间用桥对接。这看上去合理，但 Claude Code 完全没走这条路。

读 `setupComputerUseMCP()` 和 `setupClaudeInChrome()` 的源码，两者的返回值几乎同构：

```ts
return {
  mcpConfig: { [SERVER_NAME]: { type: 'stdio', command, args, scope: 'dynamic' } },
  allowedTools: [...],
  systemPrompt: '...',
}
```

桌面能力和浏览器能力，进入主系统时说的是同一种语言。这不是巧合，而是一个刻意的架构决策。

## 为什么不建第二套运行时

建独立子系统的代价，在大型系统里会被放大很多倍：

**工具发现会分叉。** 如果浏览器工具用私有 RPC，`ToolSearch`、`allowedTools` 和 skill hint 都得适配一遍新协议。

**权限链会分叉。** `request_access` 和 permission mode 是整个系统的统一门禁。一旦浏览器能力绕开它，"在浏览器里跳过所有检查"就成了合法姿势，主系统完全感知不到。

**日志和遥测会分叉。** 跨边界问题最怕两份日志对不上。`DebugLogger` 把浏览器侧日志全部翻回 `logForDebugging()`，就是为了让观测语言保持统一。

**恢复链会分叉。** 断线、重连、配对身份的持久化，这些都是整个工作台的共同设施。一旦浏览器能力自己管自己，升级、迁移、恢复都要重做。

## 翻译层是代价，也是护城河

统一性不是免费的。`setup.ts`、`mcpServer.ts`、`common.ts`、`wrapper.tsx` 这些文件都在替外部世界做翻译，代码量明显比"直接接进来"更重。

但这层翻译的另一面，是系统在任何地方都认识同一种对象：一个 MCP server，一组工具名，一套 permission mode。`main.tsx` 不需要知道"这个 MCP 是真的起了子进程还是进程内绑定"，`client.ts` 不需要知道"这台设备是浏览器还是桌面"，UI 渲染层不需要知道"这个工具调用来自哪个外部世界"。

这是成熟系统边界扩张时的标准取舍：**接入层变厚，消费层变薄**。

## 两条接入链的对比

`computerUse` 和 Claude in Chrome 并不是完全对称的，但两者的接入姿势有意保持一致：

| 维度 | computerUse | Claude in Chrome |
|------|-------------|-----------------|
| 入口 | `setupComputerUseMCP()` | `setupClaudeInChrome()` |
| 配置类型 | `type: 'stdio'` 动态 MCP | `type: 'stdio'` 动态 MCP |
| 工具命名 | `mcp__computer-use__*` | `mcp__claude-in-chrome__*` |
| 授权方式 | `request_access` 会话级 | `shouldEnableClaudeInChrome()` + feature flag |
| 进程策略 | 进程内绑定（不真起子进程） | 真起 MCP 子进程 |

两者都服从同一套宿主语言，但在实现细节上各自选了最适合自己的方案。这才是"统一口音，克制成本"的具体含义。

## 这道题的面试考察点

如果你在系统设计面试中遇到"如何给 Agent 加外延能力"这类题，Claude Code 的解法提供了一个很好的参照框架：

1. **先定义统一协议**（MCP），再让所有外部能力学这门语言
2. **把授权集中**，不在每个外延能力里单独实现权限判断
3. **日志和遥测收口**，外部世界不另起观测语言
4. **形态差异在外层消化**（bundled vs 非 bundled、进程内 vs 子进程），不污染协议层
5. **长期设施讲稳定性**（固定落点、少惊动、持久身份），而不是每次重新建立

代价是接入层代码比业务代码更重，新能力上线速度会慢一些。但如果目标是让 Agent 能力长期生长在同一个工作台上，这笔账是合算的。
