# 后端可切换说明布局不是美化，而是正式能力

如果布局只是美化，根本不用专门做后端抽象。Claude Code 反而专门做了 pane backend，说明它把“团队空间”当正式能力。

## 实现链

`teammateLayoutManager.ts` 不直接操作 tmux 或 iTerm2，而是统一调用 backend。`registry.ts` 决定用 tmux 还是 iTerm2，`PaneBackendExecutor` 则把 pane backend 包成统一执行接口。这说明布局和执行不是两个独立附加件，而是同一个团队运行时的一部分。

## 普通做法

普通做法可能会把 UI 布局留给终端或宿主环境自动处理，核心系统只关心 agent 是否启动成功。

## 为什么不用

Claude Code 不愿意把团队空间外包出去，因为空间本身影响协作秩序。lead 在哪、队友如何展开、用户怎么看见他们，都不是美观问题，而是团队控制面的问题。

## 代价

代价是得维护多套后端适配，还要处理环境探测、回退和缓存。
