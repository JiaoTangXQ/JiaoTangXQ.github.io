# MCP client.ts 不是 SDK 封装——它是一个连接状态机

## 协议设计角度：MCP 为什么不能只是一个 HTTP 客户端？

HTTP 请求是无状态的：发出去，等回来，处理结果，完成。

MCP 不是这样。一个 MCP 服务器连接后可能：
- 提供随时间变化的工具列表（`list_changed` 通知）
- 要求 OAuth 认证，且 token 会过期
- 在 session 超时后返回 404 `-32001`（需要重连）
- 在初始化阶段发送 elicitation 请求（用户还没就位）
- 通过不同的 transport（stdio、SSE、HTTP、WebSocket、SDK 内进程）通信

这些特性决定了 MCP 客户端必须是**有状态的**，必须维护一个连接生命周期。`client.ts` 是这个状态机的实现。

---

## 连接的五种状态

`src/services/mcp/types.ts` 里定义了 MCP 服务器连接的五种状态：

```typescript
export type MCPServerConnection =
  | ConnectedMCPServer     // 已连接，有能力列表
  | FailedMCPServer        // 连接失败
  | NeedsAuthMCPServer     // 需要认证
  | PendingMCPServer       // 正在连接（包含重试计数）
  | DisabledMCPServer      // 已禁用
```

```typescript
export type ConnectedMCPServer = {
  client: Client
  name: string
  type: 'connected'
  capabilities: ServerCapabilities
  serverInfo?: { name: string; version: string }
  instructions?: string
  config: ScopedMcpServerConfig
  cleanup: () => Promise<void>
}

export type PendingMCPServer = {
  name: string
  type: 'pending'
  config: ScopedMcpServerConfig
  reconnectAttempt?: number
  maxReconnectAttempts?: number
}
```

`PendingMCPServer` 里的 `reconnectAttempt` 和 `maxReconnectAttempts` 说明重连本身就是受控制的——不是无限重试，而是有上限的指数退避。

## 自定义错误类型反映状态机边界

`client.ts` 里定义了几个精心设计的错误类型：

```typescript
// MCP 工具调用失败因为认证问题（如 OAuth 401）
export class McpAuthError extends Error {
  serverName: string
  constructor(serverName: string, message: string) { ... }
}

// Session 过期，需要重新获取客户端
class McpSessionExpiredError extends Error {
  constructor(serverName: string) { ... }
}

// MCP 工具返回 isError: true
export class McpToolCallError_I_VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS
  extends TelemetrySafeError_I_VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS { ... }
```

这三个错误对应三种不同的恢复策略：
- `McpAuthError` → 把服务器标记为 `needs-auth`，触发 OAuth 流程
- `McpSessionExpiredError` → 清除连接缓存，触发重连
- `McpToolCallError` → 把错误返回给模型，由模型决定是否重试

## Session 过期检测

```typescript
export function isMcpSessionExpiredError(error: Error): boolean {
  const httpStatus =
    'code' in error ? (error as Error & { code?: number }).code : undefined
  if (httpStatus !== 404) return false

  // MCP 规范：session 不存在时返回 JSON-RPC -32001
  return (
    error.message.includes('"code":-32001') ||
    error.message.includes('"code": -32001')
  )
}
```

这个函数有两个判断条件（HTTP 404 + JSON-RPC -32001），缺一不可：
- 只看 HTTP 404：可能是服务器地址错了，不是 session 过期
- 只看 JSON-RPC code：可能是其他协议错误，不是 session 过期

两者并列，才能准确区分"session 过期"和"其他类型的 404"。

## Transport 的多态性

`client.ts` 支持 6 种 transport：

```typescript
export type Transport = z.infer<ReturnType<typeof TransportSchema>>
// = 'stdio' | 'sse' | 'sse-ide' | 'http' | 'ws' | 'sdk'
```

每种 transport 对应不同的通信方式：
- `stdio`：子进程，通过 stdin/stdout 通信（本地 MCP server 的主流方式）
- `sse`：Server-Sent Events（HTTP 长连接，单向推送）
- `http`：Streamable HTTP（新 MCP 规范，支持双向流）
- `ws`：WebSocket
- `sdk`：进程内通信（SDK embedded server）

`client.ts` 里的 transport 选择逻辑是一个 switch-case，根据配置类型选对应的 Transport 实现。这是策略模式的典型应用：连接建立方式不同，但上层状态机逻辑相同。

## 为什么工具超时设了 100,000,000ms？

```typescript
const DEFAULT_MCP_TOOL_TIMEOUT_MS = 100_000_000  // 约 27.8 小时
```

这个注释说的很直白：

```typescript
/**
 * Default timeout for MCP tool calls (effectively infinite - ~27.8 hours).
 */
```

原因：MCP 服务器可能执行长时间运行的操作（如运行测试、编译代码、等待人工确认）。设置一个"永远不会触发"的超时，但保留超时机制，是一种"有备无患"的设计：超时代码存在但不影响正常使用；如果真的需要调整，用 `MCP_TOOL_TIMEOUT` 环境变量覆盖即可。

## 面试指导

**"如何设计一个需要管理持久连接的客户端库？"**

核心要点：

1. **状态枚举先于接口设计**：把连接的所有可能状态列清楚（Pending、Connected、Failed、NeedsAuth、Disabled），再设计状态转换逻辑
2. **错误类型即状态转换信号**：不同类型的错误触发不同的状态转换（401 → NeedsAuth，404 -32001 → 重连）
3. **Transport 抽象与状态机分离**：通信方式（stdio/SSE/HTTP/WS）是可替换的实现细节，状态机逻辑不应该感知它
4. **超时策略要显式**：无论是"有限超时"还是"近似无限"，都应该是有意识的决策，而不是遗漏

说出"MCP 客户端本质上是一个状态机，而不是无状态的 HTTP 客户端"——这会让面试官知道你理解了有状态连接管理的本质。
