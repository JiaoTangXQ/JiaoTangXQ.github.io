# 面试题：状态栏命令未返回时，全屏模式下渲染占位符而非空节点——这个细节的工程动机是什么？

代码里有些决策只有读过注释才能理解。这道题就是这类——如果你只看代码，会觉得奇怪；看了注释，才知道这是一个防止布局抖动的精确工程决策。

## 代码和注释

```typescript
// StatusLine.tsx
if (statusLineText === undefined) {
  // Don't render an empty element — the footer's min-height handles the
  // placeholder row. If fullscreen, keep a blank row so the footer height
  // doesn't jump from 0→1 when the command returns.
  return isFullscreenEnvEnabled() ? <Text> </Text> : null;
}
```

关键词："footer height doesn't jump from 0→1"。

这是在描述一个终端渲染的具体问题：当底部区域高度从 0 行变成 1 行时，会发生什么。

## Yoga 布局引擎的行为

Ink 框架使用 Yoga 布局引擎（Facebook 的 CSS Flexbox C 实现）来计算终端界面的布局。

在普通 Web 中，高度变化可以用 CSS transition 平滑过渡，用户不会感到突兀。但在终端里，没有"过渡动画"——高度变化是瞬间的。

当 StatusLine 从 `null`（不渲染，高度 0）变成 `<Text>...</Text>`（渲染，高度 1）：

```
before:
┌────────────────────────┐  ← 终端顶部
│                        │
│  [消息区：rows - 1]     │
│                        │
│                        │
│                        │  ← 第 rows-1 行（消息区最后一行）
└────────────────────────┘
│  [输入区：1行]          │  ← 第 rows 行
└────────────────────────┘

after (状态栏命令返回):
┌────────────────────────┐
│                        │
│  [消息区：rows - 2]     │  ← 被压缩了！
│                        │
│                        │  ← 原来第 rows-1 行的内容不见了
└────────────────────────┘
│  [状态栏：1行]          │  ← 新出现
│  [输入区：1行]          │
└────────────────────────┘
```

用户正在阅读的最后一行消息，在状态栏出现的瞬间消失了，换成了上面一行的内容。这是"抖动"——阅读位置突然跳动。

## 为什么全屏比非全屏更需要这个保护

**全屏模式的用户期望**

在全屏（fullscreen env）模式下，整个终端都是 Claude Code 的界面。用户的眼睛固定在消息区，把它当成一个稳定的阅读区域。这个模式下，任何高度变化都会直接影响"正在看的那部分内容"。

**非全屏模式的容忍性**

在非全屏模式（Claude Code 只占了终端的一部分，或者是正常的 REPL 滚动模式）下，消息区本来就是动态增长的——每条新消息都会把内容往上推。用户已经习惯了这种"向上滚动"的体验。底部多出一行，在视觉感知上不那么突兀。

## 占位符的精确位置

```typescript
return isFullscreenEnvEnabled() ? <Text> </Text> : null;
```

- 只在全屏模式下返回占位符
- 非全屏模式返回 null（完全不渲染）
- 占位符是一个空格字符，不是空字符串（空字符串在 Yoga 里可能不产生行高）

这个条件是精确的：只在"会因为高度突变而造成阅读抖动"的模式下才做这个保护，不滥用占位符（否则非全屏模式下也会有一行空白）。

## 和 PromptInputFooterLeftSide 的配合

```typescript
// PromptInputFooterLeftSide.tsx
if (isFullscreenEnvEnabled() && !parts.length && !tasksPart && !modePart) {
  return <Text> </Text>;
}
```

StatusLine 和 PromptInputFooterLeftSide 用了**相同的策略**：在全屏模式下、没有实质内容时，都返回占位空格而不是 null。

这两个组件都在 footer 区域，都在用相同的方式维护 footer 高度的稳定性。这说明这不是某个工程师的个人习惯，而是 Claude Code 里对"全屏布局稳定性"的系统性共识。

## 权衡：占一行空白 vs 阅读稳定性

这个设计有一个明显的代价：在状态栏命令还没返回时（以及底部没有其他内容时），全屏下总是有一行看起来空白的区域。

这一行空白：
- 用户可能会注意到并觉得奇怪
- 如果终端很矮（比如只有 20 行），这一行空白占比更显眼
- 从"信息密度"角度看，是一种浪费

但换来的是：
- 状态栏命令返回时，阅读位置不会跳动
- 全屏模式下，消息区高度稳定
- 用户不会因为底部突然出现内容而失去阅读位置

对于一个主要场景是"长时间阅读对话内容"的全屏模式来说，阅读稳定性显然比那一行空白更重要。

## 面试指导

**直接问法**："为什么 statusLineText 是 undefined 时全屏下要渲染占位符？"

答：防止状态栏命令返回时底部高度从 0→1，把消息区压缩一行，导致阅读位置跳动。全屏模式下用户建立了"消息区稳定"的预期，任何高度突变都会打断阅读体验。

**追问**："为什么用 `<Text> </Text>` 而不是 `<Text></Text>`？"

答：空字符串内容的 Text 在 Yoga 里可能不产生行高（取决于实现），而空格字符确保了至少有一个字符被渲染，Yoga 会为此分配行高。这是利用了 Yoga 的"内容决定高度"行为。

**设计追问**："如果不考虑这个占位符，有什么其他方式解决布局抖动问题？"

答：可以给 footer 区域设置 `minHeight={1}`，强制它始终占至少一行。但这需要 Yoga 支持 minHeight（它应该支持，因为 Yoga 实现了大部分 CSS Flexbox），而且语义更清晰（"至少一行"）。实际代码没用这种方式，可能是历史原因或者 Yoga 的特定行为。

**类比追问**："浏览器端有类似的布局稳定性问题吗？"

答：有，而且 Google 把它定义成了一个正式的性能指标：CLS（Cumulative Layout Shift，累积布局偏移）。页面元素突然出现或大小变化，导致其他内容位置跳动，是一个影响用户体验的重要问题。浏览器端用 `min-height`、`aspect-ratio`、skeleton 组件等方式解决。终端里没有这么多工具，只能靠占位符。

---

*核心考察点：Yoga 布局引擎的行为理解、布局稳定性 vs 信息密度的 trade-off、终端 vs 浏览器 UI 的差异性处理。*
