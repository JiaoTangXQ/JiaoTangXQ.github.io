# 能提前的提前，必须收口的收口：启动阶段的双向优化

面试题版本：**"Claude Code 的启动过程包括哪些同步步骤和异步步骤？哪些操作必须在第一次模型调用之前完成，哪些可以推迟？这个区分背后的判断逻辑是什么？"**

## 启动阶段的两类操作

**必须在第一次调用前完成（同步/阻塞）**：
- 权限配置加载（`PermissionMode`）
- 工具许可初始化
- hooks 注册
- 用户身份验证（API key 有效性）
- 基本的 AppState 初始化

**可以在后台异步推迟的**（`startDeferredPrefetches()`）：
- 用户信息预取（显示名称、订阅状态）
- 模型能力列表（支持哪些参数）
- MCP 官方注册表（推荐的 MCP server 列表）
- 技能变更探测器（监控技能文件是否被修改）
- 事件循环卡顿探测（性能监控）

## 判断标准：会不会影响第一次调用的正确性

**权限配置必须先加载**：第一次工具调用就需要权限检查，如果权限还没加载，要么跳过检查（不安全）、要么阻塞等待（影响响应时间）。

**用户信息可以推迟**：第一次模型调用不需要知道用户显示名字，这是 UI 的信息，可以异步加载。

**模型能力可以推迟**：大多数情况下 Claude Code 使用标准模型参数，高级参数的能力信息（比如是否支持 `thinking` 参数）可以在第一次调用后才获取，如果用到时还没获取，再等待即可。

**事件循环监控可以推迟**：这是性能诊断工具，不影响功能正确性。

## startDeferredPrefetches 的实现

`src/setup.ts`（或类似的启动文件）里：

```typescript
async function startup() {
  // 第一阶段：同步阻塞，必须完成
  await loadPermissionConfig()
  await initTools()
  await registerHooks()
  await verifyApiKey()
  
  // 渲染第一次 UI（用户可以开始交互了）
  renderInitialUI()
  
  // 第二阶段：异步推迟，不阻塞交互
  startDeferredPrefetches({
    userInfo: prefetchUserInfo(),
    modelCapabilities: prefetchModelCapabilities(),
    officialMcpRegistry: prefetchOfficialRegistry(),
    skillWatcher: startSkillChangeWatcher(),
    perfMonitor: startEventLoopMonitor(),
  })
}
```

第二阶段的任务开始执行，但不 `await` 它们。主线程继续，用户可以立即开始使用。预取的数据在需要时可以通过 `await` 获取（如果还没完成则等待，否则立即返回缓存值）。

## "抢跑式预取"的设计细节

"能提前的提前"的一个子原则是：**把可预测的依赖提前触发，而不是等到真正需要时才开始**。

比如，模型能力信息在用户使用高级功能时才用到，但这个请求可以在启动时就发出去。当用户 5 分钟后尝试使用高级功能时，能力信息可能已经加载好了，用户感知不到任何延迟。

这是对用户行为的预判：启动 Claude Code 后，用户可能会用 X 功能，所以我们提前把 X 功能需要的数据准备好。

## 收口和开闸的平衡

"先收口再开闸"是一个时序原则：

**收口**（必须在开始前完成）：
- 权限边界清晰了吗？（工具许可配置完了）
- 约束条件满足了吗？（API key 有效、基本配置加载）

**开闸**（首次交互前的就绪状态）：
- 用户可以开始输入了
- 基本功能可以使用了

**预取**（开闸后的异步优化）：
- 高级功能的数据在后台加载
- 监控和诊断工具在后台启动

这三个阶段的顺序不能打乱：必须先收口，再开闸，预取可以和开闸同步进行，但不能早于收口。

## 面试角度的总结

启动阶段的双向优化（提前预取 + 先收口）体现了一个工程原则：**安全约束是阻塞性的前置条件，性能优化是非阻塞性的后台任务**。把这两类操作混在一起处理（要么全部阻塞、要么全部异步）都会有问题：全阻塞导致启动慢，全异步导致约束可能在第一次调用时还没建立。精确区分哪些是约束、哪些是优化，是让系统既快又安全的基础。
