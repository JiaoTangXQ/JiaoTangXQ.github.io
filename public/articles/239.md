# 注册式桥接为什么先承认打包现实

面试题版本：**"Claude Code 的插件/工具注册采用显式配置文件（~/.claude/config）而不是自动发现（扫描 PATH 或特定目录）。这两种方案各有什么优缺点？Claude Code 为什么选择显式注册？"**

## 自动发现方案的诱惑

自动发现很优雅：
```bash
# 把插件放到指定目录，自动被发现
cp my-mcp-server ~/.claude/mcp-servers/
```

不需要配置文件，不需要注册步骤，"放进去就用"。在受控的单机开发环境里，这很好用。

## 自动发现在真实环境里的问题

**安全问题**：如果任何人往 `~/.claude/mcp-servers/` 目录放文件，就能让 Claude Code 执行它，这是严重的安全漏洞。恶意软件可以通过这个路径注入代码。

**多用户环境**：公司环境里，管理员需要控制哪些 MCP server 可以被使用，哪些不行。自动发现无法实现这种管理控制（没有"这个 server 被授权使用"的声明）。

**版本和路径管理**：同一个工具可能有多个版本安装在不同路径，自动发现不知道应该用哪个。显式注册可以指定精确路径。

**可见性**：管理员或用户想知道"当前有哪些插件/MCP server 在生效"，自动发现需要扫描多个目录，结果可能不直观。显式配置文件是一个清单，所有已注册的来源一目了然。

## 注册式桥接的实现

`~/.claude/config` 里的 MCP server 配置：

```json
{
  "mcpServers": {
    "my-database-tool": {
      "command": "/usr/local/bin/my-db-mcp-server",
      "args": ["--config", "/etc/db-config.json"],
      "env": {
        "DB_PASSWORD": "..."
      }
    }
  }
}
```

这个配置是显式的：
- server 名字（命名空间）
- 可执行文件路径（精确到版本）
- 启动参数（允许同一个 server 程序用不同配置启动多个实例）
- 环境变量（安全的凭据传递）

## 多个配置层级的现实

之前提到的 7 种 ConfigScope（local/user/project/enterprise/dynamic/claudeai/managed），每一层的存在都对应一个真实的用户角色：

- `user` scope：个人开发者配置自己的工具
- `project` scope：项目维护者配置项目特定工具
- `enterprise` scope：IT 管理员配置公司范围内的工具
- `managed` scope：安全团队锁定某些配置不允许覆盖

这些层级不是理论上的分层，而是实际用户场景的直接反映。自动发现方案无法表达这些层级关系——所有发现的工具都是平等的，没有来源和优先级的区分。

## 承认打包现实的意义

"承认打包现实"意味着承认：
1. 软件是通过包管理器安装的，有明确的安装路径
2. 不同用户有不同的安装路径（全局安装 vs 用户级安装）
3. 企业环境里有版本控制和访问控制需求
4. 配置需要在不同会话、不同用户之间持久保存

显式注册配置文件正是为了应对这些现实而存在的。它不是因为"显式好于隐式"的哲学偏好，而是因为用户的真实环境就是这么运作的。

## 面试角度的总结

注册式桥接 vs 自动发现的选择，不是美学问题，而是工程诚实问题。在真实的生产环境里，工具安装有复杂的路径管理、版本控制、安全约束、多层配置需求。自动发现在这个现实面前会遇到安全漏洞、路径歧义、管理控制缺失等问题。显式注册承认了这些复杂性，把它们变成了可配置、可管理、可审计的显式声明，而不是隐式的"扫描所有地方"。
