# 工具为什么是真正伸手做事的那一层

如果说命令决定“从哪扇门进去”，那么工具决定的就是“进去以后手到底怎么动”。Claude Code 把真正碰文件、碰终端、碰网页、碰任务世界的能力，都收在工具这一层里。这不是名字上的区分，而是工程责任的区分。

在这套设计里，一个工具不是一段随便调用的脚本，而是一个被完整包装的行动单元。它有输入约束，有权限检查，有并发属性，有执行结果，也有给人看的展示方式。换句话说，工具从一开始就被当成运行时里的正式角色，而不是模型临时拼出来的一次外部调用。

这样做有两个很大的好处。第一，所有真正会改变世界的动作，都能被放进统一制度里管理。第二，系统可以在“工具怎么被展示给模型”“工具怎么被展示给人”“工具实际怎么执行”之间保持清楚分层。于是 Claude Code 既能让模型把工具当能力使用，又不会把整台系统搞成一堆难以追踪的脚本碎片。

很多 agent 看起来也会调工具，但往往只是“能调”。Claude Code 的不同在于，它把工具认真做成了行动抽象。也正因为这一层站得稳，后面的权限、并发、恢复、多 agent 协作，才不至于变成补丁堆。

## 实现链
这一章对应的实现链是角色拆分：命令负责把人带进正确入口，工具负责真正伸手做事，任务负责把劳动变成正式对象，子 agent 则站在 `AgentTool` 和任务生命周期之间。代码没有把这些角色混成一个万能 handler，而是沿 `commands.ts`、`Tool.ts`、`Task.ts` 和 AgentTool/LocalAgentTask 这几层拆开。

## 普通做法
更普通的做法是让命令既做入口又做动作，必要时再在某个回调里顺手记状态、起并发。这样局部很快，也很像脚本。

## 为什么不用那种做法
Claude Code 不这么写，因为它需要共享制度：权限、状态显示、恢复、后台托管、团队协作都要求这些角色各守边界。角色一旦混写，后面每加一种制度都只能去各处补洞。

## 代价与回报
回报是系统里谁负责入口、谁负责动作、谁负责生命周期更清楚。代价是层次比普通脚本工具更正式，理解时要接受“会做事”和“被系统正式管理”不是一回事。
