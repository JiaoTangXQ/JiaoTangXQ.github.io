# 系统设计题：如果让你设计 Tool 接口，你会少加哪些字段，然后后悔？

## 你的第一版设计

假设让你从头设计 Claude Code 的工具接口，第一版可能长这样：

```
Tool {
  name: string
  description: string
  inputSchema: ZodSchema
  call(args, context): Promise<ToolResult>
}
```

四个字段，够用，简洁。然后你开始接真实需求，一个问题一个问题地把接口逼到 Claude Code 现在的样子。

## 第一个被追加的字段：isConcurrencySafe

你的 agent 开始并行调用工具。模型同时发出两个 `FileEditTool` 请求，都要修改同一个文件。

你需要知道哪些工具可以并行、哪些不行。`BashTool` 通常可以并发，但 `FileEditTool` 对同一文件操作时不行。所以加上：

```
isConcurrencySafe(input: Input): boolean
```

注意它接收 `input`：并发安全性有时取决于具体参数，比如两个 `FileRead` 操作读不同文件是安全的，读同一个也是安全的（读操作幂等），但两个 `FileEdit` 操作编辑同一文件就可能互相覆盖。

## 第二个被追加的字段：maxResultSizeChars

`FileReadTool` 被模型用来读一个 50MB 的日志文件。整个文件内容作为工具结果塞进上下文，API 请求直接超出 token 限制或者报文大小限制。

你需要一个机制：当工具结果超过某个大小，把内容存到磁盘，给模型发一个"预览 + 文件路径"，让模型用 `FileReadTool` 再读局部内容。但 `FileReadTool` 自己的结果就不该走这条路——那会造成 Read → 结果存磁盘 → Read 读磁盘文件 → 循环。

所以 `Tool.ts` 里有：

```
maxResultSizeChars: number
```

`FileReadTool` 把这个设为 `Infinity`，明确豁免自己不走磁盘持久化路径。其他工具按实际情况设上限。

## 第三个被追加的字段：isDestructive

模型调用 `BashTool` 执行 `rm -rf /important-directory`。这是不可逆操作。

你需要在不可逆操作执行前给用户一个特殊的确认界面，和普通权限弹窗区分开来。所以加上：

```
isDestructive?(input: Input): boolean
```

注释里写着：`Only set when the tool performs irreversible operations (delete, overwrite, send).` 这三个词非常精确：删除、覆盖写（不可回退的写）、发送（消息、邮件、网络请求）。

## 第四个被追加的字段：validateInput

Zod schema 能做类型校验，但做不了业务逻辑校验。比如 `FileEditTool` 接收一个 `old_string`，如果这个字符串在文件里根本不存在，Zod 不会报错，但执行时会无声失败。

你需要在 `call()` 之前先做一次语义验证，把明显无效的调用提前告诉模型：

```
validateInput?(input: Input, context: ToolUseContext): Promise<ValidationResult>
```

`ValidationResult` 是一个 discriminated union：成功是 `{ result: true }`，失败是 `{ result: false, message: string, errorCode: number }`。失败时模型会收到一条解释性错误，可以调整参数重试，而不是等到执行时才发现问题。

## 第五个被追加的字段组：interruptBehavior 和 isReadOnly

用户在 `BashTool` 执行一个耗时命令时输入了新消息。工具应该被取消还是继续跑？

读操作（`FileReadTool`、`GrepTool`）被打断通常无所谓，但写操作被打断可能留下半写的文件。所以需要两个配合的字段：

```
isReadOnly(input: Input): boolean
interruptBehavior?(): 'cancel' | 'block'
```

`isReadOnly` 让系统知道这次调用是否会改变世界，`interruptBehavior` 决定新消息进来时怎么处理正在进行的执行。

## 第六个被追加的字段：backfillObservableInput

工具的 API 入参随版本演进会改变字段名，但已有的 hook 脚本、transcript 记录、外部监控系统已经在依赖旧字段名。

你需要在不改变发给 API 的原始 input 的前提下（不能破坏 prompt cache），给"观察者"（SDK 流、transcript、canUseTool、PreToolUse/PostToolUse hooks）看到一个补全了兼容字段的版本：

```
backfillObservableInput?(input: Record<string, unknown>): void
```

这个方法在原始 input 的副本上原地 mutate，加上派生字段或旧名字段。必须是幂等的。

## 结论：接口的每个字段背后都有一个真实失败

`Tool` 接口现在超过 20 个字段，但没有一个是出于"完整性"或"美观性"加进去的。每个字段对应一类具体的错误场景：

- 没有 `isConcurrencySafe` → 并发写数据损坏
- 没有 `maxResultSizeChars` → 上下文被大文件撑爆
- 没有 `isDestructive` → 不可逆操作无特殊提示
- 没有 `validateInput` → 语义错误的工具调用无声失败
- 没有 `interruptBehavior` → 中断策略不一致
- 没有 `backfillObservableInput` → 版本兼容性在工具边界断掉

## 面试指导

**这道题的考法**：给你看 Tool 接口的某几个字段，问"为什么要加这个，不加会怎样"。

**满分策略**：每个字段对应一个具体场景，说清楚场景 → 问题 → 字段解决的是哪个具体问题。不要说"提高灵活性"，要说"如果没有这个字段，XXX 场景会发生 XXX 错误"。

**容易丢分的地方**：把 `validateInput` 和 Zod schema 混为一谈，或者说不清楚为什么需要两层验证。答案是：Zod 验证类型（是不是字符串、是不是对象），`validateInput` 验证语义（这个字符串在文件里存不存在、这个路径有没有权限访问）。两件不同的事。
