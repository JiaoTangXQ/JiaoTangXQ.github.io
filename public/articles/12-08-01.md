# 扩张边界不等于重写骨架，外部能力仍要服从原制度

看 `computerUse` 和 Claude in Chrome 的 setup 代码，会发现它们虽然连接的是完全不同的现实世界，但进入系统的姿势几乎一样：都先产出动态 MCP 配置、统一命名的工具名和额外的 system prompt，再让现有的工具发现、权限和主循环去消费。`main.tsx` 也没有因为这些能力特殊就另建一个“外部能力主循环”，而是照旧把它们纳入原本的宿主结构。

更常见的扩张方式，是能力一变特殊，就顺势长出一套专用 runtime。比如桌面能力一套 agent，浏览器能力另一套 agent，互相之间再用桥对接。那样每个局部都能按自己最顺手的方式写，但最后会把主系统撕成几块。

Claude Code 没这么做，所以第十二卷真正要夸的不是“连得更远”，而是“连得更远以后还继续服从原制度”。这条路更难，因为每种外部能力都得先学宿主语言；但它也更稳，因为系统不会随着边界扩张而长出第二套骨架。

## 实现链
`main.tsx` 把 computer use 和 Claude in Chrome 都并成动态 MCP 配置；`usePromptsFromClaudeInChrome()` 还会把当前工具权限模式翻译成扩展侧 `ask/skip_all_permission_checks`。外部能力进来了，但仍服从原权限制度和原工具协议。

## 普通做法
普通做法是给浏览器和桌面能力单独开一套“特权通道”，绕开原来的 permission mode。

## 为什么不用
Claude Code 不开特权旁路，因为这会让“强能力”先学坏，越往后越难收回来。

## 代价
代价是新能力接入速度会慢一些，必须先过原制度的翻译层。
