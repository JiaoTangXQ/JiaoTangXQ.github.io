# 面试题：打开 App.tsx 只有三层 Provider，你能从嵌套顺序里读出什么架构决策？

这是一道区分度很高的面试题。

把 `App.tsx` 的代码放到候选人面前，问他能读出什么。初级候选人会说"用了 React Context 做状态管理"。中级候选人会说"关注点分离，三层分别管不同的事"。

高级候选人会从**嵌套顺序**本身读出一套架构原则。

---

## 源码是这样的

```tsx
export function App({ getFpsMetrics, stats, initialState, children }) {
  return (
    <FpsMetricsProvider getFpsMetrics={getFpsMetrics}>
      <StatsProvider store={stats}>
        <AppStateProvider
          initialState={initialState}
          onChangeAppState={onChangeAppState}
        >
          {children}
        </AppStateProvider>
      </StatsProvider>
    </FpsMetricsProvider>
  )
}
```

九行代码，五个参数。先回答最基础的问题：**为什么是这个顺序，而不是别的顺序？**

---

## 嵌套顺序意味着依赖方向

React Provider 的嵌套顺序决定了一件事：里层可以读外层，外层不能读里层。

`AppStateProvider` 在最里面，意味着：

1. FpsMetrics 和 Stats 可以观测整棵 AppState 管辖的子树
2. AppState 里的组件可以读取 FPS 和 Stats（如果它们需要的话）
3. FPS 和 Stats 不依赖 AppState——它们在更高层

这就是**观测层在外，事实层在内**的体现。观测系统包裹事实系统，但事实系统不应该依赖观测系统。

如果把顺序调过来——AppState 包着 Stats——那么统计系统就会变成 AppState 的下属，每次 AppState 重建，统计系统也跟着失效。这不对。

---

## 五个参数的类型签名已经在说话

```
getFpsMetrics: () => FpsMetrics | undefined   // 函数，不是值
stats?: StatsStore                             // 可选
initialState: AppState                         // 必选
onChangeAppState: (args) => void               // 外部注入的副作用出口
children: ReactNode                            // 剩下的一切
```

**`getFpsMetrics` 是函数而不是值** — 意味着 FPS 数据活在外部，组件想看就去取。没有推送，没有订阅，不触发重渲染。这是一个「侧信道」而不是「业务状态」。

**`stats` 是可选的** — 不传就用内部创建的。这是「自带电池」设计，让统计系统在测试里可以被精确控制，在生产里可以接不同的监控后端。

**`onChangeAppState` 是外部注入的** — 所有 AppState 变化的副作用都从这里出去。权限模式同步、模型持久化、认证缓存清理，全部集中在这一个回调里。这是依赖倒置：Provider 不依赖具体的副作用实现。

---

## 为什么 FPS 不能放进 AppState

这是最容易被忽视的设计决策，值得展开。

如果把 FPS 数字推进 AppState，每次 FPS 更新（可能每秒十几次）都会触发所有订阅 AppState 的组件重渲染。对一个终端界面来说这是性能灾难。

更深层的问题：FPS 是**观测数据**，不是**业务事实**。「权限模式是 default」是一个业务事实，「当前帧率是 58fps」是一个观测结果。把观测结果混入业务事实，会让状态层的语义变得模糊：消费者无法区分哪些字段是驱动业务的，哪些只是监控用的。

```tsx
// fpsMetrics.tsx
const FpsMetricsContext = createContext<FpsMetricsGetter | undefined>(undefined)

export function FpsMetricsProvider({ getFpsMetrics, children }) {
  return (
    <FpsMetricsContext.Provider value={getFpsMetrics}>
      {children}
    </FpsMetricsContext.Provider>
  )
}

export function useFpsMetrics() {
  return useContext(FpsMetricsContext)  // 返回的是 getter 函数，不是数值
}
```

`useFpsMetrics()` 返回的是 getter 函数。组件拿到函数，想查就查，但不会因为 FPS 变化而被迫重渲染。这是「拉取」而不是「推送」模式。

## StatsProvider 的 process.on('exit') flush

```tsx
useEffect(() => {
  const flush = () => {
    const metrics = store.getAll()
    if (Object.keys(metrics).length > 0) {
      saveCurrentProjectConfig(current => ({
        ...current,
        lastSessionMetrics: metrics,
      }))
    }
  }
  process.on('exit', flush)
  return () => process.off('exit', flush)
}, [store])
```

指标不是每次记录就同步写盘，而是在进程退出时统一 flush。这个决策的逻辑是：**观测不该拖慢热路径**。

但这里有一个明确的故障模式：如果进程被 SIGKILL 强杀，`exit` 事件不会触发，这次会话的指标就丢了。这是一个有意的权衡——宁可偶尔丢数据，也不要每次记录都阻塞主循环。

## 面试指导

读 Claude Code 入口层，这几个问题能区分理解深度：

**问题一：三层 Provider 的顺序能否调换？**

FpsMetrics 和 Stats 的相对顺序不影响功能，因为它们互相不依赖。但任何一个放到 AppStateProvider 里面都会造成问题——那样 AppState 子树里就无法访问观测层了。

**问题二：`onChangeAppState` 为什么从外部注入而不是硬编码？**

测试时可以传入一个什么都不做的 noop，生产里才激活真正的副作用。依赖倒置让 Provider 本身与副作用的具体实现解耦。

**问题三：`stats` 可选参数体现了什么设计模式？**

这是「可选依赖注入 + 自带电池」模式。组件对外声明它可以接受一个实现，但也知道怎么自给自足。可测试性是这个模式最直接的收益。

**问题四：如果你来设计，还有什么值得放在顶层 Provider 里？**

好的答案会提到：认证上下文（Auth）、主题/语言（Theme/i18n）、错误边界（ErrorBoundary）。差的答案会把业务状态也塞进来。区分观测层和业务层的能力，是这里真正在考查的东西。
