# 第十一卷导读：界面不是外壳，而是运行时本体

## 实现链

这一卷把同一条链路从头串到尾：`11-01` 先把终端 agent 讲成一台应用，`11-02` 把 `AppState` 钉成事实世界，`11-03` 让 `useAppState(selector)` 守住订阅纪律，`11-04` 让 `PromptInput` 成为主操纵面，`11-05` 到 `11-06` 继续把 `StatusLine` 和 `TaskListV2` 变成摘要与注意力分配层，`11-07` 到 `11-09` 再把 `Messages`、`buildMessageLookups` 和 `VirtualMessageList` 串成消息编排与长 transcript 读取层，最后才轮到 `11-10` 的 buddy 作为晚到的体验层出现。

这条链路的重点不是“功能很多”，而是每一层都依赖上一层已经把事实、状态和布局立稳：`AppStateStore` 先定义世界，`PromptInput.tsx` 和 `REPL.tsx` 再在这个世界上分配输入、布局和显示，`buddy` 只是最后插进来的可爱层。

## 普通做法

普通写法通常会先讲最显眼的角色，比如吉祥物、彩蛋或一个单独面板，再回头补状态和布局。那样读起来轻松，但会把读者带去错的结论：好像这套系统的价值主要来自“长得像产品”，而不是它如何把界面做成运行时。

## 为什么不用

Claude Code 不这么写，是因为它真正的骨架在前几章：事实世界、选择器订阅、输入面、状态栏、任务面板、消息流水线和虚拟列表先共同成立，buddy 才有资格作为体验层被讨论。把它提前，会把“运行时优先”的主旨改写成“人格化优先”。

## 代价

代价是这卷不能靠一个亮眼角色起手，必须先把抽象关系讲清楚，再让 buddy 进场。换来的好处更大：读者会先理解这台终端为什么是运行时，之后再看 buddy，才会明白它为什么只能是晚一点出现的体验层。
