# AgentTool.tsx 为什么禁止 teammate 再生 teammate？扁平组织的工程理由

在 `AgentTool.tsx` 里有一段判断，翻译成人话就是：

> 如果当前 agent 自己已经是 teammate，禁止它再创建新的 teammate。

这条限制很容易被误读为"技术上还没做好"，但实际上，这是一个有意为之的架构决策。

## 为什么要禁止嵌套

先想象允许嵌套的世界：leader 召唤 researcher，researcher 在工作中觉得需要一个助手，于是再召唤 sub-researcher。sub-researcher 又召唤……

从并发执行的角度看，这没什么问题，甚至很灵活。但从组织制度的角度看，立刻冒出一堆无法回答的问题：

**权限归宿**：sub-researcher 要执行 bash 命令，请求应该给谁批？直接给 leader？还是先给 researcher，researcher 再转给 leader？如果 researcher 正在等待自己的权限批复，这条链会死锁。

**退场顺序**：leader 决定结束任务，它通知 researcher，researcher 要通知 sub-researcher 退出。如果层次更深，这变成了广度优先还是深度优先的协商问题。

**用户心智模型**：用户在 leader 界面看到"3个 teammate 在工作"，但实际上有 3 层嵌套共 7 个 agent。用户以为自己管着 3 个人，其实管着 7 个。

**TeamFile 结构**：`TeamFile.members` 是一个扁平数组，每个成员的 `tmuxPaneId`、`agentId`、`color` 都存在这里。嵌套关系没有对应的数据结构，强行实现就要引入树节点和父子指针，整个数据模型的复杂度爆炸。

## 扁平组织的工程价值

Claude Code 选择的是**扁平组织**：一个 leader，若干 worker，所有 worker 都是 leader 的直接下属，没有中间层。

这不是因为 AI Agent 只能做简单任务，而是因为扁平组织带来了极大的实现简化：

```
权限路径：worker → leader（唯一路径，无歧义）
消息路由：worker ↔ leader（两端直连，mailbox 寻址简单）
退场顺序：leader 通知所有 worker（广播，不需要递归）
状态追踪：TeamFile.members 是数组（O(1) 查找，O(n) 遍历）
```

`AgentTool.tsx` 里判断"是否是 teammate"，用的是 `isTeammate()` 函数。这个函数检查 AsyncLocalStorage 里有没有 `TeammateContext`，或者环境变量里有没有 `CLAUDE_CODE_AGENT_ID`。一旦检测到当前执行上下文已经是 teammate，就拒绝再次进入 team creation 路径。

## 不是能力限制，而是边界主张

有意思的是，从技术实现上，允许嵌套是可以做到的——AsyncLocalStorage 支持嵌套上下文，mailbox 寻址也可以扩展成树状路径。

Claude Code 不做，不是因为做不到，而是因为**不想要**那种复杂性。

这是一个典型的"能力边界"设计——主动声明系统的边界，防止使用者因为"好像也能这么用"而踩进未经设计的组合。

一个对比：Git 也可以把 commit 做成有向图而不是有向无环图，允许循环引用。技术上没障碍，但 Git 明确不支持，因为 DAG 保证了操作的可预测性，循环引用会让合并、回溯、diff 全部变成不可解问题。

Claude Code 禁止 teammate 嵌套，同样的逻辑：**扁平约束换来了整套制度的可预测性**。

## 面试怎么答

如果有人问你"为什么你的多 agent 系统不支持 agent 嵌套"，这是一个好的回答框架：

1. **明确这是设计选择，不是限制**：技术上可以实现，主动不做。
2. **解释为什么扁平更好**：权限路径唯一，退场顺序清晰，状态结构简单。
3. **指出嵌套带来的具体问题**：死锁风险（循环权限等待）、用户心智模型失控、数据结构复杂化。
4. **类比佐证**：工程里许多好的约束不是"做不到"，而是"主动不做"——Git 的 DAG、关系数据库的范式、REST 的无状态原则都是类似的。

**加分项**：能说清楚这个限制让哪些其他部分的设计变得更简单——比如 `TeamFile.members` 能是纯数组、mailbox 的收件人只需要 `agentName@teamName` 两段格式、leader UI 只需要展示一层 worker 列表。好的约束会在全系统产生涟漪式的简化效果。
