# 工程取舍：任务框架为什么把续读和驱逐也揽进来？

`framework.ts` 的职责看起来有点杂：

- `registerTask()`：任务登记
- `updateTaskState()`：状态更新
- `generateTaskAttachments()`：增量续读输出
- `applyTaskOffsetsAndEvictions()`：应用偏移更新和驱逐
- `evictTerminalTask()`：驱逐终态任务
- `pollTasks()`：轮询所有任务

一个文件里既管登记，又管续读，又管驱逐。这看起来像是把不相干的事堆在一起。

## 为什么这不是单纯的"一锅炖"

续读和驱逐之所以在同一个文件里，是因为它们之间的判断**强耦合**，分开反而会出问题。

看 `generateTaskAttachments()` 和 `applyTaskOffsetsAndEvictions()` 这对组合：

```typescript
export async function generateTaskAttachments(state: AppState) {
  const attachments: TaskAttachment[] = []
  const updatedTaskOffsets: Record<string, number> = {}
  const evictedTaskIds: string[] = []

  for (const taskState of Object.values(tasks)) {
    // 终态 + 已通知 = 可驱逐
    if (taskState.notified) {
      switch (taskState.status) {
        case 'completed':
        case 'failed':
        case 'killed':
          evictedTaskIds.push(taskState.id)
          continue  // 跳过续读，直接标记驱逐
        case 'pending':
          continue  // 还没开始，不续读
        case 'running':
          break  // 继续往下做续读
      }
    }

    // 只对 running 任务做增量读取
    if (taskState.status === 'running') {
      const delta = await getTaskOutputDelta(taskState.id, taskState.outputOffset)
      if (delta.content) {
        updatedTaskOffsets[taskState.id] = delta.newOffset
      }
    }
  }

  return { attachments, updatedTaskOffsets, evictedTaskIds }
}
```

这段代码里：
- 决定"要不要续读"的判断依赖 `status` 和 `notified`
- 决定"要不要驱逐"的判断也依赖 `status` 和 `notified`
- 两件事在同一个 `for` 循环里，同一次遍历就都处理了

如果把续读和驱逐拆到两个文件，就需要遍历两次任务列表，或者用某种方式协调两者的判断逻辑。分开写看似更"干净"，实际上会造成冗余遍历和逻辑重复。

## TOCTOU 问题：为什么偏移量更新要两步走

`generateTaskAttachments()` 是异步函数（里面有 `await getTaskOutputDelta()`）。这意味着：

```
时间线：
T0: generateTaskAttachments 开始，读取 state 快照
T1: await getTaskOutputDelta(taskId, offset) 开始磁盘读取
T2: 磁盘读取期间，task 状态改变（比如任务完成，status 变成 completed）
T3: getTaskOutputDelta 返回新的偏移量
T4: applyTaskOffsetsAndEvictions 尝试写回偏移量
```

如果在 T4 直接用 T0 时候的状态快照来更新，会覆盖掉 T2 发生的状态改变，把任务"僵尸化"（offset 更新了，但任务其实已经完成了，应该被驱逐而不是继续跑）。

注释里明确说明了这个问题：

```typescript
/**
 * Apply the outputOffset patches and evictions from generateTaskAttachments.
 * Merges patches against FRESH prev.tasks (not the stale pre-await snapshot),
 * so concurrent status transitions aren't clobbered.
 */
export function applyTaskOffsetsAndEvictions(
  setAppState,
  updatedTaskOffsets,
  evictedTaskIds,
) {
  setAppState(prev => {
    for (const id of offsetIds) {
      const fresh = newTasks[id]
      // Re-check status on fresh state — task may have completed during the await.
      // If it's no longer running, the offset update is moot.
      if (fresh?.status === 'running') {
        newTasks[id] = { ...fresh, outputOffset: updatedTaskOffsets[id]! }
      }
    }
    // ...
  })
}
```

关键：在 `setAppState` 回调里重新拿 `prev.tasks[id]`（最新状态），而不是用 `generateTaskAttachments` 读到的快照。在更新前再次确认任务还是 `running` 状态。

这个两步走的设计，正是为了解决异步读取和状态更新之间的竞争条件。如果续读和驱逐在两个不同的文件/函数里，这个细节就很容易被遗漏，产生难以复现的并发 bug。

## PANEL_GRACE_MS 和 STOPPED_DISPLAY_MS：统一的时间常量

`framework.ts` 里还定义了两个时间常量：

```typescript
export const STOPPED_DISPLAY_MS = 3_000   // 3 秒：killed 任务显示时间
export const PANEL_GRACE_MS = 30_000      // 30 秒：终态 agent 的面板宽限期
```

这些常量同时被续读逻辑和驱逐逻辑使用。如果分散到不同文件，就需要在某个"公共常量文件"里维护它们，或者两个文件各自定义相同的数值（这更糟糕）。

## 面试要点

"单一职责原则"（SRP）的常见误用，是把"一个文件只做一件事"理解为"把所有相关判断拆开放到不同文件"。

更准确的 SRP 是：**一个模块只有一个改变的理由**。`framework.ts` 的改变理由是"任务生命周期管理规则改变"——续读规则、驱逐规则、时间常量都是这个领域的规则，它们一起改变才符合逻辑。

如果把续读拆到 `taskReader.ts`，驱逐拆到 `taskEvictor.ts`，时间常量拆到 `taskConstants.ts`，你得到的不是更清晰的代码，而是一个需要同时修改三个文件才能调整"已完成任务多久后被驱逐"这样一个规则的系统。

强耦合的逻辑放在一起，不是懒，是识别了正确的内聚边界。
