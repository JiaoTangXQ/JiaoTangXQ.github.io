# 任务登记系统为什么还负责续读和驱逐

如果任务框架只负责“写进去”，那它只是个存表工具；Claude Code 让它同时负责续读和驱逐，说明它把框架当成完整生命周期层。

## 实现链

`framework.ts` 里除了 `registerTask()`，还有 `generateTaskAttachments()`、`applyTaskOffsetsAndEvictions()`、`evictTerminalTask()`。它会基于 `outputOffset` 续读新输出，基于 `status/notified/retain/evictAfter` 决定能不能驱逐。`PANEL_GRACE_MS` 和 `STOPPED_DISPLAY_MS` 也在这里统一定义。

这表示任务框架不只是“把对象放进 AppState”，而是在替所有任务维护进入、停留、退出的共同制度。

## 普通做法

普通做法通常把这些事分散处理：登记归状态层，日志续读归输出层，驱逐归 UI 层。每层只管自己那一小块。

## 为什么不用

Claude Code 不这么分，是因为这里的判断互相缠在一起。一个任务能不能被驱逐，要同时看终态、通知、保留、宽限期；一个输出能不能续读，又依赖任务还在不在。把它们拆散，最后很容易彼此打架。

所以它宁可把这部分放在同一层，承认这是“生命周期框架”的责任。

## 代价

代价是框架层会显得偏大，也更容易被误解成“什么都管”。但从代码看，这不是懒得分层，而是有意把一组强耦合的生命周期判断放在一起。
