# 为什么同一个命令在不同入口下不应该被视为同一能力

人很容易觉得命令名就是能力本身。好像 `/files` 无论在哪里触发，都是“列出文件”；`/config` 无论在哪里触发，都是“打开配置”。Claude Code 明显不接受这种想法。

源码里至少有两层边界在改写命令。远端模式有 `REMOTE_SAFE_COMMANDS`，它会在 REPL 渲染前先筛掉那些不该在远端现场出现的命令；bridge 入口又有 `BRIDGE_SAFE_COMMANDS` 和 `isBridgeSafeCommand()`，它按另一套标准决定从手机或网页进来的命令能不能执行。更关键的是，bridge 还把 `prompt`、`local`、`local-jsx` 区分对待：`prompt` 默认安全，`local` 需要显式放行，`local-jsx` 一律拦下。

这说明“命令名相同”不等于“能力相同”。一条命令进入系统时，还背着入口、执行现场、是否有本地 UI、是否依赖本地文件系统、是否适合远端控制这些条件。离开这些条件，只谈命令名字，得到的只是一个过于天真的抽象。

Claude Code 在这里非常老练。它不让“统一命令系统”变成“假装所有入口都一样”。真正成熟的边界意识，就是承认同一个名字到了不同世界里，本来就应该被重新定义。

## 实现链
同一串 `/command` 文本，在本地、remote、bridge 入口并不一定走同一条链。`effectiveSkipSlash`、`bridgeOrigin`、命令 allowlist 和 fork 机制会一起改写它最后被当成什么能力。

## 普通做法
很多系统会强调“同名命令 everywhere 一致”，把一致性放在第一位。

## 为什么不用
因为入口背后的宿主条件不同。它宁可承认“同名不等于同能力”，也不愿用一层假一致把本地 UI、远端桥和主线程任务全混在一起。

## 代价
用户需要学习入口差异，不能只记命令名；但这至少让能力边界和宿主现实保持一致。
