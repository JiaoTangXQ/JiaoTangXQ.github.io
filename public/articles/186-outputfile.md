# outputFile：后台任务为什么要把过程写进文件而不是只在内存里跑

面试题版本：**"Claude Code 的 TaskStateBase 里有 outputFile 字段，后台任务会把输出写到磁盘文件。为什么不直接在内存里保留结果？这个设计解决了什么具体问题？"**

## 两种方案的对比

**方案 A：内存保留**
```typescript
// 任务完成后只在内存里
const taskResults = new Map<string, string>()
taskResults.set(taskId, fullOutput)
```

**方案 B：outputFile**
```typescript
// Task 对象里
{
  outputFile: '/tmp/claude/tasks/b_abc123.log',
  outputOffset: 2048,  // 已经读取到的位置
}
```

`src/Task.ts` 里 `getTaskOutputPath()` 函数生成输出文件路径，任务类型前缀（`b` 表示 bash，以此类推）确保文件名不冲突。

## outputFile 解决的具体问题

**问题一：进程重启后的数据恢复**

Claude Code 主进程崩溃或被用户强制退出时，内存里的所有数据消失。如果后台任务的输出只在内存里，任务要么重跑，要么永久丢失。

outputFile 把任务输出变成磁盘上的持久记录。下次启动时，系统可以找到未读完的 outputFile，从 `outputOffset` 位置继续读取，用户看到的是"任务完成，结果如下"而不是"找不到任务记录"。

**问题二：长输出的流式消费**

一个 bash 任务可能产生几 MB 甚至几十 MB 的输出。如果等任务完成再一次性拿到所有输出，主会话在等待期间什么都看不到。

outputFile + outputOffset 支持流式读取：主会话可以定期读取文件里新增的内容，类似 `tail -f` 的效果，任务还在跑时用户就能看到实时输出。

**问题三：跨会话的任务接续**

用户可以开一个后台任务，关掉当前会话，过一段时间再回来看结果。这在 remote agent 任务场景下很常见——远端机器在跑一个长时间 CI 任务，用户不需要保持连接。

通过 outputFile，任务的所有输出都在磁盘上，用户回来时可以读取完整的任务历史，而不是只有"任务已完成"这个通知。

## 外部痕迹的系统意义

"过程留成外部痕迹"这个表述的工程含义是：任务的执行过程不只对当前会话有意义，它对未来的任何会话都应该可查。

这与数据库的 WAL（Write-Ahead Log）有类似的思路：不是只记最终状态，而是把状态变化的过程也记录下来，让系统具备时间旅行能力。

在 Claude Code 里，这意味着：
- 任务 A 在会话 1 里开始运行
- 会话 1 结束（主动或意外）
- 会话 2 启动，可以通过 outputFile 看到任务 A 的所有历史输出
- 用户可以从任意历史位置重新理解任务的执行过程

## 代价和边界

outputFile 方案的代价：
1. **磁盘 I/O**：每次写输出都有磁盘写入开销。高频小输出时可以用写缓冲减轻，但仍然比纯内存慢。
2. **清理责任**：文件需要被清理，否则会积累大量历史任务文件。Claude Code 在 `isTerminalTaskStatus()` 返回 true 后有对应的清理路径。
3. **路径管理**：需要维护一套文件命名规则，确保不同任务、不同会话的文件不冲突。`getTaskOutputPath()` 里的任务 ID 前缀体系处理了这个问题。

什么时候不值得用 outputFile？短暂的确定性操作，比如读一个文件的内容、查询一次 API，结果小、生命周期短，不需要历史追溯。Claude Code 在这类场景下直接返回工具结果，不写 outputFile。

## 面试角度的总结

outputFile 的核心价值是：**把任务执行过程从会话生命周期里解耦出来**。这让任务的结果可以跨越进程重启、会话切换、甚至网络断线而保持可查。代价是磁盘 I/O 和文件管理的额外复杂度，但对于需要可接续、可追溯的后台任务来说，这个代价是必要的。
