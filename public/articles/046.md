# LSPTool：Claude Code 如何集成 Language Server Protocol 做代码分析？

LSP（Language Server Protocol）是编辑器和语言分析工具之间的标准协议。Claude Code 通过 `LSPTool` 把这套协议暴露给 Claude，让模型可以直接调用 Go to Definition、Find References、Hover 等编辑器级操作。

## LSPTool 支持的操作集

```typescript
operation: z.enum([
  'goToDefinition',
  'findReferences',
  'hover',
  'documentSymbol',
  'workspaceSymbol',
  'goToImplementation',
  'prepareCallHierarchy',
  'incomingCalls',
  'outgoingCalls',
])
```

9 种操作覆盖了大部分静态分析场景：
- `goToDefinition`：跳转到符号定义
- `findReferences`：查找所有引用
- `hover`：获取光标处的类型信息
- `documentSymbol`：列出文件中所有符号（函数、类、变量）
- `workspaceSymbol`：全工作区符号搜索
- `goToImplementation`：跳转到接口实现
- `prepareCallHierarchy` / `incomingCalls` / `outgoingCalls`：调用链分析

这些操作让 Claude 能做到"这个函数在哪里被调用"、"这个接口有哪些实现"、"这个变量的类型是什么"——这些信息靠 grep 很难准确获取，LSP 给出的是经过语义解析的精确答案。

## 文件大小限制

```typescript
const MAX_LSP_FILE_SIZE_BYTES = 10_000_000  // 10MB
```

LSP 操作需要把文件内容发给语言服务器，10MB 是单个文件的上限。超过这个大小会拒绝请求，避免语言服务器被超大文件卡死。

## 初始化流程：等待 LSP 连接

```typescript
import {
  getInitializationStatus,
  getLspServerManager,
  isLspConnected,
  waitForInitialization,
} from '../../services/lsp/manager.js'
```

LSP 服务器启动是异步的。Claude 可能在 LSP 初始化完成前就尝试调用 LSPTool，工具需要等待连接就绪：

- `isLspConnected()`：快速检查是否已连接（用于早期失败路径）
- `waitForInitialization()`：挂起等待直到连接建立
- `getInitializationStatus()`：获取初始化状态（用于错误提示）

## LSPDiagnosticRegistry：异步诊断收集

一个有趣的设计是诊断信息（错误、警告）的处理方式。LSP 服务器不是在工具调用时同步返回诊断——而是在任何时候主动推送 `textDocument/publishDiagnostics` 通知。

`LSPDiagnosticRegistry` 解决了这个问题：

```typescript
// 待发送的诊断信息
const pendingDiagnostics = new Map<string, PendingLSPDiagnostic>()

// 已发送的诊断，用于去重（LRU 限制内存）
const deliveredDiagnostics = new LRUCache<string, Set<string>>({
  max: MAX_DELIVERED_FILES,
})
```

流程是：
1. LSP 服务器推送 `publishDiagnostics`
2. `registerPendingLSPDiagnostic()` 把诊断存入 `pendingDiagnostics`（key 是 UUID，避免快速注册时的 key 冲突）
3. 下次 Claude 发请求时，`getAttachments()` 调用 `checkForLSPDiagnostics()` 取出 pending 诊断，作为附件注入对话

这和 `AsyncHookRegistry` 是同样的模式（注释里明确提到了）：把异步到达的信息暂存，在下一个"对话输入点"统一注入。

## 去重机制：跨对话轮次

```typescript
// 去重 key = 诊断消息 + 严重程度 + 行范围 的哈希
const deliveredDiagnostics = new LRUCache<string, Set<string>>({
  max: MAX_DELIVERED_FILES,
})
```

同一个诊断不应该在每次对话轮次都重复展示。`deliveredDiagnostics` 记录已经发送过的诊断（文件 URI → 诊断 key 集合），避免 Claude 反复看到相同的错误。

用 `LRUCache` 而不是普通 `Map` 是为了控制内存：长会话里可能积累很多文件的诊断记录，LRU 在达到 500 个文件时自动淘汰最久未访问的。

## 体积限制

```typescript
const MAX_DIAGNOSTICS_PER_FILE = 10
const MAX_TOTAL_DIAGNOSTICS = 30
```

一个文件最多传 10 条诊断，总共最多 30 条。这防止了"一个文件有 500 个编译错误"时把 Claude 的上下文塞满。超出限制的诊断被截断。

## symbolContext：符号上下文提取

`symbolContext.ts` 提供了一个辅助工具——在 LSP 结果里，位置信息是行号+列号，但 Claude 需要的是符号名称、周围代码等上下文。`symbolContext` 把 LSP 的位置信息转换成可读的代码上下文片段。

## 面试角度

LSPTool 的设计展示了几个集成外部协议的工程模式：

1. **异步推送 vs. 同步请求的分离**：诊断是异步推送的，查询是同步请求的。Registry 模式把两者解耦，统一在对话轮次边界注入
2. **LRU 控制内存边界**：长会话里无界增长的数据结构（已发送诊断记录）需要有限制。LRU 是适合"最近访问比历史记录更重要"场景的策略
3. **去重比完整性更重要**：重复诊断是噪声，去重是用户体验的保障。但去重 key 的设计（消息+严重度+范围）要能区分"同一位置的不同错误"
4. **体积限制是必须的**：不加限制地传所有诊断会导致 context 窗口被占满，Claude 读不到实际代码。截断并告知"还有更多"是合理的取舍
