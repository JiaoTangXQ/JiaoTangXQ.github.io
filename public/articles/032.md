# MCP 客户端支持三种传输协议：stdio、SSE、StreamableHTTP 各适合什么场景？

MCP（Model Context Protocol）服务器可以部署在各种地方——本地进程、局域网服务器、远程 API。Claude Code 的 MCP 客户端（`services/mcp/client.ts`）通过三种传输层适配这些场景：

```typescript
import { StdioClientTransport } from '@modelcontextprotocol/sdk/client/stdio.js'
import { SSEClientTransport } from '@modelcontextprotocol/sdk/client/sse.js'
import { StreamableHTTPClientTransport } from '@modelcontextprotocol/sdk/client/streamableHttp.js'
```

## stdio 传输：本地进程通信

`StdioClientTransport` 通过标准输入/输出与 MCP 服务器通信。服务器是一个本地进程，Claude Code 通过 `spawn` 启动它，用 stdin 发送请求，从 stdout 读取响应。

**适用场景**：
- 本地工具（git 操作、本地代码分析）
- 需要访问本地文件系统的工具
- 低延迟要求（进程内通信，无网络开销）

**特点**：服务器生命周期由 Claude Code 管理（启动、关闭），进程间隔离，无法跨机器部署。

## SSE 传输：服务器推送事件

`SSEClientTransport` 用 HTTP 连接远程 MCP 服务器：请求用 HTTP POST，响应用 Server-Sent Events（SSE）流。SSE 是 HTTP 长连接，服务器可以主动推送事件到客户端。适合"请求发出后响应可能需要一段时间，且响应是流式的"场景。

**适用场景**：远程工具 API、需要流式输出的工具、跨机器部署。

**特点**：服务器独立运行，支持认证（HTTP headers），网络连接有中断风险。

## StreamableHTTP 传输：现代替代方案

`StreamableHTTPClientTransport` 用 HTTP chunked transfer encoding 代替 SSE。

**与 SSE 的区别**：
- SSE 连接是持久的（一直保持），StreamableHTTP 是请求级别的流
- StreamableHTTP 更容易通过代理和防火墙（标准 HTTP）
- 不需要特殊的 `text/event-stream` Content-Type 支持

## OAuth 认证集成

`services/mcp/auth.ts` 处理 MCP 服务器的 OAuth 认证：

```typescript
const authTool = createMcpAuthTool(serverName, serverConfig)
// 工具列表里会出现 "__mcp_auth_..." 工具
// 让模型知道需要先完成认证
```

当 MCP 服务器返回 401 时，会触发 OAuth 流程，让用户完成授权，然后重试。这套流程集成在工具调用层，对模型透明。

## Elicitation：MCP 服务器主动请求用户输入

`services/mcp/elicitationHandler.ts` 处理一个特殊的 MCP 功能：Elicitation（主动询问）。

当 MCP 服务器需要用户提供额外信息时，它可以发送 `elicit` 请求（错误码 `-32042`）。Claude Code 捕获这个请求，展示 UI 询问用户，然后把用户的回答发回给 MCP 服务器：

```typescript
export interface ElicitationRequestEvent {
  serverName: string
  params: ElicitRequestURLParams
  signal: AbortSignal
  resolve: (result: ElicitResult) => void
}
```

这让 MCP 服务器可以实现"需要用户配置 API key"、"需要用户选择分支"这样的交互式工具，而不只是被动执行命令。

## 面试角度

三种传输协议的选择是典型的系统设计题：

1. **stdio**：最简单，最低延迟，但只能本地用；适合"Claude Code 自己管理服务器生命周期"的场景
2. **SSE**：成熟的推送方案，但连接是持久的，代理处理麻烦；适合需要服务器主动推送进度的长任务
3. **StreamableHTTP**：更现代，更容易通过网络基础设施，但是较新的标准，老版本 MCP 服务器可能不支持

在实际工程里，选传输层的依据：部署位置（本地/远程）+ 数据流向（单次响应/流式推送）+ 网络环境（直连/代理）。
