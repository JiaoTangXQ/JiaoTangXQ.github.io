# 第十三卷回收索引：外延执行里的工作台自然外延与浏览器现场

## 实现链
这页回收的实现链，主要落在 `computerUse/setup.ts`、`claudeInChrome/setup.ts`、`mcpServer.ts`、`prompt.ts`、wrapper/manifest/socket/bridge 这些接入链上。它真正解释的是：桌面和浏览器能力进入系统时，为什么先被翻译成原工作台听得懂的制度语言。

## 普通做法
更直觉的做法，是把浏览器、桌面或设备能力单独做成一套旁路子系统：自己管理协议、自己定义命令、自己做权限和恢复。

## 为什么不用
Claude Code 没这么做，因为一旦新世界各说各话，主会话、权限、日志和恢复就会裂成多套口径。它宁可多写 wrapper、manifest、bridge 和统一命名，也要把外延能力接回原骨架。

## 代价
代价是跨平台接入链明显更笨重，接缝件更多，理解门槛更高，而且平台债会持续积累。
