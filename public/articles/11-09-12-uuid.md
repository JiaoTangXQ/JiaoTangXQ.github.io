# unseenDivider 用 UUID 前缀而不是索引定位，为什么坐标不能是计数？

在会话恢复（resume）场景里，Claude Code 需要在 transcript 里标记一条线：线上方是上次离开时已看过的内容，线下方是新内容。这条线的定位是用 UUID 前缀做的，不是索引。

## 为什么索引会失效

假设上次看到的最后一条消息是 `messages[150]`，存下这个索引 150 作为分割线位置。下次打开时：

1. transcript 有 200 条消息了，前 150 条和原来一样——索引 150 还是正确的
2. 但如果之前的某些工具调用被重新 grouped（新版本的 grouping 规则变化），消息 #148 和 #149 被合并成一条 grouped_tool_use，总消息数变成 199，分割线应该在 #149，但我记录的是 150——偏了一位
3. 更严重的情况：compact（压缩早期消息），前 100 条被替换成一条 summary，现在索引 150 指向的是原来的第 250 条，完全错了

索引是"这个数组里的第 N 个"，但数组会被重新构建。

## UUID 作为稳定标识

每条消息有一个 `uuid` 字段，在消息创建时生成，不随任何处理变化。消息被 normalize、group、reorder、compact 之后，uuid 仍然不变（group 对象用组内第一条的 uuid 作为自己的 uuid，compact 前的每条消息的 uuid 会被记录在 summary 里）。

`unseenDivider.firstUnseenUuid` 存的是"第一条未看过消息的 uuid 的前缀"（不是完整 uuid，用前缀节省存储并保持一定的容错性）。

定位分割线的算法：

```typescript
function findDividerBeforeIndex(
  messages: RenderableMessage[],
  firstUnseenUuid: string
): number {
  for (let i = 0; i < messages.length; i++) {
    if (messages[i]!.uuid.startsWith(firstUnseenUuid)) {
      return i  // 分割线在这条消息之前
    }
  }
  return -1  // 未找到（可能被 compact 掉了）
}
```

每次 transcript 更新后，重新用 uuid 前缀扫描当前消息数组，找到分割线的新索引位置。这个过程 O(n)，但只在 transcript 初始化或消息数组重建时运行，不在每次渲染时运行。

## 前缀匹配的原因

为什么用前缀而不是完整 uuid？

UUID 格式是 `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`（36 字符）。存储 36 字符的精确匹配当然可以，但 Claude Code 有时候在不同上下文里对 uuid 做截断处理（比如 `deriveUUID` 用前 24 位）。

用前缀匹配（比如前 8 位）可以：
1. 减小存储体积（配置文件里只存短字符串）
2. 对 uuid 生成规则的细微变化更鲁棒（只要前缀相同就能匹配）
3. 在 compact 后，summary 保留了原消息 uuid 的关键部分，前缀仍然可以匹配

当然前缀越短碰撞概率越高。前 8 位十六进制字符（2^32 种可能）在单个会话里的碰撞概率极低（一个会话不会有 2^32 条消息），实际上是安全的。

## 面试指导

这道题考察"标识符的稳定性"——什么属性可以作为跨越数据变换的稳定锚点？

**核心原则**：如果数据会经历变换（排序、分组、压缩、重新规范化），不能用变换前的位置（索引）作为标识符，要用变换不会改变的属性（内容哈希、业务 ID、专用 UUID）。

**类比**：Git 的 commit 哈希。代码 rebase 后，commit 的"在历史里的位置"（相对于 HEAD 的偏移）会变，但 commit 哈希不变。如果你 `git cherry-pick` 时用 "3rd commit from base branch" 这样的位置描述，rebase 后位置就错了。用 commit hash 才能跨越 rebase 稳定定位。

**延伸问题**：compact 操作把前 N 条消息替换成 summary，firstUnseenUuid 指向的消息如果被 compact 进了 summary，`findDividerBeforeIndex` 会返回 -1。这时应该把分割线放在哪？答：放在 summary 消息之后（即所有历史都视为已读），因为用户在上次会话里确实看过这些内容。-1 表示"没找到分割线，全部视为已读"，这是合理的降级行为。
