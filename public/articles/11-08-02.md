# 语义很多时候不在单条消息里，而在几条消息之间

## 实现链
`Messages.tsx` 不是把原始 transcript 直接丢给 `Message`，而是先经过 `normalizeMessages`、过滤、`reorderMessagesInUI`、`applyGrouping` 和折叠，最后再把这份已经重排过的可视消息交给 `buildMessageLookups`。因此 `MessageRow` 读到的 `progressMessagesForMessage`、`siblingToolUseIDs` 和静态化判断，都是同一份 lookup 推出来的结果；`UserToolResultMessage` 则用 `toolUseByToolUseID` 找回工具定义，`HookProgressMessage` 用 hook 计数判断是否还在跑。

## 普通做法
更普通的做法，是让每个组件只看自己手头那一行数据，或者在需要的时候临时回扫整个数组。这样做表面上更直观，但每个组件都会重新解释一遍“这条消息和谁有关”。

## 为什么不用
Claude Code 这里面对的不是纯 transcript，而是会被分组、折叠、截断、补入 synthetic streaming 行的运行时视图。语义经常跨行存在，甚至要同时看 assistant、user、progress 和 attachment 才能判断。让每个组件各自解释，最后只会得到几套不一致的答案。

## 代价
共享 lookup 让代码抽象了一层，也让很多判断必须依赖同一个中间结构。代价是心智模型没那么“直”，但换来的是整个消息区始终说同一种语义语言。
