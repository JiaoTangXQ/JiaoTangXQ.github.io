# 第十三卷回收索引：外延执行里 Native Host、manifest、目录与注册表本来就是产品接入链的一部分

## 实现链
外延执行里真正难的，不是把一个功能点亮，  
而是把它放进真实系统以后，底座会不会先长出来。

这条线真正逼出来的，是第十三卷里关于产品接入现实的判断。  
Native Host、manifest、目录和注册表都不是边角料，它们就是接入链的一部分。

这页实际回收的实现链是：
- `12-06-01` 把真正难点写成 Native Host、manifest、不同浏览器目录、Windows 注册表、扩展检测和自动重连页一起把浏览器接进来，这会在 `13-06-01` 里回收到“产品级接入链本来就会长出很多笨重但必要的底座”。

很多接入底座看起来并不轻巧。  
但真正成熟的产品接入，不会先假装它们不存在。

它会先承认 Native Host、manifest、目录和注册表本来就是产品接入链的一部分。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
