# pending → running → completed/failed/killed：5 个状态为什么够用？

```typescript
export type TaskStatus =
  | 'pending'
  | 'running'
  | 'completed'
  | 'failed'
  | 'killed'
```

5 个状态。没有 `paused`，没有 `backgrounded`，没有 `suspended`，没有 `cancelling`。

Claude Code 支持任务前后台切换、用户中途取消、父任务终止时子任务连带被 kill、任务完成后宽限期展示……这些场景看起来复杂，却都被 5 个状态涵盖了。为什么？

## 数字分析：每个状态的实际作用

**`pending`**：任务已注册但尚未开始执行。这个状态在 Claude Code 里很短暂——大多数任务注册时会立刻设成 `running`。但它有语义价值：表示任务在系统里有 ID，但执行资源（`AbortController`）还没准备好。用于"先占位、后开始"的场景。

**`running`**：任务正在执行。这是最复杂的状态，几乎所有有意思的事情都在这个状态里发生：进度更新、输出写入、消息注入、前后台切换。

注意：**"后台运行"不是一个独立的状态**。前台运行和后台运行都是 `running`，区别在 `LocalAgentTaskState.isBackgrounded` 这个字段（不在 base type 里）。这是一个刻意的设计选择。

**`completed`**：正常完成。任务的 `result` 字段有返回值，`endTime` 已记录。

**`failed`**：执行出错。`error` 字段有错误信息。

**`killed`**：被外部强制终止（用户按 ESC、父任务被 kill 连带触发、批量 kill）。

## 为什么 `backgrounded` 不是一个状态

最直觉的问题：后台任务和前台任务明明不一样，为什么不加一个 `backgrounded` 状态？

因为前后台区分的是**执行环境**，不是**任务生命周期**。一个任务从前台切到后台时：
- 任务还在 `running`
- 输出还在往 `outputFile` 写
- `AbortController` 还在监听

切换的只是"UI 是否显示这个任务的详细输出"，以及"执行循环是否有一个 `backgroundSignal` 在等待"。这些是执行层的细节，不是生命周期层的状态。

把 `backgrounded` 加进 `TaskStatus`，反而会造成逻辑混乱：

- `isTerminalTaskStatus('backgrounded')` 应该返回什么？
- `backgrounded` 的任务能不能再切回 `running`？还是需要额外的 `foregrounding` 状态？
- kill 一个 `backgrounded` 的任务，是先切到 `running` 再 kill，还是直接跳到 `killed`？

这些问题的答案都是"后台任务和前台任务其实是同一个 running 任务"，所以 `backgrounded` 不应该成为状态。

## isTerminalTaskStatus：全系统的终态守卫

```typescript
export function isTerminalTaskStatus(status: TaskStatus): boolean {
  return status === 'completed' || status === 'failed' || status === 'killed'
}
```

注释："Used to guard against injecting messages into dead teammates, evicting finished tasks from AppState, and orphan-cleanup paths."

这个函数在代码库里被调用了多次，每次调用背后都有一个场景：

**消息注入保护**：
```typescript
// InProcessTeammateTask 里（推断）
if (isTerminalTaskStatus(task.status)) {
  return  // 不向已结束的 teammate 注入消息
}
```

**驱逐判断**：
```typescript
// framework.ts
export function evictTerminalTask(taskId, setAppState) {
  setAppState(prev => {
    const task = prev.tasks?.[taskId]
    if (!isTerminalTaskStatus(task.status)) return prev  // 未结束，不驱逐
    if (!task.notified) return prev  // 通知未发，不驱逐
    // ... 检查 evictAfter
    delete newTasks[taskId]  // 从内存删除
  })
}
```

**轮询跳过**：
```typescript
// generateTaskAttachments
if (taskState.notified) {
  switch (taskState.status) {
    case 'completed':
    case 'failed':
    case 'killed':
      evictedTaskIds.push(taskState.id)  // 标记可驱逐
      continue
    // ...
  }
}
```

这个单一函数是系统里"什么叫结束"的唯一真相来源。

## 状态转换的合法路径

```
pending → running → completed
                  → failed
                  → killed
```

没有从终态回到非终态的路径。一旦任务进入 `completed`/`failed`/`killed`，状态就不会再变。这是每个终止函数里的幂等检查保证的：

```typescript
updateTaskState<LocalAgentTaskState>(taskId, setAppState, task => {
  if (task.status !== 'running') return task  // 不是 running 就不改
  // ...修改状态到终态
})
```

这意味着多次调用 `killAsyncAgent(taskId, setAppState)` 是安全的——第二次调用时任务已经是 `killed`，检查 `task.status !== 'running'` 会直接返回，什么都不做。

## 面试要点

设计状态机时的一个常见错误：**把执行方式混入生命周期状态**。

`backgrounded`、`paused`、`suspended` 这些词描述的是任务的执行方式，不是任务是否还在"进行中"。把它们加进状态机会导致状态图指数级膨胀（`running-foreground`、`running-background`、`paused-foreground`...）。

正确的做法是：状态机只包含生命周期节点（`pending → running → [终态]`），执行方式的变化用独立字段（`isBackgrounded`）记录，不影响状态机的主干结构。

5 个状态够用的原因，是这 5 个状态代表了任务生命周期的 5 个真实节点，没有把"执行方式"混进来。
