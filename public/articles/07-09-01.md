# mcpSkillBuilders.ts：一个只有 40 行的文件在做什么？依赖图保险丝

## 工程考古：一个"什么都没做"的模块

`src/skills/mcpSkillBuilders.ts` 完整源码如下（精简版）：

```typescript
import type {
  createSkillCommand,
  parseSkillFrontmatterFields,
} from './loadSkillsDir.js'

export type MCPSkillBuilders = {
  createSkillCommand: typeof createSkillCommand
  parseSkillFrontmatterFields: typeof parseSkillFrontmatterFields
}

let builders: MCPSkillBuilders | null = null

export function registerMCPSkillBuilders(b: MCPSkillBuilders): void {
  builders = b
}

export function getMCPSkillBuilders(): MCPSkillBuilders {
  if (!builders) {
    throw new Error(
      'MCP skill builders not registered — loadSkillsDir.ts has not been evaluated yet',
    )
  }
  return builders
}
```

这个文件：
- 不导入任何具体实现（只导入类型）
- 不做任何数据处理
- 不包含任何业务逻辑
- 只有一个 `let builders` 变量和两个操作它的函数

为什么这 40 行代码值得单独成一个文件？

---

## 循环依赖的成因

要理解这个模块的存在，先要理解它解决的问题。

如果没有 `mcpSkillBuilders.ts`，MCP skill 相关的依赖图会是：

```
client.ts
  → mcpSkills.ts
      → loadSkillsDir.ts
          → (众多模块)
              → ... → client.ts  ← 循环！
```

具体来说：
- `client.ts` 需要 `mcpSkills.ts`（获取 MCP skills）
- `mcpSkills.ts` 需要 `loadSkillsDir.ts`（调用 `createSkillCommand`）
- `loadSkillsDir.ts` 在某条路径上需要 `client.ts` 的类型（MCP 相关）

这条循环会导致模块加载顺序问题：Node.js 处理循环依赖时，有时会得到一个部分初始化的模块，导致"类型是 undefined"这类诡异错误。

---

## 保险丝的工作原理

`mcpSkillBuilders.ts` 的注释解释了它的存在：

```typescript
/**
 * Write-once registry for the two loadSkillsDir functions that MCP skill
 * discovery needs. This module is a dependency-graph leaf: it imports nothing
 * but types, so both mcpSkills.ts and loadSkillsDir.ts can depend on it
 * without forming a cycle (client.ts → mcpSkills.ts → loadSkillsDir.ts → …
 * → client.ts).
 *
 * The non-literal dynamic-import approach ("await import(variable)") fails at
 * runtime in Bun-bundled binaries — the specifier is resolved against the
 * chunk's /$bunfs/root/… path, not the original source tree, yielding "Cannot
 * find module './loadSkillsDir.js'".
 */
```

关键词：**dependency-graph leaf**（依赖图叶子节点）。

`mcpSkillBuilders.ts` 不导入任何具体实现，只导入类型（`import type`）。类型在编译后消失，不产生运行时依赖。所以它在运行时的依赖图里是一个**叶子节点**——没有任何模块依赖链从它往下延伸。

新的依赖图：

```
client.ts
  → mcpSkills.ts
      → mcpSkillBuilders.ts  ← 叶子节点，不往下连

loadSkillsDir.ts
  → mcpSkillBuilders.ts  ← 同一个叶子节点，在启动时注册 builders

启动顺序：
1. loadSkillsDir.ts 被加载（通过 commands.ts 的静态 import）
2. loadSkillsDir.ts 模块初始化时调用 registerMCPSkillBuilders(...)
3. 后来 client.ts 连上 MCP server 时，通过 getMCPSkillBuilders() 取出构建器
```

循环被打断了。

---

## 为什么不用动态 import？

一个直觉上更简单的方案：

```typescript
// mcpSkills.ts 里
const { createSkillCommand } = await import('./loadSkillsDir.js')
```

但注释说明了为什么不行：

> "The non-literal dynamic-import approach fails at runtime in Bun-bundled binaries — the specifier is resolved against the chunk's `/$bunfs/root/…` path"

在 Bun 打包的二进制文件里，变量形式的动态 import（`import(variable)`）会失败，因为打包时路径被编译进了特殊格式（`/$bunfs/root/...`），运行时无法解析。

字面量形式的动态 import（`import('./loadSkillsDir.js')`）可以工作，但会在依赖分析工具（dependency-cruiser）里产生新的循环依赖违规。

---

## 注册式模式的代价：初始化顺序约束

```typescript
export function getMCPSkillBuilders(): MCPSkillBuilders {
  if (!builders) {
    throw new Error(
      'MCP skill builders not registered — loadSkillsDir.ts has not been evaluated yet',
    )
  }
  return builders
}
```

这个 `throw` 说明：`mcpSkillBuilders.ts` 引入了一个**初始化顺序约束**。调用 `getMCPSkillBuilders()` 之前，`registerMCPSkillBuilders()` 必须已经被调用过。

如果顺序错了（如在 `loadSkillsDir.ts` 被加载之前就有 MCP server 连接），系统会在第一次尝试使用 MCP skills 时立即抛出明确的错误，而不是默默失败或产生诡异的 undefined 行为。

早失败、早暴露——这是保险丝设计的另一个优点。

## 面试指导

**"如何处理模块间的循环依赖？"**

常见方案：
1. **重构依赖关系**：把两个模块都依赖的公共部分抽出去（本文的方案）
2. **延迟 require**（Node.js 里）：在函数里 `require`，而不是在顶层，等到真正需要时才加载
3. **依赖注入**：把依赖作为参数传进来，而不是直接 import
4. **事件系统**：A 不直接依赖 B，而是监听 B 发出的事件

`mcpSkillBuilders.ts` 的方案本质上是"注册式依赖注入的变体"：A 不导入 B，而是 B 主动把自己注入到一个中间模块，A 从中间模块取出来用。

这个模式在以下情况特别有用：
- 循环依赖涉及的模块都很大，重构成本高
- 依赖的使用时机（运行时，MCP server 连接后）远晚于注册时机（启动时）
- 打包工具（Bun）对动态 import 有特殊限制
