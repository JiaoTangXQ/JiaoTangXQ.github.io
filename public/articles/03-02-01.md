# 面试题：粘贴一张截图，Claude Code 做了哪些你看不见的工作？

## 场景推演

用户在终端里按下 Cmd+V，粘贴了一张 4MB 的 Retina 截图，然后输入「帮我看这个错误」。

从这一刻到模型收到消息，`processUserInputBase` 里发生了至少四件事：

---

## 第一步：storeImages — 把临时像素变成磁盘文件

```typescript
const storedImagePaths = pastedContents
  ? await storeImages(pastedContents)
  : new Map<number, string>()
```

`storeImages()` 把剪贴板里的图片内容写到磁盘上，返回一个 `id → path` 的 Map。

为什么要落盘？图片不只是给模型「看一眼」的临时数据。一旦存到磁盘，它就有了路径，可以被后续工具引用（比如 ImageMagick 处理、上传到 PR、被 bash 命令使用）。这一步把图片从「剪贴板里的像素」升级为「工作现场里的一个对象」。

---

## 第二步：maybeResizeAndDownsampleImageBlock — 并行 resize

```typescript
const imageProcessingResults = await Promise.all(
  imageContents.map(async pastedImage => {
    const imageBlock: ImageBlockParam = {
      type: 'image',
      source: { type: 'base64', media_type: ..., data: pastedImage.content },
    }
    logEvent('tengu_pasted_image_resize_attempt', {
      original_size_bytes: pastedImage.content.length,
    })
    const resized = await maybeResizeAndDownsampleImageBlock(imageBlock)
    return { resized, originalDimensions: pastedImage.dimensions, sourcePath: ... }
  }),
)
```

注意 `Promise.all`——多张图片并行处理，不是顺序等待。`maybeResizeAndDownsampleImageBlock` 会检查图片是否超过 API 限制（Anthropic API 对单张图片有最大分辨率和文件大小限制），超出则降采样。

这一步的目的不只是让 API 能接受，更是为了避免「4MB 原图」在 base64 编码后撑爆请求体。

---

## 第三步：收集 imageMetadataTexts

```typescript
if (resized.dimensions) {
  const metadataText = createImageMetadataText(resized.dimensions, sourcePath)
  if (metadataText) {
    imageMetadataTexts.push(metadataText)
  }
}
```

这里有一个精细的 fallback 逻辑：
1. 优先使用 resize 后的尺寸（最准确）
2. 如果 resize 没有返回尺寸，用原始尺寸
3. 如果没有尺寸但有 sourcePath，至少保留 `[Image source: /path/to/file]`

为什么要这么仔细地保留路径？因为模型可能需要告诉你「这张图保存在 /tmp/screenshot.png，你可以用 ImageMagick 裁剪它」——这类推理需要模型知道文件在哪里。

---

## 第四步：addImageMetadataMessage — 元信息进隐藏消息

```typescript
function addImageMetadataMessage(result, imageMetadataTexts) {
  if (imageMetadataTexts.length > 0) {
    result.messages.push(
      createUserMessage({
        content: imageMetadataTexts.map(text => ({ type: 'text', text })),
        isMeta: true,  // 用户看不见
      }),
    )
  }
  return result
}
```

所有图片元信息打包成一条 `isMeta: true` 的用户消息——对用户隐藏，但模型能看到。这样 transcript 保持干净，模型却能获得完整的技术上下文。

---

## 这套处理为什么不是「多此一举」

最直观的实现是：把剪贴板 base64 直接塞进消息。

问题：
1. 原始图片可能超过 API 大小限制，直接报错
2. 图片只在这一次请求里用一下，后续工具拿不到它的路径
3. 图片元信息混进用户消息，每次都要和用户的自然语言搅在一起

Claude Code 的做法：存盘 → resize → 生成 image block → 收集元信息 → 元信息进隐藏消息。每一步都在为「图片既能被模型理解，也能被后续工具使用」这个目标服务。

---

*面试指导：被问到「多模态输入是怎么处理的」时，从「存盘」和「元信息分离」这两个设计决策切入，说明为什么不直接传 base64，比泛泛地说「做了 resize」深入得多。*
