# 为什么粘贴图片要先存盘、缩放，再谈会话

粘贴图片看起来像一个很表层的小功能，好像只要把剪贴板内容塞给模型就够了。Claude Code 没这么偷懒。它会先把图片存盘，再按 API 限制做缩放和降采样，然后才把处理后的图片块放进这一轮会话里。

这样做有两个好处。第一，图片进入模型前先变成合格材料，不会因为尺寸、体积或格式问题在后面突然出错。第二，图片被存到磁盘以后，它就不只是“给模型看一眼的临时像素”，而是当前工作现场里一个可被引用、可被工具再处理、可被上传到别处的正式对象。也就是说，Claude Code 一边在准备模型输入，一边也在为后续工具世界准备坐标。

这说明它对图片的理解很成熟。图片不是 UI 上的一次花哨粘贴，而是一种既要满足模型限制、又要进入工作现场的输入材料。先存盘、再缩放、再进入会话，这种顺序感非常重要。它让图片从一开始就是系统世界里的正经成员，而不是附着在聊天气泡外面的装饰物。

## 实现链
粘贴图片后，代码先用 `storeImages()` 把内容写到磁盘，再用 `maybeResizeAndDownsampleImageBlock()` 并行缩放，最后既保留 image block 也保留源路径和尺寸元信息，让模型和后续工具都能引用。

## 普通做法
更容易想到的方式是直接把原始 base64 图片塞进会话，或者完全不落盘，只把剪贴板内容当一次性输入。

## 为什么不用
因为图片既要满足 API 限制，又要让后续 CLI 工具、PR 上传或外部命令有路径可引用。只保留原始剪贴板既大又短命。

## 代价
要多做一次磁盘写入和缩放计算；但换来的是图片既适合喂模型，也能进入真实工具世界，而不是一次性看完就没了。
