# 主 agent 写过记忆后，后台提炼 agent 为什么要主动让开？

Claude Code 的 auto-memory 系统有两条写入路径：
1. **主 agent 直接写**：用户主动要求"记住这个"，或者模型判断值得记录时主动写
2. **后台提炼 agent**：每个回合结束后，在 stop hooks 里异步运行的 `executeExtractMemories`

这两条路径可能同时处理同一段对话历史。如果都写，就会有重复。系统的解决方案：**主 agent 写过了，提炼 agent 就跳过**。

---

## hasMemoryWritesSince 的检查逻辑

```typescript
function hasMemoryWritesSince(
  messages: Message[],
  sinceUuid: string | undefined,
): boolean {
  let foundStart = sinceUuid === undefined
  for (const message of messages) {
    if (!foundStart) {
      if (message.uuid === sinceUuid) foundStart = true
      continue
    }
    if (message.type !== 'assistant') continue
    const content = (message as AssistantMessage).message.content
    for (const block of content) {
      const filePath = getWrittenFilePath(block)
      if (filePath !== undefined && isAutoMemPath(filePath)) {
        return true  // 主 agent 写过了
      }
    }
  }
  return false
}
```

这个函数从 `sinceUuid` 游标位置开始扫描，寻找主 agent 是否有过向 `isAutoMemPath` 路径的写入操作。一旦找到，返回 `true`，提炼 agent 让开。

---

## 游标机制：避免重复扫描

```typescript
// closure-scoped state in initExtractMemories
let cursor = {
  lastRunUuid: undefined,  // 上次运行到哪条消息
}
```

每次提炼完成后，游标前进到这次处理的最后一条消息：

```typescript
cursor.lastRunUuid = lastMessageUuid
```

下次触发时，只扫描 `lastRunUuid` 之后的新消息。这个增量处理设计有两个好处：
1. **避免重复**：不会重复提炼已经处理过的内容
2. **性能**：随着会话增长，每次提炼的工作量不增加（只处理新增的消息）

但有一个边界情况：如果 `sinceUuid` 对应的消息在 compact 中被移除了，函数会 fall back 到处理所有消息：

```typescript
if (!foundStart) {
  return count(messages, isModelVisibleMessage)  // 返回所有消息数
}
```

这防止了"游标指向的消息已经不存在"时永久禁用提炼的问题。

## countModelVisibleMessagesSince：提炼的触发条件

提炼不是每轮都运行，而是累积了足够多的新消息才触发：

```typescript
function countModelVisibleMessagesSince(messages, sinceUuid): number {
  // 统计 sinceUuid 之后的 user/assistant 消息数
}
```

在 `runExtraction` 里，这个计数用于判断是否"有足够的新内容值得提炼"——如果上次提炼之后只有 1-2 轮对话，可能没有新的值得长期记忆的内容，跳过。

这个阈值机制防止了"每轮都为一两句话的对话启动一个 forked agent"的开销。

## forked agent 模式的成本结构

`executeExtractMemories` 用 `runForkedAgent` 实现：

```typescript
void extractMemoriesModule!.executeExtractMemories(
  stopHookContext,
  toolUseContext.appendSystemMessage,
)
```

"fire-and-forget"（`void`）意味着这个调用不等待结果——主回合的响应已经发给用户了，提炼在后台异步运行。

forked agent 会：
1. 继承主 agent 的 prompt cache（通过 `CacheSafeParams`）
2. 扫描现有记忆文件（用 `scanMemoryFiles`）
3. 分析对话历史里值得记忆的内容
4. 写入新的记忆文件或更新现有记忆文件

这个过程会消耗额外的 API 调用，但因为是异步后台运行，不阻塞用户。

---

## 信任层级：主 agent vs 提炼 agent

这个"主 agent 优先"设计体现了一个信任层级：

**主 agent 的优势**：
- 完整的对话上下文
- 用户的实时指令（"记住这个"、"忘掉那个"）
- 更完整的项目知识
- 有明确的用户授权

**提炼 agent 的定位**：
- 是一个自动化的、无用户实时参与的过程
- 对"什么值得记忆"的判断可能不如主 agent 精确
- 作为辅助，而不是主要的记忆写入者

让主 agent 优先，让提炼 agent 在主 agent 没有主动写的情况下补充——这是"人工判断 > 自动提炼"在记忆系统里的具体体现。

## 停机时的 drain 机制

`-p`（non-interactive / pipe mode）或 SDK 调用时，会话结束前需要等待后台提炼完成：

```typescript
// stopHooks.ts 中的说明
// For -p/SDK, print.ts drains the in-flight promise after flushing
// the response but before gracefulShutdownSync (see drainPendingExtraction).
```

`drainer` 是一个可以等待所有 in-flight 提炼完成的函数。在非交互模式下，进程退出前会调用 `drainer(timeoutMs)`，确保记忆文件在进程结束前写入完成。

在交互模式下，因为用户会话可能继续，后台提炼可以在后台慢慢完成，不需要 drain。

## 并发重叠保护

有时候上一次提炼还没结束，新的回合就开始了，stop hook 再次触发：

```typescript
// initExtractMemories 里的 overlap guard
let isRunning = false

// runExtraction 里：
if (isRunning) {
  // 跳过这次，不启动新的提炼
  return
}
isRunning = true
try {
  // 执行提炼
} finally {
  isRunning = false
}
```

这个简单的锁防止了"两个提炼 agent 同时扫描相同的对话历史并写入相同的记忆文件"的竞争条件。

## 面试指导

这道题考察的是"多个自动化进程协同工作时的协调机制"。

关键机制：
1. **互斥写入**：`hasMemoryWritesSince` 确保主 agent 和提炼 agent 不重复写
2. **游标增量处理**：避免每次都扫描全部历史
3. **触发阈值**：累积足够多的新消息才运行，不是每轮都跑
4. **并发保护**：isRunning 锁防止重叠运行
5. **停机 drain**：确保后台任务在进程退出前完成

能把这五个机制串起来讲，说明你理解了"后台自动化任务"不只是"fire-and-forget"，还需要协调、节流、生命周期管理这些工程细节。
