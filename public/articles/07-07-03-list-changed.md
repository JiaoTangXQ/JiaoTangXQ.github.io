# list_changed通知说明它接的是会自己变化的能力世界

## 实现链
- `client.ts` 会接住 MCP 的 `list_changed` 类通知，然后重新拉取对应的工具、资源或 prompt 列表。代码假设 server 背后的世界会变化，所以缓存不能只靠“首次连接时那份结果”。
- 这意味着 Claude Code 接入的不是一份静态文档，而是一片会自己长出新东西、旧东西也会消失的动态能力世界。

## 普通做法
- 更常见的做法是定时轮询，或者干脆不刷新，要求用户手动重启连接。
- 这样客户端实现最简单，因为不需要处理通知驱动的失效链。

## 为什么不用
- Claude Code 没只靠轮询，是因为轮询要么太慢，要么太浪费，而且它无法准确知道哪一类缓存该失效。
- 有了 `list_changed`，客户端能把变化对准真实对象，而不是粗暴重置整个世界。

## 代价
- 代价是通知处理和缓存失效会让代码更碎，错误时也更难定位是 server 变了还是客户端没跟上。
- 但这比一直抱着一份过期快照要合理得多。
