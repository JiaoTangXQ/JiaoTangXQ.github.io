# 历史为什么要分层留存：连续性的物理基础

面试题版本：**"Claude Code 会话的消息历史随时间增长，需要被压缩。压缩后的历史和原始历史有什么区别？系统如何保证压缩后的历史还能被正确接续，而不是产生混淆？"**

## 平坦历史的问题

最简单的历史存储方式：一个消息数组，从第一条到最新一条。压缩时，把旧的消息摘要成一条，删掉原始消息。

```typescript
// 压缩前
messages = [msg1, msg2, ..., msg100, msg101, ..., msg200]

// 压缩后（平坦方式）
messages = [summary_of_1_to_100, msg101, ..., msg200]
```

问题：`summary_of_1_to_100` 和 `msg101` 之间的边界对系统来说是隐式的——系统不知道哪些消息是原始的，哪些是摘要。如果后续需要再次压缩，或者需要决定"从哪里开始是原始消息"，就找不到明确的边界。

## compact boundary 的作用

`src/utils/messages.ts` 里的 `createMicrocompactBoundaryMessage()` 和 `getMessagesAfterCompactBoundary()` 函数，维护了一个显式的压缩边界标记。

压缩后的历史不是平坦的摘要替换，而是有明确层次：

```
[compact_boundary_marker]    ← 显式标记"这里之前的历史已被摘要"
[post_compact_messages]      ← 压缩后保留的消息（通常是 LLM 生成的摘要）
[current_conversation]       ← 当前正在进行的对话
```

`compact_boundary_marker` 是一个特殊消息类型，它的存在告诉系统：
- 这个 marker 之前的历史已经被压缩过
- 不要把 marker 之后的消息和 marker 之前的历史混在一起处理
- 如果需要恢复历史，从 marker 位置开始是安全的

## 分层对推理的影响

模型在生成响应时，会读取当前会话的所有消息。如果历史是分层的，模型可以正确理解：

- "这段历史是原始对话"
- "这段历史是之前的摘要"
- "compact boundary 之后是我们刚刚发生的对话"

如果历史是平坦的，摘要和原始对话混在一起，模型可能：
- 把摘要里的信息当成原始引用（"上面你说过..."），而实际上那是摘要改写的版本
- 在摘要和当前对话之间产生连续性错觉（实际上中间有大量被压缩掉的内容）
- 在需要回忆特定细节时，无法区分"我是在回忆原始细节还是摘要里的概括"

## session memory 是独立的一层

除了 compact boundary 里的压缩摘要，Claude Code 还有 session memory（`src/services/SessionMemory/`）——一个跨会话的长期记忆层。

这两层的区别：
- **压缩摘要**：当前会话的历史压缩，在会话内可用，会话结束后消失（或被进一步提炼到 session memory）
- **session memory**：跨会话的长期记忆，每次新会话开始时加载，不随压缩而消失

这两层服务不同的需求：压缩摘要服务于"这次长对话里之前发生了什么"，session memory 服务于"我和这个用户长期合作里积累的知识"。

## 分层的接续保证

"历史分层留存，才能被正确接续"的具体含义：

**接续的起点是确定的**：`getMessagesAfterCompactBoundary()` 找到压缩边界后的消息，这是接续的起点。模型知道这里是"我们的最近历史"，之前的是"摘要"。

**上下文传递是完整的**：每次压缩都会把关键信息（当前任务状态、重要决策、进行中的工作）保留在压缩摘要里。这些信息的格式是系统控制的，确保了接续时能找到需要的内容。

**层次本身传递信息**：`compact_boundary_marker` 的存在告诉接续者"这里有一个历史边界"，而不是让接续者自己猜测哪些消息是原始的、哪些是摘要的。

## 面试角度的总结

历史分层留存不是纯粹的存储效率问题，而是连续性的物理基础。平坦的历史在压缩后会丢失层次信息，导致后续接续时产生歧义。显式的 compact boundary、压缩摘要、session memory 这三层各司其职，让任何时刻的接续都有明确的起点和可信的背景。这种显式的分层设计比"智能地猜测历史边界"更可靠，也更容易调试。
