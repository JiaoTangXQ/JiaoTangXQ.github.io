# 浏览器再特别，进系统时也尽量先说 MCP

Claude in Chrome 明明连接的是浏览器扩展和 Native Host，但它进 Claude Code 主系统时，仍然尽量先伪装成一个标准 MCP 世界。`setup.ts` 返回的是 `mcpConfig`、`allowedTools` 和 `systemPrompt`；`mcpServer.ts` 又提供统一的 `createChromeContext()`，让 subprocess MCP server 路径和 in-process 路径都共用同一份上下文。也就是说，浏览器世界当然有自己的复杂性，但它进入宿主系统时，优先说的还是“我是一个 server、这是一组 tools、这是同一套 permission mode 和 logging 约定”。

更容易想到的做法，是为浏览器单独定义一套 RPC 协议和工具发现方式。那样浏览器团队写起来可能更自由，但主系统就得额外学一门新语言，skills、ToolSearch、system prompt、日志和权限也都会被迫分叉。

Claude Code 明显不想承担这种分叉成本，所以它宁可在浏览器这一层多做翻译，也尽量让系统上层继续把它当 MCP 处理。这种选择很合理，因为越靠近中心的语言越应该少，而不是越多。

## 实现链
`setupClaudeInChrome()` 返回动态 MCP 配置，`runClaudeInChromeMcpServer()` 用 `StdioServerTransport` 起 server，连浏览器世界也先伪装成系统已经会说的 MCP。它不是直接往主循环里塞一个私有 browser API。

## 普通做法
更直觉的做法是浏览器扩展和 CLI 自己定义一套私有 RPC 协议。

## 为什么不用
Claude Code 不这么做，因为一旦不说 MCP，工具发现、权限、UI 渲染和连接管理都得单独再做一遍。

## 代价
代价是浏览器世界也要背上 MCP 的适配成本。
