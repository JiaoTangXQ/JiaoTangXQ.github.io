# 模型选择的优先级链：Claude Code 怎么决定用哪个模型？

Claude Code 支持多种 Claude 模型，有时你需要切换——用更快的 Haiku 处理简单任务，用 Opus 处理复杂分析。可以用 `/model` 命令切换，可以用 `--model` CLI 参数，可以设 `ANTHROPIC_MODEL` 环境变量，可以写进 `settings.json`。

当这四种方式同时存在时，谁优先？`utils/model/model.ts` 里有明确答案。

## 优先级顺序（高到低）

```typescript
// 注释里写得很清楚
export function getUserSpecifiedModelSetting(): ModelSetting | undefined {
  // 1. 会话内 /model 命令切换 — 最高优先级
  const modelOverride = getMainLoopModelOverride()
  if (modelOverride !== undefined) {
    specifiedModel = modelOverride
  } else {
    // 2. 启动时 --model 参数
    // 3. ANTHROPIC_MODEL 环境变量
    // 4. settings.json 里的配置
    specifiedModel = process.env.ANTHROPIC_MODEL || settings.model || undefined
  }
}
```

规则：**会话内 > 启动参数 > 环境变量 > 配置文件**

会话内的 `/model` 命令优先级最高是因为它表达的是用户的**即时意图**——你已经在用这个 Claude 了，临时想换个模型做一件事，这个决定应该立即生效，不应该被配置文件覆盖。

## 模型别名系统

直接写模型名（`claude-opus-4-5-20251101`）很繁琐，Claude Code 支持别名：

```typescript
export type ModelAlias = 'opus' | 'sonnet' | 'haiku' | 'fast' | ...

export function isModelAlias(model: string): model is ModelAlias {
  return MODEL_ALIASES.has(model)
}
```

`opus` → 最新的 Opus 模型，`haiku` → 最新的 Haiku 模型，`fast` → 专门针对速度优化的配置。

别名在 `resolveOverriddenModel` 里被解析成实际模型名，之后整个系统都用实际模型名工作。

## "Small Fast Model" 的特殊处理

```typescript
export function getSmallFastModel(): ModelName {
  return process.env.ANTHROPIC_SMALL_FAST_MODEL || getDefaultHaikuModel()
}
```

Claude Code 内部有些场景不需要最强的模型——比如生成工具使用摘要、分类请求类型、处理简单的上下文管理任务。这些场景会调用 `getSmallFastModel()` 而不是主循环模型。

这是一个重要的设计原则：**不同任务用不同规格的模型**。用 Opus 回答"这个命令该怎么格式化"是浪费。

## Extended Thinking（扩展思考）的触发

```typescript
export type ThinkingConfig =
  | { type: 'adaptive' }   // 模型自己决定是否思考
  | { type: 'enabled'; budgetTokens: number }  // 强制开启，指定 token 预算
  | { type: 'disabled' }   // 强制关闭

export function shouldEnableThinkingByDefault(model: ModelName): boolean {
  // 根据模型能力和当前模式决定默认值
}
```

`ThinkingConfig` 里的 `budgetTokens` 是 Extended Thinking 会消耗的 token 上限。不是所有模型都支持扩展思考，不是所有任务都值得消耗思考预算。这个配置让不同场景可以精确控制思考深度。

更有趣的是 `ultrathink` 关键词：

```typescript
export function hasUltrathinkKeyword(text: string): boolean {
  return /\bultrathink\b/i.test(text)
}
```

用户在输入里写 "ultrathink" 关键词，可以触发更高的思考预算。这是一个人机协作的约定——用户用一个特定词表达"我希望你深度思考这个问题"。

注意这里的 regex 是每次调用都创建新实例，而不是模块级的 `/ultrathink/gi` 变量。注释解释了原因：`/g` 标志有 `lastIndex` 状态，如果复用同一个 regex 实例，`.test()` 和 `.matchAll()` 之间会有状态泄漏。

## 1M context 的条件判断

```typescript
export function has1mContext(): boolean {
  // 检查是否满足 1M context 的使用条件
  // 包括：模型支持、订阅等级、未被 GrowthBook 关闭
}
```

1M context window 不是所有用户都能用的——需要特定的订阅等级，并且通过 GrowthBook feature flag 控制灰度发布。这种分层能力设计在 SaaS 产品里很常见，Claude Code 把它处理得比较优雅：业务代码调用 `has1mContext()` 而不是直接检查订阅类型。

## 面试角度

这套模型选择系统体现了几个工程实践：

1. **配置优先级是业务语义，不是实现细节**：用户临时指定 > 启动配置 > 环境配置 > 持久化配置，这个顺序反映了"用户意图的紧迫性"
2. **别名系统隔离接口和实现**：用户写 `opus`，系统内部处理版本迁移
3. **按任务选模型**：不是一个模型走天下，成本和质量都有要求
4. **feature flag 控制能力灰度**：新能力（1M context、ultrathink）不是直接给所有人，通过 GrowthBook 精确控制发布范围
