# 工程取舍：等待权限审批时，为什么要停止 loading 状态？

## UI 状态和系统状态的语义绑定

Loading 状态（旋转动画、进度条）在 UI 中有明确的语义：**系统正在自动执行某件事，用户只需等待**。

当远端工具请求权限审批时，系统的状态从"自动执行"切换到了"等待人工决策"。这是本质上不同的两种等待：

- Loading 表示的等待：系统在工作，我什么都不用做
- 审批等待：**我需要做一个决策，系统在等我**

如果权限审批弹出来，但底下的 loading 仍在转，用户看到的是两个互相矛盾的信号：

- loading → "继续等着，它在跑"
- 审批弹窗 → "你需要做决定"

矛盾的信号会让用户困惑：是审批完了它自己会继续，还是必须审批才能继续？审批完了是不是还需要等？

## setIsLoading(false) 的诚实

`useRemoteSession.ts` 在权限请求进入队列时，调用 `setIsLoading(false)`：

```typescript
// 权限请求到来时
setIsLoading(false)
// 添加审批项到队列
addPermissionRequest(request)
```

这是在做一个 UI 语义声明：**现在的等待不是"机器在工作"，而是"用户需要做决定"**。

停掉 loading 让用户清楚地看到：系统已经暂停了，球在你这边。

审批完成后（用户点击 Allow 或 Deny），再根据结果恢复到合适的状态——如果工具被允许，继续执行（可能需要重新显示 loading）；如果被拒绝，显示相应的结果状态。

## 流动感 vs 诚实性

UI 设计里有一个持续的张力：**体验的流动感**（seamlessness）vs **状态的诚实性**（honesty）。

追求流动感的设计：让过渡尽可能无感，用户不需要感知到系统内部的状态切换。Loading 持续，审批弹窗只是一个叠加层，底下"一切照常"。

追求诚实性的设计：让 UI 精确反映系统当前真实的状态，不在视觉上制造虚假的进度感。

Claude Code 选择了诚实性。理由很直接：**权限审批是一个明确的人工干预点，它改变了工作流的性质**。这个性质变化如果被 loading 动画掩盖，用户可能会：

- 忽视审批弹窗（以为只是一个提示，不是需要操作的）
- 不清楚自己是否还需要等待（因为 loading 还在转）
- 对审批完成后的状态感到困惑

## 与权限制度的关系

这个 UI 决策背后有一个更深的产品哲学：**权限审批是 Claude Code 安全模型的核心，不应该被降格为"流程中的一个小插曲"**。

Loading 动画持续转动，在视觉上把审批变成了一个叠层——"后台在继续工作，你顺手批准一下"。这种视觉框架下，用户更容易快速点击"允许"而不仔细查看权限请求的细节。

停掉 loading，让审批弹窗成为屏幕的主角，传达的信息是：**这件事需要你认真看一下，然后做决定**。这是对审批行为的尊重，也是对用户注意力的尊重。

## 恢复状态的精确性

审批完成后的状态恢复，也需要精确：

- 用户点击 Allow → 发送 `control_response` 允许 → 远端工具继续执行 → 如果需要，重新显示 loading
- 用户点击 Deny → 发送 `control_response` 拒绝 → 远端工具被中止 → 显示拒绝结果状态

"重新显示 loading"是关键——不是在审批完成后立刻恢复，而是等远端确实重新开始执行后再显示。如果远端因为其他原因也停止了（比如拒绝导致任务失败），就不需要重新 loading。

这种精确的状态管理，比"始终 loading"或"审批完立刻 loading"，都需要更多代码，但换来的是 UI 状态和系统状态的真实对应。

## 面试指导

"UI 状态如何反映系统状态"是前端/全栈面试里容易被忽视的深度问题。

好的回答不只是"加 loading 和 error 状态"，而是：

1. **识别状态语义**：loading 的语义是"系统在自动工作"，不是"有东西还没完成"
2. **识别状态切换点**：当系统从"自动工作"切换到"等待人工"，UI 需要对应切换
3. **讨论用户感知**：状态不准确对用户行为的影响（忽视审批、误判进度）
4. **说清楚诚实性和流动感的取舍**：在权限/安全场景下，诚实性优先于流动感

这个取舍不是所有场景都适用的——对于一个纯展示类的 loading，刻意停掉再恢复可能反而造成不必要的闪烁。具体取舍取决于状态变化的语义重要性。
