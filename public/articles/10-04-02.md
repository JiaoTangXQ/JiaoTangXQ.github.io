# /compact 的自定义指令是怎么被注入到摘要过程中的？

用户输入 `/compact 重点保留架构决策`，这段文字怎么到达摘要模型？整个注入链是什么？

---

## 数据流

```
用户输入: "/compact 重点保留架构决策"
  ↓
args = "重点保留架构决策"
  ↓
customInstructions = args.trim()
  ↓
trySessionMemoryCompaction: 因为有自定义指令，跳过
  ↓
executePreCompactHooks({trigger: 'manual', customInstructions})
  → hook 可以追加更多指令 (hookResult.newCustomInstructions)
  ↓
mergeHookInstructions(customInstructions, hookResult.newCustomInstructions)
  → 合并成最终指令字符串
  ↓
getCompactPrompt(mergedInstructions)
  → 生成带自定义指令的摘要 prompt
  ↓
摘要模型接受 prompt，生成摘要
```

---

## mergeHookInstructions：指令合并规则

```typescript
export function mergeHookInstructions(
  userInstructions: string | undefined,
  hookInstructions: string | undefined,
): string | undefined {
  if (!hookInstructions) return userInstructions || undefined
  if (!userInstructions) return hookInstructions
  return `${userInstructions}\n\n${hookInstructions}`
}
```

规则非常简单：用户指令在前，hook 指令在后，用双换行分隔。空字符串规范化为 `undefined`。

这个顺序意味着**用户指令优先级更高**——如果两者有冲突，摘要模型在解读时通常会更多关注先出现的内容。

---

## getCompactPrompt：指令如何影响摘要 prompt

`compact/prompt.ts`（源码里是 `getCompactPrompt`）负责把自定义指令嵌入到系统 prompt 里。基础 prompt 告诉摘要模型"你要做什么"，自定义指令被追加在具体要求之后：

```
请对下面的对话历史生成摘要：

[基础摘要要求：保留关键决策、工具调用结果、当前任务状态...]

额外要求：
重点保留架构决策        <- 用户的自定义指令
```

这种结构让基础摘要质量不受自定义指令干扰——即使用户写了怪异的指令，基础要求依然在。

## hook 注入指令的场景

pre-compact hooks 为什么需要注入指令？有几个实际用例：

**场景 1：CI 自动 compact**
CI 系统可以注册一个 pre-compact hook，当 trigger 为 `'auto'` 时，自动注入"保留所有测试失败的错误信息"——因为 CI 环境里测试失败的上下文特别重要，不应该在 compact 中被简化。

**场景 2：项目级别的摘要约定**
团队可以在 `.claude/hooks/precompact.sh` 里写一个脚本，根据当前项目配置（比如 `CLAUDE.md` 里的约定），动态注入"这个项目的核心约定"提醒摘要模型哪些内容不能压缩。

**场景 3：显示自定义消息**
hook 还可以返回 `userDisplayMessage`，在 compact 进行时显示给用户看：

```typescript
const userDisplayMessage = hookResult.userDisplayMessage
```

## stripImagesFromMessages：压缩前的预处理

在生成摘要之前，还有一个预处理步骤——把消息里的图片块替换成占位符：

```typescript
export function stripImagesFromMessages(messages: Message[]): Message[] {
  return messages.map(message => {
    if (message.type !== 'user') return message
    // 把 image 块替换成 '[image]'
    // 把 document 块替换成 '[document]'
    // 处理 tool_result 里嵌套的图片
  })
}
```

原因是：图片是大量 token 的来源，而摘要模型不需要"看到"图片内容来生成文字摘要。如果不去掉图片，CCD（Claude Computer Desktop）这类频繁附加截图的使用场景，compact 请求本身就会 prompt_too_long。

这个处理发生在摘要 API 调用之前，不影响 compact 结果里保留的消息。

## compact 的 suppressFollowUpQuestions 参数

```typescript
await compactConversation(
  messagesForCompact,
  context,
  cacheSafeParams,
  false,  // suppressFollowUpQuestions — 手动 compact 是 false
  customInstructions,
  false,  // isAutoCompact
)
```

手动 compact 的 `suppressFollowUpQuestions = false`，意味着摘要模型可以在摘要里提出后续问题或建议。autocompact 是 `true`，因为自动触发的 compact 应该对用户尽量透明，不应该突然生成一堆"请问下一步要怎么做"的问题。

## 面试指导

这道题考察你对"用户输入 → 系统行为"这条链路的完整理解。

面试时可以从两个角度切入：

**数据流角度**：args → customInstructions → mergeHookInstructions → getCompactPrompt → 摘要 API，每一步数据的格式和语义变化是什么。

**设计决策角度**：为什么用户指令放在 hook 指令之前？为什么 suppressFollowUpQuestions 手动和自动不同？为什么图片需要在摘要前去掉？

能把"这些不起眼的细节是为什么这样设计的"说清楚，比单纯描述代码路径更有价值。
