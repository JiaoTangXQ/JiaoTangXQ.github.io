# 挂载后还会主动纠正绕权模式说明状态层先服从更高制度

`AppStateProvider` 挂载后还会检查 bypass permissions 如果已经被更高制度禁掉，就立刻把状态纠正回来。这个动作很有代表性：界面上能不能绕过审批，不以旧缓存或旧预设为准，而以当前真正生效的规则为准。

也就是说，状态层不是只会保存事实，它还会主动修正假现实。

## 实现链

`AppStateProvider` 挂载后会检查 `toolPermissionContext.isBypassPermissionsModeAvailable` 和 `isBypassPermissionsModeDisabled()`；如果远端设置已经把绕权模式禁掉，它会立即用 `createDisabledBypassPermissionsContext()` 修正 store。状态层先服从更高制度，而不是坚持自己开局那份值。

## 普通做法

更常见的做法，是默认值定了就先用着，等用户下次交互时再慢慢修正。

## 为什么不用

Claude Code 不愿意让绕权模式哪怕短暂站在错姿态里，是因为权限制度一旦错位，后面的工具可见性和执行判断都会跟着错。它宁可在挂载后马上补一次修正。

## 代价

代价是 mount 阶段多了一次纠偏逻辑，状态初始化路径更难一眼看完；但制度正确性被放在了“先别错”这一边。
