# 面试题：一条被系统接受的输入，会携带哪些隐藏世界一起进入？

## 代码考古

翻开 `processUserInput.ts` 末尾的 `addImageMetadataMessage` 函数：

```typescript
function addImageMetadataMessage(
  result: ProcessUserInputBaseResult,
  imageMetadataTexts: string[],
): ProcessUserInputBaseResult {
  if (imageMetadataTexts.length > 0) {
    result.messages.push(
      createUserMessage({
        content: imageMetadataTexts.map(text => ({ type: 'text', text })),
        isMeta: true,  // ← 用户看不见，但模型能看到
      }),
    )
  }
  return result
}
```

这个函数被调用了三次：处理 bash 命令、处理 slash 命令、处理普通 prompt，每条路径最后都要经过它。这意味着**每条输入，不管走哪条分支，都可能带着一条用户不可见的元信息消息一起进入会话**。

这只是「隐藏乘客」的其中一个。

## 被系统接受的输入，实际上背着什么

拆开 `processUserInputBase` 的执行结果，一条典型的输入最后会产出：

**1. 图片元信息消息（`isMeta: true`）**

粘贴图片时，`imageMetadataTexts` 会收集每张图片的尺寸和来源路径。这些信息被打包成一条 `isMeta` 消息发给模型，但不出现在用户 transcript 里。模型能知道「这张图 1920×1080，来自 /tmp/screenshot.png」，用户看到的只是图片本身。

**2. 附件消息（`AttachmentMessage`）**

`getAttachmentMessages()` 从输入字符串里提取 @提及、IDE 选中的代码、MCP 资源引用、记忆文件等，生成一批独立的 `AttachmentMessage`。这些不是用户原句的一部分，而是运行时注入的背景材料。

**3. Hooks 附加上下文**

如果配置了 `UserPromptSubmit` 类型的 hooks，hook 执行结果会以 attachment message 的形式追加到 result.messages 里：

```typescript
result.messages.push(
  createAttachmentMessage({
    type: 'hook_additional_context',
    content: hookResult.additionalContexts.map(applyTruncation),
    hookName: 'UserPromptSubmit',
    ...
  }),
)
```

用户触发一次输入，hook 里的外部脚本可以在后台添加额外上下文，这些上下文会和用户消息一起进入会话。

**4. 工具许可和模型偏好**

通过 `allowedTools`、`model`、`effort` 字段，某些命令可以临时改写这一轮的工具集和推理参数。这不是消息，而是元数据层面的「隐藏乘客」——外层 QueryEngine 会读取这些字段来配置 API 请求。

**5. 命令权限附件**

对于 `prompt` 型命令，`getMessagesForPromptSlashCommand` 会额外生成一条 `command_permissions` 类型的 attachment message：

```typescript
createAttachmentMessage({
  type: 'command_permissions',
  allowedTools: additionalAllowedTools,
  model: command.model
})
```

这条消息让 attachment 系统知道这一轮激活了哪些额外权限，在压缩和会话恢复时能正确重建。

## 为什么要这样设计

用一个反问来说明：如果把所有这些信息都塞进用户消息的正文里，会发生什么？

模型会收到一堆「1920×1080 来自 /tmp/foo.png」这样的说明文字夹在用户真实意图中，transcript 会被技术说明污染，用户读日志会看到满屏元数据。更严重的是，这些信息的处理方式不同——有的只给模型看，有的要压缩时保留，有的要恢复时重建——混进正文就失去了独立管理的能力。

把它们分开成不同类型的消息，系统才能精确控制每一类「隐藏乘客」的生命周期。

## 一条输入的真实体积

用户感知：一行文字 + 一张图片
系统处理后：可能产出 5-8 条不同类型的消息，其中 2-3 条用户完全不可见

这个差距，就是 `processUserInput` 这一层存在的意义。

---

*面试指导：被问到「Claude Code 的会话消息是什么构成的」时，不要只说「用户消息和助手消息」。把 AttachmentMessage、isMeta 消息、command_permissions 消息这些隐藏层讲出来，才能展示对系统架构的真实理解。*
