# Leader 不是最强的 Worker，而是团队对外的唯一接口——从源码看这个设计决策

在 Claude Code 的 teammate 系统提示里，有一行被明确写出来的制度假设：

> The user interacts primarily with the team lead.

这不是界面说明，而是**团队架构的基础公理**。整套团队系统的许多设计决策，都从这条假设推导出来。

## 系统提示在说什么

`TEAMMATE_SYSTEM_PROMPT_ADDENDUM` 告诉每个 teammate：你在一个团队里工作。如果你要和用户沟通，优先通过 leader，而不是直接输出——因为用户主要看 leader 那边。

这带来了一个很重要的推论：**teammate 的输出在默认情况下对用户是不可见的**。用户要看到 worker 的进度，需要 worker 主动用 `SendMessage` 报告给 leader，或者 leader 主动查询。

这听起来像是信息屏蔽，但工程意图是相反的——不是屏蔽信息，而是**收束打断**。

## inProcessRunner.ts 和 leaderPermissionBridge.ts 做了什么

`leaderPermissionBridge.ts` 里注册了两个 setter：

```typescript
let registeredSetter: SetToolUseConfirmQueueFn | null = null
let registeredPermissionContextSetter: SetToolPermissionContextFn | null = null
```

这两个 setter 是 React REPL 注册进来的——当 leader 界面挂载时，它把自己的"往权限确认队列里推消息"的能力注册到这个桥上。

`inProcessRunner.ts` 里的 worker 请求权限时，找的就是这个桥：

```typescript
const setQueue = getLeaderToolUseConfirmQueue()
if (setQueue) {
  // 直接推进 leader UI 的确认队列
  setQueue(prev => [...prev, confirmRequest])
} else {
  // 桥不存在，走 mailbox permission request
  await sendPermissionRequestViaMailbox(request)
}
```

这两条路径的终点都是同一件事：**让 leader 界面展示一个权限确认弹窗**，不管 worker 是同进程还是跨进程。

## 为什么不让每个 Worker 直接弹窗

想象另一个方案：每个 worker 各自有权限确认能力，需要时直接弹给用户。

这在用户体验上会产生一个严重问题：**并发的打断**。

5 个 worker 同时工作，每个都可能在不同时刻请求权限。用户可能在 30 秒内连续收到来自不同 worker 的 5 个确认弹窗，每个弹窗来自不同 agent，要求不同权限。用户需要：

1. 理解每个弹窗来自谁
2. 理解每个请求的上下文
3. 按顺序或并行地处理它们

这不是"用户体验差"，这是"用户无法正常工作"。

leader 集中承接权限审批，本质上是在做**打断合批**——把多个 worker 的权限请求排入同一个队列，按顺序呈现给用户，每次只面对一个请求，有清晰的来源标识（worker badge）和上下文。

## 权力集中 vs 体验保护

有人会把"leader 统一承接"解读为权力集中设计，但这个解读方向搞反了。

从用户的视角，leader 只是一个**缓冲区和翻译层**——它把团队内部的并发复杂性，翻译成用户可以线性处理的请求序列。leader 没有拦截任何东西，只是整理了呈现顺序。

从工程的视角，leader 作为单一接口使得以下问题有了明确答案：

- 谁来维护确认队列的状态？（leader UI）
- 权限模式（plan/default）的权威来源在哪里？（leader 的 toolPermissionContext）
- 当多个 worker 同时需要权限时，用什么顺序处理？（先到先出队列）
- 如果 worker 完成了，但它的权限请求还在队列里怎么办？（leader 可以丢弃）

这些问题在"每个 worker 自己处理"的架构里都没有清晰答案，而在"leader 统一承接"的架构里都可以被单点处理。

## 面试怎么答

如果面试题是"多 agent 系统里，权限审批应该中心化还是分散到每个 agent"：

**分散的表面优势**：每个 agent 独立，延迟低，不存在单点瓶颈。

**分散的真实问题**：用户被多个并发弹窗轰炸；无法统一查看"当前有哪些待批请求"；每个 agent 需要各自维护审批状态。

**中心化的真实价值**：不是为了"控制"，而是为了把用户的交互成本从 O(n·并发数) 降到 O(n)。用户始终面对一个队列，一个界面，一套操作习惯。

**工程取舍**：leader 成为软瓶颈（所有权限请求排队）。这个瓶颈在实践中很少成为性能问题，因为权限请求的频率通常不高，而且用户的审批速度本身就是限速因素。

**最关键的一点**：中心化接口不等于中心化执行。Worker 仍然并行执行它们的任务；只有需要用户参与的瞬间，才会通过 leader 界面集中呈现。执行是并行的，打断是串行的。
