# Worker 请求权限时，为什么不直接弹给用户而要绕一圈到 Leader？

5 个 agent 并行处理一个大任务，每个 agent 在执行过程中可能需要：执行 bash 命令、写文件、读配置文件……这些都需要用户确认。

如果每个 agent 独立向用户申请权限，会发生什么？

## 量化一下并发打断有多严重

假设 5 个 agent 各自独立工作，平均每 2 分钟需要一次权限确认。10 分钟里，用户会收到约 25 个权限请求，来自 5 个不同的 agent，关于不同的工具调用，需要不同的上下文理解才能正确判断。

更糟的是，这些请求是**并发的**，不是排队等候的。用户正在思考 agent A 的请求时，agent B 的请求弹窗已经出现，agent C 又在等待，用户可能不小心点错。

研究人类任务切换成本的文献告诉我们：从一个任务切换到另一个任务，需要约 20 分钟才能重新进入深度专注状态。对频繁的权限确认打断来说，用户实际上永远在浅层注意力状态。

这就是"每个 worker 各批各的"方案的真实代价——**用户的注意力被团队的内部并发噪音消耗殆尽**，而不是真正用于监督 AI 的工作质量。

## leader 权限队列如何工作

`inProcessRunner.ts` 里的权限请求路径：

```typescript
// 先尝试走 leader UI 桥（in-process 快速路径）
const setQueue = getLeaderToolUseConfirmQueue()
if (setQueue) {
  // 推进 leader 的确认队列，附带 workerBadge
  setQueue(prev => [...prev, {
    ...confirmRequest,
    workerBadge: {
      name: agentName,
      color: agentColor,
    }
  }])
  // 等待 leader 处理
  return waitForLeaderApproval(requestId)
}

// 桥不可用，走 mailbox（tmux teammate 的路径）
await sendPermissionRequestViaMailbox(request)
return pollForPermissionResponse(requestId)
```

两条路径的终点都是同一件事：**请求在 leader 的 UI 队列里排队，用户看到一个弹窗，点击批准或拒绝，结果传回 worker**。

用户面对的是一个队列，一次处理一条，清楚知道"这是哪个 agent 要做什么"（因为有 workerBadge）。

## 请求上下游的完整流程

```
Worker agent 执行到需要权限的操作
  ↓
构造 ToolUseConfirm 请求（工具名、输入参数、建议规则）
  ↓
通过桥推进 leader UI 队列（或写入 mailbox）
  ↓
Worker 暂停，等待 leader 的响应
  ↓
Leader UI 显示弹窗，用户看到：
  - 这是哪个 worker 的请求（workerBadge）
  - 请求什么权限（工具名）
  - 具体参数（工具输入）
  - 可以选择：批准 / 拒绝 / 总是允许
  ↓
用户做出决定
  ↓
决定写回 permissionSync（如果是 mailbox 路径）或直接通过桥传回
  ↓
Worker 收到响应，继续或中断执行
```

这个流程里，worker 真正被"暂停"了——它不会继续往下执行，直到收到权限响应。这是有意为之的设计，防止 worker 在等待时继续做可能需要被批准的其他操作。

## permissionSync 的持久化设计

对于通过 mailbox 走的权限请求，`permissionSync.ts` 把请求写到文件系统：

```
~/.claude/teams/{teamName}/permissions/
  pending/
    perm-1234567890-abc.json   ← 待批准的请求
  resolved/
    perm-0987654321-xyz.json   ← 已处理的请求
```

这不只是临时缓冲。**把权限请求落盘意味着它可以被追溯**——leader 可以查看"这个团队在执行过程中请求了哪些权限"，哪些被批准，哪些被拒绝，谁批的。

这对 AI 安全审计来说很重要：如果一个多 agent 系统执行了敏感操作，能追溯"是哪个 agent 请求的、用户是否批准了"，比"什么都不记录、靠日志拼凑"要强得多。

## 单点瓶颈的实际影响

"所有权限请求都通过 leader"确实创造了一个串行瓶颈。如果 5 个 worker 同时需要权限，它们会在 leader 的队列里排队，第二个 worker 要等第一个处理完才能得到响应。

在实践中，这个瓶颈有多严重？

实际工作场景里，权限请求的频率通常不高（每几分钟一次），并且用户的处理速度本身就是限速因素——你批准请求需要思考时间，这个时间远大于串行等待的时间。

更重要的是：**并发的权限请求不会让事情变快，只会让用户更困惑**。如果 5 个 worker 同时等待权限，用户并不能同时处理 5 个请求——她还是一次只能处理一个，只是要在混乱的界面里找到哪个请求来自哪个 agent，判断上下文是什么。

串行队列 + 清晰标识，比并发弹窗 + 混乱来源，实际上处理得更快，也更安全。

## 面试怎么答

如果面试题是"多 agent 系统的权限控制应该中心化还是分散"：

**分散的真实问题**：不是"权限会失控"，而是"用户的注意力会被无序地并发打断"。用户管理 5 个 agent 的权限，和管理 1 个 agent 是不同量级的认知负担。

**中心化的价值**：让用户的注意力消耗是 O(n)（顺序处理 n 个请求）而不是 O(n + 并发混乱成本）。队列抹平了并发峰值，把用户的决策变成有序的线性序列。

**额外的安全价值**：权限请求落盘 = 可审计。这在需要合规追踪的场景里尤其重要——你能证明"这个操作是经过用户明确批准的"，而不是"不知道谁批的，反正执行了"。
