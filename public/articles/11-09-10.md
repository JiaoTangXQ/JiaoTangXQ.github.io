# selectedIndex 变化时为什么要主动 scrollTo？光标是视觉承诺，不是内存数字

`selectedIndex` 是键盘导航的"光标位置"。为什么改变这个状态后需要触发滚动？

## selectedIndex 的来源

`selectedIndex` 由 `Messages.tsx` 的 cursor 状态控制，用户按 `j`/`k`/`↑`/`↓` 时更新。`VirtualMessageList` 接收这个值作为 prop，渲染时把对应消息高亮。

但有一个场景：用户按 `G`（跳到底部）或 `gg`（跳到顶部），或者按 `Shift+↑` 跳到上一条用户 prompt。这些操作可能让光标跳到当前视口以外的位置——`selectedIndex` 更新了，但视口没有移动，用户看不到被选中的消息。

如果此时只更新 `selectedIndex` 而不滚动，用户会看到：
- 之前的高亮消失（旧的 selectedIndex 对应的行去掉高亮）
- 新的高亮没有出现在屏幕上（在视口外）
- 界面看起来"光标消失了"

## useEffect([selectedIndex]) 的实现

```typescript
useEffect(() => {
  if (selectedIndex === undefined) return
  const s = jumpState.current
  const el = s.getItemElement(selectedIndex)
  if (el) {
    scrollRef.current?.scrollToElement(el, 1)  // 1 = 1行 headroom
  } else {
    s.scrollToIndex(selectedIndex)  // 消息未挂载，先滚到估算位置
  }
}, [selectedIndex, scrollRef])
```

**注意：deps 里有 selectedIndex，但没有 offsets**。这是有意识的选择：

- 如果 offsets 也在 deps 里，每次消息高度变化（流式输出）都会重新 scrollTo，视口会不断跳动
- 只有 selectedIndex 变化时才滚，可以把"滚动"和"内容更新"解耦

通过 `jumpState.current` 读 `getItemElement` 和 `getItemTop`，是上一篇讲的"ref 读最新值，但函数稳定"模式的应用。

## scrollToElement vs scrollToIndex 的选择

- `getItemElement(selectedIndex)` 返回该消息已挂载的 DOM 元素
- 如果返回非空：消息在视口附近（已挂载），直接 `scrollToElement(el, 1)` 精确定位，1 行的 headroom
- 如果返回 null：消息超出虚拟滚动窗口（未挂载），`scrollToIndex` 先估算位置，之后 virtual scroll 会挂载该消息

这和 seek effect 的两阶段跳转有相似的结构，但更简单：选中消息的高亮不需要 scanElement 扫描，只需要确保消息在视口里，高亮由渲染逻辑自动处理（当 `selectedIndex === idx` 时，VirtualItem 有高亮样式）。

## 与 cursorNavRef.navigatePrev/navigateNext 的配合

`cursorNavRef.navigateNext()` 调用后不直接触发 scrollTo，而是调用 `setCursor`，`setCursor` 更新 `selectedIndex`，`selectedIndex` 变化触发上面的 `useEffect`。

这是一个反应式链路：

```
j 键
  → navigateNext()
  → setCursor({ uuid, ... })
  → Messages.tsx 的 cursor state 更新
  → selectedIndex prop 变化
  → VirtualMessageList 的 useEffect 触发
  → scrollTo
```

每一步都是单向数据流，不需要在 navigateNext 里直接调用 scrollTo。

## 面试指导

这道题本质上考的是"光标的语义"：光标是一个纯 UI 状态（我选择了哪个元素），还是一个交互承诺（我正在看哪个元素）？

**Claude Code 的答案**：光标是交互承诺。选中意味着"这条消息在用户的注意力里"，如果消息不在屏幕上，这个承诺就没有实现。主动滚动是实现承诺的机制。

**反例**：在某些列表组件里，光标（focused index）和滚动位置是完全独立的——你可以把光标移到第 500 条，但视口留在第 1 条。这种设计在文件系统树形视图里可能合理（你在折叠的节点里"记忆"了一个位置），但在 Claude Code 的 transcript 阅读场景里，光标脱离视口是没有意义的。

**追问**：如果用户在 seekEffect 运行期间（跳转动画过程中）又按了 j，selectedIndex 变化，再次触发 scrollTo，会和 seekEffect 的 scrollTo 冲突吗？答：两个 scrollTo 都是对同一个 scrollRef 的命令，后一个会覆盖前一个。因为都是 `scrollRef.current.scrollTo(n)`，最终位置是最后一次调用决定的。在用户明确按 j 导航的情况下，navigateNext 触发的 scrollTo 应该优先，seekEffect 的滚动被覆盖是正确的行为。
