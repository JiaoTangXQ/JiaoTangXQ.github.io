# 搜索语义由工具自己定义，不由消息壳猜测

Claude Code 里按 `/` 进入搜索后，用户输入关键词，系统在消息列表里查找匹配。问题是：对于工具调用的结果，「匹配」应该匹配什么？

这个问题不像文本消息那么简单。`read_file` 工具的结果包含文件内容；`bash` 工具的结果包含命令输出；`search` 工具的结果包含搜索命中的文件名和片段。每种工具结果的「重点」在不同的地方。

## 搜索文本的优先级

`Messages.tsx` 的搜索逻辑是这样的：

```typescript
const searchText = tool.extractSearchText?.(toolResult) 
  ?? renderableSearchText(message);
```

先看工具是否实现了 `extractSearchText` 方法。如果实现了，用它；没有实现，才退回到通用的 `renderableSearchText()`。

结果缓存在 `WeakMap` 里，避免用户每按一个键都重新提取所有搜索文本。

## 为什么搜索语义属于工具

通用的 `renderableSearchText()` 会提取消息里所有可见文本。对于简单的文本消息这没问题，但对工具结果，「所有可见文本」可能不等于「用户在找的内容」。

比如 `read_file` 的结果可能包含文件路径、行号标记、以及文件内容本身。如果只搜「所有可见文本」，用户搜一个变量名，可能会命中行号旁边的某个数字，而不是变量名真正出现的那行代码。

工具的作者最清楚这个工具的结果里哪些是「有意义的可搜内容」。让工具自己定义 `extractSearchText`，是在把这份知识放回它最自然的归属位置。

## 一个扩展点，不是默认要求

`extractSearchText` 是可选的。大多数工具可以不实现它，直接使用通用文本提取。只有那些「通用提取会产生错误结果」的工具，才需要实现自己的搜索文本逻辑。

这个设计让搜索系统保持简单，同时给每个工具留了一个「我知道我的结果应该怎么被搜」的表达口。
