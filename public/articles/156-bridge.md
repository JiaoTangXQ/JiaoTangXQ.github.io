# 深度分析：bridge 的克制——为什么不传有时比传更难

## 两种 bridge 设计哲学

**最大化传递**：bridge 的职责是尽可能完整地把一侧的信息传到另一侧，让两侧尽量"感觉一样"。实现上，这通常意味着把远端的请求结构直接序列化后传给本地，或者把本地的响应结构直接序列化后传给远端。

**有选择地传递**：bridge 不只是数据通道，而是**主动决定哪些信息应该跨越边界，哪些应该在边界处被过滤或转换**。

Claude Code 的 `remotePermissionBridge.ts` 是第二种设计哲学的体现。

## 不该传的东西：两个例子

**例子一：完整的 tool 对象**

`createToolStub` 函数创建了一个最小化的 Tool 对象：

```typescript
export function createToolStub(toolName: string): Tool {
  return {
    name: toolName,
    inputSchema: {} as Tool['inputSchema'],
    isEnabled: () => true,
    userFacingName: () => toolName,
    renderToolUseMessage: (input: Record<string, unknown>) => { ... }
  }
}
```

注意它缺少了真实 Tool 对象里的很多东西：没有真正的 `call()` 方法实现，没有工具特定的验证逻辑，没有 analytics 追踪，没有特定权限处理。

这些缺失是**有意的**。远端工具本地没有实现，如果试图传递完整的工具对象，会面临：

- 本地不知道怎么实现这个工具（实现在云端容器里）
- 把工具的实现代码从远端传到本地会有巨大的安全风险（远程代码执行）
- 本地运行远端代码需要沙盒，大大增加复杂度

所以 bridge 的选择是：只传工具的"外壳"（名字、展示方式），不传工具的"内脏"（实现逻辑）。

**例子二：真实的 model 信息**

在 `createSyntheticAssistantMessage` 里：

```typescript
model: '',  // 空字符串，而不是真实的模型名
stop_reason: null,
usage: {
  input_tokens: 0,
  output_tokens: 0,
  // ...
}
```

这些字段设为空值，是因为这些信息在远端执行，本地没有，也**不需要有**。

如果非要填入这些信息，选项是：
1. 从远端传过来——增加通信负担，这些信息本地审批链根本不用
2. 猜一个默认值——制造假数据，将来可能被误读

选择"不填"（设为空值或 null）是最诚实的：**告诉系统"我没有这个信息"，而不是假装有**。

## 绝不能被带上路的东西

有些信息不只是"不需要传"，而是"绝对不能传"。

**本地的私钥和 API 密钥**：远端容器不需要知道本地机器上的 API 密钥。如果 bridge 把这些透传，远端容器就能拿到用户的凭证，严重安全风险。

**本地的文件系统结构**：远端容器执行工具，不需要知道本地的文件在哪里（它有自己的文件系统）。如果 bridge 把本地路径透传，远端代码可能尝试访问本地路径，在远端失败，并暴露本地的文件系统结构。

**本地的会话历史细节**：某些会话历史包含用户的私密操作记录（比如他们访问的内部 API 等）。远端容器执行任务不需要看到这些，bridge 应该只传任务执行需要的部分。

## 克制需要主动设计

"知道什么不该传"听起来是负面的（"不做什么"），但实际上是需要主动设计的：

你需要知道每个字段的**使用者是谁**：这个字段是本地审批链用的，还是远端工具用的，还是两边都用？

你需要知道每个字段的**安全属性**：这个字段包含什么类型的信息？可以跨边界传吗？传了有什么风险？

你需要知道**哪些字段可以用空值或 stub 替代**：哪些字段可以用"安全的假数据"代替，让接收方知道"这个信息不可用"而不是崩溃？

这些问题的答案不是自动得出的，需要对系统的两侧都有深入理解。这就是 bridge 设计"需要功力"的地方。

## 面试要点

1. **bridge 是主动过滤而不是被动传递**：决定哪些信息跨边界，哪些不跨，是 bridge 最重要的设计工作。
2. **tool stub 是刻意的信息截断**：只传工具的外壳，不传内脏，避免远端代码在本地执行的安全风险。
3. **空值比假数据更诚实**：没有的信息用 null 或空值表示，不要用猜测的默认值，防止被误读。
4. **"不传"需要主动设计**：知道哪些东西不能传需要理解两侧系统的使用场景和安全需求，不是默认结果。
