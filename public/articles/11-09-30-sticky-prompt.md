# sticky prompt 选最近可见锚点：召回不该比必要的移动更多

`StickyTracker` 要决定「当前视口对应哪个 prompt」时，从 `firstVisible`（第一条可见消息）往回（往历史方向）扫描，找到最近的一条 prompt 类型的消息。

但有一条跳过规则：如果找到的 prompt 已经出现在当前视口顶边附近（基本上就贴着顶边），就跳过它，继续找更前面的那条。

## 为什么跳过已经在顶边的 prompt

如果这条 prompt 已经贴在视口顶边，那么：

1. 用户已经能看到它，sticky header 显示它的内容是多余的
2. 点击 sticky header 跳回这条 prompt，实际上只会移动很少的距离——几乎等于不动

在这种情况下，sticky header 的真正价值是：显示「比当前可见区域更早的上下文起点」。如果 header 显示的 prompt 就在视口顶边，它实际上没有提供任何「往上」的方向信息。

## 「最近可用」的含义

「最近」：离当前视口尽量近，减少点击 header 后视口需要移动的距离。

「可用」：不在视口顶边，能给用户提供真正的「往上找语境」的价值。

这两个约束合在一起，让 sticky header 指向的是「在视口外、但离当前位置最近的上下文起点」，而不是「最远的历史起点」或「就在眼前的 prompt」。

点击后的跳转距离尽量小，但目标是真正有意义的上下文节点。这是 sticky 功能最符合「召回」语义的选择方式。
