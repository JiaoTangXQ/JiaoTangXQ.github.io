# Claude Code 的权限决策链：一个工具调用经过几道检查才能执行？

当模型决定调用 `BashTool` 执行 `rm -rf /tmp/old_build`，这个操作不是立即就执行的。它要经过一条权限决策链，每一层都可能阻止或修改这个操作。

`utils/permissions/permissions.ts` 是这条链的中枢。

## 权限决策的结果类型

```typescript
type PermissionDecision =
  | PermissionAllowDecision
  | PermissionDenyDecision
  | PermissionAskDecision

type PermissionDenyDecision = {
  behavior: 'deny'
  reason: PermissionDecisionReason
  message?: string
}

type PermissionAskDecision = {
  behavior: 'ask'
  // 要弹窗请求用户确认的原因
  reason: PermissionDecisionReason
}
```

决策结果是三选一：允许、拒绝、向用户询问。

## 第一层：PermissionMode

`PermissionMode` 是系统当前的"安全姿态"，优先级最高：

```typescript
type PermissionMode =
  | 'default'           // 默认，敏感操作需要询问
  | 'plan'              // 只能计划，不能执行写操作
  | 'acceptEdits'       // 自动接受文件编辑，不询问
  | 'bypassPermissions' // 绕过所有权限检查（危险）
  | 'dontAsk'           // 不询问，但仍然检查规则
  | 'auto'              // 自动模式（内部使用，基于分类器）
```

`plan` 模式下，任何写操作都会被直接拒绝，不需要看后续规则。`bypassPermissions` 模式下，几乎所有检查都被跳过（需要显式开启，有警告）。

## 第二层：规则匹配

`ToolPermissionContext` 里有三组规则：`alwaysAllowRules`、`alwaysDenyRules`、`alwaysAskRules`。

规则按来源有优先级：

```typescript
type PermissionRuleSource =
  | 'policySettings'  // 企业 MDM 策略，最高优先级
  | 'userSettings'    // ~/.claude/settings.json
  | 'projectSettings' // .claude/settings.json
  | 'localSettings'   // 本地未提交设置
  | 'flagSettings'    // --allowedTools CLI 参数
  | 'cliArg'          // 命令行参数
  | 'command'         // 运行时命令设置的规则
  | 'session'         // 会话内临时规则
```

规则匹配用工具名+可选内容模式匹配：比如 `Bash(git *)` 只允许 git 相关命令，`Bash` 匹配所有 bash 命令。

## 第三层：工具自检

每个 `Tool` 都有 `checkPermissions` 方法，在通用规则之上做工具特定的检查：

```typescript
// Tool 接口定义
checkPermissions(
  input: z.infer<Input>,
  context: ToolUseContext,
): Promise<PermissionResult>
```

比如 `FileEditTool` 会检查目标文件是否在允许的工作目录内，`BashTool` 会检查命令是否包含危险的输出重定向（`>` 写到关键系统文件）。

## 第四层：沙箱检查

`BashTool` 有额外的沙箱机制。`shouldUseSandbox()` 判断当前环境是否需要把 bash 命令放进沙箱执行：

```typescript
// BashTool 特有
if (shouldUseSandbox(input.command)) {
  // 用 SandboxManager 在受限环境里执行
  const sandbox = new SandboxManager()
  // ...
}
```

这一层的存在让 Claude Code 可以支持"高风险环境下更严格的隔离"，而不必完全关闭 bash 能力。

## 第五层：输出重定向检查

`extractOutputRedirections` 解析 bash 命令，检查是否有危险的输出重定向：

```typescript
// 检查类似 `cat sensitive_file > /etc/passwd` 的命令
const redirections = extractOutputRedirections(command)
if (redirections.some(r => isDangerousPath(r.target))) {
  return { behavior: 'deny', reason: 'dangerous_redirect' }
}
```

这种检查无法通过规则层处理（因为规则只能匹配命令前缀），所以单独作为一层。

## 拒绝追踪与熔断

`utils/permissions/denialTracking.ts` 维护了拒绝计数器：

```typescript
type DenialTrackingState = {
  denialCount: number
  lastDenialTime?: number
}
```

如果用户频繁拒绝某个工具的请求，系统会停止弹窗，改为直接拒绝——避免"用户说了 10 次不，Claude 还在问第 11 次"的糟糕体验。

## auto 模式：机器学习辅助权限

如果 `TRANSCRIPT_CLASSIFIER` feature flag 开启，系统会有一个额外的分类器层：

```typescript
const classifierDecisionModule = feature('TRANSCRIPT_CLASSIFIER')
  ? require('./classifierDecision.js')
  : null
```

分类器读取对话历史，判断当前操作是否符合用户的"真实意图"，在用户明确告知意图后自动批准相关操作，减少重复询问。这是"auto"模式的核心。

## 面试角度

这套权限系统的架构思路在面试中可以这样展开："如何为一个能执行任意代码的 AI 设计安全边界？"答案的层次：
1. 全局模式（Policy）：粗粒度控制整体姿态
2. 规则引擎：细粒度匹配特定工具/命令
3. 工具自检：让工具自己知道什么情况下需要特别审慎
4. 行为学习：通过历史减少重复询问，提升体验
5. 熔断降级：拒绝次数过多后自动停止骚扰用户
