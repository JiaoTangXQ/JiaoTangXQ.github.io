# 代码考古：transition.reason 如何让状态机的每次跳转都可以被审计？

`query()` 的主循环是一个 `while(true)`，里面有大量的 `continue` 点。每次 `continue` 代表"这一轮没有真正结束，要重新进入循环继续推进"。

看一个典型的 continue 点：

```typescript
// max_output_tokens 恢复
if (maxOutputTokensRecoveryCount < MAX_OUTPUT_TOKENS_RECOVERY_LIMIT) {
  const recoveryMessage = createUserMessage({
    content: `Output token limit hit. Resume directly...`,
    isMeta: true,
  })

  const next: State = {
    messages: [...messagesForQuery, ...assistantMessages, recoveryMessage],
    toolUseContext,
    autoCompactTracking: tracking,
    maxOutputTokensRecoveryCount: maxOutputTokensRecoveryCount + 1,
    // ... 其他状态字段
    transition: {
      reason: 'max_output_tokens_recovery',
      attempt: maxOutputTokensRecoveryCount + 1,
    },
  }
  state = next
  continue  // ← 跳回循环顶部
}
```

注意 `transition` 字段。它不是用来做任何条件判断的——整个 `query.ts` 里只有一处读取 `state.transition`（用来防止 collapse_drain_retry 无限循环）。其他所有地方，`transition` 只是被设置，不被读取。

那它有什么用？

## transition 是给测试和调试用的可观测接口

看一下 `queryLoop` 返回的 `Terminal` 类型：

```typescript
// query/transitions.ts（推断）
export type Continue = {
  reason:
    | 'next_turn'
    | 'collapse_drain_retry'
    | 'reactive_compact_retry'
    | 'max_output_tokens_escalate'
    | 'max_output_tokens_recovery'
    | 'stop_hook_blocking'
    | 'token_budget_continuation'
  // 可能有额外字段，如 attempt 计数
  attempt?: number
  committed?: number
}
```

`transition.reason` 是一个**枚举类型的诊断信号**。测试代码可以在不解析消息内容的情况下断言"这一轮因为什么原因继续"：

```
// 测试可以做的断言（不需要看消息内容）：
assert(state.transition?.reason === 'max_output_tokens_recovery')
assert(state.transition?.attempt === 2)
```

和对比这种常见的写法：

```typescript
// 没有 transition 的版本
let shouldContinue = false
if (isMaxOutputTokens) {
  shouldContinue = true
  // 再继续...
}
```

用 `boolean shouldContinue` 的方案能知道"要继续"，但不知道"为什么继续"。这在调试生产问题时是个很大的障碍——你看到循环在跑，但看不出是走了哪条恢复路径。

## 所有的 transition.reason 和它们的含义

`query()` 里一共有这些 continue 点，每个都有不同的 reason：

**`next_turn`**
工具执行完成，要把工具结果发给模型，进入下一轮。这是最正常的循环。

**`collapse_drain_retry`**
收到了 API 413（prompt too long），先把暂存的 context collapse 提交掉，再重试。这条路只会走一次——如果 drain 后还是 413，不会再用这个 reason 继续，而是尝试 reactive compact。

```typescript
// 防止无限 drain
if (state.transition?.reason !== 'collapse_drain_retry') {
  const drained = contextCollapse.recoverFromOverflow(...)
  if (drained.committed > 0) {
    state = { ..., transition: { reason: 'collapse_drain_retry', committed: drained.committed } }
    continue
  }
}
```

**`reactive_compact_retry`**
收到 413 或媒体大小错误，触发了应急 autocompact，用压缩后的历史重试。

**`max_output_tokens_escalate`**
模型输出被截断，且这次使用的是默认的低 max_output_tokens（比如 8k）。先把限制升级到 `ESCALATED_MAX_TOKENS`（64k），再用相同的 messages 重试。只会发生一次（用 `maxOutputTokensOverride === undefined` 判断）。

**`max_output_tokens_recovery`**
输出截断，且已经用了最大 tokens 还是被截断（或者 escalate 功能未开启）。向模型发送"直接接着说，不要道歉"的恢复提示，最多恢复 `MAX_OUTPUT_TOKENS_RECOVERY_LIMIT`（3）次。

**`stop_hook_blocking`**
stop hook 执行后产生了阻塞错误（hook 认为模型的输出需要修正）。把阻塞错误写回消息流，标记 `stopHookActive: true`，让模型在"刚才的输出有问题"的新事实下重新生成。

**`token_budget_continuation`**
token budget 功能（跟踪 agent 子任务的 token 消耗）判断预算还有剩余，向模型发送"继续"nudge，让它完成还没做完的工作。

## 为什么不同的 continue 要用不同的 reason？

一个具体的反例：如果 `stop_hook_blocking` 和 `reactive_compact_retry` 用同一个 reason（比如都叫 `retry`），会发生什么？

压缩后重试时 `hasAttemptedReactiveCompact` 会被设为 `true`，防止无限压缩：

```typescript
// reactive compact 后
const next: State = {
  ...
  hasAttemptedReactiveCompact: true,  // ← 防止再次触发
  ...
}
```

但 stop hook 阻塞时需要保留这个值：

```typescript
// stop hook blocking 后
const next: State = {
  ...
  hasAttemptedReactiveCompact,  // ← 保留原来的值，不重置
  ...
}
```

这是源码注释里专门解释的一个细节：

> Preserve the reactive compact guard — if compact already ran and couldn't recover from prompt-too-long, retrying after a stop-hook blocking error will produce the same result. Resetting to false here caused an infinite loop: compact → still too long → error → stop hook blocking → compact → … burning thousands of API calls.

如果两条 continue 路径用同一个 reason，维护这种差异化行为就需要更复杂的外部 flag，而不是直接从 transition 里读出"为什么在这里"。

## 面试指导

这是一道关于"可观测性设计"的好题。面试官可能会问：**如何让一个复杂的异步状态机可被调试？**

枚举类型的 transition 是一个成熟的答案：
1. 不用 boolean flag（失去"为什么"的信息）
2. 不用错误码（没有语义）
3. 用类型安全的枚举，每个枚举值携带该路径特有的上下文数据（如 `attempt` 计数）

这个模式在 Redux action type、状态机库（XState）里都有对应。能在面试中把这个具体实现和"可观测性即设计"的原则连起来，是一个加分点。
