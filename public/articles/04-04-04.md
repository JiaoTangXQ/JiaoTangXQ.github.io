# 为什么这一切要被理解成分层修复链，而不是单一功能

把 Claude Code 的上下文治理写成“它有个压缩功能”，基本等于把最精彩的设计抹平了。它真正做的不是一个功能，而是一条修复链。

链条前面是工具结果预算和 `snip`，先处理最显眼、最局部的问题；中间是 `microcompact`，尽量在不打断会话气质的前提下温和减重；再后面才轮到 `collapse`、`autocompact`、`session memory` 这类更重的结构性整理。每一层都在回答同一个问题，但它们处理的严重程度、破坏范围、时间尺度都不一样。

这就是为什么一定要把它理解成分层修复链。单一功能的想法会逼着系统每次都用同一种手段处理所有压力，而修复链允许它先做小修，再做中修，最后才承认需要大修。Claude Code 的成熟，不在于有多少种压缩，而在于它知道不同程度的问题应该由不同层次的机制来接手。

这也是第四卷和第十卷的分界。第四卷讲的是主循环里这些修复怎样被触发、怎样接力；第十卷再去展开它们背后的连续性世界观。

在代码里，证据就是 `transition.reason` 本身。`collapse_drain_retry`、`reactive_compact_retry`、`max_output_tokens_recovery`、`stop_hook_blocking`、`token_budget_continuation` 这些名字都在说：系统不是在调用一个万能修复器，而是在选择“下一轮用哪一种理由继续”。更直觉的写法是只保留一个 `retry: true` 或 `didCompact: true` 这样的布尔量，把所有续跑都塞进同一条路里。

Claude Code 不这么做，是因为不同失败真的要接不同续写文案、不同历史切法和不同保护逻辑。比如 prompt-too-long 不能和 stop hook blocking 共用同一条重试路，否则会出现压缩失败后又被 hook 加长、再压缩、再被 hook 加长的死循环。代价是状态机更重，但换来的是每一种失败都有自己清楚的后续动作。

## 实现链
`transition.reason` 明确区分 collapse、reactive compact、stop hook、token budget 等续跑理由。

## 普通做法
只用 `shouldRetry` 这类布尔位。

## 为什么不用
因为不同失败需要不同的下一轮动作和保护逻辑。

## 代价
状态对象更重。
