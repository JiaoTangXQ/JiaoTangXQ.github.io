# Session Memory：Claude Code 如何在长会话里自动维护工作笔记？

长会话里，Claude 会忘记早期做过什么。Session Memory 试图解决这个问题——在后台定期把对话里的关键信息提取出来，写入一个 Markdown 文件，在后续请求时作为上下文附件注入。

`services/SessionMemory/sessionMemory.ts` 是这个机制的核心。

## 触发条件：双阈值 AND 逻辑

Memory 提取不是每次对话都触发，而是有明确的触发条件：

```typescript
export function shouldExtractMemory(messages: Message[]): boolean {
  const currentTokenCount = tokenCountWithEstimation(messages)

  // 阶段 1：初始化阈值（首次提取）
  if (!isSessionMemoryInitialized()) {
    if (!hasMetInitializationThreshold(currentTokenCount)) {
      return false
    }
    markSessionMemoryInitialized()
  }

  const hasMetTokenThreshold = hasMetUpdateThreshold(currentTokenCount)
  const toolCallsSinceLastUpdate = countToolCallsSince(messages, lastMemoryMessageUuid)
  const hasMetToolCallThreshold = toolCallsSinceLastUpdate >= getToolCallsBetweenUpdates()
  const hasToolCallsInLastTurn = hasToolCallsInLastAssistantTurn(messages)

  // token 增量阈值是必要条件
  const shouldExtract =
    (hasMetTokenThreshold && hasMetToolCallThreshold) ||
    (hasMetTokenThreshold && !hasToolCallsInLastTurn)

  return shouldExtract
}
```

触发逻辑（注释里明确标注了设计意图）：
- **Token 增量阈值**是必要条件——即使工具调用次数够了，如果 token 增量不够也不触发
- **工具调用次数**是充分条件之一——token 增量 + 工具调用次数足够 → 触发
- **自然对话断点**是另一个充分条件——token 增量足够 + 上一轮没有工具调用（用户在等待，是提取的好时机）→ 触发

这个双阈值设计防止了两种极端：token 增量了但没什么实质工作（纯闲聊）、做了很多工具调用但 context 还很小（刚开始的会话）。

`sequential` 是之前在 `memoize.ts` 里见过的工具——它确保同一个函数的并发调用被串行化。对 Session Memory 来说，如果两次后采样钩子快速连续触发，第二次会等第一次完成再执行，而不是同时触发两个 forked agent 去提取内存。

## 用 forked agent 提取，不打断主对话

```typescript
import { createCacheSafeParams, createSubagentContext, runForkedAgent } from '../../utils/forkedAgent.js'
```

提取操作通过 `runForkedAgent` 启动一个独立的 subagent——这个 agent 有自己独立的 API 调用，不会在主对话里产生额外的消息历史，不会消耗主 context window 的空间，也不会让用户看到"正在提取记忆"这样的中间状态。

## 文件操作的安全细节

```typescript
// 用 wx 标志（O_CREAT|O_EXCL）创建文件——文件已存在时直接跳过
await writeFile(memoryPath, '', {
  encoding: 'utf-8',
  mode: 0o600,   // 600 权限，只有所有者可读写
  flag: 'wx',
})
```

文件创建用 `O_EXCL` 标志（`wx`）——如果文件已存在，创建失败但不报错（`EEXIST` 被吞掉）。这是原子性创建的正确做法：不需要先检查文件是否存在再创建（TOCTOU 竞争），直接尝试创建，失败说明已存在。

文件权限 `0o600`——只有当前用户可读写，防止其他用户读取会话记忆（可能包含敏感信息）。目录权限 `0o700`——更严格，不允许其他用户进入目录。

## 文件缓存失效

```typescript
// 删除 FileReadTool 的缓存条目，确保拿到实际内容而不是"文件未变更"的存根
toolUseContext.readFileState.delete(memoryPath)
const result = await FileReadTool.call({ file_path: memoryPath }, toolUseContext)
```

Claude Code 有文件读取状态缓存（`readFileState`）——如果文件内容没变，FileReadTool 会返回"文件未变更"的简化结果。但这里需要读取最新内容，所以在调用 FileReadTool 前先删除缓存条目，强制重新读取。

## 配置的远程推送

Session Memory 的阈值参数（初始化 token 数、更新间隔 token 数、工具调用次数）来自 GrowthBook 远程配置 `tengu_sm_config`：

```typescript
const initSessionMemoryConfigIfNeeded = memoize((): void => {
  const remoteConfig = getSessionMemoryRemoteConfig()
  const config: SessionMemoryConfig = {
    minimumMessageTokensToInit: remoteConfig.minimumMessageTokensToInit && 
      remoteConfig.minimumMessageTokensToInit > 0
        ? remoteConfig.minimumMessageTokensToInit
        : DEFAULT_SESSION_MEMORY_CONFIG.minimumMessageTokensToInit,
    // ...
  }
  setSessionMemoryConfig(config)
})
```

`memoize` 确保这个初始化只执行一次（第一次运行后，lodash memoize 缓存结果，后续调用直接返回）。远程配置值有 `> 0` 的防御性检查——不允许把阈值设为 0（防止意外配置导致每次都触发提取）。

## 面试角度

Session Memory 的工程设计体现了几个实用原则：

1. **双阈值比单阈值更精确**：只看 token 数（可能都是闲聊）或只看工具调用（可能工作量很小）都不准，两者结合才能准确捕捉"有实质进展"的时刻
2. **自然断点提取优于打断**：在用户等待而 Claude 没有工具调用时提取，是最小化干扰的策略
3. **forked agent 做后台工作**：任何可以异步、不需要用户看到中间状态的操作，都适合用 forked agent 隔离
4. **文件权限是安全边界**：`0o600` 不是习惯性写法，而是真实的隐私保护——会话记忆不应该被其他用户读取
5. **memoize 做一次性初始化**：用 lodash memoize 代替 `let initialized = false` 标志，更简洁，也防止了并发初始化（memoize 在同步代码里天然是原子的）
