# 系统设计题：如果 Claude Code 的技术文档会长到几千篇，你怎么设计它的结构让它不散架？

## 问题的规模感

Claude Code 目前的源码有几十个工具、上百个命令、十几种任务类型、几十个 hooks 触发点，加上 MCP、权限系统、多种会话模式、连续性机制……每一块如果要深入讲清楚，都能写十几篇文章。

如果按照"每个设计决策一篇"的颗粒度持续产出，很快就会有几千篇文档。这时候最危险的事情不是内容不够，而是**读者无法在几千篇里找到他们需要的那一篇**，以及**内容开始大量重复但没有合并**（因为作者不知道另一篇已经讲过了）。

这是一个规模化内容系统设计问题，和软件架构里的"如何管理大型代码库的复杂性"是同构的。

## 借鉴 Claude Code 源码里的解法

Claude Code 自己是怎么管理几万行代码的复杂性的？

**分层**：`entrypoints/`、`tools/`、`tasks/`、`commands/`、`services/`、`utils/` 这些层次让每个文件知道自己在哪个抽象层。一个工具不需要知道命令系统的细节，一个任务不需要知道权限系统的内部实现。层次之间是单向依赖，不是随意交叉。

**统一接口**：`Tool` 类型、`TaskStateBase`、`Command` 类型——所有同类东西共享同一个接口，让任何处理工具的代码都不需要了解具体工具的内部实现，只需要知道接口形状。

**状态中心**：`AppState` 是整个系统状态的单一来源。任何需要读取或更新状态的组件，都通过 `getAppState` / `setAppState` 而不是直接互相引用。这让状态变更变得可追踪，让测试变得可能。

文档系统可以应用同样的三个原则：分层坐标 + 统一格式 + 显式关系。

## 分层坐标：四个层次，每篇文章都有定位

```
卷（Volume）
  └→ 章（Chapter）
       └→ 节（Section）
            └→ 叶子文档（Leaf）
```

- **卷**定义读者当前在哪片"大世界"：工作台架构、具体工具深度、多 agent 协作……
- **章**定义这段主线在讨论哪类问题：入口与模式、会话引擎、工具系统……
- **节**把相关问题拢在一起，让读者有"这几篇是同一组问题的不同切面"的感知
- **叶子**只处理一个小而完整的设计点：一个具体的函数、一个边界条件、一个对比分析

每篇叶子文档都知道自己的完整坐标（`01-04-03-hooks`：第一卷/第四章/第三篇），也知道自己在章节主题里的角色。

这解决了"读者迷路"的问题：就算有几千篇，每篇都有坐标，读者可以通过层次缩小范围。

## 统一格式：结构化元数据

每篇文章都有相同格式的 frontmatter：`slug`（唯一坐标）、`topics`（主题标签，用于跨章索引）、`summary`（100-200 字的问题+答案概述）、`importance`（权重，决定在某些视图里的优先级）。

这不是为了好看，而是让文档系统可以做机械化处理：

- 按 `topics` 聚合：找所有关于"权限系统"的文章，不管它们在哪一卷
- 按 `importance` 排序：推荐"最重要的 10 篇"给新读者
- 用 `slug` 做精确引用：`[toolPermissionContext 详解](01-04-02)` 而不是模糊的"见权限章节"

这类似于 Claude Code 里 `TaskStateBase` 的设计：所有任务类型共享相同的基础字段（id、type、status、startTime、outputFile……），让任何需要处理任务的代码都能用统一的方式操作，不管它是 bash 任务还是 dream 任务。

## 显式关系：防止跨主题内容孤立

Claude Code 里很多设计会同时涉及多个主题：hooks 既属于"制度系统"，又和"工具调用"深度相关；`DreamTask` 既属于"记忆机制"，又是"任务系统"的一个实例；`toolPermissionContext` 既属于"权限系统"，又是 `AppState` 的核心部分。

如果内容系统只靠树形层次，这些交叉内容要么被强制放进一个章节（其他章节找不到），要么被重复写在多个地方（不一致问题出现）。

解决方法是显式关系：每篇文章可以声明"相关文章"、"前置理解"、"深度延伸"。这些关系不是树形的，而是图形的。读者在看完 hooks 那篇时，会被导向"工具调用权限检查"那篇；读 DreamTask 时，会被导向"任务系统概述"和"auto-memory 机制"。

这相当于 Claude Code 里 `toolMatchesName` 函数的作用：当你有一个工具名字，它帮你在所有工具（包括别名）里找到匹配的——连接节点之间的关系，不假设读者已经知道从哪里跳转。

## 叶子独立但有定位

每篇叶子文档应该是**自足的**：读者不需要读完前五篇才能理解第六篇。但它同时知道自己在整棵树的哪里——通过 slug 坐标和 frontmatter 元数据。

这解决了"散架"的核心问题。单篇独立意味着可以从任何地方开始读，有坐标意味着读完之后知道"这只是整个主题的一个切面，还有更多相关内容在这些坐标里"。

类比：Claude Code 里每个工具都是独立的对象（自足），但它们都登记在 `getAllBaseTools()` 返回的列表里（有全局坐标），可以被统一的权限检查、工具搜索、schema 注入系统处理。单工具不依赖其他工具，整体工具系统又是连贯的。

## 面试指导

**考法**：设计一个可以扩展到几千条内容的技术知识库，如何保证它在增长过程中不失去可用性？

**满分答法**：类比软件架构里的同构问题，提出分层坐标（解决迷路）+ 结构化元数据（解决机械可处理）+ 显式图形关系（解决交叉主题）+ 叶子自足（解决强依赖）这四个机制。举 Claude Code 源码里的具体结构做类比（`TaskStateBase`/`Tool` 接口/`AppState`），说明为什么文档系统和软件系统面对的是同构的复杂性管理问题。

**一句话区别高低分**：低分答案说"用好的标签和搜索功能"，高分答案说"设计内容的坐标系和关系系统，让机械处理和人工浏览都能有效工作"——后者才是可以在几千篇规模下持续有效的设计。
