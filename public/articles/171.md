# AppState 为什么是 Claude Code 界面层的单一真相源：架构决策解析

面试题形式：**Claude Code 的界面层是怎么组织状态的？为什么不用多个独立的 React context？**

## REPL 的状态问题

Claude Code 是一个终端 TUI 应用，但它的复杂度远超普通命令行工具：任务列表、权限状态、MCP 服务器连接、swarm 成员、会话上下文、用户设置……这些来自不同子系统的事实都需要在界面上正确展示。

典型的 React 应用会为每类数据建一个 context，但这带来协调问题：当 MCP 服务器连接断开时，权限 context 需要更新，任务 context 也需要更新，任何遗漏都会让界面出现「各说各话」的假真相。

## AppState 的设计

Claude Code 的解决方案是让所有跨子系统的事实汇入同一个 `AppState`：

```ts
// 来自 REPL.tsx 的状态初始化
const setAppState = useSetAppState()
const foregroundedTaskId = useAppState(s => s.foregroundedTaskId)
const foregroundedTask = useAppState(s =>
  s.foregroundedTaskId ? s.tasks[s.foregroundedTaskId] : undefined,
)
```

`useAppState()` 接受一个 selector 函数，组件只订阅自己关心的切片，避免无关状态变化触发重渲染。`useSetAppState()` 返回一个 updater 函数，接受 `prev => next` 函数式更新，确保并发安全。

## 界面是运行时，不是展示壳

这个设计的关键洞察：界面层承担了原本属于「服务层」的协调职责。

`StatusLine.tsx` 里的 `buildStatusLineCommandInput()` 函数体现了这一点：它不只是格式化展示，而是把来自不同子系统的数据——权限模式、token 使用量、会话标题、rate limit 状态、worktree 信息——组装成一个结构化的命令输入对象，通过 hook 系统执行 `executeStatusLineCommand()`。状态栏不是展示结果，它是制度的执行点。

## 状态下沉的代价与收益

代价是 `AppState` 结构体会很大：任务列表、权限上下文、工具列表、MCP 连接状态、swarm 信息、展开视图、选中 agent……每个子系统的状态都在这里有一席之地。

但收益是可靠性：当任何子系统状态变化时，只需要 `setAppState(prev => ...)` 一次，所有订阅了相关 selector 的组件会自动更新，不需要手动协调多个 context。

## 为什么不用 Redux 或 Zustand

`REPL.tsx` 的注释没有解释这个选择，但从代码结构可以推断：Claude Code 的 UI 是 Ink 框架（React for CLI），不是 React DOM，外部状态库的某些特性在 terminal 环境下没有对应场景（如 Redux DevTools）。更重要的是，Ink 应用的组件树比典型 web 应用浅很多，自定义 `useAppState` + `useSetAppState` 足以满足需求，不需要引入额外依赖的复杂度。

## 面试要点

**Q：`useAppState` 的 selector 模式有什么性能优势？**

selector 函数让组件只在自己关心的数据变化时重渲染。比如 `useAppState(s => s.foregroundedTaskId)` 在 `foregroundedTaskId` 未变化时不会触发重渲染，即使 AppState 的其他字段发生了变化。这对于 CLI 应用尤为重要——终端重渲染代价高于浏览器 DOM。

**Q：`setAppState(prev => ...)` 为什么要用函数式更新？**

Claude Code 的很多状态更新来自异步回调（MCP 连接、task 文件变化、hook 执行），函数式更新确保每次都基于最新的 `prev` 状态，避免闭包陷阱导致的状态覆盖。

**Q：AppState 里的 tasks 字段是什么结构？**

`tasks` 是一个 `Record<taskId, TaskState>` 映射，包含了前台任务和后台任务。`foregroundedTaskId` 指向当前在 REPL 主视图展示的任务，后台任务通过 task navigator 访问。
