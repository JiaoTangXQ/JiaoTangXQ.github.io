# Claude Code 的启动为什么在 import 阶段就已经开始了？

**面试官：** 你描述一下 Claude Code 的启动流程吧，从用户敲下回车开始。

**候选人：** 嗯，Node.js 加载入口文件，解析模块依赖树，然后进入 `main()` 函数，开始……

**面试官：** 停。你说 Node.js 加载模块，这个过程要多长时间？

**候选人：** 对于大型工具来说，几十到上百毫秒？

**面试官：** 那这段时间，程序在干什么？

这正是 Claude Code 设计团队思考的起点。`main.tsx` 文件顶部有一段让人印象深刻的注释：

```
// These side-effects must run before all other imports:
// 1. profileCheckpoint marks entry before heavy module evaluation begins
// 2. startMdmRawRead fires MDM subprocesses (plutil/reg query) so they run
//    in parallel with the remaining ~135ms of imports below
// 3. startKeychainPrefetch fires both macOS keychain reads (OAuth + legacy
//    API key) in parallel — isRemoteManagedSettingsEligible() otherwise
//    reads them sequentially via sync spawn inside
//    applySafeConfigEnvironmentVariables() (~65ms on every macOS startup)
```

然后紧接着，在 import 语句中间夹着三行直接调用：

```typescript
profileCheckpoint('main_tsx_entry');
// ...
startMdmRawRead();
// ...
startKeychainPrefetch();
```

这不是失误，这是刻意为之的设计。

## 为什么 import 期间的副作用不算"邪门"？

JavaScript 模块系统允许模块在求值时执行代码——这本来是大家都知道、但很少主动利用的事实。大多数 CLI 工具把这段时间当作不可控的"前奏"，等进入 `main()` 才开始工作。

Claude Code 不这么看。它识别到了一个关键约束：**CLI 工具没有 loading spinner 可以掩护冷启动**。用户敲下命令后看到的是一片空白，直到第一个字符出现。这段沉默时间越长，工具就越显得笨重。

模块求值期间本来就有等待（大量 import 语句串行执行），Claude Code 选择把这段等待"利用起来"——在等待模块加载的同时，并行启动几个必然会用到的慢操作。

## 具体抢跑了什么？

**MDM（移动设备管理）原始读取**：调用 `startMdmRawRead()`，在 macOS 上会起子进程执行 `plutil`，在 Windows 上查询注册表。这类系统级配置读取往往很慢，而且企业环境下几乎一定会用到。把它放进 import 期间并行，能节省约 135ms。

**Keychain 预取**：调用 `startKeychainPrefetch()`，同时启动 OAuth token 和旧版 API key 两路 keychain 读取。注释中说明：如果不这么做，后续 `applySafeConfigEnvironmentVariables()` 会通过同步子进程 spawn 串行读取，每次 macOS 启动要多花 65ms。

两者的共同特征：
1. 本次启动中几乎一定会被消费
2. 可以安全地独立启动（没有前置依赖）
3. 操作本身有明显的等待时间（子进程、系统调用）

## 这不等于"随便往 import 里塞副作用"

**面试官：** 你说这是设计，但我在代码里看到了 `eslint-disable custom-rules/no-top-level-side-effects` 注释。这说明什么？

这说明团队知道这种模式有代价，专门写了 lint 规则来禁止滥用，然后在这几个经过仔细权衡的地方显式豁免。豁免本身就是设计决策，不是逃避规范。

关键区分在于：这两个操作满足"安全并行"的条件——它们是纯粹的 fire-and-forget 启动，结果会在后续真正用到时通过 `ensureKeychainPrefetchCompleted()` 等函数收口。不满足这个条件的操作，就算再慢，也不应该放在这里。

## 面试指导

这道题考查的不是对某个具体 API 的熟悉程度，而是对"启动时间"这个概念的理解粒度。

**低分回答**：把启动等同于 `main()` 函数执行，以为优化启动就是减少 main 里的函数调用数量。

**高分回答**：理解启动是从用户输入命令到程序可交互的整段时间，包括进程启动、模块加载、环境初始化等各阶段。能识别出哪些阶段有"已经存在的等待时间"可以复用。

**加分项**：能说出这种设计的代价——顶层副作用增加了模块的理解难度，读代码时必须同时追踪"什么已经在后台跑着"和"什么还没有结果"两个维度。这是 Claude Code 有意选择接受的工程复杂度，换取的是用户感知到的响应速度。

真正成熟的启动工程，优化的不是某个函数体，而是整段时间河流。
