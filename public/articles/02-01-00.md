# 抢跑导读：启动在模块求值阶段就已经开始

这一章要先提醒读者：Claude Code 的启动不是从 `main()` 里第一句才开始，而是从模块还在求值、用户已经开始等待的那几十毫秒就算数了。

所以这一章的三片叶子，要当成同一条抢跑链来读。先看 [02-01-01-模块刚求值的几十毫秒也值得利用.md](/Users/xxx/tmp/cc/docs/claude-code-book/02-程序怎样启动成一个agent/02-01-程序一加载就开始抢时间/02-01-01-模块刚求值的几十毫秒也值得利用.md)，再顺着昂贵工作前置和抢跑不等于乱并行往下读，这一章才会从“启动优化技巧”变成“程序怎样在一开始就尊重用户等待”的入口。

## 实现链
这一章的实现链都落在 `main.tsx` 最前面那段顶层副作用上：`profileCheckpoint()` 先记时，`startMdmRawRead()` 和 `startKeychainPrefetch()` 在 import 期间就已经启动。后面几页只是分别拆先抢什么、为什么敢抢、以及怎样保证不是乱抢。

## 普通做法
更普通的 CLI 启动会等 `main()` 正式运行后，再一项项读取配置、凭据和环境。模块求值阶段通常被当成纯准备时间。

## 为什么不用那种做法
Claude Code 没把那段时间浪费掉，因为 CLI 的冷启动会直接暴露给用户。顶层 import 本来就有等待，它就把安全可并行的慢操作塞进去重叠执行。

## 代价与回报
回报是启动更快。代价是顶层副作用变多，入口不再像传统主函数那样线性。
