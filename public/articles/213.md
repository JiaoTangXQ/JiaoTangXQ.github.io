# 界面是运行时本体：UI 不只是展示层

面试题版本：**"Claude Code 的 UI 用 React + Ink（终端 UI 框架）构建。在这个系统里，UI 是纯粹的展示层吗？还是它参与了状态管理和运行时决策？有什么具体例子？"**

## 纯展示层的定义

纯展示层意味着：
- UI 只读取状态，不拥有状态
- 所有业务逻辑在后端，UI 只是渲染结果
- 用户输入立即转换为命令，发给后端处理，UI 等待结果

这个模型在简单应用里没问题。但 Claude Code 的 UI 有几个特性打破了这个模型。

## UI 参与状态管理的证据

`src/state/AppState.ts` 是整个应用的核心状态。这个状态不是"后端状态"，而是前后端共享的状态——包括 UI 相关的字段（如 `currentFocus`、`renderMode`）和业务相关的字段（如 `tasks`、`messages`）被放在同一个状态树里。

`src/components/` 下的 React 组件不只是读取 AppState，它们通过 `setAppState` 直接修改部分状态。比如，用户在输入框里输入文字时，`currentInput` 状态是在 UI 组件里维护的，不经过后端。

这不是设计失误，而是 UI 对某些状态有合理的管辖权——输入框的当前内容是 UI 状态，不是业务状态；用户正在查看哪个任务的详情是 UI 状态，不是任务的业务逻辑。

## UI 参与运行时决策的例子

**权限确认对话**：当工具请求需要用户确认的权限时（比如删除文件），`src/components/` 里有权限确认对话。用户在 UI 里点击"允许"或"拒绝"，这不只是改变展示——它直接影响了这次工具调用是否执行，是运行时的关键决策路径。

**中断和取消**：用户按 ESC 键发起中断，这个信号通过 UI 层传递，触发 `abortController.abort()`。UI 不只是展示"用户按了 ESC"，它直接参与了取消工具执行的流程。

**显示模式切换**：用户切换到 "verbose" 模式或 "quiet" 模式，这影响后续工具执行时是否显示详细进度。UI 的状态变更直接影响了运行时的行为。

## Ink 框架的特殊性

Ink 是一个基于 React 的终端 UI 框架。它把 React 的组件模型带到终端，这意味着终端输出是声明式的、由状态驱动的。

这个选择的含义：**终端 UI 和 Web UI 一样，是状态的函数，而不是命令式的输出流**。这让 Claude Code 的 UI 可以在状态变化时自动重新渲染，而不是手动管理"什么时候 print 什么"。

## 落屏作为制度

"界面先被承认为运行时的一部分"有一个重要推论：**落屏（rendering to screen）是一种制度动作，而不只是展示**。

每次状态更新触发重新渲染，渲染结果就是"当前系统状态的官方呈现"。如果渲染的内容和实际状态不一致（比如任务已经完成但界面还在显示"运行中"），这不只是 UI bug，而是系统状态的不一致——因为界面是状态的正式呈现。

这就是为什么 AppState 里的 UI 相关字段和业务相关字段需要保持同步。界面不能独立于系统状态生活——它是系统状态可见性的组成部分。

## 跨子系统事实的共享

Claude Code 同时有多个工作线程：主会话、后台任务、MCP 连接、bridge 连接。这些不同的子系统都需要把状态汇报给 UI，让用户看到统一的全局视图。

如果 UI 只是纯展示层，每个子系统需要主动"推送"状态更新给 UI，这需要复杂的 pub-sub 或 event emitter 机制。

把 UI 作为 AppState 的消费者（React 的 hook 机制），每个子系统只需要更新 AppState，UI 自动响应更新，不需要复杂的消息传递机制。这是"界面是运行时本体"的实际工程收益之一。

## 面试角度的总结

Claude Code 的 UI 是运行时本体而不是纯展示层，体现在：UI 参与状态管理（拥有部分 AppState 字段的管辖权）、参与运行时决策（权限确认、中断信号）、是系统状态可见性的正式呈现（不是快照，而是实时映射）。这个设计让 UI 和业务逻辑共享同一套状态模型（AppState），减少了状态同步的复杂度，但也要求 UI 组件开发者理解自己不只是在"画界面"，而是在维护系统状态的一部分。
