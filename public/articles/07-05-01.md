# 缓存策略：为什么技能去重用 realpath 而不是 inode？

## 问题背景

Claude Code 支持技能从多个来源加载：用户目录（`~/.claude/skills/`）、项目目录（`.claude/skills/`）、策略目录（managed）。一个技能文件可能通过符号链接或重叠的父目录被多次访问到。

如果同一个技能被加载两次，会发生什么？命令名冲突，行为不确定，帮助文本显示两次。这不是我们想要的。

技能去重是必要的。但用什么来识别"同一个文件"？

---

## 为什么不用 inode

最直觉的回答是：用 inode 号。在 Linux/macOS 的 ext4/APFS 上，同一个文件在不同路径访问时，`stat()` 返回的 `ino`（inode 号）是相同的。

但 `loadSkillsDir.ts` 的注释解释了为什么不这么做：

```typescript
/**
 * Gets a unique identifier for a file by resolving symlinks to a canonical path.
 * This allows detection of duplicate files accessed through different paths
 * (e.g., via symlinks or overlapping parent directories).
 * Returns null if the file doesn't exist or can't be resolved.
 *
 * Uses realpath to resolve symlinks, which is filesystem-agnostic and avoids
 * issues with filesystems that report unreliable inode values (e.g., inode 0 on
 * some virtual/container/NFS filesystems, or precision loss on ExFAT).
 * See: https://github.com/anthropics/claude-code/issues/13893
 */
async function getFileIdentity(filePath: string): Promise<string | null> {
  try {
    return await realpath(filePath)
  } catch {
    return null
  }
}
```

inode 在这些情况下不可靠：
- **ExFAT 文件系统**：不支持 inode，所有文件返回 inode 0
- **NFS 挂载**：远程文件系统的 inode 可能与本地 inode 空间重叠，导致误判
- **容器/虚拟文件系统**（如 Docker overlay fs）：inode 精度可能丢失
- **Windows 子系统（WSL）**：inode 语义可能不完全一致

真实 issue #13893 就是因为这些问题触发的。

---

## realpath 的语义

`realpath()` 解析路径里的所有符号链接，返回真实的物理路径。对于"通过不同路径访问同一个文件"的场景，这是最可靠的方法：

```
/home/alice/.claude/skills/review/ → 符号链接 → /shared/team-skills/review/
/home/alice/project/.claude/skills/review/ → 符号链接 → /shared/team-skills/review/
```

两条路径的 `realpath` 都是 `/shared/team-skills/review/SKILL.md`，去重成功。

## 去重在哪里发生

```typescript
type SkillWithPath = {
  skill: Command
  filePath: string
}

// 在 getSkillDirCommands 里：
const seenPaths = new Set<string>()
const deduped = await Promise.all(
  skills.map(async ({ skill, filePath }) => {
    const identity = await getFileIdentity(filePath)
    if (identity === null) return skill  // 文件不存在，跳过去重
    if (seenPaths.has(identity)) return null  // 已见过，去重
    seenPaths.add(identity)
    return skill
  })
)
```

（实际代码结构略有差异，但核心逻辑如此。）

去重用的 `Set<string>` 存的是 `realpath` 结果，而不是原始路径。这样即使同一个技能通过三条不同的符号链接路径被发现，也只会保留第一个。

---

## 失败安全：getFileIdentity 返回 null

```typescript
async function getFileIdentity(filePath: string): Promise<string | null> {
  try {
    return await realpath(filePath)
  } catch {
    return null
  }
}
```

`realpath` 可能失败（文件不存在、权限不足、符号链接断裂）。返回 `null` 时，去重跳过这个文件而不是报错。

这是正确的降级策略：去重失败不等于"技能加载失败"。如果文件真的存在并且可读，它早在 `loadSkillsFromSkillsDir` 阶段就已经被读取成功了；`getFileIdentity` 失败只意味着"无法确定这是不是重复"，保守地保留这个技能比丢弃它更安全。

---

## 与 inode 方案的性能对比

**inode 方案：**
- `stat()` 系统调用，返回文件元数据，包含 inode 号
- 速度快，通常比 `realpath` 少一次或多次路径解析

**realpath 方案：**
- 多次 `lstat()` 调用，逐级解析符号链接
- 速度略慢，但结果跨平台一致

对于技能加载这个场景（启动时一次性执行，文件数量通常几十到几百个），性能差异可以忽略。可靠性更重要。

## 面试指导

**"你遇到过哪些跨平台文件系统的坑？"**

这道题期待的回答不是"Windows 路径用反斜杠"，而是更底层的系统问题。

几个好的切入点：
- inode 在非 Linux 文件系统上不可靠（ExFAT、FAT32 不支持 inode）
- 大小写敏感性：macOS HFS+ 默认大小写不敏感，Linux ext4 大小写敏感
- 符号链接语义：Windows 上符号链接需要管理员权限（或开发者模式）
- 路径长度限制：Windows 传统 MAX_PATH 260 字符限制

`realpath` vs `inode` 这个案例特别好，因为它展示了"看起来正确的方案（inode）在现实中遇到的边缘情况"——这正是工程经验的体现。
