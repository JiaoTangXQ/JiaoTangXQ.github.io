# 插件不是多读一个目录，而是一层正式生态

## 实现链
- `pluginLoader.ts` 处理的不只是“去某个目录读几个 Markdown”，它要面对 marketplace、session-only 插件、manifest、版本缓存、zip 缓存、策略限制和来源校验。
- `loadPluginCommands.ts` 只是插件进入命令系统的一环，前面还有安装、验证、去重、启用状态这些正式生态层才会有的工作。

## 普通做法
- 普通做法会把插件理解成“多一个自定义目录”，用户把文件放进去，系统扫描完就算接入完成。
- 这种方式很轻，也很符合很多脚手架式 CLI 的直觉。

## 为什么不用
- Claude Code 没把 plugins 做成“读目录”功能，因为它要承受第三方来源、版本升级、禁用来源、策略限制和缓存复用这些现实问题。
- 一旦扩展来自市场或远端仓库，代码就必须先管安装与治理，再谈命令内容本身。

## 代价
- 代价是插件层明显更重，调试起来也比本地 skill 麻烦得多。
- 但这正说明 plugins 在这套系统里不是“顺手开放一下自定义入口”，而是一层正式生态。
