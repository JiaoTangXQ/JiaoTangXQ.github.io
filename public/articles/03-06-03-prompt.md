# prompt 命令为什么会成为整套命令系统的灵魂

Claude Code 的命令系统之所以显得特别优雅，很大程度上就因为它没有把命令理解成“立刻做一件事”的按钮，而是发明了 `prompt` 这一层。它让命令第一次有能力去改变当前会话的制度，而不只是触发一次局部动作。

一条 `prompt` 命令能同时做几件事: 它把技能正文塞进这一轮，让模型知道该按什么方式工作；它把额外工具许可附进来，让系统知道这次能放开什么手；它还能注册 hooks、抽出附件、记录技能调用，保证这次工作方式在后面的压缩、恢复和继续里仍然认得出来。也就是说，它改写的不是一条回复，而是接下来一小段会话的行动空间。

这正是它为什么像整套命令系统的灵魂。`local` 和 `local-jsx` 很重要，但它们更像把一件具体事情办掉；`prompt` 不一样，它是在重新摆放“人、模型、系统、工具”这四者之间的关系。对普通读者来说，这恰好就是 Claude Code 和很多聊天壳的真正差别: 它不只是在回应输入，而是在允许命令临时重塑这段协作该怎样继续。

## 实现链
因为它最完整地复用了主循环：命令最终仍然回到 `query()`、工具系统和恢复链，而不是另外发明一套旁路执行引擎。

## 普通做法
很多命令系统会把高级能力拆成本地脚本、模板宏和若干独立 handler，命令越多，内部越碎。

## 为什么不用
它更想把能力继续收编回原来的 agent 运行时，而不是每来一种 skill 就另起一个子系统。

## 代价
prompt 命令会把很多复杂度重新带回主循环，对主循环质量要求很高；但这也正是它能长期扩张而不散掉的原因。
