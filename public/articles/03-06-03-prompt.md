# 面试题：为什么 prompt 命令是整个命令系统里最像 agent 的部分？

## 对比三种命令的「结束方式」

**`local` 命令结束时**：
```
messages: [用户消息, 命令输出]
shouldQuery: false
```
事情做完了，会话继续但不触发推理。

**`local-jsx` 命令结束时**：
```
messages: [用户消息, UI 交互结果]
shouldQuery: options?.shouldQuery ?? false  // 通常是 false
```
UI 交互完成，命令结束，会话继续但通常不触发推理（除非命令特意设置 shouldQuery: true）。

**`prompt` 命令结束时**：
```
messages: [技能正文(isMeta), ...附件, command_permissions]
shouldQuery: true
```
命令「结束」了，但主循环不停——它把新的工作条件装进了会话，然后让推理继续。

这不是命令做完了事情。这是命令为 agent 重新配置了工作环境，然后退出，让 agent 在新环境里运行。

## 最深的连接：复用主循环

`prompt` 命令执行完返回 `shouldQuery: true`，控制流回到 `QueryEngine.submitMessage()`，后者看到 `shouldQuery` 为 true，调用 `QueryEngine.query()`，进入主循环。

这意味着：**prompt 命令的执行结果，是通过 agent 的主循环来实现的，而不是命令系统自己完成的**。

技能说「按这套标准做 code review」，但实际的 review 是 agent 在主循环里完成的——调用工具读代码、推理、返回结果。技能只是配置了这次主循环应该怎么跑。

## 与「旁路执行引擎」的区别

想象一种替代设计：每个 skill 有自己的执行器，调用 skill 就进入这个执行器，执行完再退出。

```
命令系统
  ├── local 执行器
  ├── local-jsx 执行器
  └── skill 专用执行器  ← 平行的旁路系统
```

这种设计的问题：skill 无法使用主循环里已有的工具执行、权限检查、会话压缩、恢复机制——这些都要在旁路执行器里重新实现一遍。

Claude Code 的选择：`prompt` 命令不发明旁路执行引擎，而是通过 `shouldQuery: true` 把控制权还给主循环。这让所有 skill 自动获得主循环里的一切能力：工具执行、权限检查、流式输出、会话压缩……

这是「最像 agent 的部分」的深层原因——`prompt` 命令不自己执行任务，它配置环境，然后让 agent 执行。

## addInvokedSkill 的意义

```typescript
addInvokedSkill(command.name, skillPath, skillContent, getAgentContext()?.agentId ?? null)
```

在 session 状态里记录「这个 skill 被调用了」，让压缩算法知道需要保留这个 skill 的激活记录。

当会话被 `/compact` 压缩后，`buildPostCompactMessages` 会从 `invokedSkills` 里重新构建技能注入消息，让 agent 在压缩后的上下文里仍然「记得」这个 skill 的制度安排。

这说明 `prompt` 命令的影响不只存在于被调用的那一轮——它在 session 状态里留下了可恢复的记录。

## skill 是会话级制度，不是一次性函数调用

总结一下 `prompt` 命令和传统「函数调用」的区别：

| | 传统函数调用 | prompt 命令 |
|---|---|---|
| 执行者 | 函数本身 | 主循环（agent） |
| 生命周期 | 调用完就结束 | 通过 hooks 和 invokedSkills 持续到 session |
| 结果 | 返回值 | 修改后的会话制度 + agent 推理输出 |
| 工具访问 | 需要手动实现 | 自动继承主循环的工具系统 |
| 压缩恢复 | 不支持 | 通过 addInvokedSkill 支持 |

这五个差异共同说明：`prompt` 命令做的是「为 agent 的一次任务执行配置制度」，而不是「执行一个功能函数」。

---

*面试指导：被问到「你们的 skill 系统和其他 agent 框架的 skill/tool 有什么区别」时，从「复用主循环而不是旁路执行」这个角度切入，说明这让 skill 自动继承了 agent 运行时的所有能力，比独立执行器更系统、更可维护。*
