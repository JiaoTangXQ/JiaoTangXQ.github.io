# 为什么统一外观不等于统一内部机制

从用户视角看，`/compact`、`/config`、某个技能命令，外表都只是“打一条斜杠命令”。Claude Code 故意保留这种统一外观，因为入口越一致，学习成本越低。但它没有进一步犯另一个常见错误: 既然外观看起来一样，就硬把内部机制也做成一样。

源码里恰恰相反。`processSlashCommand()` 进入分派以后，会明确走三条完全不同的路。`prompt` 把一套工作方式装进当前会话，再把主循环继续往前推；`local` 由本地程序把事情直接做完，再把结果写回消息；`local-jsx` 则允许拉起一个临时界面，等交互结束后再回到会话。它们的等待方式、返回形态、对界面的占用、是否继续查询模型，全部不同。

这正是 Claude Code 命令系统成熟的地方。它把“统一入口”留给用户，把“不同物理机制”留给系统。前者降低理解成本，后者尊重现实差异。很多系统会在这两者之间二选一，要么入口四分五裂，要么内部被抽象压扁。Claude Code 两边都没放弃，所以它既整齐，又不僵硬。

## 实现链
入口层统一展示为 `/command args`，但到了 `getMessagesForSlashCommand()` 里会按 `command.type` 明确分流，本地结果、临时 UI、prompt 消息和 forked subagent 各走各路。

## 普通做法
最省事的方式是强迫所有命令都走同一执行机制，只保留一种内部实现。

## 为什么不用
因为命令只是用户界面上的统一外观，不代表现实世界里的动作同质。它宁可保留内部差异，也不为了架构整齐牺牲真实需求。

## 代价
从实现者视角看，统一外观反而更容易造成误解，以为所有命令都应该长得一样；但这份不整齐恰恰是诚实。
