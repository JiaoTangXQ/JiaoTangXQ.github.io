# keysRef 增量 push 而不是每次 map 重建——O(1) append 和 O(n) rebuild 的边界在哪？

Claude Code 在流式更新期间每收到一个 delta 就可能追加一条消息，这意味着 `VirtualMessageList` 可能每秒渲染几十次。每次都重建 key 数组的代价是什么，增量 push 解决了什么？

## 问题规模

假设 transcript 有 5000 条消息（一个长会话），streaming 期间每秒追加 10 条新消息。

**全量重建**：每次渲染都执行 `messages.map(m => itemKey(m))`，5000 次 itemKey 调用 + 5000 个字符串加入新数组。每秒 10 次 = 每秒 50000 次 itemKey 调用。

**增量 push**：只对新增的消息调用 itemKey，每次追加 1 条消息 = 1 次 itemKey 调用。每秒 10 次 = 每秒 10 次 itemKey 调用。

差距：全量是 O(n) × 渲染频率，增量是 O(1) × 渲染频率。在 n = 5000、每秒 10 次渲染时，差 500 倍。

## 增量 push 的实现

```typescript
const keysRef = useRef<string[]>([])
const prevMessagesRef = useRef<typeof messages>(messages)
const prevItemKeyRef = useRef(itemKey)

// 在渲染函数体里（每次渲染都执行）
if (
  prevItemKeyRef.current !== itemKey ||           // itemKey 函数变了
  messages.length < keysRef.current.length ||     // 消息数组缩短了
  messages[0] !== prevMessagesRef.current[0]     // 第一条消息变了（前缀不同）
) {
  keysRef.current = messages.map(m => itemKey(m))  // 全量重建
} else {
  // 只追加新增的
  for (let i = keysRef.current.length; i < messages.length; i++) {
    keysRef.current.push(itemKey(messages[i]!))
  }
}
prevMessagesRef.current = messages
prevItemKeyRef.current = itemKey
```

## 三个触发全量重建的条件

**条件 1：`prevItemKeyRef.current !== itemKey`（itemKey 函数变了）**

`itemKey` 是从 `Messages.tsx` 传进来的 prop，用于从消息对象提取唯一 key。如果 `Messages.tsx` 重新渲染时产生了新的 `itemKey` 函数（比如没有用 `useCallback` 稳定），keysRef 里所有 key 都要用新函数重算。

在实践中，`Messages.tsx` 的 `itemKey` 是 `useCallback` 包裹的稳定函数，这个分支很少触发。

**条件 2：`messages.length < keysRef.current.length`（消息数组缩短）**

用户执行 `/clear` 命令或 compact 后，消息数组长度减少。此时 keysRef 里有"未来"条目（原来 #200 的 key 现在可能对应不同的消息），必须全量重建。

**条件 3：`messages[0] !== prevMessagesRef.current[0]`（第一条消息对象变了）**

这个检查比较的是**对象引用**，不是内容。如果第一条消息的对象引用变了，说明 normalizedMessages 的前缀发生了变化（可能是 compact 把前面的消息换成了 summary）。

这个检查是"前缀稳定性"的代理——如果前缀不变（messages[0] 是同一个对象），说明整个数组只是在尾部增长，增量 push 安全。

## ref 而不是 state

`keysRef` 用 `useRef` 而不是 `useState`，有两个原因：

1. key 数组不需要触发渲染——渲染由 `messages` prop 变化触发，key 只是从 messages 派生的辅助数据
2. 如果用 `useState`，每次 push 都会触发额外的重渲，抵消了增量 push 的性能收益

在渲染函数体里直接修改 `keysRef.current`（而不是 setState），是 React 里"在渲染期间维护派生状态"的标准模式。

## 面试指导

这道题考察的是"追加优化"思想在 React 里的应用，也是 append-only 数据结构的工程体现。

**核心决策**：什么情况下可以假设"只是追加"？答：需要验证前缀不变（messages[0] 同一对象）且数组没有缩短（length 不减少）。前缀变化代表底层数据重排了，增量假设失效；数组缩短代表有元素被删除了，keysRef 里有悬空 key。

**加分点**：提到这个优化直接影响 VirtualItem 的 memo 效果。VirtualItem 通过 itemKey 识别，如果 key 不变，React 可以复用对应的 VirtualItem 实例（dom + 状态）；如果全量重建导致 key 在对象上重新映射，React 可能销毁并重建 VirtualItem，丢失内部状态（比如 hover 状态）。

**追问**：keysRef 里用的是什么格式的 key？答：`itemKey(msg)` 的返回值，通常是 `msg.uuid`（消息的唯一标识符）。前面讲过，streaming 消息用 `deriveUUID(contentBlock.id, 0)` 派生出稳定 key，这里是同一套机制的体现。

**和 useMemo 的比较**：`useMemo(() => messages.map(itemKey), [messages, itemKey])` 也能避免每次渲染重建。区别：useMemo 在 messages 数组引用变化时（追加消息时通常产生新数组引用）会重新计算，是 O(n)。keysRef 方案通过检查前缀和长度，把追加情况识别为 O(1)，比 useMemo 更精细。
