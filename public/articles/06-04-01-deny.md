# 事故推演：deny 规则只在运行时检查，会发生什么？

这不是假设题，而是一个真实的工程决策背景。假设有人提议简化代码：把工具池裁剪和运行时检查合并，只保留运行时层，工具装配时不做任何筛选。会发生什么？

## 场景复现

用户在 `.claude.json` 里配置：

```json
{
  “permissions”: {
    “deny”: [“Write”, “Edit”]
  }
}
```

意图是：这个项目只读，不允许写文件。

**方案 A（当前实现）**：装配工具池时，`getDenyRuleForTool()` 发现 `Write` 和 `Edit` 在 deny 名单，把它们从工具列表里剔除。Claude 在规划阶段看不到这两个工具，不会把写文件纳入可行方案。

**方案 B（简化方案）**：工具池里保留所有工具，只在 Claude 真正调用 Write 时才拦截，返回”Permission denied”。

## 方案 B 的问题

问题不在于”能不能阻止写文件”——两个方案都能阻止。问题在于 Claude 的行为：

1. **Claude 规划时把 Write 纳入计划**：它看到 Write 工具，认为它可用，所以设计了”修改 foo.ts，然后运行测试”这样的方案。

2. **第一步就被拒绝**：当 Write 调用被拒时，Claude 不知道这是”工具不存在”还是”这次调用参数不对”。它可能重试，可能换一种方式绕过（比如用 Bash 的 `echo` 命令写文件）。

3. **浪费大量 token 在注定失败的规划上**：在 `Write` 被拒后，Claude 需要重新规划，这消耗了额外的 token 和时间。

4. **安全模型变得”消极被动”**：系统成了一个拦截器，而不是边界声明者。Claude 总是先尝试、被拒再调整，而不是在知道边界的情况下主动避开。

## 方案 A 的实现：工具池筛选

在工具装配时（`tools.ts` 调用 `getToolsForDefaultPreset` 后的过滤步骤），deny 规则生效：

```typescript
// permissions.ts - 工具级 deny 检查
export function getDenyRuleForTool(
  context: ToolPermissionContext,
  tool: Pick<Tool, 'name' | 'mcpInfo'>,
): PermissionRule | null {
  return getDenyRules(context).find(rule => toolMatchesRule(tool, rule)) || null
}

// toolMatchesRule 只匹配没有 ruleContent 的规则（工具级，不是命令级）
function toolMatchesRule(tool, rule): boolean {
  if (rule.ruleValue.ruleContent !== undefined) return false  // 命令级规则不匹配
  const nameForRuleMatch = getToolNameForPermissionCheck(tool)
  if (rule.ruleValue.toolName === nameForRuleMatch) return true
  // MCP 服务器级别匹配...
}
```

当 `getDenyRuleForTool` 返回非 null 时，工具在装配阶段就被排除，不进入 Claude 的可见工具列表。

## filterDeniedAgents：Agent 类型的装配层筛选

还有一个 Agent 类型层面的装配层筛选：

```typescript
export function filterDeniedAgents<T extends { agentType: string }>(
  agents: T[],
  context: ToolPermissionContext,
  agentToolName: string,
): T[] {
  const deniedAgentTypes = new Set<string>()
  for (const rule of getDenyRules(context)) {
    if (
      rule.ruleValue.toolName === agentToolName &&
      rule.ruleValue.ruleContent !== undefined
    ) {
      deniedAgentTypes.add(rule.ruleValue.ruleContent)
    }
  }
  return agents.filter(agent => !deniedAgentTypes.has(agent.agentType))
}
```

这里有一个值得注意的性能优化：以前这个函数对每个 agent 调用一次 `getDenyRuleForTool`，时间复杂度 O(agents × rules)。现在改成先把所有被 deny 的 agent 类型收集到 Set，再统一过滤，O(rules + agents)。

注释写得很直白：”Previously this called getDenyRuleForAgent per agent, which re-parsed every deny rule for every agent (O(agents×rules) parse calls).”

## 运行时层的补充作用

装配层处理工具可见性，运行时层处理**命令级的细粒度拦截**：

```typescript
// permissions.ts - 命令级 deny 检查（运行时）
export function getDenyRules(context: ToolPermissionContext): PermissionRule[] {
  return PERMISSION_RULE_SOURCES.flatMap(source =>
    (context.alwaysDenyRules[source] || []).map(ruleString => ({
      source,
      ruleBehavior: 'deny',
      ruleValue: permissionRuleValueFromString(ruleString),
    })),
  )
}
```

比如 `Bash(rm -rf *)` 这条规则，在装配时没法通过工具级裁剪处理——Bash 工具整体是需要的。它只能在运行时，当具体命令是 `rm -rf` 时被检测到。

## 这道题考什么

**装配层 deny 的必要性**：不能只说”装配层和运行时层都有用”，要能具体描述如果缺少装配层会出现什么行为问题——Claude 的规划、token 消耗、安全模型的主动性。

**ruleContent 的分界线**：装配层的 `toolMatchesRule` 故意不处理有 `ruleContent` 的规则，这是设计意图，不是遗漏。能解释这个细节的候选人说明真正理解了两层结构的分工。

**性能考量**：`filterDeniedAgents` 的 O(agents×rules) → O(rules+agents) 优化，展示了真实系统里即使是权限检查代码也需要考虑高并发下的性能。
