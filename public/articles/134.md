# 深度分析：后台任务完成通知是怎么被结构化的

## 为什么需要结构化通知

后台任务完成时，有两种方式告诉主会话：

**方式一：自然语言**：在对话历史里注入一条消息，"任务 X 已完成，结果是 Y"。AI 读到这条消息，理解了任务完成，继续处理。

**方式二：结构化通知**：注入一条包含 XML 标签的消息，每个关键信息都有自己的标签。AI 通过解析这些标签，精确获取任务 ID、状态、输出位置等信息。

Claude Code 选择了方式二。

**面试题：结构化通知和自然语言通知相比，具体好在哪里？**

## 核心 XML 标签

在 `src/constants/xml.ts` 里（从 `LocalAgentTask.tsx` 的引用可以看到这个文件的存在），有这些常量：

```typescript
// 推断自引用关系
OUTPUT_FILE_TAG        // 输出文件路径
STATUS_TAG             // 任务状态
SUMMARY_TAG            // 阶段摘要
TASK_ID_TAG            // 任务 ID
TASK_NOTIFICATION_TAG  // 通知外层包装
TOOL_USE_ID_TAG        // 工具调用 ID
WORKTREE_BRANCH_TAG    // Git 分支（如果任务在 worktree 里）
WORKTREE_PATH_TAG      // Worktree 路径
WORKTREE_TAG           // Worktree 信息包装
```

一条完整的任务完成通知可能是这样的（推断）：

```xml
<task_notification>
  <task_id>a3f2b8c1</task_id>
  <tool_use_id>tu_xxx</tool_use_id>
  <status>completed</status>
  <output_file>/tmp/claude/tasks/a3f2b8c1.txt</output_file>
  <summary>分析了 47 个文件，发现 3 个潜在的安全漏洞</summary>
  <worktree>
    <worktree_branch>feature/security-audit</worktree_branch>
    <worktree_path>/tmp/claude/worktrees/a3f2b8c1</worktree_path>
  </worktree>
</task_notification>
```

## 每个标签解决什么问题

**`task_id`**：AI 需要能把这条通知和之前的工具调用对应起来。"我在第 N 轮发出了一个 AgentTool 调用，这条通知是那个调用的结果"——这个对应关系靠 `task_id` 建立。

**`tool_use_id`**：工具调用的 ID，用于响应的路由。主会话在收到工具调用结果时，需要知道这是哪个 `tool_use` 块的返回结果，这样才能在对话历史里正确构造 `tool_result` 消息。

**`status`**：任务成功了还是失败了。AI 需要这个信息来决定下一步：成功了继续，失败了可能要重试或者告知用户。

**`output_file`**：完整的任务输出在哪里。AI 收到通知后，如果需要查看完整输出（而不只是摘要），可以通过这个路径读取完整的 transcript。

**`summary`**：简短的任务摘要。AI 可以直接用这个摘要来回复用户，而不需要读取完整输出然后再生成摘要。

**`worktree_*`**：如果任务在一个独立的 worktree 里执行（修改了文件，但还没有 merge 到主分支），AI 需要知道这个 worktree 在哪里，分支叫什么，才能决定是否需要合并变更。

## 结构化 vs 自然语言的实际差距

**精确提取**：AI 从"任务 a3f2b8c1 已成功完成"里提取 task_id，需要用自然语言理解（可能把其他看起来像 ID 的字符串误识别）；从 `<task_id>a3f2b8c1</task_id>` 里提取，是确定性的字符串解析。

**格式一致性**：自然语言描述可能因为 AI 生成的不同而格式不同："任务已完成"、"子 agent 运行完毕"、"已执行完所有步骤"——AI 要处理这些变体。结构化标签永远是同一种格式。

**可扩展性**：需要加新信息时，增加一个新标签，不影响已有标签的解析；自然语言加新信息，可能改变整体句子结构，影响已有信息的提取。

**可测试性**：针对结构化通知写测试很直接（给定这组字段，生成这样的消息，AI 应该做出这样的决策）；自然语言通知的测试要覆盖各种措辞变体。

## XML 的选择

为什么是 XML 而不是 JSON？

Claude Code 的对话历史本身是文本流，AI 能处理夹杂在文本里的 XML 标签，而 JSON 作为独立结构体嵌入文本里会显得更突兀。

XML 标签可以自然地嵌套（`<worktree>` 包含多个子字段），在纯文本消息里也能被正确解析。

另外，XML 里有 `CDATA` 可以包含任意字符（包括可能破坏 JSON 的特殊字符），对于包含代码、路径、任意文本的摘要内容更健壮。

## 面试要点

1. **结构化通知让精确提取成为可能**：task_id、tool_use_id 等关键信息通过标签精确定位，不依赖 AI 的自然语言理解。
2. **每个标签对应一个具体的决策点**：不是随意塞信息，而是 AI 需要哪些信息才能做下一步决策，就设计哪些标签。
3. **output_file 让"通知"和"完整结果"分开**：通知里只有摘要，完整结果在文件里，AI 按需读取，避免通知消息体积过大。
4. **XML 比 JSON 更适合嵌入文本流**：在对话历史的纯文本环境里，XML 标签的嵌入更自然，嵌套更灵活。
