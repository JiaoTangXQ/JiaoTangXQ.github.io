# 第十三卷回收索引：外延执行里统一命名让后端、提示词和渲染共认同一簇能力

## 实现链
外延执行里另一半更关键的，不是能力先长成什么样，  
而是系统能不能用同一种名字把它认回来。

这条线真正逼出来的，是第十三卷里关于骨架接口的判断。  
真正站得住的命名，不是标签，而是让整条链共认同一对象的接口。

这页实际回收的实现链是：
- `12-03-02` 把统一命名写成后端、提示词和渲染一起认出同一簇能力，这会继续支撑 `13-02-01`，说明命名不是标签，而是让整条链认出同一对象的骨架接口。

很多系统会把命名当成给东西贴标签。  
Claude Code 这里更像把名字做成骨架接口，让后端、提示词和渲染都能认出同一簇能力。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
