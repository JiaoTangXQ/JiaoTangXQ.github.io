# 面试题：如何让 LLM Agent 在模型流式输出的同时执行工具？

假设你在开发一个 LLM Agent，调用 Claude API 时使用流式输出。模型流式返回时，中间会出现 `tool_use` 块：

```
流: "让我来帮你分析这段代码... [tool_use: read_file, path='src/main.ts'] 
      另外我还需要查看配置... [tool_use: read_file, path='config.json']
      [model继续生成文字...]"
```

问题是：当你看到第一个 `tool_use` 块时，模型还没有输出完成。你有两个选择：

**选择 A（批处理）**：等模型完全停止输出，收集所有 `tool_use` 块，再统一执行所有工具。
**选择 B（流式并行）**：看到 `tool_use` 块就立刻开始执行，不等模型输出完成。

Claude Code 选了 B，但实现起来比预期复杂。

## 选择 B 的好处：延迟优化

先理解为什么值得选 B。

假设模型生成需要 3 秒，`read_file` 工具执行需要 1 秒：

```
选择 A 的时间线:
[===模型生成 3s===][=工具1执行 1s=][=工具2执行 1s=]
总计: 5秒

选择 B 的时间线:
[===模型生成 3s===]
        [=工具1执行 1s=]  ← 在模型还在生成时就开始
              [=工具2执行 1s=]
总计: ~3.5秒（工具执行基本被隐藏在模型生成时间里）
```

在实际工作中，Bash 命令可能执行 10-30 秒，文件读取可能需要磁盘 IO。如果模型在一次回复里调用了 5 个工具，选择 B 的体验差异是非常显著的。

## 核心实现：StreamingToolExecutor 和查询循环的交互

`query()` 里的关键代码如下（简化版）：

```typescript
// 在流式消费循环里
for await (const message of deps.callModel({ ... })) {
  
  if (message.type === 'assistant') {
    const toolBlocks = message.message.content
      .filter(b => b.type === 'tool_use')
    
    // 看到 tool_use 块就立刻丢给执行器
    if (streamingToolExecutor && toolBlocks.length > 0) {
      for (const toolBlock of toolBlocks) {
        streamingToolExecutor.addTool(toolBlock, message)  // ← 立刻开始执行
      }
    }
  }

  // 每次循环都检查有没有工具已经完成
  if (streamingToolExecutor) {
    for (const result of streamingToolExecutor.getCompletedResults()) {
      if (result.message) {
        yield result.message  // ← 立刻输出工具结果
        toolResults.push(...)
      }
    }
  }
}
```

`streamingToolExecutor.addTool()` 立刻把工具加入执行队列，在后台异步执行；`getCompletedResults()` 是同步轮询，返回已经完成的工具的结果，保持顺序（先加入的工具先返回结果）。

## 选择 B 必须处理的三个问题

### 问题 1：Fallback 时的孤儿工具结果

如果模型输出到一半触发了 fallback（切换到另一个模型重试），已经在执行中的工具会产生"孤儿结果"——这些结果对应的是旧 `tool_use_id`，而新的模型调用会生成新的 `tool_use_id`。

Claude Code 的处理：

```typescript
if (streamingFallbackOccured) {
  // 1. 为已经输出的半截消息生成 tombstone（让 UI 删除它们）
  for (const msg of assistantMessages) {
    yield { type: 'tombstone', message: msg }
  }
  
  // 2. 清空所有状态
  assistantMessages.length = 0
  toolResults.length = 0
  toolUseBlocks.length = 0
  
  // 3. 丢弃旧的执行器，新建一个
  if (streamingToolExecutor) {
    streamingToolExecutor.discard()
    streamingToolExecutor = new StreamingToolExecutor(...)
  }
}
```

`discard()` 告诉执行器放弃所有队列中和执行中的工具，阻止它们的结果被 yield 出来。已经在执行中的工具可能会继续跑完（取决于它是否响应 abort signal），但它们的结果会被静默丢弃。

### 问题 2：中断时必须补全 tool_result

用户按 Ctrl+C 时，如果工具还在执行中，会出现这种状态：有 `tool_use` 块，但没有对应的 `tool_result`。这会导致后续 API 调用报错（API 要求每个 `tool_use` 都有对应的 `tool_result`）。

```typescript
if (toolUseContext.abortController.signal.aborted) {
  if (streamingToolExecutor) {
    // 消耗剩余结果 —— 执行器会为被中止的工具生成合成 tool_result
    for await (const update of streamingToolExecutor.getRemainingResults()) {
      if (update.message) {
        yield update.message
      }
    }
  } else {
    // 没有执行器时的降级方案：手动生成错误 tool_result
    yield* yieldMissingToolResultBlocks(
      assistantMessages,
      'Interrupted by user',
    )
  }
}
```

`getRemainingResults()` 在执行器检测到 abort signal 时，会为还没有返回结果的工具生成一个 `"Interrupted by user"` 的合成 `tool_result`，确保 tool_use/tool_result 配对完整。

### 问题 3：结果顺序必须和请求顺序一致

模型按顺序请求了 `[read_file_A, read_file_B, bash_cmd]`，但实际执行可能是 `bash_cmd` 先完成（文件 IO 慢），`read_file_A` 后完成。

如果按完成顺序返回结果，API 会收到乱序的 `tool_result`，这会破坏 transcript 的逻辑连续性，也可能让模型产生困惑。

`StreamingToolExecutor` 维护了一个**有序队列**（这就是那篇关于它的四状态机的文章详细分析的内容）：`queued → executing → completed → yielded`，保证结果按请求顺序输出，即使实际完成顺序是乱的。

## 什么时候走批处理（选择 A）？

流式工具执行不是总开着的，它被 Statsig feature gate 控制：

```typescript
const useStreamingToolExecution = config.gates.streamingToolExecution
let streamingToolExecutor = useStreamingToolExecution
  ? new StreamingToolExecutor(...)
  : null
```

当 `streamingToolExecutor` 为 null 时，走降级路径：

```typescript
const toolUpdates = streamingToolExecutor
  ? streamingToolExecutor.getRemainingResults()
  : runTools(toolUseBlocks, assistantMessages, canUseTool, toolUseContext)
```

`runTools()` 是批处理路径——等模型输出完成后，统一执行所有工具。

## 面试指导

这道题的设计权衡是：**延迟优化 vs 实现复杂度**。

选择 A（批处理）的系统：
- 实现简单，状态清晰
- 工具执行全部串行在模型输出之后
- 适合 demo 和简单场景

选择 B（流式并行）的系统：
- 更好的用户体验（工具执行被"隐藏"在模型生成时间里）
- 需要处理 fallback 孤儿、中断补偿、顺序保证
- 适合生产环境

面试时，如果面试官追问"如何保证工具结果有序"和"fallback 时如何处理已经在执行中的工具"，你能说出 `tombstone`、`discard()`、`getRemainingResults()` 这些具体机制，就是一个完整的回答。
