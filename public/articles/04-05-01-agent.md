# 为什么成熟 agent 不该等整段话说完才开始动手

如果一个 agent 总要等模型把整段回复完整吐完，再去解析里面的工具调用，再去真正执行动作，它当然也能工作，但会有一种很重的“批处理味”。先说，后做；先生成，后行动。这样的系统看起来像在思考，实际上更像在排队。

Claude Code 在主循环里选了另一种节奏。`query()` 一边接收流式输出，一边把新出现的 `tool_use` 块记下来；如果开启流式工具执行，就立刻把这些工具交给 `StreamingToolExecutor`。后面主循环还会不断检查已经完成的结果，谁先做完，谁就能先把结果回到这轮运行时里。也就是说，模型的“说”和系统的“做”不是两个完全分开的阶段，而是同一轮里彼此咬合的两条节奏。

这件事的价值不只在速度。更深的一层，是系统气质变了。它不再像一个先输出计划、再执行计划的离线程序，而更像一个边观察、边发号施令、边接收回报的现场运行时。用户感受到的是 agent 更灵、等待更少；工程上真正得到的，是工具、附件、任务通知、记忆发现都能更早进入下一轮，从而让整套循环更连续。

当然，这种设计会把复杂度迅速抬高。只要允许边说边做，就必须处理流式回退、孤儿 `tool_result`、中断补偿、顺序一致性这些麻烦。Claude Code 没有躲开这些代价，而是正面把它们做成运行时规则。这也是它比许多“会调工具的聊天程序”更像真正 agent 的地方。

代码里的关键点是：`query()` 在流式消费模型输出时，一旦看到 `tool_use` 块，就把它交给 `StreamingToolExecutor`，不等整条 assistant message 完整落地。更普通的实现是先把模型整段话收完，再统一扫描里面有哪些工具调用，最后一次性执行。这么写简单，也比较像传统请求-响应思路，但会把模型思考时间和工具执行时间完全串起来。

Claude Code 愿意提前开动工具，是为了把“模型还在往后说”和“工具已经开始干活”重叠起来，直接换取交互延迟。代价也不小：一旦流式输出后半段触发 fallback 或中断，系统就得处理 orphaned tool use、tombstone 和 synthetic tool_result，这些复杂度全是为了那份更快的体感付的账。

## 实现链
流式看到 `tool_use` 就交给 `StreamingToolExecutor`，不等整段 assistant 结束。

## 普通做法
收完整段回复后再统一执行工具。

## 为什么不用
因为系统想把模型输出时间和工具执行时间重叠起来。

## 代价
要处理 tombstone、orphan tool use 和 synthetic result。
