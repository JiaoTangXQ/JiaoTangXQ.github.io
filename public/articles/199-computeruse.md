# 第十三卷回收索引：外延执行里 computerUse 先被讲成一整套能力簇

## 实现链
外延执行里真正该先看见的，不是某个动作有多神，  
而是 `computerUse` 本身是不是一整套能力簇。

这条线真正逼出来的，是第十三卷里关于层级秩序的判断。  
真正站得住的系统，不会先堆动作，再勉强解释成整体。

这页实际回收的实现链是：
- `12-02-01` 把 `computerUse` 讲成一整套能力簇，而不是零散动作，这会在 `13-02-01` 里回收到“层级秩序先于局部技巧”，说明系统不是先堆动作，再想办法把它们解释成一个整体。

很多系统先长出动作，再事后给它们起名。  
Claude Code 这里更像先让能力成簇，再让后面的命名、调用和渲染都有东西可接。

## 普通做法
更普通的做法，是把桌面和浏览器能力做成一堆散工具，或者另起一套专用 runtime，让它们按自己的语言各跑各的。

## 为什么不用
Claude Code 不这么做，是因为命名、接口和权限一旦分裂，外部能力就会把原工作台撕成第二套系统，越往后越难收编回来。

## 代价
代价是 wrapper、manifest、bridge、统一命名和适配层都会增加，实现明显更笨重。 对单页回收附录来说，额外成本是页数更多、重复解释也会增加。
