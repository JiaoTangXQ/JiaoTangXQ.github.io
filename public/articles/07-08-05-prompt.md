# 资源刷新连prompt缓存也一起失效说明同一现实不能各刷各的

## 实现链
- `client.ts` 在处理资源变化时，连 prompt 缓存也会一起失效，因为 prompt 文案和可用参数常常依赖同一个后端现实。
- 这不是缓存写得太激进，而是代码承认了一个事实：同源对象一旦底层变化，就不能指望上层提示模板还能一直对。

## 普通做法
- 普通做法是 prompt 缓存单独管，除非用户手动刷新，否则继续沿用旧文本。
- 这样性能更稳，命中率也更高。

## 为什么不用
- Claude Code 不选择高命中率优先，是因为过期 prompt 比少刷新几次网络请求更危险。模型如果拿着旧 prompt 去解释新现实，结果往往是“说得很像，但实际已经错了”。
- 所以它宁可保守一点，把相关缓存一起打掉。

## 代价
- 代价是刷新会更频繁，某些 server 的 prompt 需要重复拉取。
- 但在动态能力系统里，这种代价比状态漂移更便宜。
