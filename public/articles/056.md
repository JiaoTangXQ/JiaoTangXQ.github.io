# 第十三卷回收索引：团队世界里的制度层与组织秩序

## 实现链
这页对应的真实实现链，主要落在 `swarm/*`、`inProcessRunner`、`teammateMailbox`、`leaderPermissionBridge`、pane backend 以及相关状态同步逻辑上。它真正回收的不是“多开几个 agent”这个表面现象，而是团队一旦出现，代码里为什么要先长出身份、通信、审批和空间布局这些组织骨架。

## 普通做法
更普通的做法，是把多 agent 当成共享上下文的并发聊天，或者把团队协作退化成几个后台线程加一个公共 transcript。

## 为什么不用
Claude Code 没走这条路，是因为那样一开始看起来轻，但很快会把身份边界、审批责任和用户视野搅在一起。它宁可多一层 mailbox、leader 和 bridge，也要先把秩序钉住。

## 代价
代价是制度件、目录状态和桥接件会变多，团队世界比普通任务世界更重、更绕，也更依赖清晰的交棒规则。
