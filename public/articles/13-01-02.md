# 为什么"优雅"必须能在出错的时候继续成立？

## 设计哲学问题的真实考点

面试官问"你觉得这套设计哪里优雅"，其实在考一件更具体的事：你所谓的优雅，是只在一切顺利时成立，还是在系统出错、用户犹豫、连接断掉之后，还能维持基本秩序？

两者之间差距很大。

## Claude Code 的优雅发生在哪里

Claude Code 代码里值得称作优雅的地方，不在于 happy path 多简洁，而在于它承认真实世界的三件事：**世界会出错、人会犹豫、系统会超长**——然后在这三件事里都选择了主动处理，而不是甩给用户。

**对"世界会出错"的处理**：`query.ts` 的主循环不假设流式调用一定完整。它针对流式中断、缺失 `tool_result`、工具重试、连接超时都有专门的收尾逻辑。不是等出错了再打一条错误日志，而是提前把这些场景做成正式路径。

**对"人会犹豫"的处理**：`permissionSetup.ts` 不是一个简单的"允许/拒绝"开关。它专门把危险的 Bash、PowerShell 和 Agent 自动放行规则挑出来单独处理，不让 auto mode 在最危险的地方偷懒——因为设计者知道用户在那个时刻可能会犹豫，也可能会被误导放行。

**对"系统会超长"的处理**：`compact.ts` 不是一刀把旧消息删掉。它先去掉图片这种对总结无益却很占 token 的内容，再把技能附件、计划等关键线索重新接回新边界。目标不是缩短，而是缩短后还能继续干活。

**对"新世界进来"的处理**：`claudeInChrome/setup.ts` 默认在非交互场景禁用浏览器能力，只有显式开关才打开。不是默认相信每个新能力都是安全的，而是先收紧，再按需开放。

## 更常见的做法

更常见的实现是把这些现实问题都留给用户自己兜底：权限太危险就靠用户小心一点，流式中断了就重试，context 爆了就手动清空，浏览器桥接连不上就让用户自己猜。这样写起来轻，但每一次出问题都把脏活推回给人。

这不是优雅，是转移成本。

## 优雅的代价

把风险前置到代码结构里，会让系统更厚、更难读。`compact.ts` 里的边界拼装逻辑不漂亮，`permissionSetup.ts` 里的危险规则检测读起来繁琐。但如果把这些东西拿掉，剩下的就只是演示级顺滑，不是产品级优雅。

产品级优雅的标志，是系统在最不该散架的时候没有散架。

## 面试指导

被问到"你觉得这套设计哪里值得学"时，不要只说"分层清晰"或"代码整洁"。

更有说服力的回答是指出：它把风险处理、失败恢复和用户犹豫都做成了代码里的正式场景，而不是把这些留给文档或用户自己兜底。这才是从"能跑"到"能用"之间真正的那道坎。
