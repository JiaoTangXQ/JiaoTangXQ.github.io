# 任务完成要重新编码成结构化通知：后台结果如何回到主会话

面试题版本：**"Claude Code 里一个后台 agent 任务完成后，它的结果是怎么通知给用户的？是直接显示还是经过某种处理？这个处理过程的设计逻辑是什么？"**

## 朴素通知 vs 结构化通知

**朴素通知**：任务完成，弹一个通知："任务 A 已完成"。用户看到通知，切到任务视图查看结果。这是 OS 级别的通知模式。

**结构化通知**：任务完成，把结果编码成 `tool_result` 消息，注入到主会话的消息队列里。下一次主循环运行时，模型看到这条消息，可以基于任务结果做进一步推理。

Claude Code 选择的是结构化通知，而不是朴素通知。为什么？

## 主会话语言的必要性

Claude Code 的主会话是一个多轮对话，每轮都有 `user` 消息、`assistant` 消息、`tool_use` 和 `tool_result` 块。模型基于这个对话历史进行推理。

如果后台任务的结果不是对话格式，而是独立的"通知对象"，模型无法直接在推理中使用这个结果——它看不到通知。要把任务结果纳入模型的推理范围，结果必须以对话格式出现在消息历史里。

把任务结果编码成 `tool_result` 消息，就是把后台世界的产物翻译成主会话的语言，让模型可以像处理任何工具调用结果一样处理这个任务结果。

## 编码的具体形式

任务完成时生成的通知消息大致结构：

```typescript
{
  role: 'tool',
  content: [
    {
      type: 'tool_result',
      tool_use_id: task.toolUseId,  // 对应创建任务时的 tool_use 块
      content: JSON.stringify({
        status: 'completed',
        summary: task.resultSummary,
        outputFile: task.outputFile,
        // 任务的关键结果
      })
    }
  ]
}
```

这条消息的 `tool_use_id` 对应的是当初调用 `TaskCreateTool` 时的那个 `tool_use` 块，这让对话历史里有完整的调用和结果配对。

## pendingMessages 的队列管理

通知消息不是立即插入对话历史，而是先放入 `pendingMessages` 队列。这个队列的作用：

**顺序保证**：主会话可能同时有多个后台任务完成，通知消息按完成顺序排队，不会打乱对话历史的时序。

**轮次控制**：通知消息在下一个合适的轮次时被插入，而不是在当前轮次的中途插入（避免破坏正在进行的推理）。

**去重和限流**：如果多个任务同时完成，可以合并通知，避免在一个轮次里塞入太多任务结果，超过上下文窗口。

## 为什么不直接修改消息历史

直接在消息历史里插入通知消息（而不是通过队列）的问题：

**时序不安全**：正在进行模型推理时修改消息历史，可能导致推理结果和历史不一致。

**破坏历史完整性**：消息历史应该是一个顺序追加的日志，中途插入会打破时序，让历史难以追溯。

**没有批处理能力**：一次只能插入一条，无法在一个轮次里合理处理多个任务同时完成的情况。

## 面试角度的总结

任务结果重新编码成结构化通知，解决的核心问题是：**让后台产生的结果成为模型可以推理的一等公民**。朴素通知只是让用户看到，结构化通知让模型也看到，这让 Claude Code 可以在任务完成后做"基于任务结果的进一步推理"——"任务 A 完成了，我看到结果是 X，接下来应该做 Y"。这个能力让多任务协作工作流成为可能。
