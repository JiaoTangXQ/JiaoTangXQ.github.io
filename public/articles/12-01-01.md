# 面试题：AI Agent 什么时候必须把手伸出终端？

## 问题背景

这是一道 Agent 架构设计题。

假设你在构建一个代码助手 CLI，用户反映它"只会在终端里聊代码，真正的工程任务还是得自己动手"。有人提议：加几个工具，让 Agent 能控制浏览器和桌面。

你的问题是：这只是"多几个工具"吗？还是需要重新思考系统边界？

## 两种不同的系统观

**第一种：CLI 聊天壳**。Agent 本质上是一个增强的 shell 命令，用户说话，它执行命令，吐出结果。终端以外的世界由用户自己处理。

**第二种：工作台**。Agent 是用户完成工作的持续环境，工作发生在哪里，Agent 就该延伸到哪里。

这两种观念的区别，在 `main.tsx` 的启动代码里看得很清楚。Claude Code 的启动期会同时决定：
- 当前的 interaction mode（interactive / non-interactive / print）
- 动态 MCP 配置（包括 computerUse 和 Chrome 能力）
- allowedTools 白名单
- system prompt 追加内容
- 权限预热

这不是 REPL 的启动逻辑，而是工作台的启动逻辑。

## 从工具到制度的跨越

从"多几个工具"到"把外部世界接进来"，中间有一道隐形的台阶。

**工具阶段**：Agent 能调用 `click(x, y)`、`type(text)`、`screenshot()` 这类操作。

**制度阶段**：Agent 知道"谁授权了这场会话"、"这台设备是什么身份"、"断线了如何恢复"、"权限边界在哪里"、"日志往哪里送"。

Claude Code 的 `setupComputerUseMCP()` 做的事情，不是往工具列表里加几个函数，而是生成一整套动态 MCP 配置：

```ts
export function setupComputerUseMCP(): {
  mcpConfig: Record<string, ScopedMcpServerConfig>
  allowedTools: string[]
} {
  const allowedTools = buildComputerUseTools(
    CLI_CU_CAPABILITIES,
    getChicagoCoordinateMode(),
  ).map(t => buildMcpToolName(COMPUTER_USE_MCP_SERVER_NAME, t.name))
  // ...
  return { mcpConfig, allowedTools }
}
```

`allowedTools` 的存在本身就在说：这套能力已经经过了会话级授权（`request_access`），不需要每次操作再单独确认。这不是工具集成，而是权限制度的前置安排。

## 把手伸出去，意味着什么必须跟着出去

外延能力不是零成本扩展。每一种真实设备世界的接入，都会拖出一套必须一起处理的问题：

**桌面控制**：谁授权（`request_access`）、哪些应用可操作（`allowedApps`）、截图尺寸（`screenshotSize`）、操作中断如何处理（`computerUseLock`）。

**浏览器控制**：扩展是否安装（`isChromeExtensionInstalled`）、manifest 有没有铺好（`installChromeNativeHostManifest`）、Native Host 在不在（`chromeNativeHost`）、设备身份是否配对（`pairedDeviceId`）、断线如何重连（reconnect URL）。

更直觉的做法是先把动作加上去，其他问题等出了再说。但 Claude Code 在 `12-01-02` 到 `12-06` 系列花了整整几章的工程量在处理这些"附带问题"。这说明它从一开始就认定：**真正难的不是把动作暴露出来，而是把真实世界接进来以后保证系统不散架**。

## 面试时怎么回答

如果面试官问你"什么时候应该给 Agent 加外延执行能力"，这道题真正要考察的判断是：

**能力边界和工作台边界是否一致？**

如果用户的工作发生在浏览器里、桌面上、真实账号里，而 Agent 只待在终端里，那工作台就是割裂的。Agent 调用的每个工具都只知道一半的现场。

把手伸出去不是炫技，而是在说：这个 Agent 认为自己的职责是"帮用户在真实工作现场完成工作"，而不只是"在终端里提供建议"。

代价是系统更重——权限、安装、连接、恢复、跨平台差异都要自己处理。但如果不这么做，外延能力最后只会是一套需要用户自己拼装的零件，Agent 和真实世界之间永远隔着一道用户手动操作的墙。
