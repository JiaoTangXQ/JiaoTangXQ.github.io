# 把手伸出终端不是炫技，而是工作台的自然外延

如果 Claude Code 真的只是一个终端里的聊天壳，那“把手伸到浏览器和桌面”当然像炫技，因为那会显得像给聊天窗口外挂机械臂。但从 `main.tsx` 一开始就能看出来，它不是这么长出来的。启动期它已经在决定模式、入口、权限、预热和能力边界；往后还有 `Tool`、`Task`、`QueryEngine`、`AppStateStore` 这些正式层级。也就是说，它本来就把自己当成一台要持续做事的工作台，而不是一张只会吐字的终端皮肤。

更普通的 CLI agent 做法，是把外部世界交给用户自己打通：你要桌面操作就另开脚本，要浏览器能力就另接 Playwright，要远端会话就再上一套新界面。那条路实现更轻，但能力一旦越过终端边界，整个产品就会开始裂成多块。

Claude Code 没这么做，所以当它引入 `computerUse` 或 Claude in Chrome 时，重点不是“多几把工具”，而是“把现实设备世界继续纳入同一工作台”。这条路更重，因为你得把权限、任务、日志、恢复和界面都一并想清楚；但如果目标真的是长期工作的 agent，这比把真实行动永远外包给用户更合理。

## 实现链
`main.tsx` 并没有把浏览器和桌面能力当成 CLI 外挂，而是在启动期就把它们并入 `dynamicMcpConfig`、`allowedTools` 和追加 system prompt 的流程里。后面的 query、task 和 transcript 仍然走原主干，只是工具集合变大了。

## 普通做法
更普通的 CLI agent 会把终端外的事情交给用户自己处理，例如再开一套 Playwright、AppleScript 或外部脚本。

## 为什么不用
Claude Code 不用那条路，因为那样动作虽然做出去了，但工作台本身并不知道发生了什么，也接不住权限、恢复和回放。

## 代价
代价是核心运行时必须认识更多宿主现实，比如订阅、交互模式、浏览器安装和系统权限；它不再是一个轻量聊天壳。
