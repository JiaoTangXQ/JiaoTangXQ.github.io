# 代码考古：local_bash 前缀为什么是 'b' 而不是 'l'？

```typescript
const TASK_ID_PREFIXES: Record<string, string> = {
  local_bash: 'b',           // Keep as 'b' for backward compatibility
  local_agent: 'a',
  remote_agent: 'r',
  in_process_teammate: 't',
  local_workflow: 'w',
  monitor_mcp: 'm',
  dream: 'd',
}
```

第一行有一个注释：`// Keep as 'b' for backward compatibility`。

为什么 `local_bash` 用 `b` 而不是 `l`？按照后面几行的命名逻辑，`local_bash` 的缩写理应是 `l`（local）或者 `lb`。但是 `b` 显然来自 `bash`，不来自 `local`。

更重要的问题是：为什么不能改？

## 任务 ID 进入文件路径的那一刻，就再也回不去了

`getTaskOutputPath(taskId)` 是 `diskOutput.ts` 里的一个函数：

```typescript
export function getTaskOutputPath(taskId: string): string {
  return join(getTaskOutputDir(), `${taskId}.output`)
}
```

任务 ID 直接进入文件名。`b8kj2m9x` 这个 ID 对应 `b8kj2m9x.output` 这个文件。

一旦这个文件被创建——或者被写进通知消息、写进日志、写进某个用户的工具调用记录——这个前缀就变成了外部可见的状态。你改了前缀，已有的文件路径就失效了，已有的通知消息里的 `output_file` 字段就指向了不存在的文件。

注释里的 `backward compatibility` 不是指代码内部的向后兼容，而是指**用户数据**的向后兼容。

## 任务 ID 的完整作用域

`generateTaskId()` 生成的 ID 在系统里出现在哪些地方？

**AppState 字典键**：
```typescript
// state/AppState.ts (推断)
tasks: Record<string, TaskState>
// 用 taskId 作为字典键
```

**输出文件路径**：
```typescript
outputFile: getTaskOutputPath(id)
// => /tmp/project/.../tasks/a8kj2m9x.output
```

**通知 XML**：
```xml
<task-notification>
<task-id>a8kj2m9x</task-id>
<output-file>/tmp/.../a8kj2m9x.output</output-file>
</task-notification>
```

**Agent transcript 软链接**：
```typescript
initTaskOutputAsSymlink(agentId, getAgentTranscriptPath(asAgentId(agentId)))
// 创建 a8kj2m9x.output -> agent_transcript_xxx.jsonl 的软链接
```

**AbortController 注册表**（推断）和各种 map 里的键。

这个 ID 一旦离开进程——写进磁盘、写进通知消息、被用户工具读取——就变成了需要长期维护的外部接口。

## 为什么不用数组下标或时间戳

容易想到的替代方案：用自增数字（`task_1`、`task_2`），或者时间戳（`task_1712567890123`）。

**自增数字的问题**：跨进程不安全。Claude Code 支持多个会话并行运行，如果每个进程各自维护计数器，ID 就会碰撞。

**时间戳的问题**：毫秒级时间戳在同一毫秒内可能有多个任务创建，不够唯一。更重要的是，时间戳不携带类型信息，调试时没有 `a` / `b` 这种一眼可辨的前缀优势。

**UUID 的问题**：前面一篇已经分析过。安全性远超需要，但可读性差，大小写不安全。

## 前缀的降级处理

```typescript
function getTaskIdPrefix(type: TaskType): string {
  return TASK_ID_PREFIXES[type] ?? 'x'
}
```

`?? 'x'` 是防御性降级：如果某种 `TaskType` 没有对应前缀（比如将来加了新类型但忘了加前缀），就用 `x` 作为通用前缀，而不是 crash。

`x` 本身也是一个语义选择——它在调试时意味着"未知类型"，足够醒目，不会被误认为是某种正常类型。

## 工程故事的启示

`// Keep as 'b' for backward compatibility` 这行注释背后是一个很常见的工程故事：

**一个早期决定**（用 `b` 作为前缀，可能因为当时这类任务直接叫 `bash_task`）**被固化为外部接口**（ID 进入文件路径），然后**再也改不动了**（改了会破坏用户数据和日志）。

好的代码留下这样的注释，是因为作者知道：如果不注明原因，下一个维护者会"修复"这个看起来不一致的命名，然后破坏掉某些用户的数据，花半天时间 debug 才发现这里改不动。

## 面试要点

被问到"如何设计系统内 ID"时，一个成熟的回答应该包含：

1. **ID 的作用域**：ID 只在内存中用，还是会写入磁盘、出现在 API 响应、进入日志？作用域越广，越难改变。
2. **向后兼容性**：一旦 ID 格式成为外部可见接口，它就需要长期维护。
3. **可读性和可调试性**：带类型前缀的 ID 在 debug 时价值极高，`a8kj2m9x` 一眼就知道是 agent 任务。
4. **安全需求**：是防暴力攻击就够，还是需要加密级随机性？

这些维度没有统一答案，但能清晰表述这些 tradeoff 的人，比背 UUID vs ULID 优劣的人更有工程判断力。
